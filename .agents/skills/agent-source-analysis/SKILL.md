# Agent 源码分析 Skill

## 核心理念：从对话框到 Agent 的质变

AI 从一个"Chat 对话框"变成能自主执行多轮任务、调用工具的 Agent，核心不是模型变聪明了，而是**外部套了一个循环**。

分析源码时，始终围绕这个叙事主线：
1. **没有这个机制之前**，AI 只能一问一答
2. **引入这个机制之后**，AI 获得了什么新能力
3. **源码中哪一行/哪一个设计**实现了这个质变

每篇笔记都要让读者（也就是未来的你自己）能回答："为什么加了这行代码，AI 就不再只是一个对话框了？"

## 分析工作流：宏观架构 → 跨工具比较 → 具体源码 → 引擎映射

### Step 1：确定分析目标

确认用户想分析的模块，常见模块映射：

| 用户提到的概念 | 对应源码文件（Claude Code） | 对应源码文件（Kimi CLI） |
|--------------|--------------------------|------------------------|
| Agent Loop / Query Loop | `src/query.ts`, `src/QueryEngine.ts` | `src/kimi_cli/wire/types.py`, `src/kimi_cli/llm.py` |
| Tool System | `src/Tool.ts`, `src/tools.ts` | `src/kimi_cli/tools/*.py` |
| MCP | `src/services/mcp/` | `src/kimi_cli/mcp_server/` |
| Context / Memory | `src/context.ts`, `src/constants/prompts.ts` | `src/kimi_cli/session.py` |
| Multi-Agent | `src/coordinator/coordinatorMode.ts`, `src/utils/swarm/` | `src/kimi_cli/subagents/` |
| UI / 事件系统 | `src/ink/`, `src/state/AppState.ts` | `src/kimi_cli/tools/display.py`, `src/kimi_cli/wire/types.py` |
| Permission / 安全 | `src/hooks/toolPermission/`, `src/utils/bash/bashSecurity.ts` | `src/kimi_cli/approval_runtime/` |

本地源码的绝对路径如下：
- Claude Code: `D:/workspace/claude-code-main/`
- Kimi CLI: `D:/workspace/kimi-cli-main/`
- 系统报告: `D:/workspace/Claude Code 0331 系统报告 - 手工川 - 20260401.pdf`
- 社区 Wiki: https://ccwiki.pages.dev/

分析时直接引用这些路径。如果文件缺失再提示用户。

### Step 1.5：本地源码验证门（反幻觉检查）

**这是防止分析沦为"纸上谈兵"的最关键步骤。**

在对任何模块做出比较结论之前，你必须先执行以下动作：

1. **读取本地源码**：根据 Step 1 的文件映射，先读取 `D:/workspace/claude-code-main/` 和 `D:/workspace/kimi-cli-main/` 中的对应文件
2. **以源码为准**：如果本地源码与已有笔记/结论矛盾，**必须推翻已有结论**，以本地源码为准
3. **禁止复读未经核实的结论**：禁止在没有读取本地源码的情况下，直接复述、引用或修正已有笔记中的"源码结论"
4. **追问黑盒**：如果某功能在 Kimi CLI 的 Agent Loop 层不可见，必须继续追问：它是否下沉到了 `kosong` 子包？是否在 `chat_provider` 层实现？必须追查到源码可见的最深层

> [!warning] 反面教材
> 之前写笔记时，因为**没有先读本地 `packages/kosong/src/kosong/_generate.py`**，就默认"Kimi CLI 等 LLM 说完再执行工具"，把实现层级差异歪曲成了能力缺失。这个错误直到用 GitHub MCP 读到 `kosong` 源码才被纠正。
> 
> **如果当时先执行了这一步，这个错误根本不会发生。**

### Step 2：宏观视角——这个架构是为了解决什么问题？

**不要一上来就钻进某个文件的某一行代码。**

先回答以下三个问题：
1. **如果没有这个机制，Agent 会怎样？**（描述瓶颈和失败场景）
2. **这个问题在游戏引擎里会以什么形式出现？**（建立跨领域直觉）
3. **业界共识的解决方案是什么？**（抽象出与具体工具无关的架构模式）

这个层面的输出应该让读者不看任何一行源码，也能理解"为什么需要这个东西"。

### Step 3：跨工具比较——不同 Agent 工具做了哪些不同的选择？

同一个架构问题，不同工具（Claude Code、Kimi CLI、Cursor、Devin 等）会有不同的实现策略。**这些差异往往反映了产品定位、技术约束或设计哲学的不同。**

比较时必须覆盖：

| 比较维度 | 说明 |
|---------|------|
| **共同解决什么问题** | 这个模块在所有 Agent 中的核心职责是什么 |
| **Claude Code 的选择** | 它的设计倾向、关键抽象、取舍 |
| **Kimi CLI 的选择** | 它的设计倾向、关键抽象、取舍 |
| **为什么会不同** | 是语言/平台差异？是产品定位？是架构演进阶段？ |
| **对引擎设计的启示** | 如果你要自己做，应该借鉴哪一种？还是融合两者？ |

> 注意：源码引用用于佐证比较结论，而不是比较本身的目的。

### 横向对比的诚实性原则

分析时必须避免把"实现层级差异"歪曲为"能力有无"。以下是强制检查清单：

| 检查项 | 说明 |
|-------|------|
| **本地源码优先** | 在做出比较结论前，必须已读取本地对应源码文件（`D:/workspace/claude-code-main/` 和 `D:/workspace/kimi-cli-main/`）。已有笔记的结论只能作为"待验证假设"，**不能作为事实依据**。 |
| **避免二元对立** | 禁止用"A 有、B 没有"描述差异。必须改为"A 在 [Layer X] 显式实现，B 在 [Layer X] 不体现/封装在 [Layer Y]"。 |
| **黑盒诚实** | 如果 B 把某能力下沉到外部库/闭源模块，不要断言"B 没有该能力"。应说明"我们无法在 B 的源码中确认同等实现，这是封装策略差异"。 |
| **追问等价设计** | 当 A 有某个特性时，必须追问：B 是否通过不同机制解决了同样的问题？即使没有，也要说明"B 的替代方案是什么"。 |
| **解释选择原因** | 不要只描述差异，要解释"为什么 Claude Code 选择显式控制，而 Kimi CLI 选择外包/下沉"——通常与产品定位、性能要求、可维护性、多客户端支持有关。 |

**反面示例**（禁止）：
> Claude Code 使用 `StreamingToolExecutor` 实现流式工具调用，而 Kimi CLI 没有类似设计，它的 LLM 和 Tool 调用是串行的。

**正面示例**（推荐）：
> Claude Code 在 Agent Loop 层显式实现了 `StreamingToolExecutor`，让循环可以在消费 LLM 流式输出的同时提前派发工具。Kimi CLI 则把 LLM 调用和工具执行整体封装在 `kosong.step()` 中（`kosong` 是 Kimi CLI monorepo 中的开源子包，也是独立的 LLM 抽象层库 `MoonshotAI/kosong`，你可以把它想象成一家"中央厨房"——负责买菜、切菜、炒菜、装盘全套流程）。根据 `kosong` 源码（`packages/kosong/src/kosong/_generate.py` 与 `__init__.py`），`generate()` 在消费 LLM 流式输出时一旦解析出完整的 `ToolCall`，就会立即回调 `on_tool_call` → `toolset.handle()` 执行工具，并通过 `ToolResultFuture` 异步回传结果。这意味着 **Kimi CLI 的 Agent Loop 层不负责显式调度流式工具**，但 `kosong` 子包内部已经完成了同等优化。两者的差异本质是**控制粒度与分层抽象的选择**：Claude Code 作为单一 TUI 应用追求端到端的调度可控性；Kimi CLI 作为多客户端服务端架构，把 LLM 交互细节下沉到库层，以降低 Loop 复杂度。

### 黑盒概念的首次出现规范

笔记中反复出现 `kosong`、`wire`、`AsyncLocalStorage` 等概念，但读者（包括未来的你自己）可能完全不了解它们是什么。**禁止在没有任何解释的情况下，直接把外部库或专有名词当作已知前提进行推理。**

任何外部库、闭源模块、专有名词**首次出现**时，必须同时给出以下三件套：

1. **它是什么**（一句话定义）
2. **它像什么**（生活化比喻）
3. **它在架构中扮演什么角色**（为什么这里要用它）

**反面示例**（禁止）：
> Kimi CLI 把 LLM 调用外包给 `kosong` 库，自己只负责点菜和上菜。

**正面示例**（推荐）：
> `kosong` 是 Kimi CLI monorepo 中的开源子包（`packages/kosong`），也是独立的 LLM 抽象层库 `MoonshotAI/kosong`（你可以把它想象成一家"中央厨房"）。它的职责是把"调用 LLM API、解析流式字节、执行工具调用、格式化返回结果"这些脏活累活全部包下来。Kimi CLI 的 Agent Loop 层不需要关心 LLM 的 HTTP 请求怎么发、流式响应怎么解析，只需要像餐厅老板一样，对 `kosong.step()` 说一声"来一单"，等它把做好的菜端上来即可。

> [!warning] 如果你说不出来它像什么
> 那就说明你自己也没真正理解这个外部库。此时有两个选择：
> 1. **先做一篇专门的前置概念笔记**（如 `kosong-架构解析.md`），在索引中标记为必读前置；
> 2. **在本次笔记中插入一个 `> [!info] 概念插播：XXX` callout**，至少给出直觉版解释。

### Step 4：具体源码——用伪代码揭示核心设计

在解释具体实现时，**优先使用伪代码或极简示意图**，而不是大段真实源码。

- 伪代码的好处是：剥离语言细节，凸显控制流和数据结构
- 只有在需要证明"这个工具确实是这样做的"时，才引用真实源码的 5~10 行关键片段
- 真实源码必须标注文件路径和行号

### Step 5：三层剥离法（验证性阅读）

当需要深入某个具体实现细节时，按以下顺序：

1. **接口层**：这个模块对外暴露了什么函数/类/事件？输入输出是什么？
2. **数据层**：核心数据结构长什么样？状态如何流转？
3. **逻辑层**：关键分支、循环、错误恢复策略是什么？

> 参考 `.agents/skills/game-engine-analysis/SKILL.md` 中的三层剥离法。

### Step 6：提取必须穿插解释的概念并映射到引擎

在分析源码的过程中，必然会遇到用户（学习者）可能不熟悉的前置概念。不要假设用户已经懂，要主动在合适的时机引入。

**更重要的是：每引入一个新概念，都要立刻审视它对 `Notes/SelfGameEngine/` 现有设计的影响。**

SelfGameEngine 是你的引擎设计知识库，Agent 源码分析的目的不是在里面新建源码笔记，而是**用源码中获得的真实洞察，回头检查、修正、完善已有的引擎设计**。

具体做法：
1. 解释概念在 Claude Code / Kimi CLI 中的实现
2. 立刻对照 `Notes/SelfGameEngine/` 中已有的相关设计
3. **找出以下问题并执行修改**：
   - 已有设计是否遗漏了这个概念的关键细节？
   - 已有设计是否与源码中的最佳实践相矛盾？
   - 是否可以借鉴源码中的具体实现来优化引擎设计？

### 2. 核心机制伪代码

用伪代码或极简代码描述该模块的核心逻辑：

```
function agent_loop(messages, tools):
    while true:
        response = stream_llm(messages, tools)
        
        for chunk in response:
            if chunk is tool_call:
                spawn_tool_executor(chunk)  // 并行执行
            yield chunk  // UI 实时更新
        
        results = collect_tool_results()
        if no results: break
        
        messages.append(response)
        messages.extend(results)
    
    return messages
```

### 常见概念的标准解释与引擎审视

以下概念在分析 Agent 源码时会反复出现，可以直接引用或改写。每个概念后面都跟一个引擎映射问题：

**AsyncGenerator / yield**
> 直觉版：想象一个广播电台，主持人不是一次性把整期节目录好给你，而是边说边播。你听到一段，他再讲下一段。
> 
> 为什么 Agent Loop 要用它：因为 LLM 的回复是流式的，工具调用可能在回复中途就被触发。`yield` 让 Loop 可以在"听到一半"时就采取行动，而不是傻等全部说完。
> 
> **引擎审视**：打开 `Notes/SelfGameEngine/从零开始的引擎骨架`，检查其中 **AgentBridge** 和 **确定性 Tick** 的设计描述。源码中的 `yield` 事件流表明，Agent 与引擎的通信应该是一个**流式协议**而非"发一个命令、等一帧结束"。如果原笔记假设 Agent 只能在帧边界观察/干预，就需要修正为：Agent 可以在 Tick 中途通过事件流插入操作。

**Function Calling / tool_calls**
> 直觉版：LLM 本身不会读文件，但它可以"假装自己有一个秘书"，写一张纸条告诉秘书："去把 `README.md` 的内容拿来。"这张纸条就是 `tool_call`。
> 
> 为什么需要它：这是把 LLM 的"想法"连接到外部世界的唯一桥梁。
> 
> **引擎审视**：打开 `Notes/SelfGameEngine/从零开始的引擎骨架`，检查 **自描述注册表** 和 **Mutate 边界** 的设计。源码中 `Tool` 对 `inputSchema` 的严格要求说明：AI 生成工具调用的可靠性完全取决于**反射信息的完整度**。如果原笔记对注册表的描述过于简略（比如只提到"有名字和字段类型"），需要补充：必须包含字段约束、默认值、依赖关系等元数据，否则 AI 会频繁生成非法调用。

**Prompt Caching**
> 直觉版：就像你点外卖，如果地址和昨天一样，平台可能会给你打折，因为它不需要重新录入地址。
> 
> 为什么重要：Agent 的 system prompt 和上下文通常很长且重复，缓存可以让长对话的费用降低 90%。
> 
> **引擎审视**：打开 `Notes/SelfGameEngine/从零开始的引擎骨架`，检查 **快照（Snapshot）** 的设计。上下文压缩与快照本质上都是"用更少的数据代表完整状态"。如果原笔记假设快照就是"全量状态拷贝"，那么根据 Claude Code 的三层压缩策略，需要修正为：快照也应该支持**增量快照 + 差异合并**，而非每次都保存完整世界状态。
> 
> 同时思考：引擎是否也应该有一份类似 `CLAUDE.md` 的 `ENGINE_AI.md`，把引擎架构、组件约定、常用模式作为"静态上下文"预加载给 AI？

## How 部分通俗易懂写作规范

用户反馈现有笔记的通病是：**Why 部分引人入胜，How 部分却名词堆砌、让人云里雾里**。以下是强制写作规范，后续所有 Agent 源码分析笔记的 `How` 部分必须遵守。

### 规范 1：对比表格必须带"人话翻译"

在抛出任何对比表格之前，**先给一个生活化的总体比喻**（如上面的"快递公司"比喻）。表格之后，必须有一段文字把表格内容"翻译"成日常语言：

```markdown
> **用人话讲**：Claude Code 之所以把所有逻辑塞在一个 `while(true)` 里，是因为它想让自己像一个"本地应用"——开机即跑，不依赖外部服务；而 Kimi CLI 把 LLM 调用外包给 `kosong` 子包（`kosong` 是 Kimi CLI monorepo 中的开源 LLM 抽象层库，你可以把它想象成一家"中央厨房"——负责买菜、切菜、炒菜、装盘全套流程），自己只负责点菜和上菜，这样前端（UI）和后厨（LLM）可以独立升级。
```

### 规范 2：每个技术名词首次出现时必须"三件套"解释

任何一个在 `Why/What` 部分没有出现过的新名词，首次出现在 `How` 部分时，必须同时给出：
1. **它是什么**（一句话定义）
2. **它像什么**（生活化比喻）
3. **不用它会怎样**（反面教材）

**反面示例**（禁止）：
> Claude Code 使用 `AsyncGenerator` 来管理流式工具执行。

**正面示例**（推荐）：
> Claude Code 使用 **`AsyncGenerator`**（异步生成器）——你可以把它想象成广播电台的"边说边播"模式。主持人不是一次性录完整期节目给你，而是说一段、播一段。如果不用它，AI 就必须等 LLM 把一整段话全部"录完"才能开始执行工具，用户就要干等着。

### 规范 3：对比维度必须附带"反面场景"

每讲完一个设计选择，必须追加一个 callout 说明"如果不这样做，系统会怎么崩溃"。

```markdown
> [!warning] 如果不这么做会怎样？
> 如果 Claude Code 不用 `StreamingToolExecutor`，而是等 LLM 全部说完再执行工具，那么"模型生成时间"和"工具执行时间"就会串行相加。用户本来可以边看到 AI 的思考边等工具跑完，结果变成了"先发呆 5 秒看 AI 写字，再发呆 10 秒等工具执行"。
```

### 规范 4：源码印证必须"先讲人话，再贴代码"

在引用任何真实源码片段之前，必须先用人话概括"这段代码在干什么"。引用之后，必须解释"为什么这里要这样写"。

**强制结构**：
```markdown
#### Claude Code：扁平循环与 StreamingToolExecutor

**这段代码在做什么**：Claude Code 的核心循环 `queryLoop` 就像一个不停转圈的水车。每一圈，它先向 LLM 要一段流式回复；在回复还没结束时，如果发现 AI 已经"写好了"一张工具调用纸条，就立刻把纸条交给 `StreamingToolExecutor` 去执行；执行过程中产生的进度也实时 `yield` 出去给用户看。

```typescript
// D:/workspace/claude-code-main/src/query.ts:241
async function* queryLoop(...) {
  ...
}
```

**为什么这样设计**：把所有控制权集中在一个生成器里，最大的好处是"没有中间商赚差价"。状态更新、流式消费、UI 通知都在同一个函数里完成，调试时只需要看这一个地方。
```

### 规范 5：禁止"清单式名词罗列"

**禁止**出现以下写法：

> Claude Code 使用 `queryLoop`、`QueryEngine`、`microCompact`、`autoCompact`、`reactiveCompact`、`compact_boundary`、`State` 对象等机制来实现上下文管理。

**必须**把概念拆成"一组一条"，每条都带解释：

> Claude Code 的上下文管理像一个"三段式垃圾处理站"：
> 1. **`microCompact`**（微压缩）：直接清空旧工具结果，相当于把过期报纸扔进垃圾桶，动作最快
> 2. **`autoCompact`**（自动压缩）：如果垃圾还太多，就派一个人（LLM）把旧报纸摘要成一句话梗概
> 3. **`reactiveCompact`**（应急压缩）：如果垃圾桶突然爆了，紧急把最不重要的东西丢出去

### 规范 6：从"What"到"How"必须有过渡桥

在讲完 `What` 进入 `How` 之前，**必须**插入一个过渡段落，告诉读者"接下来我们要看什么"。

```markdown
---

前面我们搞懂了 [Agent Loop] 的本质是一个"套在 LLM 外面的事件驱动循环"。现在我们要回答的问题是：**这个循环在 Claude Code 和 Kimi CLI 的源码里，到底长什么样？** 我们将分别从"整体架构对比"、"核心流程伪代码"、"关键源码印证"三个层面展开。不要担心，我会先把代码翻译成日常语言，再贴出真正的源码。

---
```

## 特别叮嘱

- **本地源码优先验证**。在做出任何源码比较结论之前，必须先读取 `D:/workspace/claude-code-main/` 和 `D:/workspace/kimi-cli-main/` 中的对应文件。**已有笔记的结论只能作为"待验证假设"**，禁止在没有读源码的情况下直接复述或修正。如果你因为"偷懒"或"信任既有结论"而跳过这一步，你写出的分析很可能只是幻觉。

- **不要假设用户懂 LLM**。所有专业术语首次出现时，必须用括号或 callout 给出直觉解释。
- **每篇笔记必须有一句话总结**。这是费曼学习法的核心——如果你不能用一句话讲清楚，说明你还没真懂。
