## 核心方法论：渐进式 AI 源码分析

Piccolo 虽小，但五脏俱全。分析必须遵循 **"由外向内、先广后深"** 的原则：

```
构建系统(1) → 编辑器入口(2) → 运行时核心(3) → 游戏框架(4) → 渲染管线(5) → 物理/输入/资源(6/7)
```

禁止在未理解上层启动链路的情况下，直接深入某个 Pass 或 Component 的实现细节。

### 第 1 层：接口层（What）

**目标**：这个模块对外提供了什么能力？长什么样？

- 读取模块所在目录的 `CMakeLists.txt`，确认编译目标、源文件列表、依赖项
- 阅读该模块对外暴露的所有 `.h` / `.hpp` 头文件
- 用 `Grep` 查找其他模块 `#include` 该模块的位置，确认调用方和使用方式
- 列出核心类、关键 public 方法、命名空间
- **输出**：一张"接口地图"——谁依赖它、它依赖谁、对外暴露了什么

### 第 2 层：数据层（How - Structure）

**目标**：理解核心数据结构、内存布局、状态流转。

- 找到核心类/结构体（如 `RenderSystem`、`World`、`PiccoloEditor`）
- 分析成员变量之间的关系（ownership、裸指针还是智能指针、池化索引）
- 追踪关键数据的生命周期：创建 → 初始化 → 使用 → 销毁
- 标注资源分配来源（栈、堆、Vulkan Device Memory、Jolt BodyID）

### 第 3 层：逻辑层（How - Behavior）

**目标**：理解关键算法的执行流程和动态行为。

- 选取 2~3 个最核心的成员函数，逐行追踪调用链
- 绘制 Mermaid 流程图/时序图
- 分析与上下层模块的交互点（回调、事件、命令缓冲）
- 标注性能关键路径上的设计取舍

## 工具使用策略

| 任务类型 | 推荐工具组合 |
|---------|-------------|
| **模块定位** | `Glob` + `Grep` + `Shell` (`Get-ChildItem`) |
| **接口梳理** | `ReadFile`（头文件）+ `Grep`（引用点） |
| **调用链追踪** | `Grep`（函数定义）+ `ReadFile`（函数体） |
| **复杂子系统扫描** | `Agent(subagent_type="explore")` |
| **笔记产出** | `WriteFile` / `StrReplaceFile` |

### Explore Agent 触发条件

满足以下任一条件时启动 `subagent_type="explore"`：
1. 目标模块源码文件超过 15 个，调用关系复杂（如 `render`、`meta_parser`）
2. 需要追踪一个接口在 3 个以上不同子目录中的实现
3. 对某个子系统（如 Vulkan 封装层、反射生成器）完全没有概念，需要快速建立认知
4. 怀疑存在某种设计模式，需要跨文件确认

**Prompt 模板**：
```
请对 D:/workspace/Piccolo/engine/source/<模块路径> 进行 read-only 探索。
重点关注：
1. 公共接口头文件有哪些？核心类/结构体的命名和职责
2. 依赖哪些下层模块？被哪些上层模块依赖？
3. 是否存在代码生成、配置文件或宏驱动的机制？
4. 关键设计模式推测（如单例、观察者、命令模式、ECS 变体等）
返回简洁概览：目录结构、关键文件名、核心类列表、设计模式推测。
```

## Why：为什么要学习这个模块？
- **问题背景**：这个模块解决的核心问题是什么？
- **不用它的后果**：如果去掉这个设计，Piccolo 会怎样？
- **应用场景**：2~3 个实际应用场景

## What：这个模块是什么？
- **核心定义**：一句话定义
- **关键概念表**：用表格列出核心概念
- **架构图解**：Mermaid 类图或流程图

## How：Piccolo 是如何实现的？

### 1. 接口梳理
（第 1 层：公共头文件、核心类、对外暴露的方法）

### 2. 数据结构与状态流转
（第 2 层：核心类的成员变量、生命周期、内存布局）

### 3. 关键逻辑与调用链
（第 3 层：2~3 个核心函数的逐行分析、Mermaid 流程图）

### 4. 与上下层的关系
- **上层调用者**：谁在用这个模块？通过什么接口调用？
- **下层依赖**：它依赖哪些模块？依赖方式是什么？

### 5. 设计亮点与可迁移原理
> **这是最关键的部分**：从 Piccolo 的实现中提取 1~2 个可以迁移到自研引擎的通用设计模式。
- 这个设计在小型引擎里为什么够用？
- 如果要扩展成大型引擎，需要做什么改造？
- 参考 `Notes/SelfGameEngine/` 中的已有设计，指出可借鉴或需修正的地方。

## 迭代式深化策略

每次分析分三轮进行，每轮结束后告知用户本轮产出，等待确认后再进入下一轮。

### 第一轮：骨架扫描（接口层）
**目标**：建立模块地图，确认它在 Piccolo 索引中的坐标。

- 读取 `Notes/Piccolo/索引.md` 确认阶段位置和学习重点
- 读取模块所在目录的 `CMakeLists.txt`
- 读取所有对外暴露的头文件
- Grep 查找命名空间、核心类、下游引用点
- 输出一篇快速概览笔记（约 800~1500 字），覆盖索引中该模块的学习重点（优先接口层面）

### 第二轮：血肉填充（数据层 + 逻辑层）
**目标**：深入核心类与算法，填充索引要求的学习重点。

- 根据索引学习重点选取 2~3 个核心类，读取 `.h` + `.cpp`
- 追踪关键数据结构的生命周期和内存布局
- 追踪核心成员函数的调用链，绘制 Mermaid 流程图
- 补充上下层交互点的分析

### 第三轮：关联辐射（跨模块关系 + 可迁移原理）
**目标**：将模块放回 Piccolo 的完整链路中，提取通用知识。

- Grep 查找上层调用者和同阶段相关模块的引用
- 分析数据流入和流出的完整路径
- 补充"与上下层的关系"和"设计亮点与可迁移原理"
- **打开** `Notes/SelfGameEngine/从零开始的引擎骨架` 或相关笔记，对照 Piccolo 的实现找出可借鉴点
- 更新 `Notes/Piccolo/索引.md` 中的对应状态（与用户确认后执行）

## 与 self-game-engine 的协作

Piccolo 的最终学习目的不是"读完一个小引擎"，而是**提取可带走、可复用的通用工程原理，反哺自研引擎设计**。

在"设计亮点与可迁移原理"章节中，必须执行以下动作：

1. **打开** `Notes/SelfGameEngine/索引.md` 和相关引擎设计笔记
2. **对照** Piccolo 的真实实现，检查自研引擎设计中是否存在以下问题：
   - 遗漏了某个 Piccolo 已经验证过的关键细节？
   - 设计与 Piccolo 的实际工程选择相矛盾？
   - 过于理想化，没有考虑小引擎的落地约束？
3. **执行修改**：直接在 `Notes/SelfGameEngine/` 的相关笔记中修正不准确的内容，或补充可借鉴的具体实现

> 如果发现 Piccolo 的某个设计恰好解决了 `SelfGameEngine` 中悬而未决的问题，这是最高优先级的产出。

## 用户任务格式与 AI 承诺

### 理想任务格式

鼓励用户每次请求包含：
- 目标模块（如 `runtime/function/render`）
- 目标阶段（参考 `Notes/Piccolo/索引.md`）
- 分析轮次（第一轮/第二轮/第三轮）
- 特别关注点（可覆盖索引中的"学习重点"）
- 是否允许更新索引状态

### AI 响应承诺

1. **产出物明确**：要么是一篇写入文件的笔记，要么是一份结构化发现摘要
2. **不确定性标注**：基于推测的结论明确标注"推测"或"待确认"
3. **引用可追溯**：所有源码结论都有文件路径和行号支撑
4. **不发散**：严格限定在用户指定的模块范围内
5. **索引先行**：任何分析开始前必须先读 `Notes/Piccolo/索引.md`
