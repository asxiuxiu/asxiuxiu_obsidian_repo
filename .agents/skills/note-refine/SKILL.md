# note-refine

## 触发场景

当用户说以下任何一种表达时触发：

- "优化这篇笔记"
- "用费曼学习法改这篇笔记"
- "完善笔记叙述"
- "检查笔记质量"
- "这篇笔记写得不够好"
- "重构笔记"
- "让这篇笔记更易懂"
- "review 这篇笔记"
- "笔记叙述需要改进"

## 功能

读取用户指定的 Obsidian 笔记，对照本 skill 中的**叙述原则**和**自检清单**，诊断叙述问题并执行优化。

## 核心原则

> **写笔记的本质是思考，不是记录。**
>
> 费曼说："这不是记录，这就是我思考本身。" 你的笔记应该呈现**你真正理解后的语言与逻辑**，而不是对原文的转述或摘抄。

**没有任何强制分段、没有固定标题结构、没有必须遵守的格式模板。** 唯一的要求是：

> **当你写完这篇笔记，一个对此概念零基础的人能否顺畅地读完并理解？**
>
> 检验方法：把笔记里的比喻和例子全部删掉，剩下的文字是否还能自洽？如果不能，说明术语没有建立直觉就抛出来了。

## 写作策略（可选工具箱）

以下策略根据笔记内容和你的理解过程灵活选用，不必每条都用。

### 策略 1：Why 先行（问题驱动）

不要直接抛出定义。先让读者（也就是未来的你自己）感受到"这个问题确实需要解决"。

```markdown
❌ 互斥锁（Mutex）是操作系统提供的睡眠锁，竞争时线程挂起……
（读者：哦，又一个术语。跟我有什么关系？）

✅ 想象两个工人要同时修改同一份工资表。如果不加限制，
A 读到"张三工资 5000"，B 也读到"5000"；A 改成 6000，B 改成 5500；
最后存进去的是 5500——A 的修改被覆盖了。这就是数据竞争。
互斥锁就是为了解决这个问题而生的……
（读者：原来如此，没有锁真的会出乱子。）
```

> **何时选用**：引入新概念、新机制、新算法时。如果笔记只是快速备忘一个接口用法，可以跳过。

### 策略 2：比喻贯穿（直觉先行）

每个抽象概念找一个贴近生活的比喻，而且这个比喻要**贯穿整个讲解过程**，不是开头提一下就扔掉。

```markdown
❌ 进程是资源分配的最小单位，线程是 CPU 调度的最小单位。
（比喻在哪？读者怎么建立直觉？）

✅ 进程是一间带地皮的房子，线程是房子里的工人。
同一间房子里的工人共享厨房、设备和原料（地址空间、页表）。
但每个工人有自己的工作台和工具箱（栈、寄存器）。
如果工人太多抢一间房子……如果工人被赶到另一间房子……
（比喻持续发挥作用，直到读者完全理解。）
```

> **何时选用**：讲解抽象概念、系统架构、看不见摸不着的机制时。

### 策略 3：因果推导（演进逻辑）

讲清楚**"解决了 A 的问题，又带来了 B 的问题，于是人们发明了 C"**的演进链条。避免"第一种…第二种…第三种…"地分类罗列。

```markdown
❌ 线程调度算法有四种：FCFS、RR、优先级调度、CFS。
（这是百科词条，不是学习笔记。）

✅ 最早的调度是"谁先到谁先上"（FCFS），但一个大任务会堵住所有小任务。
于是人们发明了"每人只干固定时间"（RR），解决了堵塞，但频繁换人开销大。
再于是人们发明了"紧急任务优先"，但引发了优先级反转——高优先级反而被中优先级卡住。
最后 Linux 的 CFS 说："别搞优先级数字了，咱们记账——谁跑得少谁先上。"
（读者理解了每种算法是为了解决上一个算法的什么问题。）
```

> **何时选用**：对比多种方案、梳理技术演进史、分析设计决策时。

### 策略 4：场景绑定（知识落地）

把知识绑定到具体的使用场景或故障场景。问自己："如果去掉这个机制，系统会在什么场景下出问题？"

```markdown
❌ 写时复制（COW）是一种优化技术，fork 时不立即复制内存页。
（知道了，然后呢？什么时候用？什么时候会踩坑？）

✅ 在游戏引擎中，场景图需要在逻辑线程和渲染线程之间共享。
如果每次更新都完整复制一份场景数据，16ms 一帧根本不够。
COW 让两个线程先共享同一份只读数据，直到某一方真的要修改某个节点，
才单独复制那个节点——就像两个人先看同一份图纸，
只有其中一个人要改某个零件尺寸时，才复印那一页。
但如果修改非常频繁（比如每帧都更新所有粒子位置），
COW 的"延迟复制"反而变成了"每帧都复制"，性能更差。
```

> **何时选用**：学习优化技术、并发机制、设计模式时。回答"它在什么场景下有效/失效"。

### 策略 5：问题链衔接（自然过渡）

段落结尾自然引出下一个问题。读者应该感到"知道了这个，我自然会想知道那个"。

```markdown
❌ （第一节讲完进程是什么）
（第二节突然开始）线程有三种实现方式：ULT、KLT、M:N。
（读者：等等，为什么突然讲线程？进程和线程是什么关系？）

✅ （第一节结尾）
"进程解决了'隔离'的问题，但它自带一个严重缺陷：一座房子里只有一个工人。
如果这座房子既要渲染画面，又要加载资源，该怎么办？"
（第二节自然承接）"于是操作系统给出了更聪明的方案：在同一座房子里塞进多个工人——这就是线程。"
```

> **何时选用**：长笔记、多概念串联时。让整篇笔记有"一口气读下去"的流畅感。

### 策略 6：用自己的话写（拒绝复制粘贴）

不要复制原文。合上书/关掉页面，用自己的语言重新组织。只有在表达中，思维的自我欺骗才会被识破。

```markdown
❌ 操作系统通过 TLB 加速虚拟地址到物理地址的转换。
（这是教材原话，不是你理解后的语言。）

✅ 每次地址翻译都要查页表（内存里的一张表），太慢了。
CPU 内部有一个高速缓存，专门存"最近查过的地址翻译结果"，
这样下次翻译同一个地址时就不用再访问内存。
这个缓存叫做 **TLB（Translation Lookaside Buffer）**，
Translation 是"地址翻译"，Lookaside 是"在旁边偷看一眼"——
就是说"翻译地址时，先在旁边的小本子上偷看一眼有没有记过"。
```

> **何时选用**：所有笔记。这是费曼学习法最底层的动作——"写就是思考"。

### 策略 7：原子化（一个核心观点一篇笔记）

一篇笔记只承载一个核心观点或一个核心问题。如果涉及多个并列概念，拆成多篇并互相链接。

> **何时选用**：笔记越写越长、内容越写越杂时。停下来问自己：这篇笔记到底在回答一个什么问题？

### 策略 8：渐进式摘要（分层压缩）

根据笔记的目的选择详略，不要每篇笔记都写成教科书：

| 笔记类型 | 内容深度 | 适用场景 |
|---------|---------|---------|
| **深度笔记** | 完整的因果推导 + 比喻 + 示例 + 源码分析 | 攻克核心难点、构建系统理解 |
| **速记卡片** | 核心观点 + 一句话总结 + 链接到深度笔记 | 快速记录灵感、备忘接口用法 |
| **问题笔记** | 一个具体疑问 + 目前的理解 + 待验证的假设 | 记录"我还没搞懂"的状态 |
| **冲突笔记** | "这个观点和我已有的 XX 知识矛盾" + 分析 | 发现认知盲区、纠正误解 |

> **何时选用**：根据当前的学习阶段和时间灵活选择。关键是诚实——不要为了"看起来完整"而硬凑内容。

### 策略 9：有效失败（从错误中学习）

主动记录"我曾经怎么误解这个概念"、"这个机制在什么情况下会让我踩坑"。纠正误解是最高效的学习。

```markdown
✅ 我一度以为 "volatile" 能保证线程安全，因为加了它变量就不会被缓存到寄存器。
实际上 volatile 只保证"可见性"——一个线程修改后，其他线程能读到最新值。
但它不保证"原子性"，比如 `volatile i++` 仍然是非原子的（读-改-写三步）。
真正需要原子操作还是得用 `std::atomic` 或锁。
```

> **何时选用**：任何你"以为自己懂了，实际没懂"的时刻。这是笔记最有价值的部分。

### 策略 10：表格只用于总结

对比表格、特性表格**只能用于总结已经解释过的概念**，绝对不能用于首次引入概念。表格是"复习工具"，不是"教学工具"。

```markdown
❌ 首次介绍就用表格
| Memory Order | 含义 | 使用场景 |
| relaxed | 无同步保证 | 计数器 |
（读者还没理解什么是 Memory Order，表格就是天书。）

✅ 先讲清楚故事，再用表格复习
（讲完"寄包裹的可见性合同"故事后……）
> 五种合同的严格程度对比：
> | 合同级别 | 类比 | 保证 | 性能 |
> （此时读者已经有直觉，表格帮助他整理记忆。）
```

> **何时选用**：总结多种方案的对比、整理已学过的概念关系时。

## 工作流程

### 第一步：读取目标笔记

读取用户指定的笔记全文。如果用户没有明确指定路径，询问用户。

### 第二步：诊断（必须输出诊断报告）

逐条对照以下自检清单进行分析，输出**诊断报告**：

```markdown
## 诊断报告：`<笔记名>`

### 1. 零假定（首次术语是否有人话解释）
- 状态：✅ 通过 / ⚠️ 部分通过 / ❌ 未通过
- 问题：`<具体列出哪些术语首次出现时没有解释>`
- 位置：`<行号或段落引用>`

### 2. Why 先行（每个核心概念是否有"为什么需要"的铺垫）
- 状态：...

### 3. 比喻有效（比喻是否贯穿，术语是否建立了直觉）
- 状态：...

### 4. 因果链条（并列内容是否按演进逻辑组织）
- 状态：...

### 5. 表格合规（表格是否仅用于总结已解释的概念）
- 状态：...

### 6. 衔接自然（相邻小节之间是否有因果/问答关系）
- 状态：...

### 7. 链接完整（vault 内概念是否有 wikilink）
- 状态：...

### 综合评分
- 优秀（6-7 条通过）：叙述质量高，仅需微调
- 合格（4-5 条通过）：有明显改进空间
- 不合格（0-3 条通过）：需要大面积重写
```

### 第三步：执行优化

根据诊断报告，对笔记进行**精确修改**。修改原则：

1. **优先改叙述，不改结构**：除非结构本身有严重问题，否则优先在现有结构内优化叙述方式。
2. **逐段替换**：使用 `StrReplaceFile` 进行精确替换，不要重写整个文件（避免破坏 frontmatter、代码块、wikilink 等）。
3. **修改范围**：
   - **不合格项必须改**：如果某条自检未通过，必须修改。
   - **部分通过项视情况改**：如果问题轻微，可以不改；如果问题明显，必须改。
   - **已通过项不改**：不要画蛇添足。
4. **保持原有内容**：代码块、实验数据、数学公式、已有的 wikilink 原则上不动，只优化叙述文字。
5. **比喻要贴切**：新增的比喻必须符合技术本质，不能为了比喻而比喻（错误的比喻比没有比喻更糟）。

### 第四步：输出修改摘要

修改完成后，输出摘要：

```markdown
## 修改摘要

### 修改了哪些部分
1. `<段落标题>` — 优化原因
2. ...

### 新增/改进了哪些比喻
- `<概念>` → `<比喻>`

### 修复了哪些自检项
- ❌→✅ `<自检项名称>`
```

## 优化技巧

### 如何判断"比喻是否有效"

把比喻删掉，看剩下的文字是否仍然能被零基础读者理解。如果不能，说明术语没有被解释清楚，比喻只是装饰。

### 如何把"分类罗列"改成"因果推导"

不要：
```
调度算法有四种：
1. FCFS：先来先服务
2. RR：时间片轮转
3. 优先级调度
4. CFS
```

要：
```
最早的调度是"谁先到谁先上"（FCFS），但大任务会堵住小任务。
于是人们发明了"每人只干固定时间"（RR），解决了堵塞但频繁换人开销大。
再于是……
```

### 如何给术语加人话解释

模板：
```
<人话解释：这个东西是干什么用的，在什么场景下出现>
<为什么叫这个名字>
<正式定义>
```

例如：
```
CPU 内部有一个高速缓存，专门存"最近查过的地址翻译结果"，
这样下次翻译同一个地址时就不用再访问内存。
这个缓存叫做 **TLB（Translation Lookaside Buffer）**，
Translation 是"地址翻译"，Lookaside 是"在旁边偷看一眼"。
```

### 如何判断"Why 先行"是否到位

找到每个核心概念的第一次出现位置。它前面 1~2 句话是否在回答"如果没有这个概念会怎样"或"什么场景下需要它"。

## 注意事项

- **不要只改标题**：费曼学习法不是标题结构，改标题没用。必须深入到每一段文字。
- **不要新增大量表格**：如果笔记已经有不合格的表格，应该考虑用叙述替代表格，而不是新增更多表格。
- **不要破坏已有链接**：修改文字时不要误删 wikilink。
- **不要过度优化**：如果某段叙述已经合格，不要强行改成另一种风格。合格的标准是"零基础读者能看懂"，不是"辞藻华丽"。
- **修改后必须重新自检**：改完后再次逐条对照自检清单，确认问题已修复。
