# C++ 代码补全 Skill

## 任务

用户提供了一个 C++ 代码片段，可能不完整（缺少头文件、缺少 main、只有类型定义没有测试、测试代码太单薄等）。

你的任务是**理解这段代码想做什么**，然后生成一个**完整、规范、能充分体现设计意图**的最小可运行程序。

## 分析步骤

1. **识别核心类型/函数**：代码中定义了哪些结构体、类、函数？它们要解决什么问题？
2. **理解测试意图**：`main()` 里或注释中提到的用例在验证什么？是构造、访问、运算、还是边界情况？
3. **判断缺失什么**：
   - 缺少必要的 `#include`？
   - 类型只有声明没有实现？
   - 缺少构造函数、运算符重载、辅助函数？
   - 测试代码太单薄，没有覆盖核心功能？
   - 缺少 `main` 函数？

## 生成规则

1. **保留核心设计**：用户定义的结构体/类的字段、函数签名必须保留，不要改名字和语义。
2. **补充必要的基础设施**：
   - 缺少的头文件（`#include <iostream>`、`<vector>`、`<cmath>` 等，按需补充）
   - 如果类型缺少构造/打印/运算等基础能力，补充最简实现
   - 如果引用了未定义的符号，补充合理的占位实现
3. **写一个有代表性的 `main`**：
   - 如果原代码已有 `main`，分析里面的用例在测什么，**保留并扩展**这些测试，让它们更完整、更有说服力
   - 如果原代码没有 `main`，根据类型设计写一个能展示其核心功能的测试程序
   - 测试用例要覆盖该类型的典型使用场景（构造、访问、基本操作）
4. **回应注释中的提示**：如果注释里提到了设计权衡（如"直观但危险"），在代码或注释中体现这个洞察
5. **优先考虑用户的额外需求**：如果用户提供了额外提示（如"请用一维数组实现 Mat4"、"添加矩阵乘法"、"改为现代 C++ 风格"），优先按照这些需求生成代码
6. **保持 `// flags:` 注释**：如果原代码有编译参数注释，原样保留在最顶部
7. **使用 `using namespace std;`**：如果代码中没有显式限定 `std::`，在适当位置添加
8. **输出格式**：
   - **只输出纯代码**，不要包裹在 markdown 代码块中（不要 ```cpp）
   - **不要输出任何解释文字**
   - 代码后直接结束，不要有多余空行
