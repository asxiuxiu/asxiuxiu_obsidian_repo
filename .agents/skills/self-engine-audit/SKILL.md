# 自研引擎代码审计（Self Engine Audit）

本 skill 是 vibe coding 过程中的**质量守门人**。当你在写 Entelechy 时不确定某个用法是否正确、是否符合笔记中的设计原则、是否在工业界有更好实践时，调用本 skill 进行审计。

**核心使命**：确保 Entelechy 的每一行代码都经得起三重拷问——
1. **与你自己笔记里的设计原则一致吗？**
2. **UE/Bevy/chaos 在同类问题上是怎么做的？**
3. **在 ECS 架构下，这是最优表达吗？**

## 触发条件

- "帮我看看这段代码"
- "这个写法合理吗"
- "audit 一下这个模块"
- "review 一下"
- "这和笔记里讲的不一样"
- "UE 是怎么处理这个问题的"
- "我这样实现有没有问题"
- "和工业界对比，我这代码处于什么水平"
- "我是不是把问题想简单了"
- "这段代码需要怎么改进"

## 环境配置

每次审计**必须首先读取**项目根目录下的 `.engine-source-config`，提取 `ENTELECHY_SOURCE_ROOT`。若未设置，**必须提示用户配置**，禁止回退默认路径。

```
ENTELECHY_SOURCE_ROOT=D:\workspace\Entelechy
```

## 核心方法论：三重审计

审计不是简单的"对"或"错"，而是三个维度的结构化比对。

### 维度一：理论一致性审计

**问题**：项目代码与 vault 中 `self-game-engine` 产出的费曼笔记是否一致？

- 如果笔记中明确推荐了某方案，项目是否采用了该方案？如果没有，是故意偏离还是无知偏离？
- 如果笔记中分析了多个分支并给出了默认推荐，项目的选择是否在推荐路径上？
- 如果笔记中指出了某些方案的隐藏代价或失效条件，项目是否踩了这些坑？

**读取范围**：
- `self-game-engine` 产出的费曼笔记（通用设计知识）
- `Notes/` 下与审计主题相关的所有已有笔记
- 如果相关笔记不存在或内容不足，**必须向用户说明缺口，并建议先调用 `self-game-engine` 或 `game-engine-analysis` 补充笔记**

### 维度二：工业参照审计

**问题**：与 UE/Bevy/chaos 的同类实现相比，项目代码处于什么水平？

- UE 在这个具体子问题上是怎么解决的？（优先吸收 UE 的工业级设计）
- Bevy 作为 ECS 原生引擎，在这个问题上的表达方式是什么？
- chaos 作为实战验证的参照，有什么可借鉴之处？

**读取范围**：
- `game-engine-analysis` 产出的源码分析笔记（`Game/`、`Notes/UE/`、`Notes/Bevy/`）
- 如果相关源码分析笔记缺失，**优先搜索互联网**获取开源引擎的同类实现，而非直接调用 `game-engine-analysis` 回探源码（审计流程应轻量、快速）

### 维度三：架构适配审计

**问题**：这段代码在 ECS 架构下的表达是否优雅？是否满足 AI 友好设计？

- 是否存在 OOP 全局状态？是否应该拆成 Component/Resource/System？
- 状态是否可序列化、可反射、可 diff？
- 执行是否是输入确定的？是否支持快照与回放？
- 模块的接口边界是否清晰，AI 能否通过结构化接口操作它？

## 工作流

### 步骤一：锁定审计目标

用户输入可能是以下三种形式之一：

| 形式 | 处理方式 |
|------|---------|
| **文件路径** | 读取 `ENTELECHY_SOURCE_ROOT` 下的对应文件 |
| **代码片段** | 直接分析用户提供的代码 |
| **模块/主题** | 在项目中搜索相关文件，列出候选让用户确认 |

**如果用户没有明确指定审计范围**，主动询问：
- 要审计的具体文件或代码片段
- 特别关注点（性能？架构？ECS 表达？）

### 步骤二：代码解析与主题识别

读取代码后，快速识别涉及的主题领域：

| 关键词/模式 | 主题领域 | 关联笔记路径 |
|------------|---------|------------|
| Entity/Component/System/World/Query/Archetype | ECS 核心 | `Notes/SelfGameEngine/ECS*` |
| RenderPass/CommandBuffer/Shader/Texture/FBO | 渲染管线 | `Notes/SelfGameEngine/渲染*` |
| Asset/Resource/Handle/RefCount/Loader | 资源管理 | `Notes/SelfGameEngine/资源*` |
| Log/Sink/Formatter/Async | 日志系统 | `Notes/SelfGameEngine/日志*` |
| Job/ThreadPool/Task/ParallelFor | 多线程/任务系统 | `Notes/SelfGameEngine/线程*` |
| Memory/Pool/Allocator/Arena/Stack | 内存管理 | `Notes/SelfGameEngine/内存*` |
| Reflection/Serialize/TypeRegistry | 反射与序列化 | `Notes/SelfGameEngine/反射*` |
| Window/Input/Event | 窗口与输入 | `Notes/SelfGameEngine/输入*` |
| Physics/Collision/RigidBody | 物理系统 | `Notes/SelfGameEngine/物理*` |

**主题识别后**：
1. 在 vault 中搜索相关笔记（`self-game-engine` 费曼笔记 + `game-engine-analysis` 源码分析笔记）
2. 如果相关笔记缺失，**在审计报告中标注"理论缺口"**，并建议用户后续补充

### 步骤三：执行三重审计

对每个识别出的主题，依次执行三个维度的审计。

#### 审计记录格式（内部使用）

```
审计点：<具体代码位置/设计决策>
维度一（理论一致性）：
  - 笔记推荐：XXX
  - 项目实际：YYY
  - 偏差类型：✅ 一致 / ⚠️ 轻微偏离 / ❌ 严重偏离 / ❓ 笔记未覆盖
  - 偏差说明：...

维度二（工业参照）：
  - UE 做法：...
  - Bevy 做法：...
  - 项目水平：领先 / 持平 / 差距明显 / 尚未涉及
  - 差距说明：...

维度三（架构适配）：
  - ECS 表达：优雅 / 可改进 / 需要重构
  - AI 友好性：满足 / 部分满足 / 不满足
  - 具体建议：...
```

### 步骤四：风险评级与优先级排序

所有发现按以下四级评级：

| 级别 | 标识 | 含义 | 行动建议 |
|------|------|------|---------|
| **Critical** | 🔴 | 架构级错误，会导致后续难以维护或严重性能问题 | 必须立即修复 |
| **Warning** | 🟠 | 设计缺陷，当前可能工作但存在隐藏代价或失效条件 | 建议在当前里程碑修复 |
| **Info** | 🟡 | 与工业最佳实践存在差距，有改进空间 | 可在后续迭代优化 |
| **Suggestion** | 🟢 | 风格/表达层面的优化建议，不影响正确性 | 可选采纳 |

**排序原则**：Critical > Warning > Info > Suggestion。同一级别内，按"修复成本从低到高"排序（先给容易赢的）。

## 审计原则

### 原则 1：诚实暴露代价，不夸大也不贬低

- 如果项目的某个简化方案在当前阶段是合理的，**明确说明为什么合理**，不要为批评而批评。
- 如果项目存在 Critical 级问题，**必须清晰解释"现在不修复，未来会在什么场景下爆炸"**。

### 原则 2：工业参照必须具体到子问题

- 禁止笼统地说"UE 做得更好"。必须具体到："对于「异步资源加载的取消机制」这个子问题，UE 采用了..."
- 引入 UE/Bevy/chaos 参照时，必须说明它们解决的是**同一个具体子问题**，而非泛泛对比。

### 原则 3：改进建议必须可落地

- 每个建议必须附带**代码片段**，展示改进后的关键机制。
- 如果改进涉及大范围重构，必须给出**分步迁移路径**，不要要求一次性重写。

### 原则 4：区分"知识缺口"与"实现错误"

- **实现错误**：代码逻辑有 bug 或明显反模式 → 直接指出。
- **知识缺口**：vault 中尚无相关笔记，导致项目缺乏设计参考 → 在报告中标注，并建议调用 `self-game-engine` 补充笔记。

### 原则 5：优先吸收 UE，拒绝简化妥协

与 `self-game-engine` 保持一致：
- 默认优先吸收 UE 的工业级设计。
- **禁止以"个人项目"、"当前阶段简单优先"、"先过渡后补全"为由推荐降级方案。**
- 如果 UE 的设计与 ECS 架构存在根本冲突，才允许偏离，并必须在报告中说明冲突点。

## 与相关 skill 的关系

| skill | 职责 | 何时调用 |
|-------|------|---------|
| `self-engine-audit`（本 skill）| 审计 Entelechy 项目代码 | 用户对代码用法存疑时 |
| `self-game-engine` | 产出费曼设计笔记 | 需要补充理论缺口时 |
| `game-engine-analysis` | 深入分析 UE/Bevy/chaos 源码 | 审计发现需要回探源码确认时 |
| `note-refine` | 优化笔记叙述质量 | 审计报告本身需要润色时 |

**工作流衔接**：
1. 用户 vibe coding 写出代码 → 调用 `self-engine-audit`
2. 审计发现理论缺口 → 调用 `self-game-engine` 补充费曼笔记
3. 审计发现源码结论不足 → 调用 `game-engine-analysis` 回探源码
4. 审计报告产出 → 用户继续 vibe coding

## AI 响应承诺

1. **必须先读配置**：每次审计首先读取 `.engine-source-config` 获取 `ENTELECHY_SOURCE_ROOT`
2. **必须先读相关笔记**：审计前读取 vault 中与主题相关的费曼笔记和源码分析笔记
3. **三重审计完整**：每个发现必须包含理论一致性、工业参照、架构适配三个维度的分析
4. **风险评级明确**：所有发现按 Critical/Warning/Info/Suggestion 四级评级
5. **改进建议可落地**：每个建议附带代码片段和分步迁移路径
6. **诚实暴露代价**：不为了批评而批评，也不隐瞒真正的问题
7. **区分缺口与错误**：明确标注 vault 中尚未覆盖的理论缺口
8. **UE 优先吸收**：默认推荐 UE 的工业级设计，禁止以项目规模为由推荐降级方案
9. **输出可选落地**：审计报告可直接展示，也可写入 `Notes/SelfGameEngine/审计/`
