# 图形学进阶之旅（Graphics Journey）

本 skill 负责将用户的**图形学理论知识**（软渲染器、变换、光栅化、着色模型）转化为**可运行的GPU代码能力**和**引擎级渲染架构理解**。

**核心使命**：每篇笔记必须回答一个具体的问题——"我知道这个理论，但代码该怎么写？"最终让用户能从零搭建出引擎的渲染管线。

> **关键原则：图形学的概念不能空降。** 每个API对象（VAO/VBO/Shader/Texture/FBO）必须先讲清楚"没有这个对象之前代码长什么样、有什么问题"，再引出对象的引入。拒绝"VAO是一个容器对象，它包含..."这种百科式讲解。

## 核心方法论：问题链驱动 + API对照 + 费曼提取法

图形学进阶最大的障碍是**概念堆积**。本 skill 的核心方法是把每个新概念都转化为一个"问题 → 最naive方案 → 发现问题 → 引入新概念"的链条。

### 问题链的图形学特化

图形API中的每个对象类型，都要用问题链引出：

**以 VAO 为例**：
```
问题：每次绘制都要重复配置顶点属性指针，代码又臭又长还容易错
  → 最naive方案：每次 glDrawArrays 前都重新 glVertexAttribPointer + glEnableVertexAttribArray
  → 立刻发现的问题：状态重复设置、容易遗漏、绘制多个物体时代码爆炸

问题：怎么把顶点属性配置"打包"起来一次设置重复使用？
  → 方案：引入 VAO（Vertex Array Object），把属性配置存入GPU状态对象
  → 新问题：VAO 和 VBO 的关系是什么？EBO 绑在哪里？

问题：VAO 到底存了什么？不存什么？
  → 深入 VAO 的状态机模型...
```

**以 Shader 为例**：
```
问题：固定管线只能做固定的事情，我想自定义顶点变换和像素着色怎么办？
  → 最naive方案：没有，固定管线就是固定管线
  → 立刻发现的问题：Blinn-Phong公式没法改、没法做法线贴图、没法做后处理

问题：如何让GPU执行我写的代码？
  → 方案：引入 Shader（顶点/片段着色器），GLSL编译后上传GPU
  → 新问题：Shader怎么拿到CPU的数据？怎么拿到顶点属性？怎么拿到Uniform？

问题：Uniform 和 Vertex Attribute 有什么区别？
  → 深入两种数据通道的差异...
```

### 图形学问题链的写作要求

1. **每个API对象必须让读者先感受到"没有这个对象之前的痛苦"**
   - 不要直接说"VAO是一个容器对象"
   - 要先展示"不用VAO时每次绘制都要写10行重复代码"
   - 读者感受到痛苦后，VAO的出现才是"救星"而非"新概念"

2. **每个方案必须先给出最naive的版本（尽量是能编译运行的代码）**
   - 先写出"如果我只学过软渲染器，我会怎么写OpenGL代码"
   - 然后**运行它**（或推演它），指出具体哪里会出问题（性能、正确性、可维护性）
   - 代码必须能从用户的已有知识自然延伸出来

3. **改进必须是因果推导，不是API文档罗列**
   - ❌ 错误："OpenGL有VBO、VAO、EBO、FBO、UBO等多种缓冲对象"
   - ✅ 正确："我们先写了直接drawArrays，发现顶点数据每次都要从CPU传到GPU（问题A）。于是引入VBO把数据常驻GPU（方案1），但发现属性配置还是每次都要重新设（问题B）。于是引入VAO把配置也打包（方案2）……"

4. **API对照只在决策点引入**
   - 不是每篇笔记都必须对比OpenGL/Vulkan/D3D12
   - 只有当某个子问题存在多种可行方案、且需要判断"个人项目该选哪个"时才引入对照
   - 引入时要说明：我们正在解决哪个具体问题？不同API在这个具体子问题上的差异是什么？

| 搜索时机 | 搜索目标 |

## 叙事原则（强制遵循 note-refine 的费曼精神）

### 原则 1：擅长提出问题，拒绝概念空降

在写下任何API调用或对象之前，必须先问自己三个问题：
- **这个API调用解决什么问题？** 如果回答不上来，就不该在这里出现。
- **读者此时知道什么？** 用户已经会写软渲染器，API代码必须能自然延伸自软渲染器的逻辑。
- **这个术语引入得是否过于突然？** VAO、FBO、UBO这些缩写必须先用人话解释。

### 原则 2：Why 先行（问题驱动）

```markdown
❌ VBO是顶点缓冲对象，使用glGenBuffers创建，glBindBuffer绑定...
（读者：哦，又一个API对象。跟我有什么关系？）

✅ 在软渲染器里，顶点数据存在std::vector里，CPU直接访问。
但在GPU上，顶点数据必须存在显存里。怎么把数据从内存搬到显存？
最naive的想法：每次绘制都用glBufferData传一遍。
但这意味着每帧都做一次CPU→GPU的内存拷贝，对于10万个顶点就是几MB的数据搬运。
VBO就是为了解决这个问题——一次性上传，反复绘制。
（读者：原来VBO不是可有可无的优化，而是刚需。）
```

### 原则 3：因果推导（演进逻辑）

```markdown
❌ 渲染到纹理需要以下步骤：创建FBO、创建纹理、附加纹理到FBO...
（这是API手册，不是学习笔记。）

✅ 我们先实现了直接渲染到屏幕，想做后处理时发现：
"模糊"效果需要读取已经渲染好的画面，但屏幕缓冲区是只写的。
于是我们需要一个"离屏画布"——先画到一张纹理上，再对这张纹理做处理。
OpenGL里这个"离屏画布"就是FBO。
```

### 原则 4：场景绑定（知识落地）

每个API概念都必须绑定到具体场景：
- "如果不创建VBO，10万顶点的模型每帧都要memcpy到显存，帧率会从60掉到5"
- "如果不使用FBO，后处理（模糊/泛光/色调映射）根本没法实现"
- "如果不理解Uniform vs Attribute，Shader里就拿不到正确的数据"

### 原则 5：问题链衔接（自然过渡）

段落结尾自然引出下一个问题。读者应该感到"知道了这个，我自然会想知道那个"。

```markdown
❌ （第一节讲完VBO）
（第二节突然开始）VAO是顶点数组对象，用于存储顶点属性配置...
（读者：等等，为什么要讲VAO？跟VBO有什么关系？）

✅ （第一节结尾）
"VBO把顶点数据存到了GPU显存，但每次绘制前，我们还要告诉GPU：
'第一个属性是位置，3个float；第二个属性是法线，3个float；第三个是UV，2个float'。
如果画100个物体，这套配置就要重复100次。能不能也打包起来？"
（第二节自然引出VAO）
```

### 原则 6：表格只用于总结

对比表格、特性表格**只能用于总结已经解释过的概念**，绝对不能用于首次引入概念。

## 精确性与诚实性原则

### 原则 1：操作必须精确到"状态变化图"

图形API是**状态机**，状态变化是理解的核心。任何涉及API状态变更的内容，必须提供操作前后的精确状态对比。

**必须画出来的场景包括**：
- VAO/VBO/EBO 的绑定关系图（哪个绑在哪个上）
- FBO 的附件结构（颜色附件、深度附件的布局）
- Shader 的数据流图（Attribute → Vertex Shader → Varying → Fragment Shader → Output）
- 纹理状态机（绑定目标、采样器参数、纹理单元）
- 渲染管线的帧缓冲切换（Default FBO → Custom FBO → Default FBO）

**自问 checklist**：
- 一个谨慎的读者看到这段，会不会问"那绑定VAO之前需要先绑定VBO吗"？
- 他会不会问"glDrawElements用的是哪个EBO"？
- 他会不会问"Shader里uniform的值是跟着VAO走还是跟着Shader走"？
- 如果以上任何一个问题的答案会影响理解，就必须在笔记中主动给出。

### 原则 2：性能与 trade-off 必须诚实

不能为了推崇某个API或架构而隐瞒其代价。

**必须包含的要素**：
- **承认代价**：这个API对象消除了什么成本？又引入了什么新成本？
- **对比基准**：和最naive的实现相比，在什么场景下更好、什么场景下更差？
- **反例警告**：在什么极端使用方式下，这个方案会失效或表现糟糕？
- **不连续的真相**：如果涉及"GPU并行"或"零拷贝"的宣传，必须明确说明"哪里避免了拷贝、哪里还是有开销"。

**禁止出现的说辞**：
- "VBO完全消除了数据传输开销" → 应该说"首次上传仍有CPU→GPU拷贝，但绘制时不再重复传输"
- "Shader让一切可定制" → 应该说"增加了编译链接的复杂度，且不同GPU的Shader编译器行为有差异"
- "FBO让后处理免费" → 应该说"增加了额外的纹理采样开销和显存占用"

### 代码规范

- 费曼笔记中的代码必须是**通用化重写的最小实现**，不是源码片段
- OpenGL代码使用现代Core Profile（3.3+），**禁止固定管线**
- 优先使用GLSL 330+，代码必须接近能编译运行
- 每段代码示例都应说明它解决了什么问题、还有什么问题没解决
- 必须展示"不用新概念的naive代码"和"使用新概念后的改进代码"的对比

### API对照的呈现方式

```markdown
> **参考：OpenGL / Vulkan / D3D12 对这个问题是怎么做的？**
> 
> 我们在解决的是「离屏渲染的API抽象」这个问题。
> - OpenGL 选择了 FBO（FrameBuffer Object），原因是……
> - Vulkan 选择了 RenderPass + Framebuffer + ImageView，原因是……
> - D3D12 选择了 RenderTargetView + OMSetRenderTargets，原因是……
> 
> **个人项目推荐**：默认选 OpenGL FBO，因为学习曲线最平缓；向现代API迁移时， mentally map 为 "FBO ≈ RenderPass + Framebuffer"
```

## 迭代产出策略（问题链驱动）

### 阶段一：梳理问题链（骨架）

**输入**：用户已有笔记 + 图形API文档 + 引擎源码
**步骤**：
1. **读取用户已有图形学笔记**（`Notes/计算机图形学/` 目录），了解当前知识水平
2. **搜索互联网**：搜索该概念在图形API中的标准定义、经典教程、最常见初学者陷阱
3. **梳理问题链**：基于已有知识 + API文档，列出从naive到工业级的完整问题链
   - 简单知识点（如 Uniform vs Attribute）：3~5个问题
   - 中等知识点（如纹理系统）：5~8个问题
   - 复杂知识点（如引擎渲染管线抽象）：8~12个或更多问题
4. **确定API对照点**：标注问题链中哪些子问题值得引入API对照
5. **产出问题链大纲**：向用户展示梳理好的问题链，确认方向后再进入下一阶段

**产出**：
- 该模块解决的问题背景（Why）
- 完整的问题链列表
- 每个问题对应的naive方案轮廓
- 需要引入API对照的子问题标注
- **前置依赖声明**和**本模块增量**说明

### 阶段二：逐个解决问题（血肉）

**输入**：图形API文档 + 引擎源码 + 互联网搜索
**步骤**：
1. 按问题链顺序，逐个解决每个子问题
2. 对每个子问题：
   - **先给naive方案**：写出"如果刚学完软渲染器，我会怎么写GPU代码"
   - **运行/推演它**：指出具体哪里会出问题
   - **提出改进方案**：给出使用新API概念的代码
   - **标注还遗留哪些问题**
3. **按需引入API对照**：只有当当前子问题存在多种可行方案时才引入对照
4. **搜索互联网**：针对当前正在解决的具体子问题，搜索不同API的实现差异

**产出**：
- 每个子问题的naive方案 + 问题分析 + 改进方案
- 按需引入的API对照
- 完整的代码演进过程
- **明确推荐**：在每个需要决策的子问题上，给出"推荐采用哪种方案"的结论

### 阶段三：定型与下一步

**步骤**：
1. 梳理跨模块耦合、性能优化、现代引擎抽象方式
2. **搜索互联网**：确认引擎中该模块的现代抽象方式（bgfx/Sokol/The-Forge 等）
3. 输出经过验证的、可迁移的设计清单
4. 在笔记结尾添加**下一步预告**
5. **精确性审查**：检查所有涉及API状态机、内存布局、数据流的地方
6. **叙事审查**：检查是否符合费曼精神（Why先行、因果推导、问题链衔接）
7. **最终一致性审查**：检查是否满足"必须有明确推荐"的原则

**产出**：
- 工业级设计清单
- 设计权衡表（表格只用于总结已解释的内容）
- **个人开发推荐路径**：从naive OpenGL到现代引擎抽象的具体步骤
- **更新 roadmap/索引文件**，标注该模块已完成并指向下一个推荐模块

3. **代码可运行**：所有代码示例都接近能编译运行，不是伪代码
4. **索引维护**：如 roadmap/索引文件已存在，主动提议更新；如不存在，主动创建
5. **顺序连贯**：产出的每篇笔记都明确声明前置依赖、本模块增量和下一步预告
6. **精确且诚实**：涉及API状态机、GPU数据流时，主动提供状态变化图和trade-off分析
7. **问题链驱动**：笔记的主体是"问题 → naive方案 → 发现问题 → 改进"的渐进过程
8. **按需对照**：只在存在方案分歧的具体子问题上引入OpenGL/Vulkan/D3D对照
9. **前沿探索**：每篇笔记搜索行业前沿（现代图形API抽象库），探索更优方案
10. **最正确方案**：在每个需要决策的子问题上，必须在可选方案之间给出明确推荐
11. **Roadmap 唯一权威**：所有目录结构、文件名、阶段划分、依赖关系以roadmap为准
12. **叙事质量**：笔记必须遵循费曼精神（Why先行、因果推导、问题链衔接、场景绑定）
