# 游戏引擎源码分析

系统化源码分析工作流，用于对 chaos 引擎 + wolfgang/proven_ground 游戏项目的大型代码库进行模块级架构解析。

## 触发条件

- 用户说"帮我分析这个模块的源码"
- 用户要求"阅读/解析某个开源项目的代码"
- 用户请求"生成源码分析笔记"
- 用户提到"三层剥离法"或"源码级架构解析"

## 核心方法论：三层剥离法

分析必须按以下三层递进，禁止未做完第 1 层就跳到第 3 层逐行读 `.cpp`。

### 第 1 层：接口层（What）
**目标**：知道模块对外提供了什么能力。

- 读取模块根目录的 `CMakeLists.txt`，了解目标名、依赖项、源文件列表
- 阅读对外暴露的公共头文件（`include/` 或模块根目录）
- 用 `Grep` 查找其他模块 `#include` 该模块头文件的地方，确认使用方式
- 列出核心类和关键 public 方法

### 第 2 层：数据层（How - Structure）
**目标**：理解核心数据结构、内存布局、状态流转。

- 找到核心类/结构体（关注命名空间）
- 分析成员变量之间的关系（ownership、引用、池化索引）
- 追踪关键数据的生命周期：创建 → 使用 → 销毁
- 标注内存分配来源（栈、堆、mempool、GPU memory）

### 第 3 层：逻辑层（How - Behavior）
**目标**：理解关键算法的执行流程和动态行为。

- 选取 2~3 个最核心的成员函数，逐行追踪调用链
- 分析多线程场景下的同步策略
- 分析性能关键路径上的优化手段
- 标注与上下层模块的交互点（回调、事件、命令缓冲）

## 工具使用策略

| 任务类型 | 推荐工具组合 |
|---------|-------------|
| **模块定位** | `Glob` + `Grep` |
| **接口梳理** | `ReadFile`（头文件）+ `Grep`（引用点） |
| **调用链追踪** | `Grep`（函数定义）+ `ReadFile`（函数体） |
| **跨模块关系** | `Agent(subagent_type="explore")` |
| **大型目录扫描** | `Shell` (`tree` / `dir`) |
| **批量生成笔记** | `WriteFile` / `StrReplaceFile` |

### Explore Agent 触发条件

满足以下任一条件时启动 `subagent_type="explore"`：
1. 目标模块源码文件超过 20 个，调用关系复杂
2. 需要追踪一个接口在 3 个以上不同模块中的实现/重写
3. 对某个子系统完全没有概念，需要快速建立认知
4. 怀疑存在某种设计模式，需要跨文件确认

**Prompt 模板**：
```
请对 <项目/模块路径> 进行 read-only 探索。
重点关注：
1. 公共接口头文件有哪些？
2. 核心类/结构体的命名和职责
3. 依赖哪些下层模块？被哪些上层模块依赖？
4. 是否存在代码生成、配置文件或宏驱动的机制？
返回简洁概览：目录结构、关键文件名、核心类列表、设计模式推测。
```

### Coder Agent 触发条件

当涉及以下任务时启动 `subagent_type="coder"`：
- 将大量零散发现整理为一篇连贯的笔记
- 绘制 Mermaid 类图/流程图/时序图
- 编写辅助分析脚本

## 笔记产出规范

> **注意**：产出源码分析笔记时，必须结合使用 `/obsidian-markdown` skill，确保正确使用 Obsidian 的 wikilink、embed、callout、frontmatter 等语法规范。

### 文件名与路径

**所有笔记必须输出到 `Game/` 目录下**，并严格按照 `Game/00-引擎与游戏全解析主索引.md` 中的阶段分类进行归档：

- 若索引中已规划该笔记的位置（如 `第二阶段-基础层/`），则必须放到对应的子目录中
- 若索引中尚无该条目，先向用户确认插入位置后再创建文件

文件名格式：
```
<模块>-源码解析：<主题>.md
```
例：`base-源码解析：字符串系统.md`、`core-源码解析：ECS 内存布局.md`

专题笔记：
```
专题：<主题>.md
```

> **索引对齐要求**：若索引中已存在该模块的 wikilink 名称，产出的笔记标题、文件名、aliases 必须与索引表格中的计划笔记保持一致，确保双向链接可自动建立。

### 笔记模板

完整模板参见 [references/note-template.md](references/note-template.md)。生成笔记时必须包含：
- frontmatter（title, date, tags, aliases）
- 返回索引的导航链接
- 模块定位、接口梳理、数据结构、行为分析、上下层关系、设计亮点、关键源码片段、关联阅读

### 代码引用规范

- 必须标明**文件路径**和**行号范围**
- 只引用最关键的部分，避免大段复制无关代码
- 对引用代码添加中文注释说明

示例格式：
```markdown
> 文件：`chaos/base/String.hpp`，第 45~62 行

public:
    // 默认使用全局内存池分配器
    String(Allocator* alloc = GetDefaultAllocator());

## 索引对齐规范

本 skill 服务于 `chaos` 引擎 + `wolfgang/proven_ground` 游戏项目的全解析计划。每次分析前、分析中、分析后必须参考 `Game/00-引擎与游戏全解析主索引.md`：

### 分析前
1. **必须读取** `Game/00-引擎与游戏全解析主索引.md`，确认目标模块所属的阶段（第一阶段~第九阶段）。
2. 查找索引中是否已有该模块的"计划笔记"和"分析重点"。
3. 若存在，本轮分析的 scope 必须覆盖索引表格中标注的"分析重点"。
4. 若不存在，向用户确认是否要在索引中新增条目，并提议插入位置（阶段/子分类）。

### 分析中
- 笔记内容结构应直接回应索引中的"分析重点"。
- 若索引要求分析"SoA 存储、Archetype 分组"，则笔记必须包含 SoA 和 Archetype 的专门章节，而不是泛泛而谈 ECS。
- 跨模块关系章节优先链接到索引中同阶段或相邻阶段的已有笔记。

### 分析后
- 在笔记末尾添加"索引状态"标注：所属阶段、与索引中计划笔记的对应关系。
- 如用户未明确禁止，主动提议更新 `Game/00-引擎与游戏全解析主索引.md` 中对应条目的状态（⬜ → 🔄 → ✅）。

## 链接维护规范

产出或更新笔记后，必须执行以下链接维护动作：

1. **笔记 → 索引**：在笔记顶部添加返回对应索引的导航链接。若笔记位于某阶段子目录（如 `第二阶段-基础层/`），返回链接指向 `[[00-引擎与游戏全解析主索引|全解析主索引]]`。
2. **索引 → 笔记**：检查 `Game/00-引擎与游戏全解析主索引.md`，确认该笔记的 wikilink 已存在且能正确链接。若用户允许，更新该条目的状态图标（⬜ → ✅ 或 🔄）。
3. **跨笔记链接**：在"关联阅读"章节中，使用 wikilink 链接到索引中同阶段或相邻阶段的已有笔记。优先使用索引中已有的笔记名称作为 link text。

## 迭代式深化策略

每次分析分三轮进行，每轮结束后应告知用户本轮产出，等待确认后再进入下一轮。**三轮分析必须在 `Game/00-引擎与游戏全解析主索引.md` 的框架下进行**。

### 第一轮：骨架扫描（What / 接口层）
**目标**：建立模块地图，确认它在全解析索引中的坐标。

- 读取 `Game/00-引擎与游戏全解析主索引.md`，确认目标模块所属阶段和分析重点
- 读取模块根目录的 `CMakeLists.txt`，了解目标名、依赖项、源文件列表
- 读取所有 `include/` 下的公共头文件
- Grep 查找命名空间、核心类、下游引用点
- 输出一篇快速概览笔记，覆盖索引中该模块的"分析重点"条目（优先接口层面）

### 第二轮：血肉填充（How / 数据层 + 逻辑层）
**目标**：深入核心类与算法，填充索引要求的分析重点。

- 根据索引"分析重点"选取 2~3 个核心类，读取 `.hpp` + `.cpp`
- 追踪关键数据结构的生命周期和内存布局（第 2 层）
- 追踪核心成员函数的调用链，绘制 Mermaid 流程图/时序图（第 3 层）
- 补充多线程同步、性能优化、上下层交互点的分析

### 第三轮：关联辐射（Context / 跨模块关系）
**目标**：将模块放回引擎/游戏的完整链路中。

- Grep 查找上层调用者和同阶段相关模块的引用
- 分析数据流入和流出的完整路径
- 补充"与上下层的关系"和"关联阅读"
- 若索引中存在同主题的"专题"（如`专题：内存管理全链路`），说明该模块在专题链路中的位置
- 更新 `Game/00-引擎与游戏全解析主索引.md` 中的对应状态（与用户确认后执行）

## 用户任务格式与 AI 承诺

### 理想任务格式

鼓励用户每次请求包含：
- 目标模块（如 `core/ECS`）
- 目标阶段（参考 `Game/00-引擎与游戏全解析主索引.md`）
- 分析轮次（第一轮/第二轮/第三轮）
- 特别关注点（可覆盖索引中的"分析重点"）
- 已知信息
- 是否允许更新索引状态

### AI 响应承诺

1. **产出物明确**：要么是一篇写入文件的笔记，要么是一份结构化发现摘要
2. **不确定性标注**：基于推测的结论明确标注"推测"或"待确认"
3. **引用可追溯**：所有源码结论都有文件路径和行号支撑
4. **不发散**：严格限定在用户指定的模块范围内
