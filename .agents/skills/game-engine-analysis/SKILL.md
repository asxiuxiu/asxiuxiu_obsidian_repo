# 游戏引擎源码分析

系统化源码分析工作流，用于对 chaos 引擎、Unreal Engine、Bevy 引擎及 wolfgang/proven_ground 游戏项目的大型代码库进行模块级架构解析。

## 支持的引擎

| 引擎 | 语言 | 对象模型 | 构建系统 | 典型用途 |
|------|------|---------|---------|---------|
| **chaos** | C++ | OOP + 自定义 ECS | CMake | 中型商业引擎 |
| **Unreal Engine** | C++ | UObject 继承树 | UBT (.Build.cs) | 巨型工业级引擎 |
| **Bevy** | Rust | 原生 ECS | Cargo | 现代开源 ECS 引擎 |

## 触发条件

- 用户请求分析 chaos / UE / Unreal Engine / Bevy 的源码
- 用户提到"三层剥离法"或"源码级架构解析"
- 用户请求"生成游戏引擎源码分析笔记"且明确提到上述引擎之一

## 源码分析笔记的定位

> **源码分析笔记 ≠ 学习笔记。** 源码分析面向有编程基础、想深入引擎实现的开发者；学习笔记（由 `note-refine` 规范）面向零基础初学者。

| 维度 | 学习笔记 | 源码分析笔记 |
|------|---------|-------------|
| 核心目标 | 建立直觉、理解"为什么需要" | 建立代码级精确认知、理解"怎么实现" |
| 首要标准 | 零基础能顺畅读完 | **拿着笔记能找到代码** |
| 比喻 | 贯穿全文 | **仅限引言 1~2 个** |
| 代码引用 | 少量示例 | **每个核心论点配代码片段** |

### 与 `note-refine` 的关系

- **保留**：零假定（术语首次出现给 1~2 句人话）、Why 先行、用自己的话写、因果推导
- **弱化**：比喻贯穿、问题链驱动（适合架构演进，不适合逐行代码解析）
- **放宽**：表格可用于数据结构字段说明，但首次引入概念仍需先叙述

### 核心承诺

1. **拿着笔记能找到代码**：任何结论都有精确文件路径 + 行号。
2. **数据结构可见**：核心 struct 字段关系用 ASCII 树或 Mermaid 类图呈现。
3. **执行流程可跟**：核心函数逐行/逐块解释，配 Mermaid 流程图/时序图。
4. **设计决策有解释**：不仅说"是什么"，还说"为什么选这个方案"。

## 核心方法论：三层剥离法

分析必须按以下三层递进，**严禁未读完公共接口就深入实现细节**。

### 第 1 层：接口层（What）
**目标**：建立模块地图，理解对外暴露的能力边界。

- 读取模块构建定义（`CMakeLists.txt` / `.Build.cs` / `Cargo.toml`）
- 阅读公共接口（`include/` / `Public/` / `src/lib.rs`），列出核心 struct/class/trait 和 public 方法的完整清单
- 识别反射/元数据机制，Grep 查找下游引用点

### 第 2 层：数据层（How - Structure）
**目标**：理解核心数据结构、内存布局、状态流转。

- 找到核心 struct/class，**逐一解释字段语义、类型选择理由、与其他字段的关系**
- **绘制结构关系图**（ASCII 树或 Mermaid 类图）
- 追踪生命周期（创建 → 使用 → 销毁），标注内存分配来源（栈/堆/池/ECS World）
- 对 ECS 引擎重点分析 Component 存储模型（Sparse Set / Archetype / Table）

### 第 3 层：逻辑层（How - Behavior）
**目标**：理解关键算法的执行流程和动态行为。

- **选取 2~3 个核心函数，逐行追踪调用链**，按"输入 → 关键判断 → 状态变更 → 输出"拆解
- **标注关键决策点**：每个核心分支解释"条件是什么、对应什么场景、为什么在这里判断"
- 分析多线程同步策略（chaos 线程池/锁、UE 多线程管线、Bevy System 并行调度）
- 绘制 Mermaid 流程图/时序图

## 引擎适配层

### chaos 适配

| 维度 | 约定 |
|------|------|
| 模块定义 | `CMakeLists.txt` |
| 公共接口 | `include/*.hpp` |
| 实现文件 | `.hpp` + `.cpp` |
| 反射 | 手动注册 |

### UE 适配

| 维度 | 约定 |
|------|------|
| 模块定义 | `*.Build.cs` |
| 公共接口 | `Public/*.h`、`Classes/*.h` |
| 实现文件 | `Private/*.cpp` |
| 反射 | UHT 生成 `.generated.h`，**以原始 `.h` 为准** |

**关键目录**：`Engine/Source/Runtime/`（核心）、`Editor/`、`Programs/`（UBT/UHT）、`Developer/`。

### Bevy 适配

| 维度 | 约定 |
|------|------|
| 模块定义 | `Cargo.toml` |
| 公共接口 | `crates/<crate>/src/lib.rs` 中 `pub` API |
| 实现文件 | `src/*.rs` |
| 反射 | Rust derive 宏 |
| 对象模型 | Entity + Component + System + Resource |

**特别提示**：不要以 OOP 思维找"类"，关注 Component 存储、System 的 Query 访问、Schedule 的并行依赖图、World 作为全局状态容器。

## 源码级详尽度标准

### 代码引用密度

| 层次 | 要求 |
|------|------|
| 接口层 | 每个核心 public API 配代码片段 |
| 数据层 | 每个核心数据结构配完整定义 + 字段注释 + 结构图 |
| 逻辑层 | 每个核心函数配关键代码 + 逐行/逐块解释 |

> **底线**：无源码佐证的结论必须标注为**推测**。

### 文件路径与行号

- 标明文件路径（相对于 `*_SOURCE_ROOT`）和行号范围，格式：`第 45~62 行`
- 代码生成需备注来源（如 `"由 UHT 生成"`）
- 示例：

```markdown
> 文件：`crates/bevy_ecs/src/system/system.rs`，第 45~62 行

```rust
// System trait：所有可运行的逻辑都必须实现此 trait
pub trait System: Send + Sync + 'static {
    fn run(&mut self, world: &mut World);
}
```
```

### 中文注释规范

注释回答"这句代码在做什么"和"为什么这么做"，不要只翻译变量名。

```rust
// ❌ 差的注释
self.running_systems.remove(system_index); // 从运行中系统移除

// ✅ 好的注释
self.running_systems.remove(system_index); // 标记系统已完成，释放其占用的数据访问权
```

### 数据结构图解

核心数据结构必须用 ASCII 树或 Mermaid 类图呈现内部关系。

## 设计决策分析模板

源码分析必须回答"**为什么这样设计**"。对关键设计决策，按以下模板分析：

```markdown
### 设计决策：XXX

**问题背景**：<解决什么问题？>

**naive 方案**：<不了解时可能怎么实现？>
- 优点：...
- 缺点：...

**实际方案**：<源码采用的方案>
- 优点：...
- 代价：...

**对比其他引擎**（如果已知）：
- UE 的做法：...
- chaos 的做法：...
```

**必须分析的场景**：使用了不常见的数据结构、存在明显性能优化、并发策略选择、抽象层级设计。

每次分析前、中、后参考对应引擎索引（chaos→`Game/00-引擎与游戏全解析主索引.md`，UE→`Notes/UE/00-UE全解析主索引.md`，Bevy→`Notes/Bevy/00-Bevy全解析主索引.md`）。

- **分析前**：读取索引，确认阶段和分析重点；若不存在则向用户确认新增位置。
- **分析中**：内容回应索引"分析重点"，跨模块链接优先指向同阶段已有笔记。
- **分析后**：笔记末尾添加"索引状态"；如用户未禁止，提议更新索引状态（⬜ → 🔄 → ✅）。

## 迭代式深化策略

每轮结束告知用户产出，等待确认后再进入下一轮。

### 第一轮：骨架扫描（What）
- [ ] 读取索引，确认阶段和分析重点

- [ ] Grep 下游引用点

- **不做**：不深入实现、不追踪调用链、不分析算法

### 第二轮：血肉填充（How）
- [ ] 选取 2~3 个核心类，读取实现文件
- [ ] 解剖数据结构：字段解释、结构图、内存布局、生命周期
- [ ] 选取 2~3 个核心函数，逐行追踪调用链，绘制 Mermaid 图
- [ ] 标注关键决策点，分析多线程同步和性能优化
- [ ] 按模板完成 2~3 个设计决策分析

### 第三轮：关联辐射（Context）
- [ ] Grep 上层调用者和同阶段相关模块引用
- [ ] 分析数据流入流出路径
- [ ] 补充上下层关系和关联阅读
- [ ] 跨引擎对照（如有）
- [ ] 汇总设计亮点，整理"关键源码片段"章节
- [ ] 更新索引状态（与用户确认后执行）

## 源码分析笔记质量自检清单

产出前自检，达标再交付。

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 源码可追溯性 | 每个核心 struct/function 有代码片段 + 文件路径 + 行号；无佐证结论标注"推测" |
| 2 | 数据结构解剖 | 字段逐一解释；关系用 ASCII 树/Mermaid 呈现；标注内存来源；追踪生命周期 |
| 3 | 执行流程追踪 | 2~3 个核心函数调用链；关键分支"条件→场景→原因"；有 Mermaid 图；分析多线程同步 |
| 4 | 设计决策分析 | 2~3 个决策按模板分析；说明"为什么不用 naive 方案"；标注性能优化手段 |
| 5 | 零假定 | 首次出现的术语有 1~2 句人话；缩写有全称；引擎特有概念有解释或链接 |
| 6 | 叙述策略 | 比喻 ≤2 个且仅限引言；正文精确技术描述；表格不用于首次引入概念；代码有中文注释 |
| 7 | 跨模块关系 | 有上下层关系章节；关联阅读 wikilink 可访问 |
| 8 | 索引与链接 | 顶部有返回索引链接；索引状态已填；文件名/标题/aliases 与索引对齐 |

**评分**：优秀（7-8 条通过）→ 直接交付；合格（5-6 条）→ 补充后交付；不合格（0-4 条）→ 返回对应轮次补充。

## 用户任务格式与 AI 承诺

### 理想任务格式

- 目标引擎（chaos / UE / Bevy）
- 目标模块
- 目标阶段
- 分析轮次（第一轮/第二轮/第三轮）
- 特别关注点
- 已知信息
- 是否允许更新索引状态

### AI 响应承诺

1. **产出物明确**：写入文件的笔记或结构化发现摘要。
2. **不确定性标注**：推测结论明确标注"推测"或"待确认"。
3. **引用可追溯**：所有源码结论有文件路径（相对于 `*_SOURCE_ROOT`）和行号。
4. **不发散**：严格限定在用户指定的引擎和模块范围内。
5. **引擎适配**：自动识别引擎的构建系统、目录约定和对象模型。
6. **自检前置**：产出前完成质量自检清单，确认达标再交付。
