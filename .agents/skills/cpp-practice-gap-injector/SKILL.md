# C++ 知识缺口注入器

## 定位

当用户在外部学习、阅读源码或工作中遇到不理解的 C++ 概念时，将这个概念转化为刻意练习题，沉淀到**知识缺口池**中。不强制排期，不打乱主线进度。

## 触发条件

- "这个概念我不理解，帮我加到练习计划里"
- "遇到个不懂的 C++ 概念，想练一下"
- "把这个加入训练计划"
- 用户在外部学习/阅读时遇到不理解的概念，希望转化为刻意练习

## 前置步骤

1. 读取 `workspace/cpp-recovery/.practice-tracker/state.json`
2. 读取 `Notes/学习计划/C++基础恢复70天计划.md`
3. 读取 `workspace/cpp-recovery/PROGRESS.md`

## 工作流程

### 第一步：概念分析

根据用户描述的概念，分析：

| 字段 | 分析内容 |
|------|---------|
| 概念名称 | 用户遇到的概念 |
| 能力维度 | 映射到 6 维能力矩阵（syntax/pointer/template/memoryMgmt/design/concurrency） |
| 难度评估 | ⭐ ~ ⭐⭐⭐⭐⭐，参考当前能力水平和进度 |
| 关联阶段 | 该概念属于 70 天计划中的哪个阶段主题 |

### 第二步：设计练习题

为概念设计一道**最小可练习单元**：

- 任务描述清晰（做什么）
- 考察点明确（要掌握什么）
- 只描述任务，不给实现细节或代码
- 关联到游戏引擎/实际业务场景
- 预期完成时间 30-40 分钟

> **🔴 铁律：严禁直接给出完整实现或答案。** 任务描述只能说明"要做什么"和"考察点"，不能包含"怎么做"的详细步骤或参考代码。

### 第三步：确定归入阶段

根据概念属性，归入对应阶段的**附加区域**：

| 阶段 | 主题 | 适合归入的概念类型 |
|------|------|-------------------|
| 第一阶段 | C++ 语法热身 | 语言机制（ADL、SFINAE、右值引用等） |
| 第二阶段 | 线性数据结构 | 指针技巧、内存布局 |
| 第三阶段 | 树与哈希 | 复杂数据结构变种 |
| 第四阶段 | 算法与图论 | 特定算法变体 |
| 第五阶段 | 并发与系统设计 | 设计模式、架构概念 |

### 第四步：用户确认

向用户展示：

```
📥 知识缺口捕获
━━━━━━━━━━━━━━━━━━━━━━

🎯 概念：<概念名>
📊 能力维度：<维度>
⭐ 难度：<难度>
📦 归入阶段：<阶段> · 附加
⏱️ 预计用时：30-40 分钟

📝 练习题：
<任务描述>

━━━━━━━━━━━━━━━━━━━━━━

是否加入缺口池？说"加入"确认，或告诉我你想怎么调整。
```

### 第五步：执行归入

用户确认后，执行以下修改：

1. **修改计划笔记**：在对应阶段的"附加"区域追加一行
2. **修改 PROGRESS.md**：在缺口池表格中追加一行
3. **更新 state.json**：在 `planAdjustments` 中追加记录

#### 计划笔记追加格式

在对应阶段附加区域的表格中追加：

```
| Gap-<NN> | <题目描述> | <难度> | <能力维度> | [ ] |
```

#### state.json 更新

```json
{
  "type": "gap-pool",
  "concept": "概念名",
  "dimension": ["维度1", "维度2"],
  "difficulty": 星级数字,
  "phase": "第X阶段",
  "gapId": "Gap-NN",
  "taskName": "题目描述",
  "date": "YYYY-MM-DD"
}
```

### 第六步：通知

归入完成后，告知用户：
- 练习题已沉淀到缺口池的哪个阶段
- 主线进度完全不受影响
- 何时想练都可以，工作区目录建议命名为 `gap<编号>_<主题>/`

## 注意事项

1. **不打乱主线**：70 天计划的 Day 编号永远不变，缺口池完全独立于主线
2. **阶段化归类**：每个缺口题必须归入对应阶段，不能丢在一个大杂烩里
3. **记录调整历史**：所有归入记录到 state.json 的 planAdjustments
4. **难度匹配**：新题难度不应超过用户当前能力 + 1 颗星
5. **概念粒度**：每个缺口题聚焦一个概念，不合并多个概念
6. **不强制排期**：缺口池中的题没有 deadline，用户自选时间练习
