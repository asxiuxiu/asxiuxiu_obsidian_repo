# 自研游戏引擎知识库（Self Game Engine）

本 skill 负责将 `game-engine-analysis` 产出的源码分析结论，进一步提炼为**可带走的通用工程知识**。产出的笔记完全脱敏，与公司源码解耦，是你未来 vibe coding 自己引擎时的个人知识资产。

**核心使命**：本 skill 产出的所有笔记必须能被组织成**一个连贯的、从零开始搭建游戏引擎的渐进式计划**。每篇笔记不仅是知识讲解，更是引擎构建流程中的一个具体步骤。

## 触发条件

- 用户提到"费曼笔记"、"脱敏笔记"、"通用知识"
- 用户要求"把源码分析整理成我自己能用的"
- 用户说"我想自己做引擎，从这个模块能学到什么"
- 用户提到"vibe coding" + "游戏引擎"
- 用户提到"AI 友好"、"Agent 编排"、"MCP"、"可观测性"、"确定性"与引擎设计
- 用户要求思考"源码不是 ECS，到 ECS 该怎么办"
- 在 `game-engine-analysis` 三轮分析完成后，用户要求产出配套的个人学习笔记
- 用户要求"给我一份搭建引擎的路线图"、"下一步该学什么"、"按照什么顺序 build 我的引擎"

## 与 game-engine-analysis 的关系

| skill | 产出位置 | 内容性质 | 是否脱敏 |
|-------|---------|---------|---------|
| `game-engine-analysis` | `Game/` | 公司源码的具体结构、文件路径、行号、调用链 | ❌ 不脱敏 |
| `self-game-engine` | `Notes/SelfGameEngine/` | 通用设计模式、最小实现、演进逻辑、设计权衡 | ✅ 完全脱敏 |

**工作流**：
1. `game-engine-analysis` 先完成源码分析（或读取已有的源码分析笔记）
2. `self-game-engine` 基于源码结论，重写为通用知识笔记
3. **反馈循环**：在提炼过程中，如果发现源码笔记存在盲区、设计动机不明、或无法回答"为什么这样实现"时，**主动调用 `game-engine-analysis` 回探源码**，补充 `Game/` 目录下的分析笔记
4. 结合补充后的源码分析 + 互联网检索 + 个人理解，重写为更准确的通用知识
5. 通用知识笔记是**独立自洽**的，不依赖源码分析笔记即可理解

### 信息来源

`self-game-engine` 的产出**不应局限于已有的源码分析笔记**。在构建通用知识时，AI 应主动利用互联网搜索（web search）来补充、验证和拓展以下内容：
- 通用设计模式的标准定义和最佳实践
- 行业内对该问题的常见解决方案（如开源引擎中的同类实现）
- 算法原理的准确性和边界条件
- 现代 C++ 特性或标准库对该问题的支持

**互联网是通用知识的放大器**。当笔记中仅有公司特定的实现细节时，应通过搜索将其升维为行业共识；当对某个概念的普适性存疑时，应通过搜索交叉验证。最终产出的笔记可以是"笔记灵感 + 网络求证"的融合结果，而不必是源码分析的简单改写。

### 反馈循环：发现缺口时回探源码

提炼通用知识的过程不是单向的。当你遇到以下情况时，**必须暂停当前提炼，调用 `game-engine-analysis` 深入源码**：

| 信号 | 行动 |
|------|------|
| 源码分析笔记对某个关键设计"只描述了是什么，没解释为什么" | 回探源码，追踪该设计的调用上下文和演进历史 |
| 发现两个源码模块的描述相互矛盾，或存在逻辑断层 | 回探源码，定位中间的衔接层或数据流转路径 |
| 无法写出最小实现，因为不清楚核心数据结构的实际约束 | 回探源码，提取关键结构体的字段和生命周期 |
| 互联网上的通用方案与公司实现差异巨大，且无法判断优劣 | 回探源码，理解公司实现针对的具体瓶颈或平台约束 |

**行动规范**：
1. 明确向用户说明发现了什么新问题，以及为什么需要回探源码
2. 调用 `game-engine-analysis` 完成针对性探索
3. 将新结论写入 `Game/` 目录下的相应笔记（新建或追加）
4. 返回 `self-game-engine`，继续提炼通用知识
5. 最终产出的费曼笔记应体现这一轮的认知升级

## 核心方法论：费曼提取法

基于 AGENTS.md 的费曼学习法，每篇笔记必须回答三个问题：

### 前置原则 1：擅长提出问题，拒绝概念空降

在写下任何概念之前，必须先问自己三个问题：
- **这个概念解决什么问题？** 如果回答不上来，就不该在这里出现。
- **读者此时知道什么？** 每个新概念必须稳稳地站在读者已知的台阶上，不能空降。
- **这个术语引入得是否过于突然？** 如果需要一个从未提及过的背景知识才能理解，就必须先补充那个背景知识，或换一种更朴素的表达方式。

> 费曼学习法的核心不是"解释清楚"，而是"在解释之前先质疑自己是否真正理解"。每写一个段落，都反思一次：如果我是一个完全不了解引擎的初学者，看到这里会不会愣住？

### 前置原则 2：默认面向 ECS 与 AI 友好架构

用户的自研引擎目标明确：**基于 ECS 架构**，并且**为 AI 协作（vibe coding / Agent 编排）而设计**。因此，所有产出的通用知识必须满足以下默认约束：

1. **即使源码不是 ECS，也要思考"到 ECS 该怎么办"**
   - 如果源码是 OOP 架构，提炼笔记时必须在"How"阶段明确指出：哪些状态应该被拆成组件？哪些继承关系应该变成组合？系统的迭代模式在 ECS 下会如何重写？
   - 不要把源码的 OOP 实现直接照搬到通用知识中作为"最佳实践"，而是把它当作**需要被重构到 ECS 的素材**。

2. **所有模块设计必须回答：AI 如何观察它？AI 如何操作它？**
   - **可观测性**：该模块的状态是否能被序列化、反射、diff？AI 能不能不读头文件就理解数据结构？
   - **确定性**：该模块的执行是否是输入确定的？是否支持回放（Replay）和快照（Snapshot）？
   - **工具边界**：如果 AI 要通过 MCP/Tool 接口操作这个模块，接口应该长什么样？修改的副作用是否可控？

3. **多 Agent 编排是架构的一等公民**
   - 在讨论系统调度、资源管理、编辑器框架时，默认要考虑**多个专项 Agent 并行工作**的场景。
   - 例如：物理调优 Agent 和 Gameplay Agent 能否同时工作？它们之间的状态隔离和冲突合并机制是什么？

4. **骨架先行，血肉后补**
   - 任何新模块的费曼笔记，必须先从"最小实现"开始，而且这个最小实现必须是**AI 能理解和操作的平铺数据结构**。

### Why：为什么游戏引擎需要 <X>？
- 从"没有这个系统时世界有多糟"开始
- 列出 2~3 个真实场景
- 说明这是"必须"还是"优化"

### What：最简化版本的 <X> 长什么样？
- 用 50~100 行代码写出**能跑的最小实现**
- 这是 vibe coding 时可以直接复制的骨架
- 明确指出最小实现已经解决了什么、还缺什么

### How：真实引擎的 <X> 是如何一步一步复杂起来的？
- 阶段 1：最小实现 → 能用（解决某个具体工程问题）
- 阶段 2：能用 → 好用（解决迭代效率和调试问题）
- 阶段 3：好用 → 工业级（解决性能和规模问题）
- 每个阶段说明**触发原因**和**代码层面的变化**

## 渐进式构建路线图：从零开始搭引擎

本 skill 的产出必须被组织为**一个可执行的引擎构建计划**。用户应该能够按照推荐的笔记顺序，一步一步地 vibe coding 出自己的引擎。每篇笔记不仅是知识讲解，更是引擎的一个构建步骤。

### 构建阶段与里程碑

引擎构建分为 6 个阶段，每个阶段结束时应达到一个明确的**可运行里程碑**：

| 阶段 | 里程碑 | 核心验收标准 |
|------|--------|-------------|
| **阶段 0** | 设计原则确立 | 确定了 ECS + AI 友好 + 确定性的元规则。所有后续设计决策有统一判据。 |
| **阶段 1** | 最小引擎骨架可编译运行 | `main()` 能启动，拥有最简陋的 World、Entity、Component、System。这是后续一切的基础。 |
| **阶段 2** | 基础工具就位 | 数学库、字符串、容器/分配器已集成。引擎能处理基本的数值计算和内存分配。 |
| **阶段 3** | 核心运行时闭环 | ECS 查询/存储、场景图、反射、Tick 调度全部连通。引擎能"描述一个世界并让它运转"。 |
| **阶段 4** | 第一帧画面渲染 | RHI 抽象、资源管理、渲染生命周期已接通。屏幕上有可见输出（三角形或立方体）。 |
| **阶段 5** | 工具链与 AI 桥接 | 编辑器、脚本系统、AgentBridge/MCP 已连通。AI Agent 能观察并操作引擎状态。 |
| **阶段 6** | 工业级专题增强 | 网络同步、高级内存管理、多 Agent 编排沙箱。引擎具备生产环境的扩展能力。 |

**严格依赖关系**（下游模块必须以上游模块为前提。如果上游笔记尚未产出，AI 应建议先完成上游模块）：

```
    │        └──► 容器与分配器

    │        ├──► 场景图与变换
    │        ├──► 系统调度与确定性
    │        └──► 反射与序列化
    │                    │
    │                    ├──► 编辑器框架
    │                    ├──► 脚本系统
    │                    └──► MCP与Agent桥接层
    │
    ├──► RHI抽象层
    │        │
    │        ├──► 资源管理

         │

```

**必须遵守的依赖规则**：
1. `AI友好的引擎架构` → **所有其他笔记的前置条件**。任何模块的产出都必须回扣到这篇笔记中定义的原则。
2. `从零开始的引擎骨架` → **所有运行时、渲染、工具链笔记的前置条件**。没有 ECS 骨架，讨论场景图或渲染管线都是空中楼阁。
3. `组件系统架构` → `场景图与变换`、`反射与序列化`、`系统调度与确定性` 的前置条件。场景图是 ECS 的组件组合，调度是 ECS 的系统执行。
4. `反射与序列化` → `MCP与Agent桥接层`、`编辑器框架` 的前置条件。AI 和编辑器都需要运行时自描述能力。
5. `RHI抽象层` → `渲染一帧的生命周期`、`资源管理` 的前置条件。不理解 RHI 抽象，就无法解释资源生命周期。
6. `系统调度与确定性` → `网络同步全链路`、`多Agent编排与沙箱` 的前置条件。同步和编排都建立在确定性执行模型之上。

### 连贯性强制要求

每篇产出的费曼笔记，除了回答 Why/What/How 之外，**必须**包含以下三个连贯性元素（通常放在笔记开头和结尾的醒目位置）：

1. **前置依赖声明**：在笔记开头明确列出"在写这篇笔记之前，你应该已经实现了/阅读了哪些内容"，并给出 wikilink。
2. **本模块对引擎的增量**：清晰说明"读完/实现完这篇笔记后，你的引擎相比之前新增了哪些能力"。
3. **下一步预告**：在笔记结尾预告"下一篇应该 build 什么"，以及为什么现在时机成熟了。

**示例模板**（应出现在每篇笔记中）：
```markdown
> **前置依赖**：[[从零开始的引擎骨架]]、[[组件系统架构]]
> **本模块增量**：在最小 ECS 骨架上增加 Spatial Query 和 Transform Hierarchy 能力。
> **下一步**：[[反射与序列化]]，因为场景中的实体需要被序列化保存后才能被 Agent 观察。
```

### 索引即构建手册

`索引.md` 不应只是目录列表，而是一本**构建手册**。除了列出所有笔记外，必须：
- 按构建阶段分组，标注每个模块所属的里程碑
- 明确显示推荐的阅读/实现顺序（可用流程图或有序列表）
- 在每个里程碑处说明"完成到这里的最低验收标准"
- 对于尚未产出的模块，预留位置并标注状态（待产出 / 进行中 / 已完成）

### 文件名格式

```
<主题>.md
```

例：`字符串系统.md`、`组件系统架构.md`、`RHI抽象层.md`

**禁止**在公司类名前加 `feynman-` 前缀。这些笔记是**你自己的知识**，不是源码的附庸。

### ECS 映射强制要求

当源码分析笔记基于 OOP 或非 ECS 架构时，`self-game-engine` 的产出**不得**简单复述源码结构。必须在笔记中显式包含以下内容之一：

- **ECS 重构映射表**：列出源码中的类/继承链，对应到 ECS 中的组件、系统、实体组合。
- **Why ECS 更适合该模块**：从 cache locality、组合灵活性、AI 可观测性角度解释。
- **How 演进中的 ECS 化步骤**：从最小实现到工业级的每一阶段，明确说明 ECS 存储模型如何升级（如从 `ComponentArray` → `Archetype` → `Chunk-based`）。

### AI 友好设计红线（必须遵守）

每篇产出的费曼笔记，在讨论任何模块时，**必须**检查以下清单：

| 检查项 | 要求 |
|--------|------|
| **状态平铺** | 该模块的所有可变状态是否可以用 ECS 组件表示？如果不行，必须解释为什么并给出替代方案。 |
| **自描述** | AI 是否可以在不读 C++ 头文件的情况下，通过运行时注册表知道该模块的数据结构？ |
| **确定性** | 给定相同的输入序列，该模块的输出是否可复现？是否支持快照保存与回滚？ |
| **工具边界** | 如果 AI 通过 `AgentBridge` / MCP 接口操作该模块，接口的输入输出是否是结构化的 JSON/Schema？ |
| **Agent 安全** | 该模块是否支持沙箱、事务或白名单机制，防止 AI 的误操作破坏核心状态？ |

### 脱敏红线（必须遵守）

**✅ 可以写**：
- 通用设计模式（Handle-Table、Command Pattern、Double Buffering）
- 算法原理（拓扑排序、A* 寻路、BVH 构建）
- 工程权衡的理由（"为了减少 cache miss，采用了 SoA 布局"）
- 50~100 行的最小实现示例（用通用命名重写）
- 复杂度演进路线图（"从直接 new → 对象池 → 分代回收"）
- AI 可观测性设计（Change Log、Snapshot、Replay、AgentBridge）
- 多 Agent 编排模式（Orchestrator、专项 Agent、组件白名单）

**❌ 禁止写**：
- 公司源码中的具体类名、函数名、变量名
- 具体文件路径（如 `chaos/core/ECS/private/InternalArchetype.cpp`）
- 游戏项目特有的业务逻辑（如 `ProvenGround` 中的特定技能 ID、关卡名）
- 内部工具链的详细配置（CI 脚本中的服务器 IP、密钥名）
- 任何可能暴露公司技术债务或未完成功能的负面评价

### 代码规范

- 费曼笔记中的代码必须是**通用化重写的最小实现**，不是源码片段
- 代码必须能独立编译运行（或接近能运行）
- 优先使用标准 C++，不依赖公司内部宏或基础设施
- 每段代码示例都应说明它在引擎构建路线图中的位置（"这是阶段 3 的核心运行时代码"）

## 迭代产出策略

### 第一轮：Why + What（骨架）
**输入**：源码分析笔记的接口层结论
**步骤**：
1. 阅读源码分析笔记，梳理接口层结论
2. **确认前置依赖模块的笔记是否已经产出**。如果没有，向用户说明当前模块依赖哪些上游知识，并建议优先完成上游模块或简要回顾其结论。
3. **如果发现"为什么需要这个接口/抽象"的答案不清晰，回探源码**（调用 `game-engine-analysis`），补充设计动机到 `Game/` 笔记
4. 结合补充后的理解，写出通用化的 Why + What

**产出**：
- 该模块解决的问题背景
- 50~100 行最小实现代码
- 最小实现的边界分析
- **前置依赖声明**和**本模块增量**说明

### 第二轮：How 演进（血肉）
**输入**：源码分析笔记的数据层 + 逻辑层结论
**步骤**：
1. 梳理数据结构和核心逻辑链路
2. **如果发现阶段划分模糊、或某个优化点的触发条件不明确，回探源码**（调用 `game-engine-analysis`），补充数据流转或性能测试背景到 `Game/` 笔记
3. 用更准确的认知重写演进路径

**产出**：
- 阶段 1~2 的复杂度演进
- 每个阶段新增的设计元素
- 对应的代码变化示意图

### 第三轮：工业级 + 可偷招数（定型）
**输入**：源码分析笔记的跨模块关系 + 性能优化分析
**步骤**：
1. 梳理跨模块耦合、平台适配、极端优化手段
2. **如果某个工业级技巧与通用最佳实践相悖、或原理存疑，回探源码**（调用 `game-engine-analysis`），确认其适用边界，补充到 `Game/` 笔记
3. 输出经过验证的、可迁移的设计清单
4. 在笔记结尾添加**下一步预告**

**产出**：
- 阶段 3 的工业级设计
- 设计权衡表
- "如果我要 vibe coding，该偷哪几招"清单
- 更新 `索引.md`，标注该模块已完成并指向下一个推荐模块

## AI 响应承诺

1. **产出物明确**：一篇写入 `Notes/SelfGameEngine/` 的费曼学习笔记
2. **完全脱敏**：笔记中不出现任何公司源码的具体类名、函数名、文件路径
3. **可带走**：所有代码和结论都是通用化的，可以直接用于个人项目
4. **知识独立**：笔记自成体系，不依赖源码分析笔记即可读懂
5. **索引维护**：如 `索引.md` 已存在，主动提议更新；如不存在，主动创建
6. **顺序连贯**：产出的每篇笔记都明确声明前置依赖、本模块增量和下一步预告，确保用户能按照路线图一步一步搭建引擎
7. **遵循构建阶段**：不会在没有完成前置模块的情况下，直接跳入深层专题（如不会在没有 ECS 骨架时讨论网络同步的回滚细节）
