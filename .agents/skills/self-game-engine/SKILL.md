# 自研游戏引擎知识库（Self Game Engine）

本 skill 负责将 `game-engine-analysis` 产出的源码分析结论，进一步提炼为**可复用的工业级引擎设计知识**。产出的笔记完全脱敏，与公司源码解耦，是构建对标 UE 与 Bevy 的 ECS 游戏引擎的核心设计资产。

**核心使命**：本 skill 产出的所有笔记必须能被组织成**一个工业级 ECS 游戏引擎的完整构建蓝图**。目标不是"先跑起来再说"，而是**在当前阶段把每个模块写到最好**——吸收 UE 的工业级完备性，融合 Bevy 的 ECS 原生表达，产出经得起工业场景检验的设计方案。

> **关键原则 1：笔记是问题驱动的，不是模板驱动的。** 叙事逻辑必须是：**先让人感受到问题存在 → 深入剖析问题的根源与分支 → 探讨多种解决路径 → 分析每条路径的 trade-off → 在关键决策点引入工业参照 → 逐步收敛到推荐方案。** 架构对照只在"解决某个具体子问题有多种方案可选"时才按需引入。
>
> **关键原则 2：UE 优先吸收原则，拒绝简化妥协。** 在方案选择时，**默认优先吸收 UE 的设计优点**。**禁止以"个人项目"、"当前阶段简单优先"、"先过渡后补全"为由选择降级的简化方案。** 如果 UE 的某设计在 ECS 下可直接映射，则直接采用；如果基于 OOP 全局状态，则提取其设计意图后在 ECS 中重新表达，**不得为了降低实现复杂度而丢弃工业级洞察**。最终推荐方案必须同时满足：(a) 保持 ECS 架构的一致性；(b) 在 ECS 兼容前提下，尽可能贴近 UE 的工业级设计。
>
> **关键原则 3：工业实现深度不可省略。** 每篇笔记在讲清楚设计决策（Why/What）之后，**必须提供工业实现深度剖析（How exactly）**。包括：精确到字节的数据结构内存布局、核心算法的工业级代码实现、优化手段的适用边界与代价、以及不同工业实现的对比分析。**如果单篇笔记篇幅过大，应将工业实现深度拆分为独立的子笔记**（如 `[[主题-工业实现深度剖析]]`），在 roadmap 中登记。**绝对不允许以"篇幅不够"或"当前阶段用不到"为由省略工业实现深度。**

## Roadmap 优先原则（最高优先级）

- AI 在采取任何产出行动前，**必须先读取并理解 roadmap**。roadmap 是唯一的结构权威。
- 输出目录、文件名、阶段划分、模块依赖顺序，**全部由 roadmap 定义**。
- 如果用户没有明确给出 roadmap 路径，**主动读取 vault 中可能的 roadmap 文件**；若不存在，则主动询问。
- 当 roadmap 与 skill 的其他建议发生冲突时，**以 roadmap 为准**。

## 触发条件

- 用户提到"费曼笔记"、"脱敏笔记"、"通用知识"
- 用户要求"把源码分析整理成我自己能用的"
- 用户说"我想自己做引擎，从这个模块能学到什么"
- 用户要求思考"源码不是 ECS，到 ECS 该怎么办"
- 用户要求产出配套的费曼设计笔记
- 用户要求"搭建引擎的路线图"、"下一步该学什么"
- 用户要求"对比 chaos 和 UE 的架构"

## 与 game-engine-analysis 的关系

| skill | 产出位置 | 内容性质 | 是否脱敏 |
|-------|---------|---------|---------|
| `game-engine-analysis` | `Game/` / `Notes/UE/` / `Notes/Bevy/` | 引擎源码的具体结构、文件路径、行号、调用链 | ❌ 不脱敏 |
| `self-game-engine` | **以 roadmap 定义为准** | 通用设计模式、演进逻辑、架构对照与设计权衡 | ✅ 完全脱敏 |

**工作流**：
1. `game-engine-analysis` 先完成源码分析（或读取已有的源码分析笔记）
2. `self-game-engine` 基于三端源码结论，重写为通用知识笔记
3. **反馈循环**：在提炼过程中，如果发现源码笔记存在盲区、设计动机不明、或无法回答"为什么这样实现"时，**主动调用 `game-engine-analysis` 回探源码**
4. 结合补充后的源码分析 + 互联网检索 + 工程推演，重写为更准确的通用知识
5. 通用知识笔记是**独立自洽**的，不依赖源码分析笔记即可理解

### 信息来源（四源为参考，问题为驱动）

分析输入来自以下四源，但**输出的叙事结构不以四源为章节，而以问题链为章节**：

1. **chaos 源码分析笔记**（`Game/` 目录）
2. **UE 源码分析笔记**（`Notes/UE/`）— 必须读取，禁止跳过。笔记缺失时调用 `game-engine-analysis` 回探
3. **Bevy 源码分析笔记**（`Notes/Bevy/`）— 必须读取，禁止跳过。笔记缺失时调用 `game-engine-analysis` 回探
4. **互联网搜索**（web search）— 每轮产出前必须执行，禁止省略
5. **工程推演与验证**

**搜索时机与目标**：
- **问题链梳理阶段**：搜索该模块的行业标准定义、经典论文、最被广泛引用的开源实现
- **逐个解决阶段**：搜索当前子问题，开源引擎（Bevy、Flecs、EnTT、Godot）是如何处理的
- **定型阶段**：搜索近 3~5 年的最新进展，确认是否有超越 chaos/UE 的现代方案

**搜索不是可选装饰，而是决策依据**。涉及性能 claim（如"cache friendly"、"O(1)"）时，必须搜索 benchmark 或论文验证。

**搜索后的处理**：将搜索结果与源码结论进行**三角验证**。如果冲突，优先采信**可复现的 benchmark / 学术论文 / 广泛使用的开源实现**。

### 反馈循环：发现缺口时回探源码

| 信号 | 行动 |
|------|------|
| 源码分析笔记对某个关键设计"只描述了是什么，没解释为什么" | 回探源码，追踪该设计的调用上下文和演进历史 |
| 发现两个源码模块的描述相互矛盾，或存在逻辑断层 | 回探源码，定位中间的衔接层或数据流转路径 |
| 不清楚核心数据结构的实际约束，导致无法精确分析分支方案 | 回探源码，提取关键结构体的字段和生命周期 |
| 互联网上的通用方案与公司实现差异巨大，且无法判断优劣 | 回探源码，理解公司实现针对的具体瓶颈或平台约束 |
| UE 与 chaos 对同一问题的实现差异很大，需要判断各自的适用边界 | 同时/分别回探两个源码库，提取设计动机和约束条件 |

**行动规范**：
1. 明确向用户说明发现了什么新问题，以及为什么需要回探源码
2. 调用对应 skill 完成针对性探索
3. 将新结论写入对应目录下的笔记（新建或追加）
4. 返回 `self-game-engine`，继续提炼通用知识

## 核心方法论：深度问题链驱动

本 skill 专注**深度问题链驱动**的知识组织结构。笔记叙述质量（零假定、Why 先行、比喻有效、因果链条、衔接自然等）需符合 `note-refine` skill 的通用规范。**本 skill 不重复定义叙事规范**，只规定以下核心约束。

### 深度问题链的定义

> **一篇好的引擎知识笔记，不是一份设计文档，而是一次深度思维还原——还原工程师在面对空白编辑器时，是如何从"这系统要解决什么问题"出发，逐层剥开问题的表象，发现隐藏在表象之下的分支与矛盾，最终在多条可能的路径中做出取舍，走到"最终代码长什么样"的。**

输出的叙事结构必须是**深度问题链**（Deep Problem Chain），而不是**特征对比表**、**阶段清单**或**浅层 FAQ**。

### 每个子问题必须包含的要素

1. **场景绑定与根因分析**：不只要说"有什么问题"，更要解释"为什么在这个场景下会出问题"、"问题的根因是什么"。
2. **多分支研究**：主动识别所有可行的解决方向，逐一分析其原理、适用场景、隐藏代价、失效条件。
3. **精确代码片段**：给出足以展示关键机制的代码片段——不需要完整可运行程序，但核心逻辑必须精确、无歧义。代码片段必须嵌入在问题探讨的上下文中，禁止孤立展示"完美代码"。
4. **工业实现深度（不可省略）**：在分支研究和决策分析之后，必须提供该子问题的工业级实现剖析——精确到字节的数据结构内存布局、核心算法的可运行代码骨架、工业级优化手段的具体实现、不同工业实现的对比。
5. **按需引入引擎对照**：只有当当前子问题存在**多种可行方案、且需要工业级经验辅助决策**时，才引入 chaos/UE/Bevy 的参考。引入时必须说清楚：**我们正在解决哪个具体问题**？每个引擎在这个**具体子问题**上是怎么选的？为什么？
6. **决策分析与推荐**：**必须有且仅有一条默认推荐路径**，并说明依据。禁止在笔记中呈现两种矛盾方案却不给出明确推荐。
7. **遗留问题**：明确标注"这个推荐方案还遗留哪些问题"，这些问题将成为下一个问题的引子。

### 分支研究的呈现格式

```markdown
### 分支 A：XXX 方案
**核心思路**：……
**代码片段**：……（精确展示关键机制，不需要完整程序）
**适用场景**：……
**隐藏代价**：……（必须诚实暴露）
**失效条件**：……（什么极端输入或使用方式会让它表现糟糕？）

### 分支 B：YYY 方案
……

### 决策分析
在 <具体条件> 下，分支 A 优于分支 B，因为……
在 <另一条件> 下，分支 B 更优，因为……
**默认推荐**：……（必须有且仅有一条默认路径）
**推荐依据**：该方案在 ECS 架构下的表达方式是……；吸收自 UE 的……设计；与 Bevy 的 ECS 做法的对比是……
```

### 判定"最正确方案"的优先级规则

1. **可复现的 benchmark / 性能测试数据**（最高优先级）
2. **被广泛引用的学术论文或工业界标准实现**
3. **UE 源码结论**（工业级验证。除非与 ECS 架构存在根本冲突，否则优先吸收）
4. **Bevy 源码结论**（ECS 原生验证。在"ECS 下该怎么实现"的问题上权重提升）
5. **chaos 源码结论**（实战验证，但需考虑其特定项目约束）
6. **工程推演**（最低优先级，必须标明"此为推演，待验证"）

**UE 优先吸收的具体准则**：
- UE 的方案如果可以直接映射到 ECS，**默认直接采用 UE 的设计思想**
- UE 的方案如果基于 OOP 全局状态，**提取其设计意图后在 ECS 中重新表达**，而非为了简化而丢弃其工业级洞察
- 当 UE 与 Bevy 冲突时：**ECS 表达层面**的冲突采信 Bevy；**工程完备性层面**的冲突采信 UE

## 笔记产出规范

### 输出路径与文件名

- **目标目录、子文件夹结构和文件名完全以 roadmap 定义为准**
- 文件夹按 roadmap 中定义的**构建阶段或主题分类**组织，名称不带数字前缀
- 每篇笔记的文件名统一为 `<主题>.md`，与 roadmap 中的 wikilink 名称保持一致
- **禁止**在公司类名前加 `feynman-` 前缀

### 唯一必须出现的结构性元素

虽然正文结构自由（由问题链驱动），但每篇笔记**开头和结尾**必须出现：

```markdown
> **前置依赖**：[[<roadmap 中定义的前置笔记 A>]]、[[<roadmap 中定义的前置笔记 B>]]
> **本模块增量**：读完这篇笔记后，你将对 <具体能力> 有深入理解。
> 
> 本笔记探讨的核心问题是：<具体问题>。我们将从多个分支深入分析……

（正文：按深度问题链展开，没有固定标题结构）

> **下一步**：[[<roadmap 中定义的下一篇笔记>]]，因为 <理由>。
```

### 何时引入引擎对照

1. **当前子问题存在多种可行方案，需要工业经验辅助决策**
2. **源码中的设计明显与直觉相反，需要解释动机**
3. **需要说明"为什么是 ECS"时**

对照的呈现方式：
```markdown
> **参考：chaos / UE / Bevy 对这个问题是怎么做的？**
> 
> 我们在解决的是「<具体子问题>」这个问题。
> - chaos 选择了 X，原因是……
> - UE 选择了 Y，原因是……
> - Bevy 选择了 Z，原因是……
> 
> **默认推荐**：默认选 Y，因为 UE 的工业级验证表明……；该方案在 ECS 中的映射方式是……
> **条件切换**：在 X 条件下可切换为 Z，因为……
> **扩展路径**：该方案在工业级场景下的完整形态还包括……，可在核心通路验证后逐步补全
```

### 精确性与诚实性原则

涉及内存布局、数据结构操作、性能优化时，必须遵守两条红线：

**原则 1：操作必须精确到"状态变化图"**
- 任何涉及内存布局、数据结构增删、指针/索引修改的内容，必须提供**操作前后的精确状态对比**
- 必须画出来的场景：数据结构的内存布局、增删元素前后数组/内存块的状态变化、指针解引用的完整链路、扩容/迁移/压缩等会导致地址变化的操作
- **自问 checklist**：一个谨慎的读者看到这段，会不会问"中间会不会有空洞"？"指针/索引什么时候失效"？"这两个段在内存上是连续的还是分开的"？如果答案会影响理解，必须主动给出。

**原则 2：性能与 trade-off 必须诚实**
- **承认代价**：这个优化消除了什么成本？又引入了什么新成本？
- **对比基准**：和最简单的实现相比，在什么场景下更好、什么场景下更差？
- **反例警告**：在什么极端输入或极端使用方式下，这个架构会失效或表现糟糕？
- **不连续的真相**：如果涉及"连续内存"或"cache friendly"，必须明确说明"哪里连续、哪里不连续、跳转的频率是多少"
- **禁止出现的说辞**："完全消除了 cache miss"、"零开销"、"自动处理"

### 默认面向 ECS 与 AI 友好架构

1. **即使源码不是 ECS，也要思考"到 ECS 该怎么办"——但优先吸收 UE 的工业级洞察**
   - 如果源码是 OOP 架构，提炼笔记时必须在问题链的适当位置指出：哪些状态应该被拆成组件？哪些继承关系应该变成组合？系统的迭代模式在 ECS 下会如何重写？
   - UE 的方案应当被**提取其本质设计动机后，在 ECS 中重新表达**

2. **所有模块设计必须回答：AI 如何观察它？AI 如何操作它？**
   - **可观测性**：该模块的状态是否能被序列化、反射、diff？
   - **确定性**：该模块的执行是否是输入确定的？是否支持回放和快照？
   - **工具边界**：如果 AI 要通过 MCP/Tool 接口操作这个模块，接口的输入输出是否是结构化的 JSON/Schema？

3. **多 Agent 编排是架构的一等公民**
   - 在讨论系统调度、资源管理、编辑器框架时，默认要考虑**多个专项 Agent 并行工作**的场景

### AI 友好设计红线（必须遵守）

| 检查项 | 要求 |
|--------|------|
| **状态平铺** | 该模块的所有可变状态是否可以用 ECS 组件表示？如果不行，必须解释为什么并给出替代方案。 |
| **自描述** | AI 是否可以在不读 C++ 头文件的情况下，通过运行时注册表知道该模块的数据结构？ |
| **确定性** | 给定相同的输入序列，该模块的输出是否可复现？是否支持快照保存与回滚？ |
| **工具边界** | 如果 AI 通过 MCP/Tool 接口操作这个模块，接口的输入输出是否是结构化的 JSON/Schema？ |
| **Agent 安全** | 该模块是否支持沙箱、事务或白名单机制，防止 AI 的误操作破坏核心状态？ |

### 脱敏红线（必须遵守）

**✅ 可以写**：通用设计模式、算法原理、工程权衡的理由、展示关键机制的代码片段（通用命名重写）、复杂度演进路线图、AI 可观测性设计、多 Agent 编排模式、引擎架构对照结论

**❌ 禁止写**：公司源码中的具体类名/函数名/变量名、具体文件路径、游戏项目特有的业务逻辑、内部工具链的详细配置、任何可能暴露公司技术债务或未完成功能的负面评价

### 代码规范

- 代码必须是**通用化重写的关键机制展示**，不是源码片段
- 优先使用标准 C++，不依赖公司内部宏或基础设施
- 代码片段应嵌入在问题探讨的上下文中，**每个代码片段都必须说明它正在展示哪个分支方案的哪个关键机制、该机制解决了什么问题、还遗留什么问题**
- **不需要是完整可运行程序**。以"精确展示当前讨论的分歧点"为标准决定代码片段的长度（3 行到 50 行均可）
- 孤立地展示"完美代码"是禁止的

### 索引与链接规范

- roadmap/索引文件是构建手册的唯一入口，必须按 roadmap 中定义的阶段和里程碑组织
- 费曼笔记内部应大量使用 wikilink 构建知识网络，链接名称必须与 roadmap 中定义的模块名称完全一致
- 链接不仅是引用，更是**构建顺序的导航**

## 迭代产出策略

### 阶段一：梳理深度问题链（骨架）

**输入**：源码分析笔记的接口层结论（chaos + UE + Bevy）
**步骤**：
1. **读取三端源码分析笔记**（chaos / UE / Bevy）。**如果相关笔记缺失或不足，立即调用 `game-engine-analysis` skill 回探源码**
2. **读取 roadmap，确认前置依赖模块的笔记是否已经产出**。如果没有，向用户说明依赖关系
3. **搜索互联网**：搜索该模块的标准定义、经典论文、最被广泛引用的开源实现。记录搜索关键词和关键结论来源
4. **梳理深度问题链**：基于四源 + 搜索验证，列出该知识点从 naive 到工业级的完整问题链。对每个问题，梳理：场景绑定、根因分析、主要分支、分支间的核心矛盾
   - 简单知识点（如 Handle Table）：3~5 个问题，每个问题 2~3 个分支
   - 中等知识点（如日志系统）：5~8 个问题，每个问题 3~4 个分支
   - 复杂知识点（如 ECS 核心、渲染管线）：8~12 个或更多问题，每个问题 4~5 个分支
5. **确定引擎对照点**：标注问题链中哪些子问题值得引入引擎对照（存在方案分歧时）
6. **篇幅预评估与拆分建议**：
   - 如果问题链包含 **8 个以上子问题**，或预估正文超过 **5000 字**，或代码片段超过 **8 个**，则判定为"篇幅过大"
   - **必须暂停产出，向用户展示预估拆分方案**（至少 2 种拆分方案），说明拆分理由、各篇笔记的前置依赖关系和独立增量
   - **必须获得用户明确确认后再继续**
7. **产出问题链大纲**：向用户展示梳理好的深度问题链（及拆分方案），确认方向后再进入下一阶段

**产出**：
- 该模块解决的问题背景（Why）
- 完整的深度问题链列表，每个问题包含场景、根因、主要分支
- 需要引入引擎对照的子问题标注
- 篇幅预评估结论：单篇或拆分方案
- **前置依赖声明**和**本模块增量**说明

### 阶段二：逐个深入探讨问题（血肉）

**输入**：源码分析笔记的数据层 + 逻辑层结论（chaos + UE + Bevy）
**步骤**：
1. 按深度问题链顺序，**逐个深入探讨每个子问题**
2. 对每个子问题，按"场景与根因 → 分支研究 → 引擎对照（按需） → 工业实现深度 → 决策分析与推荐 → 遗留问题"的结构展开
3. **搜索互联网**：针对当前正在解决的具体子问题，搜索开源引擎的实现差异。搜索后必须回答：行业前沿的主流方案是什么？与 chaos/UE/Bevy 的差异根源是什么？

**产出**：
- 每个子问题的场景根因分析
- 每个子问题的完整分支研究（含代码片段、适用场景、隐藏代价、失效条件）
- 工业实现深度剖析（精确内存布局、工业级代码、优化手段对比）
- 按需引入的引擎对照
- **明确推荐**：在每个需要决策的子问题上，给出默认推荐路径及依据
- **遗留问题清单**：为下一个问题的自然过渡做铺垫

### 阶段三：定型与下一步

**输入**：源码分析笔记的跨模块关系 + 性能优化分析（chaos + UE + Bevy）
**步骤**：
1. 梳理跨模块耦合、平台适配、极端优化手段
2. **如果某个工业级技巧与通用最佳实践相悖、或原理存疑，回探源码**，确认其适用边界
3. **搜索互联网**：确认是否有比 chaos/UE 更优的现代方案（近 3~5 年）。**必须给出搜索关键词和结论来源**
4. **精确性审查**：检查所有涉及内存布局、数据结构操作、性能优化的地方。自问"如果一个谨慎的读者读到这里，他会问什么问题？"，并主动在文中回答
5. **工业实现深度审查**：
   - 每个子问题是否都提供了工业实现深度？
   - 数据结构的内存布局是否精确到字节/bit？状态变化是否有前后对比图？
   - 工业级优化手段是否给出了具体实现代码，而不是模糊描述？
   - 不同工业实现的对比是否准确？
   - **如果整篇笔记的工业实现深度总量不足（少于 3 个精确数据结构图 + 3 段工业级代码 + 1 组实现对比），判定为"深度不足"，必须补充**
6. **叙事审查**（遵循 `note-refine` 的费曼精神）：
   - 每个核心概念的第一次出现前，是否有"为什么需要它"的铺垫？
   - 段落之间是否是因果/问答关系，而不是并列关系？
   - 表格是否只用于总结已解释的概念？
   - 每个问题是否都经过了"场景 → 根因 → 分支 → 工业实现 → 决策"的完整层次？
   - 代码片段是否都嵌入在问题探讨的上下文中？
7. **最终一致性审查**：检查全文是否满足"必须有明确推荐"的原则

**产出**：
- 工业级设计清单
- 设计权衡表（表格只用于总结已解释的内容）
- **默认推荐路径**：针对该模块的完整设计空间，给出默认路径与条件切换路径
- **扩展路径**：标注该方案在更极端工业场景下的完整形态
- **更新 roadmap/索引文件**，标注该模块已完成并指向下一个推荐模块

## AI 响应承诺

1. **产出物明确**：一篇或多篇写入 roadmap 指定目录的费曼学习笔记
2. **完全脱敏**：笔记中不出现任何公司源码的具体类名、函数名、文件路径
3. **可复用**：所有代码和结论都是通用化的工业级设计
4. **知识独立**：笔记自成体系，不依赖源码分析笔记即可读懂
5. **索引维护**：如 roadmap/索引文件已存在，主动提议更新；如不存在，主动创建
6. **顺序连贯**：产出的每篇笔记都明确声明前置依赖、本模块增量和下一步预告
7. **遵循构建阶段**：不会在没有完成前置模块的情况下，直接跳入深层专题
8. **精确且诚实**：涉及底层实现时，主动提供状态变化图和 trade-off 分析
9. **深度问题链驱动**：笔记的主体是"问题 → 根因分析 → 分支研究 → 工业实现深度 → 代码片段 → 决策收敛"的渐进过程
10. **按需对照**：只在存在方案分歧的具体子问题上引入 chaos/UE/Bevy 对照
11. **前沿探索**：每篇笔记必须主动搜索行业前沿。**搜索不是可选步骤，而是每轮产出的必经环节**
12. **最正确方案**：在每个需要决策的子问题上，必须给出**明确推荐**，并说明依据。推荐必须体现"在 ECS 兼容前提下吸收 UE 工业级设计"
13. **Roadmap 唯一权威**：所有目录结构、文件名、阶段划分、依赖关系以 roadmap 为准
14. **叙事质量**：遵循 `note-refine` 的费曼精神，拒绝模板化叙述
15. **深度分支研究**：每个问题必须覆盖所有主要分支，分析其核心机制、适用场景、隐藏代价和失效条件
16. **精确代码片段**：嵌入问题探讨的上下文中，精确展示关键机制
17. **UE 优先吸收，拒绝简化妥协**：禁止以"个人项目"、"先跑起来"、"当前阶段简单点"为由选择降级方案
18. **工业实现深度强制输出**：每篇笔记必须包含工业级实现剖析。如果篇幅限制导致无法在主笔记中展开，**必须**拆分为子笔记，在 roadmap 中登记。**禁止以任何理由省略工业实现深度**
19. **篇幅管控**：在阶段一时主动预估篇幅；如果预估超过合理范围（8+ 子问题 / 5000+ 字 / 8+ 代码片段），**必须向用户提出拆分方案并获得确认后再继续**
