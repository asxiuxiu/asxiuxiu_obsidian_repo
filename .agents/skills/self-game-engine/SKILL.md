# 自研游戏引擎知识库（Self Game Engine）

本 skill 负责将 `game-engine-analysis` 产出的源码分析结论，进一步提炼为**可带走的通用工程知识**。产出的笔记完全脱敏，与公司源码解耦，是你未来构建自己引擎时的个人知识资产。

**核心使命**：本 skill 产出的所有笔记必须能被组织成**一个连贯的、从零开始搭建游戏引擎的渐进式计划**。每篇笔记不仅是知识讲解，更是引擎构建流程中的一个具体步骤。

> **关键原则：笔记是问题驱动的，不是模板驱动的。** 不同的知识点（日志系统 vs 渲染管线 vs ECS 核心）复杂度完全不同，不能用同一套固定结构硬套。唯一不变的是叙事逻辑：**先让人感受到问题存在 → 深入剖析问题的根源与分支 → 探讨多种解决路径 → 分析每条路径的 trade-off → 在关键决策点引入工业参照 → 逐步收敛到推荐方案。** 架构对照只在"解决某个具体子问题有多种方案可选"时才按需引入，不是每篇笔记的必选项。

## Roadmap 优先原则（最高优先级）

**本 skill 不内置任何目标目录、文件名、阶段划分或模块依赖关系。所有结构信息必须实时从 roadmap 中读取。**

- AI 在采取任何产出行动前，**必须先读取并理解 roadmap**。roadmap 是唯一的结构权威。
- 输出目录、子文件夹名称、笔记文件名、构建阶段、里程碑、模块依赖顺序，**全部由 roadmap 定义**，skill 内部不做任何预设。
- 如果 roadmap 中某模块的 wikilink 名称是 `[[Foo]]`，则对应的笔记文件名应为 `Foo.md`，存放路径应为 roadmap 中指定的对应目录。
- 如果用户没有明确给出 roadmap 路径，**主动读取 vault 中可能的 roadmap 文件**（如 `Notes/SelfGameEngine/索引.md`、`Notes/SelfGameEngine/0_RoadMap.md` 等）；若不存在，则主动询问用户 roadmap 的位置。
- 当 roadmap 与 skill 的其他建议发生冲突时，**以 roadmap 为准**。

- 用户提到"费曼笔记"、"脱敏笔记"、"通用知识"
- 用户要求"把源码分析整理成我自己能用的"

- 用户要求"给我一份搭建引擎的路线图"、"下一步该学什么"、"按照什么顺序 build 我的引擎"

## 与 game-engine-analysis 的关系

| skill | 产出位置 | 内容性质 | 是否脱敏 |
|-------|---------|---------|---------|
| `game-engine-analysis` | `Game/` / `Notes/UE/` / `Notes/Bevy/` | 引擎源码的具体结构、文件路径、行号、调用链 | ❌ 不脱敏 |
| `self-game-engine` | **以 roadmap 定义为准** | 通用设计模式、演进逻辑、架构对照与设计权衡、深度问题分析 | ✅ 完全脱敏 |

**工作流**：
1. `game-engine-analysis` 先完成源码分析（或读取已有的源码分析笔记）
2. `self-game-engine` 基于三端源码结论，重写为通用知识笔记
3. **反馈循环**：在提炼过程中，如果发现源码笔记存在盲区、设计动机不明、或无法回答"为什么这样实现"时，**主动调用 `game-engine-analysis` 回探源码**，补充对应目录下的分析笔记
4. 结合补充后的源码分析 + 互联网检索 + 个人理解，重写为更准确的通用知识
5. 通用知识笔记是**独立自洽**的，不依赖源码分析笔记即可理解

### 信息来源（四源为参考，问题为驱动）

`self-game-engine` 的分析输入来自以下四源，但**输出的叙事结构不以四源为章节，而以问题链为章节**：

1. **chaos 源码分析笔记**（`Game/` 目录下已有笔记）
2. **UE 源码分析笔记**（`Notes/UE/` 目录下已有笔记）— 必须读取，禁止跳过
3. **Bevy 源码分析笔记**（`Notes/Bevy/` 目录下已有笔记）— 必须读取，禁止跳过
4. **互联网搜索**（web search）— 每轮产出前必须执行，禁止省略
5. **个人理解与推演**

> **重要**：四源信息是**解题的参考材料**，不是笔记的目录结构。笔记的目录结构由该知识点自身的"问题链"决定。只有在解决某个具体子问题时，如果需要参考工业实践，才引入"chaos 这样做、UE 那样做、Bevy 又一种做法"的对照。

#### UE 源码分析的具体要求

- **在产出任何费曼笔记前，必须先读取 `Notes/UE/` 下与该模块相关的所有已有笔记**。如果笔记不存在或内容不足，**必须调用 `game-engine-analysis` skill 回探 UE 源码**，不能因"UE 太复杂"或"已有 chaos 结论足够"而跳过。
- UE 的架构结论（Actor-Component 树、反射系统、UObject 生命周期、渲染管线等）是"How 演进"阶段的**参考素材**。当笔记讨论到某个具体问题时（如"如何支持运行时反射查询组件类型"），UE 的方案是一个可选的参考点。
- 如果 `Notes/UE/` 下的笔记与 chaos 笔记对同一模块的描述存在矛盾，**这是高优先级信号**，必须回探 UE 源码确认。

#### Bevy 源码分析的具体要求

- **在产出任何费曼笔记前，必须先读取 `Notes/Bevy/` 下与该模块相关的所有已有笔记**。如果笔记不存在或内容不足，**必须调用 `game-engine-analysis` skill 回探 Bevy 源码**，不能因"Rust 不熟悉"或"已有 chaos/UE 结论足够"而跳过。
- Bevy 是 ECS 原生引擎，其架构结论（Archetype 存储、System 并行调度、World 全局状态、Command 延迟执行等）是"向 ECS 演进"阶段的**核心参考**。当笔记讨论"这个模块在 ECS 下该怎么实现"时，Bevy 的做法是重要参照。
- 如果 `Notes/Bevy/` 下的笔记与 chaos/UE 笔记对同一模块的描述存在矛盾，**这是高优先级信号**，必须回探 Bevy 源码确认。

#### Web 搜索的具体要求

在**每一轮迭代产出前**（问题链梳理 / 逐个解决 / 定型），都必须执行针对性搜索：

| 搜索时机 | 搜索目标 |
|---------|---------|
| 问题链梳理阶段 | 该模块在行业内的标准定义、经典论文、最被广泛引用的开源实现 |
| 逐个解决阶段 | 当前正在解决的具体子问题，开源引擎（Bevy、Flecs、EnTT、Godot）是如何处理的 |
| 定型阶段 | 最新学术/工业进展（近 3~5 年），确认是否有超越 chaos/UE 的现代方案 |

**搜索不是可选装饰，而是决策依据**。以下情况**必须搜索**：
- 对某个设计模式的普适性存疑时
- 要判断"哪种方案更适合解决当前这个子问题"时
- 涉及性能 claim（如"cache friendly"、"O(1)"）时，必须搜索 benchmark 或论文验证
- 涉及 C++ 标准库特性或现代 C++ 惯用法时

**搜索后的处理**：将搜索结果与源码结论进行**三角验证**。如果搜索结果与某一端源码结论冲突，优先采信**可复现的 benchmark / 学术论文 / 广泛使用的开源实现**。

### 反馈循环：发现缺口时回探源码

提炼通用知识的过程不是单向的。当你遇到以下情况时，**必须暂停当前提炼，调用对应 skill 深入源码**。

可调用 skill：
- **chaos 源码缺口** → 调用 `game-engine-analysis`（指定 chaos 引擎）
- **UE 源码缺口** → 调用 `game-engine-analysis`（指定 UE 引擎）
- **Bevy 源码缺口** → 调用 `game-engine-analysis`（指定 Bevy 引擎）

| 信号 | 行动 |
|------|------|
| 源码分析笔记对某个关键设计"只描述了是什么，没解释为什么" | 回探源码，追踪该设计的调用上下文和演进历史 |
| 发现两个源码模块的描述相互矛盾，或存在逻辑断层 | 回探源码，定位中间的衔接层或数据流转路径 |
| 不清楚核心数据结构的实际约束，导致无法精确分析分支方案 | 回探源码，提取关键结构体的字段和生命周期 |
| 互联网上的通用方案与公司实现差异巨大，且无法判断优劣 | 回探源码，理解公司实现针对的具体瓶颈或平台约束 |
| UE 与 chaos 对同一问题的实现差异很大，需要判断各自的适用边界 | 同时/分别回探两个源码库，提取设计动机和约束条件 |

**行动规范**：
1. 明确向用户说明发现了什么新问题，以及为什么需要回探源码
2. 调用对应 skill 完成针对性探索
3. 将新结论写入对应目录下的笔记（新建或追加）
4. 返回 `self-game-engine`，继续提炼通用知识
5. 最终产出的费曼笔记应体现这一轮的认知升级

## 核心方法论：深度问题链驱动 + 四源参考

本 skill 专注**深度问题链驱动**的知识组织结构。笔记叙述质量（零假定、Why 先行、比喻有效、因果链条、衔接自然等）需符合 `note-refine` skill 的通用规范，`note-refine` 提供完整的费曼叙述自检清单。本 skill 不重复定义叙事规范。

本 skill 的分析基础是**深度问题链驱动**：

> **一篇好的引擎知识笔记，不是一份设计文档，而是一次深度思维还原——还原工程师在面对空白编辑器时，是如何从"这系统要解决什么问题"出发，逐层剥开问题的表象，发现隐藏在表象之下的分支与矛盾，最终在多条可能的路径中做出取舍，走到"最终代码长什么样"的。**

输出的叙事结构必须是**深度问题链**（Deep Problem Chain），而不是**特征对比表**、**阶段清单**或**浅层 FAQ**。

### 深度问题链的叙事结构

以日志系统为例，问题链应该是：

```
问题 0：为什么引擎不能直接用 std::cout？
  → 场景绑定：16ms 帧时间内输出 1000 条日志会发生什么？
  → 根因分析：std::cout 的线程安全锁、flush 策略、内存分配在哪里成为瓶颈？
  →  naive 方案：全局宏 LOG(x) { std::cout << x << std::endl; }
  → 深度推演：不仅指出"性能差"，更要量化——每次 operator<< 可能触发几次内存分配？
    std::endl 的 flush 在 Windows/Unix 下的系统调用开销分别是多少？
  → 分支研究：如果换用 std::ostringstream 预分配缓冲区，能不能缓解？能缓解多少？
    如果换用 fprintf 直接写文件，绕过 C++ stream 的 locale 开销呢？
  → 结论：std::cout 的问题不是"慢"，而是"不可控"——它的线程安全、缓冲、格式化
    全部隐藏在标准库实现中，你无法针对引擎场景调优。

问题 1：如何区分日志级别？（DEBUG / INFO / WARN / ERROR）
  → 表面需求：过滤噪音
  → 深层问题：日志级别是"编译期决策"还是"运行时决策"？两种选择的代价是什么？
  → 分支研究 A：编译期宏过滤（#if LOG_LEVEL >= DEBUG）
    - 优点：零运行时开销，完全消除 dead code
    - 代价：需要重新编译才能调整级别；无法为不同模块设置不同级别
    - 适用场景：发布版、性能敏感路径
  → 分支研究 B：运行时级别检查
    - 优点：动态调整，支持 per-module / per-category 过滤
    - 代价：每次日志调用都有分支预测失败风险
    - 优化分支：用原子变量存储当前级别 + LIKELY 宏提示分支预测器
  → 分支研究 C：混合策略——编译期硬开关 + 运行时软开关
    - 如何用两层过滤实现"发布版完全消除 DEBUG 代码，同时保留 ERROR/WARN 的动态调整"？
  → 引擎对照：UE 的编译期过滤策略是什么？chaos 的运行时过滤粒度是怎样的？
    为什么它们做了不同选择？（UE 的编辑器需要热调级别，chaos 的服务器端更关注编译期零开销）
  → 推荐结论与个人项目取舍

问题 2：如何支持多种输出目标？（控制台、文件、网络、调试器）
  → 分支研究：Sink 抽象 vs 回调注册表 vs 直接硬编码 switch-case
  → 关键决策点：每个 Sink 的格式不同怎么办？每条日志都格式化多次的成本是多少？
  → 深层问题：格式化应该发生在 Sink 之前还是 Sink 内部？延迟格式化的边界条件是什么？
  → ……（按需继续深入，每个问题都按此深度展开）
```

**在每一个子问题的讨论中**，必须包含以下要素：
1. **场景绑定与根因分析**：不只要说"有什么问题"，更要解释"为什么在这个场景下会出问题"、"问题的根因是什么"。
2. **多分支研究**：对于每个子问题，主动识别所有可行的解决方向（即使某些方向最终会被否定），逐一分析其原理、适用场景、隐藏代价。
3. **精确代码片段**：在探讨每个分支时，给出足以展示关键机制的代码片段——不需要是完整可运行程序，但核心逻辑必须精确、无歧义。
4. **按需引入引擎对照**：只有当当前子问题存在**多种可行方案、且需要工业级经验辅助决策**时，才引入 chaos/UE/Bevy/前沿的参考。

### 深度问题链的写作要求

1. **每个问题的提出必须经过三层递进**
   - **第一层（现象）**：让读者看到具体的故障场景或性能数据。
     - "如果你的游戏在 16ms 内要输出 1000 条调试日志，synchronous 的 `fwrite` 会直接吃掉 5ms，帧率从 60fps 掉到 45fps。"
   - **第二层（根因）**：剥开现象，指出问题的本质原因。
     - "这不是因为 `fwrite` 本身慢，而是因为用户态到内核态的上下文切换、文件系统缓冲区的锁竞争、以及 std::ostream 每次 << 都可能触发的内存重新分配，三重开销叠加。"
   - **第三层（矛盾）**：揭示这个根因背后的深层矛盾——通常是一个"想要 A 就必须牺牲 B"的 trade-off。
     - "你想要日志立即可见（同步 flush），就必须承受系统调用开销；你想要零开销，就必须接受崩溃时丢失缓冲区内日志。这就是异步日志设计的核心矛盾。"

2. **每个问题的探讨必须覆盖所有主要分支**
   - 不要只给出"正确方案"和"错误方案"两种对比。要主动思考：
     - "业界对这个问题的常见做法有几种？"
     - "每种做法的隐藏假设是什么？在什么条件下会失效？"
     - "有没有被主流忽略但理论上可行的做法？"
   - **分支研究的呈现格式**：
     ```markdown
     ### 分支 A：XXX 方案
     **核心思路**：……
     **代码片段**：……（精确展示关键机制，不需要完整程序）
     **适用场景**：……
     **隐藏代价**：……（必须诚实暴露）
     **失效条件**：……（什么极端输入或使用方式会让它表现糟糕？）
     
     ### 分支 B：YYY 方案
     ……
     
     ### 决策分析
     在 <具体条件> 下，分支 A 优于分支 B，因为……
     在 <另一条件> 下，分支 B 更优，因为……
     **个人项目默认推荐**：……（必须有且仅有一条默认路径）
     ```

3. **代码片段必须服务于问题的探讨**
   - ❌ 禁止孤立地展示一段"完美代码"然后说"这就是最终方案"。
   - ✅ 代码片段必须嵌入在问题探讨的上下文中：
     - "为了验证分支 A 的假设，我们先看它的核心数据结构："
     - "这段代码展示了分支 B 的致命缺陷——注意第 X 行的竞争条件："
     - "如果我们尝试将分支 A 和分支 B 结合，代码会变成这样："
   - 代码片段的长度由问题复杂度决定。可以是 3 行的关键表达式，也可以是 50 行的核心算法，**以"是否足以理解当前正在讨论的分歧点"为标准**。

4. **改进必须是因果推导，不是分类罗列**
   - ❌ 错误："日志系统有三种架构：同步、异步、混合。"
   - ✅ 正确："我们先写了同步日志，发现主线程被阻塞了（问题 A）。于是引入线程本地缓冲区（方案 1），但发现崩溃时缓冲区里的日志会丢失（问题 B）。于是引入信号量同步 + 紧急 flush（方案 2）……"

5. **引擎对照只在决策点引入**
   - 不是每篇笔记都必须有"架构对照"章节。
   - 只有当某个子问题存在**多种可行方案、且需要工业级经验辅助决策**时，才引入 chaos/UE/Bevy/前沿的参考。
   - 引入时要说清楚：**我们正在解决哪个具体问题**？每个引擎在这个**具体子问题**上是怎么选的？为什么？

### 判定"最正确方案"的优先级规则

当需要对一个具体子问题做方案决策时，按以下优先级采信：

1. **可复现的 benchmark / 性能测试数据**（最高优先级）
2. **被广泛引用的学术论文或工业界标准实现**（如 EnTT 的 sparse set、Bevy 的 archetype）
3. **Bevy 源码结论**（ECS 原生验证，但需考虑 Rust 与 C++ 生态差异）
4. **UE 源码结论**（工业级验证，但需考虑其规模约束是否适用于个人项目）
5. **chaos 源码结论**（实战验证，但需考虑其特定项目约束）
6. **个人推演**（最低优先级，必须标明"此为推演，待验证"）

**禁止在笔记中呈现两种矛盾方案却不给出明确推荐**。如果确实无法判定（如两种方案在不同尺度下各有最优），必须明确标注各自的**适用尺度**和**触发条件**，但个人开发者推荐的默认路径必须有且仅有一个。

## 叙事原则（强制遵循 note-refine 的费曼精神）

### 原则 1：擅长提出问题，拒绝概念空降

在写下任何概念之前，必须先问自己三个问题：
- **这个概念解决什么问题？** 如果回答不上来，就不该在这里出现。
- **读者此时知道什么？** 每个新概念必须稳稳地站在读者已知的台阶上，不能空降。
- **这个术语引入得是否过于突然？** 如果需要一个从未提及过的背景知识才能理解，就必须先补充那个背景知识，或换一种更朴素的表达方式。

> 费曼学习法的核心不是"解释清楚"，而是"在解释之前先质疑自己是否真正理解"。每写一个段落，都反思一次：如果我是一个完全不了解引擎的初学者，看到这里会不会愣住？

### 原则 2：Why 先行（问题驱动）

不要直接抛出定义。先让读者（也就是未来的你自己）感受到"这个问题确实需要解决"。

```markdown
❌ 互斥锁（Mutex）是操作系统提供的睡眠锁，竞争时线程挂起……
（读者：哦，又一个术语。跟我有什么关系？）

✅ 想象两个工人要同时修改同一份工资表。如果不加限制，
A 读到"张三工资 5000"，B 也读到"5000"；A 改成 6000，B 改成 5500；
最后存进去的是 5500——A 的修改被覆盖了。这就是数据竞争。
互斥锁就是为了解决这个问题而生的……
（读者：原来如此，没有锁真的会出乱子。）
```

### 原则 3：因果推导（演进逻辑）

讲清楚 **"解决了 A 的问题，又带来了 B 的问题，于是人们发明了 C"** 的演进链条。避免"第一种…第二种…第三种…"地分类罗列。

```markdown
❌ 日志系统有三种输出模式：同步、异步、混合。
（这是百科词条，不是学习笔记。）

✅ 最早的日志就是直接 `printf`（同步），但游戏主线程 16ms 一帧，
如果日志量一大，I/O 阻塞直接把帧率拉崩。
于是人们把日志先写到内存缓冲区，让后台线程慢慢刷盘（异步），
主线程不阻塞了，但崩溃时缓冲区里的日志全丢了。
再于是人们在关键路径上保留同步 flush 的逃生通道（混合模式），
正常日志走异步保证性能，崩溃前强制同步保证不丢……
（读者理解了每种设计是为了解决上一个设计的什么问题。）
```

### 原则 4：场景绑定（知识落地）

把知识绑定到具体的使用场景或故障场景。问自己："如果去掉这个机制，系统会在什么场景下出问题？"

### 原则 5：问题链衔接（自然过渡）

段落结尾自然引出下一个问题。读者应该感到"知道了这个，我自然会想知道那个"。

```markdown
❌ （第一节讲完日志级别是什么）
（第二节突然开始）日志格式化有三种方式：printf-style、stream-style、结构化。
（读者：等等，为什么要讲格式化？跟级别有什么关系？）

✅ （第一节结尾）
"日志级别帮我们过滤掉了不需要的噪音，但每条日志的内容本身——
时间、文件名、行号、线程 ID——都需要按固定格式拼接成字符串。
如果每次都手动 `sprintf`，代码又臭又长，还容易内存越界。
所以下一个问题是：日志该怎么格式化？"
（第二节自然承接）
```

### 原则 6：表格只用于总结

对比表格、特性表格**只能用于总结已经解释过的概念**，绝对不能用于首次引入概念。表格是"复习工具"，不是"教学工具"。

## 笔记结构：没有固定模板，只有深度问题链

**本 skill 不对笔记的标题结构做任何强制要求**。不同的知识点有自己的内在逻辑：

- **日志系统**的问题链可能是：为什么需要日志 → 级别与过滤（编译期 vs 运行时的深层矛盾） → 输出目标（Sink 抽象的多种形态） → 格式化（延迟格式化的边界条件） → 多线程（无锁 vs 有锁的量化对比） → 异步（线程模型的分支：单消费者 vs 多消费者 vs disruptor） → 崩溃安全（信号安全写、内存映射、core dump 集成）
- **ECS 核心**的问题链可能是：为什么 OOP 组合不行（具体到 cache miss 的量化分析） → 最 naive 的组件数组（连续存储的假象与指针失效问题） → 查询性能问题（线性扫描 vs 索引的 trade-off） → Archetype 存储（Archetype 的内存碎片与跨表查询代价） → System 并行（依赖分析的多种算法：DAG vs 区间 vs 显式声明） → Command 缓冲区（命令模式的延迟执行与顺序保证）
- **渲染管线**的问题链可能是：为什么不能用立即模式（CPU/GPU 同步的代价量化） → 渲染队列（排序键的设计空间：32 位 vs 64 位、哈希 vs 位域） → 排序与合批（合批的隐藏代价：状态切换 vs draw call 数量） → Pass 图（依赖分析的精确性 vs 开销） → 资源屏障（手动 barrier vs 自动跟踪的适用边界） → 多线程渲染（命令列表录制与提交的多种线程模型）

**AI 在产出每篇笔记前，必须先梳理该知识点的深度问题链**，然后按问题链的自然顺序组织内容。

### 唯一必须出现的结构性元素

虽然正文结构自由，但每篇笔记**开头和结尾**必须出现以下元素：

```markdown
> **前置依赖**：[[<roadmap 中定义的前置笔记 A>]]、[[<roadmap 中定义的前置笔记 B>]]
> **本模块增量**：读完这篇笔记后，你将对 <具体能力> 有深入理解。
> 
> 本笔记探讨的核心问题是：<具体问题>。我们将从多个分支深入分析……

（正文：按深度问题链展开，没有固定标题结构）

> **下一步**：[[<roadmap 中定义的下一篇笔记>]]，因为 <理由>。
```

### 何时引入引擎对照

不是每篇笔记都需要引擎对照。只在以下情况引入：

1. **当前子问题存在多种可行方案，需要工业经验辅助决策**
   - 例如："对于异步日志的线程安全队列，我们有无锁队列、双缓冲、disruptor 模式等多种选择。我们来看看工业界怎么选……"

2. **源码中的设计明显与直觉相反，需要解释动机**
   - 例如："UE 的日志系统没有选择线程本地缓冲，而是选择了全局队列加锁。这听起来很反直觉——为什么？因为 UE 的编辑器环境下，日志的消费者是多个可视化面板，而不是单一文件……"

3. **需要说明"为什么是 ECS"时**
   - 例如："chaos 的日志系统用单例模式管理全局 Logger。如果我们改写成 ECS 架构，这个'全局状态'应该放在哪里？Bevy 的做法是……"

对照的呈现方式：
```markdown
> **参考：chaos / UE / Bevy 对这个问题是怎么做的？**
> 
> 我们在解决的是「异步日志的线程安全队列」这个问题。
> - chaos 选择了 X，原因是……（不引用具体类名）
> - UE 选择了 Y，原因是……（不引用具体类名）
> - Bevy（作为 Rust/ECS 参考）选择了 Z，原因是……
> 
> **个人项目推荐**：默认选 Y，因为……；在 X 条件下可切换为 Z。
```

## 精确性与诚实性原则

费曼笔记的目标不是"听起来对"，而是"经得起追问"。所有涉及内存布局、数据结构操作、性能优化时，模糊的描述会让读者产生大量困惑。必须遵守以下两条红线。

### 原则 1：操作必须精确到"状态变化图"

任何涉及内存布局、数据结构增删、指针/索引修改的内容，不能只用文字概括，必须提供**操作前后的精确状态对比**。形式可以是 ASCII 图、表格、或伪代码，但核心要求是：一个读者看了之后，能准确画出当前内存里每个字节/每个数组元素是什么。

**必须画出来的场景包括**：
- 数据结构的内存布局（尤其是 `char*`、SoA、Pool、Ring Buffer 等 raw memory 结构）
- 增删元素前后数组/内存块的状态变化
- 指针解引用、索引映射、handle 转换的完整链路
- 扩容/迁移/压缩等会导致地址变化的操作

**自问 checklist**：
- 一个谨慎的读者看到这段，会不会问"那中间会不会有空洞"？
- 他会不会问"指针/索引什么时候失效"？
- 他会不会问"这两个段在内存上是连续的还是分开的"？
- 如果以上任何一个问题的答案会影响理解，就必须在笔记中主动给出，不能等读者追问。

### 原则 2：性能与 trade-off 必须诚实

不能为了推崇某个架构而隐瞒其代价。当讨论性能优化、架构选择时，必须**主动暴露边界条件和反例**。

**必须包含的要素**：
- **承认代价**：这个优化消除了什么成本？又引入了什么新成本？
- **对比基准**：和最简单的实现（如 `std::vector` 或 `std::unordered_map`）相比，在什么场景下更好、什么场景下更差？
- **反例警告**：在什么极端输入或极端使用方式下，这个架构会失效或表现糟糕？
- **不连续的真相**：如果涉及"连续内存"或"cache friendly"的宣传，必须明确说明"哪里连续、哪里不连续、跳转的频率是多少"。

**禁止出现的说辞**：
- "完全消除了 cache miss" → 应该说"将每实体的随机跳转降低为每 Chunk 一次"
- "零开销" → 应该说"移除了虚函数表开销，但增加了 indirection 开销"
- "自动处理" → 应该说"通过 swap-and-pop 保持密集，代价是被移动元素需要更新索引"

## 默认面向 ECS 与 AI 友好架构

用户的自研引擎目标明确：**基于 ECS 架构**，并且**为 AI 协作而设计**。因此，所有产出的通用知识必须满足以下默认约束：

1. **即使源码不是 ECS，也要思考"到 ECS 该怎么办"**
   - 如果源码是 OOP 架构（如 UE 的 Actor-Component 继承树），提炼笔记时必须在问题链的适当位置指出：哪些状态应该被拆成组件？哪些继承关系应该变成组合？系统的迭代模式在 ECS 下会如何重写？
   - 不要把源码的 OOP 实现直接照搬到通用知识中作为"最佳实践"，而是把它当作**需要被重构到 ECS 的素材**。

2. **所有模块设计必须回答：AI 如何观察它？AI 如何操作它？**
   - **可观测性**：该模块的状态是否能被序列化、反射、diff？AI 能不能不读头文件就理解数据结构？
   - **确定性**：该模块的执行是否是输入确定的？是否支持回放（Replay）和快照（Snapshot）？
   - **工具边界**：如果 AI 要通过 MCP/Tool 接口操作这个模块，接口的输入输出是否是结构化的 JSON/Schema？

3. **多 Agent 编排是架构的一等公民**
   - 在讨论系统调度、资源管理、编辑器框架时，默认要考虑**多个专项 Agent 并行工作**的场景。
   - 例如：物理调优 Agent 和 Gameplay Agent 能否同时工作？它们之间的状态隔离和冲突合并机制是什么？

## 渐进式构建路线图：从零开始搭引擎

本 skill 的产出必须被组织为**一个可执行的引擎构建计划**。用户应该能够按照 roadmap 中推荐的笔记顺序，一步一步地搭建出自己的引擎。

### 构建阶段与里程碑

**具体的阶段数量、里程碑名称和验收标准全部由 roadmap 定义**，skill 不做任何预设。roadmap 中应包含：

- 每个阶段的名称和编号（如果有）
- 每个阶段结束时的**可运行里程碑**
- 每个里程碑的**核心验收标准**

AI 在产出笔记时，必须在笔记中明确标注它属于 roadmap 中的哪个阶段/里程碑。

### 模块依赖与产出顺序

**模块之间的依赖关系和产出顺序全部由 roadmap 定义**。roadmap 中应以依赖图、流程图或分层列表的形式说明模块先后关系。

AI 在产出任何笔记前，必须：
1. 检查 roadmap 中该模块的前置依赖是否已经产出
2. 如果上游笔记尚未产出，向用户说明依赖关系，并建议优先完成上游模块
3. 如果用户坚持先写当前模块，应在笔记中简要回顾或引用前置依赖的核心结论

### 连贯性强制要求

每篇产出的费曼笔记，除了按深度问题链展开之外，**必须**包含以下三个连贯性元素（通常放在笔记开头和结尾的醒目位置）：

1. **前置依赖声明**：在笔记开头明确列出"在阅读这篇笔记之前，你应该已经阅读了哪些内容"，并给出 roadmap 中定义的 wikilink。
2. **本模块对引擎的增量**：清晰说明"读完这篇笔记后，你对引擎的哪个方面会有深入理解"。
3. **下一步预告**：在笔记结尾预告"下一篇应该深入什么"，以及为什么现在时机成熟了。下一篇的名称应从 roadmap 中的依赖顺序确定。

**示例模板**（应出现在每篇笔记中）：
```markdown
> **前置依赖**：[[<roadmap 中定义的前置笔记 A>]]、[[<roadmap 中定义的前置笔记 B>]]
> **本模块增量**：深入理解 <某模块> 的设计空间与分支选择。
> **下一步**：[[<roadmap 中定义的前篇笔记>]]，因为 <理由>。
```

### 索引即构建手册

roadmap 对应的索引文件不应只是目录列表，而是一本**构建手册**。除了列出所有笔记外，通常应包含：
- 按构建阶段分组，标注每个模块所属的里程碑
- 明确显示推荐的阅读顺序（可用流程图或有序列表）
- 在每个里程碑处说明"完成到这里的最低验收标准"
- 对于尚未产出的模块，预留位置并标注状态（待产出 / 进行中 / 已完成）

AI 在产出每篇笔记后，应主动提议更新 roadmap/索引文件，标注该模块的状态为"已完成"，并指向下一个推荐模块。

### 文件名格式

```
<主题>.md
```

**禁止**在公司类名前加 `feynman-` 前缀。这些笔记是**你自己的知识**，不是源码的附庸。

### ECS 映射要求

当源码分析笔记基于 OOP 或非 ECS 架构时，`self-game-engine` 的产出**不得**简单复述源码结构。必须在笔记中显式包含以下内容之一：

- **ECS 重构映射**：在问题链的适当位置（通常是"如果要改成 ECS 该怎么做"这个子问题下），列出源码中的类/继承链，对应到 ECS 中的组件、系统、实体组合。
- **Why ECS 更适合该模块**：从 cache locality、组合灵活性、AI 可观测性角度解释。
- **How 演进中的 ECS 化步骤**：在问题链的演进过程中，明确说明 ECS 存储模型如何升级（如从 `ComponentArray` → `Archetype` → `Chunk-based`）。

### AI 友好设计红线（必须遵守）

每篇产出的费曼笔记，在讨论任何模块时，**必须**检查以下清单：

| 检查项 | 要求 |
|--------|------|
| **状态平铺** | 该模块的所有可变状态是否可以用 ECS 组件表示？如果不行，必须解释为什么并给出替代方案。 |
| **自描述** | AI 是否可以在不读 C++ 头文件的情况下，通过运行时注册表知道该模块的数据结构？ |
| **确定性** | 给定相同的输入序列，该模块的输出是否可复现？是否支持快照保存与回滚？ |
| **工具边界** | 如果 AI 通过 `AgentBridge` / MCP 接口操作该模块，接口的输入输出是否是结构化的 JSON/Schema？ |
| **Agent 安全** | 该模块是否支持沙箱、事务或白名单机制，防止 AI 的误操作破坏核心状态？ |

### 脱敏红线（必须遵守）

**✅ 可以写**：
- 通用设计模式（Handle-Table、Command Pattern、Double Buffering）
- 算法原理（拓扑排序、A* 寻路、BVH 构建）
- 工程权衡的理由（"为了减少 cache miss，采用了 SoA 布局"）
- 展示关键机制的代码片段（用通用命名重写，嵌入问题探讨上下文中）
- 复杂度演进路线图（"从直接 new → 对象池 → 分代回收"）
- AI 可观测性设计（Change Log、Snapshot、Replay、AgentBridge）
- 多 Agent 编排模式（Orchestrator、专项 Agent、组件白名单）
- 引擎架构对照结论（"UE 采用 X，chaos 采用 Y，Bevy 采用 Z，个人项目推荐 W"）

**❌ 禁止写**：
- 公司源码中的具体类名、函数名、变量名
- 具体文件路径（如 `chaos/core/ECS/private/InternalArchetype.cpp`）
- 游戏项目特有的业务逻辑（如 `ProvenGround` 中的特定技能 ID、关卡名）
- 内部工具链的详细配置（CI 脚本中的服务器 IP、密钥名）
- 任何可能暴露公司技术债务或未完成功能的负面评价

### 代码规范

- 费曼笔记中的代码必须是**通用化重写的关键机制展示**，不是源码片段
- 优先使用标准 C++，不依赖公司内部宏或基础设施
- 代码片段应嵌入在问题探讨的上下文中，**每个代码片段都必须说明它正在展示哪个分支方案的哪个关键机制、该机制解决了什么问题、还遗留什么问题**
- **不需要是完整可运行程序**。以"精确展示当前讨论的分歧点"为标准决定代码片段的长度（3 行到 50 行均可）
- 孤立地展示"完美代码"是禁止的

## 索引与链接规范

roadmap/索引文件是构建手册的唯一入口，其位置和名称由 roadmap 自身定义或用户指定。结构可以参考 `Game/00-引擎与游戏全解析主索引.md` 和 `Notes/UE/00-UE全解析主索引.md`，但只包含通用主题，并且**必须按 roadmap 中定义的阶段和里程碑组织**。

费曼笔记内部应大量使用 wikilink 构建知识网络。链接名称必须与 roadmap 中定义的模块名称完全一致。

链接不仅是引用，更是**构建顺序的导航**。在讨论下游概念时，应使用诸如"正如我们在 [[<roadmap 中定义的前置笔记>]] 中分析的……"这样的句式，帮助读者理解当前知识点在整体引擎中的位置。

## 迭代产出策略（深度问题链驱动，非模板驱动）

### 阶段一：梳理深度问题链（骨架）

**输入**：源码分析笔记的接口层结论（chaos + UE + Bevy）
**步骤**：
1. **读取 chaos 源码分析笔记**（`Game/` 目录）
2. **读取 UE 源码分析笔记**（`Notes/UE/` 目录）。**如果相关笔记缺失或不足，立即调用 `game-engine-analysis` skill 回探 UE 源码，补充到 `Notes/UE/`。**
3. **读取 Bevy 源码分析笔记**（`Notes/Bevy/` 目录）。**如果相关笔记缺失或不足，立即调用 `game-engine-analysis` skill 回探 Bevy 源码，补充到 `Notes/Bevy/`。**
4. **读取 roadmap，确认前置依赖模块的笔记是否已经产出**。如果没有，向用户说明当前模块依赖哪些上游知识，并建议优先完成上游模块或简要回顾其结论。
5. **搜索互联网**：搜索该模块的标准定义、经典论文、最被广泛引用的开源实现（Flecs/EnTT/Godot）。记录搜索关键词和关键结论来源。
6. **梳理深度问题链**：基于四源 + 搜索验证，列出该知识点从 naive 到工业级的完整问题链。对每个问题，必须梳理：
   - **场景绑定**：该问题在什么具体场景下会爆发？
   - **根因分析**：问题的本质原因是什么？
   - **主要分支**：存在哪几种解决方向？
   - **分支间的核心矛盾**：选择 A 就必须牺牲 B 的 trade-off 是什么？
   问题链的长度由知识点复杂度决定：
   - 简单知识点（如 Handle Table）：3~5 个问题，每个问题 2~3 个分支
   - 中等知识点（如日志系统）：5~8 个问题，每个问题 3~4 个分支
   - 复杂知识点（如 ECS 核心、渲染管线）：8~12 个或更多问题，每个问题 4~5 个分支
7. **确定引擎对照点**：标注问题链中哪些子问题值得引入引擎对照（存在方案分歧时），哪些不需要。
8. **产出问题链大纲**：向用户展示梳理好的深度问题链，确认方向后再进入下一阶段。

**产出**：
- 该模块解决的问题背景（Why）
- 完整的深度问题链列表，每个问题包含场景、根因、主要分支
- 需要引入引擎对照的子问题标注
- **前置依赖声明**和**本模块增量**说明

### 阶段二：逐个深入探讨问题（血肉）

**输入**：源码分析笔记的数据层 + 逻辑层结论（chaos + UE + Bevy）
**步骤**：
1. 按深度问题链顺序，**逐个深入探讨每个子问题**。
2. 对每个子问题，必须按以下结构展开：
   - **场景与根因（详述）**：
     - 用具体数据或场景描述问题的表现
     - 剥开表象，分析问题的本质原因（通常涉及系统瓶颈、约束条件、或设计矛盾）
     - 揭示"想要 A 就必须牺牲 B"的深层 trade-off
   - **分支研究（详述）**：
     - 识别该问题的所有主要解决分支（即使某些分支最终会被否定）
     - 对每个分支：
       - 说明核心思路与假设
       - 给出**精确的代码片段**展示关键机制（不需要完整程序）
       - 分析适用场景与隐藏代价
       - 指出失效条件（什么情况下这个分支会表现糟糕）
     - 如果两个分支可以结合，探讨"混合方案"的可能性与复杂度
   - **引擎对照（按需）**：
     - 只有当当前子问题存在多种可行方案时，才引入 chaos/UE/Bevy 的参考
     - 对照格式："对于「当前子问题」，chaos 选 X 因为……，UE 选 Y 因为……，Bevy 选 Z 因为……。个人项目推荐……"
   - **决策分析与推荐**：
     - 如果四源结论冲突，按"判定最正确方案的优先级规则"给出明确采信
     - **必须有且仅有一条默认推荐路径**，并说明依据
     - 对于非默认路径，明确标注适用条件
   - **遗留问题**：
     - 不要假装这个方案是完美的。明确标注"这个推荐方案还遗留哪些问题"，这些问题将成为下一个问题的引子
3. **搜索互联网**：针对当前正在解决的具体子问题，搜索开源引擎的实现差异。搜索后必须回答：行业前沿的主流方案是什么？与 chaos/UE/Bevy 的差异根源是什么？

**产出**：
- 每个子问题的场景根因分析
- 每个子问题的完整分支研究（含代码片段、适用场景、隐藏代价、失效条件）
- 按需引入的引擎对照（不是每个子问题都需要）
- **明确推荐**：在每个需要决策的子问题上，给出"推荐采用哪种方案"的结论及依据
- **遗留问题清单**：为下一个问题的自然过渡做铺垫

### 阶段三：定型与下一步

**输入**：源码分析笔记的跨模块关系 + 性能优化分析（chaos + UE + Bevy）
**步骤**：
1. 梳理跨模块耦合、平台适配、极端优化手段
2. **如果某个工业级技巧与通用最佳实践相悖、或原理存疑，回探源码**（调用对应 skill），确认其适用边界，补充到源码笔记
3. **搜索互联网**：确认是否有比 chaos/UE 更优的现代方案（近 3~5 年的论文或开源引擎实现）。**必须给出搜索关键词和结论来源。**
4. 输出经过验证的、可迁移的设计清单
5. 在笔记结尾添加**下一步预告**（根据 roadmap 中的依赖顺序）
6. **精确性审查**：通读全文，检查所有涉及内存布局、数据结构操作、性能优化的地方。自问"如果一个谨慎的读者读到这里，他会问什么问题？"，并主动在文中回答这些问题
7. **叙事审查**：通读全文，检查是否符合 note-refine 的叙事原则（Why 先行、因果推导、问题链衔接、场景绑定）。自检清单：
   - 每个核心概念的第一次出现前，是否有"为什么需要它"的铺垫？
   - 段落之间是否是因果/问答关系，而不是并列关系？
   - 表格是否只用于总结已解释的概念？
   - 如果把比喻删掉，剩下的文字是否还能自洽？
   - 每个问题是否都经过了"场景 → 根因 → 分支 → 决策"的完整层次？
   - 代码片段是否都嵌入在问题探讨的上下文中，而不是孤立展示？
8. **最终一致性审查**：检查全文是否满足"必须有明确推荐"的原则。如果某处只说了"A 也可以，B 也可以"，必须补充为"默认推荐 A，因为 X；在 Y 条件下可切换为 B"。

**产出**：
- 工业级设计清单
- 设计权衡表（表格只用于总结已解释的内容）
- **个人开发推荐路径**：针对该模块的完整设计空间，给出默认路径与条件切换路径
- **更新 roadmap/索引文件**，标注该模块已完成并指向下一个推荐模块

## AI 响应承诺

1. **产出物明确**：一篇写入 roadmap 指定目录的费曼学习笔记
2. **完全脱敏**：笔记中不出现任何公司源码的具体类名、函数名、文件路径
3. **可带走**：所有代码和结论都是通用化的，可以直接用于个人项目
4. **知识独立**：笔记自成体系，不依赖源码分析笔记即可读懂
5. **索引维护**：如 roadmap/索引文件已存在，主动提议更新；如不存在，主动创建
6. **顺序连贯**：产出的每篇笔记都明确声明前置依赖、本模块增量和下一步预告，确保用户能按照路线图一步一步深入
7. **遵循构建阶段**：不会在没有完成前置模块的情况下，直接跳入深层专题
8. **精确且诚实**：涉及底层实现时，主动提供状态变化图和 trade-off 分析，不夸大优点，不回避代价
9. **深度问题链驱动**：笔记的主体是"问题 → 根因分析 → 分支研究 → 代码片段 → 决策收敛"的渐进过程，不是引擎特性对比表
10. **按需对照**：只在存在方案分歧的具体子问题上引入 chaos/UE/Bevy 对照，不是每篇笔记的必选项
11. **前沿探索**：每篇笔记必须主动搜索行业前沿（开源 ECS 引擎/论文），探索是否有更优方案。**搜索不是可选步骤，而是每轮产出的必经环节。**
12. **最正确方案**：在每个需要决策的子问题上，必须在可选方案之间给出**明确推荐**，并说明依据。**禁止只罗列优缺点却不给出结论。**
13. **Roadmap 唯一权威**：所有目录结构、文件名、阶段划分、依赖关系以 roadmap 为准，skill 自身不做预设
14. **叙事质量**：笔记必须遵循 note-refine 的费曼精神（Why 先行、因果推导、问题链衔接、场景绑定），拒绝模板化叙述
15. **深度分支研究**：每个问题必须覆盖所有主要分支，分析其核心机制、适用场景、隐藏代价和失效条件
16. **精确代码片段**：代码片段必须嵌入问题探讨的上下文中，精确展示关键机制，不需要是完整可运行程序
