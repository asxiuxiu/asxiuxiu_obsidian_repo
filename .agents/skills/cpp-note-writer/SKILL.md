# C++ 引擎笔记写作 Skill

## 服务定位

本 skill 是 `Notes/C++编程/` 目录下笔记的**专用生产工具**。它不替代 `note-refine`（叙事质量审查），而是与其配合：

- **本 skill**：负责「写什么」和「怎么组织内容」——确定问题链、生成正文、插入代码示例、建立引擎映射
- **`note-refine`**：负责「写得怎么样」——在初稿产出后进行费曼质量审查和叙述优化

产出笔记必须**单点聚焦**：一篇笔记只回答一个核心问题，覆盖一个核心机制。

## 触发场景

当用户表达以下任何一种意图时触发：

- "帮我写/新建一篇 C++ 笔记"
- "写一篇关于 XXX 的 C++ 笔记"
- "重写/整理 C++ 里的 XXX"
- "根据 RoadMap 写笔记"
- "补充 C++ 索引里标记为 ⬜ 的笔记"

## 前置依赖

执行本 skill 前，**必须先读取以下 skill**：

1. **`note-refine`** — 理解费曼叙述规范、问题链写作方法、自检清单
2. **`obsidian-markdown`** — 确保 frontmatter、wikilink、callout、代码块格式正确

## 笔记标题规范

**禁止用「为什么…」作为标题。** 标题应为陈述式或主题式，直接点明笔记要讨论的核心机制。

| ❌ 不推荐 | ✅ 推荐 |
|---------|--------|
| 为什么 `std::move` 不移动任何东西？ | 右值引用与 `std::move` 的本质 |
| 为什么虚函数调用比普通函数慢？ | 虚函数表的底层实现 |
| 为什么 C++ 需要四种类型转换？ | C++ 四种类型转换的分工与边界 |

**标题格式**：
- 不用前缀编号（如 `01-`、`1.`）
- 不用问号、冒号等 Windows 非法字符
- 文件名与笔记 `title` frontmatter 保持一致
- 示例：`C++ 虚函数表的底层实现.md`、`右值引用与移动语义.md`

## 正文叙事规范：问题链驱动

标题不用「为什么」，但**正文必须用问题链驱动**。每篇笔记的正文应是一条从「朴素认知」到「正确理解」的推导链条。

### 问题链的标准结构

```
问题 0：最朴素的理解是什么？它会出什么问题？
    → 给出一个「完全不懂的人」会写的代码/思路
    → 推演它在什么场景下会爆炸

问题 1：为了解决 X，人们引入了 Y。Y 解决了什么，又带来了什么新问题？
    → 讲清楚 Y 的机制
    → 绑定具体场景，说明 Y 的局限性

问题 2：为了解决 Y 带来的 Z，又演化出了什么？
    → ……

（依此类推，直到覆盖该机制的全部核心要点）
```

### 问题链的写作要点（继承自 note-refine）

1. **每个问题先让读者感受到痛**：绑定引擎中的具体场景，说清楚「如果不做，会在什么场景下出问题」
2. **先给 naive 方案**：写出「如果完全不懂，我会怎么写」，然后推演它哪里会出问题
3. **改进必须是因果推导**：「因为 A 有问题，所以引入 B；B 解决了 A 但又带来了 C，于是引入 D……」
4. **段落结尾自然引出下一个问题**：读者应该感到「知道了这个，我自然会想知道那个」

### 示例：一篇关于「虚函数表」的笔记的问题链

```
问题 0：如果不用虚函数，我怎么让「基类指针指向不同子类对象并调用各自的方法」？
    → naive：在基类里放枚举 type，每个函数里 switch-case
    → 问题：新增子类要改所有 switch，违反开闭原则，代码爆炸

问题 1：虚函数是怎么解决这个问题的？
    → 编译器在对象里偷偷塞了一个虚指针，指向虚函数表
    → 调用时通过 vptr 查表，找到实际要执行的函数
    → 代价：每个对象多 8 字节（64 位），每次调用多两次间接访问

问题 2：虚函数表存在内存的哪里？对象里只有 vptr 吗？
    → vtable 通常是每个类的全局唯一一张表，存在只读数据段
    → 对象的内存布局：vptr + 成员变量（按声明顺序，考虑对齐）
    → 用代码验证：`sizeof(Base)` 在有无虚函数时的差异

问题 3：多重继承时，一个对象有几个 vptr？调用时指针为什么会「漂移」？
    → 每个有虚函数的基类对应一个 vptr
    → this 指针调整：调用第二个基类的虚函数时，this 要偏移
    → 引擎中多重继承被禁用的底层原因

问题 4：纯虚函数和虚析构函数在 vtable 里长什么样？
    → 纯虚函数对应的 slot 填的是 `__cxa_pure_virtual`（或类似占位）
    → 虚析构函数为什么必须在 vtable 里（否则 delete 基类指针时不会调用子类析构）
    → 忘记虚析构 = 资源泄漏的典型案例
```

## C++ 代码示例规范

笔记中的代码块**统一使用 `cpp`** 语言标签，遵循 Run Code 插件规范：

```cpp
// flags: -O0 -g
#include <iostream>

struct Base {
    virtual void foo() { std::cout << "Base\n"; }
};

struct Derived : Base {
    void foo() override { std::cout << "Derived\n"; }
};

int main() {
    Base* b = new Derived();
    b->foo();  // 输出 Derived
    delete b;
    return 0;
}
```

| 规范 | 说明 |
|------|------|
| 编译参数 | 通过 `// flags:` 注释控制，放在代码块顶部 20 行内 |
| 追加参数 | `// flags: -O0 -g` 追加到默认参数之后 |
| 替换参数 | `// flags: override -O3 -DNDEBUG` 完全替换默认参数 |
| 默认参数 | `-std=c++20 -Wall -O2` |

**代码示例的选取原则**：
- 每个问题链节点至少有一个可运行的最小示例
- 示例要「小」——删掉一切与当前机制无关的代码
- 示例要「能验证」——运行后能看到明确输出，证明你的结论

## 引擎映射规范

每篇笔记必须在**结尾处**设置一个「引擎映射」小节，明确回答：

> **这个 C++ 机制在 SelfGameEngine / UE 中对应什么？**

```markdown
---

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 组件基类的接口派发 | `IComponent` 使用 CRTP 替代虚函数，避免 vptr 开销 |
| **UE** | `UObject` 的反射与序列化 | `UProperty` 宏生成的反射表替代了手写的 vtable，支持编辑器实时修改 |

> [!note] 关键取舍
> SelfGameEngine 在 ECS 组件存储中刻意避免虚函数，因为虚指针会破坏 SoA 布局的紧凑性。
> UE 则大量使用虚函数 + 宏生成反射信息，因为编辑器工具链需要运行时类型信息。
> 同一个机制（虚函数），在两个引擎中做出了相反的选择——理解底层原理，才能判断什么场景该用、什么场景该禁。
```

## 工作流程

### 第一步：确认主题范围

1. 读取 `Notes/C++编程/索引.md`，确认用户要写的笔记属于哪个主题
2. 明确这篇笔记的**核心问题**（只能有一个）
3. 确认前置知识：列出读者需要已经掌握的概念（用于文末的「相关笔记」链接）

### 第二步：设计问题链

1. 不要直接写正文，**先在思考中构建完整的问题链**
2. 检查问题链是否满足：
   - [ ] 每个问题都从「朴素认知」出发
   - [ ] 每个改进都有明确的「因为…所以…」因果
   - [ ] 最后一个问题覆盖边界情况和常见陷阱
3. 预估问题链长度：一篇聚焦笔记通常 **3~5 个问题**，不宜超过 7 个

### 第三步：生成正文

1. 按模板创建文件，填充 frontmatter
2. 逐问题撰写正文，每个问题包含：
   - 场景绑定（为什么这个问题值得问）
   - naive 方案或当前认知
   - 机制讲解（用自己的话写，不要复制 cppreference）
   - 代码验证（可运行的 minimal example）
3. 撰写「引擎映射」小节
4. 添加「相关笔记」wikilink

### 第四步：自检（在交付前执行）

对照以下清单自检，确认后再交付：

- [ ] **单点聚焦**：这篇笔记是否只回答了一个核心问题？如果涉及多个并列概念，是否该拆成多篇？
- [ ] **标题合规**：标题是否陈述式/主题式，没有「为什么」开头？
- [ ] **问题链完整**：正文是否按「朴素认知 → 发现问题 → 引入机制 → 深入边界」的链条推进？
- [ ] **零假定**：首次出现的术语是否有 1~2 句人话解释？
- [ ] **代码可运行**：每个代码块是否能在 Run Code 插件中编译运行？
- [ ] **引擎映射**：结尾是否有「引擎映射」小节，明确连接 SelfGameEngine / UE？
- [ ] **链接完整**：是否通过 wikilink 连接了 vault 中已有的相关笔记？
- [ ] **表格只用于总结**：表格是否只出现在机制讲解之后，而不是用于首次引入概念？

### 第五步：交付与后续

1. 将新笔记路径添加到 `Notes/C++编程/索引.md`，更新状态列
2. 如果笔记质量要求高，触发 `note-refine` 进行深度审查

## 注意事项

- **不要写百科**：拒绝「第一种…第二种…第三种…」的分类罗列。如果发现自己开始罗列，停下来，把每个分类改成一个「问题 X」。
- **不要过度引用标准原文**：cppreference 是查细节用的，笔记应该用你自己的话解释机制。只有在精确说明标准行为时才引用标准条款。
- **比喻要贯穿**：如果用了比喻（如「虚函数表就像电话簿」），确保这个比喻在整篇笔记中持续发挥作用，不是开头提一下就扔掉。
- **诚实面对不理解**：如果某个问题链节点你还没想透，不要硬写。写一个「问题笔记」记录当前的理解和待验证假设，比写一篇错误的笔记更有价值。
- **代码优先于文字**：能用 10 行代码说清楚的，不要用 200 字描述。C++ 笔记的核心是「可验证」。
