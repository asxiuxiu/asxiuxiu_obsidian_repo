# GAMES104 游戏引擎渲染系统

> 深入解析现代游戏引擎的渲染系统架构，从GPU硬件基础到前沿渲染技术，全面理解游戏图形渲染的核心原理与工程实践。

---

## 一、渲染概述

### 1.1 游戏渲染面临的挑战

现代游戏引擎的渲染系统需要应对以下核心挑战：

| 挑战 | 说明 |
| :---- | :---------------------------- |
| **渲染量大、算法复杂** | 海量物体、复杂光照、后处理效果 "all in one" |
| **硬件适配与优化** | 需适配不同 GPU 架构（NVIDIA/AMD/移动端） |
| **实时性要求** | 60fps/120fps 甚至更高帧率要求 |
| **高分辨率** | 1080p/2K/4K 分辨率带来的像素处理压力 |
| **CPU 带宽和内存限制** | 游戏逻辑、网络、动画、物理等均占用 CPU 资源 |

> [!note] 实践性知识
> 渲染是高度工程化、实践性的领域，技术更新换代极快，需要紧跟硬件发展。

### 1.2 渲染管线流程

![[Assets/games/rendering_pipeline.png]]
*渲染管线流程示意图 - 从顶点数据到像素输出的完整流程*

基础渲染管线流程：

```
顶点数据 → 图元装配 → 光栅化 → 片元着色 → 帧缓冲输出
   ↓          ↓         ↓         ↓           ↓
Vertex    Triangle    Raster    Fragment    Pixel Output
 Shader   Assembly    ization     Shader
```

现代渲染管线的关键特性：
- **可编程阶段**：顶点着色器、片元着色器可自定义
- **固定功能阶段**：光栅化、深度测试由硬件完成
- **并行处理**：GPU 同时处理数千个顶点/像素

## 二、了解 GPU

### 2.1 GPU 并行计算架构

#### SIMD vs SIMT

| 类型 | 全称 | 说明 |
|------|------|------|
| **SIMD** | Single Instruction Multiple Data | 单指令多数据，一条指令处理多个数据 |
| **SIMT** | Single Instruction Multiple Threads | 单指令多线程，GPU 多线程版 SIMD |

> GPU 拥有海量线程，适合处理简单、高度并行的计算任务。

#### GPU 核心组件

```
┌─────────────────────────────────────────┐
│          GPU (Graphics Card)            │
├─────────────────────────────────────────┤
│  GPC (Graphics Processing Cluster)      │
│  ├── SM (Streaming Multiprocessor) x N  │
│  │   ├── CUDA Cores (并行计算单元)       │
│  │   ├── Texture Units (纹理单元)        │
│  │   └── Shared Memory/L1 Cache          │
│  └── Raster Engine (光栅化引擎)          │
└─────────────────────────────────────────┘
```

| 组件 | 说明 |
|------|------|
| **GPC** | 图形处理集群，包含计算、光栅化、着色、纹理处理单元 |
| **SM** | 流式多处理器，运行 CUDA 核心，是 GPU 的基本计算单元 |
| **CUDA Core** | 并行处理器，支持多线程同时处理数据 |
| **Texture Unit** | 纹理处理单元，负责纹理采样和过滤 |
| **Warp** | 线程束，一组同时执行的线程（通常 32 个） |

### 2.2 CPU & GPU 数据传输

```
CPU + Main Memory          GPU + Video Memory
     ↑                           ↑
     │ ←──── High Latency ────→  │
     │     Limited Bandwidth      │
     │                           │
 Data Load/Unload          High Performance
 Data Preparation          Parallel Rendering
```

> [!warning] 关键原则
> **尽可能减少 CPU-GPU 数据传输**，推荐单向传输（CPU → GPU），避免 GPU 回调 CPU。

### 2.3 Cache 效率

- Cache 访问速度远高于显存
- **Cache Miss**（缓存未命中）可能导致数百个时钟周期的延迟
- 应充分利用硬件并行计算，避免冯诺依曼瓶颈

### 2.4 GPU 性能瓶颈分类

GPU 优化需先定位瓶颈类型，再针对性优化：

| 瓶颈类型 | 含义 | 优化方向 |
|----------|------|----------|
| **Memory Bounds** | 显存限制，数据读写速度跟不上 | 减少数据访问、优化内存布局 |
| **ALU Bounds** | 运算单元限制，计算量过大 | 简化着色计算、减少复杂运算 |
| **TMU Bounds** | 纹理单元限制，纹理采样过多 | 压缩纹理、使用 Mipmap、减少采样次数 |
| **BW (Bandwidth) Bounds** | 带宽限制，数据传输速率饱和 | 减少数据量、使用压缩格式 |

## 三、可渲染物体 (Renderable)

### 3.1 网格渲染组件

现代游戏引擎采用**组件化设计**（Component-Based）：

```cpp
// 游戏对象 = 组件容器
class GameObject {
    std::vector<Component*> components;
};

// 渲染专用组件
class MeshRenderComponent : public Component {
    Mesh* mesh;        // 几何数据
    Material* material; // 材质/着色器
    void Render();     // 提交渲染命令
};
```

**设计优势**：
- 灵活组合，运行时装配
- 关注点分离
- 组件可复用

### 3.2 网格数据结构

![[Assets/games/mesh_index_buffer.png]]
*网格顶点结构与索引缓冲区 - 通过索引复用顶点数据*

#### 基础定义

```cpp
// 顶点结构
struct Vertex {
    Vector3 position;    // 3D 空间位置
    UByte4  m_color;     // 顶点颜色 (RGBA)
    Vector3 m_normal;    // 顶点法线
    Vector2 uv;          // 纹理坐标
    // ... 其他数据
};

// 三角形（直接存储方式，不推荐）
struct Triangle {
    Vertex m_vertex[3];  // 3个顶点定义一个三角形
};
```

#### 问题：顶点冗余

直接存储三角形的缺点：
- **内存冗余**：共享顶点的三角形重复存储相同数据
- **带宽浪费**：GPU 需要传输更多数据
- **无法平滑着色**：独立顶点难以实现法线插值

#### 解决方案：索引缓冲区 (Index Buffer)

```
┌─────────────────┐    ┌─────────────────┐
│  顶点缓冲区      │◄────│  索引缓冲区      │
│ (Vertex Buffer) │    │ (Index Buffer)  │
│                 │    │                 │
│0: (0, 0, 0)    │    │三角1: [0,1,2]   │
│1: (1, 0, 0)    │    │三角2: [0,2,3]   │
│2: (1, 1, 0)    │    │...             │
│3: (0, 1, 0)    │    │                │
└─────────────────┘    └─────────────────┘
```

**改进后的数据结构**：

```cpp
// 顶点数组 - 只存唯一的顶点
std::vector<Vertex> vertices;      // 8个顶点

// 索引数组 - 定义三角形
std::vector<uint32_t> indices;     // 36个索引(12个三角形 × 3)
```

**好处**：
- 内存节省（立方体：8顶点 vs 36顶点，减少~78%）
- GPU 带宽降低
- 顶点缓存复用，着色器只需计算一次
- 共享顶点实现平滑着色

### 3.3 逐顶点法线

> **Q: 为什么顶点会有法线属性？顶点不是只是一个点，没有朝向吗？**

**A**: 顶点法线存储的是**该顶点所在表面的朝向**，用于光照计算。

#### 三角形法线 vs 顶点法线

| | 三角形法线 | 顶点法线 |
|:---|:---|:---|
| **计算方式** | 用叉积算出整个三角形的统一朝向 | 周围三角形法线的平均 |
| **效果** | 平面感强，棱角分明 | 平滑过渡，曲面效果 |
| **用途** | 平面物体（立方体） | 曲面物体（球体、角色） |

#### 硬边 vs 软边

```
【硬边立方体】             【平滑球体】
每个面独立法线             共享顶点平均法线
   ◢│◣      ←棱角分明       ●  ●  ←连续渐变
  / │ \                        )  (  
 /  │  \                       (    )
```

> **关键洞察**：当一个几何顶点被多个三角形共享，但每个三角形需要不同的法线/UV时，需要**顶点拆分**。这就是"硬边"（Hard Edge）vs "软边"（Soft Edge）的区别。

### 3.4 图元类型

| 图元类型 | 顶点共享方式 | 特点 |
|----------|--------------|------|
| **Triangle List** | 互不共享，3个顶点独立 | 最灵活，配合索引缓冲使用 |
| **Triangle Strip** | 相邻三角形共享2个顶点 | "一笔画"，顶点复用效率高 |
| **Triangle Fan** | 所有三角形共享1个公共顶点 | 放射状，适合扇形几何 |

> 现代 GPU 标准做法：**顶点数据复用 + 索引驱动绘制**（Triangle List + Index Buffer）

## 四、材质系统 (Materials)

### 4.1 材质的定义

![[Assets/games/pbr_materials.png]]
*不同 PBR 材质效果对比 - 金属度与粗糙度的组合效果*

**材质决定物体的外观，以及物体如何与光线交互。**

| 材质类型 | 特点 | 光线交互 |
|---------|------|----------|
| **Smooth metal**（光滑金属） | 高反射、镜面效果 | 镜面反射，清晰反射环境 |
| **Glossy paint**（光泽油漆） | 光滑但有漫反射 | 柔和高光，部分反射 |
| **Rough stone**（粗糙石头） | 表面不平整 | 漫反射为主，无清晰反射 |
| **Transparent glass**（透明玻璃） | 可透光 | 折射+反射，光线穿透 |

### 4.2 材质模型演进

```
传统光照模型 → 现代物理渲染
     ↓             ↓
  Phong模型      PBR + 高级效果
```

| 模型 | 组成 | 说明 |
|------|------|------|
| **Phong** | Ambient + Diffuse + Specular | 经验模型，1975年经典模型 |
| **PBR** | 基于物理原理 | 能量守恒、微表面模型 |
| **Subsurface** | + Burley SubSurface Profile | 次表面散射（皮肤、蜡、玉石） |

> **材质 vs 物理材质**：视觉材质关注"看起来如何"；物理材质基于真实物理属性（折射率、粗糙度等），在不同光照下表现一致。

### 4.3 PBR 材质工作流程

![[Assets/games/pbr_textures.jpg]]
*PBR 材质贴图组成 - Albedo、Metallic、Roughness、Normal、AO*

PBR（Physically Based Rendering）是现代游戏渲染的标准，核心是 **Metal/Roughness 工作流**：

| 贴图 | 含义 | 作用 |
| -------------------- | ------------- | ------------- |
| **Albedo**（反照率/基础颜色）| 材质本身的颜色，不含光照 | 决定表面固有颜色 |
| **Metallic**（金属度） | 0=非金属，1=纯金属 | 区分金属/非金属反射类型 |
| **Roughness**（粗糙度） | 0=镜面光滑，1=完全粗糙 | 控制微表面散射程度 |
| **Normal**（法线） | 记录表面凹凸细节 | 模拟表面起伏，不增加多边形 |
| **AO**（环境光遮蔽） | 缝隙/凹陷处的阴影 | 增强体积感和接触阴影 |

#### PBR 核心原理

1. **能量守恒**：反射光能 + 吸收光能 = 入射光能
2. **微表面理论**：表面由微观几何决定反射特性
3. **菲涅尔效应**：反射率随观察角度变化

### 4.4 Shader 与材质的关系

```
Mesh（几何） → Submesh（按材质划分） → Material（材质参数）
                                              ↓
                                         Shader（着色器程序）
                                              ↓
                                         Texture（纹理数据）
```

- **Shader**：定义如何计算光照和颜色（程序逻辑）
- **Material**：Shader 的实例化，包含具体参数
- **Texture**：材质使用的图像数据

> [!tip] Shader Graph
> 现代引擎提供可视化 Shader 编辑器（连连看），让美术人员无需编写代码即可创建 Shader。

## 五、子网格 (Submesh)

### 5.1 为什么需要 Submesh？

![[Assets/games/submesh_structure.png]]
*Submesh 内存布局 - 按材质划分的子网格结构*

**问题**：单个模型可能需要显示不同纹理（如角色的头盔、皮肤、服装、鞋子）。

**解决方案**：将 Mesh 按材质需求切分为多个 Submesh。

```
┌─────────────────────────────────────┐
│          Character Mesh             │
├─────────┬─────────┬─────────┬───────┤
│Submesh 0│Submesh 1│Submesh 2│Submesh3│
│ Helmet  │  Skin   │Clothing │ Shoes │
│   ☰    │   ☰    │   ☰    │  ☰   │
│Shader 0 │Shader 1 │Shader 2 │Shader3│
│Texture 0│Texture 1│Texture 2│Texture3│
└─────────┴─────────┴─────────┴───────┘
```

每个 Submesh 通过 `offset`（起始索引）和 `count`（数量）在共享的 Vertex/Index Buffer 中定位数据。

### 5.2 Submesh 的优势

- **避免模型碎片化**：无需拆分为独立模型文件
- **材质灵活组合**：可换皮肤、换装备，只需替换对应 Submesh 材质
- **渲染批次优化**：相同材质的 Submesh 可合并绘制

### 5.3 资源实例化 (Instance)

![[Assets/games/gpu_instancing.png]]
*GPU实例化 - 共享几何数据，独立变换渲染大量相同物体*

为避免多个模型重复存储相同数据：

```
Res Pool（资源池）         Instance（实例）
┌──────────────┐          ┌─────────────────┐
│ Mesh Data    │◄─────────│ Mesh Handle     │
│ Shader Code  │◄─────────│ Material Handle │
│ Texture Data │◄─────────│ Transform Data  │
└──────────────┘          └─────────────────┘
```

- **资源池**：存储实际资源数据（Mesh、Shader、Texture）
- **实例**：只存句柄引用 + 变换等可变数据

**优势**：1000 个士兵 = 1 个 Mesh + N 份材质，而非 1000 份完整数据。

### 5.4 按材质排序与 GPU Batch Rendering

为提高效率，渲染流程通常：

1. **按材质排序**：相同材质的物体 Group 到一起
2. **减少状态切换**：只需设置一次材质，减少 GPU 等待
3. **GPU Batch Rendering**：一个 Draw Call 渲染大量同材质物体

```
Sort by Material → Update Shader/Texture/Params → Draw Instances
```

> [!warning] 状态切换开销
> 频繁更换 Shader/贴图/缓冲区会打断 GPU 流水线，"显卡很烦" → 性能下降。Batching 是核心优化手段。

## 六、可见性剔除 (Visibility Culling)

### 6.1 包围体 (Bounding Volume)

**核心思想**：用最简单的几何体包裹复杂物体，快速剔除不可见物体。

#### 理想包围体的特点

1. 相交测试计算便宜
2. 紧密贴合原物体
3. 构建成本低
4. 变换（旋转/平移）方便
5. 内存占用小

> ⚠️ 这些特性互相矛盾，需要权衡。

#### 常见包围体对比

| 包围体 | 测试速度 | 贴合度 | 内存 | 适用场景 |
|--------|----------|--------|------|----------|
| **Sphere（球体）** | 最快 | 最差 | 最小 | 远距离粗略剔除 |
| **AABB（轴对齐包围盒）** | 较快 | 一般 | 小 | 静态物体、场景管理 |
| **OBB（定向包围盒）** | 较快 | 较好 | 较小 | 动态旋转物体 |
| **8-DOP** | 较快 | 较好 | 中等 | 折中方案 |
| **Convex Hull（凸包）** | 最慢 | 最好 | 最大 | 高精度需求 |

### 6.2 视锥剔除 (View Frustum Culling)

![[Assets/games/frustum_culling.png]]
*视锥剔除 - 只渲染视锥体内的物体，剔除外部物体*

通过判断物体与相机视锥体的空间关系进行剔除：

- **完全在外** → 剔除
- **完全在内** → 保留
- **相交** → 继续细分判断

### 6.3 分层剔除技术

![[Assets/games/bvh_culling.png]]
*BVH层级剔除 - 自顶向下的包围体层次结构加速剔除*

#### Hierarchical View Frustum Culling

从粗到细分层判断：
- 完全在外 → 剔除
- 完全在内 → 保留全部子节点
- 相交 → 继续细分判断

#### BVH (Bounding Volume Hierarchy)

```
       ┌─────┐
       │ Root│ (Scene Bounds)
       └──┬──┘
      ┌───┴───┐
   ┌──┐     ┌──┐
   │A │     │B │ (Branch Nodes)
   └┬─┘     └┬─┘
  ┌┴┐      ┌┴┐
  │●│      │●│ (Leaf Nodes - Objects)
  └─┘      └─┘
```

- 用树结构组织物体，每个节点存储包围盒
- 自顶向下遍历，父节点与视锥无关 → 整棵子树剔除

#### BVH 构建方法

| 方法 | 特点 | 适用场景 |
|------|------|----------|
| **Top-down** | 先算整体包围盒，再递归分割 | 静态场景，构建速度快 |
| **Bottom-up** | 先为每个物体建叶节点，再两两合并 | 质量较高，构建慢 |
| **Incremental insertion** | 动态逐个插入，实时更新 | 动态场景，物体频繁变化 |

### 6.4 PVS (Potential Visibility Set)

**潜在可见集**：预计算每个区域可能看到的其他区域。

**BSP-Tree 构建**：
1. 用平面递归分割空间
2. 每个叶节点（Leaf Node）代表一个凸空间区域
3. 叶节点之间通过 Portal（门/窗）连接

**PVS 计算**：
- 从每个叶节点向外发射射线，通过 Portals 可看到哪些其他叶节点
- 运行时只渲染当前叶节点及其 PVS 内的物体

**优点**：
- 比 BSP/Octree 查询更快
- 可用于资源预加载（异步加载即将可见的资源）

### 6.5 GPU Culling

利用 GPU 并行计算能力，在渲染前快速剔除被遮挡物体。

#### 遮挡剔除流程

```
1. 渲染 Occluder（遮挡物）到深度缓冲
   ↓
2. 对每个 Occludee（候选物体）：
   - 渲染其包围盒到 GPU Query
   - 查询是否被遮挡
   ↓
3. 剔除被遮挡的物体
   ↓
4. 渲染可见物体的实际模型
```

**Early-Z / Pre-Z**：
- 先渲染不透明物体的深度（关闭颜色写入）
- 后续渲染利用深度测试提前剔除被遮挡像素

> 现代 GPU 可直接在 GPU 端完成剔除决策，避免 CPU-GPU 往返延迟。

## 七、纹理压缩 (Texture Compression)

### 7.1 游戏纹理 vs 传统图像压缩

| 特点 | 传统压缩（JPG/PNG）| 游戏纹理压缩 |
|------|---------------------|--------------|
| **优化目标** | 压缩率、图像质量 | 解码速度、随机访问 |
| **处理单位** | 整张图像 | 小块纹理（如 4×4 像素块） |
| **随机访问** | 差（需解码整张图） | 快（可直接访问任意块） |
| **适用场景** | 存储/传输 | 实时渲染（GPU 快速读取） |

### 7.2 块压缩 (Block Compression)

![[Assets/games/block_compression.png]]
*块压缩原理 - 4×4 像素块压缩，BC7/ASTC 格式示意图*

GPU 纹理压缩的核心技术，将图像分为 4×4 像素块：

```
原始 4×4 块：                    压缩后 64 bits
┌──┬──┬──┬──┐                  ┌──────────────────┐
│P0│P1│P2│P3│                  │ Color 0: 16 bits │
├──┼──┼──┼──┤                  ├──────────────────┤
│P4│P5│P6│P7│  ──────→        │ Color 1: 16 bits │
├──┼──┼──┼──┤                  ├──────────────────┤
│P8│P9│PA│PB│                  │ Index: 32 bits   │
├──┼──┼──┼──┤                  │ (16 × 2 bits)    │
│PC│PD│PE│PF│                  └──────────────────┘
└──┴──┴──┴──┘

原始数据：16像素 × 24bit = 384 bits
压缩数据：64 bits
压缩比：6:1
```

**压缩过程**：
1. 将 4×4 块内像素按亮度排序
2. 记录最亮（c0）和最暗（c1）的颜色（各 16 bits）
3. 生成插值色阶（通过 2/3、1/3 混合 c0 和 c1）
4. 每个像素用 2-bit 索引选择色阶中的颜色

### 7.3 主流压缩格式

| 平台 | 现代格式 | 旧格式 |
|------|----------|--------|
| **PC/主机** | **BC7**（高质量，128 bits/块） | DXTC/BC1-BC5 |
| **移动端** | **ASTC**（灵活，可变比特率） | ETC2、PVRTC |

**BC7 vs BC1**：
- BC1：64 bits/块，颜色质量一般，有透明度支持
- BC7：128 bits/块，更高质量，支持更多模式

## 八、GPU Driven 渲染管线

### 8.1 传统 vs GPU Driven

```
传统管线：                   GPU Driven 管线：
                                              
CPU: 剔除/LOD/排序    →     CPU: 仅提交命令
      ↓                            ↓
GPU: 接收并渲染            GPU: 剔除/LOD/排序 + 渲染
```

**GPU Driven 的核心优势**：
- **减少 CPU-GPU 通信**：所有剔除计算在 GPU 完成
- **更细粒度的剔除**：Cluster 级别而非 Object 级别
- **更好的并行性**：利用 Compute Shader 进行大规模并行计算

### 8.2 Mesh Cluster 技术

将模型分割为固定大小的 Cluster（通常 64-128 个三角形）：

```
传统方式：          Cluster-Based 方式：
┌──────────┐        ┌──────────────────────────┐
│模型整体  │        │ ┌──┐┌──┐┌──┐┌──┐┌──┐    │
│ 剔除     │        │ │C1││C2││C3││C4││C5│    │
│ 渲染     │        │ └──┘└──┘└──┘└──┘└──┘    │
└──────────┘        │ Cluster（~64 三角形）    │
                     │ 单独剔除、排序、渲染    │
                     └──────────────────────────┘
```

**2021年架构三级分层**：

| 层级 | 定义 | 组成 |
|------|------|------|
| **Batch** | 单次 API 绘制调用 | 多个 Surf |
| **Surf** | 按材质划分的子网格 | 多个 Cluster |
| **Cluster** | 64 个三角形条带 | 最小剔除/调度单元 |

### 8.3 Mesh Shader 管线

现代 GPU 支持的可编程网格管线：

```
传统管线：                   新网格着色器管线：
                                                              
Vertex Shader ──→           Task Shader ──→
                ├──→ Raster     （动态决定）   ↓
Geometry Shader ─┘               ↓         ↓
                              Mesh Shader ────→
                               （直接生成几何）├──→ Raster
Hull/Domain                    Primitive
（曲面细分）                    Rasterization
```

**关键新阶段**：
- **Task Shader**：动态决定生成多少个 Mesh Shader 工作组
- **Mesh Shader**：直接生成三角形网格，替代传统顶点/几何着色器

**优势**：
- 绕过固定功能顶点处理
- GPU 驱动剔除、LOD 选择
- 更高效率的几何处理

## 九、Nanite 虚拟几何体

### 9.1 Nanite 核心思想

UE5 Nanite 是 Cluster-Based Mesh Pipeline 的极致实现：

**核心突破**：
- **虚拟微多边形几何体**：直接导入电影级资产（ZBrush雕刻、摄影测量）
- **自动 LOD**：无需手动设置，根据视角动态选择细节级别
- **无 pop-in**：LOD 过渡无缝衔接
- **内存高效**：高度压缩的数据格式，细粒度流送

### 9.2 Nanite 剔除流程

```
Nanite 剔除流程：

Instance Culling（实例剔除）
         ↓
Cluster Chunk Expansion（簇块展开）
         ↓
Cluster Culling（簇剔除：视锥/遮挡+背面）
         ↓
Index Buffer Compaction（索引压缩）
         ↓
Multi-Draw（多绘制调用）
```

### 9.3 Nanite 技术细节

**分层 LOD Clusters**：
- 边界无缝衔接，消除 LOD 切换瑕疵
- 预计 BVH 用于分层簇剔除

**持久化计算着色器**：
- 在 GPU 上持续运行，替代 Task Shader
- 通用 DX11+ GPU 即可运行，不依赖专用硬件

**Visibility Buffer**：
- 存储三角形 ID 而非完整 G-Buffer
- 着色阶段根据 ID 获取材质数据
- 极大降低显存带宽

### 9.4 Nanite 限制与未来

**当前限制**：
- 主要针对静态几何
- 变形物体支持有限
- 透明材质需要特殊处理

**发展方向**：
- 动态 Nanite（植被、布料）
- 实时光线追踪集成
- 跨平台优化（移动端/VR）

## 十、总结与要点

### 10.1 核心设计原则

1. **Do Nothing**：用 Culling 算法让引擎绘制尽可能少的内容
2. **数据与实例分离**：资源池 + 句柄复用，减少内存开销
3. **按材质排序**：减少 GPU 状态切换，提升 Batch 效率
4. **GPU Driven**：将剔除等计算从 CPU 转移到 GPU

### 10.2 关键知识脉络

```
GPU 架构理解 ──┬──→ SIMD/SIMT 并行计算
               ├──→ Memory/ALU/TMU/BW 瓶颈定位
               └──→ CPU-GPU 数据传输优化

可渲染物体 ────┬──→ Mesh & Submesh 划分
               ├──→ Vertex/Index Buffer 优化
               ├──→ Material/Shader/Texture 关系
               └──→ Instance 复用 & Batch

可见性剔除 ────┬──→ 包围体选择（Sphere/AABB/OBB）
               ├──→ BVH 构建与分层剔除
               ├──→ PVS 预计算
               └──→ GPU Culling

纹理压缩 ─────→ Block Compression（BC7/ASTC）

现代管线 ─────→ Cluster-Based / Mesh Shader / Nanite
```
