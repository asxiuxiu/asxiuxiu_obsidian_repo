# 引擎脚本构建分析（知识提炼）

> [!info] 说明
> 本笔记基于对公司 chaos 引擎构建脚本的分析，提炼出**通用的工程化知识**，不涉及保密细节。

---

## 一、分层脚本架构

### 是什么

`build.bat` 作为主入口，依次调用多个子脚本，每个子脚本承担单一职责：

```
build.bat（入口/调度）
├── setup_project_tree.bat    → 配置管理
├── delete_old_path.ps1       → 迁移清理
├── conan_install.bat         → 依赖安装
├── conanbuild.bat            → 环境注入
└── generate_launch_projects.bat → 源码树生成
```

### 为什么这样做

这是 **单一职责原则（SRP）** 在构建脚本层面的落地。每个子脚本只做一件事，互相解耦。

**好处：**
- 单个步骤可独立调试（如只跑 `conan_install.bat`）
- 团队成员可以只关注自己负责的那一层
- 替换某个步骤（如从 conan 换成 vcpkg）时，只需改一个文件
- CI/CD 流水线可以按需跳过某些步骤

**坏处：**
- 多文件跳转让新人难以理解全貌（需要"追着调用栈看"）
- Windows `.bat` 的变量作用域规则反直觉（`%var%` 在解析时展开，`!var!` 在运行时展开），调试成本高
- 跨平台性差：这套脚本只能在 Windows 上运行

### 通用经验

> 构建脚本和应用代码一样，应当模块化。超过 200 行的单一构建脚本是重构信号。

对于跨平台项目，应优先选用 CMake、Python 或 Just 等跨平台工具替代 `.bat`/`.sh`。

## 二、依赖管理：Conan 2

### 是什么

项目将 Conan 2 可执行文件直接内置于代码仓库（`misc/tools/windows/conan2/conan.exe`），通过 `conanfile.py` 声明三方依赖，生成 `conan_toolchain.cmake` 供 CMake 使用。

### 为什么用 Conan，不用其他方案

| 方案                | 优点                                                     | 缺点                             |
| ----------------- | ------------------------------------------------------ | ------------------------------ |
| **Conan 2**       | 支持多配置(Debug/Release/ASAN)、lockfile 锁版本、profile 管理跨平台编译 | 学习曲线陡，服务器需要搭 Artifactory/Nexus |
| **vcpkg**         | 微软维护，VS 集成无缝，开源包多                                      | 版本控制弱，customization 较难         |
| **Git submodule** | 简单直接                                                   | 源码编译慢，版本管理麻烦                   |
| **手动管理**          | 无依赖                                                    | 地狱级维护成本                        |

Conan 的核心优势是 **profile 系统**：可以用同一份 `conanfile.py` 针对不同平台/编译器/配置生成不同的二进制，游戏引擎需要同时维护 PC/主机/移动端时价值尤为突出。

### Conan 2 的推荐工作流

根据官方文档，标准流程是：

```bash
# 1. 先跑 conan install，生成 conan_toolchain.cmake
conan install . --output-folder=build --build=missing

# 2. 用生成的 toolchain 配置 CMake
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=build/generators/conan_toolchain.cmake

# 3. 构建
cmake --build build
```

`CMAKE_TOOLCHAIN_FILE` 是 Conan 与 CMake 之间的**桥接文件**：它自动设置头文件路径、库路径、编译器标志，让 CMake 的 `find_package()` 能找到 Conan 安装的包。

> [!note] 为什么要先 conan 再 cmake？
> CMake 在 configure 阶段就需要知道三方库的路径，所以必须先让 Conan 把库装好并生成 toolchain，CMake 才能"看到"这些库。顺序颠倒会找不到依赖。

### Lockfile 的价值

`lockfiles/windows.json` 锁定了所有依赖的精确版本。

**为什么这很重要：** 包管理器默认行为是"拉最新版本"，这在大型项目中会导致"在我机器上能跑"问题。Lockfile 确保所有人（包括 CI）使用完全相同的依赖版本，是**构建可重现性（Reproducible Builds）** 的基础。

### 内置工具链 vs 系统安装

项目把 `conan.exe` 直接放入仓库，而不是要求开发者系统安装 conan。

**好处：** 零环境依赖，clone 仓库即可构建，不存在版本不一致问题。

**坏处：** 仓库体积变大；工具需要手动更新；严格来说这是"二进制提交"，版本管理不够透明。

**替代做法：** 用 `mise`、`asdf`、`nix` 等版本管理工具在环境级别固定工具版本，比二进制提交更干净。

### 发现的 Bug：重复安装

原始脚本中，每个 build type 的 conan install 命令被执行了**两次**（一次用 `%CMD%`，一次用 `!CMD!`）。这是 Windows 批处理延迟展开（delayed expansion）的常见陷阱：

```bat
set CMD=conan install ...
call %CMD%    ← 解析时展开（可能是空字符串）
call !CMD!    ← 运行时展开（实际值）
```

如果 `enabledelayedexpansion` 已开启，`%CMD%` 在 for 循环内不会按预期展开，导致两行都在尝试执行同一个命令。正确做法是只保留 `call !CMD!`。

> **通用教训：** Windows 批处理的变量展开规则是历史包袱，生产脚本应避免使用 `.bat`，改用 PowerShell 或 Python。

## 三、Source Forest 模式

### 是什么

`project_tree_config.json` 描述了本次构建要拉哪些代码仓库、怎么组织目录结构。通过切换这个文件，可以组合出不同的"构建套餐"（仅引擎、引擎+游戏、仅工具等）。

### 为什么需要这个模式

这是大型项目**多仓库管理**的一种解法。游戏引擎项目通常有数十个独立 Git 仓库（引擎核心、各平台 SDK、工具链、示例游戏……），不可能也不应该塞进同一个仓库。Source Forest 用一个配置文件描述"我需要哪些仓库、放在哪里"，类似：

- Google 的 `repo` 工具（Android 开发用）
- `.gitmodules`（Git Submodules）
- `west.yml`（Zephyr RTOS 用）

### 利弊对比

| | Source Forest（JSON 配置） | Git Submodules | Monorepo |
|--|--|--|--|
| **灵活性** | 高，可组合任意子集 | 中 | 低，全量 |
| **原子提交** | 不支持跨仓 | 不支持跨仓 | 支持 |
| **CI 复杂度** | 高 | 中 | 低 |
| **磁盘占用** | 按需 | 按需 | 全量 |
| **新人上手** | 困难 | 中等 | 容易 |

Source Forest 模式适合**超大型项目**（引擎本身数百万行代码），Monorepo 适合中小型公司（参考 Google 的实践，他们的内部工具复杂到外部无法复制）。

## 四、模板化 CMakeLists 生成

### 是什么

`generate_launch_projects.bat` 调用 `chaos_launch_generator.exe`，将 `launch/templates/` 下的模板文件渲染成 `build/sourcetree/`，其中包含真正供 CMake 使用的 `CMakeLists.txt`。

```
launch/templates/CMakeLists.txt  →（渲染）→  build/sourcetree/CMakeLists.txt
```

### 为什么不直接用静态的 CMakeLists.txt

**原因：平台差异 + 动态内容**

模板引擎可以根据目标平台（Windows/PS5/Switch/iOS）、构建配置、功能开关，生成不同内容的 `CMakeLists.txt`，避免在 CMake 文件中堆砌大量 `if(PLATFORM_XXX)` 判断。

类比：这就像 Web 开发中的**服务端模板渲染**——同一套模板，根据请求参数生成不同的 HTML。

**好处：**
- CMake 代码更干净，无需大量平台判断
- 可以为不同平台维护不同模板，互不干扰

**坏处：**
- 增加了一个构建前置步骤，新人容易不知道要先跑这一步
- 生成文件不在源码中，IDE 的"跳转到定义"会失效
- 调试 CMake 问题时需要先找到生成的文件，增加一层间接

> **通用经验：** 代码生成是强大的工具，但每增加一层生成就增加一层认知负担。只在模板逻辑足够复杂时才值得引入。

## 五、多构建配置：Profile / ASAN / Check / Ship

### 是什么

同一份代码，用不同的编译参数生成 4 种目标：

| 配置 | 用途 | 典型编译参数 |
|------|------|------------|
| **Profile** | 日常开发，带调试符号 | `-O1 -g`，类似 Debug |
| **ASAN** | 内存错误检测 | `-fsanitize=address -g` |
| **Check** | QA 验证，接近 Release | `-O2`，部分断言保留 |
| **Ship** | 正式发布 | `-O3 -DNDEBUG`，无调试符号 |

### 为什么需要这么多配置

**ASAN（Address Sanitizer）** 是 Google 开发的内存错误检测工具，能在运行时检测越界访问、use-after-free 等问题，但有约 2x 性能开销，所以只用于开发测试，不能出现在 Ship 版本中。

**多配置同时安装的意义：** Conan 一次性安装所有配置的三方库，CI 可以并行跑不同配置的测试，无需重新 configure。这是 Visual Studio 多配置生成器（Multi-Config Generator）的核心优势——**一次 cmake 配置，多次构建**。

```cmake
-- Multi-Config 生成器（VS/Xcode/Ninja Multi-Config）
cmake -S . -B build -G "Visual Studio 17 2022"
# 此后可以随时切换配置：
cmake --build build --config Profile
cmake --build build --config ASAN
cmake --build build --config Ship
```

对比单配置生成器（Makefiles/Ninja），切换配置需要重新 configure，效率低。

> **通用经验：** 在 Windows 上用 CMake，优先选多配置生成器（Visual Studio 或 Ninja Multi-Config），避免频繁 reconfigure。在 Linux 上单配置生成器更常见，但可以通过 CMake Presets 管理多个 build 目录。

## 六、版本迁移清理脚本

### 是什么

`delete_old_path.ps1` 在每次构建前删除历史遗留文件（各次引擎升级后不再需要的文件）。

### 为什么这样做，而不是让开发者手动清理

**长生命周期项目的必然选择。** 引擎跨越多年开发，中间经历 .NET 升级、模块重构、第三方库换版等，这些变更会遗留过时的 `.dll`、`.cs`、`.cpp` 文件，如不清理可能导致：
- 链接到旧版本库
- 旧 AssemblyInfo.cs 与新版冲突
- 构建输出目录污染

让脚本自动清理比依赖开发者"记得手动删"可靠得多。

### 利弊分析

**好处：** 幂等性——无论在什么状态的机器上，跑完脚本后环境都是干净的。这是**自动化运维**的核心思想（类比 Ansible 的 `state: absent`）。

**坏处：**
- 脚本随时间增长，积累大量已注释的"历史记录"（文件里能看到多个被注释掉的旧清理逻辑），维护成本上升
- 如果清理逻辑有 bug，可能误删文件（见原脚本中的路径 pattern 匹配）

**更好的替代方案：** 用 CMake 的 `INSTALL` 目标配合 manifest 文件管理安装产物，CMake 天然知道哪些文件该存在，可以自动清理过期文件。但这需要从 CMake 层面设计，改造成本高。

## 七、安全反模式：凭证硬编码

### 现象

`conan_install.bat` 中硬编码了 GitLab Personal Access Token（`glpat-...`），用于拉取私有仓库的 conan profiles。

### 为什么这是危险的

即使代码只在内网，硬编码 token 也有严重风险：
1. Token 出现在 `git log` 历史中，即使后续删除也可通过历史找回
2. 任何有仓库读权限的人都能拿到 token，用它访问本不该访问的资源
3. Token 泄露后难以追踪是谁造成的（审计困难）

### 正确做法

```
# 方案一：环境变量
CONAN_PAT=$GITLAB_TOKEN conan config install ...

# 方案二：CI/CD 的 Secrets
# GitHub Actions: secrets.GITLAB_TOKEN
# GitLab CI: CI/CD Variables（masked）

# 方案三：Credential Helper（git credential store）
# 配置一次，脚本中不出现 token
```

> **通用经验：** 任何密钥（API token、密码、私钥）都不应出现在代码文件中，哪怕是内网项目。使用环境变量或 Secret Manager。这是 [OWASP Top 10 A02: Cryptographic Failures](https://owasp.org/Top10/) 的典型场景。

## 八、配置版本化管理

### 是什么

`setup_project_tree.bat` 的核心逻辑：检查 `project_tree_config.json` 的 `ConfigVersion` 字段，若与默认版本不一致，自动用 `.default` 文件覆盖。

### 为什么这样做

当 schema 升级时（新增必填字段、改变格式），旧的本地配置文件会不兼容。这个脚本实现了**配置文件的强制版本迁移**。

类比场景：数据库 schema 迁移（Flyway、Alembic）、应用的 `settings.json` 版本升级（VSCode 做法）。

**好处：** 开发者无需手动更新配置，减少"为什么我的配置不对"类问题。

**局限：** 这是破坏性迁移（直接覆盖），会丢失用户的本地自定义配置。更友好的做法是**合并迁移**：保留用户修改的字段，只更新新增/变更的字段。

## 九、通用经验总结

| # | 经验 | 适用场景 |
|---|------|---------|
| 1 | **构建脚本模块化** — 每个脚本单一职责，主入口只做调度 | 任何中大型项目 |
| 2 | **先装依赖再 configure** — Conan/vcpkg 需在 cmake 之前完成 | C++ 项目 |
| 3 | **Lockfile 锁依赖版本** — 可重现构建的基础 | 所有使用包管理器的项目 |
| 4 | **多构建配置并存** — Debug/ASAN/Profile/Release 各有用途，不要只用 Debug 和 Release | 性能敏感的 C++ 项目 |
| 5 | **Source Forest 适合超大型多仓项目** — 中小项目用 monorepo 更简单 | 10+ 个独立仓库的项目 |
| 6 | **代码生成增加认知层** — 只在逻辑足够复杂时引入，并做好文档 | 模板/生成器场景 |
| 7 | **迁移脚本要幂等** — 无论初始状态如何，跑完结果一致 | 长生命周期项目的升级脚本 |
| 8 | **凭证不入代码** — 用环境变量/CI Secrets，永远 | 所有项目 |
| 9 | **配置版本化** — schema 变更时提供自动迁移路径 | 有用户自定义配置的项目 |
| 10 | **避免 Windows `.bat`** — 用 PowerShell/Python 替代，减少变量展开的心智负担 | 新项目 |
