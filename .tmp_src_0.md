# platform-源码解析：窗口与输入

## Why：为什么要分析 platform 层的窗口与输入？

在 chaos 引擎的架构中，`platform/` 负责操作系统级别的抽象。**窗口管理**和**输入处理**是游戏引擎与操作系统交互的最前沿。理解 platform 层提供的窗口与输入接口，有助于搞清楚：
- 引擎如何跨平台处理剪贴板、光标、控制台输出
- 中文、日文、韩文等 CJK 输入如何通过 IME（输入法编辑器）与游戏集成
- 上层 `client/` 的窗口系统和输入事件队列如何建立在 platform 之上

**注意**：chaos 引擎的完整窗口抽象（创建、大小调整、全屏切换、事件泵）和输入设备抽象（键盘、鼠标、手柄）可能位于更上层的 `client/` 模块中。`platform/` 只提供最底层的 OS 工具函数。

---

## What：platform 层对外提供了什么？

| 类 | 文件 | 职责 |
|----|------|------|
| `Console` | `console/console.h` | 控制台/光标/剪贴板操作 |
| `IME` | `ime/ime.h` | 输入法编辑器接口 |

## How：源码级深度解析

### 第 1 层：接口层

#### 1.1 Console — 控制台与光标工具

> 文件：`chaos/_source/_engine/source/platform/public/chaos/platform/console/console.h`

```cpp
class PLATFORM_API Console
{
public:
    static const std::wstring getClipboardText();
    static void setClipboardText(const wchar_t* text, size_t len);
    static void setConsoleTitle(const std::string& title);
    static void setCursorPosition(std::int32_t x, std::int32_t y);
    static void getCursorPosition(std::int32_t& x, std::int32_t& y);
    static void clipCursor(std::int32_t x, std::int32_t y, std::uint32_t width, std::uint32_t height);
    static void cancelClipCursor();
    static bool chekcRegKey(const std::string& path);
    static bool isRemoteSession();
    static void setCursor(void* hwnd, void* cursor_handle);
    static void screenToClient(void* hwnd, std::int32_t& x, std::int32_t& y);
};
```

**接口语义**：
- `getClipboardText` / `setClipboardText`：跨平台剪贴板读写（使用宽字符，支持 Unicode）。
- `setConsoleTitle`：设置控制台窗口标题（Debug/Check 构建常用）。
- `setCursorPosition` / `getCursorPosition`：获取/设置光标在屏幕坐标系中的位置。
- `clipCursor` / `cancelClipCursor`：将鼠标光标限制在指定矩形区域内（FPS 游戏常用，防止鼠标移出窗口）。
- `isRemoteSession`：检测当前是否处于远程桌面/远程会话中（用于性能降级或反作弊）。
- `setCursor` / `screenToClient`：Win32 风格的光标句柄和坐标转换。

**拼写错误**：`chekcRegKey` 应为 `checkRegKey`，但接口已固定，上层调用时需注意。

#### 1.2 IME — 输入法编辑器

> 文件：`chaos/_source/_engine/source/platform/public/chaos/platform/ime/ime.h`

```cpp
class PLATFORM_API IME
{
public:
    static void initialize();
    static void finalize();

    static void startIME(void* hWnd);
    static void tickIME(int x, int y, void* hWnd);
    static void endIME(void* hWnd);

    static bool getIMEString(void* context, void* lparam, int type, std::string& output);
    static bool getCompositionResult(void* handle, void* lparam, std::string& composition);
    static bool getCurrentComposition(void* handle, void* lparam, std::string& composition,
                                       unsigned& cursorPos, unsigned& targetStart, unsigned& targetEnd);
};
```

**接口语义**：
- `initialize` / `finalize`：IME 子系统全局初始化和清理。
- `startIME`：在指定窗口（`hWnd`）激活输入法上下文。
- `tickIME`：每帧更新 IME 候选框位置（`x, y`），使其跟随游戏内的输入框（如聊天框、角色命名框）。
- `endIME`：关闭指定窗口的输入法上下文。
- `getIMEString`：获取输入法最终提交的字符串（如用户按空格或回车确认后的结果）。
- `getCompositionResult` / `getCurrentComposition`：获取当前正在编辑的拼音/假名字符串，以及光标位置和候选高亮区间（`targetStart/targetEnd`）。

**设计背景**：IME 是亚洲语言（中文、日文、韩文）输入的关键。如果不提供统一的 IME 抽象，上层 UI 系统需要为 Windows（IMM/TSF）、macOS（InputMethodKit）、Android/iOS（原生 IME）分别写三套实现。

### 第 2 层：数据层

#### 2.1 Console 的数据边界

`Console` 是一个纯静态工具类，没有任何成员变量。它的状态完全托管给操作系统（剪贴板全局状态、光标全局位置、控制台窗口句柄）。这种设计保证了：
- **无状态**：没有全局对象需要初始化和销毁。
- **线程不安全由 OS 保证**：剪贴板操作在 Windows 上通过 `OpenClipboard` / `CloseClipboard` 序列化。

#### 2.2 IME 的数据流

IME 的状态流涉及三方：

```
操作系统 IME 服务
        │
        ▼
  platform::IME（统一接口）
        │
        ▼
  client/UI 系统（显示候选框、接收最终字符串）
```

IME 没有缓存大量内部状态，而是将平台原生事件（如 Windows 的 `WM_IME_COMPOSITION`、`WM_IME_CHAR`）中的 `lparam` 和 `handle` 直接透传给上层。这种"薄封装"策略减少了平台差异的泄漏。

### 第 3 层：逻辑层

#### 3.1 `clipCursor` 的执行流程

> 文件：`chaos/_source/_engine/source/platform/private/chaos/platform/console/windows/console.cpp`，第 91~100 行

```cpp
void Console::clipCursor(std::int32_t x, std::int32_t y, std::uint32_t width, std::uint32_t height)
{
    ::RECT rect;
    rect.left = x;
    rect.top = y;
    rect.right = x + width;
    rect.bottom = y + height;

    ::ClipCursor(&rect);
}
```

逻辑非常简单直接：构造一个 Win32 `RECT`，然后调用 `::ClipCursor(&rect)` 将系统鼠标光标限制在该矩形内。`cancelClipCursor()`（第 102~105 行）则传入 `NULL` 解除限制：

```cpp
void Console::cancelClipCursor()
{
    ::ClipCursor(NULL);
}
```

上层 `client/` 在窗口获得焦点时调用 `clipCursor`，在失去焦点或打开菜单时调用 `cancelClipCursor`，这是 FPS 类游戏防止鼠标移出窗口的标准做法。

#### 3.2 `tickIME` 的执行流程

> 文件：`chaos/_source/_engine/source/platform/private/chaos/platform/ime/windows/ime.cpp`，第 27~48 行

```cpp
void IME::tickIME(int x, int y, void* hWnd)
{
    auto context = ::ImmGetContext((HWND)hWnd);
    if (context)
    {
        COMPOSITIONFORM cf;
        cf.dwStyle = CFS_POINT;
        cf.ptCurrentPos.x = x;
        cf.ptCurrentPos.y = y;
        cf.rcArea = {};
        ::ImmSetCompositionWindow(context, &cf);

        CANDIDATEFORM form;
        form.dwIndex = 0;
        form.dwStyle = CFS_CANDIDATEPOS;
        form.ptCurrentPos = POINT {LONG(x), LONG(y)};
        form.rcArea = {};
        ::ImmSetCandidateWindow(context, &form);

        ::ImmReleaseContext((HWND)hWnd, context);
    }
}
```

这段代码展示了 chaos 对 Windows IMM（Input Method Manager）的薄封装：

1. **`ImmGetContext`**：获取与窗口句柄关联的输入法上下文（HIMC）。
2. **`ImmSetCompositionWindow`**：设置拼音/假名的组合窗口位置，使输入中的下划线文本跟随游戏 UI 的输入框。
3. **`ImmSetCandidateWindow`**：设置候选字窗口位置，使选字框也跟随光标。
4. **`ImmReleaseContext`**：释放上下文，避免资源泄漏。

每帧 UI 系统会根据输入框在屏幕上的位置更新 `x, y`，从而保证候选框跟随光标。

#### 3.3 `getIMEString` 的字符串转换逻辑

> 文件：`chaos/_source/_engine/source/platform/private/chaos/platform/ime/windows/ime.cpp`，第 59~76 行

```cpp
bool IME::getIMEString(void* context, void* lparam, int type, std::string& output)
{
    if (!((WPARAM)lparam & type))
        return false;

    const LONG stringSz = ::ImmGetCompositionStringW((HIMC)context, type, NULL, 0);
    if (stringSz <= 0)
        return false;

    std::wstring wstr;
    wstr.resize(stringSz / sizeof(wchar_t));
    ::ImmGetCompositionStringW((HIMC)context, type, &wstr[0], stringSz);

    int size_needed = WideCharToMultiByte(CP_UTF8, 0, &wstr[0], (int)wstr.size(), NULL, 0, NULL, NULL);
    output.resize(size_needed);
    WideCharToMultiByte(CP_UTF8, 0, &wstr[0], (int)wstr.size(), &output[0], size_needed, NULL, NULL);
    return true;
}
```

核心逻辑：
1. 先用 `ImmGetCompositionStringW(..., NULL, 0)` 探测所需宽字符缓冲区大小。
2. 分配 `std::wstring` 并读取实际宽字符内容。
3. 通过 `WideCharToMultiByte(CP_UTF8, ...)` 将宽字符转换为 UTF-8 `std::string`，与引擎内部字符串编码一致。

`getCompositionResult`（第 78~88 行）和 `getCurrentComposition`（第 90~123 行）均在此基础上封装，前者读取最终确认的字符串（`GCS_RESULTSTR`），后者读取正在编辑的字符串（`GCS_COMPSTR`）并解析候选高亮区间（`GCS_COMPATTR`）。

## 设计亮点

1. **最小接口原则**：`Console` 和 `IME` 只提供最必要的 OS 功能，不试图封装完整的窗口系统。
2. **宽字符支持**：剪贴板 API 使用 `std::wstring`，确保中文等多字节字符在剪贴板中不会乱码。
3. **IME  thin wrapper**：不缓存输入法状态，直接透传平台原生参数，减少了状态同步的 bug。
4. **平台隔离**：`Console` 和 `IME` 的公共头文件是平台无关的，实现细节隐藏在 `private/chaos/platform/console/<platform>/` 和 `ime/<platform>/` 下。

## 与上下层的关系

```
client/ UI 系统（聊天框、角色命名、设置菜单）
    │ 调用 Console::clipCursor 限制鼠标
    │ 调用 IME::tickIME 更新候选框位置
    ▼
platform/ Console + IME
    │ 调用 Win32 / macOS / Android 原生 API
    ▼
操作系统（Windows IMM/TSF、macOS InputMethodKit、Android InputMethodManager）
```

**引用热度**：
- `console.h` 被约 10 个文件引用
- `ime.h` 被约 15 个文件引用

这说明它们不是高频核心类，而是特定功能（剪贴板、输入法）的专门入口。
