# CLAUDE.md 与 MEMORY.md 的区别

很多用户会有疑问：既然官方文档说公用配置放在 `CLAUDE.md`，为什么 Claude 还经常生成配置到 `MEMORY.md`？这两者到底有什么区别？

## 核心区别

| 维度 | CLAUDE.md | MEMORY.md |
|------|-----------|-----------|
| **作用域** | 项目级（仓库/工作区） | 用户级（跨项目） |
| **存储位置** | 仓库根目录或工作区 | 用户目录 `~/.claude/memory/` |
| **持久性** | 随仓库提交，团队共享 | 用户个人使用，跨会话持久 |
| **内容性质** | 项目规范、代码风格、架构约定 | 个人偏好、习惯、会话记忆 |
| **主要读者** | Claude + 团队成员 | 仅 Claude |

## 分别放什么内容？

### CLAUDE.md —— 项目的"使用说明书"

放**与项目相关**的规范和要求：

- 项目架构和技术栈约定
- 代码风格规范（缩进、命名等）
- 目录结构和文件组织方式
- 跨平台兼容性要求
- 团队共享的开发规范

**示例**（来自本 vault）：
```markdown
## 跨平台兼容规范

此 vault 会在 Windows 和 macOS 多个环境下使用...

### 代码工作区
1. **统一使用 GCC（g++）**：...
2. **CMake 生成器**：统一使用 Ninja...
```

### MEMORY.md —— Claude 的"个人笔记本"

放**与你个人相关**的偏好和习惯：

- 你喜欢的代码风格偏好
- 常用的工具或工作流
- 你的角色和背景
- 跨项目通用的个人设置
- 你纠正过 Claude 的地方（避免重复犯错）

**示例**：
```markdown
- 用户喜欢用分号结尾，即使 JavaScript 可以省略
- 用户偏好使用单引号而非双引号
- 用户是前端工程师，熟悉 React 但不熟悉 Vue
```

## 为什么有时候会混淆？

### 1. 两者都影响 Claude 的行为

无论是 `CLAUDE.md` 还是 `MEMORY.md`，Claude 都会在每次对话开始时读取，所以都可能影响输出。

### 2. 边界有时不清晰

某些内容可能既像项目规范又像个人偏好，比如：

> "我喜欢用 TypeScript 的严格模式"

这句话既可以理解为项目要求（CLAUDE.md），也可以理解为个人偏好（MEMORY.md）。

### 3. 不同场景的推荐选择

| 场景 | 推荐位置 | 原因 |
|------|----------|------|
| "这个项目用 2 空格缩进" | CLAUDE.md | 是项目规范 |
| "我个人习惯 4 空格缩进" | MEMORY.md | 是个人偏好 |
| "所有 Python 代码都要类型注解" | CLAUDE.md | 是项目要求 |
| "记得我之前说过不喜欢 f-string" | MEMORY.md | 是个人偏好 |
| "用中文写注释" | 取决于... | 如果团队都用中文 → CLAUDE.md；只有你 → MEMORY.md |

## 最佳实践建议

### 对 CLAUDE.md

1. **提交到版本控制** —— 让团队成员共享相同规范
2. **保持简洁聚焦** —— 只放与当前项目相关的内容
3. **定期回顾更新** —— 随着项目演进调整规范

### 对 MEMORY.md

1. **利用自动记忆** —— Claude 会自动将一些关键信息存入 MEMORY.md
2. **手动纠正** —— 当 Claude 反复犯同样错误时，告诉它"记住..."
3. **定期清理** —— 删除过时或不再适用的记忆

## 总结

| 问自己 | 答案 |
|--------|------|
| 这个信息团队成员需要知道吗？ | 是 → **CLAUDE.md** |
| 这个信息只在其他项目也有用吗？ | 是 → **MEMORY.md** |
| 这是项目成功的必要条件吗？ | 是 → **CLAUDE.md** |
| 这只是我个人的舒适偏好吗？ | 是 → **MEMORY.md** |

> [!tip] 一句话记忆
> **CLAUDE.md = 项目的规则书**（放在仓库里）
> **MEMORY.md = Claude 的备忘录**（放在用户目录）
