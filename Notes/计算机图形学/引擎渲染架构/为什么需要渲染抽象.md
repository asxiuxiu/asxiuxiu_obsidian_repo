# 为什么需要渲染抽象

## 问题 0：为什么引擎代码不能到处都是 `gl*`？

从 [[Notes/计算机图形学/GPU编程基础/第一个三角形|第一个三角形]] 到 [[Notes/计算机图形学/帧缓冲与后处理/FBO常见陷阱|FBO常见陷阱]]，你已经写了大量直接调用 OpenGL 的代码。这些代码在学习阶段非常宝贵——它们让你看清 GPU 到底在做什么。

但当你的目标从“一个 Demo”升级为“一个引擎”时，问题开始变得不一样：

- 你的材质系统、后处理栈、UI 渲染、阴影映射……全部都在直接调用 `glBindFramebuffer`、`glUseProgram`、`glActiveTexture`
- 每个新功能都要重复处理状态绑定、viewport 对齐、纹理解绑
- 某一天你想在 macOS 上跑，发现 Apple 已经弃用 OpenGL；或者你想用 Vulkan 获得多线程能力——所有代码都要重写

**这不是“代码洁癖”，而是“架构生死线”。**

渲染抽象层的核心作用就是：把“我要画什么”（上层逻辑）和“我怎么用 OpenGL/Vulkan/D3D12 画”（底层实现）分开。上层只说“画这个 mesh 到这个 RenderTarget”，底层决定具体调哪个 API。

---

## 问题 1：直接 API 调用，出了什么问题？

### 最 naive 的方案：`#ifdef` 做平台分支

假设你已经有了一套 OpenGL 渲染代码，现在想支持 Vulkan。最直观的想法是：在每个 API 调用处加平台分支。

```cpp
void DrawMesh(const Mesh& mesh) {
#ifdef BACKEND_OPENGL
    glUseProgram(mesh.shader.glProgram);
    glBindVertexArray(mesh.vao);
    glDrawElements(GL_TRIANGLES, mesh.indexCount, GL_UNSIGNED_INT, 0);
#elif defined(BACKEND_VULKAN)
    vkCmdBindPipeline(cmdBuf, VK_PIPELINE_BIND_POINT_GRAPHICS, mesh.vkPipeline);
    vkCmdBindVertexBuffers(cmdBuf, 0, 1, &mesh.vkVbo, offsets);
    vkCmdDrawIndexed(cmdBuf, mesh.indexCount, 1, 0, 0, 0);
#endif
}
```

**立刻发现的问题**：

- **状态语义不一致**：OpenGL 的 `glUseProgram` 修改全局状态，影响后续所有绘制；Vulkan 的 `vkCmdBindPipeline` 只影响当前命令缓冲。同样的“绑定 Shader”操作，在不同 API 里副作用范围完全不同。
- **多线程能力不同**：Vulkan 可以多个线程并行录制命令缓冲；OpenGL 上下文只能绑定到一个线程。`#ifdef` 会让“哪些能并行”变成平台相关的隐性知识。
- **资源生命周期不同**：OpenGL 删除纹理 (`glDeleteTextures`) 由驱动异步处理；Vulkan 释放资源必须等待 GPU 完成引用。统一释放逻辑很难写。
- **维护爆炸**：每修一个 bug、每加一个功能，都要在 N 个分支上同步。三个月后就没人敢动这段代码。

**结论**：`#ifdef` 分支只是把混乱分散到更多地方，没有真正解决问题。

## 问题 2：RHI 抽象层到底解决什么？

**RHI（Render Hardware Interface，渲染硬件接口）** 是引擎自己定义的一层 API，夹在“上层渲染代码”和“底层图形 API”之间。

它的核心只做一件事：

> **让上层代码只说“做什么”，不说“怎么做”。**

```cpp
// 上层代码：完全不知道底层是 OpenGL 还是 Vulkan
commandList->SetPipeline(mesh.material->pipeline);
commandList->SetVertexBuffer(mesh.vertexBuffer);
commandList->SetIndexBuffer(mesh.indexBuffer);
commandList->DrawIndexed(mesh.indexCount);
```

底层后端各自实现这套接口：

```cpp
// OpenGL 后端
void GLCommandList::SetPipeline(PipelineHandle pso) {
    GLuint program = pipelineTable.Resolve(pso)->program;
    glUseProgram(program);
}

void GLCommandList::DrawIndexed(uint32_t count) {
    glDrawElements(GL_TRIANGLES, count, GL_UNSIGNED_INT, nullptr);
}

// Vulkan 后端
void VKCommandList::SetPipeline(PipelineHandle pso) {
    VkPipeline pipeline = pipelineTable.Resolve(pso)->pipeline;
    vkCmdBindPipeline(cmdBuf, VK_PIPELINE_BIND_POINT_GRAPHICS, pipeline);
}

void VKCommandList::DrawIndexed(uint32_t count) {
    vkCmdDrawIndexed(cmdBuf, count, 1, 0, 0, 0);
}
```

**状态变化图**：

```
没有 RHI：
  上层渲染代码 ──直接调用──► OpenGL / Vulkan / D3D12
  （每个绘制函数都要知道自己在哪个平台上）

有了 RHI：
  上层渲染代码 ──调用──► ICommandList
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
          OpenGL后端      Vulkan后端       D3D12后端
              │               │               │
              ▼               ▼               ▼
           glDraw*        vkCmdDraw*      DrawIndexed*
```

**RHI 存在的三个根本原因**：

1. **跨后端**：一套上层代码跑在多个图形 API 上
2. **跨平台**：Windows/Linux/macOS/移动端的不同 API 选择被隔离在底层
3. **架构解耦**：材质系统、后处理栈、渲染管线不再关心 OpenGL 状态机细节

## 问题 3：RHI 应该多“厚”？

抽象层不是越厚越好。厚度和控制力是一对 trade-off。

### 分支 A：极薄抽象——1:1 映射 API

```cpp
class ICommandList {
public:
    virtual void SetViewport(float x, float y, float w, float h) = 0;
    virtual void DrawIndexed(uint32_t count, uint32_t start, int32_t baseVertex) = 0;
};
```

**优点**：实现简单，性能开销最小。

**缺点**：上层仍然要理解“命令列表”“Barrier”“Descriptor Set”等概念；不同 API 的行为差异仍会泄漏；无法为多线程预留统一策略。

### 分支 B：中等厚度——引擎自己的命令抽象

```cpp
enum class RenderCmdType {
    SetViewport,
    SetScissor,
    BindPipeline,
    BindVertexBuffer,
    BindIndexBuffer,
    DrawIndexed,
    ResourceBarrier,
};

class ICommandList {
public:
    virtual void SetViewport(float x, float y, float w, float h) = 0;
    virtual void BindPipeline(PipelineHandle pso) = 0;
    virtual void DrawIndexed(uint32_t count) = 0;
    // 内部把命令序列化到缓冲区，帧末统一翻译执行
};
```

**优点**：上层完全不感知底层 API 差异；天然支持延迟命令缓冲、多线程录制、资源延迟删除。

**缺点**：多一层翻译开销；需要引擎自己管理命令内存。

### 分支 C：借用第三方抽象——bgfx / Sokol / wgpu

这些库本身就是“别人写好的 RHI”。

| 库 | 后端覆盖 | 特点 |
|----|---------|------|
| **bgfx** | Vulkan, Metal, D3D12, D3D11, GL | 成熟稳定，游戏引擎广泛使用 |
| **Sokol GFX** | Vulkan, Metal, D3D11, GL, WebGPU | 极简 STB 风格，单头文件 |
| **wgpu-native** | Vulkan, Metal, D3D12, GL | 现代 WebGPU 语义，安全验证严格 |

**优点**：快速获得多后端能力，不重复造轮子。

**缺点**：API 形状锁定你的引擎设计；调试链路变长；升级大版本可能大量适配；ECS 映射不自然。

### 个人项目推荐

> **学习阶段**：先用原生 OpenGL 写到底，建立直觉。
>
> **引擎原型阶段**：自研中等厚度 RHI，接口对齐现代 API（Vulkan/D3D12）的命令语义，但初期用 OpenGL 后端实现。这样未来换后端时上层不动。
>
> **快速出产品阶段**：如果人力有限，考虑 bgfx 或 Sokol 作为后端，但引擎与第三方库之间仍保留一层薄包装。

## 问题 4：第一个后端该选谁？

这是阶段八第一个必须回答的工程问题。选错了会让 [[Notes/SelfGameEngine/0_RoadMap#阶段 5：渲染管线与画面|SelfGameEngine 阶段 5]] 的验收目标延迟数月。

| 后端 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| **OpenGL** | 初始化简单、资料多、学习曲线平缓 | 全局状态机、多线程弱、macOS 已弃用 | 阶段 1~3 学习、快速验证 |
| **Vulkan** | 现代设计、多线程原生、性能天花板高 | boilerplate 极大、调试地狱 | 深入学习 GPU、长期引擎 |
| **D3D12** | Windows/Xbox 性能最优、工具链成熟 | 仅限 Windows/Xbox | 以 Windows 为主的项目 |
| **WebGPU/wgpu** | 现代语义、跨平台、自动处理 Barrier | 外部依赖大、性能略低于原生 | 需要 Web/多端部署 |

**关键认知**：

- 第一个后端不是为了“最强性能”，而是为了“最快验证 RHI 接口设计是否正确”
- 如果你的目标是从 OpenGL 迁移到现代 API，**先用 OpenGL 实现 RHI 后端**，把上层代码全部迁移到 RHI 接口上；等上层稳定后，再实现 Vulkan/D3D12 后端
- 这样 OpenGL 代码不会白写，反而成为 RHI 接口的“第一个测试床”

## 问题 5：从直接 OpenGL 到最小 RHI 的最短路径

不要试图一次性设计完美 RHI。最小迁移路径只需要三步：

### 第一步：把资源包装成 Handle

不要在上层存 `GLuint`、`VkBuffer`、`ID3D12Resource*`。定义引擎自己的资源句柄：

```cpp
struct BufferHandle { uint32_t index; };
struct TextureHandle { uint32_t index; };
struct PipelineHandle { uint32_t index; };
struct RenderTargetHandle { uint32_t index; };
```

上层 Component 只存 Handle，后端内部维护 Handle → 实际资源的映射表。

### 第二步：把绘制命令抽象成接口

```cpp
class ICommandList {
public:
    virtual ~ICommandList() = default;

    // 状态设置
    virtual void SetViewport(int x, int y, int w, int h) = 0;
    virtual void SetScissor(int x, int y, int w, int h) = 0;

    // 资源绑定
    virtual void BindPipeline(PipelineHandle pso) = 0;
    virtual void BindVertexBuffer(BufferHandle vb) = 0;
    virtual void BindIndexBuffer(BufferHandle ib) = 0;
    virtual void BindTexture(uint32_t slot, TextureHandle tex) = 0;

    // 绘制
    virtual void DrawIndexed(uint32_t indexCount, uint32_t startIndex = 0) = 0;
    virtual void Draw(uint32_t vertexCount) = 0;

    // RenderTarget
    virtual void BeginRenderTarget(RenderTargetHandle rt, const ClearState& clear) = 0;
    virtual void EndRenderTarget() = 0;
};
```

### 第三步：用 OpenGL 实现第一版后端

```cpp
class GLCommandList : public ICommandList {
    GLuint currentFBO = 0;
    int currentWidth = 0, currentHeight = 0;

public:
    void SetViewport(int x, int y, int w, int h) override {
        glViewport(x, y, w, h);
    }

    void BindPipeline(PipelineHandle pso) override {
        auto* pipeline = g_pipelineTable.Resolve(pso);
        glUseProgram(pipeline->program);
        glBlendFunc(pipeline->srcBlend, pipeline->dstBlend);
        if (pipeline->depthTest) glEnable(GL_DEPTH_TEST);
        else glDisable(GL_DEPTH_TEST);
    }

    void BindVertexBuffer(BufferHandle vb) override {
        auto* buffer = g_bufferTable.Resolve(vb);
        glBindVertexArray(buffer->vao);  // VAO 内部已记录 VBO 绑定
    }

    void BindIndexBuffer(BufferHandle ib) override {
        auto* buffer = g_bufferTable.Resolve(ib);
        glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, buffer->ibo);
    }

    void BindTexture(uint32_t slot, TextureHandle tex) override {
        auto* texture = g_textureTable.Resolve(tex);
        glActiveTexture(GL_TEXTURE0 + slot);
        glBindTexture(GL_TEXTURE_2D, texture->id);
    }

    void DrawIndexed(uint32_t indexCount, uint32_t startIndex) override {
        glDrawElements(GL_TRIANGLES, indexCount, GL_UNSIGNED_INT,
                       reinterpret_cast<void*>(startIndex * sizeof(uint32_t)));
    }

    void BeginRenderTarget(RenderTargetHandle rt, const ClearState& clear) override {
        auto* target = g_renderTargetTable.Resolve(rt);
        currentFBO = target->fbo;
        currentWidth = target->width;
        currentHeight = target->height;

        glBindFramebuffer(GL_FRAMEBUFFER, currentFBO);
        glViewport(0, 0, currentWidth, currentHeight);

        GLbitfield clearBits = 0;
        if (clear.color) {
            clearBits |= GL_COLOR_BUFFER_BIT;
            glClearColor(clear.colorValue.r, clear.colorValue.g,
                         clear.colorValue.b, clear.colorValue.a);
        }
        if (clear.depth) clearBits |= GL_DEPTH_BUFFER_BIT;
        if (clearBits) glClear(clearBits);
    }

    void EndRenderTarget() override {
        glBindFramebuffer(GL_FRAMEBUFFER, 0);
        // viewport 会在下一次 BeginRenderTarget / Present 时重新设置
    }
};
```

> **注意**：这个最小 RHI 并没有解决 OpenGL 的所有状态机问题（比如纹理单元残留、Shader 残留），但它先把“上层代码从 gl* 中解放出来”。状态污染问题会在 [[Notes/计算机图形学/引擎渲染架构/渲染状态管理|渲染状态管理]] 中系统解决。

## 问题 6：RHI 抽象后，上层代码长什么样？

以“后处理栈”为例，迁移到 RHI 后：

```cpp
void PostProcessStack::Render(ICommandList* cmd, TextureHandle sceneColor,
                              RenderTargetHandle ping, RenderTargetHandle pong,
                              RenderTargetHandle output) {
    TextureHandle currentInput = sceneColor;

    for (size_t i = 0; i < effects.size(); i++) {
        bool isLast = (i == effects.size() - 1);

        // 绑定输出目标：中间 PingPong 或最终屏幕
        if (isLast) {
            cmd->BeginRenderTarget(output, ClearState::Color({0,0,0,1}));
        } else {
            cmd->BeginRenderTarget((i % 2 == 0) ? ping : pong,
                                   ClearState::Color({0,0,0,1}));
        }

        // 绑定输入纹理和 Shader
        cmd->BindPipeline(effects[i].pipeline);
        cmd->BindTexture(0, currentInput);
        cmd->DrawIndexed(6);  // 全屏四边形

        cmd->EndRenderTarget();

        // 更新下一步输入
        if (!isLast) {
            currentInput = (i % 2 == 0)
                ? g_renderTargetTable.GetColorTexture(ping)
                : g_renderTargetTable.GetColorTexture(pong);
        }
    }
}
```

这段代码里没有 `glBindFramebuffer`，没有 `glActiveTexture`，没有 `glViewport`。它只表达“我对这个纹理施加这个效果，输出到那个 RenderTarget”。

**这就是 RHI 带来的第一层自由**：后处理栈作者不需要是 OpenGL 专家，也能正确组织多 Pass 管线。

| 阶段 | 推荐做法 |
