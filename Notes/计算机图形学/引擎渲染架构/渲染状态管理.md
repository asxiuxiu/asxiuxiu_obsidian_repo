# 渲染状态管理

## 问题 0：合批之后，为什么画面还是错 / 还是慢？

在 [[Notes/计算机图形学/引擎渲染架构/DrawCall与合批|DrawCall与合批]] 里，我们已经会把 500 个物体按 `PSO + 材质` 排序，再用 `Instancing` 合并成几十个批次。按道理 CPU 开销应该下来了。

但如果你先画透明玻璃、再画不透明墙，或者透明物体忘了关深度写入，又或者 Alpha Test 的树叶和不透明石头混在一起画，你会发现：

- 玻璃后面的物体不见了；
- 树叶边缘出现奇怪的方块；
- 远处的树叶把近处的墙“盖”住了；
- 帧率虽然 Draw Call 少了，但 GPU 还是做了很多无效像素计算。

**问题不在 Draw Call 数量，而在状态切换的顺序和正确性。**

渲染状态就是告诉 GPU「这个像素怎么算、怎么写、怎么混合」的那一组开关和参数。管理不好，合批省下来的 CPU 时间会被 GPU 的无效工作吃掉，甚至还会得到错误的画面。

---

## 问题 1：渲染状态到底是什么？

如果你从 [[Notes/计算机图形学/GPU编程基础/OpenGL状态机模型|OpenGL状态机模型]] 过来，你已经知道 OpenGL 是一个巨大的全局状态机。决定一次绘制长什么样子的，不止 `glUseProgram`，还有一堆开关：

```cpp
glEnable(GL_DEPTH_TEST);
glDepthFunc(GL_LEQUAL);
glDepthMask(GL_TRUE);

glEnable(GL_CULL_FACE);
glCullFace(GL_BACK);
glFrontFace(GL_CCW);

glEnable(GL_BLEND);
glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
```

这些就是**渲染状态**。它们大致可以分为几类：

1. **光栅化状态**：剔除模式、正面朝向、线框/填充模式、多边形偏移。
2. **深度/模板状态**：深度测试开不开、比较函数、深度写入开不开、模板测试配置。
3. **混合状态**：混合开关、源/目标混合因子、混合方程。
4. **管线状态（PSO）**：Shader、顶点布局、渲染目标格式等现代 API 中打包成对象的状态。
5. **动态状态**：视口、裁剪矩形、线宽、混合因子、模板引用值等可以在命令列表中动态修改的参数。

在传统 OpenGL 里，这些状态散落在全局上下文中，随时可能被任何一段代码改掉。在 D3D12 / Vulkan / Metal 里，大部分状态被预先打包进 **PSO（Pipeline State Object）**，切换时只需要绑定一个对象。

但不管你是用 OpenGL 还是现代 API，**状态都需要被显式管理**，否则就会陷入「这里开了混合，忘了关，下一个 Pass 的画面全花」的泥潭。

## 问题 2：为什么全局状态机会成为痛点？

最 naive 的状态管理就是「用到什么设什么，设完不管」：

```cpp
void RenderOpaqueObject(Object* obj) {
    glUseProgram(obj->shader);
    glEnable(GL_DEPTH_TEST);
    glDepthFunc(GL_LEQUAL);
    glDepthMask(GL_TRUE);
    glDisable(GL_BLEND);
    // ... 绑定资源，绘制
}

void RenderTransparentObject(Object* obj) {
    glUseProgram(obj->shader);
    glEnable(GL_BLEND);
    glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
    // 糟糕，忘了 glDepthMask(GL_FALSE)
    // ... 绑定资源，绘制
}
```

这段代码的问题非常明显：

- **状态泄漏**：透明物体的绘制依赖上一个物体留下的深度测试状态。如果上一个物体把深度测试关了，玻璃就会画到所有东西前面。
- **重复设置**：100 个不透明物体都调用 `glEnable(GL_DEPTH_TEST)`，驱动和 GPU 都要处理这些冗余调用。
- **责任分散**：每个 `RenderXxx` 函数都要记得把状态设对，少一行就会出 bug。

这正是为什么引擎需要把渲染状态从「散落在各处的全局调用」提升为**显式的状态对象**。

---

## 问题 3：状态对象——把全局状态打包

改进方案：把一组相关的状态打包成不可变对象，绘制时只绑定对象句柄。

```cpp
struct DepthStencilState {
    bool depthTestEnabled;
    bool depthWriteEnabled;
    GLenum depthFunc;     // GL_LESS, GL_LEQUAL, GL_EQUAL, etc.
};

struct BlendState {
    bool blendEnabled;
    GLenum srcFactor;
    GLenum dstFactor;
};

struct RasterizerState {
    bool cullEnabled;
    GLenum cullFace;      // GL_FRONT, GL_BACK
    GLenum frontFace;     // GL_CCW, GL_CW
};
```

引擎创建这些状态对象时，就把它们存入缓存：

```cpp
using DepthStencilHandle = Handle<DepthStencilState>;
using BlendHandle        = Handle<BlendState>;
using RasterizerHandle   = Handle<RasterizerState>;

DepthStencilHandle CreateDepthStencilState(const DepthStencilState& desc);
BlendHandle        CreateBlendState(const BlendState& desc);
RasterizerHandle   CreateRasterizerState(const RasterizerState& desc);
```

绘制时不再直接调 `glEnable`，而是：

```cpp
cmd->SetDepthStencilState(opaqueDepthState);
cmd->SetBlendState(opaqueBlendState);
cmd->SetRasterizerState(defaultRasterState);
cmd->SetPipeline(obj->pipeline);
cmd->DrawIndexed(...);
```

**关键认知**：状态对象本身不减少状态切换次数，但它把「如何设置状态」从业务代码中隔离出来。下一步，我们可以在 RHI 层做**状态缓存**，避免重复调用 API。

## 问题 4：状态缓存——不要重复设置相同状态

即使我们已经按 PSO/材质排序，相邻 Draw Call 的状态仍然可能相同。如果每个 Draw Call 都发一次 `glEnable(GL_DEPTH_TEST)`，驱动和硬件都要处理这些冗余命令。

解决办法：在 RHI 命令列表里维护一个**当前状态快照**，只有真正改变时才调用底层 API。

```cpp
class GLCommandList : public ICommandList {
public:
    void SetPipeline(PipelineHandle pso) override {
        if (pso == currentPSO) return;
        glUseProgram(pso.program);
        currentPSO = pso;
    }

    void SetDepthStencilState(DepthStencilHandle h) override {
        if (h == currentDS) return;
        const auto& s = stateCache->Get(h);
        if (s.depthTestEnabled) glEnable(GL_DEPTH_TEST);
        else                    glDisable(GL_DEPTH_TEST);
        glDepthFunc(s.depthFunc);
        glDepthMask(s.depthWriteEnabled ? GL_TRUE : GL_FALSE);
        currentDS = h;
    }

    void SetBlendState(BlendHandle h) override {
        if (h == currentBlend) return;
        const auto& s = stateCache->Get(h);
        if (s.blendEnabled) {
            glEnable(GL_BLEND);
            glBlendFunc(s.srcFactor, s.dstFactor);
        } else {
            glDisable(GL_BLEND);
        }
        currentBlend = h;
    }

private:
    PipelineHandle        currentPSO;
    DepthStencilHandle    currentDS;
    BlendHandle           currentBlend;
    RasterizerHandle      currentRaster;
};
```

这个缓存层解决的是**冗余 API 调用**问题。它的性能收益在 OpenGL 这种“薄驱动”上尤其明显；在 D3D12 / Vulkan 上，PSO 切换本身比较重，缓存同样能减少不必要的 `vkCmdBindPipeline`。

> **注意**：状态缓存只对“连续多个 Draw Call 状态相同”有效。这也是为什么排序如此重要——排序让相同状态的物体聚集在一起。

## 问题 5：Phase 拆分——不同类别物体的状态规则不同

仅仅缓存不够。不同类型的物体有完全不同的状态规则，不能按同一套键排序。

典型拆分：

| Phase | 深度写入 | 深度测试 | 混合 | 排序方向 | 用途 |
|-------|----------|----------|------|----------|------|
| **Opaque** | 开 | 开 | 关 | 前→后 | 石头、墙、不透明角色 |
| **Alpha Test / Masked** | 开 | 开 | 关 | 前→后 | 树叶、铁丝网、镂空贴图 |
| **Transparent** | 关 | 开 | 开 | 远→近 | 玻璃、水面、粒子 |
| **Overlay / UI** | 关 | 关 | 开 | 按 UI 层级 | 血条、准星、UI 面板 |

为什么这样分？

- **Opaque**：深度写入开启，可以从近到远画。远处的像素会被 Early-Z 快速剔除，节省像素着色。
- **Alpha Test**：像素在 Shader 里用 `discard` / `clip()` 镂空。虽然它写了深度，但 `discard` 会打断某些 GPU 上的 Early-Z 优化，所以通常单独作为一个 Phase，在 Opaque 之后、Transparent 之前绘制。
- **Transparent**：必须按从远到近的顺序画，混合结果才对。同时关闭深度写入，否则后面的透明物体被前面的透明像素挡住。
- **Overlay**： often 直接覆盖在画面最上层，不需要深度测试。

这个 Phase 顺序本质上就是一套**状态模板**。每个 Phase 在进入时设置好默认状态，后续 Draw Call 只需要在此基础上做微调。

## 问题 6：Early-Z 到底是什么？为什么排序方向重要？

在 [[Notes/计算机图形学/软光栅化与3D数学/3D光栅化与深度缓冲|3D光栅化与深度缓冲]] 里，你已经知道逻辑上深度测试发生在片段着色器之后。但 GPU 为了性能，会在片段着色器**之前**做一次深度测试，这叫 **Early-Z**。

Early-Z 的工作条件：

- 当前绘制的是不透明物体；
- 片段着色器不修改深度（不写 `gl_FragDepth` / `SV_Depth`）；
- 片段着色器没有 `discard` / `clip()`；
- 当前状态允许深度测试提前进行。

如果条件满足，GPU 可以在运行片段着色器之前就丢弃被遮挡的像素。这意味着：

> **先画近的，再画远的，远处像素的片段着色器就可以被 Early-Z 跳过。**

所以 Opaque Phase 按**前→后**排序，是为了让 Early-Z 最大化剔除效率。

反过来，Transparent Phase 不能写深度，否则远处的透明物体被近处的透明像素挡住，混合顺序就错了。所以透明物体必须按**远→近**排序，并且通常关闭深度写入。

Alpha Test / Masked 的情况比较微妙：它们写深度，也有利于 Early-Z，但 `discard` 会让 GPU 在某些架构上退回到保守模式。所以多数引擎把它们单独分一个 Phase，放在 Opaque 之后、Transparent 之前。

## 问题 7：排序键里怎么放状态？

在 [[Notes/计算机图形学/引擎渲染架构/DrawCall与合批|DrawCall与合批]] 里，排序键大概是：

```cpp
key |= (uint64_t)phase       << 56;
key |= (uint64_t)psoId       << 40;
key |= (uint64_t)materialId  << 24;
key |= depthBits;
```

现在我们要把「状态」也考虑进去。因为 PSO 通常已经包含了 Depth/Blend/Rasterizer 的大部分配置，所以排序键的高位可以优先用 `phase` 和 `psoId`。

但如果你的 RHI 把深度/混合状态独立于 PSO（比如 OpenGL 风格），那就需要额外一位来确保相同状态的物体连续：

```cpp
uint64_t ComputeSortKey(RenderPhase phase,
                        uint32_t    psoId,
                        uint32_t    materialId,
                        uint32_t    stateKey,   // 深度/混合/光栅化状态的合并哈希
                        float       viewDepth) {
    uint64_t key = 0;
    key |= (uint64_t)phase       << 56;
    key |= (uint64_t)stateKey    << 48;  // 保证相同状态连续
    key |= (uint64_t)psoId       << 32;
    key |= (uint64_t)materialId  << 16;

    if (phase == RenderPhase::Transparent) {
        key |= (0xFFFFu - DepthToUint16(viewDepth));
    } else {
        key |= DepthToUint16(viewDepth);
    }
    return key;
}
```

**为什么 stateKey 放在 psoId 前面？**

因为状态切换的 CPU/GPU 开销通常比绑定材质更大。让相同状态的 Draw Call 聚集，可以让状态缓存命中率最高。

> 实际位域分配要根据你的状态数量、材质数量、场景规模调整。没有 universally optimal 的排序键，但原则是：**切换越贵的状态，位越高。**

## 问题 8：Pass 边界上的状态泄漏怎么办？

状态缓存解决的是 Pass 内部的问题。但多个 Pass 之间，上一个 Pass 的状态可能污染下一个 Pass。

比如后处理 Pass 画全屏四边形时，如果上一个场景 Pass 开了深度测试和深度写入，后处理四边形可能会被深度测试干掉；或者场景 Pass 开了混合，后处理颜色就被混合错了。

最 robust 的做法：每个 Pass 开始时**显式重置到自己需要的状态**，而不是依赖上一个 Pass 的残留。

```cpp
void PostProcessPass::Execute(ICommandList* cmd) {
    cmd->SetRenderTarget(output);
    cmd->SetDepthStencilState(noDepthTest);   // 明确关闭
    cmd->SetBlendState(noBlend);              // 明确关闭
    cmd->SetRasterizerState(defaultCullBack);
    cmd->SetPipeline(blurPipeline);
    cmd->DrawFullscreenTriangle();
}
```

在更工业化的实现里，Pass 会声明自己需要的状态，RenderGraph 或 RenderPass 框架在进入 Pass 时自动应用默认状态。这种“声明式状态”避免了人工漏设。

## 问题 9：现代 API 下的 PSO 与状态缓存

在 D3D12 / Vulkan / Metal 里，大部分渲染状态被预先烘焙进 PSO。运行时切换 PSO 的代价远高于 OpenGL 切换单个状态。因此状态缓存的核心对象从「深度/混合状态」变成了 **PSO 本身**。

```cpp
class VKCommandList : public ICommandList {
public:
    void SetPipeline(PipelineHandle pso) override {
        if (pso == currentPSO) return;
        vkCmdBindPipeline(cmd, VK_PIPELINE_BIND_POINT_GRAPHICS, pso.vkPipeline);
        currentPSO = pso;
    }

    void SetViewport(float x, float y, float w, float h) override {
        // 动态状态可以每帧改
        VkViewport vp = { x, y, w, h, 0.0f, 1.0f };
        vkCmdSetViewport(cmd, 0, 1, &vp);
    }

private:
    PipelineHandle currentPSO;
};
```

现代 API 还区分：

- **静态状态**：必须写入 PSO，运行时改不了（比如 Shader、顶点布局、大部分光栅化状态）。
- **动态状态**：可以在命令列表中动态设置（比如 viewport、scissor、blend constants、stencil reference）。

你的引擎 RHI 需要明确区分这两类状态。如果上层代码试图“动态修改”一个静态状态，RHI 应该报错或重新创建 PSO。

SelfGameEngine 的 [[Notes/SelfGameEngine/渲染管线与画面/PSO缓存与异步编译|PSO缓存与异步编译]] 专门讨论了如何用哈希缓存 PSO、异步编译 fallback、以及磁盘 Pipeline Library。这里只需要记住：**在现代 API 下，状态管理 ≈ PSO 管理**。

## 问题 10：最小可运行实现

下面是一个把「状态对象 + 状态缓存 + Phase 排序」串起来的最小实现。

```cpp
enum class RenderPhase : uint8_t {
    Opaque     = 0,
    AlphaTest  = 1,
    Transparent= 2,
    Overlay    = 3,
};

struct RenderStateDesc {
    bool depthTestEnabled  = true;
    bool depthWriteEnabled = true;
    DepthFunc depthFunc    = DepthFunc::LessEqual;
    bool blendEnabled      = false;
    BlendFactor srcBlend   = BlendFactor::One;
    BlendFactor dstBlend   = BlendFactor::Zero;
    bool cullEnabled       = true;
    CullMode cullMode      = CullMode::Back;
};

struct DrawItem {
    uint64_t sortKey;
    RenderPhase phase;
    PipelineHandle pipeline;
    RenderStateDesc state;
    MaterialHandle material;
    MeshHandle mesh;
    Mat4 worldMatrix;
    float viewDepth;
};

uint64_t ComputeSortKey(RenderPhase phase, const RenderStateDesc& state,
                        PipelineHandle pso, MaterialHandle mat, float depth) {
    uint64_t key = 0;
    key |= (uint64_t)phase            << 56;
    key |= (uint64_t)HashState(state) << 40;  // 16 bit state hash
    key |= (uint64_t)pso.index        << 24;
    key |= (uint64_t)mat.index        << 8;

    if (phase == RenderPhase::Transparent) {
        key |= (0xFFu - DepthToUint8(depth));
    } else {
        key |= DepthToUint8(depth);
    }
    return key;
}

class RenderQueue {
    Array<DrawItem> items;

public:
    void Add(const DrawItem& item) { items.push_back(item); }

    void Flush(ICommandList* cmd) {
        std::sort(items.begin(), items.end(),
                  [](const DrawItem& a, const DrawItem& b) {
                      return a.sortKey < b.sortKey;
                  });

        for (const auto& item : items) {
            cmd->SetPipeline(item.pipeline);
            cmd->SetRenderState(item.state);   // RHI 内部做缓存
            cmd->SetMaterial(item.material);
            cmd->SetVertexBuffer(item.mesh.vertexBuffer);
            cmd->SetIndexBuffer(item.mesh.indexBuffer);
            cmd->SetConstantBuffer(0, item.worldMatrix);
            cmd->DrawIndexed(item.mesh.indexCount);
        }
    }
};
```

在这个最小实现里：

- `RenderStateDesc` 是引擎层的状态描述；
- `cmd->SetRenderState()` 内部做状态缓存，避免重复调用 API；
- 排序键保证 Phase、状态、PSO、材质的连续性。

> 注意：工业级实现不会每帧都对全局做 `std::sort`，而是会分 Phase、分 bin，甚至用 GPU-Driven。但最小实现足以表达核心思想。

| **RenderGraph 自动状态推导** | 多 Pass 之间的 Barrier、RenderTarget 切换、状态重置由框架自动管理 |
| **动态状态追踪** | Viewport、Scissor、Blend Factor 等动态参数的缓存 |
| **Validation Layer** | 在 Debug 模式下检测状态冲突，例如 Opaque Phase 意外开启混合 |
| **GPU-Driven Rendering** | 状态排序和剔除进一步 offload 到 GPU，CPU 只发一次 Multi-Draw Indirect |
