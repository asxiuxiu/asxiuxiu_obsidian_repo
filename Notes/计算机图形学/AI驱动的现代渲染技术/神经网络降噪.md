# 神经网络降噪

## 问题 0：路径追踪很真实，但为什么这么“噪”？

在 [[Notes/计算机图形学/现代渲染技术/PBR基础|PBR基础]] 里我们学过，PBR 的光照计算可以用微表面模型近似直接光。但真实世界的光照远比这复杂：光线会在墙壁间多次反射、穿过玻璃、在粗糙金属表面散射。

**路径追踪（Path Tracing）** 是解决这些复杂光照的“黄金标准”算法。它的基本思想是：从相机出发，沿每条像素光线反向追踪光线路径，随机采样光线在场景中的反射/折射方向，最后把贡献累加起来。

```
相机 ──► 像素光线 ──► 击中墙面 ──► 随机反射 ──► 击中光源
                              │
                              └── 另一条路径 ──► 击中玻璃 ──► 折射 ──► 光源
```

**核心问题**：每条光线的路径是随机采样的。如果只采 1 条路径（1 sample per pixel，1 spp），每个像素的颜色就像掷骰子——可能偏高、可能偏低，画面充满噪点。

```
1 spp：像电视雪花
4 spp：好一点，但暗处还是斑斑点
64 spp：基本干净，但一帧要几秒
4096 spp：电影级干净，但一帧要几小时
```

实时渲染要 16ms 一帧，不可能采 4096 spp。**想要在低采样下获得干净图像，必须降噪。**

## 问题 1：传统滤波器降噪有什么问题？

最 naive 的降噪是**高斯模糊**：对每个像素，取周围 N×N 像素的加权平均。

```glsl
vec3 denoised = vec3(0.0);
for (int x = -2; x <= 2; x++)
    for (int y = -2; y <= 2; y++)
        denoised += texture(noisyImage, uv + offset(x, y)).rgb * gaussianWeight(x, y);
```

**立刻发现的问题**：

- 噪点是被平滑了，但边缘也被抹平了。
- 细小纹理（毛发、金属划痕）会丢失。
- 光影交界处出现“糊糊”的感觉。
- 暗处噪点和亮处噪点需要不同的模糊强度，手工调参很困难。

**根本原因**：高斯模糊不区分“噪声”和“真实细节”。它把高频信息一刀切地全部抹掉。

更聪明的**双边滤波（Bilateral Filter）** 会参考颜色相似度：只在颜色相近的像素间平均，这样能保留边缘。但它仍然不能理解场景的几何和材质，遇到复杂反射、半透明、运动物体时容易失效。

## 问题 2：神经网络降噪凭什么比双边滤波强？

神经网络降噪的核心 insight 是：

> **噪声有统计规律，真实图像细节也有统计规律。神经网络可以从大量训练数据中学到“什么是噪点、什么是纹理”。**

传统滤波器是人工设计的固定规则；神经网络是数据驱动的自适应规则。

### 输入不只是噪声图

神经网络降噪器通常输入多张 buffer：

```
主输入：Noisy Radiance（带噪的 HDR 颜色）
辅助输入：
  - Albedo（基础颜色）—— 告诉网络纹理细节在哪里
  - Normal（法线）—— 告诉网络几何边缘在哪里
  - Depth（深度）—— 告诉网络物体远近、遮挡关系
  - Roughness/Metallic（材质参数）—— 告诉网络反射特性
  - Motion Vectors（运动向量）—— 告诉网络像素移动方向
```

**为什么辅助 buffer 这么重要？**

假设噪声图里有一片亮斑。神经网络单独看颜色，无法判断这是“噪点”还是“高光”。但如果 Normal 显示这里是平面、Albedo 显示这里是灰色水泥、Roughness 显示这里很粗糙，网络就知道“这块不该有尖锐高光”，从而把它当作噪声抹掉。

> **关键认知**：神经网络降噪不是“更聪明的模糊”，而是“利用场景语义信息，区分噪声和真实信号”。

## 问题 3：降噪网络长什么样？U-Net 与 Transformer

### U-Net：图像降噪的经典骨架

U-Net 是一种编码器-解码器结构，中间用跳连（skip connection）把低层细节传到高层。

```
输入（Noisy + Albedo + Normal + Depth）
    │
    ▼
┌─────────────┐
│  下采样层   │  提取越来越抽象的特征
│  Down 1     │
└──────┬──────┘
       │ skip connection
┌──────▼──────┐
│  Down 2     │
└──────┬──────┘
       │ skip
┌──────▼──────┐
│  Bottleneck │  最抽象的特征表示
└──────┬──────┘
       │
┌──────▼──────┐
│  Up 2       │  上采样，恢复空间分辨率
└──────┬──────┘
       │ 与 skip 合并
┌──────▼──────┐
│  Up 1       │
└──────┬──────┘
       │
    输出（Denoised Radiance）
```

**为什么 U-Net 适合降噪**：

- 下采样扩大感受野：网络能看到一大片区域，判断哪些是噪点。
- 跳连保留细节：上采样时把原始分辨率的信息补回来，避免过度模糊。
- 编码器-解码器结构天然适合“输入输出同分辨率”的图像到图像任务。

### 两种输出形式

**像素预测（Pixel Prediction）**：网络直接输出每个像素去噪后的颜色。

```
输出 = DenoisedColor
```

**核预测（Kernel Prediction）**：网络输出每个像素周围一个局部滤波核的权重，然后用这个核对输入图像做加权平均。

```
输出 = Σ(邻域像素 × 学习到的权重)
```

核预测的优势是可解释性更强，网络不需要直接“猜”对颜色，只需要决定“哪些邻居更可信”。工业上很多降噪器（如 KPCN、OIDN 早期版本）用核预测。

### Transformer：更全局的注意力

近年一些方案用 **Vision Transformer（ViT）** 或 **Swin Transformer** 做降噪。核心是把图像切成 patch，用自注意力机制建立像素间的长距离关联。

**优势**：能捕捉全局光照关系（比如“这个暗角被远处窗户照亮”）。

**代价**：计算量比 U-Net 大，实时性更差。目前在实时渲染中 U-Net 仍是主流，Transformer 多用于离线或高质量预览。

## 问题 4：时域稳定性——为什么单帧降噪还不够？

只处理单帧的降噪器有个致命问题：**帧间闪烁**。

每帧噪声是独立随机产生的，单帧降噪器看到的内容每帧都在变。即使它把每一帧都处理得很干净，连续播放时画面会像“静电干扰”一样抖动。

**解决方案：引入历史帧。**

```
当前帧 Noisy Radiance
        │
        ▼
    重投影到上一帧 UV
        │
        ▼
上一帧 Denoised Output（历史帧）
        │
        ▼
    颜色约束 / AABB clamp
        │
        ▼
    与当前帧混合
        │
        ▼
    输出当前帧 Denoised Output（成为下一帧的历史）
```

这和 [[Notes/SelfGameEngine/渲染管线与画面/后处理效果算法|后处理效果算法]] 里讲的 TAA 非常像，都需要：

- **Motion Vector**：找到像素在上一帧的位置。
- **重投影（Reprojection）**：把历史颜色对齐到当前帧。
- **颜色约束（Color Clamping / AABB）**：防止历史颜色与当前邻域差异过大时产生鬼影。AABB 指把历史颜色限制在当前帧 3×3 邻域颜色最小-最大盒子里。
- **混合权重（Blend Weight）**：通常 90% 历史 + 10% 当前，动态调整。

**但神经网络降噪和 TAA 有本质区别**：

- TAA 的目标是用多帧采样“抗锯齿”。
- 神经降噪的目标是用多帧累积“抑制蒙特卡洛噪声”。

两者可以结合：先用神经降噪处理单帧，再用 TAA 做时域稳定。

## 问题 5：在引擎后处理栈里，NeuralDenoise Pass 放在哪？

在 [[Notes/计算机图形学/帧缓冲与后处理/后处理栈架构|后处理栈架构]] 里，后处理栈是一个数据流管道。神经网络降噪器作为一个**全屏图像到图像的 Pass**，可以无缝插入。

### 典型顺序（路径追踪管线）

```
Path Tracing Pass
    │ 输出：Noisy HDR Radiance + G-Buffer
    ▼
NeuralDenoise Pass
    │ 输入：Noisy Radiance + Albedo + Normal + Depth + Motion
    │ 输出：Denoised HDR Radiance
    ▼
Bloom Pass
    │ 输入：Denoised HDR Radiance
    ▼
Tone Mapping Pass
    │ 输入：HDR → 输出 LDR
    ▼
FXAA / TAA Pass
    ▼
屏幕
```

**为什么 NeuralDenoise 要在 Bloom 之前？**

因为降噪器需要 HDR 空间里的原始噪声信号。如果先做 Bloom，噪声会被 Bloom 的光晕扩散开来，降噪器很难再把它和真实光照区分开。

### RenderGraph 表达

```cpp
graph.AddPass("NeuralDenoise", [&](PassBuilder& b) {
    b.Read(noisyRadiance);
    b.Read(gBufferAlbedo);
    b.Read(gBufferNormal);
    b.Read(gBufferDepth);
    b.Read(motionVectors);
    b.Read(historyDenoised); // 上一帧输出

    auto denoised = b.CreateTransientTexture("Denoised", hdrDesc);
    b.Write(denoised);

    b.Execute([=](ICommandContext* ctx) {
        ctx->BindTexture(0, noisyRadiance);
        ctx->BindTexture(1, gBufferAlbedo);
        ctx->BindTexture(2, gBufferNormal);
        ctx->BindTexture(3, gBufferDepth);
        ctx->BindTexture(4, motionVectors);
        ctx->BindTexture(5, historyDenoised);
        ctx->Dispatch(neuralDenoiseComputeShader);
    });
});
```

> **注意**：历史帧是跨帧资源，不能放在 RenderGraph 的帧内临时纹理里。需要池化纹理 + Extract 机制，把上一帧输出保留到下一帧。

## 问题 6：工业界有哪些可用的神经降噪方案？

### Intel Open Image Denoise（OIDN）

- **定位**：开源、跨平台、CPU/GPU 均可运行。
- **输入**：Noisy beauty + Albedo + Normal。
- **特点**：质量高、易集成、适合离线渲染和实时预览。
- **缺点**：纯降噪，不做上采样；时域稳定性需要你自己做。

### NVIDIA OptiX Denoiser

- **定位**：NVIDIA GPU 上的实时降噪，集成在 OptiX 和 DLSS 管线中。
- **输入**：Noisy beauty + Albedo + Normal + Motion Vectors。
- **特点**：支持时域降噪，性能优秀。
- **缺点**：绑定 NVIDIA 生态。

### DLSS Ray Reconstruction（DLSS 3.5+）

- **定位**：把“降噪 + 上采样 + 时域稳定”打包成一个神经网络。
- **输入**：低分辨率 Noisy Radiance + G-Buffer + Motion + History。
- **输出**：高分辨率 Denoised + Upscaled 图像。
- **特点**：专门为实时光追设计，取代了传统的 multiple denoisers 串联。

### AMD Neural Supersampling and Denoising（NSSD）

- **定位**：类似 DLSS Ray Reconstruction，单网络同时完成降噪和上采样。
- **输入**：1 spp Noisy + Albedo + Normal + Roughness + Depth + Specular Hit Distance。
- **特点**：目标是在 RDNA2+ 上实现实时路径追踪。

> **个人项目推荐**：如果你用路径追踪做画面，建议先用 **OIDN** 做离线/预览验证。实时管线建议接入 **DLSS Ray Reconstruction** 或 **OptiX Denoiser**（NVIDIA 平台），或跟踪 **AMD NSSD** 的公开实现。不要自己从头训练一个通用降噪器——数据量和工程成本都太高。

## 问题 7：训练一个降噪网络需要什么？

工业级降噪器通常需要：

1. **成对训练数据**：
   - 输入：1 spp / 4 spp 的 Noisy 图像 + G-Buffer
   - 目标：4096 spp 的 Ground Truth 参考图

2. **大量场景覆盖**：
   - 不同材质（金属、玻璃、粗糙墙面）
   - 不同光照（室内、室外、夜景）
   - 不同几何复杂度

3. **损失函数**：
   - L1 / L2 像素损失：保证颜色准确
   - SSIM / LPIPS：保证感知质量
   - 时域一致性损失：减少帧间闪烁

4. **训练基础设施**：
   - 通常需要数十到数百 GB 的路径追踪数据集
   - 单场景训练可能 24 小时起步

> **这就是为什么个人项目不应该从头训练通用降噪器**。更好的做法是调用已有 SDK（OIDN/OptiX/DLSS），或只在特定场景上做迁移学习。

## 最小可运行验证路径

如果你想在引擎里验证神经降噪，建议这样走：

1. **先用 OIDN 离线验证**：
   - 渲染一帧 1 spp 的 Noisy 图 + Albedo + Normal。
   - 导出 PNG，用 OIDN 命令行工具跑降噪。
   - 对比 64 spp 参考图，确认画质收益。

2. **集成到引擎后处理栈**：
   - 新增 `NeuralDenoisePass`。
   - 输入 Noisy Radiance + G-Buffer。
   - 输出 Denoised Radiance。
   - 先做单帧版本，观察静态画面效果。

3. **加时域稳定**：
   - 引入历史帧和 Motion Vectors。
   - 做重投影 + 颜色约束 + 混合。
   - 播放动画，检查闪烁和鬼影。

4. **性能优化**：
   - 把 Pixel Shader 实现改成 Compute Shader。
   - 降低网络分辨率或通道数。
   - 考虑专用推理框架（ONNX Runtime、TensorRT）。

## 常见陷阱

### 陷阱 1：以为神经网络降噪“不消耗性能”

神经网络的推理成本不低。4K 分辨率下跑一个 U-Net 可能需要 1~3ms。在 16ms 帧预算里，这是显著开销。必须和画质收益做权衡。

### 陷阱 2：辅助 buffer 质量差导致降噪失败

如果 G-Buffer 里的 Normal 是抖动的、Depth 精度不够、Motion Vector 是错的，降噪器会被这些错误引导，产生更严重的 artifact。辅助 buffer 的质量直接决定降噪上限。

### 陷阱 3：忽略 HDR 空间

降噪应该在线性 HDR 空间进行。如果在 LDR 或 sRGB 空间降噪，Tone Mapping 的非线性会让噪声分布变形，网络学不到正确的统计规律。

### 陷阱 4：过度降噪导致画面“塑料感”

网络为了最小化损失，可能倾向于输出“平均”颜色，丢失细微高光和纹理。需要在损失函数里加入细节保留项，或限制降噪强度。

### 陷阱 5：把降噪和 TAA 顺序搞反

```
// ❌ 错误：先 TAA 再降噪
TAA → NeuralDenoise → Tone Mapping

// ✅ 正确：先降噪再 TAA（或二合一）
NeuralDenoise → Tone Mapping → TAA
```

TAA 的混合会改变噪声的统计特性，让降噪器难以工作。

## API 对照：我们在解决「把低采样噪声图变成干净图」这个问题

| 维度 | 传统双边滤波 | U-Net 神经降噪 | Transformer 神经降噪 |
|------|------------|--------------|-------------------|
| 输入 | 噪声图 | 噪声图 + G-Buffer | 噪声图 + G-Buffer + 历史帧 |
| 速度 | 快 | 中 | 慢 |
| 质量 | 中，易糊 | 高，细节保留好 | 很高，全局关系强 |
| 实现难度 | 低 | 中 | 高 |
| 训练需求 | 无 | 需要预训练模型 | 需要预训练模型 |
| 时域稳定 | 差 | 中等（需额外处理） | 强（自注意力可跨帧） |

> **个人项目推荐**：实时管线先用 U-Net + 历史帧混合。如果追求最高质量且性能预算充足，可以尝试 Transformer；否则调用 DLSS Ray Reconstruction / OIDN 等成熟方案。

## 设计 checklist

| 检查项 | 标准 |
|--------|------|
| 输入空间 | 线性 HDR，不要在 LDR 上降噪 |
| 辅助 buffer | 至少提供 Albedo + Normal + Depth |
| 时域稳定 | 使用历史帧 + Motion Vector + 颜色约束 |
| Pass 顺序 | NeuralDenoise 在 Bloom 之前、Tone Mapping 之前，保证输入是线性 HDR |
| 网络推理 | 优先用 Compute Shader，4K 分辨率需考虑性能预算 |
| 历史帧管理 | 跨帧池化纹理，RenderGraph Extract 机制 |
| 落地方案 | 个人项目优先调用 OIDN/OptiX/DLSS，避免从头训练 |
