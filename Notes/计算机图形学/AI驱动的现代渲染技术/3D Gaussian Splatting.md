# 3D Gaussian Splatting

## 问题 0：我们已经能渲染三角网格了，为什么还需要新表示？

在 [[Notes/计算机图形学/顶点数据与索引/从OBJ文件到GPU网格|从OBJ文件到GPU网格]] 里，你已经学会了把 `.obj` 里的顶点、法线、UV 解析出来，上传到 VBO/EBO，用 Shader 画在屏幕上。这套流程支撑了现代游戏引擎几十年。

但当你想表示一个**真实世界场景**时，三角网格开始暴露三个痛点：

1. **制作成本高**：一棵造型复杂的树、一座古建筑、一个人的头发，手工建模需要数小时到数周。
2. **几何细节爆炸**：真实物体表面有很多细小结构（毛发、灰尘、树叶）。要用三角网格精确表示，面数会迅速失控。
3. **重建链路重**：从照片恢复三维场景需要摄影测量管线——特征匹配、稀疏重建、稠密重建、网格化、UV 展开、贴图烘焙，每一步都可能出错。

> **换句话说**：三角网格是"为艺术家设计"的表示，不是"为照片直接学习"的表示。

## 问题 1：能不能让神经网络直接从照片中学出场景？

### 最 naive 的想法：用一个巨大神经网络记住场景

假设场景里每个点的颜色都由一个 MLP 决定：

```
输入：空间坐标 (x, y, z) + 观察方向 (θ, φ)
输出：颜色 RGB + 密度 σ
```

训练时，用多视角照片作为监督；渲染时，沿每条相机光线采样很多点，用体渲染公式把颜色和密度积分成像素颜色。这就是 **NeRF（Neural Radiance Field）** 的核心思想。

### NeRF 解决了什么

- **不需要显式几何**：不用建模、不用 UV、不用贴图。
- **从照片端到端优化**：网络自己学习场景的几何和外观。
- **新视角合成效果惊人**：给定几张照片，能生成从未拍摄过的角度的图像。

### NeRF 带来的新问题

但 NeRF 有一个致命缺点：**慢**。

渲染一张 800×800 的图像，需要对每条光线采样几百个点，每个点过一次 MLP。这意味着：

- 单帧可能需要数秒到数分钟。
- 无法直接用于实时渲染（游戏引擎要 16ms 一帧）。

> **问题转化为**：我们想要神经表示的"端到端学习能力"，但不想要体积渲染的"每像素数百次 MLP 推理"。

## 问题 2：3D Gaussian Splatting 怎么把"神经表示"变快？

### 核心洞察：用显式粒子替代隐式 MLP

3D Gaussian Splatting（3DGS）说：与其用一个 MLP 隐式存储场景，不如直接用**数百万个 3D 高斯球**来显式表示场景。

每个高斯球存储这些可学习参数：

| 参数 | 含义 |
|------|------|
| 位置 `μ` (x, y, z) | 高斯球的中心 |
| 协方差 `Σ` (3×3) | 高斯球的形状和朝向 |
| 颜色 `c` (RGB 或 SH 系数) | 各向异性颜色 |
| 不透明度 `α` | 透明度 |

渲染时，3DGS 不体渲染。它把每个 3D 高斯球**投影到屏幕**，变成一个 2D 高斯斑点，然后按深度排序，用 alpha 混合累加颜色。

### 为什么 3DGS 比 NeRF 快得多

| 操作 | NeRF | 3DGS |
|------|------|------|
| 场景查询 | 每条光线采样 N 个点，每个点过 MLP | 直接读取高斯球参数 |
| 渲染方式 | 体积渲染（光线步进） | 可微光栅化（投影 + alpha 混合） |
| 并行度 | 每个采样点串行 MLP | 高斯球之间高度并行 |
| 典型速度 | 数秒/帧 | 数十到数百 FPS |

> 3DGS 把"场景表示"从网络权重里解放出来，变成一群可以 GPU 并行光栅化的粒子。

### 可微光栅化是什么意思

传统光栅化是"把三角形画到屏幕上"，不可微（遮挡、离散采样导致无法直接求梯度）。

3DGS 的光栅化是**可微的**：每个高斯球对最终像素的贡献是连续的 alpha 混合函数，可以通过反向传播更新高斯球的参数（位置、大小、颜色、透明度）。

训练流程：
1. 从 SfM（Structure from Motion）得到初始点云。
2. 把每个点扩展为一个 3D 高斯球。
3. 光栅化得到图像，和真实照片算 L1 + SSIM 损失。
4. 反向传播，更新高斯球参数。
5. 自适应调整高斯球数量：密度不够的地方分裂/克隆，贡献低的地方删除。

## 问题 3：3DGS 会完全替代三角网格吗？

### 两种表示的 trade-off

| 维度 | 三角网格 | 3DGS |
|------|---------|------|
| 编辑性 | 强：可以手动改顶点、重拓扑 | 弱：高斯球是"学习出来的 blob" |
| 动画 | 成熟：骨骼、蒙皮、blendshape | 还在发展中：4DGS、高斯形变场 |
| 物理碰撞 | 成熟 | 困难 |
| 显存占用 | 顶点 + 索引 + 贴图 | 数百万高斯球，显存可观 |
| 从照片重建 | 链路重 | 端到端，效果好 |
| 实时渲染 | 极快 | 很快，但依赖高斯球数量 |

### 结论：共存，而非替代

3DGS 不会替代三角网格。更可能的关系是：

- **远景/背景/照片级场景**：用 3DGS，减少手工建模成本。
- **可交互主角/道具/需要精确碰撞的物体**：继续用三角网格。
- **混合管线**：同一场景里既有 mesh，也有 3DGS，渲染时分别走不同 Pass。

## 问题 4：在自研引擎里怎么接入 3DGS？

你已经有了 [[Notes/SelfGameEngine/渲染管线与画面/RHI抽象层与命令模型|RHI]] 和 [[Notes/SelfGameEngine/渲染管线与画面/RenderGraph与多Pass资源管理|RenderGraph]]。3DGS 不是"在现有管线里塞一个新 Shader"，而是新增一种 Primitive 类型。

### 引擎侧需要新增什么

1. **资源层**
   - 加载 `.ply` 高斯球数据（位置、协方差、颜色 SH、不透明度）。
   - 把数据存到 GPU buffer/texture。

2. **组件层（ECS）**
   - 新增 `GaussianSplattingComponent`：引用高斯球资源、变换矩阵、可见性开关。
   - 渲染系统遍历所有带该组件的实体，收集高斯球数据。

3. **RHI/RenderGraph 层**
   - 新增 `GaussianSplattingPass`。
   - 输入：高斯球 GPU buffer、相机参数。
   - 输出：一张 RGBA 颜色图（或者混合到 scene color）。
   - 排序：按高斯球到相机距离深度排序（不透明时从前往后，透明时从后往前，取决于混合策略）。

4. **Shader/Compute 层**
   - 选项 A：Compute Shader 实现完整光栅化（投影、tile-based 排序、alpha 混合）。
   - 选项 B：扩展 RHI 支持高斯光栅化原语（长期方向）。
   - 个人项目推荐从 Compute Shader 开始，因为不需要改动底层图形 API 驱动。

### 一个最小可运行的验证路径

如果你只是想验证"3DGS 能在我引擎里跑起来"，最小闭环是：

1. 用官方代码训练一个场景，导出 `.ply`。
2. 写一个简单的 Compute Shader，把高斯球投影到屏幕，输出到一张纹理。
3. 在 RenderGraph 里把这张纹理作为 scene color 的一个输入，alpha 混合进去。
4. 在 Inspector 里加一个开关，决定是否启用 GaussianSplattingPass。

> 不要一开始就在引擎里实现完整的训练管线。先用预训练好的 `.ply` 做推理，验证渲染通路。

## 常见误区

### 误区 1：3DGS 是"没有几何"

错。3DGS 是**显式几何**，只是这种几何是"高斯球云"而不是"三角形集合"。它有自己的空间结构、深度排序、投影公式。

### 误区 2：3DGS 不需要相机标定

错。训练 3DGS 仍然需要已知相机位姿的多视角照片。这些位姿通常先用 COLMAP 等 SfM 工具估计。相机标定质量直接影响重建效果。

### 误区 3：3DGS 比三角网格更省显存

不一定。一个复杂场景可能需要几百万高斯球，每个高斯球存位置、协方差、SH 颜色、不透明度，显存占用可以很可观。它的优势是**从照片重建的便利性**和**渲染质量**，不是无条件地省显存。
