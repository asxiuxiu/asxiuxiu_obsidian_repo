# 图形后端与渲染路线图

## 问题：我知道帧缓冲是一堆像素，但我怎么让GPU填充它？

在[[像素、帧缓冲与3D渲染管线]]里，你用 `std::vector<Pixel>` 在CPU内存里生成了一张渐变图。你逐个设置像素颜色，然后输出成PPM文件。

这建立了核心直觉：**屏幕上的图像就是内存里一块连续的像素数据**。

但这里藏着一个巨大的跳跃——你的代码运行在CPU上，而现代显示器的像素最终是由**GPU**填充的。CPU和GPU是两个完全不同的处理器，它们甚至不共享同一块物理内存。

> **关键问题**：你写的 `std::vector<Pixel>` 代码操作的是CPU内存，但显示器的像素是由GPU控制的。CPU怎么告诉GPU"把这10万个像素涂成这个颜色"？

## 最naive的想法：CPU直接操作显存

如果没有任何图形API，你能想到的最直接方案是什么？

"CPU通过PCIe总线直接写入GPU显存里的帧缓冲，然后GPU把这张帧缓冲显示到屏幕上。"

这个方案在理论上可行，但实际上会立刻撞上三个致命问题：

**问题A：不同GPU的显存布局完全不同**

NVIDIA显卡、AMD显卡、Intel集成显卡，它们的显存格式、对齐要求、颜色空间转换规则都不一样。你不可能为市面上几百款不同架构的显卡分别写像素填充代码。

**问题B：CPU直接写显存慢得无法接受**

PCIe总线的带宽虽然高，但延迟也高。CPU逐个像素地通过总线写入GPU显存，比访问自己的内存慢几十倍。假设场景有10万个顶点、每帧60fps、每个像素4字节——这种逐个写入的方式会让CPU-GPU通道彻底堵死。

**问题C："画完一帧"需要同步，否则屏幕会撕裂**

即使CPU写完了所有像素，它怎么知道GPU什么时候把这一帧显示到了屏幕上？如果CPU在GPU正在读取帧缓冲的时候写入新数据，屏幕上会出现**画面撕裂**（tearing）——上半部分是旧帧，下半部分是新帧。

## 方案：引入图形API和图形后端

这三个问题共同指向一个结论：**不能让每个程序直接操作GPU硬件，必须有一个中间层**。

这个中间层就是**图形后端**（Graphics Backend）。它向上提供统一的"画东西"接口，向下屏蔽不同GPU驱动的差异。

> [!info] 术语说明：图形后端、图形 API、驱动
> 在这篇笔记里，**"图形后端"和"图形 API"基本同义**，指 OpenGL、Vulkan、D3D12、Metal 这类标准接口。
> 
> 它们通常由两部分组成：
> - **API 规范**：定义你能调用哪些函数（如 Khronos 定义 Vulkan，Microsoft 定义 D3D12）。
> - **驱动实现**：GPU 厂商（NVIDIA/AMD/Intel/Apple）根据规范写的驱动，负责把 API 调用翻译成自家 GPU 能懂的指令。
> 
> 在后面的引擎路线中，你还会遇到另一层封装叫 **RHI（Rendering Hardware Interface）**。RHI 是引擎自己封装的抽象层，它的下面是 OpenGL/Vulkan/D3D12 这些具体的"图形后端实现"。不要混淆这两个"后端"的用法。

图形后端的核心职责只有三件：

**1. 申请渲染上下文**（Render Context）

告诉操作系统和GPU："这个窗口要用来渲染。"

在OpenGL里，这是 `glfwCreateWindow` + `glfwMakeContextCurrent` 几行代码。在Vulkan里，这是 `VkInstance` → `VkPhysicalDevice` → `VkDevice` → `VkSurfaceKHR` 的漫长链条——**同样的目的，Vulkan需要600行代码才能清一次屏**。这个数量级差异本身就是后面要讲的故事。

**2. 管理交换链**（Swap Chain）

解决"撕裂"问题。GPU维护两个帧缓冲：**前台缓冲**（正在显示到屏幕）和**后台缓冲**（你正在画）。画完后交换两者的角色。这就是双缓冲，也是垂直同步（VSync）的底层机制。

在OpenGL里，`glfwSwapBuffers` 帮你隐式管理了这一切。在Vulkan里，你必须显式创建 `VkSwapchainKHR`、处理 `vkAcquireNextImageKHR`、再调用 `vkQueuePresentKHR`——每一步都在回答一个明确的问题。

**3. 提供最小绘制能力**

哪怕只是"把屏幕清成深灰色"，这也是一种绘制能力。图形后端必须能执行这种最基本的GPU操作。

> **一句话总结**：图形后端 = OpenGL、Vulkan、D3D12 这类标准接口 + 它们的驱动实现。你通过这些统一接口写"画东西"的代码，GPU 厂商的驱动再把它翻译成自家 GPU 能懂的指令。你不会直接说"NVIDIA 方言"或"AMD 方言"。

## 问题：那接下来要学的软光栅化算什么？

现在你脑子里应该有一个疑问：

"等等，我接下来要学的阶段二（软光栅化），不就是在CPU上模拟GPU的工作吗？那软渲染器和图形后端是什么关系？"

**答案是：软渲染器是"教学工具"，图形后端是"工程现实"。**

| 维度       | 软渲染器（阶段二）    | GPU编程（阶段三~七）   | 引擎封装（阶段八）      |
| -------- | ------------ | -------------- | -------------- |
| 运行在哪     | CPU内存        | GPU显存          | GPU显存（通过抽象层）   |
| 速度       | 极慢（单线程for循环） | 快（硬件并行）        | 快（硬件并行）        |
| 代码量      | 几百行C++       | 几千行OpenGL调用    | 几千行RHI + 几百行上层 |
| 目的       | 理解GPU在做什么    | 学会调用GPU        | 让引擎不依赖具体API    |
| 和图形后端的关系 | **没有图形后端**   | **直接调用OpenGL** | **封装图形后端**     |

软渲染器的价值不在于"实用"，而在于**建立直觉**。当你在CPU上用 `for(y) for(x)` 遍历像素、判断三角形覆盖、插值颜色时，你真正理解的是：

> "哦，原来GPU的片段着色器（Fragment Shader）就是在并行地做这件事——只不过它同时跑了成千上万个实例。"

阶段二的每一个算法（Bresenham画线、边缘函数填充、Z-Buffer深度测试、透视校正插值），都是GPU硬件里某个固定功能单元或Shader程序的**CPU模拟版**。你先理解了模拟版，再看GPU版时，每个API调用都会对应到你已知的某个算法步骤上。

**阶段二明确不包含以下内容**（避免期望错位）：
- 任何OpenGL/Vulkan API调用
- 渲染上下文初始化
- 交换链管理
- 图形后端封装

这些是阶段三及以后的内容。阶段二只关心"算法本身"。

## 问题：学会了OpenGL，为什么还要封装RHI？

> [!note] 这里的"后端"换了一层含义
> 前面说的"图形后端"是指 OpenGL/Vulkan/D3D12 这些通用图形 API；而在引擎语境里，"OpenGL 后端""Vulkan 后端"通常指 RHI 抽象层下面的具体实现分支。RHI 才是那个"上层封装接口"。

如果你已经学会了OpenGL，能画三角形、贴纹理、做后处理了，为什么还要在阶段八封装一个"渲染硬件抽象层"（RHI）？

因为你直接调用的OpenGL代码，隐藏着三个工程陷阱：

**陷阱1：OpenGL是全局状态机**

`glBindVertexArray`、`glUseProgram`、`glActiveTexture`……这些调用修改的是同一个全局状态。当你的引擎有100个物体、10种材质、5个后处理Pass时，状态会在不同代码路径中被反复修改。你很难追踪"当前绑定的VAO到底是哪个"。

更糟的是，OpenGL的某些状态是**隐式关联**的：绑定纹理单元3时，它会影响当前激活的Shader中 `sampler2D` 的读取目标。上一帧某个材质改了纹理单元3的绑定，这一帧你的着色器恰好也从纹理单元3读数据——bug出现了，但你不知道原因。

**陷阱2：换API要全部重写**

如果你的引擎直接写死了 `glDrawArrays`，有一天你想支持Vulkan或D3D12，所有渲染代码都要重写。而上层的ECS渲染系统、材质系统、后处理系统——它们其实不关心你用OpenGL还是Vulkan，它们只想说"画这个网格，用这个材质"。

**陷阱3：多线程渲染几乎不可能**

OpenGL的上下文是线程绑定的。一个上下文只能在一个线程里活跃。当你未来想把渲染提交放到独立线程时，OpenGL会成为最大的绊脚石。现代API（Vulkan/D3D12/Metal）从设计上支持多线程录制命令缓冲，但OpenGL不行。

RHI抽象层就是解决这三个陷阱的——它把具体的API调用封装在一组统一接口后面，让上层代码完全无感知。

> **阶段八才会深入讲RHI**。现在你只需要建立这个预期：阶段三~七写的裸OpenGL代码，最终会被阶段八的RHI层"吸收"和"替代"。阶段三~七的学习不是白学——你是在"收集需求"，弄清楚RHI层需要封装哪些功能、解决哪些痛点。

## 与引擎路线的交叉点

图形学路线和引擎路线不是"先学完再做"，而是**在关键节点交叉**：

| 图形学阶段 | 引擎回探点 | 做什么 |
|---|---|---|
| 阶段三（第一个三角形）后 | 阶段1.2 最简图形后端 | 把"只能清屏"升级为"能画三角形" |
| 阶段五（Shader与光照）后 | 阶段5.1 RHI + 5.3 材质 | 参与Shader编译、Uniform绑定、材质参数设计 |
| 阶段六（纹理系统）后 | 阶段5.2 资源管理 | 参与纹理加载、GPU上传、生命周期管理 |
| 阶段七~八（后处理/渲染架构）后 | 阶段5.4~5.6 渲染全模块 | 在引擎里直接实现后处理管线和渲染架构 |

> **红线规则**：阶段三后必须回探引擎，不准"我再学学纹理再切入"。图形学知识在独立Demo里学一遍，再在引擎里落地一遍，才能内化为真正的工程能力。
