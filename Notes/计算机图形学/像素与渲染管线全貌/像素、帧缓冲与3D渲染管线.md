# 像素、帧缓冲与3D渲染管线

## 问题 0：屏幕上的图像到底是什么？

想象你面前有一张纯红色的图片。它「看起来」是一整块红色，但在计算机里，它根本不是「一块红色」——它是**几百万个独立的小色块**，每个小色块都有自己的颜色值。

**最 naive 的方案**：如果我想让屏幕变红，我能不能直接告诉显示器"变红"？

不能。显示器只认一件事：**每个像素该显示什么颜色**。你必须精确到「第 347 行、第 892 列的像素是 (255, 0, 0)」这种粒度。

这就是**像素（Pixel = Picture Element）**——数字图像的最小单位。所有数字图像，无论是一张自拍、一段视频，还是 3D 游戏的实时画面，最终都分解为像素。

> 图形工程师被称为"像素背后的男人"，不是比喻，是字面意思——他们的工作就是控制每一个像素的颜色。

---

## 问题 1：像素在内存里长什么样？

知道了"图像是像素的集合"，下一个问题：**像素在内存里怎么存？**

### 最 naive 的方案：用 `int` 存颜色

```cpp
// 一个像素用4个int？
int red = 255;
int green = 0;
int blue = 0;
int alpha = 255;
```

每个通道占 4 字节，一个像素 16 字节。一张 1920×1080 的图片需要 1920 × 1080 × 16 ≈ **33 MB**。

**发现的问题**：太浪费了。人眼根本分辨不出 0~255 和 0~4294967295 之间的亮度差异。

### 改进：`uint8_t`（0~255）

```cpp
struct Pixel {
    uint8_t r, g, b, a;  // 每个通道1字节，共4字节
};
```

同样 1920×1080 的图片只需要 **8 MB**，精度完全够用。

**为什么 256 级够用？**
- 人眼对亮度变化的感知是对数级的，相邻两级的差异（如 128 vs 129）在大多数场景下无法分辨
- GPU 的显存和纹理单元原生支持 `RGBA8888` 格式
- 只有 HDR（高动态范围）场景才需要 `half`（16位浮点）或 `float`（32位浮点）

### 内存布局：像素矩阵

屏幕分辨率 1920×1080 = 约 200 万个像素。它们在内存中是一个**一维数组**：

```
索引 0          索引 1          索引 2         ...  索引 2073599
┌─────────┐   ┌─────────┐   ┌─────────┐        ┌─────────┐
│ R G B A │   │ R G B A │   │ R G B A │   ...  │ R G B A │
└─────────┘   └─────────┘   └─────────┘        └─────────┘
 (0,0)         (1,0)         (2,0)               (1919,1079)
```

**二维坐标到一维索引的映射**：
```cpp
int index = y * width + x;
```

> 一个谨慎的读者会问：为什么是一维数组，不是二维数组 `image[y][x]`？
> 
> 因为 GPU 和 CPU 的内存子系统对连续内存的访问效率最高。一维数组保证像素在物理内存中连续排列，缓存友好。二维数组（指针数组）实际上是指向分散内存块的指针，随机访问时 cache miss 更高。

### 帧缓冲（Frame Buffer）

存储屏幕上所有像素颜色值的内存区域，叫做**帧缓冲**。GPU 每完成一帧渲染，就把结果写入帧缓冲，然后显示器从帧缓冲读取数据显示出来。

```
GPU 渲染 ──→ 帧缓冲（显存中的像素数组）──→ 显示器扫描输出
```

在引擎中，帧缓冲是"画面"的物理载体。后处理（如模糊、色调映射）本质上就是"读取帧缓冲 → 处理 → 写回帧缓冲"的过程。

## 问题 2：怎么验证我写的像素操作是对的？

写了一个 `std::vector<Pixel>`，但内存里的数据是看不见的。怎么验证？

### 最 naive 的方案：打印到控制台

```cpp
for (auto& p : image) {
    std::cout << (int)p.r << " " << (int)p.g << " " << (int)p.b << "\n";
}
```

**发现的问题**：200 万个像素输出到控制台，根本没法看，也没法验证颜色对不对。

### 改进：保存为图像文件

我们需要一种**最简单的图像格式**，不需要压缩、不需要第三方库、几行代码就能读写——**PPM（Portable Pixmap Format）**。

```
P3              # P3 = ASCII 文本格式
100 100         # 宽度和高度（像素）
255             # 每个通道最大值
255 0 0         # 像素1：红色
0 255 0         # 像素2：绿色
...             # 共 width × height 个像素
```

**PPM 的优势**：
- 纯文本，可以用记事本直接打开
- 无压缩，零编解码开销
- 零依赖，不需要 libpng/libjpeg
- 浏览器和系统照片应用都能打开

```cpp
void save_ppm(const char* filename,
              const std::vector<Pixel>& pixels,
              int width, int height) {
    std::ofstream file(filename);
    file << "P3\n";
    file << width << " " << height << "\n";
    file << "255\n";
    for (const auto& p : pixels) {
        file << (int)p.r << " " << (int)p.g << " " << (int)p.b << " ";
    }
}
```

## 问题 3：手动设置每个像素太痛苦了

现在我们有了一个能生成图像的完整流程：

1. 创建 `std::vector<Pixel>`（width × height）
2. 用循环设置每个像素的颜色
3. 调用 `save_ppm` 保存

但等等——画一条斜线要算几百个像素坐标，画一个三角形要判断哪些像素在内部，稍微移动一下要全部重算。

```cpp
// 画一条从 (10, 10) 到 (200, 150) 的线
// 需要计算这条线经过的所有像素...
// 这也太累了
```

**这就是问题**：我们不想手动控制每个像素。我们想定义**几何形状**（点、线、三角形），让计算机自动帮我们填充像素。

这个需求引出了光栅化——但光栅化只是 3D 渲染管线的一个环节。在深入光栅化之前，我们需要先建立 3D 渲染的**全貌直觉**。

## 问题 4：3D 世界由什么构成？

如果你刚从 2D 过来，你对"3D 世界"的想象可能是一堆复杂的数学公式。但其实，3D 世界和现实世界非常像——它是由**物体**、**相机**和**光源**组成的。

### 用"拍照"来理解 3D 渲染

想象你要给一桌美食拍一张照片：

```
3D 场景                    拍照过程                 2D 图像
┌─────────────┐           ┌─────────────┐          ┌─────────┐
│  🍰 蛋糕    │           │             │          │  🍰 🍝  │
│      🍝     │  ──────►  │   📷 相机    │  ────►  │  （照片）│
│  💡 灯光    │   变换    │   投影      │          └─────────┘
└─────────────┘           └─────────────┘
```

3D 渲染本质上就是**用虚拟相机给虚拟世界拍一张照片**。

### 四大要素

| 要素 | 现实类比 | 在图形学中的作用 |
|------|----------|----------------|
| **模型 (Model)** | 桌子上的蛋糕、红酒 | 3D 世界的"演员"，由三角形拼凑而成 |
| **材质 (Material)** | 蛋糕的奶油质感、红酒的玻璃反光 | 决定表面的颜色、光泽、透明度 |
| **光源 (Light)** | 餐厅的灯光、窗外的阳光 | 照亮场景，产生明暗和阴影 |
| **相机 (Camera)** | 你的手机/相机 | 决定从哪个角度看世界，生成 2D 图像 |

> 记忆口诀：**顶点定形状，面片拼模型，纹理加细节，材质定质感。**

### 3D 模型在计算机里到底是什么？

计算机无法直接画"一个球"或"一个人"。它只能画**三角形**。所有 3D 模型，本质上都是一堆三角形的集合：

| 概念 | 通俗解释 | 作用 |
|------|---------|------|
| **顶点 (Vertex)** | 三角形的三个角点 | 记录 3D 空间中的位置 `(x, y, z)` |
| **面片 (Face)** | 一个三角形 | 三个顶点连成的面，是 3D 模型的最小砖块 |
| **面片索引 (Index)** | 每个三角形对应的三个顶点编号 | 复用顶点，避免重复存储 |
| **网格 (Mesh)** | 所有面片的集合 | 构成模型的完整几何形状 |
| **法线 (Normal)** | 垂直于面片朝外的方向 | 告诉光源"这个面朝哪个方向"，用于计算明暗 |
| **纹理 (Texture)** | 一张贴在模型表面的 2D 图片 | 给模型添加皮肤、木纹、砖墙等细节 |
| **纹理坐标 (UV)** | 2D 图片上的坐标 `(u, v)` | 告诉计算机"三角形的这个角对应图片的哪个像素" |

## 问题 5：怎么把 3D"拍扁"成 2D？

### 三个变换步骤

```
模型坐标 → [模型变换] → 世界坐标 → [视图变换] → 相机坐标 → [投影变换] → 屏幕坐标
   (局部)                  (全局)                  (视角)                   (像素)
```

| 变换 | 作用 | 类比摄影 |
|------|------|---------|
| **模型变换 (Model)** | 把物体摆到场景中（平移/旋转/缩放）| 把道具放到拍摄场地 |
| **视图变换 (View)** | 设置相机位置和朝向 | 架设相机、调整角度 |
| **投影变换 (Projection)** | 3D → 2D 投影 | 相机成像、拍成照片 |

### 两种投影方式

| 投影类型 | 现实类比 | 特点 | 应用场景 |
|---------|---------|------|---------|
| **透视投影** | 人眼、普通相机 | 近大远小，有纵深感 | 游戏、电影、写实渲染 |
| **正交投影** | 建筑蓝图、CAD | 平行线保持平行，无近大远小 | 工程制图、策略游戏小地图 |

## 问题 6：渲染管线的全貌

现代 GPU 的渲染管线是一个**数据流**处理过程：

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│  3D 模型   │───→│   顶点处理  │───→│   光栅化   │───→│  像素处理  │
│  (OBJ/FBX) │    │ (MVP 变换)  │    │(三角形填充)│    │ (着色输出) │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
       │                │                │                │
       ↓                ↓                ↓                ↓
   顶点数据         屏幕空间         片段(Fragment)      最终颜色
   (x,y,z,u,v)      坐标 (x,y)       带插值属性          (RGBA)
```

### 关键阶段

| 阶段 | 功能 | 可编程？ |
|------|------|---------|
| **顶点着色器** | 处理每个顶点，进行 MVP 变换 | ✅ 可编程 |
| **图元装配** | 顶点组装成三角形，裁剪 | ❌ 固定 |
| **光栅化器** | 三角形 → 像素（片段） | ❌ 固定（硬件） |
| **片段着色器** | 计算每个像素的颜色 | ✅ 可编程 |
| **输出合并** | 深度测试、混合、写入帧缓冲 | ❌ 固定（可配置） |

> 现代 GPU 的核心就是两个可编程阶段（顶点/片段着色器）和三个固定阶段（图元装配、光栅化、输出合并）。理解这个数据流，是后续学习 Shader 编程的基础。
