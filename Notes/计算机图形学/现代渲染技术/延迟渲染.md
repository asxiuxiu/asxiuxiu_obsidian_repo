# 延迟渲染

## 问题 0：Shadow Map 有了，但多光源场景怎么这么快就卡了？

你在 [[Notes/计算机图形学/现代渲染技术/阴影映射|阴影映射]] 里已经让方向光能投射阴影。现在你想让场景更生动：加一个点光源做路灯，加一个聚光灯做手电筒，再加几个彩色装饰灯。

**最 naive 的方案**：在前向渲染（Forward Rendering）的片段着色器里，对每个物体遍历所有光源。

```glsl
#version 330 core
uniform int uLightCount;
uniform Light uLights[MAX_LIGHTS]; // 比如 8 个

void main() {
    vec3 result = vec3(0.0);
    for (int i = 0; i < uLightCount; i++) {
        result += CalculateLight(uLights[i], vWorldPos, vNormal, vAlbedo);
    }
    FragColor = vec4(result, 1.0);
}
```

**立刻发现的问题**：

- 假设场景有 1000 个物体、100 个光源，每个像素要算 `1000 × 100 = 10万` 次光照贡献。但很多光源只影响很小一片区域，远离光源的像素白白计算。
- 更糟的是：由于 **Overdraw**（过度绘制，指被遮挡的像素仍然被着色器执行一遍），很多本不该看见的被遮挡像素也会跑完整的光照循环。
- Uniform 数组大小有限，光源再多就传不进 Shader。

**根本原因**：前向渲染把“几何处理”和“光照计算”绑定在同一个 Pass 里。每个物体都要重新算一遍所有光，无法利用“屏幕上每个像素只需要算一次”这个事实。

---

## 问题 1：能不能把光照计算搬到屏幕空间？

延迟渲染（Deferred Rendering / Deferred Shading）的核心 insight 是：

> **屏幕上每个像素最终只显示一个片段（经过深度测试后最前面的那个）。既然这样，我们为什么不先把这个片段的材质属性存下来，然后再统一算光照？**

这样就把渲染拆成两个 Pass：

1. **Geometry Pass（几何 Pass）**：渲染所有不透明物体，把位置、法线、Albedo、Roughness/Metallic 等写入 G-Buffer。**不计算光照**。
2. **Lighting Pass（光照 Pass）**：画一个全屏四边形，读取 G-Buffer，对每个屏幕像素计算光照。

复杂度从 `物体数 × 光源数` 降到 `物体数 + 光源数`（不考虑优化）。

---

## 问题 2：G-Buffer 到底是什么？

**G-Buffer（Geometry Buffer）** 是一组屏幕大小的纹理，每个像素存储了“该位置表面计算光照所需的一切属性”。

一个 PBR 延迟渲染常见的 G-Buffer 布局：

| 附件 | 存储内容 | 精度建议 |
|------|---------|---------|
| `gPosition` | 世界空间位置（或重建的深度） | RGBA16F / RGBA32F |
| `gNormal` | 世界空间法线 | RGBA16F |
| `gAlbedo` | Albedo 颜色（Base Color） | RGBA8 SRGB |
| `gMaterial` | Metallic、Roughness、AO、Emissive Mask 等 | RGBA8 / RGBA16F |

> **为什么不用一张大纹理存所有？** MRT 允许一个 Pass 输出到多张纹理，这样几何 Pass 只跑一遍场景，效率最高。

也可以用更紧凑的布局：

```
RT0: RGB = Albedo (sRGB), A = Metallic
RT1: RGB = Normal (Octahedral 编码或 RGB), A = Roughness
RT2: RGB = Emissive / AO / Material ID, A = Specular / 预留
RT3: R = Depth (用于重建世界空间位置)
```

## 问题 3：多渲染目标（MRT）怎么用？

OpenGL 3.3+ 支持 **MRT（Multiple Render Targets）**：一个片段着色器可以同时输出到 FBO 的多个颜色附件。

```glsl
#version 330 core
layout (location = 0) out vec4 gPosition;
layout (location = 1) out vec4 gNormal;
layout (location = 2) out vec4 gAlbedo;
layout (location = 3) out vec4 gMaterial;

in vec3 vWorldPos;
in vec3 vNormal;
in vec2 vTexCoord;

uniform sampler2D uAlbedoMap;
uniform sampler2D uNormalMap;
uniform sampler2D uMetallicRoughnessMap;

void main() {
    gPosition  = vec4(vWorldPos, 1.0);
    gNormal    = vec4(normalize(vNormal), 0.0);
    gAlbedo    = texture(uAlbedoMap, vTexCoord);

    vec4 mr    = texture(uMetallicRoughnessMap, vTexCoord);
    gMaterial  = vec4(mr.r, mr.g, 0.0, 1.0); // R=metallic, G=roughness
}
```

C++ 端需要告诉 OpenGL 哪些颜色附件要写入：

```cpp
GLuint gBuffer;
glGenFramebuffers(1, &gBuffer);
glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);

// 创建 4 张纹理并挂接到 GL_COLOR_ATTACHMENT0~3
GLuint gPosition, gNormal, gAlbedo, gMaterial;
// ... glGenTextures / glTexImage2D / glFramebufferTexture2D ...

// 告诉 OpenGL：这个 FBO 要同时写入 4 个颜色附件
GLuint attachments[4] = {
    GL_COLOR_ATTACHMENT0, GL_COLOR_ATTACHMENT1,
    GL_COLOR_ATTACHMENT2, GL_COLOR_ATTACHMENT3
};
glDrawBuffers(4, attachments);

// 深度附件仍然需要，因为 Geometry Pass 要做深度测试
GLuint depthRBO;
glGenRenderbuffers(1, &depthRBO);
glBindRenderbuffer(GL_RENDERBUFFER, depthRBO);
glRenderbufferStorage(GL_RENDERBUFFER, GL_DEPTH_COMPONENT, width, height);
glFramebufferRenderbuffer(GL_FRAMEBUFFER, GL_DEPTH_ATTACHMENT, GL_RENDERBUFFER, depthRBO);
```

> **关键规则**：`glDrawBuffers` 的数组顺序对应片段着色器里 `layout(location = N)` 的 `N`。第 0 个数组元素对应 `location = 0`，以此类推。

## 问题 4：Geometry Pass 只写 G-Buffer，深度怎么办？

Geometry Pass 仍然需要深度测试，否则 G-Buffer 里会存下被遮挡像素的属性。所以 FBO 必须挂接深度附件（Renderbuffer 或深度纹理）。

```cpp
// Geometry Pass 设置
glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);
glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
glEnable(GL_DEPTH_TEST);

// 渲染所有不透明物体
RenderOpaqueObjects(geometryShader);
```

> 注意：Geometry Pass 里**不要开启混合**。因为 G-Buffer 每个像素只保留最前面一个片段的属性，混合没有意义。

## 问题 5：Lighting Pass 怎么读 G-Buffer 并算光照？

Lighting Pass 本质上就是一个全屏后处理 Pass：绑定 G-Buffer 的 4 张纹理作为输入，画一个全屏四边形，片段着色器里读取并计算。

```glsl
#version 330 core
in vec2 vUV;

layout(binding = 0) uniform sampler2D gPosition;
layout(binding = 1) uniform sampler2D gNormal;
layout(binding = 2) uniform sampler2D gAlbedo;
layout(binding = 3) uniform sampler2D gMaterial;

uniform vec3 uLightPos;
uniform vec3 uLightColor;
uniform vec3 uViewPos;

out vec4 FragColor;

void main() {
    vec3 worldPos = texture(gPosition, vUV).rgb;
    vec3 normal   = normalize(texture(gNormal, vUV).rgb);
    vec3 albedo   = texture(gAlbedo, vUV).rgb;
    vec4 material = texture(gMaterial, vUV);
    float metallic  = material.r;
    float roughness = material.g;

    // 这里可以复用你已有的 PBR / Blinn-Phong 光照代码
    vec3 color = CalculatePBR(worldPos, normal, albedo, metallic, roughness,
                              uLightPos, uLightColor, uViewPos);

    FragColor = vec4(color, 1.0);
}
```

> **关键认知**：Lighting Pass 里的光照公式和 Forward 完全一样，区别只是输入从顶点属性变成了 G-Buffer 纹理采样。

### 多光源怎么办？

最朴素的做法是在片段着色器里循环所有光源：

```glsl
vec3 result = vec3(0.0);
for (int i = 0; i < uLightCount; i++) {
    result += CalculateLight(uLights[i], worldPos, normal, albedo, ...);
}
```

这已经比 Forward 好很多——每个像素只算一次，没有 Overdraw 浪费。但如果光源很多，仍然可以优化：

- **Light Volumes（光体积）**：只在该光源影响范围内的屏幕区域跑光照 Shader，而不是全屏。
- **Tiled / Clustered Deferred**：把屏幕分成 tile 或 3D cluster，提前知道每个 tile 受哪些光源影响，减少无效计算。

> 本篇只讲基础全屏 Lighting Pass。Light Volumes 和 Tiled Deferred 是后续优化方向。

## 问题 6：延迟渲染为什么不能直接处理透明物体？

这是延迟渲染最著名的限制。

**原因**：G-Buffer 每个像素只能存一个片段的属性——深度测试后最前面的那个。透明物体需要按从远到近的顺序混合多个片段的颜色和 alpha，G-Buffer 没有地方存这种多层信息。

**解决方案：Deferred + Forward 混合管线**

```
Pass 1：Geometry Pass（不透明物体）→ G-Buffer
Pass 2：Lighting Pass → Lit Scene 纹理
Pass 3：Copy Depth → 把 G-Buffer 的深度复制到默认帧缓冲
Pass 4：Forward Pass（透明物体）→ 直接画到屏幕，复用 Lit Scene 作为背景
Pass 5：PostProcess Pass
```

在 OpenGL 里，深度复制用 `glBlitFramebuffer`：

```cpp
// 把 G-Buffer 的深度复制到默认帧缓冲
glBindFramebuffer(GL_READ_FRAMEBUFFER, gBuffer);
glBindFramebuffer(GL_DRAW_FRAMEBUFFER, 0); // 默认帧缓冲
glBlitFramebuffer(0, 0, width, height, 0, 0, width, height,
                  GL_DEPTH_BUFFER_BIT, GL_NEAREST);

// 现在默认帧缓冲有正确深度，可以前向渲染透明物体
RenderTransparentObjects(forwardShader);
```

> **透明物体仍然走前向渲染**，但数量通常远少于不透明物体，所以整体开销还是可控的。

## 问题 7：Deferred vs Forward，什么时候该用哪个？

| 维度 | Forward Rendering | Deferred Rendering |
|------|-------------------|-------------------|
| 多光源 | 光源数增加，开销线性增长 | 光源数增加，Lighting Pass 开销增长，但无 Overdraw 浪费 |
| 材质多样性 | 每个物体可以独立 Shader，容易支持复杂材质 | 所有物体共用 G-Buffer 布局，新增材质属性要改 G-Buffer |
| 透明物体 | 原生支持 | 必须额外前向 Pass |
| MSAA | 原生支持 | 不能直接 MSAA，常用 TAA 或 SSAA |
| 带宽 | 低 | 高（G-Buffer 读写） |
| 实现复杂度 | 低 | 中（需要 MRT、G-Buffer 管理） |

**决策推荐**：

- **学习阶段 / 简单场景 / 大量透明物体**：用 Forward。
- **多光源、复杂不透明场景、现代主机/PC**：用 Deferred。
- **移动端**：谨慎使用 Deferred，G-Buffer 带宽是瓶颈。可以用 Forward+（Tiled Forward，把屏幕分成小 tile 提前做光源剔除的前向渲染）替代。

> **个人项目推荐**：如果已经在写 PBR，建议把 Deferred 作为默认路径。透明物体用额外的 Forward Pass。MSAA 用 TAA 替代。

## 完整可运行示例：最小延迟渲染

### C++ 端：创建 G-Buffer

```cpp
GLuint gBuffer;
glGenFramebuffers(1, &gBuffer);
glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);

auto createGBufferTexture = [](GLuint& tex, int w, int h, GLenum internalFormat, GLenum format, GLenum type) {
    glGenTextures(1, &tex);
    glBindTexture(GL_TEXTURE_2D, tex);
    glTexImage2D(GL_TEXTURE_2D, 0, internalFormat, w, h, 0, format, type, nullptr);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);
};

GLuint gPosition, gNormal, gAlbedo, gMaterial;
createGBufferTexture(gPosition, 800, 600, GL_RGBA16F, GL_RGBA, GL_FLOAT);
createGBufferTexture(gNormal,   800, 600, GL_RGBA16F, GL_RGBA, GL_FLOAT);
createGBufferTexture(gAlbedo,   800, 600, GL_RGBA8,   GL_RGBA, GL_UNSIGNED_BYTE);
createGBufferTexture(gMaterial, 800, 600, GL_RGBA8,   GL_RGBA, GL_UNSIGNED_BYTE);

glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT0, GL_TEXTURE_2D, gPosition, 0);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT1, GL_TEXTURE_2D, gNormal,   0);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT2, GL_TEXTURE_2D, gAlbedo,   0);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT3, GL_TEXTURE_2D, gMaterial, 0);

GLuint attachments[4] = { GL_COLOR_ATTACHMENT0, GL_COLOR_ATTACHMENT1,
                          GL_COLOR_ATTACHMENT2, GL_COLOR_ATTACHMENT3 };
glDrawBuffers(4, attachments);

// 深度 Renderbuffer
GLuint depthRBO;
glGenRenderbuffers(1, &depthRBO);
glBindRenderbuffer(GL_RENDERBUFFER, depthRBO);
glRenderbufferStorage(GL_RENDERBUFFER, GL_DEPTH_COMPONENT, 800, 600);
glFramebufferRenderbuffer(GL_FRAMEBUFFER, GL_DEPTH_ATTACHMENT, GL_RENDERBUFFER, depthRBO);

if (glCheckFramebufferStatus(GL_FRAMEBUFFER) != GL_FRAMEBUFFER_COMPLETE)
    std::cerr << "G-Buffer 不完整" << std::endl;

glBindFramebuffer(GL_FRAMEBUFFER, 0);
```

### Geometry Pass 片段着色器

```glsl
#version 330 core
layout (location = 0) out vec4 gPosition;
layout (location = 1) out vec4 gNormal;
layout (location = 2) out vec4 gAlbedo;
layout (location = 3) out vec4 gMaterial;

in vec3 vWorldPos;
in vec3 vNormal;
in vec2 vTexCoord;

uniform sampler2D uAlbedoMap;
uniform sampler2D uMetallicRoughnessMap;

void main() {
    gPosition = vec4(vWorldPos, 1.0);
    gNormal   = vec4(normalize(vNormal), 0.0);
    gAlbedo   = texture(uAlbedoMap, vTexCoord);

    vec4 mr   = texture(uMetallicRoughnessMap, vTexCoord);
    gMaterial = vec4(mr.r, mr.g, 0.0, 1.0);
}
```

### Lighting Pass 片段着色器（简化 Blinn-Phong）

```glsl
#version 330 core
in vec2 vUV;

layout(binding = 0) uniform sampler2D gPosition;
layout(binding = 1) uniform sampler2D gNormal;
layout(binding = 2) uniform sampler2D gAlbedo;
layout(binding = 3) uniform sampler2D gMaterial;

uniform vec3 uLightPos;
uniform vec3 uViewPos;
uniform vec3 uLightColor;

out vec4 FragColor;

void main() {
    vec3 worldPos = texture(gPosition, vUV).rgb;
    vec3 normal   = normalize(texture(gNormal, vUV).rgb);
    vec3 albedo   = texture(gAlbedo, vUV).rgb;
    float roughness = texture(gMaterial, vUV).g;

    vec3 lightDir = normalize(uLightPos - worldPos);
    vec3 viewDir  = normalize(uViewPos - worldPos);
    vec3 halfwayDir = normalize(lightDir + viewDir);

    vec3 ambient = 0.1 * albedo;
    float diff = max(dot(normal, lightDir), 0.0);
    float spec = pow(max(dot(normal, halfwayDir), 0.0), 32.0 * (1.0 - roughness));

    vec3 color = ambient + (diff * albedo + spec * vec3(0.5)) * uLightColor;
    FragColor = vec4(color, 1.0);
}
```

### 渲染循环骨架

```cpp
void RenderFrame() {
    // ===== Pass 1：Geometry Pass =====
    glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);
    glViewport(0, 0, 800, 600);
    glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
    glEnable(GL_DEPTH_TEST);
    glDisable(GL_BLEND);

    glUseProgram(geometryProgram);
    RenderOpaqueObjects();

    // ===== Pass 2：Lighting Pass =====
    glBindFramebuffer(GL_FRAMEBUFFER, 0);
    glViewport(0, 0, 800, 600);
    glClear(GL_COLOR_BUFFER_BIT);
    glDisable(GL_DEPTH_TEST);

    glUseProgram(lightingProgram);
    glActiveTexture(GL_TEXTURE0); glBindTexture(GL_TEXTURE_2D, gPosition);
    glActiveTexture(GL_TEXTURE1); glBindTexture(GL_TEXTURE_2D, gNormal);
    glActiveTexture(GL_TEXTURE2); glBindTexture(GL_TEXTURE_2D, gAlbedo);
    glActiveTexture(GL_TEXTURE3); glBindTexture(GL_TEXTURE_2D, gMaterial);

    DrawFullscreenQuad();

    // ===== Pass 3：复制深度并渲染透明物体 =====
    glBindFramebuffer(GL_READ_FRAMEBUFFER, gBuffer);
    glBindFramebuffer(GL_DRAW_FRAMEBUFFER, 0);
    glBlitFramebuffer(0, 0, 800, 600, 0, 0, 800, 600,
                      GL_DEPTH_BUFFER_BIT, GL_NEAREST);

    glEnable(GL_DEPTH_TEST);
    glEnable(GL_BLEND);
    glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
    RenderTransparentObjects();
}
```

## 状态变化图：Deferred 管线的资源流动

```
Pass 1：Geometry Pass
  目标 = G-Buffer FBO
  输出：
    COLOR_ATTACHMENT0 → gPosition
    COLOR_ATTACHMENT1 → gNormal
    COLOR_ATTACHMENT2 → gAlbedo
    COLOR_ATTACHMENT3 → gMaterial
    DEPTH_ATTACHMENT  → depthRBO
  状态：depth test ON, blend OFF
        │
        ▼
Pass 2：Lighting Pass
  目标 = 默认帧缓冲（或 Lit Scene 纹理）
  输入纹理：
    单元 0 → gPosition
    单元 1 → gNormal
    单元 2 → gAlbedo
    单元 3 → gMaterial
  状态：depth test OFF
        │
        ▼
Pass 3：Forward Transparent Pass
  目标 = 默认帧缓冲
  输入：Lit Scene 作为背景
  状态：depth test ON, blend ON
```

> **关键状态机事实**：G-Buffer 纹理在 Lighting Pass 里被读，所以 Geometry Pass 结束后必须解绑 G-Buffer FBO，再把这些纹理绑定到纹理单元。

## 与 SelfGameEngine 的关系

### Deferred = 引擎渲染架构的“进阶模式”

在 [[Notes/SelfGameEngine/渲染管线与画面/RenderGraph与多Pass资源管理|RenderGraph与多Pass资源管理]] 里，延迟渲染管线天然适合用 RenderGraph 表达：

```cpp
graph.AddPass("GBuffer", [&](PassBuilder& b) {
    b.WriteColor(0, gPosition);
    b.WriteColor(1, gNormal);
    b.WriteColor(2, gAlbedo);
    b.WriteColor(3, gMaterial);
    b.WriteDepth(gDepth);
    b.Execute([=](ICommandContext* ctx) {
        for (auto& item : opaque_phase.items) DrawItem(ctx, item);
    });
});

graph.AddPass("Lighting", [&](PassBuilder& b) {
    b.Read(gPosition);
    b.Read(gNormal);
    b.Read(gAlbedo);
    b.Read(gMaterial);
    b.WriteColor(0, litScene);
    b.Execute([=](ICommandContext* ctx) {
        ctx->BindTexture(0, gPosition);
        ctx->BindTexture(1, gNormal);
        ctx->BindTexture(2, gAlbedo);
        ctx->BindTexture(3, gMaterial);
        ctx->DrawFullscreenQuad();
    });
});

graph.AddPass("ForwardTransparent", [&](PassBuilder& b) {
    b.Read(litScene); // 作为背景
    b.Read(gDepth);   // 复制深度
    b.WriteColor(0, swapchain);
    b.Execute([=](ICommandContext* ctx) {
        // 复制深度 + 前向渲染透明物体
    });
});
```

### 对 RHI 的要求

延迟渲染需要 RHI 支持：

- **MRT**：一个 Pass 输出到多个颜色附件。
- **深度附件可读取**：需要把 G-Buffer 深度复制到默认帧缓冲（或 Lighting Pass 后复用）。
- **纹理格式灵活**：RGBA16F、RGBA8、深度格式等。

这正是 Roadmap 里说的 “Deferred 管线 = 引擎渲染架构的进阶模式，需要 RHI 层支持 MRT”。

### 与材质系统的关系

在 [[Notes/SelfGameEngine/渲染管线与画面/材质系统架构|材质系统架构]] 中，Deferred 对材质有两层含义：

1. **G-Buffer 布局是全局契约**：所有不透明材质必须按同一套 G-Buffer 格式输出属性。如果某个材质需要额外属性（如次表面散射、各向异性），必须扩展 G-Buffer。
2. **Shader 变体分裂**：
   - `DEFERRED_GEOMETRY`：写 G-Buffer 的变体。
   - `DEFERRED_LIGHTING`：读 G-Buffer 算光照的变体（通常只有一个）。
   - `FORWARD_TRANSPARENT`：透明物体用前向渲染。

## API 对照：我们在解决「把几何和光照解耦到不同 Pass」这个问题

| 概念 | OpenGL | Vulkan | D3D12 |
|------|--------|--------|-------|
| MRT | `glDrawBuffers` + `layout(location=N)` | `VkSubpassDescription` color attachments | `OMSetRenderTargets` 多个 RTV |
| G-Buffer FBO | FBO + 多个颜色附件 + 深度附件 | `VkRenderPass` + `VkFramebuffer` | `ID3D12Resource` + RTV/DSV |
| 深度复制 | `glBlitFramebuffer` | `vkCmdCopyImage` / depth resolve | `CopyResource` / depth copy |
| 全屏 Lighting Pass | 全屏四边形 / 计算 Shader | 全屏四边形 / 计算 Shader | 全屏四边形 / 计算 Shader |
| 透明 Forward Pass | 在 Deferred 之后执行 | subpass 或额外 render pass | 额外 command list |

> **个人项目推荐**：学习阶段用 OpenGL FBO + MRT + 全屏四边形 Lighting Pass。向现代 API 迁移时，mentally map 为：`G-Buffer FBO ≈ VkRenderPass + VkFramebuffer`，`glDrawBuffers ≈ subpass color attachment references`。

## 设计 checklist

| 检查项 | 标准 |
|--------|------|
| G-Buffer 布局 | 至少包含 Position/Normal/Albedo/Material，坐标空间统一 |
| MRT 设置 | `glDrawBuffers` 顺序与片段着色器 `layout(location)` 对齐 |
| Geometry Pass | 开启深度测试，关闭混合 |
| Lighting Pass | 关闭深度测试，读取 G-Buffer 做全屏光照 |
| 透明物体 | Deferred 后用 Forward Pass，先复制深度 |
| 坐标空间 | Position/Normal/Light 必须在同一空间（推荐世界空间） |
| 精度 | Position/Normal 用 16F，Albedo 用 8-bit sRGB |

## 常见陷阱

### 陷阱 1：G-Buffer 纹理格式不支持 FBO 完整

某些驱动对 3 通道浮点纹理支持不好。优先用 4 通道格式（`GL_RGBA16F` 而不是 `GL_RGB16F`）。

### 陷阱 2：Lighting Pass 里坐标空间不一致

```glsl
// ❌ 错误：G-Buffer 存世界空间位置，但光源在观察空间
vec3 lightDir = normalize((uView * vec4(uLightPos, 1.0)).xyz - worldPos);

// ✅ 正确：所有量都在世界空间
vec3 lightDir = normalize(uLightPos - worldPos);
```

### 陷阱 3：G-Buffer 同时被读和写

Geometry Pass 结束后必须解绑 G-Buffer FBO，再绑定其纹理到采样单元。同一张纹理不能同时是渲染目标和输入纹理。

### 陷阱 4：透明物体没复制深度

如果不把 G-Buffer 深度复制到默认帧缓冲，透明物体会被错误地遮挡或显示在背景前面。

### 陷阱 5：Lighting Pass 忘记关深度测试

Lighting Pass 画的是全屏四边形，开启深度测试可能导致四边形被 G-Buffer 残留的深度剔除。
