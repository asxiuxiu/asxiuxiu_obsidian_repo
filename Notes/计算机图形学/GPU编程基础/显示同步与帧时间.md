# 显示同步与帧时间

## 问题 0：为什么 GPU 已经画完了，屏幕还是不换图？

想象你刚写好一段 OpenGL 渲染代码，屏幕上画出了一个旋转的彩色三角形。你心满意足地看着它——然后发现一个问题：

**GPU 画帧的速度，和显示器换图的速度，是两个完全独立的事情。**

GPU 是一台并行计算机器，它可能 3ms 就画完了一整帧。但显示器是一台固定节拍运行的扫描仪，它每 6.94ms（144Hz）或 16.67ms（60Hz）才从显存里抓一帧图来显示。

如果 GPU 画完了但显示器还没到换图的时刻，那帧就只能躺在显存里等着。反过来，如果 GPU 画得慢，显示器没新图可看，就只能把旧图再显示一次。

**这一章就是来回答：这两台机器怎么协作，才不会互相拖后腿？**

---

## 问题 1：显示器是怎么"刷新"的？

显示器的刷新不是"嘭"的一下全屏同时换掉。它更像一台**扫描仪**，从屏幕左上角开始，**一行一行地往下扫**，扫到右下角，这一帧才算显示完毕。

```
时间 →
[行1] ████████████████████████████████████████
[行2]  ████████████████████████████████████████
[行3]   ████████████████████████████████████████
...
[行N]                                      ████ ← 扫完最后一行
        ↑                                    ↑
      开始新帧                            垂直回扫(V-Blank)
```

扫完最后一行后，电子束（或液晶控制器）需要回到左上角准备扫下一帧，这段回程时间叫做 **垂直空白期（Vertical Blank，V-Blank）**，通常只有几百微秒到几毫秒。

**刷新率（Hz）就是这台扫描仪的固定节拍**：
- 60Hz = 每 16.67ms 扫完一帧
- 144Hz = 每 6.94ms 扫完一帧
- 240Hz = 每 4.17ms 扫完一帧

这个节拍由显示器硬件电路决定，**跟 GPU 画得多快完全无关**。

---

## 问题 2：GPU 帧率（FPS）和显示器刷新率（Hz）是什么关系？

**FPS** 是 GPU 每秒能渲染出多少帧完整图像，由 CPU 逻辑 + GPU 渲染共同决定。**Hz** 是显示器每秒能换多少次图，由显示器硬件决定。

实际你眼睛看到的流畅度，取决于这两个数中的**较小值**，以及它们之间的协作方式。

| 场景 | 结果 |
|------|------|
| GPU 跑出 200fps，接 60Hz 显示器 | 显示器每 16.67ms 才换一次图，多出来的 140 帧被浪费。快速移动时可能出现画面撕裂 |
| GPU 跑出 60fps，接 144Hz 显示器 | 显示器有一半刷新没新图看，重复显示旧帧，视觉上就是不流畅 |
| GPU 跑出 144fps，接 144Hz 显示器 + VSync | 理想状态，每画完一帧，显示器刚好刷新一次 |
| GPU 跑出 50fps，接 144Hz 显示器 | 显示器有能力刷 144 次，但 GPU 只产 50 帧，画面卡顿 |

**关键认知**：Hz 是"展示能力上限"，FPS 是"生产能力上限"。显示器 144Hz 不代表 GPU 画一帧只要 1/144 秒——GPU 画一帧需要多久，由场景复杂度（顶点数、着色器、分辨率）决定，跟显示器无关。

## 问题 3：VSync 为什么能消灭撕裂？

### 画面撕裂是什么？

如果 GPU 画完一帧就立刻让显示器换图，完全不考虑显示器扫描到哪儿了，就会出现这种情况：

```
显示器正在显示帧A，扫到屏幕中间时
GPU突然说"我画完帧B了，换！"
→ 上半屏幕是帧A，下半屏幕是帧B
```

这在快速转动视角时尤其明显——画面上半部分还是旧位置的建筑，下半部分已经切到新位置了，像被刀切开一样。

### VSync 的规则

**VSync（垂直同步）只做一件事：GPU 只能在 V-Blank 期间把新帧交给显示器。**

这样保证显示器每次从头开始扫新帧时，拿到的是一整张、同一时刻的图像，绝不会出现"上半旧下半新"。

> 要做到这一点，系统里通常有两块缓冲在轮换：
> - **前台缓冲（Front Buffer）**：显示器正在读取、正在显示的那块
> - **后台缓冲（Back Buffer）**：GPU 正在画、画好后准备交给显示器的那块
>
> `SwapBuffers` 就是交换这两块缓冲的角色。但交换不能太随便——如果显示器还在读前台缓冲，你就不能把它换成别的内容，否则屏幕会闪。

## 问题 4：VSync 为什么又会"锁帧"？

这是 VSync 最让人困惑的地方。它消灭了撕裂，但带来了三个副作用。理解这三个副作用之前，先记住一个关键事实：

**`SwapBuffers` 看起来是在“让 GPU 交帧”，其实它是 CPU 发起的 API 调用。**

### 副作用 A：CPU/GPU 被迫空等

你的渲染循环里有这么一行：

```cpp
glfwSwapBuffers(window);  // 或 Vulkan 的 vkQueuePresentKHR
```

CPU 执行到这里，会对驱动说：“后台缓冲已经画好了，请把它交给显示器。” 驱动看了看显示器：它正在扫描屏幕中间，还没进入 V-Blank。按 VSync 规则，现在不能交换。

那驱动怎么办？它**把 CPU 线程挂起**，让它睡等到下一次 V-Blank：

```
时间(ms):    0        3        6.94       10       13.88
CPU:        [逻辑+提交GPU] [SwapBuffers阻塞........] [继续下一帧]
GPU:        [    实际渲染:3ms   ]        [        空闲         ]
                          ↑
                    6.94ms 才到 V-Blank
```

**GPU 3ms 就画完了，但 CPU 在 `SwapBuffers` 上硬等了 3.94ms。** 你的 `std::chrono` 计时器测出来的是显示器的刷新周期，不是 GPU 的真实速度。

> 注意：这里“阻塞 CPU”不是 GPU 在故意拖 CPU 后腿，而是驱动为了实现 VSync，主动让调用 `SwapBuffers` 的 CPU 线程等待。CPU 被卡住了，下一帧的逻辑、输入、物理都跟着暂停；GPU 反而因为没收到新命令而空闲。

### 副作用 B：帧率断崖式下跌

如果 GPU 渲染时间**略长于**一个刷新周期，事情会变得更反直觉。

假设显示器 60Hz（周期 16.67ms），GPU 画一帧要 **17ms**：

```
时间(ms):    0        16.67     33.33
显示器:      [帧A============][帧B============]
GPU:         [====帧1(17ms)====]
                  ↑ 16.67ms 时还没画完！错过 V-Blank
                                      ↓
                                   33.33ms 才提交
```

GPU 在 16.67ms 时还没画完，错过了第一个 V-Blank，只能等到下一个（33.33ms）。于是**帧1被显示了整整 33.33ms**，帧率直接从 60fps 掉到 **30fps**。

仅仅因为慢了 0.33ms！这就是 VSync 最让人抓狂的特性：**帧率不是平滑下降的，而是断崖式掉到刷新率的整数分频**。

| 显示器   | VSync 可能锁到的帧率      |
| ----- | ------------------ |
| 60Hz  | 60 → 30 → 20 → 15  |
| 144Hz | 144 → 72 → 48 → 36 |

### 副作用 C：输入延迟增加

你按了鼠标，CPU 处理输入、GPU 画完帧，但帧在后台缓冲里等着 V-Blank 才能显示。60Hz 下最多等 16.67ms，144Hz 下最多等 6.94ms。竞技游戏里这就是生死差距。

## 问题 5：不用 VSync，撕裂怎么解决？

VSync 用“让快的等慢的”换来了不撕裂，代价是 CPU 空等、帧率断崖、输入延迟。那有没有办法**既保留不撕裂的优点，又避免这些副作用**？

有的。思路只有两条：要么让显示器适配 GPU，要么让 GPU 继续狂跑但只挑最新完整帧给显示器。下面三种方案分别走了这两条路。

### 方案 1：G-Sync / FreeSync（让显示器适配 GPU）

**核心思想**：不再让 GPU 等显示器，而是让**显示器等 GPU**。

传统 VSync:   GPU 画完 → 等显示器到 V-Blank → 显示
G-Sync:       GPU 画完 → 立刻通知显示器"换帧" → 显示器动态调整刷新时刻

显示器内部的刷新电路不再是固定节拍器，而是变成"GPU 说换就换"。只要帧率在显示器的 VRR 范围内（如 30~144Hz），就**既没有撕裂，也没有锁帧，输入延迟也极低**。

**对引擎开发者的意义**：你什么都不用做。只要用户显示器支持且驱动里开启了 G-Sync/FreeSync，关闭传统 VSync（`glfwSwapInterval(0)`）就是最优状态。

> 电竞玩家的标准配置是：144Hz G-Sync 显示器 + 游戏内锁帧到 141fps（比刷新率低 3fps，避免驱动切回传统 VSync）。

### 方案 2：Fast Sync / Enhanced Sync

针对"GPU 能跑 300fps，但显示器只有 144Hz"的高帧率竞技场景。

**核心思想**：GPU 不阻塞，继续全力渲染，但只把**最新画完的一整帧**交给显示器。

```
GPU:   [帧1][帧2][帧3][帧4][帧5][帧6]...（每 3ms 一帧，不等待）
显示:  [====帧2====][====帧4====][====帧6====]...（每 6.94ms 显示一帧）
```

多余的帧被丢弃，显示器永远拿到最新完整的一帧。代价是帧间隔时间略微不均匀（jitter），但比传统 VSync 的延迟低得多。

### 方案 3：三重缓冲（Triple Buffering）

传统双缓冲只有两块：前台显示一块，后台画一块。只要后台那块画完、前台那块还没显示完，CPU/GPU 就没地方画下一帧，只能等。

三重缓冲增加第三个缓冲，相当于多准备一个“备用画板”：

```
缓冲A: 正在显示（前台）
缓冲B: 上一帧画好的（待命）
缓冲C: GPU 正在画的（后台）
```

GPU 画完帧 C → 不用等 VSync，立刻开始在缓冲 D 画下一帧（因为总有空闲缓冲可用）。V-Blank 到来时，显示器拿 B/C 中最新完整的那一帧。

这缓解了 CPU/GPU 空等的问题，但**增加了输入延迟**（显示器可能拿到的是更早画好的一帧，而不是最新画好的那一帧），且内存占用更高。现代游戏中已经被 G-Sync/FreeSync 取代。

## 问题 6：怎么知道 GPU 画得有多快？

前面讲了各种把帧交给显示器的方案，但下一个问题自然跟上来：如果游戏卡顿，我怎么判断到底是 GPU 画得慢，还是 CPU/显示器在拖后腿？

你常用的 FPS 计数器（用 `std::chrono` 测整帧间隔）在 VSync 开启时是"假数据"——它反映的是显示器刷新率，不是 GPU 性能。要测 GPU 真实耗时，有三层方法。

### 第一层：CPU 帧时间（已有，但有局限）

```cpp
auto now = std::chrono::high_resolution_clock::now();
float dt = std::chrono::duration<float, std::milli>(now - last_time).count();
```

**它测的是**：CPU 逻辑 + GPU 渲染 + `SwapBuffers` 等待 VSync 的总和。

**局限**：VSync 开启时，这个值被人为锁定到 16.67ms 或 6.94ms，完全掩盖了 GPU 的真实速度。

### 第二层：GPU 时间戳查询（推荐）

这是测量 GPU 实际耗时的标准做法。CPU 发一对"开始/结束"命令给 GPU，GPU 执行完后把真实耗时写回：

```cpp
GLuint query;
glGenQueries(1, &query);

// 每帧
 glBeginQuery(GL_TIME_ELAPSED, query);
// ... 你的所有绘制命令 ...
glEndQuery(GL_TIME_ELAPSED);

// 延迟 1~3 帧后读取结果（GPU 是异步执行的）
GLuint64 gpu_time_ns = 0;
glGetQueryObjectui64v(query, GL_QUERY_RESULT, &gpu_time_ns);
float gpu_time_ms = gpu_time_ns / 1'000'000.0f;
```

**关键优势**：不受 VSync 干扰。GPU 可能 3ms 就画完了，但 CPU 帧时间显示 6.94ms——只有 GPU 时间戳能揭示这个真相。

### 第三层：厂商剖析工具（最精确）

| 工具 | 特点 |
|------|------|
| **RenderDoc** | 开源跨平台，抓取一帧看每次 DrawCall 的 GPU 耗时。**个人开发首选。** |
| **NVIDIA Nsight** | 深度分析 SM 占用率、显存带宽 |
| **AMD Radeon GPU Profiler** | 针对 RDNA 架构的硬件计数器 |

## 问题 7：引擎该怎么暴露这些选项？

作为引擎开发者，你不需要替用户做选择，但要把选择权清晰地交出去。

| 阶段 | 建议 |
|------|------|
| **阶段 1（OpenGL）** | 暴露一个 `vsync` bool 开关，默认开启 |
| **阶段 5（Vulkan + RHI）** | 通过 `VkPresentModeKHR` 支持多种模式：<br>• `VK_PRESENT_MODE_FIFO_KHR` = 传统 VSync（必支持）<br>• `VK_PRESENT_MODE_IMMEDIATE_KHR` = 关闭 VSync（可能撕裂）<br>• `VK_PRESENT_MODE_MAILBOX_KHR` = 单重缓冲快速替换（类似 Fast Sync） |
| **长期** | 检测用户是否开启 G-Sync/FreeSync，自动推荐 `IMMEDIATE` 或 `MAILBOX` 模式 |

---

## 一句话总结

> **Hz 是显示器的"展示节拍"，FPS 是 GPU 的"生产速度"，VSync 是用"让快的等慢的"来换取不撕裂。** 现代最优解是 G-Sync/FreeSync（让显示器适配 GPU），而引擎只需要暴露呈现模式切换，剩下的交给用户的硬件。
