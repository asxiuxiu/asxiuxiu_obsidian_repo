# GPU架构与并行渲染

## 问题 0：软渲染器那么慢，瓶颈到底在哪？

在 [[Notes/计算机图形学/软光栅化与3D数学/光栅化：从直线到三角形|光栅化：从直线到三角形]] 和 [[Notes/计算机图形学/软光栅化与3D数学/3D光栅化与深度缓冲|3D光栅化与深度缓冲]] 中，你写了一个完整的光栅化器。回顾一下它的核心循环：

```cpp
for (int y = minY; y <= maxY; y++) {
    for (int x = minX; x <= maxX; x++) {
        // 1. 判断像素是否在三角形内（边缘函数）
        // 2. 计算重心坐标
        // 3. 透视校正插值
        // 4. Z-Buffer深度测试
        // 5. 写入颜色缓冲
    }
}
```

这个循环对一个 1920×1080 的屏幕要遍历约 **200 万个像素**。每个像素做浮点运算、内存读写。一个 3D 场景有几十万个三角形，意味着这个循环要执行几十万次。

**实际表现**：在你的CPU上，这个软渲染器可能每秒只能渲染几帧。

### 最直接的 naive 想法：用多线程加速

"CPU有8个核心，我让8个线程同时处理不同的像素行不就行了？"

```cpp
// 把屏幕分成8块，每块一个线程
for (int t = 0; t < 8; t++) {
    threads.emplace_back([t]() {
        for (int y = t * height/8; y < (t+1) * height/8; y++) {
            for (int x = 0; x < width; x++) {
                // ... 光栅化逻辑
            }
        }
    });
}
```

**立刻发现的问题**：

1. **并行度不够**：8个线程×每个三角形=还是太慢。一个复杂场景可能需要同时处理成千上万个像素，8个线程远远不够。

2. **内存带宽瓶颈**：所有线程竞争访问同一个颜色缓冲区和Z-Buffer。CPU缓存行（64字节）被多个核心反复争夺，产生大量**[[Notes/C++编程/对象内存模型与底层机制/缓存行、false sharing 与内存布局#问题 2：MESI 协议如何导致伪共享？|伪共享]]（False Sharing）**——两个线程改的是不同变量，只是它们恰好落在同一缓存行，缓存行就在核心之间来回失效、写回。

3. **分支发散**：不同的像素执行路径不同——有的在三角形内（需要着色），有的在外（跳过）。CPU的分支预测器对这种不规则模式束手无策。

4. **延迟暴露**：CPU做浮点运算需要等待结果，期间只能干等。一个乘加指令可能需要几个时钟周期，而一次缓存未命中可能要等上百个周期，这段时间核心在"发呆"。**延迟暴露**就是说：运算能力被等数据的空闲时间拖住了。

> 多线程CPU渲染确实比单线程快几倍，但离"实时3D游戏"的帧率要求还差几个数量级。这说明问题的本质不是"核心不够多"，而是**CPU的架构不适合这种计算模式**。

## 问题 1：渲染任务的计算模式有什么特殊之处？

CPU擅长的事情：处理复杂的控制流、分支跳转、操作系统调度、数据库查询——这些任务的共同点是**每个任务的计算路径都不同**。

GPU面临的事情：对于屏幕上被同一个三角形覆盖的数千个像素，它们的计算路径**几乎完全相同**——

- 都执行同样的边缘函数判断
- 都执行同样的[[Notes/计算机图形学/纹理系统/纹理映射与UV坐标#三角形内部怎么确定纹理颜色？——重心坐标插值|重心坐标]]插值
- 都执行同样的[[Notes/计算机图形学/Shader与光照/GPU上的Blinn-Phong光照|Blinn-Phong]]光照公式
- 都访问同样的[[Notes/计算机图形学/纹理系统/纹理映射与UV坐标|纹理]]

只有输入数据不同：每个像素的位置、深度、UV坐标不一样。

**这就是数据并行（Data Parallelism）**：同样的程序，不同的数据。也叫 **[[Notes/操作系统/CPU与内存架构/计算机体系结构速览：CPU、内存与总线#SIMD 与图形管线|SIMD]]（Single Instruction, Multiple Data，单指令多数据）**。

```
CPU 模式（8个复杂核心）：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 任务A   │ │ 任务B   │ │ 任务C   │   ← 完全不同的事情
│ 解析JSON│ │ 编译Shader│ │ 物理模拟│
└─────────┘ └─────────┘ └─────────┘

GPU 模式（数千个简单核心）：
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│P1 │ │P2 │ │P3 │ │P4 │ │P5 │ │P6 │   ← 做同一件事：着色
│红 │ │绿 │ │蓝 │ │红 │ │绿 │ │蓝 │   ← 只是数据不同
└───┘ └───┘ └───┘ └───┘ └───┘ └───┘
```

## 问题 2：GPU 是怎么设计的？

### 核心思想：用大量简单核心换吞吐量

CPU的设计哲学：每个核心都很"聪明"——有复杂的分支预测、乱序执行、大容量缓存，追求**单线程性能**。

GPU的设计哲学：每个核心都很"笨"——没有复杂的分支预测，缓存很小，但数量极多（几千个），追求**总体吞吐量**。

> **比喻**：CPU像是8个大学教授，每个人能独立解决复杂问题；GPU像是10000个小学生，每个人只会做简单的加减乘除，但10000人同时做，总吞吐量惊人。

### GPU 的层级结构

以 NVIDIA GPU 为例：

```
GPU 芯片
├── Streaming Multiprocessor (SM) × 80~140个
│   └── CUDA Core / Shader Core × 128~256个
│   └── Shared Memory（高速共享内存）
│   └── 寄存器文件（海量寄存器）
├── L2 Cache（全局缓存）
└── 显存（GDDR6/HBM，高带宽）
```

> **这些层级分别是做什么的？**
>
> 不要被缩写吓到，这张图的每一层都有明确分工：
>
> - **SM（Streaming Multiprocessor）**：GPU 的"调度车间"。每个 SM 同时管理几十组线程，核心职责就是**延迟隐藏**——当一组线程去显存取数据卡住了，SM 立刻把另一组准备好的线程顶上来，让核心永不空闲。这些线程组叫做 **Warp**（马上讲到）。
>
> - **CUDA Core / Shader Core**：SM 里的"算术工人"，真正执行乘加运算的小核心。它们非常"笨"——没有复杂的分支预测、没有乱序执行，只会按统一指令做同样的计算，但胜在数量极多。
>
> - **Shared Memory**：每个 SM 内部的一块**超高速小内存**，比显存快几十倍，但通常只有几十到几百 KB。同一个 SM 里的线程可以通过它快速共享数据（比如一小块纹理），不用每次都去远处的显存取。
>
> - **寄存器文件**：每个线程私有的"随身工具箱"，存放临时变量。GPU 为了同时维护几千个活跃线程的状态，配备了海量的寄存器——这是 SM 能实现**零成本 Warp 切换**的硬件前提。
>
> - **L2 Cache / 显存**：L2 是整颗芯片共享的全局缓存，显存则是高带宽的大容量外部存储，用来存放纹理、[[Notes/计算机图形学/GPU编程基础/OpenGL状态机模型#VBO：Vertex Buffer Object（顶点缓冲对象）|顶点缓冲]]、[[Notes/计算机图形学/GPU编程基础/OpenGL状态机模型#FBO：Framebuffer Object（帧缓冲对象）|帧缓冲]]等数据。SM 去显存取数需要上百个时钟周期，这也是延迟隐藏机制存在的原因。

**关键数字对比**：

| 特性    | 高端CPU (i9) | 高端GPU (RTX 4090) |
| ----- | ---------- | ---------------- |
| 核心数   | 24核        | 16384个CUDA核      |
| 内存带宽  | ~100 GB/s  | ~1000 GB/s       |
| 单线程性能 | 极高         | 较低               |
| 适合的任务 | 串行、复杂分支    | 并行、相同计算          |

### 为什么 GPU 能放这么多核心？

因为GPU核心**简单**。它们共享指令解码器、共享控制逻辑。一组核心（通常是32个）执行**完全相同的指令**，只是操作不同的数据。

**代价**：如果一个核心需要走分支A，另一个核心需要走分支B，它们不能同时执行——必须串行执行两个分支。这在后面讲Shader时会详细展开。

### Warp 是什么？先剧透一下

上面反复提到"32个核心一组"执行相同指令，这个"组"在 NVIDIA GPU 里有个专门的名字：**Warp**。

你可以把 Warp 理解成 GPU 调度的**最小车队**——32辆小车（线程）排成一队，司机（SM）给它们发同一张地图（同一指令），它们在每一步都走同一条路、做同一件事，只是各自运送的货物（数据）不同。

> 所以“同一件事”指的是**同一时刻执行同一条指令**，不是“所有线程的计算结果永远相同”。数据不同，结果自然可能不同。

当车队遇到一个岔路口（分支）时，麻烦来了：有些小车该左转，有些该右转。Warp 里只有一张地图、一个方向盘，所以不能一半左转一半右转同时走。GPU 的做法是**先让该左转的车走，右转的车原地待命；再让右转的车走，左转的车原地待命**。看起来有一辆车“做了不同的事”，但实际上整个车队在任何一个瞬间仍然执行的是同一个指令，只是不同线程因为数据不同被临时“跳过”了。

如果某辆车去仓库取货（访问显存）而卡住，司机不会让整个车队干等，而是立刻换另一支车队上路——这就是后面的延迟隐藏机制。

> 为什么偏偏是 32 个？这是 NVIDIA 硬件的固定宽度，就像 CPU 的缓存行固定是 64 字节一样，属于硬件设计约定。AMD GPU 的对应概念叫 **Wavefront**，通常是 64 个线程一组。

Warp 的具体执行机制（SIMT 模型）和"一辆车卡住、全队切换"的延迟隐藏魔法，我们在下一个问题里详细展开。

## 问题 3：GPU 怎么执行 Shader 程序？

### 从 CPU 思维切换到 GPU 思维

在软渲染器里，你写了一个 `rasterizeTriangle()` 函数，CPU按顺序执行：

```cpp
// CPU 思维：按顺序处理
for (每个三角形) {
    for (每个像素) {
        computeColor();  // 一个函数搞定一切
    }
}
```

在GPU上，这个逻辑被拆分成两种**Shader程序**：

1. **[[Notes/计算机图形学/GPU编程基础/第一个三角形#顶点着色器（Vertex Shader）|顶点着色器（Vertex Shader）]]**：对每个顶点执行一次
2. **[[Notes/计算机图形学/GPU编程基础/第一个三角形#片段着色器（Fragment Shader）|片段着色器（Fragment Shader）]]**：对每个像素（片段）执行一次

### SIMT 执行模型

GPU使用 **[[Notes/游戏引擎杂谈碎片/游戏引擎渲染系统#SIMD vs SIMT|SIMT]]（Single Instruction, Multiple Threads，单指令多线程）** 模型。

**Warp**：NVIDIA GPU中，32个线程组成一个 **Warp**，它们共享一个程序计数器（PC），执行完全相同的指令。

```
Warp = 32个线程

时钟周期 1：所有32个线程同时执行"加载顶点位置"
时钟周期 2：所有32个线程同时执行"矩阵乘法"
时钟周期 3：所有32个线程同时执行"写回结果"

32个线程 × 32个核心 = 一步处理32个顶点/像素
```

> AMD GPU 的对应概念叫 **Wavefront**，通常是64个线程一组。不同厂商的宽度不同，但原理一致。

### 延迟隐藏（Latency Hiding）—— GPU 的"魔法"

GPU的核心虽然简单，但有一个绝妙的机制：**当一个Warp在等待内存数据时，马上切换到另一个Warp执行**。

```
Warp A：执行 → 请求纹理采样（等待100个周期）→ 挂起
  ↓ 切换（零开销）
Warp B：执行 → 请求内存读取（等待200个周期）→ 挂起
  ↓ 切换
Warp C：执行 → ...
```

每个SM可以同时管理几十个Warp。只要有足够的Warp在"排队"，GPU的核心就**永远不会空闲**——一个Warp卡住了，下一个立刻顶上。

**Little's Law**：并行度 = 吞吐量 × 延迟。换句话说，如果一次内存访问要等 100 个周期，而每个周期都想完成 1 个操作，那至少要有 100 个操作在同时"飞行"才能把延迟盖住。GPU正是通过海量线程来提供这个并行度。

> CPU也有超线程（Hyper-Threading），但只支持2个线程共享一个核心。GPU的一个SM可以管理64+个Warp，每个Warp 32个线程，总并发度超过2000。

## 问题 4：为什么三角形是 GPU 的基本图元？

你可能会问：既然GPU这么擅长并行，为什么不用四边形、多边形，偏偏是三角形？

答案可以总结成一句话：**三角形是“几何原子”——它永远平整、内外判断简单、能拼出任意形状，而且 GPU 的固定硬件就是为它量身定做的。**

**第一，三角形天然保证平面性。** 三个点一定共面，所以三角形在 2D 屏幕上永远是一个平整的形状。四边形就不同了：四个顶点可能不在同一平面上，渲染时容易出现裂缝或扭曲。

**第二，判断像素是否在三角形内非常简单。** 软渲染器里用的**边缘函数**本质上是叉乘，而叉乘天然只对三角形有意义。四边形的内外判断要复杂得多，硬件做起来不划算。

**第三，任何多边形都能拆成三角形。** 复杂多边形可能有凹角、自相交，但总能通过**三角化（Triangulation）**拆成一组三角形：

```
四边形 → 2个三角形
五边形 → 3个三角形
复杂模型 → 几千个三角形
```

**第四，GPU 的光栅化器是固定功能硬件。** 它并行计算三条边的边缘函数、并行插值深度/颜色/纹理坐标、硬件做 Z-Test——这些单元从设计之初就假设输入是三角形。如果换成四边形，整套光栅化硬件都要重新设计。

## 问题 5：那软渲染器的代码对应 GPU 的什么？

把软渲染器的逻辑映射到GPU管线，能帮你理解后续OpenGL API的设计：

| 软渲染器（CPU代码）                    | GPU 对应物                      | 是否可编程       |
| ------------------------------ | ---------------------------- | ----------- |
| `for(每个顶点)` + MVP矩阵乘法          | **顶点着色器（Vertex Shader）**     | ✅ 可编程       |
| 三角形组装、裁剪、透视除法                  | **图元装配（Primitive Assembly）** | ❌ 固定硬件      |
| `for(每个像素)` + 边缘函数             | **光栅化器（Rasterizer）**         | ❌ 固定硬件      |
| 透视校正插值                         | **插值器（Interpolator）**        | ❌ 固定硬件      |
| `computeColor()` + Blinn-Phong | **片段着色器（Fragment Shader）**   | ✅ 可编程       |
| Z-Buffer 深度测试                  | **深度测试单元（Z-Test）**           | ❌ 固定硬件（可配置） |
| 写入颜色缓冲                         | **ROP（Render Output Unit，渲染输出单元）**  | ❌ 固定硬件      |

**核心洞察**：

你在软渲染器里写的所有代码，在GPU上被拆成两半——
- **可编程部分**：顶点着色器、片段着色器（你写GLSL代码）
- **固定硬件部分**：光栅化、深度测试、输出合并（GPU自动做，你无法改算法）

> 这意味着：GPU上的渲染不是"用更快的CPU跑同样的代码"，而是"把代码拆成两块小Shader，中间由专用硬件处理"。
