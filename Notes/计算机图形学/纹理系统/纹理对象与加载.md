# 纹理对象与加载

## 问题 0：UV 坐标有了，但图片还在硬盘里

在 [[Notes/计算机图形学/纹理系统/纹理映射与UV坐标|纹理映射与UV坐标]] 里，我们为每个顶点配了一个 UV 坐标。但到目前为止，三角形内部采样时拿到的"纹理图片"还是抽象概念——它真实地躺在硬盘上的 `.png` 或 `.jpg` 文件里。

**现在要解决的具体问题**：
- 怎么把 PNG 文件解码成内存里的像素数组？
- 怎么把这个数组搬到 GPU 显存里？
- Shader 里的 `sampler2D` 怎么才能真的读到这张图片？

---

## 问题 1：PNG 文件怎么读？

### 最 naive 的方案：自己写一个 PNG 解码器

PNG 是一种压缩格式，涉及 zlib 解压、滤波器重建、IHDR/IDAT/IEND 等 chunk 解析。理论上你可以自己写，但这意味着：

- 光是 PNG 就要写几百行健壮代码
- 还要支持 JPG、TGA、BMP 吗？
- 美术每换一种格式，你就要重写一篇解码器

**发现的问题**：写解码器这件事和"学图形学"完全无关，却极其耗时。

### 改进：用单头文件库 stb_image

**stb_image** 是 Sean Barrett 写的著名单头文件图像加载库。使用方式非常直接：

```cpp
#define STB_IMAGE_IMPLEMENTATION
#include <stb_image.h>

int width, height, channels;
unsigned char* pixels = stbi_load("crate.png", &width, &height, &channels, 0);
if (!pixels) {
    std::cerr << "Failed to load image: " << stbi_failure_reason() << std::endl;
}
```

`stbi_load` 会把各种格式的图片都解码成统一的内存布局：

```
像素数据在内存中是按行排列的：

Row 0: R G B A | R G B A | R G B A | ...  (width 个像素)
Row 1: R G B A | R G B A | R G B A | ...
...
Row H-1: ...
```

- `channels` 告诉你每个像素有几个字节（3 表示 RGB，4 表示 RGBA）
- 返回的指针指向 CPU 内存中的像素缓冲区

> **关键认知**：stb_image 解出来的是**CPU 内存里的原始字节**。它还没进 GPU，Shader 还读不到。

## 问题 2：像素数据在 CPU 内存里，GPU 怎么访问？

### 最 naive 的方案：每次绘制都重新加载并上传

```cpp
void draw() {
    unsigned char* pixels = stbi_load("crate.png", &w, &h, &c, 0);
    // ... 马上把 pixels 传给 GPU ...
    stbi_image_free(pixels);
}
```

**发现的问题**：
- 每帧都从硬盘读文件 → IO 瓶颈
- 每帧都重新解码 → CPU 爆炸
- 每帧都做 CPU→GPU 拷贝 → 总线带宽浪费

这就好比你每次画画都要重新去书店买一本画册，而不是把画册放在手边。

### 改进：把像素数据常驻 GPU 显存

和 VBO 解决"顶点数据每次上传"的思路一样，纹理也需要一个**GPU 端的对象**来长期保存像素数据。OpenGL 里这个对象就叫**纹理对象（Texture Object）**。

创建纹理对象的流程和 VBO 几乎对称：

```cpp
GLuint texture;
glGenTextures(1, &texture);        // 1. 申请一个纹理对象 ID
glBindTexture(GL_TEXTURE_2D, texture); // 2. 把它绑定到 GL_TEXTURE_2D 目标

// 3. 上传像素数据
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, width, height,
             0, GL_RGBA, GL_UNSIGNED_BYTE, pixels);

// 4. 释放 CPU 端的 stb_image 内存（数据已经在 GPU 上了）
stbi_image_free(pixels);
```

**状态变化图**：

```
上传前：
  CPU 内存：pixels[]  ← 可读可写
  GPU 显存：无纹理数据

调用 glTexImage2D 后：
  CPU 内存：pixels[]  ← 已经没用，可以 free
  GPU 显存：texture object ← 保存了一份像素拷贝
```

> **重要**：`glTexImage2D` 会做一次**显式的 CPU→GPU 内存拷贝**。上传完成后，CPU 端的 `pixels` 就可以释放。纹理数据从此常驻 GPU 显存，绘制时不再重复上传。

## 问题 3：glTexImage2D 参数这么多，每个到底是什么？

这是初学者最容易望而生畏的 API：

```cpp
void glTexImage2D(
    GLenum  target,      // GL_TEXTURE_2D
    GLint   level,       // 0 = 基础层（mipmap 层级，后面讲）
    GLint   internalformat, // GPU 内部存储格式：GL_RGB, GL_RGBA, GL_RGB8 ...
    GLsizei width,       // 图片宽度
    GLsizei height,      // 图片高度
    GLint   border,      // 必须为 0（历史遗留）
    GLenum  format,      // 你传入的 pixels 是什么格式：GL_RGB / GL_RGBA
    GLenum  type,        // 像素分量的数据类型：GL_UNSIGNED_BYTE
    const void* pixels   // CPU 像素数据指针
);
```

### 内部格式 vs 源格式

| 参数 | 含义 | 例子 |
|------|------|------|
| `internalformat` | GPU 显存里怎么存这张纹理 | `GL_RGBA`, `GL_RGB8`, `GL_RGBA16F` |
| `format` | 你传入的 `pixels` 数组是什么布局 | `GL_RGB`, `GL_RGBA`, `GL_BGR` |
| `type` | 每个通道用多少位 | `GL_UNSIGNED_BYTE`（8bit）, `GL_FLOAT` |

**常见错误**：内部格式写 `GL_RGB`，源格式写 `GL_RGBA`。这会导致 OpenGL 把你传过去的 4 字节数据按 3 字节解析，颜色全错。

正确的匹配：

```cpp
// 3 通道图片
if (channels == 3) {
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB, width, height,
                 0, GL_RGB, GL_UNSIGNED_BYTE, pixels);
}
// 4 通道图片
else if (channels == 4) {
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, width, height,
                 0, GL_RGBA, GL_UNSIGNED_BYTE, pixels);
}
```

## 问题 4：纹理画出来上下颠倒了？

这是 99% 初学者会踩的坑。

**原因**：图片文件格式（PNG/JPG）的像素原点通常在**左上角**，而 OpenGL 的纹理坐标 `(0,0)` 在**左下角**。

```
图片文件内存布局：          OpenGL 纹理坐标期望：

(0,0) ───────→ u            v↑
  │  图片顶部                  │
  │                            │
  │  图片底部                 0.0└────────┘→ u
  ↓ v                         (0,0) 左下角
```

### 方案 A：加载时翻转（推荐）

```cpp
stbi_set_flip_vertically_on_load(true);
unsigned char* pixels = stbi_load("crate.png", &width, &height, &channels, 0);
```

### 方案 B：用 OpenGL 的 pixel store 状态翻转

```cpp
glPixelStorei(GL_UNPACK_FLIP_Y, GL_TRUE);
glTexImage2D(...);
glPixelStorei(GL_UNPACK_FLIP_Y, GL_FALSE); // 用完恢复
```

> **推荐方案 A**：在资源导入阶段就处理好方向，不要依赖全局 OpenGL 状态。这在引擎资源管线里更容易维护。

## 问题 5：纹理颜色错位，出现对角线？

这是另一个经典陷阱，通常发生在加载**RGB（3 通道）**图片时。

**原因**：OpenGL 默认假设每一行像素的起始地址是 4 字节对齐的（`GL_UNPACK_ALIGNMENT` 默认值为 4）。如果图片宽度 × 3 字节不是 4 的倍数，OpenGL 会按 4 字节对齐读取，导致下一行像素偏移出错。

### 例子

一张宽度为 401 像素的 RGB 图片：

```
每行实际字节数：401 × 3 = 1203 字节
OpenGL 按 4 字节对齐后读取：1204 字节（多补 1 字节）
结果：每一行都多读 1 字节，颜色沿对角线错位
```

### 解决方案

在调用 `glTexImage2D` 之前，把 unpack 对齐设为 1：

```cpp
glPixelStorei(GL_UNPACK_ALIGNMENT, 1);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB, width, height,
             0, GL_RGB, GL_UNSIGNED_BYTE, pixels);
```

> **工业建议**：如果你控制图片格式，**优先使用 RGBA（4 通道）**。4 字节天然对齐，性能也更好（GPU 更喜欢 32 位像素）。RGB 只在显存特别紧张时使用。

## 问题 6：纹理上传到 GPU 后，Shader 怎么才能读到？

现在 GPU 显存里有一张纹理，但 Shader 不会自动使用它。你需要完成三个动作：

1. **激活一个纹理单元（Texture Unit）**
2. **把纹理对象绑定到该纹理单元**
3. **告诉 Shader 的 `sampler2D` 从哪个纹理单元采样**

### 代码

```cpp
// 1. 激活纹理单元 0
glActiveTexture(GL_TEXTURE0);

// 2. 把 texture 对象绑定到 GL_TEXTURE_2D 目标（也就是纹理单元 0）
glBindTexture(GL_TEXTURE_2D, texture);

// 3. 告诉 shader：uniform sampler2D uTexture 使用纹理单元 0
GLint location = glGetUniformLocation(shaderProgram, "uTexture");
glUniform1i(location, 0);
```

### 状态变化图

```
绑定前：
  纹理单元 0：空
  Shader uTexture：未定义

glActiveTexture(GL_TEXTURE0) 后：
  当前活跃纹理单元 = 0

glBindTexture(GL_TEXTURE_2D, texture) 后：
  纹理单元 0 ──→ texture 对象

glUniform1i(location, 0) 后：
  Shader uTexture ──→ 纹理单元 0 ──→ texture 对象
```

### 对应的片段着色器

```glsl
#version 330 core
in vec2 vTexCoord;
uniform sampler2D uTexture;
out vec4 FragColor;

void main() {
    FragColor = texture(uTexture, vTexCoord);
}
```

> **关键认知**：`sampler2D` 在 GLSL 里是一个 **uniform**，它的值不是颜色数据，而是**纹理单元的索引**。真正的像素数据在纹理单元绑定的纹理对象里。

## 问题 7：纹理对象和 VBO 有什么异同？

| 对象 | 创建 | 绑定目标 | 上传数据 | 用途 |
|------|------|----------|----------|------|
| VBO | `glGenBuffers` | `GL_ARRAY_BUFFER` | `glBufferData` | 存顶点数据 |
| 纹理对象 | `glGenTextures` | `GL_TEXTURE_2D` | `glTexImage2D` | 存像素数据 |

两者都是 OpenGL 状态机中的"对象"：
- 先生成 ID
- 再绑定到某个目标
- 后续对该目标的操作都作用于当前绑定的对象
- 绘制前需要确保正确的对象处于绑定状态

> **和 VBO 不一样的地方**：纹理对象还多了一层**纹理单元**的概念。VBO 绑定后 Shader 通过 `layout(location = ...)` 直接访问；纹理则需要通过 `glActiveTexture + glBindTexture + uniform sampler2D` 三层间接才能访问。

## 问题 8：纹理对象的生命周期怎么管理？

和 VBO 一样，纹理对象需要显式删除：

```cpp
glDeleteTextures(1, &texture);
```

在引擎里，纹理对象通常被包装成 `TextureHandle` 或 `GpuTexture` 资源：

```cpp
struct GpuTexture {
    GLuint id = 0;
    int width = 0;
    int height = 0;
    int channels = 0;

    ~GpuTexture() {
        if (id != 0) glDeleteTextures(1, &id);
    }
};
```

## 与 SelfGameEngine 的关系

这篇笔记对应引擎 **阶段 5.2 资源管理** 中的 **Texture Import → GPU Upload** 链路。

在引擎里，这条链路会被抽象成：

```
文件系统 (VFS)
    │
    ▼
图片文件 (.png/.jpg) ──→ stb_image 解码 ──→ CPU 像素缓冲区
    │
    ▼
TextureImporter（资源导入器）
    │
    ▼
GPU Upload ──→ GpuTexture { id, width, height, format }
    │
    ▼
材质系统：把 GpuTexture 绑定到 Shader 的某个 sampler2D slot
```

你现在写的这段 OpenGL 代码，就是引擎里 `TextureImporter::Import()` 和 `RenderDevice::CreateTexture()` 的底层实现原型。

## 完整可运行示例

```cpp
// flags: -O0 -g -lGL -lGLEW -lglfw
#include <GL/glew.h>
#include <GLFW/glfw3.h>
#include <iostream>

#define STB_IMAGE_IMPLEMENTATION
#include <stb_image.h>

GLuint loadTexture(const char* path) {
    stbi_set_flip_vertically_on_load(true);

    int width, height, channels;
    unsigned char* pixels = stbi_load(path, &width, &height, &channels, 0);
    if (!pixels) {
        std::cerr << "Failed to load texture: " << path << std::endl;
        return 0;
    }

    GLuint texture;
    glGenTextures(1, &texture);
    glBindTexture(GL_TEXTURE_2D, texture);

    // 设置纹理参数（过滤和寻址将在后续笔记详细展开）
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_REPEAT);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

    // 解决 RGB 非 4 字节对齐问题
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1);

    GLenum internalFormat = (channels == 4) ? GL_RGBA : GL_RGB;
    GLenum sourceFormat   = (channels == 4) ? GL_RGBA : GL_RGB;

    glTexImage2D(GL_TEXTURE_2D, 0, internalFormat, width, height,
                 0, sourceFormat, GL_UNSIGNED_BYTE, pixels);

    stbi_image_free(pixels);
    return texture;
}

int main() {
    // ... 初始化 GLFW、GLEW、Shader、VAO ...

    GLuint texture = loadTexture("crate.png");

    // 渲染循环中
    glActiveTexture(GL_TEXTURE0);
    glBindTexture(GL_TEXTURE_2D, texture);
    glUniform1i(glGetUniformLocation(shaderProgram, "uTexture"), 0);

    glBindVertexArray(vao);
    glDrawElements(GL_TRIANGLES, 6, GL_UNSIGNED_INT, 0);

    // ... 清理 ...
    glDeleteTextures(1, &texture);
    return 0;
}
```
