# 3D光栅化与深度缓冲

## 问题 0：MVP 变换后，三角形怎么变成屏幕像素？

在 [[Notes/计算机图形学/软光栅化与3D数学/视图与投影|视图与投影]] 中，我们用 MVP 矩阵把 3D 顶点变换到了裁剪空间，再经过透视除法得到了 NDC（标准化设备坐标）`[-1, 1]³`。

但显示器是 2D 的像素阵列。从 NDC 到屏幕像素，还需要几个步骤：

1. **视口变换**：NDC `[-1, 1]²` → 屏幕像素 `[0, width] × [0, height]`
2. **光栅化**：判断哪些屏幕像素被三角形覆盖
3. **深度测试**：同一个像素可能被多个三角形覆盖，哪个在前面？

---

## 问题 1：视口变换——从 NDC 到屏幕像素

NDC 是一个与分辨率无关的标准空间。要映射到具体的屏幕分辨率，需要**视口变换**：

```cpp
// NDC [-1, 1] → 屏幕 [0, width] × [0, height]
screenX = (ndc.x + 1) * 0.5 * width;
screenY = (ndc.y + 1) * 0.5 * height;
```

**像素中心 vs 像素角点**：

```
像素 (x, y) 的覆盖区域：
┌─────────────┐
│  (x,y+1)    │
│     ●       │ ← 像素中心：(x+0.5, y+0.5)
│  (x,y)      │
└─────────────┘
```

光栅化时，我们在**像素中心点** `(x+0.5, y+0.5)` 采样，判断它是否在三角形内部。

## 问题 2：3D 光栅化和 2D 有什么不同？

在 [[Notes/计算机图形学/软光栅化与3D数学/光栅化：从直线到三角形|光栅化：从直线到三角形]] 中，我们实现了 2D 三角形光栅化。3D 光栅化的核心区别：

### 区别 1：需要考虑深度（Z 值）

3D 场景中有前后遮挡关系。一个像素可能被多个三角形覆盖，必须知道哪个三角形在最前面。

### 区别 2：透视校正插值

在 2D 中，属性（颜色、纹理坐标）可以在屏幕空间**线性插值**。但在 3D 透视投影后，屏幕空间的线性插值是**错误的**。

**为什么？**

透视投影把远处的物体压缩了。屏幕空间中"等距"的两点，在 3D 世界中可能距离不同。直接在屏幕空间线性插值纹理坐标，会导致远处纹理拉伸变形。

**正确的做法**：透视校正插值。纹理坐标 `u` 不能线性插，要插值 `u/z`，然后再除以 `1/z`。

```
屏幕空间线性插值：❌ u = (1-t) * u0 + t * u1
透视校正插值：    ✅ u = [(1-t) * u0/z0 + t * u1/z1] / [(1-t) * 1/z0 + t * 1/z1]
```

> 现代 GPU 的光栅化器自动处理透视校正插值。你在 Shader 中看到的 `varying` 变量，硬件会自动做透视校正。

## 问题 3：怎么处理遮挡？——Z-Buffer

### 最 naive 的方案：画家算法

"先画远处的物体，再画近处的物体，近处的覆盖远处的。"

**发现的问题**：
- 需要按深度排序所有三角形，排序开销大
- 循环重叠的三角形（A 在 B 前面，B 在 C 前面，C 在 A 前面）无法正确排序

### 改进：Z-Buffer（深度缓冲）

**核心思想**：为每个像素维护一个"当前最小深度值"，绘制时只更新深度更小的像素。

```cpp
// 初始化：所有像素的深度为无穷远
float zBuffer[width * height];
for (int i = 0; i < width * height; i++) zBuffer[i] = INFINITY;

// 绘制每个三角形时
for (每个像素 in 三角形覆盖区域) {
    float z = 插值后的深度值;  // 透视校正插值得到
    int idx = y * width + x;
    if (z < zBuffer[idx]) {     // 这个像素比之前绘制的更近
        zBuffer[idx] = z;       // 更新深度
        colorBuffer[idx] = color; // 更新颜色
    }
}
```

**Z-Buffer 的优势**：
- 不需要排序三角形
- 每个三角形独立绘制，与绘制顺序无关
- 硬件原生支持（现代 GPU 的 ROP 单元）

**Z-Buffer 的精度问题**：

透视投影后，Z 值的分布不是线性的——近处精度高，远处精度低。这意味着两个远处物体的深度差可能小于 Z-Buffer 的精度粒度，导致**Z-Fighting**（闪烁）。

```
透视投影后的 Z 分布：
近平面 ←──高精度──┼──低精度──→ 远平面
```

> 缓解方案：使用对数深度缓冲、增大近平面距离、使用更高精度的深度格式（24位或32位浮点）。

## 问题 4：为什么光栅化会产生锯齿？

在 [[Notes/计算机图形学/软光栅化与3D数学/光栅化：从直线到三角形|光栅化：从直线到三角形]] 中，我们用边缘函数判断像素中心点是否在三角形内部。这种"非黑即白"的判断方式，导致边缘呈现阶梯状：

```
理想直线：        光栅化结果：
    ↗              ■
   ↗              ■■
  ↗              ■■■
 ↗              ■■■■
```

**核心问题**：一个三角形是连续的几何形状，但屏幕像素是离散的格子。当几何边缘穿过像素时，我们只能选择"画"或"不画"，无法表达"画一半"。

### 走样的本质：采样率不足

光栅化本质上是**采样**：用离散的像素网格去采样连续的几何信号。

**走样（Aliasing）** 的本质：**采样率低于信号变化的频率**。

```
高频信号（锐利边缘）
    /\  /\  /\  /\      ← 变化快
   /  \/  \/  \/  \
  
低频采样（像素网格）
  ●     ●     ●     ●   ← 采样点太少
  
结果：无法区分高频信号和低频信号（混叠）
```

### 采样瑕疵的类型

| 类型 | 描述 | 例子 |
|-----|------|------|
| **锯齿（Jaggies）** | 边缘的阶梯状效果 | 斜线、三角形边缘 |
| **摩尔纹（Moire）** | 高频模式与采样频率相近时产生的干涉纹样 | 远处密集的栅栏、条纹衣服 |
| **闪烁（Shimmering）** | 微小运动时采样点变化导致颜色跳变 | 远处纹理、细小几何体 |

---

## 问题 5：怎么消除走样？

### 核心原则：先滤波，再采样

**错误的顺序**：先采样 → 再模糊（无法消除已产生的混叠）

**正确的顺序**：先滤波（去除高频） → 再采样

```
几何信号 ──[低通滤波]──→ 平滑信号 ──[采样]──→ 像素
         （去掉尖锐边缘）          （无混叠）
```

但问题是：滤波在数学上需要对连续信号做卷积积分，在实时渲染中无法直接实现。我们需要**近似的、高效的**抗锯齿方案。

## 问题 6：MSAA——多重采样抗锯齿

### 最 naive 的方案：提高分辨率

把渲染分辨率提高 4 倍（如 4K），再缩小到屏幕分辨率。每个最终像素由 4 个子采样平均得到。

**发现的问题**：像素着色器要执行 4 倍次数，性能开销巨大。

### 改进：MSAA（Multi-Sample Anti-Aliasing）

MSAA 的关键洞察：**锯齿只出现在边缘**。三角形内部的像素不需要多采样——它们完全被一个三角形覆盖。

```
MSAA 4x：每个像素分成 4 个子采样点

┌───────────┐
│  ○    ○   │  ○ = 子采样点（判断是否在三角形内）
│           │
│  ○    ○   │
└───────────┘

边缘像素：4 个子采样点分别判断覆盖率
内部像素：1 次着色，结果复制到 4 个子采样
```

**工作流程**：
1. 光栅化阶段：对每个子采样点判断是否在三角形内（只计算覆盖率，不计算颜色）
2. 像素着色阶段：只在像素中心执行一次着色计算
3. 混合阶段：根据子采样点的覆盖率，将着色结果写入对应的子采样
4. 解析阶段：将子采样平均，得到最终像素颜色

**MSAA 的代价**：
- 颜色缓冲区和深度缓冲区需要 4 倍内存
- 边缘像素的带宽增加
- 与延迟渲染（Deferred Shading）不兼容（因为 G-Buffer 无法存储多采样数据）

## 问题 7：FXAA——快速近似抗锯齿

### 核心思想：后处理检测边缘

MSAA 需要在光栅化阶段做多采样，开销较大。FXAA 完全在**后处理阶段**工作：

1. 先渲染出带有锯齿的图像
2. 通过图像处理检测边缘（寻找亮度梯度大的区域）
3. 对检测到的边缘像素做模糊混合

```cpp
// FXAA 的核心逻辑（极度简化版）
Color FXAA(sampler2D tex, vec2 uv) {
    // 采样当前像素和周围 4 个邻居的亮度
    float lumaCenter = luminance(texture(tex, uv));
    float lumaN = luminance(texture(tex, uv + vec2(0, -1) * texelSize));
    float lumaS = luminance(texture(tex, uv + vec2(0,  1) * texelSize));
    float lumaE = luminance(texture(tex, uv + vec2( 1, 0) * texelSize));
    float lumaW = luminance(texture(tex, uv + vec2(-1, 0) * texelSize));
    
    // 计算亮度梯度
    float range = max(lumaCenter, max(max(lumaN, lumaS), max(lumaE, lumaW))) -
                  min(lumaCenter, min(min(lumaN, lumaS), min(lumaE, lumaW)));
    
    // 如果梯度小于阈值，不是边缘，直接返回
    if (range < threshold) return texture(tex, uv);
    
    // 是边缘：沿垂直于边缘的方向混合
    // ... 混合逻辑 ...
}
```

**FXAA 的优势**：
- 速度极快，几乎没有额外性能开销
- 不需要多采样缓冲区，内存占用小
- 兼容延迟渲染

**FXAA 的劣势**：
- 可能引入图像模糊
- 对低对比度边缘检测效果不佳
- 有"拖影"感

## 问题 8：TAA——时间性抗锯齿

### 核心思想：把采样分布在时间维度上

MSAA 在一个像素内放多个采样点（空间多采样）。TAA 把采样分布在**多帧**上（时间多采样）：

```
第 N 帧：像素中心偏移 (-0.25, -0.25)
第 N+1 帧：像素中心偏移 ( 0.25, -0.25)
第 N+2 帧：像素中心偏移 (-0.25,  0.25)
第 N+3 帧：像素中心偏移 ( 0.25,  0.25)

4 帧混合 ≈ 4x MSAA 的效果
```

**TAA 的工作流程**：
1. 每帧渲染时，对投影矩阵做一个微小的子像素抖动（Jitter）
2. 当前帧渲染完成后，与历史帧混合
3. 使用运动向量（Motion Vector）将历史帧的像素投影到当前帧位置
4. 混合权重根据运动速度调整（运动快的像素减少历史帧权重，避免拖影）

**TAA 的优势**：
- 效率极高，几乎等同于单帧渲染的开销
- 同时抗锯齿、降噪、软化阴影
- 适合实时渲染

**TAA 的劣势**：
- 运动物体会产生鬼影（Ghosting）
- 需要运动向量支持
- 画面有轻微模糊感

---

## 三种抗锯齿方案对比

| 方案 | 原理 | 性能 | 画质 | 适用场景 |
|------|------|------|------|---------|
| **MSAA** | 空间多采样 | 中等 | 最好 | 前向渲染、高质量需求 |
| **FXAA** | 后处理边缘检测 | 极高 | 一般 | 性能敏感、移动端 |
| **TAA** | 时间多采样 | 高 | 好 | 现代游戏主流方案 |

> 现代游戏通常使用 **TAA + FXAA（备用）** 的组合，或在需要极高画质时使用 **MSAA + TAA**。
