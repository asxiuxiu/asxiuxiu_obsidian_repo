## 问题 0：MVP 变换后，三角形怎么变成屏幕像素？

在 [[Notes/计算机图形学/软光栅化与3D数学/视图与投影|视图与投影]] 中，我们用 MVP 矩阵把 3D 顶点变换到了裁剪空间，再经过透视除法得到了 NDC（标准化设备坐标）`[-1, 1]³`。

但显示器是 2D 的像素阵列。从 NDC 到屏幕像素，还需要几个步骤：

1. **视口变换**：NDC `[-1, 1]²` → 屏幕像素 `[0, width] × [0, height]`
2. **光栅化**：判断哪些屏幕像素被三角形覆盖
3. **深度测试**：同一个像素可能被多个三角形覆盖，哪个在前面？

用一句话串起来就是：**先把顶点投影到标准盒子里，再把它铺到屏幕上，最后决定谁挡在谁前面。**

---

## 问题 1：视口变换——从 NDC 到屏幕像素

NDC 是一个与分辨率无关的标准空间。要映射到具体的屏幕分辨率，需要**视口变换**：

```cpp
// NDC [-1, 1] → 屏幕 [0, width] × [0, height]
screenX = (ndc.x + 1) * 0.5 * width;
screenY = (ndc.y + 1) * 0.5 * height;
```

> [!example] 类比：NDC 就像一个标准集装箱
> 不管你的画（3D 场景）原本多大、比例如何，投影矩阵都会把它装进一个 `[-1, 1]³` 的标准盒子。快递公司（GPU）只认这个盒子；到了目的地，再根据墙面大小（屏幕分辨率）把画展开。盒子的尺寸固定，但画的比例信息已经通过打包方式保留了。

**像素中心 vs 像素角点**：

```
像素 (x, y) 的覆盖区域：
┌─────────────┐
│  (x,y+1)    │
│     ●       │ ← 像素中心：(x+0.5, y+0.5)
│  (x,y)      │
└─────────────┘
```

光栅化时，我们在**像素中心点** `(x+0.5, y+0.5)` 采样，判断它是否在三角形内部。

## 问题 2：3D 光栅化和 2D 有什么不同？

在 [[Notes/计算机图形学/软光栅化与3D数学/光栅化：从直线到三角形|光栅化：从直线到三角形]] 中，我们实现了 2D 三角形光栅化。3D 光栅化的核心区别：

### 区别 1：需要考虑深度（Z 值）

3D 场景中有前后遮挡关系。一个像素可能被多个三角形覆盖，必须知道哪个三角形在最前面。

### 区别 2：透视校正插值

在 2D 中，属性（颜色、纹理坐标）可以在屏幕空间**线性插值**。但在 3D 透视投影后，屏幕空间的线性插值是**错误的**。

**为什么？**

透视投影把远处的物体压缩了。屏幕空间中"等距"的两点，在 3D 世界中可能距离不同。直接在屏幕空间线性插值纹理坐标，会导致远处纹理拉伸变形。

**正确的做法**：透视校正插值。纹理坐标 `u` 不能线性插，要插值 `u/z`，然后再除以 `1/z`。

```
屏幕空间线性插值：❌ u = (1-t) * u0 + t * u1
透视校正插值：    ✅ u = [(1-t) * u0/z0 + t * u1/z1] / [(1-t) * 1/z0 + t * 1/z1]
```

> 现代 GPU 的光栅化器自动处理透视校正插值。你在 Shader 中看到的 `varying` 变量，硬件会自动做透视校正。

## 问题 3：怎么处理遮挡？——Z-Buffer

### 最 naive 的方案：画家算法

"先画远处的物体，再画近处的物体，近处的覆盖远处的。"

**发现的问题**：
- 需要按深度排序所有三角形，排序开销大
- 循环重叠的三角形（A 在 B 前面，B 在 C 前面，C 在 A 前面）无法正确排序

### 改进：Z-Buffer（深度缓冲）

**核心思想**：为每个像素维护一个"当前最小深度值"，绘制时只更新深度更小的像素。

```cpp
// 初始化：所有像素的深度为无穷远
float zBuffer[width * height];
for (int i = 0; i < width * height; i++) zBuffer[i] = INFINITY;

// 绘制每个三角形时
for (每个像素 in 三角形覆盖区域) {
    float z = 插值后的深度值;  // 透视校正插值得到
    int idx = y * width + x;
    if (z < zBuffer[idx]) {     // 这个像素比之前绘制的更近
        zBuffer[idx] = z;       // 更新深度
        colorBuffer[idx] = color; // 更新颜色
    }
}
```

**Z-Buffer 的优势**：
- 不需要排序三角形
- 每个三角形独立绘制，与绘制顺序无关
- 硬件原生支持（现代 GPU 的 ROP 单元）

### Z-Buffer 搞不定透明物体

Z-Buffer 默认处理的是**不透明**遮挡：近的覆盖远的。但如果前面的三角形是透明的（比如玻璃窗），直接写深度并覆盖颜色就会出错——你会看不到玻璃后面的东西。

> [!warning] 透明物体的正确姿势
> 透明物体通常要：
> 1. **保留深度测试**（读取深度，判断是否在前面）；
> 2. **关闭深度写入**（不要把透明片的深度写进 Z-Buffer）；
> 3. **从远到近排序后绘制**（因为混合结果依赖顺序）。
> 
> 多个透明面互相穿插时，还需要 **OIT（Order-Independent Transparency）** 或 **Depth Peeling** 等高级方案。

### Z-Buffer 的精度问题与 Z-Fighting

透视投影后，Z 值在 NDC 空间的分布不是线性的——**近处精度高，远处精度低**。这是 Z-Buffer 产生闪烁（Z-Fighting）的根本原因。

#### 为什么不是线性的？

> [!info] 先明确一下：`z_eye` 是什么？
> **相机空间深度 `z_eye`** 就是顶点在经过 View 变换后、在相机坐标系里沿观察方向的坐标值。你可以把相机空间想象成"以相机为原点、以相机朝向为坐标轴"的世界：相机在原点，它看向 `-Z` 方向（OpenGL 右手系约定）。
>
> 所以一个顶点的 `z_eye` 越负，它离相机就越远；`z_eye = -1` 比 `z_eye = -0.1` 更远。我们在直觉上说"深度"时，通常指的是这个 `z_eye` 的绝对值 `|z_eye|`，也就是"顶点离相机有多远"。
>
> 更详细的 View/Projection 推导见 [[Notes/计算机图形学/软光栅化与3D数学/视图与投影|视图与投影]]。

透视投影矩阵把相机空间深度 `z_eye` 变换到裁剪空间，再经过透视除法后，NDC 深度满足：

```
z_ndc = A + B / z_eye   （A、B 是由近平面 n、远平面 f 决定的常数）
```

也就是说，`z_ndc` 与 `1 / z_eye` 成线性关系，而不是与 `z_eye` 本身成线性关系。这导致深度值在近平面附近变化剧烈，在远平面附近变化平缓：

```
透视投影后的 Z 分布：
近平面 ←──高精度──┼──低精度──→ 远平面
```

#### 为什么会导致闪烁？

Z-Buffer 的存储精度是有限的（常见 24 位定点数）。硬件会把连续深度值量化到最近的离散深度上。因为远处的 `z_ndc` 变化非常平缓，两个在相机空间里距离很近的物体，深度差可能小于 Z-Buffer 能分辨的最小粒度，被量化成同一个值。

更麻烦的是，GPU 在光栅化、插值、投影过程中还会引入浮点误差。这些误差在远处被放大，导致两个几乎等深的表面在不同像素、不同帧之间"谁更近"的判断来回翻转，形成**高频闪烁**，也就是 **Z-Fighting**。

> [!example] 具体数值感受一下
> 假设 `n = 0.1`，`f = 1000`，使用 24 位深度缓冲：
> - 在 `z_eye = 1` 米处，深度精度大约能分辨 **0.0001 米** 的差别；
> - 在 `z_eye = 500` 米处，深度精度可能只能分辨 **1 米** 左右的差别。
>
> 所以近处两个硬币叠在一起，GPU 能分清前后；远处地面上一张贴花（decal）和它下面的地面，深度差只有几厘米，硬件根本分不开，就开始闪。

#### 缓解方案

| 方案            | 原理                            | 代价                     |
| ------------- | ----------------------------- | ---------------------- |
| **增大近平面距离 n** | 让 `n/f` 比值变大，深度曲线整体变平缓，远处精度提升 | 太近的物体会被裁剪掉             |
| **使用对数深度缓冲**  | 把 `log(z)` 存进深度缓冲，让精度分布更均匀    | 需要修改投影或深度写入，远处精度好但近处略降 |
| **更高精度格式**    | 用 24 位整数、32 位浮点深度缓冲           | 显存和带宽增加                |

## 问题 4：为什么光栅化会产生锯齿？

在 [[Notes/计算机图形学/软光栅化与3D数学/光栅化：从直线到三角形|光栅化：从直线到三角形]] 中，我们用边缘函数判断像素中心点是否在三角形内部。这种"非黑即白"的判断方式，导致边缘呈现阶梯状：

```
理想直线：        光栅化结果：
    ↗              ■
   ↗              ■■
  ↗              ■■■
 ↗              ■■■■
```

**核心问题**：一个三角形是连续的几何形状，但屏幕像素是离散的格子。当几何边缘穿过像素时，我们只能选择"画"或"不画"，无法表达"画一半"。

### 走样的本质：采样率不足

光栅化本质上是**采样**：用离散的像素网格去采样连续的几何信号。

**走样（Aliasing）** 的本质：**采样率低于信号变化的频率**。

```
高频信号（锐利边缘）
    /\  /\  /\  /\      ← 变化快
   /  \/  \/  \/  \
  
低频采样（像素网格）
  ●     ●     ●     ●   ← 采样点太少
  
结果：无法区分高频信号和低频信号（混叠）
```

### 采样瑕疵的类型

| 类型 | 描述 | 例子 |
|-----|------|------|
| **锯齿（Jaggies）** | 边缘的阶梯状效果 | 斜线、三角形边缘 |
| **摩尔纹（Moire）** | 高频模式与采样频率相近时产生的干涉纹样 | 远处密集的栅栏、条纹衣服 |
| **闪烁（Shimmering）** | 微小运动时采样点变化导致颜色跳变 | 远处纹理、细小几何体 |

---

## 问题 5：怎么消除走样？

### 核心原则：先滤波，再采样

**错误的顺序**：先采样 → 再模糊（无法消除已产生的混叠）

**正确的顺序**：先滤波（去除高频） → 再采样

```
几何信号 ──[低通滤波]──→ 平滑信号 ──[采样]──→ 像素
         （去掉尖锐边缘）          （无混叠）
```

但问题是：滤波在数学上需要对连续信号做卷积积分，在实时渲染中无法直接实现。我们需要**近似的、高效的**抗锯齿方案。

## 问题 6：MSAA——多重采样抗锯齿

### 最 naive 的方案：提高分辨率

把渲染分辨率提高 4 倍（如 4K），再缩小到屏幕分辨率。每个最终像素由 4 个子采样平均得到。

**发现的问题**：像素着色器要执行 4 倍次数，性能开销巨大。

### 改进：MSAA（Multi-Sample Anti-Aliasing）

MSAA 的关键洞察：**锯齿只出现在边缘**。三角形内部的像素不需要多采样——它们完全被一个三角形覆盖。

```
MSAA 4x：每个像素分成 4 个子采样点

┌───────────┐
│  ○    ○   │  ○ = 子采样点（判断是否在三角形内）
│           │
│  ○    ○   │
└───────────┘

边缘像素：4 个子采样点分别判断覆盖率
内部像素：1 次着色，结果复制到 4 个子采样
```

**工作流程**：
1. 光栅化阶段：对每个子采样点判断是否在三角形内（只计算覆盖率，不计算颜色）
2. 像素着色阶段：只在像素中心执行一次着色计算
3. 混合阶段：根据子采样点的覆盖率，将着色结果写入对应的子采样
4. 解析阶段：将子采样平均，得到最终像素颜色

**MSAA 的代价**：
- 颜色缓冲区和深度缓冲区需要 4 倍内存
- 边缘像素的带宽增加
- 与延迟渲染（Deferred Shading）不兼容（因为 G-Buffer 无法存储多采样数据）

## 问题 7：FXAA——快速近似抗锯齿

### 核心思想：后处理检测边缘

MSAA 需要在光栅化阶段做多采样，开销较大。FXAA 完全在**后处理阶段**工作：

1. 先渲染出带有锯齿的图像
2. 通过图像处理检测边缘（寻找亮度梯度大的区域）
3. 对检测到的边缘像素做模糊混合

```cpp
// FXAA 的核心逻辑（极度简化版）
Color FXAA(sampler2D tex, vec2 uv) {
    // 采样当前像素和周围 4 个邻居的亮度
    float lumaCenter = luminance(texture(tex, uv));
    float lumaN = luminance(texture(tex, uv + vec2(0, -1) * texelSize));
    float lumaS = luminance(texture(tex, uv + vec2(0,  1) * texelSize));
    float lumaE = luminance(texture(tex, uv + vec2( 1, 0) * texelSize));
    float lumaW = luminance(texture(tex, uv + vec2(-1, 0) * texelSize));
    
    // 计算亮度梯度
    float range = max(lumaCenter, max(max(lumaN, lumaS), max(lumaE, lumaW))) -
                  min(lumaCenter, min(min(lumaN, lumaS), min(lumaE, lumaW)));
    
    // 如果梯度小于阈值，不是边缘，直接返回
    if (range < threshold) return texture(tex, uv);
    
    // 是边缘：沿垂直于边缘的方向混合
    // ... 混合逻辑 ...
}
```

**FXAA 的优势**：
- 速度极快，几乎没有额外性能开销
- 不需要多采样缓冲区，内存占用小
- 兼容延迟渲染

**FXAA 的劣势**：
- 可能引入图像模糊
- 对低对比度边缘检测效果不佳
- 有"拖影"感

## 问题 8：TAA——时间性抗锯齿

### 核心思想：把采样分布在时间维度上

MSAA 在一个像素内放多个采样点（空间多采样）。TAA 把采样分布在**多帧**上（时间多采样）：

```
第 N 帧：像素中心偏移 (-0.25, -0.25)
第 N+1 帧：像素中心偏移 ( 0.25, -0.25)
第 N+2 帧：像素中心偏移 (-0.25,  0.25)
第 N+3 帧：像素中心偏移 ( 0.25,  0.25)

4 帧混合 ≈ 4x MSAA 的效果
```

每一帧我们只渲染一次（像素中心偏移一点点），但把很多帧的结果累积起来。因为每一帧的采样点位置不同，静止画面中它们共同覆盖了一个像素内的多个子采样位置。

> [!example] 类比：拍一堆轻微抖动的照片再平均
> 想象你用手机拍一张夜景，手轻微抖动了一下。单看每张照片都有点糊，但把几十张对齐后平均，噪点会被压掉，细节反而更清晰。TAA 也是这个道理：每帧图像都是一次"抖动采样"，多帧累积起来就等效于在一帧内做了多次采样。

### 为什么时间采样能抗锯齿？

从采样定理角度看，**锯齿是因为空间采样率不够**。MSAA 通过在一个像素里放多个采样点来提高空间采样率；TAA 则利用了一个事实：现实画面在时间上通常是连续的，相邻两帧之间的运动很小。

如果我们把多帧的采样点"投影回"同一帧，它们就等效于在同一帧的不同子像素位置进行了采样。这样我们用**时间上的样本**换取了**空间上的超采样效果**。

```
静止物体：
帧 N 采样点：  ●
帧 N+1 采样点：   ●
帧 N+2 采样点：      ●
帧 N+3 采样点：         ●
──────────────────────────
合并后 ≈ 一个像素内有 4 个空间采样点
```

### TAA 的工作流程

**1. 子像素抖动（Jitter）**

每帧渲染前，对投影矩阵做一个极小的子像素偏移。这个偏移量通常用 **Halton 序列** 或类似的低差异序列生成，保证多帧覆盖均匀：

```cpp
// 极简化：每帧从预定义的抖动序列中取一个偏移
vec2 jitter = haltonSequence[frameIndex % numSamples];  // 范围 [-0.5, 0.5] 像素
projectionMatrix[2][0] += jitter.x / screenWidth * 2.0;   // 修改投影矩阵
projectionMatrix[2][1] += jitter.y / screenHeight * 2.0;
```

> [!warning] 抖动幅度很小
> 这个偏移通常只有 **0.25 ~ 1.0 像素** 的量级。太大的抖动会让画面本身晃动；太小的抖动又起不到多采样的效果。

**2. 历史帧重投影（Reprojection）**

当前帧渲染完成后，不能直接和历史帧像素一一对应混合，因为场景可能在动。我们需要知道"当前像素在上帧对应哪里"。

方法就是 **Motion Vector（运动向量）**：在渲染当前帧时，同时计算每个像素在上一帧屏幕空间中的位置。

```cpp
// 当前像素的世界坐标
vec3 worldPos = reconstructWorldPosition(depth, uv);
// 用上帧的 View-Projection 矩阵投影到上帧屏幕空间
vec4 prevClip = prevViewProjection * vec4(worldPos, 1.0);
vec2 prevUV = prevClip.xy / prevClip.w * 0.5 + 0.5;
// prevUV 就是历史帧中对应的位置
```

**3. 历史帧颜色校验（Neighborhood Clamping）**

直接把历史帧颜色和当前帧颜色混合会产生鬼影：如果历史帧对应位置后面被遮挡了，或者光照变化了，历史颜色就是错的。

常见做法是取当前像素周围 3×3 邻域的颜色范围，把历史帧颜色**限制在这个范围内**：

```cpp
// 取当前像素 3x3 邻域的最小/最大颜色
vec3 minColor, maxColor;
computeNeighborhoodMinMax(uv, minColor, maxColor);

// 把历史颜色夹到当前邻域范围内
vec3 historyColor = sampleHistory(prevUV);
historyColor = clamp(historyColor, minColor, maxColor);
```

这个操作叫 **Neighborhood Clamping** 或 **Color Clamping**，是抑制鬼影的核心。

**4. 时域混合（Temporal Blend）**

最后把校验后的历史颜色与当前帧颜色按权重混合：

```cpp
vec3 currentColor = sampleCurrent(uv);
vec3 finalColor = lerp(currentColor, historyColor, blendWeight);
```

混合权重 `blendWeight` 通常在 **0.9 ~ 0.98** 之间，意味着历史帧占大头。这样静止画面会非常稳定；运动快的像素会动态降低权重，让当前帧占主导。

### TAA 的优势

- **效率极高**：每帧只渲染一次，额外开销主要是重投影和混合
- **同时抗锯齿 + 降噪**：多帧累积天然压低了噪点
- **软化阴影/反射**：对时间上抖动的软阴影、反射也有平滑效果
- **现代游戏主流方案**：几乎所有 3A 游戏都在用

### TAA 的劣势

- **运动物体容易产生鬼影（Ghosting）**：即使做了 clamping，快速运动物体仍可能留下拖影
- **需要 Motion Vector 支持**：要求渲染管线能生成运动向量
- **画面有轻微模糊感**：累积过程相当于一个小模糊核，会让高频细节变肉
- **对突然光照变化敏感**：爆炸、开关灯等场景可能出现残影

> [!tip] 如何减少 TAA 的模糊？
> 常见做法是在 TAA 之后加一个 **锐化滤波器**（如 AMD FidelityFX CAS 或 Unreal 的 TAAU Sharpen）。但要注意锐化过度会产生白边。

### TAA 与 DLSS / FSR 的关系

DLSS、FSR、XeSS 这些现代超分技术，本质上都借用了 TAA 的**时域累积**思想：

| 技术 | 核心做法 |
|------|---------|
| **TAA** | 同一分辨率下，多帧累积抗锯齿 |
| **DLSS/FSR/XeSS** | 低分辨率渲染 + 时域累积 + 上采样到高分辨率 |

所以学会 TAA，就理解了现代超分技术的一半原理。

## 三种抗锯齿方案对比

| 方案       | 原理      | 性能  | 画质  | 适用场景       |
| -------- | ------- | --- | --- | ---------- |
| **MSAA** | 空间多采样   | 中等  | 最好  | 前向渲染、高质量需求 |
| **FXAA** | 后处理边缘检测 | 极高  | 一般  | 性能敏感、移动端   |
| **TAA**  | 时间多采样   | 高   | 好   | 现代游戏主流方案   |

> 现代游戏通常使用 **TAA + FXAA（备用）** 的组合，或在需要极高画质时使用 **MSAA + TAA**。

## 术语速查（新手友好版）

| 术语 | 一句话解释 |
|------|-----------|
| **MVP 矩阵** | Model-View-Projection，把模型坐标依次变换到世界、相机、裁剪空间的矩阵组合拳。 |
| **相机空间 / Eye Space** | 以相机为原点的坐标系，相机看向 `-Z` 方向，顶点的 `z` 坐标反映它离相机的远近。 |
| **裁剪空间** | 投影矩阵输出的空间，W 分量还保留着，用来做透视除法。 |
| **NDC** | 标准化设备坐标，`[-1, 1]³` 的固定盒子，和屏幕分辨率无关。 |
| **视口变换** | 把 NDC 映射到屏幕像素坐标的步骤。 |
| **光栅化** | 把三角形变成像素的过程；每个像素叫一个 **Fragment（片元）**。 |
| **Z-Buffer / 深度缓冲** | 每个像素存一个当前最小深度，用来判断谁在前面。 |
| **Z-Fighting** | 两个表面深度值过于接近，导致深度测试结果在像素/帧之间反复翻转，产生闪烁。 |
| **Motion Vector / 运动向量** | 描述一个像素在当前帧与上一帧屏幕空间位置的偏移向量，TAA 用它做历史帧重投影。 |
| **ROP** | GPU 里负责混合颜色、写像素的单元，深度测试也在这里做。 |
| **透视校正插值** | 纹理坐标等属性不能直接在屏幕空间线性插，要除以深度再插。 |
