# 缓存一致性协议

---

## Why：为什么需要缓存一致性？

多核 CPU 的每个核心都有私有 L1/L2 缓存。如果核心 A 修改了缓存中的变量，核心 B 读到的必须是新值，否则程序逻辑会出错。

**缓存一致性协议**（如 MESI）就是硬件层面的分布式一致性解决方案。

> **历史背景**：早期多核处理器（如 Intel Pentium D）采用**总线监听（Bus Snooping）+ Write-Through** 策略——所有写操作同时更新缓存和主存，其他核心通过监听总线得知数据变化。但 Write-Through 要求每次写都访问主存，总线带宽迅速成为瓶颈。MESI 通过引入"修改后不立即写回"（Write-Back）和四状态机，将大部分同步限制在缓存之间，大幅减少总线流量。

---

## What：MESI 协议

MESI 是每个 Cache Line 的状态机。这四个状态的设计精妙之处在于：**用最少的状态覆盖了"单核独占修改"、"多核只读共享"、"无效"三种核心场景**，同时通过 E（Exclusive）状态优化了最常见的"单核读写"情况。

| 状态    | 名称        | 含义                  |
| ----- | --------- | ------------------- |
| **M** | Modified  | 仅当前缓存拥有，且已修改，与主存不一致 |
| **E** | Exclusive | 仅当前缓存拥有，与主存一致       |
| **S** | Shared    | 多缓存共享，与主存一致         |
| **I** | Invalid   | 当前缓存不拥有有效数据         |

> **为什么是 4 个状态，不是 3 个或 5 个？**
> - 3 个状态（去掉 E）：单核首次读取数据时只能设为 S，下次写时必须广播 Invalidate。E 状态的存在让单核场景下的写操作无需总线事务，显著优化了单线程性能。
> - 5 个状态（如 MOESI 的 O、MESIF 的 F）：是对多核共享写场景的进一步优化，但增加了硬件复杂度。MESI 是"最小可用状态机"，在性能和复杂度之间取得了最佳平衡。

### MESI 状态转换图

```
                        其他核心读取（BusRd）
                   ┌─────────────────────────────┐
                   │                             ▼
┌─────┐  读未命中    ┌─────┐   其他核心写(BusRdX)   ┌─────┐
│  I  │ ───────────> │  S  │ ────────────────────> │  I  │
└─────┘   (BusRd)    └─────┘                       └─────┘
   ▲                   │                              ▲
   │                   │ 写命中（BusUpgr）             │
   │写未命中(BusRdX)    ▼                              │
   │               ┌─────┐   其他核心写(BusRdX)        │
   └────────────── │  E  │ ────────────────────────────┘
                   └─────┘
                     │
                     │ 写命中（无需总线事务）
                     ▼
                   ┌─────┐   其他核心读(BusRd)：写回主存  ┌─────┐
                   │  M  │ ────────────────────────────> │  S  │
                   └─────┘                                └─────┘
                     │                                     │
                     │ 其他核心读(BusRd)：写回主存          │
                     └─────────────────────────────────────┘
```

### 关键场景与总线事务

MESI 状态转换通过**总线事务（Bus Transaction）**实现。总线是连接 CPU 核心、缓存和内存的共享通信通道。

- **读命中 E/S**：直接读，无总线事务。
- **读命中 I**：向总线发 **BusRd（Read）** 请求，其他核心若有 M 则写回并转为 S，自己拿到数据转为 S/E。
- **写命中 E**：直接写，转为 M。这是最快的情况——因为 E 表示"只有我拥有"，修改无需通知任何人。
- **写命中 S**：发 **BusUpgr（Invalidate）** 使其他核心副本失效，自己转为 M。
- **写命中 I**：发 **BusRdX（Read-Invalidate）**，读取最新数据并使其他副本失效，转为 M。

> **BusRdX（Read-Invalidate）**：这是"读 + 使其他失效"的组合事务。看似可以拆成两步（先 BusRd 读数据，再 BusUpgr 使失效），但组合成一个原子事务可以避免总线上的竞争窗口——其他核心不会在"读"和"失效"之间插入自己的写操作。

至此，MESI 的理论模型是清晰的：每个 Cache Line 有明确的状态，状态转换通过总线事务同步完成。但现实中的 CPU 并没有这样"傻傻地等待"——为了让流水线不被阻塞，硬件引入了两个关键优化，而它们正是多线程编程中各种"诡异现象"的根源。

---

## 从理想模型到现实：Store Buffer 与 Invalidate Queue

MESI 保证了缓存一致性，但严格的同步会严重拖累性能。核心 A 写一块 S 状态的 Cache Line 时，必须等所有其他核心回 Ack 才能继续——这段时间可能长达上百个周期，CPU 流水线只能干等。

现代 CPU 引入了两个异步化优化：**Store Buffer**（写方异步）和 **Invalidate Queue**（接收方异步）。**它们提升了单核性能，却也让"写操作何时对其他核心可见"变得复杂**。

### Store Buffer：写方不再等待

**没有 Store Buffer 时**，写 S/I 状态的流程是同步阻塞的：
1. 发 `BusRdX` / `BusUpgr` 使其他核心副本失效
2. **等待**所有核心回 Ack
3. 才能写入缓存，状态变 M

**Store Buffer** 是核心内部的一个 FIFO 缓冲区，把同步写变成异步写：
- 核心把写操作丢进 Store Buffer，**不等待**，继续执行后续指令。
- 等 Invalidate Ack 到齐后，Store Buffer 按 FIFO 顺序把数据真正刷入缓存。

```
核心 A 执行 data = 42 (Cache Line 状态为 S)

无 Store Buffer（同步阻塞）:         有 Store Buffer（异步非阻塞）:
                                
  发 BusUpgr ──> 总线                发 BusUpgr ──> 总线
  等核心 B Ack <─────┐               写进 Store Buffer ───┐
  等核心 C Ack <─────┤               继续执行下一条指令    │
  ...               │               ...                 │
  才能写入缓存        │               等 Ack 到齐 <───────┘
  ▲                 │               刷入缓存
  │                 │               ▲
  └─ 流水线阻塞     │               └─ 流水线不阻塞
     ~100+ 周期                      0 周期等待
```

#### Store-to-Load Forwarding

核心读数据时，硬件会先查 Store Buffer："有没有同一地址的待写入数据？" 有就直接转发，不用去读缓存。这意味着**本核心**后续读自己能立刻看到最新值；但**其他核心**看不到 Store Buffer，只能读缓存里的旧值。

```
核心 A 内部读路径

        读 data ?
           │
           ▼
    ┌──────────────┐
    │ Store Buffer │── 命中同一地址？ ──> 直接返回 42（最新值）
    │ [0x1000]=42  │      │
    └──────────────┘      │ 未命中
           ▲              ▼
           │         ┌─────────┐
           └─────────┤ L1 Cache│──> 返回旧值 0
                     └─────────┘

其他核心读 data：只能走 L1 Cache，看到旧值 0
（Store Buffer 是核心私有的，不可见）
```

> **核心问题**：Store Buffer 让写操作对**其他核心**"延迟可见"。核心 A 的 `data = 42` 还躺在 Store Buffer 里，核心 B 读缓存只能拿到旧值。这是"写操作已执行但其他线程看不到"的硬件根源。

### 什么情况下会真的发生重排序？

Store Buffer 导致的不是"任意两个写都会乱序"，而是**"不需要总线事务的写"可能越过"还在等 Ack 的写"，提前对其他核心可见**。

以经典的两变量为例（`data` 和 `flag` 通常在不同 Cache Line）：

| `data` 状态 | `flag` 状态 | 核心 A 的行为 | 其他核心可见的顺序 |
|-----------|-----------|------------|---------------|
| S/I（需 Invalidate） | E/M（直接写） | `data` 进 Store Buffer；`flag` 直接写缓存 | `flag` 可能**先于** `data` 可见 |
| S/I | S/I | 两者都进 Store Buffer | FIFO 保序，但都**延迟可见** |
| E/M | S/I | `data` 直接写；`flag` 进 Store Buffer | `data` **先于** `flag` 可见，符合程序顺序 |
| E/M | E/M | 两者都直接写缓存 | 严格按程序顺序可见 |

**最危险的正是第一行**：`data` 被多核共享过（S 状态），而 `flag` 是核心 A 独占的（E 状态）。这是多线程同步中"发布-订阅"模式的典型场景——`data` 是共享数据，`flag` 是同步标志位。

### Invalidate Queue：接收方提前回复

当核心收到 **Invalidate** 请求时，如果立即执行失效操作，可能需要暂停当前流水线。为了加速响应：
- 核心把收到的 Invalidate 请求放进 **Invalidate Queue**，立刻回复 Ack。
- 稍后在合适的时机（如流水线空闲时）才真正执行失效。

```
时间 ───────────────────────────────────────────────────────>

核心 A:     写 data=42                    等 Ack...       继续执行
               │                              ▲              │
               └──── BusRdX ──────────────────┘              │
                    总线广播                                  │
                                                             │
核心 B:         收到 BusRdX ──> 进 Invalidate Queue           │
                                  立刻回 Ack ────────────────┘
                                  (Promise: 我稍后失效)
                                    │
                                    ▼
                                  流水线继续执行其他指令
                                    │
                                    ▼
                                  稍后真正失效 0x1000
```

#### 为什么不会导致所有指令都读到错数据？

Invalidate Queue 里放的不是"我已经失效了"，而是**"别人让我失效"的请求**。核心 B 的缓存没有全部失效，只有**某一个特定地址**的 Cache Line pending 失效。

核心 B 后续执行的指令，大部分访问的是**其他地址**，完全不受影响。只有当后续指令真的要去读那个 pending 失效的地址时，硬件通常会在读缓存前先检查 Invalidate Queue，发现匹配就先处理失效、再重新去总线读最新值。

但极端情况下，这个检查可能有延迟，导致核心 B 在"回 Ack"和"真失效"的窗口期内读到旧值。

> **核心问题**：Invalidate Queue 让"失效操作"本身也延迟了。核心 B 可能已经回复了 Ack（表示"我已失效"），但实际上 Cache Line 还没失效，核心 B 的后续读操作仍可能读到旧值。

### 综合效应：一个完整的重排序场景

以下现象成立的前提是：`data` 在核心 A 中为 S 或 I（被核心 B 共享过），`flag` 为 E（只有核心 A 独占）。初始 `data = 0, flag = 0`：

```cpp
// 核心 A                // 核心 B
data = 42;              while (!flag) ;
flag = 1;               print(data);  // 可能打印 0！
```

```
时间 ───────────────────────────────────────────────────────────────>

核心 A (data=S, flag=E):
  │
  │  data = 42 ──────────────────────────────────────────────────────
  │   状态=S → 发 BusRdX → 进 Store Buffer ───────────┐
  │   (等核心 B 回 Ack...)                             │
  │                                                    │
  │              flag = 1 ─────────────────────────────┤
  │               状态=E → 直接写缓存 → 变 M           │
  │               立刻对其他核心可见 ◄─────────────────┘
  │                                                    │
  │  (Store Buffer 还在等 data 的 Ack...)              │
  │                                                    ▼

核心 B:
  │
  │              看到 flag == 1 ──────────────────────────────────
  │               跳出 while 循环
  │
  │              读 data ─────────────────────────────────────────
  │               核心 B 缓存中 data 仍是旧值 0
  │               (核心 A 的 data=42 还在 Store Buffer 里！)
  │
  │  收到 BusRdX(data) ──> 进 Invalidate Queue
  │   回 Ack ──────────────────────────────┘
  │   (但 data 在核心 B 里还没真失效)
```

**发生了什么？**

1. **核心 A**：`data = 42` 状态 S → 发 `BusRdX` → 进 Store Buffer，等 Ack
2. **核心 A**：`flag = 1` 状态 E → **直接写入缓存**，变 M
3. **核心 A**：`flag` 已对其他核心可见，`data` 还在 Store Buffer 里排队

4. **核心 B**：看到 `flag == 1`（可能因 Invalidate Queue 延迟，flag 的旧值还在，但不影响结果）
5. **核心 B**：读取 `data` → 缓存里仍是旧值 0（核心 A 的 `data = 42` 还在 Store Buffer 中）

### Store Buffer vs Invalidate Queue：责任划分

|          | Store Buffer            | Invalidate Queue        |
| -------- | ----------------------- | ----------------------- |
| **位置**   | 写核心内部                   | 接收核心内部                  |
| **缓存什么** | 自己的写操作                  | 收到的 Invalidate 请求       |
| **目的**   | 不等待 Ack，继续执行            | 快速回复 Ack，不阻塞别人          |
| **副作用**  | 自己的写**延迟对外可见**          | 自己的 Cache Line**延迟失效**  |
| **解决方式** | `sfence` / `release` 刷空 | `lfence` / `acquire` 清空 |

> **解决之道**：内存屏障（Memory Barrier / Fence）
> - **Store Fence（sfence / release）**：刷空 Store Buffer，确保之前的所有写操作对其他核心可见。
> - **Load Fence（lfence / acquire）**：清 Invalidate Queue，确保之前的所有失效操作已执行，后续读操作读到最新值。
> - **Full Fence（mfence / seq_cst）**：同时执行 Store + Load Fence。
>
> 在 C++ 中，`std::memory_order_release` 会插入 Store Fence，`std::memory_order_acquire` 会插入 Load Fence。这正是下一篇 [[Notes/操作系统/并发与多线程/原子操作与内存序]] 要深入讨论的内容。

## 物理实现细节（扩展阅读）

理解了状态转换的逻辑后，一个自然的追问是：**这些状态存在哪？初始是什么？其他核心怎么知道的？**

### 状态存在哪？

MESI 状态存储在**每个 CPU 核心私有的缓存控制器（Cache Controller）中**，具体到硬件：

- **每个 Cache Line 的 Tag 数组额外附加 2~3 bit 的状态位**
  - L1/L2/L3 缓存的每个 Cache Line，除了数据和地址 Tag，还存有 MESI 状态
  - 4 个状态只需 2 bit（00=I, 01=S, 10=E, 11=M），部分实现预留 3 bit 用于扩展（如 MOESI 的 O 状态）
- **由硬件自动维护，对软件完全不可见**
  - 没有指令能直接读取某个 Cache Line 的 MESI 状态
  - 状态转换由缓存控制器内的状态机根据总线事件自动完成

> 可以这么理解：每个核心手里有一本小册子，记着"我缓存了哪些地址，每个当前是 M/E/S/I 中的哪一个"。这本册子就是缓存的 **Tag + State 数组**。

### 初始状态是什么？

当一个核心**第一次读取某块内存**时：

1. **初始状态一定是 I（Invalid）**——因为这块数据此前不在该核心的任何一级缓存中
2. 触发**读未命中（Read Miss）**，缓存控制器向总线发出 **BusRd** 请求
3. 根据其他核心的响应，最终状态由总线仲裁决定：
   - **其他核心都没有**（或都是 I）→ 从主存读取，状态变为 **E（Exclusive）**
   - **其他核心有 S 或 E**→ 数据共享，双方（或多方）都变为 **S**
   - **其他核心有 M**→ 该核心先把脏数据写回主存（或直发给你），然后大家一起变 **S**

> **E 状态的本质**：它不是预设的，而是读完数据后发现"全世界只有我有"这个**事后判断**的结果。这正是 E 能优化单核写的原因——一旦拿到 E，后续写命中就不用再发总线事务通知任何人。

### 如何知道其他核是否持有？

这是缓存一致性的**感知机制**问题，现代 CPU 有两种实现路线：

**总线监听（Bus Snooping）—— Intel 主流路线**

所有核心的缓存控制器挂在共享总线（或环形总线 Ring Bus）上：

- 某个核心发出 BusRd / BusRdX 时，消息**广播到所有核心**
- 每个核心的缓存控制器内置 **Snooping Unit（监听单元）**，被动监听总线上的每一笔请求
- 收到请求后，Snooping Unit 查自己的 Tag 数组，看是否命中该地址
- 若命中，通过总线的响应信号（如 `HIT`、`HITM`）回传，并执行相应状态转换

**局限**：核心数多了广播成为瓶颈，适用于核心数较少的场景（桌面/服务器 8~64 核）。

**目录协议（Directory-based）—— AMD EPYC / 大型 NUMA 系统**

在内存控制器（或 Last-Level Cache）中维护一张**全局目录表**：

| 内存块地址 | 状态 | 共享者列表（Sharer List） |
|-----------|------|------------------------|
| 0x1000    | S    | Core 0, Core 3         |
| 0x2000    | E    | Core 2                 |
| 0x3000    | M    | Core 1                 |

- 读请求先发往目录，查表后**只向实际持有数据的核心定向请求**
- 状态变化时同步更新目录（如 Core 1 写 0x1000，目录清空共享者，改状态为 M）

**优劣对比**：

| | Snooping | Directory |
|--|---------|-----------|
| 延迟 | 低（单次广播，并行响应） | 高（先查目录，再定向请求） |
| 带宽 | 差（广播浪费带宽） | 好（只发必要的消息） |
| 扩展性 | 差（核心多了广播撑不住） | 好（可扩展到成百上千核） |
| 硬件复杂度 | 低 | 高（需维护目录一致性） |

> **一句话总结**：状态存在每个核心自己的缓存 Tag 里；初始是 I；通过总线广播（Snooping）或全局目录（Directory）来感知其他核心是否持有。MESI 的魔力就在于——用**局部状态 + 全局通信**，在硬件层面实现了分布式一致性。

## How：如何利用这些知识？

1. **理解 False Sharing 的底层**：正是因为 MESI 以 Cache Line 为单位同步，同一行上的无关变量写操作也会触发 Invalidate 风暴。
2. **理解 Memory Order**：`std::memory_order_release` 会插入 Store Fence，刷 Store Buffer；`acquire` 会插入 Load Fence，清 Invalidate Queue。
3. **避免跨核高频写共享变量**：无论有没有锁，只要写共享 Cache Line，性能都会崩塌。

## C++ 本地验证实验

### 实验：跨核写共享变量的性能悬崖

```cpp
// workspace/os-lab/cpu-cache/mesi_bench.cpp
#include <thread>
#include <vector>
#include <chrono>
#include <iostream>
#include <atomic>

constexpr int ITER = 10'000'000;

// 所有线程写同一变量（同一 Cache Line）
void shared_bench(int num_threads) {
    alignas(64) std::atomic<int> counter{0};
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&]() {
            for (int j = 0; j < ITER; ++j)
                counter.fetch_add(1, std::memory_order_relaxed);
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "Shared var, " << num_threads << " threads: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

// 每个线程写自己的变量（不同 Cache Line）
void separate_bench(int num_threads) {
    struct alignas(64) PaddedAtomic { std::atomic<int> v{0}; };
    std::vector<PaddedAtomic> counters(num_threads);
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&, i]() {
            for (int j = 0; j < ITER; ++j)
                counters[i].v.fetch_add(1, std::memory_order_relaxed);
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "Separate vars, " << num_threads << " threads: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    for (int t : {1, 2, 4}) {
        shared_bench(t);
        separate_bench(t);
        std::cout << "---\n";
    }
    return 0;
}
```

**预期结果**：线程数增加时，共享变量版本的耗时非线性增长（甚至超线性）；分离变量版本接近线性加速。

## 游戏引擎延伸

### 多线程渲染命令缓冲区

- 现代引擎（如 UE、Chaos）允许多个线程同时生成渲染命令。
- 为避免 MESI 风暴，通常采用 **Command List Per Thread**，每个线程写到独立的内存块，最后由主线程合并提交。
- 合并阶段也尽量顺序访问，保证提交时的空间局部性。

### 状态标记与双缓冲

- 主线程写「当前帧配置」，工作线程读「上一帧配置」。
- 通过帧索引翻转（`frameIndex & 1`）实现无锁双缓冲，避免共享 Cache Line 的频繁失效。
