# 缓存一致性协议

---

## Why：为什么需要缓存一致性？

- 多核 CPU 每个核心有私有 L1/L2 缓存，多个核心可能同时缓存同一块内存。
- 若核心 A 修改了缓存中的变量，核心 B 读到的必须是新值，否则程序逻辑出错。
- **缓存一致性协议**（如 MESI）就是硬件层面的分布式一致性解决方案。

> **历史背景**：早期多核处理器（如 Intel Pentium D）采用**总线监听（Bus Snooping）+ Write-Through** 策略——所有写操作同时更新缓存和主存，其他核心通过监听总线得知数据变化。但 Write-Through 要求每次写都访问主存，总线带宽迅速成为瓶颈。MESI 通过引入"修改后不立即写回"（Write-Back）和四状态机，将大部分同步限制在缓存之间，大幅减少总线流量。

---

## What：MESI 协议

MESI 是每个 Cache Line 的状态机。这四个状态的设计精妙之处在于：**用最少的状态覆盖了"单核独占修改"、"多核只读共享"、"无效"三种核心场景**，同时通过 E（Exclusive）状态优化了最常见的"单核读写"情况。

| 状态 | 名称 | 含义 |
|------|------|------|
| **M** | Modified | 仅当前缓存拥有，且已修改，与主存不一致 |
| **E** | Exclusive | 仅当前缓存拥有，与主存一致 |
| **S** | Shared | 多缓存共享，与主存一致 |
| **I** | Invalid | 当前缓存不拥有有效数据 |

> **为什么是 4 个状态，不是 3 个或 5 个？**
> - 3 个状态（去掉 E）：单核首次读取数据时只能设为 S，下次写时必须广播 Invalidate。E 状态的存在让单核场景下的写操作无需总线事务，显著优化了单线程性能。
> - 5 个状态（如 MOESI 的 O、MESIF 的 F）：是对多核共享写场景的进一步优化，但增加了硬件复杂度。MESI 是"最小可用状态机"，在性能和复杂度之间取得了最佳平衡。

### MESI 状态转换图

```
                        其他核心读取（BusRd）
                   ┌─────────────────────────────┐
                   │                             ▼
┌─────┐  读未命中    ┌─────┐   其他核心写(BusRdX)   ┌─────┐
│  I  │ ───────────> │  S  │ ────────────────────> │  I  │
└─────┘   (BusRd)    └─────┘                       └─────┘
   ▲                   │                              ▲
   │                   │ 写命中（BusUpgr）             │
   │写未命中(BusRdX)    ▼                              │
   │               ┌─────┐   其他核心写(BusRdX)        │
   └────────────── │  E  │ ────────────────────────────┘
                   └─────┘
                     │
                     │ 写命中（无需总线事务）
                     ▼
                   ┌─────┐   其他核心读(BusRd)：写回主存  ┌─────┐
                   │  M  │ ────────────────────────────> │  S  │
                   └─────┘                                └─────┘
                     │                                     │
                     │ 其他核心读(BusRd)：写回主存          │
                     └─────────────────────────────────────┘
```

### 关键转换场景与总线事务

MESI 状态转换通过**总线事务（Bus Transaction）**实现。总线是连接 CPU 核心、缓存和内存的共享通信通道。

- **读命中 E/S**：直接读，无总线事务。
- **读命中 I**：向总线发 **BusRd（Read）** 请求，其他核心若有 M 则写回并转为 S，自己拿到数据转为 S/E。
- **写命中 E**：直接写，转为 M。这是最快的情况——因为 E 表示"只有我拥有"，修改无需通知任何人。
- **写命中 S**：发 **BusUpgr（Invalidate）** 使其他核心副本失效，自己转为 M。
- **写命中 I**：发 **BusRdX（Read-Invalidate）**，读取最新数据并使其他副本失效，转为 M。

> **BusRdX（Read-Invalidate）**：这是"读 + 使其他失效"的组合事务。看似可以拆成两步（先 BusRd 读数据，再 BusUpgr 使失效），但组合成一个原子事务可以避免总线上的竞争窗口——其他核心不会在"读"和"失效"之间插入自己的写操作。

### Store Buffer 与 Invalidate Queue：性能优化的"副作用"

MESI 保证了缓存一致性，但严格的同步会严重拖累性能。现代 CPU 引入了两个关键优化：

#### Store Buffer

当核心想要写一个处于 **S** 或 **I** 状态的 Cache Line 时，必须先发总线事务使其他核心副本失效，等待 Ack 后才能写入。这段时间 CPU 流水线会被阻塞。

**Store Buffer** 是核心内部的一个 FIFO 缓冲区：
- 核心把写操作丢进 Store Buffer，继续执行后续指令。
- 等 Invalidate Ack 全部收到后，Store Buffer 中的写操作再真正提交到缓存。

> **问题**：Store Buffer 让写操作对其他核心"延迟可见"。核心 A 写入 `data = 42` 后进 Store Buffer，核心 B 可能立刻读到旧值。这就是多线程编程中"写操作已执行但其他线程看不到"的硬件根源。

#### Invalidate Queue

当核心收到 **Invalidate** 请求时，如果立即执行失效操作，可能需要暂停当前流水线。为了加速响应：
- 核心把收到的 Invalidate 请求放进 **Invalidate Queue**，立刻回复 Ack。
- 稍后在合适的时机（如流水线空闲时）才真正执行失效。

> **问题**：Invalidate Queue 让"失效操作"本身也延迟了。核心 B 可能已经回复了 Ack（表示"我已失效"），但实际上 Cache Line 还没失效，核心 B 的后续读操作仍可能读到旧值。

#### 综合效应：内存重排序的硬件根源

Store Buffer + Invalidate Queue 共同导致了以下可能的现象（假设初始 `data = 0, flag = 0`）：

```cpp
// 核心 A                // 核心 B
data = 42;              while (!flag) ;
flag = 1;               print(data);  // 可能打印 0！
```

在核心 A 上：
1. `data = 42` 进入 Store Buffer，等待 Invalidate Ack。
2. `flag = 1` 命中缓存（假设 flag 在 E 状态），直接写入缓存并变为 M。
3. 由于 Store Buffer 是 FIFO，`flag = 1` 虽然后写入，但可能先对其他核心可见（如果 `data` 的 Invalidate 还在等待）。

在核心 B 上：
4. 看到 `flag == 1`（Invalidate Queue 延迟处理，flag 的旧值还在）。
5. 读取 `data`，但核心 A 的 `data = 42` 还在 Store Buffer 中，读到旧值 0。

> **解决之道**：内存屏障（Memory Barrier / Fence）
> - **Store Fence（sfence / release）**：刷空 Store Buffer，确保之前的所有写操作对其他核心可见。
> - **Load Fence（lfence / acquire）**：清 Invalidate Queue，确保之前的所有失效操作已执行，后续读操作读到最新值。
> - **Full Fence（mfence / seq_cst）**：同时执行 Store + Load Fence。
>
> 在 C++ 中，`std::memory_order_release` 会插入 Store Fence，`std::memory_order_acquire` 会插入 Load Fence。这正是下一篇 [[Notes/操作系统/并发与多线程/原子操作与内存序]] 要深入讨论的内容。

## How：如何利用这些知识？

1. **理解 False Sharing 的底层**：正是因为 MESI 以 Cache Line 为单位同步，同一行上的无关变量写操作也会触发 Invalidate 风暴。
2. **理解 Memory Order**：`std::memory_order_release` 会插入 Store Fence，刷 Store Buffer；`acquire` 会插入 Load Fence，清 Invalidate Queue。
3. **避免跨核高频写共享变量**：无论有没有锁，只要写共享 Cache Line，性能都会崩塌。

## C++ 本地验证实验

### 实验：跨核写共享变量的性能悬崖

```cpp
// workspace/os-lab/cpu-cache/mesi_bench.cpp
#include <thread>
#include <vector>
#include <chrono>
#include <iostream>
#include <atomic>

constexpr int ITER = 10'000'000;

// 所有线程写同一变量（同一 Cache Line）
void shared_bench(int num_threads) {
    alignas(64) std::atomic<int> counter{0};
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&]() {
            for (int j = 0; j < ITER; ++j)
                counter.fetch_add(1, std::memory_order_relaxed);
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "Shared var, " << num_threads << " threads: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

// 每个线程写自己的变量（不同 Cache Line）
void separate_bench(int num_threads) {
    struct alignas(64) PaddedAtomic { std::atomic<int> v{0}; };
    std::vector<PaddedAtomic> counters(num_threads);
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&, i]() {
            for (int j = 0; j < ITER; ++j)
                counters[i].v.fetch_add(1, std::memory_order_relaxed);
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "Separate vars, " << num_threads << " threads: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    for (int t : {1, 2, 4}) {
        shared_bench(t);
        separate_bench(t);
        std::cout << "---\n";
    }
    return 0;
}
```

**预期结果**：线程数增加时，共享变量版本的耗时非线性增长（甚至超线性）；分离变量版本接近线性加速。

## 游戏引擎延伸

### 多线程渲染命令缓冲区

- 现代引擎（如 UE、Chaos）允许多个线程同时生成渲染命令。
- 为避免 MESI 风暴，通常采用 **Command List Per Thread**，每个线程写到独立的内存块，最后由主线程合并提交。
- 合并阶段也尽量顺序访问，保证提交时的空间局部性。

### 状态标记与双缓冲

- 主线程写「当前帧配置」，工作线程读「上一帧配置」。
- 通过帧索引翻转（`frameIndex & 1`）实现无锁双缓冲，避免共享 Cache Line 的频繁失效。
