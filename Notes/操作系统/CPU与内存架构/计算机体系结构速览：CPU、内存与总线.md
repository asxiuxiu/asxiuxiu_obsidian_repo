# 计算机体系结构速览：CPU、内存与总线

---

## Why：为什么要学习计算机体系结构？

- **性能优化的根基**：游戏引擎中 90% 的底层优化决策（Cache 友好、分支预测、SIMD）都建立在理解 CPU 如何执行指令的基础上。不了解硬件，优化就是盲目试错。
- **打破"黑盒"误区**：很多程序员把 CPU 视为"执行代码的黑盒"。当多线程程序出现诡异 Bug、或某段代码突然慢 100 倍时，体系结构知识是定位问题的唯一武器。
- **理解发展脉络**：为什么现代 CPU 主频停滞在 3-4GHz？为什么多核取代单核超频？为什么 ARM 架构在移动端胜出？这些问题的答案都在体系结构的演进史中。

---

## What：计算机体系结构的核心组成

### CPU 的基本执行模型

现代 CPU 并非简单"一条指令接一条指令"地执行，而是采用**流水线（Pipeline）**技术，将指令执行拆分为多个阶段并行处理：

| 阶段  | 英文            | 作用                    |     |
| --- | ------------- | --------------------- | --- |
| 取指  | Fetch         | 从内存中读取下一条指令到 CPU      |     |
| 译码  | Decode        | 解析指令含义，确定需要哪些操作数和运算单元 |     |
| 执行  | Execute       | ALU（算术逻辑单元）执行实际运算     |     |
| 访存  | Memory Access | 如需读写内存，在此阶段访问数据缓存     |     |
| 写回  | Write Back    | 将运算结果写回寄存器或内存         |     |

> **关键洞察**：流水线意味着 CPU 在同一时刻有 5 条不同指令处于不同执行阶段。这解释了为什么**分支预测失败**（如 `if` 判断猜错）会导致流水线清空，代价高达 10-20 个周期。

---

## 流水线的内在矛盾：依赖、并行与 PC 之谜

基础流水线模型看起来优雅，但它掩盖了两个让初学者极其困惑的问题：

1. 代码里一句强依赖下一句（比如 `d = a + e` 依赖 `a = b + c`），那所谓"多指令并行"到底并行在哪？
2. 既然多条指令同时在流水线里"飞"，为什么只有一个程序计数器 PC？难道不应该有多个"下一条"吗？

这两个问题问得非常精准。它们的分界点是：**笔记至此只讲了五级流水线的"骨架"，还没讲现代 CPU 如何在"有依赖"和"单 PC"的硬约束下，榨出真正的指令级并行**。理解这一点，是从"教科书流水线"走向"真实硬件"的关键一跃。

---

### Why：直觉上，流水线似乎根本跑不起来

写代码的人都知道，真实的指令流极少是"彼此无关"的：

```cpp
a = b + c;   // 指令 1
d = a + e;   // 指令 2，强依赖指令 1 的结果
```

如果指令 2 必须等指令 1 完全走完取指→译码→执行→访存→写回这 5 个阶段，那流水线后半段大部分时间都在空转。更糟的是，如果只有一个 PC，它每周期只推进一条指令地址，那流水线里这 5 条"在飞"的指令是怎么知道自己该干什么的？

先别急着否定流水线。CPU 设计师面对这两个矛盾，给出的不是妥协，而是一整套精妙的硬件解法。

---

### 矛盾一：数据依赖 vs. 流水线并行

#### 数据冒险（Data Hazard）

在体系结构术语里，"后一条指令需要前一条指令的结果，但前一条还没算完"叫做**数据冒险**。这是流水线最常见、也最早被解决的问题。

如果没有特殊处理，指令 2 确实要在译码阶段 stall（停顿），等指令 1 写回寄存器——这会严重拖慢流水线。

#### Forwarding（旁路前递）：依赖 ≠ 等写回

解决数据冒险的核心硬件机制叫 **Forwarding（旁路前递）** 或 **Bypassing**。它的思路极其朴素：**执行单元算出结果后，不等到"写回阶段"，直接通过专用总线把数据传给下一个需要它的指令的输入端**。

```
指令 1: a = b + c
        ├─ 执行阶段（周期 N）ALU 算出 a 的值
        └─ 旁路总线直接前递 ──────────────┐
                                          ↓
指令 2: d = a + e                         │
        └─ 执行阶段（周期 N+1）直接用上前递来的 a
```

> **关键洞察**：Forwarding 让数据从"产生处"直接流向"消费处"，跳过了"写回寄存器文件→再从寄存器文件读出"的往返。依赖造成的延迟从 3-4 个周期压缩到 **1 个周期**。

当然，并非所有依赖都能被完美前递。例如**加载-使用（Load-Use）**冒险——当数据需要先从内存加载到寄存器才能用时，即使 forwarding 也无能为力，因为内存访问本身就要 3-5 个周期。这时候流水线确实会插入一个气泡（Bubble），但这种情况只占指令流的一小部分。

---

### 矛盾二：一个 PC vs. 多条指令并行

这是另一个非常尖锐的疑问。如果流水线里真的有 5 条指令在不同阶段同时推进，为什么架构上只暴露一个 PC？

#### PC 的语义：入口调度员，而非状态快照

**PC 跟踪的是"指令流的顺序"，不是"流水线中各指令的执行状态"**。

想象一个快递分拣中心：
- **PC 就是入口处的调度员**。他每周期只说一句话："下一个包裹编号是 X"，然后把包裹放到传送带起点。
- 传送带上有 5 个包裹在不同工位处理，**每个包裹都贴着自己的编号单**——这对应流水线各阶段之间的**流水线寄存器（Pipeline Registers）**，里面保存了每条指令自己的地址副本。
- 工位们按各自包裹的编号单操作，根本不需要去问调度员。

| 层级 | 你看到的状态 | 硬件实际状态 |
|------|-----------|-----------|
| 架构层面 | **1 个 PC（RIP）** | 程序顺序中下一条待取指的地址 |
| 流水线内部 | 不可见 | **每条指令携带自己的 PC 副本**，用于异常定位、分支恢复 |
| 超标量取指 | PC 每周期 +4/8/16 字节 | 一次把多条顺序指令喂进流水线 |

所以，**架构上只有一个 PC，但流水线内部为每条指令都维护了 PC 的副本**。当发生异常（如除零）时，CPU 需要精确报告"哪条指令出错了"——靠的就是指令内部保存的那个 PC 值。

分支跳转时 PC 被改写，也印证了这个角色：预测器猜"要跳"，PC 立刻被刷成目标地址；猜错时，PC 再被刷回正确地址，流水线中按旧 PC 取进来的指令全部作废（Flush）。**PC 始终只有一个，但它是可被改写的"流指针"**。

---

### 真正的并行：超标量与乱序执行

如果 CPU 只有基础五级流水线，那它的理论上限就是**每周期完成一条指令（CPI ≈ 1）**。但现代桌面 CPU 的 IPC（Instructions Per Cycle）普遍在 3-4 之间——这意味着它们每周期能执行 3-4 条指令。这是怎么做到的？

#### 超标量（Superscalar）：多发射、多执行单元

现代 CPU 每周期可以从指令缓存取 **4-8 条指令**，译码后发给多个并行的执行单元：

| 执行单元 | 数量 | 职责 |
|---------|------|------|
| 整数 ALU | 3-4 个 | 加减、位运算、逻辑运算 |
| 乘除法单元 | 1-2 个 | 整数乘法（3-5 周期）、除法（20-100+ 周期） |
| 浮点/SIMD 单元 | 2-4 个 | SSE/AVX 向量运算 |
| Load/Store 单元 | 2-3 个 | 内存读写、地址计算 |

#### 乱序执行（Out-of-Order Execution, OoOE）

这是榨取并行的核心。当 `d = a + e` 因为等 `a` 的结果而卡住时，CPU 的**重排序缓冲（Reorder Buffer, ROB）**和**调度器（Scheduler）**不会傻等。它们会去后面的指令里挑**不依赖任何未完成结果**的指令，先发给空闲的 ALU 执行。

```
周期 N:
  ├─ ALU0: a = b + c          （正在算 a）
  ├─ ALU1: x = y + z          （完全独立，同时执行！）
  └─ ALU2: p = q + r          （也完全独立，同时执行！）

周期 N+1:
  ├─ ALU0: d = a + e          （a 已通过 Forwarding 就绪）
  └─ ALU1: m = n + o          （后面挑出来的无关指令）
```

乱序执行让 CPU 在**单线程、顺序指令流**的表象下，内部像拼图一样动态寻找可并行的指令块。对程序员而言，代码还是一行行写的；对硬件而言，指令在**满足数据依赖关系**的前提下被尽可能重叠。

> [!tip] 关键洞察
> 现代 CPU 的并行 = **流水线阶段重叠**（基础）+ **Forwarding 缩短依赖延迟**（硬件短路）+ **超标量多单元并发**（物理并行）+ **乱序执行动态调度**（智能填缝）。数据依赖没有被消除，但等待成本被压到了极限。

---

### 过渡：还有一类冒险，硬件也无法预判

Forwarding 解决了**数据冒险**，超标量和乱序执行解决了**结构冒险**（资源冲突），但流水线还面临第三种、也是最棘手的一种：**控制冒险（Control Hazard）**。

当流水线执行到一条条件分支（`if`、`for`、`while`）时，取指阶段在周期 N 就需要知道"下一条指令从哪取"——但条件判断的结果要到执行阶段（周期 N+2 或更晚）才能算出来。如果停下来等，流水线后面所有工位都会空转。

这是**分支预测**诞生的真正原因。它不是锦上添花的优化，而是**深流水线架构下的生存必需品**。

---

### 分支预测：流水线并行下的必然产物

#### Why：为什么不能"走对了再执行"？

程序员直觉里，`if (a > 0)` 的结果出来了，CPU 自然就知道该走哪条路。但这个直觉建立在**串行执行**的模型上——一条指令执行完，再执行下一条。

现代 CPU 的流水线在同一时刻有 5~20 条指令"在飞"。当流水线执行到一条条件分支时：

- **取指阶段**（周期 N）已经需要知道"下一条指令从哪个地址取"
- 但**条件判断的结果**（`a > 0` 是否为真）要到执行阶段（周期 N+2 或更晚）才能算出来

如果取指单元停下来等结果，流水线后面所有工位都会空转，性能直接崩掉。所以 CPU 只能**先猜一个方向**，让流水线保持满速运转。

#### What：分支预测如何工作？

**分支预测器（Branch Predictor）**是 CPU 内部的一个硬件模块，专门负责"赌"条件分支会往哪边走：

| 预测策略 | 原理 | 适用场景 |
|---------|------|---------|
| 静态预测 | 固定规则，如"向后跳转预测为 Taken" | 简单循环 |
| 动态预测（1-bit） | 根据该分支上一次的走向预测 | 规律性强的分支 |
| 动态预测（2-bit 饱和计数器） | 只有连续两次猜错才翻转预测方向 | 大多数通用场景 |
| 分支目标缓冲器（BTB） | 缓存分支指令的目标地址，避免重复计算 | 频繁调用的函数/循环 |

> **猜对了**：流水线无缝衔接，零额外代价。
> **猜错了**：流水线中错误分支的指令全部作废（Flush），重新从正确地址取指填充，代价 **10~20 个周期**（现代深流水线 CPU 甚至可达 20+ 周期）。

#### 类比：高速公路上的岔路口

想象你以 120km/h 行驶，导航提示"前方 500 米根据路牌决定左右"。你不可能踩刹车停在那里等看到路牌再决定——那样太慢了。你会根据经验（"这条路我天天走，80% 直行"）**先选一个方向开着**。如果对了，一路顺畅；如果错了，变道绕回来，浪费时间和油。

CPU 的分支预测器就是那位"根据历史经验先选一个方向"的司机。

#### How：写代码时如何利用分支预测？

分支预测失败的代价极高（10~20 个周期），而程序里的 `if`、`for`、`while` 无处不在。优化思路：

**1. 让分支模式可预测**

```cpp
// 糟糕：随机分支，预测器每次猜错的概率接近 50%
for (int i = 0; i < n; ++i) {
    if (data[i] > threshold)  // true/false 完全随机
        sum += data[i];
}

// 更好：先排序，让相同结果聚集，预测器更容易猜对
std::sort(data.begin(), data.end());
for (int i = 0; i < n; ++i) {
    if (data[i] > threshold)  // 前面全 false，后面全 true，模式清晰
        sum += data[i];
}
```

**2. 用条件选择指令消除分支**

```cpp
// 有分支：可能预测失败
int max_val = (a > b) ? a : b;

// 编译器优化后：生成 CMOV 条件传送指令，无分支、无预测
// 手写 SIMD 时也可用 blend 指令达到同样效果
```

**3. 减少虚函数和函数指针调用**

虚函数表跳转本质上是不可预测的分支（目标地址在运行时才能确定）。数据导向设计（DOD）中，用**类型分片（Type Splitting）**替代虚函数，既是 Cache 优化，也是分支预测优化。

> 类型分片的核心是**把不同类型的对象拆到不同的数组里**，用 `switch` 或直接函数调用替代虚函数。详见 [[Notes/C++编程/C++ 虚函数与多态本质#Type Splitting：按类型分片替代虚函数|Type Splitting 详解]]。

### CPU 主频：CPU 的"心跳速度"

**主频（Clock Rate）**描述的是 CPU 内部时钟发生器每秒钟产生的脉冲次数，单位是赫兹（Hz）。它决定了 CPU 执行操作的基本节奏。

| 主频    | 每秒脉冲数 | 一个时钟周期     |
| ----- | ----- | ---------- |
| 1 GHz | 10 亿次 | 1 纳秒（1 ns） |
| 3 GHz | 30 亿次 | 0.33 纳秒    |
| 4 GHz | 40 亿次 | 0.25 纳秒    |

> **通俗理解**：把 CPU 比作工厂流水线，**主频就是传送带的运转速度**。主频越高，传送带跑得越快，理论上单位时间内能流过的产品（指令）就越多。

但这并不意味着主频翻倍，性能就翻倍。因为：
- **功耗与电压的平方成正比**（$P \propto CV^2f$），主频提升会带来巨大的散热压力
- 内存、缓存等外围部件的速度跟不上 CPU，形成"水桶短板"
- 流水线中的分支预测失败、缓存未命中等问题会造成停顿

> [!info] 为什么现代 CPU 主频停留在 3-4GHz？
> 2005 年左右，Intel 取消了 4GHz+ 计划——因为**功耗墙（Power Wall）**到了。频率越高，芯片发热越严重，散热成本暴涨。因此行业转向**多核 + 大缓存 + 宽流水线**的方向，而非一味提升主频。

---

### 时钟周期：CPU 工作的"最小时间格子"

**时钟周期（Clock Cycle）**是主频的倒数，代表 CPU 完成一次基本操作所需的最短时间。它是 CPU 内部所有晶体管同步协作的节拍器。

如果把流水线比作汽车组装厂：

| 时钟周期 | 工位1（取指） | 工位2（译码） | 工位3（执行） | 工位4（访存） | 工位5（写回） |
|---------|------------|------------|------------|------------|------------|
| 周期 1 | 指令 A | （空） | （空） | （空） | （空） |
| 周期 2 | 指令 B | 指令 A | （空） | （空） | （空） |
| 周期 3 | 指令 C | 指令 B | 指令 A | （空） | （空） |
| 周期 5 | 指令 E | 指令 D | 指令 C | 指令 B | 指令 A 完成 |

**一个时钟周期内，流水线的每个阶段各自推进一小步。** 同一时刻可以有 5 条指令"在飞"，但它们分散在 5 个不同阶段。

> [!warning] 时钟周期 ≠ 执行完一条指令
> 早期 CPU 是单周期模型（一条指令占一个周期），但现代 CPU 采用流水线后，**一条指令通常需要多个周期才能完成**。不过，由于多条指令重叠执行，理想情况下 **CPI（Cycles Per Instruction）≈ 1**，即平均每个周期完成一条指令。

不同指令实际消耗的周期数差异很大：

| 指令类型 | 大约周期数 | 原因 |
|---------|----------|------|
| 寄存器加法 | 1 周期 | ALU 直接运算 |
| 整数乘法 | 3-5 周期 | 需要多步计算 |
| 整数除法 | 20-100+ 周期 | 最复杂的整数运算 |
| 访问 L1 缓存 | 3-5 周期 | 等待缓存响应 |
| 访问主内存 | 100-300 周期 | CPU 干等内存 |
| 分支预测失败 | 10-20 周期 | 流水线清空，重新填充 |

> [!tip] 关键洞察
> 主频只决定了每个格子有多小，但**格子里面能装多少活**，还取决于流水线深度、ALU 数量、缓存命中率等。3GHz 的 CPU 如果能做到每个周期发射 4 条指令，其吞吐量可能超过 4GHz 的单发射 CPU。

---

### 寄存器：CPU 体内的"极速缓存"

寄存器是 CPU 内部最快的存储单元，延迟通常只有 **1 个时钟周期**（~0.3ns）。

- **通用寄存器**：如 x86_64 的 `RAX`, `RBX`, `RCX`, `RDX`，用于存放临时运算数据。
- **程序计数器（PC / RIP）**：存放下一条要执行指令的地址。
- **栈指针（SP / RSP）**：指向当前函数栈帧的顶部。
- **标志寄存器（EFLAGS / RFLAGS）**：记录上一条指令的结果（零标志、进位标志、符号标志等），是条件跳转的依据。

> 编译器优化的一大目标就是尽量减少内存访问，让数据尽量留在寄存器中——这称为**寄存器分配（Register Allocation）**。

### ALU 与控制单元

- **ALU（Arithmetic Logic Unit）**：执行整数加减乘除、位运算、逻辑与或非。现代 CPU 有多个 ALU，可同时执行多条无依赖的整数指令。
- **FPU / SIMD 单元**：浮点运算单元和向量运算单元（如 x86 的 SSE/AVX，ARM 的 NEON），一次可处理 4~16 个浮点数，是图形渲染和物理模拟的硬件根基。
- **控制单元（CU）**：负责协调流水线各阶段、处理分支跳转、响应中断。

### 内存层次结构（Memory Hierarchy）

存储器存在**速度-容量-成本**的不可能三角。现代计算机采用分层设计：

```
速度 ↑  容量 ↓  成本 ↑
  │
  ├─ 寄存器（Register）      ~1KB      ~0.3ns
  ├─ L1 缓存                 ~64KB     ~1ns
  ├─ L2 缓存                 ~512KB    ~4ns
  ├─ L3 缓存（LLC）          ~32MB     ~40ns
  ├─ 主存（DRAM）            ~32GB     ~100ns
  ├─ SSD（固态硬盘）          ~1TB      ~10μs
  └─ 机械硬盘（HDD）          ~10TB     ~10ms
速度 ↓  容量 ↑  成本 ↓
```

> **核心原则**：CPU 访问数据时，会自顶向下逐层查找。越靠近顶层的"命中"，延迟越低。程序优化的本质就是让数据尽可能停留在上层。

### 总线：连接各组件的"高速公路"

CPU、内存、显卡、硬盘并非直接焊接在一起，而是通过**总线（Bus）**互联。

- **地址总线（Address Bus）**：CPU 发出"我要访问哪个地址"的信号。宽度决定寻址能力（32 位 → 4GB，64 位 → 理论 16EB）。
- **数据总线（Data Bus）**：实际传输数据的通道。宽度决定单次传输的数据量（64 位宽 → 每次传 8 字节）。
- **控制总线（Control Bus）**：传递读写控制信号、中断请求、时钟同步信号。

> **历史注记**：早期 PC 采用**北桥（Northbridge）+ 南桥（Southbridge）**芯片组架构。北桥负责连接 CPU、内存和显卡（高速设备），南桥负责硬盘、USB、网卡（低速设备）。现代 CPU（如 Intel Core、AMD Ryzen）已将内存控制器和 PCIe 控制器**集成到 CPU 封装内**，北桥消失，只剩下南桥（现称 PCH，平台控制器中枢）。

### SoC 与 NUMA（现代架构趋势）

- **SoC（System on Chip）**：将 CPU、GPU、内存控制器、I/O 控制器集成到单一芯片上。手机处理器（如 Apple M 系列、高通骁龙）和游戏主机（如 PS5、Xbox）都是典型 SoC。
- **NUMA（Non-Uniform Memory Access，非统一内存访问）**：多路服务器中，每个 CPU 插槽有自己"更近"的本地内存 bank。访问远程内存的延迟比本地高 20-50%。游戏引擎的多线程调度若忽视 NUMA，会导致隐性性能损失。

## How：如何观察和验证？

### 实验：测量不同内存层级的访问延迟

通过访问不同大小的数组（强制在不同缓存层级和主存间切换），可以粗略测出各层延迟差异：

```cpp
// workspace/os-lab/cpu-cache/memory_latency.cpp
#include <vector>
#include <chrono>
#include <iostream>
#include <cstdlib>

// 通过随机遍历，破坏硬件预取器，测出真实延迟
void benchmark(size_t bytes, const char* name) {
    size_t count = bytes / sizeof(size_t);
    std::vector<size_t> arr(count);

    // 构建随机跳转链表，确保每次访问都依赖前一次结果
    for (size_t i = 0; i < count; ++i) arr[i] = i;
    for (size_t i = count - 1; i > 0; --i) {
        size_t j = rand() % (i + 1);
        std::swap(arr[i], arr[j]);
    }

    volatile size_t idx = 0;
    auto t1 = std::chrono::high_resolution_clock::now();
    for (int iter = 0; iter < 1000000; ++iter) {
        idx = arr[idx]; // 依赖链，阻止指令重排和并行
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    double ns_per_access = std::chrono::duration<double, std::nano>(t2 - t1).count() / 1000000.0;
    std::cout << name << " (" << bytes / 1024 << " KB): " << ns_per_access << " ns/access\n";
}

int main() {
    benchmark(4 * 1024, "L1 范围");       // ~32KB 以内
    benchmark(256 * 1024, "L2 范围");     // ~256KB
    benchmark(8 * 1024 * 1024, "L3 范围"); // ~8MB
    benchmark(256 * 1024 * 1024, "主存范围"); // 远超 LLC
    return 0;
}
```

> [!warning] 关于 `volatile` 的误区澄清
> 代码中 `volatile size_t idx = 0` 和 `idx = arr[idx]` 的设计目的常被混淆，需要分两个层面理解：
>
> **1. `volatile` 的作用——阻止编译器优化**
> - 防止编译器将循环判定为死代码（Dead Code Elimination）直接删除。因为 `idx` 的最终结果没有任何外部可见副作用（不返回、不打印、不改全局状态），不开优化时编译器会保留，但开 `-O2` 可能整个循环被删掉。
> - 强制编译器每次迭代都生成真正的内存读写指令，而不是把值缓存在寄存器里做常量传播。
>
> **2. `idx = arr[idx]` 的作用——阻止 CPU 指令级并行**
> 这不是 `volatile` 的功能，而是**数据依赖链**天然导致的串行化：第 N 次加载的地址取决于第 N-1 次加载的结果（`idx`）。在 CPU 算出上一个 `idx` 之前，它根本不知道下一个地址在哪，因此无法提前发射后续加载指令，也无法乱序并行执行。
>
> **一句话总结**：`volatile` 负责让编译器"老实生成指令"；`arr[idx]` 的依赖链负责让 CPU"只能串行执行"。两者缺一不可，但各司其职。
>
> > [!caution] 注意
> > `volatile`**不是线程同步原语**，它不提供原子性，也不保证多核间的可见性顺序。多线程场景请使用 `std::atomic`。

**预期结果**：
- L1 范围内（4KB）：~3-5 ns
- L2 范围内（256KB）：~10-15 ns
- L3 范围内（8MB）：~40-60 ns
- 主存范围（256MB）：~100-150 ns

## 游戏引擎延伸

### 为什么主频不再提升？

2000 年代初期，CPU 主频以每年 50% 的速度增长。但到 2005 年左右，Intel 取消了 4GHz+ 计划——因为**功耗墙（Power Wall）**到了。频率提升带来电压提升，功耗与电压的平方成正比（$P \propto CV^2f$）。3GHz 以上的边际功耗收益急剧恶化，芯片散热成为物理极限。因此行业转向**多核 + 宽流水线 + 大缓存**的方向。

> 对引擎开发者的启示：不要指望"下一代 CPU 主频翻倍"来解决单线程瓶颈。并行化（多线程 Job System）和缓存友好化（DOD）是唯一出路。

### SIMD 与图形管线

现代 GPU 本质上是**超大规模 SIMD 处理器**。理解 CPU 的 SSE/AVX 向量指令，是理解 GPU Shader 执行模型的桥梁。两者都遵循"同一条指令，同时处理多组数据"的范式。

### 数据导向设计（DOD）的硬件根源

DOD 不是某种"编程范式偏好"，而是对内存层次结构的直接响应：
- CPU 从主存取一次数据要 100ns，足够执行 300 条指令。
- 如果数据布局不连续（如链表、虚函数表跳转），大部分时间都在等内存。
- 因此 ECS（Entity-Component-System）将同类型数据紧凑排列，让一次缓存加载能喂饱多个运算。
