# 内存对齐与填充

---

## Why：为什么要关注对齐与填充？

### 一句话本质

> **对齐是 CPU 硬件的访问偏好（甚至是强制要求），填充是编译器自动插入的补偿字节。**

想象 CPU 是货车司机，一次能运 8 吨货（8 字节）。你把一件 8 吨的大货（`double`）放在传送带地址 1 的位置——货车得跑两趟才能拉完（地址 1~7 一趟，8~15 一趟）。编译器就是那个仓库管理员，它自动在大货前面垫 7 吨空箱子，让大货对齐到地址 8，这样货车一趟就拉走。

**这就是对齐和填充的协作关系：CPU 提需求，编译器来执行。**

### 不对齐的代价

| 架构             | 未对齐访问的后果                            |
| -------------- | ----------------------------------- |
| **x86**        | 允许未对齐，但可能跨 Cache Line，导致两次加载；跨页边界更慢 |
| **ARM / MIPS** | 直接触发硬件异常（Bus Error），程序崩溃            |

除了崩溃风险，未对齐最大的性能陷阱是：

- **False Sharing（伪共享）**：两个线程修改同一 Cache Line 上的不同变量，缓存反复失效，性能暴跌 3~10 倍。
- **结构体膨胀**：字段顺序不当导致编译器插入大量填充字节，内存占用和带宽压力双双上升。

---

## What：核心概念

### 内存对齐（Alignment）

**对齐要求**：数据类型的起始地址必须是「对齐值」的整数倍。

不同数据类型因为硬件访问粒度的差异，有不同的对齐偏好。你可以把它理解为"停车位的规格"：

- **`char`** 只占 1 字节，相当于一辆自行车，停在哪都不碍事，所以它的对齐值是 1——任何地址都能停。
- **`short`** 占 2 字节，相当于一辆小轿车，喜欢停在偶数号位（0, 2, 4...），对齐值是 2。
- **`int`** 占 4 字节，相当于一辆面包车，必须停在 4 的倍数地址（0, 4, 8...），对齐值是 4。
- **`double`** 和 **64 位指针** 占 8 字节，相当于大货车，必须停在 8 的倍数地址（0, 8, 16...），对齐值是 8。

如果地址不对齐，CPU 要么花两趟才能读完（x86），要么直接抛异常（ARM）。

| 类型 | 大小 | 对齐值 | 合法起始地址示例 |
|------|------|--------|----------------|
| `char` | 1 | 1 | 0, 1, 2, 3 ... |
| `short` | 2 | 2 | 0, 2, 4, 6 ... |
| `int` | 4 | 4 | 0, 4, 8, 12 ... |
| `double` | 8 | 8 | 0, 8, 16, 24 ... |
| 指针（64位） | 8 | 8 | 0, 8, 16, 24 ... |

**结构体对齐值**：其所有成员中**最大的那个对齐值**。

### 填充（Padding）—— 编译器的"自动补位"

编译器在字段之间插入无名字节，确保每个成员都满足对齐要求。这些字节不暴露给程序员，但确实占用内存。

#### 结构体布局计算示例

```cpp
struct Bad {
    char  a; // 1 byte，偏移 0
    // 7 bytes padding ← 编译器自动插入
    double b; // 8 bytes，必须对齐到 8，所以放在偏移 8
    int    c; // 4 bytes，偏移 16
    // 4 bytes padding ← 编译器自动插入
}; // size = 24
```

**逐步拆解 `Bad` 的 24 字节怎么来的：**

1. `char a` 占 1 字节，偏移 0~0。
2. 下一个字段 `double b` 要求 8 字节对齐，当前偏移是 1，不是 8 的倍数。编译器插入 **7 字节填充**，偏移跳到 8。
3. `double b` 占 8 字节，偏移 8~15。
4. `int c` 要求 4 字节对齐，当前偏移是 16，是 4 的倍数，无需填充。`int` 占 4 字节，偏移 16~19。
5. **结构体整体大小必须是对齐值（8）的整数倍**。当前 20 字节，不是 8 的倍数。编译器在尾部再填充 **4 字节**，最终大小 24。

```cpp
struct Good {
    double b; // 8 bytes，偏移 0~7
    int    c; // 4 bytes，偏移 8~11
    char   a; // 1 byte，  偏移 12
    // 3 bytes padding ← 编译器自动插入，让整体大小为 8 的倍数
}; // size = 16
```

**字段从大到小排列，尾部填充最少。**

#### 能不能不要填充？

可以，但要有代价：

```cpp
#pragma pack(push, 1)  // 告诉编译器：按 1 字节对齐，别填充
struct Packed {
    char  a;
    double b;  // 偏移 1，未对齐！
};
#pragma pack(pop)
// sizeof(Packed) == 9，但访问 b 时性能暴跌，ARM 上直接崩溃
```

### 嵌套结构体的对齐

当结构体的成员本身也是结构体时，**把子结构体当成一个"已经编译完成的整体"**。

#### 三条核心规则

1. **子结构体的对齐值** = 它自己内部最大成员的对齐值
2. **子结构体的大小** = 编译器已经填充好的最终大小，必然是其对齐值的整数倍
3. **父结构体的整体对齐值** = 所有成员（包括子结构体）对齐值中的最大值

#### 嵌套布局计算示例

```cpp
struct Inner {
    char  a;    // 1 byte，偏移 0
    // 3 bytes padding
    int   b;    // 4 bytes，对齐值 4
};              // sizeof(Inner) = 8，对齐值 = 4

struct Outer {
    char   x;      // 1 byte，偏移 0
    // 3 bytes padding ← Inner 的对齐值是 4
    Inner  inner;  // 8 bytes，偏移 4~11
    double y;      // 8 bytes，对齐值 8
                   // 当前偏移是 12，不是 8 的倍数
    // 4 bytes padding ← 填充到偏移 16
                   // double y 占 8 字节，偏移 16~23
};                 // 总大小 24，是 max(4, 8) = 8 的倍数
```

**逐步拆解 `Outer`：**

| 步骤 | 操作 | 当前偏移 | 说明 |
|------|------|----------|------|
| 1 | `char x` | 0~0 | 占 1 字节 |
| 2 | 填充 3 字节 | 1~3 | `Inner` 对齐值是 4，下一个 4 的倍数是 4 |
| 3 | `Inner inner` | 4~11 | `sizeof(Inner) == 8`，已经是 4 的倍数 |
| 4 | 填充 4 字节 | 12~15 | `double y` 对齐值是 8，下一个 8 的倍数是 16 |
| 5 | `double y` | 16~23 | 占 8 字节 |
| 6 | 检查整体 | 24 | `Outer` 对齐值 = max(4, 8) = 8，24 是 8 的倍数 ✓ |

**最终 `sizeof(Outer) == 24`**

#### 常见陷阱

```cpp
struct A {
    char a;      // 1
    // 7 padding
    double b;    // 8
};               // sizeof(A) = 16，对齐值 = 8

struct B {
    int x;       // 4，偏移 0~3
    // 4 padding ← A 的对齐值是 8
    A inner;     // 16，偏移 8~23
    char y;      // 1，偏移 24
    // 7 padding ← B 整体对齐值是 8，需要填充到 32
};               // sizeof(B) = 32
```

嵌套一层就可能浪费十几字节，字段顺序在多级结构体中更加重要。

### `alignas` —— 显式指定对齐

```cpp
alignas(64) int counter; // 强制按 64B 对齐，常用于避免 False Sharing
```

### False Sharing（伪共享）

**场景**：两个线程分别写变量 `a` 和 `b`，但它们在物理内存上挨得太近，落在了**同一个 Cache Line**（64 字节）里。

```
CPU Core 0 写 a          CPU Core 1 写 b
     ↓                        ↓
   [a|b|...]  ← 同一 Cache Line（64B）
```

**问题根源**：CPU 缓存一致性协议（MESI）以 **Cache Line 为单位**同步，而不是以变量为单位。
- Core 0 修改了 `a` → 整个 Cache Line 标记为"我改了"
- Core 1 想修改 `b` → 发现这个 Cache Line 被改了 → 被迫从内存重新加载
- 两边来回互相 Invalidate，性能暴跌

**解决方案**：用 `alignas(64)` 或手动填充，把变量隔离到不同 Cache Line。

```cpp
struct AlignedCounters {
    alignas(64) std::atomic<int> a{0}; // a 独占一个 Cache Line
    alignas(64) std::atomic<int> b{0}; // b 独占另一个 Cache Line
};
```

### SIMD 与向量对齐

> 先澄清一个常见误区：**SIMD 不是 GPU 的概念，它是 CPU 的。**

SIMD（Single Instruction, Multiple Data）是 CPU 的**向量化并行计算**技术。x86 CPU 的 SSE/AVX 指令集就是 SIMD 的实现。一条指令可以同时处理多个数据，例如 `__m128` 一次处理 4 个 `float`。

**GPU 和 SIMD 的关系**：GPU 早期确实走 SIMD 路线，但现代 GPU 使用的是更灵活的 **SIMT**（Single Instruction, Multiple Threads）。SIMD 是显式向量寄存器，SIMT 是隐式硬件调度。

#### 为什么 SIMD 对对齐极度敏感？

SIMD 寄存器宽度远超普通标量类型：

| SIMD 类型 | 寄存器宽度 | 对齐要求 | 典型用途 |
|-----------|-----------|----------|----------|
| `__m128` (SSE) | 128 bit / 16 B | **16 B** | 4×float |
| `__m128d` (SSE2) | 128 bit / 16 B | **16 B** | 2×double |
| `__m128i` (SSE2) | 128 bit / 16 B | **16 B** | 整数向量 |
| `__m256` (AVX) | 256 bit / 32 B | **32 B** | 8×float |
| `__m512` (AVX-512) | 512 bit / 64 B | **64 B** | 16×float |

#### 对齐与未对齐的指令差异

x86 提供了两套加载指令：

- **`movaps`**（Aligned Packed Single）：要求地址严格对齐。不对齐直接触发 **#GP 异常**（程序崩溃）。
- **`movups`**（Unaligned Packed Single）：允许未对齐地址，但可能跨越 Cache Line，性能显著下降。

```cpp
#include <xmmintrin.h>

alignas(16) float data[4] = {1,2,3,4};
__m128 vec = _mm_load_ps(data);  // movaps，要求 16B 对齐
```

如果 `data` 没有 16B 对齐，`_mm_load_ps` 会崩溃。编译器通常会自动处理对齐，但在手动内存分配（如 `malloc`、`new`）或结构体布局中，程序员必须显式保证。

#### 结构体中的 SIMD 字段

```cpp
struct Vec4 {
    float x, y, z, w;
};

struct BadSIMD {
    char tag;
    // 15 bytes padding ← 编译器插入，让 vec 对齐到 16
    __m128 vec;
};

struct GoodSIMD {
    __m128 vec;  // 16B 对齐
    char tag;    // 偏移 16
    // 15 bytes padding ← 尾部填充到 32B（16 的倍数）
};
```

`__m128` 的对齐值是 16，因此包含它的结构体整体对齐值也会提升到 16。这会导致：

1. 字段前可能需要大量填充。
2. 结构体尾部也可能需要填充。
3. 动态分配时必须使用 `_mm_malloc` 或 `alignas` / `std::aligned_alloc`。

> [!tip]
> 在 ECS 或粒子系统中，将 `float x,y,z` 填充到 16B 再存储，或者直接用 `__m128`，都是为了适配 SSE 指令的加载粒度，属于典型的 SIMD 对齐优化。

---

## How：如何实践？

1. **结构体字段从大到小排列**，减少编译器插入的内部和尾部填充。
2. **对高频多线程写变量显式对齐到 Cache Line**（`alignas(64)`）。
3. **使用 `sizeof` 和 `offsetof` 检查结构体布局**，确认编译器实际生成的排布。`sizeof` 返回类型总大小，`offsetof(类型, 字段名)` 返回字段在结构体中的字节偏移——两者配合，你就能"看见"编译器塞进去的填充字节。

## C++ 本地验证实验

### 实验 1：结构体布局与大小

```cpp
// workspace/os-lab/cpu-cache/alignment_size.cpp
#include <iostream>
#include <cstddef>

struct Bad {
    char a;
    double b;
    int c;
};

struct Good {
    double b;
    int c;
    char a;
};

struct Padded {
    char a;
    char pad[7];
    double b;
    int c;
    char pad2[4];
};

int main() {
    std::cout << "Bad : " << sizeof(Bad)  << " (a@" << offsetof(Bad, a)
              << " b@" << offsetof(Bad, b) << " c@" << offsetof(Bad, c) << ")\n";
    std::cout << "Good: " << sizeof(Good) << " (b@" << offsetof(Good, b)
              << " c@" << offsetof(Good, c) << " a@" << offsetof(Good, a) << ")\n";
    std::cout << "Padded: " << sizeof(Padded) << "\n";
    return 0;
}
```

### 实验 2：False Sharing 性能对比

```cpp
// workspace/os-lab/cpu-cache/false_sharing.cpp
#include <thread>
#include <atomic>
#include <chrono>
#include <iostream>

constexpr int ITERATIONS = 10'000'000;

// 未对齐，可能共享同一 Cache Line
struct SharedCounters {
    std::atomic<int> a{0};
    std::atomic<int> b{0};
};

// 显式隔离到不同 Cache Line
struct AlignedCounters {
    alignas(64) std::atomic<int> a{0};
    alignas(64) std::atomic<int> b{0};
};

template<typename T>
void bench(const char* name) {
    T counters;
    auto t1 = std::chrono::high_resolution_clock::now();
    std::thread t([&]{ for(int i=0;i<ITERATIONS;++i) counters.a.fetch_add(1, std::memory_order_relaxed); });
    for(int i=0;i<ITERATIONS;++i) counters.b.fetch_add(1, std::memory_order_relaxed);
    t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2-t1).count()
              << " ms\n";
}

int main() {
    bench<SharedCounters>("Shared (false sharing)");
    bench<AlignedCounters>("Aligned (64B cache line)");
    return 0;
}
```

**预期结果**：对齐版本通常快 **3~10 倍**（取决于核心数与缓存架构）。

对齐和填充看似是底层细节，但在高性能引擎中，它们直接影响多线程效率和 SIMD 吞吐量。

## 游戏引擎延伸

### Job System 的任务对齐

- 多线程 Job System 中，每个工作线程有自己的任务队列或计数器。
- 若多个线程的队列头/尾指针放在相邻内存，极易发生 False Sharing。
- 最佳实践：每个线程的私有状态用 `alignas(64)` 隔离。

### 粒子/ECS 组件对齐

- ECS 的组件数组（Component Array）通常会按类型对齐分配，保证 SIMD（SSE/AVX）指令能高效加载。
- 例如 `Position` 组件包含 `float x,y,z`，可填充到 16B 对齐以配合 `__m128`。

## 核心速查表

| 问题 | 答案 |
|------|------|
| 谁要求对齐？ | **CPU 硬件架构**（x86 宽松、ARM 严格） |
| 谁插入填充？ | **编译器**（自动补位以满足对齐） |
| 对齐值谁定？ | 硬件决定（`char=1, int=4, double=8`），编译器遵守 |
| 能不能取消填充？ | 能（`#pragma pack`），但可能性能下降或崩溃 |
| False Sharing 怎么破？ | `alignas(64)` 把变量隔离到不同 Cache Line |
