# 内存对齐与填充

---

## Why：为什么要关注对齐与填充？

- **未对齐访问的代价**：x86 允许未对齐访问，但可能跨 Cache Line，导致两次加载；某些架构（ARM）会直接触发异常。
- **False Sharing（伪共享）**：两个线程修改同一 Cache Line 上的不同变量，导致缓存反复失效，性能暴跌。
- **结构体膨胀**：字段顺序不当会导致内部空洞，增加内存占用和带宽压力。

---

## What：核心概念

### 内存对齐（Alignment）

- **对齐要求**：数据类型的起始地址必须是对齐值的整数倍。
- **常见对齐**：`char=1`, `short=2`, `int=4`, `double=8`, `指针=8`（64位）。
- **结构体对齐值**：其最大成员的对齐值。

```cpp
struct Bad {
    char  a; // 1 byte
    // 7 bytes padding
    double b; // 8 bytes
    int    c; // 4 bytes
    // 4 bytes padding
}; // size = 24

struct Good {
    double b; // 8
    int    c; // 4
    char   a; // 1
    // 3 bytes padding
}; // size = 16
```

### 填充（Padding）与 `alignas`

```cpp
alignas(64) int counter; // 强制按 64B 对齐，常用于避免 False Sharing
```

### False Sharing

- 两个线程分别写 `a` 和 `b`，若 `a` 与 `b` 落在同一 Cache Line（64B），则每次写都会触发 MESI 的 Invalidate，导致另一个核心重新加载。
- **解决方案**：用 `alignas(64)` 或填充将变量隔离到不同 Cache Line。

---

## How：如何实践？

1. **结构体字段从大到小排列**，减少尾部填充。
2. **对高频多线程写变量显式对齐到 Cache Line**。
3. **使用 `offsetof` / `sizeof` 检查结构体布局**。

## C++ 本地验证实验

### 实验 1：结构体布局与大小

```cpp
// workspace/os-lab/cpu-cache/alignment_size.cpp
#include <iostream>
#include <cstddef>

struct Bad {
    char a;
    double b;
    int c;
};

struct Good {
    double b;
    int c;
    char a;
};

struct Padded {
    char a;
    char pad[7];
    double b;
    int c;
    char pad2[4];
};

int main() {
    std::cout << "Bad : " << sizeof(Bad)  << " (a@" << offsetof(Bad, a)
              << " b@" << offsetof(Bad, b) << " c@" << offsetof(Bad, c) << ")\n";
    std::cout << "Good: " << sizeof(Good) << " (b@" << offsetof(Good, b)
              << " c@" << offsetof(Good, c) << " a@" << offsetof(Good, a) << ")\n";
    std::cout << "Padded: " << sizeof(Padded) << "\n";
    return 0;
}
```

### 实验 2：False Sharing 性能对比

```cpp
// workspace/os-lab/cpu-cache/false_sharing.cpp
#include <thread>
#include <atomic>
#include <chrono>
#include <iostream>

constexpr int ITERATIONS = 10'000'000;

// 未对齐，可能共享同一 Cache Line
struct SharedCounters {
    std::atomic<int> a{0};
    std::atomic<int> b{0};
};

// 显式隔离到不同 Cache Line
struct AlignedCounters {
    alignas(64) std::atomic<int> a{0};
    alignas(64) std::atomic<int> b{0};
};

template<typename T>
void bench(const char* name) {
    T counters;
    auto t1 = std::chrono::high_resolution_clock::now();
    std::thread t([&]{ for(int i=0;i<ITERATIONS;++i) counters.a.fetch_add(1, std::memory_order_relaxed); });
    for(int i=0;i<ITERATIONS;++i) counters.b.fetch_add(1, std::memory_order_relaxed);
    t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2-t1).count()
              << " ms\n";
}

int main() {
    bench<SharedCounters>("Shared (false sharing)");
    bench<AlignedCounters>("Aligned (64B cache line)");
    return 0;
}
```

**预期结果**：对齐版本通常快 **3-10 倍**（取决于核心数与缓存架构）。

## 游戏引擎延伸

### Job System 的任务对齐

- 多线程 Job System 中，每个工作线程有自己的任务队列或计数器。
- 若多个线程的队列头/尾指针放在相邻内存，极易发生 False Sharing。
- 最佳实践：每个线程的私有状态用 `alignas(64)` 隔离。

### 粒子/ECS 组件对齐

- ECS 的组件数组（Component Array）通常会按类型对齐分配，保证 SIMD（SSE/AVX）指令能高效加载。
- 例如 `Position` 组件包含 `float x,y,z`，可填充到 16B 对齐以配合 `__m128`。
