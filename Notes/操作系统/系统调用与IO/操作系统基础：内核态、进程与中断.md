# 操作系统基础：内核态、进程与中断

---

## Why：为什么要理解内核态、进程与中断？

- **系统调用的根基**：游戏引擎中每一次文件读取、网络收发、内存分配，最终都走向系统调用。不理解内核态和用户态的边界，就无法量化"syscall 开销"，也无法理解 vDSO、io_uring 等优化技术的动机。
- **多线程编程的安全边界**：进程地址空间隔离、线程共享内存、信号量与互斥锁的实现，都建立在操作系统对进程和线程的抽象之上。
- **实时性问题**：游戏掉帧有时并非逻辑复杂，而是第 N 帧恰好触发了一次中断风暴、或触发了内核的脏页回写。理解中断机制是诊断这类问题的起点。

---

## What：操作系统的核心抽象

### 特权级与保护环（Protection Rings）

现代 CPU 提供多个**特权级（Privilege Levels）**，x86 架构称为 **Ring 0 ~ Ring 3**：

| 层级 | 名称 | 运行代码 | 权限 |
|------|------|----------|------|
| Ring 0 | 内核态（Kernel Mode） | 操作系统内核、驱动程序 | 完全访问：所有内存、所有指令、所有硬件寄存器 |
| Ring 1~2 | 保留/驱动层 | 部分设备驱动（历史遗留，现代 OS 基本不用） | 中间权限 |
| Ring 3 | 用户态（User Mode） | 应用程序（游戏引擎、浏览器） | 受限：只能访问自己的地址空间，禁止执行特权指令 |

> **关键机制**：用户态程序若尝试执行特权指令（如直接访问硬盘控制器、修改页表），CPU 会触发**通用保护故障（General Protection Fault）**，操作系统将其转换为段错误（Segfault）或非法指令信号，终止进程。这是操作系统**隔离**和**安全**的硬件根基。

### 内核态 vs 用户态：一次系统调用的全貌

当用户态程序请求操作系统服务时（如 `read()` 读文件），发生一次**系统调用（System Call）**：

1. **触发**：用户程序执行 `syscall` 指令（x86_64）或 `int 0x80`（ legacy 方式）。
2. **切换**：CPU 从 Ring 3 提升到 Ring 0，跳转到内核预设的**系统调用处理函数**。
3. **执行**：内核验证参数合法性，执行实际工作（如从磁盘读取数据到 Page Cache）。
4. **返回**：内核将结果写入用户缓冲区，执行 `sysret`/`iret` 指令，CPU 降回 Ring 3，用户程序继续执行。

> **开销来源**：状态切换本身约需 **100ns ~ 1μs**，主要来自：
> - CPU 上下文保存/恢复（寄存器、程序计数器）
> - 流水线刷新（特权级变化导致 CPU 推测执行失效）
> - TLB 刷新（部分架构在切换时刷新地址翻译缓存）
> - 缓存污染（内核代码和数据挤占用户态的缓存行）

### 进程（Process）：资源隔离的基本单位

**进程不是"运行中的程序"这么简单**。进程是操作系统分配资源的**最小单位**，包含：

- **独立的虚拟地址空间**：通过页表与物理内存隔离，进程 A 无法直接访问进程 B 的内存。
- **进程控制块（PCB / Process Control Block）**：内核中的数据结构，记录进程 ID、状态（运行/就绪/阻塞/僵尸）、寄存器快照、打开的文件描述符表、调度优先级等。
- **至少一个执行线程**：传统进程隐含一个主线程；现代操作系统（Linux）中线程本质上是"共享地址空间的轻量级进程"。

#### 进程的创建：fork 与 exec

- **`fork()`**：创建当前进程的完整副本（写时复制，Copy-On-Write）。子进程获得独立的 PCB，但初始时与父进程共享物理内存页——只有当某一方修改页面时，内核才真正复制该页。
- **`exec()`**：用新程序替换当前进程的地址空间。`fork` 后通常紧跟 `exec`，这是 Shell 执行命令的标准模式（先 fork 出子进程，再在子进程中 exec 加载新程序）。

> 游戏引擎中很少直接使用 `fork`/`exec`（多线程更常见），但理解它们有助于理解进程隔离、守护进程、以及 Linux 的 `clone` 系统调用（线程的底层实现）。

### 中断（Interrupt）与异常（Exception）

CPU 不会"被动等待"外部事件，它通过**中断机制**实现异步响应：

| 类型 | 触发源 | 示例 |
|------|--------|------|
| **硬件中断** | 外部设备 | 键盘按键、网卡收到数据包、硬盘 IO 完成 |
| **软件中断** | 指令触发 | `int 0x80`（旧式 syscall）、`syscall` 指令 |
| **异常** | CPU 执行错误 | 除零、非法指令、Page Fault（缺页） |

#### 中断处理流程

1. **中断请求（IRQ）**：设备通过中断控制器（如 APIC）向 CPU 发送信号。
2. **中断响应**：当前指令执行完毕后，CPU 查询**中断描述符表（IDT）**，找到对应处理函数的地址。
3. **上下文切换**：CPU 保存当前寄存器状态，切换到内核态，执行中断处理程序（Interrupt Service Routine, ISR）。
4. **返回**：ISR 执行完毕，恢复之前保存的寄存器，用户程序继续执行（对程序透明）。

> **关键概念**：中断有**优先级**。高优先级中断（如时钟中断、核间中断）可以打断低优先级中断的处理。操作系统利用**时钟中断**（通常每 1ms ~ 10ms）实现抢占式调度——每次时钟中断，内核检查是否有更高优先级的线程需要运行，若有则切换上下文。

#### Page Fault：一种特殊的异常

`Page Fault` 不是"错误"，而是虚拟内存机制的**正常组成部分**：
- 程序访问的虚拟页尚未映射物理页，CPU 触发 Page Fault 异常。
- 内核的 Page Fault 处理程序分配物理页（或从磁盘加载数据），更新页表，然后**恢复执行触发异常的指令**。
- 从程序视角看，仿佛什么都没发生——只是这次内存访问慢了一些（毫秒级，如果涉及磁盘）。

## How：如何观察和验证？

### 实验 1：观察系统调用的开销

```cpp
// workspace/os-lab/syscall/syscall_overhead.cpp
#include <iostream>
#include <chrono>
#include <unistd.h>  // Linux: getpid()
#include <windows.h> // Windows: GetCurrentProcessId()

constexpr int ITER = 1'000'000;

int main() {
    auto t1 = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < ITER; ++i) {
#ifdef _WIN32
        volatile auto id = GetCurrentProcessId(); // 用户态函数，无 syscall
#else
        volatile auto id = getpid(); // Linux: 早期版本是 syscall，现多为 vDSO 优化
#endif
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    auto t3 = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < ITER; ++i) {
#ifdef _WIN32
        volatile auto t = GetTickCount64(); // Windows: 可能走 syscall
#else
        volatile auto t = syscall(SYS_getpid); // 强制走 syscall 指令
#endif
    }
    auto t4 = std::chrono::high_resolution_clock::now();

    double user_ns = std::chrono::duration<double, std::nano>(t2 - t1).count() / ITER;
    double syscall_ns = std::chrono::duration<double, std::nano>(t4 - t3).count() / ITER;

    std::cout << "User call: " << user_ns << " ns\n";
    std::cout << "Syscall:   " << syscall_ns << " ns\n";
    std::cout << "Overhead:  " << (syscall_ns - user_ns) << " ns\n";
    return 0;
}
```

**预期结果**：直接系统调用比用户态函数慢 **50~300ns**（取决于 CPU 和内核版本），这就是高频 syscall 成为性能瓶颈的原因。

### 实验 2：用 strace（Linux）观察程序的系统调用

```bash
# 追踪一个简单的程序的所有系统调用
strace -c ./your_program
# 输出统计：每个 syscall 的调用次数和耗时

# 实时查看
strace -e trace=read,write,mmap ./your_program
```

**预期结果**：你会发现 `printf` 背后有大量的 `write` 系统调用，`new`/`delete` 可能触发 `brk` 或 `mmap`。

## 游戏引擎延伸

### 为什么游戏引擎要减少系统调用？

- **渲染循环的确定性**：一帧 16.6ms（60fps）中，若某帧因 `read()` 触发了磁盘 IO（Major Page Fault），可能直接掉帧。
- **批量处理策略**：引擎在启动时一次性 `mmap` 资源包，运行时通过指针访问，将磁盘 IO 转化为可控的 Page Fault。
- **vDSO 的利用**：Linux 将 `gettimeofday`、`getcpu` 等高频调用映射到用户态共享页，避免真正的 syscall 开销。引擎的计时器实现应优先使用这些接口。

### 中断与实时音频

音频回放通常由声卡通过**DMA**直接读取内存中的音频缓冲区，并通过**中断**通知 CPU"缓冲区即将耗尽"。若 CPU 被高优先级任务（如渲染）占用，未能及时响应音频中断，就会发生**爆音（Audio Pop）**。因此专业音频线程通常绑定到独立核心，并提升线程优先级（甚至使用实时调度策略）。
