# 虚拟内存与地址翻译

---

## Why：为什么要虚拟内存？

上一篇[[编译器、链接器与内存布局]]中，我们知道了程序编译链接后拥有一张"地址蓝图"——从 `0x400000` 开始的 Text 段、往上增长的 Heap、往下增长的 Stack。但程序真正跑起来时，面临三个根本性的物理现实问题：

### 问题一：内存隔离

如果进程 A 直接读写物理地址 `0x10000`，进程 B 也在用这块物理内存，就会互相踩数据。操作系统必须保证**进程之间互不干扰**。

### 问题二：地址重定位

程序编译时假设自己从某个固定地址开始运行（如 `0x400000`），但物理内存的可用区域是动态变化的。如果物理地址 `0x400000` 已经被占用了怎么办？程序不能每次运行都重新编译。

### 问题三：物理内存不够大

你只有 16GB 物理内存，但进程 A 想申请 8GB，进程 B 也想申请 8GB，进程 C 还想再申请 4GB。总和超过了物理内存，怎么办？

**虚拟内存（Virtual Memory）** 就是操作系统给出的统一解决方案：

> 给每个进程一套独立的**虚拟地址空间**，让进程"以为"自己独占整片内存；再通过**页表**把虚拟地址动态映射到物理地址；不活跃的页还能**换出到磁盘**——用磁盘空间来"假装"内存。

这样同时解决了隔离、重定位和超量使用三个问题。而**地址翻译**就是这套机制的核心引擎：CPU 最终访问的是物理内存，MMU 必须通过查表把虚拟地址翻译成物理地址。

> **一句话**：操作系统维护一张**页表**，MMU 每次内存访问时查这张表，把"虚拟页号"换成"物理页帧号"，再加上不变的页内偏移，就得到了物理地址。

## What：虚拟内存的两层含义

理解虚拟内存时，有一个常见的混淆：操作系统教科书里的"虚拟内存"和 Windows 控制面板里的"虚拟内存"并不完全是一回事。它们恰好对应了虚拟内存的**两层含义**。

### 第一层：虚拟地址空间（地址翻译）

每个进程拥有一片独立的虚拟地址空间（x86_64 上通常是低 48 位，即 256TB）。进程里的所有指针、数组地址、函数返回地址，都是虚拟地址。CPU 通过 MMU 和页表，在运行时把虚拟地址翻译成物理地址。

这一层解决的是**隔离**和**重定位**问题：
- 进程 A 的 `0x10000` 和进程 B 的 `0x10000` 通过不同页表映射到不同的物理页帧，互不干扰。
- 程序编译时硬编码的地址是虚拟地址，操作系统可以在加载时通过页表把它映射到任意可用的物理位置。

### 第二层：用磁盘扩展内存（换页 / Swapping）

虚拟内存更精髓的设计在于：**并非所有虚拟页都必须同时存在于物理内存中**。操作系统可以把暂时不用的页"换出"到磁盘，需要时再"换入"。这让进程能够使用比物理内存更大的地址空间。

```
进程的虚拟地址空间（比如 128GB 的地址范围）
       │
       ├── 活跃的页 ──→ 映射到物理内存（RAM）
       │
       └── 不活跃的页 ──→ 换出到磁盘（Page File / 交换分区）
                ↑
           需要访问时触发 Page Fault，换入到物理内存
```

> **关键洞察**：页表中的 `Present=0` 既可能表示"这一页从未加载过"（按需分页），也可能表示"这一页曾被换出到磁盘"。Page Fault 处理程序会根据具体情况，要么从磁盘读取可执行文件/数据文件，要么从交换空间读回被换出的页。

**Windows 控制面板里的"虚拟内存"，指的就是这层换页机制所用的磁盘空间——分页文件（`pagefile.sys`）的大小。** 我们在后文会详细解释它与"虚拟内存不足"报错的关系。

## What：地址翻译是如何工作的

### 大图景：虚拟地址的分解

想象你去图书馆找书：

- 图书馆有 1000 个书架，每个书架 100 层
- 书的编号规则是 `书架号 × 100 + 层号`
- 拿到编号后，管理员先看"书架号"找到书架，再看"层号"找到具体位置

虚拟地址也是类似的结构。以 x86_64 的 **4KB 页**为例：

```
虚拟地址（64位，实际用低48位）：
┌────────────────────────┬─────────────┐
│      虚拟页号           │   页内偏移   │
│   (36位，标识第几页)     │ (12位，0~4095)│
└────────────────────────┴─────────────┘
            ↓ 查页表
┌────────────────────────┬─────────────┐
│      物理页帧号         │   页内偏移   │
│   (36位，标识物理第几帧)  │ (12位，不变) │
└────────────────────────┴─────────────┘
            ↓
         物理地址
```

**关键**：页内偏移不需要翻译，直接透传。MMU 的工作只有一个——把"虚拟页号"换成"物理页帧号"。

**页表（Page Table）** 就是这张映射表，由操作系统为每个进程独立维护，存储在物理内存中。

> **关键洞察**：每次内存访问都要查页表，而页表本身也在内存里。这意味着访问一个变量可能需要**多次内存访问**——先查页表，再访问变量。这就是 TLB 存在的理由。

### TLB：给页表加缓存

如果每次访问变量都要先读内存查页表，性能损失太大。x86_64 的 4 级页表需要读 **4 次内存**才能拿到物理地址。

**TLB（Translation Lookaside Buffer）** 是 CPU 内部的高速缓存，专门缓存"虚拟页号 → 物理页帧号"的映射关系。

| 场景 | 过程 | 开销 |
|------|------|------|
| **TLB 命中** | CPU 直接在 TLB 中找到映射 | 近乎零额外开销 |
| **TLB 未命中** | 去内存查页表 → 填入 TLB → 继续执行 | 可能数百个周期 |

**类比**：TLB 就像快递站的"常用地址速查本"。大部分快递的地址都在速查本里，少数查不到的才去翻厚厚的全市地址簿（页表）。

现代 CPU 通常有两级 TLB（L1 TLB 32~64 项，L2 TLB 1024~2048 项），且分为指令 TLB（ITLB）和数据 TLB（DTLB）。

### Page Fault：按需加载的真相

前一篇我们提到操作系统采用**按需分页（Demand Paging）**——程序启动时并不加载全部代码，只有访问到某页时才加载。

这个"访问到才发现没加载"的机制，就是通过 **Page Fault（缺页中断）** 实现的：

```
CPU 访问虚拟地址
    ↓
MMU 查页表，发现该页标记为"不在内存"（Present = 0）
    ↓
触发 Page Fault（CPU 异常）
    ↓
操作系统接管：
  1. 找一个空闲物理页帧
  2. 从磁盘读取对应数据到该页帧
  3. 更新页表：虚拟页 → 物理页帧，标记 Present = 1
  4. 恢复 CPU 执行（重新执行那条指令）
    ↓
这次访问成功
```

| 类型 | 原因 | 开销 |
|------|------|------|
| **Major Fault** | 需要读磁盘：可能是程序从未加载过的代码/数据，也可能是之前被**换出到磁盘**的页 | 毫秒级（磁盘 I/O） |
| **Minor Fault** | 只需分配并清零一页物理内存（如 BSS 段、新分配的堆页） | 微秒级 |

**Major Fault 的两种来源**：

1. **按需加载（Demand Paging）**：程序首次运行某段代码或首次访问某个数据页时，操作系统从可执行文件/数据文件中读取。这是笔记前文讲的场景。
2. **换入（Swap In）**：该页之前因为物理内存紧张被操作系统换出到了磁盘的交换空间。现在进程再次访问它，必须从磁盘读回。这是虚拟内存"用磁盘扩展内存"的直接体现。

> **关键洞察**：Page Fault 对程序完全透明。CPU 自动陷入内核、加载数据、恢复执行，程序员感知不到。唯一的代价是首次访问（或换入后再次访问）时的延迟——这也是游戏加载阶段"预触摸"内存的理论基础。

### 页表长什么样

如果为整个 48 位虚拟地址空间建立一张"扁平"页表，需要 2^36 个条目 × 8 字节 = **512GB**，显然不现实。

**多级页表**的设计动机：大部分虚拟地址空间是空的（程序用不了那么多内存），多级页表只在"用到了的地址范围"才分配下级页表，大幅节省空间。

x86_64 使用 **4 级页表**：

```
CR3 寄存器（存顶级页表物理地址）
    ↓
PML4（Page Map Level 4）—— 用虚拟地址的高 9 位索引
    ↓
PDPT（Page Directory Pointer Table）—— 用接下来的 9 位索引
    ↓
PD（Page Directory）—— 再用 9 位索引
    ↓
PT（Page Table）—— 再用 9 位索引
    ↓
物理页帧号 + 12 位页内偏移 = 物理地址
```

每一级用虚拟地址的一部分作为索引，就像查字典的"部首 → 笔画 → 页码"多级索引。

**CR3** 是 x86_64 的控制寄存器，存放当前进程顶级页表（PML4）的物理基地址。进程切换时，操作系统修改 CR3，CPU 就自动使用新进程的页表。

> 每次内存访问若 TLB 未命中，需要遍历 4 级页表 = **4 次内存访问**。TLB 的作用就是把这 4 次内存访问压缩到 0 次。

### HugePage 与 mmap

#### HugePage（大页）

4KB 页虽然通用，但面对大内存工作集时 TLB 压力很大。比如访问 2GB 连续内存，4KB 页需要 512K 个页表项。

**HugePage** 使用 **2MB** 甚至 **1GB** 的大页，大幅减少页表级数和 TLB 压力：

| 页大小 | 2GB 内存需要多少个页 | 页表遍历次数 |
|--------|---------------------|-------------|
| 4KB | 524,288 个 | 4 级 |
| 2MB | 1,024 个 | 3 级（中间一级跳过） |
| 1GB | 2 个 | 2 级（跳过多级） |

游戏服务器、数据库常开启 HugePage；游戏引擎的大世界流式加载也受益于大页映射。

#### mmap（内存映射文件）

`mmap`（Memory Map）是将文件内容直接映射到进程虚拟地址空间的技术——**让文件看起来像一块内存**，用户态可以像访问数组一样读写文件，无需显式调用 `read` / `write`。

```
传统 read:  磁盘 ──→ 内核缓存 ──→ 用户缓冲区 ──→ 你的代码
mmap:       磁盘 ──→ 内核缓存 ←─────────────── 你的代码直接访问
```

| 优势 | 传统 `read()` | `mmap` |
|------|-------------|--------|
| **数据拷贝** | 磁盘 → 内核缓存 → 用户缓冲区（2 次拷贝） | 磁盘 → 内核缓存，直接映射到用户空间（零拷贝） |
| **内存占用** | 用户态需额外分配同等大小 buffer | 共享内核页缓存，无需额外大 buffer |
| **加载策略** | 一次性读入（或分块循环 read） | 按需加载（Lazy Loading），访问时才触发缺页 |
| **随机访问** | `lseek` + `read`，每次都有 syscall 开销 | 直接指针偏移 `data[offset]`，无 syscall |
| **进程共享** | 各进程独立缓存，内存冗余 | 多进程映射同一文件，共享物理页 |

> **核心原理**：`mmap` 只建立"虚拟地址 → 文件"的映射关系，并不立即读数据。当 CPU 首次访问某页时触发 **Page Fault**，操作系统才从磁盘加载对应块到页缓存，并建立页表映射。这正是虚拟内存**按需加载**思想的直接应用。

**不是银弹**：小文件（< 1MB）`mmap` 有页对齐和映射建立开销，直接用 `read` 更快；频繁顺序追加写入时，标准 I/O 的缓冲区聚合写入效率更高；文件可能被截断的场景需处理 `SIGBUS`。

### 虚拟地址与物理地址：两个关键认知

理解了地址翻译的机制后，还有两个常见的直觉误区需要澄清。它们恰好触及了"虚拟内存抽象"与"底层物理现实"之间的边界。

#### 误区一：虚拟地址连续，物理地址也连续吗？

**答案：不一定，通常是不连续的。**

页表映射的最小单位是页（4KB），MMU 只是把"虚拟页号"替换为"物理页帧号"。两个相邻的虚拟页（如虚拟地址 `0x1000` 和 `0x2000`）完全可以映射到物理内存中相隔很远的页帧（如物理页帧 `0x5000` 和 `0xA000`）。

物理内存经过长期分配和释放后高度碎片化，操作系统正是通过页表屏蔽了这种不连续性。但注意：**同一页内的地址**（`0x1000` ~ `0x1FFF`）在物理上一定是连续的——页内偏移直接透传，页是映射的最小单位。

> **例外**：HugePage（2MB/1GB）和 DMA 缓冲区在分配时，操作系统会尽量寻找物理连续的内存块。

#### 误区二：虚拟地址提出后，Cache、Cache Line、Cache Miss 还有意义吗？

**非常有意义。** Cache 是物理地址层面的硬件，但程序看到的是虚拟地址。两者之间的交互正是高性能编程的关键。

**现代 CPU 的 Cache 实现**

| 模式 | 索引用 | 标签用 | 特点 |
|------|--------|--------|------|
| VIVT | 虚拟地址 | 虚拟地址 | 简单，但有**别名问题** |
| VIPT | 虚拟地址 | 物理地址 | 现代 L1 Cache 常用 |
| PIPT | 物理地址 | 物理地址 | 无别名问题，x86_64 L2/L3 常用 |

无论哪种实现，Cache 最终都依赖**物理地址标签**判断是否命中。但这并不意味着虚拟地址层面的讨论失去了意义：

1. **Cache Line 对齐仍然有效**：Cache Line 大小通常为 64B，远小于页大小 4KB。由于页内偏移在翻译中不变，虚拟地址上的 `alignas(64)` 在物理层面完全等效。
2. **False Sharing 依然关键**：多线程访问同一 Cache Line 的不同变量（即使虚拟地址不同），仍会触发多核间的 Cache Coherence 同步（MESI 状态切换）。你在代码中控制的结构体字段排列、数组填充，正是在虚拟地址层面消除 False Sharing。
3. **TLB Miss 与 Cache Miss 是两个独立层次**：
   - 可以 TLB Hit 但 Cache Miss（无需查页表，但数据不在 Cache）
   - 可以 TLB Miss 但 Cache Hit（页表数据恰好被 Cache 缓存）

这 4 级页表遍历本身也会经历 Cache 查找——如果页表在 L1/L2 Cache 中，TLB Miss 的代价就会大幅降低。

> **一句话总结**：虚拟地址的连续性被页表打破了，但页内偏移的连续性保留了下来；Cache 工作在物理地址层面，但因为 Cache Line（64B）远小于页（4KB），你在虚拟地址层面的缓存优化在物理层面完全等效。

## 换页与交换空间

理解了地址翻译和 Page Fault 后，我们回到虚拟内存的**第二层含义**——用磁盘来扩展物理内存。这是操作系统实现"进程可以使用比物理内存更大的地址空间"的核心机制。

### 为什么需要换页？

假设你的电脑有 16GB 物理内存，同时运行着：
- 浏览器（占用 4GB）
- IDE（占用 3GB）
- 游戏（占用 8GB）
- 后台服务（占用 2GB）

总和是 17GB，超过了物理内存。没有虚拟内存的话，操作系统必须杀掉某个进程。但有了换页机制，操作系统可以：

1. 把所有进程的虚拟页都保留在页表中。
2. 只把**最近活跃**的页保留在物理内存中。
3. 把**长时间未访问**的页写到磁盘上的交换空间（Swap Space / Page File）。
4. 当进程访问到已被换出的页时，触发 Page Fault，从磁盘读回。

从进程的角度看，它仍然拥有完整的虚拟地址空间；从物理现实看，只有活跃数据在内存里，不活跃数据在磁盘上。

### 换出（Swap Out）的流程

操作系统有一个**页淘汰（Page Replacement）**守护线程，持续监控物理内存使用情况：

```
物理内存紧张时
    ↓
页淘汰线程选择"最不可能近期被访问"的页
    ↓
检查该页的 Dirty 位：
  - Dirty = 1（被修改过）→ 写回磁盘交换空间
  - Dirty = 0（未修改）→ 直接丢弃（如代码段可从原文件重新读取）
    ↓
更新页表：Present = 0，记录该页在磁盘上的位置
    ↓
释放物理页帧，供其他进程使用
```

> **关键洞察**：代码段（Text）通常不会被修改（只读），换出时不需要写磁盘——需要时直接从原始可执行文件重新读取即可。而堆、栈中的数据页如果被修改过（Dirty=1），换出时必须写回交换空间，否则数据会丢失。

### 交换空间 vs 分页文件

| 术语 | 对应系统 | 说明 |
|------|---------|------|
| **Swap Partition** | Linux | 独立的磁盘分区，专门用于换页 |
| **Swap File** | Linux | 文件系统中的大文件，功能同 Swap Partition |
| **Page File** (`pagefile.sys`) | Windows | 磁盘上的隐藏文件，用于存放被换出的页 |

无论叫什么名字，它们的本质都是同一块磁盘空间：**物理内存的后备存储（Backing Store）**。

## Windows 分页文件与"虚拟内存不足"

这是开发者在 Windows 上最常遇到的与虚拟内存相关的实际问题。

### `pagefile.sys` 是什么？

Windows 通常在系统盘根目录下创建一个隐藏文件 `pagefile.sys`。当物理内存（RAM）紧张时，操作系统把不活跃的内存页写入这个文件；当这些页再次被访问时，再从文件读回内存。

> 注意：`pagefile.sys` 不是"假装内存来骗程序"——程序访问的始终是虚拟地址，由页表翻译到物理地址。`pagefile.sys` 只是给那些**暂时被踢出物理内存**的页提供一个临时的"寄存处"。

### "所有驱动器的总分页文件大小"

Windows 控制面板 → 系统 → 高级系统设置 → 性能 → 高级 → 虚拟内存，这里调整的就是 `pagefile.sys` 的大小。

```
物理内存（RAM）：16GB
分页文件（C:\pagefile.sys）：8GB
─────────────────────────────────
系统总可用"承诺内存" = 16GB + 8GB = 24GB
```

### 为什么编译大型项目会报"虚拟内存不足"？

当你在 Visual Studio 中编译大型 C++ 项目（如 Unreal Engine）时，报错"虚拟内存不足"通常**不是**地址空间用完了（64位程序有 128TB 的虚拟地址空间，根本用不完），而是达到了 **Commit Limit**。

Windows 的内存分配有两个动作：

| 动作 | 含义 |
|------|------|
| **Reserve** | 预留一段虚拟地址范围（先占坑，不分配实际资源） |
| **Commit** | 承诺这段地址有 backing store——要么在物理内存，要么在分页文件里 |

**Commit Limit = 物理内存（RAM）+ 分页文件大小**

所有进程的 Committed 内存总和不能超过这个上限。编译和链接大型项目时，编译器（`cl.exe`）和链接器（`link.exe`）会 Commit 巨量内存来存放 AST、符号表、模板展开结果。如果 RAM + 分页文件不够大，就会报"虚拟内存不足"。

> **解决方案**：手动增大分页文件大小。很多 UE 开发者会把分页文件设为 32GB 甚至更大，专门为了给链接器足够的 Commit 额度。

### 什么时候可以关闭分页文件？

如果你的物理内存非常大（如 64GB 或 128GB），且你的 workload 不会同时 Commit 超过物理内存的总量，理论上可以缩小甚至关闭分页文件。但通常不建议：

- 某些 Windows 内核功能和驱动程序依赖分页文件进行崩溃转储（蓝屏时写入 `MEMORY.DMP`）。
- 完全关闭分页文件会把 Commit Limit 锁死在物理内存大小，某些程序可能因此无法启动。
- 现代 Windows 的内存管理已经相当智能，自动管理分页文件通常是最优选择。

以下内容是地址翻译机制的深入细节，日常开发不需要全部掌握，但在排查底层 Bug 或做引擎性能调优时会用到。

### 页表项（PTE）的控制位

每个页表项不仅有物理地址，还有控制权限的标志位：

| 位 | 名称 | 作用 |
|----|------|------|
| **P（Present）** | 该页是否在物理内存 | 0 则触发 Page Fault |
| **R/W** | 读写权限 | 0=只读，用于保护代码段 |
| **U/S** | 用户/内核权限 | 0=仅内核可访问，实现内核空间隔离 |
| **A（Accessed）** | 访问后硬件自动置 1 | OS 用于判断页是否被使用（LRU 淘汰依据） |
| **D（Dirty）** | 写入后硬件自动置 1 | 表示该页被修改过，换出时需写回磁盘 |
| **PCD** | 禁止缓存该页 | 用于 MMIO（内存映射 IO）区域 |

> **设计权衡**：Accessed 和 Dirty 位由**硬件自动设置**，但由**软件清除**。这使得 OS 可以实现高效的页淘汰策略（如 Clock 算法），无需陷入内核来追踪页的访问状态。

### TLB Shootdown

当操作系统修改了页表（如换出页、修改权限、进程结束释放内存），已缓存在 TLB 中的旧翻译条目就失效了。操作系统必须：

1. 修改页表。
2. 向所有运行该进程的 CPU 核心发送**核间中断（IPI）**。
3. 各核心收到中断后，刷新本地 TLB 中对应条目。

TLB Shootdown 是操作系统中**代价极高的操作**，因为它需要暂停其他核心的执行。游戏引擎中大量使用线程和共享内存时，频繁的 `mprotect` / `munmap` 可能触发大量 Shootdown，成为性能瓶颈。

### 为什么是 4KB？页大小的设计权衡

x86_64 的默认页大小是 **4KB**，这是**内部碎片**与**页表开销**的权衡：

| 页大小 | 优点 | 缺点 |
|--------|------|------|
| **1KB** | 内部碎片极小 | 页表条目多，页表占用内存大，TLB 覆盖范围小 |
| **4KB** | 平衡了碎片和开销 | 通用场景的最优解 |
| **64KB** | 减少页表级数，TLB 效率高 | 小对象的内部碎片严重 |
| **2MB**（HugePage） | 极大减少 TLB Miss | 分配粒度粗，不适合频繁分配释放 |

> 4KB 恰好是传统磁盘块（512B）的整数倍（4KB = 8 个块），便于磁盘 IO 与内存页对齐。现代 SSD 的页大小也是 4KB 或 16KB，进一步巩固了 4KB 的行业标准。

## How：如何实践？

### 1. 预分配并触摸（Touch）内存

在加载阶段主动读写一遍关键内存，让 Page Fault 发生在非关键时刻。实验 1 会验证：第一次触摸比第二次慢数倍，差距正是 Major Fault 的磁盘 I/O 开销。

### 2. 使用 `mmap` 映射大文件

将资源文件映射到地址空间，由 OS 按需加载。相比 `read()`，mmap 零拷贝、支持随机访问、多进程共享物理页。详见上文 [#mmap（内存映射文件）](#mmap内存映射文件)。

### 3. 适时启用 HugePage

大内存工作集（如数据库缓存、大纹理流式加载）开启 HugePage 或 Transparent HugePage，可显著降低 TLB Miss。Windows 下使用 `VirtualAlloc` + `MEM_LARGE_PAGES`。

## C++ 本地验证实验

### 实验 1：观察 Page Fault 的开销

```cpp
// workspace/os-lab/memory/page_fault.cpp
#include <iostream>
#include <vector>
#include <chrono>

int main() {
    const size_t SIZE = 512 * 1024 * 1024; // 512 MB
    std::vector<char> buf(SIZE); // 分配虚拟地址空间（此时无物理页分配）

    // 第一次触摸：每页触发一次 Page Fault，OS 从磁盘/交换区加载或分配清零页
    auto t1 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 1; // 每页写一个字节
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    // 第二次触摸：页已在物理内存，无 Page Fault
    auto t3 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 2;
    }
    auto t4 = std::chrono::high_resolution_clock::now();

    std::cout << "First touch (with page faults): "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
    std::cout << "Second touch (no page faults): "
              << std::chrono::duration<double, std::milli>(t4 - t3).count()
              << " ms\n";
    return 0;
}
```

**预期结果**：第一次触摸比第二次慢数倍到数十倍。这是因为 `std::vector` 分配的是虚拟地址空间，第一次写入时每 4KB 触发一次 Minor Page Fault，由 OS 分配并清零物理页。

> **平台差异**：Linux 上 `std::vector<char>(SIZE)` 通常使用 `mmap` 分配大内存（glibc 的 malloc 阈值），因此首次触摸表现为 Minor Fault。Windows 的堆分配行为可能不同，建议使用 `VirtualAlloc` / `mmap` 做跨平台对比。

### 实验 2：`mmap` 大文件读取

```cpp
// workspace/os-lab/memory/mmap_file.cpp
// Linux: g++ mmap_file.cpp -o mmap_file
// Windows: 使用 CreateFileMapping / MapViewOfFile
#ifdef _WIN32
#include <windows.h>
#include <iostream>
#else
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>
#include <iostream>
#endif

int main(int argc, char* argv[]) {
    if (argc < 2) { std::cerr << "Usage: " << argv[0] << " <file>\n"; return 1; }

#ifdef _WIN32
    HANDLE hFile = CreateFileA(argv[1], GENERIC_READ, FILE_SHARE_READ, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL);
    HANDLE hMap = CreateFileMapping(hFile, NULL, PAGE_READONLY, 0, 0, NULL);
    char* data = (char*)MapViewOfFile(hMap, FILE_MAP_READ, 0, 0, 0);
    LARGE_INTEGER size; GetFileSizeEx(hFile, &size);
    std::cout << "Mapped " << size.QuadPart << " bytes via Windows MMF\n";
    // 像访问数组一样遍历：for (size_t i = 0; i < size.QuadPart; ++i) { ... }
    UnmapViewOfFile(data); CloseHandle(hMap); CloseHandle(hFile);
#else
    int fd = open(argv[1], O_RDONLY);
    struct stat st; fstat(fd, &st);
    char* data = (char*)mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    std::cout << "Mapped " << st.st_size << " bytes via mmap\n";
    // 像访问数组一样遍历：for (size_t i = 0; i < st.st_size; ++i) { ... }
    munmap(data, st.st_size); close(fd);
#endif
    return 0;
}
```

**扩展对比**：可以尝试用同一个文件，分别用 `mmap` 和传统的 `ifstream::read` / `fread` 循环读取，对比大文件随机访问的性能差异。`mmap` 的随机访问几乎没有 syscall 开销，而 `read` 每次都要系统调用 + 数据拷贝。

## 游戏引擎延伸

### 大世界流式加载（World Streaming）

- 开放世界游戏无法一次性加载全部资源，采用按需加载（Lazy Loading）。
- `mmap` 映射资源包后，玩家走进新区域时触发 Page Fault，OS 自动从磁盘加载对应页。
- 引擎可预加载（`mlock` / `VirtualLock`）即将进入的区域，避免运行时卡顿。

### GPU 显存映射（ReBAR / Smart Access Memory）

- 现代 GPU 支持将显存直接映射到 CPU 地址空间，CPU 可直接读写显存页。
- 底层依赖 PCIe 的地址翻译和 OS 的页表管理，本质是虚拟内存的延伸。
