# 虚拟内存与地址翻译

---

## Why：为什么需要虚拟内存？

- **进程隔离**：每个进程认为自己独占整个地址空间，互不干扰。
- **物理内存抽象**：程序无需关心物理内存布局，操作系统负责映射。
- **按需加载（Demand Paging）**：仅加载需要的页，其余留在磁盘，节省物理内存。
- **共享内存**：不同进程可映射同一物理页（如共享库、显存映射）。

---

## What：核心概念

### 页与页表

- **页（Page）**：虚拟内存的最小管理单位，通常为 **4KB**（x86_64）。
- **页帧（Page Frame）**：物理内存中对应的 4KB 块。
- **页表（Page Table）**：多级结构（x86_64 通常为 4 级），存储虚拟页到物理帧的映射。

### 地址翻译流程

```
虚拟地址 -> [CR3 + 页表遍历] -> 物理地址
```

- 每次内存访问可能需要 **4 次内存访问**（查 4 级页表）。
- **TLB（Translation Lookaside Buffer）**：CPU 内部的页表缓存，命中时一次访问即可。
- **TLB Miss**：比 Cache Miss 更昂贵，可能需要数百个周期。

### Page Fault

- 访问的页不在物理内存中（或未分配），触发缺页异常。
- 操作系统处理：分配物理页、从磁盘加载数据、更新页表、恢复执行。
- **Major Fault**：需要磁盘 IO，代价极高（毫秒级）。
- **Minor Fault**：仅需分配清零页，代价较低（微秒级）。

### HugePage（大页）

- 使用 **2MB** 或 **1GB** 大页，减少页表级数和 TLB Miss。
- 游戏服务器、数据库常开启 HugePage；游戏引擎的大世界流式加载也受益于大页映射。

---

## How：如何实践？

1. **预分配并触摸（Touch）内存**：在加载阶段主动读写一遍，避免运行时 Page Fault。
2. **使用 `mmap` 映射大文件**：将资源文件映射到地址空间，由 OS 按需加载。
3. **Linux 下启用 Transparent HugePage**；Windows 下使用 `VirtualAlloc` + `MEM_LARGE_PAGES`。

## C++ 本地验证实验

### 实验 1：观察 Page Fault

```cpp
// workspace/os-lab/memory/page_fault.cpp
#include <iostream>
#include <vector>
#include <chrono>

int main() {
    const size_t SIZE = 512 * 1024 * 1024; // 512 MB
    std::vector<char> buf(SIZE); // 分配虚拟地址空间

    // 第一次触摸：触发大量 Page Fault
    auto t1 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 1; // 每页写一个字节
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    // 第二次触摸：已在内存，无 Page Fault
    auto t3 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 2;
    }
    auto t4 = std::chrono::high_resolution_clock::now();

    std::cout << "First touch (with page faults): "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
    std::cout << "Second touch (no page faults): "
              << std::chrono::duration<double, std::milli>(t4 - t3).count()
              << " ms\n";
    return 0;
}
```

**预期结果**：第一次触摸比第二次慢数倍到数十倍。

### 实验 2：`mmap` 大文件读取

```cpp
// workspace/os-lab/memory/mmap_file.cpp
// Linux: g++ mmap_file.cpp -o mmap_file
// Windows: 使用 CreateFileMapping / MapViewOfFile
#ifdef _WIN32
#include <windows.h>
#include <iostream>
#else
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>
#include <iostream>
#endif

int main(int argc, char* argv[]) {
    if (argc < 2) { std::cerr << "Usage: " << argv[0] << " <file>\n"; return 1; }

#ifdef _WIN32
    HANDLE hFile = CreateFileA(argv[1], GENERIC_READ, FILE_SHARE_READ, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL);
    HANDLE hMap = CreateFileMapping(hFile, NULL, PAGE_READONLY, 0, 0, NULL);
    char* data = (char*)MapViewOfFile(hMap, FILE_MAP_READ, 0, 0, 0);
    LARGE_INTEGER size; GetFileSizeEx(hFile, &size);
    std::cout << "Mapped " << size.QuadPart << " bytes via Windows MMF\n";
    // 遍历访问...
    UnmapViewOfFile(data); CloseHandle(hMap); CloseHandle(hFile);
#else
    int fd = open(argv[1], O_RDONLY);
    struct stat st; fstat(fd, &st);
    char* data = (char*)mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    std::cout << "Mapped " << st.st_size << " bytes via mmap\n";
    // 遍历访问...
    munmap(data, st.st_size); close(fd);
#endif
    return 0;
}
```

## 游戏引擎延伸

### 大世界流式加载（World Streaming）

- 开放世界游戏无法一次性加载全部资源，采用按需加载（Lazy Loading）。
- `mmap` 映射资源包后，玩家走进新区域时触发 Page Fault，OS 自动从磁盘加载对应页。
- 引擎可预加载（`mlock` / `VirtualLock`）即将进入的区域，避免运行时卡顿。

### GPU 显存映射（ReBAR / Smart Access Memory）

- 现代 GPU 支持将显存直接映射到 CPU 地址空间，CPU 可直接读写显存页。
- 底层依赖 PCIe 的地址翻译和 OS 的页表管理，本质是虚拟内存的延伸。
