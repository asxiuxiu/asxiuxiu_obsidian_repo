# 虚拟内存与地址翻译

---

## Why：为什么需要虚拟内存？

- **进程隔离**：每个进程认为自己独占整个地址空间，互不干扰。
- **物理内存抽象**：程序无需关心物理内存布局，操作系统负责映射。
- **按需加载（Demand Paging）**：仅加载需要的页，其余留在磁盘，节省物理内存。
- **共享内存**：不同进程可映射同一物理页（如共享库、显存映射）。

> **历史演进**：在虚拟内存出现之前，程序直接操作**物理地址**。这带来三个致命问题：
> 1. **内存碎片化**：程序 A 占用 0x1000~0x2000，程序 B 占用 0x3000~0x5000，中间 0x2000~0x3000 的碎片无法被大于它的程序利用。
> 2. **缺乏隔离**：程序 A 的指针 bug 可能直接覆盖程序 B 的内存。
> 3. **重定位困难**：程序编译时假设自己从地址 0 开始运行，但加载时该地址可能已被占用，需要逐个修改地址引用（重定位）。
>
> 早期解决方案是**分段（Segmentation）**——为每个进程分配几个独立的段（代码段、数据段、栈段），段内地址从 0 开始，通过段基址寄存器偏移到物理地址。但分段仍面临外部碎片问题（段大小不固定，内存中留下大量无法利用的小洞），且段的大小受限于段寄存器的位数。
>
> **分页（Paging）** 的引入彻底解决了这些问题：将内存切成固定大小的页（Page），进程的虚拟页可以映射到任意物理页帧，无外部碎片，隔离性完美，重定位简化为修改页表即可。

## What：核心概念

### 页与页表

- **页（Page）**：虚拟内存的最小管理单位，通常为 **4KB**（x86_64）。
- **页帧（Page Frame）**：物理内存中对应的 4KB 块。
- **页表（Page Table）**：多级结构（x86_64 通常为 4 级），存储虚拟页到物理帧的映射。

### 地址翻译流程

```
虚拟地址 -> [CR3 + 页表遍历] -> 物理地址
```

- 每次内存访问可能需要 **4 次内存访问**（查 4 级页表）。
- **TLB（Translation Lookaside Buffer）**：CPU 内部的页表缓存，命中时一次访问即可。
- **TLB Miss**：比 Cache Miss 更昂贵，可能需要数百个周期。

> **CR3** 是 x86_64 架构的控制寄存器（Control Register 3），存放当前进程**顶级页表（PML4）的物理基地址**。进程切换时，操作系统修改 CR3，CPU 就自动使用新进程的页表进行地址翻译。每个 CPU 核心有自己的 CR3 寄存器（在多核系统中，不同核心可以运行不同进程）。

#### 页表项（PTE）的位域详解

页表不仅存储"虚拟页→物理帧"的映射，每个页表项（Page Table Entry, PTE）还包含丰富的控制位（以 x86_64 为例）：

| 位 | 名称 | 含义 |
|----|------|------|
| 0 | **P（Present）** | 该页是否在物理内存中。0 表示未分配或在磁盘上，访问触发 Page Fault。 |
| 1 | **R/W（Read/Write）** | 0=只读，1=可读写。用于实现代码段只读保护。 |
| 2 | **U/S（User/Supervisor）** | 0=仅内核可访问，1=用户态可访问。实现内核空间隔离。 |
| 3 | **PWT（Page Write Through）** | 该页的缓存写策略（通常忽略）。 |
| 4 | **PCD（Page Cache Disable）** | 1=禁止缓存该页。用于 MMIO（内存映射 IO）区域。 |
| 5 | **A（Accessed）** | CPU 访问该页后自动置 1。操作系统用于判断页是否被使用（LRU 淘汰依据）。 |
| 6 | **D（Dirty）** | CPU 写入该页后自动置 1。表示该页被修改过，换出时需写回磁盘。 |
| 7 | **PS（Page Size）** | 在大页（HugePage）层级使用，表示该级直接映射而非继续遍历。 |
| 12~51 | **物理地址** | 页帧的物理基地址（高 12 位用于标志位）。 |

> **设计权衡**：Accessed 和 Dirty 位由**硬件自动设置**，但由**软件清除**。这使得操作系统可以实现高效的页淘汰策略（如 Clock 算法），无需陷入内核来追踪页的访问状态。

#### TLB 的结构与 TLB Shootdown

TLB 是 CPU 内部专门缓存地址翻译结果的高速缓存。由于地址翻译是**内存访问的必经路径**，TLB 的速度直接影响每一次内存访问：

| 特性 | 说明 |
|------|------|
| **容量** | 通常 32~64 项（L1 TLB），1024~2048 项（L2 TLB） |
| **关联度** | 通常全相联（Fully Associative），因为虚拟地址到物理地址的映射完全随机 |
| **组织** | 分为指令 TLB（ITLB）和数据 TLB（DTLB），与 L1 I-Cache / D-Cache 分离的理念一致 |

**TLB Shootdown（TLB 击落）**：
当操作系统修改了页表（如换出页、修改权限、进程结束释放内存），已经缓存在 TLB 中的旧翻译条目就失效了。操作系统必须：
1. 修改页表。
2. 向所有运行该进程的 CPU 核心发送**核间中断（IPI, Inter-Processor Interrupt）**。
3. 各核心收到中断后，执行 `INVLPG` 指令刷新本地 TLB 中对应条目（或刷新整个 TLB）。

> TLB Shootdown 是操作系统中**代价极高的操作**，因为它需要暂停其他核心的执行。游戏引擎中大量使用线程和共享内存时，频繁的 `mprotect` / `munmap` 可能触发大量 Shootdown，成为性能瓶颈。

### Page Fault

- 访问的页不在物理内存中（或未分配），触发缺页异常。
- 操作系统处理：分配物理页、从磁盘加载数据、更新页表、恢复执行。
- **Major Fault**：需要磁盘 IO，代价极高（毫秒级）。
- **Minor Fault**：仅需分配清零页，代价较低（微秒级）。

### 为什么是 4KB？页大小的设计权衡

x86_64 的默认页大小是 **4KB**。这不是随意选择的，而是**内部碎片**与**页表开销**的权衡：

| 页大小 | 优点 | 缺点 |
|--------|------|------|
| **1KB** | 内部碎片极小 | 页表条目多，页表占用内存大，TLB 覆盖范围小 |
| **4KB** | 平衡了碎片和开销 | 通用场景的最优解 |
| **64KB** | 减少页表级数，TLB 效率高 | 小对象（如 1KB 结构体）的内部碎片严重 |
| **2MB**（HugePage） | 极大减少 TLB Miss，适合大内存工作集 | 分配粒度粗，不适合频繁分配释放 |

> 4KB 的选择还有一个历史原因：它恰好是磁盘块（Sector）大小的整数倍（传统磁盘块 512B，4KB = 8 个块），便于磁盘 IO 与内存页的对齐。现代 SSD 的页大小也是 4KB 或 16KB，进一步巩固了 4KB 的行业标准地位。

### HugePage（大页）

- 使用 **2MB** 或 **1GB** 大页，减少页表级数和 TLB Miss。
- 游戏服务器、数据库常开启 HugePage；游戏引擎的大世界流式加载也受益于大页映射。

## How：如何实践？

1. **预分配并触摸（Touch）内存**：在加载阶段主动读写一遍，避免运行时 Page Fault。
2. **使用 `mmap` 映射大文件**：将资源文件映射到地址空间，由 OS 按需加载。
3. **Linux 下启用 Transparent HugePage**；Windows 下使用 `VirtualAlloc` + `MEM_LARGE_PAGES`。

## C++ 本地验证实验

### 实验 1：观察 Page Fault

```cpp
// workspace/os-lab/memory/page_fault.cpp
#include <iostream>
#include <vector>
#include <chrono>

int main() {
    const size_t SIZE = 512 * 1024 * 1024; // 512 MB
    std::vector<char> buf(SIZE); // 分配虚拟地址空间

    // 第一次触摸：触发大量 Page Fault
    auto t1 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 1; // 每页写一个字节
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    // 第二次触摸：已在内存，无 Page Fault
    auto t3 = std::chrono::high_resolution_clock::now();
    for (size_t i = 0; i < SIZE; i += 4096) {
        buf[i] = 2;
    }
    auto t4 = std::chrono::high_resolution_clock::now();

    std::cout << "First touch (with page faults): "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
    std::cout << "Second touch (no page faults): "
              << std::chrono::duration<double, std::milli>(t4 - t3).count()
              << " ms\n";
    return 0;
}
```

**预期结果**：第一次触摸比第二次慢数倍到数十倍。

### 实验 2：`mmap` 大文件读取

```cpp
// workspace/os-lab/memory/mmap_file.cpp
// Linux: g++ mmap_file.cpp -o mmap_file
// Windows: 使用 CreateFileMapping / MapViewOfFile
#ifdef _WIN32
#include <windows.h>
#include <iostream>
#else
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>
#include <iostream>
#endif

int main(int argc, char* argv[]) {
    if (argc < 2) { std::cerr << "Usage: " << argv[0] << " <file>\n"; return 1; }

#ifdef _WIN32
    HANDLE hFile = CreateFileA(argv[1], GENERIC_READ, FILE_SHARE_READ, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL);
    HANDLE hMap = CreateFileMapping(hFile, NULL, PAGE_READONLY, 0, 0, NULL);
    char* data = (char*)MapViewOfFile(hMap, FILE_MAP_READ, 0, 0, 0);
    LARGE_INTEGER size; GetFileSizeEx(hFile, &size);
    std::cout << "Mapped " << size.QuadPart << " bytes via Windows MMF\n";
    // 遍历访问...
    UnmapViewOfFile(data); CloseHandle(hMap); CloseHandle(hFile);
#else
    int fd = open(argv[1], O_RDONLY);
    struct stat st; fstat(fd, &st);
    char* data = (char*)mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    std::cout << "Mapped " << st.st_size << " bytes via mmap\n";
    // 遍历访问...
    munmap(data, st.st_size); close(fd);
#endif
    return 0;
}
```

## 游戏引擎延伸

### 大世界流式加载（World Streaming）

- 开放世界游戏无法一次性加载全部资源，采用按需加载（Lazy Loading）。
- `mmap` 映射资源包后，玩家走进新区域时触发 Page Fault，OS 自动从磁盘加载对应页。
- 引擎可预加载（`mlock` / `VirtualLock`）即将进入的区域，避免运行时卡顿。

### GPU 显存映射（ReBAR / Smart Access Memory）

- 现代 GPU 支持将显存直接映射到 CPU 地址空间，CPU 可直接读写显存页。
- 底层依赖 PCIe 的地址翻译和 OS 的页表管理，本质是虚拟内存的延伸。
