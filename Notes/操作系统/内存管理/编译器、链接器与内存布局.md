# 编译器、链接器与内存布局

---

## Why：为什么要理解编译链接与内存布局？

- **调试底层 Bug 的必备知识**：段错误（Segfault）为什么发生？栈溢出和堆溢出有什么区别？`double free` 为什么崩溃？这些问题的答案都在内存布局中。
- **理解编译器优化的边界**：编译器在什么情况下会重排你的代码？`volatile` 到底防什么、不防什么？理解编译流程才能写出"编译器友好"的高性能代码。
- **引擎构建系统的基础**：游戏引擎大量使用静态库/动态库拆分、插件化架构（如 UE 的模块系统）。不理解链接过程，就无法理解 `dllimport`/`dllexport`、`visibility=hidden` 等关键配置。

---

## What：从源码到可执行文件的旅程

### 编译的四个阶段

将 `.cpp` 变成可执行文件，不是"一步编译"，而是四个明确的阶段：

| 阶段 | 工具 | 输入 | 输出 | 核心任务 |
|------|------|------|------|----------|
| 预处理 | cpp / 编译器前端 | `.cpp` | `.i` | 展开 `#include`、宏替换、条件编译 |
| 编译 | cc1 / clang | `.i` | `.s` | 词法/语法/语义分析，生成汇编代码，进行优化 |
| 汇编 | as | `.s` | `.o` | 将汇编指令翻译成机器码，生成目标文件 |
| 链接 | ld / link.exe | `.o` + 库 | 可执行文件 | 符号解析、重定位、合并段 |

> **关键洞察**：`g++ -c a.cpp` 只走到汇编阶段，生成 `.o`；`g++ a.o b.o -o out` 只做链接。大型项目中分离编译（Separate Compilation）正是利用这一特性——修改一个 `.cpp` 只需重新编译该文件，再重新链接即可。

### 目标文件的段结构

每个 `.o` 文件和最终的可执行文件都按**段（Section/Segment）**组织：

| 段名 | 内容 | 权限 | 生命周期 |
|------|------|------|----------|
| `.text` | 机器指令（代码） | 读、执行 | 程序启动即存在 |
| `.data` | 已初始化的全局/静态变量 | 读、写 | 程序启动即存在 |
| `.bss` | **未初始化**的全局/静态变量 | 读、写 | 程序启动时由 OS 清零分配 |
| `.rodata` | 只读数据（字符串常量、const 全局变量） | 只读 | 程序启动即存在 |

> 为什么 `.bss` 单独存在？因为未初始化数据无需在可执行文件中占空间（反正全是 0），链接器只需记录"启动时分配 X 字节并清零"。这显著减小了可执行文件体积。

### 链接器如何合并多个 .o 文件

大型项目中，一个可执行文件通常由数十甚至数百个 `.o` 文件链接而成。链接器的核心任务之一就是将它们**同类段合并**：

```
a.o: [.text] [.data] [.bss] [.rodata]
b.o: [.text] [.data] [.bss] [.rodata]
c.o: [.text] [.data] [.bss] [.rodata]
         ↓ 链接器合并 ↓
可执行文件: [.text] [.data] [.bss] [.rodata] [...]
```

| 合并操作 | 说明 |
|----------|------|
| **代码段合并** | 所有 `.o` 的 `.text` 合并为可执行文件的一个 `.text` |
| **数据段合并** | 所有 `.o` 的 `.data` 合并为一个 `.data`，全局符号解析到唯一地址 |
| **BSS 段合并** | 所有 `.o` 的 `.bss` 大小相加，运行时 OS 一次性分配并清零 |
| **符号解析** | 将 `a.o` 中调用的外部函数绑定到 `b.o` 中的实际定义 |
| **重定位** | 将 `.o` 中的相对地址修正为最终虚拟地址空间中的绝对地址 |

> **关键结论**：无论有多少个 `.o` 文件，链接后的可执行文件只有**一套段结构**，加载到内存后也只有一个统一的虚拟地址空间。

> **补充：可执行文件的大小 ≠ 运行时内存占用**。大型程序的可执行文件可能达到数百 MB 甚至数 GB，但操作系统采用**按需分页（Demand Paging）**机制，不会一次性将整个文件载入物理内存。只有在 CPU 实际访问某页时，才会触发缺页中断将其从磁盘调入。配合动态链接（运行时加载 `.so`/`.dll`）和 BSS 段在文件中不占实际空间的特性，程序启动速度和运行时内存占用都不会随文件大小线性增长。

> **注意：下图是 Linux 的典型布局，不同操作系统布局不同**。Windows 上代码段/数据段通常被加载到较高地址，而栈的位置也与 Linux 相反，详见下文实验验证。

#### 什么是"页"？

**页（Page）是内存管理的最小单位，就像一个标准大小的箱子。**

操作系统把物理内存和虚拟地址空间都切割成固定大小的小块，每块就是一个页。在 x86_64 Linux 上，这个大小通常是 **4KB**。

| 概念 | 类比 | 大小 |
|------|------|------|
| 虚拟页 | 虚拟地址空间里的箱子 | 4KB |
| 物理页 | 物理内存里的箱子 | 4KB |
| 页表 | 快递单号查询表：虚拟页号 → 物理页号 | — |

**为什么用"页"而不是按字节管理？**

如果操作系统记录"虚拟地址 0x1000 映射到物理地址 0x8000"，粒度太细、开销太大。分页后，OS 只需记录"虚拟页 1 映射到物理页 5"，管理效率大幅提升。

#### 按需分页（Demand Paging）如何工作？

想象你去图书馆借书：

- **没有按需分页**：你一进馆，管理员就把所有书搬到你的座位上，不管你看不看得完。
- **按需分页**：你办卡了（虚拟地址空间建好了），书还在书架上。你要看哪本，管理员才去拿哪本。

具体流程：

```
程序启动时：
┌─────────────┐         ┌─────────────┐
│ 虚拟地址空间 │  ←─X──  │  物理内存    │
│  (页表)     │  未映射  │   (空的)     │
│ 全部标记为   │         │              │
│ "不在内存"  │         │              │
└─────────────┘         └─────────────┘

CPU 第一次访问某地址：
  ↓
发现页表标记为"不存在"→ 触发缺页中断（Page Fault）
  ↓
OS 从磁盘找到对应数据，装入一个空闲物理页
  ↓
更新页表：虚拟页 → 物理页
  ↓
CPU 重新执行该指令（这次正常访问）
```

**实际例子**：假设程序有 1GB 代码，但启动后只执行了 `main()`：

| 阶段 | 发生了什么 |
|------|-----------|
| 启动 | OS 建立页表，标记所有页为"未加载"。**没有任何代码被读入物理内存。** |
| 执行 `main()` | `main()` 所在页不在内存 → 缺页中断 → OS 加载该页（4KB）→ 继续执行 |
| 调用 `funcA()` | `funcA()` 在另一页 → 缺页中断 → 加载该页 |
| `funcB()` 从未调用 | 它所在的页**永远不被加载**，节省内存 |

> **关键洞察**：缺页中断对程序完全透明。CPU 自动暂停、切换内核态、加载数据、恢复执行，程序员感知不到这个过程。唯一的"代价"是首次访问时的磁盘 I/O 延迟（毫秒级）。

#### 段、页与物理内存的关系

一个容易混淆的问题：**虚拟地址空间中的 Text/Data/BSS/Heap/Stack 等段，在物理内存中是什么样子？**

答案是：**每个段由一连串的页组成，但这些页不一定都在物理内存里。**

```
虚拟地址空间（程序视角）          物理内存（OS 管理）
┌─────────────────┐
│      栈区        │ ← 只占用了 2 个页 ─────→ [物理页 3] [物理页 7]
├─────────────────┤                （其他栈页还在磁盘上，未加载）
│      ...        │
├─────────────────┤
│      堆区        │ ← 只占用了 1 个页 ─────→ [物理页 12]
├─────────────────┤
│      BSS        │ ← 只占用了 1 个页 ─────→ [物理页 5]
├─────────────────┤
│      Data       │ ← 只占用了 1 个页 ─────→ [物理页 9]
├─────────────────┤
│      Text       │ ← 只占用了 2 个页 ─────→ [物理页 1] [物理页 8]
│  (代码段)       │       （main 和当前函数所在页）
└─────────────────┘              （其他代码页还在磁盘上）
```

**两个关键区分：**

| 变化维度 | 实际含义 | 是否动态变化 |
|---------|---------|-------------|
| **哪些页在物理内存中** | 页的"存在性" | ✅ 持续变化 |
| **页内的数据内容** | 加载后的实际数据 | ❌ 代码页不变（只读），数据页按程序逻辑变化 |

**页的换出（Swap）**：当物理内存紧张时，OS 会把不活跃的页踢回磁盘（交换区），页表标记为"不存在"。下次访问时再触发缺页中断换回来。这解释了为什么 16GB 物理内存可以同时跑几十个程序——OS 只保留最活跃的页在内存里。

```
高地址
┌─────────────────┐ ← 0x7FFF...（用户态栈顶）
│      栈区        │  ↓ 向下增长（函数调用、局部变量）
│   (Stack)       │
├─────────────────┤
│      ...        │  内存映射区（mmap、动态库、共享内存）
├─────────────────┤
│      堆区        │  ↑ 向上增长（malloc/new）
│   (Heap)        │
├─────────────────┤
│      BSS        │  未初始化全局/静态变量
├─────────────────┤
│      Data       │  已初始化全局/静态变量
├─────────────────┤
│      Text       │  代码段（只读执行）
├─────────────────┤
│  保留/内核空间   │  ← 0x0（低地址）
└─────────────────┘
```

#### 栈（Stack）

- **管理方式**：由编译器自动分配和释放，无需程序员干预。
- **存储内容**：函数参数、返回地址、局部变量、寄存器保存值。
- **增长方向**：向低地址增长（压栈时地址减小）。
- **容量限制**：通常只有几 MB（Linux 默认 8MB）。**栈溢出（Stack Overflow）** 就是递归过深或局部数组过大，超出了栈的边界。

#### 堆（Heap）

- **管理方式**：由程序员通过 `malloc`/`new` 申请，`free`/`delete` 释放。操作系统通过**堆分配器**管理（如 ptmalloc、jemalloc）。
- **存储内容**：生命周期超越单个函数的对象、动态大小的数组、引擎中的资源数据。
- **增长方向**：向高地址增长。
- **碎片问题**：频繁的分配和释放不同大小的内存块，会导致**外部碎片**（空闲内存被分割成小块，无法分配大对象）。

> **栈 vs 堆的核心区别**：栈分配只需移动栈指针（一条指令），而堆分配需要遍历空闲块、可能触发系统调用、还要处理多线程竞争。因此高频小对象分配应优先考虑栈或内存池。

### 编译器优化与指令重排序

编译器在"编译"阶段会进行大量优化，常见的有：

- **常量折叠**：`int a = 2 * 3;` 直接变成 `int a = 6;`
- **死代码消除**：永远不会执行的代码被删除。
- **循环展开**：将循环体复制多次，减少跳转指令开销。
- **指令重排序（Instruction Reordering）**：在不改变**单线程**可见行为的前提下，调整指令顺序以提高流水线利用率。

> **关键陷阱**：编译器的重排序只保证**单线程**语义正确。多线程程序中，编译器不知道变量会被其他线程访问，可能将 `a = 1; flag = true;` 重排为 `flag = true; a = 1;`，导致消费者线程看到 `flag == true` 时 `a` 还未写入。这就是 `std::atomic` 和内存屏障存在的意义——告诉编译器"这里不要重排"。

### 静态链接 vs 动态链接

| 特性 | 静态链接（Static Linking） | 动态链接（Dynamic Linking） |
|------|---------------------------|---------------------------|
| 库文件 | `.a`（Linux）/ `.lib`（Windows） | `.so`（Linux）/ `.dll`（Windows） |
| 链接时机 | 编译时，库代码被复制进可执行文件 | 运行时/启动时加载 |
| 文件大小 | 大（包含库代码） | 小（只含符号引用） |
| 运行时依赖 | 无（自包含） | 需目标系统存在对应 `.so`/`.dll` |
| 更新库 | 需重新编译链接 | 替换 `.so`/`.dll` 即可 |
| 内存共享 | 每个进程独立一份库代码 | 多个进程共享同一份物理内存中的库代码 |

> 游戏引擎通常采用**混合策略**：核心框架静态链接保证稳定性，插件和第三方库（如 PhysX、FMOD）动态链接便于更新和热插拔。

## How：如何观察和验证？

### 实验 1：观察进程的内存段分布

使用 `size` 命令查看可执行文件各段大小：

```bash
# Linux
g++ -o test_layout test_layout.cpp
size test_layout
# 输出：text    data     bss     dec     hex filename
#      1895     616       8    2519     9d7 test_layout
```

使用 `nm` 查看符号（变量和函数）属于哪个段：

```bash
nm test_layout | grep " [TDB] "
# T = .text (代码), D = .data (已初始化数据), B = .bss (未初始化数据)
```

C++ 测试代码：

```cpp
// workspace/os-lab/memory/layout_demo.cpp
#include <iostream>
#include <vector>
#include <algorithm>
#include <iomanip>

int global_init = 42;             // .data
int global_uninit;                // .bss
const char* msg = "Hello";        // .data (指针), "Hello" 在 .rodata

void func() {                     // .text
    static int local_static = 1;  // .data
    int local_stack = 2;          // 栈
    int* local_heap = new int(3); // 堆

    // 收集所有变量信息：(地址, 名称, 所属段)
    std::vector<std::pair<uintptr_t, std::string>> regions;

    regions.push_back({reinterpret_cast<uintptr_t>(&global_init),    "global_init    (.data)"});
    regions.push_back({reinterpret_cast<uintptr_t>(&global_uninit),  "global_uninit  (.bss)"});
    regions.push_back({reinterpret_cast<uintptr_t>(&local_static),  "local_static   (.data)"});
    regions.push_back({reinterpret_cast<uintptr_t>(&local_stack),   "local_stack    (栈)"});
    regions.push_back({reinterpret_cast<uintptr_t>(local_heap),     "local_heap     (堆)"});
    regions.push_back({reinterpret_cast<uintptr_t>(msg),            "msg(指针值)    (.data)"});
    regions.push_back({reinterpret_cast<uintptr_t>(func),           "func(代码地址) (.text)"});

    // 按地址从高到低排序
    std::sort(regions.begin(), regions.end(),
              [](const auto& a, const auto& b) { return a.first > b.first; });

    std::cout << "===== 内存地址排序（高地址 → 低地址） =====\n";
    std::cout << std::hex << std::uppercase;

    for (const auto& [addr, name] : regions) {
        std::cout << "0x" << std::setw(16) << std::setfill('0') << addr
                  << "  " << name << "\n";
    }

    std::cout << std::dec; // 恢复十进制
    delete local_heap;
}

int main() { func(); return 0; }
```

**预期结果**：观察地址分布，验证栈地址在高地址区、堆地址在低地址区（相对栈而言）、全局变量在更底部的数据段。

> **⚠️ 平台差异**：上述预期是 **Linux** 上的典型结果。在 **Windows** 上，由于地址空间布局策略不同（代码/数据段加载到较高地址，栈和堆的位置也与 Linux 相反），你可能看到完全不同的地址分布。此外，现代操作系统开启 **ASLR（地址空间布局随机化）**，每次运行同一程序的地址都会变化，但**相对高低关系在同一系统上保持一致**。

### 实验 2：观察编译器优化带来的重排序

```cpp
// workspace/os-lab/memory/compiler_reorder.cpp
#include <iostream>

int a = 0, b = 0;

void foo() {
    a = 1;
    b = 2;
}

int main() {
    foo();
    std::cout << a << " " << b << "\n";
    return 0;
}
```

编译并查看汇编：

```bash
g++ -O2 -S compiler_reorder.cpp -o compiler_reorder.s
cat compiler_reorder.s | grep -A 20 "foo():"
```

**预期结果**：在 `-O2` 优化下，编译器可能将 `a=1` 和 `b=2` 合并为一条指令，或调整顺序。若将 `a` 和 `b` 声明为 `volatile`，则编译器会强制按源码顺序生成两条独立的内存写入指令。

## 游戏引擎延伸

### 为什么引擎需要自定义内存分配器？

通用堆分配器（ptmalloc）是为"通用程序"设计的，它不知道游戏引擎的特殊需求：
- **确定性**：游戏一帧 16.6ms，不能容忍第 137 帧突然因为堆分配触发系统调用而卡顿。
- **数据局部性**：通用分配器不保证连续分配的内存在物理上相邻，破坏 Cache 友好性。
- **碎片控制**：游戏运行时加载/卸载大量资源（纹理、模型、音效），通用堆很快碎片化。

因此引擎通常实现多层分配策略：
- **栈分配**：临时向量、矩阵计算。
- **Linear/Arena 分配器**：一帧内临时对象，帧末统一释放。
- **Pool 分配器**：相同大小的对象（如粒子、ECS Component）。
- **Buddy/Slab 分配器**：大块资源内存对齐分配。

### 动态链接与引擎插件系统

Unreal Engine 的模块系统基于动态链接。每个模块编译为 `.dll`/`.so`，通过接口类（如 `IModuleInterface`）在运行时加载。这使得：
- 编辑器可热重载 C++ 代码（重新编译模块 → 卸载旧 DLL → 加载新 DLL）。
- 第三方中间件（如 SpeedTree、Simplygon）可作为插件独立更新。
