# 内存分配器原理

---

## Why：为什么要深入 malloc？

- `malloc` / `free` 是通用分配器，设计目标是平均性能最优，而非最坏情况可控。
- **游戏引擎的痛点**：
  - **碎片化**：长时间运行后，堆变得支离破碎，大对象分配失败。
  - **锁竞争**：多线程高频 `new` 会导致分配器内部的锁竞争。
  - **不确定延迟**：通用分配器可能触发 `brk`/`mmap` 系统调用或内存整理，造成帧时间抖动。
- 理解底层后，才能在引擎中实现定制化的分配策略。

---

## What：核心概念

### 堆内存的来源

| 方式 | 机制 | 适用场景 |
|------|------|----------|
| `brk` | 移动数据段末尾指针，扩展/收缩连续堆空间 | 小内存（< 128KB） |
| `mmap` | 匿名映射独立内存区域 | 大内存（>= 128KB） |

### 分配器算法

- **伙伴系统（Buddy System）**：将内存按 2 的幂次分块，合并时只与"伙伴"合并，简单高效但内部碎片较多。
- **Slab Allocator**：按固定大小分类（size class），每类维护一个空闲链表，适合同尺寸高频分配（如内核对象）。
- **ptmalloc（glibc）**：
  - 主分配区（arena）用 `brk`，非主分配区用 `mmap`。
  - 多线程时每个线程有线程缓存（tcache），减少锁竞争。
  - 复杂但通用，最坏情况性能不可预测。
- **tcmalloc（Google）**：
  - 线程本地缓存（Thread Cache）→ 中央 Free List → 向 OS 申请。
  - 小对象分配几乎无锁，适合高并发服务器。
- **jemalloc（Facebook）**：
  - 按 size class 分 arena，碎片控制优秀，广泛用于游戏客户端（如《原神》）。

---

## How：如何实践？

1. **观察分配行为**：用 `strace -e brk,mmap,munmap` 跟踪程序的内存申请。
2. **对比分配器**：链接 `tcmalloc` 或 `jemalloc`，对比多线程分配性能。
3. **重载 `new/delete`**：统计分配次数、大小、热点，定位高频分配点。

## C++ 本地验证实验

### 实验 1：高频分配性能对比

```cpp
// workspace/os-lab/memory/alloc_bench.cpp
#include <vector>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int ITER = 100'000;
constexpr int NUM_THREADS = 4;

void bench(const char* name, auto alloc_fn, auto free_fn) {
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int t = 0; t < NUM_THREADS; ++t) {
        threads.emplace_back([&]() {
            for (int i = 0; i < ITER; ++i) {
                void* p = alloc_fn(256);
                free_fn(p);
            }
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    bench("malloc/free", [](size_t n){ return malloc(n); }, [](void* p){ free(p); });
    // bench("new/delete", [](size_t n){ return operator new(n); }, [](void* p){ operator delete(p); });
    return 0;
}
```

**进阶**：链接 `-ltcmalloc` 或 `-ljemalloc` 重新编译，对比耗时。

### 实验 2：重载 new/delete 统计分配

```cpp
// workspace/os-lab/memory/hook_new_delete.cpp
#include <new>
#include <cstdlib>
#include <atomic>
#include <iostream>

static std::atomic<size_t> g_alloc_count{0};
static std::atomic<size_t> g_alloc_bytes{0};

void* operator new(size_t size) {
    g_alloc_count.fetch_add(1, std::memory_order_relaxed);
    g_alloc_bytes.fetch_add(size, std::memory_order_relaxed);
    return std::malloc(size);
}
void operator delete(void* p) noexcept {
    std::free(p);
}

// 同样重载 new[] / delete[] ...

int main() {
    for (int i = 0; i < 1000; ++i) {
        auto p = new char[128];
        delete[] p;
    }
    std::cout << "Allocs: " << g_alloc_count.load() << "\n";
    std::cout << "Bytes : " << g_alloc_bytes.load() << "\n";
    return 0;
}
```

## 游戏引擎延伸

### UE 的 FMemory

- UE 不直接使用 `malloc`，而是通过 `FMemory::Malloc` / `Free` / `Realloc` 抽象。
- 可全局替换为自定义分配器，或按模块使用不同策略。
- `FMallocBinned`：按 size class 分桶，类似 slab，适合小对象。
- `FMallocMimalloc`：可接入微软的 mimalloc，进一步提升性能。

### Chaos 引擎的 TMemoryPool

- 物理模拟模块需要大量同构对象（如刚体、碰撞体）。
- 使用 `TMemoryPool` 预分配一批对象，O(1) 取出/归还，无碎片，无锁（ per-pool ）。
- 对象按 Cache Line 对齐，保证 SIMD 运算效率。
