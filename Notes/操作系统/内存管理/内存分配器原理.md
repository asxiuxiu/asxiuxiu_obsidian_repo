# 内存分配器原理

---

## Why：为什么要深入 malloc？

上一篇[[虚拟内存与地址翻译]]中，我们理解了操作系统如何通过页表把虚拟地址映射到物理地址。程序启动时，操作系统为其建立了一个完整的虚拟地址空间，其中包含一个**堆区（Heap）**——这是程序运行时动态申请内存的主战场。

但操作系统并不会直接帮你管理这块区域的"谁用了哪一块、哪块是空闲的"。当你调用 `malloc(256)` 时，真正发生的是：

1. **向 OS 申请虚拟内存映射**：如果堆空间不够，分配器通过 `brk` 或 `mmap` 在进程已有的虚拟地址空间内申请更多的虚拟页映射（扩展堆区或可独立释放的映射区域）。
2. **管理已获得的内存**：在已申请的虚拟内存中，找到一块 256 字节的空闲区域返回给你。
3. **回收与复用**：当你 `free(p)` 时，这块内存被标记为空闲，供后续分配复用。

`malloc` / `free` 是**通用分配器**——它的设计目标是在各种未知场景下"平均性能最优"，而非"最坏情况可控"。这意味着：

- **碎片化**：长时间运行后，堆中的空闲内存被切割成碎片，大对象分配失败。
- **锁竞争**：多线程高频 `new` 时，分配器内部的全局锁成为瓶颈。
- **不确定延迟**：通用分配器可能随时触发 `brk`/`mmap` 系统调用，或进行内存整理，造成帧时间抖动。
- **局部性缺失**：连续 `malloc` 返回的地址在物理内存中可能相距甚远，破坏 Cache 友好性。

这些问题对普通后台服务可能无关紧要，但对一帧只有 16.6ms 的游戏引擎是致命的。理解分配器的底层原理，才能知道**通用分配器的边界在哪里**，以及**下一篇我们自己实现分配器时要解决什么问题**。

---

## What：堆分配器的核心机制

### 堆内存从哪里来？

操作系统为进程创建的虚拟地址空间是固定的（如 x86-64 用户空间 128TB），其中包含堆区（Heap）。进程启动时堆区很小，且真正的物理内存是在**首次访问时才通过 Page Fault 分配的**（详见上一篇的按需分页）。分配器向操作系统"批发"虚拟内存的方式有两种：

| 方式 | 机制 | 阈值（glibc） | 核心特点 |
|------|------|-------------|----------|
| `brk` | 移动数据段末尾指针，扩展/收缩连续堆空间 | < 128 KB | 地址连续，但无法独立归还物理页 |
| `mmap`（匿名映射） | 向 OS 申请独立的虚拟内存区域 | >= 128 KB | 独立区域，可精确 `munmap` 归还 |

**为什么小内存用 `brk`，大内存用 `mmap`？**

`brk` 管理的堆是一段连续的虚拟地址空间。分配器在其中维护各种大小的空闲块，小对象频繁分配释放时，不需要反复系统调用，只需在用户态管理这块区域。但 `brk` 有一个致命问题：**即使你把中间某块内存 `free` 了，只要堆顶还有被占用的内存，就无法把物理页归还给 OS**——因为 `brk` 只能收缩堆顶指针。

`mmap` 每次映射独立的虚拟内存区域（在进程的 mmap 区域中），释放时可以直接 `munmap` 把对应的虚拟页和物理页一起还给操作系统。因此大内存用 `mmap`，避免长期占用不释放的物理页。

> 128 KB 这个阈值是 glibc 的经验值，不同系统/版本可能不同。

### 分配器算法

#### 伙伴系统（Buddy System）

伙伴系统是一种经典的**按 2 的幂次分块**的管理策略，常用于操作系统内核管理物理页帧。

**核心思想**：把整个内存区域初始化为一个 2^n 大小的大块。当申请大小为 `2^k` 的内存时：

1. 找一个足够大的空闲块（`2^m`，其中 `m >= k`）。
2. 如果太大，就**递归对半分裂（split）**，直到得到恰好 `2^k` 的块。
3. 释放时，检查它的**伙伴块**（与它大小相同、地址相邻的另一半）是否空闲。
4. 如果伙伴也空闲，就**合并（coalesce）**成一个更大的 `2^(k+1)` 块，继续向上合并。

```
初始：一个 1024 大小的块

分配 100 字节 → 向上取整到 128
  1024 分裂为 512 + 512
  第一个 512 分裂为 256 + 256
  第一个 256 分裂为 128 + 128
  → 返回第一个 128 块

内存树：
        [1024]
       /      \
    [512]     [512(空闲)]
   /    \
[256]  [256(空闲)]
 /   \
[128] [128(空闲)] ← 返回给用户

释放 128 块后：
  伙伴（相邻的 128）是空闲的 → 合并为 256
  伙伴（相邻的 256）是空闲的 → 合并为 512
  伙伴（相邻的 512）不是空闲的 → 停止
```

**优点**：合并逻辑简单高效，没有复杂的空闲块搜索。
**缺点**：内部碎片严重——申请 100 字节却要分配 128 字节，多出的 28 字节被浪费。只适合管理固定粒度的内存（如内核物理页）。

---

#### Slab Allocator

Slab Allocator 最初由 SunOS 引入 Linux 内核，核心思想是**按尺寸分类（size class）**。

**为什么按尺寸分类？**

假设你频繁分配 32 字节、64 字节、256 字节的对象。如果把这些对象混存在一个通用空闲链表中，分配时你需要遍历链表找到"第一个足够大的块"，释放时还要考虑与相邻空闲块合并。而如果你预先把内存划分成若干条"专线"：

- 32 字节专线：只存 32 字节的块
- 64 字节专线：只存 64 字节的块
- 256 字节专线：只存 256 字节的块

那么分配和释放都变成 **O(1)**：从对应尺寸的链表中弹出一个节点，或把节点压回去。无需搜索，无需合并。

**Slab 的结构**：

每个 size class 维护一个或多个 **Slab**（一块连续的物理页）。Slab 被切成固定大小的对象：

```
Slab（4KB 页）for 64-byte size class：
┌─────┬─────┬─────┬─────┬─────┬─────┐
│ 64B │ 64B │ 64B │ 64B │ ... │ 64B │  ← 64 个对象
└─────┴─────┴─────┴─────┴─────┴─────┘
        ↑
   空闲链表：obj3 → obj7 → obj12 → null
```

**适用场景**：内核对象（task_struct、inode）、游戏引擎中大量同构对象（粒子、刚体）。Slab 的变体就是下一篇要讲的 **Pool Allocator**。

---

#### ptmalloc（glibc）

ptmalloc（pthreads malloc）是 Linux 上最广泛使用的通用分配器，它的设计反映了"通用场景下的折中"。

**核心结构：Arena**

ptmalloc 用 **arena** 管理从 OS 申请来的内存。一个进程有一个**主 arena**（用 `brk`），多线程时还会创建多个**非主 arena**（用 `mmap`）。每个 arena 内部维护复杂的空闲块数据结构（bins）。

```
进程
├── 主 arena（brk 管理）
│   ├── fast bins（小对象缓存，单向链表）
│   ├── small bins（精确尺寸，双向链表）
│   ├── large bins（范围尺寸，双向链表）
│   └── unsorted bin（刚释放的块，先放在这里）
├── 非主 arena 1（mmap 管理）
├── 非主 arena 2（mmap 管理）
└── ...
```

**tcache（Thread Cache）**

ptmalloc 2.26+ 引入了 **per-thread cache**。每个线程有一个私有的小对象缓存（默认最多 7 个同尺寸对象）。分配小对象时先从 tcache 取，释放时先放回 tcache，**完全无锁**。只有当 tcache 空/满时才与 arena 交互（需要加锁）。

**ptmalloc 的问题**：

- **复杂度高**：为了通用性，维护了 fast/small/large/unsorted 多种 bin，分配路径长。
- **锁竞争**：虽然 tcache 缓解了小对象竞争，但中等对象和大对象仍需要在 arena 上加锁。线程数多、分配尺寸分散时，竞争显著。
- **碎片不可控**：长期运行后，大量小空闲块散布在 arena 中，虽然总计空闲内存很多，但没有足够大的连续块满足大对象分配。
- **非确定性**：`malloc` 可能触发 arena 扩展（`brk`/`mmap`）、空闲块合并、甚至 `madvise` 归还内存，任何一次调用都可能耗时剧增。

---

#### tcmalloc（Google）

tcmalloc 的设计目标很明确：**高并发场景下的小对象分配要快，且几乎无锁**。

**三层架构**：

```
Thread 0          Thread 1          Thread 2
   │                 │                 │
┌──▼──┐          ┌──▼──┐          ┌──▼──┐
│TC 0 │          │TC 1 │          │TC 2 │   ← Thread Cache（per-thread，无锁）
└──┬──┘          └──┬──┘          └──┬──┘
   │  批量申请/归还   │  批量申请/归还   │
   └────────┬────────┘
            ▼
     ┌─────────────┐
     │ Central     │   ← Central Free List（按 size class，需要原子操作或轻量锁）
     │ Free List   │
     └──────┬──────┘
            │  整页申请/归还
            ▼
     ┌─────────────┐
     │ Span        │   ← Page Heap（管理从 OS mmap 来的页）
     │ Management  │
     └─────────────┘
```

1. **Thread Cache**：每个线程私有，按 size class（如 8, 16, 32, 48...）缓存小对象。分配/释放只需操作本地链表，**完全无锁**。但容量有限（默认 2MB），超限后批量归还到 Central Free List。
2. **Central Free List**：全局的 size class 缓存。Thread Cache 空时从这里批量取（如一次取 32 个）；Thread Cache 满时批量归还。用原子操作或自旋锁，竞争比 ptmalloc 的全局 arena 锁轻得多。
3. **Span Management**：向 OS 申请内存以 **span**（连续的 1~128 页）为单位。Span 按 size class 切成小对象，或作为大对象直接分配。

**tcmalloc 的优势**：
- 小对象分配路径极短，无锁。
- 大对象直接走 Page Heap，与 GC/小对象路径分离。
- 适合高并发服务器（如 Google 内部服务）。

**局限**：线程退出时，其 Thread Cache 中的内存不会立即释放给其他线程，可能造成内存占用偏高。

---

#### jemalloc（Facebook）

jemalloc 最初为 FreeBSD 编写，后被 Firefox、Facebook、《原神》等广泛采用。它的核心设计目标是**控制碎片**和**可预测的扩展性**。

**Arena + Size Class 分桶**：

jemalloc 把内存划分为多个 **arena**（默认 CPU 核心数 × 4 个）。线程通过哈希绑定到某个 arena，避免全局锁。每个 arena 内部按严格的 size class 管理：

| 类别 | Size Class 间隔 | 碎片上限 |
|------|----------------|---------|
| Small（< 4KB）| 2 的幂次附近，间隔 16B~512B | 约 25% |
| Large（4KB ~ 2MB）| 页大小的整数倍 | 接近 1 页 |
| Huge（> 2MB）| chunk 大小的整数倍 | 接近 1 chunk |

**关键设计：严格的 size class 对齐**

jemalloc 的 size class 表经过精心计算，确保任何分配请求向上取整到最近的 size class 时，**内部碎片被严格限制**。例如 48 字节的请求可能被分配到 48 字节的精确槽位，而不是像伙伴系统那样跳到 64 字节。

**jemalloc vs tcmalloc vs ptmalloc**：

| 维度 | ptmalloc | tcmalloc | jemalloc |
|------|----------|----------|----------|
| 小对象性能 | 中（tcache 缓解后较好） | 极快（无锁 Thread Cache） | 很快（arena 绑定，几乎无锁） |
| 碎片控制 | 差 | 中 | 优（严格 size class） |
| 大对象性能 | 中 | 快（独立路径） | 快（独立路径） |
| 内存足迹 | 低 | 偏高（Thread Cache 不立即释放） | 中 |
| 线程扩展性 | 差（arena 锁竞争） | 优 | 优 |

### 碎片：分配器的宿敌

理解分配器必须理解**碎片（Fragmentation）**，它是所有通用分配器无法根治的问题。

**外部碎片（External Fragmentation）**：空闲内存总量足够，但被分割成许多不连续的小块，无法满足大对象分配。

```
堆内存：
┌─────┬─────┬─────┬─────┬─────┬─────┐
│ 已用 │ 空闲 │ 已用 │ 空闲 │ 已用 │ 空闲 │
│ 4KB │ 4KB │ 4KB │ 4KB │ 4KB │ 4KB │
└─────┴─────┴─────┴─────┴─────┴─────┘
                ↑
       空闲总量 12KB，但最大连续块只有 4KB
       申请 8KB 会失败！
```

**内部碎片（Internal Fragmentation）**：分配给用户的内存比实际请求大，多余部分被浪费。

```
请求 100 字节，分配器按 128 字节 size class 分配
→ 内部碎片 28 字节
```

通用分配器为了服务任意尺寸、任意顺序的分配/释放请求，必然产生碎片。**下一篇的自定义分配器之所以高效，正是因为我们放弃了"通用性"**——知道对象尺寸、知道生命周期、知道分配模式，才能设计出零碎片或低碎片的策略。

## How：如何实践？

### 1. 观察分配行为

用 `strace` 跟踪程序的内存申请，观察 `brk` 和 `mmap` 的触发时机：

```bash
# Linux
strace -e brk,mmap,munmap ./your_program
```

### 2. 对比分配器

```bash
# 链接 tcmalloc
g++ main.cpp -o main_tcmalloc -ltcmalloc

# 链接 jemalloc
g++ main.cpp -o main_jemalloc -ljemalloc
```

### 3. 重载 `new/delete` 统计分配

在引擎开发中，第一步通常是重载全局 `new/delete`，统计分配热点：

```cpp
// workspace/os-lab/memory/hook_new_delete.cpp
#include <new>
#include <cstdlib>
#include <atomic>
#include <iostream>

static std::atomic<size_t> g_alloc_count{0};
static std::atomic<size_t> g_alloc_bytes{0};

void* operator new(size_t size) {
    g_alloc_count.fetch_add(1, std::memory_order_relaxed);
    g_alloc_bytes.fetch_add(size, std::memory_order_relaxed);
    return std::malloc(size);
}
void operator delete(void* p) noexcept {
    std::free(p);
}
// 同样重载 new[] / delete[] ...

int main() {
    for (int i = 0; i < 1000; ++i) {
        auto p = new char[128];
        delete[] p;
    }
    std::cout << "Allocs: " << g_alloc_count.load() << "\n";
    std::cout << "Bytes : " << g_alloc_bytes.load() << "\n";
    return 0;
}
```

## C++ 本地验证实验

### 实验 1：高频分配性能对比

```cpp
// workspace/os-lab/memory/alloc_bench.cpp
#include <vector>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int ITER = 100'000;
constexpr int NUM_THREADS = 4;

void bench(const char* name, auto alloc_fn, auto free_fn) {
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int t = 0; t < NUM_THREADS; ++t) {
        threads.emplace_back([&]() {
            for (int i = 0; i < ITER; ++i) {
                void* p = alloc_fn(256);
                free_fn(p);
            }
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    bench("malloc/free",
          [](size_t n){ return malloc(n); },
          [](void* p){ free(p); });
    return 0;
}
```

**进阶**：分别链接 `-ltcmalloc` 和 `-ljemalloc`，观察多线程小对象分配的性能差异。通常 tcmalloc/jemalloc 比 ptmalloc 快 **2-10 倍**（线程数越多差距越大）。

## 游戏引擎延伸

### 为什么 UE 不用 malloc？

Unreal Engine 不直接使用 `malloc`，而是通过 `FMemory::Malloc` / `Free` / `Realloc` 抽象一层。原因正是我们上面分析的通用分配器的缺陷：

- **帧时间确定性**：`malloc` 可能触发系统调用或内存整理，导致某一帧突然耗时翻倍。
- **数据局部性**：通用分配器不保证连续分配的内存在物理上相邻，破坏 CPU Cache 和 GPU 传输效率。
- **调试需求**：引擎需要在分配时嵌入 guard band（越界检测）、分配栈跟踪、泄漏检测，通用分配器无法灵活支持。

**UE 的分配器家族**：

| 分配器 | 原理 | 用途 |
|--------|------|------|
| `FMallocBinned` | 按 size class 分桶，类似 Slab | 默认小对象分配 |
| `FMallocBinned2` | Binned 的改进版，更好的线程扩展性 | 现代 UE 默认 |
| `FMallocMimalloc` | 接入微软 mimalloc | 可选高性能后端 |
| `TMemStack` | Linear Allocator | 渲染命令、临时数据 |
| `TBlockAllocator` | Pool Allocator | UObject、组件索引管理 |

### Chaos 引擎的 TMemoryPool

物理模拟模块（Chaos）需要大量同构对象（如刚体、碰撞体、约束）。使用 `TMemoryPool`：

- **预分配**：启动时向 OS 申请一大块虚拟内存，切成固定大小的对象槽位。
- **O(1) 取出/归还**：用自由链表管理空闲槽位，分配释放只需指针操作。
- **无碎片**：所有对象同尺寸，不存在外部碎片；尺寸精确匹配，无内部碎片。
- **按 Cache Line 对齐**：对象起始地址对齐到 64B，保证 SIMD 运算效率。

这正是 **Slab / Pool Allocator 思想**在引擎中的直接应用。通用分配器无法提供这种确定性，因为通用分配器不知道"你只会分配这一种对象"。
