# 线程基础与调度

---

## Why：为什么游戏引擎需要多线程？

- **硬件趋势**：CPU 频率停滞，核心数增加；单核性能提升有限，必须并行。
- **任务天然可并行**：渲染、物理、音频、网络、AI 可相对独立运行。
- **隐藏延迟**：在 IO 等待时，其他线程继续工作（如资源加载不阻塞渲染）。

---

## What：核心概念

### 进程 vs 线程

| 特性 | 进程 | 线程 |
|------|------|------|
| 地址空间 | 独立 | 共享 |
| 切换代价 | 高（需切换页表、刷新 TLB） | 低（仅保存寄存器、栈指针） |
| 通信 | IPC（管道、共享内存、Socket） | 直接读写共享内存 |
| 崩溃隔离 | 进程隔离 | 一个线程崩溃，整个进程崩溃 |

### 线程调度

- **抢占式调度**：操作系统定时器中断强制切换线程（时间片通常 10-20ms）。
- **上下文切换开销**：保存/恢复寄存器、切换内核栈、可能刷新缓存，**数微秒到数十微秒**。
- **线程数陷阱**：线程数远大于核心数时，上下文切换开销会吞噬计算收益。

### 线程亲和性（Affinity）

- 将线程绑定到特定核心，避免跨 NUMA 节点调度，提升缓存命中率。
- Windows：`SetThreadAffinityMask`；Linux：`pthread_setaffinity_np`。

---

## How：如何实践？

1. **线程数 = 硬件线程数**（或略少），留一个核心给操作系统和驱动。
2. **工作线程不阻塞**：避免在线程池中执行可能阻塞的 IO 操作。
3. **绑定关键线程**：如渲染线程、音频线程绑定到独立核心，减少调度抖动。

## C++ 本地验证实验

### 实验 1：线程数与加速比

```cpp
// workspace/os-lab/concurrency/thread_scaling.cpp
#include <thread>
#include <vector>
#include <chrono>
#include <iostream>
#include <atomic>

constexpr int WORK_PER_THREAD = 10'000'000;

void cpu_work() {
    volatile int sum = 0;
    for (int i = 0; i < WORK_PER_THREAD; ++i) sum += i;
    (void)sum;
}

void bench(int num_threads) {
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < num_threads; ++i) threads.emplace_back(cpu_work);
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << num_threads << " threads: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    for (int t = 1; t <= std::thread::hardware_concurrency(); ++t) {
        bench(t);
    }
    return 0;
}
```

**预期结果**：线程数 <= 核心数时接近线性加速；超过后收益递减甚至下降。

### 实验 2：线程亲和性绑定

```cpp
// workspace/os-lab/concurrency/affinity.cpp
#include <thread>
#include <iostream>

#ifdef _WIN32
#include <windows.h>
void set_affinity(int core_id) {
    SetThreadAffinityMask(GetCurrentThread(), 1ULL << core_id);
}
#else
#include <pthread.h>
void set_affinity(int core_id) {
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(core_id, &cpuset);
    pthread_setaffinity_np(pthread_self(), sizeof(cpuset), &cpuset);
}
#endif

int main() {
    std::thread t([]() {
        set_affinity(2); // 绑定到核心 2
        std::cout << "Thread running on core 2\n";
        volatile int s = 0;
        for (int i = 0; i < 1e8; ++i) s += i;
    });
    t.join();
    return 0;
}
```

## 游戏引擎延伸

### 游戏主循环与线程分离

- **经典模型**：
  - **主线程**：游戏逻辑 Tick、输入处理。
  - **渲染线程**：GPU 命令提交、Present（或单独 RHI 线程）。
  - **工作线程池**：Job System，执行可并行的任务（Culling、动画、物理步进）。
- UE 的 `FRunnableThread` 封装了平台线程，可设置优先级和亲和性。

### Job System 的线程管理

- 固定数量的工作线程（通常 `hardware_concurrency - 1`），绑定到物理核心。
- 主线程生成 Job，推入无锁队列，工作线程窃取（Work Stealing）执行。
- 避免 OS 调度器干扰：线程数不超过核心数，不阻塞，不频繁创建/销毁。
