# 进程与线程——为什么需要多线程

---

## 从一个问题开始：你的游戏卡了

想象你正在玩一款开放世界游戏。画面里同时在发生很多事：

- 远处的树木和山丘需要渲染
- 物理引擎在计算车辆碰撞
- 网络线程在接收其他玩家的位置
- 音频系统在播放引擎轰鸣和背景音乐
- AI 在决定 NPC 下一步往哪走

如果这些事情全部由一个"工人"（主线程）按顺序做完——先渲染，再算物理，再处理网络，再混音——那么每一帧要花的时间就是所有这些工作的总和。当场景复杂起来，一帧可能需要 50 毫秒甚至 100 毫秒，画面就会明显卡顿（20 FPS 以下）。

**多线程的目的很简单：让多个工人同时干活，把一帧的时间压下去。**

但"同时干活"不是免费的。如果几个工人要修改同一份数据，就会出问题。理解多线程编程，就是理解"如何让多个工人安全地共享资源"。这一整系列笔记，都会围绕这个问题展开。

---

## 程序 vs 进程：从图纸到房子

在谈"多个工人"之前，先搞清楚"房子"是什么。

你电脑里有一个 `.exe` 文件，比如 `Game.exe`。它静静地躺在硬盘上，是一堆字节——这叫**程序（Program）**。程序就像一张建筑图纸，描述了"该做什么"，但本身不会动。

当你双击 `Game.exe`，操作系统开始做几件事：

1. **划一块地皮**：在内存中创建一个独立的**虚拟地址空间**，进程 A 的地址 `0x1000` 和进程 B 的地址 `0x1000` 是完全不同的物理内存。
2. **按图纸盖房子**：把程序文件里的代码和数据搬进内存，安排好各自的位置。
3. **登记信息**：创建一个 **PCB（Process Control Block，进程控制块）**，记录进程 ID、页表位置、打开的文件等。
4. **派第一个工人入住**：创建**主线程**，从 `main()` 开始执行。

到这里，程序就变成了**进程（Process）**——一个正在运行的、活的实体。

> **关键认知**：一个进程里至少有一个线程（主线程）。进程是操作系统**分配资源**的最小单位，线程是操作系统**调度执行**的最小单位。

---

## 为什么要发明"线程"？一座房子里的多个工人

既然进程已经能跑了，为什么还要线程？答案一句话：**创建和切换进程太贵了**。

进程之间是完全隔离的——不同的虚拟地址空间、不同的页表。如果两个进程想交换数据，必须通过操作系统提供的 IPC 机制（管道、Socket、共享内存），速度慢且麻烦。更关键的是，进程切换时要更换整个页表、刷新 TLB（快表），代价极高。

**线程就是在同一座房子里塞进更多工人**。所有线程共享同一块虚拟地址空间、同一个页表、同一套打开的文件。但每个线程有自己的：

- **程序计数器**（PC）：执行到哪一行代码了
- **寄存器**：临时计算结果放在哪里
- **栈**：函数调用历史和局部变量

线程切换时，操作系统只需要保存/恢复这三个东西（记到 **TCB，Thread Control Block** 里），不需要碰页表——这就是为什么线程切换比进程切换快一个数量级。

```cpp
// 验证：进程隔离 vs 线程共享
#include <iostream>
#include <thread>
#include <unistd.h>  // Linux: fork()
#include <sys/wait.h>

int global_var = 42;

void test_process_isolation() {
#ifdef __linux__
    pid_t pid = fork();
    if (pid == 0) {
        // 子进程：修改 global_var
        global_var = 999;
        std::cout << "[子进程] global_var = " << global_var << "\n";
        _exit(0);
    } else {
        wait(nullptr);
        // 父进程：global_var 不变！
        std::cout << "[父进程] global_var = " << global_var << "\n";
    }
#endif
}

void test_thread_sharing() {
    std::thread t([]() {
        global_var = 777;
        std::cout << "[子线程] global_var = " << global_var << "\n";
    });
    t.join();
    // 主线程：global_var 被改了！
    std::cout << "[主线程] global_var = " << global_var << "\n";
}

int main() {
    std::cout << "=== 进程隔离测试 ===\n";
    test_process_isolation();
    
    global_var = 42;  // 重置
    
    std::cout << "\n=== 线程共享测试 ===\n";
    test_thread_sharing();
    return 0;
}
```

**编译运行（Linux）**：
```bash
g++ -std=c++17 process_vs_thread.cpp -o process_vs_thread -pthread
./process_vs_thread
```

**预期输出**：
```
=== 进程隔离测试 ===
[子进程] global_var = 999
[父进程] global_var = 42

=== 线程共享测试 ===
[子线程] global_var = 777
[主线程] global_var = 777
```

> **核心结论**：子进程修改 `global_var` 不影响父进程（隔离）；子线程修改 `global_var` 主线程立刻看到（共享）。这个"共享"是多线程性能的来源，也是所有 bug 的根源。

## 共享内存带来的问题：数据竞争

既然线程共享内存，那如果两个线程同时修改同一个变量会怎样？

```cpp
int counter = 0;

void increment() {
    for (int i = 0; i < 100000; ++i) {
        ++counter;  // 两个线程同时执行这一行……
    }
}
```

在 CPU 层面，`++counter` 不是"一步完成的"，它分三步：

1. 从内存读取 `counter` 的值到寄存器
2. 寄存器里的值加 1
3. 把新值写回内存

如果线程 A 执行完第 1 步（读到 100），这时候操作系统把 A 换下去，换线程 B 上来，B 也执行第 1 步（也读到 100）。然后 B 执行完 2、3（内存变成 101），A 被换回来继续执行 2、3（也写回 101）。**原本应该加两次，结果只加了一次**。

这就是**数据竞争（Data Race）**——多个线程同时读写同一块内存，且至少有一个是写操作，没有任何同步措施。数据竞争会导致未定义行为：程序可能 99% 的时间运行正常，但在高负载或特定时序下崩溃或产生错误结果。

下一篇笔记，你会学到在 C++ 中如何创建线程。再下一篇，我们会用代码实际演示这个数据竞争，然后一步步学习如何解决它。

## 小结

| 概念 | 比喻 | 核心作用 |
|------|------|---------|
| 程序 | 建筑图纸 | 描述"该做什么"，静态 |
| 进程 | 盖好的房子 | 资源容器，独立地址空间 |
| 线程 | 房子里的工人 | 执行流，共享进程资源 |
| 数据竞争 | 两个工人同时改一份账本 | 多线程需要同步的原因 |
