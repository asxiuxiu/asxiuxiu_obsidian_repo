# 自旋锁——另一种等待方式

---

## 从一个问题开始：Mutex 在临界区很短时太"重"了

上一篇我们学了 `std::mutex`。它工作得很好，但当临界区很短时（比如就一句 `++counter`），Mutex 的代价反而成了瓶颈。

为什么？传统认知里，Mutex 竞争失败时操作系统会把线程**挂起（睡眠）**，切换到别的线程，等锁释放后再**唤醒**。一次上下文切换就是几微秒。如果临界区本身只要几十纳秒，你为了等这几十纳秒，付出了几千倍的代价。

但这里有个重要的时代背景：**现代操作系统的 Mutex 已经不是"纯睡眠"了**。glibc 的 `pthread_mutex` 和 macOS 的 `os_unfair_lock` 普遍采用**自适应自旋**——竞争失败时先自旋一小段时间，如果锁很快释放就直接拿到，只有真的等很久才进入内核睡眠。所以 `std::mutex` 实际上已经内置了一个"优化版的自旋锁"。

那么手写自旋锁的意义在哪？在于**理解它**，以及在**嵌入式或内核态**等无法使用系统同步原语的场景中派上用场。我们来从头实现一个正确的版本。

**自旋锁（Spinlock）** 的思路就是："我就站在这儿等，不停敲门问'好了没？'"

---

## Spinlock：站着等

自旋锁的逻辑很简单：线程在循环中反复检查锁是否释放，**不放弃 CPU**。如果锁很快释放（下一个时钟周期就解了），自旋锁几乎没有额外开销。

用 `std::atomic` 可以实现一个自旋锁，但有个陷阱要注意：

```cpp
#include <atomic>
#include <thread>

class SpinlockBad {
    std::atomic<bool> flag{false};
public:
    void lock() {
        // ❌ 问题：即使锁已经被占了，每次循环都在写 flag
        while (flag.exchange(true, std::memory_order_acquire)) {
            #if defined(__x86_64__) || defined(__i386__)
            __asm__ volatile("pause" ::: "memory");
            #elif defined(__aarch64__)
            __asm__ volatile("yield" ::: "memory");
            #else
            std::this_thread::yield();
            #endif
        }
    }
    void unlock() { flag.store(false, std::memory_order_release); }
};
```

**上面的实现有个隐蔽的性能陷阱**：`exchange(true)` 是**写操作**。即使 `flag` 已经是 `true`（锁被占了），它每次循环都强行把 `true` 写回去。这会导致其他 CPU 核心上的缓存行被反复**失效（Invalidate）**，缓存一致性协议（MESI）产生大量额外流量。

**这就像一个电梯门铃，你每 10 毫秒按一次。锁被占时按门铃毫无用处，但你的每次"按"都会惊动整栋楼的门禁系统。**

正确的做法是 **TTAS（Test-Test-And-Set）**：锁被占时只读检查，看到空闲了才尝试写：

```cpp
class Spinlock {
    std::atomic<bool> flag{false};
public:
    void lock() {
        while (true) {
            // 只有看到 flag == false 时，才尝试抢锁（写操作）
            if (!flag.exchange(true, std::memory_order_acquire)) {
                return; // 抢到了
            }
            // 锁被占时，只读检查，不污染缓存
            while (flag.load(std::memory_order_relaxed)) {
                #if defined(__x86_64__) || defined(__i386__)
                __asm__ volatile("pause" ::: "memory");
                #elif defined(__aarch64__)
                __asm__ volatile("yield" ::: "memory");
                #else
                std::this_thread::yield();
                #endif
            }
        }
    }
    void unlock() { flag.store(false, std::memory_order_release); }
};
```
```

**关键代码解析**：

- `flag.exchange(true, acquire)`：这是**原子操作**。它把 `flag` 设为 `true`，同时返回原来的值。如果返回 `false`，说明之前没人持锁，你成功抢到了。如果返回 `true`，说明锁被占了。

- **外层 `while(true)` + `exchange`**：尝试抢锁。只有看到 `flag == false` 时才做写操作。

- **内层 `while(flag.load(...))`**：锁被占时只做**读操作**（`load` 是读，不会触发缓存一致性协议的 Invalidate），安静等待。

- `pause` / `yield` 指令：纯粹自旋会疯狂占用 CPU，`pause` 告诉 CPU"我在忙等，可以优化功耗和流水线"，`yield` 则主动让出 CPU 给同核心的其他超线程。

> [!warning] `pause` 指令在现代 CPU 上变"慢"了
> Intel Skylake 及以后架构中，`pause` 的执行周期从 ~10 cycles 暴涨到 ~100+ cycles。这意味着自旋间隔变长，自旋锁的"站着等"效率有所下降。

### 另一种实现：CAS

除了 `exchange`，自旋锁也可以用 **CAS（Compare-And-Swap）** 实现。同样要注意 TTAS 优化：

```cpp
void lock() {
    while (true) {
        bool expected = false;
        // 只有当 flag == false（锁空闲）时，才把它换成 true
        if (flag.compare_exchange_weak(expected, true, std::memory_order_acquire)) {
            return;
        }
        // 锁被占时，只读等待
        while (flag.load(std::memory_order_relaxed)) {
            #if defined(__x86_64__) || defined(__i386__)
            __asm__ volatile("pause" ::: "memory");
            #elif defined(__aarch64__)
            __asm__ volatile("yield" ::: "memory");
            #endif
        }
    }
}
```

**为什么 CAS 版本用 `weak` 而不是 `strong`？**

- `compare_exchange_weak` 在 x86 上表现和 `strong` 一样，但在 ARM 等架构上可能**假失败**（即使条件满足也返回失败），优点是性能更好。
- 自旋锁本来就是 `while` 循环，假失败大不了再转一圈，所以用 `weak` 更划算。
- 如果你不能容忍假失败（比如一次 CAS 成功后要做不可逆操作），才用 `compare_exchange_strong`。

> CAS 的优势在于需要"先判断再更新"的复杂逻辑（比如无锁队列的指针交换）。对于简单的自旋锁，`exchange` 更简洁。

---

## 什么时候用 Spinlock，什么时候用 Mutex？

| 场景 | 推荐 | 原因 |
|------|------|------|
| 临界区很短（几行代码） | Spinlock | 避免上下文切换开销 |
| 临界区长或竞争激烈 | Mutex | 自旋会浪费大量 CPU |
| 单核 CPU | Mutex | 自旋时占着 CPU，持锁线程没机会释放锁 |
| 需要睡眠（如等 IO） | Mutex | Spinlock 里不能睡眠 |
| 用户态应用（现代系统） | Mutex | `std::mutex` 已内置自适应自旋，通常更优 |

> **核心判断**：预期等待时间 < 两次上下文切换时间 → Spinlock；否则 → Mutex。
> 
> 但在现代系统上，`std::mutex` 通常已经是最佳选择——它内部会根据竞争情况自动在"自旋"和"睡眠"之间切换。手写自旋锁更多是为了理解原理，或在无系统库支持的环境（如内核、裸机）中使用。

## 动手实验

### 实验：Mutex vs Spinlock 性能对比

```cpp
// flags: -pthread -O2
#include <mutex>
#include <atomic>
#include <thread>
#include <vector>
#include <chrono>
#include <iostream>

constexpr int ITER = 1'000'000;
constexpr int NUM_THREADS = 4;

// TTAS 自旋锁实现
class Spinlock {
    std::atomic<bool> flag{false};
public:
    void lock() {
        while (true) {
            if (!flag.exchange(true, std::memory_order_acquire)) {
                return; // 抢到锁
            }
            while (flag.load(std::memory_order_relaxed)) {
                #if defined(__x86_64__) || defined(__i386__)
                __asm__ volatile("pause" ::: "memory");
                #elif defined(__aarch64__)
                __asm__ volatile("yield" ::: "memory");
                #else
                std::this_thread::yield();
                #endif
            }
        }
    }
    void unlock() { flag.store(false, std::memory_order_release); }
};

template<typename Lock>
void bench(const char* name) {
    Lock lk;
    int counter = 0;
    
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < NUM_THREADS; ++i) {
        threads.emplace_back([&]() {
            for (int j = 0; j < ITER; ++j) {
                lk.lock();
                ++counter;  // 临界区：只有一行！
                lk.unlock();
            }
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    
    double ms = std::chrono::duration<double, std::milli>(t2 - t1).count();
    std::cout << name << ": " << ms << " ms (counter=" << counter << ")\n";
}

int main() {
    bench<std::mutex>("Mutex   ");
    bench<Spinlock>("Spinlock");
    return 0;
}
```

**编译运行**：
```bash
g++ -std=c++17 mutex_vs_spinlock.cpp -o mutex_vs_spinlock -pthread -O2
./mutex_vs_spinlock
```

**预期结果**（x86-64，4线程）：
- **Mutex**：通常 100~300 ms（现代 Mutex 已内置自旋优化，不一定比手写 Spinlock 慢）
- **TTAS Spinlock**：通常 100~300 ms，与 Mutex 接近或略快
- **错误的 TAS Spinlock**（每次循环都 `exchange`）：通常比两者都慢，因为缓存一致性开销

> [!info] 为什么我的机器上自旋锁更慢？
> 常见原因：
> 1. **用了 TAS 而不是 TTAS**：笔记之前的版本是 TAS，每次循环写缓存，缓存一致性开销巨大
> 2. **现代 Mutex 已进化**：`std::mutex` 底层通常先自旋后睡眠，不是纯睡眠
> 3. **`pause` 指令周期变长**：Skylake 以后 `pause` 从 ~10 cycles 变成 ~100+ cycles
> 4. **核心数不足或调度干扰**：如果物理核心少于线程数，自旋线程被切走时纯粹浪费 CPU

## 小结

| 特性 | Mutex | Spinlock |
|------|-------|----------|
| 等待方式 | 线程睡眠，不占用 CPU | 循环检查，占用 CPU |
| 获得锁时的开销 | 上下文切换（几微秒） | 几乎没有 |
| 适用场景 | 临界区长、竞争激烈 | 临界区短、竞争低 |
| 实现复杂度 | 简单（标准库直接提供） | 需自己实现或第三方库 |

> **问题链**：现在你有两种保护临界区的方式了。但无论是 Mutex 还是 Spinlock，消费者线程在队列为空时都在"干等"——Mutex 在反复加锁解锁，Spinlock 在疯狂自旋。能不能让线程"有任务了再叫醒我"？→ [[条件变量——有消息了叫你]]
