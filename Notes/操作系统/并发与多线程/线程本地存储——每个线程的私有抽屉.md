# 线程本地存储——每个线程的私有抽屉

---

## 从一个问题开始：如果每个线程都有自己的副本，还需要锁吗？

上一篇我们学了 `std::atomic`——它保证"多个线程竞争同一个变量时，操作不会被打断"。但这里隐藏了一个前提：**这个变量必须被多个线程共享**。

想象这样一个场景：你的引擎有一个性能计数器，每个 Worker 线程每完成一个任务就 `++taskCount`，最后主线程汇总所有线程的计数。

用全局 atomic：

```cpp
std::atomic<int> g_taskCount{0};

// Worker 线程每完成一个任务：
g_taskCount.fetch_add(1, std::memory_order_relaxed);
```

这能工作，但 8 个 Worker 每帧各加几千次，竞争的是**同一个缓存行**。CPU 核心之间不断互相失效缓存，MESI 协议产生大量流量。

但如果换个思路：**每个线程自己计数，最后统一汇总**？每个 Worker 有一个独立的计数器，没有任何竞争，连 atomic 都不需要。

这就是 **线程本地存储（Thread Local Storage，TLS）** 的核心思想：给每个线程发一个"私有抽屉"，放它自己的数据。

---

## thread_local：每个线程独立一份

C++11 用 `thread_local` 关键字声明线程本地变量：

```cpp
thread_local int s_taskCount = 0;

void WorkerThread() {
    for (...) {
        ++s_taskCount;  // 操作的是本线程的副本，零竞争！
    }
}
```

每个线程第一次访问 `s_taskCount` 时，操作系统/运行时会为它分配一块独立的内存。线程 A 修改自己的 `s_taskCount` 完全不会影响线程 B 的副本。

### 和全局变量的本质区别

```cpp
int g_counter = 0;                  // 全局：所有线程共享同一块内存
thread_local int t_counter = 0;     // TLS：每个线程有独立副本
```

| 特性 | 全局变量 | `thread_local` |
|------|---------|----------------|
| 内存位置 | 进程全局唯一 | 每个线程各自独立 |
| 多线程访问 | 需要同步（atomic/mutex） | 天然线程安全，无需同步 |
| 生命周期 | 程序启动到结束 | 线程创建到结束 |
| 适用场景 | 真正的共享状态 | 线程私有的临时状态/缓存 |

> [!warning] TLS 不是银弹
> `thread_local` **不能解决共享数据的同步问题**。如果多个线程真的需要读写同一份数据，TLS 帮不了你——它只是把数据拆成了多份独立的拷贝。

## 编译器和操作系统在背后做了什么？

`thread_local` 不是魔法，它的实现依赖操作系统和编译器的协作。

**Linux/glibc**：
- 每个线程有一个 `Thread Control Block（TCB）`，里面存了指向 TLS 内存块的指针。
- 编译器把 `thread_local int x` 翻译成通过 `fs` 段寄存器（x86-64）或 `tpidr_el0` 寄存器（ARM64）的偏移访问。
- 访问速度几乎和栈变量一样快（一条寄存器寻址指令）。

**Windows**：
- 通过 `TlsAlloc()` / `TlsSetValue()` / `TlsGetValue()` API 管理。
- C++11 的 `thread_local` 在 MSVC 上直接用 `__declspec(thread)` 实现，同样通过段寄存器快速访问。

**关键结论**：TLS 访问的代价极低，通常只有 1~2 条指令，不需要系统调用，不需要锁。

## 引擎中的经典用法

### 用法 1：线程池 Worker 的身份识别

你的线程池笔记里有这段代码：

```cpp
class ThreadPool {
    thread_local static Worker* s_localWorker;  // 每个线程知道自己是谁
    // ...
    void SubmitLocal(std::function<void()> task) {
        if (s_localWorker) {
            s_localWorker->queue.Push(std::move(task));  // 直接访问自己的队列
        } else {
            Submit(std::move(task));  // 非 Worker 线程，走全局分发
        }
    }
};
```

为什么不用全局变量存"当前 Worker"？
- 如果用 `Worker* g_currentWorker`，每次切换 Worker 身份都要改这个全局指针，而且如果系统回调在 Worker 线程上运行，会错误地被认为是 Worker。
- `thread_local` 让每个线程的身份是**固有属性**，不需要任何同步。

### 用法 2：内存分配器的线程缓存

工业级内存分配器（如 jemalloc、tcmalloc、UE 的 `FMallocBinned`）都使用 TLS：

```cpp
thread_local ThreadCache* s_threadCache;

void* Allocate(size_t size) {
    if (size <= kSmallThreshold) {
        // 小对象直接从线程本地缓存分配，零竞争
        return s_threadCache->Pop(size);
    }
    // 大对象才走全局分配器（可能需要锁）
    return GlobalAlloc(size);
}
```

原理：大多数内存分配请求都是小对象（< 256KB）。如果每次分配都竞争全局锁，多线程性能会崩溃。TLS 让每个线程维护一个"小对象缓存池"，99% 的分配都是无竞争的。

### 用法 3：UE FNamePool 的批量分配

UE 的 `FName` 系统需要给字符串分配唯一的 ID。如果每个线程每次创建 `FName` 都竞争全局 ID 计数器，性能会很差。

UE 的做法：
1. 每个线程有一个 TLS 缓存，预分配一段 ID 范围（比如"这个线程可以分配 1000~1999"）。
2. 线程创建 `FName` 时，先从 TLS 缓存里拿一个未使用的 ID，不需要任何原子操作。
3. 只有当 TLS 缓存用完时，才用原子操作向全局池申请下一段范围。

这被称为 **TLS Batch Allocation（TLS 批量分配）**——把高频的原子操作变成了低频的批量申请。

## TLS 的局限与陷阱

### 陷阱 1：它制造的是真隔离，但也真浪费

如果 8 个线程各自有一份 1MB 的 TLS 缓存，总共就是 8MB。如果你的数据确实需要共享，TLS 不仅不能解决问题，还会让状态同步变得更复杂（你要手动汇总各个线程的副本）。

### 陷阱 2：动态加载库（DLL）中的 TLS

Windows 上，DLL 动态加载/卸载时，`thread_local` 变量的生命周期管理非常棘手。跨 DLL 边界访问 TLS 可能导致崩溃。这也是为什么有些引擎选择自己管理 TLS，而不是依赖 C++11 的 `thread_local`。

### 陷阱 3：和 `static` 的混淆

```cpp
void Foo() {
    thread_local int counter = 0;  // 每个线程第一次进 Foo 时初始化
    ++counter;
}
```

`thread_local` 变量是**延迟初始化**的（第一次使用时才构造），这和全局 `static` 的"程序启动时初始化"不同。如果构造函数有副作用，要注意它发生在线程上下文中。

## 动手实验

### 实验：验证 TLS 的竞争消除效果

```cpp
// flags: -pthread -O2
#include <thread>
#include <vector>
#include <atomic>
#include <chrono>
#include <iostream>

constexpr int ITERATIONS = 10'000'000;
constexpr int NUM_THREADS = 8;

// 方案 A：全局 atomic 计数器
std::atomic<int> g_atomicCounter{0};

void AtomicWorker() {
    for (int i = 0; i < ITERATIONS; ++i) {
        g_atomicCounter.fetch_add(1, std::memory_order_relaxed);
    }
}

// 方案 B：TLS 计数器，最后汇总
thread_local int s_tlsCounter = 0;
int g_tlsResults[NUM_THREADS] = {0};

void TLSWorker(int threadIndex) {
    for (int i = 0; i < ITERATIONS; ++i) {
        ++s_tlsCounter;  // 零竞争！
    }
    g_tlsResults[threadIndex] = s_tlsCounter;
}

template<typename SetupFn, typename WorkerFn>
void Bench(const char* name, SetupFn setup, WorkerFn worker) {
    setup();
    auto t1 = std::chrono::high_resolution_clock::now();

    std::vector<std::thread> threads;
    for (int i = 0; i < NUM_THREADS; ++i) {
        threads.emplace_back([&, i]() { worker(i); });
    }
    for (auto& t : threads) t.join();

    auto t2 = std::chrono::high_resolution_clock::now();
    double ms = std::chrono::duration<double, std::milli>(t2 - t1).count();

    int total = 0;
    for (int i = 0; i < NUM_THREADS; ++i) total += g_tlsResults[i];

    std::cout << name << ": " << ms << " ms, total=" << total << "\n";
}

int main() {
    Bench("Atomic Global",
        []() { g_atomicCounter = 0; },
        [](int) { AtomicWorker(); });

    Bench("TLS Local",
        []() { for (int i = 0; i < NUM_THREADS; ++i) g_tlsResults[i] = 0; },
        [](int i) { TLSWorker(i); });

    return 0;
}
```

**预期结果**（x86-64，8 线程）：
- Atomic Global：通常 50~150ms
- TLS Local：通常 5~20ms，**快 5~10 倍**

> [!info] 为什么差距这么大？
> Atomic Global 的每次 `fetch_add` 都要触发缓存一致性协议（MESI），8 个核心互相 invalidate。TLS Local 的 `++s_tlsCounter` 完全在自己的 L1 cache 里完成，没有任何跨核通信。

## 小结

| 策略 | 核心思想 | 适用场景 | 代价 |
|------|---------|---------|------|
| 全局 + atomic | 共享数据，原子操作保护 | 确实需要全局汇总的计数 | 缓存一致性开销 |
| 全局 + mutex | 共享数据，锁保护临界区 | 复杂逻辑的原子性 | 上下文切换/阻塞 |
| **TLS** | **每个线程独立一份，最后汇总** | **线程私有的临时状态、缓存、身份** | **内存占用增加；最后需要汇总步骤** |

> **问题链**：`thread_local` 让数据天然隔离，但引擎里总有些数据是**真的需要共享**的（如任务队列、渲染状态）。当数据必须共享时，我们该用什么工具保护它？→ [[互斥锁——保护临界区]]
