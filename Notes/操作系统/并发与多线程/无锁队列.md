# 无锁队列

---

## Why：为什么要无锁？

- **锁的问题**：Mutex 在竞争激烈时需要操作系统介入（睡眠/唤醒），上下文切换开销大；Spinlock 忙等浪费 CPU。
- **无锁的优势**：
  - 无死锁、无优先级反转。
  - 最坏情况下仍有系统级进度保证（Lock-free = 总有一个线程在前进）。
- **引擎中的应用**：Job System 的任务队列、渲染命令提交、音频包队列。

---

## What：核心概念

### SPSC vs MPMC

| 类型 | 场景 | 复杂度 |
|------|------|--------|
| **SPSC**（单生产者单消费者） | 一个工作线程与主线程通信 | 简单，只需原子头尾指针 |
| **MPMC**（多生产者多消费者） | 多线程共享任务池 | 复杂，需 CAS 循环、ABA 防护 |

### SPSC Ring Buffer 原理

- 固定大小的环形数组，头指针（`head`）由生产者更新，尾指针（`tail`）由消费者更新。
- 生产者写前检查 `(head + 1) % size != tail`，消费者读前检查 `head != tail`。
- 关键点：
  - 生产者只写 `head` 之前的位置，写完再原子更新 `head`（release）。
  - 消费者只读到 `head` 的位置，读完再原子更新 `tail`（release）。

### ABA 问题

- 某位置先为 A，被改为 B，再改回 A；CAS 检查值相同却实际经历过变化。
- **危害**：在无锁数据结构中，ABA 可能导致内存重复释放、或插入到错误的位置。
- **解决**：
  - **Tagged Pointer**：将指针与序列号打包到 64/128 位，每次操作序列号递增，即使指针值回到 A，序列号也不同。
  - **Hazard Pointer**：读线程将访问的指针注册到"危险指针"列表，写线程释放内存前检查是否有读线程仍在使用。
  - **Epoch-Based Reclamation**：将内存释放延迟到"所有可能访问过该内存的线程都进入新 Epoch"之后。

## How：实现一个 SPSC Lock-free Queue

```cpp
// workspace/os-lab/concurrency/spsc_queue.hpp
#include <atomic>
#include <vector>
#include <cassert>

template<typename T>
class SPSCQueue {
    struct Slot { T data; std::atomic<bool> ready{false}; };
    std::vector<Slot> buffer;
    alignas(64) std::atomic<size_t> head{0};
    alignas(64) std::atomic<size_t> tail{0};
    size_t capacity;
public:
    explicit SPSCQueue(size_t cap) : capacity(cap) { buffer.resize(cap); }

    bool enqueue(const T& item) {
        size_t h = head.load(std::memory_order_relaxed);
        size_t next = (h + 1) % capacity;
        if (next == tail.load(std::memory_order_acquire)) return false; // 满
        buffer[h].data = item;
        // 防止编译器重排：保证 data 写入在 ready 标志之前对其他线程可见
        std::atomic_thread_fence(std::memory_order_release);
        buffer[h].ready.store(true, std::memory_order_release);
        head.store(next, std::memory_order_release);
        return true;
    }

    bool dequeue(T& item) {
        size_t t = tail.load(std::memory_order_relaxed);
        if (t == head.load(std::memory_order_acquire)) return false; // 空
        if (!buffer[t].ready.load(std::memory_order_acquire)) return false;
        item = buffer[t].data;
        buffer[t].ready.store(false, std::memory_order_release);
        tail.store((t + 1) % capacity, std::memory_order_release);
        return true;
    }
};
```

## C++ 本地验证实验

### 实验：SPSC Queue vs Mutex Queue

```cpp
// workspace/os-lab/concurrency/spsc_bench.cpp
#include "spsc_queue.hpp"
#include <queue>
#include <mutex>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int COUNT = 5'000'000;

void bench_spsc() {
    SPSCQueue<int> q(1024);
    auto t1 = std::chrono::high_resolution_clock::now();
    std::thread producer([&]() {
        for (int i = 0; i < COUNT; ++i) {
            while (!q.enqueue(i)) { /* 自旋或yield */ }
        }
    });
    std::thread consumer([&]() {
        int v, received = 0;
        while (received < COUNT) {
            if (q.dequeue(v)) ++received;
        }
    });
    producer.join(); consumer.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "SPSC: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

void bench_mutex() {
    std::queue<int> q;
    std::mutex mtx;
    auto t1 = std::chrono::high_resolution_clock::now();
    std::thread producer([&]() {
        for (int i = 0; i < COUNT; ++i) {
            std::lock_guard lk(mtx);
            q.push(i);
        }
    });
    std::thread consumer([&]() {
        int received = 0;
        while (received < COUNT) {
            std::lock_guard lk(mtx);
            if (!q.empty()) { q.pop(); ++received; }
        }
    });
    producer.join(); consumer.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << "Mutex: "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms\n";
}

int main() {
    bench_spsc();
    bench_mutex();
    return 0;
}
```

**预期结果**：SPSC 无锁队列通常比 `std::queue` + `std::mutex` 快 **5-20 倍**。

## 游戏引擎延伸

### Job System 的任务队列

- UE 的 `FAsyncTask` 系统使用 per-thread 队列 + Work Stealing。
- 每个工作线程的本地队列用 SPSC（或 MPMC with CAS）实现，主线程 push，工作线程 pop。
- 偷取（Stealing）时从其他线程队列尾部用 CAS 拿任务，避免全局锁。

### 渲染命令提交

- 游戏逻辑线程每帧生成渲染命令，推入 SPSC Queue。
- 渲染线程在 GPU 空闲时从 Queue 取出并翻译为 GPU API 调用。
- 无锁保证逻辑线程不会因锁竞争而掉帧。
