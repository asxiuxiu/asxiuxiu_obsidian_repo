# 原子操作与内存序

---

## Why：为什么需要理解内存序？

- 编译器和 CPU 都会**重排序**指令，以提升单线程性能。
- 但多线程程序中，重排序可能破坏程序逻辑（如先看到 flag=true 却看到数据未初始化）。
- `std::atomic` + Memory Order 是 C++ 提供给程序员的「可见性契约」。

> **历史演进**：在单核 CPU 时代，编译器重排序是唯一的威胁——CPU 按顺序执行指令，只是编译器为了优化可能调整指令顺序。进入多核时代后，即使编译器不做任何重排，**CPU 自身的乱序执行（Out-of-Order Execution）** 也会导致问题：每个核心有独立的 Store Buffer 和 Invalidate Queue，写操作对其他核心的可见顺序可能与程序顺序不同。这催生了不同架构的内存模型：x86 采用相对强的 TSO（Total Store Order），而 ARM/POWER 采用极弱的内存模型，允许几乎任何重排。

---

## What：核心概念

### 原子操作

- `std::atomic<T>` 保证对 T 的读/写/RMW（Read-Modify-Write）是不可分割的。
- 常见操作：`load`, `store`, `exchange`, `compare_exchange_weak/strong`, `fetch_add` 等。

### 重排序的根源：编译器 vs CPU

理解内存序之前，必须分清**两种重排序**的来源和防御手段：

#### 编译器重排序

编译器在单线程假设下，会合法地调整指令顺序以提高流水线利用率。例如：

```cpp
// 源码顺序          // 编译器优化后可能的顺序
data = 42;          flag = true;
flag = true;        data = 42;   // 如果编译器认为 data 和 flag 无依赖
```

- **防御手段**：编译屏障（Compiler Barrier）。在 MSVC 中是 `_ReadWriteBarrier()`，在 GCC/Clang 中是 `asm volatile("" ::: "memory")`。C++11 的 `std::atomic_thread_fence` 也包含编译屏障语义。

#### CPU 乱序执行

即使编译器按源码顺序生成汇编，现代 CPU 的**超标量流水线**也会：
- **Store Buffer 延迟**：写入先进入 Store Buffer，对其他核心延迟可见（见 [[Notes/操作系统/CPU与内存架构/缓存一致性协议]]）。
- **Load 乱序**：CPU 可能提前执行后面的 Load 指令，如果它与前面的 Store 无地址依赖。
- **指令级并行**：无依赖的指令会被并行发射到不同执行单元。

- **防御手段**：CPU 内存屏障指令。x86 提供 `lfence`（Load Fence）、`sfence`（Store Fence）、`mfence`（Full Fence）。ARM 提供 `dmb`（Data Memory Barrier）、`dsb`（Data Synchronization Barrier）。

#### 编译屏障 vs CPU 屏障映射表

| C++ 语义 | 编译屏障 | x86 CPU 指令 | ARM CPU 指令 | 作用 |
|----------|----------|-------------|-------------|------|
| `memory_order_relaxed` | ❌ | 无 | 无 | 仅保证原子性，不重排同地址操作 |
| `memory_order_acquire` | ✅ | 无（x86 TSO 天然保证） | `ldar` / `dmb ishld` | 阻止后续读/写重排到本操作之前 |
| `memory_order_release` | ✅ | 无（x86 TSO 天然保证） | `stlr` / `dmb ish` | 阻止前面读/写重排到本操作之后 |
| `memory_order_acq_rel` | ✅ | 无 | `dmb ish` | 同时 acquire + release |
| `memory_order_seq_cst` | ✅ | `mfence`（仅 Store-Load 场景） | `dmb ish` | 全局顺序一致性 |

> **关键洞察**：x86-64 采用 **TSO（Total Store Order）** 强内存模型，除了"Store-Load"可能被重排外，其他顺序都严格保持。这意味着在 x86 上，`acquire`/`release` 通常不需要额外 CPU 指令（编译屏障足矣）。但 ARM64 采用**弱内存模型**，几乎任何重排都可能发生，`acquire`/`release` 会生成真实的 `ldar`/`stlr` 屏障指令。编写跨平台引擎时，**不能假设 x86 的行为在 ARM 上成立**。

### 内存序（Memory Order）

| 顺序 | 含义 | 使用场景 |
|------|------|----------|
| `memory_order_relaxed` | 无同步或顺序保证，仅原子性 | 纯计数器、标志位 |
| `memory_order_acquire` | 本操作之后的读写不能被重排序到本操作之前 | 读锁/标志 |
| `memory_order_release` | 本操作之前的读写不能被重排序到本操作之后 | 写锁/标志 |
| `memory_order_acq_rel` | 同时具有 acquire 和 release 语义 | RMW 操作 |
| `memory_order_seq_cst` | 全局顺序一致性，最强也最慢 | 默认，简单但性能差 |
| `memory_order_consume` | 类似 acquire，但只同步与当前操作有**数据依赖**的后续操作 | 罕见，C++17 起推荐使用 acquire 替代 |

### Happens-Before：形式化定义与直观理解

Happens-Before 是内存模型的**核心形式化概念**，由 Leslie Lamport 提出。

**定义**：
- 如果操作 A **sequenced-before** 操作 B（同一线程中 A 在源码顺序上先于 B），且两者都不是 `relaxed`，则 A happens-before B。
- 如果线程 A 对原子变量 M 执行 `release` 写，线程 B 对同一变量 M 执行 `acquire` 读，且 B 读到了 A 写入的值，则 A 中所有在 release 之前的操作 **happens-before** B 中在 acquire 之后的操作。

**直观理解**：Happens-Before 不是"物理时间上的先后"，而是"因果上的先后"。如果事件 A 的副作用必须在事件 B 中被观察到，那么 A happens-before B。

```
线程 A                              线程 B
─────────────────────────────────────────────────────────
...                                 ...
data = 42;      ──────┐             ...
...              被 release-acquire   if (ready.load(acquire))
ready.store(true, release) ────┼──>  {  guarantee
                                 │       data == 42
                                 │    }
                              happens-before
```

> **注意**：Happens-Before 是**传递的**。如果 A happens-before B，且 B happens-before C，则 A happens-before C。这使得通过中间变量可以建立复杂的多线程同步链。

#### Release-Acquire 的典型模式：数据发布

```cpp
// 典型模式：数据发布
std::atomic<int*> data{nullptr};

// 生产者
int* p = new int(42);
data.store(p, std::memory_order_release); // 发布

// 消费者
int* p = data.load(std::memory_order_acquire); // 获取
if (p) std::cout << *p; // 保证看到 42
```

> 这里 `data.store(release)` 不仅把指针值发布出去，还保证了"`new int(42)` 的构造" happens-before "消费者读取指针"。这是无锁数据结构中最基础的模式。

---

## How：如何正确使用？

1. **默认用 `seq_cst`**，直到性能分析证明是瓶颈。
2. **生产者-消费者用 `release`/`acquire`**：发布数据用 release，消费用 acquire。
3. **纯计数器用 `relaxed`**：如引用计数、统计计数，不用于同步。
4. **绝不混合同一变量上的不同顺序**：除非你是 Herb Sutter。

## C++ 本地验证实验

### 实验：不同 Memory Order 的性能差异

```cpp
// workspace/os-lab/concurrency/memory_order_bench.cpp
#include <atomic>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int ITER = 10'000'000;

void bench(const char* name, std::memory_order order) {
    std::atomic<int> counter{0};
    auto start = std::chrono::high_resolution_clock::now();
    std::thread th1([&]() {
        for (int i = 0; i < ITER; ++i) counter.fetch_add(1, order);
    });
    std::thread th2([&]() {
        for (int i = 0; i < ITER; ++i) counter.fetch_add(1, order);
    });
    th1.join(); th2.join();
    auto end = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(end - start).count()
              << " ms (counter=" << counter.load() << ")\n";
}

int main() {
    bench("relaxed", std::memory_order_relaxed);
    bench("acq_rel", std::memory_order_acq_rel);
    bench("seq_cst", std::memory_order_seq_cst);
    return 0;
}
```

**预期结果**：`relaxed` 通常比 `seq_cst` 快 **2-5 倍**（取决于 CPU 架构）。

### 实验 2：验证 Release-Acquire 的可见性

```cpp
// workspace/os-lab/concurrency/acquire_release.cpp
#include <atomic>
#include <thread>
#include <cassert>
#include <iostream>

std::atomic<int> data{0};
std::atomic<bool> ready{false};

void producer() {
    data.store(42, std::memory_order_relaxed);
    ready.store(true, std::memory_order_release); // 发布
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) { /* 自旋 */ }
    assert(data.load(std::memory_order_relaxed) == 42); // 一定成立！
    std::cout << "Consumer saw data=" << data.load() << "\n";
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join(); t2.join();
    return 0;
}
```

## 游戏引擎延伸

### 原子引用计数

- 智能指针（如 `std::shared_ptr`）的引用计数更新是 `std::atomic` 的典型场景。
- 使用 `fetch_add(1, memory_order_relaxed)` 增加计数（不需要同步）。
- 使用 `fetch_sub(1, memory_order_acq_rel)` 减少计数：release 保证之前对对象的访问先完成，acquire 保证若计数归零，析构时能看到所有更新。

### Job System 的任务完成标记

- 主线程提交任务后，用 `std::atomic<int> pending{ N };` 跟踪未完成数。
- 每个工作线程完成后 `pending.fetch_sub(1, std::memory_order_release)`。
- 主线程 `while(pending.load(std::memory_order_acquire) > 0);` 等待，保证看到所有任务副作用。
