# 原子操作与内存序

---

## Why：为什么需要理解内存序？

- 编译器和 CPU 都会**重排序**指令，以提升单线程性能。
- 但多线程程序中，重排序可能破坏程序逻辑（如先看到 flag=true 却看到数据未初始化）。
- `std::atomic` + Memory Order 是 C++ 提供给程序员的「可见性契约」。

---

## What：核心概念

### 原子操作

- `std::atomic<T>` 保证对 T 的读/写/RMW（Read-Modify-Write）是不可分割的。
- 常见操作：`load`, `store`, `exchange`, `compare_exchange_weak/strong`, `fetch_add` 等。

### 内存序（Memory Order）

| 顺序 | 含义 | 使用场景 |
|------|------|----------|
| `memory_order_relaxed` | 无同步或顺序保证，仅原子性 | 纯计数器、标志位 |
| `memory_order_acquire` | 本操作之后的读写不能被重排序到本操作之前 | 读锁/标志 |
| `memory_order_release` | 本操作之前的读写不能被重排序到本操作之后 | 写锁/标志 |
| `memory_order_acq_rel` | 同时具有 acquire 和 release 语义 | RMW 操作 |
| `memory_order_seq_cst` | 全局顺序一致性，最强也最慢 | 默认，简单但性能差 |

### Happens-Before

- 如果操作 A **happens-before** 操作 B，则 A 的效果对 B 可见。
- **Release-Acquire 对**：线程 A `release` 写 flag，线程 B `acquire` 读 flag，则 A 在写 flag 之前的所有操作对 B 可见。

```cpp
// 典型模式：数据发布
std::atomic<int*> data{nullptr};

// 生产者
int* p = new int(42);
data.store(p, std::memory_order_release); // 发布

// 消费者
int* p = data.load(std::memory_order_acquire); // 获取
if (p) std::cout << *p; // 保证看到 42
```

---

## How：如何正确使用？

1. **默认用 `seq_cst`**，直到性能分析证明是瓶颈。
2. **生产者-消费者用 `release`/`acquire`**：发布数据用 release，消费用 acquire。
3. **纯计数器用 `relaxed`**：如引用计数、统计计数，不用于同步。
4. **绝不混合同一变量上的不同顺序**：除非你是 Herb Sutter。

## C++ 本地验证实验

### 实验：不同 Memory Order 的性能差异

```cpp
// workspace/os-lab/concurrency/memory_order_bench.cpp
#include <atomic>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int ITER = 10'000'000;

void bench(const char* name, std::memory_order order) {
    std::atomic<int> counter{0};
    auto t1 = std::chrono::high_resolution_clock::now();
    std::thread t1([&]() {
        for (int i = 0; i < ITER; ++i) counter.fetch_add(1, order);
    });
    std::thread t2([&]() {
        for (int i = 0; i < ITER; ++i) counter.fetch_add(1, order);
    });
    t1.join(); t2.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms (counter=" << counter.load() << ")\n";
}

int main() {
    bench("relaxed", std::memory_order_relaxed);
    bench("acq_rel", std::memory_order_acq_rel);
    bench("seq_cst", std::memory_order_seq_cst);
    return 0;
}
```

**预期结果**：`relaxed` 通常比 `seq_cst` 快 **2-5 倍**（取决于 CPU 架构）。

### 实验 2：验证 Release-Acquire 的可见性

```cpp
// workspace/os-lab/concurrency/acquire_release.cpp
#include <atomic>
#include <thread>
#include <cassert>
#include <iostream>

std::atomic<int> data{0};
std::atomic<bool> ready{false};

void producer() {
    data.store(42, std::memory_order_relaxed);
    ready.store(true, std::memory_order_release); // 发布
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) { /* 自旋 */ }
    assert(data.load(std::memory_order_relaxed) == 42); // 一定成立！
    std::cout << "Consumer saw data=" << data.load() << "\n";
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join(); t2.join();
    return 0;
}
```

## 游戏引擎延伸

### 原子引用计数

- 智能指针（如 `std::shared_ptr`）的引用计数更新是 `std::atomic` 的典型场景。
- 使用 `fetch_add(1, memory_order_relaxed)` 增加计数（不需要同步）。
- 使用 `fetch_sub(1, memory_order_acq_rel)` 减少计数：release 保证之前对对象的访问先完成，acquire 保证若计数归零，析构时能看到所有更新。

### Job System 的任务完成标记

- 主线程提交任务后，用 `std::atomic<int> pending{ N };` 跟踪未完成数。
- 每个工作线程完成后 `pending.fetch_sub(1, std::memory_order_release)`。
- 主线程 `while(pending.load(std::memory_order_acquire) > 0);` 等待，保证看到所有任务副作用。
