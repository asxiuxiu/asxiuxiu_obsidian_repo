# 数据竞争——多线程的第一个敌人

---

## 从一个 bug 开始：为什么计数总是少？

上一篇你学会了创建线程。现在让我们写一个"简单"的程序：两个线程各自把计数器加 100 万次，最后应该是 200 万。

> 记得关闭编译器优化。

```cpp
#include <thread>
#include <iostream>

int counter = 0; // 阻止优化

void increment() {
    for (int i = 0; i < 1'000'000; ++i) {
        ++counter;
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    t1.join();
    t2.join();
    std::cout << "counter = " << counter << " (expected 2000000)\n";
    return 0;
}
```

**编译运行**：
```bash
g++ -std=c++17 race_condition.cpp -o race_condition -pthread
./race_condition
```

**预期结果**：counter 几乎总是**小于** 2000000。典型的结果可能在 130 万到 180 万之间，而且每次运行结果都不一样。

这就是你写的第一个多线程 bug——它不会崩溃，不会报错，只是**静静地给出错误答案**。

---

## 为什么 `++counter` 不是原子操作？

在 C++ 源代码里，`++counter` 是一行。但在 CPU 层面，它被拆成了三步：

```
1. LOAD  counter -> 寄存器 R0    // 从内存读取 counter 的值
2. ADD   R0, 1 -> R0             // 寄存器里加 1
3. STORE R0 -> counter           // 写回内存
```

如果只有一个线程执行这三步，没有问题。但如果两个线程交错执行，就会变成这样：

```
时间 →
线程 A:  LOAD counter (100)  ──────────────────────  ADD (101)  STORE (101)
线程 B:  ───────────────  LOAD counter (100)  ADD (101)  STORE (101)  ─────
                                                             ↑
                                                     两个线程都写回 101
                                                     丢了一次递增！
```

这种"读到同一个旧值，各自修改后写回，导致更新丢失"的问题，就是**数据竞争（Data Race）**。

> **定义**：数据竞争发生在多个线程同时访问同一块内存，且至少有一个是写操作，且没有任何同步机制保证访问顺序时。C++ 标准规定，有数据竞争的程序属于**未定义行为（Undefined Behavior）**——它可能今天运行正常，明天在客户机器上崩溃。

---

## 用代码"放大"数据竞争

上面的例子中，因为循环 100 万次，竞争发生的概率很高，但你仍然看不到"具体发生了什么"。让我们写一个更可控的版本，**强制让两个线程在关键步骤交错**：

```cpp
#include <thread>
#include <iostream>
#include <atomic>

int counter = 0;
std::atomic<bool> go{false};

void worker() {
    while (!go.load()) {}  // 等待发令枪
    
    // 故意把 ++counter 拆成三步，模拟 CPU 实际行为
    int local = counter;      // 步骤 1：读
    std::this_thread::sleep_for(std::chrono::milliseconds(1));  // 模拟"被中断"
    local = local + 1;        // 步骤 2：改
    counter = local;          // 步骤 3：写
}

int main() {
    std::thread t1(worker);
    std::thread t2(worker);
    
    go.store(true);  // 发令枪响，两个线程同时跑
    
    t1.join();
    t2.join();
    std::cout << "counter = " << counter << " (expected 2)\n";
    return 0;
}
```

这个例子中，我们手动把 `++counter` 拆成了"读-改-写"三步，并在中间加了睡眠。结果几乎一定是 `counter = 1`——两个线程都读到了 0，各自加 1 写回，最终只加了 1 次。

> **关键洞察**：真实的 CPU 不会恰好在中间 sleep，但**上下文切换可以在任何两条指令之间发生**。如果你的循环次数足够多，总有一天会发生"读完还没写回就被切换"的情况——这就是数据竞争"偶发性"的本质。

---

## 解决思路：两种路线

知道了问题根源，解决方向有两个：

**路线 1：让 `++counter` 变成"一步完成"**

如果 CPU 能提供一条"读取并加 1"的硬件指令，且这条指令执行期间不允许其他线程插手，问题就解决了。这就是**原子操作（Atomic Operation）**的思路——下一篇笔记会详细讲 `std::atomic`。

**路线 2：让"读-改-写"这段代码变成"同一时间只能一个线程执行"**

即使 `++counter` 分三步，如果我们能保证"一个线程执行这三步期间，其他线程不能进入"，也不会出问题。这就是**锁（Lock）**的思路——把一段代码标记为**临界区（Critical Section）**。

```
// 锁的思路：
加锁();
  int local = counter;   // 步骤 1
  local = local + 1;     // 步骤 2
  counter = local;       // 步骤 3
解锁();
// 保证：一个线程在 1-2-3 期间，其他线程不能执行 1-2-3
```

> **问题链**：我们知道了数据竞争的根源——"读-改-写"三步被线程交错。最直接的解决方法是：让这三步变成"原子"的，或者让这三步成为"临界区"。先来看第一种方案：→ [[std::atomic——让操作不可分割]]
