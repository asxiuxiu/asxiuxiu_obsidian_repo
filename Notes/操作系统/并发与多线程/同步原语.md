# 同步原语

---

## Why：为什么需要同步？

- 多线程共享内存时，**数据竞争（Data Race）**会导致未定义行为。
- 即使是简单的 `++counter`，在汇编层面也是"读-改-写"三步，可能被其他线程打断。
- 同步原语提供互斥访问和执行顺序保证。

---

## What：核心概念

### 互斥锁（Mutex）

- `std::mutex`：操作系统提供的睡眠锁。竞争时线程挂起，不占用 CPU，适合**临界区长或竞争激烈**的场景。
- `std::shared_mutex`（C++17）：读写锁。多读单写，读多写少时性能优异。

### 自旋锁（Spinlock）

- 线程在循环中忙等（`while (locked) ;`），不释放 CPU。
- 适合**临界区极短**（几行代码）且竞争不激烈的场景。
- 实现（C++11）：

```cpp
class Spinlock {
    std::atomic<bool> flag{false};
public:
    void lock() {
        while (flag.exchange(true, std::memory_order_acquire)) {
            // 可选：pause / yield
            #if defined(_MSC_VER)
            _mm_pause();
            #elif defined(__x86_64__) || defined(__i386__)
            __asm__ volatile("pause" ::: "memory");
            #elif defined(__aarch64__)
            __asm__ volatile("yield" ::: "memory");
            #else
            // 非 x86/ARM 架构，回退到线程让出
            std::this_thread::yield();
            #endif
        }
    }
    void unlock() { flag.store(false, std::memory_order_release); }
};
```

### 信号量与条件变量

- **信号量（Semaphore）**：控制同时访问某资源的线程数，如限制并发 IO 数。
- **条件变量（Condition Variable）**：线程等待某个条件成立，避免忙等。经典模式：

```cpp
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

// 等待方
std::unique_lock lk(mtx);
cv.wait(lk, []{ return ready; });

// 通知方
{
    std::lock_guard lk(mtx);
    ready = true;
}
cv.notify_one();
```

---

## How：如何选择？

| 场景 | 推荐原语 | 原因 |
|------|----------|------|
| 临界区短、竞争低 | Spinlock / `std::atomic` | 避免上下文切换 |
| 临界区长、竞争高 | `std::mutex` | 线程睡眠，不浪费 CPU |
| 读多写少 | `std::shared_mutex` | 并行读，提升吞吐 |
| 生产者-消费者 | `std::mutex` + `std::condition_variable` | 阻塞等待，无忙等 |
| 高并发队列 | Lock-free Queue | 见 [[无锁队列]] |

## C++ 本地验证实验

### 实验：Mutex vs Spinlock 性能对比

```cpp
// workspace/os-lab/concurrency/mutex_vs_spinlock.cpp
#include <mutex>
#include <atomic>
#include <thread>
#include <vector>
#include <chrono>
#include <iostream>

constexpr int ITER = 1'000'000;
constexpr int NUM_THREADS = 4;

class Spinlock {
    std::atomic<bool> flag{false};
public:
    void lock() {
        while (flag.exchange(true, std::memory_order_acquire)) {
            #if defined(_MSC_VER)
            _mm_pause();
            #elif defined(__x86_64__) || defined(__i386__)
            __asm__ volatile("pause" ::: "memory");
            #elif defined(__aarch64__)
            __asm__ volatile("yield" ::: "memory");
            #else
            std::this_thread::yield();
            #endif
        }
    }
    void unlock() { flag.store(false, std::memory_order_release); }
};

template<typename Lock>
void bench(const char* name) {
    Lock lk;
    int counter = 0;
    auto t1 = std::chrono::high_resolution_clock::now();
    std::vector<std::thread> threads;
    for (int i = 0; i < NUM_THREADS; ++i) {
        threads.emplace_back([&]() {
            for (int j = 0; j < ITER; ++j) {
                lk.lock();
                ++counter;
                lk.unlock();
            }
        });
    }
    for (auto& t : threads) t.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    std::cout << name << ": "
              << std::chrono::duration<double, std::milli>(t2 - t1).count()
              << " ms (counter=" << counter << ")\n";
}

int main() {
    bench<std::mutex>("Mutex");
    bench<Spinlock>("Spinlock");
    return 0;
}
```

**预期结果**：
- 低竞争（线程少、ITER 小）：Spinlock 可能略快。
- 高竞争（线程多）：Spinlock 因忙等浪费大量 CPU，Mutex 更优。

## 游戏引擎延伸

### 双缓冲与命令队列

- 渲染线程消费命令时，逻辑线程继续生产下一帧命令。
- 使用双缓冲 + 原子指针交换（或 mutex 保护队列头尾），避免每帧锁竞争。
- UE 的 `FRHICommandList` 使用线程本地列表 + 最后合并，避免全局锁。

### 资源加载的读写锁

- 游戏资源（纹理、模型）加载完成后很少修改，大量渲染线程只读。
- 使用 `shared_mutex`：加载线程写，渲染线程读，提升并行度。
