# 条件变量——有消息了叫你

---

## 从一个问题开始：消费者线程在空队列上浪费 CPU

上一篇我们对比了 Mutex 和 Spinlock。但无论是哪种锁，消费者线程在发现队列为空时都在"干等"：

- **Spinlock 方案**：`while (queue.empty()) {}` —— 疯狂占用 CPU， poll 检查。
- **Mutex 方案**：反复 `lock()` → 检查 → `unlock()` → `sleep(1ms)` → 重试 —— 频繁加锁解锁，要么延迟高，要么 CPU 浪费。

更好的方案应该是：消费者发现没任务，说"我先睡了，有任务了叫我"，然后**真正释放 CPU**。生产者放完任务后，喊一声"有任务了！"，操作系统把消费者唤醒。

这就是**条件变量（Condition Variable）**要做的事。

---

## 条件变量：不要站着干等，有消息了叫你

想象你去餐厅等外卖。最傻的办法是每分钟跑到前台问"好了吗？"（Spinlock）。稍微好点的办法是每 5 分钟去问一次，中间坐着玩手机（Mutex + sleep）。

最好的办法是：你坐下跟服务员说"我的餐好了叫我"，然后**真的睡着**。餐好了服务员拍醒你，你去取餐。

条件变量就是这个"服务员"。它有两个核心操作：

- **`wait(lock, predicate)`**：原子地"释放锁 + 进入睡眠"。被唤醒后重新加锁，并检查条件是否满足。
- **`notify_one()` / `notify_all()`**：喊醒一个或所有等待的线程。

### 基本用法：生产者-消费者

```cpp
#include <mutex>
#include <condition_variable>
#include <queue>
#include <thread>
#include <iostream>

std::mutex mtx;
std::condition_variable cv;
std::queue<int> task_queue;
bool done = false;

void producer() {
    for (int i = 0; i < 10; ++i) {
        {
            std::lock_guard<std::mutex> lock(mtx);
            task_queue.push(i);
            std::cout << "Produced: " << i << "\n";
        }
        cv.notify_one();  // 喊醒一个消费者
    }
    
    {
        std::lock_guard<std::mutex> lock(mtx);
        done = true;
    }
    cv.notify_all();  // 喊醒所有消费者，告诉它们结束了
}

void consumer(int id) {
    while (true) {
        std::unique_lock<std::mutex> lock(mtx);
        
        // wait 会自动：1. 释放锁  2. 睡眠  3. 被唤醒后重新加锁
        // 第二个参数是"退出条件"——防止虚假唤醒
        cv.wait(lock, []() {
            return !task_queue.empty() || done;
        });
        
        if (!task_queue.empty()) {
            int task = task_queue.front();
            task_queue.pop();
            lock.unlock();  // 尽早释放锁，减少临界区
            
            std::cout << "Consumer " << id << " processed: " << task << "\n";
        } else if (done) {
            break;  // 生产者说结束了，退出
        }
    }
}

int main() {
    std::thread p(producer);
    std::thread c1(consumer, 1);
    std::thread c2(consumer, 2);
    
    p.join();
    c1.join();
    c2.join();
    return 0;
}
```

**编译运行**：
```bash
g++ -std=c++17 condition_variable.cpp -o condition_variable -pthread
./condition_variable
```

## 为什么 `wait()` 必须带条件？

你可能注意到上面的代码用的是 `cv.wait(lock, predicate)` 而不是简单的 `cv.wait(lock)`。这是因为**虚假唤醒（Spurious Wakeup）**。

操作系统可能在没有 `notify` 的情况下唤醒等待线程（比如收到信号）。如果你醒来后直接往下执行，可能发现队列还是空的。

`wait(lock, predicate)` 的等价实现：

```cpp
while (!predicate()) {  // 醒来后检查条件
    cv.wait(lock);       // 如果不满足，继续睡
}
```

> **原则**：永远用 `cv.wait(lock, predicate)`，不要用裸 `cv.wait(lock)`。

---

## `notify_one` vs `notify_all`

| 函数 | 作用 | 适用场景 |
|------|------|---------|
| `notify_one()` | 喊醒一个等待线程 | 只有一个任务，一个消费者处理就够了 |
| `notify_all()` | 喊醒所有等待线程 | 所有消费者都应该检查新状态（如"结束"信号） |

> [!warning] 惊群效应
> `notify_all()` 唤醒所有消费者，但它们要排队抢锁。只有一个能先拿到锁取出任务，其他消费者醒来后可能发现队列又空了，继续睡。这是正常的，不是 bug。

## 动手实验

### 实验：条件变量 vs 忙等的 CPU 占用

```cpp
// cv_vs_busywait.cpp
#include <mutex>
#include <condition_variable>
#include <thread>
#include <iostream>
#include <chrono>

// 方案 A：条件变量
std::mutex mtx_a;
std::condition_variable cv_a;
bool ready_a = false;

void consumer_cv() {
    std::unique_lock<std::mutex> lock(mtx_a);
    cv_a.wait(lock, []() { return ready_a; });
    std::cout << "Consumer woke up!\n";
}

// 方案 B：忙等
std::atomic<bool> ready_b{false};

void consumer_busywait() {
    while (!ready_b.load()) {
        // 疯狂自旋！
    }
    std::cout << "Consumer (busy) woke up!\n";
}

int main() {
    // 测试条件变量：消费者立刻睡眠，不占 CPU
    {
        std::thread t(consumer_cv);
        std::this_thread::sleep_for(std::chrono::seconds(2));
        {
            std::lock_guard<std::mutex> lock(mtx_a);
            ready_a = true;
        }
        cv_a.notify_one();
        t.join();
    }
    
    // 测试忙等：消费者疯狂占 CPU
    {
        std::thread t(consumer_busywait);
        std::this_thread::sleep_for(std::chrono::seconds(2));
        ready_b.store(true);
        t.join();
    }
    
    return 0;
}
```

**观察方法**：运行时用 `htop` 或任务管理器观察 CPU 占用。
- 方案 A（条件变量）：消费者睡眠期间 CPU 占用接近 0%。
- 方案 B（忙等）：消费者把 CPU 核心跑满 100%。

## 小结

| 等待方式 | CPU 占用 | 延迟 | 适用场景 |
|---------|---------|------|---------|
| Spinlock（忙等） | 100% | 最低 | 等待时间极短（微秒级） |
| Mutex + sleep | 低 | 高（sleep 间隔） | 不要求实时性 |
| 条件变量 | 0%（睡眠） | 低（唤醒即响应） | 生产者-消费者、事件通知 |
