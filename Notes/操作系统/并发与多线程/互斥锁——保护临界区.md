# 互斥锁——保护临界区

---

## 从一个问题开始：atomic 不够用了

上一篇我们学了 `std::atomic`，它能保证单个变量的读/写/加减是原子的。但现实中的同步问题远比"保护一个变量"复杂。

想象你要实现一个任务队列：主线程往队列里塞任务，工作线程从队列里取任务。这里涉及多少共享状态？
- 队列的头部指针
- 队列的尾部指针
- 队列里元素的计数
- 队列的容量

你不可能用 `std::atomic` 分别保护这四个变量——如果工作线程在读 head 的时候，主线程正在改 tail，中间某个瞬间队列处于"head 已更新、tail 还没更新"的不一致状态。

你需要的是：**把一段代码（而不是一个变量）标记为"同一时间只能有一个人执行"**。这段代码就叫**临界区（Critical Section）**。

---

## Mutex：抢厕所

**Mutex（Mutual Exclusion，互斥锁）**是最直观的同步工具。它的逻辑和抢公共厕所一样：

- 厕所门上有一把锁。进去的人把门锁上，外面的人只能排队等。
- 里面的人出来了，开锁，下一个排队的人进去。
- 如果厕所被占了，外面的人**去旁边坐着歇会儿，轮到你了叫你**（睡眠）。

`std::mutex` 选择的是"去旁边坐着"。当线程竞争失败时，操作系统把它挂起（睡眠），切换到其他线程执行——被挂起的线程不占用 CPU。这非常适合临界区较长、竞争激烈的场景。

### 基本用法

```cpp
#include <mutex>
#include <thread>
#include <iostream>
#include <vector>

std::mutex mtx;      // 创建一把锁
int counter = 0;     // 普通变量，不需要 atomic

void increment() {
    for (int i = 0; i < 1'000'000; ++i) {
        mtx.lock();    // 抢厕所
        ++counter;     // 临界区：同一时间只有一个线程能执行这里
        mtx.unlock();  // 开锁
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    t1.join(); t2.join();
    std::cout << "counter = " << counter << "\n";  // 一定是 2000000
    return 0;
}
```

**编译运行**：
```bash
g++ -std=c++17 mutex_basic.cpp -o mutex_basic -pthread
./mutex_basic
```

> [!warning] `lock()` 和 `unlock()` 必须成对出现
> 如果在 `lock()` 之后、`unlock()` 之前抛出了异常，锁永远不会被释放，其他线程永远进不来——这就是**死锁**。现代 C++ 用 **RAII** 来避免这个问题。

---

## RAII：让锁自动释放

`std::lock_guard` 是一个 RAII（资源获取即初始化）包装器：它在构造时加锁，在析构时自动解锁——即使函数中间 return 或抛出异常，析构函数也一定会在栈展开时执行。

```cpp
#include <mutex>

std::mutex mtx;
int counter = 0;

void safe_increment() {
    std::lock_guard<std::mutex> lock(mtx);  // 构造时加锁
    ++counter;                               // 临界区
}  // 函数结束时，lock 自动析构，锁被释放
```

`std::unique_lock` 比 `lock_guard` 更灵活：它支持手动 `unlock()`，也可以和条件变量配合使用（后面会讲）。

```cpp
std::unique_lock<std::mutex> lock(mtx);
++counter;
lock.unlock();  // 提前解锁，减少临界区长度
// 做一些不需要锁保护的事
```

> **原则**：临界区越短越好。锁内只放必须保护的代码，把不需要锁的计算移到外面。

## 动手实验

### 实验 1：用 mutex 保护队列

```cpp
// mutex_queue.cpp
#include <mutex>
#include <queue>
#include <thread>
#include <iostream>

class TaskQueue {
    std::queue<int> q;
    std::mutex mtx;
public:
    void push(int value) {
        std::lock_guard<std::mutex> lock(mtx);
        q.push(value);
    }
    
    bool pop(int& value) {
        std::lock_guard<std::mutex> lock(mtx);
        if (q.empty()) return false;
        value = q.front();
        q.pop();
        return true;
    }
    
    size_t size() {
        std::lock_guard<std::mutex> lock(mtx);
        return q.size();
    }
};

TaskQueue queue;

void producer() {
    for (int i = 0; i < 1000; ++i) {
        queue.push(i);
    }
}

void consumer() {
    int value;
    int count = 0;
    while (count < 1000) {
        if (queue.pop(value)) {
            ++count;
        }
    }
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join(); t2.join();
    std::cout << "Queue size = " << queue.size() << "\n";  // 应该是 0
    return 0;
}
```

### 实验 2：死锁演示

```cpp
// deadlock_demo.cpp
#include <mutex>
#include <thread>
#include <iostream>

std::mutex mtx1;
std::mutex mtx2;

void thread_a() {
    std::lock_guard<std::mutex> lock1(mtx1);
    std::this_thread::sleep_for(std::chrono::milliseconds(10));
    std::lock_guard<std::mutex> lock2(mtx2);  // 等 mtx2，但 mtx2 被 B 拿着
    std::cout << "Thread A got both locks\n";
}

void thread_b() {
    std::lock_guard<std::mutex> lock2(mtx2);
    std::this_thread::sleep_for(std::chrono::milliseconds(10));
    std::lock_guard<std::mutex> lock1(mtx1);  // 等 mtx1，但 mtx1 被 A 拿着
    std::cout << "Thread B got both locks\n";
}

int main() {
    std::thread t1(thread_a);
    std::thread t2(thread_b);
    t1.join();  // 永远不会返回！
    t2.join();
    return 0;
}
```

**这个程序会永远卡住**。线程 A 拿着 mtx1 等 mtx2，线程 B 拿着 mtx2 等 mtx1——这就是**死锁（Deadlock）**。

**解决死锁的方法**：
1. **统一加锁顺序**：所有线程都先锁 mtx1，再锁 mtx2。
2. **同时获取多把锁**：`std::lock(mtx1, mtx2)` 会自动处理顺序，避免死锁。

```cpp
// 安全的写法
void safe_thread() {
    std::lock(mtx1, mtx2);  // 同时获取，避免死锁
    std::lock_guard<std::mutex> lock1(mtx1, std::adopt_lock);
    std::lock_guard<std::mutex> lock2(mtx2, std::adopt_lock);
    // 安全地操作...
}
```

## Mutex 的代价：上下文切换

Mutex 不是免费的。当一个线程竞争失败时，操作系统要做的事：

1. 把当前线程的状态保存到 TCB
2. 把线程标记为"睡眠"，移到等待队列
3. 调度另一个线程上 CPU
4. 等锁释放后，再把这个线程唤醒，恢复状态

这一套操作就是**上下文切换**，直接开销大约 **1~10 微秒**。如果临界区本身只有几十纳秒（比如就一句 `++counter`），上下文切换的代价反而占大头。

这就像你等电梯——如果电梯马上就来，站在门口等比先去旁边咖啡店再回来更快。如果电梯要很久才来，去坐着等更划算。

## 小结

| 特性 | `std::atomic` | `std::mutex` |
|------|--------------|-------------|
| 保护对象 | 单个变量 | 一段代码（临界区） |
| 竞争失败时 | 硬件重试 | 线程睡眠，上下文切换 |
| 适用场景 | 计数器、标志位 | 队列、复杂数据结构 |
| 临界区长度 | 单条指令 | 任意长度 |
| 开销 | 低（硬件指令） | 高（可能上下文切换） |

> **问题链**：Mutex 在临界区短时效率低（上下文切换太贵）。有没有一种锁，在临界区很短时"站着等"而不是"去睡觉"？→ [[自旋锁——另一种等待方式]]
