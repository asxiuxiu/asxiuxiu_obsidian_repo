# 无锁队列——综合运用

---

## 从一个问题开始：锁不是万能的

你已经学会了用 Mutex 和 Spinlock 保护共享数据。但在某些极致性能场景下，**锁本身就是瓶颈**。

想象你的游戏主线程每帧都要把渲染命令发给渲染线程。如果用 `std::queue` + `std::mutex`：

1. 主线程抢锁
2. 主线程把命令拷贝进队列
3. 主线程释放锁
4. （如果渲染线程正在处理）主线程竞争失败，被操作系统挂起，上下文切换几微秒

对于追求 16.6ms（60FPS）的游戏来说，这几微秒在高压场景下可能是掉帧的最后一根稻草。

**无锁队列**的思路是：两个线程通过原子操作直接协调，不经过操作系统，不睡眠，不抢锁。就像两个人在一个传送带上放东西和取东西，不需要打电话请示对方。

---

## SPSC Ring Buffer：一条环形传送带

游戏引擎中最常用的无锁队列是 **SPSC（Single Producer Single Consumer，单生产者单消费者）环形缓冲区**。为什么？因为场景非常明确：

- 主线程每帧生成渲染命令（唯一生产者）
- 渲染线程读取命令提交 GPU（唯一消费者）
- 不需要多个生产者竞争，也不需要多个消费者竞争

### 设计思路

想象一个环形跑道，跑道上有一圈位置（固定大小的数组）。两个人在跑道上：

- **生产者**拿着一支笔，在跑道上写数据。他有一个自己的位置标记（`head`），写完后把标记往前推。
- **消费者**拿着一块橡皮，擦掉已经读过的数据。他也有自己的位置标记（`tail`），读完后把标记往前推。

规则很简单：
- 生产者写之前，先看一眼前面一个位置是不是被消费者追上了。如果是，说明跑道满了，等一等。
- 消费者读之前，先看一眼自己的位置是不是被生产者甩开了。如果是，说明跑道空了，等一等。

> 关键：**head 和 tail 是唯一的共享状态**，而且它们各自只被一个线程写、被另一个线程读。通过 `release`/`acquire` 保证可见性，不需要锁。

## 代码实现

```cpp
// spsc_queue.hpp
#include <atomic>
#include <vector>
#include <cassert>
#include <new>

template<typename T>
class SPSCQueue {
    struct Slot {
        T data;
        std::atomic<bool> ready{false};
    };
    
    std::vector<Slot> buffer;
    alignas(64) std::atomic<size_t> head{0};   // 生产者写位置
    alignas(64) std::atomic<size_t> tail{0};   // 消费者读位置
    size_t capacity;
    size_t mask;  // 容量为 2^n 时，用位运算代替取模

public:
    explicit SPSCQueue(size_t cap) : capacity(cap) {
        // 容量必须是 2 的幂，这样可以用位运算 & mask 代替慢速的 %
        assert((cap & (cap - 1)) == 0 && "Capacity must be power of 2");
        mask = cap - 1;
        buffer.resize(cap);
    }

    // 生产者调用
    bool enqueue(const T& item) {
        size_t h = head.load(std::memory_order_relaxed);
        size_t next = (h + 1) & mask;
        
        // 检查队列是否满：如果 next 追上了 tail，说明满了
        if (next == tail.load(std::memory_order_acquire)) {
            return false;  // 队列满
        }
        
        // 写入数据
        buffer[h].data = item;
        
        // release 保证：data 写入一定在 ready 标志之前对其他线程可见
        buffer[h].ready.store(true, std::memory_order_release);
        head.store(next, std::memory_order_release);
        return true;
    }

    // 消费者调用
    bool dequeue(T& item) {
        size_t t = tail.load(std::memory_order_relaxed);
        
        // 检查队列是否空
        if (t == head.load(std::memory_order_acquire)) {
            return false;  // 队列空
        }
        
        // acquire 保证：看到 ready==true 后，data 一定已写入
        if (!buffer[t].ready.load(std::memory_order_acquire)) {
            return false;  // 数据还没准备好（理论上不会发生，但防御性编程）
        }
        
        item = buffer[t].data;
        buffer[t].ready.store(false, std::memory_order_release);
        tail.store((t + 1) & mask, std::memory_order_release);
        return true;
    }
};
```

## 逐行解析：我们在哪里用了前面学过的知识？

### 1. `std::atomic` 保护 head 和 tail

```cpp
alignas(64) std::atomic<size_t> head{0};
alignas(64) std::atomic<size_t> tail{0};
```

`head` 只被生产者写、被消费者读；`tail` 只被消费者写、被生产者读。用 `std::atomic` 保证读写是原子的，不会出现"读到一半的旧值、一半的新值"。

`alignas(64)` 是为了避免 **False Sharing**：如果 `head` 和 `tail` 在同一个缓存行（64 字节），生产者和消费者会争用同一个缓存行，导致缓存频繁失效。把它们分开到不同缓存行，两者可以独立工作。

### 2. `release`/`acquire` 保证数据可见性

```cpp
// 生产者
buffer[h].data = item;                              // 写数据
buffer[h].ready.store(true, std::memory_order_release);  // release：data 写入不会重排到这之后
head.store(next, std::memory_order_release);        // release：ready 写入不会重排到这之后

// 消费者
if (!buffer[t].ready.load(std::memory_order_acquire))  // acquire：后面的 data 读取不会重排到这之前
    return false;
item = buffer[t].data;                              // 安全读取
```

这里有两层 Release-Acquire：
- `ready.store(release)` → `ready.load(acquire)`：保证消费者看到数据已准备好
- `head.store(release)` → `head.load(acquire)`：保证消费者看到最新的 head 位置

### 3. 容量为 2 的幂，用位运算代替取模

```cpp
size_t next = (h + 1) & mask;  // mask = capacity - 1
// 等价于 (h + 1) % capacity，但快一个数量级
```

### 4. 忙等策略由调用方决定

```cpp
// 调用方可以选择自旋、yield、或 sleep
while (!q.enqueue(item)) {
    std::this_thread::yield();  // 队列满了，让出 CPU
}
```

队列本身不阻塞，只是把"满/空"状态返回给调用方。调用方根据场景决定等待策略——这正是无锁设计的灵活性。

## ABA 问题：无锁编程的幽灵

SPSC 队列因为只有一个生产者和一个消费者，且数组固定，不存在 ABA 问题。但在更复杂的 MPMC（多生产者多消费者）无锁结构中，ABA 是一个致命陷阱。

**什么是 ABA 问题？**

想象一个 MPMC 场景：线程 A 看到一个指针是 `0x1000`（状态 A），准备用 CAS 更新它。但在 A 操作的过程中：

1. 线程 B 把 `0x1000` 改成了 `0x2000`（状态 B）
2. 线程 B 释放了 `0x1000` 的内存
3. 线程 C 新申请了一块内存，操作系统恰好又把 `0x1000` 分给了它（状态 A'）

现在指针又回到 `0x1000`，但里面的内容完全不同了。线程 A 的 CAS 检查通过，但它以为自己操作的是原来的对象——**use-after-free 立刻发生**。

**解决方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **Tagged Pointer** | 指针 + 序列号打包到 64/128 位，每次操作序列号 +1 | 通用，无额外开销 |
| **Hazard Pointer** | 读线程注册"我正在读"，写线程检查"没人读才能释放" | 读多写少 |
| **Epoch-Based Reclamation** | 延迟释放到"所有可能读过的线程都进入新 Epoch" | 批量释放 |

> 对于 SPSC Ring Buffer，ABA 问题基本不存在（数组固定，指针循环），所以是最安全的无锁入门练习。

## 动手实验

### 实验：SPSC Queue vs Mutex Queue 性能对比

```cpp
// spsc_bench.cpp
#include "spsc_queue.hpp"
#include <queue>
#include <mutex>
#include <thread>
#include <chrono>
#include <iostream>

constexpr int COUNT = 5'000'000;

// 无锁队列
void bench_spsc() {
    SPSCQueue<int> q(1024);
    auto t1 = std::chrono::high_resolution_clock::now();
    
    std::thread producer([&]() {
        for (int i = 0; i < COUNT; ++i) {
            while (!q.enqueue(i)) {
                std::this_thread::yield();
            }
        }
    });
    
    std::thread consumer([&]() {
        int v, received = 0;
        while (received < COUNT) {
            if (q.dequeue(v)) ++received;
        }
    });
    
    producer.join(); consumer.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    auto ms = std::chrono::duration<double, std::milli>(t2 - t1).count();
    std::cout << "SPSC:    " << ms << " ms\n";
}

// Mutex 队列
void bench_mutex() {
    std::queue<int> q;
    std::mutex mtx;
    auto t1 = std::chrono::high_resolution_clock::now();
    
    std::thread producer([&]() {
        for (int i = 0; i < COUNT; ++i) {
            std::lock_guard lk(mtx);
            q.push(i);
        }
    });
    
    std::thread consumer([&]() {
        int received = 0;
        while (received < COUNT) {
            std::lock_guard lk(mtx);
            if (!q.empty()) { q.pop(); ++received; }
        }
    });
    
    producer.join(); consumer.join();
    auto t2 = std::chrono::high_resolution_clock::now();
    auto ms = std::chrono::duration<double, std::milli>(t2 - t1).count();
    std::cout << "Mutex:   " << ms << " ms\n";
}

int main() {
    bench_spsc();
    bench_mutex();
    return 0;
}
```

**预期结果**：SPSC 无锁队列通常比 `std::queue` + `std::mutex` 快 **5-20 倍**。

## 小结：知识串联

这个 SPSC 队列用到了本系列所有核心知识：

| 知识点 | 在哪里用了 |
|--------|-----------|
| `std::atomic` | `head` 和 `tail` 的原子读写 |
| `memory_order_release` | 保证数据写入在指针更新之前可见 |
| `memory_order_acquire` | 保证看到指针更新后，数据一定可读 |
| 数据竞争防范 | 生产者和消费者不同时对同一位置读写 |
| False Sharing 防范 | `alignas(64)` 隔离 head 和 tail |
| 忙等策略 | 调用方决定 `yield` 或自旋 |
