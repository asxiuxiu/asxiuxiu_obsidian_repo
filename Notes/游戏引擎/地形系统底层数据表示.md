> [!abstract] 核心问题
> 地形系统在引擎中的本质是一个**空间高度函数**的离散采样与存储问题。本文从数据结构的视角，剖析不同地形技术的底层表示方式及其固有限制。

## 1. 高度图（Heightmap）：最基础的表示

### 1.1 数据定义

Heightmap 是最经典的地形表示，其数学本质是**单值函数**的离散采样：

$$
y = f(x, z)
$$

其中：
- 定义域：二维网格 $(x, z) \in [0, W] \times [0, H]$
- 值域：高度值 $y \in [y_{min}, y_{max}]$
- 采样点：均匀分布于网格交点

**内存布局**：
```
Heightmap[Width × Height] → 每个元素存储一个标量高度值

示例：1024×1024 地形，16位精度
- 原始数据：1024 × 1024 × 2 bytes = 2 MB
- 对比：同等分辨率网格模型约 12-24 MB（含位置、法线、UV）
```

### 1.2 固有限制

Heightmap 的数学性质决定了其表达能力边界：

| 限制 | 数学原因 | 典型反例 |
|-----|---------|---------|
| **无垂直结构** | 单值函数：每个 $(x,z)$ 对应唯一 $y$ | 悬崖、洞穴、悬空岩石 |
| **无自相交** | 函数不能折叠 | 拱门、隧道、多层建筑 |
| **拓扑固定** | 二维流形，亏格为 0 | 带洞的地面（需特殊处理） |

> [!note] 为什么 Heightmap 如此流行？
> 尽管存在上述限制，Heightmap 仍是工业标准，原因在于：
> 1. **数据密度极低**：仅需 $W \times H$ 个标量
> 2. **LOD 天然支持**：Mipmap 直接提供多精度层级
> 3. **GPU 友好**：纹理采样硬件加速
> 4. **物理碰撞高效**：高度查询为 O(1) 纹理采样

### 1.3 精度问题

高度值的存储格式直接影响地形细节和范围：

| 格式 | 范围 | 精度 | 适用场景 |
|-----|-----|-----|---------|
| **8-bit UNORM** | $[0, 1]$ | $1/256 \approx 0.4\%$ | 移动端、小范围起伏 |
| **16-bit UNORM** | $[0, 1]$ | $1/65536 \approx 0.0015\%$ | 标准选择，平衡精度与内存 |
| **16-bit SNORM** | $[-1, 1]$ | $1/65536$ | 允许负高度（海平面以下） |
| **32-bit Float** | 任意 | 机器精度 | 程序化生成、需要精确计算 |

**精度陷阱**：
```
问题：16-bit 存储 1000m 高度范围时
每单位精度 = 1000m / 65536 ≈ 1.5cm

后果：
- 平缓坡面可能出现阶梯状 artifacts
- 远处地形可能出现带状条纹（banding）
- 碰撞检测可能出现"地面抖动"

解决方案：
- 使用高度偏移（Height Offset）分层存储
- 局部使用更高精度（如岩石细节用单独高度图）
```

## 2. 网格表示（Mesh-based）：直接几何存储

### 2.1 顶点数据结构

与 Heightmap 的隐式表示不同，Mesh-based 地形显式存储每个顶点的完整信息：

```cpp
// 基础顶点格式
struct TerrainVertex {
    float3 position;    // 世界空间位置 (x, y, z)
    float3 normal;      // 顶点法线 (nx, ny, nz)
    float2 uv;          // 纹理坐标 (u, v)
    // 可选属性
    float4 color;       // 顶点色（可用于混合权重）
    float3 tangent;     // 切线空间
    uint   materialID;  // 材质索引
};
```

**索引缓冲布局**：
```cpp
// 规则的网格拓扑，索引呈规律性排列
// 对于 Width × Height 的顶点网格，生成 (Width-1) × (Height-1) × 2 个三角形

示例：4×4 顶点网格 → 3×3×2 = 18 个三角形，54 个索引
索引模式（每行）：
(0, W, 1), (1, W, W+1), (1, W+1, 2), (2, W+1, W+2), ...
```

### 2.2 LOD 网格链

静态 LOD 系统预生成多个精度层级的网格：

```cpp
struct LODLevel {
    int subdivision;           // 细分级别（如 1, 2, 4, 8...）
    VertexBuffer*  vertices;   // 顶点数据
    IndexBuffer*   indices;    // 索引数据
    float switchDistance;      // 切换距离阈值
};

LODChain lodLevels[4] = {
    {1,  vb_high, ib_high,   0.0f},   // 0-50m
    {2,  vb_med,  ib_med,   50.0f},   // 50-150m
    {4,  vb_low,  ib_low,  150.0f},   // 150-400m
    {8,  vb_far,  ib_far,  400.0f}    // 400m+
};
```

**切换时的突变问题（Popping）**：
```
当相机从 49m 移动到 51m 时：
- 49m：使用 LOD0（高模，约 10k 三角形）
- 51m：切换到 LOD1（中模，约 2.5k 三角形）

视觉突变：地形轮廓突然简化，产生明显的"跳变"

缓解方案：
1. 延迟切换（Hysteresis）：设置不同的进入/退出阈值
2. 几何过渡（Geomorphing）：在切换时顶点插值变形
3. 屏幕空间误差（Screen Space Error）：基于像素误差而非距离
```

## 3. 四叉树表示（QuadTree）：层次化空间分割

### 3.1 树节点结构

四叉树将地形空间递归分割，每个节点代表一个地形块：

```cpp
struct QuadTreeNode {
    // 空间信息
    AABB bounds;              // 包围盒
    float minHeight, maxHeight; // 高度范围（用于剔除）
    
    // 树结构
    QuadTreeNode* children[4]; // 子节点（NE, NW, SE, SW）
    QuadTreeNode* parent;      // 父节点
    int depth;                 // 树深度
    
    // 渲染数据
    Mesh* mesh;               // 本节点网格（可能为 null，表示继续细分）
    Texture* heightmap;       // 局部高度图
    Material* material;       // 材质
    
    // LOD 状态
    float desiredLod;         // 当前帧计算的期望 LOD
    bool isLeaf;              // 是否为叶节点
};
```

### 3.2 节点细分判定

四叉树的核心是**自适应细分算法**：

```cpp
float CalculateLOD(QuadTreeNode* node, Vector3 cameraPos, float fov) {
    float distance = Distance(node->bounds.center, cameraPos);
    float nodeSize = node->bounds.size.x;
    
    // 屏幕空间投影大小
    float screenSize = (nodeSize * fov) / distance;
    
    // 转换为 LOD 级别
    return log2(maxScreenSize / screenSize);
}

void Subdivide(QuadTreeNode* node) {
    if (node->depth >= maxDepth) return;
    
    // 创建 4 个子节点
    for (int i = 0; i < 4; i++) {
        node->children[i] = new QuadTreeNode();
        node->children[i]->bounds = CalculateChildBounds(node->bounds, i);
        node->children[i]->depth = node->depth + 1;
        node->children[i]->parent = node;
    }
    
    node->isLeaf = false;
}
```

### 3.3 T-Junction 的数据层面解释

四叉树细分的核心问题是**相邻节点层级不一致导致的拓扑裂缝**：

```
场景：
┌───┬───┐
│ A │ B │  节点 A 和 B 相邻
└───┴───┘

A 细分后：
┌───┬───┐
│ A1│ A2│
├───┼───┤  B 保持原样
│ A3│ A4│
└───┴───┘

边界问题：
- A 的右边界有 3 个顶点（A2底部、A4顶部、中间点）
- B 的左边界只有 2 个顶点（顶部、底部）
- 中间点悬空，形成 T 型连接（T-Junction）

后果：渲染时该点两侧三角形不共享边，产生裂缝
```

**缝合（Stitching）的数据处理**：
```cpp
// 方案 1：生成退化三角形
void GenerateStitchIndices(QuadTreeNode* fine, QuadTreeNode* coarse, Edge edge) {
    // 细边有 N 个顶点，粗边有 N/2 个顶点
    // 将细边的偶数顶点与粗边顶点连接
    // 奇数顶点与其相邻偶数顶点形成退化三角形
    
    for (int i = 0; i < fineVertexCount; i++) {
        if (i % 2 == 0) {
            // 正常连接
            indices.push_back(fineVertex[i]);
            indices.push_back(coarseVertex[i/2]);
        } else {
            // 生成退化三角形（面积为零，但闭合网格）
            indices.push_back(fineVertex[i]);
            indices.push_back(fineVertex[i-1]); // 与前一个点重合
        }
    }
}

// 方案 2：顶点吸附
void SnapVertices(QuadTreeNode* fine, QuadTreeNode* coarse, Edge edge) {
    // 将细边奇数顶点的位置修改为相邻两偶数顶点的中点
    for (int i = 1; i < fineVertexCount; i += 2) {
        Vector3 p0 = GetVertexPosition(fine, i - 1);
        Vector3 p1 = GetVertexPosition(fine, i + 1);
        SetVertexPosition(fine, i, (p0 + p1) * 0.5f);
    }
}
```

## 4. GPU 细分的数据流

### 4.1 Tessellation 管线数据

硬件曲面细分（DX11+）的数据表示涉及多个阶段：

```
CPU → Input Assembler → Vertex Shader → Hull Shader → 
Tessellator → Domain Shader → Pixel Shader

数据流：
1. CPU 提交 Patch（如 4 个控制点的四边形面片）
2. Vertex Shader：变换控制点到世界空间
3. Hull Shader：
   - 计算边缘细分因子（Edge Tessellation Factor）
   - 输出控制点（Hull Points）
4. Tessellator：
   - 硬件根据因子生成细分后的顶点坐标（UV 空间）
   - 不访问任何数据，纯几何生成
5. Domain Shader：
   - 接收 UV 坐标和 Hull Points
   - 从高度图采样计算实际高度
   - 输出世界空间顶点位置
```

**Hull Shader 常量数据**：
```hlsl
// 每个 Patch 的细分配置
struct PatchConstantOutput {
    float edges[4]    : SV_TessFactor;       // 四条边的细分级别
    float inside[2]   : SV_InsideTessFactor; // 内部细分级别
    float lodLevel    : TEXCOORD0;           // 用于 Domain Shader 的 LOD
};

PatchConstantOutput ConstantHS(InputPatch<VertexOutput, 4> patch) {
    PatchConstantOutput output;
    
    // 基于到相机距离计算细分因子
    float dist = distance(patch[0].worldPos, cameraPos);
    float tessFactor = maxTessLevel * (1.0 - saturate(dist / maxDistance));
    
    output.edges[0] = output.edges[1] = 
    output.edges[2] = output.edges[3] = tessFactor;
    output.inside[0] = output.inside[1] = tessFactor;
    
    return output;
}
```

### 4.2 Displacement Mapping 的数据依赖

Domain Shader 需要从高度图采样，这引入了**纹理读取延迟**：

```hlsl
[domain("quad")]
float4 DomainShader(PatchConstantOutput input, 
                    float2 uv : SV_DomainLocation,
                    const OutputPatch<HullOutput, 4> patch) {
    // 双线性插值计算基础位置
    float3 pos = lerp(
        lerp(patch[0].worldPos, patch[1].worldPos, uv.x),
        lerp(patch[2].worldPos, patch[3].worldPos, uv.x),
        uv.y
    );
    
    // 从高度图采样（纹理读取！）
    float height = Heightmap.SampleLevel(LinearSampler, uv, input.lodLevel);
    
    // 位移计算
    pos.y += height * heightScale;
    
    return mul(ViewProjMatrix, float4(pos, 1.0));
}
```

**性能问题**：
```
Tessellation 的隐藏成本：
1. 每帧动态生成大量顶点（可能数百万）
2. 每个顶点都需要纹理采样（随机访问）
3. 高度图可能不在缓存中（Cache Miss）
4. 过度细分会导致三角形小于像素（Overdraw 放大）

优化策略：
- 预计算高度图的 Mipmap，使用适当层级
- 限制最大细分级别（通常 8x8 或 16x16）
- 使用屏幕空间误差控制细分（距离远时减少细分）
```

## 5. 体素表示（Voxel）：体积数据

### 5.1 密度场定义

体素地形使用**有向距离场（Signed Distance Field）**或**密度场**：

```cpp
// 密度场定义
// density > 0：实体（地面内部）
// density < 0：空气（地面外部）
// density = 0：表面（等值面）

struct VoxelTerrain {
    float density[Width][Height][Depth];  // 3D 密度数组
    // 或使用稀疏表示
    HashMap<Vector3Int, float> sparseVoxels;
};
```

### 5.2 Marching Cubes 算法数据流

从体素数据生成网格的经典算法：

```cpp
// 每个体素（立方体）的 8 个角点密度值
corners[8] = {
    density[x][y][z],       density[x+1][y][z],
    density[x][y+1][z],     density[x+1][y+1][z],
    density[x][y][z+1],     density[x+1][y][z+1],
    density[x][y+1][z+1],   density[x+1][y+1][z+1]
};

// 计算索引（8 位，每个角点 1 位）
int cubeIndex = 0;
for (int i = 0; i < 8; i++) {
    if (corners[i] < 0) cubeIndex |= (1 << i);
}

// 查表获取该构型的三角形配置
// 预计算的查找表有 256 种可能的构型
Triangle* triangles = MarchingCubesTable[cubeIndex];

// 顶点位置通过线性插值确定
for (每个需要生成的顶点) {
    float t = corners[i] / (corners[i] - corners[j]); // 插值因子
    Vector3 pos = lerp(cornerPos[i], cornerPos[j], t);
}
```

### 5.3 存储效率问题

体素的核心问题是**内存爆炸**：

```
Heightmap  vs  Voxel（相同 1024×1024 范围，256 米高度）：

Heightmap：
- 数据：1024 × 1024 × 2 bytes = 2 MB
- 表示范围：1024 × 1024 水平，无垂直限制

Voxel（相同分辨率）：
- 假设垂直分辨率也是 256
- 数据：1024 × 256 × 1024 × 1 byte ≈ 256 MB
- 是 Heightmap 的 128 倍！

稀疏优化（Sparse Voxel Octree）：
- 只存储密度变化的区域
- 使用八叉树（Octree）组织
- 实际内存：取决于地形复杂度
- 平坦区域：接近 Heightmap
- 复杂洞穴：可能超过密集存储
```

**稀疏体素表示**：
```cpp
struct SparseVoxelOctree {
    struct Node {
        bool isLeaf;
        union {
            Node* children[8];     // 非叶节点：子节点指针
            float density[8];       // 叶节点：8 个角点的密度值
        };
    };
    
    Node* root;
    float voxelSize;  // 最小体素尺寸
};

// 查询某点的密度（O(log n)）
float QueryDensity(SparseVoxelOctree* tree, Vector3 pos) {
    Node* node = tree->root;
    float size = tree->worldSize;
    
    while (!node->isLeaf) {
        int octant = DetermineOctant(pos, size);
        node = node->children[octant];
        size *= 0.5f;
    }
    
    // 在叶节点内三线性插值
    return TrilinearInterpolate(node->density, pos);
}
```

## 6. 虚拟纹理（Virtual Texture）的数据抽象

### 6.1 虚拟地址空间

虚拟纹理创建了**巨大的虚拟地址空间**，物理上只存储活跃区域：

```cpp
// 虚拟纹理地址（64-bit）
struct VirtualAddress {
    uint32_t pageX;      // 虚拟页 X 坐标
    uint32_t pageY;      // 虚拟页 Y 坐标
    uint32_t mipLevel;   // Mipmap 层级
};

// 虚拟纹理尺寸示例
// 虚拟尺寸：65536 × 65536 texels
// 页大小：128 × 128 texels
// 总页数：512 × 512 = 262,144 pages per mip
// Mip 层级：0-8（共 9 级）
// 如果全开需要：约 349,525,333 pages × 128 × 128 × 4 bytes ≈ 21 TB！
```

### 6.2 页表（Page Table）

页表是虚拟到物理地址的映射结构：

```cpp
// CPU 端页表（用于管理）
struct PageTableEntry {
    bool resident;        // 是否在物理内存中
    PhysicalPageID physID; // 物理页索引
    uint32_t lastUsed;    // LRU 时间戳
    FileOffset fileOffset; // 在磁盘文件中的位置
};

PageTableEntry pageTable[NumMips][NumPagesY][NumPagesX];

// GPU 端页表（用于渲染时查表）
// 通常以纹理形式上传
// RGBA32F：R=物理X, G=物理Y, B=mip层级, A=有效性
Texture2D<uint4> PageTableTexture;
```

**渲染时的地址转换**：
```hlsl
// 在 Pixel Shader 中进行虚拟地址解析
float4 VirtualTextureSample(float2 virtualUV) {
    // 计算所需的 mip 层级
    float mipLevel = CalculateMipLevel(virtualUV, screenSpaceDerivatives);
    int mip = floor(mipLevel);
    
    // 计算虚拟页坐标
    int2 virtualPage = int2(virtualUV * virtualTextureSize / pageSize);
    
    // 查页表
    uint4 pageEntry = PageTableTexture.Load(int3(virtualPage, mip));
    
    if (pageEntry.a == 0) {
        // 页未加载，返回默认颜色或请求加载
        return RequestPageLoad(virtualPage, mip);
    }
    
    // 计算物理 UV
    float2 pageUV = frac(virtualUV * virtualTextureSize / pageSize);
    float2 physicalUV = (float2(pageEntry.rg) + pageUV) * pageSize / physicalTextureSize;
    
    // 采样物理纹理
    return PhysicalTexture.Sample(AnisoSampler, physicalUV);
}
```

### 6.3 Feedback Pass 数据收集

确定需要哪些页的关键步骤：

```cpp
// Feedback Pass：渲染低分辨率 buffer，输出需要的页 ID
struct FeedbackPixel {
    uint pageX : SV_Target0;
    uint pageY : SV_Target1;
    uint mipLevel : SV_Target2;
};

FeedbackPixel FeedbackPS(float2 virtualUV : TEXCOORD) {
    // 计算该像素需要的 mip 层级
    float2 dx = ddx(virtualUV * virtualSize);
    float2 dy = ddy(virtualUV * virtualSize);
    float mip = log2(max(length(dx), length(dy)));
    mip = clamp(mip, 0, maxMipLevel);
    
    int mipInt = floor(mip);
    int2 page = int2(virtualUV * virtualSize / pageSize) >> mipInt;
    
    return FeedbackPixel(page.x, page.y, mipInt);
}

// CPU 读取并处理
void ProcessFeedbackBuffer(Texture2D feedbackBuffer) {
    // 收集所有唯一的页请求
    std::set<PageID> requestedPages;
    
    for (每个像素) {
        PageID page = feedbackBuffer.Load(pixelPos);
        if (!IsResident(page)) {
            requestedPages.insert(page);
        }
    }
    
    // 按优先级排序（如距离屏幕中心、LOD 层级）
    SortByPriority(requestedPages);
    
    // 异步加载
    for (PageID page : requestedPages) {
        if (freePhysicalPages.empty()) {
            EvictLRUPage();
        }
        LoadPageAsync(page);
    }
}
```

## 7. 物理碰撞的数据表示

### 7.1 HeightField 碰撞形状

物理引擎针对地形有专门的碰撞数据结构：

```cpp
// PhysX HeightField 数据结构
struct PxHeightFieldDesc {
    uint32_t nbRows;           // 行数
    uint32_t nbColumns;        // 列数
    
    // 高度数据格式
    PxHeightFieldFormat format; // 16-bit 或 32-bit
    void* samples;              // 高度值数组
    
    // 可选数据
    uint8_t* materialIndices;   // 每三角形的材质索引（用于不同物理属性）
    float thickness;            // 厚度（用于双面碰撞）
    
    // 缩放
    PxVec3 rowScale;           // 行间距
    PxVec3 columnScale;        // 列间距
    PxReal heightScale;        // 高度值缩放
};

// 内存布局（紧凑存储）
// samples 指向的内存：
// [Height0, Material0, Height1, Material1, ...]
// 或交错存储，取决于格式
```

### 7.2 查询优化

HeightField 的核心优势是**O(1) 高度查询**：

```cpp
// 查询某 (x,z) 位置的高度和法线
bool SampleHeightField(const HeightField* hf, float x, float z, 
                       float& outHeight, Vector3& outNormal) {
    // 1. 转换为网格坐标
    float colF = x / hf->columnScale;
    float rowF = z / hf->rowScale;
    
    int col = floor(colF);
    int row = floor(rowF);
    
    // 2. 边界检查
    if (col < 0 || col >= hf->nbColumns - 1 || 
        row < 0 || row >= hf->nbRows - 1) {
        return false; // 超出范围
    }
    
    // 3. 确定所在三角形（通过小数部分）
    float dc = colF - col;
    float dr = rowF - row;
    
    float h00 = GetHeight(hf, row, col);
    float h01 = GetHeight(hf, row, col + 1);
    float h10 = GetHeight(hf, row + 1, col);
    float h11 = GetHeight(hf, row + 1, col + 1);
    
    // 4. 双线性插值（或根据对角线选择三角形进行重心插值）
    if (dc + dr < 1.0f) {
        // 左上三角形
        outHeight = BarycentricInterpolate(h00, h01, h10, dc, dr);
        outNormal = Cross(h01 - h00, h10 - h00);
    } else {
        // 右下三角形
        outHeight = BarycentricInterpolate(h11, h01, h10, 1-dc, 1-dr);
        outNormal = Cross(h01 - h11, h10 - h11);
    }
    
    outNormal.Normalize();
    return true;
}
```

### 7.3 与渲染数据的同步问题

当地形可编辑时，物理数据需要与渲染数据同步：

```cpp
class EditableTerrain {
    // 渲染数据
    Texture2D heightmapTexture;      // GPU 高度图
    VertexBuffer terrainMesh;        // GPU 网格
    
    // 物理数据
    PxHeightField* physicsHeightField; // PhysX 高度场
    
    // 源数据（共享）
    std::vector<uint16_t> heightData;  // CPU 高度值
    
public:
    void ModifyHeight(int x, int z, float newHeight) {
        // 1. 更新源数据
        heightData[z * width + x] = FloatToHeight(newHeight);
        
        // 2. 更新 GPU 纹理（部分更新）
        heightmapTexture.UpdateRegion(x-1, z-1, 3, 3, &heightData);
        
        // 3. 更新物理高度场
        physicsHeightField->ModifySamples(x-1, z-1, 3, 3, &heightData);
        
        // 4. 标记受影响的碰撞体需要重新计算
        physicsWorld->WakeUpObjectsInRegion(x-1, z-1, 3, 3);
        
        // 5. 更新 GPU 网格（如果是预生成网格而非 Tessellation）
        if (!useTessellation) {
            UpdateMeshRegion(x-1, z-1, 3, 3);
        }
    }
};
```

## 8. 浮点精度与坐标空间

### 8.1 大世界的精度问题

当世界坐标很大时，float32 的精度不足：

```
float32 精度分析：
- 尾数位：23 bits
- 有效精度：约 7 位十进制数字

坐标范围 vs 最小精度：
- 0-1:         ~0.0000001  (0.1 微米)
- 1-10:        ~0.000001   (1 微米)
- 10-100:      ~0.00001    (10 微米)
- 100-1000:    ~0.0001     (0.1 毫米)
- 1000-10000:  ~0.001      (1 毫米)  ← 开始出现问题
- 10000+:      ~0.1        (10 厘米) ← 严重抖动！

问题表现：
- 远处地形顶点抖动
- 相机移动时物体"一跳一跳"
- 碰撞检测穿透
- 阴影闪烁
```

### 8.2 Camera-Relative Rendering 的数据变换

解决方案：每帧将世界原点平移到相机位置：

```cpp
void RenderFrame() {
    Vector3 cameraPos = mainCamera.GetPosition();
    
    // 构建相对变换矩阵
    Matrix4x4 worldToRelative = Matrix4x4::Translation(-cameraPos);
    Matrix4x4 viewMatrix = mainCamera.GetViewMatrix();
    
    // 合并：世界 → 相对 → 相机空间
    // 实际上 viewMatrix 已经包含了相机位置，这里需要特殊处理
    
    foreach (RenderObject& obj : renderObjects) {
        // 物体位置相对于相机
        Vector3 relativePos = obj.worldPosition - cameraPos;
        
        // 此时 relativePos 在合理范围内（如 ±1000m），精度足够
        Matrix4x4 modelMatrix = Matrix4x4::Translation(relativePos) * obj.rotation * obj.scale;
        
        // 提交渲染
        renderQueue.Submit(obj.mesh, obj.material, modelMatrix);
    }
    
    // 相机位于原点
    viewMatrix = Matrix4x4::LookAt(Vector3::Zero, mainCamera.GetForward(), mainCamera.GetUp());
    
    // 渲染...
}
```

### 8.3 双精度 CPU + 单精度 GPU

更复杂的方案：CPU 用 double，GPU 用 float：

```cpp
struct RenderObject {
    Vector3d worldPosition;      // CPU：双精度存储
    // GPU 提交时使用：
    // relativePos = worldPosition - cameraPosition（在 CPU 完成）
    // 结果在 float32 有效范围内
};

void UpdateRenderData() {
    Vector3d cameraPosD = mainCamera.GetPositionDouble();
    
    for (auto& obj : renderObjects) {
        // CPU 侧双精度减法
        Vector3d relativeD = obj.worldPosition - cameraPosD;
        
        // 转换到 float（此时数值范围安全）
        obj.gpuData.position = Vector3f(relativeD.x, relativeD.y, relativeD.z);
    }
    
    // 上传到 GPU...
}
```

## 9. 数据表示对比总结

| 表示方式 | 核心数据结构 | 内存占用 | 表达能力 | 主要问题 |
|---------|------------|---------|---------|---------|
| **Heightmap** | 2D 高度数组 | $O(W \times H)$ | 单值函数地形 | 无垂直结构、浮点精度 |
| **Static Mesh LOD** | 顶点缓冲链 | $O(W \times H \times L)$ | 任意静态几何 | 存储冗余、切换突变 |
| **QuadTree** | 层次节点树 | $O(\text{活跃节点})$ | 单值函数 + LOD | T-Junction、树维护开销 |
| **GPU Tessellation** | Patch + 高度图 | $O(W \times H)$（CPU 侧）| 单值函数 + 动态细分 | 过度绘制、纹理带宽 |
| **Voxel (Dense)** | 3D 密度数组 | $O(W \times H \times D)$ | 任意体积几何 | 内存爆炸 |
| **Voxel (Sparse)** | 稀疏八叉树 | $O(\text{表面体素})$ | 任意体积几何 | 遍历开销、碎片化 |
| **Virtual Texture** | 页表 + 物理池 | $O(\text{物理池大小})$ | 超高分辨率材质 | 流式延迟、页表更新 |

## 10. 技术选型决策树

```
地形是否可以表示为单值函数 y = f(x,z)？
├── 是 → 继续
│   ├── 是否需要运行时动态修改？
│   │   ├── 是 → Heightmap + 动态更新
│   │   └── 否 → 继续
│   ├── 规模多大？
│   │   ├── < 1km² → Static Mesh LOD
│   │   ├── 1-10km² → QuadTree LOD
│   │   ├── 10-100km² → GPU Tessellation + Virtual Texture
│   │   └── > 100km² → GPU-Driven + Sparse Virtual Texture
│   └── 目标平台？
│       ├── 移动端 → Heightmap + 手动 LOD
│       └── PC/主机 → GPU Tessellation / Mesh Shader
│
└── 否（需要洞穴、悬崖、悬空）
    ├── 主要是表面细节？
    │   ├── 是 → Heightmap + Displacement + 垂直 Mesh 补丁
    │   └── 否 → 继续
    ├── 地形复杂度？
    │   ├── 中等 → Sparse Voxel Octree + Marching Cubes
    │   └── 高 → Dual Contouring + 自适应体素
    └── 是否需要完全可破坏？
        ├── 是 → 体素 + 实时重网格化
        └── 否 → 预计算 Signed Distance Field
```
