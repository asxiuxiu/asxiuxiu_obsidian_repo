# 构建系统演进：从单文件到游戏引擎

> 这篇笔记从演进视角回答：为什么一个 `build.bat` 需要如此复杂？
> 我们从最简单的 C++ 程序开始，一步步看每个阶段遇到的问题，以及解决方案如何层层叠加，最终形成游戏引擎的构建体系。

---

## 阶段一：单文件时代 —— 直接调用编译器

### 最简单的程序

```cpp
// hello.cpp
#include <iostream>
int main() {
    std::cout << "Hello, World!" << std::endl;
    return 0;
}
```

### 编译命令

```bash
# 所有平台
$ g++ hello.cpp -o hello

# Windows (MSVC)
$ cl hello.cpp /Fe:hello.exe
```

**核心问题**：无
**解决方案**：直接调用编译器

**这个阶段的思维模型**：
```
源码文件 ──→ 编译器 ──→ 可执行文件
```

## 阶段二：多文件项目 —— 手动管理依赖

### 代码结构复杂化

```
my_project/
├── main.cpp
├── utils.cpp
├── utils.h
├── math.cpp
└── math.h
```

### 编译命令

```bash
# 需要分别编译每个 .cpp，然后链接
$ g++ -c main.cpp -o main.o
$ g++ -c utils.cpp -o utils.o
$ g++ -c math.cpp -o math.o
$ g++ main.o utils.o math.o -o my_app
```

### 遇到的问题

| 问题        | 说明                               |
| --------- | -------------------------------- |
| **重复编译**  | 每次改一个文件，所有文件都重新编译                |
| **头文件依赖** | 如果 `math.h` 改了，哪些 `.cpp` 需要重新编译？ |
| **链接顺序**  | 静态库链接有顺序要求，顺序错了会报错               |
| **命令冗长**  | 文件多了之后命令无法维护                     |

**解决方案：Makefile**

```makefile
# 第一次引入"构建系统"的概念
CXX = g++
CXXFLAGS = -std=c++17 -Wall

SRCS = main.cpp utils.cpp math.cpp
OBJS = $(SRCS:.cpp=.o)
TARGET = my_app

$(TARGET): $(OBJS)
    $(CXX) $(OBJS) -o $(TARGET)

%.o: %.cpp
    $(CXX) $(CXXFLAGS) -c $< -o $@

# 关键：自动处理头文件依赖
-include $(SRCS:.cpp=.d)
```

**这个阶段的思维模型**：
```
源码文件 ──┬─→ .o 文件 ──┐
          ├─→ .o 文件 ──┤──→ 链接器 ──→ 可执行文件
          └─→ .o 文件 ──┘
              ↑
         增量编译：只重新编译修改过的文件
```

## 阶段三：库的分发与复用 —— 静态库与动态库

### 新问题：如何复用代码？

多个项目需要相同的工具函数，直接复制代码会导致：
- 重复代码
- 更新困难（一个地方改了，所有项目都要改）

### 解决方案：打包成库

```bash
# 将通用代码打包成静态库
$ ar rcs libutils.a utils.o math.o

# 使用静态库编译
$ g++ main.cpp -L. -lutils -o my_app
```

### 静态库 vs 动态库

| 特性   | 静态库 (.a/.lib) | 动态库 (.so/.dll) |
| ---- | ------------- | -------------- |
| 链接时机 | 编译时复制到可执行文件   | 运行时加载          |
| 文件大小 | 可执行文件大        | 可执行文件小         |
| 更新方式 | 需要重新编译        | 替换 dll 即可      |
| 内存占用 | 每个进程独立        | 多个进程共享         |

### 遇到的构建问题

| 问题        | 说明                                              |
| --------- | ----------------------------------------------- |
| **库路径管理** | `-L` 指定的路径在不同机器上可能不同                            |
| **头文件路径** | 库的头文件放在哪里？`-I` 如何指定？                            |
| **版本兼容**  | 动态库版本不匹配导致运行时崩溃                                 |
| **跨平台**   | Linux 用 `.so`，Windows 用 `.dll`，macOS 用 `.dylib` |

**Makefile 变得更复杂**：
```makefile
# 需要处理不同平台
UNAME := $(shell uname)
ifeq ($(UNAME), Linux)
    LIB_EXT = .so
    LDFLAGS = -shared -fPIC
endif
ifeq ($(UNAME), Windows)
    LIB_EXT = .dll
    LDFLAGS = -shared
endif

# 库的安装路径、查找路径、rpath 设置...
```

## 阶段四：跨平台需求 —— CMake 的引入

### 核心问题

一个项目需要在 Windows (MSVC)、Linux (GCC)、macOS (Clang) 上都能编译。Makefile 是平台相关的，无法跨平台。

### CMake 的解决方案

**"描述你想构建什么，而不是如何构建"**

```cmake
# CMakeLists.txt
cmake_minimum_required(VERSION 3.15)
project(MyApp)

# 描述：我有一个可执行文件，由这些源文件组成
add_executable(my_app
    main.cpp
    utils.cpp
    math.cpp
)

# 描述：我需要 C++17 标准
target_compile_features(my_app PRIVATE cxx_std_17)

# 描述：我需要链接这个库
target_link_libraries(my_app PRIVATE utils_lib)
```

CMake 根据不同平台生成对应的构建文件：
- Windows → Visual Studio 工程 (.sln)
- Linux → Makefile
- macOS → Xcode 工程

**这个阶段的思维模型**：
```
CMakeLists.txt ──→ CMake ──┬─→ Visual Studio 工程 ──→ MSVC
                          ├─→ Makefile ────────────→ GCC
                          └─→ Xcode 工程 ──────────→ Clang
```

### CMake 生成器与默认行为

CMake 通过 **`-G` 生成器选项** 控制输出什么类型的工程文件。如果不指定 `-G`，CMake 会按优先级**自动检测**：

| 检测顺序 | Windows 环境 | 生成的构建系统 |
|---------|-------------|---------------|
| 1 | 安装了 Visual Studio | Visual Studio 工程 (`.sln`) |
| 2 | 安装了 MinGW | MinGW Makefiles |
| 3 | 以上都不是 | 报错，需要手动指定 |

**显式指定生成器**：

```bash
# Windows - Visual Studio（最常用）
cmake -B build -G "Visual Studio 17 2022" -A x64

# Windows - MinGW GCC
cmake -B build -G "MinGW Makefiles"
cmake --build build

# 跨平台 - Ninja（最快，需单独安装）
cmake -B build -G Ninja
cmake --build build
```

**Windows 上使用 GCC 的方式**：

| 方式 | 说明 |
|------|------|
| **MinGW-w64** | 最常用，直接提供 `g++.exe` |
| **MSYS2** | 完整的类 Unix 环境 + GCC |
| **WSL** | Windows 子系统 Linux，原生 GCC |
| **LLVM/Clang** | 兼容 GCC 参数，Windows 官方支持 |

```bash
# MinGW 示例 - 与 Linux 命令完全一致
g++ hello.cpp -o hello.exe
```

## 阶段五：多目标、多配置 —— CMake 进阶

### 新问题：一个项目产出多个产物

游戏引擎需要同时构建：
- 编辑器（可执行文件）
- 运行时库（供游戏使用）
- 多个工具（资源转换器、调试工具等）
- 各种插件（渲染、物理、网络）

### CMake 解决方案

```cmake
# 添加库（供其他项目链接）
add_library(engine_core
    src/core/entity.cpp
    src/core/scene.cpp
    src/core/resource.cpp
)

# 添加可执行文件（编辑器）
add_executable(editor
    editor/main.cpp
    editor/ui.cpp
)
target_link_libraries(editor PRIVATE engine_core)

# 添加另一个可执行文件（资源转换器）
add_executable(asset_converter
    tools/converter/main.cpp
)
target_link_libraries(asset_converter PRIVATE engine_core)

# 添加插件（动态库）
add_library(render_plugin SHARED
    plugins/render/renderer.cpp
)
```

### 多配置构建

```bash
# 不同配置：Debug / Release / Profile
$ cmake -B build -DCMAKE_BUILD_TYPE=Release
$ cmake --build build --config Release
```

**这个阶段的思维模型**：
```
                    ┌─→ editor.exe
                    │
源代码 ──→ CMake ──┼─→ engine_core.lib
                    │
                    ├─→ asset_converter.exe
                    │
                    └─→ render_plugin.dll
```

## 阶段六：依赖管理 —— 第三方库

### 新问题：项目依赖大量第三方库

游戏引擎通常依赖：
- 图形：DirectX, Vulkan, OpenGL
- 物理：PhysX, Havok
- 网络：ENet, RakNet
- 音频：FMOD, Wwise
- 其他：zlib, jsoncpp, fmt, spdlog...

### 遇到的问题

| 问题       | 说明                     |
| -------- | ---------------------- |
| **获取困难** | 每个库下载方式不同，有些需要注册       |
| **编译配置** | 每个库的 CMake 选项不同，需要统一配置 |
| **版本锁定** | 升级一个库版本可能导致兼容性问题       |
| **传递依赖** | 库 A 依赖库 B，库 B 依赖库 C... |
| **平台差异** | 不同平台需要不同的预编译库          |

### 解决方案：包管理器（Conan / vcpkg）

```python
# conanfile.py - 声明依赖
from conan import ConanFile

class EngineRecipe(ConanFile):
    requires = [
        "zlib/1.2.13",
        "fmt/9.1.0",
        "spdlog/1.11.0",
        "jsoncpp/1.9.5",
        # ... 更多依赖
    ]
```

Conan 处理的事情：
1. 从远程仓库下载预编译库（或源码编译）
2. 管理传递依赖
3. 生成 `conan_toolchain.cmake`，让 CMake 能找到库
4. 版本锁定（lockfile）

**思维模型扩展**：
```
conanfile.py ──→ Conan ──┬─→ 下载第三方库
                          ├─→ 处理传递依赖
                          └─→ 生成 toolchain.cmake
                                       ↓
                              CMakeLists.txt ──→ CMake ──→ 构建
```

## 阶段七：超大规模项目 —— 源码组织与模块化

### 游戏引擎的真实规模

```
chaos/                          # 引擎仓库
├── _source/_engine/source/     # 引擎核心（数百个子模块）
├── _source/_plugin/            # 插件系统
├── _source/_tools/             # 工具链
└── ...

wolfgang/_games/proven_ground/  # 游戏项目仓库
├── _source/                    # 游戏代码
└── ...

# 总计可能包含：
# - 数千个 C++ 文件
# - 数百个构建目标（库/可执行文件）
# - 多个代码仓库
# - 多种语言（C++, C#, Python, Shader...）
```

### 新问题

| 问题 | 说明 |
|------|------|
| **源码树组织** | 引擎代码、游戏代码、插件代码如何在 IDE 中组织？ |
| **条件编译** | 编辑器版本需要包含编辑器代码，服务器版本不需要 |
| **多仓库协作** | 引擎和游戏代码在不同仓库，如何一起构建？ |
| **构建时间** | 完整构建可能需要数小时，如何支持增量和并行？ |
| **IDE 工程生成** | 数百个项目在 VS 中如何分组显示？ |

## 阶段八：Source Forest —— 终极解决方案

这就是 `build.bat` 真正要解决的问题。

### 核心思想

**"源码树的动态组装"**

不是静态地写死所有 `add_subdirectory()`，而是通过配置决定：
- 本次构建包含哪些模块？
- 这些模块来自哪个目录？
- 在什么条件下才包含某个模块？

### 两层 JSON 配置

#### 第一层：`project_tree_config.json`

决定本次构建包含哪些**源码仓库**：

```json
{
    "SolutionName": "ChaosCB2",
    "SourceTree": [
        "chaos/_source/cmake_projects.json",           // 引擎核心
        "wolfgang/_games/proven_ground/_source/...",   // 游戏代码
        "chaos/_source/_plugin/...",                   // 插件
        "chaos/_source/_tools/cmake/..."               // 工具链
    ]
}
```

#### 第二层：`cmake_projects.json`

决定每个仓库内的**目录结构和条件编译**：

```json
{
    "Projects": {
        "_engine/source": {
            "1base": "",                    // 空字符串 = 总是包含
            "6client": "@BUILD_CLIENT@",    // CMake 变量控制
            "7editor": "@BUILD_EDITOR@ AND @WIN32@",
            "8server": {
                "1backend": "@BUILD_SERVER@",
                "3game_server": "@BUILD_SERVER@"
            }
        }
    }
}
```

### 为什么需要生成器 (`chaos_launch_generator.exe`)

因为 CMake 的 `add_subdirectory()` 是**静态的**，不能运行时决定。所以引擎采用了一个技巧：

1. **模板阶段**：`launch/templates/` 包含 CMake 模板和 JSON 配置
2. **生成阶段**：`chaos_launch_generator.exe` 根据配置**生成**临时的 `build/sourcetree/` 目录
3. **代理模式**：每个子目录生成一个代理 `CMakeLists.txt`，再 include 真实源码

```cmake
# 生成的代理文件：build/sourcetree/Engine/_engine/source/base/CMakeLists.txt
set(CHAOS_SOURCETREE_SOURCE_CMAKELISTS_PATH
    E:/cb2_master/dev/chaos/_source/_engine/source/base/)
include(${CHAOS_SOURCETREE_SOURCE_CMAKELISTS_PATH}/CMakeLists.txt)
```

这样真实源码目录**不需要知道** sourcetree 的存在，保持干净。

## 从简单到复杂的演进总结

| 阶段 | 触发条件 | 核心问题 | 解决方案 |
|------|---------|---------|----------|
| 单文件 | 1 个 .cpp | 无 | 直接编译 |
| 多文件 | 多个 .cpp | 重复编译、依赖管理 | Makefile |
| 库复用 | 多个项目 | 代码复用、链接 | 静态/动态库 |
| 跨平台 | 多平台需求 | 平台差异 | CMake |
| 多目标 | 库 + 多个可执行文件 | 目标管理 | CMake targets |
| 第三方依赖 | 使用外部库 | 获取、版本、传递依赖 | Conan / vcpkg |
| 超大规模 | 游戏引擎规模 | 源码组织、条件编译、多仓库 | Source Forest |

### 为什么游戏引擎需要这么复杂？

**不是因为单个功能复杂，而是因为问题的叠加：**

1. **多平台** (CMake) + **多配置** (Debug/Release/Profile/ASAN) = 需要灵活的构建配置
2. **多目标** (数百个库/可执行文件) + **条件编译** (编辑器版/服务器版/客户端版) = 需要动态决定构建内容
3. **多仓库** (引擎 + 游戏 + 插件) + **第三方依赖** (数十个库) = 需要统一的源码组织和依赖管理
4. **IDE 友好** (Visual Studio) + **构建效率** (增量编译) = 需要复杂的工程生成策略

`build.bat` 的每一步都对应解决一个层级的复杂性：

| 脚本 | 解决的问题 |
|------|-----------|
| `setup_project_tree.bat` | 配置版本管理 |
| `conan_install.bat` | 第三方依赖获取 |
| `generate_launch_projects.bat` | 源码树动态组装 |
| `build.bat` | 全流程编排 |

## 给新手的建议

### 学习路径

1. **掌握阶段 1-2**：先用 `g++` 手动编译，理解预处理、编译、链接的全过程
2. **理解阶段 3**：学会创建和使用静态库/动态库
3. **实践阶段 4-5**：用 CMake 管理一个多目标项目
4. **了解阶段 6**：使用 Conan 或 vcpkg 管理依赖
5. **研究阶段 7-8**：当你真的需要处理超大规模项目时，再深入研究 Source Forest 模式

### 什么时候需要复杂构建系统？

| 项目规模 | 推荐方案 |
|---------|---------|
| 1-10 个文件 | 直接编译 |
| 10-50 个文件 | Makefile / 简单 CMake |
| 50-200 个文件 | CMake + 多 target |
| 200+ 个文件，多平台 | CMake + 包管理器 |
| 多仓库、数百+ 目标、条件编译 | Source Forest 模式 |
