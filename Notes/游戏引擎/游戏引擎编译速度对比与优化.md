# 游戏引擎编译速度对比与优化

## Why：为什么编译速度很重要？

**编译速度直接影响开发效率：**

| 编译时间 | 每天编译次数 | 日等待时间 | 年等待时间（250工作日） |
|----------|-------------|-----------|----------------------|
| 3秒 | 200次 | 10分钟 | 42小时 |
| 5分钟 | 20次 | 1.7小时 | 416小时（17天） |
| 30分钟 | 5次 | 2.5小时 | 625小时（26天） |

> 在UE项目中，开发者每年可能花费**超过一个月**的时间只是等待编译。

**编译慢的心理影响**：
- 减少尝试意愿（"改这个要编译5分钟，算了不改了"）
- 打断心流状态（context switching 成本）
- 增加bug风险（一次改多行代码，因为不想频繁编译）

## What：各引擎编译速度现状

### 实际数据对比（2024-2025社区反馈）

| 引擎/框架 | 修改一行.cpp | 修改头文件 | 完整编译 | 主要瓶颈 |
|-----------|-------------|-----------|---------|---------|
| **Unity (C#)** | 1-3秒 | N/A | 2-5分钟 | Domain Reload |
| **Unity DOTS** | 2-5秒 | N/A | 5-10分钟 | Burst编译 |
| **Godot (C++)** | 10-30秒 | 1-2分钟 | 10-30分钟 | 模块依赖 |
| **Unreal Engine 4** | 3-8分钟 | 5-15分钟 | 22-26分钟 | UHT + PCH |
| **Unreal Engine 5** | 5-15分钟 | 10-30分钟 | 55-70分钟 | UHT + 模板爆炸 |
| **Flecs (C)** | **<3秒** | **<3秒** | **<10秒** | 无 |
| **EnTT (C++)** | 30秒-5分钟 | 2-10分钟 | 5-20分钟 | 模板实例化 |
| **Bevy (Rust)** | 10-30秒 | 10-30秒 | 3-10分钟 | LLVM优化 |

### 为什么 UE 这么慢？

UE编译慢不是单一原因，而是**多重债务叠加**：

#### 1. Unreal Header Tool (UHT) - 最大元凶

```cpp
// 你的头文件
UCLASS()
class MYGAME_API AMyCharacter : public ACharacter {
    GENERATED_BODY()  // ← 这行触发UHT
    UFUNCTION(BlueprintCallable)
    void Jump();
};
```

**UHT的工作流程**：
```
修改 MyCharacter.h
    ↓
UHT 扫描所有 .h 文件（非增量！）
    ↓
生成 MyCharacter.generated.h（包含反射元数据）
    ↓
该文件被数百个 .cpp 包含
    ↓
所有包含者重编译
```

**问题根源**：
- UHT 是**外部进程**，在编译前运行
- 某些模式（如 `GenerateClangDatabase`）默认**全量重新生成**
- 即使只改一个数字，也会重新扫描所有 `UCLASS`/`UFUNCTION`

#### 2. 头文件级联依赖（Header Cascades）

```cpp
// ACharacter.h 包含链（简化）
#include "GameFramework/Pawn.h"
    #include "GameFramework/Actor.h"
        #include "Engine/EngineBaseTypes.h"
            #include "UObject/Object.h"
                #include "UObject/ObjectMacros.h"
                    // ... 数百个头文件

// 结论：包含 ACharacter.h = 包含大半个引擎
```

**PCH（预编译头）陷阱**：
- PCH 本意是加速完整编译
- 但 PCH 包含的头文件一旦变化（如 Engine.h），**整个项目重编译**
- UE的PCH管理混乱，频繁触发全量重编

#### 3. UE5 模板爆炸

UE5为了支持**双精度坐标**，将核心数学类型全部模板化：

```cpp
// UE4: 具体类型
struct FVector { float X, Y, Z; };

// UE5: 模板类型
template<typename T>
struct TVector { T X, Y, Z; };
using FVector = TVector<float>;
using FVector3d = TVector<double>;
```

**后果**：
- 每个使用 `FVector` 的地方变成模板实例化点
- 修改模板定义 → 所有实例化点重编译
- 编译单元膨胀 2-3 倍

## How：编译优化策略

### 1. 架构层面：ECS的编译优势

**为什么ECS能改善编译速度？**

#### 传统OOP的编译耦合
```cpp
// 修改前
class Character {
public:
    void Jump();  // 虚函数
};

// 修改后 - 添加参数
class Character {
public:
    void Jump(float height);  // 签名变化
};
```

**连锁反应**：
1. `Character.h` 变化
2. `Player.h` 包含 `Character.h` → 重编译
3. `Enemy.h` 包含 `Character.h` → 重编译
4. `GameMode.cpp` 包含 `Player.h` → 重编译
5. ... 级联数十个文件

#### ECS的编译隔离
```cpp
// Component：纯数据， rarely changes
struct Position { float x, y, z; };

// System：纯函数，无继承、无接口
void MovementSystem(World& w) {
    w.query<Position, Velocity>().each([](auto& p, auto& v) {
        p.value += v.value * dt;
    });
}
```

**隔离性体现**：
| 修改位置 | OOP影响 | ECS影响 |
|----------|---------|---------|
| Component 添加字段 | 所有包含该头文件的文件 | 仅使用该 Component 的 System |
| System 实现 | 派生类、使用者全部重编 | **仅该 System 文件** |
| System 添加 | 无 | **仅新 System 文件** |
| System 删除 | 使用者重编 | **仅该 System 文件** |

### 2. C++工程技巧

#### PImpl 模式（Pointer to Implementation）

```cpp
// MySystem.h - 稳定接口
class MySystem {
public:
    void Update(World& w);
    
private:
    class Impl;           // 前向声明
    std::unique_ptr<Impl> pImpl;  // 隐藏实现
};

// MySystem.cpp - 实现细节
class MySystem::Impl {
public:
    std::vector<Entity> cache;  // 频繁修改的数据
    void InternalUpdate() { ... }
};

void MySystem::Update(World& w) {
    pImpl->InternalUpdate();
}
```

**收益**：修改 `Impl` 不影响任何包含 `MySystem.h` 的文件。

#### 前向声明

```cpp
// ❌ 坏：包含整个头文件
#include "GameFramework/Character.h"
class AMyPlayer : public ACharacter { ... };

// ✅ 好：前向声明 + 指针
class ACharacter;  // 只声明，不定义
class AMyPlayer {
    ACharacter* character;  // 指针/引用可以前向声明
};
```

#### 显式模板实例化

```cpp
// Template.h - 声明
#pragma once
template<typename T>
class Container {
    void Add(T item);
};

// Template.cpp - 定义 + 显式实例化
#include "Template.h"
template<typename T>
void Container<T>::Add(T item) { ... }

// 显式实例化常用类型
template class Container<int>;
template class Container<float>;
template class Container<Entity>;
```

**收益**：其他文件包含 `Template.h` 时，不需要实例化模板，编译器直接使用 `.cpp` 中已编译的版本。

### 3. 引擎特定优化

#### Unreal Engine

```csharp
// Build.cs 配置
public class MyGame : ModuleRules {
    public MyGame(ReadOnlyTargetRules Target) : base(Target) {
        // 禁用Unity Build（小项目）
        bUseUnityBuild = false;
        
        // 最小化PCH使用
        MinFilesUsingPrecompiledHeader = 1;
        
        // 增加并行编译
        BuildConfiguration.ProcessorCountMultiplier = 2;
        
        // 禁用非必要插件
        bEnablePlugins = false;
    }
}
```

**硬件建议**：
- **SSD**：将UE源码、项目、IDE全部放在NVMe SSD上
- **内存**：64GB+（减少链接时的磁盘交换）
- **CPU**：高主频 > 多核（UE编译更吃单核性能）

#### Unity

```csharp
// 减少Domain Reload时间
[InitializeOnLoad]
public static class DomainReloadOptimizer {
    static DomainReloadOptimizer() {
        // 禁用PlayMode下的Domain Reload
        EditorSettings.enterPlayModeOptions = 
            EnterPlayModeOptions.DisableDomainReload;
    }
}
```

## 各ECS框架编译速度对比详解

| 框架 | 实现方式 | 编译模型 | 修改System代价 | 模板使用 |
|------|---------|---------|---------------|---------|
| **Flecs** | C99 | 非模板，纯函数 | **<3秒** | 无 |
| **EnTT** | C++17 | 重度模板 | 30秒-5分钟 | `registry.view<A,B,C>()` |
| **gaia-ecs** | C++20 | 可选模板 | 10-60秒 | 可控 |
| **UE Mass** | C++17 | 重度模板 | 5-15分钟 | 查询系统 |

### 为什么 EnTT 比 Flecs 编译慢？

**EnTT的模板实现**：
```cpp
// 每次不同的组件组合都生成新模板实例
auto view = registry.view<Position, Velocity>();
view.each([](auto& pos, auto& vel) { ... });  // 实例化1

auto view2 = registry.view<Position, Health>();
view2.each([](auto& pos, auto& hp) { ... });  // 实例化2

auto view3 = registry.view<Position, Velocity, Health>();
view3.each([](auto& pos, auto& vel, auto& hp) { ... });  // 实例化3
```

**后果**：
- 模板代码在头文件中 → 修改头文件导致所有使用者重编
- 模板实例化是编译期计算 → 增加编译时间
- 链接时去重 → 增加链接时间

**Flecs的非模板实现**：
```cpp
// 纯C函数，运行时查询
ecs_query_t* q = ecs_query_new(world, "Position, Velocity");
ecs_iter_t it = ecs_query_iter(world, q);
while (ecs_query_next(&it)) {
    Position* p = ecs_field(&it, Position, 1);
    // ...
}
```

**优势**：
- 无模板实例化开销
- 查询字符串在运行时解析（一次）
- 编译器只需编译一次 `ecs_query_iter` 函数

## 决策建议

| 你的情况 | 推荐方案 |
|----------|----------|
| 编译速度 > 一切 | Flecs + C |
| 需要C++生态 + 编译速度 | Flecs C++ wrapper |
| 需要C++ + 极致运行时性能 | EnTT（架构稳定后引入） |
| 需要完整引擎 + 可接受编译慢 | Unity DOTS |
| 在UE项目中 | 使用传统Actor，接受编译慢，或用第三方ECS插件 |
