## Why：为什么游戏引擎需要编辑器框架？

### 没有编辑器框架的世界有多糟

想象一下：你的 ECS 核心运行时已经闭环，渲染管线也能在屏幕上画出一个带纹理的立方体。但如果你想把这个立方体从 `(0, 0, 0)` 移动到 `(5, 0, 0)`，唯一的方式是修改 C++ 源码、重新编译、再次运行。调整相机视角？改代码。给实体换一个材质？改代码。测试一个物理碰撞参数？改代码、编译、等待 30 秒。

没有编辑器框架，引擎只是一个**黑箱运行时**。开发者（无论是人还是 AI）无法直接观察世界状态，也无法实时干预。

### 三个真实场景

1. **关卡搭建**：美术需要在一个场景中摆放 300 个道具、调整 50 盏灯光的参数、设置 Trigger 区域。没有编辑器，这些工作将完全依赖手写配置文件或脚本，迭代效率归零。
2. **参数调优**：物理系统的重力、碰撞体的摩擦系数、动画的 Blend 权重——这些参数的最佳值无法在设计阶段确定，必须在运行时反复微调并即时观察效果。
3. **AI 协作开发**：Agent 需要读取当前场景中所有实体的组件状态，修改某个组件的字段值，然后推进一帧验证结果。没有编辑器框架提供的可观测性和可操控性，Agent 将无从下手。

### 这是"必须"还是"优化"

编辑器框架是**必须**。一个没有编辑器的引擎可以运行游戏，但无法**开发**游戏。它相当于工厂有了机床却没有操作面板—— technically 能工作， practically 不可用。

## What：最简化版本的编辑器框架长什么样？

一个最小可运行的编辑器框架只需要四样东西：
- **场景树（Hierarchy）**：列出当前世界中的所有实体及其层级关系
- **Inspector**：显示选中实体的所有组件及其字段值，并允许编辑
- **视口（Viewport）**：渲染世界的相机视角，接收鼠标/键盘输入
- **Gizmo**：在视口中可视化地操控实体的位置、旋转、缩放

下面是一个约 150 行的最小实现骨架。它基于我们已经搭建好的 ECS + 反射系统。

```cpp
// ============================================================
// 阶段 5 最小编辑器骨架 —— 基于 ECS + 反射
// ============================================================
#include <vector>
#include <string>
#include <functional>
#include <iostream>

// 前置：假设这些类型已在 [[从零开始的引擎骨架]] 和 [[反射系统]] 中定义
struct Entity { uint32_t id; };
struct World;
struct TypeRegistry;
struct ComponentDesc;

// ------------------------------------------------------------
// 1. 场景树：简单的实体列表 + 父子关系
// ------------------------------------------------------------
struct HierarchyPanel {
    World* world;

    void drawEntityNode(Entity e, int depth = 0);
    void onDraw();
};

// ------------------------------------------------------------
// 2. Inspector：通过反射读取和写入组件字段
// ------------------------------------------------------------
struct InspectorPanel {
    Entity selectedEntity;
    TypeRegistry* registry;

    void onDraw() {
        if (selectedEntity.id == 0) return;
        std::cout << "=== Inspector: Entity " << selectedEntity.id << " ===\n";

        // 遍历该实体的所有组件类型
        for (auto* compDesc : registry->getEntityComponents(selectedEntity)) {
            std::cout << "Component: " << compDesc->name << "\n";
            void* instance = compDesc->getInstance(world, selectedEntity);

            // 遍历组件的所有字段
            for (auto& field : compDesc->fields) {
                std::cout << "  " << field.name << " = ";
                if (field.type == FieldType::Float) {
                    float* ptr = (float*)field.getPtr(instance);
                    std::cout << *ptr << "\n";
                    // 在实际编辑器中，这里会渲染一个 float input
                } else if (field.type == FieldType::Vector3) {
                    Vector3* ptr = (Vector3*)field.getPtr(instance);
                    std::cout << "(" << ptr->x << ", " << ptr->y << ", " << ptr->z << ")\n";
                }
            }
        }
    }

    void setFieldFloat(const std::string& componentName,
                       const std::string& fieldName,
                       float value) {
        auto* desc = registry->findComponent(componentName);
        void* instance = desc->getInstance(world, selectedEntity);
        for (auto& field : desc->fields) {
            if (field.name == fieldName && field.type == FieldType::Float) {
                *(float*)field.getPtr(instance) = value;
                desc->markDirty(world, selectedEntity); // 通知运行时更新
                return;
            }
        }
    }
};

// ------------------------------------------------------------
// 3. 视口：绑定相机 + 接收输入
// ------------------------------------------------------------
struct Viewport {
    uint32_t width = 1280, height = 720;
    Entity   cameraEntity; // 视口使用的相机实体

    void onInput(float mouseX, float mouseY, bool leftButtonDown);
    void onRender(World* world); // 调用渲染管线画一帧
};

// ------------------------------------------------------------
// 4. Gizmo：最小平移操控器
// ------------------------------------------------------------
struct Gizmo {
    enum class Mode { Translate, Rotate, Scale };
    Mode mode = Mode::Translate;
    Entity targetEntity;

    // 射线与 X/Y/Z 轴的交互逻辑（简化版）
    void onMouseDrag(const Ray& mouseRay, const Vector3& dragDelta) {
        if (targetEntity.id == 0) return;
        auto* transform = world->getComponent<TransformComponent>(targetEntity);
        if (!transform) return;

        switch (mode) {
            case Mode::Translate:
                transform->position = transform->position + dragDelta;
                break;
            case Mode::Rotate:
                // 简化：绕 Y 轴旋转
                transform->rotation = Quaternion::fromAxisAngle({0,1,0}, dragDelta.x * 0.01f) * transform->rotation;
                break;
            case Mode::Scale:
                transform->scale = transform->scale + dragDelta * 0.1f;
                break;
        }
        world->markComponentDirty<TransformComponent>(targetEntity);
    }
};

// ------------------------------------------------------------
// 5. 最小编辑器核心：把面板组合起来
// ------------------------------------------------------------
struct Editor {
    World* world;
    HierarchyPanel hierarchy;
    InspectorPanel inspector;
    Viewport       viewport;
    Gizmo          gizmo;

    void onTick(float dt) {
        // 1. 处理输入（选择、Gizmo 拖拽、相机漫游）
        // 2. 更新场景树和 Inspector UI
        hierarchy.onDraw();
        inspector.onDraw();
        // 3. 渲染视口
        viewport.onRender(world);
    }
};
```

### 这个最小实现已经解决了什么

- **可观测性**：通过 `TypeRegistry`，外部无需读 C++ 头文件即可知道每个实体有什么组件、每个组件有什么字段。
- **可操控性**：Inspector 可以直接修改组件字段；Gizmo 可以在 3D 空间中直观地移动、旋转、缩放实体。
- **即时反馈**：修改字段后会调用 `markDirty`，触发相应的 System（如 `TransformHierarchySystem`）在下一帧更新世界。

### 还缺什么

- **Undo/Redo**：改错了位置无法回退。
- **编辑态/运行态隔离**：直接修改运行中的世界会导致状态混乱，也无法保存编辑结果。
- **多视口/布局**：只能从一个固定相机观察场景。
- **AI 标准化接口**：人类可以点 UI，但 AI 需要结构化的 `query` / `mutate` / `step` 协议（见 [[MCP与Agent桥接层]]）。

## How：真实引擎的编辑器框架是如何一步一步复杂起来的？

### 阶段 1：最小实现 → 能用（解决直接操作的问题）

**触发原因**：开发者需要不重新编译就能调整场景。核心诉求是"看到什么改什么"。

**代码层面的变化**：

1. **反射驱动的 Inspector 自动生成**
   - 从手写每个组件的 UI 面板，升级为基于 `ComponentDesc` 的自动生成。
   - 基础类型（`int`, `float`, `bool`, `Vector3`, `Color`）有对应的通用 UI 控件。
   - 枚举类型通过反射的元数据自动渲染为下拉框。

2. **场景树的层级展开**
   - `HierarchyPanel` 不再只是平铺列表，而是根据 `ParentComponent` 递归渲染为树形结构。
   - 支持拖拽改变父子关系，触发 `TransformHierarchySystem` 重新计算世界矩阵。

3. **Gizmo 的射线交互精确化**
   - 从简单的屏幕空间偏移，升级为射线与平面的求交。
   - 平移模式：根据选中的轴（X/Y/Z/XY/YZ/XZ/中心球），构造不同的约束平面，计算鼠标射线与该平面的交点，得到精确的 world-space delta。
   - 旋转模式：计算鼠标射线与以 Gizmo 中心为原点的坐标平面的交点，通过两帧交点之间的夹角得到旋转量。
   - 缩放模式：将鼠标位移投影到各轴在屏幕空间的投影方向，得到各轴缩放比例。

4. **Picking（对象选择）**
   - 单选：从视口相机发出射线，与场景中的 Mesh 做相交测试（或利用 GPU Picking Buffer），返回选中的 `Entity`。
   - 框选：定义屏幕空间的矩形区域，利用视锥体或 GPU 颜色编码批量筛选可见实体。

**阶段 1 验收标准**：
- [x] 能在 Inspector 中修改任意组件的可见字段
- [x] 能在视口中用 Gizmo 平移/旋转/缩放选中实体
- [x] 能通过点击或框选改变当前选中实体

---

### 阶段 2：能用 → 好用（解决迭代效率和调试问题）

**触发原因**：直接修改运行时的 ECS 世界会导致两个问题：（1）改错了无法撤销；（2）测试游戏逻辑时，编辑器的存在会干扰运行态。需要引入**事务边界**和**世界隔离**。

**代码层面的变化**：

1. **命令模式（Command Pattern）与 Undo/Redo**
   - 不再直接修改组件字段，而是将每次修改包装为一个 `ICommand`：
     ```cpp
     struct ICommand {
         virtual void execute() = 0;
         virtual void undo() = 0;
     };
     struct SetComponentFieldCommand : ICommand {
         Entity entity;
         ComponentID comp;
         FieldID field;
         void* oldValue;
         void* newValue;
         void execute() override { /* set newValue */ }
         void undo() override { /* restore oldValue */ }
     };
     ```
   - `CommandStack` 维护 `undoStack` 和 `redoStack`，支持 `Ctrl+Z` / `Ctrl+Y`。
   - **与 ECS 的融合**：Command 的 `execute()` 实际上是在向世界的 `CommandBuffer` 提交变更，确保所有编辑操作在正确的系统边界内应用。这与 [[系统调度与确定性]] 中的命令缓冲一脉相承。

2. **编辑态（Edit World）与运行态（Play World）隔离**
   - 这是 PIE（Play In Editor）模式的核心。
   - 编辑器维护两个独立的世界实例：`g_editWorld` 和 `g_playWorld`。
   - 当用户点击"Play"时：
     1. 序列化 `g_editWorld` 的当前状态（或直接使用同一关卡资产重新加载）。
     2. 创建全新的 `g_playWorld` 实例并加载关卡。
     3. 将输入路由从"编辑方案"（相机漫游、Gizmo 操作）切换到"游戏方案"（玩家移动、射击）。
   - 当用户点击"Stop"时：
     1. 销毁 `g_playWorld`。
     2. 将活跃世界指针切换回 `g_editWorld`。
     3. 恢复输入路由到编辑方案。
   - **关键设计**：视口窗口（HWND / SwapChain）本身不销毁，只切换它绑定的 `World` 上下文。这是 PIE 能做到"秒级启停"的根本原因。

3. **选中高亮（Highlight）**
   - 选中实体后，渲染管线需要额外绘制一层轮廓（Outline）或覆盖材质（Override Material）以提供视觉反馈。
   - 采用**修饰器模式（Modifier Pattern）**：`HighlightModifier` 基类下有 `OutlineModifier`、`WireframeModifier`、`ColorOverrideModifier` 等实现。
   - 每帧在渲染主 Pass 之后，遍历所有激活的修饰器，注入额外的 Draw Call。

**阶段 2 验收标准**：
- [x] 任何 Inspector 修改都支持 Undo/Redo
- [x] 能一键进入/退出 PIE 模式，编辑态和运行态互不干扰
- [x] 选中实体有清晰的视觉反馈（轮廓线或变色）

### 阶段 3：好用 → 工业级（解决扩展性、跨平台工具链和 AI 集成问题）

**触发原因**：引擎要支撑大型团队、跨平台开发和 AI Agent 协作。编辑器框架必须从一个"内部工具"升级为"可扩展的平台"。

**阶段 3 的详细内容已迁移至 [[编辑器基础设施]]，主要包括：**

1. **跨语言绑定与插件化编辑器前端**
2. **数据桥与增量同步协议**

4. **PIE 的进阶模式**（Pause / Step Frame、Multiplayer PIE、Play From Here）
5. **窗口布局与 Docking 系统**

7. **独立工具程序架构**

> **核心洞察**：编辑器框架和编辑器基础设施是两个层次。框架回答"如何操作场景"，基础设施回答"如何扩展工具链"。在 vibe coding 的早期阶段，先做好框架；当团队规模和资产量增长后，再逐步引入基础设施。

## AI 友好设计检查清单

每篇费曼笔记在讨论具体模块时，必须回扣到 AI 友好的核心约束。以下是编辑器框架的 checklist：

| 检查项 | 编辑器框架的满足方式 |
|--------|---------------------|
| **状态平铺** | ✅ 编辑器操作的对象全部是 ECS 组件和字段，没有隐藏的状态机。Gizmo 的 delta 最终也归结为 `TransformComponent` 的字段变更。 |
| **自描述** | ✅ Inspector 的面板完全由 `TypeRegistry` 驱动。AI 不需要读头文件，通过反射注册表即可知道每个组件有什么字段、什么类型。 |
| **确定性** | ⚠️ 需要特别注意：编辑器的 Undo/Redo 和 PIE 模式切换必须基于确定性的序列化/反序列化。如果编辑态到运行态的切换依赖于指针地址或内存布局，AI 将无法可靠地复现场景。应确保 World Snapshot 是平台无关的。 |
| **工具边界** | ✅ 编辑器的所有操作都可以被抽象为 `ICommand`。这是 AI 通过 MCP 操作引擎的完美接口：`mutate(entity, component, field, value)` 本质上就是构造并执行一个 Command。 |
| **Agent 安全** | ⚠️ PIE 模式本身就是一种沙箱——AI 在运行态的修改不会污染编辑态。更进一步的，可以为 AI Agent 分配一个只读或白名单模式的 Inspector，限制它能修改的组件类型（详见 [[多Agent编排与沙箱]]）。 |

## ECS 重构映射：从传统编辑器到 ECS 编辑器

传统 OOP 引擎（如早期 Unity、Unreal 的部分子系统）的编辑器通常直接操作 GameObject / Actor 对象及其成员变量。在 ECS 架构下，这一切需要被重构为对**组件和字段**的操作。

### 映射表

| OOP 编辑器概念 | ECS 编辑器等价物 | 说明 |
|---------------|-----------------|------|
| `Actor::SetPosition()` | 修改 `TransformComponent.position` | 没有"对象方法"，只有对组件字段的 set |
| `Actor::GetComponents()` | `Query<AllComponents>(entity)` | 通过 ECS 查询获取实体的所有组件 |
| `SceneGraph::root` | 所有没有 `ParentComponent` 的实体 | 层级关系由组件表达，而非树节点指针 |
| `EditorCamera` | 一个带有 `CameraComponent` 和 `EditorTag` 的特殊实体 | 相机只是世界中的一种实体组合 |
| `GizmoHandle` | 一个带有 `GizmoComponent` 和 `RenderMeshComponent` 的实体 | Gizmo 本身也是 ECS 中的普通实体 |
| `SelectionSet` | `std::vector<Entity>` + `SelectedTag` 组件 | 可以给选中实体临时附加一个标记组件 |

### 为什么 ECS 更适合编辑器

1. **Cache Locality**：Inspector 批量修改 100 个相同组件的同一字段时，可以按 Archetype 批量处理，而不是逐个对象反射 get/set。
2. **组合灵活性**：编辑器独有的功能（如选中高亮、Gizmo）可以通过临时附加 `EditorOnly` 组件实现，不会污染核心游戏组件的定义。
3. **AI 可观测性**：ECS 的平铺数据结构天然适合序列化为 JSON/表格。AI 可以直接用 `SELECT entity, position FROM TransformComponent WHERE SelectedTag EXISTS` 这样的心智模型理解场景状态。

## 设计权衡表

| 决策 | 选项 A | 选项 B | 本笔记推荐 |
|------|--------|--------|-----------|
| Inspector 数据同步 | 实时反射读写（简单直接） | 数据桥批量同步（性能高） | **阶段 1 用 A，阶段 2 起用 B** |
| PIE 世界隔离 | 深拷贝编辑世界（状态完全一致） | 重新加载关卡资源（速度快） | **推荐 B**，配合关卡序列化确保一致性 |
| 编辑器前端技术栈 | 原生 C++ UI（如 Qt/ImGui） | 脚本语言 UI（如 C#/Electron） | **核心逻辑 C++，UI 可插拔** |
| Gizmo 交互计算 | 屏幕空间投影（简单） | 射线-平面求交（精确） | **阶段 1 用 A，阶段 2 起用 B** |
| Undo 存储粒度 | 字段级（精细，内存大） | 组件级（粗糙，内存小） | **推荐字段级**，但可配置为组件级以优化内存 |

## 如果我要 vibe coding，该偷哪几招？

1. **反射驱动的 Inspector 自动生成**：不要为每个组件手写 UI 面板。只要你的 ECS 组件有自描述能力，Inspector 就应该是免费的。
2. **Command Pattern 就是 AI 的 MCP 接口**：人类的 Undo/Redo 栈和 AI 的 `mutate` 工具本质上是一回事。设计好 Command 接口，一举两得。
3. **PIE = 世界切换，不是窗口切换**：切换编辑态和运行态时，保留窗口和渲染上下文，只换 World 实例。这是实现"秒级 Play/Stop"的关键。
4. **Gizmo 也是 ECS 实体**：不要把 Gizmo 写成特殊的渲染对象。把它当作一个普通实体，有 `TransformComponent`、`MeshComponent`、`GizmoTag`，利用已有的渲染管线绘制。新增旋转/缩放 Gizmo 只需要换 Mesh，不需要改渲染代码。
5. **数据桥协议是 AI 友好性的分水岭**：在 C++ 和 UI 之间建立一层结构化数据桥，人类用滑块操作它，AI 用 JSON 操作它。这会让你的引擎立刻具备"可被 Agent 理解"的属性。
