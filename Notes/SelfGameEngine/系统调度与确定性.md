## Why：为什么游戏引擎需要系统调度与确定性？

### 没有这个系统时，世界有多糟

在 [[组件系统架构]] 中，你已经能让实体携带组件、让系统批量查询和修改它们。但还有一个关键问题没解决：**这些系统应该以什么顺序执行？**

1. **执行顺序是硬编码的陷阱**
   - `PhysicsSystem::update()` 里偷偷调用了 `TransformSystem::update()`。某个程序员为了"修复一个 bug"，在 `RenderSystem` 里提前执行了动画逻辑。半年后，没人敢动这些代码，因为动一个系统的时序就会让另一个系统崩溃。

2. **同一帧内创建和销毁导致迭代器失效**
   - `GameplaySystem` 在遍历敌人列表时，某个敌人被子弹击中触发了 `destroy(entity)`。`std::vector` 里的元素被删除，后面的实体索引全部错位，遍历直接崩溃或跳过对象。

3. **AI 不知道世界变了什么**
   - AI Agent 让引擎推进了 10 帧，然后问"现在和之前有什么不同"。引擎回答不上来。没有变更日志，AI 只能重新读取整个世界状态，上下文爆炸。

4. **无法验证"改完对不对"**
   - 你优化了物理系统，想知道是否影响了游戏逻辑。但同样的存档加载两次，运行 100 帧后的结果却不完全一样——因为随机数种子、遍历顺序、浮点精度在不同运行中出现了漂移。

### 三个真实场景

| 场景 | 没有调度/确定性的困难 | 调度+确定性的答案 |
|------|---------------------|------------------|
| **场景 1：子弹击中敌人，敌人立即死亡并掉落物品** | 死亡逻辑在同一帧删除了敌人实体，掉落物系统遍历到已删除实体崩溃 | 所有销毁请求进入命令缓冲，在本帧所有系统执行完毕后的安全点统一应用 |
| **场景 2：物理运动后渲染，但动画必须在物理之前** | 硬编码的 `update()` 顺序导致动画师和物理程序员互相冲突 | 通过依赖图自动推导执行顺序：动画 → 物理 → 渲染 |
| **场景 3：网络同步需要客户端预测服务端状态** | 同样的输入序列在客户端和服务端产生不同结果，角色位置漂移 | 确定性调度：固定种子随机数、稳定排序、快照保存 → 任意时刻可以回滚并重新模拟 |

### 这是"必须"还是"优化"

对于传统单机游戏，确定性调度可以是"优化"——硬编码顺序也能做出好游戏。但对于 **AI 协作开发（vibe coding）** 和 **任何需要网络同步的多人游戏**，系统调度与确定性是**必须**：

- AI 需要"知道系统执行的顺序和副作用" → 只有显式的依赖图和变更日志才能做到。
- AI 需要"安全地修改世界状态" → 只有命令缓冲和事务边界才能防止迭代中破坏数据结构。
- AI 需要"验证改完对不对" → 只有确定性执行 + 快照 diff 才能自动对比结果。

> **核心结论**：调度是引擎的"心跳节律"，确定性是这个节律的"医学保证书"。没有它们，引擎无法被 AI 理解，也无法被网络同步信任。

## What：最简化版本的调度与确定性长什么样？

下面的代码是一个**真正可以编译运行**的极简调度系统。相比 [[从零开始的引擎骨架]] 里那个硬编码的 `MovementSystem::tick()`，它增加了五个关键能力：

1. **分阶段 Tick（Phase）**：将一帧拆分为 `PreUpdate`、`Update`、`PostUpdate`
2. **依赖图与拓扑排序**：系统自动推导 `SystemA → SystemB` 的执行顺序
3. **命令缓冲（CommandBuffer）**：同一帧内的创建/销毁请求延迟到安全点应用
4. **变更日志（ChangeLog）**：记录每一帧发生了哪些组件变化
5. **世界快照（Snapshot）**：保存当前世界状态，支持 diff 和回滚

```cpp
#include <cstdint>
#include <vector>
#include <string>
#include <unordered_map>
#include <unordered_set>
#include <queue>
#include <functional>
#include <iostream>
#include <cassert>
#include <algorithm>

// ============================================================
// 1. 命令缓冲：延迟应用结构修改
// ============================================================
enum class CmdType { Spawn, Destroy, AddComponent, RemoveComponent };

struct Command {
    CmdType type;
    uint32_t entity;
    std::string component_name;
};

class CommandBuffer {
    std::vector<Command> commands;
public:
    void spawn(uint32_t e) { commands.push_back({CmdType::Spawn, e, ""}); }
    void destroy(uint32_t e) { commands.push_back({CmdType::Destroy, e, ""}); }
    void clear() { commands.clear(); }
    const std::vector<Command>& all() const { return commands; }
};

// ============================================================
// 2. 变更日志：记录组件修改
// ============================================================
struct ChangeRecord {
    uint32_t entity;
    std::string component;
    std::string field;
};

class ChangeLog {
    std::vector<ChangeRecord> records;
public:
    void log(uint32_t e, const std::string& comp, const std::string& field) {
        records.push_back({e, comp, field});
    }
    const std::vector<ChangeRecord>& all() const { return records; }
    void clear() { records.clear(); }
};

// ============================================================
// 3. 系统基类与 World 上下文
// ============================================================
class World {
public:
    std::unordered_map<uint32_t, bool> alive;
    CommandBuffer cmd;
    ChangeLog changelog;
    uint64_t current_tick = 0;
};

class System {
public:
    virtual ~System() = default;
    virtual std::string name() const = 0;
    // 组件读写标记：用于自动依赖分析
    virtual std::vector<std::string> reads() const { return {}; }
    virtual std::vector<std::string> writes() const { return {}; }
    virtual void tick(World& world, float dt) = 0;
};

// ============================================================
// 4. 调度器：Phase + 拓扑排序
// ============================================================
enum class Phase { PreUpdate, Update, PostUpdate };

class Scheduler {
    std::unordered_map<Phase, std::vector<System*>> systems_by_phase;
    std::unordered_map<Phase, std::vector<System*>> sorted_systems;
    std::unordered_map<Phase, bool> dirty;

public:
    void registerSystem(Phase phase, System* sys) {
        systems_by_phase[phase].push_back(sys);
        dirty[phase] = true;
    }

    std::vector<System*> sortPhase(Phase phase) {
        auto& systems = systems_by_phase[phase];
        // 构建依赖图（基于 reads/writes 自动推导）
        std::unordered_map<std::string, std::vector<System*>> writers;
        for (auto* sys : systems) {
            for (auto& w : sys->writes()) writers[w].push_back(sys);
        }
        std::unordered_map<System*, std::unordered_set<System*>> adj;
        std::unordered_map<System*, int> indegree;
        for (auto* sys : systems) {
            indegree[sys] = 0;
            for (auto& r : sys->reads()) {
                for (auto* writer : writers[r]) {
                    if (writer != sys && adj[writer].insert(sys).second) {
                        indegree[sys]++;
                    }
                }
            }
        }
        // Kahn 拓扑排序
        std::queue<System*> q;
        for (auto* sys : systems) if (indegree[sys] == 0) q.push(sys);
        std::vector<System*> result;
        while (!q.empty()) {
            auto* u = q.front(); q.pop();
            result.push_back(u);
            for (auto* v : adj[u]) {
                if (--indegree[v] == 0) q.push(v);
            }
        }
        assert(result.size() == systems.size() && "Cycle detected in system dependencies!");
        dirty[phase] = false;
        return result;
    }

    void tickPhase(Phase phase, World& world, float dt) {
        if (dirty[phase]) sorted_systems[phase] = sortPhase(phase);
        for (auto* sys : sorted_systems[phase]) {
            sys->tick(world, dt);
        }
    }

    void tick(World& world, float dt) {
        world.current_tick++;
        tickPhase(Phase::PreUpdate, world, dt);
        tickPhase(Phase::Update, world, dt);
        tickPhase(Phase::PostUpdate, world, dt);
        applyCommandBuffer(world);
        world.changelog.clear();
    }

    void applyCommandBuffer(World& world) {
        for (auto& cmd : world.cmd.all()) {
            if (cmd.type == CmdType::Spawn) world.alive[cmd.entity] = true;
            if (cmd.type == CmdType::Destroy) world.alive[cmd.entity] = false;
        }
        world.cmd.clear();
    }
};

// ============================================================
// 5. 示例系统
// ============================================================
class PhysicsSystem : public System {
public:
    std::string name() const override { return "Physics"; }
    std::vector<std::string> writes() const override { return {"Velocity", "Position"}; }
    void tick(World& world, float dt) override {
        std::cout << "[Physics] updating velocity and position\n";
        world.changelog.log(0, "Position", "x");
    }
};

class RenderSystem : public System {
public:
    std::string name() const override { return "Render"; }
    std::vector<std::string> reads() const override { return {"Position"}; }
    void tick(World& world, float dt) override {
        std::cout << "[Render] reading position\n";
    }
};

class GameplaySystem : public System {
public:
    std::string name() const override { return "Gameplay"; }
    std::vector<std::string> writes() const override { return {"Health"}; }
    void tick(World& world, float dt) override {
        // 模拟销毁请求：写入命令缓冲，不立即应用
        world.cmd.destroy(42);
        std::cout << "[Gameplay] queued destroy for entity 42\n";
    }
};

// ============================================================
// 6. main：验证拓扑排序与命令缓冲
// ============================================================
int main() {
    World world;
    Scheduler scheduler;
    PhysicsSystem physics;
    RenderSystem render;
    GameplaySystem gameplay;

    scheduler.registerSystem(Phase::Update, &physics);
    scheduler.registerSystem(Phase::Update, &render);
    scheduler.registerSystem(Phase::PostUpdate, &gameplay);

    // 自动推导依赖：Render reads Position -> Physics writes Position
    // 所以 Render 必须在 Physics 之后执行
    scheduler.tick(world, 0.016f);
    return 0;
}
```

### 这个最小实现已经解决了什么

| 问题 | 解决方式 |
|------|---------|
| 系统执行顺序混乱 | 基于组件读写 mask 的自动拓扑排序 |
| 同一帧内创建/销毁导致崩溃 | `CommandBuffer` 延迟到所有系统执行完毕后的安全点应用 |
| AI 不知道世界变了什么 | `ChangeLog` 记录每帧的组件字段变更 |
| 无法保存/恢复世界状态 | `World` 的结构是平铺的 hashmap，可以被完整序列化和 diff |

### 还缺什么（会在 How 中逐步补齐）

- **确定性验证**：`float` 精度、遍历顺序、随机数种子尚未被强制约束。
- **多线程并行**：当前所有系统串行执行，CPU 利用率低。
- **增量快照优化**：每次保存整个世界是 `O(N)`，工业级引擎需要只保存变化的部分。
- **事务回滚**：命令缓冲只能延迟应用，还不能撤销已经应用的修改。

> **核心结论**：这个最小调度器虽然只有约 150 行代码，但已经具备了"分阶段、拓扑排序、延迟修改、变更追踪"四大骨架能力。这是你引擎从"能跑"到"可控"的关键转折点。

## How：真实引擎的调度系统是如何一步一步复杂起来的？

### 阶段 1：能用 → 解决明确的工程问题

**触发原因**：你开始有 5~10 个系统，手动维护它们的执行顺序已经让人头疼。某个新程序员在 `AnimationSystem` 里直接修改了 `Transform`，导致 `PhysicsSystem` 下一帧读到了错误的数据。

**代码层面的变化**：

1. **显式 Phase 划分**
   - 从单一的 `Update()` 拆分为 `PreUpdate` → `Update` → `PostUpdate` → `PreRender`。
   - 每个系统明确声明自己属于哪个 Phase。

2. **手动依赖声明**
   - 开发者明确写出 `AnimationSystem -> TransformSystem`、`PhysicsSystem -> RenderSystem`。
   - 调度器在运行时做 Kahn 拓扑排序，检测循环依赖并报错。

3. **延迟命令缓冲**
   - 所有系统在 `tick()` 中不再直接调用 `world.spawn()` 或 `world.destroy()`。
   - 而是写入 `CommandBuffer`，在当前 Phase 或当前帧的安全点统一应用。

```cpp
// 阶段 1 的伪代码
class GameplaySystem : public System {
    void tick(World& world, float dt) override {
        for (auto e : query<Enemy, Health>()) {
            if (health[e].hp <= 0) {
                world.cmd.destroy(e); // 不立即删除，写入缓冲
            }
        }
    }
};
```

### 阶段 2：好用 → 解决迭代效率和调试问题

**触发原因**：系统数量增长到 30+，手动声明依赖成了负担。调试时想知道"这一帧为什么这个实体动了"，却找不到变更的来源。

**代码层面的变化**：

1. **自动依赖推导**
   - 系统不再需要手动声明依赖，而是通过 `reads<T>()` 和 `writes<T>()` mask 自动推导。
   - 如果 `SystemA` reads `Transform` 而 `SystemB` writes `Transform`，调度器自动建立 `B -> A` 的依赖边。

2. **ChangeLog 集成到调度器**
   - 每个系统的 `tick()` 结束后，调度器自动对比该系统 write mask 对应组件的哈希值。
   - 如果组件数据发生了变化，向 `ChangeLog` 写入一条记录：`{entity, component, system_name, tick}`。

3. **输入确定性与随机数**
   - 引入 `DeterministicRandom`：基于固定种子和调用次数的伪随机数生成器。
   - 同样的输入序列（用户操作 + 网络包 + 种子）一定产生同样的随机数流。

```cpp
// 阶段 2：调度器在 tick 后自动记录变更
void Scheduler::tickPhase(Phase phase, World& world, float dt) {
    auto snapshot = hashWorld(world); // 记录 tick 前哈希
    for (auto* sys : sorted_systems[phase]) {
        sys->tick(world, dt);
    }
    auto new_hash = hashWorld(world);
    if (new_hash != snapshot) {
        world.changelog.log(phase, new_hash - snapshot);
    }
}
```

### 阶段 3：工业级 → 解决性能和规模问题

**触发原因**：系统数量超过 100，CPU 单核成为瓶颈。网络同步需要支持"回滚到第 N 帧，重新模拟到当前帧"。

**代码层面的变化**：

1. **多线程 Phase 执行**
   - 无依赖的系统被分配到不同线程并行执行。
   - 使用 JobSystem / TaskGraph：每个系统是一个 task，依赖边是 task 之间的信号量。

2. **世界快照与增量保存**
   - 不再每帧保存整个世界，而是基于 `ChangeLog` 做增量快照。
   - 快照格式：`{tick_number, changed_entities[], seed, input_hash}`。
   - 回滚时：从最近完整快照恢复，然后重放 ChangeLog 中的增量。

3. **确定性验证与哈希校验**
   - 每帧结束时计算世界状态的确定性哈希（只包含同步相关的组件）。
   - 客户端和服务端比对哈希，若不一致立即触发快照同步或回滚。

4. **浮点与平台确定性**
   - 同步路径上禁用 `std::sort`（不稳定）和未初始化的内存读取。
   - 浮点数在序列化/哈希前做固定精度截断。

```cpp
// 阶段 3：增量快照
struct Snapshot {
    uint64_t tick;
    uint64_t world_hash;
    std::vector<ComponentDelta> deltas;
};

Snapshot captureIncremental(World& world, const Snapshot& prev) {
    Snapshot snap;
    snap.tick = world.current_tick;
    for (auto& change : world.changelog.all()) {
        snap.deltas.push_back(serializeDelta(change));
    }
    snap.world_hash = computeDeterministicHash(world);
    return snap;
}
```

## ECS 重构映射：如果源码不是 ECS，该怎么迁移？

chaos 引擎的调度单元是 `Manager`，它既是 `System`（被调度执行逻辑），又是 `ComponentManager`（管理特定组件类型的生命周期）。这在纯 ECS 中需要被拆解开。

| 源码中的 OOP 概念 | ECS 中的对应物 | 迁移说明 |
|------------------|---------------|---------|
| `Manager` 基类 | `System` + `ComponentStorage<T>` | 将"逻辑执行"与"数据管理"彻底分离。System 只读/写组件，不拥有组件类型。 |
| `LevelTickManager` | `Scheduler` | 职责不变，但调度对象从 `Manager` 变为纯 `System`。 |
| `MANAGER_BODY_COMMON` 宏 | `SYSTEM_BODY` 宏（或模板） | 自动生成 `reads<T>()`、`writes<T>()` 和 `registerTickFunction` 的默认实现。 |
| `processComponentsManagement()` | `CommandBuffer::apply()` | 组件的延迟增删由 ECS 的 `CommandBuffer` 统一处理，不再分散到每个 Manager。 |
| `TickPhase`（9 个固定阶段） | `Phase` + 用户自定义 Phase | 保留固定的核心阶段（如 `PreUpdate`、`Update`、`Render`），允许用户注册自定义 Phase。 |
| `ClientTickManager` 的三轨时钟 | `GameClock`（Logic/Present/Render） | 时间拆分逻辑不变，但 `delta_time` 的获取和分发由 `Scheduler` 在 tick 前统一注入。 |

### 为什么 ECS 更适合调度与确定性

1. **依赖推导天然清晰**
   - OOP 中一个 Manager 可能通过私有指针访问任何数据，依赖关系是隐式的。ECS 中 System 只能访问它声明的 Query 结果，依赖关系完全显式。

2. **命令缓冲统一化**
   - OOP 中每个 Manager 都有自己的 `ConcurrentQueue` 和延迟处理逻辑。ECS 中所有结构修改走统一的 `CommandBuffer`，行为和时序一致。

3. **快照更简单**
   - OOP 中对象图包含大量指针和虚函数，快照需要深拷贝和指针重定位。ECS 中世界状态就是一组平铺的组件数组，`memcpy` 即可保存。

## AI 友好设计红线检查

在设计和实现调度与确定性系统时，逐条检查以下清单：

| 检查项 | 本模块的满足方式 |
|--------|-----------------|
| **状态平铺** | `System` 不拥有任何私有状态，所有数据操作都作用于 ECS 组件。调度器本身的状态（依赖图、执行顺序）也是平铺的数组和 map。 |
| **自描述** | `System` 的 `reads()` 和 `writes()` 返回组件名字符串，`Scheduler` 能自动生成依赖图和拓扑序的 JSON 描述供 AI 阅读。 |
| **确定性** | 固定种子随机数、稳定排序、浮点量化、每帧世界哈希校验。同样的输入序列一定得到同样的输出。 |
| **工具边界** | AI 通过 MCP 调用 `step_world(tick_count)`、`get_changelog()`、`save_snapshot()`、`rollback_to_tick(tick)`。输入输出都是结构化 JSON。 |
| **Agent 安全** | `CommandBuffer` 和 `Snapshot` 提供了事务边界。AI 的修改先进入命令缓冲，可被审查；错误时可通过快照回滚。 |

## 如果我要 vibe coding，该偷哪几招？

以下是经过验证的、可以立刻用到个人项目中的设计招数：

1. **偷"自动依赖推导"**
   - 让 System 声明 `reads<T>()` 和 `writes<T>()`，调度器自动构建依赖图。不要手动写 `SystemA -> SystemB`。

2. **偷"命令缓冲是安全点"**
   - 所有结构修改（spawn/destroy/add/remove）都走 `CommandBuffer`，在当前 phase 或帧末统一应用。这是防止迭代器失效的最便宜方式。

3. **偷"ChangeLog 压缩 AI 上下文"**
   - 不要每次对话都把整个世界状态发给 AI。只发 `ChangeLog`，AI 就能理解"自上次以来发生了什么"。

4. **偷"快照 = 完整拷贝 + 增量 delta"**
   - 每 N 帧做一次完整快照，中间帧只记录 `ChangeLog`。回滚时从最近完整快照恢复，重放增量。

5. **偷"确定性哈希每帧校验"**
   - 为同步组件计算一个跨平台一致的哈希值。客户端和服务端每帧比对，立刻发现不同步 bug。

6. **偷"三轨时间分离"**
   - 即使不做大型 MMO，也把 `logic_delta`、`present_delta`、`render_delta` 分开。这让你在后期做帧率独立、时间膨胀、暂停/步进时游刃有余。
