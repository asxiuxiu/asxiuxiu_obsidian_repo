# 可视化日志系统

> **前置依赖**：[[窗口与输入系统]]
> **本模块增量**：引擎拥有了第一个"可视化调试工具"——日志面板。支持异步多线程写入、文件滚动持久化、屏幕实时过滤显示，以及调用点追踪。从此所有系统验证都通过日志或窗口反馈，不再硬编码 `printf`。
> **下一步**：[[最小ECS数据层]] — 在窗口里直接观察 ECS 世界。

---

## Why：为什么 `printf` 不够？

刚开始写引擎时，`printf` 或 `std::cout` 是最快的调试方式。但当系统从 1 个变成 10 个、线程从 1 条变成多条时，你会遇到四个致命问题。

### 没有专业日志系统时，世界有多糟

1. **多线程下日志交错**
   - `printf` 不是线程安全的。两条线程同时输出，控制台里会看到半截消息混在一起，完全无法阅读。

2. **崩溃后丢失最后几条关键日志**
   - `printf` 是带缓冲的。程序 segfault 时，缓冲区里最后几条最能定位问题的日志往往还没来得及刷到控制台。

3. **信息噪音淹没重点**
   - 当你只想看渲染系统的提交状态时，物理引擎的碰撞日志会把屏幕刷满。没有级别过滤，调试效率极低。

4. **AI 无法高效观察世界**
   - Agent 从控制台读取原始文本是脆弱且低效的。没有结构化分类，AI 很难在数千行日志中定位"刚才为什么掉帧"。

### 真实应用场景

| 场景 | 用 `printf` | 用可视化日志系统 |
|------|------------|----------------|
| 多线程物理模拟 | 日志交错，无法阅读 | 异步队列保证顺序，按时间戳排列 |
| 崩溃分析 | 最后几条日志丢失 | 后台线程实时 flush 到文件 |
| 性能调优 | 满屏数字，找不到 FPS | 按 `Render` 分类过滤，只看帧时间 |
| AI 行为验证 | 读控制台文本猜状态 | 读取结构化 `Event` 日志，直接解析 |

> **核心结论**：可视化日志是引擎从"能编译"升级为"可调试"的第一个里程碑。它是后续所有系统（ECS、渲染、物理）的验证基础设施。

## What：最小能跑的可视化日志系统长什么样？

下面的代码是一个**真正可以编译运行**的最小实现。它包含：异步日志队列、控制台/文件双端输出、以及一个基于 ImGui 的屏幕滚动面板。

```cpp
#include <vector>
#include <string>
#include <mutex>
#include <queue>
#include <fstream>
#include <iostream>
#include <chrono>
#include <iomanip>

// ============================================================
// 1. 日志核心数据结构
// ============================================================
enum class LogLevel { Debug, Info, Warning, Error };

struct LogEntry {
    LogLevel level;
    std::string category;   // 如 "Render", "Physics"
    std::string message;
    std::string timestamp;
};

// ============================================================
// 2. 异步日志后端
// ============================================================
class Logger {
    std::mutex mutex_;
    std::queue<LogEntry> queue_;
    std::vector<LogEntry> history_;  // 用于屏幕面板显示
    std::ofstream file_;

    const char* levelStr(LogLevel l) const {
        switch (l) {
            case LogLevel::Debug:   return "DBG";
            case LogLevel::Info:    return "INF";
            case LogLevel::Warning: return "WRN";
            case LogLevel::Error:   return "ERR";
        }
        return "???";
    }

public:
    Logger(const char* filename = "engine.log") {
        file_.open(filename, std::ios::out | std::ios::trunc);
    }

    void log(LogLevel level, const char* category, const char* msg) {
        auto now = std::chrono::system_clock::now();
        auto t = std::chrono::system_clock::to_time_t(now);
        std::tm tm;
        localtime_s(&tm, &t);

        char time_buf[32];
        std::strftime(time_buf, sizeof(time_buf), "%H:%M:%S", &tm);

        LogEntry entry{level, category, msg, time_buf};

        // 异步写入：先锁队列
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_.push(entry);
        }
    }

    // 每帧在主线程调用一次
    void flush() {
        std::vector<LogEntry> batch;
        {
            std::lock_guard<std::mutex> lock(mutex_);
            while (!queue_.empty()) {
                batch.push_back(queue_.front());
                queue_.pop();
            }
        }

        for (auto& e : batch) {
            // 输出到文件
            file_ << "[" << e.timestamp << "][" << levelStr(e.level)
                  << "][" << e.category << "] " << e.message << "\n";
            // 输出到控制台
            std::cout << "[" << levelStr(e.level) << "][" << e.category << "] "
                      << e.message << "\n";
            // 保留到历史（用于 ImGui 面板）
            history_.push_back(e);
        }
        file_.flush();
    }

    const std::vector<LogEntry>& history() const { return history_; }
};

// 全局实例（实际引擎中会注入到 Context 或作为单例）
Logger g_logger;

// 宏封装（提供调用点信息）
#define LOG_INFO(cat, msg)  g_logger.log(LogLevel::Info,  cat, msg)
#define LOG_ERROR(cat, msg) g_logger.log(LogLevel::Error, cat, msg)

// ============================================================
// 3. 使用示例
// ============================================================
int main() {
    LOG_INFO("Engine", "Startup");

    // 模拟工作线程产生日志
    auto worker = []() {
        LOG_INFO("Physics", "Simulation step 1");
        LOG_ERROR("Render", "Shader compile failed");
    };

    // 在主循环中：
    // worker();      // 产生日志
    // g_logger.flush(); // 消费并输出

    worker();
    g_logger.flush();

    return 0;
}
```

### 最小实现已经解决了什么？

1. **线程安全写入**：工作线程通过 `mutex` 把日志推入队列，主线程在 `flush()` 中批量消费。这是**单生产者-单消费者**模型的雏形。
2. **文件持久化**：即使程序崩溃，已经 `flush()` 到文件的内容不会丢失。
3. **分类与级别**：每条日志都带 `category` 和 `level`，为后续过滤打下基础。
4. **屏幕面板数据源**：`history_` 数组可以直接被 ImGui 面板读取并渲染。

### 最小实现还缺什么？

- 没有**真正的零分配**：`std::string` 和 `std::queue` 会在堆上频繁分配，高频日志（如每帧 1000 条渲染事件）会成为瓶颈。
- 没有**调用点信息**：不知道日志来自哪个文件、哪一行。
- 没有**运行时级别过滤**：无法在游戏运行时把 `Debug` 日志关掉以减少噪音。
- 没有**结构化输出**：AI 无法直接以 JSON/Schema 形式消费日志内容。

## How：真实引擎的日志系统是如何复杂起来的？

### 阶段 1：最小实现 → 能用（异步双端 + ImGui 面板）

**触发原因**：单线程 demo 很快就能跑通，但一旦引入多线程（物理线程、加载线程、网络线程），日志交错和线程安全问题立刻暴露。

**代码层面的变化**：

**A. 引入批量消费 + 双缓冲队列**

上面的最小实现已经是这个模式：

```
工作线程：分配 LogEntry → 加锁 → push_back 到 queue_A
主线程 flush()：加锁 → swap(queue_A, queue_B) → 解锁 → 遍历 queue_B 输出
```

`swap` 替代了逐个 `pop`，把临界区缩到最短。

**B. 增加 ImGui 滚动面板**

```cpp
// 在窗口与输入系统的主循环中
void drawLogPanel(const Logger& logger) {
    ImGui::Begin("Logs");
    for (const auto& e : logger.history()) {
        ImVec4 color = ImVec4(1,1,1,1);
        if (e.level == LogLevel::Error)   color = ImVec4(1,0.3f,0.3f,1);
        if (e.level == LogLevel::Warning) color = ImVec4(1,1,0.3f,1);
        ImGui::TextColored(color, "[%s][%s] %s", e.timestamp.c_str(),
                           e.category.c_str(), e.message.c_str());
    }
    if (ImGui::GetScrollY() >= ImGui::GetScrollMaxY()) {
        ImGui::SetScrollHereY(1.0f); // 自动滚动到底部
    }
    ImGui::End();
}
```

**C. 日志级别过滤**

```cpp
LogLevel g_min_level = LogLevel::Info;

bool shouldLog(LogLevel level) {
    return static_cast<int>(level) >= static_cast<int>(g_min_level);
}
```

### 阶段 2：能用 → 好用（调用点追踪 + 分类过滤 + 文件滚动）

**触发原因**：当问题出现时，你不仅想知道"发生了什么"，还想知道"在哪一行代码发生的"。同时，运行数小时后日志文件可能膨胀到数 GB。

**代码层面的变化**：

**A. 宏注入调用点**

```cpp
#define LOG_INFO(cat, fmt, ...) \
    g_logger.log(LogLevel::Info, cat, __FILE__, __LINE__, fmt, ##__VA_ARGS__)
```

`__FILE__` 和 `__LINE__` 让你能一键定位到源码。在发布版中，这些宏可以被编译为空以节省开销。

**B. 分类过滤面板**

```cpp
// 用户可以在 ImGui 中勾选/取消勾选某个分类
bool show_render = true;
bool show_physics = false;

for (const auto& e : history) {
    if (e.category == "Render" && !show_render) continue;
    if (e.category == "Physics" && !show_physics) continue;
    // 渲染...
}
```

**C. 文件滚动（Rolling File）**

```cpp
struct LogFileConfig {
    std::string base_path = "logs/engine";
    size_t max_size_mb = 10;        // 单个文件最大 10MB
    size_t max_files = 5;           // 保留最近 5 个
};
```

当当前日志文件超过阈值时，关闭旧文件、创建新文件（如 `engine_001.log`、`engine_002.log`），并删除最老的文件。

### 阶段 3：好用 → 工业级（零分配 + 结构化日志 + 崩溃集成）

**触发原因**：大型引擎每秒可能产生数千条日志。堆分配、字符串拷贝、文件 IO 会成为性能热点。同时，自动化测试和 AI Agent 需要机器可读的日志格式。

**代码层面的变化**：

**A. 零分配环形缓冲区（Ring Buffer）**

不再为每条日志分配 `std::string`，而是预分配一大块连续内存（如 8MB），用环形缓冲区存储固定格式的日志：

```
Ring Buffer 内存布局（8MB）：
┌────────────────────────────────────────────────────────────┐
│ [Header: timestamp|level|category_len|msg_len] [category] [msg] │
│ [Header] [category] [msg]                                    │
│ ...                                                          │
└────────────────────────────────────────────────────────────┘
                    ^write_pos (循环覆盖最老日志)
```

**操作状态变化**：
- **写入前**：`write_pos` 指向空闲区起始。
- **写入后**：`write_pos` 向前移动 `header + category_len + msg_len` 字节。如果超过缓冲区末尾，则回绕到开头。
- **覆盖处理**：如果新日志会覆盖尚未读取的旧日志，则推进 `read_pos` 跳过被覆盖的完整条目，保证消费者永远读到完整的日志。

> **诚实的边界**：环形缓冲区消除了每日志的堆分配，但引入了**容量限制**和**覆盖丢失**。如果你的调试场景需要保留 24 小时完整日志，环形缓冲区不适用，应继续使用滚动文件。

**B. 结构化日志输出**

为 AI 和自动化测试提供 JSON 格式的日志行：

```json
{"timestamp":"14:32:01","level":"Error","category":"Render","message":"Shader compile failed","file":"shader.cpp","line":128}
```

这让 Agent 可以直接用 JSON 解析器读取引擎状态，而不用写脆弱的正则表达式。

**C. 与崩溃转储集成**

注册一个 `std::set_terminate` 或信号处理函数，在崩溃前执行 `logger.emergencyFlush()`：

```cpp
void onCrash() {
    g_logger.emergencyFlush(); // 无锁，直接把缓冲区内容写入文件
    // 然后生成 minidump / stacktrace
}
```

这样即使主线程已经崩溃，最后几条日志（往往包含崩溃原因）也能被保留下来。

## 性能与 Trade-off

### 选择 1：`printf` vs 异步队列 vs 环形缓冲区

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| `printf` | 最简单 | 线程不安全、无持久化、无过滤 | 单线程原型（< 1000 行） |
| 异步队列 + 文件 | 线程安全、可持久化 | 每日志都有堆分配和拷贝 | 一般开发调试 |
| 环形缓冲区 | 零分配、极高吞吐 | 容量固定、老日志会被覆盖 | 发布版运行、性能敏感场景 |

> **反例警告**：不要在发布版中保留 `Debug` 级别的日志输出。即使被过滤掉，格式化字符串的参数求值和函数调用开销仍然存在。发布版日志应通过编译期宏完全抹除。

### 选择 2：文本日志 vs 结构化日志

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 纯文本 | 人类可读、体积小 | AI/机器解析困难 | 人类调试 |
| 结构化（JSON） | AI 可直接解析、可搜索 | 体积大、有序列化开销 | AI 观察、自动化测试 |

> **折中策略**：人类可读日志写入 `.log` 文本文件；结构化 JSON 日志写入 `.jsonl` 文件（每行一个 JSON 对象）。两者并行输出，互不干扰。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|----------------|
| **状态平铺** | 日志历史是一个时间序列数组，每条日志都是自描述的结构体。AI 可以直接遍历查询。 |
| **自描述** | `LogEntry` 的结构固定（level/category/message/timestamp），通过反射注册，AI 无需读头文件即可理解。 |
| **确定性** | 异步队列 + 主线程批量 flush 保证了日志输出的时序与帧边界对齐。相同输入序列产生相同日志序列。 |
| **工具边界** | AgentBridge 提供 `log.query(category, level, max_count)` 和 `log.subscribe(category)` 等结构化接口。 |
| **Agent 安全** | AI 只能读取日志和订阅分类，不能直接修改日志历史或伪造日志内容。 |

## 如果我要 vibe coding，该偷哪几招？

1. **异步队列 + 主线程批量 flush**
   - 这是保证多线程日志不交错的最小可行方案。不要在工作线程里直接写文件。

2. **用 `__FILE__` 和 `__LINE__` 武装日志宏**
   - 调试时最想知道的就是"这条日志从哪来的"。

3. **日志分级 + 分类过滤**
   - 从第一天就引入 `Debug/Info/Warning/Error` 和 `category`。这是避免日志噪音的唯一办法。

4. **为 AI 保留一条 JSON 日志流**
   - 即使人类读文本日志，也并行输出 `.jsonl`。未来接入 Agent 时，它可以直接解析。

5. **崩溃前 emergencyFlush**
   - 这是救命的设计。崩溃时最后几条日志往往比 stacktrace 更能说明问题。
