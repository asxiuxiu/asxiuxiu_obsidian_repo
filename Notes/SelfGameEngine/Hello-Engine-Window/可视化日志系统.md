# 可视化日志系统

> **前置依赖**：[[ImGui接入与基础面板]]
> **本模块增量**：引擎拥有了第一个"可视化调试工具"——日志面板。支持异步多线程写入、编译期+运行时双层过滤、文件滚动持久化、屏幕实时过滤显示、调用点追踪，以及结构化输出。从此所有系统验证都通过日志或窗口反馈，不再硬编码 `printf`。
> **下一步**：[[最小ECS数据层]] — 在窗口里直接观察 ECS 世界。

## 问题 0：为什么 `printf` 不够？

刚开始写引擎时，`printf` 或 `std::cout` 是最快的调试方式。但当你从单线程 demo 走向真正的多系统协作时，四个致命问题会同时砸过来。

**场景一：多线程日志交错**

假设你的物理线程和渲染线程同时报错：

```cpp
// 物理线程
printf("[Physics] Collision detected at (");
// 渲染线程
printf("[Render] Shader compile failed\n");
// 物理线程继续
printf("%.2f, %.2f, %.2f)\n", x, y, z);
```

控制台输出可能变成：

```
[Physics] Collision detected at ([Render] Shader compile failed
0.00, 1.00, 2.00)
```

`printf` 不是线程安全的。两条线程同时输出，消息会互相插入，完全无法阅读。

**场景二：崩溃后丢失最后几条关键日志**

程序 segfault 时，操作系统不会帮你把 `stdout` 缓冲区里的内容刷出来。最后几条最能定位问题的日志——比如"即将访问纹理指针"——往往就这么消失了。

**场景三：信息噪音淹没重点**

当你只想看渲染系统的提交状态时，物理引擎的碰撞日志会把屏幕刷满。没有级别过滤，调试效率极低。

**场景四：AI 无法高效观察世界**

Agent 从控制台读取原始文本是脆弱且低效的。没有结构化分类，AI 很难在数千行文本中定位"刚才为什么掉帧"。

> **核心结论**：可视化日志是引擎从"能编译"升级为"可调试"的第一个里程碑。它是后续所有系统（ECS、渲染、物理）的验证基础设施。

## 问题 1：如何区分日志级别？

### 最 naive 的方案

```cpp
enum class LogLevel { Debug, Info, Warning, Error };

void log(LogLevel level, const char* msg) {
    if (level >= g_min_level) {
        printf("[%s] %s\n", levelToStr(level), msg);
    }
}
```

这看起来合理，但有一个致命缺陷：**即使日志被运行时过滤掉了，参数求值的开销仍然存在**。

```cpp
// 假设 g_min_level = Warning，这条 Debug 日志不会输出
log(LogLevel::Debug, format("entity count: %d", world.entity_count()).c_str());
```

问题在哪？`format()` 函数仍然会被调用，`std::string` 仍然会在堆上分配，几百条这样的日志每帧执行一次，Shipping 构建的帧率会直接崩塌。

### 改进：编译期零开销过滤

工业级引擎（如 UE、TraceLog 体系）的解法是**在预处理阶段就消除代码**，而不是在运行时跳过。

```cpp
// 构建系统根据配置定义 SHIPPING_BUILD
#ifdef SHIPPING_BUILD
    #define LOG_COMPILE_LEVEL LogLevel::Error
#else
    #define LOG_COMPILE_LEVEL LogLevel::Debug
#endif

#define LOG_IF_ENABLED(level, cat, msg) \
    do { \
        if constexpr (static_cast<int>(level) >= static_cast<int>(LOG_COMPILE_LEVEL)) { \
            g_logger.log(level, cat, msg, __FILE__, __LINE__); \
        } \
    } while(0)
```

`if constexpr` 在 C++17 中会在编译期求值。如果 `LOG_COMPILE_LEVEL` 是 `Error`，`LOG_INFO` 的宏体在编译后**完全消失**——不仅不输出，连参数求值都不会执行。这比运行时 `if` 更高效，因为后者仍然需要计算参数。

> **诚实边界**：编译期过滤的代价是灵活性——你无法在 Shipping 构建中临时打开 Debug 日志。因此需要第二层过滤。

### 第二层：运行时动态过滤

开发调试时，你需要临时关闭某个系统的噪音：

```cpp
std::atomic<LogLevel> g_min_level{LogLevel::Debug};

void setLogLevel(LogLevel level) {
    g_min_level.store(level, std::memory_order_relaxed);
}

// 在 log() 函数中无锁快速路径
if (level < g_min_level.load(std::memory_order_relaxed)) return;
```

运行时过滤用于开发调试时动态关闭噪音，但不替代编译期过滤——Shipping 构建不应依赖它。

> **参考**：UE 的日志体系采用这种**编译期 + 运行时双层过滤**。`CompileTimeVerbosity` 在 Shipping 中完全剔除低级别日志；`RuntimeVerbosity` 允许编辑器中动态调整。Bevy/tracing 则通过 `EnvFilter` 在运行时按模块精确控制，但 Rust 生态的宏优化在编译期同样能消除死代码。

## 问题 2：多线程下日志如何不交错？

### 最 naive 的方案

```cpp
std::mutex g_log_mutex;

void log(LogLevel level, const char* msg) {
    std::lock_guard<std::mutex> lock(g_log_mutex);
    printf("[%s] %s\n", levelToStr(level), msg);
}
```

这解决了交错问题，但引入了新的灾难：**主线程被 I/O 阻塞**。

假设一帧内有 1000 条日志。上面的代码意味着 1000 次加锁、1000 次 `printf`、1000 次系统调用。在 16ms 的帧预算里，`printf` 可能直接吃掉 5ms，帧率从 60fps 掉到 45fps。

### 改进：异步双缓冲队列

核心思想是**把日志先写到内存，让后台慢慢刷盘**。

```cpp
class Logger {
    std::mutex mutex_;
    std::vector<LogEntry> queue_a_;  // 生产者写入
    std::vector<LogEntry> queue_b_;  // 消费者读取（swap 后）
    std::ofstream file_;

public:
    void log(LogLevel level, const char* category, const char* msg,
             const char* file, int line) {
        auto now = std::chrono::steady_clock::now();
        auto sec = std::chrono::duration<double>(now.time_since_epoch()).count();

        LogEntry entry{level, 0, msg, sec, file, line};

        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_a_.push_back(entry);  // 只 push，不格式化，不写文件
        }
    }

    // 每帧在主线程调用一次
    void flush() {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_a_.swap(queue_b_);    // 临界区仅一次指针交换
        }

        for (auto& e : queue_b_) {
            file_ << "[" << e.timestamp_sec << "][" << levelStr(e.level)
                  << "][" << e.file << ":" << e.line << "] "
                  << e.message << "\n";
        }
        queue_b_.clear();
        file_.flush();
    }
};
```

工作线程只执行 `push_back`（内存写），主线程在 `flush()` 中一次性 `swap` 后批量处理（格式化 + 文件 I/O）。临界区从"1000 次加锁"缩减为"1 次 swap"。

> **参考**：这是 chaos 等中型引擎的起步基准，也是 spdlog 异步模式的核心思想。区别在于 spdlog 使用环形缓冲区（ring buffer）而这里是双 `std::vector`，环形缓冲区在预分配内存和避免扩容方面更优，但双 vector 的实现更直观。

### 还遗留的问题

所有线程仍然竞争同一把 `mutex_`。如果日志量极高（>1000 条/帧），这把锁仍会成为瓶颈。

## 问题 3：如何支持多种输出目标？

### 最 naive 的方案

在 `flush()` 里写死 if-else：

```cpp
void flush() {
    // ... swap ...
    for (auto& e : queue_b_) {
        if (b_console_enabled) printf(...);
        if (b_file_enabled) file_ << ...;
        if (b_network_enabled) socket.send(...);
        // 新增输出？继续加 if...
    }
}
```

问题很明显：每新增一种输出目标，都要修改 `Logger` 类。控制台的格式、文件的格式、网络的格式混在一起，同一条日志可能需要格式化三次。

### 改进：Sink 抽象（设备抽象）

把"输出到哪里"和"日志核心"解耦：

```cpp
struct LogSink {
    virtual ~LogSink() = default;
    virtual void write(const LogEntry& entry) = 0;
    virtual void flush() = 0;
};

struct ConsoleSink : LogSink {
    void write(const LogEntry& e) override {
        std::cout << "[" << levelStr(e.level) << "] " << e.message << "\n";
    }
    void flush() override { std::cout.flush(); }
};

struct FileSink : LogSink {
    std::ofstream file_;
    void write(const LogEntry& e) override { /* 格式化并写入文件 */ }
    void flush() override { file_.flush(); }
};
```

Logger 只维护一个 `std::vector<std::unique_ptr<LogSink>> sinks_`，`flush()` 时遍历每个 Sink：

```cpp
for (auto& e : queue_b_) {
    for (auto& sink : sinks_) {
        sink->write(e);
    }
}
for (auto& sink : sinks_) {
    sink->flush();
}
```

> **关键认知**：Sink 列表是**平铺的回调数组**，不是虚函数继承树。在 ECS 化之后，你可以把每个 Sink 注册为一个 System，日志事件作为 ECS Event 被每个 Sink System 消费。这消除了 OOP 的虚函数表开销，同时保持了扩展性。

> **参考**：UE 的 `FOutputDevice` 就是这种设备抽象思想的工业级实现——文件、控制台、Debugger、编辑器 Output Log 窗口都是设备的实现。Bevy/tracing 的 `Layer` 架构则更进一步：过滤、格式化、输出三者完全解耦，你可以同时挂载控制台 Layer、JSON Layer、Tracy Layer，彼此不干扰。

## 问题 4：日志该怎么格式化？

### 最 naive 的方案

`sprintf` 或 `std::format` 每次拼接字符串：

```cpp
char buffer[1024];
snprintf(buffer, sizeof(buffer), "[%.3f][%s] %s", timestamp, level, msg);
```

这在功能上没问题，但有两个隐藏成本：

1. **栈缓冲溢出风险**：消息太长会截断或越界。
2. **AI 无法直接消费**：如果 Agent 想查询"最近 FPS 低于 30 的帧"，它必须写正则表达式去解析文本，极其脆弱。

### 改进：结构化日志（JSON Lines）

不要只输出文本字符串，要输出**带字段的结构化事件**：

```cpp
// ❌ 旧方式：AI 需要正则解析
LOG_INFO("Render", "FPS: %.2f, draw_calls: %d", fps, draw_calls);

// ✅ 新方式：AI 直接按字段查询
LOG_STRUCT("Render", "frame_stats",
    field("fps", fps),
    field("draw_calls", draw_calls),
    field("frame_time_ms", dt * 1000.0f));
```

输出到 `.jsonl` 文件（每行一个 JSON 对象）：

```json
{"timestamp":1234.56,"level":"Info","category":"Render","event":"frame_stats",
 "fields":{"fps":60.0,"draw_calls":128,"frame_time_ms":16.67},
 "file":"render.cpp","line":256}
```

这让 Agent 可以直接用 `fields.fps < 30` 作为过滤条件，而不必写正则。

> **诚实边界**：结构化日志的体积比纯文本大约 30%~50%。但磁盘空间便宜，解析器脆弱性昂贵。**个人项目从第一天就保留 `.jsonl` 管道**，未来接入 Agent 时无需改造。为人类保留文本 `.log`，为 AI 保留结构化 `.jsonl`，两者并行输出。

> **参考**：Bevy/tracing 原生支持结构化字段，`info!(?secret_message, "Here's a log")` 会把字段以键值对形式嵌入日志。UE TraceLog 则使用自定义二进制协议输出结构化数据，供 UnrealInsights 解析。对于 C++ 个人项目，JSON Lines 是最简单且生态最成熟的结构化方案。

## 问题 5：高频日志下的性能瓶颈怎么破？

### 回到问题 2 的遗留问题

异步双缓冲队列把临界区缩到了最短，但"最短"不等于"零"。当多个工作线程同时产出日志时，`mutex` 的竞争仍然会发生。如果你测到 profiler 里 `std::mutex::lock` 占了帧时间的 2% 以上，就到了升级的时候。

### 改进：TLS 写缓冲

核心思想是**每个线程拥有自己的写缓冲区，完全避免锁竞争**。

```
线程 A TLS：┌─────────────────┐
            │ Cursor          │  <-- 当前写入位置
            │ Committed       │  <-- 已提交给 Worker 的边界
            │ Reaped          │  <-- Worker 已回收的边界
            │ Buffer[64KB]    │
            └─────────────────┘
线程 B TLS：┌─────────────────┐
            │ Cursor          │
            │ Committed       │
            │ Reaped          │
            │ Buffer[64KB]    │
            └─────────────────┘

Worker 线程：定期遍历所有线程的 TLS 缓冲区，
             把 Committed 之前的数据批量收集并输出
```

**三指针状态变化**：

| 指针 | 含义 | 谁修改 |
|------|------|--------|
| `Cursor` | 当前线程的写入位置 | 生产者线程 |
| `Committed` | 已提交给 Worker 的边界 | 生产者线程（批量提交时） |
| `Reaped` | Worker 已回收的边界 | Worker 线程 |

- **写入前**：`Cursor` 指向空闲区起始，`Committed = Reaped`。
- **写入后**：`Cursor` 向前移动 `entry_size` 字节。
- **批量提交时**：`Committed = Cursor`，告知 Worker"这批数据可以读了"。
- **Worker 回收后**：`Reaped = Committed`，缓冲区空间可被复用。

> **诚实的边界**：TLS 写缓冲消除了锁竞争，但引入了**内存开销**（每个线程预分配 64KB）和**延迟**（Worker 定期收集，不是实时输出）。如果你的项目只有 2~3 个线程且日志量不大，双缓冲队列已经足够，不需要升级到 TLS。

> **参考**：UE TraceLog 是 TLS 写缓冲的工业级实现，每个线程通过 `thread_local` 拥有 `FWriteBuffer`，后台 Worker 批量收集。Google Abseil / Folly 的日志也采用类似模式。对于个人项目，建议在**实测到锁竞争是瓶颈**后再引入 TLS，而非 prematurely optimize。

## 问题 6：崩溃时如何不丢最后几条日志？

### 问题背景

异步日志的代价是**延迟**。日志先进入内存缓冲区，再被后台线程刷到文件。如果程序突然 segfault，缓冲区里"最后几条"往往包含最关键的信息——比如"即将解引用空指针"——但它们还没来得及持久化。

### 改进：Emergency Flush

注册一个崩溃处理函数，在程序终止前强制把所有已提交的日志写入文件：

```cpp
void onCrash() {
    // 无锁，直接遍历所有 Sink 并 flush
    for (auto& sink : g_logger.sinks()) {
        sink->emergencyFlush();
    }
}

// 在 main() 中注册
std::set_terminate(onCrash);
// 或信号处理
signal(SIGSEGV, [](int) { onCrash(); std::abort(); });
```

对于 TLS 写缓冲架构，emergency flush 需要把每个线程 `Committed` 之前的数据直接写出，不等待 Worker 线程。

> **参考**：UE TraceLog 的 `Important` 事件标志会额外缓存到独立缓冲区，即使分析工具在引擎启动后才连接也能收到早期事件。Bevy 则通过自定义 panic hook 在崩溃时输出当前 span trace。个人项目的最小可行方案是 `set_terminate` + 强制 flush。

## 问题 7：日志在 ECS 架构中该如何表达？

你的引擎目标是 ECS，所以日志不能永远是一个游离于 ECS 之外的全局单例。

### OOP → ECS 的重构映射

| OOP 概念 | ECS 等价物 | 说明 |
|----------|-----------|------|
| `LogEntry` | `LogEvent`（ECS Event） | 瞬时事件，被 System 消费后自动清除 |
| `Logger` | `LogSink`（Component） | 每个输出目标是一个 Entity，带 `LogSink` 组件 |
| `category` | `LogCategory`（Tag Component） | 用 Tag 标记日志分类，Query 可按 Tag 过滤 |
| `min_level` | `LogFilter`（Resource） | 全局配置，存储在 ECS World 的 Resource 中 |
| `history_` | `LogBuffer`（Component） | ImGui 面板对应的 Entity，持有环形缓冲组件 |

### System 调度图

```
[任意 System 产生日志]
        │
        ▼
[LogEvent Queue]  ←── ECS Event 队列（帧结束时清空）
        │
        ├──► [ConsoleSinkSystem]  → stdout
        ├──► [FileSinkSystem]     → engine.log
        ├──► [JsonSinkSystem]     → engine.jsonl
        └──► [ImGuiBufferSystem]  → 更新 LogBuffer 组件
```

**关键变化**：
- 日志不再是"调用全局函数"，而是"向 ECS World 发送 Event"。
- 每个输出目标是一个独立的 System，可以并行执行（只要它们只读 Event 队列）。
- AI 可以通过 Query `LogEvent` 直接遍历所有日志，无需读文件或访问全局状态。

### 数据流改造步骤

1. **阶段 A（当前）**：`LOG_INFO(cat, msg)` 调用全局 `g_logger.log()`。
2. **阶段 B（引入 ECS Event）**：`LOG_INFO(cat, msg)` 向 `World` 的 `EventWriter<LogEvent>` 写入事件。
3. **阶段 C（完全 ECS 化）**：移除全局 `g_logger`，所有日志消费由 System 在 Schedule 中按阶段执行。

> **迁移时机**：建议在完成 [[最小ECS数据层]] 和 [[事件总线]] 后再执行阶段 B~C。当前阶段 1 用 OOP 单例完全合理。

## 问题 8：AI 如何观察/操作日志系统？

日志是 AI 观察引擎世界的核心窗口。如果 Agent 只能读原始文本，它的理解能力会被严重限制。

### AI 可观测性设计

**状态平铺**：日志历史是一个时间序列数组，每条日志都是自描述的结构体。AI 可以直接遍历 `LogEvent` Query。

**自描述**：`LogEntry` 的结构固定（`level/category/message/timestamp/file/line`），通过反射注册，AI 无需读头文件即可理解。

**确定性**：异步队列 + 主线程批量 flush 保证了日志输出的时序与帧边界对齐。相同输入序列产生相同日志序列。

### AgentBridge 结构化接口

```cpp
// AgentBridge 提供以下 JSON-RPC 接口供 AI 调用
// log.query(category, level, max_count) -> LogEntry[]
// log.subscribe(category, level) -> stream of LogEntry
// log.set_level(category, level) -> bool

// 示例：AI 查询最近 10 条 Render 系统的 Error
{
    "method": "log.query",
    "params": {
        "category": "Render",
        "level": "Error",
        "max_count": 10
    }
}

// 返回结构化 JSON
{
    "result": [
        {"timestamp": 1234.56, "level": "Error", "file": "shader.cpp", "line": 128,
         "msg": "Shader compile failed: missing uniform 'u_MVP'"}
    ]
}
```

**Agent 安全**：AI 只能读取日志和订阅分类，不能直接修改日志历史或伪造日志内容。所有写操作由引擎内部 System 执行。

## 完整最小实现

下面的代码整合了上述所有阶段 1~2 的改进，是一个**真正可以编译运行**的最小实现：

```cpp
#include <vector>
#include <string>
#include <mutex>
#include <queue>
#include <fstream>
#include <iostream>
#include <chrono>
#include <iomanip>
#include <atomic>
#include <memory>
#include <unordered_map>

// ============================================================
// 1. 核心数据结构（ECS 友好：纯数据、无行为）
// ============================================================
enum class LogLevel { Debug, Info, Warning, Error };

struct LogEntry {
    LogLevel    level;
    uint32_t    category_hash;
    const char* message;
    double      timestamp_sec;
    const char* file;
    int         line;
};

// ============================================================

// ============================================================
// 4. 异步日志核心（双缓冲 + Sink 列表）
// ============================================================
class Logger {
    std::mutex mutex_;
    std::vector<LogEntry> queue_a_;
    std::vector<LogEntry> queue_b_;
    std::vector<LogEntry> history_;  // 用于 ImGui 面板
    std::vector<std::unique_ptr<LogSink>> sinks_;
    std::atomic<LogLevel> min_level_{LogLevel::Debug};

public:
    void addSink(std::unique_ptr<LogSink> sink) {
        sinks_.push_back(std::move(sink));
    }

    void setMinLevel(LogLevel level) { min_level_.store(level); }

    void log(LogLevel level, const char* category, const char* msg,
             const char* file, int line) {
        if (static_cast<int>(level) < static_cast<int>(min_level_.load())) return;

        auto now = std::chrono::steady_clock::now();
        auto sec = std::chrono::duration<double>(now.time_since_epoch()).count();

        LogEntry entry{level, 0, msg, sec, file, line};

        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_a_.push_back(entry);
        }
    }

    void flush() {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_a_.swap(queue_b_);
        }

        for (auto& e : queue_b_) {
            for (auto& sink : sinks_) {
                sink->write(e);
            }
            // 保留到历史（上限 4096 条）
            if (history_.size() < 4096) {
                history_.push_back(e);
            } else {
                history_.erase(history_.begin());
                history_.push_back(e);
            }
        }

        for (auto& sink : sinks_) {
            sink->flush();
        }
        queue_b_.clear();
    }

    void emergencyFlush() {
        for (auto& sink : sinks_) {
            sink->emergencyFlush();
        }
    }

    const std::vector<LogEntry>& history() const { return history_; }
    const std::vector<std::unique_ptr<LogSink>>& sinks() const { return sinks_; }
};
```

## 性能与 Trade-off 总结

### 方案演进路线

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| `printf` | 最简单 | 线程不安全、无持久化、无过滤 | 单线程原型（< 1000 行） |
| 异步双缓冲队列 | 线程安全、可持久化、实现简单 | 所有线程竞争同一把锁 | **一般开发调试（推荐默认）** |
| TLS 写缓冲 | 零锁竞争、极高吞吐 | 每个线程预分配内存、非实时输出 | 高频日志、发布版运行 |

> **反例警告**：不要在发布版中保留 `Debug` 级别的日志输出。即使被运行时过滤掉，格式化字符串的参数求值和函数调用开销仍然存在。发布版日志应通过**编译期宏**完全抹除。

### 文本日志 vs 结构化日志

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 纯文本 `.log` | 人类可读、体积小 | AI/机器解析困难 | 人类调试 |
| 结构化 `.jsonl` | AI 可直接解析、可搜索、可过滤 | 体积大约 30%~50% | **AI 观察、自动化测试（推荐并行保留）** |

### 演进路径总结（个人开发推荐）

1. **立即采纳**：编译期过滤 + 运行时原子级别过滤（双层过滤思想）。
2. **第一阶段**：异步双缓冲队列 + Sink 抽象 + 文本/JSON 双端输出。
3. **第二阶段**：当高频日志（>1000 条/帧）成为瓶颈时，升级为 TLS 写缓冲。
4. **始终并行**：为人类保留文本 `.log`，为 AI 保留结构化 `.jsonl`。
5. **崩溃保护**：注册 `set_terminate` + `emergencyFlush`。

> **推荐依据**：个人项目的首要矛盾不是"每秒能写多少条日志"，而是" Shipping 构建不能因为有日志而掉帧"和"AI 能不能读懂日志"。编译期过滤解决前者，结构化输出解决后者。TLS 写缓冲是 beautiful but optional 的优化，只有当你真的测到锁竞争是瓶颈时才需要升级。

## 如果我要 vibe coding，该偷哪几招？

1. **编译期过滤 + 运行时原子过滤**
   - 这是保证 Shipping 构建性能的最小可行方案。`if constexpr` 一行代码就能消除 Shipping 中的 Debug 日志。

2. **异步双缓冲队列 + swap**
   - 保证多线程日志不交错的最小可行方案。不要在工作线程里直接写文件。

3. **用 `__FILE__` 和 `__LINE__` 武装日志宏**
   - 调试时最想知道的就是"这条日志从哪来的"。

4. **日志分级 + 分类过滤**
   - 从第一天就引入 `Debug/Info/Warning/Error` 和 `category`。这是避免日志噪音的唯一办法。

5. **为 AI 保留一条 `.jsonl` 日志流**
   - 即使人类读文本日志，也并行输出 `.jsonl`。未来接入 Agent 时，它可以直接解析字段。

6. **Once 宏**
   - 对于警告类日志，用 `LOG_WARN_ONCE` 避免一帧内重复数千次。

7. **崩溃前 emergencyFlush**
   - 这是救命的设计。崩溃时最后几条日志往往比 stacktrace 更能说明问题。

8. **Sink 抽象而非写死 if-else**
   - 新增输出目标时只需新增一个类，不改 Logger 核心。
