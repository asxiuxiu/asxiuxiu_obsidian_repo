# 窗口与输入系统

## 问题 0：为什么引擎的第一步是"窗口"，而不是"渲染管线"？

很多教程教你先画一个三角形。这很爽，但那个三角形通常跑在一个第三方库帮你隐藏的窗口里——你**还没真正理解**引擎是怎么跟操作系统对话的。

想象你要造一辆车。渲染管线是发动机，但如果没有车身和方向盘（窗口和输入），你连点火测试都没法做。你想调试一个着色器参数，结果引擎是个黑框命令行程序——你改了数值，只能盯着 `printf` 输出的数字盲猜画面变没变。这就是**没有窗口时的世界**。

更实际的问题是：
- 没有窗口 = 没有可视化输出。调整参数后无法观察效果。
- 没有输入 = 无法与引擎交互。想测试相机灵敏度？每次修改都要重新编译、重新运行。
- 没有窗口抽象 = 后续做编辑器时，所有"直接渲染到屏幕"的代码几乎要重写。

所以窗口与输入不是"锦上添花"，它是引擎从 demo 变成**可长期迭代工具**的临界点。

## 问题 1：如何创建一个能显示的窗口？

### 最 naive 的方案：直接调用操作系统 API

如果你只在 Windows 上开发，最直接的做法是：

```cpp
// Windows 专属
#include <windows.h>

int main() {
    HWND hwnd = CreateWindowExW(0, L"MyClass", L"Hello",
        WS_OVERLAPPEDWINDOW, CW_USEDEFAULT, CW_USEDEFAULT,
        800, 600, nullptr, nullptr, hInstance, nullptr);
    ShowWindow(hwnd, SW_SHOW);

    MSG msg;
    while (GetMessage(&msg, nullptr, 0, 0)) {
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }
    return 0;
}
```

这段代码在 Windows 上能跑。但如果你想让 Mac 和 Linux 的用户也能跑你的引擎？Mac 需要写 Cocoa（Objective-C），Linux 需要写 X11 或 Wayland。三套代码、三种消息循环模型、三种坐标系。

**维护三个平台的窗口代码，对个人开发者来说是自杀式行为。**

### 改进：引入 GLFW

GLFW 是一个专门做"窗口 + 输入"的跨平台库。它把 Win32/Cocoa/X11 的复杂性封装在统一的 C API 后面。

```cpp
// flags: -std=c++20 -lglfw
#include <GLFW/glfw3.h>

int main() {
    glfwInit();
    GLFWwindow* window = glfwCreateWindow(800, 600, "Hello Engine", nullptr, nullptr);

    while (!glfwWindowShouldClose(window)) {
        glfwPollEvents();  // 泵取 OS 消息
        // 这里将来会放渲染代码
    }

    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}
```

> [!tip] "泵取 OS 消息"是什么意思？
> **"泵取"**（Pump）来自 Windows 的 **Message Pump**（消息泵）概念。操作系统内部维护着一个**消息队列**——用户的每一次按键、鼠标点击、窗口拖拽、甚至显示器插拔，都会被系统打包成一条"消息"丢进这个队列。
>
> 但队列里的消息不会自动跑到你的程序里来。`glfwPollEvents()` 就像一个**水泵**：它主动到系统队列里把属于当前窗口的消息一条条"抽"出来，翻译成 GLFW 能理解的格式，再分发给你注册的回调函数（或者更新按键状态）。如果没有这个"泵"，你的程序永远收不到用户的输入，窗口也无法响应关闭、拖拽等操作。
>
> 这些消息具体包括：**按键按下/释放**、**鼠标移动/点击/滚轮**、**窗口尺寸变化**、**窗口关闭请求**、**获得/失去焦点**等。

这 10 行代码同时支持 Windows、macOS 和 Linux。GLFW 的选择是经过验证的——它专注只做一件事（窗口管理和输入），代码量仅约 12K 行，不绑架你使用它的渲染器或文件系统（不像 SDL 是个"全能框架"）。

> [!tip] 什么是 SDL？
> **SDL**（Simple DirectMedia Layer，简单直接媒体层）是一套跨平台的多媒体开发库。它把不同操作系统上的窗口创建、音频播放、手柄输入、线程管理等功能封装成统一的 C 接口，让你写一份代码就能在各平台运行。
>
> 你可以把它理解成一个"全能工具箱"——不必自己跟 Windows API 或 macOS Cocoa 打交道，SDL 帮你把这些平台差异都抹平了。代价是它很"重"：代码量约 150K 行，且自带 2D 渲染器，与窗口系统深度耦合。

> **GLFW vs SDL 的取舍**：SDL 功能更全面（音频、线程、文件抽象），但代码量 150K 行，存在"锁定效应"。对于自研引擎，GLFW 的"Unix 哲学"更合适——窗口和输入交给 GLFW，其他模块自己掌控。当你未来需要 SDL 的特定功能（如游戏手柄的高级支持），可以局部引入，不必全盘迁移。

> **为什么不用 SDL？** 因为 SDL 的 2D 渲染器和窗口管理深度耦合，而我们要用 Vulkan 做 3D 渲染。GLFW 从设计之初就是为 OpenGL/Vulkan 服务的，它不会悄悄帮你创建一个你不需要的渲染上下文。

## 问题 2：程序怎么知道"用户按了键"？

### 最 naive 的方案：每帧轮询

GLFW 提供了轮询 API：

```cpp
while (!glfwWindowShouldClose(window)) {
    glfwPollEvents();

    if (glfwGetKey(window, GLFW_KEY_ESCAPE) == GLFW_PRESS) {
        glfwSetWindowShouldClose(window, 1);
    }
}
```

这看起来简单，但有两个致命问题：

**问题 A：按键事件可能丢失**

假设你的游戏跑在 60fps（每帧约 16.6ms）。如果用户快速按了一下空格键——按下和释放都在同一帧内完成——`glfwGetKey` 只能告诉你"这帧按键是按下的"，你完全不知道用户是"刚刚按下"还是"已经按住 5 分钟"。对于跳跃这种需要检测"按下瞬间"的操作，轮询会让你错过时机。

**问题 B：无法区分 Press 和 Release**

`glfwGetKey` 返回的是"当前状态"。你想做"按住 Shift 加速跑"？轮询可以。但你想做"按一下空格跳一次"？轮询会让你每帧都跳，因为只要按键处于按下状态，返回值就不变。

### 改进：事件回调

GLFW 允许你注册回调函数，在按键状态变化时立即通知你：

```cpp
void on_key(GLFWwindow* w, int key, int scancode, int action, int mods) {
    // action == GLFW_PRESS / GLFW_RELEASE / GLFW_REPEAT
    if (key == GLFW_KEY_ESCAPE && action == GLFW_PRESS) {
        glfwSetWindowShouldClose(w, 1);
    }
}

glfwSetKeyCallback(window, on_key);
```

回调精确捕获了"哪个键"、"发生了什么动作"、"什么时候"。快速按键不会丢失，Press/Release 明确区分。

但新的问题又来了：

**问题 C：`GLFW_REPEAT` 怎么处理？**

GLFW 的 `action` 实际上有三种状态：`GLFW_PRESS`、`GLFW_RELEASE` 和 `GLFW_REPEAT`。当你按住空格不放时，操作系统会按固定频率发送"重复"事件——在 Windows 上大约是每秒 30 次。如果你把 `GLFW_REPEAT` 也当成 `GLFW_PRESS` 处理，角色会每隔 33ms 跳一次；如果你忽略它，"按住方向键持续移动"的功能就实现不了。

这意味着输入事件不能只分成"按下"和"释放"两类，必须保留三种状态：
- **Press**：按键从释放变为按下的瞬间（"刚刚按下"）
- **Release**：按键从按下变为释放的瞬间（"刚刚松开"）
- **Repeat**：按键保持按下状态时，操作系统自动重复触发

这三个状态的语义完全不同。Press 用于"按一下跳一次"，Repeat 用于"按住持续移动"，Release 用于"松开刹车"。

## 问题 3：回调直接修改游戏状态，有什么隐患？

上面的回调里，我们直接调用了 `glfwSetWindowShouldClose`。这在简单场景下没问题，但当你的引擎复杂起来后：

**隐患 A：多线程不安全**

GLFW 的回调运行在 OS 的消息线程（在 Windows 上是主线程的消息泵）。如果你的游戏逻辑跑在另一个线程，回调直接修改共享状态就是数据竞争。

**隐患 B：逻辑分散**

每个回调都是孤立的函数。按键处理逻辑散落在各个回调里，找不到统一的入口。你想加一个"全局暂停时忽略所有输入"的逻辑？要在每个回调里加判断。

**隐患 C：与 ECS 的 Tick 模型冲突**

ECS 引擎的核心范式是"System 每帧批量处理组件"。如果输入事件在任意时刻通过回调修改组件，System 就失去了"给定输入序列产生确定性输出"的保证。同一帧内，事件 A 在 System X 之前到达、事件 B 在之后到达，结果就不可复现。

### 改进：原始输入事件队列

核心原则：**窗口层只负责"收集"原始事件，不负责"解释"事件含义。**

```cpp
// 通用事件结构（不泄漏 GLFW 类型到上层）
enum class InputAction { Press, Release, Repeat };

struct RawInputEvent {
    enum Type { Key, MouseButton, MouseMove, MouseScroll, WindowResize, FocusChange, Close } type;
    
    // Key / MouseButton 共用
    int key_code = 0;           // GLFW_KEY_xxx 或 GLFW_MOUSE_BUTTON_xxx
    InputAction action = InputAction::Press;
    
    // MouseMove
    float mx = 0, my = 0;
    
    // MouseScroll
    float scroll_x = 0, scroll_y = 0;
    
    // WindowResize
    int width = 0, height = 0;
    
    // FocusChange
    bool focused = true;
};

std::vector<RawInputEvent> g_input_queue;

// GLFW 回调：只做一件事——把平台事件翻译成通用事件，写入队列
void glfw_key_callback(GLFWwindow*, int key, int, int action, int) {
    InputAction ia = InputAction::Press;
    if (action == GLFW_RELEASE) ia = InputAction::Release;
    else if (action == GLFW_REPEAT) ia = InputAction::Repeat;
    g_input_queue.push_back({RawInputEvent::Key, key, ia});
}

void glfw_mouse_button_callback(GLFWwindow*, int button, int action, int) {
    InputAction ia = (action == GLFW_RELEASE) ? InputAction::Release : InputAction::Press;
    g_input_queue.push_back({RawInputEvent::MouseButton, button, ia});
}

void glfw_cursor_callback(GLFWwindow*, double x, double y) {
    g_input_queue.push_back({RawInputEvent::MouseMove, 0, InputAction::Press, (float)x, (float)y});
}

void glfw_scroll_callback(GLFWwindow*, double x, double y) {
    g_input_queue.push_back({RawInputEvent::MouseScroll, 0, InputAction::Press, 0, 0, (float)x, (float)y});
}

void glfw_resize_callback(GLFWwindow*, int w, int h) {
    g_input_queue.push_back({RawInputEvent::WindowResize, 0, InputAction::Press, 0, 0, 0, 0, w, h});
}

void glfw_focus_callback(GLFWwindow*, int focused) {
    g_input_queue.push_back({RawInputEvent::FocusChange, 0, InputAction::Press, 0, 0, 0, 0, 0, 0, focused == GLFW_TRUE});
}
```

主循环里，每帧统一消费：

```cpp
while (!glfwWindowShouldClose(window)) {
    glfwPollEvents();  // 触发回调，填充队列

    // === 帧边界：统一消费原始输入 ===
    for (const auto& e : g_input_queue) {
        // 这里只收集，不直接修改游戏状态
        // 具体的"Space = Jump"映射在后续 [[事件总线]] 中解决
        if (e.type == RawInputEvent::WindowResize) {
            std::cout << "Resized to " << e.width << "x" << e.height << "\n";
        }
    }
    g_input_queue.clear();
}
```

这个设计的收益：
1. **确定性**：同一帧内的所有输入事件，在 System 执行前已经收集完毕，处理顺序固定。
2. **解耦**：窗口层不懂"Jump"是什么意思，它只报告"Space 被按下了"。语义映射交给上层。
3. **AI 友好**：`RawInputEvent` 是平铺的结构体，AI 可以直接读取队列内容，不需要解析 GLFW 的回调签名。

> [!tip] 为什么用 `clear()` 而不是每帧新建 vector？
> `std::vector::clear()` 只重置大小为 0，**不释放已分配的内存**。这意味着 vector 的容量（capacity）会保留，下一帧重新 `push_back` 时不会触发内存分配。如果每帧都 `g_input_queue = std::vector<RawInputEvent>()`，就会反复分配/释放内存，造成不必要的开销。
>
> 但这个方案有个前提：GLFW 回调和主循环跑在同一线程。如果未来引入多窗口或多线程事件泵，需要用锁或环形缓冲区保护队列。

## 问题 4：窗口大小变了，渲染怎么跟着变？

### 最 naive 的方案：直接通知渲染器

```cpp
void glfw_resize_callback(GLFWwindow*, int w, int h) {
    render_set_viewport(w, h);  // 直接改！
}
```

这在单线程程序里似乎没问题。但现代引擎的渲染通常在独立线程（或至少独立的命令缓冲区提交队列）。当用户拖拽窗口边框时，OS 每秒产生数十个 Resize 事件，而 GPU 可能还在用旧的 SwapChain Image 执行上一帧的渲染命令。

**在 Vulkan 下，这会直接崩溃。**

Vulkan 的 SwapChain 与窗口 Surface 的尺寸严格绑定。当窗口尺寸变化时：
- `vkAcquireNextImageKHR` 可能返回 `VK_ERROR_OUT_OF_DATE_KHR`
- 旧的 `VkImageView` 和 `VkFramebuffer` 必须全部销毁后才能重建 SwapChain
- 如果 GPU 仍在引用旧 SwapChain 的 Image，重建会导致未定义行为

### 改进：pending resize + 帧边界安全处理

```cpp
struct WindowState {
    int width = 800, height = 600;
    int pending_width = 0, pending_height = 0;  // 0 = 无待处理 resize
    bool focused = true;
};

// 注意：当前是单窗口全局变量。未来多窗口时，改用 WindowHandle → WindowState 映射表
WindowState g_window_state;

void glfw_resize_callback(GLFWwindow*, int w, int h) {
    // 回调里只标记"有 resize 待处理"，不立即操作 GPU
    g_window_state.pending_width = w;
    g_window_state.pending_height = h;
}
```

渲染循环的安全点：

```cpp
// 在 render() 的开头检查 pending resize
void render_frame() {
    if (g_window_state.pending_width != 0) {
        // 1. 等待 GPU 完成当前所有工作
        vkDeviceWaitIdle(device);

        // 2. 销毁依赖旧 SwapChain 的所有资源
        destroy_framebuffers();
        destroy_image_views();

        // 3. 重建 SwapChain
        recreate_swapchain(g_window_state.pending_width, g_window_state.pending_height);

        // 4. 重建依赖资源
        create_image_views();
        create_framebuffers();

        g_window_state.width = g_window_state.pending_width;
        g_window_state.height = g_window_state.pending_height;
        g_window_state.pending_width = 0;
        g_window_state.pending_height = 0;
    }

    // 正常渲染...
}
```

> **与 OpenGL 的差异**：OpenGL 的 `glViewport` 和 `SwapBuffer` 是隐式的，窗口 Resize 通常由驱动"自动处理"。Vulkan 的显式设计迫使你在代码中显式处理 Resize，这反而让多线程安全更容易推理——你知道确切哪些资源需要等待、哪些需要重建。

## 问题 5：不同电脑的帧率不同，怎么保证行为一致？

### 最 naive 的方案：固定 sleep

```cpp
while (running) {
    update();
    render();
    Sleep(16);  // 约 60fps
}
```

`Sleep` 的精度通常在 1ms~15ms 之间波动（取决于操作系统调度）。你请求睡 16ms，实际可能睡 12ms 或 20ms。帧率忽高忽低，动画速度也随之忽快忽慢。

更糟的是，如果你的 `update()` + `render()` 本身就耗时 10ms，加上 `Sleep(16)`，总帧时间就是 26ms（约 38fps），而不是你期望的 16ms（60fps）。

### 改进：测量实际耗时，用 DeltaTime 驱动逻辑

```cpp
#include <chrono>
#include <algorithm>

using Clock = std::chrono::high_resolution_clock;

Clock::time_point last_time = Clock::now();

while (!glfwWindowShouldClose(window)) {
    auto now = Clock::now();
    float dt = std::chrono::duration<float>(now - last_time).count();
    last_time = now;

    // 保护：dt 过大时 clamp，防止断点调试导致"穿墙"
    dt = std::clamp(dt, 0.0f, 0.1f);  // 最大允许 100ms（约 10fps）

    glfwPollEvents();

    // 消费输入队列（同上）

    // update(dt) —— 所有位移、动画、物理都乘以 dt
    // render()
}
```

`std::chrono::high_resolution_clock` 在现代平台上精度达到微秒级。有了 `dt`：
- 角色移动从"每帧移 0.1"变成"每秒移 5.0 单位，本次移 5.0 * dt"
- 动画帧从"每 3 帧切一张图"变成"每 0.1 秒切一张图"
- 在 144Hz 显示器和 30Hz 的低端电脑上，角色移动速度一致

> **为什么需要 clamp？** 想象你在调试时打了一个断点，程序暂停了 5 秒钟。恢复执行后，`dt` = 5000ms。如果没有 clamp，角色会瞬间移动 300 个单位——直接穿墙出地图。`clamp(dt, 0.0f, 0.1f)` 把极端值限制在 100ms 以内，即使卡顿也只是"稍微慢动作"，不会物理爆炸。

### 帧率控制：PollEvents vs WaitEvents

GLFW 提供两种事件获取方式，它们的 CPU 占用截然不同：

```cpp
// 方式 A：轮询（游戏运行时常用）
glfwPollEvents();  // 立即返回，无论有没有事件。没有事件时 CPU 空转。

// 方式 B：等待（编辑器/空闲时常用）
glfwWaitEvents();  // 阻塞直到有事件到达。没有事件时 CPU 休眠。
```

**`PollEvents`** 适合游戏主循环：即使没有用户输入，渲染也要继续（角色在行走、粒子在飘动）。但代价是 CPU 100% 占用一个核心。

**`WaitEvents`** 适合编辑器或菜单界面：画面不动时，不需要浪费 CPU 周期。但代价是如果游戏中使用它，动画会在没有输入时完全冻结。

**个人项目推荐**：默认用 `PollEvents`，但在窗口最小化或失去焦点时切换到 `WaitEvents`（配合 `glfwWaitEventsTimeout` 设置一个上限，如 100ms，防止完全睡死）。

```cpp
if (g_window_state.focused && !is_minimized) {
    glfwPollEvents();
} else {
    glfwWaitEventsTimeout(0.1);  // 最多等 100ms，保证还能响应关闭事件
}
```

> **VSync 的取舍**：启用 VSync（垂直同步）后，`glfwSwapBuffers` 会阻塞直到显示器刷新，天然限制帧率到 60Hz/144Hz。这解决了"CPU 空转"问题，但引入了**输入延迟**——你的按键可能要等 16ms 才能反映在下一帧。竞技游戏通常关闭 VSync，用独立的帧率限制器（如 `dt` 累加器）来控制节奏。
>
> 阶段 1 建议**开启 VSync**：简单、省电、不会跑 3000fps 把显卡烧坏。阶段 5 深入渲染管线后再讨论如何平衡延迟与功耗。

## 问题 6：窗口失去焦点时，"粘键"怎么解决？

想象这个场景：用户按住 W 键前进，然后 Alt+Tab 切出去回一条消息。他在窗口外松开了 W 键，再切回游戏时——引擎的回调没有收到 Release 事件（因为焦点已经丢了）。于是引擎认为 W 键一直被按住，角色**不受控制地一直向前走**。

这就是**粘键问题**（Sticky Keys）。它是"窗口焦点丢失 + 输入事件泵在窗口层"的必然产物。

### 检测焦点丢失，清空按键状态

GLFW 提供焦点变化回调：

```cpp
void glfw_focus_callback(GLFWwindow*, int focused) {
    g_input_queue.push_back({RawInputEvent::FocusChange, 0, InputAction::Press, 0, 0, 0, 0, 0, 0, focused == GLFW_TRUE});
}
```

主循环里，消费事件时检测焦点变化：

```cpp
for (const auto& e : g_input_queue) {
    if (e.type == RawInputEvent::FocusChange) {
        g_window_state.focused = e.focused;
        if (!e.focused) {
            // 失去焦点：向所有订阅者发送"所有按键已释放"的虚拟事件
            // 或者让上层 System 自行丢弃失去焦点期间的所有输入
        }
    }
}
```

更简单的策略：在失去焦点时，直接忽略所有输入事件。上层的 `MovementSystem` 在 `!window_focused` 时直接返回，不做任何移动计算。等焦点恢复后，用户需要重新按下方向键才能移动——这比"角色自己跑"好一万倍。

> **为什么不能在回调里直接清空？** 因为回调里你不知道当前有哪些键被按住了。正确的做法是在上层维护一个"当前按键状态表"（`bool keys[512]`），失去焦点时把整张表重置为 false。这个状态表在阶段 6.6 的 Action Mapping 系统中会正式引入，阶段 1 只需要知道这个问题存在即可。

## 问题 7：未来要做编辑器，3D 视口怎么嵌进 UI 面板？

这个问题对于阶段 1 来说还太远，但你需要在阶段 1 就知道**路线选择**，因为它影响你早期的接口设计。

### 路线 A：外部句柄桥接（第三方 UI + 原生窗口嵌入）

编辑器用 Qt/Electron 构建，引擎通过 `attachNativeHandle(void*)` 绑定到 Qt Widget 的底层 OS 句柄。

**优势**：复用 Qt 成熟的布局、Docking、控件库，编辑器开发速度快。
**代价**：3D 视口是"黑盒"原生窗口，无法参与 Qt 的布局流。两套输入路由（Qt 事件循环 vs 引擎事件队列），焦点切换时容易丢事件。每个平台都要处理 Qt 句柄到 Vulkan Surface 的映射。

### 路线 B：自研 UI 框架（UE Slate 模式）

从窗口抽象到控件树、从输入路由到渲染提交，全部自己掌控。3D 视口不是"嵌进去的原生窗口"，而是 UI 树中的一个普通节点。

**优势**：视口与编辑器 UI 完全无缝融合。同一套布局系统、同一套输入路由、同一套渲染管线。PIE/Editor 切换无需重建窗口。
**代价**：需要自研完整的 UI 框架（布局、事件、控件库、样式系统）。Epic 花了数年迭代，UE4 早期 Slate 的 bug 数量远超想象。**个人开发者在阶段 1~3 绝不应该从头写 Slate。**

### 路线 C：RenderTarget 桥接（离屏渲染，推荐）

引擎用 Vulkan 渲染到一张纹理（`VkImage`），第三方 UI 只负责显示这张纹理。视口不再是原生窗口，而是一个绑定纹理的 Image Widget。

**优势**：
- 没有跨窗口坐标映射问题（鼠标坐标直接就是纹理 UV）
- 没有多窗口 Surface 创建的平台化灾难
- 视口尺寸变化不需要重建 SwapChain（只改 RenderTarget 尺寸）
- Qt/ImGui 不需要知道 Vulkan 的存在

**代价**：需要实现离屏渲染管线，比直接渲染到屏幕多一个"纹理 → 屏幕"的 blit 步骤。但这个代价在 Vulkan 下极小（本质就是一次全屏贴图）。

> **参考 UE**：UE 的 Slate 虽然是自研 UI 框架，但它的 `FWidgetRenderer` 也支持将 Slate Widget 渲染到 RenderTarget，用于 3D WidgetComponent。这说明即使是 Slate 体系，RenderTarget 桥接也是重要能力。

> **参考 Bevy**：Bevy 的 UI 系统本身就是 ECS 的一部分——UI 节点是 ECS Entity，通过 `Node`、`Style`、`Parent` 组件表达。3D 场景和 2D UI 共享同一套渲染管线，没有"嵌入"的概念，因为它们是同一个 World 里的不同 Entity。这是长期目标，但阶段 1 不必追求。

**个人开发者默认推荐：路线 C（RenderTarget 桥接）**。阶段 1 先用 GLFW 直接渲染到窗口（最简单），阶段 2~3 引入 `IWindow` 抽象层，阶段 4 开始把主视口改为 RenderTarget 输出，为编辑器做准备。

## 问题 8：怎么干净退出，不泄漏资源？

销毁顺序是 GPU 资源的生死线：

```cpp
// ✅ 正确顺序：先 GPU，再窗口，再 GLFW
// 1. 等待 GPU 完成所有 pending 工作
vkDeviceWaitIdle(device);

// 2. 销毁 GPU 资源（SwapChain、ImageView、Framebuffer、CommandPool）
// ...

// 3. 销毁窗口
glfwDestroyWindow(window);

// 4. 终止 GLFW
glfwTerminate();
```

**反例**：如果你先 `glfwDestroyWindow` 再等待 GPU，`vkDeviceWaitIdle` 可能试图向一个已经销毁的 Surface 查询状态，直接崩溃。

## 阶段 1 的完整代码

把上面的改进组合起来，这是阶段 1 能编译运行的完整代码：

```cpp
// flags: -std=c++20 -O2 -lglfw
#include <GLFW/glfw3.h>
#include <vector>
#include <cstdint>
#include <iostream>
#include <chrono>
#include <algorithm>

// ========== 通用原始输入事件 ==========
enum class InputAction { Press, Release, Repeat };

struct RawInputEvent {
    enum Type { Key, MouseButton, MouseMove, MouseScroll, WindowResize, FocusChange, Close } type;
    int key_code = 0;
    InputAction action = InputAction::Press;
    float mx = 0, my = 0;
    float scroll_x = 0, scroll_y = 0;
    int width = 0, height = 0;
    bool focused = true;
};

std::vector<RawInputEvent> g_input_queue;

// ========== 窗口状态 ==========
struct WindowState {
    int width = 800, height = 600;
    int pending_width = 0, pending_height = 0;
    bool focused = true;
};

// 当前是单窗口全局变量。未来多窗口时，改用 WindowHandle → WindowState 映射表
WindowState g_window_state;

// ========== GLFW 回调 ==========
void glfw_key_callback(GLFWwindow*, int key, int, int action, int) {
    InputAction ia = InputAction::Press;
    if (action == GLFW_RELEASE) ia = InputAction::Release;
    else if (action == GLFW_REPEAT) ia = InputAction::Repeat;
    g_input_queue.push_back({RawInputEvent::Key, key, ia});
}

void glfw_mouse_button_callback(GLFWwindow*, int button, int action, int) {
    InputAction ia = (action == GLFW_RELEASE) ? InputAction::Release : InputAction::Press;
    g_input_queue.push_back({RawInputEvent::MouseButton, button, ia});
}

void glfw_cursor_callback(GLFWwindow*, double x, double y) {
    g_input_queue.push_back({RawInputEvent::MouseMove, 0, InputAction::Press, (float)x, (float)y});
}

void glfw_scroll_callback(GLFWwindow*, double x, double y) {
    g_input_queue.push_back({RawInputEvent::MouseScroll, 0, InputAction::Press, 0, 0, (float)x, (float)y});
}

void glfw_resize_callback(GLFWwindow*, int w, int h) {
    g_window_state.pending_width = w;
    g_window_state.pending_height = h;
}

void glfw_focus_callback(GLFWwindow*, int focused) {
    g_input_queue.push_back({RawInputEvent::FocusChange, 0, InputAction::Press, 0, 0, 0, 0, 0, 0, focused == GLFW_TRUE});
}

// ========== 主循环 ==========
int main() {
    glfwInit();
    
    // 不创建 OpenGL 上下文，为 Vulkan 做准备
    // GLFW 默认会创建一个 OpenGL 上下文，但我们计划用 Vulkan（或 bgfx），
    // 所以需要显式禁用，避免 GLFW 自动创建我们不需要的 GL 上下文
    glfwWindowHint(GLFW_CLIENT_API, GLFW_NO_API);
    
    GLFWwindow* window = glfwCreateWindow(800, 600, "Hello Engine", nullptr, nullptr);

    glfwSetKeyCallback(window, glfw_key_callback);
    glfwSetMouseButtonCallback(window, glfw_mouse_button_callback);
    glfwSetCursorPosCallback(window, glfw_cursor_callback);
    glfwSetScrollCallback(window, glfw_scroll_callback);
    glfwSetWindowSizeCallback(window, glfw_resize_callback);
    glfwSetWindowFocusCallback(window, glfw_focus_callback);

    using Clock = std::chrono::high_resolution_clock;
    Clock::time_point last_time = Clock::now();

    while (!glfwWindowShouldClose(window)) {
        auto now = Clock::now();
        float dt = std::chrono::duration<float>(now - last_time).count();
        last_time = now;

        // 断点保护：dt 过大时 clamp，防止穿墙
        dt = std::clamp(dt, 0.0f, 0.1f);

        // 焦点切换时的事件获取策略
        if (g_window_state.focused) {
            glfwPollEvents();
        } else {
            glfwWaitEventsTimeout(0.1);
        }

        // 消费原始输入
        for (const auto& e : g_input_queue) {
            if (e.type == RawInputEvent::KeyPress && e.key_code == GLFW_KEY_ESCAPE) {
                glfwSetWindowShouldClose(window, 1);
            }
            if (e.type == RawInputEvent::WindowResize) {
                std::cout << "Resized to " << e.width << "x" << e.height << "\n";
            }
            if (e.type == RawInputEvent::FocusChange) {
                g_window_state.focused = e.focused;
                std::cout << (e.focused ? "Window focused\n" : "Window lost focus\n");
            }
        }
        g_input_queue.clear();

        // TODO: 检查 pending resize，重建 SwapChain（在 [[RHI抽象层]] 中实现）
        // TODO: update(dt)
        // TODO: render()

        // 临时：显示 FPS
        static float fps_timer = 0;
        static int frame_count = 0;
        fps_timer += dt;
        frame_count++;
        if (fps_timer >= 1.0f) {
            std::cout << "FPS: " << frame_count << "\n";
            fps_timer = 0;
            frame_count = 0;
        }
    }

    // 干净退出
    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}
```

> **编译指南**：上述代码需要链接 GLFW 库。如果你用 CMake：
> ```cmake
> find_package(glfw3 REQUIRED)
> target_link_libraries(your_target PRIVATE glfw)
> ```
> 如果你手动编译：
> ```bash

> brew install glfw
> g++ main.cpp -std=c++20 -O2 -lglfw -framework Cocoa -framework OpenGL -framework IOKit
>
> # Linux
> sudo apt install libglfw3-dev
> g++ main.cpp -std=c++20 -O2 -lglfw
>
> # Windows (vcpkg)
> vcpkg install glfw3
> cl main.cpp /std:c++20 /O2 glfw3.lib user32.lib gdi32.lib shell32.lib
> ```

这段代码已经解决了阶段 1 的所有核心问题：

## 向 ECS 演进：窗口状态如何表达？

虽然阶段 1 不需要完整 ECS，但你要提前知道"这些数据在 ECS 下长什么样"，避免后期重写。

### 当前代码中的状态 → ECS 映射

| 当前代码 | ECS 表达 |
|---------|---------|
| `g_window_state`（全局变量） | `WindowState` Resource（单例组件） |
| `g_input_queue`（全局向量） | `Events<RawInputEvent>`（每帧自动清空的 Event 队列） |
| 窗口回调函数 | `WindowSystem`（负责 pump 事件、写入 Event 队列） |
| FPS 计数器 | `FpsCounter` Component（附加到 Diagnostics Entity） |
| 按键状态表 | `InputState` Resource（映射 key_code → 当前是否按下） |

### Bevy 的参考模式

Bevy 中：
- 窗口是 ECS Entity，`Window` 组件存储尺寸、位置、模式
- 输入事件是 ECS Event：`KeyboardInput`、`MouseMotion`、`WindowResized`
- System 通过 `EventReader<T>` 读取事件，通过 `ResMut<Window>` 修改窗口状态
- `bevy_winit`（基于 winit crate）负责底层窗口和事件泵

这与我们的设计方向一致：窗口层只管"把 OS 事件翻译成 ECS Event"，System 管"怎么响应"。

> **AI 友好性检查**：`RawInputEvent` 是纯 POD（Plain Old Data）结构体，没有指针、没有虚函数、没有深层嵌套。AI 可以通过 `TypeRegistry` 直接读取其字段列表和内存布局，不需要理解 GLFW 的回调签名。这满足 [[AI友好的引擎架构]] 中的"自描述"和"状态平铺"红线。
