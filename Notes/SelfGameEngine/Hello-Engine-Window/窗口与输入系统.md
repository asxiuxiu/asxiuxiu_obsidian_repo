# 窗口与输入系统

> **前置依赖**：[[AI友好的引擎架构]]
> **本模块增量**：引擎能创建窗口、进入稳定的事件循环、捕获键盘鼠标原始输入，并安全处理窗口 Resize。这是从"黑框调试"迈向"可交互开发"的关键一跃，也是后续 [[RHI抽象层]] 和 [[编辑器框架]] 的硬前提。
> **下一步**：[[可视化日志系统]] — 先让"能看见的日志"就位，再 build 上层。

---

## Why：为什么窗口与输入是引擎的硬骨头？

很多人做引擎的第一步是画一个三角形。这很爽，但问题是：这个三角形通常跑在一个第三方库（如 GLFW）帮你隐藏的窗口里。你**还没真正理解**引擎是如何与操作系统对话的。

### 没有这个系统时，世界有多糟

1. **调试是纯黑盒**
   - 没有窗口意味着没有可视化输出。你调整了一个参数，却不知道是否生效，因为没有视口可以观察。

2. **无法与引擎交互**
   - 想测试相机灵敏度？每次修改都要重新编译、重新运行。一个按键就能微调参数的世界遥不可及。

3. **渲染与编辑器无法桥接**
   - 当你开始写编辑器时，会发现 GLFW 创建的窗口无法直接嵌入到 Qt 的面板里。你之前写的所有"直接渲染到屏幕"的代码，在编辑器模式下几乎要重写一遍。

### 真实应用场景

| 场景 | 没有窗口/输入抽象 | 有窗口/输入抽象 |
|------|------------------|----------------|
| 运行时调试 | 盯着控制台数字盲猜 | 用屏幕面板实时看状态 |
| 参数调优 | 改代码→编译→运行（5分钟一次） | 按方向键实时微调，立刻看到反馈 |
| 编辑器嵌入 | 为编辑器单独写一套渲染入口 | 同一渲染器，通过外部句柄无缝嵌入 |
| AI 观察 | Agent 只能读日志 | Agent 能直接读取窗口状态、注入输入事件验证行为 |

> **核心结论**：窗口与输入系统不是"锦上添花"，它是引擎从 demo 变成**可长期迭代工具**的临界点。

## What：最小能跑的窗口 + 输入长什么样？

下面的代码是一个**真正可以编译运行**的最小实现。它使用 GLFW 做跨平台窗口（因为最简洁），并将原始输入事件写入队列。上层 System 按帧消费，保证确定性。

```cpp
#include <GLFW/glfw3.h>
#include <vector>
#include <cstdint>
#include <iostream>

// ============================================================
// 1. 极简原始输入事件（通用，不依赖 GLFW 类型泄漏到上层）
// ============================================================
struct RawInputEvent {
    enum Type { KeyPress, KeyRelease, MouseMove, WindowResize, Close } type;
    int key_code = 0;
    float mx = 0, my = 0;
    int width = 0, height = 0;
};

std::vector<RawInputEvent> g_input_queue;

// GLFW 回调：把平台事件翻译成通用事件，写入队列
void glfw_key_callback(GLFWwindow*, int key, int, int action, int) {
    g_input_queue.push_back({
        action == GLFW_RELEASE ? RawInputEvent::KeyRelease : RawInputEvent::KeyPress,
        key
    });
}
void glfw_cursor_callback(GLFWwindow*, double x, double y) {
    g_input_queue.push_back({RawInputEvent::MouseMove, 0, (float)x, (float)y});
}
void glfw_resize_callback(GLFWwindow*, int w, int h) {
    g_input_queue.push_back({RawInputEvent::WindowResize, 0, 0, 0, w, h});
}

// ============================================================
// 2. 主循环
// ============================================================
int main() {
    glfwInit();
    GLFWwindow* window = glfwCreateWindow(800, 600, "SGE Window", nullptr, nullptr);
    glfwSetKeyCallback(window, glfw_key_callback);
    glfwSetCursorPosCallback(window, glfw_cursor_callback);
    glfwSetWindowSizeCallback(window, glfw_resize_callback);

    while (!glfwWindowShouldClose(window)) {
        glfwPollEvents();

        // 消费原始输入事件
        for (const auto& e : g_input_queue) {
            if (e.type == RawInputEvent::KeyPress && e.key_code == GLFW_KEY_ESCAPE) {
                glfwSetWindowShouldClose(window, 1);
            }
            if (e.type == RawInputEvent::WindowResize) {
                std::cout << "Resized to " << e.width << "x" << e.height << "\n";
            }
        }
        g_input_queue.clear();

        // 下一篇 [[RHI抽象层]] 将在这里调用：presentFrame(window);
    }

    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}
```

### 最小实现已经解决了什么？

1. **窗口创建与事件泵**：`glfwCreateWindow` + `glfwPollEvents` 构成了最基础的消息循环。
2. **输入与引擎解耦**：GLFW 回调不直接修改游戏状态，而是先写入通用事件队列。上层 System 按帧消费，保证**确定性**。
3. **Resize 可观测**：窗口尺寸变化被捕获为事件，后续渲染系统可以据此调整视口/SwapChain。

### 最小实现还缺什么？

- 没有**自研窗口抽象**，无法嵌入编辑器面板。
- 没有**DeltaTime 计算**，无法保证不同帧率下行为一致。
- 没有**输入映射层**，原始按键无法直接驱动游戏逻辑（这将在 [[事件总线]] 中解决）。
- 没有**多窗口、高 DPI、全屏切换**等工业级能力。

## How：真实引擎的窗口与输入是如何复杂起来的？

### 阶段 1：最小实现 → 能用（自研窗口抽象 + DeltaTime）

**触发原因**：GLFW/SDL 虽好，但无法满足两个核心需求：
1. **编辑器嵌入**：编辑器用 Qt 构建，需要把引擎渲染器绑定到一个**外部传入**的窗口句柄。
2. **平台深度控制**：全屏切换、光标锁定、IME 输入法、高 DPI 感知等细节，第三方库的封装粒度往往不够。

**代码层面的变化**：引入自研的 `Window` 抽象层。

```cpp
class Window {
public:
    virtual void create(const char* title, int w, int h) = 0;
    virtual void attachNativeHandle(void* native_handle) = 0;
    virtual void* getNativeHandle() const = 0;
    virtual void setSize(int w, int h) = 0;
    virtual void pollEvents() = 0;
    virtual bool shouldClose() const = 0;
    virtual void getSize(int& w, int& h) const = 0;
};
```

**两种模式的本质差异**：

| 维度 | 引擎自创建（Game 模式） | 外部句柄桥接（Editor 模式） |
|------|----------------------|--------------------------|
| 窗口生命周期 | 引擎负责创建和销毁 | 外部框架管理 |
| 句柄来源 | 调用 `RegisterClassExW` + `CreateWindow` | 接收 Panel 的 `HWND` |
| 析构行为 | 调用 `DestroyWindow` | **禁止**销毁外部句柄 |
| 全屏切换 | 引擎自由切换窗口样式 | 通常受限于外部容器布局 |

> **关键设计**：用一个 `bool is_external_handle` 标记区分两种模式，避免析构时误杀编辑器面板。

**DeltaTime 的计算**：

```cpp
#include <chrono>

using Clock = std::chrono::high_resolution_clock;

Clock::time_point last_time = Clock::now();
while (!window->shouldClose()) {
    auto now = Clock::now();
    float dt = std::chrono::duration<float>(now - last_time).count();
    last_time = now;

    window->pollEvents();
    // update(dt);
    // render();
}
```

**延迟创建机制**：

编辑器 UI 事件线程不能直接操作 GPU 资源（创建 SwapChain 必须在渲染线程执行）。因此引入 pending 队列：

```
UI 线程调用 attachNativeHandle(native_handle)
    │ 只写入 pending_handle
    │ 不立即创建
    ▼
渲染线程的安全点
    │ 检测到 pending_handle 存在
    │ 执行真正的 attach + createSwapChain
    ▼
窗口与 SwapChain 就绪，开始渲染
```

这个机制消除了**多线程竞态**，是编辑器稳定性的关键。

### 阶段 2：能用 → 好用（输入系统分层 + 窗口状态组件化）

**触发原因**：原始按键/鼠标坐标无法直接驱动游戏逻辑。玩家按的是 `Space`，游戏想要的是 `Jump`；窗口的尺寸、焦点状态需要被 ECS 世界中的系统读取。

**窗口状态的 ECS 表达**：

窗口本身是**操作系统资源**，不适合直接作为 ECS Entity。但窗口的**状态**可以通过组件附加到 World 上：

```cpp
struct WindowViewport {
    int width = 1280, height = 720;
    float dpi_scale = 1.0f;
    bool has_focus = true;
};

struct WindowMode {
    enum Type { Windowed, BorderlessFullscreen, ExclusiveFullscreen } mode;
};
```

**原始输入的 ECS 映射**：

原始输入事件是**瞬态的**，不应该长期存储在组件中。推荐模式：
- 窗口回调将事件写入全局/World 级别的原始事件队列。
- `InputSystem` 在 Tick 开始时消费队列，输出 `MoveInput`、`JumpInput` 等语义组件（详见 [[事件总线]]）。

### 阶段 3：好用 → 工业级（规模与性能）

**触发原因**：单窗口 demo 无法覆盖多显示器、高 DPI、竞技级输入延迟等真实需求。

#### 3.1 多窗口与高 DPI

- **主窗口 + 辅助窗口**：`WindowManager` 维护窗口数组和 ID→窗口的映射表。
- **DPI 感知**：查询 `dpi_scale_factor`，并传播给 UI 系统和渲染管线，确保文字和按钮不会忽大忽小。

#### 3.2 全屏切换的精确还原

窗口化 → 全屏 → 窗口化 的切换中，保存窗口的 `placement`（位置、尺寸、最大化状态），退出时还原。

```cpp
struct WindowPlacement {
    int x, y, width, height;
    bool maximized;
};

saved_placement = getPlacement();   // 进入全屏前
setPlacement(saved_placement);      // 退出全屏后
```

#### 3.3 SwapChain 与窗口 Resize 的安全窗口

当用户拖拽窗口边框改变大小时，底层图形 API 的 BackBuffer 尺寸必须同步变化。但如果在 GPU 仍有未完成的绘制命令时直接 Resize，会导致崩溃。

**安全策略**：
1. 逻辑线程只把新尺寸推入 `pending_resize` 队列。
2. 渲染线程在 `preRender()` 中执行 Resize。
3. Resize 前执行 `Flush`（等待 GPU 完成所有命令）+ `ResetRenderTargets`（释放旧 BackBuffer 引用）。
4. 调用底层 `SwapChain.Resize(width, height)`。
5. 重建 BackBuffer 的包装纹理和渲染目标视图。

## 性能与 Trade-off

### 选择 1：GLFW/SDL vs 自研窗口抽象

| 方案 | 优点 | 缺点 | 适用阶段 |
|------|------|------|----------|
| GLFW/SDL | 跨平台、API 简洁、社区成熟 | 难以嵌入编辑器面板、某些平台细节不可控 | **阶段 1** 最小实现 |
| 自研抽象 | 可控、支持编辑器嵌入、IME/光标可深度定制 | 工作量大、每个平台都要写后端 | **阶段 2+** 正式开发 |

> **反例警告**：如果你预计半年内就要做编辑器，**不要**在前期过度依赖 GLFW。迁移成本远大于一开始就用薄封装 + 平台后端的方案。

### 选择 2：轮询 vs 事件驱动

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 轮询（Polling） | 代码简单、每帧状态一致 | CPU 开销随设备数增加、离散事件可能丢失 | 摇杆、鼠标位置等**连续状态** |
| 事件驱动（Event） | 精确捕获离散事件、延迟低 | 逻辑分散在回调中、与 ECS Tick 模型需桥接 | 按键按下/释放、窗口 Resize 等**离散事件** |

> **诚实的边界**：事件驱动不是"零开销"。将 OS 事件桥接到 ECS 事件队列需要内存分配和拷贝；极端高频输入（如高 DPI 鼠标每秒产生数千个移动事件）可能导致队列膨胀，需要合并或采样。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|----------------|
| **状态平铺** | Window 状态（尺寸、DPI、焦点）以组件形式显式存储；原始输入事件以队列形式暴露。 |
| **自描述** | 窗口属性、原始输入事件结构通过反射注册，AI 不读头文件即可查询。 |
| **确定性** | 异步事件（Resize、窗口创建）全部通过 pending 队列延迟到帧边界处理。 |
| **工具边界** | AgentBridge 提供 `window.resize(w, h)`、`window.setFullscreen(mode)` 等结构化接口。 |
| **Agent 安全** | 输入注入接口只影响原始事件队列，不直接修改任意组件；危险操作（如窗口关闭）需显式确认。 |

## 如果我要 vibe coding，该偷哪几招？

1. **外部句柄桥接**：用 `attachNativeHandle(void*)` + `is_external` 标记，让引擎渲染器能塞进任何编辑器框架。
2. **延迟创建/Resize 队列**：任何涉及 GPU 的窗口操作都推到渲染线程的安全点执行，能避免 90% 的多线程崩溃。
3. **全屏前保存 placement**：一个 `WindowPlacement` 结构体，让玩家切回窗口化时位置和尺寸完全不变。
4. **原始输入与语义输入分离**：窗口层只负责把 OS 事件变成通用队列，映射层交给后续的事件系统。不要让 GLFW 回调直接修改游戏状态。
