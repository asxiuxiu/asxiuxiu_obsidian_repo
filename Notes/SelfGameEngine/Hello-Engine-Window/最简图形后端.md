# 最简图形后端

## 问题 0：为什么有了窗口，屏幕还是黑的？

想象你已经用 GLFW 弹出了一个 800×600 的窗口，标题写着 "Hello Engine"。你运行程序，满怀期待——结果看到的是一块漆黑的窗口，或者更糟，是操作系统默认的灰白色背景。

**窗口和渲染是两个完全不同的东西。**

窗口是操作系统提供的"画布框架"——它给你一块矩形区域、一个标题栏、一个关闭按钮。但操作系统完全不知道你想在这块区域里画什么。它不会自动帮你调用 GPU，不会帮你清屏，更不会帮你画一个三角形。

如果你不做任何渲染相关的事情，这块区域里的像素值就是未定义的——可能是黑的，可能是上一帧窗口残留的垃圾像素，可能是显卡驱动初始化时的默认颜色。

**图形后端**（Graphics Backend）就是负责"让 GPU 把像素写到窗口里"的那一层。它要完成三件事：
1. 跟操作系统申请一个**渲染上下文**（告诉 GPU"我要用这个窗口画图"）
2. 管理**交换链**（Swap Chain，让 GPU 画好的图像最终显示到屏幕上）
3. 提供最起码的绘制能力（哪怕只是清个屏）

没有图形后端，你的引擎就是一个空窗口。后续所有酷炫效果——ImGui 面板、3D 立方体、后处理 Bloom——都建立在这一层之上。

## 问题 1：选哪个图形 API？

### 最 naive 的方案：跟着教程走 OpenGL

OpenGL 的教程最多、社区最大、报错信息最友好。创建一个 OpenGL 上下文只需要几行 GLFW 代码：

```cpp
glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 4);
glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 6);
glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
GLFWwindow* window = glfwCreateWindow(800, 600, "Hello", nullptr, nullptr);
glfwMakeContextCurrent(window);
```

然后你就能调用 `glClearColor` 和 `glClear` 把屏幕染成任意颜色。看起来完美。

但当你继续往下写，三个隐性问题会逐渐暴露：

**问题 A：OpenGL 是"状态机"，状态隐式且全局**

OpenGL 没有一个叫 `RenderPass` 的对象来显式描述"我这帧要做什么"。相反，它维护着一个巨大的全局状态机——当前绑定的顶点缓冲、当前激活的纹理单元、当前启用的混合模式……这些状态散落在各处，互相影响，极难追踪。

当你在阶段 5 写渲染管线时，你会发现自己一直在问："为什么这个纹理没生效？哦，因为上一帧某个材质修改了纹理单元 3 的绑定，而这帧我的着色器恰好也从纹理单元 3 读数据。"

**问题 B：驱动做太多隐式优化，行为不可预测**

同样的 OpenGL 代码，在 NVIDIA 显卡上跑 60fps，在 AMD 上可能掉帧，在集成显卡上可能直接黑屏。驱动在背后做了大量的自动优化——它可能偷偷把你的纹理格式转换了，可能把某个缓冲区从显存挪到了内存。这些隐式行为在多线程环境下尤其致命。

**问题 C：多线程渲染几乎不可能**

OpenGL 的上下文是线程绑定的。一个上下文只能在一个线程里活跃（`glMakeContextCurrent`），而且驱动层面的线程安全由实现厂商决定。当你未来想把渲染提交放到独立线程时，OpenGL 会成为最大的绊脚石。

### 改进方案：Vulkan（长期正确，但阶段 1 太重）

Vulkan 是 OpenGL 的继任者，它的设计哲学是"把控制权交给开发者，把责任也交给开发者"。

在 Vulkan 里，没有隐式状态机。你要显式创建每一个对象：
- `VkInstance` —— Vulkan 运行时实例
- `VkSurfaceKHR` —— 与窗口系统的连接
- `VkPhysicalDevice` —— 显卡硬件
- `VkDevice` —— 逻辑设备（驱动接口）
- `VkSwapchainKHR` —— 交换链（管理屏幕缓冲）
- `VkCommandBuffer` —— 命令缓冲（记录 GPU 指令）
- `VkSemaphore` / `VkFence` —— 同步原语

然后你才能在命令缓冲里记录一条 `vkCmdClearColorImage`，提交到队列，等待 GPU 执行，再把结果呈现到屏幕。

**这需要 600~1000 行代码才能清一次屏。**

对于阶段 1 的个人开发者来说，这个代价太高了。我们的目标是"尽快在屏幕上看见东西"，而不是"在阶段 1 就写出生产级 Vulkan 封装"。

### 中间路线：阶段 1 用 OpenGL，阶段 5 回探升级为 RHI

这是个人项目最务实的路径：

```
阶段 1（现在）：OpenGL 清屏 + ImGui
    │
    ▼
阶段 5（将来）：引入 RHI 抽象层
    ├── 保留 OpenGL 后端作为"简单后端"
    ├── 新增 Vulkan 后端作为"高性能后端"
    └── 上层代码（ECS 渲染系统、材质系统）完全无感知切换
```

这样做的好处是：**阶段 1 的 OpenGL 代码不是白写的**。你清屏、画 ImGui、加载纹理的这些调用，在 RHI 抽象层里会被封装为统一的 `IRenderDevice::Clear()`、`IRenderDevice::DrawIndexed()` 接口。阶段 5 只是"把实现从 OpenGL 换成 Vulkan"，而上层调用方式不变。

> **参考 UE**：UE 的 `FDynamicRHI` 就是这种多后端工厂的工业级实现。D3D12、Vulkan、Metal 都继承自同一个纯虚接口，运行时通过全局指针 `GDynamicRHI` 分发。个人项目不需要这么复杂，但核心思想——"上层不依赖具体 API"——是一样的。

> **参考 Bevy**：Bevy 的 `RenderPlugin` 默认使用 wgpu 作为跨平台渲染后端。wgpu 本身提供了类似 Vulkan 的现代 API，但内部自动处理了很多繁琐的初始化。对于 Rust 生态这是好选择，但 C++ 侧没有同等成熟度的替代品。

**个人开发者默认推荐**：阶段 1 用 **OpenGL 4.6 Core Profile**，理由只有一个——用最少代码最快看到结果。阶段 5 再引入正式的 RHI 抽象层和 Vulkan 后端。

## 问题 2：OpenGL 最小初始化需要做什么？

假设你已经有了 GLFW 窗口（来自 [[窗口与输入系统]]），现在要让 GPU 开始工作。

### Step 1：请求 OpenGL 上下文

创建窗口前，先告诉 GLFW 你要用哪个版本的 OpenGL：

```cpp
glfwInit();
glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 4);
glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 6);
glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
// macOS 需要这行，因为它只支持到 OpenGL 4.1
glfwWindowHint(GLFW_OPENGL_FORWARD_COMPAT, GL_TRUE);

GLFWwindow* window = glfwCreateWindow(800, 600, "Hello Engine", nullptr, nullptr);
glfwMakeContextCurrent(window);
```

`CORE_PROFILE` 的意思是：只启用现代 OpenGL 功能，禁用所有已废弃的兼容层 API（如 `glBegin`/`glEnd` 立即模式）。这强迫你从一开始就使用顶点缓冲、着色器、VAO 等现代做法，避免技术债务。

### Step 2：加载 OpenGL 函数指针

OpenGL 的一个历史包袱是：操作系统只提供旧版本（Windows 默认 OpenGL 1.1）的入口点。要调用 4.6 的新函数，你必须在运行时从显卡驱动里动态加载函数指针。

最流行做法是使用 **GLAD**（或 GLEW）：

```cpp
// 在使用任何 OpenGL 函数前，先初始化 GLAD
if (!gladLoadGLLoader((GLADloadproc)glfwGetProcAddress)) {
    // 加载失败：显卡不支持 OpenGL 4.6，或驱动有问题
    return -1;
}
```

GLAD 本质上是一个代码生成器：你告诉它"我要 OpenGL 4.6 Core + 某些扩展"，它生成一个 `glad.c` 文件，里面包含所有 OpenGL 4.6 函数的函数指针声明和加载逻辑。

> **诚实边界**：GLEW 比 GLAD 更老、更臃肿，而且初始化时会把所有扩展一次性全部加载，启动慢。GLAD 支持按需加载，且代码生成器可以精确控制只生成你需要的版本和扩展。**新项目首选 GLAD**。

### Step 3：清屏 + 交换缓冲

现在你已经可以渲染了。最小渲染循环长这样：

```cpp
while (!glfwWindowShouldClose(window)) {
    glfwPollEvents();

    // 设置清屏颜色（深灰色）
    glClearColor(0.1f, 0.1f, 0.1f, 1.0f);
    glClear(GL_COLOR_BUFFER_BIT);

    // TODO: 在这里画东西

    glfwSwapBuffers(window);
}
```

`glClearColor` 设置 RGBA 清屏颜色。`glClear(GL_COLOR_BUFFER_BIT)` 把颜色缓冲区的每个像素都填充为这个颜色。`glfwSwapBuffers` 把后台缓冲区的内容交换到前台显示。

> **双缓冲是什么？** 为了防止用户看到"画到一半"的画面，GPU 维护了两个缓冲区：**前台缓冲**（正在显示到屏幕）和**后台缓冲**（你正在画）。`glfwSwapBuffers` 的作用就是交换这两个缓冲区的角色。旧的前台变成后台（下一帧覆盖绘制），旧的后台变成前台（显示到屏幕）。这保证了用户永远只看到完整的一帧。

## 问题 3：Vulkan 最小初始化长什么样？

我们不会在这个阶段用 Vulkan 做主后端，但你必须了解它的轮廓——因为阶段 5 的 RHI 抽象层就是围绕 Vulkan 的设计理念构建的。而且，理解 Vulkan 的显式设计，能帮助你理解为什么 OpenGL 的某些行为那么"玄学"。

### Vulkan 清屏的完整链路

```cpp
// 1. 创建 Vulkan Instance（连接 Vulkan 运行时）
VkInstance instance = createInstance();

// 2. 创建 Surface（连接窗口系统）
VkSurfaceKHR surface = createSurface(instance, window);

// 3. 选择物理设备并创建逻辑设备
VkPhysicalDevice gpu = pickPhysicalDevice(instance);
VkDevice device = createLogicalDevice(gpu, surface);

// 4. 创建 SwapChain（管理呈现图像）
VkSwapchainKHR swapchain = createSwapChain(device, surface, width, height);

// 5. 获取 SwapChain 图像和视图
std::vector<VkImage> swapImages = getSwapImages(device, swapchain);
std::vector<VkImageView> swapViews = createImageViews(swapImages);

// 6. 创建 Command Pool 和 Command Buffer
VkCommandPool cmdPool = createCommandPool(device);
VkCommandBuffer cmdBuf = allocateCommandBuffer(device, cmdPool);

// 7. 创建同步原语（Semaphore + Fence）
VkSemaphore imgAvailable = createSemaphore(device);
VkSemaphore renderFinished = createSemaphore(device);
VkFence fence = createFence(device);
```

这是**初始化**。然后每帧还要做：

```cpp
// 8. 从 SwapChain 获取下一帧图像
uint32_t imageIndex;
vkAcquireNextImageKHR(device, swapchain, UINT64_MAX, imgAvailable, VK_NULL_HANDLE, &imageIndex);

// 9. 等待上一帧完成（CPU-GPU 同步）
vkWaitForFences(device, 1, &fence, VK_TRUE, UINT64_MAX);
vkResetFences(device, 1, &fence);

// 10. 记录命令缓冲
vkBeginCommandBuffer(cmdBuf, ...);
//   插入图像布局转换屏障（Undefined → TransferDst）
//   vkCmdClearColorImage(cmdBuf, swapImages[imageIndex], ...)
//   插入图像布局转换屏障（TransferDst → PresentSrc）
vkEndCommandBuffer(cmdBuf);

// 11. 提交命令到队列
VkSubmitInfo submit{...};
//   waitSemaphore = imgAvailable
//   signalSemaphore = renderFinished
//   commandBuffer = cmdBuf
//   fence = fence
vkQueueSubmit(queue, 1, &submit, fence);

// 12. 呈现到屏幕
VkPresentInfoKHR present{...};
//   waitSemaphore = renderFinished
//   swapchain = swapchain
//   imageIndex = imageIndex
vkQueuePresentKHR(queue, &present);
```

同样是清屏，OpenGL 需要 3 行（`glClearColor`、`glClear`、`SwapBuffers`），Vulkan 需要 200 行以上。

**但 Vulkan 的每一行都在回答一个明确的问题**：
- `vkAcquireNextImageKHR`："我要开始画下一帧了，给我一张可用的图像"
- `vkWaitForFences`："上一帧画完了吗？没画完的话 CPU 等一下"
- `VkImageMemoryBarrier`："这张图像现在要从'未定义'状态变成'可被清除'状态"
- `vkQueuePresentKHR`："这张图像画好了，请显示到屏幕上"

OpenGL 把这些全部藏在了驱动里。驱动帮你做了，但你也失去了控制权和可见性。

> **为什么阶段 1 不直接用 Vulkan？** 因为阶段 1 的目标是"弹出窗口，看见颜色"。用 Vulkan 的话，你需要先理解实例、设备、队列族、交换链、命令池、管线屏障……等 20 多个概念，才能看见第一个像素。这在学习曲线上是反人类的。先用 OpenGL 建立直觉（"哦，原来清屏就是填充颜色缓冲区"），再用 Vulkan 理解"驱动在背后到底做了什么"，是更自然的认知路径。

## 问题 4：窗口 Resize 时，渲染资源怎么重建？

这个问题在 [[窗口与输入系统]] 中已经讲过策略（pending resize + 帧边界安全处理），但那时候只讲了"为什么不要立即操作 GPU"。现在图形后端就位了，我们要回答"具体怎么重建"。

### OpenGL 路径（简单）

OpenGL 的 SwapBuffer 是隐式的。窗口 Resize 时，驱动会自动调整后台缓冲区的大小。你只需要在每帧开始时同步视口：

```cpp
// 在主循环里
int width, height;
glfwGetFramebufferSize(window, &width, &height);
glViewport(0, 0, width, height);
```

`glViewport` 告诉 OpenGL"我这帧要画到哪个矩形区域"。即使窗口从 800×600 变成了 1920×1080，这一行代码也能让渲染正确适配。

### Vulkan 路径（显式控制）

Vulkan 的 SwapChain 与窗口 Surface 的尺寸严格绑定。窗口 Resize 时：

1. `vkAcquireNextImageKHR` 可能返回 `VK_ERROR_OUT_OF_DATE_KHR`
2. 你必须**销毁**旧的 SwapChain、ImageView、Framebuffer
3. 用新尺寸**重建** SwapChain
4. 重建所有依赖 SwapChain 图像的资源

这在代码上很繁琐，但逻辑上很清晰——你知道每一步在做什么，不会遇到"驱动偷偷帮你做了某件事但你不确定是什么"的困惑。

> **阶段 1 的务实做法**：用 OpenGL，只需 `glViewport`。阶段 5 引入 Vulkan 后端时，再把完整的 SwapChain 重建逻辑写进 RHI 抽象层。

## 问题 5：多帧在途时，CPU 和 GPU 如何同步？

### 最 naive 的方案：每帧等 GPU

```cpp
while (running) {
    // ... 记录渲染命令 ...
    glfwSwapBuffers(window);
    glFinish();  // ❌ 强制 CPU 等 GPU 完成所有工作
}
```

`glFinish` 会阻塞 CPU，直到 GPU 把队列里的所有命令都执行完。你的游戏循环变成了：CPU 画 → CPU 等 GPU → GPU 画 → 下一帧。如果 GPU 耗时 10ms，CPU 就白白等 10ms。在 144Hz 显示器上，这意味着你最多只能跑到 50~60fps，因为 CPU 和 GPU 无法并行。

### 改进：双缓冲 / 三缓冲 + 显式同步

现代引擎的做法是：**让 CPU 提前准备第 N+1 帧，同时 GPU 正在画第 N 帧**。

```
时间线：
CPU: [准备帧0][准备帧1][准备帧2][准备帧3]...
GPU:          [执行帧0][执行帧1][执行帧2]...
```

这要求你有多套帧资源（命令缓冲、 uniform 缓冲、描述符集），每帧用一套，循环复用。

**OpenGL**：`glfwSwapInterval(1)` 启用 VSync，驱动隐式管理双缓冲。对于阶段 1 足够了。

**Vulkan**：你需要显式创建 `VkFence`（CPU-GPU 同步）和 `VkSemaphore`（GPU-GPU 同步）：
- `vkAcquireNextImageKHR` 返回前，需要 `imageAvailableSemaphore` 信号
- 提交渲染命令时，等待 `imageAvailableSemaphore`，完成后发出 `renderFinishedSemaphore`
- `vkQueuePresentKHR` 等待 `renderFinishedSemaphore`
- `vkWaitForFences` 确保 CPU 不会超前 GPU 超过 FRAME_COUNT 帧

```cpp
// FRAME_COUNT = 2（双缓冲）或 3（三缓冲）
constexpr int MAX_FRAMES_IN_FLIGHT = 2;
VkFence inFlightFences[MAX_FRAMES_IN_FLIGHT];
VkSemaphore imageAvailable[MAX_FRAMES_IN_FLIGHT];
VkSemaphore renderFinished[MAX_FRAMES_IN_FLIGHT];

uint32_t currentFrame = 0;

void drawFrame() {
    // 等上一帧（currentFrame）完成
    vkWaitForFences(device, 1, &inFlightFences[currentFrame], VK_TRUE, UINT64_MAX);
    vkResetFences(device, 1, &inFlightFences[currentFrame]);

    // 获取 SwapChain 图像
    uint32_t imageIndex;
    vkAcquireNextImageKHR(device, swapchain, UINT64_MAX,
        imageAvailable[currentFrame], VK_NULL_HANDLE, &imageIndex);

    // 记录命令...

    // 提交，用 Fence 标记完成
    VkSubmitInfo submit{...};
    submit.pWaitSemaphores = &imageAvailable[currentFrame];
    submit.pSignalSemaphores = &renderFinished[currentFrame];
    vkQueueSubmit(queue, 1, &submit, inFlightFences[currentFrame]);

    // 呈现
    VkPresentInfoKHR present{...};
    present.pWaitSemaphores = &renderFinished[currentFrame];
    present.pImageIndices = &imageIndex;
    vkQueuePresentKHR(queue, &present);

    currentFrame = (currentFrame + 1) % MAX_FRAMES_IN_FLIGHT;
}
```

> **参考 UE**：UE 的渲染线程使用了多层队列同步。Game Thread 通过 `ENQUEUE_RENDER_COMMAND` 把渲染命令投递到 Render Thread，Render Thread 构建 `FRHICommandList`，再由 RHI Thread（可选）翻译成具体 API 调用。Fence 和 Semaphore 的管理隐藏在 `FRHICommandList` 的提交逻辑中，上层无需关心。

> **个人项目推荐**：阶段 1 用 OpenGL + `glfwSwapInterval(1)`，VSync 自动处理 tearing。阶段 5 引入 Vulkan 后，再实现显式的双缓冲 Fence 同步。

## 问题 6：向 ECS 演进，图形后端状态该怎么表达？

阶段 1 的图形后端是一堆全局变量和 GLFW 回调。但你的引擎目标是 ECS，所以你需要提前知道"这些数据在 ECS 下长什么样"。

### 当前代码中的状态 → ECS 映射

| 当前代码 | ECS 表达 | 说明 |
|----------|---------|------|
| `GLFWwindow*`（全局指针） | `WindowHandle` Resource | 单例资源，存储原生窗口句柄 |
| OpenGL/Vulkan 上下文 | `RenderContext` Resource | 单例资源，存储 `VkDevice` / `GLFWwindow* context` |
| SwapChain / 帧缓冲 | `SwapChain` Component | 多视口时可以有多个 SwapChain Entity |
| 当前视口尺寸 `(width, height)` | `Viewport` Component | 附加到主相机 Entity |
| 清屏颜色 | `ClearColor` Component | 附加到 RenderSettings Entity |
| VSync 开关 | `VSync` Component | 附加到 RenderSettings Entity |
| `currentFrame`（双缓冲索引） | `FrameIndex` Resource | 每帧递增，模 MAX_FRAMES_IN_FLIGHT |

### 渲染 System 的调度

```
[WindowSystem]          ──► pump 事件、更新 WindowHandle Resource
    │
    ▼
[AcquireImageSystem]    ──► 调用 vkAcquireNextImageKHR / glfwSwapBuffers 准备
    │
[RenderSceneSystem]     ──► 执行 ECS 渲染查询，生成 DrawCall（阶段 5 实现）
    │
[PresentSystem]         ──► 提交命令队列，呈现到屏幕
```

阶段 1 不需要真的实现这些 System。但你在写 OpenGL 渲染循环时，要有意识地把它想象成"一个 System 调度图"——这会让阶段 5 的迁移更顺畅。

> **参考 Bevy**：Bevy 的渲染管线完全由 ECS System 驱动。`RenderApp` 是一个独立的 ECS World，拥有 `Extract`、`Prepare`、`Queue`、`Render`、`Cleanup` 等固定阶段。每个阶段都是一组 System，通过 `RenderGraph` 组织依赖关系。这是长期目标，但阶段 1 只需要知道"渲染也可以被 ECS 化"这个概念。

## 问题 7：AI 如何观察/操作图形后端？

### 可观测性

图形后端的状态应该能被序列化和查询：

```cpp
struct RenderStats {
    uint32_t frame_count;
    float    frame_time_ms;
    uint32_t swapchain_width;
    uint32_t swapchain_height;
    bool     vsync_enabled;
    const char* backend_name;  // "OpenGL 4.6" / "Vulkan 1.3"
};
```

AI 可以通过 AgentBridge 查询这些状态，而不需要读 C++ 头文件。

### 确定性

给定相同的输入序列（窗口尺寸、清屏颜色、VSync 设置），图形后端的输出必须是确定的：
- 清屏颜色严格按 `ClearColor` 组件的值执行
- 每帧的 `frame_time_ms` 由 `std::chrono` 测量，不依赖 GPU 负载的随机波动
- SwapBuffer 时机由 VSync 设置固定（开启时绑定显示器刷新率）

### 工具边界

AI 不应该直接操作 GPU 资源（如创建纹理、修改着色器）。图形后端的 MCP 接口应该只暴露：
- `render.set_clear_color(r, g, b, a)`
- `render.set_vsync(enabled)`
- `render.query_stats()` → `RenderStats`
- `render.take_screenshot()` → PNG bytes

所有涉及 GPU 资源创建的操作（加载纹理、编译着色器、创建缓冲区）属于阶段 5 的 [[资源管理]] 和 [[RHI抽象层]] 范畴。

## 阶段 1 的完整代码

下面是整合了上述所有改进的阶段 1 完整代码。它能编译运行，弹出一个深灰色窗口，按 ESC 退出，窗口 Resize 时自动适配：

```cpp
#include <GLFW/glfw3.h>
#include <glad/glad.h>  // 需要提前用 GLAD 生成器生成
#include <iostream>
#include <chrono>

// ========== 图形后端状态 ==========
struct GraphicsBackend {
    GLFWwindow* window = nullptr;
    float clear_color[4] = {0.1f, 0.1f, 0.1f, 1.0f};
    bool vsync = true;
};

GraphicsBackend g_graphics;

// ========== 初始化 ==========
bool initGraphics(GLFWwindow* window) {
    g_graphics.window = window;

    // GLAD 加载 OpenGL 函数指针
    if (!gladLoadGLLoader((GLADloadproc)glfwGetProcAddress)) {
        std::cerr << "Failed to initialize GLAD\n";
        return false;
    }

    // 启用 VSync
    glfwSwapInterval(g_graphics.vsync ? 1 : 0);

    // 初始视口
    int w, h;
    glfwGetFramebufferSize(window, &w, &h);
    glViewport(0, 0, w, h);

    std::cout << "OpenGL " << glGetString(GL_VERSION) << "\n";
    std::cout << "GPU: " << glGetString(GL_RENDERER) << "\n";
    return true;
}

// ========== 渲染一帧 ==========
void renderFrame() {
    // 自动适配窗口尺寸
    int width, height;
    glfwGetFramebufferSize(g_graphics.window, &width, &height);
    glViewport(0, 0, width, height);

    // 清屏
    glClearColor(
        g_graphics.clear_color[0],
        g_graphics.clear_color[1],
        g_graphics.clear_color[2],
        g_graphics.clear_color[3]
    );
    glClear(GL_COLOR_BUFFER_BIT);

    // TODO: 在这里画 ImGui（[[ImGui接入与基础面板]]）
    // TODO: 在这里画三角形/立方体（阶段 5）
}

// ========== 干净退出 ==========
void shutdownGraphics() {
    // OpenGL 上下文随窗口销毁自动释放
    g_graphics.window = nullptr;
}

// ========== 主循环（与窗口系统结合） ==========
int main() {
    glfwInit();
    glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 4);
    glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 6);
    glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);

    GLFWwindow* window = glfwCreateWindow(800, 600, "Hello Engine", nullptr, nullptr);
    glfwMakeContextCurrent(window);

    if (!initGraphics(window)) {
        glfwDestroyWindow(window);
        glfwTerminate();
        return -1;
    }

    using Clock = std::chrono::high_resolution_clock;
    Clock::time_point last_time = Clock::now();

    while (!glfwWindowShouldClose(window)) {
        auto now = Clock::now();
        float dt = std::chrono::duration<float>(now - last_time).count();
        last_time = now;

        glfwPollEvents();

        // 消费输入队列（来自 [[窗口与输入系统]]）
        // ...

        renderFrame();

        glfwSwapBuffers(window);

        // 临时 FPS 显示
        static float fps_timer = 0;
        static int frame_count = 0;
        fps_timer += dt;
        frame_count++;
        if (fps_timer >= 1.0f) {
            std::cout << "FPS: " << frame_count << " | "
                      << "Backend: OpenGL " << glGetString(GL_VERSION) << "\n";
            fps_timer = 0;
            frame_count = 0;
        }
    }

    shutdownGraphics();
    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}
```

## 性能与 Trade-off 总结

### OpenGL vs Vulkan 决策树

| 维度     | OpenGL（阶段 1） | Vulkan（阶段 5） |
| ------ | ------------ | ------------ |
| 初始化代码量 | ~30 行        | ~600 行       |
| 多线程渲染  | ❌ 困难         | ✅ 原生支持       |
| 跨平台一致性 | ⚠️ 驱动依赖      | ✅ 规范严格       |
| 调试难度   | ✅ 简单         | ⚠️ 需验证层      |
| 性能天花板  | ⚠️ 驱动开销      | ✅ 低开销        |
| 学习曲线   | ✅ 平缓         | ⚠️ 陡峭        |

**个人项目默认路径**：阶段 1 用 OpenGL，阶段 5 引入 RHI 抽象层时并行支持 OpenGL + Vulkan。

### 为什么阶段 1 不直接上 bgfx？

bgfx 是一个跨平台的渲染抽象层，支持 OpenGL / Vulkan / D3D12 / Metal 多后端。它的 API 比 Vulkan 简单，比 OpenGL 现代。

不选它的原因是：**bgfx 隐藏了太多细节**。阶段 1~2 的目标是"建立直觉"，而 bgfx 的抽象会让你不知道"清屏到底涉及哪些 GPU 操作"。先用原生 OpenGL 看清每个步骤，再用 bgfx（或自研 RHI）做封装，是更扎实的学习路径。
