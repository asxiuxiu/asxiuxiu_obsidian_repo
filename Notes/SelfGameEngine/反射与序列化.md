# 反射与序列化

> **前置依赖**：[[从零开始的引擎骨架]]、[[组件系统架构]]
> **本模块增量**：为 ECS 组件建立运行时自描述能力（反射），并让世界状态可以被保存、加载和跨网络传输（序列化）。这是 AI 观察引擎、编辑器操作实体、游戏存档与网络同步的共同基础。
> **下一步**：[[系统调度与确定性]] — 有数据结构和反射后，需要让系统按正确顺序运转，并保证给定相同输入一定得到相同输出。

---

## Why：为什么游戏引擎需要反射与序列化？

### 没有这个系统时，世界有多糟

在 [[组件系统架构]] 中，你已经能让实体携带组件、让系统批量查询和修改它们。但还有一个根本问题没解决：**引擎自己不知道这些组件里有什么**。

1. **AI 是个瞎子**
   - 你想让 AI Agent 帮你调整所有敌人的血量上限，但 AI 不读 C++ 头文件。它不知道 `Health` 组件有哪些字段、每个字段叫什么、类型是什么。没有运行时自描述能力，AI 无法安全地操作引擎状态。

2. **编辑器得为每个组件手写 UI**
   - 每新增一个组件，程序就要在 Inspector 里手写一套 `if (type == "Health") { drawFloatField("max_hp", ...); }`。组件类型越多，编辑器代码越臃肿，越容易遗漏。

3. **存档就是内存 dump**
   - 保存游戏时只能把整片内存 fwrite 到磁盘。里面夹杂着指针、虚表指针、平台相关的句柄。换一个编译器或架构，存档就打不开了。

4. **网络同步一团糟**
   - 你想同步玩家的位置，却不知道 `Transform` 组件在内存中的布局是否和远端一致。只能手动写死每个字段的读写逻辑，新增一个字段就要改两端代码。

### 三个真实场景

| 场景 | 没有反射/序列化的困难 | 反射+序列化的答案 |
|------|---------------------|------------------|
| **场景 1：AI Agent 查询并修改实体属性** | AI 需要读头文件才能知道 `Health` 有 `max_hp` 字段 | 通过 `TypeRegistry` 运行时查询字段名和类型，AI 用结构化接口安全读写 |
| **场景 2：编辑器显示任意组件的属性面板** | 每个组件都要手写 UI 绘制代码 | Inspector 遍历反射元数据，自动生成 float、bool、enum 的输入框 |
| **场景 3：保存/加载游戏世界，或网络同步组件状态** | 手动为每个组件写二进制读写函数 | 基于反射元数据自动生成序列化代码，新增字段无需改逻辑 |

### 这是"必须"还是"优化"

对于传统引擎，反射可以是"优化"——手写序列化也能做出游戏。但对于 **AI 协作开发（vibe coding）**，反射与序列化是**必须**：

- AI 需要"不读头文件就能理解数据结构" → 只有运行时反射注册表才能做到。
- AI 需要"安全地修改引擎状态" → 只有基于字段元数据的访问器才能避免类型擦除和越界。
- AI 需要"验证改完对不对" → 只有序列化后的状态才能做 diff、保存快照、回滚对比。

> **核心结论**：反射是 AI 的"眼睛"，序列化是 AI 的"手"与"记忆"。没有它们，AI 无法与引擎形成闭环。

## What：最简化版本的反射与序列化长什么样？

下面的代码是一个**真正可以编译运行**的极简反射+序列化系统。相比 [[从零开始的引擎骨架]] 里那个只能写死的 `TypeRegistry`，它增加了四个关键能力：

1. **`FieldDesc`**：描述一个字段的名字、类型、偏移量
2. **`TypeDesc`**：描述一个组件类型有哪些字段
3. **`TypeRegistry`**：全局注册表，运行时按名字查找类型描述
4. **基于反射元数据的 JSON 序列化**：不需要为每个组件手写 `to_json`

```cpp
#include <cstdint>
#include <vector>
#include <string>
#include <unordered_map>
#include <functional>
#include <iostream>
#include <sstream>
#include <type_traits>

// ============================================================
// 1. 反射元数据：让运行时"理解"C++ 结构体
// ============================================================
enum class FieldType {
    Float, Int, Bool, Vec3, String, EntityRef, Unknown
};

struct FieldDesc {
    std::string name;
    FieldType   type;
    size_t      offset;
    size_t      size;
};

struct TypeDesc {
    std::string              name;
    size_t                   size;
    std::vector<FieldDesc>   fields;
};

// 全局类型注册表
class TypeRegistry {
public:
    static TypeRegistry& instance() {
        static TypeRegistry reg;
        return reg;
    }

    void registerType(const TypeDesc& desc) {
        m_types[desc.name] = desc;
    }

    const TypeDesc* find(const std::string& name) const {
        auto it = m_types.find(name);
        return (it != m_types.end()) ? &it->second : nullptr;
    }

private:
    std::unordered_map<std::string, TypeDesc> m_types;
};

// 字段访问器：运行时按偏移量安全读写（最小实现用 void*）
struct FieldAccessor {
    const FieldDesc* desc;

    void* get(void* instance) const {
        return static_cast<char*>(instance) + desc->offset;
    }

    void set(void* instance, const void* value) const {
        std::memcpy(get(instance), value, desc->size);
    }
};

// ============================================================
// 2. 示例组件：Health 和 Transform
// ============================================================
struct Vec3 { float x{0}, y{0}, z{0}; };

struct Health {
    float max_hp{100.0f};
    float current_hp{100.0f};
    bool  is_invincible{false};
};

struct Transform {
    Vec3 position{0, 0, 0};
    Vec3 scale{1, 1, 1};
};

// ============================================================
// 3. 手动注册组件类型（工业级会用宏或代码生成替代）
// ============================================================
#define OFFSET_OF(T, M) offsetof(T, M)

void registerBuiltinTypes() {
    TypeRegistry::instance().registerType({
        "Health", sizeof(Health), {
            {"max_hp",     FieldType::Float, OFFSET_OF(Health, max_hp),     sizeof(float)},
            {"current_hp", FieldType::Float, OFFSET_OF(Health, current_hp), sizeof(float)},
            {"is_invincible", FieldType::Bool, OFFSET_OF(Health, is_invincible), sizeof(bool)}
        }
    });

    TypeRegistry::instance().registerType({
        "Transform", sizeof(Transform), {
            {"position", FieldType::Vec3, OFFSET_OF(Transform, position), sizeof(Vec3)},
            {"scale",    FieldType::Vec3, OFFSET_OF(Transform, scale),    sizeof(Vec3)}
        }
    });
}

// ============================================================
// 4. 基于反射的 JSON 序列化（最小实现）
// ============================================================
std::string serializeToJson(const std::string& type_name, void* instance) {
    const TypeDesc* desc = TypeRegistry::instance().find(type_name);
    if (!desc) return "{}";

    std::ostringstream oss;
    oss << "{\n";
    for (size_t i = 0; i < desc->fields.size(); ++i) {
        const auto& f = desc->fields[i];
        void* ptr = static_cast<char*>(instance) + f.offset;
        oss << "  \"" << f.name << "\": ";
        switch (f.type) {
            case FieldType::Float:  oss << *static_cast<float*>(ptr); break;
            case FieldType::Int:    oss << *static_cast<int*>(ptr); break;
            case FieldType::Bool:   oss << (*static_cast<bool*>(ptr) ? "true" : "false"); break;
            case FieldType::Vec3: {
                Vec3* v = static_cast<Vec3*>(ptr);
                oss << "{" << v->x << ", " << v->y << ", " << v->z << "}";
                break;
            }
            default: oss << "null"; break;
        }
        if (i + 1 < desc->fields.size()) oss << ",";
        oss << "\n";
    }
    oss << "}";
    return oss.str();
}

// ============================================================
// 5. 让 AI 能"看见"和"操作"
// ============================================================
int main() {
    registerBuiltinTypes();

    Health h{200.0f, 150.0f, false};

    // AI 查询：Health 有哪些字段？
    const TypeDesc* desc = TypeRegistry::instance().find("Health");
    std::cout << "Type '" << desc->name << "' has " << desc->fields.size() << " fields:\n";
    for (const auto& f : desc->fields) {
        std::cout << "  - " << f.name << " (type=" << static_cast<int>(f.type) << ")\n";
    }

    // AI 操作：把 current_hp 改成 180
    FieldAccessor accessor{&desc->fields[1]};
    float new_hp = 180.0f;
    accessor.set(&h, &new_hp);
    std::cout << "Modified current_hp = " << h.current_hp << "\n";

    // 序列化：让状态可以被保存和传输
    std::cout << serializeToJson("Health", &h) << "\n";

    return 0;
}
```

### 最小实现已经解决了什么？

1. **运行时自描述**：`TypeRegistry` 让外部（包括 AI）无需读 C++ 头文件就能知道 `Health` 有三个字段。
2. **通用序列化**：`serializeToJson` 不依赖任何组件的具体类型，只依赖反射元数据。新增组件时，只要注册类型描述，就能自动被序列化。
3. **字段安全访问**：`FieldAccessor` 通过编译期确定的 `offset` 和 `size` 读写内存，避免了裸指针转换的未定义行为。

### 还缺什么？

1. **手动注册仍然繁琐**：每个字段都要手写 `OFFSET_OF` 和类型映射。工业级引擎会用宏或构建时代码生成来自动填充注册表。
2. **没有版本兼容**：JSON 里没写类型版本号，字段增删会导致旧存档无法读取。
3. **没有多态/引用解析**：如果组件里存的是 `Entity` 引用或基类指针，最小实现无法自动处理。
4. **二进制效率低**：JSON 文本格式占用空间大、解析慢，不适合网络同步和大型存档。

> 但这 150 行代码已经是**一个能跑通闭环的骨架**。你可以立刻让 AI 查询组件结构、修改字段值、保存世界状态。

## How：真实引擎的反射与序列化是如何一步一步复杂起来的？

### 阶段 1：从手动注册到自动生成（能用）

**触发原因**：手动写 `registerType` 在组件数量超过 20 个后就会变成维护噩梦。字段增删、类型重命名都可能导致注册表与真实结构体不一致。你需要一种"结构体定义即注册"的自动化手段。

#### 1.1 宏驱动的半自动注册

最常见的过渡方案是用宏包裹结构体定义，让编译器在展开时顺便收集字段信息：

```cpp
// 宏在 Debug/编辑器模式下展开为字段描述，在 Release 下展开为空
#define REFLECT_FIELD(type, name) \
    { #name, mapTypeToEnum<type>(), offsetof(CurrentClass, name), sizeof(type) }

#define REFLECT_COMPONENT(name, ...) \
    struct name { __VA_ARGS__ }; \
    inline void register_##name() { \
        using CurrentClass = name; \
        TypeRegistry::instance().registerType({ \
            #name, sizeof(name), { __VA_ARGS__ } \
        }); \
    }

// 使用方式
REFLECT_COMPONENT(Health,
    REFLECT_FIELD(float, max_hp);
    REFLECT_FIELD(float, current_hp);
    REFLECT_FIELD(bool,  is_invincible);
)
```

**宏方案的优缺点**：
- **优点**：零额外构建步骤，所有代码都在头文件里，IDE 能直接跳转。
- **缺点**：语法侵入性强，复杂模板类型（如 `DynamicArray<Handle<Mesh>>`）很难用宏表达；宏展开错误时调试困难。

#### 1.2 构建时代码生成（Annotation-Driven）

当项目规模进一步增长，宏方案会变得笨重。工业级引擎通常采用**代码生成器**：

1. 在头文件里用注解或特殊宏标记需要反射的类和字段：
   ```cpp
   struct Meta(VisibleToEditor, Serialize)
   Health {
       float max_hp;
       float current_hp;
       bool  is_invincible;
   };
   ```

2. 构建时运行一个独立工具（如 `header_parser`），扫描所有头文件，解析出类名、字段名、类型。

3. 生成对应的注册代码文件（通常放在 `generated/` 目录）：
   ```cpp
   // generated/health_reflection.cpp
   static void auto_register_health() {
       TypeRegistry::instance().registerType({
           "Health", sizeof(Health), {
               {"max_hp",      FieldType::Float, offsetof(Health, max_hp),      sizeof(float)},
               {"current_hp",  FieldType::Float, offsetof(Health, current_hp),  sizeof(float)},
               {"is_invincible", FieldType::Bool, offsetof(Health, is_invincible), sizeof(bool)}
           }
       });
   }
   static struct HealthRegistrator { HealthRegistrator() { auto_register_health(); } } s_health_registrator;
   ```

**代码生成 vs 宏**：

| 维度 | 宏驱动 | 代码生成 |
|------|--------|---------|
| 构建复杂度 | 低（纯 C++） | 高（需要 parser + 生成工具链） |
| 表达能力 | 弱（复杂类型难处理） | 强（可解析模板、继承、命名空间） |
| 编译时间 | 宏展开增加编译负担 | 生成代码是 plain C++，编译快 |
| 调试体验 | 宏错误信息晦涩 | 生成代码可直接阅读、断点 |
| 灵活性 | 低 | 高（可一并生成序列化、脚本绑定、网络同步代码） |

> **决策建议**：个人项目或 vibe coding 早期用**宏**快速验证；当组件类型超过 50 个或需要生成脚本绑定时，迁移到**代码生成**。

#### 1.3 从 JSON 到二进制序列化

JSON 适合人类阅读和调试，但对于存档、网络包、资产文件来说太臃肿。阶段 1 的核心工程目标是把序列化格式切换为**紧凑的二进制**。

最小二进制序列化的骨架：

```cpp
class BinarySerializer {
public:
    std::vector<uint8_t> data;

    void write(const void* ptr, size_t size) {
        const uint8_t* bytes = static_cast<const uint8_t*>(ptr);
        data.insert(data.end(), bytes, bytes + size);
    }

    void read(void* ptr, size_t size, size_t& offset) {
        std::memcpy(ptr, data.data() + offset, size);
        offset += size;
    }
};

// 基于 TypeDesc 的通用二进制序列化
template<typename T>
void serializeBinary(BinarySerializer& s, const std::string& type_name, T* instance) {
    const TypeDesc* desc = TypeRegistry::instance().find(type_name);
    if (!desc) return;
    for (const auto& f : desc->fields) {
        void* ptr = static_cast<char*>(instance) + f.offset;
        s.write(ptr, f.size);
    }
}
```

**阶段 1 的关键设计：数据块边界**

工业级二进制格式会给每个对象包裹一个 `data_block_size`：

```cpp
void serializeObject(BinarySerializer& s, const std::string& type_name, void* instance) {
    size_t block_start = s.data.size();
    s.write(uint32_t(0));  // 预留 data_block_size

    // ... 写入字段 ...

    size_t block_end = s.data.size();
    uint32_t block_size = block_end - block_start - sizeof(uint32_t);
    std::memcpy(s.data.data() + block_start, &block_size, sizeof(uint32_t));
}
```

**为什么需要块边界？**
- **快速跳过**：反序列化时如果不知道某个类型的细节，可以直接 `seek` 跳过整个数据块。
- **严格校验**：加载结束后断言 `used_size == data_block_size`，及早发现格式不匹配。
- **向前兼容的基础**：虽然当前严格匹配，但未来可以改为"读取已识别字段，跳过未知字节"。

#### 1.4 POD 数组批量读取优化

对于 `std::vector<float>` 这类 POD 数组，不要逐元素调用序列化器，而是直接 `memcpy` 整片内存：

```cpp
if (is_pod_array) {
    s.write(array.data(), sizeof(T) * array.size());
} else {
    for (auto& elem : array) {
        serializeBinary(s, elem_type_name, &elem);
    }
}
```

这个优化让大量数值数据（如顶点缓冲、骨骼权重）的加载性能接近裸 `memcpy`。

---

### 阶段 2：多态、引用与编辑器桥接（好用）

**触发原因**：二进制格式跑通后，你会遇到三个新的工程问题：
1. 组件里存的是基类指针或 `Entity` 引用，二进制格式不知道怎么创建对象或解析引用。
2. 编辑器需要实时查看和修改组件字段，但你不想为每个组件手写 UI。
3. 资产文件增删字段后，旧存档全部报废，迭代成本极高。

#### 2.1 类型注册表与多态创建

当字段类型是基类指针（如 `Mesh* renderable`）时，序列化器需要在运行时知道"这块数据到底是什么派生类"。解决方案是**全局类型工厂**：

```cpp
using FactoryFunc = std::function<void*()>;

class TypeFactory {
public:
    static TypeFactory& instance() {
        static TypeFactory f;
        return f;
    }

    void registerFactory(const std::string& type_name, FactoryFunc factory) {
        m_factories[type_name] = factory;
    }

    void* create(const std::string& type_name) const {
        auto it = m_factories.find(type_name);
        return (it != m_factories.end()) ? it->second() : nullptr;
    }

private:
    std::unordered_map<std::string, FactoryFunc> m_factories;
};

// 注册时自动填入
template<typename T>
struct AutoFactoryRegistrator {
    AutoFactoryRegistrator(const std::string& name) {
        TypeFactory::instance().registerFactory(name, []() -> void* { return new T(); });
    }
};
```

序列化 `Base*` 字段时：
1. 写入派生类的类型名哈希/字符串。
2. 读取时查询 `TypeFactory`，运行时创建正确类型的对象。
3. 递归序列化/反序列化对象内容。

#### 2.2 引用延迟解析（Handle/EntityRef）

ECS 中组件经常引用其他实体（如 `Entity parent` 或 `Handle<Texture> texture`）。这些引用在二进制流中不能写内存地址，而必须写**可重建的标识符**（如 `EntityID` 或 `FileID`）。

```cpp
struct EntityRef {
    uint32_t target_id{0xFFFFFFFF};
};

// 序列化时只写 target_id
// 反序列化时：
// 1. 读取 target_id
// 2. 把 (this_entity, field_index, target_id) 放入 pending_list
// 3. 等所有对象都创建完后，统一把 target_id 解析为实际 Entity 指针/索引
```

**延迟解析的好处**：
- 支持循环引用（A 引用 B，B 引用 A）。
- 支持异步资产加载（纹理还没从磁盘读完，可以先占位）。
- 网络同步时可以只传 `FileID`，接收端按本地资源表映射为实际句柄。

#### 2.3 Inspector 的自动生成：从反射到 UI

有了 `TypeDesc` 和 `FieldDesc`，编辑器属性面板可以**完全通用化**：

```cpp
void drawInspector(const std::string& type_name, void* instance) {
    const TypeDesc* desc = TypeRegistry::instance().find(type_name);
    if (!desc) return;

    for (const auto& f : desc->fields) {
        void* ptr = static_cast<char*>(instance) + f.offset;
        switch (f.type) {
            case FieldType::Float:  ImGui::DragFloat(f.name.c_str(), static_cast<float*>(ptr)); break;
            case FieldType::Bool:   ImGui::Checkbox(f.name.c_str(), static_cast<bool*>(ptr)); break;
            case FieldType::Vec3:   ImGui::DragFloat3(f.name.c_str(), &static_cast<Vec3*>(ptr)->x); break;
            // ...
        }
    }
}
```

**但这只是"能用"。好用的 Inspector 还需要：**
- **分组与标签**：通过反射注解 `Meta(Category = "Rendering")` 把字段分组显示。
- **范围限制**：`Meta(Range(0, 100))` 让血量拖拽条有上下界。
- **自定义绘制器**：某些字段（如颜色、枚举）需要特殊 UI，可以通过类型名注册自定义绘制回调。

> **ECS 映射注意**：Inspector 操作的是 ECS 组件的**裸数据**。在 ECS 架构下，编辑器不直接操作"对象"，而是批量选中若干实体，然后对每个实体的组件数组做统一修改。修改完成后，需要把对应实体的 `Dirty` 标记位设为 true。

#### 2.4 版本兼容与 Migration

严格匹配的二进制格式在迭代期非常痛苦。阶段 2 需要引入**版本号**和**可选字段**机制：

```cpp
struct TypeVersion {
    std::string type_name;
    uint32_t    version{1};
};

struct VersionedFieldDesc : FieldDesc {
    uint32_t since_version{1};  // 从哪个版本开始有这个字段
    bool     optional{false};   // 缺失时是否使用默认值
};
```

**加载策略**：
- 写入时总是带上 `type_version`。
- 读取时如果磁盘版本 < 内存版本：缺失的 optional 字段使用默认值；非 optional 字段报错或触发 Migration 回调。
- 读取时如果磁盘版本 > 内存版本：跳过未知字段（利用 `data_block_size` 边界）。

**Migration 回调**：
```cpp
TypeRegistry::instance().registerMigration("Health", 1, 2, [](void* old_data, void* new_data) {
    // 从版本 1 迁移到版本 2：新增 regen_rate 字段
    auto* h = static_cast<Health*>(new_data);
    h->regen_rate = 0.0f;  // 默认值
});
```

		### 阶段 3：AI 可观测性、网络裁剪与工业级优化（工业级）

**触发原因**：当引擎进入生产环境，你需要回答三个问题：
1. AI 怎么能**高效地**理解整个世界的状态？（不能每次都把整棵树序列化给 AI）
2. 网络同步时，怎么只传**变化了的数据**？（不能每帧把整个场景广播给所有客户端）
3. 跨平台部署时，怎么保证不同 CPU 架构上的序列化格式一致？

#### 3.1 Schema 导出：AI 的"眼睛"升级

反射系统不仅服务编辑器，更要服务外部 AI Agent。阶段 3 的核心产物是**自动生成的 Schema 文档**，让 AI 在对话上下文里就能理解引擎数据模型：

```json
{
  "components": {
    "Health": {
      "fields": {
        "max_hp": { "type": "float", "default": 100.0, "range": [1, 10000] },
        "current_hp": { "type": "float", "default": 100.0 },
        "is_invincible": { "type": "bool", "default": false }
      }
    },
    "Transform": {
      "fields": {
        "position": { "type": "Vec3", "default": [0, 0, 0] },
        "scale": { "type": "Vec3", "default": [1, 1, 1] }
      }
    }
  }
}
```

这个 Schema 可以被直接写入 `ENGINE_AI.md`，作为 AI 操作引擎的**静态前缀上下文**。当 AI 通过 MCP 调用 `query_entity(entity_id)` 时，返回的组件数据严格遵循这个 Schema，AI 无需猜测字段含义。

#### 3.2 增量序列化与 ChangeLog

不要把整个世界的状态每帧都发给 AI 或网络对端。你需要一个 **ChangeLog（变更日志）**：

```cpp
struct ComponentChange {
    Entity    entity;
    std::string component_type;
    std::string field_name;
    std::vector<uint8_t> old_value;
    std::vector<uint8_t> new_value;
};

class ChangeLog {
public:
    std::vector<ComponentChange> changes;

    void record(Entity e, const std::string& comp, const std::string& field,
                const void* old_val, const void* new_val, size_t size) {
        ComponentChange cc;
        cc.entity = e;
        cc.component_type = comp;
        cc.field_name = field;
        cc.old_value.assign(static_cast<const uint8_t*>(old_val),
                            static_cast<const uint8_t*>(old_val) + size);
        cc.new_value.assign(static_cast<const uint8_t*>(new_val),
                            static_cast<const uint8_t*>(new_val) + size);
        changes.push_back(std::move(cc));
    }

    void clear() { changes.clear(); }
};
```

**AI 上下文压缩**：
- AI 不需要看到"当前状态是什么"，而是"**自上次对话以来，状态发生了什么变化**"。
- 把 ChangeLog 格式化为 Markdown 表格发给 AI，token 消耗大幅降低。
- 如果 AI 需要完整状态，可以提供 `snapshot` 工具让它主动调用。

**网络同步优化**：
- 只序列化 `ChangeLog` 中标记为 `Replicated` 的字段。
- 对高频变化数据（如 Transform）做 delta compression 或量化（把 float 压缩为 16bit 定点数）。

#### 3.3 网络裁剪序列化（Net Saver）

网络包和本地存档对序列化的需求不同。网络序列化需要"裁剪"掉不需要同步的信息：

```cpp
enum class SerializeFlags {
    None       = 0,
    Network    = 1 << 0,  // 只同步标记为 Network 的字段
    SaveGame   = 1 << 1,  // 只保存标记为 Persistent 的字段
    Editor     = 1 << 2,  // 编辑器场景文件：保存所有字段
};

struct FieldDesc {
    std::string name;
    FieldType   type;
    size_t      offset;
    size_t      size;
    uint32_t    flags;  // 该字段在哪些场景下序列化
};
```

例如：
- `Transform::position` 标记 `Network | Persistent | Editor`
- `Health::current_hp` 标记 `Network | Persistent | Editor`
- `Health::last_damage_time` 标记 `Editor`（不存档、不同步，运行时临时计算）

序列化器根据 `flags` 过滤字段，自动生成三种不同的二进制流。

#### 3.4 跨平台与确定性保障

**字节序**：网络同步和存档必须在不同平台（x86、ARM、主机）上保持一致。方案有两种：
1. **统一小端**：所有写入都按小端序，读取时根据平台做字节交换。
2. **统一大端**：同上，只是选择大端作为标准格式。

现代游戏引擎几乎都选择**小端统一**（因为 x86/ARM 默认都是小端，只有少量主机和旧网络协议是大端）。

**内存对齐**：
- 序列化格式中不能包含编译器插入的 padding 字节。
- 方案 A：按字段逐个紧凑写入，忽略结构体对齐。
- 方案 B：定义 `#pragma pack(1)` 的序列化镜像结构体。

**确定性**：
- 所有浮点数运算结果在不同平台上可能略有差异（如 x86 80bit 扩展精度 vs SSE 32bit）。
- 对于需要确定性回放的场景（如网络同步），序列化前应对浮点数做**固定精度量化**（如保留 3 位小数），或完全避免在同步路径上使用浮点逻辑状态。

## ECS 重构映射：如果源码不是 ECS，该怎么迁移？

许多传统引擎的反射系统服务于深继承链的 OOP 对象。当你把这些知识迁移到自研 ECS 引擎时，必须做一次**结构重映射**：

| 源码中的 OOP 概念 | ECS 中的对应物 | 迁移说明 |
|------------------|---------------|---------|
| `Actor` / `GameObject` 基类 | `Entity`（轻量 ID） | 删除所有继承关系，把原基类中的字段拆成独立组件 |
| 继承链上的字段（如 `name`、`guid`、`is_dirty`） | `Name` 组件、`GUID` 组件、`Dirty` 标记组件 | 这些字段现在作为普通组件平铺在 ECS 中，一样可以被反射和序列化 |
| `BaseComponent::getReflectionInstance()` | `System::getReflectionInstance(Entity)` | 编辑器通过 System 查询组件数组，按 Entity ID 构造反射实例 |
| 对象树中的子对象指针（如 `Transform* parent`） | `Parent` 组件 + `EntityRef` | 反射系统把 `EntityRef` 识别为特殊类型，序列化时只写 `target_id`，加载后延迟解析 |
| 虚函数 `Serialize(Archive&)` | 自动生成的 `ComponentSerializer<T>` | 不再在每个类里手写虚函数，而是由代码生成器基于 `TypeDesc` 生成静态特化模板 |
| `SketumBridge` 数据桥 | `ComponentSchema` + `ChangeLog` | ECS 下不需要专门的数据桥对象，Schema 描述组件结构，ChangeLog 记录组件变化 |

### 为什么 ECS 更适合反射与序列化

1. **状态完全平铺**
   - OOP 中对象图深埋指针和虚函数，反射访问器需要逐层解引用。ECS 中所有状态都在组件数组里，`FieldAccessor::set` 直接操作连续内存的固定偏移，性能更高、行为更可预测。

2. **序列化粒度天然可控**
   - OOP 序列化往往以"对象"为单位，不得不序列化大量无关字段。ECS 以"组件类型"为单位，网络同步时只选标记了 `Network` 的组件，存档时只选 `Persistent` 组件，粒度精确到字段。

3. **AI 可直接批量操作**
   - AI 通过 Query 获取所有带 `Health` 的实体，然后批量应用反射修改。OOP 中需要遍历 `std::vector<Actor*>`，逐个虚函数调用来获取反射实例，cache miss 严重。

## AI 友好设计红线检查

在设计和实现反射与序列化系统时，逐条检查以下清单：

| 检查项 | 本模块的满足方式 |
|--------|-----------------|
| **状态平铺** | 所有被反射描述的数据都对应 ECS 组件的字段。`TypeDesc` 不描述指针关系，只描述平铺的数值/向量/EntityRef。 |
| **自描述** | `TypeRegistry` 和自动生成的 `ENGINE_AI.md` Schema 让 AI 无需读头文件即可理解所有组件结构。 |
| **确定性** | 序列化格式统一小端、无 padding、浮点做量化。同一世界快照在不同平台上反序列化后完全一致。 |
| **工具边界** | AI 通过 MCP 调用 `query_entity`、`set_component_field`、`save_snapshot`。输入输出都是基于 Schema 的结构化 JSON。 |
| **Agent 安全** | `FieldDesc` 中的 `flags` 控制字段可写性。核心运行时字段（如内部句柄、缓存指针）标记为 `NoEdit | NoNetwork`，AI 和编辑器都无法误改。 |

## 如果我要 vibe coding，该偷哪几招？

以下是经过验证的、可以立刻用到个人项目中的设计招数：

1. **偷"字段描述即一切"**
   - 用 `FieldDesc`（name + type + offset + size）作为反射的最小原子。所有高级功能（序列化、Inspector、脚本绑定）都建立在这四个字段上。

2. **偷"数据块边界封装"**
   - 每个对象在二进制流中包裹 `data_block_size`。这是版本兼容、快速跳过、严格校验的共同基础。

3. **偷"POD 数组批量 memcpy"**
   - 对 `std::vector<float>` 这类连续 POD 数据不要逐元素序列化。直接 `write(array.data(), sizeof(T) * count)`。

4. **偷"引用延迟解析"**
   - 对 `EntityRef` 和 `Handle` 采用 pending list 机制。反序列化第一阶段读 ID 占位，第二阶段统一解析为实际索引。支持循环引用和异步加载。

5. **偷"ChangeLog 压缩 AI 上下文"**
   - 不要每次对话都把整个世界发给 AI。维护一个 `ChangeLog`，只向 AI 汇报"自上次以来变了什么"。

6. **偷"字段 flags 控制序列化场景"**
   - 用 `Network | Persistent | Editor` 等 flag 标记字段，自动生成三种不同的序列化路径，避免手写三套逻辑。

7. **偷"Schema 作为 AI 静态前缀"**
   - 把 `TypeRegistry` 导出为 JSON Schema，写入 `ENGINE_AI.md`。这是让 AI 在第一次对话前就"懂"你引擎结构的最便宜方式。
