## 问题 0：为什么 GPU 自身的背面剔除不够？

### 场景与根因

想象你的场景里有 10000 棵树，分布在一座山的两侧。相机朝向山的东坡，按理说西坡的 5000 棵树根本不该出现在这帧画面里。

最 naive 的做法是：遍历这 10000 棵树，每棵树发一个 Draw Call，让 GPU 自己处理。反正 GPU 有 Early-Z 和背面剔除，看不见的像素会自动被丢弃，对吧？

**错。** 如果你这样做，会发现空场景能跑 120fps，加了 10000 棵树后掉到 30fps——而 GPU 使用率可能只有 40%。瓶颈不在 GPU，而在 **CPU**。

GPU 的背面剔除和 Early-Z 确实能丢弃不可见的**像素**，但它们发生在管线的**极后端**。在像素被丢弃之前，CPU 已经为这 10000 棵树付出了：
- **遍历开销**：遍历 10000 个实体，读取它们的 `Transform`、`MeshHandle`、`MaterialHandle`（约 0.2ms）。
- **管线状态切换**：每棵树绑定自己的 Vertex Buffer、Index Buffer、PSO、Descriptor Set（约 0.5ms）。
- **命令缓冲写入**：把 10000 条绘制命令写入 Command Buffer（约 2~5ms，取决于 API 和驱动）。
- **矩阵上传**：每棵树上传一个 64 字节的 Model-View-Projection 矩阵（约 0.6ms）。

这些开销加起来，在 16ms 的帧预算里直接吃掉 **3~6ms**——帧率从 60fps 掉到 45fps。

**根因在于：GPU 的剔除发生在顶点着色器之后，太晚了。** CPU 侧已经为了这些不可见物体做了大量的"准备工作"。

所以我们需要在 **CPU 侧** 做可见性判断，而且要在**尽可能早的阶段**做——最好在"遍历实体生成 Draw Call"之前就筛掉大部分不可见物体。越早剔除，后续阶段处理的数据量就越小。

### 一个量化对比

| 阶段 | 10000 实体无剔除 | 10000 实体经视锥剔除后剩 500 | 节省 |
|------|----------------|---------------------------|------|
| 实体遍历 | 10000 次 | 10000 次（剔除本身要遍历） | — |
| Draw Call 录制 | 10000 次 | 500 次 | **95%** |
| PSO 绑定 | 10000 次 | ~50 次（合批后） | **99.5%** |
| 矩阵上传 | 10000 个 | 500 个 | **95%** |

> **表格说明**：上述数字基于 D3D12/Vulkan 的 CPU 开销经验值。剔除的代价是"遍历 10000 次做相交测试"，但测试本身极快（问题 3 会量化），而省下的 Draw Call 录制和状态切换是实实在在的。

但这带来了一个自然的问题："判断可见"不是免费的。我们需要一个**几何体**来代表每棵树的空间占位，再用一个**体积**来代表相机的视野，然后问："这个占位和视野有交集吗？"这就引出了问题 1 和问题 2。

## 问题 1：用什么几何体表示物体的"空间占位"？

### 场景与根因

一棵树的模型可能有 5000 个三角形，形状不规则。如果每次做可见性测试都要拿这 5000 个三角形去和相机视野做相交，测试开销会远大于节省的 Draw Call 开销——剔除反而成了瓶颈。

**根因在于：真实物体的几何太复杂，我们需要一种"简单近似体"来代表它的空间占位，用近似体做快速测试。** 只要近似体在视野外，真实物体一定在视野外；近似体在视野内，真实物体"大概率"在视野内（需要更精细的测试确认，但引擎通常接受这种保守估计）。

### 分支研究

#### 分支 A：AABB（轴对齐包围盒）

**核心思路**：用三个轴（X/Y/Z）上的最小/最大值定义一个与坐标轴对齐的长方体。

```cpp
struct AABB {
    Vec3 min;
    Vec3 max;
};

// 从三角形列表构建
AABB FromTriangles(Span<Vec3> vertices) {
    AABB box;
    box.min = Vec3(FLT_MAX, FLT_MAX, FLT_MAX);
    box.max = Vec3(-FLT_MAX, -FLT_MAX, -FLT_MAX);
    for (const Vec3& v : vertices) {
        box.min = Min(box.min, v);
        box.max = Max(box.max, v);
    }
    return box;
}
```

**适用场景**：绝大多数静态和动态物体。构建简单，测试极快。

**隐藏代价**：
- **旋转后膨胀**。AABB 要求边与坐标轴对齐。一个 45 度旋转的立方体，其 AABB 的对角线长度是原边长的 $\sqrt{3}$ 倍——包围盒里充满了"空空间"。这意味着一个实际上离相机很远的旋转立方体，可能因为 AABB 膨胀而被误判为"需要测试"。
- 旋转后的 AABB 需要重新计算（不能简单旋转 min/max 角点），每帧更新有额外开销。

**失效条件**：物体频繁大幅旋转，且对剔除精度要求极高（如狭窄走廊里的长条形物体斜着放）。

#### 分支 B：OBB（定向包围盒）

**核心思路**：与 AABB 类似，但允许长方体随物体一起旋转，保持紧致贴合。

```cpp
struct OBB {
    Vec3 center;
    Vec3 axes[3];    // 本地坐标系的三个正交轴（单位向量）
    Vec3 extents;    // 沿三个轴的半边长
};
```

**适用场景**：长条形物体（如车辆、飞机、长桌）在旋转后仍需要紧致包围盒。

**隐藏代价**：
- **测试开销大**。OBB-Frustum 测试需要把 Frustum 平面变换到 OBB 的本地空间，或把 OBB 角点变换到世界空间，计算量远高于 AABB。
- **存储开销**。AABB 只需 6 个 float，OBB 需要 15 个 float（center 3 + axes 9 + extents 3），缓存不友好。

**失效条件**：物体数量极大时，OBB 的测试开销会抵消剔除带来的收益。

#### 分支 C：包围球（Bounding Sphere）

**核心思路**：用一个球体包围物体，只需中心点和半径。

```cpp
struct Sphere {
    Vec3 center;
    float radius;
};
```

**适用场景**：快速预筛选。球体-平面测试只需一次点积和一次比较，是**所有包围体中最快的测试**。

**隐藏代价**：
- **紧致性差**。立方体的最小包围球直径是对角线长度，球体里的大量空空间会导致大量误判。一个扁平的地面板块，其包围球的半径是长边的一半，高度方向几乎全部被浪费。

**失效条件**：扁平或细长的物体。球体对这类物体的近似极差。

#### 分支 D：k-DOP（k-Discrete Oriented Polytope）

**核心思路**：用 k 个固定方向的平面包围物体。AABB 是 6-DOP（6 个方向：+X, -X, +Y, -Y, +Z, -Z）。18-DOP 增加了对角线方向，包围更紧致。

**适用场景**：对包围精度要求极高、且物体形状复杂的离线渲染或物理碰撞检测。

**隐藏代价**：
- **测试复杂度随 k 线性增长**。18-DOP 需要 18 次平面测试，比 AABB 的 6 次慢 3 倍。
- **构建复杂**。需要沿每个方向投影所有顶点求极值。

**失效条件**：实时渲染的可见性剔除。k-DOP 的额外精度不值得它的开销。

### 决策分析与推荐

**默认推荐：AABB 为主，包围球做快速预筛选。**

理由如下：

1. **AABB 的测试速度足够快**。问题 2 会展示，AABB-Frustum 测试只需 6 次比较，在 CPU 上约 30~50 周期。
2. **AABB 的存储最紧凑**。6 个 float（24 字节），一个 64 字节缓存行能装 2 个 AABB。这对 ECS 的 SoA 布局非常友好——可以把所有实体的 `min` 存一个数组，`max` 存一个数组，批量读取。
3. **包围球是完美的"预筛选器"**。UE 的 `FConvexVolume::IntersectSphere` 被用作**第一层测试**——如果球体完全在视锥外，直接丢弃；如果球体在视锥内，再用 AABB 做精确测试。这种"球体快速淘汰 + AABB 精确确认"的两层策略，在 SIMD 实现中尤其高效。
4. **OBB 和 k-DOP 的精度优势，在剔除场景下收益有限**。剔除是"保守估计"——宁可多画几个物体，也不能漏掉可见物体。AABB 的膨胀只会导致"多画"（False Positive），不会导致"漏画"（False Negative）。而多画几个物体的 GPU 开销，通常远小于 OBB/k-DOP 增加的 CPU 测试开销。

**ECS 映射**：
- 主 World 组件：`LocalAABB`（模型空间 AABB，静态，构建时计算一次）、`GlobalAABB`（世界空间 AABB，每帧由 `HierarchySystem` 或 `TransformSystem` 更新）。
- 可选组件：`BoundingSphere`（用于快速预筛选）。
- Render World 组件：`RenderAABB`（Extract 阶段从 `GlobalAABB` 复制）。

**遗留问题**：我们有了包围盒，但还需要知道"相机的视野"到底怎么数学化表达。一个 FOV 60°、近裁剪面 0.1m、远裁剪面 1000m 的透视相机，它的"视野体积"是什么形状？如何与 AABB 做相交测试？

## 问题 2：如何数学化表达"相机能看到的空间"？

### 场景与根因

相机有位置、朝向、FOV、宽高比、近裁剪面、远裁剪面。这些参数定义了一个**四棱锥被两个平行平面截断后的几何体**——也就是**视锥体（Frustum）**。

我们需要一种表示法，能快速回答："给定一个 AABB，它是否与这个视锥体相交？" 测试必须在 CPU 上执行，且要足够快以支撑每秒数千到数万次调用。

**根因在于：视锥体的表示方式直接影响相交测试的速度和实现的复杂度。** 不同的表示法适合不同的测试策略。

### 分支研究

#### 分支 A：六个平面（Frustum Planes）

**核心思路**：用六个无限平面（左、右、上、下、近、远）定义视锥体。每个平面用一个方程 $n \cdot p + d = 0$ 表示，其中 $n$ 是朝内的单位法向量，$d$ 是到原点的有符号距离。一个点在视锥体内，当且仅当它位于所有六个平面的"正面"（$n \cdot p + d \geq 0$）。

```cpp
struct Plane {
    Vec3 normal;   // 朝内，已归一化
    float distance;
};

struct Frustum {
    Plane planes[6]; // Left, Right, Bottom, Top, Near, Far
};

// 从 View-Projection 矩阵提取六个平面
Frustum FrustumFromMatrix(const Mat4& vp) {
    Frustum f;
    // 第 i 个平面 = vp 的第 i 行 + 或 - vp 的第 3 行（取决于平面）
    // 具体符号见标准推导
    f.planes[0] = NormalizePlane(vp.Row(3) + vp.Row(0)); // Left
    f.planes[1] = NormalizePlane(vp.Row(3) - vp.Row(0)); // Right
    f.planes[2] = NormalizePlane(vp.Row(3) + vp.Row(1)); // Bottom
    f.planes[3] = NormalizePlane(vp.Row(3) - vp.Row(1)); // Top
    f.planes[4] = NormalizePlane(vp.Row(3) + vp.Row(2)); // Near
    f.planes[5] = NormalizePlane(vp.Row(3) - vp.Row(2)); // Far
    return f;
}
```

**适用场景**：CPU 侧的 AABB 和包围球相交测试。工业级标准做法。

**隐藏代价**：
- 平面提取需要从 View-Projection 矩阵做浮点运算，每帧相机移动时都需要重新提取。
- 六个平面之间没有显式的"角点"信息，如果需要知道视锥的 8 个角点（如用于调试线框绘制），需要额外计算。

#### 分支 B：八个角点（Frustum Corners）

**核心思路**：存储视锥体在裁剪空间（Clip Space）的 8 个角点，变换到世界空间。

```cpp
struct FrustumCorners {
    Vec3 corners[8]; // NBL, NBR, NTL, NTR, FBL, FBR, FTL, FTR
};
```

**适用场景**：需要精确知道视锥几何边界的场景，如调试绘制、阴影贴图的视锥拟合（Cascade Shadow Map 的包围计算）。

**隐藏代价**：
- AABB-Frustum 相交测试用角点做非常麻烦。标准的 Separating Axis Theorem（SAT）需要测试 13 条分离轴，计算量大。
- 不适合高频的可见性剔除。

**失效条件**：逐物体的可见性剔除。角点表示更适合几何运算，不适合快速筛选。

#### 分支 C：直接用投影矩阵做裁剪空间测试

**核心思路**：不提取平面，而是把 AABB 的 8 个角点变换到裁剪空间，检查是否都在 [-w, w] 范围内。

**适用场景**：GPU 顶点着色器中的裁剪（硬件自动做）。

**隐藏代价**：
- CPU 侧做 8 次 Vec4-Mat4 乘法（每次 16 次乘加）比 6 次平面-点测试慢得多。
- 不适合批量剔除。

**失效条件**：CPU 侧的批量可见性测试。

### 关键机制：AABB-Frustum 相交测试

分支 A（六平面法）是 CPU 剔除的工业标准。下面是精确的 AABB-Frustum 测试：

```cpp
// 测试 AABB 是否完全位于某一平面的背面
// 如果在背面 → 整个 AABB 不可见 → 立即返回 false
bool AABBBehindPlane(const AABB& box, const Plane& plane) {
    // 找到 AABB 在平面法向量方向上的"最正面点"（p-vertex）
    // 如果 even p-vertex 都在平面背面，则整个 box 都在背面
    Vec3 p_vertex;
    p_vertex.x = (plane.normal.x > 0) ? box.max.x : box.min.x;
    p_vertex.y = (plane.normal.y > 0) ? box.max.y : box.min.y;
    p_vertex.z = (plane.normal.z > 0) ? box.max.z : box.min.z;
    
    float dist = Dot(plane.normal, p_vertex) + plane.distance;
    return dist < 0; // p-vertex 在背面 → 整个 box 不可见
}

bool FrustumIntersectsAABB(const Frustum& frustum, const AABB& box) {
    for (int i = 0; i < 6; ++i) {
        if (AABBBehindPlane(box, frustum.planes[i]))
            return false; // 被某一个平面完全剔除
    }
    return true; // 与视锥相交或在内部
}
```

**关键洞察**：这个测试是**保守的**。它只能告诉我们"AABB 完全在视锥外"或"AABB 可能与视锥相交"。它不能区分"部分相交"和"完全在内"——但对剔除来说，"完全在外 → 丢弃"是唯一重要的决策。"完全在内"和"部分相交"的物体都会被保留到下一阶段。

### 状态变化图：平面测试的核心逻辑

```
平面法向量 n 指向视锥内部

    n 指向这边（视锥内部）
         ↑
    ─────┼─────  平面
         │
    [AABB]          p-vertex 是 AABB 在 n 方向上的最远点
         
如果 p-vertex 到平面的距离 < 0
→ 整个 AABB 都在平面背面 → 不可见
```

**SIMD 优化**：UE 的 `FConvexVolume::PermutedPlanes` 把 8 个平面（6 个视锥面 + 2 个 padding）打包成 SIMD 友好的结构，一次测试 4 个或 8 个平面。对于大批量剔除，SIMD 版本可以把吞吐提升 2~4 倍。

### 决策分析与推荐

**默认推荐：分支 A（六平面法），并实施 SIMD 优化。**

理由：
1. **六平面法是工业标准**。UE 的 `FConvexVolume`、Bevy 的 `Frustum` 组件、几乎所有图形学教材都采用此方案。
2. **测试复杂度固定**。无论 AABB 在哪，都是 6 次平面测试，没有分支依赖（除了 early-out）。
3. **与包围球天然兼容**。同一个六平面表示可以直接用于球体测试：`if (dot(n, center) + d < -radius) reject`。

**ECS 映射**：
- 主 World：`Camera` 组件包含 `projection`、`view` 矩阵。
- Extract 阶段：`ExtractCameraSystem` 从 `Camera` 的 VP 矩阵提取 `Frustum` 组件，存入 Render World 的 `ExtractedCamera`。
- Render World：`FrustumCullSystem` 读取 `Frustum` Resource，遍历 `Query<(&RenderAABB, Entity)>` 执行测试。

**遗留问题**：六平面测试虽然快，但如果场景有 10 万个实体，即使是 6 次比较 × 10 万 = 60 万次浮点运算，也需要约 2ms。而且 ECS 的组件遍历顺序与空间位置无关——相邻实体在内存中可能相距甚远，缓存行利用率极低。当实体数量超过某个阈值后，逐实体测试本身就会成为瓶颈。如何避免逐实体遍历？

## 问题 3：逐实体视锥测试的瓶颈在哪里？

### 场景与根因

假设你实现了问题 2 的六平面测试，在 Render World 的 `FrustumCullSystem` 中遍历所有可渲染实体：

```cpp
void FrustumCullSystem(
    Query<(Entity, &RenderAABB)> query,
    Res<ExtractedCamera> camera,
    ResMut<ViewVisibleList> visible
) {
    visible->clear();
    for (auto [e, aabb] : query) {
        if (FrustumIntersectsAABB(camera->frustum, aabb)) {
            visible->push_back(e);
        }
    }
}
```

在 5000 个实体的场景中，这完全够用——实测约 0.1~0.3ms。但在 10 万实体的开放世界中，同样的代码可能需要 **2~5ms**，直接吃掉帧预算的 1/3。

**根因有三层**：

1. **O(n) 线性复杂度**。10 万实体就是 10 万次测试。六平面法虽然快，但总量累积不可忽视。
2. **缓存不友好**。ECS 的 `Query<(&RenderAABB, Entity)>` 遍历顺序由 ECS 存储布局决定（Archetype 遍历或 Sparse Set 遍历），这与实体在三维空间中的位置**完全无关**。相机看向东方时，内存中连续读取的实体可能一个在东方、一个在西方、一个在地下——每次读取都是随机访问模式，CPU 缓存行利用率极低。
3. **无空间早期淘汰**。如果相机看向一座城市的一个街区，理论上 99% 的实体应该在 100ms 内被"一眼排除"（因为它们在城市的另一头）。但逐实体测试没有这种"大局观"——每个实体都被单独测试，无法利用"这一片区域全不可见"的信息批量跳过。

### 分支研究

#### 分支 A：线性遍历 + Early-Out

**核心思路**：就是上面的代码。对每个实体做测试，一旦发现不可见立即跳到下一个。

**适用场景**：实体数量 < 5000、实现简单优先、原型验证。

**隐藏代价**：如上述三层根因所示，O(n) + 缓存不友好 + 无空间批量淘汰。

**失效条件**：实体数量 > 1 万、或场景空间分布广。

#### 分支 B：SIMD 批量测试

**核心思路**：一次处理 4 个或 8 个 AABB，用 SSE/AVX 指令同时与 6 个平面做测试。

```cpp
// SSE 版本：一次测试 4 个 AABB 的一个平面
// aabb_min_x[4], aabb_max_x[4] 等是 SoA 布局的数组
__m128 min_x = _mm_load_ps(aabb_min_x + i);
__m128 max_x = _mm_load_ps(aabb_max_x + i);
// 根据平面法向量符号选择 min 或 max
__m128 px = (plane_normal_x > 0) ? max_x : min_x;
// ... 对 y, z 同样处理
// dist = dot(n, p) + d，对 4 个 AABB 同时计算
__m128 dist = _mm_add_ps(
    _mm_add_ps(_mm_mul_ps(nx, px), _mm_mul_ps(ny, py)),
    _mm_mul_ps(nz, pz)
);
dist = _mm_add_ps(dist, _mm_set1_ps(plane_d));
// 如果 dist < 0，对应位设为 0xFFFFFFFF
__m128 mask = _mm_cmplt_ps(dist, _mm_setzero_ps());
```

**适用场景**：CPU 支持 SSE/AVX、实体数量 1~5 万、需要压榨最后一滴 CPU 性能。

**隐藏代价**：
- 需要把 AABB 数据从 AoS（Array of Structs）转为 SoA（Struct of Arrays）布局，增加内存搬运。
- 代码可移植性降低（需要为 x86 ARM 分别实现）。

**失效条件**：实体数量 > 10 万，或 AABB 数据不在连续内存中（如 Sparse Set 存储导致随机访问）。

#### 分支 C：SoA 连续存储 + 预过滤

**核心思路**：在 Extract 阶段，不只提取单个实体的 AABB，而是把所有可见实体的 AABB 打包到一个连续的 SoA 数组中。这样 SIMD 批量测试可以连续读取，缓存命中率高。

```cpp
struct AABBSoA {
    Array<float> min_x, min_y, min_z;
    Array<float> max_x, max_y, max_z;
};

// Extract 阶段打包
void ExtractAABBSoASystem(Query<(&RenderAABB)> q, ResMut<AABBSoA> soa) {
    soa->clear();
    for (auto [aabb] : q) {
        soa->min_x.push_back(aabb.min.x);
        soa->min_y.push_back(aabb.min.y);
        soa->min_z.push_back(aabb.min.z);
        soa->max_x.push_back(aabb.max.x);
        soa->max_y.push_back(aabb.max.y);
        soa->max_z.push_back(aabb.max.z);
    }
}
```

**适用场景**：配合 SIMD 批量测试使用，解决缓存不友好问题。

**隐藏代价**：Extract 阶段增加了内存拷贝和额外内存分配。

### 决策分析与推荐

**默认推荐：分支 A 为起点，分支 B/C 为中期优化，长期必须引入空间加速结构（问题 4）。**

理由：
1. **线性遍历在 < 5000 实体时是"足够好的"**。0.1~0.3ms 的开销在 16ms 帧预算内可以忽略。Bevy 的默认 CPU culling 就是线性遍历，已在大量项目中验证。
2. **SIMD 优化是"锦上添花"，不是"雪中送炭"**。它能提升 2~4 倍吞吐，但无法解决 O(n) 的根本问题。从 2ms 降到 0.5ms 很好，但如果实体增长到 50 万，SIMD 也会崩溃。
3. **真正的解法是空间加速结构**。只有当实体按空间组织后，才能把 O(n) 降到 O(log n) 或 O(1)（平均）。问题 4 将深入讨论。

**量化参考**：
- 5000 实体，线性遍历，六平面法：~0.15ms
- 5000 实体，SSE 批量，SoA 布局：~0.04ms
- 10 万实体，线性遍历：~3ms（ unacceptable ）
- 10 万实体，SSE 批量：~0.8ms（仍偏高）
- 10 万实体，BVH 加速：~0.1~0.3ms（ acceptable ）

> **注意**：上述数据为经验估算，实际取决于 CPU 型号、内存带宽、编译器优化级别。"此为推演，待实际 benchmark 验证"。

**遗留问题**：线性遍历的瓶颈是"没有利用空间局部性"。如果相机只看城市的一个街区，我们不该测试城市另一头的建筑。如何把实体按三维空间预先分组，让测试从"逐个体"变成"逐区域"？

## 问题 4：如何把实体按空间组织起来，避免逐实体测试？

### 场景与根因

你的场景是一座城市：1000 栋建筑、5000 辆汽车、20000 个街道设施，分布在 10km × 10km 的区域内。相机站在城市中心的一个十字路口，只能看到周围 200m 的范围。

逐实体测试意味着你要对 26000 个实体做 Frustum-AABB 测试，其中 95% 的实体在 1km 之外——它们的位置一眼就能看出"不可能看见"，但 CPU 仍然要逐个测试。

**根因在于：空间位置是一种天然的"索引键"，但我们没有建立这种索引。** 就像在一本没有目录的电话簿里找人——你只能从头翻到尾。

### 分支研究

#### 分支 A：均匀网格（Uniform Grid）

**核心思路**：把世界空间切成等大小的立方体格子（Cell）。每个实体根据其 AABB 中心点或角点落入对应的格子。测试时，只遍历与视锥相交的格子，测试格子内的实体。

```cpp
struct UniformGrid {
    Vec3 world_min;      // 网格覆盖范围
    Vec3 cell_size;      // 每个格子的大小
    IVec3 resolution;    // 网格分辨率 (Nx, Ny, Nz)
    Array<Array<Entity>> cells; // 每个格子存储实体列表
};

// 插入实体
void Insert(UniformGrid* grid, Entity e, const Vec3& pos) {
    IVec3 cell = Clamp((pos - grid->world_min) / grid->cell_size, IVec3(0), grid->resolution - 1);
    int index = cell.x + cell.y * grid->resolution.x + cell.z * grid->resolution.x * grid->resolution.y;
    grid->cells[index].push_back(e);
}

// 查询与视锥相交的格子
void QueryFrustum(const UniformGrid& grid, const Frustum& frustum, Array<Entity>* out) {
    // 遍历所有格子（或做 Bresenham / 3D DDA 遍历）
    for (int z = 0; z < grid.resolution.z; ++z) {
        for (int y = 0; y < grid.resolution.y; ++y) {
            for (int x = 0; x < grid.resolution.x; ++x) {
                AABB cell_aabb = ComputeCellAABB(grid, x, y, z);
                if (FrustumIntersectsAABB(frustum, cell_aabb)) {
                    // 格子与视锥相交，测试其中的所有实体
                    for (Entity e : grid.cells[CellIndex(grid, x, y, z)]) {
                        out->push_back(e);
                    }
                }
            }
        }
    }
}
```

**适用场景**：实体分布相对均匀、场景范围有限、动态物体多（插入/删除快）。

**隐藏代价**：
- **内存开销大且不均匀**。如果城市有一条密集的街道和一个空旷的公园，街道格子里可能有 500 个实体，公园格子里只有 2 个。最坏情况下一个格子爆炸（如大型集会场景），遍历该格子的开销退化为逐实体测试。
- **大物体跨格子问题**。一栋摩天大楼的 AABB 可能跨越数十个格子，需要插入到所有重叠的格子中，存储冗余。
- **网格分辨率调参困难**。格子太大 → 每个格子里实体太多，失去加速意义；格子太小 → 大物体跨几十个格子，插入和删除开销大。

**失效条件**：实体分布极不均匀、或有大范围跨度物体。

#### 分支 B：八叉树（Octree）

**核心思路**：递归细分三维空间。根节点覆盖整个场景，如果某个节点内的实体数量超过阈值，就将其沿 X/Y/Z 轴一分为八，生成 8 个子节点。实体只存储在完全包含它的最细粒度节点中（或存储在所有相交的叶节点中）。

```cpp
struct OctreeNode {
    AABB bounds;
    Array<Entity> entities; // 仅叶节点存储，或少量时存储在内节点
    OctreeNode* children[8]; // 内节点有子节点，叶节点为 nullptr
    bool is_leaf;
};

// 递归查询
void QueryOctree(OctreeNode* node, const Frustum& frustum, Array<Entity>* out) {
    if (!FrustumIntersectsAABB(frustum, node->bounds))
        return; // 整个分支被剔除
    
    if (node->is_leaf) {
        for (Entity e : node->entities)
            out->push_back(e);
    } else {
        for (int i = 0; i < 8; ++i) {
            if (node->children[i])
                QueryOctree(node->children[i], frustum, out);
        }
    }
}
```

**适用场景**：大规模开放场景、实体分布不均匀、需要 LOD 或远距离剔除（可配合节点层级做距离裁剪）。

**隐藏代价**：
- **树深度不均**。密集区域可能细分到 10+ 层，空旷区域只有 2 层。遍历时的递归深度不可预测，可能导致栈溢出或缓存不友好。
- **动态物体更新昂贵**。一个移动物体可能需要从旧叶节点删除，插入到新叶节点，路径上所有节点的实体计数都要更新。最坏情况下每帧 O(log n) 的节点操作 + 内存分配/释放。
- **内存碎片化**。每个节点是独立分配的对象（指针跳跃），缓存局部性差。

**失效条件**：动态物体极多、或需要极致缓存效率的场景。

#### 分支 C：BVH（Bounding Volume Hierarchy）

**核心思路**：不是先切分空间再分配实体，而是先把实体分组，再用包围盒包裹每组。每个节点的包围盒是其子节点包围盒的并集。树是自底向上或自顶向下构建的，节点的空间划分天然贴合实体的实际分布。

```cpp
struct BVHNode {
    AABB bounds;
    union {
        struct { BVHNode* left; BVHNode* right; }; // 内节点
        struct { Entity* entities; uint32_t count; }; // 叶节点
    };
    bool is_leaf;
};

// 递归查询 —— 与 Octree 结构类似，但分裂策略完全不同
void QueryBVH(BVHNode* node, const Frustum& frustum, Array<Entity>* out) {
    if (!FrustumIntersectsAABB(frustum, node->bounds))
        return;
    
    if (node->is_leaf) {
        for (uint32_t i = 0; i < node->count; ++i)
            out->push_back(node->entities[i]);
    } else {
        QueryBVH(node->left, frustum, out);
        QueryBVH(node->right, frustum, out);
    }
}
```

**适用场景**：静态物体占主体、需要极致查询效率、内存预算有限。

**隐藏代价**：
- **构建开销**。BVH 构建需要排序和分组，对于 10 万实体可能需要 10~100ms。静态场景可以预构建；动态场景每帧重建不可接受。
- **动态更新困难**。与八叉树类似，动态物体插入/删除需要重新计算路径上的包围盒，最坏情况可能触发局部重建。
- 不像八叉树那样天然支持"按距离裁剪"（Octree 的层级可直接对应 LOD 距离）。

**失效条件**：场景极度动态、或需要频繁大范围重构。

#### 分支 D：场景图层级包围盒（Scene-Graph BVH）

**核心思路**：利用 ECS 或场景图（Scene Graph）中天然的父子层级关系。父节点的包围盒是其所有子节点包围盒的并集。这样，场景图本身就是一棵 BVH。

```cpp
// 场景图节点天然携带层级包围盒
struct SceneNode {
    AABB local_aabb;
    AABB world_aabb;      // 子树包围盒的并集
    Array<SceneNode*> children;
};

// 更新世界包围盒（自底向上）
void UpdateWorldAABB(SceneNode* node) {
    node->world_aabb = TransformAABB(node->local_aabb, node->local_transform);
    for (auto* child : node->children) {
        UpdateWorldAABB(child);
        node->world_aabb = Merge(node->world_aabb, child->world_aabb);
    }
}
```

**适用场景**：已经有父子层级结构的场景（如骨骼模型、建筑组合、车辆及其挂载部件）。

**隐藏代价**：
- 依赖场景的层级结构质量。如果层级是扁平的（所有实体都是根节点的子节点），则退化为逐实体测试。
- 自底向上更新包围盒需要遍历整棵树，每帧 O(n)。

**失效条件**：场景扁平、或层级关系与空间关系不一致（如按功能分组而非按空间分组）。

### 状态变化图：均匀网格 vs BVH 的空间组织差异

```
均匀网格（先切空间，再塞物体）:
┌───┬───┬───┐
│ A │   │ B │   物体 A 和 B 可能很接近，但被格子边界切开
├───┼───┼───┤   每个格子大小固定
│   │ C │   │
├───┼───┼───┤
│ D │   │ E │
└───┴───┴───┘

BVH（先分组物体，再算包围盒）:
       [ABCDE]
       /     \
    [ABC]   [DE]
    /   \    /  \
  [AB]  [C] [D] [E]
  /  \
[A]  [B]

物体的分组由空间接近度决定，包围盒贴合实际分布
```

### 引擎对照

> **参考：chaos / UE / Bevy 对「空间加速结构」是怎么做的？**
>
> - **chaos** 从源码推断采用了**混合策略**：静态世界几何（地形、建筑）使用预构建的空间加速结构（可能是 BVH 或均匀网格的变体）；动态物体（角色、车辆）单独维护列表，每帧暴力测试或插入到简化的动态网格中。这种双轨制的动机是：静态数据占渲染实体的 80% 以上，值得预构建加速结构；动态数据量少但更新频繁，不适合复杂结构。
> - **UE** 的渲染线程中，静态网格通过 `FStaticMeshSceneProxy` 参与剔除，其底层使用基于场景管理的加速结构（具体实现随版本变化，早期是均匀网格，后期转向更紧凑的 BVH 变体）。对于地形，`FLandscapeComponent` 使用四叉树（Quadtree）做 LOD 和裁剪。对于 Nanite 虚拟几何，UE5 使用基于 Cluster 的 BVH 做 GPU-Driven 剔除——这在下篇会深入。
> - **Bevy** 在 0.19 版本中**默认不使用空间加速结构做视锥剔除**。`queue_material_meshes` System 直接线性遍历 `Query<(&Aabb, &GlobalTransform)>`，用 `Aabb::intersects_frustum` 测试。Bevy 的动机是：ECS 的 Query 遍历本身已经相当快（cache friendly），对于中小规模场景，空间加速结构的构建和维护开销可能超过其收益。但 Bevy 支持插件化的扩展路径，用户可自行接入 `bevy_spatial` 等第三方 crate。

### 决策分析与推荐

**默认推荐：阶段 5 初期用线性遍历（分支 A 的朴素版），阶段 5 中期引入双轨制——静态物体用 BVH（分支 C），动态物体用单独列表 + 球体预筛选。**

理由如下：

1. **Bevy 的线性遍历验证了"简单优先"在中小规模场景的可行性**。对于 < 1 万实体的场景，线性遍历 + SIMD 已经足够。过早引入 BVH 会增加不必要的架构复杂度。

2. **BVH 对静态场景是性价比极高的中期优化**。构建一次，每帧遍历。对于开放世界的地形、建筑等静态物体，BVH 可以将剔除复杂度从 O(n) 降到 O(log n)。推荐的构建策略是 Top-down SAH（问题 5 详述）。

3. **双轨制是工业级引擎的共识**。chaos 和 UE 都采用"静态加速结构 + 动态列表"的混合策略。动态物体（角色、车辆、粒子）数量通常远少于静态物体（建筑、地形、植被），单独维护一个 `Array<DynamicEntity>` 每帧做暴力测试是可接受的。

4. **均匀网格和八叉树在通用性上不如 BVH**。均匀网格的内存开销和跨格子问题使其更适合特殊场景（如粒子系统或 2D 游戏）；八叉树的动态更新成本和内存碎片化在 C++ 实现中较难控制。BVH 的内存布局可以做成连续数组（问题 5 会展示），缓存效率更高。

**具体实施策略**：

```cpp
// 阶段 5 初期：线性遍历（直接从 ECS Query 遍历）
void FrustumCullSystem(Query<(Entity, &RenderAABB)> q, ...) {
    for (auto [e, aabb] : q) {
        if (FrustumIntersectsAABB(frustum, aabb))
            visible->push_back(e);
    }
}

// 阶段 5 中期：双轨制
void FrustumCullSystemV2(
    Query<(Entity, &RenderAABB)> dynamic_query,  // 动态物体
    Res<StaticBVH> static_bvh,                    // 静态 BVH
    Res<ExtractedCamera> camera,
    ResMut<ViewVisibleList> visible
) {
    visible->clear();
    
    // 1. 遍历静态 BVH
    QueryBVH(static_bvh->root, camera->frustum, &visible->entities);
    
    // 2. 暴力测试动态物体（数量少）
    for (auto [e, aabb] : dynamic_query) {
        if (FrustumIntersectsAABB(camera->frustum, aabb))
            visible->push_back(e);
    }
}
```

**遗留问题**：BVH 听起来很好，但"构建一次"具体怎么做？如果把 10 万个三角形的静态场景建成 BVH，该按什么规则分组？动态物体如果很多，有没有比"暴力列表"更好的方案？

## 问题 5：BVH 如何构建？动态物体怎么办？

### 场景与根因

你决定为城市的静态建筑引入 BVH。这些建筑不会移动，但数量有 5000 栋，每栋建筑的 AABB 大小和位置各不相同。简单粗暴地"前 2500 个放左子树，后 2500 个放右子树"会生成一棵极不平衡的树——左子树全是市中心的高层建筑（AABB 很大），右子树全是郊区的平房（AABB 很小）。测试时，左子树的巨大包围盒几乎总是与视锥相交，导致大量无效遍历。

**根因在于：BVH 的查询效率取决于树的"质量"——包围盒的紧致度和树的平衡度。** 一棵高质量的 BVH 能让"不可见分支"在根节点附近就被剔除；一棵劣质的 BVH 会让测试穿透到深层叶节点才发现不可见。

### 分支研究

#### 分支 A：Top-Down SAH 构建

**核心思路**：自顶向下递归分裂。每次选择一个**分割轴**（X/Y/Z），在轴上找一个**分割点**，把实体分成左右两组。选择分割点的方式是**最小化 Surface Area Heuristic（SAH）成本**。

SAH 的核心假设是：一条随机光线（或视锥测试射线）与某个节点相交的概率，正比于该节点包围盒的**表面积**。因此，好的分割应该让"左子树的表面积 × 左子树实体数 + 右子树的表面积 × 右子树实体数"最小。

```cpp
// 简化版 SAH 构建
BVHNode* BuildBVH(Span<EntityAABB> entities, int depth) {
    if (entities.count <= LEAF_SIZE || depth >= MAX_DEPTH) {
        return CreateLeaf(entities);
    }
    
    // 1. 计算当前节点的包围盒
    AABB node_bounds = ComputeBounds(entities);
    
    // 2. 选择最长的轴作为分割轴
    Vec3 extent = node_bounds.max - node_bounds.min;
    int axis = (extent.x > extent.y && extent.x > extent.z) ? 0 :
               (extent.y > extent.z) ? 1 : 2;
    
    // 3. 按中心点沿分割轴排序
    Sort(entities, [axis](const EntityAABB& a, const EntityAABB& b) {
        return a.center[axis] < b.center[axis];
    });
    
    // 4. 找 SAH 最优分割点（简化版：取中位数）
    int mid = entities.count / 2;
    
    // 5. 递归构建
    BVHNode* node = new BVHNode;
    node->bounds = node_bounds;
    node->left = BuildBVH(entities.Slice(0, mid), depth + 1);
    node->right = BuildBVH(entities.Slice(mid, entities.count - mid), depth + 1);
    node->is_leaf = false;
    return node;
}
```

**适用场景**：静态场景预构建、离线工具（如编辑器中的场景预处理）。

**隐藏代价**：
- **排序开销**。每次递归都要排序，总复杂度 O(n log² n)。对于 10 万实体，构建可能需要 50~200ms。
- **内存分配**。每个节点独立 `new`，内存碎片化，缓存不友好。
- **中位数分割不是真正的 SAH 最优**。真正的 SAH 需要在当前轴上扫描所有可能的分割位置（Binned SAH），计算量更大。

**优化：Binned SAH**。不扫描所有分割点，而是把轴分成 16 个桶（bin），只评估桶边界。复杂度降到 O(n log n)，且质量接近最优 SAH。

#### 分支 B：连续内存 BVH（Linear BVH / LBVH）

**核心思路**：利用 Morton Code（Z-Order Curve）把三维空间位置映射为一维排序键。实体按 Morton Code 排序后，相邻实体在空间中也相邻。然后自底向上或自顶向下构建，树可以用连续数组存储，无需指针。

```cpp
// Morton Code 把 (x,y,z) 编码为一个 30-bit 整数（每轴 10 bit）
uint32_t MortonCode3(Vec3 pos_normalized) {
    uint32_t x = ExpandBits10((uint32_t)(pos_normalized.x * 1023.0f));
    uint32_t y = ExpandBits10((uint32_t)(pos_normalized.y * 1023.0f));
    uint32_t z = ExpandBits10((uint32_t)(pos_normalized.z * 1023.0f));
    return x * 4 + y * 2 + z; // 交错位
}

// LBVH 构建流程
// 1. 计算每个实体中心点的 Morton Code
// 2. 按 Morton Code 排序
// 3. 相邻实体自动形成叶节点和内节点
// 4. 整棵树存储在一个 Array<BVHNode> 中，用索引代替指针
```

**适用场景**：需要极致构建速度和缓存效率的大规模静态场景。GPU 上构建 BVH（如光追的 TLAS）通常用 LBVH。

**隐藏代价**：
- Morton Code 的精度有限（10 bit 每轴 = 1024 格），如果两个实体落在同一格，排序不再保证空间相邻。
- 对非均匀分布的场景，质量略逊于全 SAH。

#### 分支 C：动态物体的增量更新

**核心思路**：为动态物体维护一个独立的、支持快速插入/删除/移动的 BVH。使用基于 AABB 重叠的增量更新：当一个物体移动后，更新其叶节点包围盒，然后自底向上修正路径上所有祖先节点的包围盒。如果物体的移动导致其超出父节点的范围，则触发局部重构。

```cpp
// 增量更新示意
void UpdateDynamicEntity(BVHTree* tree, Entity e, const AABB& new_aabb) {
    BVHNode* leaf = tree->entity_to_leaf[e];
    leaf->bounds = new_aabb;
    
    // 自底向上修正包围盒
    BVHNode* node = leaf->parent;
    while (node) {
        AABB old_bounds = node->bounds;
        node->bounds = Merge(node->left->bounds, node->right->bounds);
        if (node->bounds == old_bounds)
            break; // 父节点未变，上方无需更新
        node = node->parent;
    }
    
    // 如果新 AABB 超出原根节点范围，可能需要全局重构或重新插入
}
```

**适用场景**：动态物体数量中等（< 1000）、移动范围有限、需要比暴力列表更好的效率。

**隐藏代价**：
- 局部修正只能处理"包围盒小幅变化"。如果物体跨越了大半个场景，修正会传播到根节点，实际效果接近全局重构。
- 需要维护父指针和实体到叶节点的映射，增加内存开销。

**失效条件**：动态物体数量大、或移动剧烈（如高速飞行器）。

#### 分支 D：双轨制——静态 BVH + 动态列表

**核心思路**：不为动态物体维护复杂加速结构，而是单独维护一个 `Array<DynamicRenderable>`。每帧直接遍历这个数组做视锥测试。动态物体的数量通常远少于静态物体（如 100 个角色 vs 10000 栋建筑），暴力测试的开销可接受。

```cpp
struct DynamicRenderable {
    Entity entity;
    AABB world_aabb;
    Sphere world_sphere; // 用于快速预筛选
};

struct DynamicRenderList {
    Array<DynamicRenderable> entries;
};

// 每帧更新动态列表的 AABB（从 ECS 组件读取最新 Transform）
void UpdateDynamicListSystem(
    Query<(Entity, &GlobalTransform, &LocalAABB)> dynamic_query,
    ResMut<DynamicRenderList> list
) {
    list->entries.clear();
    for (auto [e, transform, local_aabb] : dynamic_query) {
        AABB world_aabb = TransformAABB(local_aabb, transform.matrix);
        Sphere world_sphere = BoundingSphereFromAABB(world_aabb);
        list->entries.push_back({e, world_aabb, world_sphere});
    }
}
```

**适用场景**：绝大多数游戏场景。动态物体数量占总实体数的 5~20%。

**隐藏代价**：动态物体数量如果异常增多（如万人同屏的 MMO），暴力测试会成为瓶颈。但这是可预期的边界条件，可以通过增加 LOD 距离或强制限制同屏动态实体数来缓解。

```
[A][B][C][D][E][F][G][H]

第 1 层分裂（中位数）:
        [AABB: A-H]
       /           \
  [AABB: A-D]   [AABB: E-H]

第 2 层分裂:
    [AABB: A-D]       [AABB: E-H]
     /         \       /         \
  [A-B]     [C-D]   [E-F]     [G-H]

叶节点（每叶 2 个实体）:
[A][B]  [C][D]  [E][F]  [G][H]

→ 跳过 [A-B] 和 [C-D] 共 4 个实体，只需一次测试
```

### 决策分析与推荐

**默认推荐：静态物体用 Top-Down Binned SAH BVH（连续数组存储），动态物体用双轨列表。**

理由：

1. **Binned SAH 是工业级标准**。UE、Embree、OptiX 等主流光追和渲染引擎都采用 Binned SAH 构建 BVH。它在构建质量和速度之间取得了最佳平衡——16 个桶的评估足够产生高质量树，而 O(n log n) 的复杂度让 10 万实体在 100ms 内完成构建。

2. **连续数组存储是缓存友好的关键**。把 `BVHNode` 存到 `Array<BVHNode>` 中，用 `uint32_t` 索引代替指针。这样遍历树时，节点在内存中大概率相邻，CPU 预取能有效工作。指针跳跃的 BVH（每个节点 `new` 出来的）在遍历时会因为缓存未命中损失 30~50% 性能。

3. **LBVH 适合 GPU 构建，但 CPU 侧 Binned SAH 质量更高**。如果未来需要 GPU 光追的 TLAS（顶层加速结构），LBVH 是优秀选择。但对于 CPU 剔除，Binned SAH 生成的树更紧致，查询更快。

4. **双轨制对动态物体的处理经过工业验证**。UE 的渲染线程中，动态 `FPrimitiveSceneProxy` 不走静态加速结构的批量剔除，而是参与视锥测试的"动态列表"。chaos 也采用类似策略。这不是妥协，而是"正确的工程判断"——为 5% 的动态实体维护复杂加速结构，其更新开销通常超过剔除收益。

**ECS 映射**：
- **主 World**：`StaticGeometry` 标签组件（标记该实体参与静态 BVH）；`DynamicGeometry` 标签组件。
- **编辑器/工具**：`BuildStaticBVHSystem` 在场景保存或导入时离线运行，产出 `StaticBVH` 资源文件。
- **Render World**：`StaticBVH` 作为 Resource 存在；`DynamicRenderList` 作为每帧更新的 Resource。
- `FrustumCullSystem` 先查询 `StaticBVH`，再遍历 `DynamicRenderList`。

**AI 友好设计检查**：
- **状态平铺**：`StaticBVH` 是 Render World 的 Resource，其节点数组可被序列化；`DynamicRenderList` 是每帧重建的 Resource。
- **自描述**：BVH 的构建参数（叶节点大小、最大深度、Binned SAH 的桶数）是 `StaticBVH` Resource 的字段，AI 可通过 Schema 读取。
- **确定性**：相同的实体输入 + 相同的构建参数 = 相同的 BVH 树结构。支持离线构建后保存，加载时确定性复现。
- **工具边界**：AI 不应直接修改 BVH 节点，但可以操作主 World 的 `StaticGeometry` 组件（增删静态实体后触发 BVH 重建）。

**遗留问题**：到这一步，我们解决了"相机视野外"的剔除。但还有一种更隐蔽的不可见情况：物体虽然在视锥内，但它被前面的墙、山、建筑完全挡住了。视锥剔除对这种遮挡无能为力。如果不处理遮挡，室内场景中玩家看向一堵墙，墙后的整个房间仍然会被 CPU 提交给 GPU 绘制。如何剔除被遮挡的物体？
