> **本模块增量**：深入理解现代 GPU API 的多线程命令录制能力，掌握从单线程到引擎级并行渲染的完整演进路径。你将能够设计一个支持 Pass 级并行、MeshDrawCommand 并行生成的渲染调度系统。
>
> 本笔记探讨的核心问题是：**当单线程 CPU 录制成为瓶颈时，如何在保证 GPU 命令顺序正确的前提下，充分利用多核 CPU 并行录制渲染命令？**

## 问题 0：为什么单线程 CPU 录制会成为瓶颈？

### 场景与根因

一帧画 5000 个物体不是 GPU 的瓶颈——现代 GPU 轻松处理数万个 Draw Call。但 CPU 录制这 5000 条命令的过程，可能成为帧时间的杀手。

**量化场景**：假设你在 D3D12 下录制一条典型的 `DrawIndexedInstanced` 调用，CPU 侧需要完成的工作包括：
1. **PSO 验证**：确认当前绑定的管线状态对象是否合法（~50-100 ns）
2. **描述符表解析**：将引擎的 `BindGroup` 索引转换为 GPU 可访问的描述符堆偏移（~100-200 ns）
3. **命令缓冲内存写入**：将编码后的命令字节序列写入 `ID3D12CommandList` 的内部缓冲区（~30-50 ns）
4. **资源状态跟踪**：D3D12 驱动内部维护资源状态的 shadow copy，每次绘制都可能触发状态位更新（~20-40 ns）

单条 Draw Call 的 CPU 开销约 **200-400 ns**。5000 条就是 **1.0-2.0 ms**。这还没算 `SetPipelineState`、`BindVertexBuffer`、`ResourceBarrier` 等状态切换命令。在复杂的延迟渲染管线中，一帧的 API 调用总数（含状态设置）可能达到 2-3 万条，单线程录制耗时 **5-10 ms** 并不罕见。

16.6 ms 的 60fps 帧预算中，如果录制吃掉 8 ms，留给逻辑更新、物理模拟、动画评估的时间只剩 8 ms——这还没算 GPU 的实际执行时间。

**深层矛盾**：现代 CPU 有 8 核、16 核甚至更多，但渲染命令录制天然有顺序依赖。GBuffer Pass 必须在 Shadow Pass 之后执行（如果 shadow 是前一帧的则另说），同一 Pass 内的物体可以按任意顺序绘制（只要排序正确），但 Render Pass 的 `Begin` 和 `End` 必须成对出现。

你想利用多核并行录制，但命令之间有两类依赖：
1. **提交顺序依赖**：GBuffer -> Lighting -> PostProcess 不能乱序提交给 GPU。
2. **状态依赖**：如果线程 A 录制了一个改变全局渲染状态的命令（如切换 Render Target），线程 B 录制的命令不能在这个切换之前执行。

**关键洞察**：瓶颈不在"画"，而在"准备画的过程中 CPU 做了太多事"。多线程并行渲染的目标不是让 GPU 画得更快，而是让 CPU 准备命令的速度追上 GPU 消费命令的速度。

## 问题 1：CPU 侧并行的切入点在哪里？

单线程录制成为瓶颈后，直觉的想法是："让多个线程一起调 `DrawIndexed`"。但这只是最 naive 的方案。实际上，CPU 侧与渲染相关的工作可以分成三个层次，每个层次的并行策略完全不同。

### 层次划分：生成 → 翻译 → 提交

```
┌─────────────────────────────────────────────────────────────┐
│  层次 1：生成（Generation）                                   │
│  ─────────────────────────                                   │
│  视锥剔除、LOD 选择、材质排序、构建 DrawPacket               │
│  → 完全无状态依赖，天然适合数据并行                          │
├─────────────────────────────────────────────────────────────┤
│  层次 2：翻译（Translation）                                  │
│  ─────────────────────────                                   │
│  将引擎级 DrawPacket 转换为底层 API 调用                      │
│  → 需要统一的命令内存输出，但有策略可分片并行                  │
├─────────────────────────────────────────────────────────────┤
│  层次 3：提交（Submission）                                   │
│  ─────────────────────────                                   │
│  将命令列表送入 GPU 队列                                      │
│  → 通常单线程完成，驱动对 QueueSubmit 的线程安全有开销        │
└─────────────────────────────────────────────────────────────┘
```

**为什么必须区分这三个层次？**

因为"生成"的瓶颈和"翻译"的瓶颈来源完全不同。生成阶段的主要开销是场景遍历、矩阵计算、排序——这些是纯计算密集型工作，并行收益极高。翻译阶段的主要开销是 API 调用本身的固定开销和驱动内部状态跟踪——并行收益取决于底层 API 是否支持多线程命令列表。如果在生成阶段还没有成为瓶颈时就盲目并行翻译，可能引入的线程同步开销反而拖慢整体帧率。

### 分支 A：原生 API 多线程命令缓冲（Vulkan Secondary CB / D3D12 Multi-thread CL）

**核心思路**：直接使用底层 API 提供的多线程录制能力。Vulkan 有 Primary/Secondary Command Buffer 机制：多个线程各自录制 Secondary CB，最后由主线程把它们 `vkCmdExecuteCommands` 进 Primary CB 再提交。D3D12 允许多个线程各自持有独立的 `ID3D12CommandList`，只要它们从不同的 `CommandAllocator` 分配。

```cpp
// Vulkan 风格：多线程录制 Secondary Command Buffer
void WorkerThreadRecord(uint32_t worker_id, VkCommandPool pool,
                        const std::vector<DrawItem>& items) {
    VkCommandBuffer sec_cb;
    vkAllocateCommandBuffers(device, &(VkCommandBufferAllocateInfo{
        .commandPool = pool,
        .level = VK_COMMAND_BUFFER_LEVEL_SECONDARY,
        .commandBufferCount = 1,
    }), &sec_cb);
    
    vkBeginCommandBuffer(sec_cb, &begin_info);
    for (const auto& item : items) {
        vkCmdBindPipeline(sec_cb, VK_PIPELINE_BIND_POINT_GRAPHICS, item.pso);
        vkCmdBindVertexBuffers(sec_cb, 0, 1, &item.vb, &item.vb_offset);
        vkCmdDrawIndexed(sec_cb, item.index_count, 1, item.first_index, 0, 0);
    }
    vkEndCommandBuffer(sec_cb);
    // 把 sec_cb 交给主线程
}
```

**适用场景**：你直接使用 Vulkan/D3D12 作为唯一后端，团队熟悉这些 API 的线程安全规则，且 Draw Call 数量极大（>5000/帧）。

**隐藏代价**：
- **极高的复杂度**：Vulkan 的 Secondary CB 有"继承"概念（可以继承 Render Pass 状态，也可以不继承），选错会导致验证层报错或渲染错误。
- **Barrier 地狱**：如果多线程录制的命令涉及同一资源的布局转换，你必须精确知道"其他线程最后一次对这个资源做了什么"——这在多线程下几乎无法追踪。Godot 引擎的维护者曾明确指出这是 Vulkan 多线程录制最致命的问题。
- **Secondary CB 调用开销**：ARM Mali GPU（G710 之前）没有原生支持 Secondary CB 的硬件能力，每次 `vkCmdExecuteCommands` 都有显著的 CPU 和 GPU 开销。
- **内存爆炸**：每个线程每帧需要一个独立的 Command Pool，每个 Pool 按最坏情况分配命令缓冲内存。如果线程数 x 帧缓冲数很大，内存占用可观。

**失效条件**：当你的引擎需要支持 OpenGL（没有 Secondary CB 概念）或移动平台（ARM Mali 的 Secondary CB 开销高）时，这套模型完全无法映射。

### 分支 B：引擎级命令收集 + 单线程翻译

**核心思路**：多线程不参与底层 API 录制，而是并行生成**引擎级绘制命令包**（如 `DrawItem` 数组或 `MeshDrawCommand` 结构）。所有命令包被收集到一个线程安全的队列或数组中，最后由**单个 RHI 线程**把它们翻译成底层 API 调用。

```cpp
// 引擎级命令：纯数据结构，不含任何 API 句柄
struct DrawPacket {
    PipelineHandle pipeline;
    BufferHandle vertex_buffer;
    BufferHandle index_buffer;
    uint32_t index_count;
    // ... 排序键、材质参数索引等
};

// 多线程并行生成 DrawPacket
void CullAndFillPackets(JobSystem* js, Viewport* view,
                        std::atomic<DrawPacket*>& out_packets) {
    // 每个 worker 处理一部分可见物体
    ParallelFor(js, view->visible_objects, [&](Object* obj) {
        DrawPacket pkt;
        pkt.pipeline = obj->material->pipeline;
        pkt.vertex_buffer = obj->mesh->vb;
        // ...
        out_packets[alloc_index.fetch_add(1)] = pkt;
    });
}

// 单线程 RHI 线程：按排序键顺序翻译为底层 API 命令
void RHISubmitThread(const DrawPacket* packets, uint32_t count) {
    std::sort(packets, packets + count, [](const DrawPacket& a, const DrawPacket& b) {
        return a.sort_key < b.sort_key;  // 按 PSO、材质等排序以最小化状态切换
    });
    
    auto* cmd = rhi->BeginCommandBuffer();
    for (uint32_t i = 0; i < count; ++i) {
        const auto& pkt = packets[i];
        cmd->BindPipeline(pkt.pipeline);
        cmd->BindVertexBuffer(pkt.vertex_buffer);
        cmd->DrawIndexed(pkt.index_count, ...);
    }
    rhi->Submit(cmd);
}
```

**适用场景**：需要跨平台一致性（OpenGL 也能用）、多线程生成逻辑复杂（如视锥剔除、LOD 选择）、但底层 API 录制开销可控。

**隐藏代价**：
- **单线程翻译仍是瓶颈**：如果 Draw Call 数量极高（> 5000），单线程把它们全部翻译成底层 API 调用仍然会很慢。
- **排序与合批的延迟**：所有包必须先收集完才能排序，这意味着多线程生成阶段和单线程录制阶段是串行的，无法流水线重叠。

**失效条件**：当单线程翻译的 CPU 开销超过帧预算的 20% 时。

### 分支 C：Pass 级并行 + 依赖图调度

**核心思路**：不并行录制单个 Pass 内的命令，而是**并行录制不同的 Pass**——前提是这些 Pass 之间没有资源依赖。通过 Render Graph 显式声明 Pass 间的输入输出依赖，自动推导出哪些 Pass 可以并行录制。

```cpp
// 简化的 Pass 依赖图
render_graph
    .AddPass("ShadowMap", [&](PassBuilder& b) {
        b.Write(shadow_tex);
    })
    .AddPass("GBuffer", [&](PassBuilder& b) {
        b.Write(gbuffer_tex);
        b.Read(scene_uniforms);
    })
    .AddPass("Lighting", [&](PassBuilder& b) {
        b.Read(gbuffer_tex);
        b.Read(shadow_tex);  // 依赖 ShadowMap Pass
        b.Write(lit_tex);
    })
    .AddPass("PostProcess", [&](PassBuilder& b) {
        b.Read(lit_tex);
        b.Write(swapchain_tex);
    });

// 依赖分析：ShadowMap 和 GBuffer 无互相依赖，可并行录制！
// Lighting 必须等两者完成。
```

**适用场景**：引擎已经引入了 Render Graph（阶段 5.7 的内容），Pass 数量较多、依赖关系相对规则。

**隐藏代价**：
- Render Graph 本身的实现复杂度很高（需要资源别名分析、Barrier 自动推导、内存复用优化）。
- 不是所有 Pass 都能并行：许多渲染管线有严格的顺序依赖（如透明物体必须在 opaque 之后）。
- 对于小项目，Pass 数量少（可能只有 3~5 个），并行收益有限。

**失效条件**：当引擎的渲染管线极度简单（单前向 Pass + 后处理）时，依赖图的调度开销反而大于收益。

### 引擎对照

> 我们在解决的是「跨线程并行录制」这个宏观问题。上述三个分支不是互斥的，而是作用于不同层次的组合策略。
>
> - **chaos** 使用 `MeshDrawCommand` 封装单个绘制所需的一切状态，支持多线程并行生成命令后统一提交——接近**分支 B** 的"引擎级命令 + 单线程翻译"模型。其 `RenderTaskGraph` 进一步在 Pass 级别并行化无依赖的渲染任务，吸收了**分支 C** 的思想。
> - **UE** 使用了最完整的分层模型：渲染线程并行生成 `FMeshBatch` / `FMeshDrawCommand`（分支 B 的生成层），然后通过**并行翻译线程**（Parallel Translation Threads）把命令翻译成底层 API 调用（分支 A 的翻译层）。UE 的 `FRHICommandList` 体系本身就是线程安全的，每个线程可以持有自己的命令列表，最后合并提交——这是**分支 A + B 的工业级混合**。
> - **Bevy** 的渲染发生在独立的 `Render World` 中，由 ECS System 驱动。System 的并行调度由 Bevy 的调度器自动处理（基于显式依赖声明），这相当于在**生成层**天然并行。但 wgpu 的 `CommandEncoder` 不是线程安全的，所以 Bevy 的 system 并行只到生成绘制数据为止，最终的命令编码仍是在单线程的 `render_system` 中完成——本质上是**分支 B** 的 ECS 化表达。

### 决策分析

**默认推荐：以分支 B 的 ECS 化扩展为骨架，分层引入并行能力。**

理由：
1. **UE 没有选纯分支 A，也没有选纯分支 B——它选择了两者的优势组合**。UE 的渲染线程并行生成引擎级命令（分支 B），然后通过并行翻译线程（分支 A）把这些命令翻译成底层 API。这种设计让 UE 在高端硬件上能充分利用 8+ 核心的并行能力，同时避免了原生 API 多线程的复杂度直接暴露给上层。
2. **分支 A（原生 API 多线程）确实超纲，但"超纲"不等于"要放弃"**。UE 的做法告诉我们：你不需要让上层 System 直接操作 Vulkan Secondary CB 或 D3D12 Command Allocator——你只需要让上层生成引擎级的 `DrawPacket`，然后让专门的"翻译线程池"把这些包翻译成底层命令。Barrier 地狱、Allocator 同步、继承规则……这些都在翻译层内部处理，上层不感知。
3. **分支 C（Pass 级并行）依赖 Render Graph**，在阶段 5.2 过早引入全部复杂度不现实。但 Render Graph 解决的是"Pass 之间资源依赖"问题，而不是"命令录制并行"问题。UE 的并行翻译甚至不依赖 Render Graph——只要 Draw Call 够多，同一 Pass 内的命令也可以分片给多个线程翻译。
4. **Bevy 的做法验证了分支 B 的 ECS 适配性**：Bevy 的 System 并行生成渲染数据，最终单线程编码。这证明"引擎级命令 + ECS 并行"是天然契合的。

**具体实施策略**：
- **Phase 1**（阶段 5 初期）：单线程录制 + 单线程翻译。`IRHICommandList` 在单个 System 中顺序生成命令，帧末由 `RHISubmitSystem` 顺序翻译。这个阶段的重点是**验证命令缓冲的内存模型和接口设计是否正确**。
- **Phase 2**（当 Draw Call > 2000 且 Profiling 确认 CPU 录制是瓶颈时）：引入**多线程并行生成 `DrawPacket`**。ECS 的 `MeshRenderSystem` 可以拆分为多个并行的 `GenerateDrawPacketsSystem`（按实体分片），各自写入线程本地的 `DrawPacket` 数组。然后**单线程合并排序**（按 `sort_key`），最后**单线程翻译**。这一步的收益通常极大，因为"生成绘制数据"（视锥剔除、LOD 选择、材质排序）比"翻译 API 命令"更耗 CPU。
- **Phase 3**（渲染管线复杂到需要时）：引入**并行翻译线程池**。把排序后的 `DrawPacket` 数组按区间分片，多个线程各自翻译成底层 API 命令（D3D12 的 `ID3D12CommandList` 或 Vulkan 的 Secondary CB）。主线程最后合并提交。这是 UE 的工业级方案，实现复杂度较高，但架构上从 Phase 2 迁移过来时，只需要把"单线程翻译"替换为"区间分片 + 线程池翻译"，上层不感知。

**遗留问题**：多线程并行生成了 `DrawPacket`，但这些包需要被收集到一个统一的地方才能排序和翻译。多个 worker 同时往一个数组里写数据，如何避免锁竞争？这是问题 2 的核心。

## 问题 2：多线程生成阶段，线程安全的数据结构如何设计？

### 场景与根因

假设 Phase 2 已经启动：8 个 worker 线程并行处理可见实体列表，每个线程要为它负责的实体生成 `DrawPacket`。最 naive 的做法是：所有线程共享一个 `std::vector<DrawPacket>`，每次生成一个包就 `push_back`——但 `std::vector` 不是线程安全的，并发 `push_back` 会导致数据竞争和崩溃。

加锁？`std::mutex` 保护每次 `push_back`。但 8 个线程每帧竞争同一个锁数千次，锁竞争的开销可能抵消掉并行化的全部收益。

**深层矛盾**：你想要一个**全局有序的**命令数组（以便后续排序），但多个线程**并行写入**时，全局有序和并行写入在物理上是冲突的——除非你愿意付出同步代价。

### 分支 A：全局数组 + 原子偏移量（无锁追加）

**核心思路**：预分配一个足够大的全局 `DrawPacket` 数组，每个 worker 通过原子操作 `fetch_add` 获取自己的写入区间，然后在这个区间内无锁写入。

```cpp
struct DrawPacketBuffer {
    DrawPacket* buffer;           // 预分配的全局数组
    std::atomic<uint32_t> count;  // 当前已写入的数量
    uint32_t capacity;
};

// Worker 线程生成
void GeneratePacketsWorker(DrawPacketBuffer* shared,
                           const EntityRange& range,
                           const SceneView& view) {
    // 先统计本 worker 会产生多少个包
    uint32_t local_count = CountVisiblePackets(range, view);
    
    // 原子分配全局数组中的连续区间
    uint32_t start = shared->count.fetch_add(local_count);
    
    // 在分配到的区间内无锁写入
    uint32_t write_idx = start;
    for (Entity e : range) {
        if (IsVisible(e, view)) {
            shared->buffer[write_idx++] = BuildPacket(e, view);
        }
    }
}
```

**适用场景**：生成阶段能预先知道（或快速估算）每个 worker 的输出数量，且 `DrawPacket` 大小固定。

**隐藏代价**：
- **预分配容量估计困难**：如果估算不足，需要 fallback 到动态扩容或截断。一帧的最大 Draw Call 数取决于视角、场景复杂度、粒子数量，波动很大。
- **内存碎片不严重，但数组不紧凑**：虽然每个 worker 写入连续区间，但不同 worker 负责的实体在场景中的位置是随机的，所以 `buffer` 中相邻的包可能对应场景中相距很远的物体——这不影响排序阶段，但影响生成阶段的缓存局部性。

**失效条件**：当 `DrawPacket` 的大小不固定（如某些包需要附加变长数据），或生成数量完全无法预估时。

### 分支 B：线程本地缓冲区 + 最终合并

**核心思路**：每个 worker 拥有自己独立的 `ThreadLocalBuffer`（一个小的局部数组或线性分配器），并行生成阶段只写本地缓冲区，全部完成后再把所有本地缓冲区的内容合并到全局数组中。

```cpp
struct ThreadLocalPacketBuffer {
    DrawPacket local[1024];  // 固定大小本地缓冲
    uint32_t local_count = 0;
    std::vector<DrawPacket>* overflow;  // 溢出时回退到堆
};

void GeneratePacketsWorkerTLS(ThreadLocalPacketBuffer* tls,
                              const EntityRange& range,
                              const SceneView& view) {
    for (Entity e : range) {
        if (!IsVisible(e, view)) continue;
        
        if (tls->local_count < 1024) {
            tls->local[tls->local_count++] = BuildPacket(e, view);
        } else {
            tls->overflow->push_back(BuildPacket(e, view));
        }
    }
}

// 合并阶段（单线程）
void MergeThreadLocalBuffers(DrawPacketBuffer* global,
                             ThreadLocalPacketBuffer* tls_array,
                             uint32_t num_threads) {
    uint32_t total = 0;
    for (uint32_t i = 0; i < num_threads; ++i) {
        total += tls_array[i].local_count;
        total += tls_array[i].overflow->size();
    }
    global->resize(total);  // 或确保预分配足够
    
    uint32_t write = 0;
    for (uint32_t i = 0; i < num_threads; ++i) {
        auto& tls = tls_array[i];
        memcpy(&global->buffer[write], tls.local, tls.local_count * sizeof(DrawPacket));
        write += tls.local_count;
        if (!tls.overflow->empty()) {
            memcpy(&global->buffer[write], tls.overflow->data(),
                   tls.overflow->size() * sizeof(DrawPacket));
            write += tls.overflow->size();
        }
    }
}
```

**适用场景**：Worker 数量固定（等于线程池核心数），本地缓冲能覆盖大多数情况（90%+ 的 worker 不需要 overflow）。

**隐藏代价**：
- **合并阶段的 memcpy 开销**：虽然 `memcpy` 很快，但如果 8 个线程各产生了 1000 个包，合并时要拷贝 8000 个 `DrawPacket`（假设 48 字节/个，共 384 KB），这会在主线程上引入约 0.05-0.1 ms 的开销。
- **overflow 分配的不可预测性**：如果某个角度下大量实体同时可见（如俯瞰城市），单个 worker 的 local 缓冲可能集体溢出，导致大量堆分配。

**失效条件**：当 Worker 数量动态变化（如根据负载调整），或本地缓冲的命中率极低时。

### 分支 C：每个 Pass 独立的命令数组（按 Pass 分片）

**核心思路**：不是按实体分片给 worker，而是按 Pass 分片。GBuffer Pass 的 DrawPacket 只进 GBuffer 数组，Shadow Pass 只进 Shadow 数组。同一 Pass 内的多线程写入仍可用分支 A 或 B 的策略。

```cpp
struct PassPacketBucket {
    DrawPacket* packets;
    uint32_t count;
    uint32_t capacity;
};

struct FramePacketBuckets {
    PassPacketBucket shadow;
    PassPacketBucket gbuffer;
    PassPacketBucket lighting;
    PassPacketBucket transparent;
    // ...
};
```

**适用场景**：渲染管线 Pass 数量较多，且不同 Pass 的绘制逻辑差异大（如 Shadow Pass 只需要深度，GBuffer Pass 需要完整材质状态）。

**隐藏代价**：
- **负载不均衡**：如果 Shadow Pass 只有 100 个 Draw Call，而 GBuffer Pass 有 3000 个，按 Pass 分片会导致部分线程空闲。
- **跨 Pass 的实体排序困难**：如果同一个实体出现在多个 Pass（如先画深度预渲染，再画 GBuffer），按 Pass 分片后需要在每个 Pass 中独立处理该实体，增加了数据提取的重复工作。

**失效条件**：当渲染管线以单一 Pass 为主（如前向渲染只有一个 Main Pass）时，按 Pass 分片退化为单 Pass 内部分片，没有额外收益。

### 决策分析

**默认推荐：分支 B 的变体——线程本地缓冲区 + 全局原子追加 fallback。**

理由：
1. **本地缓冲区消除了 99% 的原子竞争**。在大多数帧中，每个 worker 处理的实体数量均匀分布，本地 1024 个包的缓冲足够覆盖。只有当某个 worker 遇到"热点"（如大量粒子在同一区域）时才需要原子追加到全局数组。
2. **UE 的 `FParallelCommandListSet` 本质上就是分支 B 的工业级形态**。UE 为每个并行 worker 分配独立的 `FRHICommandList`（线程本地命令内存）， worker 完成后再把这些列表合并到主列表中。这验证了"先本地后合并"模式的工程可行性。
3. **Bevy 的 ECS 并行调度天然适合分支 B**。Bevy 的 System 并行执行时，每个 System 可以通过 `Local<T>` 获取线程本地状态，这正对应"线程本地缓冲区"的概念。
4. **分支 A 的原子追加在极端竞争下会退化**。如果所有 worker 的输出都不可预估，且持续触发 `fetch_add`，缓存行 bouncing 会让多核扩展性归零。

**ECS 映射**：
- `DrawPacketBuffer` 可以作为 ECS `Resource`（全局单例）。
- `ThreadLocalPacketBuffer` 用 `Local<PacketBuffer>` 在每个 System 实例中自动分配。
- `GenerateDrawPacketsSystem` 是一个按实体分片并行执行的 ECS System。
- 合并阶段由一个顺序执行的 `MergeDrawPacketsSystem` 完成，它在 `GenerateDrawPacketsSystem` 之后运行。

**遗留问题**：合并后的全局 `DrawPacket` 数组是乱序的（按 worker 完成顺序排列）。为了最小化 GPU 状态切换，必须在翻译前按 `sort_key` 排序。但 5000 个元素的排序在 CPU 上需要 0.1-0.3 ms，这个开销是否可以接受？有没有办法在生成阶段就维持近似有序？这是问题 3 的核心。

## 问题 3：跨线程生成的命令如何排序与合批？

### 场景与根因

多线程生成后的 `DrawPacket` 数组是乱序的：线程 A 先处理了一批远处建筑，线程 B 后处理了近处角色，线程 C 处理了半透明粒子。如果直接按这个顺序翻译成 GPU 命令，状态切换会极其频繁——画一个建筑（PSO_A）→ 画一个角色（PSO_B）→ 画一个粒子（PSO_C）→ 再画一个建筑（PSO_A）……这种来回切换是 GPU 执行效率的噩梦。

**根因**：现代 GPU 的状态切换开销极高。切换 PSO 可能需要重新配置固定功能管线（Rasterizer State、DepthStencil State、Blend State），切换纹理/缓冲区需要更新描述符表。理想情况是：所有使用相同 PSO 的物体连续绘制，所有使用相同材质的物体连续绘制。

这就是**排序键（Sort Key）**存在的意义：在翻译前，按一个精心设计的 64 位整数对所有 `DrawPacket` 排序，让状态相似的包聚集在一起。

### Sort Key 的设计空间

```cpp
// 典型的 64 位排序键布局（位域编码）
struct SortKey {
    uint64_t value;
    
    // 位域分配示例：
    // [63:56]  Pass ID          (8 bits)  — 确保不同 Pass 完全隔离
    // [55:48]  PSO 哈希高 8 位   (8 bits)  — 相同管线状态聚集
    // [47:32]  Material ID      (16 bits) — 相同材质聚集
    // [31:16]  Depth Key        (16 bits) — 透明物体按深度排序（从远到近）
    // [15:0 ]  Draw ID          (16 bits) — 打破平局，保持确定性
};
```

**关键决策**：排序键的位数分配直接决定了合批质量。
- 如果 PSO 只占 8 位，但引擎有 300 种 PSO，哈希冲突会导致不同 PSO 被错误地"合批"在一起——翻译阶段发现 PSO 不同再拆分，浪费排序时间。
- 如果 Depth Key 不占独立位域，透明物体的从远到近排序会与 PSO 排序冲突，导致半透明渲染错误。

**UE 的做法**：`FMeshDrawCommand` 的排序键是一个 64 位整数，按优先级从高到低编码：Pass → PSO → Material → Depth。这种编码让 `std::sort` 的整数比较直接实现了多级排序，无需自定义比较函数（自定义比较函数会显著降低排序速度）。

### 分支 A：全局排序（单线程 qsort/std::sort）

**核心思路**：所有 `DrawPacket` 生成并合并后，由主线程调用一次 `std::sort`。

```cpp
void SortDrawPackets(DrawPacket* packets, uint32_t count) {
    std::sort(packets, packets + count,
        [](const DrawPacket& a, const DrawPacket& b) {
            return a.sort_key < b.sort_key;
        });
}
```

**适用场景**：Draw Call 数量中等（< 5000），排序开销可接受。

**隐藏代价**：
- **O(N log N) 的不可压缩性**：5000 个元素约需 5000 * log2(5000) ≈ 60,000 次比较。每次比较是 64 位整数比较（1 个时钟周期），但 `std::sort` 的间接访问模式对缓存不友好。
- **单线程瓶颈**：当 Draw Call > 10000 时，全局排序可能消耗 0.5-1.0 ms，成为新的单线程瓶颈。

**失效条件**：Draw Call > 8000 或需要每帧多次排序（如多 viewport 渲染）时。

### 分支 B：基数排序（Radix Sort）

**核心思路**：Sort Key 是 64 位整数，基数排序对固定位宽的整数排序是 O(N)，且是稳定的。对于 5000-20000 个元素，基数排序通常比 `std::sort` 快 2-5 倍。

```cpp
void RadixSortDrawPackets(DrawPacket* packets, uint32_t count) {
    // 标准 LSD Radix Sort，按 11 位分 6 轮（或按 8 位分 8 轮）
    // 每轮统计 histogram，然后输出到临时缓冲
    // 实现略，可用现成的 radix sort 库
}
```

**适用场景**：Draw Call 数量大（> 5000），Sort Key 是纯整数。

**隐藏代价**：
- **需要 2x 内存**：基数排序需要源数组 + 临时数组两份内存。
- **对非整数 Sort Key 不友好**：如果 Sort Key 包含浮点深度值，需要先转换为整数表示（如翻转符号位以支持负值比较）。

**失效条件**：当 Sort Key 结构复杂（如包含指针或字符串）时，基数排序无法直接应用。

### 分支 C：生成阶段预分箱（Binning）

**核心思路**：与其生成后排序，不如在生成阶段就把包放进对应的"箱子"里。每个箱子对应一种 PSO + Material 组合，箱子内的包天然就是合批的。

```cpp
struct DrawBin {
    PipelineHandle pipeline;
    MaterialHandle material;
    DrawPacket* packets;  // 动态数组或线性分配器
    uint32_t count;
};

// 使用 HashMap：Key = (pipeline, material) 的 64 位混合哈希
HashMap<uint64_t, DrawBin> bins;

// Worker 生成时直接入箱
void GenerateAndBin(Entity e, const SceneView& view, HashMap<uint64_t, DrawBin>* bins) {
    DrawPacket pkt = BuildPacket(e, view);
    uint64_t bin_key = HashCombine(pkt.pipeline.value, pkt.material.value);
    
    auto it = bins->find(bin_key);
    if (it == bins->end()) {
        DrawBin new_bin;
        new_bin.pipeline = pkt.pipeline;
        new_bin.material = pkt.material;
        new_bin.count = 0;
        bins->insert({bin_key, new_bin});
        it = bins->find(bin_key);
    }
    it->second.packets[it->second.count++] = pkt;
}
```

**适用场景**：PSO + Material 的组合数远小于 Draw Call 数量（如 5000 个 Draw Call 只有 50 种不同的 PSO/Material 组合）。

**隐藏代价**：
- **HashMap 的并发访问需要锁或线程本地 Map**：如果多个 worker 同时往同一个 `bins` 里写，需要复杂的同步机制。更实用的做法是：每个 worker 有自己的本地 `bins`，最后再合并。
- **箱子数量不确定**：如果每个物体都有独特材质，箱子数 = Draw Call 数，分箱退化为排序。
- **透明物体的深度排序难以在箱内维持**：透明物体需要在同一材质内按深度排序，这要求每个箱子内部再做一次排序。

**失效条件**：当材质/PSO 组合数接近 Draw Call 数量时（如每个物体都有独特材质）。

### 决策分析

**默认推荐：分支 A 的优化版——64 位整数 Sort Key + 基数排序（Radix Sort）。**

理由：
1. **UE 使用 64 位整数 Sort Key + 自定义排序**，但 UE 的排序实现高度优化（可能使用 SIMD 加速的基数排序）。对于个人项目，直接使用成熟的基数排序实现（如 `gfx-rs` 的 `radix_sort` 或自研 LSD radix sort）即可获得接近线性的排序性能。
2. **分箱（分支 C）在理想情况下更快，但并发复杂度更高**。每个 worker 维护本地 HashMap 的方案在工程上是可行的，但 HashMap 的内存分配和合并开销在中小规模场景下不一定比基数排序更快。
3. **全局排序的优势是确定性**：无论输入如何乱序，排序后的顺序只取决于 Sort Key，这对调试、回放、单元测试极其重要。分箱方案的输出顺序取决于 HashMap 的遍历顺序，可能引入非确定性。
4. **基数排序的 O(N) 复杂度让它在 Draw Call 增长时表现稳定**。当场景从 5000 个物体扩展到 15000 个时，`std::sort` 的时间会增长约 4.5 倍（N log N），而基数排序只增长 3 倍（线性）。

**具体实施**：
- Sort Key 设计为 64 位无符号整数，按从高到低的位域编码：Pass ID → PSO Hash → Material ID → Depth → Draw ID。
- 不透明物体：Depth 位域填 0（或最大深度），让 PSO 和 Material 的排序优先。
- 透明物体：Depth 位域编码 view-space depth（翻转符号位使远处值更小），确保从远到近绘制。
- 排序使用稳定基数排序，保证相同 Sort Key 的包保持原始顺序（确定性）。

**遗留问题**：排序后的 `DrawPacket` 数组在翻译阶段需要被转换成底层 API 命令。如果是单线程翻译，这一步又可能成为瓶颈。如何把翻译也并行化，同时保证 GPU 看到的命令顺序与排序后的顺序一致？这是问题 4 的核心。

## 问题 4：并行翻译后，如何保持 GPU 提交顺序正确？

### 场景与根因

假设 Phase 3 启动：你把排序后的 `DrawPacket` 数组按区间分片，4 个翻译线程各负责一个区间：
- 线程 0：packets[0:1000] → 翻译成 Command List A
- 线程 1：packets[1000:2000] → 翻译成 Command List B
- 线程 2：packets[2000:3000] → 翻译成 Command List C
- 线程 3：packets[3000:4000] → 翻译成 Command List D

现在你有 4 个独立的底层命令列表（D3D12 的 `ID3D12CommandList` 或 Vulkan 的 Secondary CB）。如果你随意提交它们，GPU 可能按 A→B→C→D 执行，也可能按其他顺序——而你的排序逻辑要求它们必须按 A→B→C→D 的严格顺序执行，否则状态切换优化会失效。

**根因**：并行翻译打破了"命令在单一列表中的自然顺序"，你必须在提交阶段重建这个顺序。

### 分支 A：命令列表数组 + 顺序提交

**核心思路**：所有翻译线程完成后，主线程按数组索引顺序依次提交每个命令列表。

```cpp
// 并行翻译
ParallelFor(4, [&](uint32_t thread_id) {
    auto* native_list = rhi->CreateCommandList();
    uint32_t start = thread_id * packets_per_thread;
    uint32_t end = start + packets_per_thread;
    
    for (uint32_t i = start; i < end; ++i) {
        TranslatePacketToNative(packets[i], native_list);
    }
    native_list->Close();
    command_lists[thread_id] = native_list;
});

// 主线程：严格按顺序提交
for (uint32_t i = 0; i < 4; ++i) {
    rhi->Submit(command_lists[i]);
}
```

**适用场景**：D3D12/Vulkan 后端，命令列表数量少（< 16 个）。

**隐藏代价**：
- **Submit 的顺序性**：`vkQueueSubmit` 或 `ExecuteCommandLists` 虽然是 O(1) 的队列追加，但多次 Submit 比一次 Submit 一个大列表有更高的驱动开销。NVIDIA 官方建议"尽量减少 Queue 提交的次数"。
- **翻译线程的同步点**：主线程必须等待所有翻译线程完成后才能开始 Submit，这意味着翻译和提交无法流水线重叠。

**失效条件**：当命令列表数量很多（如每个 Pass 都分片成 8 个列表，一帧有 10 个 Pass，共 80 个列表）时，多次 Submit 的驱动开销会变得显著。

### 分支 B：单一命令列表 + 并行内存写入

**核心思路**：不使用多个独立的底层命令列表，而是让所有翻译线程并行写入同一个引擎级命令缓冲区（如 `IRHICommandList` 的内存缓冲），最后由单线程一次性翻译成原生 API 并提交。

```cpp
// 引擎级命令缓冲区（问题 3 的延迟缓冲模型）
class CommandBuffer {
    LinearAllocator mem;
    std::atomic<uint8_t*> write_ptr;
};

// 并行翻译线程只负责把引擎命令写到线性缓冲区
void TranslateWorker(CommandBuffer* cmd_buf,
                     const DrawPacket* packets,
                     uint32_t start, uint32_t end) {
    for (uint32_t i = start; i < end; ++i) {
        auto* cmd = cmd_buf->Alloc<CmdDrawIndexed>();
        cmd->pipeline = packets[i].pipeline;
        cmd->vertex_buffer = packets[i].vertex_buffer;
        // ...
    }
}

// 单线程：顺序遍历引擎命令缓冲区，翻译成原生 API
void FlushCommandBuffer(CommandBuffer* cmd_buf, NativeContext* ctx) {
    uint8_t* ptr = cmd_buf->mem.GetBase();
    while (ptr < cmd_buf->mem.GetCurrent()) {
        auto* header = reinterpret_cast<RenderCmdHeader*>(ptr);
        ptr += sizeof(RenderCmdHeader);
        switch (header->type) {
            case RenderCmdType::DrawIndexed: {
                auto* cmd = reinterpret_cast<CmdDrawIndexed*>(ptr);
                ctx->BindPipeline(cmd->pipeline);
                ctx->DrawIndexed(cmd->index_count, ...);
                break;
            }
            // ...
        }
        ptr += header->payload_size;
    }
}
```

**适用场景**：希望最大化跨平台一致性，且能接受单线程翻译的瓶颈（或 Draw Call 数量未达到翻译瓶颈）。

**隐藏代价**：
- **单线程翻译仍是瓶颈**：如果 Draw Call 数量极高，单线程 `FlushCommandBuffer` 的 `switch-case` 分发和 API 调用仍是帧时间的主要消耗。
- **引擎命令缓冲区的内存带宽**：一帧数万条命令，每条命令 48-64 字节，命令缓冲区本身可能占用数 MB 内存带宽。

**失效条件**：当单线程翻译的 CPU 时间超过帧预算的 15% 时。

### 分支 C：混合模型——区间分片翻译 + 单一列表合并

**核心思路**：这是 UE 的工业级方案。翻译阶段按区间并行生成多个**原生**命令列表（D3D12/Vulkan），但这些列表不是独立提交给 GPU，而是被"回放"到一个**主命令列表**中。D3D12 的 `ExecuteBundle` 和 Vulkan 的 `vkCmdExecuteCommands` 都支持这种"在一个主列表中按顺序执行多个子列表"的语义。

```cpp
// D3D12 风格：Bundle / CommandList 并行录制 + 主列表合并
void ParallelTranslateAndMerge(uint32_t num_threads) {
    ID3D12GraphicsCommandList* primary_list = ...;
    primary_list->BeginRenderPass(...);
    
    // 并行录制多个 secondary/bundle 列表
    ParallelFor(num_threads, [&](uint32_t tid) {
        ID3D12GraphicsCommandList* sub_list = sub_lists[tid];
        sub_list->SetPipelineState(...);
        // ... 录制命令
        sub_list->Close();
    });
    
    // 主线程：按顺序把子列表合并到主列表
    for (uint32_t i = 0; i < num_threads; ++i) {
        primary_list->ExecuteBundle(sub_lists[i]);  // D3D12 Bundle
        // 或 Vulkan: vkCmdExecuteCommands(primary, 1, &sub_lists[i]);
    }
    
    primary_list->EndRenderPass();
    primary_list->Close();
    queue->ExecuteCommandLists(1, &primary_list);
}
```

**适用场景**：D3D12 或 Vulkan 后端，需要极致的翻译并行性，同时保证严格的 GPU 执行顺序。

**隐藏代价**：
- **Bundle/Secondary CB 的限制**：D3D12 Bundle 不能改变 Render Target、不能设置 Pipeline State（在创建时就固定）——这些限制意味着 Bundle 只适用于状态相对固定的绘制序列，不是所有 Pass 都能用。
- **Secondary CB 的继承复杂度**：Vulkan 的 Secondary CB 必须正确设置 `VkCommandBufferInheritanceInfo`，否则在 `vkCmdExecuteCommands` 时会导致验证错误或崩溃。
- **移动端 Secondary CB 开销**：ARM Mali GPU（G710 之前）对 Secondary CB 的支持是软件模拟，每次执行都有额外开销。

**失效条件**：OpenGL 后端（不支持命令列表）、或需要频繁切换 Render Target 的 Pass。

### 决策分析

**默认推荐：分支 A（命令列表数组 + 顺序提交）作为 Phase 3 的实现，但接口设计预留分支 C 的扩展能力。**

理由：
1. **UE 在 D3D12/Vulkan 后端使用分支 C 的变体**（`FParallelCommandListSet` + `ExecuteBundle`/`vkCmdExecuteCommands`），但在 D3D11/OpenGL 后端回退到分支 B（单线程翻译）。这说明**不同后端的并行能力差异巨大，不能一刀切**。
2. **分支 A 的实现最简单，且跨后端一致**。无论底层是 D3D12、Vulkan 还是 OpenGL（通过模拟），"生成多个命令列表再按顺序提交"的语义都能正确表达。
3. **分支 C 的 Bundle/Secondary CB 限制太多**，在通用引擎中只能作为特定 Pass（如大量静态物体的 BasePass）的优化手段，不能作为默认路径。
4. **分支 B 是 Phase 1/2 的延续**，在 Phase 3 如果翻译成为瓶颈，应该优先尝试分支 A，而不是直接把引擎改成分支 B。

**ECS 映射**：
- 并行翻译可以表达为多个 ECS System 实例（分片）同时执行，每个实例输出一个 `NativeCommandList` 组件。
- 顺序提交由一个顺序执行的 `SubmitSystem` 完成，它读取所有 `NativeCommandList` 组件，按索引顺序调用 `Submit`。
- 这种设计天然保证了 GPU 顺序：ECS System 的依赖图确保 `SubmitSystem` 在所有翻译 System 完成后才执行。

**遗留问题**：并行翻译解决了 CPU 效率，但多线程架构引入了一个更深层的工程问题：如果生成和翻译都在并行，逻辑线程（Gameplay）是否也应该与渲染线程并行？如果逻辑线程修改了 Transform，而渲染线程正在读取同一个 Transform 生成 DrawPacket，数据竞争怎么办？这是问题 5 的核心。

## 问题 5：Render Thread 与 RHI Thread 的分离——必要性与 ECS 映射

### 场景与根因

前面的四个问题都假设"有一个线程（或一组线程）在负责渲染"。但现代引擎通常不止一组渲染线程。UE 有三级线程分离：

```
Game Thread（逻辑更新：Transform、动画、物理）
    ↓ 投递渲染命令 / 同步点
Render Thread（场景遍历、剔除、生成 DrawPacket、排序）
    ↓ 投递 RHI 命令 / 同步点
RHI Thread（翻译引擎命令为 D3D12/Vulkan API 调用、Submit 到 GPU）
    ↓
GPU（执行命令）
```

**为什么需要这种分离？**

如果没有分离，单线程要依次完成：逻辑更新 → 场景遍历 → 生成命令 → 翻译 API → Submit。一帧的 CPU 时间是这四者的总和。分离后，Game Thread 处理第 N+1 帧的逻辑时，Render Thread 可以并行处理第 N 帧的生成，RHI Thread 可以并行翻译第 N-1 帧的命令——理论上三帧的 CPU 工作可以重叠，帧时间取三者中的最大值而非总和。

**深层矛盾**：分离带来并行性，但也带来延迟和同步复杂度。Game Thread 修改的 Transform 不会立即反映在屏幕上——它要经过"Game Thread → Render Thread → RHI Thread → GPU → 显示器"的完整流水线，延迟可能达到 2-3 帧。对于输入响应敏感的游戏（如射击游戏的准星移动），这种延迟是不可接受的。

### 分支 A：单线程渲染（无分离）

**核心思路**：所有 CPU 渲染工作（逻辑更新后的生成、翻译、提交）都在同一个线程上顺序执行。这是最简单、延迟最低的模型。

**适用场景**：小型项目、移动平台、对输入延迟极度敏感的游戏、或 CPU 核心数受限的平台（如 Nintendo Switch 的 3 核可用）。

**隐藏代价**：
- **CPU 时间无法重叠**：逻辑更新和渲染录制串行执行，帧时间是两者之和。
- **无法利用多核**：即使 CPU 有 8 核，渲染只用一个核。

**失效条件**：当场景复杂度增长到单线程 CPU 时间超过帧预算时。

### 分支 B：双线程分离（Logic + Render）

**核心思路**：Game Thread 和 Render Thread 分离。Game Thread 完成逻辑更新后，把渲染所需的数据（如 Transform 数组、Camera 参数）复制到一个"渲染侧快照"中，然后继续下一帧的逻辑。Render Thread 读取快照，独立执行生成和翻译。

```cpp
// 帧边界的数据移交
void GameThreadFrameEnd() {
    // 1. 把当前帧的渲染数据写入双缓冲的 Front Buffer
    RenderFrameData* front = render_data_pool.GetFront();
    CopyTransformsToRenderData(front);
    CopyCameraToRenderData(front);
    
    // 2. 交换 Front/Back，通知 Render Thread 有新的数据可用
    render_data_pool.Swap();
    render_thread_semaphore.signal();
}

// Render Thread 主循环
void RenderThreadMain() {
    while (running) {
        render_thread_semaphore.wait();
        RenderFrameData* back = render_data_pool.GetBack();
        GenerateAndSubmitFrame(back);
    }
}
```

**适用场景**：大多数中型项目的起点。平衡了并行性和实现复杂度。

**隐藏代价**：
- **数据复制开销**：每帧需要把 Transform、Camera 等数据从 Game Thread 复制到 Render Thread。Entity 数量大时（10万+），复制可能消耗 0.2-0.5 ms。
- **延迟增加**：Game Thread 修改的 Transform 最早在下一帧才可见（如果 Render Thread 处理的是前一帧的数据）。

**失效条件**：当数据复制开销超过并行节省的时间时（通常发生在 Entity 数量极少但渲染极重的场景）。

### 分支 C：三线程分离（Logic + Render + RHI）

**核心思路**：在分支 B 的基础上，再增加一个独立的 RHI Thread，专门负责翻译和 Submit。Render Thread 只负责生成引擎级命令（`DrawPacket` 和 `IRHICommandList`），RHI Thread 负责把这些命令翻译成底层 API。

**适用场景**：大型项目、D3D12/Vulkan 后端、Draw Call 数量极大（>5000）。

**隐藏代价**：
- **同步复杂度指数级增长**：Game Thread ↔ Render Thread、Render Thread ↔ RHI Thread、RHI Thread ↔ GPU，三对同步关系需要精确管理。任何一个同步点出错都会导致画面撕裂、资源竞争或 GPU 崩溃。
- **RHI Thread 的瓶颈**：如果 Render Thread 生成命令的速度超过 RHI Thread 翻译的速度，RHI Thread 仍然会成为瓶颈。此时需要引入问题 4 的并行翻译方案。

**失效条件**：D3D11/OpenGL 后端（这些 API 的单线程提交效率已经很高，RHI Thread 的收益有限）、或团队缺乏多线程调试经验时。

### 分支 D：ECS 原生并行（Bevy 风格）

**核心思路**：不区分"Game Thread"和"Render Thread"，而是把渲染工作拆分为多个 ECS System，由 ECS 调度器自动决定哪些 System 可以并行、哪些必须串行。渲染所需的"快照"通过 Extract 阶段（把数据从 Logic World 复制到 Render World）完成，Extract 之后所有渲染 System 都在 Render World 中执行。

```cpp
// Bevy 风格的 ECS 渲染调度
// Extract 阶段：把 Logic World 的 Transform 复制到 Render World
ExtractTransformsSystem.run();

// 以下 System 可以自动并行（如果无依赖）
GenerateShadowPacketsSystem.run();    // 可以并行
GenerateGBufferPacketsSystem.run();   // 可以并行
// ...

// 以下 System 必须串行（有依赖）
SortDrawPacketsSystem.run();          // 依赖所有 Generate 完成
TranslateAndSubmitSystem.run();       // 依赖 Sort 完成
```

**适用场景**：ECS 原生引擎，System 依赖关系清晰，希望最大化自动并行能力。

**隐藏代价**：
- **Extract 阶段的单线程瓶颈**：如果 Extract 阶段需要遍历所有 Entity 复制数据，这一步是单线程的（或有限并行），可能成为瓶颈。
- **调度器的不透明性**：开发者难以精确控制哪些 System 在哪个线程上运行，调试并行问题更困难。

**失效条件**：当渲染管线的并行依赖关系极其复杂（如大量跨 Pass 的数据依赖），ECS 调度器的自动分析可能做出次优调度决策。

### 引擎对照

> 我们在解决的是「逻辑-渲染-提交的线程分离」这个架构级问题。
>
> - **chaos** 采用了**分支 B 的变体**：`ClientTickManager::tickMain()` 中，`present_logic_task` 被投递到任务队列并行执行，而 `tickRender()` 在主渲染线程上执行。渲染开始前会等待逻辑任务完成（`waitForSingleTaskCounter`），所以逻辑和渲染的并行度受限于这种显式同步。chaos 没有独立的 RHI Thread，翻译和提交都在渲染线程内完成。
> - **UE** 采用了**分支 C 的完整形态**：Game Thread → Render Thread → RHI Thread 三级分离，每级之间通过命令队列异步通信。UE 的 `FFrameEndSync` 和 `FRHICommandListImmediate::RHIThreadFence` 精确管理三级线程的帧边界同步。这是工业级引擎的标杆设计，但实现复杂度极高。
> - **Bevy** 采用了**分支 D 的 ECS 原生方案**：没有固定的"Render Thread"，渲染是 `RenderApp` 中的一组普通 System，由 `bevy_tasks` 的线程池自动调度。`ExtractCommands` System 负责数据复制，之后所有渲染 System 并行执行。但 wgpu 的 `CommandEncoder` 非线程安全，最终编码阶段（`render_system`）仍回归单线程。

### 决策分析

**默认推荐：分支 B（Logic + Render 双线程分离）作为阶段 5 的目标架构，接口预留向分支 C 演进的能力。**

理由：
1. **分支 B 是并行性和复杂度的最佳平衡点**。单线程（分支 A）在阶段 5 的验收标准（带纹理旋转立方体）下足够，但一旦加入后处理、多 Pass、UI，CPU 时间必然紧张。分支 C 的三级分离对于个人项目来说同步复杂度太高，容易陷入调试地狱。
2. **UE 的三级分离不是一步到位的**。UE 早期只有 Game Thread + Render Thread，RHI Thread 是在 D3D12/Vulkan 普及后才引入的。这说明**线程分离应该是渐进式演进，而非一次性设计**。
3. **Bevy 的 ECS 原生方案验证了"渲染作为普通 System"的可行性**。在我们的 ECS 引擎中，可以把"生成 DrawPacket"、"排序"、"翻译"都表达为 ECS System，由调度器自动并行。但"最终的 Submit"必须是一个顺序执行的 System，因为它涉及与 GPU 驱动的单点交互。
4. **分支 B 在 ECS 下的自然映射**：
   - `LogicTickSystem` 在 Logic World 中并行执行。
   - `ExtractRenderDataSystem` 在帧边界把 Logic World 的数据复制到 Render World（顺序执行）。
   - `GeneratePacketsSystem`、 `SortPacketsSystem`、 `TranslateSystem`、 `SubmitSystem` 在 Render World 中按依赖链执行。
   - 这种模型让"逻辑"和"渲染"天然解耦，且 Extract 阶段的显式数据复制是 AI 可观测的（AI 可以 diff 两个 World 的状态差异）。

**关键架构决策**：
- **双缓冲（Double Buffering）渲染数据**：Logic World 每帧写入 Buffer A，Render World 读取 Buffer B，帧末交换。这避免了 Render Thread 读取时被 Game Thread 修改的竞争条件。
- **延迟一帧渲染**：Render World 处理的是 Logic World 上一帧的数据。这是 UE 的默认模式，也是保证并行性最简单的方式。如果需要降低输入延迟，可以引入 Predictive Rendering 或 Input Warping 技术，但那是后期优化。
- **RHI Thread 作为可插拔选项**：接口层设计 `IRHIThread` 抽象，阶段 5 默认实现为"空"（翻译在 Render Thread 内完成），未来可以通过配置启用独立的 RHI Thread。

**遗留问题**：多线程渲染解决了 CPU 效率，但 GPU 资源的生命周期管理引入了更棘手的 race condition——CPU 删除了纹理，GPU 可能还在读。如何安全释放？这是 [[GPU资源生命周期与PSO管理]] 的核心。

## 工业级设计清单

| 设计点 | 默认推荐 | 条件切换 | ECS 映射 |
|--------|---------|---------|---------|
| **并行层次** | 先生成（并行），后翻译（先单线程，后可并行） | Draw Call < 1000 时全单线程 | 不同 System 负责不同层次 |
| **生成阶段数据结构** | 线程本地缓冲 + 最终合并 | 内存极度紧张时用全局原子追加 | `Local<PacketBuffer>` |
| **排序策略** | 64 位整数 Sort Key + 基数排序 | 材质组合极少时用分箱（Binning） | `SortSystem` 顺序执行 |
| **翻译并行** | 区间分片 + 命令列表数组顺序提交 | D3D12 Bundle/Vulkan Secondary CB 优化特定 Pass | 分片 System + 顺序 SubmitSystem |
| **线程分离** | Logic + Render 双线程 | 大型项目启用独立 RHI Thread | Extract → Render World |
| **Sort Key 编码** | Pass(8) → PSO(8) → Material(16) → Depth(16) → DrawID(16) | 透明物体启用深度位域 | `u64` 组件字段 |
