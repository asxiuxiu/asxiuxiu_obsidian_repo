# 为什么需要材质系统？

想象你正在写一个最 naive 的渲染程序。你有一堆模型要画，每个模型颜色不一样。最直白的做法是什么？

```cpp
// 分支 A：完全硬编码（naive 方案）
void DrawRedCube() {
    shader_red.Use();          // 每套颜色一个 shader
    SetUniform("color", 1,0,0);
    DrawMesh(cube_mesh);
}
void DrawBlueCube() {
    shader_blue.Use();         // 复制粘贴，只是颜色不同
    SetUniform("color", 0,0,1);
    DrawMesh(cube_mesh);
}
```

如果只有红蓝两个立方体，这勉强能跑。但场景里有 100 个物体、10 种贴图、开关骨骼动画、切换光照模型呢？你很快会陷入**着色器代码的复制粘贴地狱**——不是美术在调材质，而是程序员在硬编码 `shader_xxx.cpp`。

更深层的问题是：GPU 执行的底层代码（着色器）与美术想要的"外观调整"（颜色、光泽、贴图）处于完全不同的抽象层次。我们需要一座桥，把"美术语言"翻译成"GPU 语言"。这座桥就是**材质系统**。

但材质系统不只是"给 shader 传参数"那么简单。现代渲染管线中，一个"材质"可能触发数十种 GPU 状态组合（有无法线贴图？延迟还是前向？骨骼动画还是静态？），这些组合如果处理不好，会导致**着色器变体爆炸**——编译时间以小时计、内存占用以 GB 计、首次加载时帧率暴跌。材质系统的设计目标，就是在"美术灵活性"和"运行时可控性"之间找到平衡。

# 问题 1：材质与着色器之间应该是什么关系？

## 场景与根因

假设美术做了一张"生锈金属"的材质。这个材质被用到三种物体上：静态雕像、带动画的士兵、UI 图标。三种物体的顶点格式不同（静态模型只有位置+法线+UV，士兵多了骨骼权重，UI 只需要位置+UV），渲染路径也不同（士兵走延迟渲染，UI 走前向透明）。

如果"材质"直接等于"着色器程序"，那么"生锈金属"需要复制成三份着色器代码，分别处理三种顶点格式和渲染路径。美术改一下粗糙度，三份代码都要同步修改——这是维护噩梦。

问题的根因在于：**材质描述的是"外观属性"（颜色、贴图、参数），着色器描述的是"GPU 执行指令"（顶点变换、像素着色）。两者是多对多的关系，不能硬绑在一起。**

## 分支研究

### 分支 A：一对一绑定（每个材质一个独立着色器）

**核心思路**：材质资产直接包含完整的 VS/PS 源码。

```cpp
struct Material {
    std::string vertex_shader_source;
    std::string fragment_shader_source;
    Texture* albedo_texture;
    float roughness;
};
```

**适用场景**：极小型的 demo、教学项目、每个物体确实需要完全不同的着色逻辑。

**隐藏代价**：
- 共享困难：10 个物体都想要"金属"外观，但顶点格式不同，就得复制 10 份源码。
- 编译爆炸：每个材质都是独立的着色器编译单元，缓存无法共享。
- 美术无法工作：要求美术直接写 GLSL/HLSL。

**失效条件**：项目规模超过 10 个材质，或需要支持多种顶点格式/渲染路径。

### 分支 B：模板-实例分层（工业级方案）

**核心思路**：把"材质"拆成三层——**模板（Template）**定义变体规则，**资产（Asset）**定义具体参数，**实例（Instance）**绑定运行时资源。

```cpp
// 模板层：定义 "这个材质支持哪些关键字组合"
struct ShaderTemplate {
    // 如：skinning_type, lighting_model, mesh_type
    Array<ShaderCategory> categories;
    
    // 根据关键字组合生成唯一的变体 ID
    uint64_t GeneratePermutationHash(Array<StringId> keywords);
};

// 资产层：美术编辑的"材质定义"
struct MaterialAsset {
    ShaderTemplate* template_ref;
    PassHint pass_hint;              // deferred / translucent / ui
    RenderState raster_state;
    RenderState depth_state;
    Array<MaterialParameter> parameters;  // 颜色、贴图引用等
};

// 实例层：运行时对象，关联具体 GPU 资源
struct MaterialInstance {
    MaterialAsset* asset_ref;
    // permutation_id → Technique（实际可执行的 Shader 程序集）
    HashMap<uint32_t, ShaderTechnique> technique_cache;
};
```

**关键机制**：当渲染系统请求一个具体的着色器变体时（如"生锈金属 + 有骨骼 + 延迟渲染"），`MaterialInstance` 用关键字组合生成 `permutation_id`，查找缓存。如果未命中，就触发异步构建——从模板定义中读取各阶段的着色器源码，组合成完整的 `ShaderTechnique`（包含 VS/PS/SRG/PSO）。

**适用场景**：任何需要支持多种变体组合的工业级引擎。

**隐藏代价**：
- 关键字组合过多时，`technique_cache` 可能急剧膨胀（Permutation 爆炸）。
- 异步构建有帧延迟：首帧请求某个变体时，该物体可能漏绘或使用 fallback。

**失效条件**：Category 维度失控（如同时定义 10 个二值开关，产生 1024 种组合）。

### 分支 C：纯数据驱动 + ECS 组件映射（ECS 原生方案）

**核心思路**：材质不是继承树中的对象，而是 ECS 世界中的数据和系统。Bevy 的做法是：材质作为 `Asset<Material>` 存在，渲染阶段通过 `AsBindGroup` trait 自动转换为 GPU 绑定。

```rust
// Bevy 风格的 ECS 材质定义（概念转写为 C++）
struct StandardMaterial {
    Handle<Texture> base_color_texture;
    Vec4 base_color;
    float roughness;
    float metallic;
    // ...
};

// 渲染系统查询所有带 StandardMaterial 的实体
// 自动提取参数、创建 BindGroup、提交绘制
```

**关键机制**：`AsBindGroup` 把材质字段映射到着色器的 binding 槽位。`#[uniform(0)]` 表示这个字段对应 binding 0 的 uniform buffer，`#[texture(1)]` 表示对应 binding 1 的纹理。渲染系统遍历 ECS 查询结果时，为每个（材质类型，视图条件）组合生成特化的渲染管线。

**适用场景**：ECS 原生引擎，希望材质系统与 ECS 架构深度集成。

**隐藏代价**：
- 每个材质类型需要独立的渲染管线特化逻辑。
- 对"节点图编辑"的支持不如 OOP 引擎直观（Bevy 目前主要面向代码定义材质）。

**失效条件**：需要复杂的可视化节点图编辑（如 UE 材质编辑器）。

## 决策分析与推荐

**默认推荐：分支 B（模板-实例分层）**，因为 UE 的工业级验证表明这是处理复杂材质关系的最稳健方案。该方案在 ECS 中的映射方式是：

- `ShaderTemplate` 和 `MaterialAsset` 作为**共享资源**（类似 Bevy 的 Asset），由资源管理系统统一管理生命周期。
- `MaterialInstance` 作为**运行时句柄**存储在 ECS Component 中（如 `MaterialHandle` 组件）。
- 渲染 System 在 `prepare` 阶段查询 `Query<(MeshHandle, MaterialHandle, Transform)>`，根据当前渲染路径和网格特性，向 `MaterialInstance` 请求对应的 `ShaderTechnique`。

**条件切换**：如果项目规模很小（<20 种材质、无复杂变体需求），可先用分支 C 的简化数据驱动方案，但需明确标注"这是工业级方案的简化版，未来需补全模板-实例分层和异步构建机制"。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：采用严格的三级体系（Template → Asset → Instance），通过 `Category/Keyword` 机制管理变体，延迟异步构建 Technique。
- **UE**：`UMaterialInterface → UMaterial → UMaterialInstance`，`FMaterialResource` 按 (Platform, QualityLevel, FeatureLevel) 投影出多个编译目标，`FMaterialShaderMap` 管理所有变体，DDC 缓存避免重复编译。
- **Bevy**：`Asset<Material>` 通过 `AsBindGroup` 直接生成 BindGroup，材质类型即 Rust 类型，渲染管线通过 `SpecializedMeshPipeline` 按 Key 特化。

**默认推荐依据**：chaos 和 UE 的分层思路一致，都解决了"美术资产"到"GPU 状态"的多对多映射。Bevy 的 ECS 原生表达更简洁，但在复杂变体管理（如按平台/质量等级生成不同代码）上不如 UE 完备。因此我们取 UE 的分层架构，用 ECS 组件存储运行时句柄。

# 问题 2：如何处理着色器变体爆炸？

## 场景与根因

回到"生锈金属"材质。美术可能给它配置了这些开关：
- 是否使用法线贴图？
- 是否使用骨骼动画？
- 延迟渲染还是前向渲染？
- 低画质还是高画质？
- 是否开启视差遮蔽映射？

如果每个开关都是二值的，5 个开关就是 32 种组合。10 个开关就是 1024 种。每种组合都需要独立的 GPU 管线状态对象（PSO）和编译后的着色器字节码。这被称为**着色器变体爆炸（Shader Permutation Explosion）**。

根因有两个层面：
1. **GPU 执行层面**：GPU 的 SIMD 架构喜欢执行完全相同的字节码。如果同一次绘制中有的像素要算法线贴图、有的不要，就会产生分支分歧（branch divergence），降低利用率。
2. **API 设计层面**：现代图形 API（D3D12/Vulkan）要求 PSO 提前烘焙几乎所有状态（着色器代码、顶点格式、混合模式、深度模式等），导致状态组合的笛卡尔积爆炸。Sebastian Aaltonen 指出，现代游戏的本地 PSO 缓存可能超过 100GB。

## 分支研究

### 分支 A：运行时动态分支（if/else）

**核心思路**：用一个统一的着色器，通过 uniform 变量在运行时选择分支。

```glsl
// 分支 A：运行时 if
uniform bool u_use_normal_map;

void main() {
    vec3 N;
    if (u_use_normal_map) {
        N = texture(normal_map, uv).rgb * 2.0 - 1.0;
    } else {
        N = vertex_normal;
    }
    // ... lighting
}
```

**适用场景**：开关影响的代码路径极短、分歧概率低、对性能不敏感的平台（如移动端某些简化场景）。

**隐藏代价**：
- **Wave divergence**：一个 wave（如 32 或 64 线程）中只要有一个像素走 `true` 分支、一个走 `false` 分支，GPU 就要串行执行两个分支，有效利用率腰斩。
- **寄存器压力**：为了容纳所有分支的临时变量，编译器会分配更多寄存器，导致 occupancy 下降。

**失效条件**：分支内有纹理采样（高延迟）或复杂数学运算，且场景中同时存在两种状态的物体被绘制到同一批次。

### 分支 B：编译期全排列（静态变体）

**核心思路**：每个开关组合在编译期生成独立的着色器源码，通过宏定义或源码生成控制。

```cpp
// 分支 B：编译期宏控制（概念）
// 生成的源码之一："有骨骼 + 有法线 + 延迟渲染"
#define ENABLE_SKINNING 1
#define ENABLE_NORMAL_MAP 1
#define PASS_DEFERRED 1
#include "material_template.glsl"
```

**适用场景**：对性能极度敏感的主机游戏、已知固定组合的项目。

**隐藏代价**：
- **编译时间爆炸**：N 个开关产生 2^N 个编译任务。UE5 中大型项目的首次着色器编译可达数小时。
- **存储爆炸**：每个变体都存储独立的字节码。DDC 缓存体积膨胀。
- **运行时内存爆炸**：加载关卡时可能需要同时驻留数百种变体。

**失效条件**：开关数量超过 6~7 个，或项目规模中等以上。

### 分支 C：按需编译 + 多级缓存（UE 方案的核心）

**核心思路**：不预编译所有变体，而是**运行时按需请求 → 异步编译 → 全局缓存**。

```cpp
// 分支 C：按需异步编译（概念）
enum class TechniqueState { Invalid, Pending, Valid, Empty };

struct ShaderTechnique {
    std::atomic<TechniqueState> state;
    // 异步构建完成前，state = Pending
    // 构建完成后，state = Valid，触发等待的回调
};

MaterialInstance::GetTechnique(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (cache.contains(id)) return cache[id];
    
    // 未命中：加入 pending 队列，返回 fallback
    pending_builds.push_back(id);
    return fallback_technique;
}
```

**关键机制**：
1. **延迟构建**：首帧请求变体时，不阻塞渲染线程，而是加入后台队列，当前帧使用一个默认的 fallback 材质（如纯粉色或简化光照）。
2. **异步编译**：后台线程（或独立进程，如 UE 的 ShaderCompileWorker）执行实际编译。
3. **全局缓存**：编译产物写入 Derived Data Cache（DDC），跨会话、跨项目共享。下次启动时直接命中缓存，零编译时间。

**适用场景**：大型项目、编辑器环境、变体组合无法预先穷举的情况。

**隐藏代价**：
- **首帧/切换闪烁**：fallback 材质与真实材质差异明显时，玩家能看到" pop-in"。
- **缓存失效敏感**：引擎版本升级、着色器源码修改、平台切换都会导致 DDC 键变化，整批缓存失效。
- **缓存存储成本**：团队共享的 DDC 服务器可能存储 TB 级数据。

**失效条件**：对加载时间要求极高的竞技游戏（如 FPS），fallback 材质的视觉差异不可接受。

### 分支 D：特化常量（Specialization Constants）

**核心思路**：Vulkan 提供的机制，允许在管线创建时传入常量值，驱动在编译时做常量折叠和死代码消除，但不需要生成独立的源码文件。

```cpp
// Vulkan 特化常量（概念）
VkSpecializationInfo specInfo;
specInfo.mapEntryCount = 1;
specInfo.pMapEntries = &entry;  // 映射到常量 ID
specInfo.dataSize = sizeof(int);
specInfo.pData = &enable_normal_map;  // 0 或 1

// 管线创建时传入，驱动内部生成优化后的机器码
vkCreateGraphicsPipelines(..., &specInfo, ...);
```

**适用场景**：Vulkan 平台、变体差异较小的简单开关。

**隐藏代价**：
- 只有 Vulkan（和部分 Metal）原生支持，D3D12 没有直接等价物。
- 如果特化后的代码差异很大，驱动编译时间仍然不可忽略。
- 无法改变顶点输入格式（vertex layout），只能做片段着色器级别的开关。

**失效条件**：需要改变顶点格式或大量代码路径差异的复杂变体。

### 分支 E：动态状态 + Bindless（现代 API 演进方向）

**核心思路**：既然 PSO 爆炸是因为 API 把太多状态烘焙进去了，那我们就让 API 支持更多动态状态，并用 bindless 描述符减少绑定差异。

```cpp
// Vulkan 1.3 动态状态（概念）
VkPipelineDynamicStateCreateInfo dynamicState{};
dynamicState.dynamicStateCount = 3;
VkDynamicState states[] = {
    VK_DYNAMIC_STATE_DEPTH_TEST_ENABLE,    // 运行时开关深度测试
    VK_DYNAMIC_STATE_RASTERIZER_DISCARD,   // 运行时开关光栅化
    VK_DYNAMIC_STATE_COLOR_BLEND_ENABLE,   // 运行时开关混合
};
dynamicState.pDynamicStates = states;
```

**Bindless** 则把所有纹理/缓冲描述符放进一个大数组（Descriptor Heap），着色器通过索引访问，CPU 不需要为每种纹理组合创建不同的 BindGroup/DescriptorSet。

**适用场景**：D3D12 Ultimate / Vulkan 1.3+ / 现代主机平台。

**隐藏代价**：
- 需要 GPU 和驱动支持 bindless（所有现代桌面 GPU 都支持，但移动端支持度不一）。
- 着色器代码需要适配 bindless 风格（使用 `ResourceDescriptorHeap[]` 而非具名 sampler）。
- 调试复杂度上升：纹理索引错误不会立即报错，而是采样到随机数据。

**失效条件**：目标平台包含低端移动设备或旧版 API。

## 决策分析与推荐

**默认推荐：分支 C（按需编译 + 多级缓存）为主，分支 E（动态状态 + Bindless）为长期演进方向。**

具体策略：
1. **默认实现初期**：实现简化版的分支 C——运行时按需触发编译，内存中缓存 Technique，暂不需要进程外编译和分布式 DDC。
2. **变体控制**：严格限制材质的关键字维度（建议最多 4~5 个二值开关），超过此限制的方案应被否决。
3. **长期演进**：当目标平台确定只面向现代 GPU 时，逐步引入动态状态和 bindless，将 PSO 差异收敛到"顶点格式 + 着色器代码"两个维度，大幅削减变体数量。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：`ShaderTemplate` 的 Category/Keyword 机制生成 `permutation_id`，`MaterialInstance` 维护 `pending_build` 队列，`MaterialInstanceManager::postTick()` 异步处理构建。Technique 池使用固定索引数组提供稳定 ID。
- **UE**：`FMaterialShaderMapId` 编码所有影响编译的维度（QualityLevel、FeatureLevel、StaticSwitch、VertexFactoryType），DDC 缓存键精确到这些维度。UE5.1+ 引入 On-Demand Shader Compilation（ODSC），只编译当前关卡需要的变体，减少约 60% 的编译量。PSO Precaching + Bundled PSOs 双系统缓解运行时卡顿。
- **Bevy**：`PipelineCache` 按 `PipelineKey` 缓存渲染管线，材质通过 `SpecializedMeshPipeline` 按视图条件和网格特性特化。Bevy 的变体管理相对简单，因为 `bevy_pbr` 内置的 `StandardMaterial` 变体维度固定，用户自定义材质通过 Rust 类型系统自然隔离。

**默认推荐依据**：UE 的按需编译 + DDC 缓存是工业级验证最充分的方案。chaos 的 pending 队列 + Technique 池是同一思路的轻量实现。Bevy 的 PipelineCache 在 ECS 下表达更自然，但缺乏 UE 的进程外编译和分布式缓存能力。默认实现优先 chaos 级别的轻量异步编译 + 内存缓存，未来可扩展为 UE 级别的 DDC。

# 问题 3：材质参数如何在运行时高效绑定到 GPU？

## 场景与根因

场景中有 1000 个物体，都使用"生锈金属"这个材质模板，但每个物体的生锈程度、颜色深浅、使用的贴图不同。渲染每一帧时，GPU 需要知道**当前这个 draw call 的具体参数值和纹理在哪里**。

根因在于：GPU 是独立处理器，有自己的地址空间和描述符表。CPU 不能直接把 C++ 结构体的指针递给 GPU，必须通过 API 提供的绑定机制（Uniform Buffer、Descriptor Set、BindGroup）把数据上传到 GPU 可见的内存，然后告诉着色器"你的参数在 binding 0，你的纹理在 binding 1"。

这个绑定过程如果设计不好，会成为 CPU 侧的性能瓶颈。让我们看看几种做法。

## 分支研究

### 分支 A：每 Draw Call 独立上传

**核心思路**：每次绘制一个物体前，用 `glUniform*` 或 `vkCmdPushConstants` 把参数直接塞进命令缓冲。

```cpp
// 分支 A：即时上传（OpenGL 风格，概念）
for (auto& obj : objects) {
    cmd->SetUniform("baseColor", obj.material.base_color);
    cmd->SetTexture("albedoMap", obj.material.albedo_texture);
    cmd->DrawIndexed(...);
}
```

**适用场景**：极小型项目、学习用渲染器、物体数量 <100。

**隐藏代价**：
- **CPU 开销**：每次 `SetUniform` 都会修改命令缓冲的状态跟踪，大量小数据拷贝消耗 CPU 时间。
- **GPU 同步**：某些 API 下频繁修改同一资源会导致驱动层面的同步开销。
- **无法合批**：即使两个物体参数完全相同，也无法合并为一次 draw call（因为绑定操作在中间打断）。

**失效条件**：物体数量 >500 或需要每帧更新大量参数。

### 分支 B：Uniform Buffer 数组 + 索引

**核心思路**：把所有物体的材质参数打包到一个大的 Uniform Buffer（或 Storage Buffer）中，每个物体占一段，draw call 时只传一个索引。

```cpp
// 分支 B：UBO 数组（概念）
struct MaterialData {
    Vec4 base_color;
    float roughness;
    float metallic;
    uint albedo_texture_index;  // bindless 索引
};

// GPU 端
layout(std140, binding = 0) readonly buffer MaterialBuffer {
    MaterialData materials[];
};

// 绘制时只传 material_id
push_constants { uint material_id; }
void main() {
    MaterialData mat = materials[material_id];
    vec4 albedo = texture(bindless_textures[mat.albedo_texture_index], uv);
}
```

**适用场景**：现代 GPU、支持 SSBO/Storage Buffer 的 API、物体数量大且需要合批。

**隐藏代价**：
- 需要 bindless 支持（或至少 large UBO）。
- 所有材质必须使用统一的数据布局（Struct of Arrays 思维），灵活性受限。
- 如果某个材质需要额外参数（如 subsurface scattering），所有材质都要为这个字段预留空间，造成内存浪费。

**失效条件**：材质类型差异极大，无法用统一结构表达；或目标平台不支持 large buffer。

### 分支 C：BindGroup/DescriptorSet 池（Bevy 方案）

**核心思路**：每种材质类型定义自己的 BindGroupLayout（binding 0 = uniform，binding 1 = texture），运行时从池中分配 BindGroup 实例。

```cpp
// 分支 C：BindGroup 池（概念，受 Bevy 启发）
struct BindGroupLayout {
    // binding 0: Uniform buffer (材质参数)
    // binding 1: Texture2D (albedo)
    // binding 2: Sampler
};

struct MaterialBindGroupAllocator {
    // 按 layout 类型分池
    HashMap<TypeId, BindGroupPool> pools;
    
    BindGroupHandle Allocate(MaterialData data, GpuTexture texture) {
        // 从对应池中获取或创建 BindGroup
        // BindGroup 持有 uniform buffer 和 texture view 的引用
    }
};
```

**关键机制**：Bevy 的 `MaterialBindGroupAllocator` 为每种材质类型维护一个分配器。当材质参数变化时，释放旧的 BindGroup，分配新的。材质参数通过 `OwnedBindingResource::Data` 变体延迟写入统一的上传缓冲区。

**适用场景**：ECS 原生引擎、材质类型相对固定、需要自动处理参数上传。

**隐藏代价**：
- BindGroup 对象本身有创建开销，频繁变化的材质（如每帧更新的 UI）可能产生大量分配。
- 不同材质类型的 BindGroupLayout 不同，导致 PSO 无法共享（即使着色器代码相同）。

**失效条件**：动态材质数量极大（如粒子系统每个粒子不同材质），或需要超高频率更新。

### 分支 D：Push Constants + Bindless 描述符堆（现代高效方案）

**核心思路**：用 Push Constants（D3D12 Root Constants / Vulkan Push Constants）传少量高频变化数据（如材质索引、变换矩阵索引），所有纹理和缓冲通过 bindless 描述符数组访问。

```cpp
// 分支 D：Push Constants + Bindless（概念）
struct DrawConstants {
    uint material_index;    // 指向 MaterialBuffer 的索引
    uint transform_index;   // 指向 TransformBuffer 的索引
    uint custom_flags;      // 材质特化标志
};

// 命令缓冲中只需要一个 push constant
CmdPushConstants(sizeof(DrawConstants), &constants);
CmdDrawIndexed(index_count, 1, 0, 0, 0);
```

**适用场景**：现代 API（D3D12/Vulkan）、高性能渲染、Draw Call 数量极大。

**隐藏代价**：
- Push Constants 有大小限制（如 Vulkan 最小保证只有 128 字节）。
- 需要 bindless 支持。
- 调试困难：描述符索引错误不会立即崩溃。

**失效条件**：目标平台不支持 Push Constants 或 Bindless。

## 决策分析与推荐

**默认推荐：分支 C（BindGroup 池）作为基础架构，分支 D（Push Constants + Bindless）作为高性能优化路径。**

具体策略：
1. **基础实现**：每种材质类型定义 `BindGroupLayout`，`MaterialBindGroupAllocator` 按类型分池管理。材质参数通过统一的 CPU → GPU 上传缓冲区更新。这是 ECS 架构下最自然的表达方式。
2. **优化路径**：当 profiling 表明 BindGroup 切换成为瓶颈时，将纹理访问迁移到 bindless 描述符堆，将材质参数迁移到大型 Storage Buffer，Draw Call 只保留一个 `material_index` push constant。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：`ShaderContext` 内部持有 `ParameterPool` 和 `SRG`（Shader Resource Group），运行时快速设置常量缓冲和纹理绑定。后处理系统绕过材质实例的纹理池，直接通过 `ParameterPool` 设置参数。
- **UE**：`FShaderParameterMapInfo` 描述着色器参数布局，`FUniformBuffer` 按更新频率分层（每帧、每视图、每物体、每材质）。材质参数通过 `FMaterialRenderProxy` 传递到渲染线程，最终写入 GPU Uniform Buffer。
- **Bevy**：`AsBindGroup` trait 自动生成 BindGroupLayout，`MaterialBindGroupAllocator` 管理每种类型的 BindGroup 分配。`OwnedBindingResource::Data` 允许延迟将原始数据打包到统一缓冲区。

**默认推荐依据**：Bevy 的 `BindGroupAllocator` 在 ECS 下表达最为优雅，与 ECS 查询-提取-准备的生命周期天然契合。UE 的 UniformBuffer 分层策略在工程完备性上更优（考虑了不同更新频率），我们吸收其"按更新频率分层"的思想：将世界全局参数（View、Light）放在 binding 0~1，材质参数放在 binding 2，物体参数（Transform）放在 binding 3。这样可以最大化 BindGroup 的复用——同一材质的不同实例可以共享同一个材质参数 BindGroup，只需切换物体级绑定。

# 问题 4：着色器编译如何不阻塞渲染？

## 场景与根因

你在编辑器中修改了一个材质的节点图——把粗糙度从 0.5 改成 0.8。点击保存后，整个编辑器卡死了 3 秒钟。这就是**同步编译着色器**的代价。

根因在于：现代着色器编译（尤其是涉及 DXIL/SPIR-V 优化、PSO 创建时调用驱动级编译器）是 CPU 密集型任务。一个复杂材质的 PSO 创建可能需要 50~200ms。如果在渲染线程同步执行，16ms 一帧的预算会被瞬间击穿，帧率暴跌到 5fps。

更隐蔽的场景是游戏运行时：玩家走进一个新区域，加载了一套新材质。如果这些材质的着色器变体没有提前编译，首帧会出现严重卡顿（stutter）——这就是 Steam Deck 和主机游戏中常见的"着色器编译卡顿"问题。

## 分支研究

### 分支 A：同步阻塞编译

**核心思路**：请求材质时，如果对应变体未编译，直接在当前线程编译，完成后才继续渲染。

```cpp
// 分支 A：同步阻塞（反例，禁止在生产环境使用）
ShaderTechnique* GetTechniqueSync(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (!cache.contains(id)) {
        cache[id] = CompileTechnique(id);  // 阻塞！可能耗时 100ms+
    }
    return cache[id];
}
```

**适用场景**：离线工具、单元测试、极其简化的学习项目。

**隐藏代价**：显而易见——任何未缓存的变体都会导致卡顿。

**失效条件**：任何对帧率有要求的场景。

### 分支 B：异步后台编译 + Fallback 材质

**核心思路**：未命中缓存时，立即返回一个预定义的 fallback 材质（如简化光照或纯色），同时把编译任务提交到后台队列。

```cpp
// 分支 B：异步编译 + Fallback（概念）
enum class CompileState { Missing, Pending, Ready };

struct AsyncTechniqueCache {
    HashMap<uint32_t, ShaderTechnique*> ready_cache;
    Queue<CompileJob> pending_jobs;
    ShaderTechnique* fallback;  // 预编译好的简化材质
};

ShaderTechnique* GetTechniqueAsync(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (ready_cache.contains(id)) return ready_cache[id];
    
    if (!pending_jobs.contains(id)) {
        pending_jobs.push({id, keywords});  // 提交后台
    }
    return fallback;  // 立即返回，不阻塞
}

// 后台线程或每帧尾部的 PostTick 中处理
void ProcessPendingJobs() {
    while (!pending_jobs.empty() && time_budget > 0) {
        auto job = pending_jobs.pop();
        auto tech = CompileTechnique(job.keywords);
        ready_cache[job.id] = tech;
        NotifyReady(job.id);  // 触发回调，下一帧使用真实材质
    }
}
```

**关键机制**：
1. **Fallback 材质**：必须是预编译好的、保证随时可用的简化着色器。UE 使用"默认材质"（亮粉色棋盘格）作为视觉提示；chaos 使用后处理快捷路径或全局实例。
2. **回调/轮询**：编译完成后，通过回调函数或下一帧的状态检查，将渲染对象从 fallback 切换到真实材质。
3. **时间片预算**：每帧只处理有限数量的编译任务，避免后台编译挤占 CPU 影响主线程。

**适用场景**：编辑器环境、游戏运行时动态加载、变体无法预先穷举的情况。

**隐藏代价**：
- **视觉 pop-in**：fallback 与真实材质差异明显时，玩家能看到物体"突然变好"。
- **状态管理复杂**：需要跟踪每个 pending 请求的编译进度，处理编译失败（如平台不支持某变体）。

**失效条件**：竞技游戏等对视觉一致性要求极高的场景。

### 分支 C：预编译缓存（PSO Precache）

**核心思路**：在构建期或加载期，预先编译所有可能用到的 PSO，运行时只查缓存。

```cpp
// 分支 C：预编译缓存（概念）
// 构建期：遍历所有材质 + 所有变体组合 + 所有顶点格式，生成 PSO 清单
void BuildPSOCache() {
    for (auto& material : all_materials) {
        for (auto& variant : material.variants) {
            for (auto& vf : vertex_formats) {
                auto pso = CreatePSO(material.shader, variant, vf);
                pso_cache[ComputeKey(material, variant, vf)] = pso;
            }
        }
    }
}

// 运行时：O(1) 查表
PipelineState* GetPSO(material, variant, vf) {
    return pso_cache[ComputeKey(material, variant, vf)];  //  guaranteed hit
}
```

**适用场景**：主机游戏、关卡内容固定的游戏、对卡顿零容忍的项目。

**隐藏代价**：
- **构建时间剧增**：构建期需要枚举所有可能的组合并编译，可能增加数小时。
- **缓存体积庞大**：PSO 缓存文件可能达到数 GB。
- **无法覆盖动态内容**：DLC、Mod、用户生成内容无法预先缓存。

**失效条件**：内容动态生成、构建时间预算有限、存储空间敏感的平台。

### 分支 D：进程外编译 + 分布式加速（UE 工业级方案）

**核心思路**：把编译任务从游戏进程完全剥离，通过独立进程（甚至远程机器）执行，主进程只负责收发结果。

```cpp
// 分支 D：进程外编译（概念）
// 主进程：序列化编译任务到文件
void SubmitCompileJob(CompileJob job) {
    auto input_file = WriteJobToFile(job);
    LaunchWorkerProcess(input_file);  // ShaderCompileWorker
}

// 工作进程：读取文件、编译、写回结果
void WorkerMain() {
    while (true) {
        auto job = ReadJobFromFile(input_file);
        auto result = CompileShader(job);
        WriteResultToFile(output_file);
        RenameAtomically(output_file, done_file);  // 原子通知
    }
}

// 主进程：轮询结果文件
void PollResults() {
    for (auto& done_file : ScanDoneFiles()) {
        auto result = ReadResult(done_file);
        InstallToCache(result);
    }
}
```

**关键机制**：
1. **进程隔离**：编译器崩溃不会拖垮主进程。
2. **分布式扩展**：通过 IncrediBuild (XGE) 或 Unreal Build Accelerator (UBA) 将任务分发到局域网内的多台机器。
3. **文件系统 IPC**：UE 使用输入/输出文件作为进程间通信通道，简单可靠，无需共享内存或 Socket。

**适用场景**：大型团队、编辑器环境、需要最大化利用多核/多机的场景。

**隐藏代价**：
- **I/O 开销**：大量小文件的读写对网络文件系统（如云桌面）不友好。
- **复杂度**：需要管理进程生命周期、超时、崩溃恢复。

**失效条件**：单兵项目、对工具链复杂度极度敏感的项目。

## 决策分析与推荐

**默认推荐：分支 B（异步后台编译 + Fallback）作为运行时核心，分支 C（预编译缓存）为发布版优化，分支 D（进程外编译）为团队扩展预留。**

具体策略：
1. **开发期**：异步编译 + 内存缓存。编辑器中修改材质后，fallback 材质的短暂显示是可接受的。
2. **发布期**：引入 PSO 预缓存。在构建期遍历所有打包材质和已知变体组合，生成 PSO 缓存文件。运行时优先查缓存，缓存未命中再走异步编译。
3. **团队扩展**：当项目规模增大、编译时间成为瓶颈时，将编译任务序列化到文件，通过外部工具链（如 `make -jN` 或分布式编译系统）加速。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：`MaterialInstance` 维护 `m_pending_build_permutation_techs` 队列，`MaterialInstanceManager::postTick()` 每帧批量处理 pending 构建。Technique 状态机（Invalid → Pending → Valid）管理生命周期。
- **UE**：`FShaderCompilingManager` 全局单例调度 ShaderCompileWorker 进程，支持优先级队列和分布式编译。UE5.1+ PSO Precaching 在 `PostLoad()` 时预编译，Bundled PSOs 捕获运行时使用的 PSO 清单用于下轮构建。
- **Bevy**：`PipelineCache` 在渲染世界按 `PipelineKey` 缓存管线，创建管线是异步的（通过 wgpu），但 Bevy 本身不管理进程外编译——它依赖 wgpu 的底层缓存和 `naga` 的即时编译。

**默认推荐依据**：UE 的进程外编译 + DDC 缓存是工业级验证最充分的方案，但工具链复杂度高。chaos 的 pending 队列 + postTick 处理是同一思路的轻量实现，适合默认实现起步。Bevy 的 `PipelineCache` 在 ECS 下足够简洁，但缺乏主动的后台编译调度。因此我们推荐以 chaos 的异步队列模型为基础，保留向 UE 级进程外编译扩展的接口。

# 问题 5：在 ECS 架构下，材质系统该如何表达？

## 场景与根因

在 OOP 引擎中，材质通常是 `MeshRenderer` 类的一个成员变量：`mesh_renderer.material = some_material;`。但 ECS 的世界里没有"成员变量"，只有独立的 Component 和批量执行的 System。

假设你有 1000 个实体，每个实体都有 `Mesh`、`Material`、`Transform` 三个组件。渲染系统需要把这三组数据组合成 GPU Draw Call。如果设计不好，ECS 的优势（缓存友好、批量处理）会被材质系统的复杂性抵消。

根因在于：**材质系统天然带有"异构性"——不同材质类型需要不同的着色器、不同的绑定组、不同的渲染管线。而 ECS 的优势在于"同构批量"。我们需要在异构材质需求和同构批量处理之间找到桥梁。**

## 分支研究

### 分支 A：材质作为 Mesh 组件的一部分

**核心思路**：`Mesh` 组件内部包含材质数据，渲染系统只查询 `Mesh` 组件。

```cpp
// 分支 A：材质内嵌（反模式）
struct Mesh {
    VertexBufferHandle vb;
    IndexBufferHandle ib;
    MaterialData material;  // 内嵌材质参数
};

// 渲染系统
for (auto [mesh, transform] : Query<Mesh, Transform>()) {
    // 每个 Mesh 的材质类型可能不同，无法批量处理
}
```

**适用场景**：无。这是 ECS 反模式。

**隐藏代价**：
- `Mesh` 组件大小因材质类型不同而变化，破坏 ECS 的 SoA 布局。
- 无法共享材质：100 个物体用同一套参数，内存中存了 100 份拷贝。
- 渲染系统需要处理每种材质类型的分支，失去批量优势。

**失效条件**：任何使用 ECS 的场景。

### 分支 B：Handle 分离 + 按材质类型分 System（Bevy 方案）

**核心思路**：`Mesh` 和 `Material` 是两个独立组件，`Material` 存储的是资源句柄（`Handle<MaterialAsset>`），而非材质数据本身。渲染阶段按材质类型分多个 System 处理。

```cpp
// 分支 B：Handle 分离（ECS 原生，概念）
struct MeshHandle { AssetId mesh; };
struct MaterialHandle { AssetId material; };
struct Transform { Mat4 matrix; };

// 渲染提取阶段：把主世界的 ECS 数据搬到渲染世界
void ExtractMaterialsSystem(
    Query<(&MeshHandle, &MaterialHandle, &Transform)> query,
    RenderWorld* render_world
) {
    for (auto [mesh, mat, transform] : query) {
        render_world->phase_items.push_back({
            mesh_id: mesh.id,
            material_id: mat.id,
            transform: transform.matrix
        });
    }
}

// 渲染准备阶段：按材质类型批量创建 BindGroup
void PrepareStandardMaterials(
    RenderAssets<StandardMaterial>* materials,
    RenderAssets<GpuImage>* textures
) {
    for (auto& [id, mat] : materials->changed()) {
        mat.prepared_bind_group = CreateBindGroup(mat, textures);
    }
}

// 渲染队列阶段：按（材质类型，管线 key）分组
void QueueOpaqueDraws(RenderPhase* phase) {
    for (auto& item : phase->items) {
        auto material = GetPreparedMaterial(item.material_id);
        auto pipeline_key = ComputeKey(material.flags, mesh.vertex_layout, view.hdr);
        auto pipeline = pipeline_cache->Get(pipeline_key);
        phase->bins[material.type][pipeline].push_back(item);
    }
}
```

**关键机制**：Bevy 的渲染架构分为 Extract → Prepare → Queue → PhaseSort → Render 五个阶段：
1. **Extract**：把主世界 ECS 中的组件数据（Handle、Transform）复制到渲染世界。
2. **Prepare**：将 `Asset<Material>` 转换为 GPU 就绪的 `PreparedMaterial`（BindGroup、PipelineKey）。
3. **Queue**：按渲染阶段（Opaque/Transparent/Shadow）将物体放入对应的 Phase。
4. **PhaseSort**：在 Phase 内按（管线、深度、材质）排序，最大化合批。
5. **Render**：遍历排序后的列表，提交 Draw Call。

**适用场景**：ECS 原生引擎、需要清晰分离逻辑线程和渲染线程的项目。

**隐藏代价**：
- Extract 阶段需要遍历所有可见实体，数据拷贝有 CPU 开销。
- 不同材质类型的 `Prepare` System 可能竞争 GPU 资源上传带宽。

**失效条件**：对延迟极度敏感的竞技游戏（如 FPS），Extract 阶段的遍历开销可能成为瓶颈。

### 分支 C：统一渲染项 + 材质数据平铺（Data-Oriented 极端方案）

**核心思路**：所有材质参数平铺到大型缓冲区中，渲染项只存索引。完全不按材质类型区分 System，只有一个超通用的渲染 System。

```cpp
// 分支 C：平铺数据（概念）
struct RenderItem {
    uint mesh_id;
    uint material_data_offset;  // 指向 MaterialBuffer 的偏移
    uint pipeline_id;           // 预计算好的管线 ID
    uint bindless_texture_base; // bindless 描述符数组基址
};

// 一个通用的渲染 System，零分支
void RenderAllItems(
    Array<RenderItem> items,
    Buffer* material_buffer,
    Buffer* transform_buffer
) {
    for (auto& batch : GroupByPipeline(items)) {
        cmd->SetPipeline(batch.pipeline);
        for (auto& item : batch.items) {
            cmd->PushConstants({
                material_offset = item.material_data_offset,
                transform_index = item.transform_index
            });
            cmd->DrawMesh(item.mesh_id);
        }
    }
}
```

**适用场景**：现代 GPU、bindless 支持、材质类型差异可完全用数据表达（无不同着色器代码）的情况。

**隐藏代价**：
- 所有材质必须使用同一套着色器代码，通过 uniform 控制分支——回到了问题 2 的分支 A（运行时 if/else）的困境。
- 无法支持本质不同的渲染技术（如 PBR vs 卡通 vs 后处理）。

**失效条件**：需要多种 fundamentally different 的着色器技术。

## 决策分析与推荐

**默认推荐：分支 B（Handle 分离 + 按材质类型分 System）**，这是 ECS 兼容前提下吸收 UE 工业级设计的最优折中。

具体 ECS 映射：

| OOP 概念 | ECS 映射 | 说明 |
|---------|---------|------|
| 材质资产（UMaterial） | `Asset<MaterialAsset>` | 资源系统管理的共享数据 |
| 材质实例（UMaterialInstance） | `MaterialHandle` Component | 实体上的句柄，指向具体资产 |
| 渲染状态（FMaterialResource） | `PreparedMaterial`（渲染世界） | Extract 后生成的 GPU 就绪数据 |
| ShaderMap / Technique 池 | `PipelineCache`（渲染世界 Resource） | 按 PipelineKey 缓存的管线对象 |
| 参数绑定（UniformBuffer/SRG） | `BindGroup`（渲染世界） | Prepare 阶段创建，Queue 阶段引用 |

**数据流**：
```
主世界 ECS
├── Entity: [MeshHandle, MaterialHandle, Transform]
└── AssetServer: MaterialAsset { template_ref, parameters, textures }

Extract 阶段 → 渲染世界 ECS
├── [RenderMeshInstance { mesh_id, material_id, transform }]

Prepare 阶段
├── MaterialAsset → PreparedMaterial { bind_group, pipeline_key_bits }

Queue 阶段
├── RenderPhase::Opaque bins[Pipeline][MaterialType] ← 排序、合批

Render 阶段
├── 遍历 bins → SetPipeline → SetBindGroup → Draw
```

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：OOP 架构中 `MaterialInstance` 被 `RenderEntity` 引用，渲染系统遍历 `RenderEntity` 列表时获取对应 Technique。向 ECS 迁移时，Technique 句柄应成为 Component，渲染 System 按 Technique 类型分组处理。
- **UE**：`UMeshComponent` 持有 `UMaterialInterface`，渲染线程创建 `FPrimitiveSceneProxy` 和 `FMaterialRenderProxy`。ECS 化时，`FMaterialRenderProxy` 应变为 Extract 后的渲染组件。
- **Bevy**：`Mesh3d` + `MeshMaterial3d<T>` 是 Component，`RenderMeshInstances` 和 `RenderMaterialInstances` 是渲染世界的映射表。`Prepare` System 按材质类型并行处理。

**默认推荐依据**：Bevy 的 Extract-Prepare-Queue-Render 分阶段架构是 ECS 下最清晰的材质数据流模型。UE 的 Proxy 双缓冲机制（GameThread Proxy → RenderThread Proxy）可转化为 Extract 阶段的数据搬运。chaos 的 `MaterialInstanceManager` 全局池可转化为渲染世界的 `PipelineCache` Resource。该方案同时满足：(a) ECS 一致性——所有数据都是 Component/Resource；(b) 工业级完备性——保留了 UE 的分层设计意图。

# 问题 6：AI 如何观察和操作材质系统？

## 场景与根因

假设一个外部 AI Agent 想要修改场景中某个物体的颜色。它需要：
1. **知道**这个物体的材质有哪些可修改参数。
2. **安全地**修改参数，不会破坏渲染状态。
3. **看到**修改后的效果，确认操作成功。

根因在于：材质系统的状态分散在多个层次（资产定义、运行时实例、GPU 绑定组、渲染管线缓存）。如果 AI 只能操作最底层（如直接改 GPU 缓冲区的字节），它既难以理解语义，也容易引发不一致。

## 分支研究

### 分支 A：直接操作 GPU 资源

**核心思路**：AI 通过底层 API 直接修改 uniform buffer 或纹理数据。

**隐藏代价**：GPU 资源没有自描述性。AI 不知道偏移量 64 处是 roughness 还是 metallic，也不知道修改后是否需要重新创建 BindGroup 或 PSO。

**失效条件**：任何需要可理解、可维护的 AI 操作场景。

### 分支 B：操作 ECS Component + 自动同步

**核心思路**：材质参数作为 ECS Component 的字段，AI 修改 Component 后，渲染 System 自动将变化同步到 GPU。

```cpp
// 分支 B：AI 友好的材质 Component（概念）
struct PbrMaterial {
    Vec4 base_color;
    float roughness;
    float metallic;
    AssetId albedo_texture;
    // ... 其他参数
};

// AI 通过 MCP/Schema 接口操作
// {
//   "entity": 42,
//   "component": "PbrMaterial",
//   "field": "roughness",
//   "value": 0.3
// }

// 渲染 System 检测到 Component 变化（Change Detection）
void PreparePbrMaterials(Query<&PbrMaterial> query) {
    for (auto [mat] : query.changed()) {
        auto prepared = GetPreparedMaterial(mat);
        UpdateUniformBuffer(prepared.ubo, mat);  // 自动同步到 GPU
        prepared.bind_group_dirty = true;        // 标记需要重建 BindGroup
    }
}
```

**关键机制**：
1. **Schema 导出**：反射系统（阶段 4.4）自动导出 `PbrMaterial` 的字段类型和范围，AI 无需读头文件。
2. **变更追踪**：ECS 的 Change Detection（Bevy 的 `Changed<T>`）让渲染 System 只处理被 AI 修改的实体。
3. **事务边界**：AI 的修改先在 Component 层生效，GPU 同步是延迟的、可回滚的。如果 AI 操作被撤销，只需恢复 Component 值，GPU 状态会在下一帧自动重建。

**适用场景**：所有 AI 协作场景。

**隐藏代价**：
- 需要完善的反射系统支持（已在阶段 4.4 建立）。
- 高频修改（如每帧都改）可能导致每帧都重建 BindGroup，需要优化路径（如动态 UBO 子范围更新）。

**失效条件**：反射系统缺失、或需要绕过 ECS 直接操作 GPU 的极端性能优化场景。

## 决策分析与推荐

**默认推荐：分支 B（操作 ECS Component + 自动同步）。**

这是唯一同时满足 AI 友好四原则的方案：
- **状态平铺**：所有可变材质参数都是 ECS Component 的字段。
- **自描述**：通过反射注册表，AI 可以查询任意材质类型的 Schema。
- **确定性**：给定相同的 Component 修改序列，GPU 状态变化是确定的（因为同步逻辑是纯函数）。
- **工具边界**：MCP 接口的输入输出是结构化的 JSON/Schema。

**Agent 安全**：材质修改应受白名单约束。AI 不应被允许修改引擎内部材质（如后处理材质、GBuffer 写入材质），只能修改用户定义的 PBR/卡通等材质类型。这一约束通过阶段 8.6 的"组件白名单"机制实现。

# 工业级设计清单与简化路径

## 默认推荐路径（工业级）

```
MaterialAsset（磁盘/内存资产）
    ├── ShaderTemplate（共享模板：Category/Keyword 定义）
    ├── MaterialPass（渲染路径：Deferred/Forward/UI）
    ├── RenderState（Rasterizer/Depth/Blend）
    └── Parameters（Scalar/Vector/Texture 参数定义）
        ↓
MaterialInstance（运行时）
    ├── AssetRef → MaterialAsset
    ├── TechniqueCache[permutation_id] → ShaderTechnique
    └── PendingBuildQueue（异步变体构建队列）
        ↓
ShaderTechnique（GPU 就绪）
    ├── VertexShader + FragmentShader（编译后的模块）
    ├── PipelineStateObject（烘焙的管线状态）
    ├── BindGroupLayout（参数绑定接口声明）
    └── ParameterPool/SRG（运行时参数值）
```

**异步编译生命周期**：
```
渲染请求 Technique(keywords)
    → Cache 命中 → 直接返回
    → Cache 未命中 → 返回 Fallback
        → 提交 Pending Job
        → 后台线程编译（或 PostTick 时间片）
        → 编译完成 → 状态变为 Valid → 下一帧自动切换
```

## 简化路径（默认实现起步）

如果当前阶段工程负担过重，可采用以下简化子集：

1. **无模板-实例分层**：材质直接引用单个着色器程序，无 Category/Keyword 变体系统。
2. **无异步编译**：着色器在加载期同步编译（加载屏幕期间可接受）。
3. **单 BindGroup 分配器**：所有材质共用同一种 BindGroupLayout（固定 binding 位置：0=参数 UBO，1=albedo，2=normal，3=metallic/roughness）。
4. **手动 Fallback**：未加载完成时显示纯色，不需要复杂的 Technique 状态机。

**必须明确标注的差距**：
- 缺少变体管理意味着无法高效支持"有法线/无法线"等开关，要么全用统一分支（性能损失），要么手动维护多个材质资产（维护负担）。
- 同步编译意味着首次加载新材质时有卡顿，不适合开放世界动态流送。
- 统一 BindGroupLayout 会浪费 binding 槽位（如不需要法线的材质也要留空位），且无法支持 Bindless 优化。

# 总结

材质与着色器系统的核心矛盾是：**美术想要灵活、直观的"外观调整"，而 GPU 想要确定、同构的"执行指令"**。我们走过的决策链是：

1. **分层抽象**：通过 Template → Asset → Instance 三层，解耦"美术资产"与"GPU 状态"。
2. **变体控制**：按需异步编译 + 多级缓存，缓解静态变体爆炸；长期向动态状态和 bindless 演进。
3. **参数绑定**：ECS 架构下用 Component 存储材质句柄，渲染阶段按类型分 System 处理，保留批量优势。
4. **异步安全**：fallback 材质保证帧率不阻塞，后台编译完成后无缝切换。
5. **AI 友好**：所有材质参数作为 ECS Component 字段，通过反射 Schema 暴露给外部 Agent。

**设计权衡表**：

| 子问题 | 默认推荐 | 简化路径 | 长期演进 |
|--------|---------|---------|---------|
| 材质-着色器关系 | 模板-实例三层 | 资产直接引用单个 shader | 可视化节点图编辑器 |
| 变体管理 | 按需异步编译 + 内存缓存 | 加载期同步编译 | 进程外编译 + DDC + Bindless |
| 参数绑定 | BindGroup 池 + 按类型分 System | 统一 BindGroupLayout | Push Constants + Bindless Heap |
| ECS 表达 | Handle Component + Extract-Prepare-Queue | Mesh Component 内嵌材质数据 | 纯数据驱动平铺（需 bindless） |
| AI 操作 | Component 字段 + Change Detection | 暂不支持 AI 修改 | 组件白名单 + 事务回滚 |

> **下一步**：[[渲染一帧的生命周期]]，因为材质系统的最终输出（`ShaderTechnique` + `BindGroup` + `PSO`）需要在"一帧"的时间轴上找到正确的插入点——何时提取 ECS 数据、何时准备 GPU 资源、何时提交 Draw Call、何时处理资源屏障。理解了材质，现在需要把它放进帧的节奏里。
