## 问题 0：为什么 ECS 引擎不能直接用裸指针持有纹理和网格？

想象你正在写一个最简单的 3D 场景：一个旋转的立方体。你写了一个 `MeshComponent`，里面存了一个 `Mesh*` 指针，指向从磁盘加载的立方体网格数据。一帧过去，一切正常。

然后你加了第二个立方体。为了省内存，你让两个 `MeshComponent` 指向**同一个** `Mesh` 实例。很好，内存只花了一份。

接着你删除了第一个 Entity。`MeshComponent` 跟着销毁了——但 `Mesh*` 指向的内存怎么办？如果 `MeshComponent` 的析构函数里 `delete mesh`，第二个立方体立刻崩溃（use-after-free）。如果不 delete，内存泄漏。

这还只是两个 Entity 共享一个 Mesh。实际游戏里，同一张纹理可能被几百个 Material 引用，同一个骨骼动画被几十个角色共享。裸指针根本无法回答一个核心问题：**"现在还有谁在用它？"**

### 根因分析：ECS 的平铺存储与所有权矛盾

ECS 的核心优势是**组件数据平铺、批量迭代**。`MeshComponent` 应该是一个很小的 POD 结构，能够被成千上万个 Entity 连续存储，在 System 中被 SIMD 批量处理。如果在 `MeshComponent` 里塞一个智能指针（如 `std::shared_ptr<Mesh>`），会发生两件事：

1. **内存膨胀**：`std::shared_ptr` 通常占 16 字节（两个指针），而且指向的 control block 是独立堆分配。一万个 Entity 就是一万份 shared_ptr 拷贝，每次拷贝都要原子操作修改引用计数。
2. **缓存不友好**：System 迭代 `MeshComponent` 数组时，真正需要的数据是变换矩阵和渲染状态，但 `shared_ptr` 的间接跳转把 cache line 搞乱了。

更深层的问题是：**ECS 的组件存储不负责对象生命周期**。Entity 的创建和销毁由 `EntityRegistry` 管理，组件的增删由 `World` 的 Archetype 迁移处理——这些操作都是批量的、机械的。如果把资源的生命周期管理（引用计数、加载、卸载）也塞进组件里，就等于让 ECS 存储层承担了它不该承担的责任。

### 深层矛盾："引用"与"数据"必须分离

所以问题的根因可以归结为一句话：

> **ECS 组件应该只持有"如何找到资源"的最小信息，而资源本身的生命周期和数据存储应该由专门的资源管理系统负责。**

这个"最小信息"就是**句柄（Handle）**——一个轻量的、无所有权的标识符。组件存句柄，资源系统管理句柄到实际数据的映射和生命周期。这和我们之前在 [[实体生命周期与注册表]] 中讨论的"Entity ID 只是索引，不是指针"是同一套哲学在资源领域的延伸。

但句柄本身有多种形态，选择哪一种直接决定了系统的性能特征和安全性。这就引出了第一个关键决策点。

## 问题 1：资源句柄应该长什么样？

如果我们不能用裸指针，那组件里该存什么？业界对这个"最小信息"的设计有几种截然不同的思路。

### 分支 A：裸指针 + 手动生命周期（不推荐，但理解其局限很重要）

```cpp
struct MeshComponent {
    Mesh* mesh;  // 指向资源池中的某个 Mesh
};
```

**核心思路**：资源由一个全局的"资源池"分配，返回指针。用户保证不在资源被释放后使用指针。

**适用场景**：原型验证、单人开发的小项目、资源生命周期与关卡完全绑定的简单游戏。

**隐藏代价**：
- 没有 ABA 防护。资源 A 被释放后，新分配的资源 B 可能恰好拿到同一块内存。旧指针突然"复活"指向一个完全不同的资源。
- 没有引用计数。不知道多少组件在引用同一个资源，卸载决策全靠猜。
- 多线程环境下，资源池的分配/释放需要全局锁，与 ECS 的并行迭代冲突。

**失效条件**：任何涉及动态加载/卸载、共享资源、多线程的场景都会暴露问题。

### 分支 B：整数索引 + 世代计数（Handle Table）

```cpp
struct AssetHandle {
    uint32_t index;      // 在资源表中的位置
    uint32_t generation; // 世代计数，每次该槽位被回收后重新分配时递增
};

// 资源表中的每个槽位
template<typename T>
struct AssetSlot {
    T        data;
    uint32_t generation;
    bool     occupied;
};
```

**核心思路**：把所有同类型资源存在一个密集数组（`Array<AssetSlot<T>>`）中。Handle 是"索引 + 世代"的二元组。访问时先检查 `slots[index].generation == handle.generation`，不匹配则说明原资源已释放，这是一个"悬空句柄"。

**适用场景**：ECS 原生引擎（Bevy 的 `AssetIndex` 就是这种思想的变体）、需要 O(1) 访问且内存连续的场景。

**代码片段：访问校验**
```cpp
template<typename T>
T* AssetPool<T>::try_get(AssetHandle handle) {
    if (handle.index >= slots.size()) return nullptr;
    auto& slot = slots[handle.index];
    if (!slot.occupied || slot.generation != handle.generation)
        return nullptr;  // ABA 防护：世代不匹配，说明是旧句柄
    return &slot.data;
}
```

**适用场景**：
- 需要频繁遍历所有活跃资源的场景（如每帧更新所有纹理的流送状态）
- 内存敏感、追求 cache locality 的场合

**隐藏代价**：
- 资源表是单块连续内存，如果资源类型差异很大（纹理几十MB，材质几KB），统一池化会造成内存碎片或浪费。
- 世代计数溢出？`uint32_t` 的世代在正常使用中几乎不可能溢出（40亿次回收），但如果真的发生，需要额外的 salt 机制。
- 索引本身没有类型安全。一个 `AssetHandle {index=5, gen=1}` 既可以指向 Mesh 也可以指向 Texture，全凭调用者上下文。

**失效条件**：当资源类型差异极大、或需要跨类型统一引用时（如一个材质既引用纹理又引用着色器），单一类型的 Handle Table 需要额外的类型擦除层。

### 分支 C：128-bit 值类型 + 类型 ID + 引用计数（chaos 风格）

```cpp
union Handle {
    struct {
        uint32_t type_id;    // 资源类型（Mesh/Texture/Shader...）
        uint32_t salt;       // ABA 防护盐值
        void*    payload;    // 指向类型擦除的资源管理器内部节点
    };
    uint128_t value;         // 整体作为值类型传递
};
```

**核心思路**：Handle 是一个"自描述"的 128-bit 值，内部包含类型信息、盐值和一个指向管理器内部节点的指针。类型擦除的管理器统一处理所有资源的引用计数和生命周期。

**适用场景**：大型商业引擎、需要统一资源管理接口、多种资源类型共存且需要顶层统一调度的项目。

**代码片段：引用计数的内部实现**
```cpp
// Handle 本身不管理生命周期，只是标识符
// 但资源管理器内部为每个 payload 维护引用计数
class ResourceManager {
    struct ResourceNode {
        void* data;
        uint32_t ref_count;
        uint32_t salt;
        // ... 类型信息、加载状态等
    };
    // type_id -> DenseArray<ResourceNode>
    HashMap<uint32_t, void*> type_tables;
};

// RAII 包装（可选）
class OwnedHandle {
    Handle handle;
public:
    ~OwnedHandle() { release_reference(handle); }
};
```

**隐藏代价**：
- `payload` 是一个指针，Handle 不再是一个纯值类型，跨线程传递时需要考虑指针有效性。
- 128-bit 在某些 32-bit 平台上不友好（虽然现代游戏引擎基本抛弃 32-bit 了）。
- 类型擦除层带来虚函数或函数指针开销，批量访问同一类型资源时不利于编译器优化。

**失效条件**：对极致 cache locality 有要求的场景（如每帧访问数千个 Mesh 的顶点数据），类型擦除的间接层会成为瓶颈。

### 分支 D：强类型 Handle + 全局类型擦除存储（Bevy 风格）

```rust
// Bevy 的做法：编译期强类型 + 运行期世代索引
pub struct Handle<A: Asset> {
    id: AssetId<A>,  // Index 或 Uuid
    // ...
}

pub enum AssetId<A> {
    Index { index: AssetIndex, marker: PhantomData<fn() -> A> },
    Uuid { uuid: Uuid },
}
```

**核心思路**：Handle 在编译期携带类型参数 `A`，防止把 `Handle<Mesh>` 误传给需要 `Handle<Texture>` 的函数。运行期内部是一个世代索引（`AssetIndex = index + generation`），指向 `Assets<A>` 这个类型专属的稠密存储。

**适用场景**：Rust 等强类型语言、ECS 原生架构、编译期错误检查优先于运行期检查。

**隐藏代价**：
- C++ 模板元编程实现同等程度的编译期类型安全代价较高（虽然可行，但编译时间和错误信息不友好）。
- 跨类型统一处理困难（如"卸载所有资源"需要遍历所有已注册的类型）。
- `Uuid` 路径的访问是 HashMap 查找，比数组索引慢一个数量级。

**失效条件**：需要运行时动态类型识别的场景（如脚本系统传入一个 untyped handle，需要查询它的实际类型）。

### 决策分析

| 条件 | 推荐方案 | 理由 |
|------|---------|------|
| 纯 ECS 架构、类型安全优先 | 分支 B（Handle Table）+ 编译期类型包装 | Bevy 证明了这条路径在 ECS 下的优雅性；C++ 可用模板做 `Handle<Mesh>` 包装 |
| 大型商业引擎、多类型统一调度 | 分支 C（128-bit 自描述 Handle）| UE/chaos 都选择了某种形式的类型擦除，因为上层系统需要统一接口 |
| 原型/教学项目、快速迭代 | 分支 A（裸指针池）| 不是不能选，但必须清楚自己在牺牲什么。建议预留 Handle Table 的迁移接口 |

**默认推荐：分支 B（Handle Table + 世代索引）作为核心机制，外层用模板提供类型安全。**

推荐依据：
1. **ECS 兼容性**：Handle Table 的密集存储与 ECS 的批量处理哲学完全一致。`Array<AssetSlot<T>>` 可以被 System 直接批量遍历。
2. **UE 优先吸收**：UE 的 `FName` + 索引哲学本质上也是"小标识符指向大表"。虽然 UE 的 UObject 是全局继承树，但其"通过轻量索引访问大量数据"的设计意图与 Handle Table 一致。
3. **Bevy 验证**：Bevy 的 `DenseAssetStorage` + `AssetIndex` 已经在生产环境中验证了这条路径的可行性。
4. **性能可预测**：数组索引的访问延迟是固定的、可量化的，没有虚函数或哈希表的波动。

**在 ECS 中的映射**：
- `MeshComponent` 只存 `Handle<Mesh>`（8 字节：4 字节 index + 4 字节 generation）。
- `TextureComponent` 或 `MaterialComponent` 中的 `Handle<Texture>` 同理。
- 实际的 `Mesh` 数据存在 `Assets<Mesh>` 这个 ECS Resource 中（或者一个独立的资源子系统）。
- System 通过 `Res<Assets<Mesh>>` 查询句柄对应的实际数据。

**遗留问题**：Handle 解决了"如何标识资源"，但没有解决"什么时候释放资源"。如果一个 Entity 被销毁了，它持有的 `Handle<Mesh>` 随组件一起消失——但 Mesh 本身可能还被其他 Entity 引用。我们需要引用计数。

## 问题 2：引用计数该由谁管理？

组件里的 Handle 只是一个标识符，没有所有权语义。当持有 `Handle<Mesh>` 的 Entity 被销毁时，Mesh 数据本身该不该释放？这取决于还有没有其他 Entity 在引用它。

### 场景绑定：千 Entity 共享一 Mesh

一个场景里有 1000 棵树，每棵树是一个 Entity，有 `MeshComponent` 引用同一个 oak_tree.mesh。如果玩家砍倒一棵树（销毁一个 Entity），你绝对不想把 oak_tree.mesh 从内存中卸载——其他 999 棵树还在用它。

### 分支 A：每个 Handle 自带原子引用计数（类似 `std::shared_ptr`）

```cpp
template<typename T>
struct SmartHandle {
    uint32_t index;
    uint32_t generation;
    std::atomic<uint32_t>* ref_count;  // 指向全局引用计数表的指针
};
```

**核心思路**：每个 Handle 实例在拷贝时原子递增 `*ref_count`，析构时原子递减。当计数归零，通知资源管理器释放该槽位。

**适用场景**：简单直观、需要 RAII 自动释放的场合。

**隐藏代价**：
- **拷贝开销**：每次拷贝 Handle（如组件数组扩容、System 参数传递）都要原子操作。10000 个 Entity 的 `MeshComponent` 数组，每次 System 读取 Handle 都不会触发拷贝——但任何涉及 Handle 复制的操作（如序列化、Inspector 编辑）都要付原子税。
- **伪共享**：多个 CPU 核心同时拷贝不同 Handle，但它们的 `ref_count` 恰好落在同一 cache line 上，导致缓存一致性风暴。

**失效条件**：高频 Handle 拷贝、大规模并行场景。

### 分支 B：全局引用计数表，延迟批量回收（ECS 友好方案）

```cpp
template<typename T>
struct AssetStorage {
    Array<AssetSlot<T>> slots;
    Array<uint32_t>     ref_counts;     // 与 slots 平行数组
    Array<uint32_t>     pending_free;   // 待回收索引队列
    
    // System 批量操作：扫描所有组件数组，统计引用
    void batch_update_ref_counts(const Span<Handle<T>>& all_handles);
    void flush_pending_frees();         // 每帧末或定时执行
};
```

**核心思路**：引用计数不放在 Handle 里，而是放在资源存储的平行数组中。ECS System 可以批量扫描所有组件，用 SIMD 或并行 for 快速统计引用。回收延迟到帧边界统一执行。

**适用场景**：ECS 架构、需要批量处理、对拷贝开销敏感的项目。

**代码片段：批量引用计数更新**
```cpp
// 在 System 中，遍历所有 MeshComponent，统计哪些 Mesh 被引用了
void update_mesh_ref_counts(
    Query<const MeshComponent> meshes,
    ResMut<AssetStorage<Mesh>> storage
) {
    // 先清零
    for (auto& rc : storage->ref_counts) rc = 0;
    
    // 批量统计
    for (auto [mesh_comp] : meshes) {
        if (mesh_comp.handle.index < storage->ref_counts.size())
            storage->ref_counts[mesh_comp.handle.index]++;
    }
    
    // 标记零引用的槽位为待回收
    for (size_t i = 0; i < storage->ref_counts.size(); ++i) {
        if (storage->ref_counts[i] == 0 && storage->slots[i].occupied) {
            storage->pending_free.push_back(i);
            storage->slots[i].occupied = false;
            storage->slots[i].generation++;  // 递增世代，使旧 Handle 失效
        }
    }
}
```

**适用场景**：
- 大规模 ECS 场景，组件数量远超资源数量
- 需要确定性的帧边界行为（资源不会在渲染中途被释放）

**隐藏代价**：
- 需要遍历所有持有 Handle 的组件，时间复杂度 O(Entity 数量)。如果 Entity 数量极大（百万级），每帧全量扫描有性能压力。
- 延迟回收意味着资源不会立即释放，内存峰值可能略高于即时回收。

**优化分支：增量更新**
不用每帧全量扫描，而是只在 Entity 创建/销毁/组件修改时更新对应 Handle 的引用计数。这需要 ECS 的变更检测或生命周期事件支持。我们在 [[事件总线]] 中讨论过延迟事件队列，这里可以用同样的机制：Entity 销毁时发送 `ComponentRemoved<MeshComponent>` 事件，资源系统消费事件并递减对应计数。

**失效条件**：资源生命周期需要立即响应的场景（如内存极度受限，必须毫秒级释放）。

### 分支 C：不引用计数，靠显式作用域/区域卸载

```cpp
class Level {
    Array<Handle<Mesh>> level_meshes;
public:
    ~Level() {
        for (auto h : level_meshes) asset_manager->unload(h);
    }
};
```

**核心思路**：资源的生命周期与关卡/场景/区域绑定。进入关卡时加载一批资源，离开关卡时全部卸载。组件里的 Handle 只是"借用"，不负责生命周期。

**适用场景**：线性关卡制游戏、资源需求可预测、内存预算宽松的项目。

**隐藏代价**：
- 无法处理跨关卡共享资源（如全局 UI 纹理、角色基础模型）。需要额外的"常驻资源"机制。
- 卸载粒度粗，可能造成内存浪费（关卡 A 和关卡 B 共享 90% 资源，但按关卡卸载会重复加载）。

**失效条件**：开放世界、动态流送、资源高度复用的场景。

### 引擎对照

> **参考：chaos / UE / Bevy 对引用计数是怎么做的？**
>
> 我们在解决的是「资源生命周期由谁决定」这个问题。
>
> - **chaos** 选择了全局引用计数表 + `PossessedHandle` RAII 包装。`AssetManager` 维护 `m_top_asset_file_id_map`，用 `PossessedHandle`（自动增减引用）持有顶层资源。底层 `DatumManager` 统一 GC。这种设计的动机是：chaos 需要支持运行时异步加载和编辑器热重载，引用计数是最安全的默认策略。
> - **UE** 选择了 UObject 垃圾回收（GC）而非引用计数。`UPROPERTY()` 标记的引用会被 GC 遍历，unreachable 对象定期批量回收。UE 的动机是：C++ 的循环引用（A 引用 B，B 引用 A）会让纯引用计数失效，而 GC 可以处理循环依赖。代价是 GC 暂停和内存峰值不可控。
> - **Bevy** 选择了 channel-based drop 通知 + 延迟回收。`StrongHandle` 被 Drop 时发送事件到 channel，`Assets::track_assets` System 在帧边界消费事件。Bevy 的动机是：ECS 的 System 调度天然适合"批量处理事件"，不需要每帧遍历所有组件。
>
> **默认推荐**：**分支 B 的变体——增量引用计数 + 延迟帧边界回收**。原因如下：
> 1. UE 的 GC 机制与 ECS 架构存在根本性冲突。ECS 强调数据局部性和确定性调度，GC 的"标记-清扫"是非确定性的、会暂停世界的。在 ECS 中重新表达 UE 的设计意图，应该保留"自动生命周期管理"的核心目标，但用 ECS 友好的事件驱动机制替代 GC。
> 2. Bevy 的 channel-based drop 通知是 ECS 下最优雅的引用计数表达。我们在 C++ 中可以用无锁队列（如 `moodycamel::ConcurrentQueue` 或自研的 SPSC 队列）实现同等语义。
> 3. chaos 的 `PossessedHandle` RAII 包装可以作为简化过渡路径（默认实现早期不需要 channel 事件，直接用 RAII 递减计数即可），但未来向 ECS 批量回收演进时，channel 事件是更干净的边界。
>
> **简化路径**：如果当前阶段工程负担过重，可先实现「Handle Table + RAII 包装（`OwnedHandle`）+ 手动 `unload`」。这相当于 chaos 的 `PossessedHandle` 简化版。但需在代码中预留 `DropEvent` 接口，明确标注"未来需替换为 ECS 事件驱动的批量回收"。

### 状态变化图：引用计数与回收

```
n 个 Entity 持有 Handle<Mesh> {index=3, gen=7}
n >= 1 时：
+-------------------------------------+
| slots[3] = { data: Mesh(...),       |
|             generation: 7,          |
|             occupied: true }        |
| ref_counts[3] = n                   |
+-------------------------------------+
        v 所有 Entity 被销毁，无新引用
ref_counts[3] 变为 0
        v 帧边界 flush_pending_frees()
slots[3].occupied = false
slots[3].generation = 8  <- 旧 Handle 的 gen=7 不再匹配
        v 新资源申请槽位
slots[3] 被复用，generation=8
新 Handle {index=3, gen=8} 有效
旧 Handle {index=3, gen=7} 访问时 try_get() 返回 nullptr
```

**诚实性检查**：这个方案引入了什么新成本？
- 引用计数表是额外的内存（每个资源一个 `uint32_t`，对于几千个资源可以忽略）。
- 延迟回收意味着资源可能多存活几帧。如果内存极度紧张，可以改为"零引用立即回收"，但会失去帧边界安全性。
- 循环引用仍然无法自动处理。如果资源 A 引用资源 B，资源 B 又引用资源 A，引用计数永远不会归零。解决方案在下一个问题中讨论。

**遗留问题**：引用计数解决了"什么时候释放"，但资源的加载过程本身可能很慢（磁盘 I/O、网络下载、解压解码）。如果在主线程阻塞加载，帧率会直接崩溃。我们需要异步加载。

## 问题 3：异步加载的核心矛盾是什么？

游戏主线程的运行节奏是固定的：16ms 一帧（60fps），或者 33ms（30fps）。在这个时间预算里，它需要完成输入处理、Gameplay 模拟、物理更新、渲染提交。如果突然需要加载一张 4K 纹理（从 SSD 读取可能耗时 50~100ms），主线程不能停下来等。

### 根因分析：I/O 延迟与帧率预算的根本冲突

这不是"优化"问题，而是**架构层面的不可调和矛盾**：
- CPU 内存访问：纳秒级
- SSD 顺序读取：微秒~毫秒级
- SSD 随机读取：毫秒级
- 网络加载：几十到几百毫秒
- 纹理解码（PNG/JPEG -> raw）：每兆字节数毫秒

一帧只有 16ms。任何涉及磁盘或网络的加载操作，如果阻塞主线程，必然导致帧率下降。

### naive 方案：开一个新线程 `std::thread` 去读文件

```cpp
Mesh* load_mesh_blocking(const char* path) {
    auto bytes = read_file(path);           // 50ms 阻塞
    auto mesh = parse_mesh(bytes);          // 20ms 阻塞
    return mesh;
}

// 在某个 System 里
void spawn_enemy(...) {
    std::thread t([](const char* path){
        auto mesh = load_mesh_blocking(path);
        // ??? 怎么把 mesh 交给主线程？
    }, "enemy.mesh");
    t.detach();  // 灾难的开始
}
```

**问题**：
1. 线程创建销毁开销高，不能每加载一个资源就开一个线程。
2. 加载完成后怎么通知主线程？直接操作 ECS World 是线程不安全的。
3. 如果加载失败（文件不存在、格式错误），异常怎么传播？
4. 加载顺序怎么控制？敌人模型和它的纹理应该先加载哪个？

### 分支 A：单后台线程 + 双缓冲队列（chaos 模型）

```cpp
class AssetManager {
    std::thread* loading_thread;
    std::thread* postprocess_thread;
    BlockingQueue<LoadRequest>  pending_queue;
    BlockingQueue<LoadedResult> postprocess_channel;
};
```

**核心思路**：主线程把加载请求推入队列，一个专门的"加载线程"从队列取出请求、读取文件、反序列化。加载完成后把结果推入第二个队列，"后处理线程"负责依赖解析和回调触发。

**适用场景**：中型项目、加载任务类型相对统一、不需要极度并行化。

**代码片段：加载线程主循环**
```cpp
void AssetManager::loading_thread_loop() {
    while (!exit_flag) {
        LoadRequest req;
        if (pending_queue.wait_dequeue_timed(req, 100ms)) {
            auto data = archive->read(req.path);
            auto asset = deserializer->parse(data);
            postprocess_channel.enqueue(LoadedResult{req.token, asset});
        }
    }
}
```

**隐藏代价**：
- 单线程加载无法利用多核 CPU。如果同时需要加载 10 个资源，它们只能串行处理。
- 反序列化（如解析 JSON/XML/二进制格式）可能是 CPU 密集型的，放在加载线程会阻塞后续的文件读取。
- 两个线程之间的队列竞争成为瓶颈。

**失效条件**：开放世界、大量并发加载请求、CPU 解码耗时长的场景。

### 分支 B：线程池 + 任务图（现代 C++ 方案）

```cpp
struct LoadTask {
    String path;
    Function<void(AssetHandle)> on_complete;
    Array<LoadTask*> dependencies;  // 必须先完成的依赖任务
};

class AssetLoader {
    ThreadPool thread_pool{4};  // 4 个加载 worker
    
    TaskToken async_load(const String& path, Callback cb) {
        auto task = make_load_task(path);
        // 如果有依赖，先提交依赖任务
        for (auto dep : task->dependencies)
            thread_pool.submit(dep);
        thread_pool.submit(task);
        return task->token;
    }
};
```

**核心思路**：把加载拆分为可并行的子任务（读文件、解码、GPU 上传），用线程池调度。任务之间有显式依赖关系，构成 DAG（有向无环图）。

**适用场景**：大型项目、需要最大化磁盘和 CPU 并行度、任务依赖复杂的场景。

**隐藏代价**：
- 任务图调度本身有开销。对于小型项目，调度开销可能超过并行收益。
- 依赖图的管理复杂，循环依赖检测、任务取消、优先级调整都是棘手的工程问题。
- 回调/续体（continuation）的内存分配和生命周期管理需要小心。

**失效条件**：加载任务数量少、依赖简单的场景（任务图调度的固定开销占主导）。

### 分支 C：IO 任务池 + Channel 回调 + ECS 事件（Bevy 模型）

```rust
// Bevy 的核心设计
asset_server.load("player.png");  // 非阻塞，立刻返回 Handle<Image>

// 在 IoTaskPool（异步线程池）中执行
// 加载完成后发送 InternalAssetEvent::Loaded
// ECS System 在 PreUpdate 阶段消费事件，插入 Assets<T>
```

**核心思路**：加载完全异步化，主线程只发起请求和接收结果。结果通过 channel 传回，由 ECS System 在固定的调度点统一处理。这严格分离了"异步边界"和"同步边界"。

**在 C++ 中的映射**：
```cpp
class AssetServer {
    ThreadPool io_pool;
    ConcurrentQueue<InternalAssetEvent> event_queue;
    
public:
    Handle<Mesh> load(const Path& path) {
        Handle<Mesh> handle = allocate_handle<Mesh>();
        io_pool.submit([this, path, handle]() {
            auto data = vfs->read(path);
            auto mesh = mesh_loader->load(data);
            event_queue.enqueue(AssetLoadedEvent{handle, mesh});
        });
        return handle;  // 立刻返回，此时 mesh 还不存在
    }
    
    // 在主线程的 System 中每帧调用
    void process_events(World& world) {
        InternalAssetEvent evt;
        while (event_queue.try_dequeue(evt)) {
            match(evt, [&](AssetLoadedEvent& e) {
                world.resource_mut<Assets<Mesh>>()->insert(e.handle, e.data);
                world.send_event(AssetEvent<Mesh>::Added{e.handle});
            });
        }
    }
};
```

**适用场景**：ECS 原生引擎、需要严格主线程安全、事件驱动架构的项目。

**隐藏代价**：
- 加载完成到实际可用有一帧延迟（因为 System 在下一 Tick 才处理事件）。对于需要"加载完立刻使用"的场景（如脚本动态加载），这一帧延迟可能需要特殊处理。
- Channel 的内存分配和无锁实现需要精心设计。高并发场景下，MPSC queue 的伪共享是潜在瓶颈。

**失效条件**：需要同步加载的场景（如关卡切换时，必须等所有资源就绪才能继续）。需要同步路径作为补充。

### 引擎对照

> **参考：chaos / UE / Bevy 对异步加载是怎么做的？**
>
> 我们在解决的是「主线程不能被 I/O 阻塞」这个具体问题。
>
> - **chaos** 选择了**双后台线程 + 无锁队列**。加载线程专注 I/O 和反序列化，后处理线程专注依赖解析和回调。chaos 的动机是：资源加载后需要解析内部引用（如材质引用的纹理句柄），这个解析必须在被引用资源已加载后才能进行，因此需要一个独立的拓扑排序阶段。双线程模型把"读取"和"解析引用"解耦，避免加载线程被复杂的依赖逻辑阻塞。
> - **UE** 选择了**事件驱动加载器（EDL, AsyncLoading2）+ IO Dispatcher**。UE5 的 Zen/IoStore 架构把加载彻底异步化：后台线程通过 `FIoDispatcher` 读取数据块，主线程 Tick 时消费事件、创建 UObject 壳、执行 PostLoad。UE 的动机是：开放世界需要流送（streaming）大量资源，传统的同步 `LoadPackage` 完全无法应对。
> - **Bevy** 选择了**IoTaskPool + crossbeam channel + ECS System 网关**。Bevy 的动机非常纯粹：ECS 的 World 不是线程安全的，异步任务绝对不能直接修改 World。因此所有加载结果必须通过 channel 路由到一个持有 `&mut World` 的 System（`handle_internal_asset_events`），这是唯一能在异步和同步之间安全过渡的网关。
>
> **默认推荐：分支 C（IO 任务池 + Channel 事件 + ECS System 网关）**。
>
> 推荐依据：
> 1. **ECS 兼容性**：Bevy 的网关模式是 ECS 架构下处理异步结果的最安全方式。C++ 中可以用无锁 MPSC 队列（如 `rigtorp::SPSCQueue` 或自研的基于 `std::atomic` 的环形缓冲区）替代 Rust 的 crossbeam。
> 2. **UE 优先吸收**：UE5 EDL 的"事件驱动 + 后台 IO"思想应该被完全吸收。虽然我们不需要 UObject 的复杂性，但"后台线程读取数据 -> 主线程消费事件 -> 批量创建资源"的流水线是工业级验证的最优路径。
> 3. **chaos 的借鉴**：chaos 的 LoadingTree 拓扑后处理是一个重要洞察——依赖解析不能和 I/O 混在一起。在我们的 ECS 表达中，可以设计一个 `AssetDependencySystem`，它在 `process_events` 之后运行，负责解析刚加载资源的内部引用（如材质中的纹理句柄 -> 实际纹理数据）。
>
> **简化路径**：如果当前阶段不想引入线程池和 channel，可以先实现「单后台线程 + 互斥锁保护的事件队列」。这是 chaos 双线程模型的简化版。但需标注：单线程 I/O 吞吐是瓶颈，未来需升级到线程池 + 无锁队列。

### 精确性补充：加载状态机

一个资源的加载不是"非黑即白"的。精确的状态定义对调试和 AI 可观测性至关重要：

```cpp
enum class AssetLoadState : uint8_t {
    Unloaded,           // 未开始加载
    Pending,            // 已提交加载请求，等待 IO 线程处理
    Loading,            // IO 线程正在读取/解码
    Loaded,             // 数据已在内存，但依赖尚未解析
    Ready,              // 数据就绪，所有依赖也 Ready，可被 System 使用
    Failed,             // 加载失败（文件不存在、格式错误等）
    Unloading,          // 正在释放（引用计数归零后的过渡态）
};
```

**为什么需要区分 Loaded 和 Ready？**

假设材质 M 引用了纹理 T。T 先完成加载（状态变为 Loaded），但 M 还没加载完。此时 T 的数据虽然在内存中，但如果某个 System 试图渲染 M，会发现 M 还不存在。只有当 M 也完成加载，并且 M 内部的 `Handle<Texture>` 被解析为指向 T 的有效引用后，M 才算 Ready。

Bevy 用 `load_state` / `dep_load_state` / `rec_dep_load_state` 三层状态精确表达这个过程。我们的简化版可以合并为上述六态，但在调试工具和 AI Schema 中应暴露足够细粒度的信息。

**遗留问题**：异步加载解决了"不阻塞主线程"，但引入了新的复杂度——如果系统请求加载一个资源，立刻返回了 Handle，但此时资源还没加载完，其他 System 拿到这个 Handle 该怎么办？我们需要一个"占位符"或"默认回退"机制。更重要的是，资源之间有依赖关系（材质依赖纹理），这些依赖如何被自动追踪和解析？

## 问题 4：依赖关系怎么处理？

资源很少孤立存在。一个材质定义了表面的光照模型，但它需要引用一张漫反射纹理、一张法线纹理、可能还有一张粗糙度纹理。一个角色网格需要引用骨骼绑定和多个材质。一个关卡需要引用成百上千个静态网格和纹理。

### 场景绑定：加载一个 Material 需要多少步？

你调用 `asset_server.load("wood.mat")`，返回一个 `Handle<Material>`。你以为这就完了？不。解析 `wood.mat` 文件时发现它引用了 `wood_diffuse.png`、`wood_normal.png`、`wood_roughness.png`。这三张纹理必须先加载完，`wood.mat` 才算真正可用。

如果系统不处理这种依赖关系，开发者就必须手动保证加载顺序——先加载所有纹理，再加载材质。这在小型项目中还能应付，但在大型项目中，一个材质可能引用十几个资源，一个关卡可能引用几千个资源，手动管理是不可能的。

### 根因分析：依赖图是有向无环图（DAG），不是线性列表

资源的依赖关系构成一张 DAG。"木材材质"依赖"木材纹理"，"木材纹理"不依赖"木材材质"。但"门模型"可能依赖"木材材质"，"木材材质"又依赖"木材纹理"——这是一条链。更复杂的是，"地板材质"也依赖"木材纹理"，所以 DAG 中有合并点。

加载的挑战在于：**DAG 中的节点必须在所有前驱节点加载完成后才能被解析**。你不能先解析材质（因为纹理句柄还没指向有效数据），也不能无限期等待（因为材质本身可能正被渲染系统急需）。

### 分支 A：手动加载，不处理依赖（naive）

```cpp
// 开发者手动保证顺序
auto diffuse = asset_server.load("wood_diffuse.png");
auto normal  = asset_server.load("wood_normal.png");
// ... 等待所有纹理加载完成 ...
auto mat = asset_server.load("wood.mat");  // 现在安全了？
```

**问题**：
- 开发者必须知道每个资源的内部依赖，这是不可能的（资源文件格式是引擎内部细节）。
- 无法处理嵌套依赖（纹理 A 依赖调色板 B）。
- 代码与资源结构强耦合，美术修改资源后代码可能失效。

**适用场景**：仅用于理解"为什么需要自动依赖管理"。

### 分支 B：同步递归加载（阻塞式）

```cpp
AssetHandle load_with_dependencies(const Path& path) {
    auto asset = load_single(path);  // 阻塞 I/O
    for (auto dep_path : asset->internal_dependencies) {
        auto dep = load_with_dependencies(dep_path);  // 递归阻塞
        asset->resolve_reference(dep_path, dep);
    }
    return asset;
}
```

**核心思路**：加载一个资源时，递归加载它的所有依赖，全部完成后才返回。

**适用场景**：工具链（Cooker、编辑器导入）、启动时的强制预加载、小型项目。

**隐藏代价**：
- 阻塞主线程。如果依赖树很深（A→B→C→D→E），总阻塞时间是所有节点加载时间的和。
- 重复加载。如果 A 依赖 C，B 也依赖 C，递归会在 A 和 B 的加载过程中分别加载 C 两次（除非有去重缓存）。

**失效条件**：运行时动态加载、大型依赖树、需要保持帧率的场景。

### 分支 C：异步依赖图 + 拓扑排序（chaos LoadingTree 模型）

```cpp
struct LoadingNode {
    AssetHandle handle;
    Array<LoadingNode*> dependencies;   // 我依赖谁
    Array<LoadingNode*> dependents;     // 谁依赖我
    uint32_t unresolved_deps = 0;       // 还剩多少依赖没加载完
};

class LoadingGraph {
    HashMap<Path, LoadingNode*> nodes;
    
    void on_asset_loaded(AssetHandle h) {
        auto node = nodes[h.path];
        for (auto dep : node->dependents) {
            dep->unresolved_deps--;
            if (dep->unresolved_deps == 0) {
                // 所有依赖都完成了，可以解析引用了
                submit_for_postprocess(dep);
            }
        }
    }
};
```

**核心思路**：加载请求提交时，先扫描资源的依赖列表，构建一张 LoadingGraph。每个节点记录"我还剩多少依赖没完成"。当某个资源加载完成时，通知所有依赖它的节点，递减计数器。当计数器归零，该节点进入"可解析"状态，提交到后处理队列。

**适用场景**：大型项目、复杂依赖关系、需要精确控制加载顺序的场景。

**代码片段：反向依赖传播**
```cpp
// 加载线程完成一个资源后
void AssetManager::on_load_complete(const LoadedResult& result) {
    auto& node = loading_graph[result.path];
    node.state = LoadState::Loaded;
    
    // 反向通知：所有等待这个资源的节点
    for (auto* dependent : node.dependents) {
        dependent->pending_deps--;
        if (dependent->pending_deps == 0) {
            // 拓扑条件满足，提交后处理
            postprocess_queue.enqueue(dependent->path);
        }
    }
}

// 后处理线程
void AssetManager::postprocess_loop() {
    while (running) {
        Path path;
        if (postprocess_queue.wait_dequeue(path)) {
            auto& node = loading_graph[path];
            auto asset = storage->get(node.handle);
            for (auto& ref : asset->internal_references) {
                ref.resolve(storage->get(ref.handle));  // 此时 Handle 已指向有效数据
            }
            node.state = LoadState::Ready;
            fire_callback(node.on_ready);
        }
    }
}
```

**隐藏代价**：
- 依赖图需要内存。对于一次性加载几千个资源的关卡切换，图本身可能有数万个节点。
- 循环依赖检测。如果资源 A 依赖 B，B 又依赖 A（这通常是资源制作错误），加载会死锁。需要显式检测循环并报错。
- 取消加载的复杂度。如果用户在加载中途切换关卡，需要安全地取消整个图的所有 pending 任务。

**失效条件**：依赖关系极简单（如所有资源无依赖、或只有一层依赖），图的维护开销得不偿失。

### 分支 D：反向依赖传播 + ECS 事件（Bevy 模型）

Bevy 的依赖处理与 chaos 类似，但完全嵌入 ECS 事件系统：

```rust
// Bevy 的 AssetInfo 同时维护了两组信息
struct AssetInfo {
    loading_dependencies: HashSet<ErasedAssetIndex>,      // 我在等谁
    dependents_waiting_on_load: HashSet<ErasedAssetIndex>, // 谁在等我
}

// 当资源 C 加载完成
fn propagate_loaded_state(infos: &mut AssetInfos, loaded: ErasedAssetIndex) {
    // 找到所有等待 loaded 的节点
    for waiting_id in infos[loaded].dependents_waiting_on_load {
        infos[waiting_id].loading_dependencies.remove(&loaded);
        if infos[waiting_id].loading_dependencies.is_empty() {
            // 所有直接依赖完成了，但递归依赖呢？
            // Bevy 还会检查 rec_dep_load_state...
        }
    }
}
```

**核心思路**：与分支 C 相同，都是反向传播 + 拓扑排序，但实现上完全使用 ECS 的 Resource（`AssetInfos`）和事件（`InternalAssetEvent::LoadedWithDependencies`）。

**适用场景**：ECS 原生引擎、希望所有状态都在 ECS 框架内表达的项目。

**隐藏代价**：
- `AssetInfos` 作为 ECS Resource，每次修改需要 `ResMut<AssetInfos>`，这会与读取 `AssetInfos` 的其他 System 产生调度冲突。
- Bevy 用 `RwLock<AssetInfos>` 来缓解，但高频的加载/卸载场景下锁竞争不可忽视。

### 决策分析

| 条件 | 推荐方案 |
|------|---------|
| 依赖关系简单（≤2 层） | 分支 B（同步递归）在工具链中可用；运行时推荐分支 C/D |
| 复杂 DAG、大型项目 | 分支 C 或 D |
| ECS 原生、事件驱动优先 | 分支 D |
| 需要精确控制加载顺序和取消 | 分支 C（图结构更易操作） |

**默认推荐：分支 C 的图结构 + ECS 事件包装**。

推荐依据：
1. **UE 优先吸收**：UE 的 `FLinkerLoad` 在处理 Package 依赖时本质上也是拓扑排序（ImportMap → 递归加载外部包）。UE 的 EndLoad 阶段按磁盘偏移排序后批量序列化，也是为了最大化依赖加载的效率。我们提取 UE 的"依赖必须拓扑处理"的设计意图，在 ECS 中重新表达为 LoadingGraph + 反向传播。
2. **chaos 的验证**：chaos 的 `LoadingTree` + `pushFront` 深度优先加载 + 拓扑后处理已经在生产环境中验证了这个模式的正确性。
3. **ECS 映射**：LoadingGraph 本身可以作为 ECS Resource（`ResMut<LoadingGraph>`），后处理可以是一个 ECS System（`AssetPostprocessSystem`），在 `PreUpdate` 阶段运行。这比 Bevy 的 `RwLock<AssetInfos>` 更符合 ECS 的"System 调度"哲学——数据是 Resource，处理是 System。

**遗留问题**：依赖解析完成后，资源数据存在哪里？ECS 组件里只存 Handle，实际的 Mesh 顶点数据、纹理像素数据是巨大的。如果把这些大对象作为 ECS Resource，会不会有性能和设计问题？

## 问题 5：资源数据在 ECS 中怎么存储？

这是一个架构层面的问题：ECS 的哲学是"数据平铺在组件中、System 批量处理"，但资源数据（如 100MB 的纹理、带 LOD 的网格）显然不能平铺在每个 Entity 的组件里。

### 根因分析：资源数据的"规模不兼容性"

ECS 组件的理想特征是：**小、固定大小、可预测、可批量迭代**。一个 `Transform` 组件是 12 字节的 position + 16 字节的 rotation，完美。但一个 `Mesh` 资源可能包含：
- 顶点数组（数万~数百万个 `Vertex`，每个 32~64 字节）
- 索引数组
- LOD 层级
- 骨骼绑定数据
- 材质引用

把这些塞进 `MeshComponent` 里，组件数组的步长会暴涨，缓存彻底失效。而且多个 Entity 共享同一个 Mesh 时，数据会被复制多份。

所以资源数据需要**集中存储、组件只存引用**。但这引出了一个新的问题：集中存储应该是什么形态？

### 分支 A：每类型一个 ECS Resource（Bevy 风格）

```rust
// Bevy 的做法
#[derive(Resource)]
pub struct Assets<T: Asset> {
    dense_storage: DenseAssetStorage<T>,
    // ...
}

// System 访问
fn render_meshes(
    meshes: Query<&Handle<Mesh>>,
    assets: Res<Assets<Mesh>>,  // 通过 Resource 访问实际数据
) {
    for handle in &meshes {
        if let Some(mesh) = assets.get(handle) {
            // 渲染 mesh
        }
    }
}
```

**核心思路**：每种资源类型对应一个 ECS Resource。`Assets<Mesh>` 是 Resource，`Assets<Texture>` 是另一个 Resource。System 通过 `Res<Assets<T>>` 访问。

**适用场景**：ECS 原生引擎、类型安全优先、资源类型数量可控的项目。

**隐藏代价**：
- 在 C++ 中，模板化的 `Assets<T>` 需要显式实例化每种类型。虽然可以通过宏简化，但编译时间和二进制体积会增加。
- System 的依赖声明会变得冗长。一个渲染 System 可能需要 `Res<Assets<Mesh>>`、`Res<Assets<Texture>>`、`Res<Assets<Material>>`……参数列表膨胀。
- 跨类型统一操作困难。如"获取所有资源的内存占用"需要遍历所有已注册的类型。

**失效条件**：资源类型极多（几十上百种）、需要跨类型统一查询的场景。

### 分支 B：专门的资源子系统，ECS 只存门面（UE/chaos 风格）

```cpp
// 资源子系统是引擎核心子系统之一，不直接是 ECS Resource
class AssetManager {
    HashMap<TypeId, void*> type_storages;  // type-erased storage
public:
    template<typename T>
    T* get(Handle<T> handle);
    
    template<typename T>
    Handle<T> load(const Path& path);
};

// ECS 中只存一个门面引用（可选）
struct AssetManagerRef : Resource {
    AssetManager* ptr;
};
```

**核心思路**：资源管理是独立的子系统，有自己的存储、加载、生命周期管理。ECS 世界可以持有一个 `AssetManagerRef` Resource 作为访问门面，但核心逻辑不在 ECS System 中。

**适用场景**：大型引擎、资源系统先于 ECS 存在、或需要与编辑器/工具链深度集成的项目。

**隐藏代价**：
- ECS 和资源系统之间的边界不够清晰。System 既要查询 ECS 组件，又要访问外部资源系统，代码风格不统一。
- AI 可观测性受损。如果资源数据不在 ECS 的组件/Resource 中，AI 通过 MCP 查询资源状态时需要额外的桥接层。

**失效条件**：追求纯 ECS 架构、希望所有状态都可被 ECS System 批量处理的项目。

### 分支 C：混合方案——ECS Resource 门面 + 类型擦除后端

```cpp
// 前端：ECS Resource，提供类型安全的访问接口
template<typename T>
struct Assets : Resource {
    HandleTable<T> storage;
    
    T* get(Handle<T> h) { return storage.try_get(h); }
    void insert(Handle<T> h, T value) { storage.insert(h, value); }
};

// 后端：统一的内存管理和加载调度
class AssetBackend {
    // 线程池、加载队列、依赖图、引用计数表...
};

// System 使用
void render_system(
    Query<const MeshComponent> meshes,
    Res<Assets<Mesh>> mesh_assets,
    Res<Assets<Texture>> tex_assets,
    Res<AssetBackend> backend  // 用于发起异步加载
) {
    for (auto [mesh_comp] : meshes) {
        if (auto* mesh = mesh_assets->get(mesh_comp.handle))
            submit_to_renderer(mesh);
    }
}
```

**核心思路**：每种资源的实际数据存储是独立的 ECS Resource（`Assets<T>`），提供 O(1) 类型安全访问。但加载调度、线程池、依赖图等"基础设施"是共享的 `AssetBackend` Resource。

**适用场景**：大多数 ECS 引擎项目。兼顾了类型安全、ECS 哲学和工程实用性。

**隐藏代价**：
- 需要为每种资源类型注册 `Assets<T>`。可以用宏简化（`REGISTER_ASSET_TYPE(Mesh)`）。
- `Assets<T>` 的生命周期与 ECS World 绑定。World 销毁时所有资源一起释放，这在大多数情况下是期望的行为。

### 决策分析

**默认推荐：分支 C（混合方案）**。

推荐依据：
1. **Bevy 验证**：Bevy 的 `Assets<T>` 作为 ECS Resource 已经验证了这个设计的可行性。我们在 C++ 中可以用模板实现同等的类型安全。
2. **UE 设计意图吸收**：UE 的 `UObject` 全局表本质上也是"类型擦除的后端 + 类型化的前端（`Cast<T>()`）"。我们提取 UE 的"集中存储 + 类型安全访问"意图，用 ECS Resource 重新表达。
3. **AI 可观测性**：`Assets<T>` 作为 ECS Resource，可以通过反射系统导出 Schema，AI 可以通过 MCP 查询 `Assets<Mesh>` 中所有活跃资源的句柄和元数据。
4. **确定性**：资源存储在 ECS World 中，World 快照（我们在 [[系统调度与确定性]] 中讨论过）可以包含资源状态，支持回放和检查点。

**简化路径**：如果当前阶段资源类型很少（只有 Mesh 和 Texture），可以直接手写 `MeshStorage` 和 `TextureStorage` 两个独立的 ECS Resource，不需要模板泛化。但接口设计时应预留泛化空间（如统一的 `get/insert/remove` 命名约定）。

**遗留问题**：资源加载和管理的基础设施已经讨论得差不多了，但还有一个开发期效率的关键问题——热重载。当美术师修改了一张纹理，引擎能不能在不重启的情况下自动更新？

## 问题 6：热重载怎么实现？

开发期迭代效率直接决定项目进度。想象一个场景：美术师在 Photoshop 中修改了一张纹理，保存后，引擎中的材质应该自动更新，而不需要重启游戏、重新加载关卡、重新定位摄像机。

### 根因分析：开发期与运行期的需求差异

运行期的资源加载是"一次性"的：加载 → 使用 → 卸载。开发期的资源加载是"反复性"的：加载 → 修改 → 重载 → 再修改 → 再重载。这两套需求不能硬塞进同一个代码路径。

### 分支 A：手动重载（按按钮触发）

```cpp
// 开发者按下一个键或点击按钮
if (input.pressed(Key::F5)) {
    asset_manager->reload("wood_diffuse.png");
}
```

**核心思路**：显式触发重载。引擎提供 `reload(handle)` 或 `reload(path)` 接口，开发者手动调用。

**适用场景**：调试特定资源、验证修复效果。

**隐藏代价**：
- 开发者必须知道哪个资源被修改了。对于复杂材质，一张纹理可能被多个材质引用，手动逐个重载效率低下。
- 没有自动发现机制。美术师保存了文件，开发者可能忘记按 F5。

**失效条件**：频繁修改、大量资源的场景。

### 分支 B：文件系统 Watch + 脏标记 + 延迟替换（工业标准）

这是 chaos、UE、Bevy 都采用的核心方案。

**核心思路**：
1. **Watch**：启动一个文件系统监听器（如 Windows 的 `ReadDirectoryChangesW`、Linux 的 `inotify`、跨平台的 `notify` 库），监控资源目录的变更。
2. **Dirty Handle**：文件变更时，不立即替换资源，而是将对应的 Handle 标记为"脏"。
3. **帧边界替换**：在主线程的帧边界（如 `PostUpdate` 之后，`PreRender` 之前），统一处理所有 Dirty Handle，执行替换。

**代码片段：热重载流程**
```cpp
class HotReloadSystem {
    FileWatcher watcher;
    ConcurrentQueue<Path> changed_files;
    
public:
    void startup() {
        watcher.watch("assets/", [&](const Path& path) {
            changed_files.enqueue(path);
        });
    }
    
    // 每帧在主线程运行
    void tick(World& world, ResMut<AssetBackend> backend) {
        Path path;
        while (changed_files.try_dequeue(path)) {
            auto handle = backend->find_handle(path);
            if (handle.valid()) {
                // 标记为脏，延迟到帧边界替换
                backend->mark_dirty(handle);
            }
        }
    }
    
    // 帧边界执行
    void flush_dirty(World& world) {
        for (auto handle : backend->dirty_handles()) {
            // 1. 异步重新加载资源数据
            backend->reload_async(handle);
            
            // 2. 新数据加载完成后，在 Assets<T> 中原地替换
            // 注意：这里不是删除旧数据再插入新数据，而是复用同一个 Handle
            // 这样所有引用该 Handle 的组件都不需要更新
        }
        backend->clear_dirty();
    }
};
```

**适用场景**：所有需要开发期热重载的项目。

**隐藏代价**：
- 文件系统 Watch 有平台差异。Windows、macOS、Linux 的 API 完全不同，需要抽象层。
- "变更"的定义复杂。有些编辑器保存文件时会先写入临时文件再重命名，Watch 可能捕获到多次事件。需要防抖（debounce）机制。
- 原地替换的线程安全。如果渲染线程正在读取纹理数据，主线程不能同时替换它。必须确保替换发生在渲染使用的安全点（如帧边界、或配合 GPU fence）。

**失效条件**：某些资源类型不支持原地替换（如需要重新编译的着色器、需要重建加速结构的网格）。这些资源可能需要更重的"卸载再加载"流程。

### 引擎对照

> **参考：chaos / UE / Bevy 对热重载是怎么做的？**
>
> 我们在解决的是「开发期资源修改后如何自动更新」这个问题。
>
> - **chaos** 选择了**时间戳比对 + 脏 Handle 队列**。编辑器模式下，`AssetManager` 维护 `m_file_last_modified_time_when_load_map`，通过文件修改时间判断是否需要重载。重载时通过 `override_asset_handle` 复用已有内存，保持所有外部引用有效。chaos 的动机是：编辑器需要频繁热更，且重载时必须保证所有引用不断裂。
> - **UE** 选择了**DirectoryWatcher + 资产重新导入 + Live Coding**。UE 的编辑器有完整的资产导入管线，源文件（如 FBX、PNG）变更后，DirectoryWatcher 触发重新导入（Reimport），生成新的 `.uasset`，然后通知运行时加载新包。UE 的动机是：资产导入涉及格式转换（如 PNG → 平台特定纹理格式），热重载必须走完整的导入管线。
> - **Bevy** 选择了**AssetWatcher + AssetProcessor 两阶段热重载**。Bevy 0.15+ 的 `AssetProcessor` 会先处理源文件变更（如压缩、转换），输出到 `imported_assets/` 目录，然后主 `AssetServer` 从 processed 目录重新加载。Bevy 的动机是：分离"源资产"和"运行时资产"，确保引擎加载的永远是优化后的格式。
>
> **默认推荐：分支 B（文件 Watch + 脏 Handle + 帧边界原地替换）**。
>
> 推荐依据：
> 1. **UE 优先吸收**：UE 的"脏检测 + 延迟替换"思想是工业标准。我们不需要 UE 的完整导入管线（那是 [[资产导入管线]] 的主题），但"检测变更 → 标记脏 → 安全点替换"的流程应该直接吸收。
> 2. **chaos 的洞察**：chaos 的 `override_asset_handle` 复用内存是一个关键技巧——Handle 不变，只替换 Handle 指向的数据。这保证了所有组件中的 Handle 引用自动生效，不需要遍历所有 Entity 更新组件。
> 3. **ECS 映射**：热重载可以是一个 ECS System（`HotReloadSystem`），在 `PreUpdate` 阶段检查文件变更，在 `PostUpdate` 阶段执行替换。这与 ECS 的调度模型完全一致。
>
> **简化路径**：如果当前阶段不想引入文件系统 Watch 库，可以先实现「手动 F5 触发 reload()」。这是分支 A 的简化版，但代码结构应预留 Watch 接口。

### 关键决策：替换时如何保证渲染安全？

热重载最危险的瞬间是"替换发生时，GPU 是否正在使用旧数据"。对于 CPU 端资源（如网格顶点数据），如果渲染 System 已经提交了 Draw Call，GPU 可能还在读取旧顶点缓冲区。此时如果主线程释放旧缓冲区，会导致 GPU 崩溃。

解决方案：
1. **延迟释放**：替换后，旧数据不立即释放，而是标记为"待释放"，等待一帧（或直到 GPU fence 信号）后再释放。
2. **双缓冲**：对于关键资源（如纹理），维护两个 GPU 资源句柄。替换时切换到新句柄，旧句柄在几帧后释放。
3. **帧边界同步**：所有替换操作严格发生在 `Present` 之后、`BeginFrame` 之前。此时 GPU 已完成上一帧的所有工作。

对于默认实现，**方案 3（帧边界同步）** 是最简单可靠的。在 [[RHI抽象层]] 中，我们已经讨论了帧生命周期和命令列表提交。热重载的替换可以绑定到 `EndFrame` 阶段。

**遗留问题**：热重载解决了开发期效率，但资源的"原始格式"和"运行时格式"通常是不同的（如 PNG vs GPU 压缩纹理）。谁来负责格式转换？这就是导入管线（Import Pipeline）的职责。

## 问题 7：导入管线与格式转换

游戏资源不是从美术软件直接塞进引擎的。Photoshop 导出 PNG，Blender 导出 FBX，但这些格式都不适合直接 GPU 渲染。PNG 需要解码、可能还需要生成 mipmap、压缩为 BC7/ASTC；FBX 需要解析场景图、提取顶点数据、生成 LOD、烘焙动画。

### 场景绑定：为什么不能用原始文件直接运行？

假设你直接用 PNG 作为纹理运行时格式。每次加载纹理时，引擎都需要：
1. 从磁盘读取 PNG（可能 5MB）
2. CPU 解码为 raw RGBA（可能 64MB）
3. 生成 mipmap（额外 21MB）
4. 上传到 GPU

这个过程在运行时每次加载都要执行。如果关卡有 100 张纹理，启动时间会被拖慢几十秒。而且不同 GPU 支持的压缩格式不同（PC 用 BC7，iOS 用 ASTC，旧 Android 用 ETC2），运行时动态压缩太慢。

### 根因分析：源格式与运行时格式的根本差异

| 维度 | 源格式（PNG/FBX/OBJ） | 运行时格式 |
|------|----------------------|-----------|
| 目标 | 人类可读、可编辑、无损 | GPU 可直接使用、内存紧凑、加载快 |
| 压缩 | 通用压缩（PNG deflate、ZIP） | GPU 原生压缩（BC7/ASTC/ETC2） |
| 结构 | 保留编辑信息（图层、骨骼、关键帧） | 扁平化（顶点缓冲区、索引缓冲区、纹理数组） |
| 平台 | 平台无关 | 平台特定（不同 GPU 用不同格式） |

源格式和运行时格式的差异如此之大，以至于它们不能是同一个文件。

### 分支 A：运行时即时转换（最简单，但最慢）

**核心思路**：引擎在加载原始文件时，现场执行所有转换（解码、生成 mipmap、压缩、平台适配）。

**适用场景**：原型验证、资源数量极少、没有性能要求的教学项目。

**隐藏代价**：
- 启动极慢。每次运行都要重复相同的转换工作。
- 运行时内存峰值高。解码后的 raw 数据可能比最终 GPU 格式大 4~8 倍。
- 无法利用平台特定优化。如主机平台的专用纹理布局、GPU 的异步解压特性。

**失效条件**：任何超过十几个资源的项目。

### 分支 B：Cook 管线——源资产与引擎资产分离（UE/chaos 标准）

```
美术工作区                  Cooker（构建时）              运行时
------------                --------------               --------
wood.png  ──►  导入管线  ──► wood.texture  ──► Pak 包 ──► 引擎加载
wood.fbx  ──►  （编辑器）  ──► wood.mesh     ──►        ──► GPU 直接使用
```

**核心思路**：
1. **源资产**（Source Assets）：美术直接编辑的文件，存放在版本控制中（如 `SourceAssets/`）。
2. **引擎资产**（Engine Assets）：Cooker 生成的运行时格式，存放在构建输出目录（如 `CookOutput/`）。
3. **导入管线**：定义每种源格式到引擎格式的转换规则。如 PNG → 纹理 Cooker → `.texture` 二进制文件。
4. **增量 Cook**：只处理修改过的文件。通过时间戳或哈希判断哪些源资产需要重新 Cook。

**代码片段：Cook 配置示例**
```json
{
    "importers": {
        "*.png": {
            "target_format": "texture",
            "generate_mipmaps": true,
            "compression": "bc7",
            "platform_overrides": {
                "ios": { "compression": "astc_4x4" }
            }
        },
        "*.fbx": {
            "target_format": "mesh",
            "generate_lods": [0.5, 0.25, 0.125],
            "collision": "convex_hull"
        }
    }
}
```

**适用场景**：所有商业游戏项目、任何需要发布到多平台的项目。

**隐藏代价**：
- Cook 管线是重大工程。需要为每种资源类型写 Cooker，处理错误、依赖追踪、增量更新。
- 源资产和引擎资产之间的同步问题。如果美术修改了 PNG，但 Cooker 没有运行，运行时加载的是旧版本。
- 构建时间增加。完整 Cook 可能需要数分钟到数小时（取决于资源总量）。

**失效条件**：纯原型项目、没有美术管线的小型项目。

### 分支 C：按需生成 + 缓存（中间路线）

**核心思路**：没有独立的 Cooker 程序，但引擎在首次加载某个源资产时，自动转换为运行时格式并缓存到磁盘。下次加载直接读取缓存。

**适用场景**：小型团队、快速原型、不想维护独立 Cooker 管线的项目。

**隐藏代价**：
- 首次加载仍然慢（需要现场转换）。
- 缓存失效逻辑复杂。如果转换规则变了（如从 BC7 改为 ASTC），需要清空缓存重新生成。
- 多平台支持困难。缓存文件通常是平台特定的，需要按平台分目录。

**失效条件**：资源量大、需要精确控制构建产物的项目。

### 决策分析

**默认推荐：分支 B（Cook 管线）作为最终目标，分支 C（按需生成）作为开发期简化路径。**

推荐依据：
1. **UE 优先吸收**：UE 的 Cooker (`UnrealFrontend` / `UE4Editor-Cmd`) 是游戏行业的事实标准。"源资产 → 导入 → Cook → Pak → 运行时加载"的链路经过了无数项目的验证。
2. **chaos 的验证**：chaos 的 `AssetImporter`（三阶段：PreProcessing → Import → Attach）和 `ResourceCooker`（Dirty 检查 + 增量 Cook）证明了这条路径在自研引擎中的可行性。
3. **默认实现取舍**：开发初期，维护完整的 Cooker 管线可能过重。可以先实现「运行时按需转换 + 本地缓存」，等资源量增大后再升级为独立 Cooker。

**ECS 映射**：Cooker 本身不是 ECS 的一部分，它是编辑器/工具链模块。但 Cook 产物的运行时加载完全由 ECS 资源系统管理。在 [[资产导入管线]]（阶段 8.1）中，我们会深入讨论 Cooker 的架构设计。

**遗留问题**：资源系统的设计空间已经覆盖得差不多了。但作为 AI 友好引擎，还有一个必须回答的问题：AI 如何观察资源状态？AI 如何安全地操作资源？

## 问题 8：AI 可观测性与确定性设计

我们的引擎目标明确：**为 AI 协作开发而设计**。资源系统作为引擎的核心基础设施，必须回答以下问题。

### 可观测性：AI 如何理解资源状态？

AI 不应该需要读 C++ 头文件才能知道"当前加载了哪些纹理"。资源系统的所有状态必须是**自描述的**。

**具体设计**：

1. **资源类型注册表**：每种资源类型在初始化时向反射系统注册，提供 Schema（字段名、类型、默认值）。
   ```json
   {
     "type": "Texture",
     "fields": {
       "width": { "type": "u32", "readonly": true },
       "height": { "type": "u32", "readonly": true },
       "format": { "type": "enum", "values": ["RGBA8", "BC7", "ASTC"] },
       "mip_count": { "type": "u32", "readonly": true },
       "memory_size": { "type": "u64", "readonly": true }
     }
   }
   ```

2. **加载状态查询**：AI 可以通过 MCP 接口查询任意资源的加载状态。
   ```json
   // AI 请求
   { "tool": "query_asset_state", "args": { "handle": "Texture#42" } }
   
   // 引擎响应
   { "state": "Ready", "path": "wood_diffuse.png", "refs": 15, "memory": 16777216 }
   ```

3. **引用图导出**：AI 可以查询"哪些 Entity 引用了这张纹理"，用于影响分析和变更传播。

### 确定性：给定相同输入，资源加载是否可复现？

确定性对调试、回放、测试至关重要。如果资源加载顺序受线程调度影响，两次运行的内存布局可能不同，导致难以复现的 bug。

**具体设计**：

1. **加载顺序确定性**：如果两个资源没有依赖关系，它们的加载顺序仍然应该是确定的（如按路径字母序）。避免使用无序容器（`std::unordered_map`）存储待加载队列。
2. **异步完成事件顺序**：`process_events` System 应该按事件到达顺序处理，而不是按资源类型或其他非确定性规则。
3. **World 快照包含资源**：我们在 [[系统调度与确定性]] 中讨论过 World 快照。资源存储作为 ECS Resource，其内容应可被序列化为快照的一部分。这样，从快照恢复时，资源状态与 ECS 状态完全一致。

### 工具边界：AI 通过 MCP 能做什么？

AI 操作资源的接口必须**结构化、可验证、可撤销**。

| 操作 | MCP 工具 | 验证规则 | 安全级别 |
|------|---------|---------|---------|
| 查询资源状态 | `query_asset` | 只读，无限制 | 安全 |
| 发起异步加载 | `load_asset` | 路径白名单 | 安全 |
| 热重载资源 | `reload_asset` | 仅限开发模式 | 需确认 |
| 卸载资源 | `unload_asset` | 检查引用计数 | 危险 |
| 修改资源数据 | `edit_asset` | 类型校验 + 范围检查 | 危险 |

**Agent 安全**：危险操作（如卸载、修改）应通过事务机制实现：
1. AI 提交操作请求。
2. 引擎生成变更预览（diff）。
3. 人类开发者审批（或自动白名单通过）。
4. 操作在帧边界原子提交，支持 Undo。

这在 [[多Agent编排与沙箱]]（阶段 8.6）中会深入讨论。

## 总结：默认路径与简化路径

### 工业级默认推荐

基于以上全部分析，个人 ECS 引擎的资源管理系统默认推荐以下架构：

```
+---------------------------------------------------+
|                     主线程（ECS World）              |
|  +------------------+  +------------------------+  |
|  | MeshComponent    |  | Assets<Mesh> (Resource) |  |
|  |   Handle<Mesh>   |  |   HandleTable<Mesh>     |  |
|  +------------------+  +------------------------+  |
|  +------------------+  +------------------------+  |
|  | MaterialComponent|  | Assets<Texture>         |  |
|  |   Handle<Mat>    |  |   HandleTable<Texture>  |  |
|  +------------------+  +------------------------+  |
|                                                    |
|  Systems:                                          |
|  - AssetEventProcessSystem (PreUpdate)            |
|  - AssetDependencySystem   (PreUpdate)            |
|  - HotReloadSystem         (PostUpdate)           |
|  - RefCountUpdateSystem    (EndFrame)             |
+---------------------------------------------------+
                        ^
                        | 无锁 MPSC Channel
                        v
+---------------------------------------------------+
|                   IO 线程池                          |
|  - 文件读取 / 解码 / 反序列化                         |
|  - 完成后发送 AssetLoadedEvent 到 Channel            |
+---------------------------------------------------+
                        ^
                        | VFS 抽象
                        v
+---------------------------------------------------+
|                文件系统 / Pak 包                      |
+---------------------------------------------------+
```

**核心组件清单**：

| 组件 | 职责 | 对应笔记 |
|------|------|---------|
| `Handle<T>` | 类型安全句柄（index + generation）| 问题 1 |
| `HandleTable<T>` | 密集存储 + ABA 防护 | 问题 1 |
| `Assets<T>` | ECS Resource，封装 HandleTable | 问题 5 |
| `AssetBackend` | ECS Resource，加载调度、线程池、引用计数 | 问题 2~4 |
| `LoadingGraph` | ECS Resource，异步依赖 DAG | 问题 4 |
| `AssetEvent<T>` | ECS 事件（Added/Modified/Removed/Ready）| 问题 3 |
| `FileWatcher` | 文件系统监控（开发期）| 问题 6 |
| `AssetImporter` | Cook 管线入口（工具链）| 问题 7 |

### 简化路径（阶段 5 初版可落地）

如果当前阶段工程负担过重，可按以下优先级实现：

1. **必做**：`Handle<T>` + `HandleTable<T>` + `Assets<T>`（ECS Resource）。这是整个系统的基石。
2. **必做**：单后台线程 + 互斥锁队列 + 同步 `load()` 接口。至少支持非阻塞的异步加载回调。
3. **必做**：引用计数表 + 手动 `unload()`。先不实现自动回收，但接口预留。
4. **选做**：文件系统 Watch + 手动 `reload()` 触发。先不做自动检测。
5. **选做**：Cook 管线。开发期先用原始格式（PNG/FBX）直接加载，阶段 8.1 再升级为正式 Cook 管线。

**简化路径与工业级方案的差距**：
- 缺少自动依赖图（需要手动保证加载顺序）。
- 缺少无锁 channel（使用互斥锁队列）。
- 缺少帧边界批量回收（使用即时 unload）。
- 缺少自动热重载（使用手动触发）。
- 缺少多平台 Cook（使用运行时即时转换）。

这些差距应在后续阶段（阶段 8~9）逐步补齐。

### 设计权衡表

| 决策点 | 默认推荐 | 简化替代 | 切换条件 |
|--------|---------|---------|---------|
| 句柄类型 | Handle Table + 世代索引 | 裸指针（仅原型） | 需要 ABA 防护时 |
| 引用计数 | 全局表 + 延迟批量回收 | 手动 unload | 需要自动生命周期时 |
| 异步加载 | IO 线程池 + Channel 事件 | 单后台线程 + 锁队列 | 并发加载 > 5 个时 |
| 依赖处理 | LoadingGraph + 拓扑后处理 | 手动顺序加载 | 资源有内部引用时 |
| 资源存储 | `Assets<T>` ECS Resource | 全局裸数组 | 任何非原型阶段 |
| 热重载 | 文件 Watch + 帧边界替换 | 手动 F5 触发 | 开发期迭代频繁时 |
| 导入管线 | Cooker + 源/运行时分离 | 运行时即时转换 | 资源量 > 50 或需多平台时 |
