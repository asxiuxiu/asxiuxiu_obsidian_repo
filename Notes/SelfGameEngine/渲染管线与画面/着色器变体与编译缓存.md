> **前置依赖**：[[材质系统架构]]
> **本模块增量**：掌握着色器变体爆炸的工业级控制方案——动态分支、静态变体、特化常量、按需异步编译、多级缓存。你将能够设计一个编译不卡顿、内存不爆炸的着色器管理系统。
>
> 本笔记探讨的核心问题是：**一个材质模板可能产生数十种 GPU 状态组合（有无法线贴图、延迟/前向、骨骼动画开关等）。这些组合如果全编译，时间和内存都会爆炸。如何只编译"需要的"，且不让玩家感受到编译卡顿？**

# 问题 2：如何处理着色器变体爆炸？

## 场景与根因

回到"生锈金属"材质。美术可能给它配置了这些开关：
- 是否使用法线贴图？
- 是否使用骨骼动画？
- 延迟渲染还是前向渲染？
- 低画质还是高画质？
- 是否开启视差遮蔽映射？

如果每个开关都是二值的，5 个开关就是 32 种组合。10 个开关就是 1024 种。每种组合都需要独立的 GPU 管线状态对象（PSO）和编译后的着色器字节码。这被称为**着色器变体爆炸（Shader Permutation Explosion）**。

根因有两个层面：
1. **GPU 执行层面**：GPU 的 SIMD 架构喜欢执行完全相同的字节码。如果同一次绘制中有的像素要算法线贴图、有的不要，就会产生分支分歧（branch divergence），降低利用率。
2. **API 设计层面**：现代图形 API（D3D12/Vulkan）要求 PSO 提前烘焙几乎所有状态（着色器代码、顶点格式、混合模式、深度模式等），导致状态组合的笛卡尔积爆炸。Sebastian Aaltonen 指出，现代游戏的本地 PSO 缓存可能超过 100GB。

## 分支研究

### 分支 A：运行时动态分支（if/else）

**核心思路**：用一个统一的着色器，通过 uniform 变量在运行时选择分支。

```glsl
// 分支 A：运行时 if
uniform bool u_use_normal_map;

void main() {
    vec3 N;
    if (u_use_normal_map) {
        N = texture(normal_map, uv).rgb * 2.0 - 1.0;
    } else {
        N = vertex_normal;
    }
    // ... lighting
}
```

**适用场景**：开关影响的代码路径极短、分歧概率低、对性能不敏感的平台（如移动端某些简化场景）。

**隐藏代价**：
- **Wave divergence**：一个 wave（如 32 或 64 线程）中只要有一个像素走 `true` 分支、一个走 `false` 分支，GPU 就要串行执行两个分支，有效利用率腰斩。
- **寄存器压力**：为了容纳所有分支的临时变量，编译器会分配更多寄存器，导致 occupancy 下降。

**失效条件**：分支内有纹理采样（高延迟）或复杂数学运算，且场景中同时存在两种状态的物体被绘制到同一批次。

### 分支 B：编译期全排列（静态变体）

**核心思路**：每个开关组合在编译期生成独立的着色器源码，通过宏定义或源码生成控制。

```cpp
// 分支 B：编译期宏控制（概念）
// 生成的源码之一："有骨骼 + 有法线 + 延迟渲染"
#define ENABLE_SKINNING 1
#define ENABLE_NORMAL_MAP 1
#define PASS_DEFERRED 1
#include "material_template.glsl"
```

**适用场景**：对性能极度敏感的主机游戏、已知固定组合的项目。

**隐藏代价**：
- **编译时间爆炸**：N 个开关产生 2^N 个编译任务。UE5 中大型项目的首次着色器编译可达数小时。
- **存储爆炸**：每个变体都存储独立的字节码。DDC 缓存体积膨胀。
- **运行时内存爆炸**：加载关卡时可能需要同时驻留数百种变体。

**失效条件**：开关数量超过 6~7 个，或项目规模中等以上。

### 分支 C：按需编译 + 多级缓存（UE 方案的核心）

**核心思路**：不预编译所有变体，而是**运行时按需请求 → 异步编译 → 全局缓存**。

```cpp
// 分支 C：按需异步编译（概念）
enum class TechniqueState { Invalid, Pending, Valid, Empty };

struct ShaderTechnique {
    std::atomic<TechniqueState> state;
    // 异步构建完成前，state = Pending
    // 构建完成后，state = Valid，触发等待的回调
};

MaterialInstance::GetTechnique(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (cache.contains(id)) return cache[id];
    
    // 未命中：加入 pending 队列，返回 fallback
    pending_builds.push_back(id);
    return fallback_technique;
}
```

**关键机制**：
1. **延迟构建**：首帧请求变体时，不阻塞渲染线程，而是加入后台队列，当前帧使用一个默认的 fallback 材质（如纯粉色或简化光照）。
2. **异步编译**：后台线程（或独立进程，如 UE 的 ShaderCompileWorker）执行实际编译。
3. **全局缓存**：编译产物写入 Derived Data Cache（DDC），跨会话、跨项目共享。下次启动时直接命中缓存，零编译时间。

**适用场景**：大型项目、编辑器环境、变体组合无法预先穷举的情况。

**隐藏代价**：
- **首帧/切换闪烁**：fallback 材质与真实材质差异明显时，玩家能看到" pop-in"。
- **缓存失效敏感**：引擎版本升级、着色器源码修改、平台切换都会导致 DDC 键变化，整批缓存失效。
- **缓存存储成本**：团队共享的 DDC 服务器可能存储 TB 级数据。

**失效条件**：对加载时间要求极高的竞技游戏（如 FPS），fallback 材质的视觉差异不可接受。

### 分支 D：特化常量（Specialization Constants）

**核心思路**：Vulkan 提供的机制，允许在管线创建时传入常量值，驱动在编译时做常量折叠和死代码消除，但不需要生成独立的源码文件。

```cpp
// Vulkan 特化常量（概念）
VkSpecializationInfo specInfo;
specInfo.mapEntryCount = 1;
specInfo.pMapEntries = &entry;  // 映射到常量 ID
specInfo.dataSize = sizeof(int);
specInfo.pData = &enable_normal_map;  // 0 或 1

// 管线创建时传入，驱动内部生成优化后的机器码
vkCreateGraphicsPipelines(..., &specInfo, ...);
```

**适用场景**：Vulkan 平台、变体差异较小的简单开关。

**隐藏代价**：
- 只有 Vulkan（和部分 Metal）原生支持，D3D12 没有直接等价物。
- 如果特化后的代码差异很大，驱动编译时间仍然不可忽略。
- 无法改变顶点输入格式（vertex layout），只能做片段着色器级别的开关。

**失效条件**：需要改变顶点格式或大量代码路径差异的复杂变体。

### 分支 E：动态状态 + Bindless（现代 API 演进方向）

**核心思路**：既然 PSO 爆炸是因为 API 把太多状态烘焙进去了，那我们就让 API 支持更多动态状态，并用 bindless 描述符减少绑定差异。

```cpp
// Vulkan 1.3 动态状态（概念）
VkPipelineDynamicStateCreateInfo dynamicState{};
dynamicState.dynamicStateCount = 3;
VkDynamicState states[] = {
    VK_DYNAMIC_STATE_DEPTH_TEST_ENABLE,    // 运行时开关深度测试
    VK_DYNAMIC_STATE_RASTERIZER_DISCARD,   // 运行时开关光栅化
    VK_DYNAMIC_STATE_COLOR_BLEND_ENABLE,   // 运行时开关混合
};
dynamicState.pDynamicStates = states;
```

**Bindless** 则把所有纹理/缓冲描述符放进一个大数组（Descriptor Heap），着色器通过索引访问，CPU 不需要为每种纹理组合创建不同的 BindGroup/DescriptorSet。

**适用场景**：D3D12 Ultimate / Vulkan 1.3+ / 现代主机平台。

**隐藏代价**：
- 需要 GPU 和驱动支持 bindless（所有现代桌面 GPU 都支持，但移动端支持度不一）。
- 着色器代码需要适配 bindless 风格（使用 `ResourceDescriptorHeap[]` 而非具名 sampler）。
- 调试复杂度上升：纹理索引错误不会立即报错，而是采样到随机数据。

**失效条件**：目标平台包含低端移动设备或旧版 API。

## 决策分析与推荐

**默认推荐：分支 C（按需编译 + 多级缓存）为主，分支 E（动态状态 + Bindless）为长期演进方向。**

具体策略：
1. **默认实现初期**：实现简化版的分支 C——运行时按需触发编译，内存中缓存 Technique，暂不需要进程外编译和分布式 DDC。
2. **变体控制**：严格限制材质的关键字维度（建议最多 4~5 个二值开关），超过此限制的方案应被否决。
3. **长期演进**：当目标平台确定只面向现代 GPU 时，逐步引入动态状态和 bindless，将 PSO 差异收敛到"顶点格式 + 着色器代码"两个维度，大幅削减变体数量。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：`ShaderTemplate` 的 Category/Keyword 机制生成 `permutation_id`，`MaterialInstance` 维护 `pending_build` 队列，`MaterialInstanceManager::postTick()` 异步处理构建。Technique 池使用固定索引数组提供稳定 ID。
- **UE**：`FMaterialShaderMapId` 编码所有影响编译的维度（QualityLevel、FeatureLevel、StaticSwitch、VertexFactoryType），DDC 缓存键精确到这些维度。UE5.1+ 引入 On-Demand Shader Compilation（ODSC），只编译当前关卡需要的变体，减少约 60% 的编译量。PSO Precaching + Bundled PSOs 双系统缓解运行时卡顿。
- **Bevy**：`PipelineCache` 按 `PipelineKey` 缓存渲染管线，材质通过 `SpecializedMeshPipeline` 按视图条件和网格特性特化。Bevy 的变体管理相对简单，因为 `bevy_pbr` 内置的 `StandardMaterial` 变体维度固定，用户自定义材质通过 Rust 类型系统自然隔离。

**默认推荐依据**：UE 的按需编译 + DDC 缓存是工业级验证最充分的方案。chaos 的 pending 队列 + Technique 池是同一思路的轻量实现。Bevy 的 PipelineCache 在 ECS 下表达更自然，但缺乏 UE 的进程外编译和分布式缓存能力。默认实现优先 chaos 级别的轻量异步编译 + 内存缓存，未来可扩展为 UE 级别的 DDC。


# 问题 4：着色器编译如何不阻塞渲染？

## 场景与根因

你在编辑器中修改了一个材质的节点图——把粗糙度从 0.5 改成 0.8。点击保存后，整个编辑器卡死了 3 秒钟。这就是**同步编译着色器**的代价。

根因在于：现代着色器编译（尤其是涉及 DXIL/SPIR-V 优化、PSO 创建时调用驱动级编译器）是 CPU 密集型任务。一个复杂材质的 PSO 创建可能需要 50~200ms。如果在渲染线程同步执行，16ms 一帧的预算会被瞬间击穿，帧率暴跌到 5fps。

更隐蔽的场景是游戏运行时：玩家走进一个新区域，加载了一套新材质。如果这些材质的着色器变体没有提前编译，首帧会出现严重卡顿（stutter）——这就是 Steam Deck 和主机游戏中常见的"着色器编译卡顿"问题。

## 分支研究

### 分支 A：同步阻塞编译

**核心思路**：请求材质时，如果对应变体未编译，直接在当前线程编译，完成后才继续渲染。

```cpp
// 分支 A：同步阻塞（反例，禁止在生产环境使用）
ShaderTechnique* GetTechniqueSync(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (!cache.contains(id)) {
        cache[id] = CompileTechnique(id);  // 阻塞！可能耗时 100ms+
    }
    return cache[id];
}
```

**适用场景**：离线工具、单元测试、极其简化的学习项目。

**隐藏代价**：显而易见——任何未缓存的变体都会导致卡顿。

**失效条件**：任何对帧率有要求的场景。

### 分支 B：异步后台编译 + Fallback 材质

**核心思路**：未命中缓存时，立即返回一个预定义的 fallback 材质（如简化光照或纯色），同时把编译任务提交到后台队列。

```cpp
// 分支 B：异步编译 + Fallback（概念）
enum class CompileState { Missing, Pending, Ready };

struct AsyncTechniqueCache {
    HashMap<uint32_t, ShaderTechnique*> ready_cache;
    Queue<CompileJob> pending_jobs;
    ShaderTechnique* fallback;  // 预编译好的简化材质
};

ShaderTechnique* GetTechniqueAsync(keywords) {
    auto id = GeneratePermutationId(keywords);
    if (ready_cache.contains(id)) return ready_cache[id];
    
    if (!pending_jobs.contains(id)) {
        pending_jobs.push({id, keywords});  // 提交后台
    }
    return fallback;  // 立即返回，不阻塞
}

// 后台线程或每帧尾部的 PostTick 中处理
void ProcessPendingJobs() {
    while (!pending_jobs.empty() && time_budget > 0) {
        auto job = pending_jobs.pop();
        auto tech = CompileTechnique(job.keywords);
        ready_cache[job.id] = tech;
        NotifyReady(job.id);  // 触发回调，下一帧使用真实材质
    }
}
```

**关键机制**：
1. **Fallback 材质**：必须是预编译好的、保证随时可用的简化着色器。UE 使用"默认材质"（亮粉色棋盘格）作为视觉提示；chaos 使用后处理快捷路径或全局实例。
2. **回调/轮询**：编译完成后，通过回调函数或下一帧的状态检查，将渲染对象从 fallback 切换到真实材质。
3. **时间片预算**：每帧只处理有限数量的编译任务，避免后台编译挤占 CPU 影响主线程。

**适用场景**：编辑器环境、游戏运行时动态加载、变体无法预先穷举的情况。

**隐藏代价**：
- **视觉 pop-in**：fallback 与真实材质差异明显时，玩家能看到物体"突然变好"。
- **状态管理复杂**：需要跟踪每个 pending 请求的编译进度，处理编译失败（如平台不支持某变体）。

**失效条件**：竞技游戏等对视觉一致性要求极高的场景。

### 分支 C：预编译缓存（PSO Precache）

**核心思路**：在构建期或加载期，预先编译所有可能用到的 PSO，运行时只查缓存。

```cpp
// 分支 C：预编译缓存（概念）
// 构建期：遍历所有材质 + 所有变体组合 + 所有顶点格式，生成 PSO 清单
void BuildPSOCache() {
    for (auto& material : all_materials) {
        for (auto& variant : material.variants) {
            for (auto& vf : vertex_formats) {
                auto pso = CreatePSO(material.shader, variant, vf);
                pso_cache[ComputeKey(material, variant, vf)] = pso;
            }
        }
    }
}

// 运行时：O(1) 查表
PipelineState* GetPSO(material, variant, vf) {
    return pso_cache[ComputeKey(material, variant, vf)];  //  guaranteed hit
}
```

**适用场景**：主机游戏、关卡内容固定的游戏、对卡顿零容忍的项目。

**隐藏代价**：
- **构建时间剧增**：构建期需要枚举所有可能的组合并编译，可能增加数小时。
- **缓存体积庞大**：PSO 缓存文件可能达到数 GB。
- **无法覆盖动态内容**：DLC、Mod、用户生成内容无法预先缓存。

**失效条件**：内容动态生成、构建时间预算有限、存储空间敏感的平台。

### 分支 D：进程外编译 + 分布式加速（UE 工业级方案）

**核心思路**：把编译任务从游戏进程完全剥离，通过独立进程（甚至远程机器）执行，主进程只负责收发结果。

```cpp
// 分支 D：进程外编译（概念）
// 主进程：序列化编译任务到文件
void SubmitCompileJob(CompileJob job) {
    auto input_file = WriteJobToFile(job);
    LaunchWorkerProcess(input_file);  // ShaderCompileWorker
}

// 工作进程：读取文件、编译、写回结果
void WorkerMain() {
    while (true) {
        auto job = ReadJobFromFile(input_file);
        auto result = CompileShader(job);
        WriteResultToFile(output_file);
        RenameAtomically(output_file, done_file);  // 原子通知
    }
}

// 主进程：轮询结果文件
void PollResults() {
    for (auto& done_file : ScanDoneFiles()) {
        auto result = ReadResult(done_file);
        InstallToCache(result);
    }
}
```

**关键机制**：
1. **进程隔离**：编译器崩溃不会拖垮主进程。
2. **分布式扩展**：通过 IncrediBuild (XGE) 或 Unreal Build Accelerator (UBA) 将任务分发到局域网内的多台机器。
3. **文件系统 IPC**：UE 使用输入/输出文件作为进程间通信通道，简单可靠，无需共享内存或 Socket。

**适用场景**：大型团队、编辑器环境、需要最大化利用多核/多机的场景。

**隐藏代价**：
- **I/O 开销**：大量小文件的读写对网络文件系统（如云桌面）不友好。
- **复杂度**：需要管理进程生命周期、超时、崩溃恢复。

**失效条件**：单兵项目、对工具链复杂度极度敏感的项目。

## 决策分析与推荐

**默认推荐：分支 B（异步后台编译 + Fallback）作为运行时核心，分支 C（预编译缓存）为发布版优化，分支 D（进程外编译）为团队扩展预留。**

具体策略：
1. **开发期**：异步编译 + 内存缓存。编辑器中修改材质后，fallback 材质的短暂显示是可接受的。
2. **发布期**：引入 PSO 预缓存。在构建期遍历所有打包材质和已知变体组合，生成 PSO 缓存文件。运行时优先查缓存，缓存未命中再走异步编译。
3. **团队扩展**：当项目规模增大、编译时间成为瓶颈时，将编译任务序列化到文件，通过外部工具链（如 `make -jN` 或分布式编译系统）加速。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：`MaterialInstance` 维护 `m_pending_build_permutation_techs` 队列，`MaterialInstanceManager::postTick()` 每帧批量处理 pending 构建。Technique 状态机（Invalid → Pending → Valid）管理生命周期。
- **UE**：`FShaderCompilingManager` 全局单例调度 ShaderCompileWorker 进程，支持优先级队列和分布式编译。UE5.1+ PSO Precaching 在 `PostLoad()` 时预编译，Bundled PSOs 捕获运行时使用的 PSO 清单用于下轮构建。
- **Bevy**：`PipelineCache` 在渲染世界按 `PipelineKey` 缓存管线，创建管线是异步的（通过 wgpu），但 Bevy 本身不管理进程外编译——它依赖 wgpu 的底层缓存和 `naga` 的即时编译。

**默认推荐依据**：UE 的进程外编译 + DDC 缓存是工业级验证最充分的方案，但工具链复杂度高。chaos 的 pending 队列 + postTick 处理是同一思路的轻量实现，适合默认实现起步。Bevy 的 `PipelineCache` 在 ECS 下足够简洁，但缺乏主动的后台编译调度。因此我们推荐以 chaos 的异步队列模型为基础，保留向 UE 级进程外编译扩展的接口。
