> **前置依赖**：[[组件系统架构]]
> **本模块增量**：建立 ECS 引擎安全引用外部资源的基础设施——轻量 Handle、ABA 防护、自动生命周期。你将理解为什么裸指针和 shared_ptr 都不适合 ECS，并掌握 Handle Table 的工业级设计。
>
> 本笔记探讨的核心问题是：**在 ECS 架构下，组件如何安全、高效、可预测地引用外部资源（纹理、网格、材质等）？**

## 问题 0：为什么 ECS 引擎不能直接用裸指针持有纹理和网格？

想象你正在写一个最简单的 3D 场景：一个旋转的立方体。你写了一个 `MeshComponent`，里面存了一个 `Mesh*` 指针，指向从磁盘加载的立方体网格数据。一帧过去，一切正常。

然后你加了第二个立方体。为了省内存，你让两个 `MeshComponent` 指向**同一个** `Mesh` 实例。很好，内存只花了一份。

接着你删除了第一个 Entity。`MeshComponent` 跟着销毁了——但 `Mesh*` 指向的内存怎么办？如果 `MeshComponent` 的析构函数里 `delete mesh`，第二个立方体立刻崩溃（use-after-free）。如果不 delete，内存泄漏。

这还只是两个 Entity 共享一个 Mesh。实际游戏里，同一张纹理可能被几百个 Material 引用，同一个骨骼动画被几十个角色共享。裸指针根本无法回答一个核心问题：**"现在还有谁在用它？"**

### 根因分析：ECS 的平铺存储与所有权矛盾

ECS 的核心优势是**组件数据平铺、批量迭代**。`MeshComponent` 应该是一个很小的 POD 结构，能够被成千上万个 Entity 连续存储，在 System 中被 SIMD 批量处理。如果在 `MeshComponent` 里塞一个智能指针（如 `std::shared_ptr<Mesh>`），会发生两件事：

1. **内存膨胀**：`std::shared_ptr` 通常占 16 字节（两个指针），而且指向的 control block 是独立堆分配。一万个 Entity 就是一万份 shared_ptr 拷贝，每次拷贝都要原子操作修改引用计数。
2. **缓存不友好**：System 迭代 `MeshComponent` 数组时，真正需要的数据是变换矩阵和渲染状态，但 `shared_ptr` 的间接跳转把 cache line 搞乱了。

更深层的问题是：**ECS 的组件存储不负责对象生命周期**。Entity 的创建和销毁由 `EntityRegistry` 管理，组件的增删由 `World` 的 Archetype 迁移处理——这些操作都是批量的、机械的。如果把资源的生命周期管理（引用计数、加载、卸载）也塞进组件里，就等于让 ECS 存储层承担了它不该承担的责任。

### 深层矛盾："引用"与"数据"必须分离

所以问题的根因可以归结为一句话：

> **ECS 组件应该只持有"如何找到资源"的最小信息，而资源本身的生命周期和数据存储应该由专门的资源管理系统负责。**

这个"最小信息"就是**句柄（Handle）**——一个轻量的、无所有权的标识符。组件存句柄，资源系统管理句柄到实际数据的映射和生命周期。这和我们之前在 [[实体生命周期与注册表]] 中讨论的"Entity ID 只是索引，不是指针"是同一套哲学在资源领域的延伸。

但句柄本身有多种形态，选择哪一种直接决定了系统的性能特征和安全性。这就引出了第一个关键决策点。

## 问题 1：资源句柄应该长什么样？

如果我们不能用裸指针，那组件里该存什么？业界对这个"最小信息"的设计有几种截然不同的思路。

### 分支 A：裸指针 + 手动生命周期（不推荐，但理解其局限很重要）

```cpp
struct MeshComponent {
    Mesh* mesh;  // 指向资源池中的某个 Mesh
};
```

**核心思路**：资源由一个全局的"资源池"分配，返回指针。用户保证不在资源被释放后使用指针。

**适用场景**：原型验证、单人开发的小项目、资源生命周期与关卡完全绑定的简单游戏。

**隐藏代价**：
- 没有 ABA 防护。资源 A 被释放后，新分配的资源 B 可能恰好拿到同一块内存。旧指针突然"复活"指向一个完全不同的资源。
- 没有引用计数。不知道多少组件在引用同一个资源，卸载决策全靠猜。
- 多线程环境下，资源池的分配/释放需要全局锁，与 ECS 的并行迭代冲突。

**失效条件**：任何涉及动态加载/卸载、共享资源、多线程的场景都会暴露问题。

### 分支 B：整数索引 + 世代计数（Handle Table）

```cpp
struct AssetHandle {
    uint32_t index;      // 在资源表中的位置
    uint32_t generation; // 世代计数，每次该槽位被回收后重新分配时递增
};

// 资源表中的每个槽位
template<typename T>
struct AssetSlot {
    T        data;
    uint32_t generation;
    bool     occupied;
};
```

**核心思路**：把所有同类型资源存在一个密集数组（`Array<AssetSlot<T>>`）中。Handle 是"索引 + 世代"的二元组。访问时先检查 `slots[index].generation == handle.generation`，不匹配则说明原资源已释放，这是一个"悬空句柄"。

**适用场景**：ECS 原生引擎（Bevy 的 `AssetIndex` 就是这种思想的变体）、需要 O(1) 访问且内存连续的场景。

**代码片段：访问校验**
```cpp
template<typename T>
T* AssetPool<T>::try_get(AssetHandle handle) {
    if (handle.index >= slots.size()) return nullptr;
    auto& slot = slots[handle.index];
    if (!slot.occupied || slot.generation != handle.generation)
        return nullptr;  // ABA 防护：世代不匹配，说明是旧句柄
    return &slot.data;
}
```

**适用场景**：
- 需要频繁遍历所有活跃资源的场景（如每帧更新所有纹理的流送状态）
- 内存敏感、追求 cache locality 的场合

**隐藏代价**：
- 资源表是单块连续内存，如果资源类型差异很大（纹理几十MB，材质几KB），统一池化会造成内存碎片或浪费。
- 世代计数溢出？`uint32_t` 的世代在正常使用中几乎不可能溢出（40亿次回收），但如果真的发生，需要额外的 salt 机制。
- 索引本身没有类型安全。一个 `AssetHandle {index=5, gen=1}` 既可以指向 Mesh 也可以指向 Texture，全凭调用者上下文。

**失效条件**：当资源类型差异极大、或需要跨类型统一引用时（如一个材质既引用纹理又引用着色器），单一类型的 Handle Table 需要额外的类型擦除层。

### 分支 C：128-bit 值类型 + 类型 ID + 引用计数（chaos 风格）

```cpp
union Handle {
    struct {
        uint32_t type_id;    // 资源类型（Mesh/Texture/Shader...）
        uint32_t salt;       // ABA 防护盐值
        void*    payload;    // 指向类型擦除的资源管理器内部节点
    };
    uint128_t value;         // 整体作为值类型传递
};
```

**核心思路**：Handle 是一个"自描述"的 128-bit 值，内部包含类型信息、盐值和一个指向管理器内部节点的指针。类型擦除的管理器统一处理所有资源的引用计数和生命周期。

**适用场景**：大型商业引擎、需要统一资源管理接口、多种资源类型共存且需要顶层统一调度的项目。

**代码片段：引用计数的内部实现**
```cpp
// Handle 本身不管理生命周期，只是标识符
// 但资源管理器内部为每个 payload 维护引用计数
class ResourceManager {
    struct ResourceNode {
        void* data;
        uint32_t ref_count;
        uint32_t salt;
        // ... 类型信息、加载状态等
    };
    // type_id -> DenseArray<ResourceNode>
    HashMap<uint32_t, void*> type_tables;
};

// RAII 包装（可选）
class OwnedHandle {
    Handle handle;
public:
    ~OwnedHandle() { release_reference(handle); }
};
```

**隐藏代价**：
- `payload` 是一个指针，Handle 不再是一个纯值类型，跨线程传递时需要考虑指针有效性。
- 128-bit 在某些 32-bit 平台上不友好（虽然现代游戏引擎基本抛弃 32-bit 了）。
- 类型擦除层带来虚函数或函数指针开销，批量访问同一类型资源时不利于编译器优化。

**失效条件**：对极致 cache locality 有要求的场景（如每帧访问数千个 Mesh 的顶点数据），类型擦除的间接层会成为瓶颈。

### 分支 D：强类型 Handle + 全局类型擦除存储（Bevy 风格）

```rust
// Bevy 的做法：编译期强类型 + 运行期世代索引
pub struct Handle<A: Asset> {
    id: AssetId<A>,  // Index 或 Uuid
    // ...
}

pub enum AssetId<A> {
    Index { index: AssetIndex, marker: PhantomData<fn() -> A> },
    Uuid { uuid: Uuid },
}
```

**核心思路**：Handle 在编译期携带类型参数 `A`，防止把 `Handle<Mesh>` 误传给需要 `Handle<Texture>` 的函数。运行期内部是一个世代索引（`AssetIndex = index + generation`），指向 `Assets<A>` 这个类型专属的稠密存储。

**适用场景**：Rust 等强类型语言、ECS 原生架构、编译期错误检查优先于运行期检查。

**隐藏代价**：
- C++ 模板元编程实现同等程度的编译期类型安全代价较高（虽然可行，但编译时间和错误信息不友好）。
- 跨类型统一处理困难（如"卸载所有资源"需要遍历所有已注册的类型）。
- `Uuid` 路径的访问是 HashMap 查找，比数组索引慢一个数量级。

**失效条件**：需要运行时动态类型识别的场景（如脚本系统传入一个 untyped handle，需要查询它的实际类型）。

### 决策分析

| 条件 | 推荐方案 | 理由 |
|------|---------|------|
| 纯 ECS 架构、类型安全优先 | 分支 B（Handle Table）+ 编译期类型包装 | Bevy 证明了这条路径在 ECS 下的优雅性；C++ 可用模板做 `Handle<Mesh>` 包装 |
| 大型商业引擎、多类型统一调度 | 分支 C（128-bit 自描述 Handle）| UE/chaos 都选择了某种形式的类型擦除，因为上层系统需要统一接口 |
| 原型/教学项目、快速迭代 | 分支 A（裸指针池）| 不是不能选，但必须清楚自己在牺牲什么。建议预留 Handle Table 的迁移接口 |

**默认推荐：分支 B（Handle Table + 世代索引）作为核心机制，外层用模板提供类型安全。**

推荐依据：
1. **ECS 兼容性**：Handle Table 的密集存储与 ECS 的批量处理哲学完全一致。`Array<AssetSlot<T>>` 可以被 System 直接批量遍历。
2. **UE 优先吸收**：UE 的 `FName` + 索引哲学本质上也是"小标识符指向大表"。虽然 UE 的 UObject 是全局继承树，但其"通过轻量索引访问大量数据"的设计意图与 Handle Table 一致。
3. **Bevy 验证**：Bevy 的 `DenseAssetStorage` + `AssetIndex` 已经在生产环境中验证了这条路径的可行性。
4. **性能可预测**：数组索引的访问延迟是固定的、可量化的，没有虚函数或哈希表的波动。

**在 ECS 中的映射**：
- `MeshComponent` 只存 `Handle<Mesh>`（8 字节：4 字节 index + 4 字节 generation）。
- `TextureComponent` 或 `MaterialComponent` 中的 `Handle<Texture>` 同理。
- 实际的 `Mesh` 数据存在 `Assets<Mesh>` 这个 ECS Resource 中（或者一个独立的资源子系统）。
- System 通过 `Res<Assets<Mesh>>` 查询句柄对应的实际数据。

**遗留问题**：Handle 解决了"如何标识资源"，但没有解决"什么时候释放资源"。如果一个 Entity 被销毁了，它持有的 `Handle<Mesh>` 随组件一起消失——但 Mesh 本身可能还被其他 Entity 引用。我们需要引用计数。


## 问题 2：引用计数该由谁管理？

组件里的 Handle 只是一个标识符，没有所有权语义。当持有 `Handle<Mesh>` 的 Entity 被销毁时，Mesh 数据本身该不该释放？这取决于还有没有其他 Entity 在引用它。

### 场景绑定：千 Entity 共享一 Mesh

一个场景里有 1000 棵树，每棵树是一个 Entity，有 `MeshComponent` 引用同一个 oak_tree.mesh。如果玩家砍倒一棵树（销毁一个 Entity），你绝对不想把 oak_tree.mesh 从内存中卸载——其他 999 棵树还在用它。

### 分支 A：每个 Handle 自带原子引用计数（类似 `std::shared_ptr`）

```cpp
template<typename T>
struct SmartHandle {
    uint32_t index;
    uint32_t generation;
    std::atomic<uint32_t>* ref_count;  // 指向全局引用计数表的指针
};
```

**核心思路**：每个 Handle 实例在拷贝时原子递增 `*ref_count`，析构时原子递减。当计数归零，通知资源管理器释放该槽位。

**适用场景**：简单直观、需要 RAII 自动释放的场合。

**隐藏代价**：
- **拷贝开销**：每次拷贝 Handle（如组件数组扩容、System 参数传递）都要原子操作。10000 个 Entity 的 `MeshComponent` 数组，每次 System 读取 Handle 都不会触发拷贝——但任何涉及 Handle 复制的操作（如序列化、Inspector 编辑）都要付原子税。
- **伪共享**：多个 CPU 核心同时拷贝不同 Handle，但它们的 `ref_count` 恰好落在同一 cache line 上，导致缓存一致性风暴。

**失效条件**：高频 Handle 拷贝、大规模并行场景。

### 分支 B：全局引用计数表，延迟批量回收（ECS 友好方案）

```cpp
template<typename T>
struct AssetStorage {
    Array<AssetSlot<T>> slots;
    Array<uint32_t>     ref_counts;     // 与 slots 平行数组
    Array<uint32_t>     pending_free;   // 待回收索引队列
    
    // System 批量操作：扫描所有组件数组，统计引用
    void batch_update_ref_counts(const Span<Handle<T>>& all_handles);
    void flush_pending_frees();         // 每帧末或定时执行
};
```

**核心思路**：引用计数不放在 Handle 里，而是放在资源存储的平行数组中。ECS System 可以批量扫描所有组件，用 SIMD 或并行 for 快速统计引用。回收延迟到帧边界统一执行。

**适用场景**：ECS 架构、需要批量处理、对拷贝开销敏感的项目。

**代码片段：批量引用计数更新**
```cpp
// 在 System 中，遍历所有 MeshComponent，统计哪些 Mesh 被引用了
void update_mesh_ref_counts(
    Query<const MeshComponent> meshes,
    ResMut<AssetStorage<Mesh>> storage
) {
    // 先清零
    for (auto& rc : storage->ref_counts) rc = 0;
    
    // 批量统计
    for (auto [mesh_comp] : meshes) {
        if (mesh_comp.handle.index < storage->ref_counts.size())
            storage->ref_counts[mesh_comp.handle.index]++;
    }
    
    // 标记零引用的槽位为待回收
    for (size_t i = 0; i < storage->ref_counts.size(); ++i) {
        if (storage->ref_counts[i] == 0 && storage->slots[i].occupied) {
            storage->pending_free.push_back(i);
            storage->slots[i].occupied = false;
            storage->slots[i].generation++;  // 递增世代，使旧 Handle 失效
        }
    }
}
```

**适用场景**：
- 大规模 ECS 场景，组件数量远超资源数量
- 需要确定性的帧边界行为（资源不会在渲染中途被释放）

**隐藏代价**：
- 需要遍历所有持有 Handle 的组件，时间复杂度 O(Entity 数量)。如果 Entity 数量极大（百万级），每帧全量扫描有性能压力。
- 延迟回收意味着资源不会立即释放，内存峰值可能略高于即时回收。

**优化分支：增量更新**
不用每帧全量扫描，而是只在 Entity 创建/销毁/组件修改时更新对应 Handle 的引用计数。这需要 ECS 的变更检测或生命周期事件支持。我们在 [[事件总线]] 中讨论过延迟事件队列，这里可以用同样的机制：Entity 销毁时发送 `ComponentRemoved<MeshComponent>` 事件，资源系统消费事件并递减对应计数。

**失效条件**：资源生命周期需要立即响应的场景（如内存极度受限，必须毫秒级释放）。

### 分支 C：不引用计数，靠显式作用域/区域卸载

```cpp
class Level {
    Array<Handle<Mesh>> level_meshes;
public:
    ~Level() {
        for (auto h : level_meshes) asset_manager->unload(h);
    }
};
```

**核心思路**：资源的生命周期与关卡/场景/区域绑定。进入关卡时加载一批资源，离开关卡时全部卸载。组件里的 Handle 只是"借用"，不负责生命周期。

**适用场景**：线性关卡制游戏、资源需求可预测、内存预算宽松的项目。

**隐藏代价**：
- 无法处理跨关卡共享资源（如全局 UI 纹理、角色基础模型）。需要额外的"常驻资源"机制。
- 卸载粒度粗，可能造成内存浪费（关卡 A 和关卡 B 共享 90% 资源，但按关卡卸载会重复加载）。

**失效条件**：开放世界、动态流送、资源高度复用的场景。

### 引擎对照

> **参考：chaos / UE / Bevy 对引用计数是怎么做的？**
>
> 我们在解决的是「资源生命周期由谁决定」这个问题。
>
> - **chaos** 选择了全局引用计数表 + `PossessedHandle` RAII 包装。`AssetManager` 维护 `m_top_asset_file_id_map`，用 `PossessedHandle`（自动增减引用）持有顶层资源。底层 `DatumManager` 统一 GC。这种设计的动机是：chaos 需要支持运行时异步加载和编辑器热重载，引用计数是最安全的默认策略。
> - **UE** 选择了 UObject 垃圾回收（GC）而非引用计数。`UPROPERTY()` 标记的引用会被 GC 遍历，unreachable 对象定期批量回收。UE 的动机是：C++ 的循环引用（A 引用 B，B 引用 A）会让纯引用计数失效，而 GC 可以处理循环依赖。代价是 GC 暂停和内存峰值不可控。
> - **Bevy** 选择了 channel-based drop 通知 + 延迟回收。`StrongHandle` 被 Drop 时发送事件到 channel，`Assets::track_assets` System 在帧边界消费事件。Bevy 的动机是：ECS 的 System 调度天然适合"批量处理事件"，不需要每帧遍历所有组件。
>
> **默认推荐**：**分支 B 的变体——增量引用计数 + 延迟帧边界回收**。原因如下：
> 1. UE 的 GC 机制与 ECS 架构存在根本性冲突。ECS 强调数据局部性和确定性调度，GC 的"标记-清扫"是非确定性的、会暂停世界的。在 ECS 中重新表达 UE 的设计意图，应该保留"自动生命周期管理"的核心目标，但用 ECS 友好的事件驱动机制替代 GC。
> 2. Bevy 的 channel-based drop 通知是 ECS 下最优雅的引用计数表达。我们在 C++ 中可以用无锁队列（如 `moodycamel::ConcurrentQueue` 或自研的 SPSC 队列）实现同等语义。
> 3. chaos 的 `PossessedHandle` RAII 包装可以作为简化过渡路径（默认实现早期不需要 channel 事件，直接用 RAII 递减计数即可），但未来向 ECS 批量回收演进时，channel 事件是更干净的边界。
>
> **简化路径**：如果当前阶段工程负担过重，可先实现「Handle Table + RAII 包装（`OwnedHandle`）+ 手动 `unload`」。这相当于 chaos 的 `PossessedHandle` 简化版。但需在代码中预留 `DropEvent` 接口，明确标注"未来需替换为 ECS 事件驱动的批量回收"。

### 状态变化图：引用计数与回收

```
n 个 Entity 持有 Handle<Mesh> {index=3, gen=7}
n >= 1 时：
+-------------------------------------+
| slots[3] = { data: Mesh(...),       |
|             generation: 7,          |
|             occupied: true }        |
| ref_counts[3] = n                   |
+-------------------------------------+
        v 所有 Entity 被销毁，无新引用
ref_counts[3] 变为 0
        v 帧边界 flush_pending_frees()
slots[3].occupied = false
slots[3].generation = 8  <- 旧 Handle 的 gen=7 不再匹配
        v 新资源申请槽位
slots[3] 被复用，generation=8
新 Handle {index=3, gen=8} 有效
旧 Handle {index=3, gen=7} 访问时 try_get() 返回 nullptr
```

**诚实性检查**：这个方案引入了什么新成本？
- 引用计数表是额外的内存（每个资源一个 `uint32_t`，对于几千个资源可以忽略）。
- 延迟回收意味着资源可能多存活几帧。如果内存极度紧张，可以改为"零引用立即回收"，但会失去帧边界安全性。
- 循环引用仍然无法自动处理。如果资源 A 引用资源 B，资源 B 又引用资源 A，引用计数永远不会归零。解决方案在下一个问题中讨论。

**遗留问题**：引用计数解决了"什么时候释放"，但资源的加载过程本身可能很慢（磁盘 I/O、网络下载、解压解码）。如果在主线程阻塞加载，帧率会直接崩溃。我们需要异步加载。


## 问题 5：资源数据在 ECS 中怎么存储？

这是一个架构层面的问题：ECS 的哲学是"数据平铺在组件中、System 批量处理"，但资源数据（如 100MB 的纹理、带 LOD 的网格）显然不能平铺在每个 Entity 的组件里。

### 根因分析：资源数据的"规模不兼容性"

ECS 组件的理想特征是：**小、固定大小、可预测、可批量迭代**。一个 `Transform` 组件是 12 字节的 position + 16 字节的 rotation，完美。但一个 `Mesh` 资源可能包含：
- 顶点数组（数万~数百万个 `Vertex`，每个 32~64 字节）
- 索引数组
- LOD 层级
- 骨骼绑定数据
- 材质引用

把这些塞进 `MeshComponent` 里，组件数组的步长会暴涨，缓存彻底失效。而且多个 Entity 共享同一个 Mesh 时，数据会被复制多份。

所以资源数据需要**集中存储、组件只存引用**。但这引出了一个新的问题：集中存储应该是什么形态？

### 分支 A：每类型一个 ECS Resource（Bevy 风格）

```rust
// Bevy 的做法
#[derive(Resource)]
pub struct Assets<T: Asset> {
    dense_storage: DenseAssetStorage<T>,
    // ...
}

// System 访问
fn render_meshes(
    meshes: Query<&Handle<Mesh>>,
    assets: Res<Assets<Mesh>>,  // 通过 Resource 访问实际数据
) {
    for handle in &meshes {
        if let Some(mesh) = assets.get(handle) {
            // 渲染 mesh
        }
    }
}
```

**核心思路**：每种资源类型对应一个 ECS Resource。`Assets<Mesh>` 是 Resource，`Assets<Texture>` 是另一个 Resource。System 通过 `Res<Assets<T>>` 访问。

**适用场景**：ECS 原生引擎、类型安全优先、资源类型数量可控的项目。

**隐藏代价**：
- 在 C++ 中，模板化的 `Assets<T>` 需要显式实例化每种类型。虽然可以通过宏简化，但编译时间和二进制体积会增加。
- System 的依赖声明会变得冗长。一个渲染 System 可能需要 `Res<Assets<Mesh>>`、`Res<Assets<Texture>>`、`Res<Assets<Material>>`……参数列表膨胀。
- 跨类型统一操作困难。如"获取所有资源的内存占用"需要遍历所有已注册的类型。

**失效条件**：资源类型极多（几十上百种）、需要跨类型统一查询的场景。

### 分支 B：专门的资源子系统，ECS 只存门面（UE/chaos 风格）

```cpp
// 资源子系统是引擎核心子系统之一，不直接是 ECS Resource
class AssetManager {
    HashMap<TypeId, void*> type_storages;  // type-erased storage
public:
    template<typename T>
    T* get(Handle<T> handle);
    
    template<typename T>
    Handle<T> load(const Path& path);
};

// ECS 中只存一个门面引用（可选）
struct AssetManagerRef : Resource {
    AssetManager* ptr;
};
```

**核心思路**：资源管理是独立的子系统，有自己的存储、加载、生命周期管理。ECS 世界可以持有一个 `AssetManagerRef` Resource 作为访问门面，但核心逻辑不在 ECS System 中。

**适用场景**：大型引擎、资源系统先于 ECS 存在、或需要与编辑器/工具链深度集成的项目。

**隐藏代价**：
- ECS 和资源系统之间的边界不够清晰。System 既要查询 ECS 组件，又要访问外部资源系统，代码风格不统一。
- AI 可观测性受损。如果资源数据不在 ECS 的组件/Resource 中，AI 通过 MCP 查询资源状态时需要额外的桥接层。

**失效条件**：追求纯 ECS 架构、希望所有状态都可被 ECS System 批量处理的项目。

### 分支 C：混合方案——ECS Resource 门面 + 类型擦除后端

```cpp
// 前端：ECS Resource，提供类型安全的访问接口
template<typename T>
struct Assets : Resource {
    HandleTable<T> storage;
    
    T* get(Handle<T> h) { return storage.try_get(h); }
    void insert(Handle<T> h, T value) { storage.insert(h, value); }
};

// 后端：统一的内存管理和加载调度
class AssetBackend {
    // 线程池、加载队列、依赖图、引用计数表...
};

// System 使用
void render_system(
    Query<const MeshComponent> meshes,
    Res<Assets<Mesh>> mesh_assets,
    Res<Assets<Texture>> tex_assets,
    Res<AssetBackend> backend  // 用于发起异步加载
) {
    for (auto [mesh_comp] : meshes) {
        if (auto* mesh = mesh_assets->get(mesh_comp.handle))
            submit_to_renderer(mesh);
    }
}
```

**核心思路**：每种资源的实际数据存储是独立的 ECS Resource（`Assets<T>`），提供 O(1) 类型安全访问。但加载调度、线程池、依赖图等"基础设施"是共享的 `AssetBackend` Resource。

**适用场景**：大多数 ECS 引擎项目。兼顾了类型安全、ECS 哲学和工程实用性。

**隐藏代价**：
- 需要为每种资源类型注册 `Assets<T>`。可以用宏简化（`REGISTER_ASSET_TYPE(Mesh)`）。
- `Assets<T>` 的生命周期与 ECS World 绑定。World 销毁时所有资源一起释放，这在大多数情况下是期望的行为。

### 决策分析

**默认推荐：分支 C（混合方案）**。

推荐依据：
1. **Bevy 验证**：Bevy 的 `Assets<T>` 作为 ECS Resource 已经验证了这个设计的可行性。我们在 C++ 中可以用模板实现同等的类型安全。
2. **UE 设计意图吸收**：UE 的 `UObject` 全局表本质上也是"类型擦除的后端 + 类型化的前端（`Cast<T>()`）"。我们提取 UE 的"集中存储 + 类型安全访问"意图，用 ECS Resource 重新表达。
3. **AI 可观测性**：`Assets<T>` 作为 ECS Resource，可以通过反射系统导出 Schema，AI 可以通过 MCP 查询 `Assets<Mesh>` 中所有活跃资源的句柄和元数据。
4. **确定性**：资源存储在 ECS World 中，World 快照（我们在 [[系统调度与确定性]] 中讨论过）可以包含资源状态，支持回放和检查点。

**简化路径**：如果当前阶段资源类型很少（只有 Mesh 和 Texture），可以直接手写 `MeshStorage` 和 `TextureStorage` 两个独立的 ECS Resource，不需要模板泛化。但接口设计时应预留泛化空间（如统一的 `get/insert/remove` 命名约定）。

**遗留问题**：资源加载和管理的基础设施已经讨论得差不多了，但还有一个开发期效率的关键问题——热重载。当美术师修改了一张纹理，引擎能不能在不重启的情况下自动更新？
