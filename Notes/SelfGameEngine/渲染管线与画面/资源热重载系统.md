> **前置依赖**：[[异步加载管线]]
> **本模块增量**：掌握开发期资源热重载的完整实现——文件系统 Watch、脏 Handle 标记、帧边界原地替换。你将能够让美术修改纹理后引擎自动更新，大幅缩短迭代周期。
>
> 本笔记探讨的核心问题是：**开发期迭代中，美术修改了资源文件，引擎如何在不影响运行稳定性的前提下自动加载新版本？**

## 问题 6：热重载怎么实现？

开发期迭代效率直接决定项目进度。想象一个场景：美术师在 Photoshop 中修改了一张纹理，保存后，引擎中的材质应该自动更新，而不需要重启游戏、重新加载关卡、重新定位摄像机。

### 根因分析：开发期与运行期的需求差异

运行期的资源加载是"一次性"的：加载 → 使用 → 卸载。开发期的资源加载是"反复性"的：加载 → 修改 → 重载 → 再修改 → 再重载。这两套需求不能硬塞进同一个代码路径。

### 分支 A：手动重载（按按钮触发）

```cpp
// 开发者按下一个键或点击按钮
if (input.pressed(Key::F5)) {
    asset_manager->reload("wood_diffuse.png");
}
```

**核心思路**：显式触发重载。引擎提供 `reload(handle)` 或 `reload(path)` 接口，开发者手动调用。

**适用场景**：调试特定资源、验证修复效果。

**隐藏代价**：
- 开发者必须知道哪个资源被修改了。对于复杂材质，一张纹理可能被多个材质引用，手动逐个重载效率低下。
- 没有自动发现机制。美术师保存了文件，开发者可能忘记按 F5。

**失效条件**：频繁修改、大量资源的场景。

### 分支 B：文件系统 Watch + 脏标记 + 延迟替换（工业标准）

这是 chaos、UE、Bevy 都采用的核心方案。

**核心思路**：
1. **Watch**：启动一个文件系统监听器（如 Windows 的 `ReadDirectoryChangesW`、Linux 的 `inotify`、跨平台的 `notify` 库），监控资源目录的变更。
2. **Dirty Handle**：文件变更时，不立即替换资源，而是将对应的 Handle 标记为"脏"。
3. **帧边界替换**：在主线程的帧边界（如 `PostUpdate` 之后，`PreRender` 之前），统一处理所有 Dirty Handle，执行替换。

**代码片段：热重载流程**
```cpp
class HotReloadSystem {
    FileWatcher watcher;
    ConcurrentQueue<Path> changed_files;
    
public:
    void startup() {
        watcher.watch("assets/", [&](const Path& path) {
            changed_files.enqueue(path);
        });
    }
    
    // 每帧在主线程运行
    void tick(World& world, ResMut<AssetBackend> backend) {
        Path path;
        while (changed_files.try_dequeue(path)) {
            auto handle = backend->find_handle(path);
            if (handle.valid()) {
                // 标记为脏，延迟到帧边界替换
                backend->mark_dirty(handle);
            }
        }
    }
    
    // 帧边界执行
    void flush_dirty(World& world) {
        for (auto handle : backend->dirty_handles()) {
            // 1. 异步重新加载资源数据
            backend->reload_async(handle);
            
            // 2. 新数据加载完成后，在 Assets<T> 中原地替换
            // 注意：这里不是删除旧数据再插入新数据，而是复用同一个 Handle
            // 这样所有引用该 Handle 的组件都不需要更新
        }
        backend->clear_dirty();
    }
};
```

**适用场景**：所有需要开发期热重载的项目。

**隐藏代价**：
- 文件系统 Watch 有平台差异。Windows、macOS、Linux 的 API 完全不同，需要抽象层。
- "变更"的定义复杂。有些编辑器保存文件时会先写入临时文件再重命名，Watch 可能捕获到多次事件。需要防抖（debounce）机制。
- 原地替换的线程安全。如果渲染线程正在读取纹理数据，主线程不能同时替换它。必须确保替换发生在渲染使用的安全点（如帧边界、或配合 GPU fence）。

**失效条件**：某些资源类型不支持原地替换（如需要重新编译的着色器、需要重建加速结构的网格）。这些资源可能需要更重的"卸载再加载"流程。

### 引擎对照

> **参考：chaos / UE / Bevy 对热重载是怎么做的？**
>
> 我们在解决的是「开发期资源修改后如何自动更新」这个问题。
>
> - **chaos** 选择了**时间戳比对 + 脏 Handle 队列**。编辑器模式下，`AssetManager` 维护 `m_file_last_modified_time_when_load_map`，通过文件修改时间判断是否需要重载。重载时通过 `override_asset_handle` 复用已有内存，保持所有外部引用有效。chaos 的动机是：编辑器需要频繁热更，且重载时必须保证所有引用不断裂。
> - **UE** 选择了**DirectoryWatcher + 资产重新导入 + Live Coding**。UE 的编辑器有完整的资产导入管线，源文件（如 FBX、PNG）变更后，DirectoryWatcher 触发重新导入（Reimport），生成新的 `.uasset`，然后通知运行时加载新包。UE 的动机是：资产导入涉及格式转换（如 PNG → 平台特定纹理格式），热重载必须走完整的导入管线。
> - **Bevy** 选择了**AssetWatcher + AssetProcessor 两阶段热重载**。Bevy 0.15+ 的 `AssetProcessor` 会先处理源文件变更（如压缩、转换），输出到 `imported_assets/` 目录，然后主 `AssetServer` 从 processed 目录重新加载。Bevy 的动机是：分离"源资产"和"运行时资产"，确保引擎加载的永远是优化后的格式。
>
> **默认推荐：分支 B（文件 Watch + 脏 Handle + 帧边界原地替换）**。
>
> 推荐依据：
> 1. **UE 优先吸收**：UE 的"脏检测 + 延迟替换"思想是工业标准。我们不需要 UE 的完整导入管线（那是 [[资产导入管线]] 的主题），但"检测变更 → 标记脏 → 安全点替换"的流程应该直接吸收。
> 2. **chaos 的洞察**：chaos 的 `override_asset_handle` 复用内存是一个关键技巧——Handle 不变，只替换 Handle 指向的数据。这保证了所有组件中的 Handle 引用自动生效，不需要遍历所有 Entity 更新组件。
> 3. **ECS 映射**：热重载可以是一个 ECS System（`HotReloadSystem`），在 `PreUpdate` 阶段检查文件变更，在 `PostUpdate` 阶段执行替换。这与 ECS 的调度模型完全一致。
>
> **简化路径**：如果当前阶段不想引入文件系统 Watch 库，可以先实现「手动 F5 触发 reload()」。这是分支 A 的简化版，但代码结构应预留 Watch 接口。

### 关键决策：替换时如何保证渲染安全？

热重载最危险的瞬间是"替换发生时，GPU 是否正在使用旧数据"。对于 CPU 端资源（如网格顶点数据），如果渲染 System 已经提交了 Draw Call，GPU 可能还在读取旧顶点缓冲区。此时如果主线程释放旧缓冲区，会导致 GPU 崩溃。

解决方案：
1. **延迟释放**：替换后，旧数据不立即释放，而是标记为"待释放"，等待一帧（或直到 GPU fence 信号）后再释放。
2. **双缓冲**：对于关键资源（如纹理），维护两个 GPU 资源句柄。替换时切换到新句柄，旧句柄在几帧后释放。
3. **帧边界同步**：所有替换操作严格发生在 `Present` 之后、`BeginFrame` 之前。此时 GPU 已完成上一帧的所有工作。

对于默认实现，**方案 3（帧边界同步）** 是最简单可靠的。在 [[RHI抽象层与命令模型]] 中，我们已经讨论了帧生命周期和命令列表提交。热重载的替换可以绑定到 `EndFrame` 阶段。

**遗留问题**：热重载解决了开发期效率，但资源的"原始格式"和"运行时格式"通常是不同的（如 PNG vs GPU 压缩纹理）。谁来负责格式转换？这就是导入管线（Import Pipeline）的职责。


## 问题 7：导入管线与格式转换

游戏资源不是从美术软件直接塞进引擎的。Photoshop 导出 PNG，Blender 导出 FBX，但这些格式都不适合直接 GPU 渲染。PNG 需要解码、可能还需要生成 mipmap、压缩为 BC7/ASTC；FBX 需要解析场景图、提取顶点数据、生成 LOD、烘焙动画。

### 场景绑定：为什么不能用原始文件直接运行？

假设你直接用 PNG 作为纹理运行时格式。每次加载纹理时，引擎都需要：
1. 从磁盘读取 PNG（可能 5MB）
2. CPU 解码为 raw RGBA（可能 64MB）
3. 生成 mipmap（额外 21MB）
4. 上传到 GPU

这个过程在运行时每次加载都要执行。如果关卡有 100 张纹理，启动时间会被拖慢几十秒。而且不同 GPU 支持的压缩格式不同（PC 用 BC7，iOS 用 ASTC，旧 Android 用 ETC2），运行时动态压缩太慢。

### 根因分析：源格式与运行时格式的根本差异

| 维度 | 源格式（PNG/FBX/OBJ） | 运行时格式 |
|------|----------------------|-----------|
| 目标 | 人类可读、可编辑、无损 | GPU 可直接使用、内存紧凑、加载快 |
| 压缩 | 通用压缩（PNG deflate、ZIP） | GPU 原生压缩（BC7/ASTC/ETC2） |
| 结构 | 保留编辑信息（图层、骨骼、关键帧） | 扁平化（顶点缓冲区、索引缓冲区、纹理数组） |
| 平台 | 平台无关 | 平台特定（不同 GPU 用不同格式） |

源格式和运行时格式的差异如此之大，以至于它们不能是同一个文件。

### 分支 A：运行时即时转换（最简单，但最慢）

**核心思路**：引擎在加载原始文件时，现场执行所有转换（解码、生成 mipmap、压缩、平台适配）。

**适用场景**：原型验证、资源数量极少、没有性能要求的教学项目。

**隐藏代价**：
- 启动极慢。每次运行都要重复相同的转换工作。
- 运行时内存峰值高。解码后的 raw 数据可能比最终 GPU 格式大 4~8 倍。
- 无法利用平台特定优化。如主机平台的专用纹理布局、GPU 的异步解压特性。

**失效条件**：任何超过十几个资源的项目。

### 分支 B：Cook 管线——源资产与引擎资产分离（UE/chaos 标准）

```
美术工作区                  Cooker（构建时）              运行时
------------                --------------               --------
wood.png  ──►  导入管线  ──► wood.texture  ──► Pak 包 ──► 引擎加载
wood.fbx  ──►  （编辑器）  ──► wood.mesh     ──►        ──► GPU 直接使用
```

**核心思路**：
1. **源资产**（Source Assets）：美术直接编辑的文件，存放在版本控制中（如 `SourceAssets/`）。
2. **引擎资产**（Engine Assets）：Cooker 生成的运行时格式，存放在构建输出目录（如 `CookOutput/`）。
3. **导入管线**：定义每种源格式到引擎格式的转换规则。如 PNG → 纹理 Cooker → `.texture` 二进制文件。
4. **增量 Cook**：只处理修改过的文件。通过时间戳或哈希判断哪些源资产需要重新 Cook。

**代码片段：Cook 配置示例**
```json
{
    "importers": {
        "*.png": {
            "target_format": "texture",
            "generate_mipmaps": true,
            "compression": "bc7",
            "platform_overrides": {
                "ios": { "compression": "astc_4x4" }
            }
        },
        "*.fbx": {
            "target_format": "mesh",
            "generate_lods": [0.5, 0.25, 0.125],
            "collision": "convex_hull"
        }
    }
}
```

**适用场景**：所有商业游戏项目、任何需要发布到多平台的项目。

**隐藏代价**：
- Cook 管线是重大工程。需要为每种资源类型写 Cooker，处理错误、依赖追踪、增量更新。
- 源资产和引擎资产之间的同步问题。如果美术修改了 PNG，但 Cooker 没有运行，运行时加载的是旧版本。
- 构建时间增加。完整 Cook 可能需要数分钟到数小时（取决于资源总量）。

**失效条件**：纯原型项目、没有美术管线的小型项目。

### 分支 C：按需生成 + 缓存（中间路线）

**核心思路**：没有独立的 Cooker 程序，但引擎在首次加载某个源资产时，自动转换为运行时格式并缓存到磁盘。下次加载直接读取缓存。

**适用场景**：小型团队、快速原型、不想维护独立 Cooker 管线的项目。

**隐藏代价**：
- 首次加载仍然慢（需要现场转换）。
- 缓存失效逻辑复杂。如果转换规则变了（如从 BC7 改为 ASTC），需要清空缓存重新生成。
- 多平台支持困难。缓存文件通常是平台特定的，需要按平台分目录。

**失效条件**：资源量大、需要精确控制构建产物的项目。

### 决策分析

**默认推荐：分支 B（Cook 管线）作为最终目标，分支 C（按需生成）作为开发期简化路径。**

推荐依据：
1. **UE 优先吸收**：UE 的 Cooker (`UnrealFrontend` / `UE4Editor-Cmd`) 是游戏行业的事实标准。"源资产 → 导入 → Cook → Pak → 运行时加载"的链路经过了无数项目的验证。
2. **chaos 的验证**：chaos 的 `AssetImporter`（三阶段：PreProcessing → Import → Attach）和 `ResourceCooker`（Dirty 检查 + 增量 Cook）证明了这条路径在自研引擎中的可行性。
3. **默认实现取舍**：开发初期，维护完整的 Cooker 管线可能过重。可以先实现「运行时按需转换 + 本地缓存」，等资源量增大后再升级为独立 Cooker。

**ECS 映射**：Cooker 本身不是 ECS 的一部分，它是编辑器/工具链模块。但 Cook 产物的运行时加载完全由 ECS 资源系统管理。在 [[资产导入管线]]（阶段 8.1）中，我们会深入讨论 Cooker 的架构设计。

**遗留问题**：资源系统的设计空间已经覆盖得差不多了。但作为 AI 友好引擎，还有一个必须回答的问题：AI 如何观察资源状态？AI 如何安全地操作资源？

## 问题 8：AI 可观测性与确定性设计

我们的引擎目标明确：**为 AI 协作开发而设计**。资源系统作为引擎的核心基础设施，必须回答以下问题。

### 可观测性：AI 如何理解资源状态？

AI 不应该需要读 C++ 头文件才能知道"当前加载了哪些纹理"。资源系统的所有状态必须是**自描述的**。

**具体设计**：

1. **资源类型注册表**：每种资源类型在初始化时向反射系统注册，提供 Schema（字段名、类型、默认值）。
   ```json
   {
     "type": "Texture",
     "fields": {
       "width": { "type": "u32", "readonly": true },
       "height": { "type": "u32", "readonly": true },
       "format": { "type": "enum", "values": ["RGBA8", "BC7", "ASTC"] },
       "mip_count": { "type": "u32", "readonly": true },
       "memory_size": { "type": "u64", "readonly": true }
     }
   }
   ```

2. **加载状态查询**：AI 可以通过 MCP 接口查询任意资源的加载状态。
   ```json
   // AI 请求
   { "tool": "query_asset_state", "args": { "handle": "Texture#42" } }
   
   // 引擎响应
   { "state": "Ready", "path": "wood_diffuse.png", "refs": 15, "memory": 16777216 }
   ```

3. **引用图导出**：AI 可以查询"哪些 Entity 引用了这张纹理"，用于影响分析和变更传播。

### 确定性：给定相同输入，资源加载是否可复现？

确定性对调试、回放、测试至关重要。如果资源加载顺序受线程调度影响，两次运行的内存布局可能不同，导致难以复现的 bug。

**具体设计**：

1. **加载顺序确定性**：如果两个资源没有依赖关系，它们的加载顺序仍然应该是确定的（如按路径字母序）。避免使用无序容器（`std::unordered_map`）存储待加载队列。
2. **异步完成事件顺序**：`process_events` System 应该按事件到达顺序处理，而不是按资源类型或其他非确定性规则。
3. **World 快照包含资源**：我们在 [[系统调度与确定性]] 中讨论过 World 快照。资源存储作为 ECS Resource，其内容应可被序列化为快照的一部分。这样，从快照恢复时，资源状态与 ECS 状态完全一致。

### 工具边界：AI 通过 MCP 能做什么？

AI 操作资源的接口必须**结构化、可验证、可撤销**。

| 操作 | MCP 工具 | 验证规则 | 安全级别 |
|------|---------|---------|---------|
| 查询资源状态 | `query_asset` | 只读，无限制 | 安全 |
| 发起异步加载 | `load_asset` | 路径白名单 | 安全 |
| 热重载资源 | `reload_asset` | 仅限开发模式 | 需确认 |
| 卸载资源 | `unload_asset` | 检查引用计数 | 危险 |
| 修改资源数据 | `edit_asset` | 类型校验 + 范围检查 | 危险 |

**Agent 安全**：危险操作（如卸载、修改）应通过事务机制实现：
1. AI 提交操作请求。
2. 引擎生成变更预览（diff）。
3. 人类开发者审批（或自动白名单通过）。
4. 操作在帧边界原子提交，支持 Undo。

这在 [[多Agent编排与沙箱]]（阶段 8.6）中会深入讨论。

## 总结：默认路径与简化路径

### 工业级默认推荐

基于以上全部分析，个人 ECS 引擎的资源管理系统默认推荐以下架构：

```
+---------------------------------------------------+
|                     主线程（ECS World）              |
|  +------------------+  +------------------------+  |
|  | MeshComponent    |  | Assets<Mesh> (Resource) |  |
|  |   Handle<Mesh>   |  |   HandleTable<Mesh>     |  |
|  +------------------+  +------------------------+  |
|  +------------------+  +------------------------+  |
|  | MaterialComponent|  | Assets<Texture>         |  |
|  |   Handle<Mat>    |  |   HandleTable<Texture>  |  |
|  +------------------+  +------------------------+  |
|                                                    |
|  Systems:                                          |
|  - AssetEventProcessSystem (PreUpdate)            |
|  - AssetDependencySystem   (PreUpdate)            |
|  - HotReloadSystem         (PostUpdate)           |
|  - RefCountUpdateSystem    (EndFrame)             |
+---------------------------------------------------+
                        ^
                        | 无锁 MPSC Channel
                        v
+---------------------------------------------------+
|                   IO 线程池                          |
|  - 文件读取 / 解码 / 反序列化                         |
|  - 完成后发送 AssetLoadedEvent 到 Channel            |
+---------------------------------------------------+
                        ^
                        | VFS 抽象
                        v
+---------------------------------------------------+
|                文件系统 / Pak 包                      |
+---------------------------------------------------+
```

**核心组件清单**：

| 组件 | 职责 | 对应笔记 |
|------|------|---------|
| `Handle<T>` | 类型安全句柄（index + generation）| 问题 1 |

### 简化路径（阶段 5 初版可落地）

如果当前阶段工程负担过重，可按以下优先级实现：

1. **必做**：`Handle<T>` + `HandleTable<T>` + `Assets<T>`（ECS Resource）。这是整个系统的基石。
2. **必做**：单后台线程 + 互斥锁队列 + 同步 `load()` 接口。至少支持非阻塞的异步加载回调。
3. **必做**：引用计数表 + 手动 `unload()`。先不实现自动回收，但接口预留。
4. **选做**：文件系统 Watch + 手动 `reload()` 触发。先不做自动检测。
5. **选做**：Cook 管线。开发期先用原始格式（PNG/FBX）直接加载，阶段 8.1 再升级为正式 Cook 管线。

**简化路径与工业级方案的差距**：
- 缺少自动依赖图（需要手动保证加载顺序）。
- 缺少无锁 channel（使用互斥锁队列）。
- 缺少帧边界批量回收（使用即时 unload）。
- 缺少自动热重载（使用手动触发）。
- 缺少多平台 Cook（使用运行时即时转换）。
