> 本笔记探讨的核心问题是：**主场景渲染完成后，具体该对像素做什么计算，才能让画面从"正确"变成"好看"？**

## 问题 1：HDR 场景如何变回能显示的画面？（Tone Mapping）

你的渲染器内部可能产生高达 10000.0 的亮度值，但显示器只能显示 0~255。直接截断（clamp）会让高光全部变成死白，暗部全部变成死黑——画面像一张过曝的相片。

**Tone Mapping 就是解决这个问题的数学工具**：它把无限的 HDR 亮度范围，非线性地压缩到有限的 LDR 范围，同时尽量保留对比度和色彩层次。

### 最 naive 的做法：Reinhard

Erik Reinhard 在 2002 年提出的经典算子，思路简单到一句话就能说完："把颜色除以（1 + 颜色）"。

```cpp
float3 reinhard(float3 hdr) {
    return hdr / (1.0 + hdr);
}
```

- **为什么有效**：当亮度很小时（hdr ≈ 0），结果 ≈ hdr，几乎不变；当亮度很大时（hdr → ∞），结果 → 1.0，平滑收敛。
- **隐藏代价**：
  - **整体发灰**：Reinhard 对高光的压缩过于"温柔"，导致中高亮度区域被挤压在一起，画面像蒙了一层雾。
  - **没有白点概念**：理论上任何亮度都能被映射，但实际上显示器有自己的峰值亮度限制，Reinhard 不考虑这个物理约束。
- **失效条件**：当场景有高对比度光源（如室内暗处 + 窗外阳光），Reinhard 会让整个画面失去冲击力。

### 分支 B：Filmic Tone Mapping（Uncharted 2）

John Hable 为《神秘海域 2》设计的曲线，后来被称为 "Hable Filmic" 或 "Uncharted 2 Tone Map"。它在 Reinhard 的基础上引入了 S 形曲线，让暗部更暗、亮部更亮，同时用 shoulder 模拟胶片的高光压缩特性。

```cpp
float3 uncharted2Tonemap(float3 x) {
    const float A = 0.15; // shoulder strength
    const float B = 0.50; // linear section
    const float C = 0.10; // toe strength
    const float D = 0.20; // linear white
    const float E = 0.02; // toe numerator
    const float F = 0.30; // toe denominator
    return ((x * (A * x + C * B) + D * E) / (x * (A * x + B) + D * F)) - E / F;
}

float3 filmicTonemap(float3 hdr) {
    float3 mapped = uncharted2Tonemap(hdr * 2.0); // exposure bias
    float3 whiteScale = 1.0 / uncharted2Tonemap(float3(11.2, 11.2, 11.2));
    return mapped * whiteScale;
}
```

- **优点**：比 Reinhard 更有"电影感"，暗部对比度和高光压缩都更自然。
- **隐藏代价**：参数（A/B/C/D/E/F）的调优需要美术经验，且不同场景可能需要不同参数。
- **失效条件**：对于需要精确物理还原的应用（如建筑可视化），Filmic 的"艺术加工"反而成为干扰。

### 分支 C：ACES（Academy Color Encoding System）

ACES 是美国电影艺术与科学学院制定的色彩标准，被好莱坞广泛使用。它的 Tone Mapping 曲线经过精心设计，以匹配胶片和数字电影的观感。

```cpp
// 简化的 ACES fitted curve（Narkowicz 2015 近似）
float3 acesTonemap(float3 x) {
    const float a = 2.51;
    const float b = 0.03;
    const float c = 2.43;
    const float d = 0.59;
    const float e = 0.14;
    return clamp((x * (a * x + b)) / (x * (c * x + d) + e), 0.0, 1.0);
}
```

- **优点**：工业级标准，跨项目、跨引擎的色彩一致性最好。高光的 shoulder 和暗部的 toe 都处理得非常细腻。
- **隐藏代价**：
  - 完整的 ACES 流程包含多个色彩空间转换（ACES2065-1 → ACEScg → ODT），计算量较大。
  - 对于游戏实时渲染，通常使用简化版（ACES fitted），但简化版在某些极端色域下可能产生色偏。
- **失效条件**：ACES 默认假设输入是线性光，如果你的渲染管线在非线性空间做光照计算，ACES 的结果会出错。

### 分支 D：AgX 与 TonyMcMapface（现代方案）

AgX 是 Troy Sobotka 等人近年推出的显示渲染变换（DRT），设计目标是解决 ACES 在某些场景下的"高光过紫"和"暗部过青"问题。Bevy 的默认 Tonemapping 是 TonyMcMapface（由 Tomasz Stachowiak 设计），它在保留细节和抑制 firefly 方面表现优异。

- **优点**：现代方案对极端 HDR 值的处理更稳健，暗部细节保留更好，且计算成本与 ACES 简化版相当。
- **隐藏代价**：部分算法需要 3D LUT 纹理查找（如 AgX、TonyMcMapface、BlenderFilmic），增加了显存占用和纹理采样开销。
- **失效条件**：在不支持 3D 纹理采样或 LUT 加载的平台（如某些 WebGL 2 环境），需要回退到纯 ALU 版本。

### 决策分析

| 方案 | 计算成本 | 观感 | 是否需要 LUT | 适用场景 |
|------|---------|------|-------------|---------|
| Reinhard | 极低（1 次除法） | 发灰，不推荐 | 否 | 仅作教学/快速原型 |
| Filmic (Uncharted 2) | 低（多项式） | 电影感强 | 否 | 追求电影风格的项目 |
| ACES Fitted | 中（多项式） | 工业标准，一致性强 | 否 | 需要跨项目色彩一致性 |
| TonyMcMapface / AgX | 中（多项式 + 可选 LUT） | 现代，稳健，细节保留好 | 部分需要 | 现代 PBR 项目的默认选择 |

**默认推荐：TonyMcMapface 或 ACES Fitted 作为默认 Tonemapping**，原因：
- Bevy 的工业验证表明 TonyMcMapface 在多种场景下表现均衡；
- ACES 是电影行业标准，如果你的项目需要与影视管线对接，ACES 更优；
- 两者都是纯 ALU 实现（简化版），无需 LUT，兼容性最好。

**ECS 映射**：`Tonemapping` 作为相机实体的组件，枚举值决定算法选择。`Exposure` 作为独立组件（或同一组件的字段），控制曝光补偿。这样 AI 可以通过 `set_component(camera, Tonemapping, TonyMcMapface)` 直接切换算法。

## 问题 2：Bloom 的"光晕"是怎么产生的？

真实相机拍摄强光时，镜头散射会在光源周围产生柔和的光晕。Bloom 就是模拟这种光学现象的屏幕空间技术。

### 根因分析：为什么 Bloom 不能"直接画"？

 Bloom 不是简单的"在亮的东西周围加一个光圈"。光源可能只占屏幕几个像素，但光晕需要扩散到周围几十甚至几百个像素。如果直接在主渲染 Pass 里做这种宽核模糊，性能会崩溃——每个像素都要采样周围 100 个邻居。

**核心洞察：Bloom 必须通过"降分辨率 + 多级模糊 + 上采样混合"来实现。**

### Bloom 的完整管线

标准 Bloom 管线分为三步：

```
SceneColor (HDR)
    │
    ▼ 阈值过滤 + 首次 Downsample
┌─────────────┐
│ Bloom Mip 0 │  (1/2 分辨率)
└─────────────┘
    │
    ▼ Downsample
┌─────────────┐
│ Bloom Mip 1 │  (1/4 分辨率)
└─────────────┘
    │
    ▼ ...继续降采样到 Mip N
┌─────────────┐
│ Bloom Mip N │  (1/2^N 分辨率)
└─────────────┘
    │
    ▼ Upsample + Tent Filter 混合
┌─────────────┐
│ Bloom Mip 0 │  (混合回 1/2 分辨率)
└─────────────┘
    │
    ▼ 最终 Upsample，叠加回 SceneColor
SceneColor + Bloom
```

#### Step 1：Prefilter + 首次 Downsample

从全分辨率 SceneColor 中提取过亮区域，同时降采样到半分辨率。这一步通常使用一个 13-tap 采样核（参考 Call of Duty: Advanced Warfare 的实现），在降采样的同时进行轻微模糊：

```wgsl
// 首次 Downsample 的 WGSL 伪代码
fn sample_input_13_tap(uv: vec2<f32>) -> vec3<f32> {
    // 13 个采样点分布在 4x4 网格的角落和中心
    var group0 = (a + b + d + e) * (0.125 / 4.0);
    var group1 = (b + c + e + f) * (0.125 / 4.0);
    var group2 = (d + e + g + h) * (0.125 / 4.0);
    var group3 = (e + f + h + i) * (0.125 / 4.0);
    var group4 = (j + k + l + m) * (0.5 / 4.0); // 中心权重更高

    // Firefly reduction：用 Karis average 抑制单像素过亮（萤火虫）
    group0 *= karis_average(group0);
    group1 *= karis_average(group1);
    group2 *= karis_average(group2);
    group3 *= karis_average(group3);
    group4 *= karis_average(group4);

    return group0 + group1 + group2 + group3 + group4;
}
```

**Karis average** 的作用是：如果 4 个像素中有一个特别亮（firefly），降低它的权重，防止单像素噪声扩散成大面积光斑。

#### Step 2：Downsample 链

从 Mip 0 到 Mip N，每次分辨率减半。非首次的 Downsample 使用更简单的盒式滤波：

```wgsl
fn downsample(uv: vec2<f32>) -> vec3<f32> {
    var sample = (a + c + g + i) * 0.03125;     // 角落
    sample += (b + d + f + h) * 0.0625;          // 边中
    sample += (e + j + k + l + m) * 0.125;       // 中心和内部
    return sample;
}
```

**为什么降采样就是模糊？** 因为降采样时使用了双线性过滤（bilinear filter），相邻像素的加权平均天然实现了 2×2 的 box blur。连续降采样多级，等效于一个非常大的高斯核，但计算量呈指数下降。

#### Step 3：Upsample + 混合

从最小 Mip 开始，逐级上采样并混合。上采样使用 tent filter（3×3）：

```wgsl
fn upsample_tent(uv: vec2<f32>) -> vec3<f32> {
    let texel = 1.0 / vec2<f32>(textureDimensions(input));
    var sum = sample(e) * 0.25;                  // 中心
    sum += sample(b + d + f + h) * 0.125;        // 四边
    sum += sample(a + c + g + i) * 0.0625;       // 四角
    return sum;
}
```

每一级 Upsample 时，将当前 Mip 的模糊结果与下一级（更大、更模糊）的结果混合。混合权重由 parametric curve 控制，实现"低频光晕更多、高频光晕更少"的艺术效果。

### 分支研究：Bloom 的混合模式

#### 分支 A：Energy Conserving（能量守恒）

```
dest = src * bloomIntensity + dest * (1.0 - bloomIntensity)
```

- **优点**：物理上更合理，总光量不变，只是重新分布。自然、柔和。
- **代价**：强度太高时画面会变暗（因为能量从原图"抽走"了）。

#### 分支 B：Additive（加法叠加）

```
dest = src * bloomIntensity + dest
```

- **优点**：明亮、发光感强，适合霓虹、魔法等风格化场景。
- **代价**：能量不守恒，容易过曝。需要谨慎控制强度。

**默认推荐：Energy Conserving 作为默认模式**，Additive 作为可选风格化开关。

### 内存布局与状态变化

Bloom 的 Downsample 链需要一张带 Mipmap 的纹理（或在 WebGL2 下用多张纹理）：

```
初始状态：BloomTexture[Mip0..MipN] 全部未初始化

Pass 0 (First Downsample):
  读取：SceneColor (full res)
  写入：BloomTexture[Mip0] (1/2 res)
  
Pass 1..N (Downsample):
  读取：BloomTexture[Mip i-1]
  写入：BloomTexture[Mip i]
  
Pass N+1..2N-1 (Upsample):
  读取：BloomTexture[Mip i] + BloomTexture[Mip i-1] (作为混合源)
  写入：BloomTexture[Mip i-1] (LoadOp::Load，保留之前内容并叠加)
  
最终 Pass (Composite):
  读取：BloomTexture[Mip0]
  写入：ViewTarget (SceneColor)
```

> **注意**：Upsample 阶段的 LoadOp 必须是 `Load` 而非 `Clear`，因为要在之前写入的内容上叠加。这是 Bloom 实现中最常见的坑之一。

## 问题 3：时域抗锯齿 TAA

锯齿（Aliasing）是实时渲染的顽疾。MSAA 开销大，FXAA 会模糊细节。TAA 的核心思想是：**用时间换空间**——每帧只渲染一个偏移采样点，但复用过去多帧的采样结果，累积成等效的超采样。

### 最 naive 的做法：直接混合历史帧

```cpp
float3 taa_naive(float3 currentColor, float3 historyColor) {
    return lerp(currentColor, historyColor, 0.9); // 90% 历史 + 10% 当前
}
```

- **问题**：如果场景中有物体移动，历史像素对应的是**上一帧的不同位置**。直接混合会产生可怕的鬼影（Ghosting）——移动的物体拖着长长的尾巴。

### 分支 A：Velocity Buffer + 重投影

解决 Ghosting 的关键是知道每个像素"上一帧在哪里"。Velocity Buffer 记录了这个信息：

```cpp
// 基础 Pass 中写入 Velocity
float2 velocity = (currentClipPos.xy - prevClipPos.xy) * 0.5; // NDC 空间差值
```

TAA Pass 中，用当前像素的 velocity 回溯到上一帧的对应位置，采样历史颜色：

```cpp
float2 historyUV = currentUV - velocity;
float3 historyColor = sampleHistory(historyUV);
```

- **优点**：运动物体的历史采样点跟随移动，基本消除 Ghosting。
- **隐藏代价**：
  - **Velocity Buffer 的精度**：如果 velocity 计算有误（如静态物体因为相机移动产生了微小 velocity），重投影会偏移到错误位置。
  - **遮挡边缘**：物体 A 从物体 B 前面经过，B 被遮挡的区域上一帧不存在于当前位置，重投影会采样到 A 的颜色，导致边缘闪烁。

### 分支 B：颜色约束（Color Clamping / AABB）

即使重投影正确，历史颜色也可能与当前邻域颜色差异巨大（如光源突然亮起、新物体出现）。需要对历史颜色进行**可信度校验**。

最常见的方法是 **YCoCg 颜色空间的 AABB 约束**：

```cpp
// 1. 收集当前帧 3x3 邻域的颜色
float3 neighbors[9];
for (int i = 0; i < 9; i++) {
    neighbors[i] = sampleCurrent(neighborOffsets[i]);
}

// 2. 转换到 YCoCg 空间（亮度与色度分离）
for (int i = 0; i < 9; i++) {
    neighbors[i] = RGB_to_YCoCg(neighbors[i]);
}

// 3. 计算邻域的 AABB（最小/最大值）
float3 aabbMin = minOf(neighbors);
float3 aabbMax = maxOf(neighbors);

// 4. 将历史颜色限制在 AABB 内
float3 historyYCoCg = RGB_to_YCoCg(historyColor);
historyYCoCg = clamp(historyYCoCg, aabbMin, aabbMax);

// 5. 混合
float3 clampedHistory = YCoCg_to_RGB(historyYCoCg);
float3 result = lerp(currentColor, clampedHistory, blendWeight);
```

- **为什么用 YCoCg**：亮度（Y）和色度（Co/Cg）分离后，可以单独对亮度做更严格的约束（因为人眼对亮度闪烁更敏感），对色度放宽约束（保留更多历史颜色信息）。
- **变体**：一些实现使用颜色方差而非 AABB，或结合深度差异进行更精细的拒绝。

### 分支 C：Motion Blur 的集成与分离

Motion Blur 和 TAA 都依赖 Velocity Buffer，但目标不同：
- **TAA**：用历史帧填充单帧采样不足，目标是"更清晰"。
- **Motion Blur**：用快门模型模糊运动物体，目标是"更有动态感"。

如果两者同时开启，顺序很重要：
- **TAA 在 Motion Blur 之前**：TAA 先稳定锯齿，Motion Blur 在此基础上模糊运动。这是 UE 和多数引擎的选择。
- **Motion Blur 在 TAA 之前**：Motion Blur 会改变当前帧的颜色分布，TAA 的历史约束可能失效。

**默认推荐：TAA → Motion Blur 的顺序**。

## 问题 4：屏幕空间环境光遮蔽 SSAO

环境光遮蔽（Ambient Occlusion）解决的是：间接光照下，角落、缝隙、凹陷处应该更暗。实时全局光照太昂贵，SSAO 用屏幕空间近似实现。

### 最 naive 的做法：经典 SSAO

Crytek 在 2007 年提出的原始算法：对每个像素，在周围半球内随机采样几个点，投影到屏幕空间查深度，如果被遮挡则贡献 AO。

```cpp
float ssao_naive(float3 position, float3 normal, float2 uv) {
    float occlusion = 0.0;
    for (int i = 0; i < SAMPLE_COUNT; i++) {
        float3 samplePos = position + randomHemisphereVector(normal, i) * radius;
        float4 projected = mul(viewProjMatrix, float4(samplePos, 1.0));
        float2 sampleUV = projected.xy / projected.w * 0.5 + 0.5;
        float sampleDepth = sampleSceneDepth(sampleUV);
        
        if (sampleDepth < projected.z) { // 被遮挡
            occlusion += 1.0;
        }
    }
    return 1.0 - (occlusion / SAMPLE_COUNT);
}
```

- **问题**：
  - **噪点**：采样数少时噪点严重，采样数多时性能崩溃。
  - **Halo**：在物体边缘，背景深度会被误判为遮挡，产生一圈黑边。
  - **平面错误**：纯平面上如果没有倾斜，半球采样会错误地认为平面自身遮挡了自己。

### 分支 B：HBAO+（Horizon-Based Ambient Occlusion）

NVIDIA 提出的改进方案，核心思想是：在像素周围的多个方向上扫描地平线角度，计算被地形遮挡的天空比例。

- **优点**：
  - 大幅减少了 Halo  artifacts，因为基于地平线角度的计算对深度不连续更鲁棒。
  - 方向性扫描产生的噪点有结构化特征，更容易被后续模糊 Pass 消除。
- **隐藏代价**：需要更多纹理采样（每个方向 6~8 个采样点，4~6 个方向），带宽更高。

### 分支 C：GTAO（Ground Truth Ambient Occlusion）

近年提出的"ground truth"方案，通过更精确的几何积分模型，在更少采样下获得更高质量。

- **优点**：质量接近光线追踪 AO，同时保持屏幕空间性能。
- **隐藏代价**：算法更复杂，需要多个 Pass（AO 计算 + 多尺度 Temporal Filter + Spatial Filter），总耗时可能高于 HBAO+。

### 决策分析

| 方案 | 质量 | 性能 | 实现复杂度 |
|------|------|------|-----------|
| 经典 SSAO | 低 | 中 | 低 |
| HBAO+ | 中 | 中 | 中 |
| GTAO | 高 | 中高 | 高 |

**默认推荐：HBAO+ 作为默认推荐的 SSAO**，原因：
- 质量足以满足大多数场景；
- 实现复杂度可控，网上有大量参考实现；
- 性能在 1080p 下通常 < 1ms。
