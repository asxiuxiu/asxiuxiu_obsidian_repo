## 问题 5：基础粒子系统

后处理是"对已有图像的操作"，但粒子系统创造的是**全新的动态图像元素**——火焰、烟雾、火花、魔法光点。虽然技术上粒子不属于"后处理"，但它与后处理有深度交互（粒子的自发光是 Bloom 的重要来源，粒子的透明度排序与后处理管线共享 RenderTarget）。

### 为什么 ECS 架构下粒子是特殊问题？

如果每个粒子都是一个 ECS Entity，高频的创建和销毁会对 Archetype 存储造成巨大压力——Archetype 的内存布局假设组件类型相对稳定，而粒子的生命周期通常只有几秒甚至几帧。

### 分支研究：粒子系统的架构选择

#### 分支 A：CPU 粒子（简单、可控）

在 CPU 上模拟粒子位置、速度、生命周期，每帧将结果写入 GPU buffer，然后实例化渲染：

```cpp
struct Particle {
    Vec3 position;
    Vec3 velocity;
    float life;      // 剩余生命
    float maxLife;   // 总生命
    Color color;
    float size;
};

// CPU 更新（每帧）
for (auto& p : particles) {
    p.velocity.y -= gravity * dt;
    p.position += p.velocity * dt;
    p.life -= dt;
    p.color.a = p.life / p.maxLife; // 淡出
}

// 移除死亡粒子
remove_if(particles, [](const Particle& p) { return p.life <= 0; });

// 上传到 GPU 并渲染
uploadToGPU(particles);
drawInstanced(particleQuadMesh, particles.size());
```

- **优点**：逻辑简单，易于调试，支持复杂的 CPU 端逻辑（如碰撞检测、物理交互）。
- **隐藏代价**：
  - **CPU-GPU 带宽瓶颈**：粒子数量超过 10k 时，每帧上传数据量显著。
  - **CPU 并行度有限**：粒子模拟的向量化程度不高，大量粒子会吃掉主线程时间。
- **失效条件**：粒子数量 > 50k 或需要复杂物理模拟（如流体、烟雾）时，CPU 成为瓶颈。

#### 分支 B：GPU 粒子（Compute Shader 驱动）

将粒子的生成（Spawn）、初始化（Init）、更新（Update）、排序（Sort）全部放到 GPU Compute Shader 中：

```cpp
// Compute Shader：粒子更新
[numthreads(256, 1, 1)]
void updateParticles(uint id : SV_DispatchThreadID) {
    if (id >= particleCount) return;
    
    Particle p = particleBuffer[id];
    p.velocity.y -= gravity * dt;
    p.position += p.velocity * dt;
    p.life -= dt;
    
    if (p.life <= 0.0) {
        // 标记为死亡，由 compact shader 回收
        aliveBuffer[id] = 0;
    } else {
        particleBuffer[id] = p;
        aliveBuffer[id] = 1;
    }
}
```

- **优点**：
  - **零 CPU-GPU 数据传输**：粒子数据始终驻留 GPU，支持百万级粒子。
  - **与渲染共享内存**：更新后的粒子 buffer 直接作为实例化渲染的顶点数据。
- **隐藏代价**：
  - **调试极其困难**：无法直接断点 GPU 逻辑。
  - **需要 Compute Shader 支持**：WebGL2 不支持 Compute，必须要求 WebGPU。
  - **Spawn 逻辑复杂**：CPU 需要知道每帧该 spawn 多少粒子（基于发射率、burst 等），这个计算通常在 CPU 端完成，然后写入 GPU 的 spawn counter。

#### 分支 C：Expression / Modifier 驱动的 GPU 粒子（bevy_hanabi 风格）

社区方案 `bevy_hanabi` 采用的架构：在 CPU 侧用 Expression API 构建粒子行为树，然后动态生成 WGSL Compute Shader。

- **优点**：极高的灵活性，用户可组合任意 modifier（重力、力场、碰撞、颜色渐变）而不手写 Shader。
- **隐藏代价**：引入了一套 DSL 和 Shader 拼接逻辑，内部复杂度很高。

### ECS 映射与决策分析

**默认推荐：分支 A（CPU 粒子）作为默认推荐的起点**，原因：
- 实现简单，与 ECS 架构自然融合（粒子发射器是 Entity，粒子数据用 SoA 或连续数组存储）。
- 对于阶段 5 的里程碑（"后处理 Bloom 生效"），CPU 粒子足以产生自发光效果。
- GPU 粒子的复杂度（Compute Shader、Buffer 管理、平台兼容性）应留到阶段 9 或更后期。

**ECS 映射**：
- `ParticleEmitter` 组件：定义发射率、生命周期范围、初始速度分布。
- `ParticleSystem` Resource：管理所有活跃粒子的连续内存池（不是每个粒子一个 Entity，而是用 `Array<Particle>` 批量存储，避免 Archetype 抖动）。
- 渲染时，粒子作为透明物体插入主渲染的 Transparent Phase，在 Bloom 之前绘制，确保自发光能被 Bloom 捕捉。

**条件切换**：当项目需要 > 10k 粒子或复杂物理模拟时，迁移到分支 B（GPU 粒子）。迁移路径：保留 CPU 端的 `ParticleEmitter` 组件和参数配置，将更新逻辑从 CPU System 替换为 Compute Dispatch。
