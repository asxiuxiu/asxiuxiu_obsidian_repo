# 为什么需要材质系统？

想象你正在写一个最 naive 的渲染程序。你有一堆模型要画，每个模型颜色不一样。最直白的做法是什么？

```cpp
// 分支 A：完全硬编码（naive 方案）
void DrawRedCube() {
    shader_red.Use();          // 每套颜色一个 shader
    SetUniform("color", 1,0,0);
    DrawMesh(cube_mesh);
}
void DrawBlueCube() {
    shader_blue.Use();         // 复制粘贴，只是颜色不同
    SetUniform("color", 0,0,1);
    DrawMesh(cube_mesh);
}
```

如果只有红蓝两个立方体，这勉强能跑。但场景里有 100 个物体、10 种贴图、开关骨骼动画、切换光照模型呢？你很快会陷入**着色器代码的复制粘贴地狱**——不是美术在调材质，而是程序员在硬编码 `shader_xxx.cpp`。

更深层的问题是：GPU 执行的底层代码（着色器）与美术想要的"外观调整"（颜色、光泽、贴图）处于完全不同的抽象层次。我们需要一座桥，把"美术语言"翻译成"GPU 语言"。这座桥就是**材质系统**。

但材质系统不只是"给 shader 传参数"那么简单。现代渲染管线中，一个"材质"可能触发数十种 GPU 状态组合（有无法线贴图？延迟还是前向？骨骼动画还是静态？），这些组合如果处理不好，会导致**着色器变体爆炸**——编译时间以小时计、内存占用以 GB 计、首次加载时帧率暴跌。材质系统的设计目标，就是在"美术灵活性"和"运行时可控性"之间找到平衡。


# 问题 1：材质与着色器之间应该是什么关系？

## 场景与根因

假设美术做了一张"生锈金属"的材质。这个材质被用到三种物体上：静态雕像、带动画的士兵、UI 图标。三种物体的顶点格式不同（静态模型只有位置+法线+UV，士兵多了骨骼权重，UI 只需要位置+UV），渲染路径也不同（士兵走延迟渲染，UI 走前向透明）。

如果"材质"直接等于"着色器程序"，那么"生锈金属"需要复制成三份着色器代码，分别处理三种顶点格式和渲染路径。美术改一下粗糙度，三份代码都要同步修改——这是维护噩梦。

问题的根因在于：**材质描述的是"外观属性"（颜色、贴图、参数），着色器描述的是"GPU 执行指令"（顶点变换、像素着色）。两者是多对多的关系，不能硬绑在一起。**

## 分支研究

### 分支 A：一对一绑定（每个材质一个独立着色器）

**核心思路**：材质资产直接包含完整的 VS/PS 源码。

```cpp
struct Material {
    std::string vertex_shader_source;
    std::string fragment_shader_source;
    Texture* albedo_texture;
    float roughness;
};
```

**适用场景**：极小型的 demo、教学项目、每个物体确实需要完全不同的着色逻辑。

**隐藏代价**：
- 共享困难：10 个物体都想要"金属"外观，但顶点格式不同，就得复制 10 份源码。
- 编译爆炸：每个材质都是独立的着色器编译单元，缓存无法共享。
- 美术无法工作：要求美术直接写 GLSL/HLSL。

**失效条件**：项目规模超过 10 个材质，或需要支持多种顶点格式/渲染路径。

### 分支 B：模板-实例分层（工业级方案）

**核心思路**：把"材质"拆成三层——**模板（Template）**定义变体规则，**资产（Asset）**定义具体参数，**实例（Instance）**绑定运行时资源。

```cpp
// 模板层：定义 "这个材质支持哪些关键字组合"
struct ShaderTemplate {
    // 如：skinning_type, lighting_model, mesh_type
    Array<ShaderCategory> categories;
    
    // 根据关键字组合生成唯一的变体 ID
    uint64_t GeneratePermutationHash(Array<StringId> keywords);
};

// 资产层：美术编辑的"材质定义"
struct MaterialAsset {
    ShaderTemplate* template_ref;
    PassHint pass_hint;              // deferred / translucent / ui
    RenderState raster_state;
    RenderState depth_state;
    Array<MaterialParameter> parameters;  // 颜色、贴图引用等
};

// 实例层：运行时对象，关联具体 GPU 资源
struct MaterialInstance {
    MaterialAsset* asset_ref;
    // permutation_id → Technique（实际可执行的 Shader 程序集）
    HashMap<uint32_t, ShaderTechnique> technique_cache;
};
```

**关键机制**：当渲染系统请求一个具体的着色器变体时（如"生锈金属 + 有骨骼 + 延迟渲染"），`MaterialInstance` 用关键字组合生成 `permutation_id`，查找缓存。如果未命中，就触发异步构建——从模板定义中读取各阶段的着色器源码，组合成完整的 `ShaderTechnique`（包含 VS/PS/SRG/PSO）。

**适用场景**：任何需要支持多种变体组合的工业级引擎。

**隐藏代价**：
- 关键字组合过多时，`technique_cache` 可能急剧膨胀（Permutation 爆炸）。
- 异步构建有帧延迟：首帧请求某个变体时，该物体可能漏绘或使用 fallback。

**失效条件**：Category 维度失控（如同时定义 10 个二值开关，产生 1024 种组合）。

### 分支 C：纯数据驱动 + ECS 组件映射（ECS 原生方案）

**核心思路**：材质不是继承树中的对象，而是 ECS 世界中的数据和系统。Bevy 的做法是：材质作为 `Asset<Material>` 存在，渲染阶段通过 `AsBindGroup` trait 自动转换为 GPU 绑定。

```rust
// Bevy 风格的 ECS 材质定义（概念转写为 C++）
struct StandardMaterial {
    Handle<Texture> base_color_texture;
    Vec4 base_color;
    float roughness;
    float metallic;
    // ...
};

// 渲染系统查询所有带 StandardMaterial 的实体
// 自动提取参数、创建 BindGroup、提交绘制
```

**关键机制**：`AsBindGroup` 把材质字段映射到着色器的 binding 槽位。`#[uniform(0)]` 表示这个字段对应 binding 0 的 uniform buffer，`#[texture(1)]` 表示对应 binding 1 的纹理。渲染系统遍历 ECS 查询结果时，为每个（材质类型，视图条件）组合生成特化的渲染管线。

**适用场景**：ECS 原生引擎，希望材质系统与 ECS 架构深度集成。

**隐藏代价**：
- 每个材质类型需要独立的渲染管线特化逻辑。
- 对"节点图编辑"的支持不如 OOP 引擎直观（Bevy 目前主要面向代码定义材质）。

**失效条件**：需要复杂的可视化节点图编辑（如 UE 材质编辑器）。

## 决策分析与推荐

**默认推荐：分支 B（模板-实例分层）**，因为 UE 的工业级验证表明这是处理复杂材质关系的最稳健方案。该方案在 ECS 中的映射方式是：

- `ShaderTemplate` 和 `MaterialAsset` 作为**共享资源**（类似 Bevy 的 Asset），由资源管理系统统一管理生命周期。
- `MaterialInstance` 作为**运行时句柄**存储在 ECS Component 中（如 `MaterialHandle` 组件）。
- 渲染 System 在 `prepare` 阶段查询 `Query<(MeshHandle, MaterialHandle, Transform)>`，根据当前渲染路径和网格特性，向 `MaterialInstance` 请求对应的 `ShaderTechnique`。

**条件切换**：如果项目规模很小（<20 种材质、无复杂变体需求），可先用分支 C 的简化数据驱动方案，但需明确标注"这是工业级方案的简化版，未来需补全模板-实例分层和异步构建机制"。

**参考：chaos / UE / Bevy 对这个问题是怎么做的？**

- **chaos**：采用严格的三级体系（Template → Asset → Instance），通过 `Category/Keyword` 机制管理变体，延迟异步构建 Technique。
- **UE**：`UMaterialInterface → UMaterial → UMaterialInstance`，`FMaterialResource` 按 (Platform, QualityLevel, FeatureLevel) 投影出多个编译目标，`FMaterialShaderMap` 管理所有变体，DDC 缓存避免重复编译。
- **Bevy**：`Asset<Material>` 通过 `AsBindGroup` 直接生成 BindGroup，材质类型即 Rust 类型，渲染管线通过 `SpecializedMeshPipeline` 按 Key 特化。

**默认推荐依据**：chaos 和 UE 的分层思路一致，都解决了"美术资产"到"GPU 状态"的多对多映射。Bevy 的 ECS 原生表达更简洁，但在复杂变体管理（如按平台/质量等级生成不同代码）上不如 UE 完备。因此我们取 UE 的分层架构，用 ECS 组件存储运行时句柄。
