## 问题 0：为什么 ECS 引擎需要事件总线？

### 场景绑定

假设你正在实现一个 2D 平台跳跃游戏。玩家按下空格键，此时引擎需要同时响应：让玩家跳跃、播放跳跃音效、在脚下生成灰尘粒子、UI 显示跳跃连击数。紧接着，玩家击杀了敌人，引擎又要同时触发：掉落物品、更新击杀任务、检查成就、3 秒后召唤增援。

如果所有这些连锁反应都通过"System 直接调用 System"来完成，你的代码会变成一张无法维护的蜘蛛网。

### 根因分析

ECS 通过依赖图保证 System 按正确顺序执行，但依赖图只解决**执行顺序**问题，不解决**通信耦合**问题。当 System A 需要通知 System B、C、D 发生了某事时，如果 A 直接持有 B/C/D 的引用：

1. **新增消费者必须修改生产者**：每新增一个对"跳跃"感兴趣的 System，都要回头修改 `InputSystem`。
2. **依赖图爆炸**：生产者和消费者之间形成 N×M 的显式依赖，依赖图失去意义。
3. **时序不可控**：如果 A 在 B 的遍历中途触发副作用（如删除实体），B 的迭代器会失效，导致崩溃或脏读。

**本质矛盾**：System 需要**知道发生了什么**，但不应该**知道谁在听**。

### 分支方案

| 方案 | 核心思想 | 代码示意 | 缺点 |
|------|---------|---------|------|
| **直接引用调用** | A 直接持有 B/C/D 的指针并调用 | `movement_sys->on_jump(e); audio_sys->on_jump(e);` | 耦合度最高，违反开闭原则 |
| **组件隐式传递** | A 写 `JumpRequest` 组件，B/C/D Query 该组件 | `world.add<JumpRequest>(player);` | 组件是持久状态，事件是瞬态数据，混用会污染 ECS 数据模型 |
| **事件总线（Pub-Sub）** | A 向总线 `publish`，B/C/D 向总线 `subscribe` | `bus.publish(JumpEvent{e});` | 引入间接层，但彻底解耦生产者和消费者 |

### 决策分析

默认推荐**事件总线**。组件隐式传递看似 ECS-native，但它混淆了"状态"和"事件"的语义边界——组件表示"这个实体当前处于什么状态"，事件表示"这个帧内发生了什么事"。把事件当成组件留在实体上，会导致 Query 结果在帧与帧之间产生不可预期的残留。事件总线在语义上更精确，且为后续的延迟队列、事件过滤、多 Agent 路由提供了统一的扩展点。

### 引擎对照

| 引擎 | 事件通信机制 | 核心特点 |
|------|-------------|---------|
| **chaos** | `MessageBus::publish<TMessage>(msg)` + `subscribe(callback)` | 纯回调式 Pub-Sub，无延迟队列，publish 期间 unsubscribe 不安全 |
| **UE** | `DECLARE_DELEGATE` / `DECLARE_MULTICAST_DELEGATE` | 基于委托的多播机制，支持弱引用绑定（`UObject` 销毁时自动解绑），但仍在 UObject 体系内运行 |
| **Bevy** | `EventWriter<T>` / `EventReader<T>` | ECS-native 的事件队列，事件存储在 `Events<T>` 资源中，System 通过 Reader 按帧消费，天然支持批量读取 |

> **核心洞察**：引擎需要一种**解耦的、安全的、可延迟的**通信机制，让 System 可以只关心"发生了什么"，而不关心"谁在处理"。

## 问题 1：如何设计一个类型安全且遍历安全的即时事件总线？

### 场景绑定

你实现了一个 `EventBus<DamageEvent>`，`CombatSystem` 在 `publish` 期间遍历所有 subscriber。此时 `UISystem` 收到事件后决定切换场景，于是调用了 `unsubscribe` 销毁自己的监听器。如果 `unsubscribe` 立即从 subscriber 数组中删除元素，正在遍历的 `CombatSystem` 会遇到迭代器失效或跳过下一个 subscriber。

### 根因分析

标准容器的**迭代期间修改**是 C++ 的经典陷阱。`std::vector` 在遍历过程中执行 `erase` 会导致所有后续元素的迭代器失效；即使使用 `swap-and-pop`，也可能跳过元素或触发 use-after-free。事件总线的核心矛盾是：**publish 是读操作，unsubscribe 是写操作，两者在同一时间尺度上冲突**。

### 分支方案

**方案 A：读写锁保护**
```cpp
std::shared_mutex mtx;
void publish(const TEvent& e) {
    std::shared_lock lock(mtx);
    for (auto& sub : subscribers) sub.cb(e);
}
void unsubscribe(uint64_t id) {
    std::unique_lock lock(mtx);
    // erase...
}
```
- 优点：语义清晰， publish 可并发。
- 缺点：锁粒度粗，`publish` 内部如果嵌套 `publish` 会导致死锁；单线程 ECS 中锁是纯粹的开销。

**方案 B：延迟清理（Swap-and-Pop）**
```cpp
struct Subscriber {
    uint64_t id;
    Callback cb;
    bool     pending_remove = false;
};

void unsubscribe(uint64_t id) {
    for (auto& sub : subscribers) {
        if (sub.id == id) {
            sub.pending_remove = true;
            dirty_queue.push(id);
            break;
        }
    }
}

void publish(const TEvent& event) {
    for (auto& sub : subscribers) {
        if (!sub.pending_remove) {
            sub.cb(event);
        }
    }
}

void tick() {
    // 延迟清理：swap-and-pop 保持连续
    while (!dirty_queue.empty()) dirty_queue.pop();
    for (int i = (int)subscribers.size() - 1; i >= 0; --i) {
        if (subscribers[i].pending_remove) {
            subscribers[i] = subscribers.back();
            subscribers.pop_back();
        }
    }
}
```
- 优点：无锁，`publish` 期间安全标记删除，在下一安全点（如帧末 `tick`）统一清理。
- 缺点：subscriber 数组在清理前会短暂保留僵尸条目，占用极少量内存。

**方案 C：链表节点 + 引用计数**
每个 subscriber 是堆分配的链表节点，`unsubscribe` 只减少引用计数并标记为 dead，`publish` 跳过 dead 节点，由后台线程定期回收。
- 优点：无遍历期修改问题，支持跨线程。
- 缺点：堆分配开销大，缓存不友好，不适合高频 publish。

### 决策分析

默认推荐**方案 B（延迟清理）**。单线程 ECS 中，锁（方案 A）是过度设计，链表（方案 C）牺牲了缓存局部性。延迟清理在零同步开销的前提下，用 O(1) 的标记代价解决了迭代期修改问题。`tick()` 应在 Scheduler 的安全点调用——如每个 Phase 末尾或帧末——确保没有 System 正在遍历 subscriber 列表。

### 引擎对照

| 引擎 | 遍历安全策略 | 评价 |
|------|-------------|------|
| **chaos** | 无延迟清理，`publish` 期间 `unsubscribe` 未定义行为 | 需要使用者自行保证时序 |
| **UE** | `TMulticastDelegate` 内部维护绑定列表，遍历时支持 `RemoveAll` 但非原子 | 依赖 UObject GC 解绑，非 ECS 架构 |
| **Bevy** | 无回调式 subscriber，`EventReader` 通过索引消费事件队列 | 从根本上消除了"遍历 subscriber 列表"的问题 |

### 最简化实现

以下是类型安全的模板化 `EventBus`，支持延迟订阅清理：

```cpp
#include <vector>
#include <queue>
#include <functional>
#include <cstdint>
#include <typeindex>
#include <unordered_map>

template<typename TEvent>
struct EventBus {
    using Callback = std::function<void(const TEvent&)>;

    struct Subscriber {
        uint64_t id;
        Callback cb;
        bool     pending_remove = false;
    };

    uint64_t next_id = 1;
    std::vector<Subscriber> subscribers;
    std::queue<uint64_t>    dirty_queue; // 延迟清理队列

    uint64_t subscribe(Callback cb) {
        uint64_t id = next_id++;
        subscribers.push_back({id, cb, false});
        return id;
    }

    void unsubscribe(uint64_t id) {
        for (auto& sub : subscribers) {
            if (sub.id == id) {
                sub.pending_remove = true;
                dirty_queue.push(id);
                break;
            }
        }
    }

    void publish(const TEvent& event) {
        for (auto& sub : subscribers) {
            if (!sub.pending_remove) {
                sub.cb(event);
            }
        }
    }

    void tick() {
        while (!dirty_queue.empty()) dirty_queue.pop();
        for (int i = (int)subscribers.size() - 1; i >= 0; --i) {
            if (subscribers[i].pending_remove) {
                subscribers[i] = subscribers.back();
                subscribers.pop_back();
            }
        }
    }
};

// ---------- 全局事件总线管理器 ----------
struct EventBusManager {
    std::unordered_map<std::type_index, void*> buses; // 类型擦除存储

    template<typename TEvent>
    EventBus<TEvent>& get() {
        std::type_index key = typeid(TEvent);
        if (buses.find(key) == buses.end()) {
            buses[key] = new EventBus<TEvent>();
        }
        return *static_cast<EventBus<TEvent>*>(buses[key]);
    }

    void tick_all() {
        // 遍历所有 bus 并调用 tick()
        // 实际项目中可用 RTTI 或自定义注册表替代
    }
};
```

这个最小实现解决了：
- **解耦通信**：`InputSystem` 只需要 `bus.publish(JumpEvent{entity_id})`，不需要知道谁在听。
- **延迟清理**：`unsubscribe` 不会立即修改数组，避免了 `publish` 迭代期间迭代器失效。
- **类型安全**：每种事件类型有独立的 `EventBus<TEvent>`，编译期保证不会把 `JumpEvent` 错发成 `ShootEvent`。

还缺什么？这正是后续问题要解决的：事件无法被 ECS Query、没有延迟事件队列、没有输入映射层。

## 问题 2：如何解决"事件在 System tick 中途触发"导致的时序灾难？

### 场景绑定

`CombatSystem` 在 `Update` Phase 遍历所有带 `Health` 组件的存活实体。遍历到第 5 个实体时，该实体受到致命伤害，`CombatSystem` 调用 `publish(EnemyDiedEvent{e=42})`。如果事件是即时回调的，`LootSystem` 的回调立即执行，在 `LootSystem` 内部创建了一堆掉落物实体——而此时 `CombatSystem` 的遍历还没有结束，新实体可能被错误地纳入遍历范围，或者更糟：回调里触发了 `destroy_entity`，导致 `CombatSystem` 的 `for` 循环访问已释放内存。

### 根因分析

ECS 的 System 通常假设"本轮遍历期间，世界结构是稳定的"。即时回调破坏了这一假设：它把**同步的、不可中断的遍历**变成了**嵌套的、可重入的修改链**。这不仅是迭代器失效问题，更是**时序语义**问题——如果 `AISystem` 在 `CombatSystem` 之前运行，`AISystem` 发布的 `DamageEvent` 是否应该在本帧影响 `CombatSystem`？即时回调会让时序依赖于 System 的注册顺序，而注册顺序是隐式的、脆弱的。

### 分支方案

**方案 A：递归锁 + 即时回调**
给每个 `EventBus` 加 `std::recursive_mutex`，`publish` 时加锁，回调内部允许再次 `publish`。
- 优点：实现简单，事件响应无延迟。
- 缺点：递归锁隐藏了重入风险，调试困难；锁竞争严重；无法解决"遍历中途结构突变"的语义问题。

**方案 B：延迟事件队列（Event Queue）**
`publish` 只将事件写入 `pending_events`，不调用回调。`dispatch()` 在 Scheduler 的安全点（Phase 末尾）统一调用：
```cpp
template<typename TEvent>
struct EventBus {
    std::vector<TEvent> pending_events;
    std::vector<Subscriber> subscribers;

    void publish(const TEvent& e) {
        pending_events.push_back(e); // 只入队，不回调
    }

    void dispatch() {
        for (const auto& e : pending_events) {
            for (auto& sub : subscribers) {
                if (!sub.pending_remove) sub.cb(e);
            }
        }
        pending_events.clear();
    }
};
```
- 优点：结构安全，时序确定，同一 Phase 内所有 System 稳定。
- 缺点：事件响应延迟一截（Phase 末尾），对于需要立即反馈的输入事件不够灵敏。

**方案 C：命令模式（Command Pattern）**
System 不直接发布事件，而是生成 `Command` 对象压入 `CommandBuffer`，由 Scheduler 在 Phase 切换时统一执行。
- 优点：命令可撤销、可序列化、可用于网络同步。
- 缺点：引入额外抽象层，命令对象的类型擦除和内存管理更复杂。

### 决策分析

默认推荐**方案 B（延迟事件队列）**作为基础机制，**方案 C（命令模式）**用于需要网络回滚或调试回放的场景。方案 A 的递归锁在单线程 ECS 中没有价值，只是把局部问题变成全局问题。

对于输入事件（如跳跃），延迟一整个 Phase 是不可接受的。解决方案不是回到即时回调，而是**在 Input Phase 尽早采集输入、生成 `InputAction` 事件，在后续 Phase 中消费**。输入延迟被控制在 1~2 个 Phase 之间，对于 60fps 游戏约为 16ms，远低于人类感知阈值。

### 状态变化图：延迟事件队列的时序

```
Frame N, Update Phase
│
├─► AISystem.tick()
│   └─► bus.publish(EnemyDiedEvent{e=42})  →  写入 pending_events
│
├─► MovementSystem.tick()
│   （看不到 EnemyDiedEvent，因为还没 dispatch）
│
├─► CombatSystem.tick()
│   └─► bus.publish(DamageEvent{e=42, dmg=10})  →  写入 pending_events
│
└─► Update Phase 结束，Scheduler 调用 bus.dispatch()
    ├─► LootSystem 收到 EnemyDiedEvent → 生成掉落物
    ├─► QuestSystem 收到 EnemyDiedEvent → 击杀数 +1
    └─► HealthSystem 收到 DamageEvent → HP -10
```

> **精确性补充**：注意这里的**内存布局**。`pending_events` 是一个 `std::vector<TEvent>`，同一类型的事件在内存上是连续的。但如果有 10 种不同类型的事件，就会有 10 个不同的 `EventBus<T>` 和 10 个不同的 `pending_events` 数组。它们分散在内存各处，不存在一个"所有事件的统一连续数组"。如果需要更紧凑的布局，可以考虑 SOA（Structure of Arrays）的事件存储，但这会牺牲类型安全和代码简洁性。

## 问题 3：如何让 AI 和 System 都能"看见"并批量消费事件？

### 场景绑定

你的 AI Agent 需要通过 MCP 接口查询"当前帧有哪些 JumpEvent"，以便理解玩家行为并做出响应。但回调式 EventBus 的事件流隐藏在函数调用栈中，AI 无法在不执行一遍游戏逻辑的情况下"读取"事件历史。同时，你的 `QuestSystem` 需要在本帧处理所有 `EnemyDiedEvent` 并统一更新击杀计数——如果每个事件都触发一次回调，计数逻辑会被打散到多次函数调用中，难以优化（如批量数据库写入）。

### 根因分析

回调式通信的**控制流是隐式的**：当 `publish` 被调用时，程序计数器跳转到未知的 subscriber 函数中，执行完再跳回来。这种"隐式跳转"对 AI 极不友好——AI 无法通过静态分析或运行时查询知道"当前有哪些事件待处理"。同时，回调也破坏了**数据局部性**：每个 subscriber 可能访问完全不同的内存区域，导致缓存未命中。

### 分支方案

**方案 A：保留纯回调**
继续用 `EventBus<T>::subscribe(callback)`，AI 通过注入特殊 subscriber 来"监听"事件。
- 优点：实现简单，对现有代码侵入小。
- 缺点：AI 观测是"被动推送"而非"主动查询"，无法控制采样频率；批量处理困难。

**方案 B：事件即实体（Event as Entity）**
在 ECS 中，**事件可以是一种特殊的实体**：
```cpp
struct EventJump {
    Entity trigger_entity;
    float  jump_force;
};

struct EventEnemyDied {
    Entity victim;
    Entity killer;
};

// 创建一个只存活一帧的事件实体
Entity evt = world.create_entity();
world.add_component<EventJump>(evt, {player_entity, 10.0f});
// 不需要额外注册，System 直接 Query<EventJump>
```

事件清理 System 在帧末统一销毁：
```cpp
void EventCleanupSystem::tick(float dt) {
    for (Entity e : query<EventJump>())   world.destroy_entity(e);
    for (Entity e : query<EventEnemyDied>()) world.destroy_entity(e);
}
```
- 优点：事件完全白盒，AI 可通过 Query 读取；利用 ECS 的批量迭代优化；生命周期由 World 管理，无泄漏风险。
- 缺点：实体创建/销毁有开销，高频事件（如碰撞）可能拖垮 ECS。

**方案 C：混合方案——回调队列 + 事件实体**
高频事件（如 `EventCollision`）走延迟回调队列，不走实体创建；中低频事件（如 `EventJump`、`EnemyDiedEvent`）走事件实体路线。
- 优点：兼顾性能和可观测性。
- 缺点：同一引擎中维护两套事件语义，需要清晰的约定区分。

### 决策分析

默认推荐**方案 C（混合方案）**。方案 B 的"所有事件都走实体"在工业级引擎中是不现实的——Bevy 的 `Event<T>` 本质上就是连续数组而非实体，正是为了避免高频事件的实体开销。方案 C 的划分标准是：**如果事件频率 > 100 次/帧，或不需要 AI 观测，走延迟队列；否则走事件实体**。

### 引擎对照

| 引擎 | 事件消费方式 | AI 可观测性 | 高频事件优化 |
|------|-------------|------------|-------------|
| **chaos** | 纯回调 | ❌ 黑盒 | ❌ 无特殊优化 |
| **UE** | `AActor::ReceiveTick` + GameplayEvent 委托 | ⚠️ 需通过反射遍历 Actor | ⚠️ GameplayAbility 有批量处理，但底层仍是 UObject 消息 |
| **Bevy** | `EventReader<T>` 读取 `Events<T>` 资源 | ✅ 事件存储在资源中，可外部读取 | ✅ `Events<T>` 是连续数组，无实体开销 |

> **代价诚实**：
> - "事件即实体" 消除了什么成本？消除了回调的隐式控制流，让事件的消费变得可追踪、可批量处理。
> - 引入了什么新成本？**实体创建/销毁的开销**。每帧可能创建数十个事件实体，如果 ECS 的实体分配器不够快，这会成为瓶颈。
> - **优化策略**：对于高频事件（如 `EventCollision`，每帧数百次），保留传统的回调式或数组式事件队列，不走实体创建路径。只有中低频、需要 AI 观测的事件才走"事件实体"路线。

### AI 可观测性

由于事件是带有组件的实体，[[反射系统]] 的 `TypeRegistry` 可以直接导出它们的 Schema。AI 可以通过 MCP 接口查询：
```json
{
  "events_this_frame": [
    { "type": "EventJump", "trigger_entity": 5, "jump_force": 10.0 }
  ]
}
```

## 问题 4：如何从原始输入走到可跨平台的 Gameplay 事件？

### 场景绑定

你的游戏最初在 PC 上用键盘开发：`SDLK_SPACE` 触发跳跃。现在要移植到主机，跳跃映射到手柄的 A 键；再移植到移动端，跳跃变成屏幕上的虚拟按钮触摸。如果 `MovementSystem` 里写死了 `if (key == SDLK_SPACE) jump()`，每换一个平台就要改一遍 gameplay 代码。

### 根因分析

硬件输入（keycode、button index、touch coordinate）是**平台相关的瞬态信号**，而 gameplay 动作（Jump、Attack、Move）是**平台无关的语义意图**。直接让 gameplay System 读取硬件输入，等于把平台抽象层和业务逻辑层焊死在一起。

### 分支方案

**方案 A：Gameplay System 直接检查硬件输入**
```cpp
// 在 MovementSystem::tick() 中
if (keyboard.is_pressed(SDLK_SPACE)) { jump(); }
```
- 优点：零抽象，零开销。
- 缺点：平台迁移时代码大面积重写；无法支持按键重映射（玩家自定义键位）；同一个物理按键在不同场景语义不同（空格在对话中是"下一句"，在战斗中是"跳跃"）。

**方案 B：InputAction 映射层**
```cpp
struct InputAction {
    std::string name;      // "Jump", "Attack", "Move"
    float       value;     // 0.0 ~ 1.0（用于模拟摇杆）
    bool        is_pressed;
    bool        is_released;
};
```
`InputSystem` 每帧读取硬件状态，通过映射表生成 `InputAction` 事件。`MovementSystem` 订阅的是 `InputAction("Jump")`，而不是 `SDLK_SPACE`。
- 优点：平台无关，支持按键重映射，同一按键在不同上下文语义不同。
- 缺点：引入一层间接查询，对极简 demo 来说略重。

**方案 C：Enhanced Input（上下文敏感 + 修饰器）**
在方案 B 基础上增加：
- **Input Context**：不同游戏状态（菜单、战斗、驾驶）启用不同的映射表。
- **Modifiers**：摇杆输入经过死区、灵敏度曲线、轴向反转处理后再生成 Action。
- **Triggers**：Action 不只在"按下"时触发，还支持"按住 0.5 秒"、"双击"、"组合键"等复杂条件。
- 优点：工业级输入体验，复用 UE 成熟设计。
- 缺点：实现复杂度高，需要配置系统和编辑器支持。

### 决策分析

默认推荐**方案 B（InputAction 映射层）**作为基础实现，**方案 C（Enhanced Input）**作为未来里程碑。方案 A 的直接硬件读取在任何长期项目中都是技术债务——即使你现在只支持一个平台，也应保留映射层的扩展点。方案 B 和 C 的额外开销主要在配置解析和一层字符串查找上，对于 60fps 游戏可以忽略不计。

### 引擎对照

| 引擎 | 输入抽象层 | 特点 |
|------|-----------|------|
| **chaos** | 无统一抽象，各模块直接读 SDL 输入 | 平台绑定严重 |
| **UE** | **Enhanced Input**（UE5 取代 Legacy Input） | Action Mapping + Input Context + Modifiers + Triggers，工业标杆 |
| **Bevy** | `Input<KeyCode>` / `Input<GamepadButton>` + `InputMap` 插件 | ECS-native，但 InputMap 是社区插件，官方支持较基础 |

## 问题 5：事件激增后，如何防止事件泛滥和误消费？

### 场景绑定

你的 RPG 游戏有 100 个敌人在同屏战斗，每帧产生大量 `DamageEvent`、`CollisionEvent`、`EnemyDiedEvent`。同时，你的 AI Agent 正在通过 MCP 向引擎发送 `EventMCPCommand` 来批量修改关卡数据。突然，一个 bug 导致 `DamageEvent` 的发布频率从每帧 10 次暴涨到每帧 10000 次，事件队列撑爆了内存。更糟糕的是，Agent A 发送的 `DestroyEntity` 命令被 Agent B 的 `RestoreSystem` 误消费，导致数据不一致。

### 根因分析

事件总线如果没有**路由**和**过滤**机制，就是一个不加区分的广播器。当系统规模扩大时， broadcast 的复杂度是 O(N×M)（N 个事件 × M 个 subscriber），任何事件都可能触发任何 listener。对于多 Agent 场景，这种无差别广播是致命的——Agent 的操作必须具备**边界隔离**能力。

### 分支方案

**方案 A：全局广播，信任所有消费者**
所有事件对所有 System 可见，靠"约定"避免误消费。
- 优点：实现最简单。
- 缺点：不可扩展，调试困难，多 Agent 场景不可接受。

**方案 B：事件过滤与优先级消费**
给事件实体增加 `EventPriority` 和 `EventConsumed` 标记。System 可以设置 `consumed = true`，下游 System 检查到后跳过处理。
```cpp
struct EventFilter {
    int  priority = 0;
    bool consumed = false;
};
```
- 优点：实现轻量，适合单 Agent 场景的状态拦截（如 Stunned 状态拦截 Jump）。
- 缺点：consumer 顺序仍然重要，consumed 标记是隐式协议。

**方案 C：多 Agent 的事件命名空间 + 白名单路由**
```cpp
struct EventMCPCommand {
    std::string agent_id;
    std::string command;
    json        payload;
};
```
`AgentOrchestratorSystem` 根据 `agent_id` 将事件路由到对应的白名单世界或 staging buffer，防止 Agent A 的事件被 Agent B 的 System 误消费。
- 优点：严格的 Agent 隔离，支持沙箱和权限控制。
- 缺点：需要额外的路由表和权限系统。

### 决策分析

默认推荐**方案 B 用于单 Agent 游戏逻辑，方案 C 用于多 Agent 编排**。两者不是互斥的：`EventConsumed` 标记解决的是**同一条事件在同一命名空间内的消费顺序**问题，`agent_id` 解决的是**不同命名空间之间的隔离**问题。对于单玩家游戏，只需要方案 B；对于 AI 辅助开发或多人协作编辑，必须引入方案 C。

### 引擎对照

| 引擎 | 事件过滤机制 | 多 Agent / 多用户隔离 |
|------|-------------|---------------------|
| **chaos** | 无 | 无 |
| **UE** | **GameplayTag** 过滤 + Ability Task 优先级 | 多人游戏通过 `Role` / `RemoteRole` 和 Net Ownership 隔离，Editor 通过 PIE 实例隔离 |
| **Bevy** | 无内置过滤，靠 System 条件编译（`run_if`） | 无内置多用户沙箱，需自行实现 |

## 从源码到 ECS 的重构映射

chaos 引擎使用 `MessageBus`（Pub-Sub 回调）和 `DataBus`（双缓冲跨线程数据交换）两套并行方案。迁移到 ECS 时，可以按以下方式重构：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `MessageBus::publish<TMessage>(msg)` | **事件实体 + `EventT` 组件** 或 **延迟事件队列** | 对于需要 AI 观测的事件，走"事件实体"路线；对于高频内部通知，保留延迟队列。 |
| `MessageBus::subscribe<TMessage>(callback)` | **`Query<EventT>`** | System 不再注册回调，而是在 tick 中直接查询当前帧的事件组件。 |
| `ThreadSafeMessage` 分级 | **事件实体的 Phase 绑定** | 不需要 recursive_mutex。事件在哪个 Phase 产生、在哪个 Phase 被消费，由调度器保证。 |
| `DataBus`（Logic ↔ Present 双缓冲）| **渲染命令列表（Render Command List）** | 将跨线程数据交换显式化为渲染 System 的输出：逻辑线程生成 `DrawCommand` 列表，渲染线程读取并执行。 |
| `ScopedSubscriptionToken`（RAII 取消订阅）| **事件实体的自动生命周期** | 不需要手动 unsubscribe。事件实体在帧末被 `EventCleanupSystem` 统一销毁，不存在野回调。 |

> **Why ECS 更适合这个模块？**
> - **可观测性**：回调式 EventBus 的事件流是隐藏在函数调用栈中的黑盒。ECS 的事件实体让 AI 可以通过 Query 直接读取事件列表，完全白盒。
> - **确定性**：回调的执行顺序取决于订阅的注册顺序，难以保证跨会话一致。ECS 中事件由调度器和 Query 机制统一消费，顺序完全确定。
> - **无泄漏风险**：传统回调容易因忘记 unsubscribe 导致内存泄漏或野指针。ECS 中事件是实体，生命周期由 World 管理，不存在悬挂的订阅者。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ 事件实体是 World 中的普通实体，所有事件状态都在组件中。延迟事件队列是简单的 `std::vector`，无隐藏状态。 |
| **自描述** | ✅ `EventJump`、`EventEnemyDied` 等事件组件可以通过反射注册表导出 Schema。AI 不需要读 C++ 头文件就能理解事件结构。 |
| **确定性** | ✅ 事件在 Phase 安全点统一 dispatch。同一帧内的事件消费顺序由 System 调度顺序唯一确定。 |
| **工具边界** | ✅ MCP 接口可暴露：`publish_event(type_name, json_payload)`、`query_events(type_name)`、`subscribe_to_event(type_name, agent_id)`。 |
| **Agent 安全** | ✅ 多 Agent 场景下，事件可以绑定 `agent_id` 或 `EventSource` 组件。`AgentOrchestratorSystem` 负责过滤和路由，防止误消费。 |

## "如果我要 vibe coding，该偷哪几招？"

1. **先偷延迟队列，再偷事件实体**
   如果你只有 5~10 个 System，一个模板化的 `EventBus<T>` 加延迟队列完全够用。当你的 AI 需要"读取事件"而不是"响应回调"时，再引入事件实体。

2. **输入必须做 Action 映射**
   不要让 `MovementSystem` 直接检查 `SDLK_SPACE`。引入 `InputAction` 层，把硬件输入和 gameplay 逻辑彻底解耦。这是从 demo 走向可发布游戏的分水岭。

3. **高频事件不走实体，低频事件走实体**
   碰撞检测可能每帧产生 500 次碰撞事件，走实体创建会拖垮 ECS。保留数组队列。而"玩家升级""任务完成"这类低频事件非常适合走事件实体，方便 AI 观测和调试。

4. **给事件加时间戳和源实体**
   ```cpp
   struct EventBase {
       uint64_t frame_id;
       Entity   source_entity;
   };
   ```
   这在调试和回放时 invaluable。你可以精确知道"是哪个实体在第 1247 帧触发了这个事件"。

5. **事件 System 放在最后执行**
   确保 `EventCleanupSystem` 是整帧的最后一个 System（或最后一个 Phase 的最后一个 System）。这样所有其他 System 都有机会在本帧消费事件，而不会在下一帧留下垃圾。

## 默认推荐路径总结

> **默认推荐路径**
> - **即时通信层**：类型安全的 `EventBus<T>` 模板，支持延迟清理（swap-and-pop），`tick()` 在 Phase 安全点调用。
> - **延迟调度层**：`publish` 只入队，`dispatch()` 在 Phase 末尾统一触发，保证遍历安全。
> - **ECS 观测层**：中低频事件（< 100 次/帧）走"事件即实体"路线，高频事件走延迟回调队列。
> - **输入抽象层**：`InputAction` 映射表，将硬件输入解耦为平台无关的语义动作。
> - **安全隔离层**：单 Agent 场景用 `EventConsumed` 标记实现优先级消费；多 Agent 场景用 `agent_id` + `AgentOrchestratorSystem` 实现路由隔离。

> **UE 工业级设计吸收**
> - **延迟队列的 Phase 绑定**：从 UE 的 Latent Action 和 Tick Group 中学习，把事件分发绑定到明确的执行阶段，而非任意时刻。
> - **输入映射层**：吸收 UE5 Enhanced Input 的 Action Mapping + Input Context 思想，预留 Modifier 和 Trigger 的扩展接口。
> - **GameplayTag 过滤**：参考 UE 的 GameplayTag 系统，未来给事件实体增加 `GameplayTags` 组件，实现基于标签的细粒度路由和过滤。
> - **弱引用绑定**：学习 UE Delegate 的 `UObject` 弱引用机制，在回调总线中引入 subscriber 的弱引用检查，防止 listener 销毁后的野指针回调。

> **ECS 映射**
> - **组件**：`EventJump`、`EventEnemyDied` 等事件组件；`EventFilter`（优先级/消费标记）；`EventSource`（agent_id / 源实体）。
> - **资源**：`EventBusManager`（全局总线注册表）；`InputActionMap`（输入映射配置）。
> - **系统**：`InputSystem`（硬件读取 → Action 生成）；`EventDispatchSystem`（Phase 末尾 dispatch）；`EventCleanupSystem`（帧末销毁事件实体）；`AgentOrchestratorSystem`（多 Agent 路由）。

> **条件切换路径**
> - 如果项目**完全没有 AI 观测需求**且事件频率极低（< 10 次/帧）：可以暂时只用纯回调式 `EventBus<T>`，不引入事件实体。但一旦 AI 介入，必须迁移。
> - 如果项目**需要网络同步或回放**：延迟事件队列不足以支持命令回滚，必须升级为**命令模式**（Command Pattern），将事件序列化为可撤销、可重放的 `CommandBuffer`。
> - 如果项目**只支持单平台且永不扩展**：InputAction 映射层可以简化，但必须在代码中显式注释"此处假设平台不变，未来迁移时需重构"。

> **扩展路径**
> - **Milestone 1：事件历史与回放**：给事件实体增加 `frame_id` 和 `timestamp`，构建环形缓冲区 `EventHistory`，支持"查看过去 N 帧的事件流"。
> - **Milestone 2：GameplayTag 路由**：引入 `GameplayTags` 组件和 `TagQuery` 系统，让 System 可以按标签订阅事件子集，替代硬编码的 `Query<EventT>`。
> - **Milestone 3：跨世界事件桥接**：实现 `WorldBridge` 系统，支持不同 ECS World 之间的事件转发（如 Editor World ↔ PIE World，或 Server World ↔ Client World）。
> - **Milestone 4：可视化调试面板**：开发 ImGui 事件监视器，实时显示当前帧的所有事件实体、来源、消费状态，支持按类型/源实体过滤。
