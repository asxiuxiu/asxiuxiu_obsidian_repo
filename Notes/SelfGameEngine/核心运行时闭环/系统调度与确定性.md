# 系统调度与确定性

> **前置依赖**：[[组件系统架构]]、[[反射系统]]
> **本模块增量**：深入理解 ECS 引擎的调度设计空间——从"按列表顺序跑 System"到工业级的分阶段并行调度，再到确定性回放与多世界协同。你将能做出每一个关键决策（Phase 模型、依赖声明方式、同步点策略、并行粒度、时间步方案）并知道各自的代价。
> **下一步**：[[事件总线]]，因为调度好的 System 之间需要一种解耦的通信机制，而不必通过直接的读写依赖硬编码所有交互。

---

## 问题 0：为什么 ECS 引擎不能简单地"按列表顺序逐个执行 System"？

最天真的做法是：把所有 System 塞到一个 `std::vector` 里，每帧从头到尾 `for (auto& sys : systems) sys.tick(dt)`。这在 Demo 阶段能跑，但一旦系统超过 5 个，三个噩梦会同时降临。

### 场景绑定：三个会实际爆发的故障

**故障一：顺序不同，结果不同**。`MovementSystem` 把玩家向前推了 1 米，`CollisionSystem` 发现穿墙了又推回来。如果 Collision 先执行、Movement 后执行，玩家就会穿墙。两个 System 的执行顺序不是"性能问题"，是"正确性问题"。

**故障二：遍历中间销毁实体**。`BulletSystem` 正在遍历所有带 `Bullet` 组件的实体，`CollisionSystem` 突然销毁了其中一个（子弹击中敌人）。如果 `BulletSystem` 用的是一个普通数组索引迭代，销毁后数组收缩，迭代器立刻指向错误位置——崩溃或访问已回收内存。

**故障三：联机对战不同步**。主机 A 的 `AISystem` 和 `MovementSystem` 执行顺序与客机 B 不同。10 秒后，两个客户端的世界状态出现微小偏差，偏差像滚雪球一样放大，最终导致"明明没被打中却扣血"的奇观。

### 根因分析：这三个故障共享同一个根

它们都源于 ECS 的一条核心假设被打破了：**System 不应该在彼此执行的过程中干扰对方的"世界观"**。如果 System A 执行时，System B 能直接修改 A 正在遍历的数据结构，那么 A 看到的就是一个"半更新"的中间状态——这不是 A 的 bug，是调度层的失职。

换言之，ECS 把"数据"和"逻辑"拆开了，但如果"逻辑的执行顺序"仍然是一片混沌，拆分就毫无意义。我们需要一个**显式的、可验证的、输入确定则输出确定**的调度层。

> **关键结论**：调度系统不是"优化选项"，是 ECS 架构的**必要基础设施**。没有它，System 数量超过个位数后，引擎的正确性就不可证明了。

## 问题 1：Phase 划分——是硬编码枚举好，还是可扩展标签好？

知道了"必须有序"，下一个问题是：这个"序"应该用什么方式表达？

### 分支 A：不做 Phase，纯依赖图

最极端的做法是：没有 Phase，所有 System 都在一个全局 DAG 里，靠 `before` / `after` 声明决定顺序。

```cpp
// 分支 A：纯依赖图，无 Phase
scheduler.add_system(movement);
scheduler.add_system(collision);
scheduler.add_dependency(collision, movement); // collision after movement
```

**适用场景**：System 数量很少（<10 个）、领域单一（比如只有一个物理模块）。

**隐藏代价**：当 System 数量到 30 个以上时，依赖图变成一张密密麻麻的蜘蛛网。程序员为了表达"这些 System 负责输入处理、那些负责物理"，不得不写大量 `before` / `after` 来维护隐式的阶段语义。

**失效条件**：跨领域 System 增加时，维护依赖图的成本指数级上升。

### 分支 B：硬编码 Phase 枚举

给 System 分配固定的阶段，按阶段顺序执行。UE 的 `ETickingGroup` 是这种思路的工业级实现：

```cpp
// 分支 B：硬编码 Phase 枚举（类似 UE 的 TickGroup）
enum class Phase {
    PrePhysics,     // 物理模拟前：输入、动画评估
    StartPhysics,   // 启动物理（特殊标记，不可降级）
    DuringPhysics,  // 可与物理并行
    EndPhysics,     // 结束物理（特殊标记）
    PostPhysics,    // 物理后：碰撞响应、Transform 传播
    PostUpdateWork, // 所有更新完成后
    LastDemotable,  // 末尾兜底
};
```

每个 System 注册时绑定到一个 Phase。调度器先跑完 `PrePhysics` 里的所有 System，再进入 `StartPhysics`，以此类推。

**适用场景**：大型商业引擎，阶段语义稳定（物理必须在逻辑之后、渲染必须在 Transform 之后）。

**隐藏代价**：
- **扩展困难**：如果你想在 `PrePhysics` 和 `StartPhysics` 之间插入一个"导航网格更新"阶段，必须修改枚举定义并重新编译所有模块。
- **动态降级复杂**：UE 的 `Prerequisites` 机制允许一个 Tick 函数因依赖被"降级"到更晚的 Phase。但如果降级的目标恰好是 `StartPhysics` 或 `EndPhysics`，UE 会阻止这种降级——因为物理阶段是特殊的硬边界。

**失效条件**：需要频繁插入新阶段、或阶段边界在不同游戏类型中差异很大时（如 RTS 不需要物理阶段，但格斗游戏需要极细的输入响应阶段）。

### 分支 C：可扩展的 ScheduleLabel（插件驱动）

Bevy 走了另一条路：`ScheduleLabel` 不是硬编码枚举，而是一个可以被插件动态插入的"时隙"列表。

```rust
// 分支 C：可扩展 ScheduleLabel（Bevy 风格）
// 默认主循环顺序
MainScheduleOrder::default() = [
    First, PreUpdate, RunFixedMainLoop,
    Update, SpawnScene, PostUpdate, Last
];
// StatesPlugin 在运行时插入新阶段
order.insert_after(PreUpdate, StateTransition);
```

**适用场景**：模块化引擎，不同游戏类型需要不同的阶段组合。插件可以插入自己的阶段而不修改核心代码。

**隐藏代价**：
- 阶段的相对顺序在运行时才能确定，调试时比硬编码枚举更难一眼看出"某个 System 到底在第几步执行"。
- 如果两个插件都试图在同一个位置插入阶段，可能产生顺序冲突。

**失效条件**：对阶段顺序有极强的硬性要求（如物理引擎要求某些阶段绝对不能被插入），运行时扩展反而引入不确定性。

### 引擎对照与默认实现推荐

| 维度 | 分支 B（硬编码 Phase） | 分支 C（可扩展 Label） |
|------|----------------------|----------------------|
| 扩展性 | 需改枚举、重新编译 | 插件运行时插入 |
| 可预测性 | 阶段顺序写死在代码里，一眼可知 | 运行时组合，需查配置 |
| 调试友好度 | 高（Phase 名即语义） | 中（需打印实际顺序） |
| 适用规模 | 大型、阶段语义稳定 | 模块化、阶段需求多变 |

> **参考：工业界怎么选的？**
> - UE 选了分支 B 的变体：9 个硬编码 TickGroup，加上 `Prerequisites` 实现动态降级。它的阶段语义是二十年演化的结果，非常稳定。
> - Bevy 选了分支 C：`MainScheduleOrder` 是一个普通的 `Resource`，插件可以在运行时修改它。`StatesPlugin` 插入 `StateTransition`、`RenderPlugin` 通过 SubApp 外挂渲染阶段。
> - 一个半 ECS 引擎的参考实现也用了硬编码 Phase（9 个固定阶段），但所有 System 串行执行，并行标记为"TODO"——这说明 Phase 模型与并行策略是**正交**的，可以独立选择。

**默认推荐**：**分支 C 的简化版**。定义一个 `Phase` 枚举作为默认顺序（`PreUpdate` → `Update` → `PostUpdate` → `Render`），但允许插件通过 `insert_before` / `insert_after` 动态扩展。这样既保留了"4 个默认阶段足够支撑 Demo"的简单性，又为未来扩展留了口子。

**明确条件**：如果项目类型确定（如纯 2D 平台跳跃，不需要物理），可以将阶段裁剪到 2~3 个，降低心智负担。

## 问题 2：依赖声明——让程序员手动写，还是让调度器自动推导？

Phase 解决的是"粗粒度阶段顺序"，但同一 Phase 内的 System 谁先谁后？这需要依赖声明机制。

### 分支 A：纯手动显式依赖

程序员为每对 System 手动声明 `before` / `after`：

```cpp
// 分支 A：纯手动依赖
scheduler.add_system(input_system);
scheduler.add_system(ai_system);
scheduler.add_system(movement_system);
scheduler.add_dependency(ai_system, input_system);     // ai after input
scheduler.add_dependency(movement_system, ai_system);  // movement after ai
```

**适用场景**：System 数量少、依赖关系简单直观。

**隐藏代价**：当 System 数量达到 20+ 时，手动维护依赖关系变成一场灾难。遗漏一条依赖不会导致编译错误——它会导致**运行时静默的时序 bug**，极难调试。

**失效条件**：System 数量超过个位数，或多人协作开发时。

### 分支 B：自动推导 + 手动覆盖

让每个 System 显式声明自己读/写哪些组件，调度器自动推导依赖图：

```cpp
// 分支 B：自动推导依赖
struct SystemDesc {
    std::string name;
    std::vector<ComponentTypeID> reads;
    std::vector<ComponentTypeID> writes;
    std::function<void(float)> tick;
};

// 调度器自动推导：若 A 写 Position，B 读 Position，则 B 隐式依赖 A
```

**适用场景**：ECS 架构下的中等规模项目。自动推导消除了"遗漏依赖"的沉默错误。

**隐藏代价**：
- **过度串行化**：如果 System A 写 `Position.x`，System B 读 `Position.y`，自动推导会认为两者冲突（因为都是 `Position` 组件），强制串行。但现实中它们可以并行。
- **无法表达非数据依赖**：例如 `AISystem` 必须在 `MovementSystem` 之前执行，不是因为它们读写相同组件，而是因为 AI 决定"往哪走"、Movement 决定"实际移动"。这种**语义依赖**无法从组件读写中推导。

**失效条件**：当大量 System 存在细粒度组件访问（如只读写组件的某个字段）或语义依赖时。

### 分支 C：混合策略——自动推导 + 显式 before/after + 动态降级

Bevy 和 UE 都采用了混合策略，但实现方式不同：

**Bevy 的混合策略**：
- 每个 System 通过 `SystemParam` 的 `FilteredAccessSet` 自动申报数据访问（读/写哪些组件、哪些 Resource）。
- 开发者通过 `.before()` / `.after()` 声明显式语义依赖。
- 调度器在构建期将两者合并为一张系统级 DAG，并通过**歧义检测**找出"数据冲突但未排序"的系统对，向开发者报警。

```rust
// Bevy 风格：自动冲突检测 + 显式顺序
app.add_systems(Update, (
    input_system,
    (ai_system, movement_system)
        .after(input_system)
        .before(collision_system),
));
// 如果 ai_system 和 movement_system 都写 Position，
// 但开发者没有为它们指定顺序，Bevy 会报 Ambiguity Warning
```

**UE 的混合策略**：
- `TickGroup` 提供粗粒度顺序（类似 Phase）。
- `FTickFunction::Prerequisites` 提供显式前置依赖。
- `FTickTaskManager` 在帧开始时解析依赖图，动态计算 `ActualStartTickGroup`。如果一个 Tick 的前置在更晚的组，它会被**降级**（demote）到更晚的组执行——但不会被降入 `StartPhysics` / `EndPhysics` 这两个特殊组。

```cpp
// UE 风格：TickGroup + 动态降级
actor.PrimaryActorTick.TickGroup = TG_PrePhysics;
actor.PrimaryActorTick.AddPrerequisite(other_actor, other_actor.PrimaryActorTick);
// 如果 other_actor 在 TG_PostPhysics，则本 actor 的实际开始组会被降级
```

**适用场景**：工业级引擎。自动推导减少人为错误，显式依赖表达语义，动态降级提供灵活性。

**隐藏代价**：
- 构建期分析更复杂（需要环检测、传递约简、歧义检测）。
- 调试时"为什么这个 System 被推迟了"需要理解调度器的推导逻辑。

**失效条件**：对调度开销极度敏感的场景（如嵌入式平台），构建期分析可能不可接受。

### 决策分析与推荐

> **默认推荐**：**分支 C 的简化版**。
> 1. 每个 System 注册时声明 `reads` / `writes` 组件列表，调度器**自动推导隐式依赖**。
> 2. 对于语义依赖（如 AI → Movement），开发者显式写 `.after()` / `.before()`。
> 3. **自动环检测**：构建 DAG 时用 Kahn 算法检测循环依赖，发现环时立即崩溃并打印环路径（而不是静默失败）。
> 4. **可选的歧义检测**：开发模式下，如果两个 System 写同一个组件但没有显式排序，打印警告（生产模式关闭，避免噪音）。

为什么不用纯自动推导（分支 B）？因为语义依赖无法被推导。为什么不用纯手动（分支 A）？因为人为遗漏的代价太高。

> **参考：三引擎的依赖策略对比**
> - 一个半 ECS 引擎：Manager 级别手动声明 `addDependency<Dependent, Dependency>()`，在 Phase 内按 Kahn 拓扑排序。缺点是 Manager 同时是 System + ComponentManager 的混合体，依赖粒度粗。
> - UE：`TickGroup` + `Prerequisites` + 动态降级。粗粒度阶段保证基本顺序，细粒度依赖处理复杂交互。
> - Bevy：`before` / `after` + `FilteredAccessSet` 自动推导 + 歧义检测。开发者写高层语义，底层冲突由调度器静态分析。

## 问题 3：命令缓冲与同步点——何时应用结构变更？

ECS 中 System 遍历组件时，如果另一个 System 同时创建/销毁实体，迭代器会失效。命令缓冲（Command Buffer）是解决这个问题的经典方案：System 不直接修改世界结构，而是把"创建实体"、"销毁实体"、"添加组件"等请求写入缓冲区，在**安全点**统一应用。

但安全点选在哪里？这是问题 3 的核心矛盾。

### 分支 A：Phase 级别手动安全点

每个 Phase 开始前统一应用上一个 Phase 积累的命令：

```cpp
// 分支 A：Phase 级别安全点
for (Phase p : {PreUpdate, Update, PostUpdate, Render}) {
    command_buffers[p].apply();  // 应用上一阶段的所有命令
    scheduler.run_phase(p, dt);
    // System 执行中产生的命令进入 command_buffers[p]
}
```

**适用场景**：Phase 数量少、命令产生者和消费者之间的时序要求简单。

**隐藏代价**：
- **延迟问题**：如果 System A 在 `Update` Phase 创建了实体，System B（也在 `Update` Phase，排在 A 后面）看不到这个新实体——因为它要等到 `PostUpdate` 的安全点才被创建。
- 程序员为了"让 B 看到 A 的创建"，要么把 B 移到 `PostUpdate`，要么放弃命令缓冲直接修改——两者都破坏架构。

**失效条件**：同一 Phase 内存在"创建实体 → 立即使用"的链式需求时。

### 分支 B：System 级别同步点（自动插入 ApplyDeferred）

Bevy 的做法更细：如果 System A 使用了 `Commands`（延迟参数），而 System B 显式依赖 A（`B.after(A)`），Bevy 的构建器会自动在 A 和 B 之间插入一个 `ApplyDeferred` 同步点。

```rust
// 分支 B：自动同步点（Bevy 的 distance 算法）
app.add_systems(Update, (
    spawn_enemies,      // 使用 Commands 生成敌人
    track_enemies.after(spawn_enemies),  // 需要看到新生成的敌人
));
// Bevy 自动在两者之间插入 ApplyDeferred
```

核心算法是**基于 distance 的同步点插入**：
1. 对每个节点计算 `distance`——从起点到该节点路径上经过的同步点数量。
2. 如果一个系统使用了 `Commands`，它会将出边目标的 `distance` 增加 1（意味着需要一个同步点）。
3. 遍历所有边：如果边的两端 `distance` 不同，且目标不是已有的 `ApplyDeferred`，就插入同步点。

**适用场景**：需要细粒度同步点、同一 Phase 内存在复杂的命令依赖链。

**隐藏代价**：
- **构建开销**：每次修改 Schedule 都需要重新运行 distance 算法。
- **隐式魔法**：开发者可能不理解"为什么这里自动插入了一个同步点"，导致对执行顺序的误判。
- **过度同步**：如果 A 和 B 之间插入了 ApplyDeferred，但 B 其实不依赖 A 的结构变更（只是语义依赖），这个同步点就是不必要开销。

**失效条件**：Schedule 频繁修改、或开发者需要精确控制同步点位置时。

### 分支 C：混合策略——默认 Phase 安全点 + 显式局部同步

默认实现的推荐路径：默认按 Phase 级别安全点（简单、可预测），同时允许开发者显式插入局部同步点处理特殊需求。

```cpp
// 分支 C：混合策略
for (Phase p : phase_order) {
    // 1. 应用全局命令缓冲（默认安全点）
    global_cmd.apply();
    
    // 2. 运行 Phase 内 System
    for (auto& sys : systems_in_phase[p]) {
        sys.tick(dt, local_cmd);  // System 写入线程本地命令队列
        
        // 3. 如果下一个 System 显式依赖当前 System，
        //    且当前 System 产生了结构变更，局部 flush
        if (next_sys.depends_on(sys) && local_cmd.has_structural_changes()) {
            local_cmd.apply();
        }
    }
    
    // 4. 合并线程本地命令到全局缓冲
    merge_thread_local_cmds(global_cmd);
}
```

**适用场景**：默认实现，需要简单默认值 + 精细控制的能力。

### 决策分析与推荐

> **默认推荐**：**分支 C（混合策略）**。
> - 默认行为：Phase 边界是安全点。90% 的场景下这足够用，且心智负担最低。
> - 逃生通道：提供 `apply_deferred()` 显式系统，开发者可以在 Phase 内手动插入同步点。
> - 从 Bevy 学到的：如果自动插入同步点，必须提供 `before_ignore_deferred` / `after_ignore_deferred` 等显式关闭选项，否则"魔法"会变成调试噩梦。

为什么不是纯自动（分支 B）？因为默认实现初期 System 数量少，Phase 安全点足够，自动算法的构建开销和心智负担不划算。为什么不是纯手动（分支 A 的严格版）？因为同一 Phase 内的链式创建需求确实会出现，需要局部逃生通道。

> **精确性补充**：注意命令缓冲的**线程安全**。多线程 System 可能同时向命令缓冲写入。如果命令缓冲是全局共享的，需要锁——这会抵消并行收益。正确做法是：每个工作线程维护**线程本地命令队列**，在 Phase 安全点统一合并。

```cpp
// 线程安全的多线程命令缓冲
thread_local CommandBuffer local_cmds;

void System::tick(float dt) {
    local_cmds.push([&](World& w) { w.destroy_entity(e); });
}

void apply_commands() {
    for (auto& buf : all_thread_local_buffers) {
        for (auto& cmd : buf) cmd(world);
    }
}
```

## 问题 4：并行调度——System 级并行的冲突检测该怎么做？

当 System 数量到十几个、实体数量到上万个时，串行执行会让单核 CPU 爆满。但多线程 ECS 不能简单地把 System 丢进线程池——两个 System 同时写同一个组件会导致数据竞争。

### 分支 A：Chunk 级并行（数据并行）

不并行 System，而是并行**单个 System 内部的处理**：

```cpp
// 分支 A：Chunk 级并行
void MovementSystem::tick(float dt) {
    // 将所有 Chunk 拆分为批次，交给线程池
    parallel_for(chunks, [&](Chunk* chunk) {
        for (Entity e : chunk->entities_with<Position, Velocity>()) {
            auto [pos, vel] = chunk->get<Position, Velocity>(e);
            pos.value += vel.value * dt;
        }
    });
}
```

**适用场景**：单个 System 处理量大（如粒子更新、批量 Transform 计算），且处理逻辑可以按 Chunk 完美划分。

**隐藏代价**：
- **无法并行不同 System**：如果 `AISystem` 和 `MovementSystem` 各自只处理 1000 个实体，但加起来占 2ms，Chunk 级并行无法把它们分布到两个核心上。
- **Chunk 之间如果有依赖**（如 Transform 父子层级需要按 rank 顺序更新），Chunk 级并行需要额外的同步机制。

**失效条件**：瓶颈来自"System 数量多但每个处理量小"，而非"单个 System 处理量大"。

### 分支 B：System 级并行 + 静态冲突检测（Wavefront）

调度器在构建期分析所有 System 的 `reads` / `writes`，生成一张**冲突图**。运行时按波次（wavefront）执行：同一波次内的 System 没有冲突，可以并行；波次之间串行。

```cpp
// 分支 B：Wavefront 并行调度
// 构建期预计算冲突图
for (size_t i = 0; i < systems.size(); ++i) {
    for (size_t j = i + 1; j < systems.size(); ++j) {
        if (conflicts(systems[i].access, systems[j].access)) {
            conflict_graph[i].set(j);
            conflict_graph[j].set(i);
        }
    }
}

// 运行时：拓扑排序后分组为 wavefront
// Wave 0: InputSystem, SpawnSystem   (互不冲突)
// Wave 1: AISystem, MovementSystem   (读 Input，互不写相同组件)
// Wave 2: CollisionSystem            (写 Position，必须等 Wave 1)
```

**适用场景**：System 数量中等（10~50 个），依赖链不是特别深。

**隐藏代价**：
- **波次尾延迟**：如果 Wave 1 有 5 个 System，其中 4 个 0.1ms 完成，1 个 1ms 完成，那么 Wave 2 必须等这 1ms 的 System——4 个核心空转 0.9ms。
- **构建开销**：每次增删 System 都需要重新分析冲突图。

**失效条件**：依赖链很深（几乎无法形成宽波次）、或 System 执行时间差异极大时。

### 分支 C：System 级并行 + 位图运行时检测（Bevy 方案）

Bevy 的多线程执行器不做静态波次分组，而是**运行时动态调度**：

1. 构建期预计算每个 System 的 `conflicting_systems` 位图（`FixedBitSet`）。
2. 运行时维护 `running_systems` 位图。
3. 一个 System 能否启动 = 检查它的冲突位图是否与 `running_systems` 有交集。如果没有交集，立即启动。

```rust
// 分支 C：位图运行时检测（Bevy 核心思想，改写为 C++ 风格）
bool can_run(size_t sys_idx) const {
    const auto& meta = system_metadata[sys_idx];
    if (meta.is_exclusive && num_running > 0) return false;
    // 检查冲突：conflicting_systems 与 running_systems 的交集是否为空
    return meta.conflicting_systems.is_disjoint(running_systems);
}

// 当一个 System 完成时，唤醒下游
void on_system_complete(size_t sys_idx) {
    running_systems.clear(sys_idx);
    for (size_t dependent : metadata[sys_idx].dependents) {
        if (--remaining_deps[dependent] == 0) {
            ready_queue.push(dependent);
        }
    }
}
```

**适用场景**：System 数量多、依赖关系复杂、需要最大化 CPU 利用率。

**隐藏代价**：
- **调度器本身的竞争**：主线程需要频繁获取锁来更新 `running_systems` 和 `ready_queue`。如果 System 执行时间很短（<0.1ms），调度开销会超过并行收益。
- **位图大小**：如果有 1000 个 System，`FixedBitSet` 需要 1000 bit（125 字节），`is_disjoint` 是 125 字节的逐位比较——对小规模 System 很快，但对超大规模需要分层优化。

**失效条件**：System 数量极少（<5 个）或执行时间极短（每次 <0.05ms）。

### 引擎对照与推荐

> **参考：三引擎的并行策略**
> - 一个半 ECS 引擎：`ExecutionFlowType::Parallel` 标记为 TODO，当前全部串行。Manager 之间的依赖在 Phase 级别通过 Kahn 拓扑排序解决，但 Manager 内部（如 `TransformManager` 的按 rank 更新）可以按 Chunk 并行——这是**分支 A** 的局部应用。
> - UE：`bRunOnAnyThread` 允许非关键 Tick 在任务系统上并行执行。但 UE 的并行粒度是 **Actor/Component 级**（多个 Actor 的 Tick 并行），而非 System 级。这是因为 UE 不是纯 ECS，Tick 函数附着在 UObject 上。
> - Bevy：纯 ECS 的 **System 级并行**，基于 `FilteredAccessSet` 的位图冲突检测。这是最贴近 ECS 哲学的方案——System 是无状态的 pure function，只要数据访问不冲突就可以并行。

> **默认推荐**：**从分支 A 开始，逐步升级到分支 B**。
> - **阶段 1（Demo 期）**：只实现 Chunk 级并行。大部分 CPU 密集型 System（Transform 更新、粒子、动画评估）都能从中受益，且实现简单。
> - **阶段 2（System 数量 >10）**：引入分支 B 的 Wavefront 调度。构建期分析冲突图，按波次并行。波次尾延迟问题在 System 数量不多时不明显。
> - **阶段 3（工业级）**：如果 Wavefront 的尾延迟成为瓶颈，再考虑分支 C 的位图运行时调度。Bevy 的方案更优，但实现复杂度显著更高——对于默认实现，先让 Wavefront 跑起来，比追求完美调度器更重要。

**性能诚实**：
- "并行 System"消除了什么成本？消除了串行执行的 CPU 瓶颈。
- 引入了什么新成本？任务提交、同步、冲突检测的开销。如果每个 System 只处理 100 个实体，拆成多线程的任务提交和同步开销可能比串行更慢。
- **反例**：System 处理量很小（<1ms）或依赖链很长（几乎无法并行）时，强行并行化只会增加代码复杂度而不提升性能。

## 问题 5：固定时间步 vs 可变时间步——如何兼顾渲染流畅性与逻辑确定性？

渲染帧率随硬件负载波动（30~144fps 不等），但物理模拟和 AI 逻辑需要稳定的时间间隔才能保持确定性。这是游戏引擎最经典的矛盾之一。

### 分支 A：纯可变时间步

直接用上一帧的真实耗时作为 `dt`：

```cpp
// 分支 A：纯可变时间步
auto now = clock::now();
float dt = duration<float>(now - last_time).count();
last_time = now;
world.tick(dt);  // dt 每帧不同
```

**适用场景**：单机休闲游戏、对确定性没有要求的项目。

**隐藏代价**：
- **物理不稳定**：大 `dt`（如从 60fps 掉到 30fps，dt 从 16ms 变成 33ms）会导致物理模拟爆炸（物体穿透、关节断裂）。
- **非确定性**：同一台机器两次运行，帧率波动不同，导致 `dt` 序列不同，最终状态 diverge。
- **浮点累积误差**：不同 `dt` 的浮点运算顺序不同，累积误差路径不同。

**失效条件**：任何需要网络同步、物理稳定、或确定性回放的项目。

### 分支 B：纯固定时间步

逻辑以固定频率更新（如 60Hz），渲染独立插值：

```cpp
// 分支 B：纯固定时间步
float fixed_dt = 1.0f / 60.0f;
accumulator += real_dt;

while (accumulator >= fixed_dt) {
    world.tick(fixed_dt);  // 逻辑更新
    accumulator -= fixed_dt;
}

// 渲染用插值：在上一帧状态和当前状态之间插值
float alpha = accumulator / fixed_dt;
render_interpolated(previous_state, current_state, alpha);
```

**适用场景**：物理模拟、网络同步、确定性回放。

**隐藏代价**：
- **渲染延迟**：渲染的是插值后的状态，比逻辑状态最多延迟一个 `fixed_dt`（约 16ms）。对于极高响应要求的游戏（如格斗、音游），这 16ms 可能可感知。
- **螺旋死亡**：如果逻辑更新耗时超过 `fixed_dt`，`accumulator` 永远无法归零，导致"一帧内跑多次逻辑 → 更慢 → 跑更多次"的死亡螺旋。

**失效条件**：逻辑更新不稳定、或需要极低输入延迟的场景。

### 分支 C：双轨制——可变渲染 + 固定逻辑（Bevy 方案）

Bevy 采用了最清晰的双轨分离：`Update`（可变步长）和 `FixedUpdate`（固定步长）是两个独立的 Schedule。

```rust
// 分支 C：双轨制（Bevy 的 RunFixedMainLoop）
// 主循环（可变步长）：First → PreUpdate → RunFixedMainLoop → Update → PostUpdate → Last
// RunFixedMainLoop 内部：根据累积时间决定调用 FixedMain 0~N 次
// FixedMain 内部（固定步长）：FixedFirst → FixedPreUpdate → FixedUpdate → FixedPostUpdate → FixedLast
```

**关键设计**：
- `RunFixedMainLoop` 是一个特殊的 Schedule，它根据 `Time<Virtual>` 的累积量决定一帧内调用多少次 `FixedMain`。
- 物理、AI、网络逻辑挂在 `FixedUpdate`；渲染准备、相机跟随、UI 更新挂在 `Update`。
- 相机跟随物理的问题通过"跨越边界的数据依赖"解决：`RunFixedMainLoopSystems::AfterFixedMainLoop` 在固定更新后执行插值。

**适用场景**：需要同时满足"渲染流畅"和"逻辑确定"的项目。

**隐藏代价**：
- **复杂度**：开发者必须理解"我的 System 应该挂在 Update 还是 FixedUpdate"。挂错会导致抖动（物理在 Update 跑）或延迟（渲染在 FixedUpdate 跑）。
- **状态双份**：如果需要插值，必须保存上一帧 FixedUpdate 的状态和当前状态。

**失效条件**：极简项目（不需要物理、不需要网络），双轨制的复杂度不值得。

### 引擎对照与推荐

> **参考：三引擎的时间步策略**
> - UE：编辑器中支持 `PIEFixedTickSeconds`（固定步长一帧多 Tick），但运行时默认是可变步长。物理通过 Sub-Stepping 在可变帧内多次更新来逼近固定步长效果。
> - Bevy：显式双轨。`Update` 和 `FixedUpdate` 是两个完全独立的 Schedule，由 `RunFixedMainLoop` 作为"变速齿轮"桥接。
> - 一个半 ECS 引擎：三轨时间模型——Logic / Present / Render 各自有独立的 `delta_time`。但实现上三轨的同步和插值逻辑较复杂。

> **默认推荐**：**分支 C 的简化版**。
> 1. 定义两个 Schedule：`Main`（可变步长）和 `FixedMain`（固定步长，如 60Hz）。
> 2. `Main` 每帧执行一次；`FixedMain` 根据 `accumulator / fixed_dt` 决定执行 0~N 次。
> 3. 物理、确定性逻辑挂在 `FixedMain`；渲染准备、相机、UI 挂在 `Main`。
> 4. **防螺旋死亡**：设置 `max_fixed_steps_per_frame`（如 5）。如果一帧需要跑超过 5 次 FixedUpdate，说明逻辑已经追不上实时，应慢动作运行（`TimeScale` 降低）而不是崩溃。

为什么不用纯固定（分支 B）？因为渲染插值增加了代码量。为什么不用纯可变（分支 A）？因为确定性和物理稳定性是自研引擎的核心目标之一，不能妥协。

```cpp
// 双轨制核心循环实现
float fixed_dt = 1.0f / 60.0f;
const int max_fixed_steps = 5;

void engine_tick(float real_dt) {
    // 可变步长：Update 阶段
    world.run_schedule(Main, real_dt);
    
    // 固定步长：FixedMain 阶段
    accumulator += real_dt;
    int steps = 0;
    while (accumulator >= fixed_dt && steps < max_fixed_steps) {
        world.run_schedule(FixedMain, fixed_dt);
        accumulator -= fixed_dt;
        steps++;
    }
    
    // 防螺旋死亡：如果 steps 达到上限，不要丢弃剩余 accumulator
    // 而是让它留到下一帧，同时日志报警
    if (steps >= max_fixed_steps) {
        LOG_WARN("Fixed update spiral detected! TimeScale reduced.");
        time_scale *= 0.9f;  // 慢动作运行，给逻辑喘息时间
    }
}
```

## 问题 6：世界快照与确定性回放——如何实现"给定相同输入，第 N 帧 bit 级一致"？

如果一台机器能保存第 1000 帧的完整状态，另一台机器加载这个状态后从第 1000 帧继续执行，两台机器的输出必须完全一致。这是网络同步、调试回放、AI 沙箱的基石。

### 根因分析：为什么"相同代码相同输入"不等于"相同输出"？

C++ 程序中有大量**隐式非确定性来源**：
1. **浮点运算**：`a + b + c` 与 `a + (b + c)` 在浮点下可能不同；不同编译器/CPU 的 FMA（乘加融合）指令行为不同。
2. **容器遍历顺序**：`std::unordered_map` 的遍历顺序在不同运行间不同。
3. **随机数**：`rand()` 依赖全局种子，线程安全版本可能使用 TLS，跨平台行为不同。
4. **动态内存地址**：指针值影响 `std::sort` 的不稳定排序结果（比较相等时交换顺序取决于地址）。
5. **并行执行顺序**：多线程 System 的完成顺序不可预测，如果后续 System 依赖"谁先完成谁的数据先可见"，就会 diverge。

### 确定性保证清单

要让引擎 bit 级确定，调度层和所有 System 必须共同遵守以下规则：

**规则 1：固定 System 执行顺序**
拓扑排序必须稳定。对于相同入度的节点，按**注册顺序**打破平局。不能依赖 `std::sort` 的未定义稳定行为。

```cpp
// 稳定拓扑排序：入度相同时按注册索引排序
std::queue<SystemID> q;
for (SystemID i = 0; i < systems.size(); ++i) {
    if (indegree[i] == 0) q.push(i);
}

while (!q.empty()) {
    // 关键：用 FIFO 队列保证同入度节点按注册顺序出队
    SystemID u = q.front(); q.pop();
    order.push_back(u);
    for (SystemID v : dependents[u]) {
        if (--indegree[v] == 0) q.push(v);
    }
}
```

**规则 2：固定时间步**
逻辑更新必须用固定 `dt`（如 1/60 秒）。渲染可以用可变步长插值，但逻辑轨道必须是时钟驱动的。

**规则 3：确定性随机数**
禁用 `rand()` 和 `std::random_device`。使用确定性的伪随机数生成器（如 XorShift128+），且每个需要随机数的 System 持有独立的种子流，避免全局状态竞争。

```cpp
struct DeterministicRng {
    uint64_t state[2];
    uint64_t next() {
        uint64_t s1 = state[0];
        uint64_t s0 = state[1];
        state[0] = s0;
        s1 ^= s1 << 23;
        state[1] = s1 ^ s0 ^ (s1 >> 17) ^ (s0 >> 26);
        return state[1] + s0;
    }
};
```

**规则 4：禁止未初始化的内存**
所有组件必须通过默认构造函数初始化为确定值。C++ 的 `struct Vec3 { float x, y, z; };` 如果不显式初始化，`x/y/z` 可能是任意垃圾值——在快照序列化时表现为不可预测的 bit 模式。

**规则 5：禁止非确定性容器遍历**
`Query` 返回的实体顺序必须稳定。如果 ECS 存储使用 `Archetype` + `Chunk`，同一 Archetype 内的实体顺序按创建时间排列（swap-and-pop 销毁时保持连续），查询结果天然稳定。但如果使用 `SparseSet`，遍历顺序取决于稀疏索引的哈希分布——需要额外排序或使用稳定遍历策略。

**规则 6：单线程确定性热路径**
对于需要严格确定性的 Phase（如物理、战斗逻辑校验），即使在多核机器上也可以强制串行执行。并行收益在这里不如确定性重要。

### 世界快照的实现

利用 [[反射系统]] 的元数据，遍历所有 Archetype → Chunk → Entity → Component，将字段值序列化为二进制块：

```cpp
struct WorldSnapshot {
    uint64_t frame_number;
    float fixed_time;
    std::vector<uint8_t> data;
};

WorldSnapshot capture(const World& world) {
    WorldSnapshot snap;
    snap.frame_number = world.current_frame();
    snap.fixed_time = world.accumulated_fixed_time();
    
    // 遍历所有 Archetype
    for (const Archetype& arch : world.archetypes()) {
        // 只序列化带 Replicated 标记的组件（可选优化）
        for (ComponentTypeID type : arch.component_types()) {
            if (!type.has_flag(ComponentFlag::Replicated)) continue;
            
            const TypeDesc& desc = type_registry[type];
            for (size_t i = 0; i < arch.entity_count(); ++i) {
                const void* component_ptr = arch.get_component_raw(type, i);
                desc.serialize(component_ptr, snap.data);
            }
        }
    }
    return snap;
}
```

**快照用途**：
- **网络同步**：服务器每 N 帧发送一次完整快照，客户端从快照恢复。
- **调试回放**：记录每秒一个快照，bug 发生后加载最近快照并重放后续输入。
- **AI 沙箱**：Agent 在副本世界上做实验，如果结果不好就回滚到快照。

**快照优化**：
- **增量 Diff**：连续两帧的快照做差异编码，只发送变化的字段。
- **Chunk 哈希**：用 CRC32 快速跳过未变化的 Chunk。
- **频率分层**：关键组件（位置、血量）每帧快照；次要组件（粒子颜色）每 10 帧快照。

## 问题 7：多世界/子世界调度——编辑器 PIE、渲染提取怎么与主世界协同？

单机引擎可能只有一个世界，但一旦引入编辑器、渲染线程、或 AI 沙箱，就会出现"多个世界需要以不同频率、不同规则运行"的需求。

### 分支 A：单世界 + 多上下文切换（UE 方案）

UE 的编辑器在 Tick 时先 Tick EditorWorld，再逐个 Tick PIE World。通过 `FWorldContext` 和 `SetPlayInEditorWorld` / `RestoreEditorWorld` 切换 `GWorld` 指针。

```cpp
// 分支 A：UE 风格的多世界切换
void UEditorEngine::Tick(float delta) {
    // 1. Tick Editor World（只更新 Viewport）
    EditorWorld->Tick(LEVELTICK_ViewportsOnly, delta);
    
    // 2. 逐个 Tick PIE Worlds
    for (auto& ctx : WorldList) {
        SetPlayInEditorWorld(ctx.World);
        ctx.World->Tick(LEVELTICK_All, delta);
    }
    RestoreEditorWorld();
}
```

**适用场景**：编辑器与运行时需要严格隔离（EditorWorld 和 PIE World 的状态完全不同）。

**隐藏代价**：
- `GWorld` 全局指针切换是**线程不安全**的隐患源。如果 Render Thread 正在读取 `GWorld`，而 Game Thread 切换了指针，会导致崩溃。
- 每个世界有自己的 Tick 管理器，内存和调度开销随世界数量线性增长。

**失效条件**：需要高频跨世界数据同步时（如渲染世界每帧从逻辑世界提取数据）。

### 分支 B：主世界 + SubApp 提取（Bevy 方案）

Bevy 的 `SubApps` 结构更清晰：主世界先完整执行一帧，然后对每个子世界执行 `extract` + `update`。

```rust
// 分支 B：Bevy 风格的 SubApp
fn update(sub_apps: &mut SubApps) {
    // 1. 主世界完整执行一帧
    sub_apps.main.run_default_schedule();
    
    // 2. 子世界：先 extract 数据，再执行自己的 schedule
    for (_label, sub_app) in &mut sub_apps.sub_apps {
        sub_app.extract(&mut sub_apps.main.world);
        sub_app.update();
    }
    
    // 3. 清除主世界变更追踪器
    sub_apps.main.world.clear_trackers();
}
```

**关键设计**：
- `extract` 是显式的数据拷贝函数（`FnMut(&mut World, &mut World)`），第一个 World 是主世界（数据源），第二个是子世界（目标）。
- 子世界的 `update()` 在 `extract` 之后执行，因此子世界能看到主世界本帧的所有变更。
- 子世界的变更不会影响主世界（除非子世界主动写回）。

**适用场景**：渲染世界需要从逻辑世界提取数据、AI 沙箱需要逻辑世界的副本。

**隐藏代价**：
- `extract` 是一次性数据拷贝，如果数据量大（如上万个实体的 Transform），拷贝开销不可忽视。
- `clear_trackers` 在主世界所有子世界处理完后才执行，意味着子世界的 `extract` 可以看到主世界的变更标记，但主世界看不到子世界的。

**失效条件**：需要双向高频同步时（如物理世界和逻辑世界互相反馈）。

### 默认实现推荐

> **默认推荐**：**分支 B 的简化版**。
> - 主世界：运行 `Main` 和 `FixedMain` Schedule。
> - 渲染子世界：每帧 `extract` 主世界的 `GlobalTransform` + `Mesh` + `Material`，然后在自己的 `Render` Schedule 中构建渲染命令。
> - AI 沙箱子世界：按需 `extract` 主世界状态到副本，Agent 在副本上实验，不影响主世界。

为什么不是分支 A？因为 `GWorld` 全局切换是隐式魔法，调试困难。Bevy 的显式 `extract` 函数让数据流向一目了然，更符合"AI 可观测性"的设计目标。

## 默认推荐路径总结

### 默认推荐路径
- **Phase 模型**：可扩展的 `ScheduleLabel` 简化版。默认 4 阶段（`PreUpdate` → `Update` → `PostUpdate` → `Render`），允许插件通过 `insert_before` / `insert_after` 动态扩展。
- **依赖声明**：自动推导 + 显式 `before` / `after` 的混合策略。每个 System 声明 `reads` / `writes`，调度器自动推导隐式依赖；语义依赖通过显式声明覆盖。
- **同步点策略**：默认 Phase 边界为安全点，全局命令缓冲统一应用。提供显式 `apply_deferred()` 作为逃生通道。
- **并行粒度**：Chunk 级数据并行起步，System 数量 >10 且 Profile 证明存在串行瓶颈时引入 Wavefront 调度。
- **时间步**：双轨制。`Main`（可变步长）与 `FixedMain`（固定步长，如 60Hz），物理和确定性逻辑挂 `FixedMain`，渲染和 UI 挂 `Main`。
- **确定性**：固定 `dt` + 稳定拓扑排序（FIFO 打破平局）+ 确定性 RNG + 禁止未初始化内存。
- **多世界**：主世界 + SubApp 提取。渲染子世界每帧 `extract` 主世界数据，AI 沙箱子世界按需提取实验。

### UE 工业级设计吸收
- **TickGroup 粗粒度阶段**：UE 的 9 个硬编码 TickGroup → 映射为可扩展 Phase 枚举的默认顺序。
- **Prerequisites 动态降级**：UE 的 `AddPrerequisite` 显式依赖 + 实际开始组降级 → 映射为混合依赖声明（自动推导 + 显式覆盖）。
- **多世界切换**：UE 的 `FWorldContext` + `GWorld` 指针切换 → 映射为显式 SubApp `extract` 函数，避免隐式全局状态。
- **物理 Sub-Stepping**：UE 在可变帧内多次物理更新 → 映射为独立 `FixedMain` Schedule 的固定步长执行。

### ECS 映射
- System 是无状态的纯函数，所有状态在组件中。调度器只维护只读的依赖图和 Phase 配置。
- `CommandBuffer` 统一了结构变更的延迟应用，消除了 OOP 引擎中分散的延迟队列。
- 并行策略基于组件访问冲突检测（`FilteredAccessSet` / 位图），而非 OOP 的 Actor/Component 级 Tick 并行。

### 条件切换路径
| 条件 | 切换方案 |
|------|---------|
| 阶段语义极度稳定、不需扩展 | 硬编码 Phase 枚举（UE 风格） |
| 系统数量少（<10）、领域单一 | 纯依赖图，无 Phase |
| 同一 Phase 内复杂链式创建需求 | 自动 `ApplyDeferred` 插入（Bevy 风格 distance 算法） |
| 确定性回放 / 网络预测回滚 | 双缓冲世界（内存换确定性） |
| System 数量多、依赖复杂、CPU 瓶颈 | 位图运行时调度（Bevy 风格） |
| 极简项目（无物理、无网络） | 纯可变时间步（需接受非确定性） |

### 扩展路径
- 自动歧义检测（开发模式警告数据冲突但未排序的 System 对）。
- 渲染线程与逻辑线程的显式 `extract` + `clear_trackers` 多帧流水线。
- 世界快照的增量 Diff、Chunk 哈希跳过、频率分层。
- 跨平台浮点确定性（FMA 控制、编译器标志统一）。

## "如果我要 vibe coding，该偷哪几招？"

1. **从 3 个 Phase 开始，不要从 9 个开始**
   `PreUpdate → Update → PostUpdate → Render` 足够支撑一个可玩的 demo。Phase 太多会增加理解成本，而且你很可能根本用不满。

2. **拓扑排序是调试神器**
   在日志中打印每帧的 System 执行顺序。当发现"AI 在移动之后执行导致决策延迟一帧"时，看一眼日志就能定位依赖问题。

3. **命令缓冲默认在 Phase 边界 flush，但留一个逃生通道**
   不要让每个 System 自己维护一套延迟队列。一个全局的 `CommandBuffer` 不仅代码更少，而且 AI 只需要监控一个入口就能知道所有的结构变更。但同一 Phase 内有链式创建需求时，允许局部 `apply_deferred()`。

4. **固定步长 dt 是确定性的前提，双轨制是最佳折中**
   如果你的 `dt` 直接用上一帧的真实耗时，浮点数累积会让不同机器的结果出现微小差异。使用固定时间步（如 1/60 秒），并将余数 carry over 到下一帧。渲染走可变步长插值，逻辑走固定步长。

5. **快照是最高杠杆的调试工具**
   在游戏开发中，80% 的 bug 都发生在"某次状态变化之后"。如果能保存快照并在 10 秒内重放，调试效率会提升一个数量级。快照代码可以和序列化代码共用同一套反射基础设施。

6. **并行从 Chunk 级开始，System 级并行是进阶优化**
   不要一开始就做 Wavefront 或位图调度。先让单个 System 能按 Chunk 并行跑起来（Transform 更新、粒子、动画评估），这能解决 80% 的 CPU 瓶颈。System 级并行是 System 数量 >10 且 Profile 证明有串行瓶颈时的优化。
