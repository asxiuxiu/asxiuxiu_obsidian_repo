# 系统调度与确定性

> **前置依赖**：[[组件系统架构]]、[[反射系统]]
> **本模块增量**：在 ECS 骨架上增加 Tick 阶段划分、System 依赖图与拓扑排序、命令缓冲、世界快照与确定性回放能力。
> **下一步**：[[事件总线]]，因为调度好的 System 之间需要一种解耦的通信机制。

---

## Why：为什么 ECS 引擎需要一个确定的调度系统？

在 [[组件系统架构]] 中，我们让 System 能批量查询组件。但如果所有 System 都胡乱执行，引擎很快就会陷入混乱。

### 场景 1：移动必须在碰撞检测之前

`MovementSystem` 把玩家向前推了 1 米，`CollisionSystem` 发现这 1 米穿墙了，又把玩家推回来。如果 `CollisionSystem` 先执行，`MovementSystem` 后执行，玩家就会穿墙。执行顺序不同，结果不同。

### 场景 2：一帧的中间销毁了正在被遍历的实体

`SpawnSystem` 在遍历 `Bullet` 组件时，`CollisionSystem` 突然销毁了一个子弹（因为它击中了敌人）。`SpawnSystem` 的迭代器瞬间失效，崩溃或访问到已回收的内存。

### 场景 3：联机对战需要帧同步

两个客户端运行完全相同的逻辑，但主机 A 的 `AI System` 和 `Movement System` 执行顺序与客机 B 不同。10 秒后，两个客户端的世界状态出现了微小的偏差，然后偏差像滚雪球一样越来越大，最终导致不同步。

> **核心需求**：引擎需要一个**显式的、可验证的、输入确定则输出确定**的调度系统。它必须能回答："这 20 个 System 谁先谁后？" "谁依赖谁？" "增删实体什么时候是安全的？" "给定相同的输入序列，第 N 帧的状态是否完全一致？"

## What：最简化版本的调度系统长什么样？

下面是一个包含**拓扑排序 + 命令缓冲**的最小 ECS 调度器。它足够让 5~10 个 System 安全地串行运行。

```cpp
#include <vector>
#include <queue>
#include <unordered_map>
#include <unordered_set>
#include <functional>
#include <cstdint>

using SystemID = uint32_t;

struct System {
    std::string name;
    std::function<void(float)> tick;
    std::unordered_set<SystemID> dependencies;
};

// ---------- 命令缓冲 ----------
struct CommandBuffer {
    std::vector<std::function<void()>> commands;
    void push(std::function<void()> cmd) { commands.push_back(cmd); }
    void execute() {
        for (auto& cmd : commands) cmd();
        commands.clear();
    }
};

// ---------- 调度器 ----------
struct Scheduler {
    std::vector<System> systems;
    std::unordered_map<std::string, SystemID> name_to_id;
    std::vector<SystemID> execution_order;
    bool dirty = true;

    SystemID register_system(const std::string& name, std::function<void(float)> tick) {
        SystemID id = (SystemID)systems.size();
        systems.push_back({name, tick, {}});
        name_to_id[name] = id;
        dirty = true;
        return id;
    }

    void add_dependency(SystemID dependent, SystemID dependency) {
        systems[dependent].dependencies.insert(dependency);
        dirty = true;
    }

    // Kahn 算法拓扑排序
    bool sort() {
        if (!dirty) return true;
        execution_order.clear();

        std::unordered_map<SystemID, int> indegree;
        for (SystemID i = 0; i < systems.size(); ++i) indegree[i] = 0;
        for (SystemID i = 0; i < systems.size(); ++i) {
            for (SystemID dep : systems[i].dependencies) {
                indegree[i]++;
            }
        }

        std::queue<SystemID> q;
        for (auto& [id, deg] : indegree) {
            if (deg == 0) q.push(id);
        }

        while (!q.empty()) {
            SystemID u = q.front(); q.pop();
            execution_order.push_back(u);
            for (SystemID v = 0; v < systems.size(); ++v) {
                if (systems[v].dependencies.count(u)) {
                    if (--indegree[v] == 0) q.push(v);
                }
            }
        }

        dirty = false;
        return execution_order.size() == systems.size();
    }

    void tick_all(float dt, CommandBuffer& cmd) {
        if (!sort()) {
            // 循环依赖，应提前崩溃或记录
            return;
        }
        // 1. 先执行命令缓冲（安全点）
        cmd.execute();
        // 2. 按拓扑序执行 System
        for (SystemID id : execution_order) {
            systems[id].tick(dt);
        }
    }
};
```

这个最小实现解决了什么？
- **执行顺序可验证**：通过 Kahn 算法，System 之间的依赖关系被显式化。如果程序员错误地让 A 依赖 B、B 又依赖 A，调度器能立刻检测到循环依赖。
- **结构安全**：所有创建/销毁实体的操作都先进入 `CommandBuffer`，在 tick 的最开始统一执行。System 遍历组件时，ECS 的结构是稳定的。
- **确定性**：给定相同的初始状态和相同的 `dt`，只要依赖图不变，每帧的执行顺序就是固定的，输出也固定。

还缺什么？
- **没有 Tick Phase**：所有 System 都在同一个大桶里。真实引擎需要 `PreUpdate`、`Update`、`PostUpdate`、`Render` 等阶段。
- **没有并行**：所有 System 串行执行，无法利用多核。
- **没有世界快照**：无法保存某一帧的完整状态用于回放或网络同步。
- **循环依赖报告太粗糙**：只知道"有环"，但不知道具体是哪几个 System 形成了环。

## How：真实引擎的调度系统是如何一步一步复杂起来的？

### 阶段 1：从单桶调度 → Tick Phase + 分层命令缓冲（解决组织混乱）

**触发原因**：当 System 数量超过 10 个时，所有 System 挤在一个列表里变得难以维护。你需要一种自然的方式来表达"这些 System 负责输入处理"、"那些 System 负责物理"、"另一些负责渲染准备"。

**代码层面的变化**：

1. **引入 Tick Phase**
   ```cpp
   enum class Phase {
       PreUpdate,   // 输入采样、命令缓冲应用
       Update,      // 主要逻辑：移动、AI、动画
       PostUpdate,  // 碰撞响应、后处理
       Render       // 渲染数据准备
   };
   ```
   每个 System 注册时绑定到一个 Phase。`World::tick(dt)` 按 Phase 顺序执行：
   ```cpp
   for (Phase p : {PreUpdate, Update, PostUpdate, Render}) {
       scheduler.tick_phase(p, dt);
   }
   ```

2. **Phase 级别的命令缓冲**
   不再只有一个全局命令缓冲，而是每个 Phase 配一个：
   ```cpp
   CommandBuffer pre_update_cmds;
   CommandBuffer update_cmds;
   ```
   这样 `Update` Phase 中的 System 提交的命令可以在 `PostUpdate` Phase 之前应用，给中间系统留出处理窗口。

3. **System 的读写声明**
   让每个 System 显式声明它读什么、写什么：
   ```cpp
   struct SystemDesc {
       std::string name;
       Phase phase;
       std::vector<ComponentTypeID> reads;
       std::vector<ComponentTypeID> writes;
       std::function<void(float)> tick;
   };
   ```
   调度器可以据此自动推导依赖关系：如果 System A 写 `Position`，System B 读 `Position`，则 B 隐式依赖 A。

**状态变化图：Phase + 命令缓冲的执行时序**

```
Frame N 开始
│
├─► PreUpdate Phase
│   ├─► 执行 pre_update_cmds  （安全点：应用上一帧的输入/创建请求）
│   └─► InputSystem.tick()     （采样键盘鼠标，写入 Input 组件）
│
├─► Update Phase
│   ├─► 执行 update_cmds       （安全点）
│   ├─► AISystem.tick()        （读 Position，写 Target）
│   ├─► MovementSystem.tick()  （读 Velocity/Position，写 Position）
│   └─► AnimationSystem.tick() （读 State，写 BonePose）
│
├─► PostUpdate Phase
│   ├─► 执行 post_update_cmds
│   ├─► CollisionSystem.tick() （读 Position/Collider，写 Position/Velocity）
│   └─► TransformSystem.tick() （读 LocalTransform/Parent，写 GlobalTransform）
│
└─► Render Phase
    ├─► 执行 render_cmds
    └─► RenderSystem.tick()    （读 GlobalTransform/Mesh/Material）
```

> **精确性补充**：注意**命令缓冲的应用时机**。如果 `MovementSystem` 在 `Update` Phase 中调用 `cmd.destroy_entity(e)`，这个实体会立即进入 `update_cmds` 队列，但**不会**在 `MovementSystem` 的执行过程中被销毁。它会在 `PostUpdate` Phase 开始前的安全点才被真正移除。这保证了 `AnimationSystem`（同 Phase 中排在 Movement 后面的 System）仍然能看到这个实体。

### 阶段 2：从串行到并行调度（解决 CPU 利用率）

**触发原因**：当场景里有上万个实体、十几个 System 时，串行执行会让单核 CPU 爆满，而其他核心闲置。

**代码层面的变化**：

1. **Chunk 级并行**
   ECS 的 Archetype 将实体分成了多个 Chunk。同一个 System 对不同 Chunk 的访问通常是互不重叠的内存区域。因此可以将 Chunk 列表拆分为多个批次，交给线程池并行处理：
   ```cpp
   parallel_for(chunks, [](Chunk* chunk) {
       system.tick_chunk(chunk, dt);
   });
   ```

2. **基于读写依赖的 System 并行**
   调度器分析所有 System 的 `reads` 和 `writes`：
   - 若 System A 和 System B 的写集合不相交，且都不读对方写过的组件，则可以并行执行。
   - 若 System C 读 System A 写的组件，则 C 必须在 A 之后执行。

   这是一个经典的**任务图调度问题**。调度器在帧开始时会构建一个 DAG，将可并行的 System 打包成任务波次（wavefront），每一波内部的 System 并行执行，波次之间串行：
   ```
   Wave 0: InputSystem, SpawnSystem        (无依赖，并行)
   Wave 1: AISystem, MovementSystem        (读 Input，互不写相同组件，并行)
   Wave 2: CollisionSystem                 (读/写 Position，必须等 Wave 1)
   Wave 3: TransformSystem                 (读 LocalTransform/Parent，写 GlobalTransform)
   Wave 4: RenderSystem                    (读 GlobalTransform/Mesh)
   ```

3. **命令缓冲的线程安全**
   多线程 System 都可能向命令缓冲写入请求。命令缓冲需要是**线程本地**的（每个线程一个队列），在 Phase 安全点统一合并：
   ```cpp
   void apply_commands() {
       for (auto& local_buf : thread_local_buffers) {
           for (auto& cmd : local_buf) cmd();
       }
   }
   ```

> **性能诚实**：
> - "并行 System" 消除了什么成本？消除了串行执行的 CPU 瓶颈，将多个独立 System 的计算分布到多核。
> - 引入了什么新成本？**任务调度的开销**。如果每个 System 只处理 100 个实体，拆成多线程的任务提交和同步开销可能比串行更慢。
> - **反例**：System 处理量很小（<1ms）或依赖链很长（几乎无法并行）时，强行并行化只会增加代码复杂度而不提升性能。

### 阶段 3：工业级 → 世界快照、确定性回放与 AI 可观测性

**触发原因**：网络同步需要发送"世界状态"；调试需要"回放 bug 发生时的那一帧"；AI Agent 需要观察引擎的完整状态而不破坏它。

**代码层面的变化**：

1. **世界快照（World Snapshot）**
   利用 [[反射系统]] 的元数据，遍历所有实体的所有组件，将它们的字段值序列化为一个二进制块：
   ```cpp
   struct WorldSnapshot {
       uint64_t frame_number;
       std::vector<uint8_t> data;
   };

   WorldSnapshot capture(World& world) {
       // 遍历所有 Archetype → Chunk → Entity → Component
       // 通过 TypeDesc 逐个字段写入 data
   }
   ```
   快照可以用于：
   - **网络同步**：服务器每 N 帧发送一次完整快照，客户端从快照恢复。
   - **调试回放**：记录每秒一个快照，bug 发生后可以加载最近快照并重放后续输入。
   - **AI 沙箱**：Agent 在副本世界上做实验，如果结果不好就回滚到快照。

2. **确定性保证清单**
   为了确保"相同输入 → 相同输出"，调度器和所有 System 必须遵守以下规则：
   - **固定的 System 执行顺序**：拓扑排序必须稳定（相同入度的节点按注册顺序打破平局）。
   - **固定的 `dt` 处理**：帧时间不是直接用 `delta_time`，而是量化到固定步长（如 16.666ms），余数累积到下一帧。
   - **禁止未初始化的内存**：所有组件必须通过默认构造函数初始化为确定值。
   - **禁止非确定性的 API**：`rand()`、`std::unordered_map` 的遍历顺序、`Query` 返回的实体顺序（如果不稳定）都必须替换为确定性版本。
   - **单线程热路径**：对于需要严格确定性的 System（如物理、战斗逻辑），即使在多核机器上也可以在关键 Phase 强制串行执行。

3. **快照压缩与增量 Diff**
   完整快照可能很大（数 MB）。工业级实现会：
   - 只序列化**带 `Replicated` 标记**的组件。
   - 对连续两帧的快照做差异编码，只发送变化的字段。
   - 使用哈希（如 CRC32）快速跳过未变化的 Chunk。

## 从源码到 ECS 的重构映射

chaos 引擎的调度单元是 `Manager`，它是一个 System + ComponentManager 的混合体。迁移到纯 ECS 时，需要将其职责拆分：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `LevelTickManager` 的 9 个 Phase | **自定义 Phase 枚举** + **Scheduler** | 保留 Phase 思想，但 Phase 数量可根据项目需求简化（如 Pre/Update/Post/Render 4 个即可）。 |
| `Manager::processComponentsManagement()` | **全局 CommandBuffer** | Manager 的延迟队列被提升为 World 级别的命令缓冲，所有 System 共用同一套安全点机制。 |
| `addManagerDependency<Dependent, Dependency>()` | **System 的显式 `reads`/`writes` 声明** | 不再手动写 Manager 之间的依赖字符串，而是通过组件读写关系自动推导依赖图。 |
| `sortPhaseByDependencies()`（Kahn 算法）| **Scheduler::sort()** | 算法本身完全复用，但排序对象是 pure System 而非混合 Manager。 |
| `ConcurrentQueue<ComponentCreationData>` | **线程本地命令队列** | 从 Manager 各自维护队列变为每个线程一个队列，Phase 安全点统一合并。 |
| `ExecutionFlowType::Sequential`（当前强制串行）| **Wavefront 并行调度** | ECS 的 Chunk 划分让 System 级并行变得天然可行，不再需要"TODO: NOT YET IMPLEMENTED"。 |

> **Why ECS 更适合这个模块？**
> - **依赖自动推导**：chaos 的 Manager 依赖是程序员手动声明的，容易遗漏或写错。ECS 中 System 的依赖可以通过 `reads`/`writes` 自动推导，减少人为错误。
> - **并行粒度更细**：chaos 只能在单个 Manager 内部（如 `TransformManager` 的按 rank 更新）并行，Manager 之间必须串行。ECS 可以在 System 之间和 Chunk 之间同时并行。
> - **快照更简单**：chaos 的 Manager 持有大量内部状态（`m_registered_entities`、`m_create_component_funcs` 等），世界快照不仅要拍组件数据，还要拍 Manager 状态。ECS 中 System 是无状态的 pure function，快照只需要序列化组件数据即可。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ System 本身不持有状态，所有状态都在组件中。调度器只维护只读的依赖图和 Phase 配置。 |
| **自描述** | ✅ 调度器的 Phase 列表、System 执行顺序、读写声明都可以导出为 JSON。AI 可以知道"MovementSystem 在 Update Phase 执行，读 Velocity、写 Position"。 |
| **确定性** | ✅ 固定步长 `dt` + 稳定拓扑排序 + 确定性随机数种子。给定相同的输入序列，第 N 帧的状态 bit 级一致。 |
| **工具边界** | ✅ MCP 接口可暴露：`list_systems()`、`get_system_order()`、`step_frame(dt)`、`capture_snapshot()`、`restore_snapshot(data)`。 |
| **Agent 安全** | ✅ 命令缓冲让 AI 的修改请求不会立即生效。可以设置"审批模式"：AI 的命令进入 staging buffer，人类确认后再 apply。快照功能让 Agent 可以在副本世界上做实验并随时回滚。 |

## "如果我要 vibe coding，该偷哪几招？"

1. **从 3 个 Phase 开始，不要从 9 个开始**
   `PreUpdate → Update → PostUpdate → Render` 足够支撑一个可玩的 demo。Phase 太多会增加理解成本，而且你很可能根本用不满。

2. **拓扑排序是调试神器**
   在日志中打印每帧的 System 执行顺序。当发现"AI 在移动之后执行导致决策延迟一帧"时，看一眼日志就能定位依赖问题。

3. **命令缓冲必须全局且统一**
   不要让每个 System 自己维护一套延迟队列。一个全局的 `CommandBuffer` 不仅代码更少，而且 AI 只需要监控一个入口就能知道所有的结构变更。

4. **固定步长 dt 是确定性的前提**
   如果你的 `dt` 直接用上一帧的真实耗时，浮点数累积会让不同机器的结果出现微小差异。使用固定时间步长（如 1/60 秒），并将余数 carry over 到下一帧。

5. **快照是最高杠杆的调试工具**
   在游戏开发中，80% 的 bug 都发生在"某次状态变化之后"。如果能保存快照并在 10 秒内重放，调试效率会提升一个数量级。快照代码可以和序列化代码共用同一套反射基础设施。
