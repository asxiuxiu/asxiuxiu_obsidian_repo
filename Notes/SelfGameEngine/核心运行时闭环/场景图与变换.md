# 场景图与变换

> **前置依赖**：[[组件系统架构]]、[[数学基础]]
> **本模块增量**：在 ECS 骨架上增加 Transform 组件、父子层级、脏标记传播与批量矩阵更新能力。实体从此能在三维空间中建立关系。
> **下一步**：[[反射系统]]，因为 Inspector 需要自动识别新组件，才能实时调试 Transform 数值。

---

## Why：为什么 ECS 世界需要场景图与变换系统？

在 [[组件系统架构]] 中，我们让实体可以拥有组件，也能被 System 批量查询。但如果世界里的实体都是彼此孤立的点，引擎只能渲染一堆漂浮在坐标原点附近的方块。没有空间关系，就没有"世界"。

### 场景 1：一个角色拿着一把剑

剑必须始终挂在角色的右手上。如果角色移动、旋转，剑必须同步跟随。没有父子层级，你每帧都要手动计算剑的世界坐标，不仅繁琐，还容易因更新顺序错误导致剑"抖一帧"。

### 场景 2：摄像机跟随玩家

摄像机作为玩家的子节点，其局部偏移是固定的 `(0, 2, -5)`。玩家跳跃时，摄像机应该平滑地一起上升。如果没有世界/局部变换的自动级联，所有跟随逻辑都要硬编码在专门的 CameraSystem 中，无法复用于其他需要跟随的物体。

### 场景 3：批量渲染 10,000 个实体

渲染系统需要每个实体的**世界矩阵**（4×4）来做视锥剔除和 GPU 上传。如果每个 System 都自己从 `LocalTransform` 临时合成矩阵，不仅重复计算，而且无法利用"只有 5% 的实体每帧实际移动"这一事实。需要一种机制，**只重新计算那些真的变了的东西**。

> **核心需求**：引擎需要一个机制来描述"谁挂在谁下面"，并能**高效、确定性地**将局部变换转换为世界变换，且只计算那些"脏了的"分支。

## What：最简化版本的三维变换系统长什么样？

下面是一个**纯 ECS 风格**的最小实现。它包含三个组件和一个 System：

- `LocalTransform`：局部 TRS（相对于父节点）
- `GlobalTransform`：世界空间的 4×4 矩阵
- `Parent`：指向父实体的 EntityID
- `TransformSystem`：按 hierarchy 深度排序后批量更新

```cpp
#include <vector>
#include <algorithm>
#include <cstdint>

using Entity = uint32_t;
const Entity INVALID_ENTITY = ~0u;

// 假设已有 Vec3, Quat, Mat4 数学类型
struct Vec3 { float x, y, z; };
struct Quat { float x, y, z, w; };
struct Mat4 {
    float m[16];
    static Mat4 TRS(const Vec3& t, const Quat& r, const Vec3& s);
};

// ---------- ECS 组件 ----------
struct LocalTransform {
    Vec3    position = {0,0,0};
    Quat    rotation = {0,0,0,1};
    Vec3    scale    = {1,1,1};
};

struct GlobalTransform {
    Mat4 matrix;
    bool is_dirty = true;  // 是否需要重新计算
};

struct Parent {
    Entity parent = INVALID_ENTITY;
};

// ---------- 最小 TransformSystem ----------
struct TransformSystem {
    // 辅助：计算 hierarchy 深度（rank）
    uint32_t getRank(Entity e, const std::vector<Parent>& parents) {
        uint32_t rank = 0;
        while (e < parents.size() && parents[e].parent != INVALID_ENTITY) {
            e = parents[e].parent;
            ++rank;
            if (rank > 100) break; // 防止循环
        }
        return rank;
    }

    void update(
        std::vector<LocalTransform>&  locals,
        std::vector<GlobalTransform>& globals,
        const std::vector<Parent>&    parents,
        const std::vector<Entity>&    entities)
    {
        // 1. 按 rank 排序，保证父节点先更新
        std::vector<std::pair<uint32_t, Entity>> ranked;
        for (Entity e : entities) {
            if (e < parents.size())
                ranked.push_back({getRank(e, parents), e});
        }
        std::sort(ranked.begin(), ranked.end(),
            [](auto& a, auto& b){ return a.first < b.first; });

        // 2. 逐个更新
        for (auto& [rank, e] : ranked) {
            if (e >= locals.size() || e >= globals.size()) continue;

            const Parent& p = parents[e];
            bool parent_dirty = (p.parent != INVALID_ENTITY && p.parent < globals.size())
                                ? globals[p.parent].is_dirty
                                : false;

            if (!globals[e].is_dirty && !parent_dirty) continue;

            Mat4 local = Mat4::TRS(locals[e].position, locals[e].rotation, locals[e].scale);
            if (p.parent != INVALID_ENTITY && p.parent < globals.size()) {
                // 世界矩阵 = 父世界矩阵 × 局部矩阵
                globals[e].matrix = multiply(globals[p.parent].matrix, local);
            } else {
                globals[e].matrix = local;
            }
            globals[e].is_dirty = false;
        }
    }

    Mat4 multiply(const Mat4& a, const Mat4& b); // 矩阵乘法实现
};
```

这个最小实现解决了什么？
- **父子层级**：通过 `Parent` 组件，任意实体可以挂在另一个实体下。
- **确定性更新**：按 `rank` 排序后父节点一定先于子节点计算，不会出现"子节点用父节点上一帧矩阵"的抖动。
- **脏标记跳过**：静止的实体（`is_dirty == false` 且父节点也干净）完全跳过矩阵合成。

还缺什么？
- **动态增删父子关系**时，`getRank()` 的递归开销很大。
- **没有循环依赖检测**：如果 A 是 B 的父节点，B 又变成 A 的父节点，系统会死循环或崩溃。
- **没有缓存局部性**：`std::vector` 按 EntityID 索引，但遍历顺序是按 rank 排序后的随机访问，cache 不友好。
- **矩阵实时合成开销**：每帧都对每个脏节点做 `TRS` + `mat4×mat4`，对于大量骨骼节点仍有优化空间。

## How：真实引擎的场景图是如何一步一步复杂起来的？

### 阶段 1：从朴素递归 → 显式 Hierarchy 节点池（解决动态增删和循环检测）

**触发原因**：最小实现每次更新都要递归算 `rank`，而且父子关系变更时无法快速找到所有子节点做级联更新。

**代码层面的变化**：

1. **引入独立的 `HierarchyNode` 池**
   ```cpp
   struct HierarchyNode {
       Entity   entity;
       Entity   parent;
       std::vector<Entity> children;
       uint32_t rank;
       bool     is_dirty;
   };
   ```
   `SceneGraph` 维护一个 `std::vector<HierarchyNode>` 或 `SparseSet<HierarchyNode>`，并通过 `Map<Entity, NodeIndex>` 快速定位。

2. **`setParent(child, parent)` 的原子操作**
   ```cpp
   void setParent(Entity child, Entity parent) {
       // 1. 循环检测：从 parent 向上遍历，如果遇到 child，报错
       // 2. 从旧父节点的 children 列表中移除 child
       // 3. 将 child 加入新父节点的 children 列表
       // 4. 更新 child 及其所有后代的 rank
       // 5. 标记 child 为 dirty（以及所有后代）
   }
   ```

3. **Rank 的增量维护**
   不再每帧递归计算 rank。仅在父子关系变更时，用 BFS/DFS 更新受影响子树的 `rank`：
   ```cpp
   std::queue<Entity> q;
   q.push(child);
   while (!q.empty()) {
       Entity e = q.front(); q.pop();
       for (Entity c : nodes[e].children) {
           nodes[c].rank = nodes[e].rank + 1;
           q.push(c);
       }
   }
   ```

**状态变化图：父节点变更时的级联 dirty 标记**

变更前：
```
Entity A (rank=0, dirty=false)
  └── Entity B (rank=1, dirty=false)
        └── Entity C (rank=2, dirty=false)
```

执行 `setParent(B, new_parent_D)`：
```
1. 从 A 的 children 中移除 B
2. 将 B 加入 D 的 children
3. BFS 更新 B 的子树 rank:
   B.rank = D.rank + 1  (假设 D.rank=0 → B.rank=1)
   C.rank = B.rank + 1  (C.rank=2)
4. 标记 B.dirty = true, C.dirty = true
```

变更后：
```
Entity D (rank=0, dirty=false)
  └── Entity B (rank=1, dirty=true)
        └── Entity C (rank=2, dirty=true)

Entity A (rank=0, dirty=false)  // A 本身不变
```

> **精确性补充**：这里的关键是 **"哪里连续、哪里不连续"**。`HierarchyNode` 池本身是连续的（`vector` 存储），但子节点的 `children` 数组是各自独立的动态分配。这意味着遍历整个 hierarchy 时，会在 `NodePool` 连续内存和各个 `children` 小数组之间反复跳转。对于深度很大的树（如 100 根骨骼），这种跳转可能带来显著的 cache miss。

---

### 阶段 2：按 Rank 分批并行 + 矩阵缓存（解决缓存局部性和 CPU 开销）

**触发原因**：当场景里有大量静态物体和少量动态物体时，阶段 1 的 `std::sort` 按 rank 全量排序是浪费；而且每个脏节点都做 `mat4×mat4` 对 CPU 仍是负担。

**代码层面的变化**：

1. **将实体按 rank 分桶（Bucket）**
   ```cpp
   std::vector<std::vector<Entity>> rank_buckets;
   ```
   在父子关系变更时，直接把实体放入对应 bucket，而不是每帧重新排序。

2. **按 bucket 顺序、桶内并行更新**
   ```cpp
   for (auto& bucket : rank_buckets) {
       parallel_for(bucket, [](Entity e) {
           update_global_transform(e);
       });
   }
   ```
   同一 bucket 内的实体互不依赖（它们的父节点都在更小的 rank 中），可以安全并行。

3. **缓存 `LocalToWorld` 矩阵，TRS 分离存储**
   最小实现中 `GlobalTransform` 只存矩阵。工业级实现通常同时存储：
   - `LocalTransform`：TRS（float3 + quaternion + float3）
   - `GlobalTransform`：float4×4 矩阵
   - 可选的 `PreviousGlobalTransform`：用于运动模糊和 TAA

4. **按需矩阵合成 vs 全量合成**
   像 chaos 引擎那样，**不缓存矩阵**，只在需要时从 `Pose3D` 合成。或者像 Unity DOTS 那样，**每帧统一合成并缓存**。
   - **不缓存**：节省内存（每个 Transform 省 64 bytes），适合移动平台或 Transform 数量极多的开放世界。
   - **缓存**：渲染、物理、动画系统可以直接读取连续矩阵数组做 SIMD 批量处理，适合 CPU 充裕的 PC/主机平台。

> **性能诚实**：
> - "缓存矩阵" 不是零开销。它增加了 64 bytes/实体 的内存占用，而且每帧都要做 `mat4×mat4`。
> - "不缓存矩阵" 也不是零开销。每次渲染该实体时都要临时合成矩阵，如果渲染系统对同一个实体请求多次矩阵，就会重复计算。
> - **选择基准**：如果 Transform 数量 > 50 万且大部分是静态的，不缓存更省内存；如果每帧有上万个实体被渲染且骨骼动画密集，缓存更省 CPU。

### 阶段 3：工业级增强 → 延迟挂接、命令缓冲与空间加速结构分离

**触发原因**：编辑器中用户可能在一帧内拖拽 100 个物体改变父子关系；运行时物理系统可能在一帧内生成大量碎片并挂接到爆炸中心。直接立即修改 hierarchy 会导致遍历中的 System 崩溃。

**代码层面的变化**：

1. **延迟父子关系处理（Pending Parentship）**
   - `setParent` 不再直接修改 `HierarchyNode`，而是将请求写入 `pending_parentship_queue`。
   - 在 `TransformSystem` 的帧开始阶段（或专门的 `PreUpdate` Phase）统一处理队列：
     ```cpp
     struct PendingParentship {
         Entity child;
         Entity parent;
         bool preserve_world_transform;
     };
     ```
   - `preserve_world_transform` 为 true 时，系统会先用当前世界矩阵反推局部矩阵，再挂接，保证编辑器拖拽时物体不会瞬移。

2. **Transform 命令缓冲与 ECS 命令缓冲合并**
   在 [[组件系统架构]] 中，我们已经有了添加/移除组件的命令缓冲。场景图的延迟挂接可以复用同一套机制：
   ```cpp
   world.commands().set_parent(child, parent);
   world.commands().apply(); // 安全点统一执行
   ```

3. **场景图与空间加速结构彻底分离**
   `SceneGraph` 只负责**父子层级和变换传播**，不承担任何空间查询职责。
   - 视锥剔除、射线检测、碰撞 broadphase 由专门的 **BVH / Octree / Spatial Hash** 负责。
   - 这些空间结构在 Transform 更新完毕后，从 `GlobalTransform` 组件读取世界包围盒或位置，重建或增量更新自己的加速结构。
   - 这种分离让 hierarchy 可以很深（如骨骼链），而不影响空间查询的效率。

## 从源码到 ECS 的重构映射

chaos 引擎的场景图实现是 OOP 风格的（`SceneHierarchy` + `TransformManager` + `TransformComponent` 基类）。迁移到 ECS 的映射如下：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `SceneHierarchy` + `HierarchyNode` | **`Parent` 组件** + 可选的 **`Children` 组件** | 不再维护独立的 hierarchy 节点池。`Parent(Entity)` 就是一个普通的 ECS 组件，由同一个 World 存储和查询。 |
| `TransformComponent`（含 `m_parent_handle`、`m_local_pose`、`m_pose`） | **`LocalTransform`** + **`GlobalTransform`** + **`Parent`** 三个独立组件 | 将 TRS、世界矩阵、父子引用拆成三个 POD 组件，分别由不同 System 读写。 |
| `TransformManager::calculateAndUpdateTransform()` | **`TransformSystem`** | 一个普通的 ECS System，通过 `Query<(&LocalTransform, &mut GlobalTransform, &Parent)>` 批量更新，不需要是特殊的 Manager 子类。 |
| `DesiredPoseMap` + `PendingUpdateEntityMap` | **命令缓冲（Command Buffer）** | 外部系统想修改 Transform 时，写入命令缓冲，在 System 执行前统一应用。 |
| `getAllChildren()` 递归收集 | **按需查询 `Parent` 组件** 或维护 **`Children` 组件** | ECS 中可以通过 `Query<(&Parent,)>` 反向查找谁是某个实体的子节点；或者给父实体附加 `Children(std::vector<Entity>)` 组件，直接读取。 |
| `Pose3D::toMat44()` 按需合成 | **可选的 `GlobalTransform` 矩阵缓存** | 如果每帧渲染量大，在 `TransformSystem` 中统一预计算矩阵并写入 `GlobalTransform`；如果内存更紧张，保留按需合成策略。 |

> **Why ECS 更适合这个模块？**
> - **数据连续**：chaos 的 `TransformManager` 需要通过 `EntityComponentManager` 的多层哈希表逐个获取 `TransformComponent`。ECS 的 `Query<LocalTransform, GlobalTransform, Parent>` 可以直接线性遍历匹配的 Archetype Chunk，cache 友好得多。
> - **并行天然**：同一 rank 的节点在 chaos 中需要显式用 `g_task_executor` 分批任务化。在 ECS 中，只要 `TransformSystem` 只写 `GlobalTransform`、读 `LocalTransform` 和 `Parent`，调度器就可以自动按 Chunk 并行，无需手写任务拆分。
> - **组合灵活**：chaos 中每个实体必须有且只有一个 `TransformComponent`。ECS 中，静态障碍物可以只有 `GlobalTransform`（没有 `LocalTransform` 和 `Parent`），进一步节省内存和计算。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ `LocalTransform`、`GlobalTransform`、`Parent` 都是 POD 组件。Hierarchy 结构完全由组件的有无决定，没有隐藏状态。 |
| **自描述** | ✅ `LocalTransform` 和 `GlobalTransform` 的字段可通过反射注册表导出为 `{position: [f32;3], rotation: [f32;4], scale: [f32;3]}`。AI 不需要读 C++ 头文件。 |
| **确定性** | ✅ 给定相同的父子关系序列和相同的局部变换输入，每帧输出的世界矩阵bit级一致。更新顺序由 rank 排序唯一确定。 |
| **工具边界** | ✅ MCP 接口可暴露：`set_local_transform(entity, json)`、`set_parent(child, parent)`、`get_world_matrix(entity)`。 |
| **Agent 安全** | ✅ 父子关系变更通过命令缓冲延迟执行，Agent 的误操作不会在中途破坏正在遍历的 hierarchy。支持 `preserve_world_transform` 选项防止视觉跳变。 |

## "如果我要 vibe coding，该偷哪几招？"

1. **EntityID + Parent 组件 = 最小 hierarchy**
   不要一上来就写四叉树、八叉树、BVH。先让 `Parent(Entity)` 组件跑起来，90% 的游戏场景都够用。

2. **rank 分桶比每帧排序快 10 倍**
   不要在每帧的 `System.update()` 里 `std::sort` 所有实体。在挂接/解挂时直接把实体移入对应 rank 的 bucket，更新时按 bucket 顺序遍历即可。

3. **脏标记是性能的生命线**
   静态物体应该占场景的 80% 以上。如果没有 dirty flag，你每帧都在为不动的物体重复计算矩阵。`is_dirty` + `parent_dirty` 的双重检查是最小且必要的优化。

4. **preserve_world_transform 是编辑器刚需**
   在拖拽 GameObject 改变父子关系时，如果没有这个选项，物体会瞬间跳到新父节点的局部坐标系里。实现方法：先记录旧世界矩阵 → 挂接新父节点 → 用 `world_matrix × inverse(parent_world_matrix)` 反推局部矩阵。

5. **场景图 ≠ 空间加速结构**
   不要把视锥剔除、射线检测逻辑塞进 TransformSystem。场景图只负责"谁是谁的孩子"，空间查询交给专门的模块，在 Transform 更新完毕后执行。
