# 场景图与变换

> **前置依赖**：[[组件系统架构]]、[[数学基础]]
> **本模块增量**：深入理解 ECS 场景图的设计空间——从"能用"到"工业级"的完整分支与决策路径。你将掌握父子层级的组件化表达、局部/世界变换的存储矛盾、拓扑序并行更新、子树脏标记传播、动态挂接语义，以及如何将 OOP 风格的场景图重构为 ECS 架构。
>
> 本笔记探讨的核心问题是：**在 ECS 架构中，如何让实体在三维空间中建立父子关系，并高效、确定性地将局部变换传播为世界变换？** 我们将从多个分支深入分析每种方案的原理、代价与适用边界。

---

## 问题 0：为什么 ECS 世界不能只有孤立的点？

在 [[组件系统架构]] 中，我们让实体可以拥有组件，也能被 System 批量查询。但如果世界里的实体都是彼此孤立的点，引擎只能渲染一堆漂浮在坐标原点附近的方块。没有空间关系，就没有"世界"。

想象一个具体场景：你做了一个角色，角色右手上应该握着一把剑。角色向前奔跑时，剑必须始终挂在手上，跟着一起移动、旋转。如果没有父子层级，你每帧都要在代码里写：

```cpp
// 没有层级时的噩梦
sword.position = character.position + character.rotation * Vec3(0.5f, 1.2f, 0.3f);
sword.rotation = character.rotation * hand_socket_rotation;
```

这不仅繁琐，而且**更新顺序**一旦出错——比如先更新了剑的位置，然后角色才移动——剑就会"抖一帧"，在屏幕上跳一下。这种抖动在 60fps 下极其刺眼。

再比如摄像机跟随玩家。摄像机作为玩家的子节点，其局部偏移是固定的 `(0, 2, -5)`。玩家跳跃时，摄像机应该平滑地一起上升。如果没有世界/局部变换的自动级联，所有跟随逻辑都要硬编码在专门的 CameraSystem 中，无法复用于其他需要跟随的物体（比如宠物、浮空血条）。

第三个场景更底层：批量渲染 10,000 个实体时，渲染系统需要每个实体的**世界矩阵**（4×4）来做视锥剔除和 GPU 上传。如果每个 System 都自己从局部变换临时合成矩阵，不仅重复计算，而且无法利用"场景中 80% 的物体每帧其实根本不动"这一事实。

> **核心需求**：引擎需要一个机制来描述"谁挂在谁下面"，并能**高效、确定性地**将局部变换转换为世界变换，且只计算那些"真的变了"的分支。

这就引出了第一个深层矛盾：**如何用 ECS 的"平铺组件"来表达"树形层级"？**

## 问题 1：如何用 ECS 组件表达父子关系？

树形结构和 ECS 的扁平组件模型天然不对付。树需要指针/引用连接节点，而 ECS 鼓励把所有数据摊平成密集的 POD 数组。我们先看最 naive 的做法，再分析它为什么不够用。

### Naive 方案：独立的 Hierarchy 节点池

这是 OOP 引擎的常见做法——用一个专门的类/池子来维护树结构：

```cpp
struct HierarchyNode {
    Entity   entity;
    Entity   parent;
    std::vector<Entity> children;
    uint32_t rank;
};

std::vector<HierarchyNode> node_pool;
```

每个实体在 `node_pool` 中占一个节点，`children` 数组存着直接子节点。chaos 引擎的 `SceneHierarchy` 和 UE 的 `USceneComponent`（内部的 `AttachChildren` 数组）都是这种思路的变体。

**它解决了什么问题**：树操作非常直观。找子节点读 `children` 数组，循环检测从节点向上遍历 parent，变更父子关系时直接改指针和数组。

**但它带来了什么新问题**：
1. **与 ECS 存储分离**：`node_pool` 是独立分配的内存，ECS 的 `Query<LocalTransform>` 线性遍历不到这里。TransformSystem 更新时，要从 ECS 存储跳转到 `node_pool` 读父子关系，再跳回 ECS 存储写矩阵——两次间接跳转，cache 不友好。
2. **Children 数组的内存碎片**：每个节点的 `children` 是独立的小数组（`std::vector`），散落在堆上。遍历整棵树时，会在 `node_pool` 的连续内存和各个 `children` 小数组之间反复跳转。
3. **双轨同步**：实体创建了 Transform，还要同步创建 HierarchyNode；实体销毁了，node_pool 中的节点也要回收。如果两边不同步，就会出现"有 Transform 但没节点"或"有节点但实体已死"的悬空状态。

### 分支 A：只在子节点存 Parent(Entity)

ECS 最自然的表达方式是：父子关系只是一个**组件**。

```cpp
struct ChildOf {
    Entity parent;
};
```

子节点挂一个 `ChildOf { parent = A }`，就表示"我是 A 的孩子"。没有 `ChildOf` 的实体就是根节点。

**核心思路**：把"树边"摊平成组件数组。ECS 的 `Query<(&ChildOf, &LocalTransform)>` 可以直接遍历所有有父节点的实体。

**适用场景**：层级查询以"子节点找父节点"为主的场景，如 Transform 传播（子节点需要父节点的世界矩阵来计算自己的世界矩阵）。

**隐藏代价**：如果频繁需要"找一个实体的所有子节点"（比如编辑器中选中一个物体要高亮所有后代），就必须做全表扫描——遍历所有 `ChildOf` 组件，找出 `parent == 目标` 的那些。这是 O(n) 的，在 10 万实体场景中不可接受。

**失效条件**：编辑器操作、场景管理、UI 树遍历等需要频繁"父节点查子节点"的场景。

### 分支 B：双向存储——ChildOf + Children

为了解决反向查询问题，可以同时在父节点上维护一个子节点列表：

```cpp
struct ChildOf {
    Entity parent;
};

struct Children {
    Array<Entity> entities;  // 引擎自定义动态数组
};
```

**核心思路**：子节点有 `ChildOf`，父节点有 `Children`，两边同步维护。Bevy 0.16 引入的 **ECS Relationships** 就是这种机制的内建版本——你只需给子节点挂一个 `ChildOf(parent)`，ECS 框架会自动在父节点上维护 `Children` 组件，并保证两边的一致性。

**适用场景**：需要频繁双向遍历的场景，如编辑器、场景序列化、递归销毁子树。

**隐藏代价**：
1. **`Children` 组件大小不固定**：如果父节点的子节点数量变化（如 UI 面板动态添加按钮），`Children` 组件的大小会改变，可能导致实体在不同 Archetype 之间迁移。Archetype 迁移有内存拷贝开销。
2. **一致性维护成本**：如果绕过 ECS 框架直接修改 `ChildOf`，`Children` 就会不同步。Bevy 的 Relationship 系统通过专用 API 和命令缓冲来保证一致性，但默认实现如果手写，容易出 bug。

**失效条件**：子节点数量巨大且频繁变动（如每帧都重组的粒子簇），导致 `Children` 组件和 Archetype 迁移开销成为瓶颈。

### 分支 C：线性化层级——用 Implicit Tree / Depth-First 编码

还有一个被主流忽略但理论上可行的做法：把树编码成一个线性数组，用 DFS 序或 Level-Order 序存储，每个节点只存 `first_child_index` 和 `next_sibling_index`。

```cpp
struct HierarchyNode {
    Entity entity;
    uint32_t first_child;   // 第一个子节点的数组索引，INVALID 表示无子节点
    uint32_t next_sibling;  // 下一个兄弟节点的数组索引
    uint32_t parent;        // 父节点的数组索引
};
```

**核心思路**：整棵树被拍平成一个连续数组，遍历子节点只需沿着 `first_child` → `next_sibling` 链表走。没有散落在堆上的小数组。

**适用场景**：层级结构相对静态、遍历性能极度敏感的场景（如骨骼动画的 100+ 根骨头）。

**隐藏代价**：动态增删节点时需要维护链表指针，操作复杂度从"改一个数组"变成"改三个索引 + 可能的重排"。如果实体 ID 和数组索引脱钩（通过 handle 映射），又引入了额外的 indirection。

**失效条件**：运行时频繁重组层级（如技能系统将召唤物挂接到不同目标）。

### 引擎对照：三端在这个问题上是怎么选的？

> **参考：chaos / UE / Bevy 对「层级存储」是怎么做的？**
>
> - **chaos** 选择了**独立节点池**（分支的 OOP 变体）。`SceneHierarchy` 维护 `HierarchyNode` 池，与 `EntityComponentManager` 的组件存储分离。原因是引擎早期基于 OOP 组件模型，ECS 是后期演进而非原生设计。
> - **UE** 选择了**对象内嵌双向链表**。`USceneComponent` 内部直接持有 `AttachParent` 指针和 `AttachChildren` 数组，每个场景组件就是一个树节点。这是典型的 OOP 继承树方案，在 UObject 垃圾回收体系内工作良好，但无法直接映射到 ECS 的密集存储。
> - **Bevy** 选择了**原生 ECS Relationship**（分支 B 的框架级实现）。`ChildOf` 和 `Children` 是普通的 ECS 组件，但由框架保证双向一致性。Bevy 0.16 的 Relationships 还允许自定义关系类型（如 `Likes`/`LikedBy`），说明这套机制是通用的图/树抽象。
>
> **默认推荐**：**分支 B（双向 Relationship）**。编辑器需要频繁"父查子"，没有 `Children` 组件体验极差。如果子节点数量极少且固定（如武器挂点），可以只在父节点存 `Children` 而子节点不存 `ChildOf`，用约定代替双向同步。但如果使用 Bevy 风格的 ECS 框架，直接采用框架提供的 Relationship 机制是最省心的。

## 问题 2：局部变换 vs 世界变换——该存什么？

层级关系只是"谁挂在谁下面"。真正消耗 CPU 的是**变换计算**：把每个实体的局部位置、旋转、缩放，层层累加成世界空间中的 4×4 矩阵。

这里存在一个根本矛盾：Gameplay 系统喜欢操作**局部变换**（"把剑往右移 0.5 米"，这是相对于角色手的局部坐标），但渲染和物理系统需要**世界变换**（GPU 着色器里的 `world_matrix`，物理引擎里的全局刚体位置）。

### Naive 方案：只存一个 4×4 矩阵

```cpp
struct Transform {
    Mat4 matrix;
};
```

**它有什么问题**：
1. **无法直观编辑**。你想把剑往右移 0.5 米，但矩阵里存的是混合了位置、旋转、缩放的 16 个浮点数。你得先反解出 TRS，修改位置，再重新合成矩阵。数值误差会累积。
2. **父子层级无法表达**。子节点的矩阵应该是 `parent_world × local_matrix`，但"局部"这个概念在只有一个矩阵时很模糊——你修改了矩阵，是改局部还是改世界？

所以工业界共识是：**必须分离局部和世界**。

### 分支 A：只存局部 TRS，世界矩阵按需合成

```cpp
struct LocalTransform {
    Vec3 position;
    Quat rotation;
    Vec3 scale;
};

// 没有 GlobalTransform 组件！
// 需要世界矩阵时：
Mat4 world = parent_world * Mat4::TRS(local.position, local.rotation, local.scale);
```

这是 chaos 引擎的做法。它的 `TransformComponent` 只存 `Pose3D`（TRS），没有持久缓存的 4×4 世界矩阵。需要矩阵时调用 `Pose3D::toMat44()` 实时生成。

**核心思路**：省内存。`Pose3D` 约 40 字节（3×4 + 4×4 + 3×4 = 40 bytes，考虑对齐可能 48 bytes），而一个 `Mat4` 就要 64 字节。对于 50 万实体的开放世界，不缓存矩阵能省几十 MB。

**适用场景**：内存极度紧张的移动平台；实体数量极大但每帧实际需要矩阵的实体很少（如远处的 LOD0 物体被 culled 了）。

**隐藏代价**：
1. **重复计算**。渲染管线可能多次读取同一实体的世界矩阵（视锥剔除一次、渲染提交一次、阴影贴图一次）。每次都要重新 `TRS → Mat4`，CPU 乘法开销叠加。
2. **无法批量 SIMD**。如果世界矩阵不连续缓存，渲染系统无法直接上传一个连续的世界矩阵数组到 GPU。必须先遍历实体、逐个合成矩阵、拷贝到临时上传缓冲区——多了一次线性遍历。

**失效条件**：每帧有上万个实体需要世界矩阵（高画质 PC/主机场景），或骨骼动画密集（每根骨头都需要矩阵上传 GPU）。

### 分支 B：局部 TRS + 世界 Affine3A/Mat4 缓存

```cpp
struct LocalTransform {
    Vec3 translation;
    Quat rotation;
    Vec3 scale;
};

struct GlobalTransform {
    Affine3A affine;  // Bevy 的选择，12 floats，支持 shear
    // 或者 Mat4 matrix;  // 传统选择，16 floats
};
```

这是 Bevy 和 UE 的主流做法。Bevy 使用 `Affine3A`（3×4 的仿射矩阵，12 个 float）而非完整 `Mat4`，因为它更紧凑，且层级叠加非均匀缩放时可能产生 shear，`Affine3A` 能正确表达而 TRS 不能。

**核心思路**：用 64 字节（Mat4）或 48 字节（Affine3A）的内存换取"渲染/物理直接读取连续数组"的性能。

**适用场景**：PC/主机平台，每帧渲染实体数多，骨骼动画密集，需要批量上传矩阵到 GPU。

**隐藏代价**：
1. **内存占用**。每个有 Transform 的实体都多了 48~64 字节。10 万实体就是 4.8~6.4 MB。在 PC 上微不足道，在移动端可能需要考虑。
2. **同步开销**。Gameplay 改了 `LocalTransform`，`GlobalTransform` 必须在同一帧的某个安全点（如 PostUpdate）被更新。如果某个 System 在 TransformSystem 之后读取 `GlobalTransform`，然后在同一帧又修改了 `LocalTransform`，就会有一帧延迟。

**失效条件**：内存预算极其紧张（如 32MB 总内存的掌机），或几乎没有需要世界矩阵的消费者（如纯服务器逻辑，不渲染）。

### 分支 C：只存世界矩阵，局部通过逆矩阵反推

```cpp
struct GlobalTransform {
    Mat4 matrix;
};
```

Gameplay 想改局部位置时，先算 `parent_world.inverse()`，再乘以目标世界矩阵，反解出局部 TRS。

**核心思路**：砍掉 LocalTransform，世界矩阵是唯一真相。

**隐藏代价**：
1. **数值不稳定**。父节点有非均匀缩放时，逆矩阵可能条件数极差，反推的局部位置会有明显误差。
2. **无法唯一分解**。世界矩阵如果包含 shear（多个非均匀缩放父节点叠加的结果），无法唯一分解为 TRS。Bevy 的 `GlobalTransform` 就明确警告：`.compute_transform()` "may be invalid"。

**失效条件**：任何含非均匀缩放的层级，或需要精确局部编辑的编辑器场景。

### 决策分析

> **默认推荐**：**分支 B（LocalTransform + GlobalTransform）**。
>
> 理由：默认实现的首要瓶颈不是内存，而是"能不能快速看到结果"。分支 B 让渲染、物理、动画系统都能直接读取连续缓存的世界变换，SIMD 和 GPU 上传都更高效。Bevy 的 `Affine3A` 是比 `Mat4` 更优的选择——它更紧凑（48 vs 64 字节），且能正确表达层级叠加后的 shear，避免了"非均匀缩放父节点下子节点变换异常"的隐藏 bug。
>
> **切换条件**：如果目标平台是移动端或 WebAssembly，且场景中有 50 万+ 大部分是静态的实体，可切换为分支 A（不缓存矩阵），但必须在渲染阶段做批量矩阵合成缓存，避免重复计算。

## 问题 3：如何按正确顺序更新，还能并行？

父子层级引入了一个致命的依赖关系：**父节点的世界矩阵必须在子节点之前计算完成**。如果子节点先算，它读到的父矩阵是上一帧的，结果就是"抖一帧"。

### Naive 方案：每帧递归算 depth，然后 std::sort

```cpp
uint32_t getDepth(Entity e, const std::vector<ChildOf>& parents) {
    uint32_t d = 0;
    while (e != INVALID_ENTITY) {
        e = parents[e].parent;
        ++d;
        if (d > 100) break; // 防循环
    }
    return d;
}

// update() 中：
std::vector<std::pair<uint32_t, Entity>> ranked;
for (Entity e : all_entities) ranked.push_back({getDepth(e, parents), e});
std::sort(ranked.begin(), ranked.end(), /*按 depth*/);
```

**它的问题**：
1. **每帧 O(n log n)**。即使没有任何实体移动，也要重新排序。
2. **递归算 depth 是 O(树深)**。对于 100 根骨骼的链，每个实体要向上走 100 步。10 万实体就是 1000 万次跳转。
3. **cache 灾难**。排序后的遍历顺序是随机的（按 depth，不是按 EntityID），ECS 的连续内存遍历优势被彻底破坏。

### 分支 A：维护 rank 字段，变更时 BFS 增量更新

不再每帧算 depth。给每个实体一个 `rank`（层级深度），仅在父子关系变更时用 BFS 更新受影响子树的 rank：

```cpp
struct HierarchyNode {
    Entity entity;
    uint32_t rank;   // 0 = 根节点
    // ...
};

void onParentChanged(Entity child, Entity new_parent) {
    uint32_t new_rank = (new_parent == INVALID) ? 0 : nodes[new_parent].rank + 1;
    
    // BFS 更新整个子树
    std::queue<Entity> q;
    q.push(child);
    nodes[child].rank = new_rank;
    
    while (!q.empty()) {
        Entity e = q.front(); q.pop();
        for (Entity c : children_of(e)) {
            nodes[c].rank = nodes[e].rank + 1;
            q.push(c);
        }
    }
}
```

chaos 引擎的 `SceneHierarchy` 和 UE 的 `USceneComponent` 都维护了一个隐式的层级深度，用于确定更新顺序。

**核心思路**：用增量维护替代每帧全量计算。

**适用场景**：层级结构相对静态，变更不频繁的场景。

**隐藏代价**：
1. **深层树（100+ 级骨骼）的 BFS 级联开销**。移动一个根骨骼，要更新 100 个骨头的 rank。
2. **需要显式维护 children 列表**。没有 children 列表，BFS 无法继续。

### 分支 B：按 rank 分桶，同层并行

维护 rank 的目的不是为了"每帧排序"，而是为了"分桶"。把相同 rank 的实体放进同一个数组：

```cpp
std::vector<std::vector<Entity>> rank_buckets;

// 变更时直接移桶
void setRank(Entity e, uint32_t new_rank) {
    uint32_t old_rank = entity_rank[e];
    remove_from_bucket(old_rank, e);
    add_to_bucket(new_rank, e);
    entity_rank[e] = new_rank;
}

// 更新时按桶顺序遍历，桶内并行
void updateTransforms() {
    for (auto& bucket : rank_buckets) {
        parallel_for(bucket, [](Entity e) {
            // 父节点已在更小的 rank 中更新完毕
            update_global_transform(e);
        });
    }
}
```

**核心思路**：同一桶内的实体之间没有父子依赖（它们的父节点都在更小的 rank 中），可以安全并行。chaos 的 `TransformManager` 按 rank 分批后，用 `g_task_executor` 将同一 depth 的节点拆成 250 个一批的任务并行执行。Bevy 0.16 则进一步优化为**跨树并行 + leaf/non-leaf 拆分**，优化 cache 局部性。

**适用场景**：需要并行化的大型场景，有成熟的任务调度系统。

**隐藏代价**：
1. **桶的数量 = 最大深度**。如果树很扁（深度 2，宽度 50000），只有 2 个桶，第二个桶有 50000 个实体，并行粒度虽然够粗，但第一个桶只有几个根节点，会成为并行瓶颈。
2. **桶维护开销**。实体频繁变更 rank 时，移桶操作是 O(1)，但需要额外的数组管理。

**失效条件**：树极度不平衡（一个根节点下挂 10 万子节点），导致只有一个大桶能并行，根节点桶成为串行瓶颈。

### 分支 C：利用 ECS Scheduler 的隐式阶段（Bevy 做法）

Bevy 没有显式的 rank 桶，而是通过 ECS 的 **System Schedule** 来保证顺序：

```rust
// Bevy 的 Transform 系统运行在 PostUpdate 阶段
app.add_systems(PostUpdate, (
    mark_dirty_trees,               // 向上传播脏标记
    propagate_parent_transforms,    // 父→子传播矩阵
    sync_simple_transforms,         // 无父节点的实体直接复制
).chain());
```

所有修改 `Transform` 的 System 都在 `Update` 阶段运行。`PostUpdate` 保证它们全部完成后，Transform 传播才开始。Bevy 的 ECS 调度器会自动处理 System 依赖和并行——`propagate_parent_transforms` 内部通过查询 `ChildOf` 组件来遍历，ECS 以 Archetype Chunk 为单位分配并行任务。

**核心思路**：把"拓扑序"问题交给 ECS 框架的调度阶段，而不是在 TransformSystem 内部手写排序。

**适用场景**：使用 Bevy 或自研的、支持 System 阶段和依赖声明的 ECS 调度器。

**隐藏代价**：
1. **需要框架支持**。如果你的 ECS 没有 PostUpdate / SystemSet 概念，就无法使用。
2. **一帧延迟风险**。如果某个 System 错误地注册在 `PostUpdate` 之后修改 `Transform`，它的变更要到下一帧的 `PostUpdate` 才会被传播到 `GlobalTransform`。

### 引擎对照

> **参考：chaos / UE / Bevy 对「更新顺序与并行」是怎么做的？**
>
> - **chaos** 选择了**显式 rank 分桶 + 任务调度器并行**。`TransformManager::update()` 将待更新实体按 `m_update_rank` 排序，统计每个 rank 的数量，然后逐个 rank 调用 `updateEntitiesInRank()`，内部使用 `g_task_executor` 按 250 个实体一批拆成并行任务。这是典型的"手写并行"方案。
> - **UE** 选择了**递归传播 + 延迟更新**。`PropagateTransformUpdate` → `UpdateChildTransforms` 是递归调用，不是批量并行。UE 的优化点是 `ScopedMovement`——批量变换更新期间推迟传播，避免中间状态的多次递归。UE 的渲染线程也会通过 `MarkRenderTransformDirty` 把变换计算和渲染提交解耦。
> - **Bevy** 选择了**ECS Scheduler 隐式阶段 + Chunk 并行**。Transform 传播是一个普通的 ECS System，Bevy 的调度器自动按 Archetype Chunk 分配并行任务。Bevy 0.16 额外引入了 leaf/non-leaf 拆分和跨树并行策略，使得即使在深层树中也能充分利用多核。
>
> **默认推荐**：
> - 如果使用自研 C++ ECS（无高级 Scheduler），采用**分支 B（显式 rank 分桶 + parallel_for）**。实现简单，不依赖特定框架。
> - 如果使用 Bevy 风格的 ECS（有 System Stage），采用**分支 C（PostUpdate 阶段 + ECS 内部并行）**，代码更简洁。

## 问题 4：静态物体占 80%，如何避免为不动的物体重复计算？

这是场景图性能优化中最重要的问题，也是 Bevy 0.16 实现 11 倍性能提升的核心。

### Naive 方案：每帧全量更新所有 GlobalTransform

不管动没动，遍历所有有层级的实体，重新计算世界矩阵。

**量化代价**：Bevy 团队在 Caldera Hotel 场景（来自《使命召唤：战区》，127,515 个物体）上测试，全量更新在 M4 Max MacBook 上耗时 **1.1 ms**。这听起来不多，但在中低端设备上可能吃掉整整 **25% 的帧预算**（以 60fps、16.6ms 帧时间计算，就是 4ms+）。

### 分支 A：实体级 dirty flag（is_dirty）

```cpp
struct GlobalTransform {
    Mat4 matrix;
    bool is_dirty = true;
};

void update(Entity e) {
    bool parent_dirty = (has_parent(e)) ? globals[parent(e)].is_dirty : false;
    if (!globals[e].is_dirty && !parent_dirty) return;
    
    // 重新计算...
    globals[e].is_dirty = false;
}
```

**核心思路**：实体自身没变（`is_dirty == false`）且父节点也没变，就跳过。

**隐藏代价**：对于深层静态子树，仍需**逐层检查** `parent_dirty`。想象一棵深度 10 的静态树：根节点干净 → 检查子节点（父脏？否）→ 检查孙节点（父脏？否）……虽然最终跳过了矩阵计算，但 10 次分支判断和内存读取仍然发生。如果这棵树有 1000 棵（静态建筑群的 1000 个实例），就是 1000 × 10 = 10000 次无效检查。

### 分支 B：子树级 dirty flag，向上传播（Bevy 0.16 的突破性优化）

Bevy 0.16 引入了一个 ZST（Zero-Sized Type）标记组件 `TransformTreeChanged`，以及一个关键优化：**dirty bit 向上传播**。

机制分为两个阶段：

**阶段 1：mark_dirty_trees（向上传播）**

当任意实体的 `Transform` 发生变化时，将 `TransformTreeChanged` 组件沿 `ChildOf` 链**向上**传播到根节点：

```
初始状态：
        A(根)
       /   \
      B     C
     / \
    D   E

修改 E.Transform：
1. E 获得 TransformTreeChanged
2. 向上传播：B 获得 TransformTreeChanged
3. 向上传播：A 获得 TransformTreeChanged

最终 dirty 标记：
        A(dirty)
       /        \
      B(dirty)   C(干净)
     /     \
    D(?)    E(dirty)
```

注意 D 没有直接获得 dirty 标记！这是关键。

**阶段 2：propagate_parent_transforms（向下传播矩阵）**

更新系统从根节点开始向下遍历：

```
遍历到 A（有 dirty）→ 更新 A，继续遍历子树
遍历到 B（有 dirty）→ 更新 B，继续遍历子树
遍历到 D（无 dirty）→ 【跳过整个 D 子树】
遍历到 E（有 dirty）→ 更新 E
遍历到 C（无 dirty）→ 【跳过整个 C 子树】
```

**核心突破**：不是"跳过单个实体"，而是**跳过整棵子树**。对于 127k 物体的静态场景，这意味着绝大部分子树被直接忽略，无需任何检查。

**量化收益**：Caldera 场景 1.1 ms → **0.1 ms**，**11 倍提升**。在 60fps 下，这意味着节省了 **6% 的整帧 CPU 预算**。

**适用场景**：静态物体占主体的任何场景（建筑、地形、道具）。

**隐藏代价**：
1. **向上传播本身是写操作**。如果一帧内修改了 1000 个实体的 `Transform`，就要沿 Parent 链向上写 1000 × 平均深度 次 dirty 标记。在"全动态场景"（如粒子系统每帧全动）中，向上传播无法跳过任何节点，收益降为零，但传播开销仍在。
2. **额外一个组件**。`TransformTreeChanged` 是 ZST，不占内存，但 ECS 的 archetype 匹配会多一个条件。

**失效条件**：全动态场景，或极扁平的层级（所有实体都是根节点的直接子节点，深度为 1）。在深度为 1 的情况下，向上传播只有一个步骤，"跳过子树"的收益很小。

### 分支 C：隐式脏标记（ECS Change Detection）

Bevy 的 ECS 本身有 change detection 机制：`Query<Changed<Transform>>` 可以只遍历本帧被修改过 `Transform` 的实体。

**核心思路**：利用 ECS 框架自带的 change detection，无需手动维护 dirty 标记。

**隐藏代价**：
1. **父节点移动不会触发子节点的 `Changed<Transform>`**。子节点的 `Transform` 组件本身没有被修改，但它的 `GlobalTransform` 必须更新。纯 change detection 会漏掉这种情况。
2. **有一帧延迟风险**。某些 ECS 的 change detection 在 System 执行后才标记，导致本轮读取不到本轮的修改。

**失效条件**：任何有父子层级的场景（这正是场景图的核心场景）。

### 决策分析

> **默认推荐**：**分支 B（子树级 dirty flag，向上传播）**。
>
> 这是 Bevy 0.16 用真实 127k 场景验证过的方案，实现极其简单（一个 ZST marker + 一次向上遍历），收益巨大。全动态场景下它自然退化成全量更新，没有负收益。
>
> **状态变化图：dirty bit 向上传播**
>
> ```
> 变更前（所有节点干净）：
>         A
>       /   \
>      B     C
>     / \
>    D   E
>
> 执行：set E.local_position += Vec3(1,0,0)
>
> 步骤 1：E 添加 TransformTreeChanged
>         A
>       /   \
>      B     C
>     / \
>    D   E[*]
>
> 步骤 2：沿 Parent 链向上传播
>         A[*]
>       /      \
>      B[*]     C
>     /   \
>    D     E[*]
>
> 更新阶段遍历行为：
> - A 有 [*] → 更新 A，继续向下
> - B 有 [*] → 更新 B，继续向下
> - D 无 [*] → 【剪枝，跳过 D 及其所有后代】
> - E 有 [*] → 更新 E
> - C 无 [*] → 【剪枝，跳过 C 及其所有后代】
> ```
>
> 注意：D 虽然最终也需要正确的世界矩阵，但由于 D 的 `Transform` 本帧未变，且 B 的旧世界矩阵也未变（B 的 `Transform` 本帧未变，只是 dirty 标记被 E 的传播点亮），等一下——这里有一个细节：B 的 `GlobalTransform` 本帧是否需要更新？
>
> 答案是：**如果 B 的 `Transform` 没变，B 的 `GlobalTransform` 也不需要重新计算**。`TransformTreeChanged` 只是告诉传播系统"这个子树里有人动了，请继续向下检查"，而不是"这个节点本身需要重算"。B 被标记 dirty 后，传播系统会继续检查 B 的子节点，发现 D 没有 dirty 就跳过，发现 E 有 dirty 就更新 E。B 自己的 `GlobalTransform` 如果未变，可以直接复用缓存。这是这个方案精妙的地方：dirty 标记是**遍历许可**，不是**重算指令**。

## 问题 5：动态改父子关系时，物体为什么突然"瞬移"？

这是编辑器体验和运行时动态挂接的核心问题。

### 场景绑定

你在编辑器里选中一个立方体，把它从"场景根"拖拽到"玩家角色"下面。拖拽前，立方体在世界坐标 `(10, 0, 0)`。拖拽后，如果系统直接把立方体当成角色的子节点，并用它原来的局部坐标 `(10, 0, 0)` 来解释，那它在世界中的位置就变成了 `player_world * (10, 0, 0)`——可能瞬间跳到 `(100, 50, 20)`，在屏幕上"瞬移"了。

用户期望的是：**拖拽前后，物体在屏幕上的位置不变**。

### 根因分析

局部坐标系切换。原父节点（根）下的 `(10, 0, 0)` 和新父节点（角色）下的 `(10, 0, 0)` 在世界中完全不是同一个点。变换的本质是"相对关系"，相对对象变了，同一个数字代表的含义就变了。

### 分支 A：KeepRelative——保持局部 TRS 不变

```cpp
void setParentKeepRelative(Entity child, Entity new_parent) {
    // 只改 Parent 引用，LocalTransform 完全不动
    child.local_transform = child.local_transform;  // 不变！
    child.parent = new_parent;
    child.global_transform.is_dirty = true;
}
```

**适用场景**：运行时逻辑挂接。比如把剑挂到角色手上，剑的局部偏移 `(0.3, -0.2, 0.1)` 是美术设计好的，不需要随世界位置变化。

**隐藏代价**：编辑器中会导致"瞬移"，用户体验极差。

### 分支 B：KeepWorld / preserve_world_transform——保持世界坐标不变

```cpp
void setParentKeepWorld(Entity child, Entity new_parent) {
    // 1. 记录当前世界矩阵
    Mat4 old_world = child.global_transform.matrix;
    
    // 2. 获取新父节点的当前世界矩阵
    Mat4 parent_world = new_parent.global_transform.matrix;
    
    // 3. 反推局部矩阵：local = inverse(parent_world) * old_world
    Mat4 local = inverse(parent_world) * old_world;
    
    // 4. 从局部矩阵分解出 TRS（可能不唯一）
    Vec3 t; Quat r; Vec3 s;
    decompose(local, &t, &r, &s);
    
    child.local_transform = {t, r, s};
    child.parent = new_parent;
    child.global_transform.is_dirty = true;
}
```

UE 的 `FAttachmentTransformRules::KeepWorld` 和 Bevy 的 `PushChildInPlace` / `RemoveParentInPlace` 命令都是这个语义。

**适用场景**：编辑器拖拽、运行时动态重组层级但要求视觉连续性（如技能将敌人吸到手中）。

**隐藏代价**：
1. **非均匀缩放的父节点会导致异常**。如果父节点的缩放是 `(2, 1, 1)`，那么 `inverse(parent_world)` 会包含 `(0.5, 1, 1)` 的缩放。子节点原来的世界旋转在反推局部旋转时会被"吸收"进缩放中，导致局部缩放不再是 `(1,1,1)`，局部旋转也可能扭曲。
2. **Shear 无法分解**。如果父节点链叠加后产生了 shear（非刚性变换），`inverse(parent) * world` 的结果无法用 TRS 精确表达，decompose 会给出近似值，引入数值误差。

**失效条件**：父节点有非均匀缩放或 shear，且对局部 TRS 的精确性有要求（如物理碰撞体）。

### 分支 C：SnapToTarget——对齐到父节点的 Socket/锚点

```cpp
void attachToSocket(Entity child, Entity parent, SocketID socket) {
    // 直接把子节点的局部 TRS 设为目标 Socket 的局部偏移
    child.local_transform = socket.local_offset;
    child.parent = parent;
}
```

UE 的 `FAttachmentTransformRules::SnapToTarget` 和骨骼挂点系统（`AttachToComponent(InSocketName)`）就是这个做法。

**适用场景**：武器挂到角色骨骼的"WeaponSocket"、UI 元素挂到面板锚点。

**隐藏代价**：完全覆盖子节点的原有变换，不适合需要保持原位的编辑器场景。

### 引擎对照

> **参考：chaos / UE / Bevy 对「动态挂接语义」是怎么做的？**
>
> - **chaos** 的 `setParentHandle(parent_handle, preserve_global_pose)` 直接支持分支 B。`preserve_global_pose == true` 时调用 `tryUpdateLocalPoseFromGlobalPose()`，用数学公式反推局部 Pose。
> - **UE** 提供了最完整的语义：`FAttachmentTransformRules` 有 `KeepRelative`、`KeepWorld`、`SnapToTarget` 三种。`SnapToTarget` 还可以指定 `InSocketName`，挂接到骨骼 Socket。UE 的 `FDetachmentTransformRules` 也有 `KeepRelative` / `KeepWorld`，Detach 时同样避免瞬移。
> - **Bevy** 通过命令扩展支持分支 B：`PushChildInPlace` 和 `RemoveParentInPlace` 命令会在执行时自动调整子节点的 `Transform`，使其 `GlobalTransform` 保持不变。Bevy 0.16 的 Relationships 让这种命令更易于实现。
>
> **默认推荐**：API 必须**同时暴露分支 A 和分支 B**。`set_parent(child, parent)` 默认 **KeepRelative**（运行时逻辑的直觉行为）；提供显式的 `set_parent_in_place(child, parent)` 用于 **KeepWorld**（编辑器刚需）。分支 C（SnapToTarget）可作为高级 API 在骨骼/动画模块中单独提供。
>
> **关于非均匀缩放的警告**：如果你的引擎允许非均匀缩放（如 `(2, 0.5, 1)`），`KeepWorld` 反推的局部变换可能包含异常缩放。建议在编辑器中检测到这种情况时给出警告，或在文档中明确说明"非均匀缩放父节点下的 KeepWorld 是近似结果"。

## 问题 6：物理、渲染、动画都要读取 Transform，如何解耦？

场景图不是孤岛。Transform 更新完毕后，有一堆消费者等着用它。

### OOP 方案：虚回调 + 委托广播

UE 的做法是典型的 OOP 事件驱动：

```cpp
void USceneComponent::PropagateTransformUpdate(bool bTransformChanged, ...) {
    if (bTransformChanged) {
        UpdateBounds();
        OnUpdateTransform(...);           // 虚函数！子类（如 PrimitiveComponent）重写
        TransformUpdated.Broadcast(...);  // 委托广播，任何对象可以注册监听
        MarkRenderTransformDirty();       // 标记渲染脏
        UpdateChildTransforms();          // 递归子节点
        UpdateNavigationData();           // 导航数据更新
    }
}
```

`UPrimitiveComponent::OnUpdateTransform` 重写中同步物理引擎：`SendPhysicsTransform(Teleport)`。

**核心思路**：Transform 变更后，主动通知所有消费者。

**隐藏代价**：
1. **虚函数调用开销**。每个子节点更新都要走一遍虚函数表，cache 不友好。
2. **消费者被硬编码在传播链中**。想新增一个消费者（比如 AI 感知系统），必须改 `PropagateTransformUpdate` 或注册委托——耦合了所有模块。
3. **中间状态多次传播**。如果一帧内先改位置、再改旋转、再改缩放，OOP 方案可能触发三次传播。UE 用 `ScopedMovement` 延迟到作用域结束才统一传播，但这增加了复杂度。

### ECS 方案：Pipeline 阶段 + Query 按需读取

Bevy 的做法是数据驱动的：

```rust
// 阶段划分（简化）
app.add_systems(Update, player_movement);      // 修改 Transform
app.add_systems(PostUpdate, (
    mark_dirty_trees,
    propagate_parent_transforms,   // TransformSystem 在这里
    sync_simple_transforms,
).chain());

// 渲染系统、物理系统都在 PostUpdate 或之后
app.add_systems(PostUpdate, extract_meshes);   // 读取 GlobalTransform
```

**核心思路**：不通知消费者，而是让消费者在正确的**阶段**通过 `Query<&GlobalTransform>` 按需读取。TransformSystem 保证在它执行完毕后，所有 `GlobalTransform` 都是最新的。

**隐藏代价**：
1. **阶段设计必须正确**。如果某个 System 错误地注册在 TransformSystem 之前读取 `GlobalTransform`，就会拿到旧数据。这是"约定优于通知"的代价——开发者必须理解 Pipeline 顺序。
2. **一帧延迟风险**。如果物理系统在本帧修改了刚体位置，希望反馈到 `Transform`，必须等到下一帧的 `Update` 阶段才能修改 `Transform`，再等到 `PostUpdate` 才能传播到 `GlobalTransform`——最多两帧延迟。

### 深层问题：物理和 Transform 的双向反馈

物理引擎（如 PhysX、Jolt）内部有自己的刚体变换。如果物理 System 在 `PostUpdate` 阶段运行，它读取的是本帧最新的 `GlobalTransform`。但物理模拟后，刚体位置可能变了——这个变化怎么反馈回 ECS？

**分支 A：物理输出到 DesiredPose，下帧应用**
物理 System 把新的刚体位置写入一个 `DesiredPose` 或 `PhysicsOutput` 组件，下一帧的 TransformSystem 在 `Update` 阶段把它应用到 `LocalTransform`。chaos 的 `DesiredPoseMap` 就是这个思路。

**分支 B：物理直接写 GlobalTransform，TransformSystem 逆推 LocalTransform**
物理 System 直接修改 `GlobalTransform`，然后触发一次逆向传播（`global → local`）。这在 ECS 中不太自然，因为 `GlobalTransform` 是"派生数据"，不应该被直接修改。

**默认推荐**：**分支 A**。物理和 Transform 的同步延迟一帧在大多数游戏中可接受（渲染也是延迟一帧的）。关键是把物理输出显式地放在一个中间组件里，而不是偷偷改 `GlobalTransform`。

### 决策分析

> **默认推荐**：**ECS 方案（Pipeline 阶段 + Query 按需读取）**。
>
> 彻底摒弃虚回调。TransformSystem 在 `PostUpdate` 阶段执行完毕后，渲染、物理、动画等消费者通过各自的 System 读取 `GlobalTransform`。新增消费者只需新增一个 System，无需改动 TransformSystem 的核心代码。
>
> Pipeline 阶段建议：
> - `PreUpdate`：输入处理、AI 决策
> - `Update`：Gameplay 修改 `LocalTransform`、物理输出 DesiredPose
> - `PostUpdate`：**TransformSystem**（mark_dirty → propagate → sync）
> - `RenderExtract`：从 `GlobalTransform` 提取渲染数据

## 问题 7：从 OOP 到 ECS 的重构映射

chaos 和 UE 的场景图都是 OOP 风格的。如果我们从零开始用 ECS，或者需要把 OOP 代码重构为 ECS，该怎么映射？

### 核心映射表

| OOP 源码结构 | ECS 等价物 | 迁移说明 |
|-------------|-----------|---------|
| `SceneHierarchy` + `HierarchyNode` 池 | **`ChildOf` 组件** + 可选 **`Children` 组件** | 不再维护独立的 hierarchy 节点池。树结构完全由 ECS 组件的有无决定，存储在 Archetype Chunk 中。 |
| `TransformComponent`（含 `m_parent_handle`、`m_local_pose`、`m_pose`） | **`LocalTransform`** + **`GlobalTransform`** + **`ChildOf`** 三个独立组件 | 将 TRS、世界变换、父子引用拆成三个 POD 组件。静态物体可以只有 `GlobalTransform`（无 `LocalTransform` 和 `ChildOf`），节省内存。 |
| `TransformManager::calculateAndUpdateTransform()` | **`TransformSystem`** | 一个普通的 ECS System，通过 `Query<(&LocalTransform, &mut GlobalTransform, &ChildOf)>` 批量更新。不需要是特殊的 Manager 子类。 |
| `DesiredPoseMap` + `PendingUpdateEntityMap` | **命令缓冲（Command Buffer）** + **`DesiredPose` 组件** | 外部系统想修改 Transform 时，写入 `Commands` 或附加 `DesiredPose` 组件，在 TransformSystem 的安全点统一应用。 |
| `getAllChildren()` 递归收集 | **ECS Query `&Children`** 或 **Relationship 查询** | Bevy 的 `Children` 组件直接提供子节点列表；如果没有 `Children`，可通过 `Query<&ChildOf>` 反向扫描（适用于低频查询）。 |
| `Pose3D::toMat44()` 按需合成 | **预计算的 `GlobalTransform`** | ECS 中优先缓存世界变换。如果内存极度紧张，可保留按需合成，但需在渲染阶段做批量缓存。 |
| `USceneComponent::OnUpdateTransform` 虚回调 | **Pipeline 阶段划分** | 渲染、物理不再被回调通知，而是在 TransformSystem 之后的阶段通过 Query 读取 `GlobalTransform`。 |
| `FAttachmentTransformRules` | **`set_parent` API 的参数枚举** | `AttachRules::KeepRelative`、`KeepWorld`、`SnapToTarget` 映射为函数参数或命令变体。 |
| `bAbsoluteLocation/Rotation/Scale` | **可选的 `AbsoluteTransform` 标记组件** | UE 中某些组件（如 UI、特效）不需要跟随父级旋转/缩放。在 ECS 中可附加 `AbsoluteTransform` 组件，TransformSystem 传播时跳过对应通道。 |

### Why ECS 更适合这个模块？

1. **数据连续**：chaos 的 `TransformManager` 需要通过 `EntityComponentManager` 的多层哈希表逐个获取 `TransformComponent`。ECS 的 `Query<LocalTransform, GlobalTransform, ChildOf>` 可以直接线性遍历匹配的 Archetype Chunk，cache 友好得多。
2. **并行天然**：同一 rank 的节点在 chaos 中需要显式用 `g_task_executor` 分批任务化。在 ECS 中，只要 `TransformSystem` 只写 `GlobalTransform`、读 `LocalTransform` 和 `ChildOf`，调度器就可以自动按 Chunk 并行，无需手写任务拆分。
3. **组合灵活**：chaos 中每个实体必须有且只有一个 `TransformComponent`。ECS 中，静态障碍物可以只有 `GlobalTransform`（没有 `LocalTransform` 和 `ChildOf`），进一步节省内存和计算。UI 元素可以只有 `LocalTransform` 和 `GlobalTransform` 但没有旋转——任意组合。

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ `LocalTransform`、`GlobalTransform`、`ChildOf` 都是 POD 组件。Hierarchy 结构完全由组件的有无决定，没有隐藏状态。 |

## 默认推荐路径总结

### 默认推荐路径
- **层级存储**：`ChildOf` + `Children` 双向 Relationship。子节点存 `ChildOf(parent)`，父节点由框架自动维护 `Children` 列表。子节点极少且静态时可退化为仅 `ChildOf`。
- **变换存储**：`LocalTransform`（TRS）+ `GlobalTransform`（`Affine3A`，48 字节）。Gameplay 系统只写 `LocalTransform`，渲染/物理/动画只读 `GlobalTransform`。
- **更新顺序**：自研 ECS 采用显式 `rank` 分桶 + 同层 `parallel_for`；Bevy 风格 Scheduler 使用 `PostUpdate` 阶段链式执行（`mark_dirty_trees` → `propagate_parent_transforms` → `sync_simple_transforms`）。
- **脏标记**：`TransformTreeChanged` ZST 组件，变更时沿 `ChildOf` 链向上传播到根。更新阶段从根向下遍历，无 dirty 标记的子树直接剪枝。
- **动态挂接**：`set_parent(child, parent)` 默认 `KeepRelative`；`set_parent_in_place(child, parent)` 显式 `KeepWorld`，编辑器拖拽刚需。
- **消费者解耦**：Pipeline 阶段驱动。`PostUpdate` 阶段完成 Transform 传播后，渲染/物理/动画在后续阶段通过 `Query<&GlobalTransform>` 按需读取，彻底替代虚回调。
- **物理同步**：物理 System 输出到 `DesiredPose` 中间组件，下帧 `Update` 阶段应用到 `LocalTransform`，避免直接修改派生的 `GlobalTransform`。

### UE 工业级设计吸收
- **`FAttachmentTransformRules` 语义完整吸收**：`KeepRelative`、`KeepWorld`、`SnapToTarget` 三种挂接模式映射为 API 参数或命令变体，编辑器体验与 UE 对齐。
- **`ScopedMovement` 延迟传播思想**：批量变换更新期间推迟传播，避免中间状态多次触发。ECS 中通过命令缓冲 + Pipeline 阶段实现同等效果。
- **`MarkRenderTransformDirty` 的脏标记解耦**：UE 把"变换计算"和"渲染提交"通过脏标记解耦到不同线程。ECS 中对应为 `PostUpdate` 的 TransformSystem 和 `RenderExtract` 阶段的提取 System。
- **USceneComponent 的 `Absolute` 标记**：UE 中某些组件（UI、特效）可选择不跟随父级旋转/缩放。ECS 中映射为可选的 `AbsoluteTransform` 标记组件，TransformSystem 传播时跳过对应通道。

### ECS 映射
- **`SceneHierarchy` + `HierarchyNode` 池** → **`ChildOf` 组件** + 可选 **`Children` 组件**。树结构完全由 ECS 组件的有无决定，存储在 Archetype Chunk 中，不再维护独立节点池。
- **`TransformComponent`（含 `m_parent_handle`、`m_local_pose`、`m_pose`）** → **`LocalTransform`** + **`GlobalTransform`** + **`ChildOf`** 三个独立组件。TRS、世界变换、父子引用拆成三个 POD 组件；静态障碍物可只有 `GlobalTransform`，进一步节省内存。
- **`TransformManager::calculateAndUpdateTransform()`** → **`TransformSystem`**。普通 ECS System，通过 `Query<(&LocalTransform, &mut GlobalTransform, &ChildOf)>` 批量更新，无需 Manager 子类。
- **`DesiredPoseMap` + `PendingUpdateEntityMap`** → **命令缓冲** + **`DesiredPose` 组件**。外部系统写入 `Commands` 或附加 `DesiredPose`，在 TransformSystem 的安全点统一应用。
- **`USceneComponent::OnUpdateTransform` 虚回调** → **Pipeline 阶段划分**。渲染、物理不再被回调通知，而是在 TransformSystem 之后的阶段通过 Query 读取 `GlobalTransform`。
- **`FAttachmentTransformRules`** → **`set_parent` API 的参数枚举**。`KeepRelative`、`KeepWorld`、`SnapToTarget` 映射为函数参数或命令变体。

### 条件切换路径
| 条件 | 切换方案 | 触发信号 |
|------|---------|---------|
| 内存预算极其紧张（移动端 / WebAssembly，50 万+ 静态实体） | `GlobalTransform` 缓存 → 按需合成 | 内存 Profiler 显示 Transform 占用超标 |
| 全动态场景（粒子系统、物理刚体每帧全动） | 子树脏标记 → 全量更新 | 脏标记传播开销 > 全量更新 50% |
| 层级结构极度扁平（深度 ≈ 1，宽度 10 万+） | 子树 dirty → 实体级 `is_dirty` | 向上传播无剪枝收益 |
| 使用 Bevy 风格 ECS Scheduler | 显式 rank 分桶 → `PostUpdate` 阶段链 | 已有 Stage / SystemSet 基础设施 |
| 运行时频繁重组层级（技能系统、动态 UI） | `ChildOf` + `Children` → 线性化 Implicit Tree | Archetype Churn 成为瓶颈 |

### 扩展路径
- **跨树并行 + leaf/non-leaf 拆分**：Bevy 0.16 的进一步优化，同一 rank 内不同子树可并行，叶子节点和非叶子节点分 kernel 处理，提升 cache 局部性。
- **BVH / Octree 空间加速结构**：场景图只负责父子层级，空间查询（视锥剔除、射线检测）交给独立的空间索引模块，在 Transform 更新完毕后执行。
- **骨骼动画专用层级优化**：骨骼链（100+ 深度）用线性化数组 + `first_child` / `next_sibling` 编码，避免 Archetype 因 `Children` 大小变化而频繁迁移。
- **网络同步的 Transform 插值**：客户端接收服务器 authoritative 的 `GlobalTransform`，本地不直接覆盖，而是写入 `TargetGlobalTransform`，由插值 System 平滑过渡，消除抖动。

### "如果我要 vibe coding，该偷哪几招？"

1. **EntityID + ChildOf 组件 = 最小 hierarchy**
   不要一上来就写四叉树、八叉树、BVH。先让 `ChildOf(Entity)` 组件跑起来，90% 的游戏场景都够用。

2. **rank 分桶比每帧排序快 10 倍**
   不要在每帧的 `System.update()` 里 `std::sort` 所有实体。在挂接/解挂时直接把实体移入对应 rank 的 bucket，更新时按 bucket 顺序遍历即可。

3. **子树脏标记是性能的生命线**
   Bevy 0.16 的 `TransformTreeChanged` 向上传播方案，让 127k 静态场景的 transform 开销从 1.1ms 降到 0.1ms。实现成本极低（一个 ZST 组件 + 一次向上遍历），是默认实现最值得优先吸收的一招。

4. **`set_parent_in_place` 是编辑器刚需**
   在拖拽 GameObject 改变父子关系时，如果没有 KeepWorld 选项，物体会瞬间跳到新父节点的局部坐标系里。实现方法：先记录旧世界矩阵 → 挂接新父节点 → 用 `inverse(parent_world) * old_world` 反推局部矩阵。

5. **场景图 ≠ 空间加速结构**
   不要把视锥剔除、射线检测逻辑塞进 TransformSystem。场景图只负责"谁是谁的孩子"，空间查询交给专门的 BVH/Octree 模块，在 Transform 更新完毕后执行。

6. **用 Affine3A 代替 Mat4 存 GlobalTransform**
   省 4 个 float（64→48 字节），且能正确表达层级叠加后的 shear。Bevy 的选择经过了生产验证。
