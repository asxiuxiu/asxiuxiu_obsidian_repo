# 实体生命周期与注册表

> **前置依赖**：[[Prefab与数据层]]、[[组件系统架构]]
> **本模块增量**：在 ECS 骨架上完善 `EntityRegistry`、EntityID 的生成与回收、延迟创建与销毁队列。实体的完整生命周期从此可控。
> **下一步**：[[RHI抽象层]]，因为 ECS 核心运行时闭环已经完成，终于可以开始真正画东西了。

---

## Why：为什么 ECS 引擎需要一个完善的实体生命周期系统？

在 [[Prefab与数据层]] 中，我们学会了从文件批量创建实体。但如果只懂"生"不懂"死"、只懂"立即创建"不懂"安全延迟销毁"，引擎运行一段时间后就会内存泄漏、ID 溢出、迭代器崩溃。

### 场景 1：射击游戏中每秒生成 100 颗子弹，5 秒后消失

如果没有回收机制，EntityID 单调递增，10 分钟后 ID 计数器可能超过 200 万。虽然 32 位整数理论上能撑到 40 亿，但你还需要为每个已销毁实体保留 sparse 数组槽位，内存浪费惊人。

### 场景 2：`CombatSystem` 在遍历中销毁了一个敌人

`CombatSystem` 正在遍历 `Query<HealthComponent>` 处理伤害结算。某次伤害导致敌人 HP 归零，代码立即调用 `destroy_entity`。如果 ECS 的组件存储是 `SparsePool`（见 [[组件系统架构]]），`swap-and-pop` 会立即移动另一个实体的组件到当前索引，导致 `CombatSystem` 的遍历跳过某个实体或访问到已回收的内存。

### 场景 3：Prefab 实例化需要创建 50 个有父子关系的实体

如果你逐个立即创建并挂接父子关系，中间的某个 System 可能在只创建了 30 个实体时就开始执行，看到一棵"缺胳膊少腿"的 hierarchy 树。需要一种机制，让 50 个实体在"全部创建完毕"后才对外界可见。

> **核心需求**：引擎需要一个**安全、可回收、延迟生效**的实体生命周期管理中心。它必须能回答："EntityID 用完了怎么办？" "遍历中销毁实体安全吗？" "批量创建的原子性如何保证？"

---

## What：最简化版本的实体生命周期系统长什么样？

下面是一个基于 **generation + 回收队列** 的 `EntityRegistry`，配合延迟创建/销毁队列，保证结构安全。

```cpp
#include <vector>
#include <queue>
#include <cstdint>

// ---------- EntityID：索引 + 世代 ----------
struct EntityID {
    uint32_t index;
    uint32_t generation;

    bool operator==(const EntityID& other) const {
        return index == other.index && generation == other.generation;
    }
};

const EntityID INVALID_ENTITY = { ~0u, ~0u };

// ---------- 实体槽位 ----------
struct EntitySlot {
    uint32_t generation = 0;
    bool     alive = false;
};

// ---------- 命令缓冲 ----------
struct EntityCommandBuffer {
    std::vector<std::function<void()>> commands;
    void push(std::function<void()> cmd) { commands.push_back(cmd); }
    void execute() {
        for (auto& cmd : commands) cmd();
        commands.clear();
    }
};

// ---------- 实体注册表 ----------
struct EntityRegistry {
    std::vector<EntitySlot> slots;
    std::queue<uint32_t>    free_indices;
    EntityCommandBuffer     pending_create;
    EntityCommandBuffer     pending_destroy;

    EntityID create() {
        uint32_t index;
        uint32_t generation;

        if (!free_indices.empty()) {
            index = free_indices.front();
            free_indices.pop();
            generation = slots[index].generation;
        } else {
            index = (uint32_t)slots.size();
            slots.push_back({1, true});
            generation = 1;
        }

        slots[index].alive = true;
        return {index, generation};
    }

    bool is_valid(EntityID id) const {
        return id.index < slots.size() &&
               slots[id.index].generation == id.generation &&
               slots[id.index].alive;
    }

    void destroy(EntityID id) {
        if (!is_valid(id)) return;
        slots[id.index].alive = false;
        slots[id.index].generation++; // 世代递增，旧句柄失效
        free_indices.push(id.index);
    }

    // 延迟创建：将创建逻辑封入命令缓冲
    template<typename F>
    void deferred_create(F&& fn) {
        pending_create.push(std::forward<F>(fn));
    }

    // 延迟销毁
    void deferred_destroy(EntityID id) {
        pending_destroy.push([this, id]() { this->destroy(id); });
    }

    // 安全点：统一执行延迟命令
    void apply_pending() {
        pending_create.execute();
        pending_destroy.execute();
    }
};
```

这个最小实现解决了什么？
- **ID 回收**：实体销毁后，index 被放入 `free_indices` 复用，不会无限增长。
- **悬空访问检测**：旧 `EntityID` 的 generation 与新复用的 slot 不匹配，`is_valid()` 会返回 false。
- **延迟生效**：`deferred_create` / `deferred_destroy` 让批量操作和遍历中的操作都能安全推迟到安全点执行。

还缺什么？
- **没有组件联动**：`destroy()` 只标记了 slot 死亡，没有通知各个组件池移除该实体的组件。
- **没有版本化的 sparse 数组同步**：组件池（如 `SparsePool`）也有自己的 generation 或 sparse 数组，需要与 `EntityRegistry` 同步回收。
- **没有批量创建的原子性**：延迟创建队列中的命令还是逐个执行的，没有"全部创建完成后统一对外可见"的提交机制。
- **没有父子联删**：销毁一个父实体时，子实体不会自动被销毁，可能导致 hierarchy 上出现 orphan 节点。

## How：真实引擎的实体生命周期系统是如何一步一步复杂起来的？

### 阶段 1：从单注册表 → 注册表 + 组件池联动回收（解决组件泄漏）

**触发原因**：最小实现的 `destroy()` 只回收了 EntityID，但没有通知 `SparsePool<Position>`、`SparsePool<Health>` 等组件池清理数据。这会导致组件内存泄漏，而且 `Query<Position>` 会遍历到已死亡实体的组件。

**代码层面的变化**：

1. **组件池注册机制**
   ```cpp
   struct IComponentPool {
       virtual void on_entity_destroyed(uint32_t index) = 0;
   };

   struct EntityRegistry {
       std::vector<IComponentPool*> pools;
       // ...
       void register_pool(IComponentPool* pool) { pools.push_back(pool); }

       void destroy(EntityID id) {
           if (!is_valid(id)) return;
           for (auto* pool : pools) {
               pool->on_entity_destroyed(id.index);
           }
           slots[id.index].alive = false;
           slots[id.index].generation++;
           free_indices.push(id.index);
       }
   };
   ```
   每个 `SparsePool<T>` 都实现 `on_entity_destroyed`，在实体死亡时将自己对应的 dense 条目移除。

2. **Sparse Pool 的联动清理**
   ```cpp
   template<typename T>
   struct SparsePool : IComponentPool {
       // ...
       void on_entity_destroyed(uint32_t index) override {
           if (has(index)) remove(index);
       }
   };
   ```

**状态变化图：实体销毁时的联动清理**

销毁前：
```
EntityRegistry:
  slots[3] = {generation=2, alive=true}

PositionPool:
  dense: [P0, P1, P3, P5]
  dense_to_entity: [E0, E1, E3, E5]
  sparse[3] = 2  (E3 → dense[2] = P3)
```

执行 `destroy(E3{index=3, gen=2})`：
```
1. EntityRegistry 通知所有 pools: on_entity_destroyed(3)
2. PositionPool 发现 sparse[3]=2 有效，执行 swap-and-pop:
   dense: [P0, P1, P5]          // P3 被移除
   dense_to_entity: [E0, E1, E5] // E3 被移除
   sparse[5] = 2                // E5 从 dense[3] 移到 dense[2]
   sparse[3] = INVALID
3. EntityRegistry 回收 slot:
   slots[3] = {generation=3, alive=false}
   free_indices.push(3)
```

> **精确性补充**：注意这里的关键是 **"不同数据结构的回收动作是同步发生的"**。`EntityRegistry` 的 `generation` 增加和 `SparsePool` 的 `remove` 必须在同一个 `destroy()` 调用内完成。如果分两步执行（比如先标记死亡、等到帧末再清理组件），在这两步之间，其他 System 可能通过旧 `EntityID` 访问到"活着但无组件"的实体，或遍历到已死亡实体的组件。

### 阶段 2：延迟队列 + 命令缓冲合并（解决遍历安全和批量原子性）

**触发原因**：阶段 1 的 `destroy()` 是立即执行的。如果在 System tick 中直接调用它，仍然会导致迭代器失效。而且 Prefab 实例化时需要创建 50 个实体，中间过程不应被其他 System 观测到。

**代码层面的变化**：

1. **将延迟命令合并到全局命令缓冲**
   不再让 `EntityRegistry` 自己维护独立的 `pending_create` / `pending_destroy`，而是复用 [[系统调度与确定性]] 中的全局 `CommandBuffer`：
   ```cpp
   world.commands().destroy_entity(e);   // 只入队
   world.commands().apply();             // 安全点统一执行
   ```

2. **批量创建的原子性**
   Prefab 实例化时，所有创建命令都写入同一个 `CommandBuffer`，并附加一个 `batch_id`：
   ```cpp
   uint64_t batch_id = world.commands().begin_batch();
   for (auto& desc : prefab_entities) {
       world.commands().create_entity(batch_id, desc);
   }
   world.commands().end_batch(batch_id);
   ```
   `CommandBuffer::apply()` 保证同一个 `batch_id` 下的所有命令在同一安全点内连续执行，不会被其他 System 穿插观测。

3. **父子联删的延迟处理**
   当某个实体被标记为延迟销毁时，系统会递归收集其所有子实体，一并加入销毁队列：
   ```cpp
   void deferred_destroy_with_children(EntityID parent) {
       std::vector<EntityID> to_destroy;
       collect_children_recursive(parent, to_destroy);
       for (EntityID e : to_destroy) {
           world.commands().destroy_entity(e);
       }
   }
   ```

### 阶段 3：工业级 → 分代回收、Entity 版本缓存与网络同步支持

**触发原因**：在大型场景中，实体的创建和销毁非常频繁。简单的 `free_indices` 队列可能导致 generation 快速溢出（虽然 32 位需要 40 亿次，但 16 位 generation 在极端场景下可能不够）。网络同步需要 EntityID 在客户端和服务器之间保持一致或可映射。

**代码层面的变化**：

1. **分代回收池（Generational Pool）**
   不再只有一个 `free_indices` 队列，而是维护多个队列，按 slot 的"死亡时间"分组：
   - 刚死亡的 slot 放入"年轻代"队列，延迟 N 帧后才真正回收。
   - 这给了所有持有旧 `EntityID` 的 System 一个缓冲期，即使它们错过了安全点的通知，也不会立刻复用到错误的 slot。

2. **Entity 版本缓存（Version Cache）**
   某些 System 会缓存 `EntityID → ComponentPointer` 的映射。如果实体死亡后立即复用，这些缓存的指针就会指向错误的对象。引入 `EntityVersion` 组件或全局 `VersionRegistry`：
   ```cpp
   uint64_t get_global_version(EntityID id); // 每次实体复活时递增
   ```
   缓存的映射可以附带版本号，访问前校验版本是否匹配。

3. **网络同步的 EntityID 映射**
   服务器和客户端的 EntityID 分配可能不同步。引入 `NetEntityID` 组件，由服务器统一分配，客户端在收到实体创建消息时建立映射：
   ```cpp
   std::unordered_map<NetEntityID, EntityID> net_to_local;
   ```
   这样网络消息中传输的是稳定的 `NetEntityID`，而不是本地的 `EntityID`。

4. **热更新与实体迁移**
   编辑器中修改 Prefab 后，可能需要"重建"某些实体（销毁旧实体并用新的 Prefab 版本重新实例化）。为了保持子实体覆盖和运行时状态：
   - 先对旧实体做快照（保存组件值）。
   - 销毁旧实体。
   - 用新 Prefab 实例化新实体。
   - 将快照中未受 Prefab 变更影响的字段回写到新实体。

## 从源码到 ECS 的重构映射

chaos 引擎的实体生命周期系统有以下特点：EntityID 单调递增不回收、重量级 `EntityProxy`、Manager 各自维护延迟队列。迁移到 ECS 时，应做如下改造：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `EntityProxyManager::allocateID()`（单调递增）| **`EntityRegistry::create()`（带 generation 的回收池）** | 引入 generation 和 free list，解决长时间运行的 ID 溢出和内存浪费问题。 |
| `EntityProxy`（重量级对象，含 Prefab 元数据、GameActor 配置）| **普通 ECS 实体 + `PrefabInstance` 组件** | 将 Proxy 中的状态完全拆分为组件，实体本身只是一个轻量 ID。 |
| `EntityRegistry::deleteEntity()`（双阶段延迟删除）| **`CommandBuffer::destroy_entity()`** | 将延迟删除统一到全局命令缓冲，而不是每个 Manager 各自维护队列。 |
| `Manager::processComponentsManagement()` | **全局安全点 `world.commands().apply()`** | 所有组件的增删都在同一个安全点执行，Scheduler 负责驱动。 |
| `DataReferenceManager`（DataRef ↔ EntityID 映射）| **可选的 `NetEntityID` 组件或 `Name` 组件** | 不需要额外的 XOR DataRef 系统。跨系统引用可以通过命名实体或网络 ID 实现。 |
| `checkPendingParentship()`（延迟父子挂接）| **命令缓冲中的 `set_parent` 命令** | 父子关系变更也走命令缓冲，在 `TransformSystem` 执行前的安全点统一处理。 |

> **Why ECS 更适合这个模块？**
> - **无重量级代理**：chaos 的 `EntityProxy` 是每个实体一个的 OOP 对象，内存开销大且生命周期复杂。ECS 中实体只是一个 ID，所有元数据都是可选组件，没有 Proxy 就没有代理层的泄漏风险。
> - **统一的命令缓冲**：chaos 中每个 Manager 有自己的 `ConcurrentQueue`，调试时需要追踪多个队列的状态。ECS 的全局命令缓冲让"这帧到底创建/销毁了什么"一目了然。
> - **更好的可观测性**：AI 可以通过 Query 直接查看所有待销毁的实体（如果命令缓冲以"事件实体"形式暴露），而不需要穿透 Manager 的私有队列。

## "如果我要 vibe coding，该偷哪几招？"

1. **EntityID 必须带 generation**
   这是 ECS 101，但很多人会忽略。没有 generation，你迟早会遇到"销毁了 E5，然后另一个 System 拿着旧的 E5 指针访问了新复用的 E5"这种噩梦级 bug。

2. **延迟销毁不是可选优化，是必需品**
   即使你的引擎只有 10 个 System，也要从第一天起把 `destroy_entity` 做成命令缓冲。立即销毁在 demo 期可能不崩溃，但在生产环境它是 100% 的定时炸弹。

3. **父子联删默认开启**
   在 99% 的场景下，销毁父实体时你都想同时销毁子实体。把这个作为默认行为，提供一个可选参数 `keep_children` 给那 1% 的特殊情况。

4. **命令缓冲的日志输出是调试神器**
   在 Debug 模式下，每帧打印命令缓冲的内容："Created: [E5, E6], Destroyed: [E3], SetParent: [E7→E5]"。当你看到"为什么这帧结束后 E5 还存在？"这类问题时，这条日志能救命。

5. **不要过早优化分代回收池**
   对于个人项目或中小型游戏，简单的 `free_indices` 队列完全够用。分代回收池是 MMO 级别项目才需要的优化，不要为了"看起来更工业级"而增加不必要的复杂度。
