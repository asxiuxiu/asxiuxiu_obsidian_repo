# 实体生命周期与注册表

> **前置依赖**：[[组件系统架构]]、[[最小ECS数据层]]
> **本模块增量**：在 ECS 骨架上完善 `EntityRegistry`、EntityID 的生成与回收、延迟创建与销毁队列、生命周期钩子与批量原子性。实体的完整生命周期从此可控。
> 
> 本笔记探讨的核心问题是：**在一个每帧可能创建/销毁上千个实体的 ECS 世界中，如何保证 ID 不泄漏、遍历不崩溃、批量操作原子、且 AI 能安全地远程操作？** 我们将从多个分支深入分析……

---

## 问题 0：为什么 ECS 引擎需要一个完善的实体生命周期系统？

在 [[最小ECS数据层]] 中，我们验证了一个最简 ECS 能跑起来：创建实体、附加组件、遍历查询、修改数值。但如果只懂"生"不懂"死"、只懂"立即创建"不懂"安全延迟销毁"，引擎运行一段时间后就会内存泄漏、ID 溢出、迭代器崩溃。

### 场景绑定：10,000 个包裹的分拣中心出了什么问题？

沿用 [[组件系统架构]] 中的分拣中心比喻。现在中心里的包裹（实体）不是静态的——每秒钟有数百个新包裹入库，数百个旧包裹被取走。

**场景 1：单号溢出**。包裹单号单调递增，运行 10 分钟后单号超过了 200 万。虽然 32 位整数理论上能到 40 亿，但你还得为每个已销毁包裹保留查询台（sparse 数组）的槽位。10,000 个活跃包裹占用了 200 万个槽位的查询台——内存浪费惊人。

**场景 2：分拣员扫描时包裹被移走**。分拣员 A 正在扫描"加急件"货架，扫描到第 50 个包裹时，另一个分拣员突然把第 50 个包裹取走了——货架用 `swap-and-pop` 保持紧凑，把末尾的包裹补到了第 50 位。分拣员 A 下一步读到的是原来末尾的包裹，跳过了原本在第 51 位的包裹。

**场景 3：批量入库的中间状态**。一辆货车同时卸下 50 个有父子关系的包裹（大箱子里套小箱子）。如果逐个入库，在入库到第 30 个时，某个分拣系统开始扫描——它看到的是一箱"缺胳膊少腿"的层级关系：大箱子存在，但里面只挂了 15 个小箱子，另外 35 个还没入库。

> **核心需求**：引擎需要一个**安全、可回收、延迟生效**的实体生命周期管理中心。它必须能回答："EntityID 用完了怎么办？" "遍历中销毁实体安全吗？" "批量创建的原子性如何保证？"

## 问题 1：Entity ID 用完了怎么办？——回收与世代计数

### 场景绑定：把包裹发给"死人"

当前实现里 Entity ID 单调递增。更致命的是：如果一个包裹被销毁后单号立刻复用，其他拿着旧单号的分拣员会误把新包裹当旧包裹处理。

```cpp
// 危险的做法
Entity e = world.create();  // e = {index=42, gen=0}
world.destroy(e);           // 42 被回收
Entity e2 = world.create(); // e2 = {index=42, gen=0}（复用了！）
// 某个 System 还拿着旧单号 42，以为它指向原来的实体
auto* pos = world.get<Position>(e); // 实际上拿到了新实体的 Position！
```

这不是理论问题——在没有 generation 的 ECS 中，这是调试期**最高频的 bug 类型**。你看到一个实体的 `Position` 突然跳到奇怪的位置，花了两小时追踪，最后发现是因为某个 System 缓存了旧的 Entity ID，而这个 ID 已被新实体复用。

### 根因分析：复用的必然性与悬空访问的矛盾

32 位 index 不可能无限增长。即使你不主动回收，内存中的 sparse 数组也会随最大 index 膨胀。所以**回收是必然的**。但回收意味着旧 ID 可能指向新对象——除非有一种机制能区分"同一个 index 的不同世代"。

### 分支 A：单调递增 + 延迟回收

永远不复用 ID，直到 32 位溢出。对于运行几分钟的 demo 这足够，但对于长时间运行的服务器或开放世界游戏，overflow 后的行为是未定义的。而且 sparse 数组的内存随最大 ID 线性增长——100 万个历史实体即使只剩 100 个活跃，sparse 数组仍占 4MB × 组件类型数量。

**适用场景**：极短生命周期的工具程序、原型验证。
**隐藏代价**：内存浪费、ID 溢出风险、无法用于生产环境。

### 分支 B：Index + Generation（世代计数）

这是 ECS 引擎的标准做法。Entity ID 拆分为两部分：

```cpp
struct EntityID {
    uint32_t index;      // 回收后复用
    uint32_t generation; // 检测悬空访问
};
```

销毁实体时，index 放入空闲队列，generation 递增。下次复用该 index 时，旧单号因 generation 不匹配而被拒绝。

```cpp
struct EntityRegistry {
    struct Slot {
        uint32_t generation = 0;
        bool     alive = false;
    };
    Array<Slot>      slots;      // generations + alive 状态
    Array<uint32_t>  free_list;  // 可回收的 index
    uint32_t         next_index = 0;

    EntityID create() {
        if (!free_list.empty()) {
            uint32_t idx = free_list.back();
            free_list.pop_back();
            slots[idx].alive = true;
            return {idx, slots[idx].generation};
        }
        // 新槽位
        slots.push_back({0, true});
        return {next_index++, 0};
    }

    bool is_alive(EntityID e) const {
        return e.index < slots.size() 
            && slots[e.index].generation == e.generation 
            && slots[e.index].alive;
    }

    void destroy(EntityID e) {
        if (!is_alive(e)) return;
        slots[e.index].alive = false;
        slots[e.index].generation++;  // 世代递增，旧句柄失效
        free_list.push_back(e.index);
    }
};
```

**状态变化图：generation 的防别名机制**

```
初始状态：
  slots[3] = {generation=2, alive=true}
  free_list = []

执行 destroy(E3{index=3, gen=2})：
  slots[3] = {generation=3, alive=false}
  free_list = [3]

执行 create()（复用 index 3）：
  slots[3] = {generation=3, alive=true}
  返回 E3'{index=3, gen=3}

旧句柄 E3{index=3, gen=2} 再访问：
  is_alive(E3) → slots[3].generation=3 ≠ 2 → false ✅ 被拒绝
```

**适用场景**：几乎所有现代 ECS 引擎。
**隐藏代价**：`slots` 数组随最大 index 增长，但每个槽位只有 `generation + alive`（约 8 字节），100 万个实体约 8MB，可接受。

### 分支 C：UUID（128 位全局唯一标识符）

某些 OOP 引擎使用 UUID 作为实体标识符，彻底消除复用冲突。但 UUID 的随机性让它无法直接做数组索引，必须配合哈希表使用——这摧毁了 Sparse Set 的 O(1) 索引性能，也让 Archetype 的连续内存布局无法实现。

**适用场景**：需要跨会话持久标识的分布式系统（如持久化 MMO 的角色 ID）。
**隐藏代价**：失去 ECS 的缓存局部性优势，查询和迭代性能灾难。

### 引擎对照：工业界如何处理实体标识？

> **参考：UE、Bevy、chaos 对这个问题是怎么做的？**
>
> 我们在解决的是「实体标识的回收与悬空检测」这个问题。
> - **chaos** 选择了单调递增不回收。这意味着长时间运行后 ID 会溢出，且 sparse 数组无限膨胀。默认实现不应效仿。
> - **UE** 没有 ECS 式的 Entity ID——Actor 是重量级对象，通过全局数组索引在对象注册表中定位。对象销毁后，旧指针不会自动指向新对象，因为 UE 依赖 C++ 裸指针和 GC 标记。UE 的对象注册表元数据槽中还有一个序列号字段，作用类似 generation：它保证弱引用指针在对象复用同一数组索引时不会误判。这是 OOP 世界对同一问题的不同解答。
> - **Bevy** 的 `Entity` 是 `index + generation` 的 64 位打包（低 32 位 index，高 32 位 generation）。`Entities` 元数据数组按 index 索引，`is_alive` 检查是单次数组访问。`EntityAllocator` 甚至支持原子并发分配 ID——多个线程可以同时"预订"实体 ID，而无需独占 World。
>
> **默认推荐**：`index + generation`。这是防止悬空访问的最小代价方案，没有 generation 的 ECS 在调试期会给你带来地狱般的 use-after-free。

## 问题 2：遍历中销毁实体安全吗？——延迟命令缓冲

### 场景绑定：扫描到一半，货架塌了

分拣员正在扫描"易碎品+加急件"托盘。扫描到第 50 个包裹时，系统判定这个包裹已损坏需要销毁——如果立即执行 `swap-and-pop` 移除，分拣员的扫描指针就踩空了。

```cpp
// 危险：在遍历中直接销毁
for (auto [e, pos, vel] : query<Position, Velocity>()) {
    if (pos.x > 100.0f) {
        world.destroy(e);  // 当前实体从 dense 数组中移走了！
        // 迭代器可能已经失效，或者跳过某些实体，
        // 或者访问到已释放的内存（如果组件存储立即释放）
    }
}
```

### 根因分析：结构变更与迭代稳定性的矛盾

ECS 的批量迭代依赖**遍历期间数据结构不变**。`destroy()` 会触发：
1. 从组件池的 dense 数组中 `swap_remove`——被移动元素的索引变了。
2. 从 Archetype 的 Table 中移除行——同一 Table 中其他实体的行号变了。
3. 从 `EntityRegistry` 中回收 slot——`free_list` 变了。

任何一步都会破坏正在进行的迭代器状态。

### 分支 A：立即销毁（最简单，最危险）

直接调用 `world.destroy(e)`，所有清理立即发生。

**适用场景**：仅在遍历结束后、且确定没有其他迭代器活跃时。
**隐藏代价**：在 System tick 内部调用是 100% 的定时炸弹。即使当前没崩溃，随着 System 增多、Query 嵌套，迟早会遇到迭代器失效。

### 分支 B：命令缓冲（Command Buffer）

所有 "创建/销毁实体"、"添加/移除组件" 的请求不再立即执行，而是写入当前帧的命令缓冲队列。在 tick 的**安全点**（通常是帧尾或专门的延迟应用阶段）统一执行。这保证了分拣员扫描 Chunk 时，Chunk 的结构是稳定的。

```cpp
// 安全：只入队，不立即执行
world.commands().destroy(e);        // 写入延迟队列
world.commands().spawn(bundle);     // 同上
world.commands().add_component(e2, Velocity{1,0,0}); // 同上

// 在安全点统一执行（由 Scheduler 在 Phase 末尾自动调用）
world.commands().apply();
```

**状态变化图：命令缓冲的" staging → commit "模型**

```
Tick 开始：
  CommandBuffer: []

System A 遍历中：
  发现 E5 需要销毁 → cmd: Destroy(E5)
  发现需要新建子弹 → cmd: Spawn(BulletBundle)
  CommandBuffer: [Destroy(E5), Spawn(BulletBundle)]

System B 遍历中：
  E5 仍然"活着"（因为还没 apply）
  System B 可以正常读到 E5 的组件
  
ApplyDeferred 安全点：
  1. 执行 Spawn(BulletBundle) → 新实体 E100 诞生
  2. 执行 Destroy(E5) → E5 的组件被清理、slot 回收
  CommandBuffer: []

Tick 结束：
  所有 System 看到的都是 apply 后的最终状态
```

**关键洞察**：命令缓冲不是"可选优化"，而是 ECS 的**结构性安全设施**。没有它，System 无法在遍历中安全地发起任何结构变更。

**隐藏代价**：
- **一帧延迟**：组件增删在提交后不会立即生效。如果 System A 在 Tick 前半段添加了一个组件，System B 在同 Tick 的后半段查询这个组件（且在同一个安全点之前），是查不到的。
- **内存分配**：如果每帧产生数千条命令，缓冲区的分配和释放可能成为瓶颈。

### 分支 C：双缓冲世界（Double-Buffered World）

维护两个世界状态：当前帧只读，下一帧可写。System 遍历当前帧的数据，所有修改写入下一帧。帧尾时交换两个世界的指针。

**适用场景**：确定性回放（Replay）和网络预测回滚——这些场景需要保存完整的历史状态。
**隐藏代价**：内存翻倍。对于大型世界，两个完整副本的内存开销不可接受。

### 分支 D：延迟标记 + 延迟清理

在遍历中不真正移除组件，而是打一个"待移除"标记（如 `IsDestroyed` 组件）。遍历完成后，统一处理所有标记。

**适用场景**：标记组件的批量清理。
**隐藏代价**：System 必须在逻辑上忽略已标记的组件，这增加了每个 System 的复杂度。而且标记本身也是一种组件增删——如果走 Archetype 存储，标记会让实体从原 Archetype 迁移到 `{原组件, IsDestroyed}` Archetype，引发 Archetype Churn。

### 引擎对照：工业界如何保证遍历安全？

> **参考：UE、Bevy、chaos 对这个问题是怎么做的？**
>
> 我们在解决的是「遍历中发起结构变更的安全性」这个问题。
> - **chaos** 的每个 Manager（System 的抽象基类）各自维护私有的延迟组件队列，在 tick 的组件管理阶段统一处理。这与 ECS 的命令缓冲思想一致，但分散在每个 Manager 内部——调试时需要追踪多个队列的状态。
> - **UE** 没有这个问题——因为 UE 的组件增删是在主线程上同步执行的。添加组件和注销组件会立即修改组件容器数组。UE 不担心"遍历中修改"，因为 UE 的组件更新是逐个调用 Tick 的，不是批量迭代。但这也意味着 UE 无法在组件级别做自动并行。
> - **Bevy** 使用全局延迟命令队列。延迟应用是一个显式的 System，在 Schedule 的特定阶段运行。Bevy 的命令队列采用**密集字节缓冲**策略——每个命令不单独堆分配，而是直接序列化到连续内存中，极大降低了命令缓冲的内存开销。它还提供了针对单个实体的链式 API，所有操作最终都绑定到同一个预分配的 Entity ID 上。
>
> **默认推荐**：全局命令缓冲。不要在 System 遍历中直接 `add_component` 或 `destroy_entity`。哪怕你的 ECS 只有 100 个实体，这也是定时炸弹。
>
> **进阶优化**：如果 Profiler 显示命令缓冲的内存分配是瓶颈，可以参考 Bevy 的密集字节缓冲策略——将命令序列化到 `Array<uint8_t>` 中，配合函数指针表来执行，而不是用虚函数或 `std::function`。

## 问题 3：批量创建的原子性如何保证？

### 场景绑定：50 个包裹的层级关系只建了一半

一辆货车同时卸下 50 个有父子关系的包裹（大箱子里套小箱子、套更小的箱子）。如果你逐个立即创建并挂接父子关系，中间的某个 System 可能在只创建了 30 个实体时就开始执行——它看到的是一棵"缺胳膊少腿"的 hierarchy 树。更糟的是，如果父子关系通过组件表示（如 `Parent` 和 `Children`），中间状态下 `Parent` 指向的实体可能还不存在。

### 根因分析：结构变更的"可见性"边界

ECS 没有事务隔离级别。任何结构变更都是立即对 World 可见的（除非你使用命令缓冲）。但即使使用命令缓冲，如果命令是逐条执行的，执行到第 N 条和第 N+1 条之间仍然有一个"中间状态窗口"。

### 分支 A：逐个立即创建（最 naive）

```cpp
Entity parent = world.create();
for (int i = 0; i < 50; ++i) {
    Entity child = world.create();
    world.add_component(child, Parent{parent});
    // 此时如果有 System 在运行，它只看到 i+1 个子实体
}
```

**隐藏代价**：中间状态导致依赖层级关系的 System（如 Transform 传播）读取到不一致的数据。

### 分支 B：命令缓冲 + 批量提交

所有创建命令都写入同一个 `CommandBuffer`，并附加一个 `batch_id`。`CommandBuffer::apply()` 保证同一个 `batch_id` 下的所有命令在同一安全点内连续执行，不会被其他 System 穿插观测。

```cpp
uint64_t batch = world.commands().begin_batch();
Entity parent = world.commands().create_entity(batch);
for (int i = 0; i < 50; ++i) {
    Entity child = world.commands().create_entity(batch);
    world.commands().add_component(batch, child, Parent{parent});
}
world.commands().end_batch(batch);
// 在 ApplyDeferred 时，batch 内所有命令原子执行
```

**关键机制**：Scheduler 在延迟应用阶段执行命令缓冲时，必须保证：
1. 同一个 batch 内的命令按顺序连续执行，中间不插入其他 System。
2. batch 执行期间，任何 Query 都看到的是 batch 开始前的状态，或 batch 完全执行后的状态——没有中间状态。

**适用场景**：Prefab 实例化、关卡加载、网络快照批量重建。

### 分支 C：预分配 + 延迟初始化（Bevy 风格）

Bevy 的 实体 ID 分配器支持**并发预分配** ID——多个线程可以先并行"预订"实体 ID，再串行地实际生成实体。这意味着：
1. 父实体和 50 个子实体的 ID 可以先全部分配好（原子操作，线程安全）。
2. `Commands` 中捕获的是已经确定的 ID，父子关系在命令层面就是完整的。
3. 在延迟应用同步点时一次性创建所有实体并写入组件——因为 ID 已经预先知道，不存在"指向不存在的实体"的中间状态。

```cpp
// Bevy 风格的预分配
Entity parent = allocator.alloc();  // 原子操作，线程安全
Array<Entity> children;
for (int i = 0; i < 50; ++i) {
    children.push_back(allocator.alloc());
}
// 所有 ID 已确定，现在写入命令缓冲
commands.spawn_at(parent, ParentBundle{});
for (Entity child : children) {
    commands.spawn_at(child, ChildBundle{parent});
}
```

**适用场景**：高并发场景下需要大量创建实体（如粒子系统、子弹风暴）。
**隐藏代价**：预分配的 ID 如果最终未被使用（如命令被取消），会造成 ID 浪费。需要配套的"取消预分配"机制。

### 引擎对照

> **参考：Bevy、UE 对批量创建是怎么做的？**
>
> 我们在解决的是「批量结构变更的原子可见性」这个问题。
> - **Bevy** 的 创建命令会先预分配 ID，再构建延迟执行的命令对象。这意味着即使分两步调用（先创建再插入组件），它们操作的是同一个预先知道的 Entity ID，不存在中间悬空。Bundle 实例化器在延迟应用同步点时原子地完成所有组件写入。
> - **UE** 的 Actor 创建是一个漫长的同步链：从创建对象到加载数据、注册组件、初始化、开始运行，每一步都可能被其他系统观测到。UE 通过创建完成标志来标记"创建是否完成"，但中间状态仍然是可见的。这是 OOP 模型的代价——没有命令缓冲的隔离层。
>
> **默认推荐**：命令缓冲 + 预分配 ID。这是 ECS 架构的天然优势——你可以先分配 ID、构建完整的命令序列，再在安全点原子提交。OOP 引擎很难做到这一点，因为对象的构造函数本身就是副作用。

## 问题 4：父子联删该如何实现？

### 场景绑定：大箱子被销毁，里面的东西还在飘

销毁一个父实体时，如果不同时销毁其子实体，hierarchy 上会出现 orphan 节点——子实体仍然持有指向已销毁父实体的 `Parent` 组件，而场景图系统试图更新这个子实体的世界变换时，会访问到无效的父实体数据。

### 分支 A：手动逐个销毁

每个销毁父实体的地方，手动递归收集所有子实体并逐个销毁。

```cpp
void destroy_recursive(EntityRegistry& reg, EntityID parent) {
    // 手动查询所有 Parent == parent 的子实体
    for (auto [e, p] : query<Parent>()) {
        if (p.entity == parent) {
            destroy_recursive(reg, e);  // 递归销毁
        }
    }
    reg.destroy(parent);
}
```

**隐藏代价**：
- 每次销毁都要做一次全表扫描（或至少扫描 `Parent` 组件的 dense 数组），复杂度 O(N)。
- 如果层级很深，递归栈可能溢出。
- 开发者容易忘记调用递归版本，导致 orphan 泄漏。

### 分支 B：递归收集 + 批量入队

在命令缓冲层面实现自动的父子联删。当某个实体被标记为延迟销毁时，系统递归收集其所有后代，一并加入销毁队列：

```cpp
void deferred_destroy_with_children(CommandBuffer& cmds, EntityID parent) {
    Array<EntityID> to_destroy;
    // 通过后缀遍历收集所有后代
    collect_descendants(parent, to_destroy);
    to_destroy.push_back(parent);
    
    // 按"先子后父"的顺序入队，避免中间状态
    for (EntityID e : to_destroy) {
        cmds.destroy(e);
    }
}
```

**关键机制**：销毁顺序必须是**先子后父**。如果先销毁父实体，子实体在销毁前的短暂窗口内会持有无效的 `Parent` 引用。

**适用场景**：几乎所有需要层级关系的 ECS 引擎。

### 分支 C：ECS 关系组件（Bevy 风格）

Bevy 不通过手动递归实现父子关系，而是通过**关系（Relationship）**组件——一个表示"是谁的子实体"的组件，以及引擎自动维护的反向索引。当父实体被销毁时，级联销毁操作会自动查找所有子实体关系，并一并销毁。

```rust
// Bevy 风格：关系是组件，但由引擎自动维护双向索引
commands.entity(parent).despawn_recursive();
// 级联销毁 parent 及其所有后代
```

**状态变化图：关系组件的双向索引**

```
World 状态：
  Parent:  E0
  ChildOf: E1→E0, E2→E0, E3→E1
  
  层级关系：
    E0
    ├── E1
    │   └── E3
    └── E2

despawn_recursive(E0)：
  1. 查询所有 ChildOf(E0) → [E1, E2]
  2. 对 E1 递归：查询 ChildOf(E1) → [E3]
  3. 对 E3 递归：无子节点 → 销毁 E3
  4. 销毁 E1
  5. 对 E2 递归：无子节点 → 销毁 E2
  6. 销毁 E0
  
  结果：所有相关实体和关系组件一并清理
```

**适用场景**：层级关系频繁变化、需要自动维护双向索引的复杂场景。
**隐藏代价**：关系组件的自动索引维护需要额外的内存（反向索引表）和更新开销。每次添加/移除子实体关系组件时，都要同步更新父实体的子实体列表。

### 决策分析

| 条件 | 推荐方案 | 理由 |
|------|---------|------|
| 层级简单、手动控制 | 分支 B：递归收集 + 批量入队 | 实现简单，无额外索引开销。 |
| 层级复杂、需要自动维护 | 分支 C：ECS 关系组件 | Bevy 已验证该模式在 UI 树、场景图中的有效性。 |
| 父子联删作为默认行为 | 任何方案都行 | 关键是**默认开启父子联删**，提供 `keep_children` 参数给那 1% 的特殊情况。 |

> **默认推荐**：分支 B（递归收集 + 批量入队）起步。当场景图复杂度增长到一定程度（如需要 UI 树、骨骼层级），再引入关系组件的自动双向索引。

## 问题 5：工业级引擎的生命周期复杂度——从 OOP 到 ECS 的映射

前面四个问题解决了 ECS 实体生命周期的"基础生存需求"。但工业级引擎还有更复杂的需求：异步资源释放、网络同步、编辑器热重载。这些在 OOP 引擎（如 UE）中有成熟的解决方案，我们需要思考：**在 ECS 架构下，这些能力该如何映射？**

### 5.1 异步销毁的两阶段模型

UE 的对象销毁是**异步的**：先启动异步清理，等待就绪检查，再完成最终销毁。为什么拆分？因为启动清理阶段可能发起了跨线程的资源释放请求（如向渲染线程发送释放 GPU 缓冲区的信号），必须等待渲染线程确认完成后才能继续。

ECS 中实体本身没有析构函数——它只是一个 ID。但**组件可能有外部资源**：网格组件可能持有 GPU 缓冲区句柄，纹理组件可能持有异步加载中的纹理。如果实体被销毁时立即释放这些资源，而渲染线程还在使用它们，就会崩溃。

**ECS 映射方案**：

```cpp
// 组件可以注册 OnRemove 钩子，钩子分两个阶段
struct ComponentHooks {
    void (*on_remove)(EntityID e, void* component);           // 类似 BeginDestroy
    bool (*is_ready_for_drop)(EntityID e, void* component);   // 类似 IsReadyForFinishDestroy
    void (*on_drop)(EntityID e, void* component);             // 类似 FinishDestroy
};

// 销毁流程
void destroy_with_async_cleanup(EntityRegistry& reg, EntityID e) {
    // 阶段 1：立即执行 on_remove（启动异步清理）
    for (each component of e) {
        if (hooks.on_remove) hooks.on_remove(e, ptr);
    }
    // 实体标记为"待最终清理"，index 不立即回收
    reg.mark_dying(e);
}

// 每帧的安全点
tick_end() {
    // 阶段 2：检查所有 dying 实体
    for (EntityID e : dying_list) {
        bool ready = true;
        for (each component of e) {
            if (hooks.is_ready_for_drop && !hooks.is_ready_for_drop(e, ptr)) {
                ready = false; break;
            }
        }
        if (ready) {
            // 执行 on_drop 并真正回收
            for (each component of e) {
                if (hooks.on_drop) hooks.on_drop(e, ptr);
            }
            reg.destroy(e);  // 现在才回收 generation
        }
    }
}
```

**代价诚实**：
- 引入了"dying"中间状态——实体已逻辑死亡，但资源未释放，index 也未回收。
- 如果 `is_ready_for_drop` 永远返回 false（如 GPU fence 丢失），dying 列表会无限增长。需要设置超时强制清理。

> **参考：UE 的异步销毁两阶段模型在 ECS 中没有直接等价物，因为 ECS 的实体没有虚函数表。但通过组件级别的 `on_remove` / `is_ready_for_drop` / `on_drop` 钩子，可以实现相同语义。**

### 5.2 网络同步的 EntityID 映射

在联机游戏中，服务器和客户端各自管理自己的 ECS World。服务器创建了一个敌人实体 E_S_42，客户端收到网络消息后需要在自己的 World 中创建对应的实体 E_C_17。如果网络消息中直接传输本地 ID（如 "E42 的位置更新了"），客户端和服务器必须保证 ID 同步——这在分布式系统中几乎不可能。

**解决方案：网络 ID 映射表**

```cpp
// 服务器端：所有网络可见实体分配稳定的 NetEntityID
struct NetSync {
    NetEntityID net_id;  // 由服务器统一分配，全局唯一
};

// 客户端：维护 net_id → local_id 的映射
HashMap<NetEntityID, EntityID> net_to_local;

// 收到网络消息 "NetID=7, Position=(10,0,5)" 时：
EntityID local = net_to_local[7];
world.get<Position>(local) = {10, 0, 5};
```

**关键设计**：
- `NetEntityID` 由服务器统一分配，单调递增不回收（或采用 64 位 UUID）。
- 客户端在收到实体创建消息时，本地创建实体并建立映射。
- 实体销毁时，网络消息携带 `NetEntityID`，客户端查找映射后销毁本地对应实体。

> **参考：UE 的 Actor 网络同步通过网络 GUID 映射系统实现类似映射。ECS 中这个映射更简单——因为实体只是一个 ID，没有 OOP 对象的复杂外层容器层级。**

### 5.3 编辑器热重载与实体迁移

编辑器中修改 Prefab 后，可能需要"重建"某些实体（销毁旧实体并用新的 Prefab 版本重新实例化）。为了保留子实体覆盖和运行时状态：

```cpp
void hot_reload_prefab(EntityID root, PrefabID new_prefab) {
    // 1. 对旧实体子树做快照（保存组件值）
    Snapshot snap = capture_snapshot(root);
    
    // 2. 销毁旧实体子树（走延迟销毁队列）
    deferred_destroy_with_children(commands, root);
    
    // 3. 用新 Prefab 实例化新实体子树
    EntityID new_root = instantiate_prefab(commands, new_prefab);
    
    // 4. 将快照中未受 Prefab 变更影响的字段回写到新实体
    apply_snapshot(commands, new_root, snap);
}
```

**关键约束**：步骤 2 和 3 必须走命令缓冲，且在同一个 batch 中原子执行，避免编辑器面板在中间状态看到"旧实体已死、新实体未生"的空白期。

## 问题 6：AI 如何安全地操作实体生命周期？

自研引擎的一个核心设计目标是**AI 友好**。实体生命周期管理直接关系到 AI 的可观测性和安全性。

### 6.1 命令缓冲作为 Agent 操作的天然隔离层

当外部 AI 通过 MCP 接口发送 `create_entity` 或 `destroy_entity` 命令时，这些命令不应直接修改 World，而是进入**Staging CommandBuffer**。这为 Agent 操作提供了三层保护：

1. **延迟生效**：AI 的命令不会立即破坏正在进行的 System 迭代。
2. **可审查**：在 `apply` 之前，可以审查命令缓冲的内容（"AI 想要销毁哪些实体？"）。
3. **可回滚**：如果审查不通过，可以清空 Staging Buffer 而不影响主世界。

```cpp
// Agent 操作接口
AgentResult agent_create_entity(const Json& desc) {
    // 1. 解析 desc，生成 Command
    auto cmd = parse_spawn_command(desc);
    
    // 2. 写入 Staging Buffer（不是主 World）
    staging_buffer.push(cmd);
    
    // 3. 返回预览信息（"如果执行，将创建实体 E123"）
    return AgentResult::staging(cmd.preview());
}

// 审批后应用
void commit_agent_commands() {
    staging_buffer.merge_into(world.commands());
    world.commands().apply();
}
```

### 6.2 组件白名单与 Agent 安全

不是所有实体都应该被 AI 随意销毁。引擎应该支持**组件白名单**：只有带有 `AgentDeletable` 标记组件的实体，才允许 AI 通过命令缓冲销毁。

```cpp
bool can_agent_destroy(EntityID e) {
    return world.has<AgentDeletable>(e);  // 白名单检查
}
```

### 6.3 确定性保证

给定相同的初始实体集合和相同的命令序列，第 N 帧的 `EntityRegistry` 状态必须完全一致。这意味着：
- `free_list` 的顺序由销毁顺序唯一确定。
- 命令缓冲的执行顺序是确定的（按入队顺序）。
- 预分配的 Entity ID 顺序由 `EntityAllocator` 的原子计数器唯一确定。

这是网络同步（快照 + 回放）和多 Agent 协作的前提。

## 从三引擎源码到 ECS 的重构映射

| 源码概念 | OOP 引擎中的形态 | ECS 等价物 | 迁移说明 |
|---------|-----------------|-----------|---------|
| 实体标识回收 | UE 全局对象注册表的索引复用 + 序列号 | `EntityRegistry` 的 `free_list` + `generation` | 引入 generation 防止悬空访问，从 OOP 的全局数组索引映射为 ECS 的轻量 ID。 |
| 异步销毁 | UE 异步销毁两阶段模型（启动清理 → 等待就绪 → 完成销毁） | 组件级别的 `on_remove` / `is_ready_for_drop` / `on_drop` 钩子 | ECS 实体没有虚函数，组件钩子提供等价语义。 |
| 延迟结构变更 | chaos 各 Manager 的延迟组件队列 | 全局 `CommandBuffer` | 将分散的延迟队列统一到全局命令缓冲，Scheduler 负责驱动。 |
| 网络实体映射 | UE 网络 GUID 映射系统 | `NetSync` 组件 + `net_to_local` 映射表 | ECS 中映射更简单，因为实体无外层容器层级。 |
| 父子联删 | UE Actor 销毁流程递归销毁组件容器 | `deferred_destroy_with_children` 或关系组件 | ECS 中通过组件查询或关系索引实现级联。 |
| 批量创建 | UE Actor 创建同步链（多阶段、中间状态可见） | `CommandBuffer` + `begin_batch` / `end_batch` + 预分配 ID | ECS 的原子 batch 提交消除了中间状态。 |

> **Why ECS 更适合这个模块？**
> - **轻量标识**：OOP 中每个实体是一个完整对象（UE 的 Actor 对象有 40+ 字节的对象头开销）。ECS 中实体只是一个 64 位 ID，所有元数据都是可选组件。
> - **统一命令缓冲**：OOP 中每个子系统有自己的延迟队列（chaos 的 Manager 队列、UE 的对象销毁回调链）。ECS 的全局命令缓冲让"这帧到底创建/销毁了什么"一目了然，AI 可以直接读取。
> - **更好的可观测性**：AI 可以通过 Query 直接查看所有待销毁的实体（如果命令缓冲以"事件实体"形式暴露），而不需要穿透 Manager 的私有队列。
> - **确定性更强**：ECS 的命令序列化比 OOP 的虚函数调用更容易保证确定性——没有隐藏的 `TickComponent()` 副作用链。

## 默认推荐路径总结

### 默认推荐路径
- **实体标识**：`index(32bit) + generation(32bit)` 的 64 位打包 EntityID，配合简单 `free_list` 数组实现回收。
- **结构变更**：全局 `CommandBuffer`，在 Phase 安全点统一应用。
- **命令缓冲内存**：密集字节缓冲 + 函数指针表（参考 Bevy 策略）。
- **遍历安全**：安全点统一 `apply`，禁止在 System 遍历中直接修改结构。
- **父子联删**：递归收集 + 批量入队，默认开启联删，提供 `keep_children` 可选参数。
- **异步资源释放**：组件级别 `on_remove` / `is_ready_for_drop` / `on_drop` 两阶段钩子。
- **AI 操作隔离**：Staging CommandBuffer + `AgentDeletable` 组件白名单。

### UE 工业级设计吸收
- **序列号防悬空**：UE 对象注册表的元数据槽序列号机制 → 映射为 `index + generation` 的世代计数。
- **异步销毁两阶段**：UE 的 `BeginDestroy` → `IsReadyForFinishDestroy` → `FinishDestroy` → 映射为组件级三阶段钩子（`on_remove` / `is_ready_for_drop` / `on_drop`）。
- **网络 GUID 映射**：UE 的 `NetGUID` 系统 → 映射为 `NetSync` 组件 + `net_to_local` 映射表。

### ECS 映射
- 实体在 ECS 中只是一个 64 位 ID，所有生命周期元数据（generation、alive 状态、net_id 映射）都是可选组件或外部注册表数据。
- 全局 `CommandBuffer` 统一了 OOP 引擎中分散的延迟队列（chaos Manager 延迟队列、UE 对象销毁回调链）。
- 父子层级通过 `Parent` / `Children` 组件或关系组件表达，级联销毁通过组件查询或反向索引实现。

### 条件切换路径
| 条件 | 切换方案 |
|------|---------|
| 需要跨会话持久 ID | 增加 `NetEntityID(64bit)` 组件 |
| 极高频率创建/销毁（>1万/秒） | 分代延迟回收池 |
| 确定性回放场景 | 双缓冲世界（内存换确定性） |
| 复杂 UI/场景图 | 关系组件自动维护双向索引 |
| 单用户编辑器，无 AI 操作 | 可省略 Staging 层 |

### 扩展路径
- 网络同步的完整 EntityID 映射与快照增量 Diff。
- 编辑器热重载的 Prefab 重建与快照迁移。
- 多线程并发下的原子 ID 预分配（Bevy 风格 `EntityAllocator`）。
- 命令缓冲的链式 API 与单实体批量操作优化。

## "如果我要 vibe coding，该偷哪几招？"

1. **EntityID 必须带 generation**
   这是 ECS 101。没有 generation，你迟早会遇到"销毁了 E5，然后另一个 System 拿着旧的 E5 指针访问了新复用的 E5"这种噩梦级 bug。Bevy 把 index 和 generation 打包成 64 位整数，比较和哈希都极快。

2. **延迟销毁不是可选优化，是必需品**
   即使你的引擎只有 10 个 System，也要从第一天起把 `destroy_entity` 做成命令缓冲。立即销毁在 demo 期可能不崩溃，但在生产环境它是 100% 的定时炸弹。参考 Bevy 的延迟命令队列和 Unity DOTS 的实体命令缓冲。

3. **预分配 ID 解决批量创建的原子性**
   如果你需要创建 50 个有父子关系的实体，不要逐个 `create()`。先一次性预分配 50 个 ID（原子操作），然后用这些已知 ID 构建完整的命令序列，最后原子提交。这是 ECS 比 OOP 更容易实现的批量原子操作。

4. **父子联删默认开启**
   在 99% 的场景下，销毁父实体时你都想同时销毁子实体。把这个作为默认行为，提供一个可选参数 `keep_children` 给那 1% 的特殊情况。

5. **命令缓冲的日志输出是调试神器**
   在 Debug 模式下，每帧打印命令缓冲的内容："Created: [E5, E6], Destroyed: [E3], SetParent: [E7→E5]"。当你看到"为什么这帧结束后 E5 还存在？"这类问题时，这条日志能救命。

6. **不要过早优化分代回收池和 dying 列表**
   默认实现采用简单的 `free_list` 队列。分代延迟回收池和异步销毁的 dying 列表是应对极高频率实体 churn 或跨线程资源释放的工业级优化，应在 Profiler 证明存在瓶颈或出现跨线程释放需求时再引入。UE 的两阶段异步销毁是为了适配 Render Thread / Async Loading Thread 的复杂交互——如果你的引擎还没有多线程渲染，先跳过它。
