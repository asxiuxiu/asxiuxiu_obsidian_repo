# 反射系统

> **前置依赖**：[[组件系统架构]]、[[场景图与变换]]
> **本模块增量**：在 ECS 骨架上增加类型注册表与字段反射能力。Inspector 可以自动识别和编辑任意组件字段，AI 可以通过 Schema 理解数据结构。
> **下一步**：[[系统调度与确定性]]，因为多个 System 需要按正确顺序执行，而反射系统本身也需要在调度框架中提供脏标记通知。

---

## Why：为什么 ECS 引擎需要反射系统？

在 [[场景图与变换]] 中，我们定义了 `LocalTransform`、`GlobalTransform`、`Parent` 等组件。但如果只有 C++ 编译器认识这些类型，引擎的调试工具和外部 AI 就都是"瞎子"。

### 场景 1：Inspector 要编辑一个你自己新增的组件

你写了一个 `HealthComponent { float current; float max; }`。没有反射，Inspector 根本不知道这个组件里有什么字段、每个字段是什么类型、偏移量是多少。你必须为每个新组件手写一份 C# 或 ImGui 的绘制代码——这在迭代期是不可接受的。

### 场景 2：AI Agent 想读取当前实体的状态

外部 AI 通过 MCP 发送命令：`get_component(entity_42, "LocalTransform")`。如果引擎没有反射，AI 收到的只能是一坨原始字节。AI 无法理解 `position` 是前 12 个字节、`rotation` 是接下来 16 个字节。没有 Schema，就没有可观测性。

### 场景 3：保存/加载场景、网络同步、脚本绑定

序列化一个组件时，你需要知道它的每个字段如何转成 JSON/Binary。脚本系统绑定一个组件时，你需要知道它的字段名和内存布局。如果每个系统都手写这部分逻辑，维护成本会随组件数量线性爆炸。

> **核心需求**：引擎需要一张**类型注册表**，在运行期就能回答"`HealthComponent` 有哪些字段？每个字段叫什么名字、什么类型、在结构体中偏移多少字节？"

---

## What：最简化版本的反射系统长什么样？

下面是一个**零外部依赖**、**无需代码生成器**的最小实现。它使用 C++ 模板和宏手动构建类型注册表，足以让 Inspector 自动遍历任意组件的字段。

```cpp
#include <vector>
#include <string>
#include <unordered_map>
#include <cstdint>
#include <cstddef>

// ---------- 字段描述符 ----------
enum class FieldType { Float, Int, Bool, Vec3, String, Unknown };

struct FieldDesc {
    std::string name;
    std::string type_name;
    FieldType   type;
    size_t      offset;
    size_t      size;
};

// ---------- 类型描述符 ----------
struct TypeDesc {
    std::string name;
    size_t      size;
    std::vector<FieldDesc> fields;

    const FieldDesc* find_field(const std::string& field_name) const {
        for (const auto& f : fields) {
            if (f.name == field_name) return &f;
        }
        return nullptr;
    }
};

// ---------- 全局注册表 ----------
struct TypeRegistry {
    std::unordered_map<std::string, TypeDesc> types;

    void register_type(const TypeDesc& desc) {
        types[desc.name] = desc;
    }

    const TypeDesc* find(const std::string& name) const {
        auto it = types.find(name);
        return (it != types.end()) ? &it->second : nullptr;
    }
};

inline TypeRegistry& g_registry() {
    static TypeRegistry r;
    return r;
}

// ---------- 辅助宏：自动计算偏移和大小 ----------
#define REFLECT_FIELD(struct_type, field_name, field_type) \
    FieldDesc{ #field_name, #field_type, field_type, \
               offsetof(struct_type, field_name), \
               sizeof(((struct_type*)0)->field_name) }

#define REGISTER_TYPE(struct_type, ...) \
    static bool _reflect_##struct_type = [](){ \
        g_registry().register_type({ #struct_type, sizeof(struct_type), { __VA_ARGS__ } }); \
        return true; \
    }()
```

**使用示例**：
```cpp
struct HealthComponent {
    float current = 100.0f;
    float max     = 100.0f;
};

REGISTER_TYPE(HealthComponent,
    REFLECT_FIELD(HealthComponent, current, FieldType::Float),
    REFLECT_FIELD(HealthComponent, max,     FieldType::Float)
);

// 运行时查询
const TypeDesc* desc = g_registry().find("HealthComponent");
for (const auto& f : desc->fields) {
    printf("Field: %s, type: %s, offset: %zu\n", f.name.c_string(), f.type_name.c_string(), f.offset);
}
```

这个最小实现解决了什么？
- **运行期类型发现**：通过 `TypeRegistry` 可以在不读 C++ 头文件的情况下知道任意组件的字段列表。
- **通用 Inspector 绘制**：遍历 `fields` 数组，根据 `type` 和 `offset` 用 ImGui 绘制对应的输入框。
- **零外部工具链**：不需要代码生成器、不需要 Python 脚本、不需要特殊编译器支持。

还缺什么？
- **手动注册繁琐**：每新增一个组件都要写 `REGISTER_TYPE`，字段多了容易出错。
- **不支持嵌套类型**：`struct Foo { Bar inner; }` 中的 `Bar` 无法自动展开。
- **不支持数组/容器**：`std::vector<int>`、`DynamicArray` 等复杂类型没有通用的反射表示。
- **没有方法反射**：只能反射字段，不能调用组件的方法。
- **FieldType 枚举是硬编码的**：新增一种原子类型就要改枚举和 Inspector 的绘制逻辑。

## How：真实引擎的反射系统是如何一步一步复杂起来的？

### 阶段 1：手动宏注册 → 自动代码生成（解决维护成本）

**触发原因**：当组件数量超过 50 个时，手动写 `REGISTER_TYPE` 变成体力劳动，而且字段改名后容易遗漏更新反射注册代码。

**代码层面的变化**：

1. **标记驱动的代码生成器**
   在头文件中使用特殊宏或注解标记需要反射的类和字段：
   ```cpp
   REFLECT()
   struct HealthComponent {
       REFLECT_FIELD()
       float current;

       REFLECT_FIELD()
       float max;
   };
   ```

2. **构建时生成注册代码**
   用一个 Python/Clang 脚本在编译前扫描所有头文件：
   - 提取 `REFLECT()` 标记的结构体
   - 提取字段名、类型、注解（如 `range(0, 100)`、`tooltip("生命值")`）
   - 生成 `generated_reflection.cpp`，其中包含所有 `REGISTER_TYPE` 的调用

3. **构建系统集成**
   在 CMake 中添加一个预构建步骤：
   ```cmake
   add_custom_command(
       OUTPUT ${CMAKE_BINARY_DIR}/generated_reflection.cpp
       COMMAND python ${PROJECT_SOURCE_DIR}/tools/reflection_generator.py
       DEPENDS ${HEADER_FILES}
   )
   ```

**状态变化图：代码生成器的工作流程**

```
输入：
  HealthComponent.h
  └── struct HealthComponent { float current; float max; };

处理：
  1. Parser 读取 AST → 发现 struct "HealthComponent"
  2. 遍历成员 → ["current", "float"], ["max", "float"]
  3. Mustache 模板渲染 → generated_reflection.cpp

输出：
  generated_reflection.cpp
  └── g_registry().register_type({
        "HealthComponent", sizeof(HealthComponent), {
          {"current", "float", Float, 0, 4},
          {"max",     "float", Float, 4, 4}
        }
      });
```

> **代价诚实**：代码生成器引入了什么新成本？
> - **构建时间增加**：每次头文件改动都要重新跑生成器，对于大型项目可能增加数秒到数十秒。
> - **生成的代码不可见**：如果生成的 `.cpp` 文件不被提交到仓库，新成员首次构建时可能会困惑"这些函数定义在哪里"。
> - **Parser 的局限性**：自定义的 Python 脚本无法处理复杂的 C++ 模板特化、宏展开、条件编译。往往需要对标记语法做严格限制。

---

### 阶段 2：从字段反射 → 完整的对象图反射（解决嵌套、数组、方法）

**触发原因**：组件字段不再只是 `float` 和 `int`，而是嵌套结构、动态数组、资源引用（如 `MaterialHandle`、`TextureID`）。Inspector 需要能递归展开这些复杂类型。

**代码层面的变化**：

1. **统一的原子类型注册表**
   不再用硬编码的 `FieldType` 枚举，而是用一个可扩展的原子类型注册表：
   ```cpp
   struct AtomType {
       std::string name;
       size_t size;
       std::function<void*(void*)> getter;  // 返回指向实际值的指针
       std::function<std::string(void*)> to_json;
       std::function<void(void*, const std::string&)> from_json;
   };
   ```
   基础类型（`float`, `int`, `bool`, `Vec3`）预先注册，用户也可以注册自定义原子类型。

2. **复合类型与递归展开**
   `FieldDesc` 增加 `is_composite` 标志和指向子 `TypeDesc` 的指针：
   ```cpp
   struct FieldDesc {
       // ...
       bool is_composite = false;
       const TypeDesc* subtype = nullptr;
   };
   ```
   Inspector 绘制时，如果遇到 `is_composite`，递归绘制子字段，并正确计算偏移量叠加。

3. **数组/容器的反射**
   引入 `ArrayDesc` 和 `ArrayAccessor`：
   ```cpp
   struct ArrayDesc {
       size_t (*size)(void* instance);
       void*  (*get)(void* instance, size_t index);
       void   (*resize)(void* instance, size_t new_size);
   };
   ```
   通过模板为 `std::vector<T>` 生成特化的 `ArrayDesc`，让 Inspector 能动态增删数组元素。

4. **方法反射（可选）**
   为需要脚本调用或编辑器按钮触发的组件方法生成访问器：
   ```cpp
   struct MethodDesc {
       std::string name;
       std::function<void(void*, void** args)> invoke;
   };
   ```
   这使得 Inspector 可以显示一个"调用 Reset()"按钮，或让脚本系统直接调用组件方法。

---

### 阶段 3：工业级 → AI Schema 导出、网络同步与脚本一体化

**触发原因**：反射系统不再只服务于 Inspector，而是成为**序列化、网络同步、脚本绑定、AI 交互**的统一基础设施。任何新增组件都必须"反射一下就能到处用"。

**代码层面的变化**：

1. **JSON / Binary 序列化的自动生成**
   基于反射元数据，实现一个通用的 `Serializer`：
   ```cpp
   json serialize(const void* obj, const TypeDesc& desc);
   void deserialize(void* obj, const TypeDesc& desc, const json& data);
   ```
   保存场景时，遍历所有实体的组件，调用 `serialize` 即可。无需为每个组件手写序列化逻辑。

2. **AI Schema 导出**
   将 `TypeRegistry` 导出为结构化的 JSON Schema，供外部 AI Agent 消费：
   ```json
   {
     "HealthComponent": {
       "fields": {
         "current": { "type": "float", "range": [0, "max"] },
         "max":     { "type": "float", "default": 100.0 }
       }
     }
   }
   ```
   AI 在连接引擎前，先读取这份 Schema，从而知道如何构造合法的 `set_component` 请求。

3. **网络同步的字段级 Diff**
   利用反射元数据遍历组件的每个字段，计算上一帧和当前帧的差异，只同步变化了的字段。
   ```cpp
   void diff_component(void* old_state, void* new_state, const TypeDesc& desc, BitStream& out);
   ```
   这避免了整组件同步的带宽浪费，尤其对 `Transform` 这种大部分实体每帧都不变的组件至关重要。

4. **脚本绑定的零成本扩展**
   Lua / Python 绑定层不再手写 `lua_pushfloat`、`pybind11::def`，而是遍历 `TypeRegistry`，自动生成绑定代码：
   ```cpp
   for (const auto& [type_name, desc] : registry.types) {
       lua.new_usertype<type>(type_name, /* 自动展开字段 */);
   }
   ```

> **性能诚实**：
> - "通用序列化" 不是零开销。与手写序列化相比，基于反射的版本需要运行时遍历字段表、查类型、做字符串比较。对于高频网络同步（如每帧同步上千个实体），可能成为瓶颈。
> - **优化策略**：对热路径组件（如 `Transform`）手写特化序列化，反射序列化只用于编辑器、存档、低频网络包。
> - **C++20/23 静态反射**：如果编译器支持，未来可以用编译期反射生成零开销的序列化代码，但目前（2026 年）主流编译器对静态反射的支持仍不完整，代码生成器仍是更稳健的选择。

## 从源码到 ECS 的重构映射

chaos 引擎的反射系统采用 "`__attribute__((annotate))` + `chaos_header_tool` 代码生成器 + `std::tuple` 函数指针" 的架构。迁移到自研 ECS 时，核心思想可以保留，但实现应更贴近 ECS 的数据平铺哲学：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `Meta(...)` / `CLASS(...)` 注解宏 | **`REFLECT()` / `REFLECT_FIELD()` 宏** | 保留标记驱动的代码生成思想，但标记语法更简单，只针对 POD 结构体，不涉及多态基类。 |
| `chaos_header_tool`（Mustache 模板生成器）| **Python/Clang 代码生成器** | 生成内容从"函数指针元组 + 虚函数桥接"简化为"字段描述符数组 + 偏移量列表"。 |
| `TypeMeta` + `FieldAccessor`（`std::tuple` 存储函数指针）| **`TypeDesc` + `FieldDesc`** | 不再用函数指针元组做字段访问，而是直接用 `offset` 读写内存。ECS 组件是 POD，不需要 setter/getter 封装。 |
| `TypeInfoManager`（全局类型 ID 注册表）| **`ComponentRegistry`** | 将类型注册与 ECS 的组件类型 ID 分配合并。每个组件类型在注册反射的同时获得一个稳定的 `ComponentTypeID`。 |
| `EngineEditorSketumManager` + `DetailViewModel.cs`（SWIG + Sketum 双轨制）| **ImGui Inspector + JSON Schema 桥接** | 对于 vibe coding 的个人引擎，Inspector 直接用 ImGui 绘制；AI 接口直接输出 JSON Schema，无需 SWIG 和 C# 前端。 |
| `ReflectionInstance(void* instance, TypeMeta meta)` | **`(Entity, ComponentTypeID)` 坐标** | ECS 中不需要"实例指针 + 元数据"的包装器。给定 Entity 和 ComponentTypeID，就能从 World 中定位到组件内存地址，再结合 `TypeDesc` 直接解释字节。 |

> **Why ECS 更适合这个模块？**
> - **内存可解释**：chaos 的组件是多态对象，反射访问需要通过 `FieldAccessor::set(void*, void*)` 做间接调用。ECS 的组件是平铺的 POD，反射系统只需要知道 `base_ptr + offset`，直接 cast 即可读写。这更简单、更快、更容易被 AI 理解。
> - **无继承树**：chaos 的反射需要处理基类字段透查（`getBaseClassReflectionInstanceList`）。ECS 组件禁止继承，没有基类字段的问题，反射元数据是一张扁平的表。
> - **与 Query 天然结合**：反射注册表可以直接作为 ECS Query 的过滤条件。例如 `"列出所有带 HealthComponent 的实体"` 不需要字符串比较，而是用 `ComponentTypeID` 做位掩码匹配。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ 反射元数据（`TypeDesc`、`FieldDesc`）是只读的注册表数据，没有隐藏状态。组件实例的内存布局完全由这些元数据描述。 |
| **自描述** | ✅ `TypeRegistry` 可以随时导出完整的 JSON Schema。AI 不需要读 C++ 头文件就能知道 `HealthComponent.current` 是 `float`、偏移量是 0。 |
| **确定性** | ✅ 反射注册表在引擎启动时构建完成，之后只读。给定相同的编译产物，注册表内容完全一致。 |
| **工具边界** | ✅ MCP 接口可暴露：`get_component_types()`、`get_component_schema(type_name)`、`set_component_field(entity, type_name, field_name, json_value)`。 |
| **Agent 安全** | ✅ `set_component_field` 在写入前通过 `TypeDesc` 校验字段类型和大小，拒绝非法的 JSON 输入。对于数值字段，可以附加 `range` 注解做边界检查。 |

## "如果我要 vibe coding，该偷哪几招？"

1. **从 offsetof 开始，不要从代码生成器开始**
   如果你的引擎只有 10~20 个组件，手动写 `REGISTER_TYPE` 配合 `offsetof` 完全够用。代码生成器是 50+ 组件时才值得引入的优化。

2. **反射元数据 = Inspector + 序列化 + AI Schema 的唯一真相来源**
   不要为 Inspector 写一套描述、为序列化写另一套。让 `TypeRegistry` 成为所有工具的共同入口。

3. **组件必须是 POD，禁止虚函数和继承**
   一旦组件有了虚函数，`sizeof` 和 `offsetof` 就不再是你想象的那样，反射系统会迅速复杂化。坚持 POD，反射就是一张简单的偏移量表。

4. **给字段加注解，不要只存 name + type**
   `range(0, 100)`、`tooltip("生命值")`、`color_picker` 这些注解不会增加运行期开销（它们存在只读的 `FieldDesc` 中），但能让 Inspector 和 AI 的理解能力提升一个数量级。

5. **AI Schema 是反射系统的最高 ROI 输出**
   花 10 分钟写一个 `registry.to_json_schema()` 函数，你的 Agent 就能立刻理解引擎中所有组件的结构。这是 vibe coding 时最值得优先实现的功能之一。
