## Why：为什么游戏引擎需要一个工业级的组件系统架构？

在 [[最小ECS数据层]] 中，我们搭了一个最原始的 ECS：每个组件类型对应一个 `std::vector`，Entity 只是一个隐含的索引。这个骨架能跑 demo，但它在真实场景中会遇到三个致命问题。

想象你经营一个快递分拣中心。每个包裹（Entity）可以贴多种标签（组件）——有的贴"易碎品"，有的贴"加急件"，有的同时贴两者。你的分拣员（System）每天要做各种批量操作："所有加急件提前出库"、"易碎品加急件走特殊通道"。

### 场景 1：只想找"同时贴了易碎品和加急件"的包裹

中心里有 10,000 个包裹，其中只有 200 个同时贴了两张标签。如果没有高效的查询系统，分拣员必须翻遍全部 10,000 个包裹，逐个检查标签。这是 **O(N)** 的浪费，而且每小时都发生一次。

### 场景 2：贴新标签时，货架突然倒塌重组

一车的中间，某个包裹被判定需要加贴"冷链"标签。如果冷链包裹存放在一个普通货架上，新包裹入库可能导致整个货架推倒重建——原来摆在第 3 层的包裹被移到第 7 层。另一名正在第 3 层取货的分拣员，手里的地址单瞬间失效，可能把包裹发错城市。

### 场景 3：分拣员在不同仓库之间来回跑

即使你能快速定位那 200 个"易碎+加急"包裹，如果"易碎品信息"存在东仓库，"加急件信息"存在西仓库，分拣员每处理一个包裹就要在两栋楼之间跑一趟。CPU 管这叫**缓存未命中（Cache Miss）**——处理器不断从主内存拉取数据，批量处理的速度可能还不如逐个处理。

> **核心需求**：引擎需要一个**知道哪些实体有哪些组件**、**增删组件不崩溃**、**批量迭代时数据相邻**的存储与查询系统。这不是优化，而是"能规模化"的前提。

## What：最简化版本的组件系统长什么样？

我们先实现一个基于 **Sparse Set（稀疏集）** 的最小 ECS。它的核心思想是：把同一类组件像图书一样紧凑地排列在"密集书架"上，同时在"查询台"维护一张快递单号到书架位置的快速对照表。

```cpp
#include <vector>
#include <cstdint>
#include <cassert>
#include <algorithm>

using Entity = uint32_t;
const Entity INVALID_ENTITY = ~0u;

// ---------- SparseSet 组件池 ----------
// 想象成一个专用货架：只放某一种组件（比如"加急件"包裹的详细资料）
template<typename T>
struct SparsePool {
    std::vector<T>        dense;           // 密集书架：实际组件数据，紧凑排列
    std::vector<Entity>   dense_to_entity; // 书架第 i 本书属于哪个包裹
    std::vector<uint32_t> sparse;          // 查询台：sparse[entity] = 书架位置

    bool has(Entity e) const {
        return e < sparse.size() && sparse[e] != INVALID_ENTITY;
    }

    T& get(Entity e) {
        assert(has(e));
        return dense[sparse[e]];
    }

    void add(Entity e, const T& value) {
        if (e >= sparse.size()) sparse.resize(e + 1, INVALID_ENTITY);
        if (has(e)) { get(e) = value; return; }
        // 新书放到书架末尾，查询台登记位置
        sparse[e] = (uint32_t)dense.size();
        dense.push_back(value);
        dense_to_entity.push_back(e);
    }

    void remove(Entity e) {
        if (!has(e)) return;
        uint32_t index = sparse[e];
        uint32_t last  = (uint32_t)dense.size() - 1;
        // swap-and-pop：把最后一本书移到空位，保持书架紧凑
        dense[index] = dense[last];
        dense_to_entity[index] = dense_to_entity[last];
        sparse[dense_to_entity[last]] = index;
        dense.pop_back();
        dense_to_entity.pop_back();
        sparse[e] = INVALID_ENTITY;
    }
};

// ---------- 最小 World ----------
struct World {
    Entity next_entity = 0;

    template<typename T>
    SparsePool<T>& pool() {
        static SparsePool<T> p;
        return p;
    }

    Entity create() { return next_entity++; }

    template<typename T>
    void add(Entity e, const T& value) { pool<T>().add(e, value); }

    template<typename T>
    T& get(Entity e) { return pool<T>().get(e); }

    template<typename T>
    bool has(Entity e) { return pool<T>().has(e); }
};
```

这个最小实现解决了什么？
- **单组件遍历**：`pool.dense` 是完美连续的，迭代速度极快——就像沿着一条直线书架扫过去。
- **O(1) 增删**：`add` / `remove` 只需修改查询台的索引和书架末端的几个位置。
- **O(1) 存在性查询**：`has(entity)` 只需查一次查询台。

还缺什么？我们接下来用**问题链**的方式逐一补全。

## How：真实引擎的组件系统是如何一步一步复杂起来的？

### 问题 1：如何快速找出"同时贴了 A 和 B 标签"的包裹？

**最 naive 的做法**：分拣员拿到"易碎品+加急件"的任务单后，先去易碎品仓库翻遍所有包裹，对每一个都跑到加急件仓库查对照表确认。这样做比翻遍全中心快（如果易碎品只有 500 件），但仍然要查 500 次对照表。

**问题出在哪？** 对照表查询虽然也是 O(1)，但每次查询都要访问另一座仓库的查询台——CPU 需要不断从主内存加载不同的数据结构，缓存命中率很低。

**改进：引入 Query/View 抽象，选最短的货架遍历**

```cpp
template<typename... Cs>
struct Query {
    // 自动找出组件数量最少的货架（比如加急件只有 200 个）
    // 遍历这 200 个包裹，对每个包裹查其他仓库的对照表
};
```

这已经比 naive 全遍历好得多，但还不够理想。如果"易碎品"有 8,000 个，"加急件"有 200 个，Query 会选加急件货架遍历，但仍然要做 200 次"跨仓库查表"。

**进一步的改进：EntityID 带世代号（Generation）**

在继续优化查询之前，我们先解决一个更基础的问题：包裹单号（EntityID）用完了怎么办？

当前实现里 `next_entity` 单调递增，运行久了会溢出。而且如果一个包裹被销毁后单号立刻复用，其他拿着旧单号的分拣员会误把新包裹当旧包裹处理。

```cpp
struct EntityID {
    uint32_t index;      // 回收后复用
    uint32_t generation; // 检测悬空访问
};
```

销毁包裹时，index 放入空闲队列，generation 递增。下次复用该 index 时，旧单号因 generation 不匹配而被拒绝。这是防止"把包裹发给死人"的最小代价方案。

**代价诚实**：Sparse Set 的 `swap-and-pop` 有个隐蔽的陷阱——被移动元素的索引发生了变化。如果某个分拣员曾记住了"易碎品书架第 3 层是包裹 E5"，在 E1 被移除后，E5 可能被移到了第 2 层。因此 Sparse Set 的 ECS **严禁外部持有组件指针跨帧**。

> **swap-and-pop 状态变化图解**
>
> 假设 `Position` 货架当前状态：
> ```
> dense:          [P0, P1, P2, P3]
> dense_to_entity:[E0, E1, E2, E3]
> sparse:[0, 1, 2, 3, ...]  (E0->0, E1->1, E2->2, E3->3)
> ```
> 移除 E1 的 Position：
> ```
> 1. 查对照表：sparse[E1] = 1
> 2. 把最后一本书 P3 移到空位 1：
>    dense:          [P0, P3, P2]
>    dense_to_entity:[E0, E3, E2]
> 3. 更新 E3 的对照表：sparse[E3] = 1
> 4. 标记 E1 为无效：sparse[E1] = INVALID
> ```
> 结果：书架依然紧凑，但 E3 的位置变了。

### 问题 2：如何让"经常一起处理的包裹"住得更近？

**触发原因**：Sparse Set 在查询 `<Position, Velocity>` 时，虽然可以选最短的货架遍历，但每次都要通过查询台去另一座仓库查找对应组件。当两个仓库都很大且重叠率不高时，分拣员在两栋楼之间跑来跑去的时间成了瓶颈。

**核心洞察**：如果中心里有大量包裹都同时拥有"易碎品+加急件+冷链"标签，为什么不让它们**住在同一个托盘上**？这样分拣员只需走到托盘前，就能同时看到所有需要的信息。

**代码层面的变化**：

#### 1. Archetype（原型）概念

**Archetype** 是一组组件类型的唯一组合。例如 `{Position, Velocity}` 是一个 Archetype，`{Position, Mesh, Material}` 是另一个。每个 Archetype 管理一个 **Chunk 数组**，每个 Chunk 是一块固定大小的连续内存（如 16KB），就像物流中的标准化托盘。

```cpp
struct Chunk {
    uint8_t* data;           // 16KB 连续内存，一个托盘
    uint16_t count;          // 当前容纳的包裹数
    uint16_t capacity;       // 最大容量（由组件总大小决定）
    Archetype* archetype;    // 所属原型
};
```

在 Chunk 的 16KB 内存内部，组件按 **SoA（Structure of Arrays）** 排列——你可以理解为托盘上先放一排快递单号，再放一排易碎品资料，再放一排加急件资料：

```
[EntityID_0, EntityID_1, ...]          // 实体索引数组
[Position_0, Position_1, ...]          // Position 数组
[Velocity_0, Velocity_1, ...]          // Velocity 数组
```

这样查询 `<Position, Velocity>` 时，只需遍历该 Archetype 的所有 Chunk，一次线性扫描就能拿到两个组件的连续数组。分拣员不再需要在仓库之间穿梭。

#### 2. Archetype 图（Archetype Graph）

当包裹增加或撕掉一个标签时，它必须从一个 Archetype "迁移"到另一个 Archetype——就像把包裹从一个托盘搬到另一个托盘。

引擎维护一个有向图：节点是 Archetype，边是 "添加组件 X" 或 "移除组件 X"。例如：

```
Archetype {Pos} --add Vel--> Archetype {Pos, Vel}
```

这样 `add<Velocity>(entity)` 时，不需要遍历所有 Archetype 去查找目标，直接沿图边走即可。这就像物流中心预先规划好了"加贴冷链标签后应转入哪个区域"，而不是每次临时翻区域目录。

**状态变化图：包裹从一个 Archetype 迁移到另一个**

假设包裹 E5 当前在 Archetype `{Position}` 的 Chunk A 中：
```
Chunk A ({Position}):
  entities: [E0, E5, E9]
  positions:[P0, P5, P9]
```

执行 `add<Velocity>(E5)`：
```
1. 沿 Archetype 图找到目标：{Position, Velocity}
2. 在目标 Archetype 的某个 Chunk B 中分配一个新槽位（假设 index 1）
3. 复制 E5 的 Position 到新槽位
4. 构造新的 Velocity 组件
5. 在 Chunk A 中用 swap-and-pop 移除 E5
   Chunk A: entities:[E0, E9], positions:[P0, P9]
   Chunk B: entities:[E2, E5], positions:[P2, P5], velocities:[V2, V5]
```

> **代价诚实**：
> - **添加/移除组件不再是 O(1)**。它变成了"复制组件数据 + 构造/析构"，代价与组件大小成正比——就像把包裹从旧托盘取出来，重新打包进新托盘。
> - **组件指针更不稳定**。只要包裹的标签组合发生变化，它的所有数据都会被搬到新 Chunk 中，旧引用全部失效。
> - **内存碎片化**。Archetype 组合爆炸时，可能出现很多只装了一两个包裹的小托盘，浪费托盘空间。
>
> 因此工业级 ECS（如 Flecs、Unity DOTS）通常提供 `Sparse` 或 `DontFragment` 标记：让那些频繁增删的组件（如 Buff、状态标记）走 out-of-line 存储，不参与 Archetype 迁移。

> #### Bevy 的 Archetype 双轨存储有何特别？
>
> 我们在上面描述的 Archetype + Chunk 模型，并不是凭空想象——它几乎就是 Bevy `bevy_ecs` 的存储核心。Bevy 使用 **Archetype + Table + SparseSet 的双轨存储**（详见 [[Bevy-bevy_ecs-源码解析：Component 存储与 Archetype]]）。
>
> 在 Bevy 中：
> - `Archetype` 是"组件组合的指纹"——具有完全相同组件集合的所有实体属于同一个 Archetype。
> - `Table` 是实际的 SoA 列式存储，相当于我们说的 Chunk。但 Bevy 的 Table 不限于 16KB，它是一个动态增长的数组，按 Archetype 的组件类型分列存储。
> - `SparseSet` 作为第二轨存储，专门服务那些被标记为 `#[component(storage = "SparseSet")]` 的组件。
>
> 这种设计的精妙之处在于：**热数据走 Table（Archetype），冷数据走 SparseSet**。Bevy 的 `Edges` 机制还缓存了 Archetype 之间的迁移路径，让实体增删组件时的图查找是 O(1) 的。
>
> 如果你读 chaos 源码时觉得 `EntityComponentManager` 的嵌套哈希表很绕，那么 Bevy 的 `bevy_ecs` 是一个绝佳的对照——它用 Rust 的类型系统把 Archetype 模型表达得极为紧凑。

### 问题 3：10 万包裹、几百个分拣员同时工作，如何不撞车？

**触发原因**：当中心里有 10 万个包裹、数百个 System、数十种 Archetype 时，直接在分拣过程中修改托盘结构会导致遍历崩溃，Query 重复构建也成为瓶颈。

想象一个分拣员正在扫描"易碎品+加急件"托盘，另一个分拣员突然把其中某个包裹的"易碎品"标签撕掉了——这个包裹需要立刻搬到另一个托盘。第一个分拣员的扫描指针突然就踩空了。

**代码层面的变化**：

#### 1. 命令缓冲（Command Buffer）

所有 "添加/移除组件"、"创建/销毁包裹" 的请求不再立即执行，而是写入当前帧的命令缓冲队列。在 tick 的**安全点**（通常是帧尾或专门的 `ComponentManagement` Phase）统一执行。这保证了分拣员在扫描 Chunk 时，Chunk 的结构是稳定的。

```cpp
world.commands().add_component(e, Velocity{1,0,0});  // 只入队
world.commands().apply();                            // 安全点统一执行
```

#### 2. Query 缓存与增量更新

第一次构造 `Query<Position, Velocity>` 时，引擎扫描所有 Archetype，把匹配的 Chunk 加入缓存列表。当新 Archetype 被创建（因为出现了新的组件组合），引擎只检查这个新 Archetype 是否匹配已有 Query，增量更新缓存。后续迭代直接遍历缓存列表，无需重新匹配。

这就像分拣主管第一次接到"易碎+加急"任务时，走遍全中心把所有符合条件的托盘记下来画成地图。之后每天直接按地图走，只有新区域建成时才去确认一下。

#### 3. Chunk 内的并行调度

同一个 Archetype 可能有几十个 Chunk。System 可以将这些 Chunk 分发给线程池并行处理。由于不同 Chunk 的内存互不重叠，不存在数据竞争（前提是 System 只写组件数据、不改结构）。

更精细的做法是：在 Query 级别声明 `Read<Position>` / `Write<Velocity>`，调度器自动检测 System 间的读写依赖，决定哪些可以并行、哪些必须串行。

#### 4. 非碎片化组件（Sparse / Shared）

对于 `Cooldown`、`IsSelected` 这种小而多变的组件，标记为 `Sparse` 后它们存储在单独的 Sparse Set 中，不进入 Chunk。这样既保留了 Archetype 对核心热数据的缓存友好性，又避免了边缘数据导致的频繁迁移。
> #### UE 与 Bevy 的并行策略为何截然不同？
>
> 上面的"Chunk 内并行"和"Query 级读写依赖"听起来很自然，但如果你是 UE 开发者，可能会觉得陌生——因为 UE 的并行模型完全不同。
>
> **UE 的并行是"线程级"的**：Game Thread 负责逻辑 Tick，Render Thread 负责渲染准备，RHI Thread 负责 GPU 提交，Async Loading Thread 负责资源加载（详见 [[UE-专题：引擎 Tick 结构与编辑器差异]]）。一个 Actor 的 `Tick()` 通常在 Game Thread 上串行执行，Component 之间没有"基于数据依赖自动并行"的概念。UE 的并行粒度是"任务"（如 `ParallelFor` 批量更新实例化组件），而不是"System"。
>
> **Bevy 的并行是"System 级"的**：Bevy 的 `Schedule` 会在启动时分析所有 System 的 `SystemParam` 访问模式（`&ComponentA` 是读，`&mut ComponentB` 是写），构建一张依赖图，自动决定哪些 System 可以并行、哪些必须串行（详见 [[Bevy-bevy_ecs-源码解析：Schedule 与 System 并行调度]]）。这不需要开发者手动分配线程，调度器会自动把无冲突的 System 分派到任务池。
>
> 两者的根本差异源于对象模型：UE 的 Actor-Component 是**紧耦合的继承树**，一个 Actor 的多个 Component 可能通过虚函数互相调用，无法静态分析数据依赖；而 ECS 的 System 是**纯函数**，它声明了自己要读什么、写什么，调度器才能安全地并行。
>
> 对于自研引擎的选择：
> - 如果你从 UE 背景转过来，**不要试图在 ECS 中复刻 UE 的 Tick Group 模型**。ECS 的并行是"细粒度的、数据驱动的"，不是"把大任务扔到不同线程"。
> - 如果你想学习 System 级并行的最佳实践，Bevy 的 `multi_threaded.rs` 执行器比 Flecs 的 pipeline 更容易读懂（Rust 的类型系统让依赖分析非常显式）。

## 从源码到 ECS 的重构映射

本笔记的灵感部分来自对 chaos 引擎组件系统的源码分析。chaos 当前采用的是**面向对象的 Component-Based 架构**（而非 ECS）。如果我们要把它的设计思想迁移到 ECS，映射关系如下：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| EntityProxy（重量级代理对象，含 Prefab 元数据、GameActor 创建配置） | **轻量 EntityID** + 组件组合 | 将 Proxy 中的状态拆分为 PrefabInstance、GameActorConfig 等独立组件，不再把元数据绑在实体对象上。 |
| Component（多态基类，含虚函数 initialize() / clear()） | **POD 结构体** | 移除虚函数和继承，组件变为纯数据切片。初始化逻辑移到 System 的 OnAdd Hook 中。 |
| PossessedComponentHandle + DatumTable（按类型隔离的 AoS 对象池） | **SparsePool** 或 **Archetype Chunk** | DatumTable 的按类型分池思想可映射为 Sparse Set；若追求更高查询性能，则升级为 Archetype Chunk。 |
| EntityComponentManager（Map<EntityID, Map<Group, Array<Handle>>>) | **Archetype 注册表** + **Query 缓存** | 去掉嵌套哈希表，用 Archetype 图和 Query Cache 替代。实体只记录它属于哪个 Archetype 和 Chunk 中的哪个索引。 |
| Manager（System 的抽象基类，通过 processComponentsManagement 处理延迟增删） | **System** + **命令缓冲** | Manager 的延迟队列思想直接对应 ECS 的命令缓冲。Tick 分阶段执行：先处理命令缓冲，再执行业务 System。 |
| LevelTickManager（拓扑排序调度） | **System Scheduler** | 保留依赖图和拓扑排序，但调度粒度从 Manager 实例变为 Query 迭代任务，支持 Chunk 级并行。 |

> **Why ECS 更适合这个模块？**
> - **Cache Locality**：chaos 的 EntityComponentManager 是多层哈希表，查询一个组件需要多次指针追逐。ECS 的 Archetype 让相关组件在物理内存上相邻，批量迭代时 cache miss 大幅减少。
> - **组合灵活性**：chaos 的 EntityObjectType（Entity / GameActor / GameObject）是硬编码的枚举，新增类型需要改框架。ECS 下，是不是 GameActor 只是一个 GameActorTag 组件的有无，完全数据驱动。
> - **AI 可观测性**：chaos 的组件存储在 DatumTable 中，AI 要理解一个模块必须先读它的 C++ 头文件。ECS 的组件是平铺的 POD，可以通过反射注册表自动导出 Schema，AI 无需读源码就能知道世界的状态结构。

## 跨引擎速查：三者为何走向不同的路？

chaos、UE、Bevy 在"如何组织游戏世界的数据"这个问题上，给出了三种截然不同的答案。理解它们的差异，能帮你更坚定地选择 ECS。

**chaos：OOP + 手动分池**
chaos 的 `EntityProxy` 是重量级对象，`DatumTable` 按组件类型隔离存储，但每个实体在每种组件类型中最多只有一个实例。查询一个实体有哪些组件需要遍历多层哈希表。这种设计的优点是概念简单（C++ 开发者熟悉），缺点是缓存局部性差、组合灵活性低。

**UE：Actor-Component 继承树**
UE 不是 ECS。一个 `AActor` 拥有多个 `UActorComponent`，Component 通过虚函数 `TickComponent()` 更新。`UObject` 垃圾回收负责生命周期，Tick Group（`TG_PrePhysics`、`TG_PostPhysics` 等）负责调度顺序。这种模型的优点是编辑器集成极深、蓝图可视化极成熟；缺点是 Actor 是重量级对象、Component 更新顺序靠人工配置、无法自动并行。

**Bevy：原生 ECS + Archetype 双轨**
Bevy 从第一天就是 ECS。Entity 只是一个 `u32` ID，Component 是纯数据，System 是普通函数。存储层用 Archetype + Table 保证缓存局部性，用 SparseSet 处理边缘数据，用 Schedule 的依赖图实现自动并行。这种模型的优点是性能天花板高、组合极度灵活；缺点是调试时需要适应"数据在哪"的思维转换，对习惯了 OOP 的开发者有学习曲线。

| 维度 | chaos | UE | Bevy |
|------|-------|-----|------|
| **对象模型** | OOP + 手动 ECS | Actor-Component 继承树 | 原生 ECS（Entity + Component + System） |
| **组件存储** | `DatumTable`（按类型隔离的 AoS 池） | `UActorComponent` 实例挂在 Actor 上 | Archetype + Table（SoA）+ SparseSet 双轨 |
| **查询机制** | `EntityComponentManager` 嵌套哈希表 | `GetComponents()` 遍历 Actor 的组件数组 | `Query<&A, &mut B>`，基于 Archetype 匹配 |
| **生命周期** | `EntityProxyManager` 单调递增 ID | UObject GC + 引用计数 | EntityCommands + generation 回收 |
| **并行策略** | `LevelTickManager` 拓扑排序 + 手动任务化 | 线程级分离（GT/RT/RHI） | System 依赖图 + 任务池自动并行 |
| **命令缓冲** | 每个 Manager 各自维护延迟队列 | 无（直接修改） | `Commands` 全局延迟队列 + `ApplyDeferred` 同步点 |

> **为什么自研引擎应该参考 Bevy 而不是 UE？**
> 如果你从零开始写引擎，UE 的 Actor-Component 模型会给你一个"看起来很工业级"的起步，但很快你会遇到"Actor 太重、Tick 太串行、并行太难加"的瓶颈。Bevy 的 ECS 设计已经被证明可以在现代硬件上高效扩展，而且它的 Rust 源码比 UE 的 C++ 宏反射更容易理解数据流向。
>
> 当然，**编辑器成熟度是 UE 的绝对优势**。阶段 6 的自研 UI 框架和编辑器基础，需要吸收 UE Slate 和编辑器的经验，而不是 Bevy 的极简风格。在"运行时数据层"参考 Bevy，在"编辑器交互层"参考 UE，是自研引擎的务实路线。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | Entity 只是 index + generation，所有可变状态都在组件中。Archetype 和 Chunk 是只读的存储元数据。 |
| **自描述** | 组件类型通过模板元编程或宏自动注册到 ComponentRegistry，AI 可以在运行期获取组件名、字段列表、大小。 |
| **确定性** | 给定相同的初始世界 + 相同的命令缓冲序列，第 N 帧的状态 bit 级一致。Chunk 分配顺序不影响组件数据的逻辑顺序。 |
| **工具边界** | MCP 接口可以暴露 query_entities(component_mask)、add_component(entity, type_name, json_data)、remove_component(entity, type_name)。 |
| **Agent 安全** | 命令缓冲天然支持事务：AI 的批量操作先进入 staging buffer，经审批（或沙箱回滚）后再合并到主世界。 |

## 如果我要 vibe coding，该偷哪几招？

1. **先偷 Sparse Set，再考虑 Archetype**
   如果你只有不到 10 种组件、几千个实体，Sparse Set 完全够用，代码量小、调试友好。当你的 Profiler 显示 System 迭代耗时过高且瓶颈在 cache miss 时，再引入 Archetype。

2. **EntityID 必须带 generation**
   这是防止悬空访问的最小代价方案。没有 generation 的 ECS 在调试期会给你带来地狱般的 use-after-free。

3. **命令缓冲是生命线**
   不要在 System 遍历中直接 `add_component` 或 `destroy_entity`。哪怕你的 ECS 只有 100 个实体，这也是定时炸弹。

4. **热数据进 Chunk，冷数据进 Sparse**
   `Position`、`Velocity`、`Transform` 放 Archetype Chunk；`IsHovered`、`DebugLabel`、`TemporaryBuff` 放 Sparse Set 或 out-of-line。

5. **Query 必须缓存**
   每次 System tick 都重新扫描所有 Archetype 是自杀行为。构造 Query 时只匹配一次，之后通过事件或增量维护缓存列表。
