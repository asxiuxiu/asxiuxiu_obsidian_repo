# 组件系统架构

> **前置依赖**：[[最小ECS数据层]]、[[容器与分配器]]、[[引擎基础类型与平台抽象]]
> **本模块增量**：在最小 ECS 骨架上增加 Sparse Set 查询、Archetype 存储模型、Chunk-based 批量迭代与并行调度能力。
> **下一步**：[[场景图与变换]]，因为实体需要在三维空间中建立关系才能被渲染和物理系统消费。

---

## Why：为什么游戏引擎需要一个工业级的组件系统架构？

在 [[最小ECS数据层]] 中，我们搭建了一个最原始的 ECS：每个组件类型对应一个 `std::vector`，Entity 只是一个隐含的索引。这个骨架能跑，但它会在三个真实场景中迅速崩溃：

### 场景 1：只想遍历"有 Position 和 Velocity 的实体"

假设世界里有 10,000 个实体，其中只有 200 个是"可移动的"。如果没有查询系统，你的 MovementSystem 必须遍历全部 10,000 个实体，逐一检查它是否同时拥有 `Position` 和 `Velocity`。这是 **O(N)** 的浪费，而且每帧都发生。

### 场景 2：动态增删组件导致指针/索引失效

一帧的中间，某个逻辑决定给实体添加一个 `Collider` 组件。如果 `Collider` 存储在 `std::vector<Collider>` 中，`push_back` 可能导致整个数组重新分配，所有指向旧元素的指针和引用瞬间失效。正在遍历该数组的渲染线程可能因此崩溃。

### 场景 3：Cache Miss 让批量计算变成随机内存访问

即使你能找到那 200 个可移动实体，如果它们的 `Position` 和 `Velocity` 分散在内存各处（比如每个组件都是 `new` 出来的独立对象），CPU 需要不断从主存拉取数据。批量更新的速度可能还不如逐个 `GameObject.Update()`。

> **核心需求**：引擎需要一个**知道哪些实体有哪些组件**、**增删组件不崩溃**、**批量迭代时内存连续**的存储与查询系统。这不是优化，而是"能规模化"的前提。

---

## What：最简化版本的组件系统架构长什么样？

我们先实现一个基于 **Sparse Set** 的最小 ECS。它的核心思想是：
- 每个组件类型维护一个**稀疏数组**（sparse）和一个**密集数组**（dense）。
- sparse 数组以 `EntityID` 为索引，快速定位；dense 数组存储真正的组件数据，保证内存连续。
- 查询多组件时，遍历最小的 dense 数组，再用 sparse 检查其他组件是否存在。

```cpp
#include <vector>
#include <cstdint>
#include <cassert>
#include <algorithm>

using Entity = uint32_t;
const Entity INVALID_ENTITY = ~0u;

// ---------- SparseSet 组件池 ----------
template<typename T>
struct SparsePool {
    std::vector<T>     dense;      // 实际组件数据，连续存储
    std::vector<Entity> dense_to_entity; // dense[i] 属于哪个实体
    std::vector<uint32_t> sparse;  // sparse[entity] = index in dense (或 INVALID)

    bool has(Entity e) const {
        return e < sparse.size() && sparse[e] != INVALID_ENTITY;
    }

    T& get(Entity e) {
        assert(has(e));
        return dense[sparse[e]];
    }

    void add(Entity e, const T& value) {
        if (e >= sparse.size()) sparse.resize(e + 1, INVALID_ENTITY);
        if (has(e)) { get(e) = value; return; }
        sparse[e] = (uint32_t)dense.size();
        dense.push_back(value);
        dense_to_entity.push_back(e);
    }

    void remove(Entity e) {
        if (!has(e)) return;
        uint32_t index = sparse[e];
        uint32_t last  = (uint32_t)dense.size() - 1;
        // swap-and-pop 保持 dense 连续
        dense[index] = dense[last];
        dense_to_entity[index] = dense_to_entity[last];
        sparse[dense_to_entity[last]] = index;
        dense.pop_back();
        dense_to_entity.pop_back();
        sparse[e] = INVALID_ENTITY;
    }
};

// ---------- 最小 World ----------
struct World {
    Entity next_entity = 0;

    template<typename T>
    SparsePool<T>& pool() {
        static SparsePool<T> p;
        return p;
    }

    Entity create() { return next_entity++; }

    template<typename T>
    void add(Entity e, const T& value) { pool<T>().add(e, value); }

    template<typename T>
    T& get(Entity e) { return pool<T>().get(e); }

    template<typename T>
    bool has(Entity e) { return pool<T>().has(e); }
};
```

这个最小实现解决了什么？
- **单组件遍历**：`pool.dense` 是完美连续的，迭代速度极快。
- **O(1) 增删**：`add` / `remove` 都只需修改几个数组索引。
- **O(1) 存在性查询**：`has(entity)` 只需一次数组索引访问。

还缺什么？
- **多组件联合查询**还不够快（虽然比全遍历好，但仍需对每个实体做多次 `has` 检查）。
- **EntityID 永不回收**，长时间运行会溢出。
- **没有线程安全**，多线程添加组件会竞争。
- **没有内存上限控制**，sparse 数组可能随最大 EntityID 线性膨胀。

## How：真实引擎的组件系统是如何一步一步复杂起来的？

### 阶段 1：Sparse Set → 能用（解决查询与增删性能）

**触发原因**：最小实现的 `std::vector` 查询和遍历已经比 OOP 的 `new/delete` + 虚函数快一个数量级，但多组件查询仍是痛点。

**代码层面的变化**：

1. **引入 Query/View 抽象**
   ```cpp
   template<typename... Cs>
   struct Query {
       // 自动找出最小组件池，遍历它的 dense 数组
       // 对每个实体检查是否拥有所有其他组件
   };
   ```

2. **EntityID 带版本号（Generation）**
   ```cpp
   struct EntityID {
       uint32_t index;      // 回收后复用
       uint32_t generation; // 检测悬空访问
   };
   ```
   - 销毁实体时，index 放入空闲队列；generation 递增。
   - 下次复用该 index 时，旧 Handle 因 generation 不匹配而被拒绝。

3. **Sparse Set 的内存优化**
   - sparse 数组不再用 `std::vector<uint32_t>` 直接以 Entity index 为下标，而是分**页存储**（page table）：每页 4096 个条目，按需分配。
   - 这避免了"创建了一个 ID 为 100 万的实体后，sparse 数组瞬间膨胀到 4MB"的问题。

**状态变化图：swap-and-pop 如何保持 dense 连续**

假设 `Position` 池当前状态：
```
dense:          [P0, P1, P2, P3]
dense_to_entity:[E0, E1, E2, E3]
sparse:[0, 1, 2, 3, ...]  (E0→0, E1→1, E2→2, E3→3)
```

现在移除 `E1` 的 `Position`：
```
1. 取出 E1 的 dense index: sparse[E1] = 1
2. 将最后一个元素 P3 swap 到 index 1:
   dense:          [P0, P3, P2]
   dense_to_entity:[E0, E3, E2]
3. 更新 E3 的 sparse 指向新位置: sparse[E3] = 1
4. 标记 E1 为无效: sparse[E1] = INVALID
```

结果：
```
dense:          [P0, P3, P2]   ← 依然连续，无空洞
dense_to_entity:[E0, E3, E2]
sparse:[0, INV, 2, 1, ...]
```

> **代价诚实**：swap-and-pop 的代价是**被移动元素的索引发生了变化**。如果外部有人缓存了 `&dense[3]`，这个指针会在移除 E1 后失效。因此 Sparse Set 的 ECS 严禁外部持有组件指针跨帧。

---

### 阶段 2：Archetype + Chunk → 好用（解决多组件联合查询的缓存局部性）

**触发原因**：Sparse Set 在查询 `<Position, Velocity>` 时，虽然可以选最短的 dense 数组遍历，但每次都要通过 sparse 数组去另一个池查找对应组件。当两个池都很大且重叠率不高时，缓存命中率下降。

**核心洞察**：如果世界里有大量实体都同时拥有 `Position` + `Velocity` + `Mesh`，为什么不让它们**住在同一个内存块里**？

**代码层面的变化**：

1. **Archetype（原型）概念**
   - Archetype = 一组组件类型的唯一组合。例如 `{Position, Velocity}` 是一个 Archetype，`{Position, Mesh, Material}` 是另一个。
   - 每个 Archetype 管理一个 **Chunk 数组**，每个 Chunk 是一块固定大小的连续内存（如 16KB）。

2. **Chunk 内的 SoA 布局**
   ```cpp
   struct Chunk {
       uint8_t* data;           // 16KB 连续内存
       uint16_t count;          // 当前容纳的实体数
       uint16_t capacity;       // 最大容量（由组件总大小决定）
       Archetype* archetype;    // 所属原型
   };
   ```
   在 Chunk 的 16KB 内存内部，组件按 **SoA（Structure of Arrays）** 排列：
   ```
   [EntityID_0, EntityID_1, ...]          // 实体索引数组
   [Position_0, Position_1, ...]          // Position 数组
   [Velocity_0, Velocity_1, ...]          // Velocity 数组
   ```
   这样查询 `<Position, Velocity>` 时，只需遍历该 Archetype 的所有 Chunk，一次线性扫描就能拿到两个组件的连续数组。

3. **Archetype 图（Archetype Graph）**
   - 当实体增加/删除组件时，它从一个 Archetype "迁移"到另一个 Archetype。
   - 引擎维护一个有向图：节点是 Archetype，边是 "添加组件 X" 或 "移除组件 X"。
   - 例如：Archetype `{Pos}` --add Vel--> Archetype `{Pos, Vel}`。
   - 这样 `add<Velocity>(entity)` 时，不需要遍历所有 Archetype 去查找目标，直接沿图边走即可。

**状态变化图：实体从一个 Archetype 迁移到另一个**

假设实体 `E5` 当前在 Archetype `{Position}` 的 Chunk A 中：
```
Chunk A ({Position}):
  entities: [E0, E5, E9]
  positions:[P0, P5, P9]
```

执行 `add<Velocity>(E5)`：
```
1. 沿 Archetype 图找到目标：{Position, Velocity}
2. 在目标 Archetype 的某个 Chunk B 中分配一个新槽位（假设 index 1）
3. 复制 E5 的 Position 到新槽位
4. 构造新的 Velocity 组件
5. 在 Chunk A 中用 swap-and-pop 移除 E5
   Chunk A: entities:[E0, E9], positions:[P0, P9]
   Chunk B: entities:[E2, E5], positions:[P2, P5], velocities:[V2, V5]
```

> **代价诚实**：
> - **添加/移除组件不再是 O(1)**。它变成了"复制组件数据 + 构造/析构"，代价与组件大小成正比。
> - **组件指针更不稳定**。只要实体的组件组合发生变化，它的所有组件都会被搬到新 Chunk 中，旧指针全部失效。
> - **内存碎片化**。Archetype 组合爆炸时，可能出现很多只装了一两个实体的小 Chunk。
>
> 因此工业级 ECS（如 Flecs、Unity DOTS）通常提供 `Sparse` 或 `DontFragment` 标记：让那些频繁增删的组件（如 Buff、状态标记）走 out-of-line 存储，不参与 Archetype 迁移。

---

### 阶段 3：工业级 → 命令缓冲、并行调度与 Query 缓存

**触发原因**：当场景里有 10 万实体、数百个 System、数十种 Archetype 时，直接在 Tick 中修改组件结构会导致遍历崩溃，Query 重复构建也成为瓶颈。

**代码层面的变化**：

1. **命令缓冲（Command Buffer）**
   - 所有 "添加/移除组件"、"创建/销毁实体" 的请求不再立即执行，而是写入当前帧的命令缓冲队列。
   - 在 tick 的**安全点**（通常是帧尾或专门的 `ComponentManagement` Phase）统一执行。
   - 这保证了 System 在遍历 Chunk 时，Chunk 的结构是稳定的。

2. **Query 缓存与增量更新**
   - 第一次构造 `Query<Position, Velocity>` 时，引擎扫描所有 Archetype，把匹配的 Chunk 加入缓存列表。
   - 当新 Archetype 被创建（因为出现了新的组件组合），引擎只检查这个新 Archetype 是否匹配已有 Query，增量更新缓存。
   - 后续迭代直接遍历缓存列表，无需重新匹配。

3. **Chunk 内的并行调度**
   - 同一个 Archetype 可能有几十个 Chunk。System 可以将这些 Chunk 分发给线程池并行处理。
   - 由于不同 Chunk 的内存互不重叠，不存在数据竞争（前提是 System 只写组件数据、不改结构）。
   - 更精细的做法是：在 Query 级别声明 `Read<Position>` / `Write<Velocity>`，调度器自动检测 System 间的读写依赖，决定哪些可以并行、哪些必须串行。

4. **非碎片化组件（Sparse / Shared）**
   - 对于 `Cooldown`、`IsSelected` 这种小而多变的组件，标记为 `Sparse` 后它们存储在单独的 Sparse Set 中，不进入 Chunk。
   - 这样既保留了 Archetype 对核心热数据的缓存友好性，又避免了边缘数据导致的频繁迁移。

## 从源码到 ECS 的重构映射

本笔记的灵感部分来自对 chaos 引擎组件系统的源码分析。chaos 当前采用的是**面向对象的 Component-Based 架构**（而非 ECS）。如果我们要把它的设计思想迁移到 ECS，映射关系如下：

| chaos 源码结构 | ECS 等价物 | 迁移说明 |
|---------------|-----------|---------|
| `EntityProxy`（重量级代理对象，含 Prefab 元数据、GameActor 创建配置） | **轻量 EntityID** + 组件组合 | 将 Proxy 中的状态拆分为 `PrefabInstance`、`GameActorConfig` 等独立组件，不再把元数据绑在实体对象上。 |
| `Component`（多态基类，含虚函数 `initialize()` / `clear()`） | **POD 结构体** | 移除虚函数和继承，组件变为纯数据切片。初始化逻辑移到 System 的 `OnAdd` Hook 中。 |
| `PossessedComponentHandle` + `DatumTable`（按类型隔离的 AoS 对象池） | **SparsePool** 或 **Archetype Chunk** | DatumTable 的"按类型分池"思想可映射为 Sparse Set；若追求更高查询性能，则升级为 Archetype Chunk。 |
| `EntityComponentManager`（`Map<EntityID, Map<Group, Array<Handle>>>`） | **Archetype 注册表** + **Query 缓存** | 去掉嵌套哈希表，用 Archetype 图和 Query Cache 替代。实体只记录它属于哪个 Archetype 和 Chunk 中的哪个索引。 |
| `Manager`（System 的抽象基类，通过 `processComponentsManagement` 处理延迟增删） | **System** + **命令缓冲** | Manager 的延迟队列思想直接对应 ECS 的命令缓冲。Tick 分阶段执行：先处理命令缓冲，再执行业务 System。 |
| `LevelTickManager`（拓扑排序调度） | **System Scheduler** | 保留依赖图和拓扑排序，但调度粒度从"Manager 实例"变为"Query 迭代任务"，支持 Chunk 级并行。 |

> **Why ECS 更适合这个模块？**
> - **Cache Locality**：chaos 的 `EntityComponentManager` 是多层哈希表，查询一个组件需要多次指针追逐。ECS 的 Archetype 让相关组件在物理内存上相邻，批量迭代时 cache miss 大幅减少。
> - **组合灵活性**：chaos 的 `EntityObjectType`（Entity / GameActor / GameObject）是硬编码的枚举，新增类型需要改框架。ECS 下，"是不是 GameActor" 只是一个 `GameActorTag` 组件的有无，完全数据驱动。
> - **AI 可观测性**：chaos 的组件存储在 `DatumTable` 中，AI 要理解一个模块必须先读它的 C++ 头文件。ECS 的组件是平铺的 POD，可以通过反射注册表自动导出 Schema，AI 无需读源码就能知道世界的状态结构。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | ✅ Entity 只是 `index + generation`，所有可变状态都在组件中。Archetype 和 Chunk 是只读的存储元数据。 |
| **自描述** | ✅ 组件类型通过模板元编程或宏自动注册到 `ComponentRegistry`，AI 可以在运行期获取组件名、字段列表、大小。 |
| **确定性** | ✅ 给定相同的初始世界 + 相同的命令缓冲序列，第 N 帧的状态bit级一致。Chunk 分配顺序不影响组件数据的逻辑顺序。 |
| **工具边界** | ✅ MCP 接口可以暴露 `query_entities(component_mask)`、`add_component(entity, type_name, json_data)`、`remove_component(entity, type_name)`。 |
| **Agent 安全** | ✅ 命令缓冲天然支持事务：AI 的批量操作先进入 staging buffer，经审批（或沙箱回滚）后再合并到主世界。 |

## "如果我要 vibe coding，该偷哪几招？"

1. **先偷 Sparse Set，再考虑 Archetype**
   - 如果你只有不到 10 种组件、几千个实体，Sparse Set 完全够用，代码量小、调试友好。
   - 当你的 Profiler 显示 "System 迭代耗时过高" 且瓶颈在 cache miss 时，再引入 Archetype。

2. **EntityID 必须带 generation**
   - 这是防止悬空访问的最小代价方案。没有 generation 的 ECS 在调试期会给你带来地狱般的 use-after-free。

3. **命令缓冲是生命线**
   - 不要在 System 遍历中直接 `add_component` 或 `destroy_entity`。哪怕你的 ECS 只有 100 个实体，这也是定时炸弹。

4. **热数据进 Chunk，冷数据进 Sparse**
   - `Position`、`Velocity`、`Transform` 放 Archetype Chunk；`IsHovered`、`DebugLabel`、`TemporaryBuff` 放 Sparse Set 或 out-of-line。

5. **Query 必须缓存**
   - 每次 System tick 都重新扫描所有 Archetype 是自杀行为。构造 Query 时只匹配一次，之后通过事件或增量维护缓存列表。
