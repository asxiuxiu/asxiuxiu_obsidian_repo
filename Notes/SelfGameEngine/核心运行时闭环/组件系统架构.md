## 问题 0：为什么阶段 2 的 `std::vector` 组件数组不够用？

在 [[最小ECS数据层]] 中，我们为每种组件类型维护了一个 `std::vector`，Entity 只是一个隐含的索引。这个方案写起来只有几十行代码，但它有三个在真实场景中会被放大到致命的缺陷。

### 场景绑定：10,000 个包裹的分拣中心

想象你经营一个快递分拣中心。每个包裹（Entity）可以贴多种标签（组件）——有的贴"易碎品"，有的贴"加急件"，有的同时贴两者。你的分拣员（System）每天要做各种批量操作："所有加急件提前出库"、"易碎品加急件走特殊通道"。

**缺陷一：空槽位浪费**。中心里有 10,000 个包裹，但只有 3,000 个贴了"加急件"标签。如果你的"加急件货架"是一个固定 10,000 格的阵列，其中 7,000 格是空的，分拣员每扫一格就有 70% 的概率扑空。`std::vector` 虽然不会物理分配 10,000 个对象，但如果你想用 Entity ID 直接做数组索引，就必须接受这种稀疏浪费。

**缺陷二：多标签查询的跨仓库跳转**。分拣员拿到"易碎品+加急件"的任务单后，发现易碎品存在东仓库、加急件存在西仓库。即使他手里有两张清单，每处理一个包裹都要在两栋楼之间跑一趟——CPU 管这叫**缓存未命中（Cache Miss）**。当两个仓库都很大且重叠率不高时，跨仓库查表的时间会成为瓶颈。

**缺陷三：增删标签时地址失效**。某个包裹突然被判定需要加贴"冷链"标签。如果冷链货架是独立分配的，这个包裹的"加急件"资料可能从货架第 3 层被移到第 7 层（因为 `std::vector` 中间插入需要挪动后续所有元素）。另一名分拣员昨天记住了"第 3 层是包裹 E5"，今天再去取就拿了别人的资料。

> **核心矛盾**：`std::vector` 用 Entity ID 做索引的方案，假设了"每个实体都有这个组件"和"组件顺序永不变"。这两个假设在真实引擎中都不成立。

## 问题 1：如何让单组件遍历达到极致速度？

既然 `std::vector` 的稀疏索引方案有缺陷，我们先解决最简单的问题：**如何只遍历拥有某个组件的所有实体，且数据在内存上完全连续？**

### 分支 A：Sparse Set（稀疏集）

Sparse Set 的核心思想是双数组结构：
- **密集数组（dense）**：只存实际有该组件的实体数据，紧凑排列。
- **稀疏数组（sparse）**：以 Entity ID 为索引，值是 dense 数组中的位置（或无效标记）。

```cpp
// flags: -O2
template<typename T>
struct SparsePool {
    Array<T>        dense;           // 紧凑书架：实际组件数据
    Array<Entity>   dense_to_entity; // 书架第 i 本书属于哪个包裹
    Array<uint32_t> sparse;          // 查询台：sparse[entity] = 书架位置

    bool has(Entity e) const {
        return e.index < sparse.size() && sparse[e.index] != INVALID;
    }

    T& get(Entity e) {
        assert(has(e));
        return dense[sparse[e.index]];
    }

    void add(Entity e, const T& value) {
        if (e.index >= sparse.size()) sparse.resize(e.index + 1, INVALID);
        if (has(e)) { get(e) = value; return; }
        sparse[e.index] = dense.size();
        dense.push_back(value);
        dense_to_entity.push_back(e);
    }

    void remove(Entity e) {
        if (!has(e)) return;
        uint32_t index = sparse[e.index];
        uint32_t last  = dense.size() - 1;
        // swap-and-pop：把最后一本书移到空位，保持书架紧凑
        dense[index] = dense[last];
        dense_to_entity[index] = dense_to_entity[last];
        sparse[dense_to_entity[last].index] = index;
        dense.pop_back();
        dense_to_entity.pop_back();
        sparse[e.index] = INVALID;
    }
};
```

**为什么 Sparse Set 比 `std::vector` 快？**
- **遍历**：`dense` 数组是完美连续的，System 可以用一个 `for (size_t i = 0; i < dense.size(); ++i)` 线性扫过，没有空槽位。
- **存在性查询**：`has(entity)` 是 O(1) 的数组索引。
- **增删**：`add` / `remove` 也是 O(1)，但有个代价——`remove` 用 `swap-and-pop` 保持密集，被移动的元素索引变了。

**swap-and-pop 的状态变化图**：

假设 `Position` 货架当前状态：
```
dense:          [P0, P1, P2, P3]
dense_to_entity:[E0, E1, E2, E3]
sparse:          [0,  1,  2,  3]  (E0->0, E1->1, E2->2, E3->3)
```
移除 E1 的 Position：
```
1. 查查询台：sparse[E1] = 1
2. 把最后一本书 P3 移到空位 1：
   dense:          [P0, P3, P2]
   dense_to_entity:[E0, E3, E2]
3. 更新 E3 的查询台：sparse[E3] = 1
4. 标记 E1 为无效：sparse[E1] = INVALID
```
结果：书架依然紧凑，但 **E3 的位置变了**。

> **代价诚实**：Sparse Set 严禁外部持有组件指针跨帧。任何指向 `dense[i]` 的指针，在另一次 `remove` 后都可能指向错误的实体或变成悬空指针。这是 ECS 设计的核心约束之一——System 只能通过 Entity ID 访问组件，不能缓存裸指针。

### 分支 B：直接链表或哈希表

有人可能会想：既然要紧凑，为什么不维护一个 `HashMap<Entity, T>`？

哈希表确实支持 O(1) 的随机访问，但遍历一个 `HashMap` 时，元素在内存上是不连续的。当 System 要处理 10,000 个 `Position` 组件时，哈希表的桶跳转会让 CPU 的预取器失效，实际性能可能比 `std::vector` 的稀疏扫描还差。

链表则更糟——每个节点一次指针追逐，cache miss 是百分之百。

**结论**：在"单组件批量遍历"这个子问题上，Sparse Set 是压倒性优选的方案。EnTT（最广泛使用的 C++ ECS 库）和 chaos 的早期组件池都选择了这条路径。

> **参考：工业界怎么选？**
> - **chaos** 的 `TypedDatumTable` 本质上是一种按类型隔离的稀疏集变体——它为每种组件类型维护独立的对象池，通过句柄间接引用。
> - **EnTT** 的 `basic_storage` 是 Sparse Set 的工业级实现，支持稳定迭代器（通过 swapped 标记延迟更新）和分页式 sparse 数组（减少稀疏数组的内存浪费）。
> - **UE 没有这个问题**——因为 UE 的 `UActorComponent` 不是按"批量遍历所有同类型组件"优化的。UE 的渲染系统遍历的是 `UWorld` 中的 `UPrimitiveComponent` 列表，这个列表是注册时维护的，不是 ECS 式的存储。

## 问题 2：Entity ID 的设计空间——如何防止"把包裹发给死人"？

在继续优化查询之前，我们必须先解决一个更基础的问题：包裹单号用完了怎么办？

### 场景：单号复用的灾难

当前实现里 Entity ID 单调递增，运行久了会溢出。更致命的是：如果一个包裹被销毁后单号立刻复用，其他拿着旧单号的分拣员会误把新包裹当旧包裹处理。

```cpp
// 危险的做法
Entity e = world.create();  // e = 42
world.destroy(e);           // 42 被回收
Entity e2 = world.create(); // e2 = 42（复用了！）
// 某个 System 还拿着旧单号 42，以为它指向原来的实体
```

### 分支 A：单调递增 + 延迟回收

永远不复用 ID，直到溢出。这对于 32 位 ID 来说，在大多数游戏场景中其实是够用的（几十亿个实体）。但 overflow 后的行为是未定义的，而且内存中的 sparse 数组会随最大 ID 增长，即使大部分实体已被销毁。

### 分支 B：Index + Generation（世代计数）

这是 ECS 引擎的标准做法。Entity ID 拆分为两部分：

```cpp
struct Entity {
    uint32_t index;      // 回收后复用
    uint32_t generation; // 检测悬空访问
};
```

销毁实体时，index 放入空闲队列，generation 递增。下次复用该 index 时，旧单号因 generation 不匹配而被拒绝。

```cpp
struct EntityRegistry {
    Array<uint32_t> generations;  // generations[index] = 当前世代
    Array<uint32_t> free_list;    // 可回收的 index
    uint32_t next_index = 0;

    Entity create() {
        if (!free_list.empty()) {
            uint32_t idx = free_list.back();
            free_list.pop_back();
            return {idx, generations[idx]};
        }
        generations.push_back(0);
        return {next_index++, 0};
    }

    bool is_alive(Entity e) const {
        return e.index < generations.size() && generations[e.index] == e.generation;
    }

    void destroy(Entity e) {
        if (!is_alive(e)) return;
        generations[e.index]++;  // 世代递增
        free_list.push_back(e.index);
    }
};
```

**适用场景**：几乎所有现代 ECS 引擎。
**隐藏代价**：`generation` 数组是随最大 index 增长的，但这是一个连续的 `uint32_t` 数组，内存开销极小（100 万个实体约 4MB）。

### 分支 C：UUID（128 位全局唯一标识符）

某些 OOP 引擎使用 UUID 作为实体标识符，彻底消除复用冲突。但 UUID 的随机性让它无法直接做数组索引，必须配合哈希表使用——这摧毁了 Sparse Set 的 O(1) 索引性能。

> **参考：工业界怎么选？**
> - **Bevy** 的 `Entity` 就是 `index + generation` 的 64 位打包（低 32 位 index，高 32 位 generation）。`Entities` 元数据数组按 index 索引，`is_alive` 检查是单次数组访问。
> - **UE** 的 `AActor` 没有 generation 概念——`AActor` 是 UObject，通过 `InternalIndex` 和垃圾回收（GC）管理生命周期。Actor 被销毁后，`AActor::IsPendingKill()` 返回 true，但旧指针不会自动指向新 Actor。UE 依赖 C++ 的裸指针和 GC 标记，而不是 generation 检测。
> - **chaos** 的 `EntityID` 是单调递增的强类型包装，没有 generation。这意味着 chaos 必须依赖外部系统（如 `EntityProxyManager`）来保证不访问已销毁实体。

**默认推荐**：`index + generation`。这是防止悬空访问的最小代价方案，没有 generation 的 ECS 在调试期会给你带来地狱般的 use-after-free。

## 问题 3：如何高效找出"同时贴了 A 和 B 标签"的包裹？

Sparse Set 解决了"遍历所有有加急件的包裹"，但没有解决"遍历所有同时有易碎品和加急件的包裹"。

### 最 naive 的做法：最短列表遍历

分拣员拿到"易碎品+加急件"的任务单后，发现加急件只有 200 个，易碎品有 8,000 个。他选择去加急件货架遍历，对每一个加急件包裹查易碎品查询台确认。

```cpp
template<typename A, typename B>
void naive_query(World& world, auto&& callback) {
    auto& pool_a = world.pool<A>();
    auto& pool_b = world.pool<B>();
    // 选最短的货架遍历
    auto& shorter = pool_a.dense.size() < pool_b.dense.size() ? pool_a : pool_b;
    auto& other   = pool_a.dense.size() < pool_b.dense.size() ? pool_b : pool_a;
    for (Entity e : shorter.dense_to_entity) {
        if (other.has(e)) {
            callback(e, shorter.get(e), other.get(e));
        }
    }
}
```

这已经比翻遍全中心好得多，但还不够理想。200 次"跨仓库查表"意味着 200 次随机内存访问——如果两个仓库都不在 CPU 缓存里，每次查表都要从主内存拉取数据。

### 更深层的矛盾：缓存局部性 vs 查询灵活性

Sparse Set 的问题在于：**相关数据在物理内存上不相邻**。`Position` 在东仓库，`Velocity` 在西仓库，System 每处理一个实体都要在两栋楼之间跑一趟。

如果中心里有大量包裹都同时拥有"易碎品+加急件+冷链"标签，为什么不让它们**住在同一个托盘上**？这样分拣员只需走到托盘前，就能同时看到所有需要的信息。

这就是 **Archetype（原型）** 存储的核心洞察。

### 分支 A：Archetype + Chunk（按组件组合分组）

**Archetype** 是一组组件类型的唯一组合。例如 `{Position, Velocity}` 是一个 Archetype，`{Position, Mesh, Material}` 是另一个。每个 Archetype 管理一个 **Chunk 数组**，每个 Chunk 是一块固定大小的连续内存（如 16KB），就像物流中的标准化托盘。

```cpp
struct Chunk {
    uint8_t* data;           // 16KB 连续内存
    uint16_t count;          // 当前容纳的实体数
    uint16_t capacity;       // 最大容量（由组件总大小决定）
    Archetype* archetype;    // 所属原型
};
```

在 Chunk 的 16KB 内存内部，组件按 **SoA（Structure of Arrays）** 排列：

```
[EntityID_0, EntityID_1, EntityID_2, ...]   // 实体索引数组
[Position_0, Position_1, Position_2, ...]   // Position 数组
[Velocity_0, Velocity_1, Velocity_2, ...]   // Velocity 数组
```

这样查询 `<Position, Velocity>` 时，只需遍历该 Archetype 的所有 Chunk，一次线性扫描就能拿到两个组件的连续数组。分拣员不再需要在仓库之间穿梭。

**状态变化图：实体从一个 Archetype 迁移到另一个**

假设实体 E5 当前在 Archetype `{Position}` 的 Chunk A 中：
```
Chunk A ({Position}):
  entities: [E0, E5, E9]
  positions:[P0, P5, P9]
```

执行 `add<Velocity>(E5)`：
```
1. 沿 Archetype 图找到目标：{Position, Velocity}
2. 在目标 Archetype 的某个 Chunk B 中分配新槽位（假设 index 1）
3. 复制 E5 的 Position 到新槽位
4. 构造新的 Velocity 组件
5. 在 Chunk A 中用 swap-and-pop 移除 E5
   Chunk A: entities:[E0, E9], positions:[P0, P9]
   Chunk B: entities:[E2, E5], positions:[P2, P5], velocities:[V2, V5]
```

> **代价诚实**：
> - **增删组件不再是 O(1)**。它变成了"复制组件数据 + 构造/析构"，代价与组件大小成正比。
> - **组件指针更不稳定**。只要实体的组件组合发生变化，它的所有数据都会被搬到新 Chunk 中，旧引用全部失效。
> - **内存碎片化**。Archetype 组合爆炸时，可能出现很多只装了一两个实体的小 Chunk，浪费托盘空间。

### 分支 B：多 Sparse Set 的联合索引

另一种思路是保留 Sparse Set，但为每种组件组合维护一个预计算的交集列表。当实体增删组件时，更新所有受影响的交集列表。

这个方案的代价是：组件组合的数量随组件类型指数增长，维护交集列表的更新成本可能超过 Archetype 的数据迁移成本。

> **参考：工业界怎么选？**
> - **Bevy** 选择了 **Archetype + Table + SparseSet 双轨存储**。
>   - `Archetype` 是组件组合的指纹，具有完全相同组件集合的所有实体属于同一个 Archetype。
>   - `Table` 是实际的 SoA 列式存储，相当于 Chunk。但 Bevy 的 Table 是不限大小的动态数组（不是固定 16KB），按 Archetype 的组件类型分列存储。
>   - `SparseSet` 作为第二轨存储，专门服务那些被标记为稀疏存储的组件。
>   - 这种设计的精妙之处在于：**热数据走 Table（Archetype），冷数据走 SparseSet**。`Edges` 机制还缓存了 Archetype 之间的迁移路径，让实体增删组件时的图查找是 O(1) 的。
> - **Unity DOTS** 选择了 **固定 16KB Chunk** 的 Archetype 模型。16KB 的选取是为了适配 L1 Cache（通常 32KB），保证单个 Chunk 能被完整缓存。DOTS 还引入了 `SharedComponent`——值相同的实体才能放在同一个 Chunk 中，用于材质、LOD 等批量分组。
> - **EnTT** 选择了 **纯 Sparse Set（Columnar）** 路线。每种组件类型一个 Sparse Set，多组件查询时选最短的 dense 数组遍历。EnTT 通过 `group` 机制（把多个 Sparse Set 的 dense 数组同步排序）来优化特定查询模式的缓存局部性，但这不是通用方案。
> - **UE 没有这个选择**——UE 的 Actor-Component 模型下，批量查询所有"同时有 Mesh 和 Collision 的 Actor"需要遍历世界中的全部 Actor，然后对每个 Actor 检查 `OwnedComponents`。这在 ECS 视角下是 O(N×M) 的灾难，但 UE 通过渲染线程的 `PrimitiveSceneProxy` 列表和物理的 `BodySetup` 注册表，在子系统层面做了各自的手动优化，而不是在引擎核心层解决。

### 决策分析：Sparse Set vs Archetype

这里的选择不是"谁更好"，而是"你的瓶颈在哪"。

| 条件 | 推荐方案 | 理由 |
|------|---------|------|
| 实体数 < 5,000，组件类型 < 20 | **Sparse Set** | 代码简单、增删快、调试友好。Archetype 的迁移开销在这个阶段无法回收。 |
| 实体数 > 50,000，批量迭代是瓶颈 | **Archetype** | Cache locality 的收益远超迁移开销。Unity DOTS 和 Bevy 都验证了这一点。 |
| 频繁增删组件（如 Buff、状态标记） | **双轨存储** | 热数据（Transform、Mesh）进 Archetype，冷数据（IsHovered、TemporaryBuff）进 Sparse Set。 |
| 需要 Unity DOTS 级别的极致 SIMD | **固定 Chunk (16KB)** | 固定大小 Chunk 更容易预测内存访问模式，适配 Burst 编译器和 Job System。 |

**默认推荐**：从 Sparse Set 起步，在 Profiler 确认迭代耗时过高且瓶颈在 cache miss 时，再引入 Archetype。过早优化是万恶之源——Archetype 的代码复杂度（Archetype 图、Chunk 分配、跨 Table 迁移）是 Sparse Set 的 5~10 倍。

## 问题 4：组件增删与数据稳定性——为什么不能在遍历中改货架结构？

### 场景：遍历中修改的崩溃

想象一个分拣员正在扫描"易碎品+加急件"托盘，另一个分拣员突然把其中某个包裹的"易碎品"标签撕掉了——这个包裹需要立刻搬到另一个托盘。第一个分拣员的扫描指针突然就踩空了：他刚读完第 5 个包裹，准备读第 6 个，但第 5 个被 swap_remove 后，原来的第 6 个已经移到了第 5 个的位置，而现在的第 6 个是原来第 10 个。

在代码中，这意味着：

```cpp
// 危险：在遍历中直接增删组件
for (auto [e, pos, vel] : query<Position, Velocity>()) {
    if (pos.x > 100.0f) {
        world.remove<Velocity>(e);  // 当前实体从 Chunk 中移走了！
        // 迭代器可能已经失效，或者跳过某些实体，或者访问已释放内存
    }
}
```

### 分支 A：命令缓冲（Command Buffer）

所有 "添加/移除组件"、"创建/销毁实体" 的请求不再立即执行，而是写入当前帧的命令缓冲队列。在 tick 的**安全点**（通常是帧尾或专门的 `CommandFlush` Phase）统一执行。这保证了分拣员在扫描 Chunk 时，Chunk 的结构是稳定的。

```cpp
// 安全：只入队，不立即执行
world.commands().add_component(e, Velocity{1,0,0});
world.commands().remove<Velocity>(e2);
world.commands().destroy(e3);

// 在安全点统一执行
world.commands().apply();
```

**适用场景**：几乎所有 ECS 引擎的标准做法。
**隐藏代价**：
- 命令缓冲引入了**一帧延迟**——组件增删在提交后不会立即生效。如果 System A 在 Tick 前半段添加了一个组件，System B 在同 Tick 的后半段查询这个组件，是查不到的（除非在安全点之间插入显式的 flush）。
- 命令缓冲本身需要内存分配。如果每帧产生数千条命令，缓冲区的分配和释放可能成为瓶颈。

### 分支 B：双缓冲世界（Double-Buffered World）

维护两个世界状态：当前帧只读，下一帧可写。System 遍历当前帧的数据，所有修改写入下一帧。帧尾时交换两个世界的指针。

**适用场景**：确定性回放（Replay）和网络预测回滚。
**隐藏代价**：内存翻倍。对于大型世界，两个完整副本的内存开销不可接受。

### 分支 C：延迟标记 + 延迟清理

在遍历中不真正移除组件，而是打一个"待移除"标记。遍历完成后，统一处理所有标记。

**适用场景**：标记组件（如 `IsDestroyed`）的批量清理。
**隐藏代价**：System 必须在逻辑上忽略已标记的组件，这增加了每个 System 的复杂度。

> **参考：工业界怎么选？**
> - **Bevy** 使用 `Commands` 全局延迟队列。`apply_deferred` 是一个显式的 System，在 Schedule 的特定阶段运行。这意味着 Bevy 的开发者可以精确控制命令何时生效——如果你需要在两个 System 之间看到结构变化，就在它们之间插入一个 `apply_deferred` 同步点。
> - **chaos** 的每个 `Manager`（System 的抽象基类）各自维护延迟队列 `m_pending_registering_components` 和 `m_pending_unregistering_components`，在 tick 的 `ComponentManagement` Phase 统一处理。这与 ECS 的命令缓冲思想一致，但分散在每个 Manager 内部，而不是全局统一。
> - **UE 没有这个问题**——因为 UE 的 Component 增删是在 Game Thread 上同步执行的。`AActor::AddOwnedComponent` 和 `UnregisterComponent` 会立即修改 `OwnedComponents` 数组。UE 不担心"遍历中修改"，因为 UE 的 Component Tick 是通过 `PrimaryComponentTick` 逐个调用的，不是批量迭代。但这也意味着 UE 无法在 Component 级别做自动并行——每个 Component 的 `TickComponent()` 可能调用其他 Component 的方法，数据依赖是隐式的。

**默认推荐**：命令缓冲。不要在 System 遍历中直接 `add_component` 或 `destroy_entity`。哪怕你的 ECS 只有 100 个实体，这也是定时炸弹。

## 问题 5：System 并行的两种哲学

当中心里有 10 万个包裹、数百个 System 时，如何让多个分拣员同时工作而不撞车？

### 分支 A：手动线程分配（UE 的线程级分离）

UE 的并行模型是**线程级**的：
- **Game Thread (GT)**：负责逻辑 Tick、蓝图、输入、动画评估。
- **Render Thread (RT)**：负责渲染准备、视锥剔除、Draw Call 收集。
- **RHI Thread**：负责 GPU 命令提交。
- **Async Loading Thread**：负责资源加载。

一个 Actor 的 `Tick()` 通常在 Game Thread 上串行执行。Component 之间没有"基于数据依赖自动并行"的概念。UE 的并行粒度是"任务"（如 `ParallelFor` 批量更新实例化组件），而不是"System"。

**为什么 UE 选择手动线程？**
因为 UE 的 Actor-Component 是**紧耦合的继承树**。一个 Actor 的多个 Component 可能通过虚函数互相调用——`USkeletalMeshComponent::TickComponent()` 可能调用 `UAnimInstance::NativeUpdateAnimation()`，后者又可能访问同一个 Actor 上的 `UCharacterMovementComponent`。这种调用关系是动态的、运行时才确定的，调度器无法在编译期或启动时分析数据依赖。

### 分支 B：数据依赖分析自动并行（Bevy 的 System 级并行）

Bevy 的 `Schedule` 会在启动时分析所有 System 的访问模式：
- `&ComponentA` 是只读访问
- `&mut ComponentB` 是可写访问
- `Query<&A, &mut B>` 声明了这个 System 要读 A、写 B

基于这些声明，调度器构建一张依赖图：
- 两个只读 A 的 System 可以并行。
- 一个读 A、一个读 B（A ≠ B）的 System 可以并行。
- 一个写 A、另一个读 A 的 System 必须串行。

```cpp
// 伪代码：Bevy 的依赖分析逻辑
for (system_a in systems) {
    for (system_b in systems) {
        if (system_a.writes & system_b.reads)  conflict!;
        if (system_a.writes & system_b.writes) conflict!;
    }
}
```

这种模型的前提是：**System 是纯函数**，它声明了自己要读什么、写什么，调度器才能安全地并行。

**为什么 Bevy 可以自动并行？**
因为 ECS 的 System 不通过虚函数互相调用。一个处理 `Position` 和 `Velocity` 的 System 不会偷偷去修改另一个 Actor 的 `Health`——除非它在 Query 中显式声明了 `&mut Health`。这种**显式数据依赖**是自动并行的前提。

### 决策分析

| 维度 | UE 手动线程 | Bevy 自动 System 并行 |
|------|------------|----------------------|
| **前提条件** | Actor-Component 动态调用 | System 纯函数 + 显式依赖声明 |
| **并行粒度** | 粗（线程级：GT/RT/RHI） | 细（System 级：任意无冲突 System） |
| **开发者负担** | 需要手动分配任务到线程 | 自动，但要求正确声明读写 |
| **适用场景** | 复杂继承树、编辑器深度集成 | 批量数据处理、大规模实体 |
| **默认实现** | 不推荐从零复刻 | 推荐，但需理解依赖声明规则 |

**默认推荐**：采用 Bevy 的 System 级自动并行，但保留 UE 式的 Phase（阶段）概念。把 System 按功能分组（如 `PreUpdate`、`Update`、`PostUpdate`、`Render`），在同一 Phase 内部自动并行，跨 Phase 严格串行。这既保证了并行的粒度，又避免了不同生命周期阶段的 System 互相干扰。

## 问题 6：热数据与冷数据的双轨存储

### 场景：一小撮标记组件拖垮整个 Archetype

假设你的世界有 10,000 个实体，其中 9,990 个都有 `{Position, Velocity, Mesh}`，这是核心 Archetype。但其中有 100 个实体正处于"中毒"状态，你给它加了一个 `PoisonDebuff` 组件。如果 `PoisonDebuff` 也走 Archetype，这 100 个实体必须从主 Archetype 迁移到一个新的 `{Position, Velocity, Mesh, PoisonDebuff}` Archetype。更糟糕的是，中毒效果每 3 秒消失一次——这些实体在主 Archetype 和毒 Archetype 之间来回搬家。

这就是 **Archetype Churn（原型搅动）**——频繁的结构变化让 Archetype 的优势变成劣势。

### 分支 A：全部走 Archetype

简单统一，但遇到频繁增删的小组件时性能雪崩。

### 分支 B：双轨存储（Bevy 的 Table + SparseSet）

Bevy 的解决方案是允许开发者为组件选择存储策略：

```rust
#[derive(Component)]
struct Position { x: f32, y: f32, z: f32 }  // 默认：Table 存储

#[derive(Component)]
#[component(storage = "SparseSet")]
struct PoisonDebuff { damage_per_sec: f32 }   // 稀疏存储
```

- `Position`、`Velocity`、`Mesh` 等热数据走 `Table`（Archetype），享受批量迭代的缓存局部性。
- `PoisonDebuff`、`IsHovered`、`DebugLabel` 等冷数据走 `SparseSet`，增删时不需要迁移实体，只需更新一个独立 Sparse Set 的 dense/sparse 数组。

**代价诚实**：
- 查询 `<Position, PoisonDebuff>` 时，System 需要遍历 `Position` 的 Table（很快），对每个实体查 `PoisonDebuff` 的 Sparse Set（随机访问）。这比两个组件都在同一个 Table 里慢，但比因为 `PoisonDebuff` 频繁增删导致大量实体在 Archetype 间迁移要快得多。
- 双轨存储增加了存储层的复杂度——World 需要同时管理 Tables 和 SparseSets，Query 需要知道每种组件在哪个存储里。

### 分支 C：Chunk 内标记（Unity DOTS 的 Enableable Component）

Unity DOTS 提供了 `IEnableableComponent`——组件物理上存在，但可以通过 `Enabled` 标记临时禁用。查询时可以筛选只返回 Enabled 的组件。

这避免了增删组件导致的 Archetype 迁移，但代价是：禁用组件仍然占用 Chunk 内存，而且查询时需要额外检查标记位。

> **参考：工业界怎么选？**
> - **Bevy** 的 `StorageType::Table` vs `StorageType::SparseSet` 是开发者的显式选择。Bevy 的默认是 Table，因为大多数组件是热数据。
> - **Unity DOTS** 的 `EnableableComponent` 适合"经常开关但很少真正移除"的场景，如技能冷却、碰撞体开关。
> - **EnTT** 没有 Archetype，所以不存在 Churn 问题。但 EnTT 的 `storage` 支持 `erase`（标记删除）和 `remove`（真正删除），`erase` 是 O(1) 的标记操作，不影响其他实体。
> - **UE 的对照**：UE 的 `UActorComponent` 可以通过 `SetActive(false)` 禁用，但 Component 实例仍然存在。这与 DOTS 的 Enableable Component 类似，但 UE 的禁用是 UObject 级别的，不涉及 ECS 式的存储优化。

**默认推荐**：
- 核心组件（Transform、Mesh、Material、PhysicsBody）默认走 Archetype/Table。
- 提供显式注解或模板特化，让开发者标记"稀疏组件"（如 `struct PoisonDebuff : SparseComponent`）。
- 如果某类组件的增删频率超过每帧实体数的 1%，考虑将其改为稀疏存储。

## 从 OOP 到 ECS：三引擎的组件模型对照

chaos、UE、Bevy 在"如何组织游戏世界的数据"这个问题上，给出了三种截然不同的答案。理解它们的差异，能帮你更坚定地选择 ECS，也更清楚 ECS 化迁移时该保留什么、该抛弃什么。

### chaos：OOP + 手动分池

chaos 的组件系统采用**面向对象的 Component-Based 架构**。每个实体有一个代理对象，组件通过句柄指向按类型隔离的对象池。查询一个实体有哪些组件需要遍历多层哈希表。

**ECS 重构映射**：
- 代理对象中的状态 → 拆分为独立的 ECS 组件（如 `PrefabInstance`、`GameActorConfig`）。
- 组件基类的虚函数 `initialize()` / `clear()` → 移除虚函数，初始化逻辑移到 System 的 OnAdd Hook。
- 按类型隔离的对象池 → 映射为 Sparse Set 或 Archetype Chunk。
- 嵌套哈希表的 Entity-Component 映射 → 替换为 Archetype 注册表 + Query Cache。
- Manager 的延迟队列 → 映射为全局命令缓冲。

### UE：Actor-Component 继承树

UE 不是 ECS。一个 Actor 拥有多个 Component，Component 通过虚函数 `TickComponent()` 更新。`UObject` 垃圾回收负责生命周期，Tick Group 负责调度顺序。

**为什么 UE 不用 ECS？**
不是因为 ECS 不好，而是因为 UE 的编辑器生态、蓝图可视化、Actor 继承树已经深度绑定了 OOP 模型。迁移到 ECS 意味着重写整个编辑器工作流——这是一个数十人年的工程。

**ECS 重构映射**：
- `AActor` → 轻量 EntityID + `ActorTag` 组件。
- `UActorComponent` 继承树 → 扁平化为独立组件。`USceneComponent` 的 Transform 能力变成 `Transform` 组件，`UPrimitiveComponent` 的渲染能力变成 `Mesh` + `Material` 组件。
- `OwnedComponents` 数组 → 不再需要，组件存在性由 Archetype 或 Sparse Set 隐式表示。
- `TickComponent()` 虚函数 → System 的普通函数，通过 Query 批量迭代。
- GC 生命周期 → EntityRegistry 的 generation 回收。

### Bevy：原生 ECS + Archetype 双轨

Bevy 从第一天就是 ECS。Entity 只是一个 ID，Component 是纯数据，System 是普通函数。存储层用 Archetype + Table 保证缓存局部性，用 SparseSet 处理边缘数据，用 Schedule 的依赖图实现自动并行。

**为什么自研引擎应该参考 Bevy 而不是 UE？**
如果你从零开始写引擎，UE 的 Actor-Component 模型会给你一个"看起来很工业级"的起步，但很快你会遇到"Actor 太重、Tick 太串行、并行太难加"的瓶颈。Bevy 的 ECS 设计已经被证明可以在现代硬件上高效扩展，而且它的 Rust 源码比 UE 的 C++ 宏反射更容易理解数据流向。

当然，**编辑器成熟度是 UE 的绝对优势**。阶段 6 的自研 UI 框架和编辑器基础，需要吸收 UE Slate 和编辑器的经验，而不是 Bevy 的极简风格。在"运行时数据层"参考 Bevy，在"编辑器交互层"参考 UE，是自研引擎的务实路线。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|-----------------|
| **状态平铺** | Entity 只是 index + generation，所有可变状态都在组件中。Archetype 和 Chunk 是只读的存储元数据。 |
| **自描述** | 组件类型通过模板元编程或宏自动注册到 ComponentRegistry，AI 可以在运行期获取组件名、字段列表、大小。 |
| **确定性** | 给定相同的初始世界 + 相同的命令缓冲序列，第 N 帧的状态 bit 级一致。Chunk 分配顺序不影响组件数据的逻辑顺序。 |
| **工具边界** | MCP 接口可以暴露 `query_entities(component_mask)`、`add_component(entity, type_name, json_data)`、`remove_component(entity, type_name)`。 |
| **Agent 安全** | 命令缓冲天然支持事务：AI 的批量操作先进入 staging buffer，经审批（或沙箱回滚）后再合并到主世界。 |

## 默认推荐路径总结

### 默认推荐路径
- **组件存储**：每种组件类型一个 Sparse Set（dense + sparse 双数组），支持 O(1) 增删和完美连续遍历。
- **Entity ID**：`index + generation` 64 位打包，防止悬空访问和 ID 复用冲突。
- **多组件查询**：选最短的 dense 数组遍历，配合 `has()` 交叉验证；Profiler 确认 cache miss 为瓶颈后，再引入 Archetype + Chunk。
- **结构安全**：全局命令缓冲（Command Buffer），所有增删组件、创建销毁实体操作延迟到帧尾安全点统一执行。
- **System 调度**：Bevy 风格的依赖分析自动并行，但保留 UE 式 Phase（`PreUpdate` / `Update` / `PostUpdate` / `Render`），同 Phase 内自动并行，跨 Phase 严格串行。
- **双轨存储**：热数据（`Transform`、`Mesh`、`Material`、`PhysicsBody`）走 Archetype/Table；冷数据（`IsHovered`、`DebugLabel`、`TemporaryBuff`）显式标记为稀疏存储。

### UE 工业级设计吸收
- **Phase（阶段）概念**：UE 的 `TickGroup`（`TG_PrePhysics`、`TG_DuringPhysics`、`TG_PostPhysics`）把不同生命周期的逻辑严格隔离。自研 ECS 的 Phase 划分直接吸收这一思想，避免 System 在错误的时机互相干扰。
- **命令缓冲的延迟执行**：UE 的 `ScopedMovement` 在批量变换期间推迟传播，避免中间状态多次触发回调。ECS 的命令缓冲把这一思想推广到所有结构变更——增删组件、挂接父子关系都延迟到安全点。
- **热/冷数据分离**：UE 的 `UActorComponent` 有 `PrimaryComponentTick` 和 `SecondaryComponentTick` 之分，本质上也是把"每帧必 tick"和"偶尔 tick"的组件分开处理。ECS 的双轨存储（Table vs SparseSet）是同一原理在数据层的表达。

### ECS 映射
- **组件存储**：`TypedDatumTable` / `basic_storage` → `SparsePool<T>` 或 `ArchetypeChunk`。不再按类型隔离对象池，而是把同类型组件拍平到密集数组。
- **Entity 生命周期**：`InternalIndex` + GC 标记 → `index + generation` 的显式回收。不再依赖 UObject GC，Entity 的生死由 Registry 的世代计数精确控制。
- **System 调度**：`TickComponent()` 虚函数 + `TickGroup` → 纯函数 System + 依赖图 + Phase。每个 System 声明读写签名，调度器自动分析并行性。
- **查询缓存**：`PrimitiveSceneProxy` 列表等手动维护的遍历数组 → Query Cache。Archetype 的匹配结果在 System 构造时缓存，运行时只增量更新。

### 条件切换路径
| 条件 | 切换方案 | 触发信号 |
|------|---------|---------|
| 实体数 > 50,000 且 Profiler 显示 System 迭代 cache miss > 30% | Sparse Set → Archetype + Chunk | 帧时间中 ECS 迭代占比突然跳升 |
| 某类组件增删频率 > 每帧实体数的 1% | Archetype → SparseSet（双轨） | Archetype Churn 导致 Chunk 碎片化 |
| 需要 Unity DOTS 级别的 Burst SIMD | 动态 Table → 固定 16KB Chunk | 矩阵批量运算成为瓶颈 |
| 需要确定性回放（Deterministic Replay） | 命令缓冲 → 双缓冲 World | 网络同步或回放系统接入 |

### 扩展路径
- **固定 Chunk 分配器**：16KB Chunk 对齐 L1 Cache，支持 SIMD 批量加载。
- **跨 World 实体引用**：`EntityRef { world_id, entity }` 支持子场景、流式加载、网络复制时的跨世界引用。
- **Deterministic Scheduling**：System 执行顺序完全确定，消除并行调度中的竞争不确定性，服务网络同步和回放。
- **Component 版本控制**：每个组件数组维护 `version` 计数器，支持 `Query<Changed<T>>` 级别的增量查询，替代全量遍历。

## 如果我要 vibe coding，该偷哪几招？

1. **先偷 Sparse Set，再考虑 Archetype**
   如果你只有不到 10 种组件、几千个实体，Sparse Set 完全够用，代码量小、调试友好。当你的 Profiler 显示 System 迭代耗时过高且瓶颈在 cache miss 时，再引入 Archetype。

2. **EntityID 必须带 generation**
   这是防止悬空访问的最小代价方案。没有 generation 的 ECS 在调试期会给你带来地狱般的 use-after-free。

3. **命令缓冲是生命线**
   不要在 System 遍历中直接 `add_component` 或 `destroy_entity`。哪怕你的 ECS 只有 100 个实体，这也是定时炸弹。

4. **热数据进 Table，冷数据进 Sparse**
   `Position`、`Velocity`、`Transform` 放 Archetype Chunk；`IsHovered`、`DebugLabel`、`TemporaryBuff` 放 Sparse Set。

5. **Query 必须缓存**
   每次 System tick 都重新扫描所有 Archetype 是自杀行为。构造 Query 时只匹配一次，之后通过事件或增量维护缓存列表。

6. **并行从 Phase 开始，不要从单个 System 开始**
   先把 System 分到 `PreUpdate` / `Update` / `PostUpdate` 等大阶段，保证阶段间的执行顺序。再在阶段内部用依赖分析自动并行。这比手动管理线程安全简单 100 倍。
