## 问题 0：为什么主场景渲染完后，画面还需要全屏后处理？

想象你刚刚完成了一款小游戏的场景渲染——角色、地形、光照全部就位。你兴奋地截图发给朋友，得到的反馈却是："画面好灰 ""太阳为什么不刺眼 ""快速转镜头时眼睛不疼吗 "

这些反馈指向同一个根源：**物理正确的光照结果，与显示设备的输出能力之间存在鸿沟。**

在真实世界中，太阳亮度可能是室内灯光的数千倍。你的渲染器在内部用浮点数精确记录了这种差异（HDR，High Dynamic Range），但普通显示器只能显示 0~255 的整数亮度（LDR，Low Dynamic Range）。如果直接把 HDR 值截断到 LDR，太阳和灯泡会看起来一样亮——画面发灰、失去层次感。

这不是唯一的断层：
- **相机光学特性**：真实相机镜头遇到强光会产生光晕（Bloom），快速移动会产生运动模糊（Motion Blur），对焦范围外的物体会虚化（Depth of Field）。
- **显示设备限制**：显示器分辨率有限，像素边界会产生锯齿（Aliasing），需要抗锯齿（TAA / FXAA）来平滑。
- **人眼适应机制**：从暗处走到亮处，人眼会自动调节敏感度（Eye Adaptation），游戏画面若不做这种模拟，玩家会感到不适。

**后处理（Post-Processing）就是填补这些断层的"最后一公里"。** 它不负责"画什么"，而是负责"怎么让画出来的东西看起来对"。

### 最 naive 的做法

最早的后处理就是"什么都不做"——直接在主渲染 Pass 里把颜色输出到屏幕。HDR 值超过 1.0 的部分直接截断（clamp），画面 flat 且生硬。

```cpp
// 主渲染 shader 的 naive 版本
void main() {
    vec3 hdrColor = computeLighting(); // 可能超过 1.0
    vec3 ldrColor = clamp(hdrColor, 0.0, 1.0); // 直接截断
    outColor = vec4(ldrColor, 1.0);
}
```

这个方案的问题不仅是"不好看"：
- **信息永久丢失**：一旦 clamp，你就永远失去了 HDR 信息，后续无法再提取高光做 Bloom。
- **无法热插拔效果**：如果某天想加 Bloom，必须回改主渲染 Shader，牵一发而动全身。
- **多效果耦合**：Tone Mapping、Bloom、FXAA 全挤在一个 Shader 里，维护噩梦。

> 这引出了第一个关键洞察：**后处理必须与主渲染解耦，以独立的 Pass 链形式存在。**

## 问题 1：后处理的"原材料"从哪来？

后处理不是凭空计算。Bloom 需要知道"哪里亮"，Motion Blur 需要知道"什么东西在动"，SSAO 需要知道"表面的深度和朝向"。这些信息从哪里来？

### 场景绑定：Bloom 为什么需要 HDR 场景颜色？

假设你想做一个最简单的 Bloom——让亮的东西发光。如果没有 HDR 中间纹理，主渲染已经把所有亮度截断到 0~1 了。此时一盏亮度 0.9 的灯和一盏亮度 10.0 的灯看起来几乎一样，Bloom 无法区分它们。

**根因分析**：后处理效果的输入质量，直接决定了输出质量。如果前端渲染阶段丢弃了关键信息，后处理无法"无中生有"。

### 分支研究：输入数据的组织方式

#### 分支 A：延迟管线的天然 GBuffer

在延迟渲染（Deferred Shading）中，主渲染阶段已经生成了 GBuffer——通常包含 Albedo、Normal、Metallic/Roughness、Emissive、Depth 等。后处理可以直接读取这些缓冲：

- **优点**：SSAO、SSR 等需要法线和深度的效果可以直接使用，无需额外 Pass。
- **隐藏代价**：GBuffer 占用大量显存带宽。如果后处理只需要 Depth，整个 GBuffer 的读取仍是带宽浪费。
- **失效条件**：前向渲染（Forward Rendering）管线没有 GBuffer，此分支不可用。

#### 分支 B：前向管线的按需生成

前向渲染不生成完整 GBuffer，而是根据后处理需求单独生成所需缓冲：

```cpp
// 伪代码：按需生成速度缓冲（Velocity Buffer）
if (motionBlurEnabled || taaEnabled) {
    renderVelocityPass(); // 仅在此情况下才生成
}
```

- **优点**：不浪费带宽，只生成实际需要的数据。
- **隐藏代价**：需要运行时判断"哪些缓冲需要生成"，增加了管线复杂度。如果运行时动态开关效果，缓冲的分配和释放需要谨慎管理。
- **失效条件**：如果多个效果分别需要不同缓冲，按需生成可能导致多个 Prepass，总开销反而高于一次性生成 GBuffer。

#### 分支 C：统一场景纹理集（Scene Texture Set）

工业级引擎通常采用折中方案：定义一组"场景纹理"（SceneColor、SceneDepth、SceneVelocity、CustomDepth 等），由渲染管线统一生成和管理，后处理阶段只读不写（除了 SceneColor 会被后处理链修改）。

```cpp
struct SceneTextures {
    TextureHandle sceneColor;      // HDR 颜色，R16G16B16A16 或类似格式
    TextureHandle sceneDepth;      // 深度缓冲
    TextureHandle velocity;        // 屏幕空间速度（XY 通道）
    TextureHandle customDepth;     // 可选：自定义深度（用于描边等）
};
```

- **优点**：接口清晰，后处理效果只需声明自己需要哪些 Scene Texture；管线可以统一决定生成策略。
- **隐藏代价**：SceneColor 必须保持 HDR 格式直到 Tone Mapping 之前，这意味着它比 LDR 纹理多占用一倍（或更多）显存。
- **失效条件**：在显存极度受限的平台（如移动端），HDR SceneColor 可能成为负担。

### 决策分析

**默认推荐：分支 C（统一场景纹理集）**，在 ECS 架构下的映射方式是：
- `SceneTextures` 作为 ECS **Resource**（全局单例），由渲染管线 System 在每帧开始时准备。
- 每个后处理效果通过 System 参数声明自己读取哪些纹理（如 `Res<SceneTextures>`）。
- 这与 UE 的 Scene Texture 参数统一设计理念一致，且被 Bevy 的 `ViewTarget` 模式验证可行。

**条件切换**：在确定使用延迟渲染且后处理重度依赖 GBuffer 数据时，可让 `SceneTextures` 直接引用 GBuffer 子资源，避免额外拷贝。

**简化路径**：早期实现可以固定生成 SceneColor（HDR）+ SceneDepth，待需要 Motion Blur / TAA 时再添加 Velocity Pass。

> **遗留问题**：SceneColor 在后处理链中会被反复读写——从主渲染输出，经过 Bloom、Tone Mapping 等多次修改。如果每个 Pass 都分配新纹理，显存爆炸。下一个问题自然浮现：这些中间结果该怎么管理？

## 问题 2：多个后处理效果按什么顺序执行？

这是后处理架构的**核心设计问题**。不同效果之间存在严格的依赖关系：Bloom 必须在 Tone Mapping 之前（因为 Bloom 需要 HDR），FXAA 必须在 Tone Mapping 之后（因为 FXAA 工作在 LDR），TAA 必须在 Tone Mapping 之前（历史帧混合在 HDR 空间更准确）。

### 场景绑定：顺序错了会怎样？

假设你把 FXAA 放在了 Tone Mapping 之前。FXAA 会平滑 HDR 场景中的边缘，然后 Tone Mapping 把 HDR 压缩到 LDR——这个压缩是非线性的，会重新引入锯齿。最终画面比不做 FXAA 还糟。

又或者你把 Bloom 放在 Tone Mapping 之后。Bloom 的输入已经是 LDR 了，光晕的亮度被提前截断，效果会变得暗淡无力。

### 根因分析：后处理链是一个**数据流管道**

每个效果都读取上一阶段的输出，处理后产生新的输出。这种依赖关系本质上是一个有向无环图（DAG）。如果顺序违反数据依赖，就相当于在图中引入了反向边，导致语义错误。

### 分支研究：Pass 顺序的编排策略

#### 分支 A：硬编码顺序

直接在代码里按固定顺序调用每个效果：

```cpp
void applyPostProcess() {
    runSSAO();
    runBloom();
    runToneMapping();
    runFXAA();
    runColorGrading();
}
```

- **优点**：简单直观，无运行时开销，调试时一目了然。
- **隐藏代价**：
  - 新增效果必须修改核心代码并重新编译。
  - 无法根据场景动态禁用某个效果（如室内关 Bloom、室外开 Bloom），除非每个 `runXxx()` 内部自己判断。
  - 不同项目对顺序有不同需求（如某些项目希望 Color Grading 在 Tone Mapping 之前），硬编码缺乏灵活性。
- **失效条件**：当效果数量超过 5~6 个，且需要频繁调整顺序或动态开关时，维护成本急剧上升。

#### 分支 B：可配置效果栈（Effect Stack）

把每个效果封装为一个独立对象，按配置数组顺序执行：

```cpp
struct PostProcessEffect {
    std::string name;
    std::function<void(TextureHandle input, TextureHandle output)> execute;
    bool enabled;
};

std::vector<PostProcessEffect> effectStack;
// 从配置文件加载顺序
effectStack = loadFromConfig("postprocess.json");

TextureHandle current = sceneColor;
for (auto& effect : effectStack) {
    if (!effect.enabled) continue;
    TextureHandle next = allocateTempTexture();
    effect.execute(current, next);
    current = next;
}
```

- **优点**：顺序可配置，效果可热插拔，适合编辑器可视化调整。
- **隐藏代价**：
  - 配置文件中容易写错顺序（如把 FXAA 放在 Tone Mapping 前），导致画面错误，且错误原因难以排查。
  - 每个效果之间需要显式分配中间纹理，如果 `allocateTempTexture()` 实现粗糙，会导致显存爆炸。
  - 栈模型是线性的，无法表达分支（如"同时生成 Bloom 和 SSAO，再合并"）。
- **失效条件**：当效果之间存在复杂的资源依赖（如 TAA 需要同时读取当前帧和历史帧，而历史帧由上一帧的 TAA 产生），线性栈难以优雅表达。

#### 分支 C：渲染依赖图（RenderGraph / DAG）

将后处理链建模为图：每个效果是一个**节点（Node/PASS）**，每个中间纹理是一条**边（资源依赖）**。图的执行顺序由拓扑排序自动推导。

```cpp
// 伪代码：RenderGraph 风格的后处理编排
RenderGraph graph;

// 注册外部输入（主渲染输出）
auto sceneColor = graph.registerExternalTexture(hdrSceneColor);
auto sceneDepth = graph.registerExternalTexture(depthBuffer);

// 每个效果声明自己读取和写入的资源
auto bloomOutput = graph.addPass("Bloom", [=](RenderPassContext& ctx) {
    ctx.read(sceneColor);       // 读取 SceneColor
    auto bloomTex = ctx.createTransientTexture("BloomTex", halfResDesc);
    ctx.write(bloomTex);        // 输出 Bloom 纹理
    // ... 录制 GPU 命令
});

auto toneMapOutput = graph.addPass("ToneMapping", [=](RenderPassContext& ctx) {
    ctx.read(sceneColor);       // 读取原始 SceneColor
    ctx.read(bloomOutput);      // 读取 Bloom 结果
    auto output = ctx.createTransientTexture("PostTonemap", ldrDesc);
    ctx.write(output);
});

// 自动推导执行顺序：Bloom → ToneMapping
// 自动分配中间纹理，自动插入资源屏障
graph.compile();
graph.execute(commandList);
```

- **优点**：
  - **顺序由数据依赖决定**，不会写错。如果 Tone Mapping 读取了 Bloom 的输出，图自然保证 Bloom 先执行。
  - **自动剔除未使用节点**：如果某个效果被禁用，其输出不被任何后续节点读取，编译阶段自动剔除（Dead Pass Elimination）。
  - **内存别名**：编译器可以分析纹理生命周期，让生命周期不重叠的纹理复用同一块 GPU 内存。
  - **天然支持并行**：无依赖的节点可以并行录制命令或异步执行。
- **隐藏代价**：
  - 框架本身有学习和实现成本。需要实现图的构建、拓扑排序、资源分配、屏障推导等。
  - 调试难度增加：执行顺序是自动推导的，开发者需要工具（如图可视化）才能理解实际运行顺序。
  - 对于简单项目（只有 2~3 个后处理效果），RenderGraph 的 overhead 可能大于收益。
- **失效条件**：在极度简单的场景（如 2D 游戏只有一个 Bloom），RenderGraph 的框架复杂度不值得。

### 引擎对照

> **参考：chaos / UE / Bevy 对这个问题是怎么做的？**
> 
> 我们在解决的是「后处理 Pass 顺序编排」这个问题。
> 
> - chaos 选择了**硬编码主干顺序 + RenderGraph 子图**的混合方案。主干顺序（如 SSAO → DOF → TAA → Bloom → ToneMap → FXAA）在初始化时固定，但每个效果内部通过 RenderGraph 的 CompositePass 管理自己的子 Pass 链。这样兼顾了可预测性和资源管理优势。
> - UE 选择了**显式顺序调用 + RDG 资源管理**。`AddPostProcessingPasses` 函数内部按严格顺序调用 `AddBloomPass`、`AddTonemapPass` 等，但每个 Pass 通过 RDG（Rendering Dependency Graph）声明资源依赖。RDG 负责自动屏障和内存别名。同时 UE 通过 PostProcessVolume 的 BlendWeight 实现参数混合，但 Pass 顺序本身不由 Volume 控制。
> - Bevy（作为 ECS 原生参考）选择了**类型安全的 Schedule 编排**。`Core3d` Schedule 定义了 `PostProcess` 阶段，各效果插件通过 `.before(tonemapping)` / `.after(tonemapping)` 声明相对顺序，Rust 编译器保证约束的合法性。这种方案没有显式图结构，但依赖关系同样由数据流向决定。
> 
> **默认推荐**：默认选 **分支 C（RenderGraph / DAG）**，因为 UE 的工业级验证表明，当后处理效果超过 3 个时，自动资源管理和屏障推导的收益远超框架成本。该方案在 ECS 中的映射方式是：后处理效果的注册和参数配置通过 ECS System 完成，但执行顺序由全局 RenderGraph Resource 统一编译和调度。
> **条件切换**：如果当前阶段工程负担过重，可先实现**简化版 RenderGraph**（只支持线性链 + 自动双缓冲翻转），但需明确标注"这是完整 RenderGraph 的线性子集，未来需补全 DAG 能力和自动内存别名"。
> **简化路径**：如果项目当前只有 Bloom + ToneMapping 两个效果，可以先用硬编码顺序（分支 A）快速验证，但接口设计必须预留向 RenderGraph 迁移的扩展点。

## 问题 3：每个全屏效果如何高效执行？

后处理链中的每个效果，本质上都是"读取整张纹理，对每个像素做计算，写出新纹理"。在 1080p 分辨率下，这意味着每帧要处理 200 万个像素，且通常有 5~10 个这样的 Pass。

### 场景绑定：后处理的性能瓶颈在哪里？

假设你的游戏在 1080p@60fps 下运行，每帧只有 16.67ms。如果后处理链有 8 个全屏 Pass，每个 Pass 耗时 0.5ms，后处理就吃掉 4ms——接近总预算的 25%。在 4K 分辨率下，像素数量是 1080p 的 4 倍，同样算法可能直接吃掉 16ms，帧率暴跌到 30fps。

**根因分析**：全屏 Pass 的性能瓶颈通常不是 ALU（计算），而是**带宽**——每次 Pass 都要从显存读取整张输入纹理，再写入整张输出纹理。8 个 Pass 意味着 SceneColor 被读了 8 次、写了 8 次，带宽消耗惊人。

### 分支研究：全屏 Pass 的执行模型

#### 分支 A：传统全屏四边形 / 三角形（Pixel Shader 主导）

这是最常见的实现：画一个覆盖整个屏幕的四边形（或更优的单个 fullscreen triangle），在 Pixel Shader 中对每个像素采样输入纹理并计算输出。

```cpp
// 全屏三角形顶点 Shader（无顶点输入，用 SV_VertexID 生成）
float4 vs_main(uint vertexID : SV_VertexID) : SV_Position {
    float2 uv = float2((vertexID << 1) & 2, vertexID & 2);
    return float4(uv * 2.0f - 1.0f, 0.0f, 1.0f);
}

// Pixel Shader：Tone Mapping 示例
float4 ps_main(float2 uv : TEXCOORD0) : SV_Target0 {
    float3 hdrColor = sampleSceneColor(uv);
    float3 ldrColor = acesFilmicToneMap(hdrColor);
    return float4(ldrColor, 1.0);
}
```

- **优点**：
  - 兼容所有图形 API 和硬件，包括移动端。
  - 与图形管线的其他阶段（如 Blend State）无缝集成，容易实现"Bloom 混合到 SceneColor"这种操作。
- **隐藏代价**：
  - **Quad Overdraw**：虽然全屏四边形本身没有 overdraw，但如果后处理链中每个 Pass 都开启单独的 RenderPass，驱动和硬件的 RenderPass 切换开销不可忽视。
  - **线程组利用率**：Pixel Shader 以 2×2 quad 为单位调度。对于简单的后处理（如纯 Tone Mapping，每个像素只有几次 ALU 操作），quad 内的线程可能大部分是内存等待，ALU 利用率低。
- **失效条件**：在需要大量线程协作的场景（如降分辨率生成 Bloom Mipmap、Histogram 统计），PS 模型不如 Compute Shader 灵活。

#### 分支 B：Compute Shader

使用后处理效果的计算 Shader 版本，直接以线程组（thread group）为单位处理图像块：

```cpp
// Compute Shader：Tone Mapping
[numthreads(8, 8, 1)]
void cs_main(uint3 dispatchID : SV_DispatchThreadID) {
    uint2 pixelCoord = dispatchID.xy;
    float2 uv = (pixelCoord + 0.5) / textureSize;
    float3 hdrColor = sceneColor[pixelCoord].rgb;
    float3 ldrColor = acesFilmicToneMap(hdrColor);
    outputTexture[pixelCoord] = float4(ldrColor, 1.0);
}
```

- **优点**：
  - **无 RenderPass 开销**：不需要 `BeginRenderPass`/`EndRenderPass`，直接 `Dispatch`。
  - **更好的线程调度**：计算管线有自己的调度器，不受图形管线的 rasterizer 约束。
  - **共享内存（LDS/Shared Memory）**：可以在 thread group 内共享数据，适合需要局部邻域信息的操作（如 Blur、Histogram）。
  - **Async Compute**：在支持异步计算的平台上，可以把后处理 offload 到 Compute Queue，与 Graphics Queue 并行。
- **隐藏代价**：
  - **Barrier 手动管理**：如果 Compute Shader 的输出要作为下一个 Pixel Shader Pass 的输入，需要手动插入 UAV → SRV 的屏障。在 RenderGraph 框架下这被自动化了，但裸写 API 时容易出错。
  - **混合限制**：Compute Shader 输出到纹理时，无法使用硬件 Blend State。如果效果需要"混合到已有纹理"（如 Bloom Upsample 的叠加），Compute Shader 需要在 Shader 内手动实现混合公式，可能损失精度或性能。
  - **平台兼容性**：某些旧硬件（尤其是移动端早期的 Vulkan 驱动）对 Compute Shader 的支持和优化不如 Pixel Shader 成熟。
- **失效条件**：在需要利用硬件 Blend State 的 Pass（如 Bloom 的多级叠加），Compute Shader 反而更复杂。

#### 分支 C：Uber Shader（多效果合并）

将多个简单效果合并到一个 Shader 中，减少 Pass 数量：

```cpp
// Uber PostProcess Shader：ToneMap + ColorGrading + Vignette
float4 ps_main(float2 uv : TEXCOORD0) : SV_Target0 {
    float3 color = sampleSceneColor(uv);
    color = applyToneMap(color);
    color = applyColorGradingLUT(color);
    color = applyVignette(color, uv);
    return float4(color, 1.0);
}
```

- **优点**：将 3 个 Pass 合并为 1 个，带宽减少 66%。
- **隐藏代价**：
  - **Shader 变体爆炸**：如果每个效果都有开关（如"开/关 Vignette"），Uber Shader 的排列组合数量呈指数增长。
  - **可维护性**：不同效果的开发者需要修改同一个 Shader 文件，容易产生冲突。
  - **调试困难**：如果合并后的画面有问题，难以定位是哪个子效果导致的。
- **失效条件**：当子效果的复杂度差异很大（如简单的 Vignette 和复杂的 TAA 合并），Shader 的 register pressure 和指令缓存效率会下降。

### 引擎对照

> **参考：chaos / UE / Bevy 对这个问题是怎么做的？**
> 
> 我们在解决的是「全屏 Pass 的执行模型选择」这个问题。
> 
> - chaos 的 Legacy 后处理基类使用**全屏三角形 + Pixel Shader** 模型，但部分效果（如某些 TAA 实现）允许走 Compute Shader。基类提供了统一的 `executePrimitive()` 骨架，但具体用 PS 还是 CS 由子类决定。
> - UE 的每种后处理效果通常**同时实现 PS 和 CS 两个版本**，根据 `View.bUseComputePasses` 和平台能力动态选择。例如 Tonemap Pass 在 PC 上可能走 Compute 以减少 Graphics Queue 压力，在移动端走 Pixel Shader 以保证兼容性。同时 UE 在 Tone Mapping 阶段合并了 Color Grading、Vignette、Film Grain 等效果，形成事实上的 Uber Pass。
> - Bevy 当前版本（0.19）的后处理效果**主要基于 Pixel Shader 全屏三角形**（通过 `FullscreenShader` 辅助）。Bevy 的 `TrackedRenderPass` 在 CPU 录制阶段就去重了冗余的管线状态设置，缓解了多 Pass 的命令缓冲开销。
> 
> **默认推荐**：默认选 **分支 A（Pixel Shader 全屏三角形）作为基础，分支 B（Compute Shader）作为可选优化**，原因如下：
> - Pixel Shader 的硬件 Blend State 支持对 Bloom 等需要叠加的效果至关重要；
> - Compute Shader 的 Async Compute 优势在复杂后处理链中才能体现，个人项目初期收益有限；
> - UE 的"双版本"策略虽然最优，但实现和维护成本极高。推荐先实现 PS 版本，在性能分析确认后处理是瓶颈后，再对关键 Pass（如 Histogram、Downsample）追加 CS 版本。
> 
> **Uber Shader 的取舍**：推荐在** Tone Mapping 的最终合成阶段**采用 Uber 策略，将 ToneMap + ColorGrading + Vignette + Dither 合并为一个 Pass。这是 UE 和 Far Cry 4 都验证过的"高回报合并点"。其他效果（Bloom、TAA、SSAO）保持独立 Pass，避免 Shader 变体爆炸。

## 问题 4：时域效果的历史帧管理

TAA（Temporal Anti-Aliasing）和 Motion Blur 等效果需要复用上一帧的渲染结果。这引入了一个独特的架构挑战：**跨帧资源的生命周期管理**。

### 场景绑定：TAA 的"记忆"

TAA 的核心思想是：这帧只渲染一个偏移的采样点，然后与上一帧的历史颜色混合。如果上一帧的颜色被扔掉了，TAA 就退化为普通的单帧图像，失去抗锯齿能力。

但历史缓冲不能简单地在每帧重新分配——它需要**跨帧保持数据**。

### 根因分析：RenderGraph 的"瞬时性"与历史缓冲的"持久性"矛盾

现代 RenderGraph 框架（如 UE 的 RDG、chaos 的 RenderGraph）倾向于将纹理视为"帧内临时资源"——创建、使用、销毁，都在单帧内完成，通过内存别名最大化复用。但历史缓冲是例外：它在本帧结束时不能被销毁，而必须存活到下一帧。

### 分支研究：跨帧资源的生存策略

#### 分支 A：每帧新建 + 显式拷贝

每帧在 RenderGraph 内创建新历史缓冲，在 TAA Pass 结束时通过显式 Blit 将结果拷贝到外部持久纹理：

```cpp
// 伪代码：显式拷贝方案
TextureHandle historyIn = getPersistentHistoryTexture(); // 来自上一帧
TextureHandle historyOut = graph.createTexture(desc);    // 本帧新建

taaPass.execute(historyIn, sceneColor, historyOut);

// Pass 结束后，显式拷贝到持久存储
commandList.copyTexture(historyOut, persistentHistoryTexture);
```

- **优点**：概念简单，RenderGraph 不需要感知跨帧资源。
- **隐藏代价**：
  - **额外的拷贝开销**：每帧一次全屏 Blit，消耗带宽。
  - **同步问题**：拷贝必须在 TAA Pass 完成后执行，可能阻塞后续 Pass。
- **失效条件**：在 TAA 已经是性能瓶颈时，额外拷贝不可接受。

#### 分支 B：池化纹理 + 提取-注册循环

使用池化纹理（Pooled Texture）作为历史缓冲的持久存储。RenderGraph 提供两个关键操作：
- **提取（Extract）**：在帧结束时，将 RenderGraph 内的瞬时纹理"提取"到池化纹理中，延长其生命周期。
- **注册（Register External）**：在下一帧开始时，将池化纹理"注册"回 RenderGraph，作为外部输入。

```cpp
// 帧 N：TAA 输出后，提取到池化纹理
graph.queueTextureExtraction(taaOutput, &pooledHistoryTexture);
graph.execute();
// execute 结束后，taaOutput 的数据保留在 pooledHistoryTexture 中

// 帧 N+1：将池化纹理注册为 RenderGraph 输入
auto historyInput = graph.registerExternalTexture(pooledHistoryTexture);
taaPass.execute(historyInput, currentSceneColor, newTaaOutput);
```

- **优点**：
  - **零额外拷贝**：提取和注册操作在底层可能是同一块 GPU 内存的句柄传递，无需数据搬运。
  - **分辨率变化自动处理**：如果视口大小改变，池化纹理的尺寸不匹配，TAA 可以检测到并标记历史无效，重新累积。
- **隐藏代价**：
  - 需要框架支持提取/注册机制，增加了 RenderGraph 的复杂度。
  - 池化纹理的管理（何时真正释放、内存上限控制）需要额外的策略。
- **失效条件**：无——这是工业标准做法。

#### 分支 C：双缓冲 Ping-Pong

不依赖 RenderGraph 的提取机制，而是显式维护两个历史缓冲，每帧交换：

```cpp
struct TemporalHistory {
    TextureHandle buffers[2];
    uint32_t currentIndex = 0;
    
    TextureHandle read() { return buffers[currentIndex ^ 1]; }
    TextureHandle write() { return buffers[currentIndex]; }
    void swap() { currentIndex ^= 1; }
};
```

- **优点**：实现简单，不依赖 RenderGraph 的高级特性。
- **隐藏代价**：
  - 需要手动管理双缓冲的同步，容易出现"读了本帧正在写入的缓冲"的竞争条件。
  - 在动态分辨率场景下，双缓冲的尺寸管理更复杂。
- **失效条件**：当需要超过一帧的历史（如某些高级 TAA 或路径追踪降噪），双缓冲不够用。

### 决策分析

**默认推荐：分支 B（池化纹理 + 提取-注册循环）**，因为 UE 的 RDG 和 chaos 的 RenderGraph 都验证了这一模式。该方案在 ECS 中的映射：
- 历史缓冲作为 **Resource**（如 `TemporalHistoryResource`），由渲染系统每帧管理提取和注册。
- TAA System 读取 `Res<TemporalHistoryResource>` 作为输入，写入新的历史缓冲，帧结束时由调度 System 执行提取。

**简化路径**：如果 RenderGraph 尚未实现提取/注册机制，可以先用分支 C（双缓冲 Ping-Pong）过渡，但必须在代码中标注"未来需迁移到池化提取机制以支持动态分辨率和内存别名"。

## 问题 5：后处理链的性能预算与优化手段

### 场景绑定：帧率骤降的元凶

你加入了 Bloom、TAA、Tone Mapping、FXAA 后，在 4K 显示器上帧率从 60fps 跌到了 35fps。Profiler 显示 GPU 时间增加了 8ms——后处理成了瓶颈。

### 根因分析：后处理是带宽密集型操作

每个全屏 Pass 的耗时 ≈ (纹理读取带宽 + 纹理写入带宽 + ALU 计算) / GPU 带宽。在 4K 下，一张 RGBA16F 纹理的读写就是 4K × 2K × 8 字节 × 2（读+写）≈ 256MB 每 Pass。8 个 Pass 就是 2GB 的显存带宽——接近中端 GPU 的带宽上限。

### 分支研究：优化策略

#### 分支 A：降分辨率后处理

对部分效果使用半分辨率（1/2）或四分之一分辨率（1/4）：

```cpp
// Bloom 下采样链天然低分辨率
TextureDesc bloomDesc = sceneColorDesc;
bloomDesc.width /= 2;
bloomDesc.height /= 2;
// Bloom 在 1/2 分辨率下计算，最后上采样混合回全分辨率
```

- **适用效果**：Bloom（人眼对光晕高频细节不敏感）、SSAO（低频环境遮蔽）、Motion Blur（模糊本身掩盖了分辨率损失）。
- **不适用效果**：TAA（需要全分辨率历史匹配）、FXAA（边缘精度要求）、Tone Mapping（色彩精度要求）。

#### 分支 B：效果质量动态缩放

根据当前帧时间，动态调整后处理质量：

```cpp
if (gpuFrameTime > 16.0f) { // 掉帧了
    bloomSettings.mipCount = 3; // 从 6 级降到 3 级
    ssaoSettings.sampleCount = 4; // 从 8 采样降到 4 采样
}
```

- **优点**：保证帧率稳定，用户优先感受到流畅而非最高画质。
- **代价**：画面质量波动，需要平滑过渡避免闪烁。

#### 分支 C：Pass 合并（Uber Pass）

如问题 3 中讨论的，将多个效果合并为单个 Shader，减少带宽往返。

### 决策分析

**默认推荐**：
1. **Bloom、SSAO 默认走半分辨率**（这是 UE、Bevy、Far Cry 4 的共同选择）。
2. **Tone Mapping + Color Grading + Vignette + Dither 合并为单一 Uber Pass**。
3. **预留动态质量缩放接口**，但不作为默认开启项——个人项目初期先保证画质一致性，性能问题出现时再启用。

## 问题 6：AI 如何观察/操作后处理？

作为为 AI 协作设计的引擎，每个模块都必须回答：AI 能不能在不读 C++ 头文件的情况下理解后处理的状态？AI 能不能安全地调整后处理参数？

### 根因分析：后处理参数的"黑盒"问题

传统引擎中，后处理参数可能分散在：Shader 的 `#define`、C++ 的硬编码常量、材质实例的参数、体积触发器的属性……AI 若想调整 Bloom 强度，可能需要理解 Volume 混合权重、Priority、Blendable 材质参数等多层概念。

### 推荐方案：ECS 组件驱动的后处理配置

在 ECS 架构下，后处理参数应该**平铺为组件**：

```cpp
// 后处理参数组件（主世界，可序列化、可反射）
struct PostProcessStack {
    bool bloomEnabled;
    float bloomIntensity;
    float bloomThreshold;
    
    TonemappingMode tonemapping; // enum: None / Reinhard / ACES / AgX
    float exposure;
    
    bool fxaaEnabled;
    // ... 其他效果参数
};

// AI Schema 自动生成
// {
//   "type": "PostProcessStack",
//   "fields": {
//     "bloomIntensity": { "type": "float", "min": 0.0, "max": 10.0 },
//     "tonemapping": { "type": "enum", "values": ["None", "Reinhard", "ACES", "AgX"] }
//   }
// }
```

**关键设计点**：
- **组件即配置**：相机实体挂载 `PostProcessStack` 组件，AI 通过 MCP 接口 `set_component(camera, PostProcessStack, { bloomIntensity: 2.5 })` 即可调整。
- **效果开关即时生效**：移除 `bloomEnabled = false` 时，渲染系统下一帧自动跳过 Bloom Pass（RenderGraph 的 Dead Pass Elimination）。
- **Undo/Redo 天然支持**：ECS 组件的修改走事件总线，自动记录到命令历史。
- **多 Agent 安全**：后处理参数属于相机实体的组件，多个 Agent 同时修改时，可通过组件白名单隔离冲突。

> **参考：UE 的 PostProcessVolume 提供了极其丰富的参数混合能力，但其配置分散在 Volume、Camera、Material、CVar 等多个层面。在 ECS 架构下，我们将这种能力重新表达为"相机实体上的组件堆叠"——每个相机独立配置，天然支持 Inspector 实时编辑和 AI Schema 导出。**

## 工业级设计清单

| 设计点 | 推荐方案 | 简化路径 | 需补全的差距 |
|--------|---------|---------|-------------|
| **输入数据** | 统一 SceneTextures（HDR SceneColor + Depth + Velocity） | 固定 SceneColor + Depth | 按需生成 Velocity / CustomDepth |
| **Pass 编排** | RenderGraph / DAG（自动拓扑排序、Pass 剔除、内存别名） | 硬编码顺序 + 手动双缓冲 | DAG 编译器、自动屏障推导 |
| **执行模型** | Pixel Shader 全屏三角为主，Compute Shader 为可选优化 | 全部 Pixel Shader | Async Compute、LDS 优化 |
| **跨帧资源** | 池化纹理 + 提取-注册循环 | 双缓冲 Ping-Pong | 动态分辨率适配、池大小管理 |
| **性能优化** | Bloom/SSAO 半分辨率 + ToneMap Uber Pass | 全分辨率 + 独立 Pass | 动态质量缩放、自动 Pass 合并 |
| **AI 接口** | ECS 组件 `PostProcessStack` + Schema 导出 | 硬编码配置 + 手动暴露 | 多 Agent 白名单、参数混合权重 |

## 默认推荐路径总结

在 ECS 兼容前提下，后处理栈架构的默认推荐路径为：

1. **渲染管线输出统一 SceneTextures**，其中 SceneColor 保持 HDR 格式直到 Tone Mapping 之前。
2. **后处理链通过 RenderGraph 编排**，每个效果声明输入/输出资源依赖，由图编译器自动推导顺序、剔除未使用 Pass、管理中间纹理生命周期。
3. **全屏 Pass 默认使用 Pixel Shader + 全屏三角形**，在性能分析确认瓶颈后，对特定 Pass（如 Downsample、Histogram）追加 Compute Shader 版本。
4. **时域效果的历史缓冲通过池化纹理 + 提取-注册循环管理**，避免额外拷贝开销。
5. **后处理参数平铺为 ECS 组件**（`PostProcessStack`），支持 Inspector 实时编辑、AI Schema 导出、Undo/Redo 和多 Agent 安全操作。

这条路径吸收了 UE 的工业级设计（RDG 资源管理、显式 Pass 顺序、历史缓冲提取模式）和 Bevy 的 ECS 原生实践（Component 驱动、Schedule 编排），同时通过 RenderGraph 的自动推导降低了手动管理复杂度。
