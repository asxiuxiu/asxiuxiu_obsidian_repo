# 场景图与变换

> **前置依赖**：[[从零开始的引擎骨架]]、[[组件系统架构]]
> **本模块增量**：在最小 ECS 骨架上增加层级空间关系与世界坐标变换能力，让实体能在三维空间中建立父子关系并正确传播变换。
> **下一步**：[[反射与序列化]] — 场景中的实体需要被序列化保存后才能被 Agent 观察和世界持久化。

---

## Why：为什么游戏引擎需要场景图与变换？

### 没有这个系统时，世界有多糟

在 [[从零开始的引擎骨架]] 中，实体只是漂浮在虚空中的孤立 ID。你可以给 `Entity(5)` 一个 `Position`，但无法表达以下事实：

1. **椅子放在桌子下面**
   - 移动桌子时，椅子应该跟着动。没有层级关系，你只能手动遍历所有"可能相关"的实体并逐一手动更新坐标。

2. **摄像机跟随玩家**
   - 玩家每移动一帧，你就要写一段代码把摄像机的世界坐标重新计算一遍。如果还有肩扛武器、头盔装饰，这些挂接物都要单独处理。

3. **骨骼动画**
   - 一个角色有 50 块骨骼，每块骨骼的变换都依赖于父骨骼。没有统一的层级更新机制，动画系统必须自己维护一套独立的树结构。

### 三个真实场景

| 场景 | 没有场景图的困难 | ECS 场景图的答案 |
|------|-----------------|-----------------|
| **场景 1：编辑器中拖拽一个父物体** | 需要写脚本遍历所有子物体并手动偏移坐标 | `TransformSystem` 自动将父物体的变换级联到所有后代 |
| **场景 2：网络同步只需要角色的根位置** | 必须同步 50 块骨骼的世界矩阵 | 只同步根实体的 `LocalTransform`，子骨骼在客户端本地按层级重算 |
| **场景 3：AI 想查询"门把手相对于门的位置"** | 需要在对象图中追踪深层指针关系 | `Parent` 组件明确记录了这种关系，AI 可直接读取 |

### 这是"必须"还是"优化"

对于任何需要渲染三维对象、模拟物理或播放骨骼动画的引擎，**场景图与变换系统是必需的**。它不是性能优化，而是让"空间存在"成为可能的基础能力。

> **核心结论**：场景图不是渲染的附属品，而是引擎中**空间语义的表达层**。在 ECS 架构下，这层语义必须被拆成可平铺、可查询、可批量更新的组件。

## What：最简化版本的 ECS 场景图长什么样？

下面的代码是一个**真正可以编译运行**的极简 ECS 场景图系统。相比 [[组件系统架构]] 中的孤立实体，它增加了四个关键能力：

1. **`LocalTransform`**：存储局部坐标（相对于父物体）
2. **`GlobalTransform`**：存储世界坐标（经过层级传播后）
3. **`Parent`**：记录父实体 ID
4. **`TransformSystem`**：按层级深度（rank）排序后批量更新

```cpp
#include <cstdint>
#include <vector>
#include <array>
#include <algorithm>
#include <cmath>
#include <iostream>

// ============================================================
// 1. 最小数学原语（假设你还没有引入完整数学库）
// ============================================================
struct Vec3 { float x, y, z; };
struct Quat { float x, y, z, w; };

Vec3 operator+(Vec3 a, Vec3 b) { return {a.x+b.x, a.y+b.y, a.z+b.z}; }
Vec3 operator*(Vec3 a, float s) { return {a.x*s, a.y*s, a.z*s}; }

Quat mul(Quat a, Quat b) {
    return {
        a.w*b.x + a.x*b.w + a.y*b.z - a.z*b.y,
        a.w*b.y - a.x*b.z + a.y*b.w + a.z*b.x,
        a.w*b.z + a.x*b.y - a.y*b.x + a.z*b.w,
        a.w*b.w - a.x*b.x - a.y*b.y - a.z*b.z
    };
}

Vec3 rotate(Quat q, Vec3 v) {
    Vec3 u = {q.x, q.y, q.z};
    float s = q.w;
    Vec3 cross = {
        u.y*v.z - u.z*v.y,
        u.z*v.x - u.x*v.z,
        u.x*v.y - u.y*v.x
    };
    Vec3 cross2 = {
        u.y*cross.z - u.z*cross.y,
        u.z*cross.x - u.x*cross.z,
        u.x*cross.y - u.y*cross.x
    };
    return v + cross*(2*s) + cross2*2;
}

// ============================================================
// 2. ECS 组件（纯 POD）
// ============================================================
using Entity = uint32_t;
constexpr Entity MAX_ENTITIES = 1024;

struct LocalTransform {
    Vec3 position{0,0,0};
    Quat rotation{0,0,0,1};
    Vec3 scale{1,1,1};
};

struct GlobalTransform {
    Vec3 position{0,0,0};
    Quat rotation{0,0,0,1};
    Vec3 scale{1,1,1};
};

struct Parent {
    Entity parent{0xFFFFFFFF};
    bool valid() const { return parent != 0xFFFFFFFF; }
};

struct Children {
    std::vector<Entity> list;
};

struct TransformDirty {
    bool value{true}; // 是否需要重新计算世界坐标
};

// ============================================================
// 3. World：沿用引擎骨架的平铺数组
// ============================================================
struct World {
    std::array<LocalTransform, MAX_ENTITIES> locals{};
    std::array<GlobalTransform, MAX_ENTITIES> globals{};
    std::array<Parent, MAX_ENTITIES> parents{};
    std::array<Children, MAX_ENTITIES> children{};
    std::array<TransformDirty, MAX_ENTITIES> dirty{};
    std::array<bool, MAX_ENTITIES> alive{};

    Entity spawn() {
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!alive[e]) { alive[e] = true; return e; }
        }
        return 0;
    }

    void setParent(Entity child, Entity parent) {
        // 解除旧父关系
        if (parents[child].valid()) {
            auto& old = children[parents[child].parent].list;
            old.erase(std::remove(old.begin(), old.end(), child), old.end());
        }
        // 建立新父关系
        parents[child] = {parent};
        children[parent].list.push_back(child);
        dirty[child].value = true;
    }

    void setLocalPosition(Entity e, Vec3 p) {
        locals[e].position = p;
        dirty[e].value = true;
    }
};

// ============================================================
// 4. TransformSystem：按层级深度排序更新
// ============================================================
class TransformSystem {
    struct Entry { Entity e; uint32_t rank; };
    std::vector<Entry> entries;

    uint32_t computeRank(World& w, Entity e) {
        uint32_t rank = 0;
        while (w.parents[e].valid()) {
            e = w.parents[e].parent;
            ++rank;
        }
        return rank;
    }

    void updateEntity(World& w, Entity e) {
        if (!w.parents[e].valid()) {
            // 根节点：局部 = 世界
            w.globals[e].position = w.locals[e].position;
            w.globals[e].rotation = w.locals[e].rotation;
            w.globals[e].scale    = w.locals[e].scale;
        } else {
            Entity p = w.parents[e].parent;
            const auto& gp = w.globals[p];
            const auto& lc = w.locals[e];
            // world = parent * local
            w.globals[e].scale    = {gp.scale.x * lc.scale.x,
                                     gp.scale.y * lc.scale.y,
                                     gp.scale.z * lc.scale.z};
            w.globals[e].rotation = mul(gp.rotation, lc.rotation);
            w.globals[e].position = gp.position + rotate(gp.rotation, gp.scale * lc.position);
        }
        w.dirty[e].value = false;
    }

public:
    void tick(World& w) {
        entries.clear();
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!w.alive[e]) continue;
            entries.push_back({e, computeRank(w, e)});
        }
        // 按 rank 升序排序：父节点先于子节点更新
        std::sort(entries.begin(), entries.end(),
            [](const Entry& a, const Entry& b){ return a.rank < b.rank; });

        for (const auto& entry : entries) {
            updateEntity(w, entry.e);
        }
    }
};

// ============================================================
// 主循环
// ============================================================
int main() {
    World world;
    TransformSystem transformSys;

    Entity table = world.spawn();
    world.setLocalPosition(table, {10, 0, 0});

    Entity chair = world.spawn();
    world.setLocalPosition(chair, {2, 0, 0}); // 椅子在桌子局部坐标系中偏移 2
    world.setParent(chair, table);

    transformSys.tick(world);

    auto& chairWorld = world.globals[chair];
    std::cout << "Chair world position: (" << chairWorld.position.x << ", " << chairWorld.position.y << ", " << chairWorld.position.z << ")\n";
    // 输出应为 (12, 0, 0)

    // 移动桌子
    world.setLocalPosition(table, {20, 0, 0});
    transformSys.tick(world);

    std::cout << "After table moved: (" << world.globals[chair].position.x << ", "
              << world.globals[chair].position.y << ", " << world.globals[chair].position.z << ")\n";
    // 输出应为 (22, 0, 0)

    return 0;
}
```

### 这个最小实现已经解决了什么

| 能力 | 说明 |
|------|------|
| **层级父子关系** | 通过 `Parent` 和 `Children` 组件显式表达 |
| **世界坐标自动传播** | 父物体移动后，子物体的 `GlobalTransform` 在下一帧自动更新 |
| **拓扑有序更新** | 按 `rank` 排序确保父节点先于子节点计算，结果正确 |
| **纯 POD 组件** | 所有状态都是平铺的 struct，可直接序列化、diff、快照 |

### 这个最小实现还缺什么

- **没有矩阵缓存**：`GlobalTransform` 只存储 TRS，渲染系统如果需要 `Mat4`，每次都要实时合成
- **没有脏标记子树传播**：这里每个实体都重新计算了 `rank` 和 `GlobalTransform`，没有利用 `dirty` 做增量更新
- **没有并行更新**：同 `rank` 的节点其实可以并行计算
- **父子关系变更时没有 preserve world transform**：`setParent` 只保持了局部坐标，编辑器拖拽时可能需要保持世界坐标不变

这些正是接下来要演进的步骤。

## ECS 重构映射：从 OOP 场景图到纯 ECS

很多成熟引擎（包括我们分析的源码）使用的是**OOP 风格的场景图**：`SceneHierarchy` 维护 `HierarchyNode` 树，`TransformComponent` 携带 `Pose3D` 和父节点句柄，`TransformManager` 专门负责更新。如果目标是纯 ECS，这些结构应该如何重构？

| 源码中的 OOP 结构 | 在 ECS 中的对应 | 重构原因 |
|------------------|----------------|---------|
| **`SceneHierarchy` + `HierarchyNode` 树** | **`Parent` 组件 + `Children` 组件** | 场景图本身不需要独立的数据结构。父子关系作为普通组件存储在 ECS 中，`TransformSystem` 通过 Query 读取即可。 |
| **`TransformComponent`（含 `Pose3D` + `m_parent_handle` + 虚函数）** | **`LocalTransform` + `GlobalTransform` + `Parent`（三个纯 POD 组件）** | 虚函数和句柄解引用破坏数据连续性。拆成三个独立组件后，不需要 `TransformComponentHandle` 即可直接线性访问。 |
| **`TransformManager`（专门的管理器类）** | **`TransformSystem`（普通 System）** | Manager 既是调度对象又是数据所有者，职责过重。ECS 中 `TransformSystem` 只读取 `LocalTransform` 和 `Parent`，写入 `GlobalTransform`，不拥有任何组件。 |
| **按 `rank` 分批并行更新** | **System 内按 depth 排序后批量处理** | OOP 引擎用 `g_task_executor` 对同 rank 节点做任务拆分；ECS 中可以直接用 Job System 对排序后的实体数组分片并行。 |
| **显式 `getAllChildren()` 子树收集** | **`TransformDirty` 标记组件 + Query 筛选** | 不需要 Manager 主动遍历树。父节点变化时，给子树打上 `TransformDirty` 标记，下一帧 `TransformSystem` 只处理 dirty 实体。 |
| **`DynamicFixedIndexArray` 节点池** | **直接复用 Entity ID + `ComponentArray`** | 在 ECS 中，Entity 本身就是轻量 ID，不需要额外的 slot-map 来存储场景图节点。 |

> **关键洞察**：源码中的 `SceneHierarchy` 和 `TransformManager` 分离，本质上是想把"树结构维护"和"姿态更新"解耦。在 ECS 中，这种解耦更加彻底——树结构就是数据（组件），姿态更新就是逻辑（System），没有任何中间层的 Handle 或 Manager。

## How：真实引擎的场景图是如何一步一步复杂起来的？

### 阶段 1：能用 — 从朴素 ECS 到脏标记增量更新

#### 触发原因

当场景中有几千个实体时，朴素实现的问题出现了：

1. **每帧都重新计算所有实体的 rank 和世界坐标**：即使 99% 的物体根本没动，也要做一遍拓扑排序和矩阵运算。
2. **渲染系统频繁索要 4×4 矩阵**：`GlobalTransform` 只有 TRS，渲染时每次都要 `makeTransform(position, scale, rotation)`，重复计算。

#### 代码层面的变化

**A. 引入真正的脏标记子树传播**

```cpp
struct TransformDirty {
    bool value{false};
};

class TransformSystem {
public:
    void markDirtyRecursive(World& w, Entity root) {
        w.dirty[root].value = true;
        for (Entity child : w.children[root].list) {
            markDirtyRecursive(w, child);
        }
    }

    void tick(World& w) {
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!w.alive[e] || !w.dirty[e].value) continue;
            // 只更新 dirty 的实体
            updateEntity(w, e);
        }
    }
};
```

当 `setLocalPosition(e, ...)` 被调用时，系统调用 `markDirtyRecursive(e)`，一次性将子树全部标记为 dirty。下一帧 `TransformSystem` 只处理这些被标记的节点。

**B. 缓存 `GlobalTransform` 的 4×4 矩阵**

```cpp
struct GlobalTransform {
    Vec3 position;
    Quat rotation;
    Vec3 scale;
    float matrix[16];
    bool matrixDirty{true};

    const float* getMatrix() {
        if (matrixDirty) {
            // compute TRS -> Mat4
            computeMatrix(position, rotation, scale, matrix);
            matrixDirty = false;
        }
        return matrix;
    }
};
```

> 源码中的 `TransformComponent` **不缓存矩阵**，矩阵是按需实时计算的。对于小型 demo 这没问题，但对于大量物体的渲染管线，频繁合成矩阵会增加 CPU 开销。在自研引擎中，建议缓存矩阵并配合 `matrixDirty` 延迟计算。

### 阶段 2：好用 — 父子关系变更、并行更新与编辑器友好

#### 触发原因

- 编辑器中拖拽一个对象到另一个对象下当子物体，需要保持它在世界中的位置不变（preserve world transform）。
- 场景中有大量静态物体和少量动态物体，希望静态物体的变换完全跳过更新。
- 多核 CPU 普及，希望同层节点的矩阵计算能并行化。

#### 代码层面的变化

**A. 父子关系变更时 preserve world transform**

```cpp
void setParentPreserveWorld(World& w, Entity child, Entity newParent) {
    Vec3 worldPos = w.globals[child].position;
    Quat worldRot = w.globals[child].rotation;
    Vec3 worldScl = w.globals[child].scale;

    if (newParent != 0xFFFFFFFF) {
        const auto& p = w.globals[newParent];
        // local = inv(parent) * world
        w.locals[child].scale = {worldScl.x / p.scale.x, worldScl.y / p.scale.y, worldScl.z / p.scale.z};
        w.locals[child].rotation = inverse(p.rotation) * worldRot; // 简化表示
        w.locals[child].position = rotate(inverse(p.rotation), (worldPos - p.position) / p.scale);
    } else {
        w.locals[child].position = worldPos;
        w.locals[child].rotation = worldRot;
        w.locals[child].scale = worldScl;
    }

    // 解除旧关系、建立新关系...
    w.dirty[child].value = true;
}
```

这是编辑器操作的基础能力：用户拖拽物体时，眼睛看到的世界坐标不变，引擎自动反推局部坐标。

**B. 静态 vs 动态分离：跳过完全不变的子树**

```cpp
struct TransformStatic {
    // 标记该实体及其子树是否长期不变
    bool value{false};
};
```

如果一棵子树的所有节点都带有 `TransformStatic`，且上一帧没有被标记 dirty，那么 `TransformSystem` 可以直接跳过整棵子树的遍历。

**C. 按 rank 并行更新**

```cpp
void tickParallel(World& w, JobSystem& jobs) {
    // 1. 收集所有 dirty 实体并按 rank 分组
    std::vector<std::vector<Entity>> ranks;
    for (Entity e = 0; e < MAX_ENTITIES; ++e) {
        if (!w.alive[e] || !w.dirty[e].value) continue;
        uint32_t r = computeRank(w, e);
        if (ranks.size() <= r) ranks.resize(r + 1);
        ranks[r].push_back(e);
    }

    // 2. 逐 rank 更新，同一 rank 内并行
    for (auto& rankEntities : ranks) {
        jobs.parallel_for(rankEntities.size(), [&](size_t i) {
            updateEntity(w, rankEntities[i]);
        });
    }
}
```

同一 `rank` 的节点之间没有父子依赖，可以安全地并行更新。这正是源码中 `TransformManager` 使用 `g_task_executor` 的核心思想。

### 阶段 3：工业级 — 骨骼优化、空间查询分离与延迟更新

#### 触发原因

- 角色骨骼链有 50~100 个层级深度，递归计算 rank 和 dirty 传播的开销变高。
- 需要做视锥剔除、射线检测，但场景图本身只维护父子关系，不是空间加速结构。
- 网络同步和回放要求：变换更新必须在确定的 Tick 边界统一发生。

#### 代码层面的变化

**A. 骨骼链优化：扁平化索引**

对于深度很大的骨骼链，递归 `computeRank` 和 `markDirtyRecursive` 的开销不可忽略。工业级实现通常会：

- **预计算拓扑序数组**：在骨骼创建时，按从根到叶的顺序生成一个扁平的 `boneOrder` 数组。更新时直接线性遍历这个数组，无需递归或排序。
- **使用 `TransformHierarchy` 专用组件**：将深度树和一般场景图分开处理。骨骼使用扁平数组，普通场景对象使用通用 ECS 组件。

**B. 明确区分：场景图 ≠ 空间加速结构**

这是源码分析中一个重要的结论：引擎的 `SceneHierarchy` 只维护父子关系，不承担 BVH、Octree、网格化空间查询的职责。

| 结构 | 用途 | 在 ECS 中的位置 |
|------|------|----------------|
| **场景图（Transform Hierarchy）** | 父子关系、世界坐标传播 | `Parent` / `Children` 组件 + `TransformSystem` |
| **空间加速结构（BVH / Octree / Spatial Hash）** | 视锥剔除、射线检测、近邻查询 | 独立的 `SpatialIndexSystem`，读取 `GlobalTransform` + `BoundingBox` |

> **不要把场景图当成空间查询结构**。如果渲染系统需要做视锥剔除，应该另外维护一个 BVH 或 Octree，而不是遍历场景图。

**C. 命令缓冲驱动的延迟更新**

```cpp
struct TransformCommand {
    enum Type { SetParent, SetLocal, SetWorld } type;
    Entity target;
    // 联合存储参数...
};

class TransformSystem {
    std::vector<TransformCommand> commandBuffer;
public:
    void stageSetParent(Entity child, Entity parent, bool preserveWorld) {
        commandBuffer.push_back({TransformCommand::SetParent, child, parent, preserveWorld});
    }

    void applyCommands(World& w) {
        for (auto& cmd : commandBuffer) {
            if (cmd.type == TransformCommand::SetParent) {
                // ...
            }
        }
        commandBuffer.clear();
    }

    void tick(World& w) {
        applyCommands(w);      // Tick 边界统一应用变更
        updateDirtyNodes(w);   // 然后传播变换
    }
};
```

所有编辑器操作、AI 工具调用、网络同步包都先把变更写入 `commandBuffer`，在 `tick()` 开始时才统一应用。这保证了：
- 迭代安全：System 不会在运行中读到半成品的父子关系。
- 确定性：所有变更都在固定的时序点发生，回放可复现。
- 事务性：如果命令校验失败，可以整批丢弃。

## 设计权衡表

| 决策点 | 原型阶段（< 1000 实体） | 工业级阶段（> 10 万实体 + 复杂骨骼） |
|--------|----------------------|-----------------------------------|
| 存储模型 | `std::array<Transform, N>` 裸数组 | `ComponentArray<T>` Sparse Set + 按需分配 `GlobalTransform` |
| 脏标记策略 | 每帧全量更新（简单、无 bug） | 精确子树 dirty 传播 + 静态标记跳过 |
| 并行更新 | 单线程串行 | 按 rank 分批，Job System 并行 |
| 骨骼处理 | 与场景图共用 `Parent` 组件 | 骨骼使用预计算拓扑序扁平数组，单独 `SkeletonSystem` |
| 矩阵缓存 | 实时按需合成 TRS → Mat4 | `GlobalTransform` 内嵌 `Mat4` + `matrixDirty` 标记 |
| 空间查询 | 线性遍历所有实体做 AABB 测试 | 独立 `BVHSystem` 或 `OctreeSystem`，与场景图解耦 |

## 如果我要 vibe coding，该偷哪几招？

### 骨架层必偷

1. **`Parent` 和 `Children` 组件从 Day 1 就要有**
   - 不要等渲染管线出来了才补。任何需要在空间中存在的实体都需要它们。

2. **按 rank 排序更新是层级变换的黄金法则**
   - 父节点先于子节点更新，这是数学上的必然要求。不要试图用事件回调或消息队列来替代拓扑排序。

3. **父子关系变更时支持 preserve world transform**
   - 编辑器拖拽、运行时动态挂接、网络同步重插父节点，都需要这个能力。

### ECS 重构层必偷

4. **如果源码是 OOP 场景图，把它拆成 `Parent` + `LocalTransform` + `GlobalTransform` 三个组件**
   - 这是从 Component-Based 到纯 ECS 的最核心重构。

5. **场景图不负责空间查询**
   - 视锥剔除和射线检测交给独立的 BVH/Octree。场景图只回答"谁是谁的父节点"。

### AI 层必偷

6. **给 AI 暴露 `get_world_transform(entity)` 和 `set_parent(child, parent)` 工具**
   - 这是 Agent 理解空间关系的基础 API。

7. **在 `TransformCommandBuffer` 中记录所有 AI 发起的父子关系变更**
   - 如果 AI 的改动导致循环父子关系（A 的父是 B，B 的父是 A），可以在应用命令前统一检测并拒绝。
