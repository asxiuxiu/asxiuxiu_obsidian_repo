# 从零开始的引擎骨架

这篇笔记分两部分：
1. **第一部分**：一个**没有 AI 参与**的引擎骨架，应该长什么样？
2. **第二部分**：当这个引擎需要被 **AI Agent 长时间自主操作**时，哪些部分需要改变、增强或新增？

---

## 第一部分：没有 AI 的引擎骨架

### Why：为什么先做骨架，而不是直接写渲染？

很多人开始做引擎时，第一步是创建一个窗口、初始化 Vulkan、在屏幕上画一个三角形。这很爽，但它只是一个渲染 demo。

如果你的目标是构建一个**可长期演进、可调试、可多人协作**的引擎，那么第一步不应该是"让画面出来"，而应该是**搭一个清晰的状态骨架**。

#### 没有骨架时，世界有多糟

1. **状态是一片黑盒**
   - 对象图深埋指针、副作用散落在虚函数表里。你想调整"玩家移动速度"，却发现这个字段继承自五层之上的基类，运行时根本不被读取。

2. **调试成本高**
   - 运行时崩溃，你拿到调用栈却不知道内存中真正发生了什么。没有统一的观测接口，只能逐个断点追踪。

3. **系统耦合无法扩展**
   - 新加一个物理系统，却不知道怎么保证它在 Gameplay 逻辑之前执行。系统之间互相依赖、隐式修改全局状态。

#### 一个清晰的骨架解决什么

| 问题 | 骨架的答案 |
|------|-----------|
| 某个对象有哪些属性？ | 自描述的类型注册表（TypeRegistry），运行时可直接查询 |
| 状态是否显式、可追踪？ | 平铺的 ECS 组件数据，所有状态都是显式的、可序列化的 |
| 新系统怎么接入？ | 分阶段 Tick + 依赖排序：系统以纯函数方式注册到调度器 |
| 改完对不对？ | 确定性 Tick + 快照回放：保存状态 → 改动 → 对比 diff |

> **核心结论**：这个骨架是为了让引擎状态透明、行为可预测、系统可扩展。

### What：最小能跑的引擎骨架长什么样？

下面的代码是一个**真正可以编译运行**的极简引擎骨架。它不包含任何图形 API，但已经具备了引擎最核心的三个能力：
1. **平铺 ECS 数据层**：所有状态透明
2. **自描述 TypeRegistry**：运行时即可理解数据结构
3. **确定性 Tick 调度**：同样的输入一定得到同样的状态

```cpp
#include <cstdint>
#include <vector>
#include <string>
#include <unordered_map>
#include <functional>
#include <iostream>
#include <cassert>

// ============================================================
// 1. 平铺 ECS 数据层
// ============================================================
using Entity = uint32_t;
constexpr Entity MAX_ENTITIES = 1024;

struct Position { float x{0}, y{0}, z{0}; };
struct Velocity { float x{0}, y{0}, z{0}; };

class World {
public:
    std::vector<Position> positions;
    std::vector<Velocity> velocities;
    std::vector<bool>     alive;

    World() {
        positions.resize(MAX_ENTITIES);
        velocities.resize(MAX_ENTITIES);
        alive.resize(MAX_ENTITIES, false);
    }

    Entity spawn() {
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!alive[e]) { alive[e] = true; return e; }
        }
        assert(false); return 0;
    }
    void destroy(Entity e) { alive[e] = false; }
    bool valid(Entity e) const { return e < MAX_ENTITIES && alive[e]; }
};

// ============================================================
// 2. 自描述类型注册表
// ============================================================
enum class FieldType { Float };
struct FieldDesc { std::string name; FieldType type; size_t offset; };
struct ComponentDesc {
    std::string name;
    std::vector<FieldDesc> fields;
    size_t size;
};

class TypeRegistry {
public:
    std::unordered_map<std::string, ComponentDesc> components;

    void register_all() {
        components["Position"] = {
            "Position",
            { {"x", FieldType::Float, offsetof(Position, x)},
              {"y", FieldType::Float, offsetof(Position, y)},
              {"z", FieldType::Float, offsetof(Position, z)} },
            sizeof(Position)
        };
        components["Velocity"] = {
            "Velocity",
            { {"x", FieldType::Float, offsetof(Velocity, x)},
              {"y", FieldType::Float, offsetof(Velocity, y)},
              {"z", FieldType::Float, offsetof(Velocity, z)} },
            sizeof(Velocity)
        };
    }
};

// ============================================================
// 3. 确定性 System 调度
// ============================================================
class System {
public:
    virtual ~System() = default;
    virtual void tick(World&, float dt) {}
};

class MovementSystem : public System {
public:
    void tick(World& world, float dt) override {
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!world.alive[e]) continue;
            world.positions[e].x += world.velocities[e].x * dt;
            world.positions[e].y += world.velocities[e].y * dt;
            world.positions[e].z += world.velocities[e].z * dt;
        }
    }
};

class Scheduler {
    std::vector<std::unique_ptr<System>> systems;
public:
    void add(std::unique_ptr<System> sys) {
        systems.push_back(std::move(sys));
    }
    void tick(World& world, float dt) {
        for (auto& sys : systems) {
            sys->tick(world, dt);  // 确定性：固定顺序，无隐藏状态
        }
    }
};

// ============================================================
// 主循环
// ============================================================
int main() {
    World world;
    TypeRegistry registry; registry.register_all();
    Scheduler scheduler;
    scheduler.add(std::make_unique<MovementSystem>());

    Entity player = world.spawn();
    world.velocities[player] = {1.0f, 0.0f, 0.0f};

    // 推进 3 帧
    for (int i = 0; i < 3; ++i) {
        scheduler.tick(world, 0.016f);
        auto* t = &world.positions[player];
        std::cout << "Frame " << i << ": pos = (" << t->x << ", " << t->y << ")\n";
    }
    return 0;
}
```

### 这个最小实现已经解决了什么

| 能力 | 说明 |
|------|------|
| **状态完全平铺** | `World` 里的所有数据都是连续数组，可直接序列化、diff、分析 |
| **零指针追踪** | 没有深层对象树、没有虚函数副作用，不需要"猜"引用关系 |
| **运行时自描述** | `TypeRegistry` 让外部工具知道每个组件有哪些字段、偏移量和类型 |
| **确定性执行** | `scheduler.tick(world, dt)` 是纯函数式调用，输入固定则输出固定 |

### 这个最小实现还缺什么

- **没有自动化反射**：`TypeRegistry` 还是手写的，真实引擎需要宏或代码生成
- **没有系统依赖排序**：`PhysicsSystem` 必须在 `MovementSystem` 之前执行时，当前代码无法表达
- **没有增量变更日志**：每次都要全量扫描世界才能知道发生了什么变化
- **没有快照与回滚**：还不能"尝试一个改动，不满意就撤销"

这些不是缺陷，而是**下一阶段的明确目标**。

### How：纯引擎骨架是如何复杂起来的？

#### 阶段 1：能用 —— 状态可见与系统扩展

##### 触发原因

当引擎从 1 个系统变成 10 个系统时：
- "`Player` 类继承自 `Actor`，`Actor` 里有一个 `transform_` 指针……" 追踪真正的状态位置变得困难。
- "找出所有速度大于 100 的物体"，需要遍历复杂的对象关系。

##### 代码层面的变化

**A. 引入 ECS 强制所有状态平铺**

这是本知识库的默认假设。ECS 的"数据与逻辑分离"天然适合批量分析和系统扩展。

**B. 手写 `TypeRegistry` → 自动化反射生成**

```cpp
// 阶段 1：手写
// 阶段 2：通过宏自动生成
#define REFLECT_COMPONENT(Name, ...) \
    static ComponentDesc get_desc() { return { #Name, { __VA_ARGS__ }, sizeof(Name) }; }

struct Transform {
    float x, y, z;
    REFLECT_COMPONENT(Transform,
        {"x", FieldType::Float, offsetof(Transform, x)},
        {"y", FieldType::Float, offsetof(Transform, y)},
        {"z", FieldType::Float, offsetof(Transform, z)}
    )
};
```

**C. `ToolRegistry` 与工具自注册**

引擎需要对外提供操作接口，供编辑器、调试器、自动化测试脚本调用。每个 `System` 在初始化时，可以向 `ToolRegistry` 注册自己的观测/调参工具：

```cpp
struct ToolResult {
    bool ok;
    std::string json;
};

struct ToolDesc {
    std::string name;
    std::string description;
    std::string inputSchema;
    bool isReadOnly;
    bool isConcurrencySafe;
    bool isDestructive;
    std::function<ToolResult(const std::string& json_args)> call;
};

class ToolRegistry {
public:
    std::unordered_map<std::string, ToolDesc> tools;
    void register_tool(ToolDesc desc) { tools[desc.name] = std::move(desc); }
};

class MovementSystem : public System {
public:
    void register_tools(ToolRegistry& registry) {
        registry.register_tool({
            .name = "set_global_gravity",
            .description = "Set the global gravity vector",
            .inputSchema = R"({"type":"object","properties":{"x":{"type":"number"},...}})",
            .isReadOnly = false,
            .isConcurrencySafe = false,
            .isDestructive = false,
            .call = [&](const std::string& args) -> ToolResult {
                // 解析 JSON 并修改系统内部状态
                return {true, R"({"status":"ok"})"};
            }
        });
    }
};
```

> 注：`ToolRegistry` 在纯引擎中作为编辑器、调试器、自动化测试的统一接口。后面引入 AI 时，这些工具将成为 AI 的操作边界。

#### 阶段 2：好用 —— 可调试与可协作

##### 触发原因

- 引擎规模扩大，需要知道"这一帧到底发生了什么变化"
- 多个模块（编辑器、Gameplay、物理）需要同时读写状态，不能互相覆盖
- 需要尝试性改动并回滚

##### 代码层面的变化

**A. 增量状态变更日志（Change Log）**

```cpp
struct ChangeLog {
    struct Entry {
        Entity e;
        std::string component;
        std::string field;
        std::string oldValue;
        std::string newValue;
    };
    std::vector<Entry> entries;
};

class World {
    ChangeLog changelog;
public:
    void notify_change(Entity e, const std::string& comp,
                       const std::string& field,
                       const std::string& oldVal,
                       const std::string& newVal) {
        changelog.entries.push_back({e, comp, field, oldVal, newVal});
    }
    const ChangeLog& get_changes() const { return changelog; }
    void clear_changes() { changelog.entries.clear(); }
};
```

**A+. ChangeLog 滚动窗口与过期清理**

`ChangeLog` 不能无限增长：

```cpp
class World {
    static constexpr size_t MAX_CHANGELOG_FRAMES = 16;
    ChangeLog changelog;
    std::unordered_map<std::string, size_t> compactedStats; // 组件 -> 累计变更次数

public:
    void notify_change(Entity e, const std::string& comp,
                       const std::string& field,
                       const std::string& oldVal,
                       const std::string& newVal) {
        changelog.entries.push_back({e, comp, field, oldVal, newVal});
    }

    void compact_changelog() {
        if (changelog.entries.size() <= MAX_CHANGELOG_FRAMES) return;
        auto split = changelog.entries.begin() + (changelog.entries.size() - MAX_CHANGELOG_FRAMES);
        for (auto it = changelog.entries.begin(); it != split; ++it) {
            compactedStats[it->component]++;
        }
        changelog.entries.erase(changelog.entries.begin(), split);
    }

    const ChangeLog& get_changes() const { return changelog; }
    const auto& get_compacted_stats() const { return compactedStats; }
    void clear_changes() {
        changelog.entries.clear();
        compactedStats.clear();
    }
};
```

- 最近 16 帧保留完整记录（约 1 秒 @ 60fps）
- 超过 16 帧的旧记录自动折叠为每个组件的"累计变更次数"

**B. 命令缓冲（Command Buffer）**

```cpp
struct CommandBuffer {
    struct Command {
        Entity e;
        std::string component;
        std::string field;
        std::string newValue;
    };
    std::vector<Command> commands;
};

class EngineBridge {
    CommandBuffer pendingCommands;
public:
    void stage_change(Entity e, const std::string& comp,
                      const std::string& field,
                      const std::string& newVal) {
        pendingCommands.commands.push_back({e, comp, field, newVal});
    }

    void clear_pending() { pendingCommands.commands.clear(); }

```

> 在纯引擎中，`CommandBuffer` 可用于编辑器操作、网络同步指令、录制回放等。它是一种"延迟应用修改"的通用机制。

**C. 系统依赖图 + 分阶段 Tick**

```cpp
enum class Phase { PrePhysics, Physics, PostPhysics, Gameplay, Render };

class Scheduler {
    std::unordered_map<Phase, std::vector<std::unique_ptr<System>>> phases;
public:
    void add(Phase p, std::unique_ptr<System> sys) { phases[p].push_back(std::move(sys)); }
    void tick(World& world, float dt) {
        for (auto p : {Phase::PrePhysics, Phase::Physics, Phase::PostPhysics,
                       Phase::Gameplay, Phase::Render}) {
            for (auto& sys : phases[p]) {
                sys->tick(world, dt);
            }
        }
    }
};
```

**D. 世界快照与回放（Snapshot / Replay）**

```cpp
class WorldSnapshot {
    std::vector<char> rawData;  // 压缩后的组件数组 dump
public:
    static WorldSnapshot capture(const World& w);
    static void restore(World& w, const WorldSnapshot& snap);
};

class ContextCompressor {
public:
    void microcompact(World& w) { w.compact_changelog(); }
    void checkpoint(World& w, const std::string& name);
    void revert_to(World& w, const std::string& name);
private:
    std::unordered_map<std::string, WorldSnapshot> snapshots;
};
```

工作流：
1. `snapshot = capture(world)` —— 保存
2. 尝试改动 → 运行几帧观察效果
3. 不满意？`restore(world, snapshot)` —— 回滚

这在游戏调试、自动化测试、网络同步中都是核心能力。

#### 阶段 3：工业级 —— 规模与性能

##### 触发原因

- 实体数量从 100 增长到 10 万
- 需要网络同步
- 需要防止"幻觉改动"破坏核心玩法逻辑

##### 代码层面的变化

**A. 从 `std::vector` 到 Archetype/Chunk 存储**

```cpp
struct Archetype {
    uint32_t componentMask;       // 位掩码表示包含哪些组件
    std::vector<Entity> entities; // 实体 ID 列表
    std::vector<std::vector<char>> columns; // 每列是一个组件类型的 SoA 数组
};
```

Archetype 不仅提升 cache locality，也极大加速批量查询。

**B. 确定性回放网络（Deterministic Replay Net）**

```cpp
struct FrameInput {
    uint32_t frameId;
    float fixedDt;
    std::vector<PlayerInput> playerInputs;
    uint64_t randomSeed;
};

class ReplayRecorder {
    std::vector<FrameInput> history;
public:
    void record(uint32_t frame, const FrameInput& input) { history.push_back(input); }
```

这是网络同步、自动化测试的基石：给定相同的输入序列，世界状态必须比特级一致。

让引擎本身成为改动的"自动审核员"。

## 第二部分：引入 AI 后，骨架需要哪些改变？

### 为什么纯骨架还不够？

当目标是 **vibe coding**——让 AI 能够长时间无人干预地帮你构建、调试、迭代这个引擎——纯引擎骨架还需要额外的四个能力：

| AI 的诉求 | 纯引擎的问题 | 需要的补充 |
|-----------|-------------|-----------|
| AI 怎么知道某个对象有哪些属性？ | TypeRegistry 存在，但没有面向 AI 的查询接口 | 结构化工具：query_entities / get_component |
| AI 怎么修改状态而不引发副作用？ | ECS 数据平铺，但 AI 直接改内存不可控 | AgentBridge 工具边界 + CommandBuffer |
| AI 怎么验证"改完对不对"？ | 有快照和 ChangeLog，但上下文太大 | 增量 ChangeLog + 滚动窗口 + ENGINE_AI.md |
| 多个 AI 怎么同时工作？ | 没有隔离机制 | Orchestrator + 组件白名单 + ApprovalRuntime |

> **核心结论**：AI 不是在写一个新的引擎，而是在操作一个已有的引擎。所以骨架的目标是让 AI **能看懂、能安全操作、能验证结果、能与其他 AI 协作**。

---

### AgentBridge：AI 与引擎之间的边界

AI 不应该直接生成代码来操作引擎内存，也不应该绑定某个特定的 AI 客户端。引擎需要一个**统一的桥接层**，即 `AgentBridge`。

#### 结构化显示与事件流反馈

借鉴 Kimi CLI 的 DisplayBlock 设计，把引擎内部状态转化为 AI 和用户都能理解的结构化反馈：

```cpp
struct EngineDisplayBlock {
    std::string type;        // "changelog", "profile", "screenshot", "diff", "task_status"
    std::string title;       // 人类可读的标题
    std::string payload;     // JSON 格式的详细内容
};

enum class AgentEventType {
    TurnBegin,      // 新对话轮次开始
    StepBegin,      // 新步骤开始（一次 LLM 调用）
    ToolCall,       // AI 决定调用某个引擎工具
    ToolResult,     // 工具执行完毕
    DisplayBlock,   // 结构化状态反馈（供 UI 渲染）
    StepEnd,        // 步骤结束
    TurnEnd         // 对话轮次结束
};

struct AgentEvent {
    AgentEventType type;
    std::string jsonPayload;
};

struct ToolResult {
    bool ok;
    std::string json;                       // 工具原始返回值（给 AI 消费）
    std::vector<EngineDisplayBlock> blocks; // 结构化显示块（给 UI 消费）
};

// 工具的完整自描述元数据（借鉴 Claude Code 的 Tool 接口）
struct ToolDesc {
    std::string name;
    std::string description;
    std::string inputSchema;   // JSON Schema 字符串，约束 AI 生成合法调用
    bool isReadOnly;
    bool isConcurrencySafe;
    bool isDestructive;
    std::function<ToolResult(const std::string& json_args)> call;
};
```

#### 完整的 AgentBridge 示例

把之前第一部分提到的 `ToolRegistry` 和 `AgentBridge` 结合起来：

```cpp
class AgentBridge {
    World& world;
    TypeRegistry& registry;
    Scheduler& scheduler;
    ToolRegistry& toolRegistry;
    std::vector<AgentEvent> eventBuffer;
public:
    AgentBridge(World& w, TypeRegistry& r, Scheduler& s, ToolRegistry& t)
        : world(w), registry(r), scheduler(s), toolRegistry(t) {}

    // AgentBridge 的通信模型是流式事件协议，而非纯批处理。
    // 借鉴 Claude Code 的 StreamingToolExecutor 与 Kimi CLI 的 Wire 协议：
    // 读操作可以在 Tick 执行过程中通过事件流实时响应，
    // 写操作累积到 Tick 边界统一提交，
    // 同时每步产生 DisplayBlock 供 Editor UI 渲染。

    void emitEvent(AgentEventType type, const std::string& payload = "{}") {
        eventBuffer.push_back({type, payload});
    }

    std::vector<AgentEvent> consumeEvents() {
        auto out = std::move(eventBuffer);
        eventBuffer.clear();
        return out;
    }

    // 统一的工具执行入口：所有工具调用都必须经过这里
    // 未来可在此嵌入权限检查、Undo 快照、参数校验等安全机制
    ToolResult execute_tool(const std::string& name, const std::string& argsJson) {
        auto it = toolRegistry.tools.find(name);
        if (it == toolRegistry.tools.end())
            return {false, R"({"error":"unknown tool"})", {}};
        return it->second.call(argsJson);
    }

    // Tool: 查询带有某组件的所有实体
    ToolResult query_entities(const std::string& comp) {
        emitEvent(AgentEventType::ToolCall, R"({"tool":"query_entities","args":{"comp":" )" + comp + "\"}}");
        if (!registry.components.count(comp))
            return {false, R"({"error":"unknown component"})", {}};
        std::string json = "[";
        for (Entity e = 0; e < MAX_ENTITIES; ++e) {
            if (!world.alive[e]) continue;
            if (json.size() > 1) json += ",";
            json += std::to_string(e);
        }
        json += "]";
        emitEvent(AgentEventType::ToolResult, R"({"tool":"query_entities","ok":true})");
        return {true, json, {{
            "changelog",
            "Query Result",
            R"({"tool":"query_entities","matched_count":)" + std::to_string(json.size()) + "}"
        }}};
    }

    // Tool: 修改实体的 Velocity
    ToolResult set_velocity(Entity e, float vx, float vy, float vz) {
        emitEvent(AgentEventType::ToolCall, R"({"tool":"set_velocity","args":{"e":)" + std::to_string(e) + "}}");
        if (!world.valid(e))
            return {false, R"({"error":"invalid entity"})", {}};
        world.velocities[e] = {vx, vy, vz};
        emitEvent(AgentEventType::ToolResult, R"({"tool":"set_velocity","ok":true})");
        return {true, R"({"status":"ok"})", {{
            "diff",
            "Velocity Changed",
            R"({"entity":)" + std::to_string(e) + R"(","component":"Velocity","new":{"x":)" +
            std::to_string(vx) + R"(","y":)" + std::to_string(vy) + R"(","z":)" + std::to_string(vz) + "}}"
        }}};
    }

    // Tool: 推进一帧
    ToolResult step(float dt) {
        emitEvent(AgentEventType::ToolCall, R"({"tool":"step","args":{"dt":)" + std::to_string(dt) + "}}");
        scheduler.tick(world, dt);
        // 生成这一帧的 ChangeLog 显示块
        auto changes = world.get_changes();
        std::string changePayload = "{\"entries\":[";
        for (size_t i = 0; i < changes.entries.size(); ++i) {
            if (i > 0) changePayload += ",";
            changePayload += "{\"e\":" + std::to_string(changes.entries[i].e) +
                ",\"component\":\"" + changes.entries[i].component + "\"}";
        }
        changePayload += "]}";
        emitEvent(AgentEventType::ToolResult, R"({"tool":"step","ok":true})");
        return {true, R"({"status":"tick_complete"})", {{
            "changelog",
            "Frame Changes",
            changePayload
        }}};
    }
};
```

### 引入 AI 后的三阶段演进

引擎骨架本身已经经历了"能用 → 好用 → 工业级"的演进。引入 AI 后，**每一阶段都需要补充 AI 专属能力**。

#### 阶段 1 补充：让 AI 能看见状态

纯引擎的 `TypeRegistry` 解决了"运行时自描述"问题。对 AI 来说，需要把它暴露为**结构化工具**：

| 工具名 | 作用 |
|--------|------|
| `query_entities(mask)` | 按组件掩码查询实体 |
| `get_component(e, name)` | 读取某个实体的某个组件字段（JSON 返回）|
| `set_component(e, name, json)` | 修改某个实体的某个组件字段 |
| `step(dt)` | 确定性推进一帧 |

AI 无需头文件，通过 `ToolRegistry` 的自描述元数据即可知道每个工具的参数约束、安全标记和返回格式。

**标准工具集 + 工具自注册**：`ToolRegistry` 自动从 `TypeRegistry` 生成默认的 `get_component` / `set_component` 工具，而 specialized 系统可以覆盖默认工具，提供更语义化的接口（如 `set_global_gravity`）。

#### 阶段 2 补充：让 AI 能高效协作

##### A. ChangeLog 对 AI 的特殊价值

纯引擎的 `ChangeLog` 已经用于调试。对 AI 来说，它是**避免上下文爆炸**的关键：
- AI 不需要读整个世界，只需要读这一帧的 `changelog`。

配合**滚动窗口清理**，超过 16 帧的旧记录折叠为统计，防止上下文无限膨胀。

##### D. 权限控制与安全边界

阅读 Claude Code `permissions.ts` / `bashSecurity.ts` 和 Kimi CLI `approval_runtime` 后，对原有设计做了以下落地化补充：引入三级决策树、拒绝追踪、工具级参数检查、Sandbox 配置。

```cpp
struct ApprovalSource {
    std::string kind;      // "foreground_turn" / "background_agent"
    std::string id;        // uuid
    std::string agentId;   // 所属 Agent
};

enum class PermissionBehavior { Deny, Ask, Allow };

struct PermissionResult {
    PermissionBehavior behavior;
    std::string reason;    // 哪条规则或哪个检查触发了结果
    std::string ruleSource; // flagSettings / projectSettings / cliArg / session
};

// 拒绝追踪：防止 Auto 模式陷入死循环
struct DenialTracker {
    int consecutiveDenials = 0;
    int totalDenials = 0;
    static constexpr int MAX_CONSECUTIVE = 3;
    static constexpr int MAX_TOTAL = 20;

    bool shouldFallbackToPrompting() const {
        return consecutiveDenials >= MAX_CONSECUTIVE || totalDenials >= MAX_TOTAL;
    }
    void recordDenial() { consecutiveDenials++; totalDenials++; }
    void recordAllow() { consecutiveDenials = 0; }
};

// 四级规则来源（优先级从高到低）
using RuleSource = std::string; // flag > policy > cli > command > session > user > project > local

struct PermissionRule {
    std::string toolName;   // "*" 表示通配所有工具
    std::string pattern;    // 精确 / 前缀 / 通配
    PermissionBehavior behavior;
    RuleSource source;
};

class PermissionEngine {
    std::vector<PermissionRule> rules;
public:
    void loadFromProjectSettings(const std::string& path);  // ENGINE_AI.json
    void loadFromCLIArgs(const std::vector<std::string>& args);
    void loadFromSession(const std::unordered_map<std::string, std::string>& session);

    PermissionResult evaluate(const std::string& toolName, const std::string& input) const {
        // 1. Deny 规则优先
        for (const auto& r : rules) {
            if (r.behavior == PermissionBehavior::Deny && matches(r, toolName, input))
                return {PermissionBehavior::Deny, "matched deny rule", r.source};
        }
        // 2. Ask 规则
        for (const auto& r : rules) {
            if (r.behavior == PermissionBehavior::Ask && matches(r, toolName, input))
                return {PermissionBehavior::Ask, "matched ask rule", r.source};
        }
        // 3. Allow 规则
        for (const auto& r : rules) {
            if (r.behavior == PermissionBehavior::Allow && matches(r, toolName, input))
                return {PermissionBehavior::Allow, "matched allow rule", r.source};
        }
        // 4. 默认回退 Ask
        return {PermissionBehavior::Ask, "no rule matched", "default"};
    }

    bool matches(const PermissionRule& r, const std::string& toolName, const std::string& input) const;
};

// 工具特定检查器（以 set_component 为例）
class ToolPermissionChecker {
public:
    virtual PermissionResult check(const std::string& inputJson) const = 0;
};

class SetComponentChecker : public ToolPermissionChecker {
public:
    PermissionResult check(const std::string& inputJson) const override {
        // 示例：禁止修改受保护字段，拦截越界值
        if (inputJson.find("entity_id") != std::string::npos)
            return {PermissionBehavior::Deny, "entity_id is immutable", "safety"};
        if (inputJson.find("health_max_99999") != std::string::npos)
            return {PermissionBehavior::Ask, "suspicious max health value", "safety"};
        return {PermissionBehavior::Allow, "passed", "safety"};
    }
};

// 统一审批运行时（借鉴 Kimi CLI ApprovalRuntime）
class ApprovalRuntime {
    std::unordered_map<std::string, std::vector<std::string>> pendingRequests;
    ApprovalSource currentSource;
    DenialTracker tracker;
    PermissionEngine engine;
    std::unordered_map<std::string, std::unique_ptr<ToolPermissionChecker>> checkers;
public:
    void setCurrentSource(const ApprovalSource& source) { currentSource = source; }

    PermissionResult resolve(const std::string& toolName, const std::string& input) {
        // 步骤 1：规则引擎
        auto result = engine.evaluate(toolName, input);
        if (result.behavior == PermissionBehavior::Deny) {
            tracker.recordDenial();
            return result;
        }

        // 步骤 2：工具特定安全检查
        auto it = checkers.find(toolName);
        if (it != checkers.end()) {
            auto safety = it->second->check(input);
            if (safety.behavior == PermissionBehavior::Deny) {
                tracker.recordDenial();
                return safety;
            }
            if (safety.behavior == PermissionBehavior::Ask) {
                return safety;
            }
        }

        // 步骤 3：拒绝追踪回退
        if (tracker.shouldFallbackToPrompting()) {
            return {PermissionBehavior::Ask, "denial limit exceeded, fallback to user", "tracker"};
        }

        // 步骤 4：最终结果
        if (result.behavior == PermissionBehavior::Allow) {
            tracker.recordAllow();
        }
        return result;
    }

    void checkAndRecord(const ApprovalSource& source, const std::string& toolName);
    void cancelBySource(const std::string& kind, const std::string& id);
    void cancelByAgentId(const std::string& agentId);
};

// 组件白名单 + 沙箱配置
struct SandboxConfig {
    std::vector<std::string> allowWrite;
    std::vector<std::string> denyWrite;   // engine.ini, .agents/skills, .git/
    std::vector<std::string> allowRead;
    std::vector<std::string> allowedDomains; // 网络白名单
};

class AgentBridge {
    std::unordered_set<std::string> allowedComponents;
    ApprovalRuntime* approvalRuntime = nullptr;
    SandboxConfig sandbox;
    ToolRegistry& toolRegistry;
    ContextCompressor* compressor = nullptr;
public:
    AgentBridge(ToolRegistry& tr) : toolRegistry(tr) {}

    bool can_mutate(const std::string& comp) const {
        return allowedComponents.count(comp);
    }

    PermissionResult request_permission(const std::string& toolName, const std::string& input) {
        if (!approvalRuntime)
            return {PermissionBehavior::Ask, "no runtime attached", "default"};
        return approvalRuntime->resolve(toolName, input);
    }

    void setApprovalRuntime(ApprovalRuntime* runtime) { approvalRuntime = runtime; }
    void setSandboxConfig(const SandboxConfig& cfg) { sandbox = cfg; }
    void setCompressor(ContextCompressor* c) { compressor = c; }

    // 统一的工具执行入口：权限 gate + Undo 快照 + 实际执行
    ToolResult execute_tool(const std::string& name, const std::string& input) {
        // 1. 权限检查
        auto perm = request_permission(name, input);
        if (perm.behavior == PermissionBehavior::Deny)
            return {false, R"({"error":"permission denied"})", {}};
        if (perm.behavior == PermissionBehavior::Ask)
            return {false, R"({"error":"waiting for user approval"})", {}};

        auto it = toolRegistry.tools.find(name);
        if (it == toolRegistry.tools.end())
            return {false, R"({"error":"unknown tool"})", {}};

        // 2. 写操作前自动 checkpoint（支撑 Undo）
        if (!it->second.isReadOnly && compressor) {
            compressor->checkpoint("auto_before_" + name);
        }

        // 3. 执行工具
        return it->second.call(input);
    }
};
```

> **借鉴点说明**：
> - 引入 **PermissionEngine** 实现四级规则来源合并与三级决策树（Deny → Ask → Allow），与 Claude Code `hasPermissionsToUseToolInner` 对齐。
> - 引入 **DenialTracker**，连续拒绝 3 次后强制回退到人工 Ask，避免 Auto 模式死循环。
> - 引入 **ToolPermissionChecker** 接口，让 `set_component` 等工具可以像 Claude Code BashTool 一样做参数级安全检查。
> - 保留 Kimi CLI 风格的 **ApprovalSource**，支持按 Agent 批量取消 pending 审批。
> - 增加 **SandboxConfig**，把 `TransactionalWorld` 的抽象想法落地为具体的读写路径和网络域规则。

#### 阶段 3 补充：自治与规模

##### A. Archetype 查询作为 AI 的第一类接口

纯引擎的 Archetype 已经提升了 cache locality。对 AI 来说，它还有一个额外好处：
- AI 说"找出所有带 `Enemy` + `LowHealth` 的实体"，引擎直接定位到对应 Archetype，无需遍历整个世界。

##### B. Deterministic Replay Pipeline 的 AI 测试价值

纯引擎的确定性回放已经用于网络同步。对 AI 来说，它是**自主跑回归测试**的基石：
- AI 可以改动参数 → 重放同一批输入 → 验证输出是否一致。

##### C. 沙箱事务系统（Transactional Sandbox）

纯引擎已有 `TransactionalWorld` 的雏形。对 AI 来说，标准工作流是：
1. `enter_sandbox()`
2. `bridge.set_component(...)` → 尝试改动
3. `bridge.step(dt)` → 运行几帧观察效果
4. 不满意？`discard_sandbox()` → 回滚；满意则 `apply_to_main()`

##### D. Orchestrator 层调度多 Agent

```cpp
enum class AgentTaskType { ReadOnly, WriteHeavy, Verification };

class AgentOrchestrator {
    struct AgentTask {
        std::string agentId;
        AgentTaskType type;
        std::unordered_set<std::string> targetComponents;
        std::unordered_set<Entity> targetEntities;
        std::function<ToolResult(AgentBridge&)> execute;
        bool reuseContext = false;  // true = Continue, false = Spawn Fresh
    };

    std::vector<AgentTask> pendingTasks;

public:
    void submit(AgentTask task);
    void dispatchAll(AgentBridge& bridge, World& world);
};

void AgentOrchestrator::dispatchAll(AgentBridge& bridge, World& world) {
    std::vector<AgentTask> reads, writes, verifies;
    for (auto& t : pendingTasks) {
        if (t.type == AgentTaskType::ReadOnly) reads.push_back(t);
        else if (t.type == AgentTaskType::WriteHeavy) writes.push_back(t);
        else verifies.push_back(t);
    }

    // 1. 读任务：自由并行
    parallel_for(reads, [&](auto& t) { t.execute(bridge); });

    // 2. 写任务：按目标实体/组件集合检测冲突，有重叠则串行
    for (size_t i = 0; i < writes.size(); ++i) {
        bool conflict = false;
        for (size_t j = 0; j < i; ++j) {
            if (hasOverlap(writes[i].targetComponents, writes[j].targetComponents) &&
                hasOverlap(writes[i].targetEntities, writes[j].targetEntities)) {
                conflict = true;
                break;
            }
        }
        if (!conflict) {
            writes[i].execute(bridge);  // 并行执行（实际实现中可放入 job queue）
        } else {
            writes[i].execute(bridge);  // 冲突则等待前一个完成后再执行
        }
    }

    // 3. 验证任务：如果验证区域与正在执行的写区域不重叠，可并行
    for (auto& t : verifies) {
        t.execute(bridge);
    }

    pendingTasks.clear();
}
```

- `PhysicsAgent`：只读写 `RigidBody`、`Collider`
- `RenderAgent`：只读写 `Material`、`Mesh`
- `GameplayAgent`：只读写 `Health`、`Inventory`
- Orchestrator 负责合并它们的改动、检测冲突、决定执行顺序。

> 借鉴 Claude Code Coordinator 的并发哲学：**读任务自由并行，写任务按文件集/组件集串行化**。同时引入 `reuseContext` 标志，让 Orchestrator 决定是复用同一个 Agent 实例的上下文（Continue）还是开启新实例（Spawn Fresh），避免错误路径的上下文污染。

### 设计权衡表

| 决策点 | 原型阶段 | 商业引擎阶段 |
|--------|---------|-------------|
| ECS 存储 | 每个类型一个 `std::vector` | Archetype/Chunk-based 或 Sparse Set |
| 反射 | 宏 + 手动注册 | 编译器插件 / clang 解析生成 |
| AI 接口 | `AgentBridge` 手写 3~5 个工具 | 完整 MCP 协议 + Schema 验证 |
| 并发模型 | 单线程分阶段 Tick | 依赖图并行 + Job System |
| 状态变更可见性 | 全量轮询 | 增量 Change Log |
| 错误恢复 | 手动调试 | 快照 + 确定性回放 + 事务沙箱 |
| 渲染抽象 | 不需要（或最小 OpenGL 封装） | RHI + RenderGraph |
| 编辑器 | JSON Inspector 就够了 | 完整工具链 + MCP 桥接 |

> **关键洞察**：原型阶段的最大错误是过早引入商业引擎的复杂度；原型阶段欠下的技术债如果不包含"可观测性"和"确定性"，那么当你引入 AI 协作时，还债成本会指数级上升。

### 如果我要 vibe coding，该偷哪几招？

#### 骨架层必偷（没有 AI 也需要）

1. **所有游戏状态必须是 ECS 组件**
   - 不要在脚本里藏局部变量，不要依赖隐式的单例状态。

2. **运行时自描述（TypeRegistry）**
   - 让外部工具（编辑器、调试器、AI）无需头文件即可理解数据结构。

3. **每一帧生成 `ChangeLog`**
   - 记录哪些实体的哪些组件被增删改。

4. **世界快照必须是廉价操作**
   - 使用 memory arena 或 copy-on-write，让"保存 → 尝试 → 回滚"在几毫秒内完成。

5. **Tick 必须是确定性的**
   - 固定步长、固定随机种子、固定系统执行顺序。

#### AI 层必偷（引入 AI 后需要）

6. **暴露 `query_entities` + `get_component` 工具**
   - 让 AI 能用一句话拿到它关心的所有实体，并读取/修改具体字段。

7. **ChangeLog 实施滚动窗口清理**
   - 超过 16 帧的旧记录自动折叠为统计，防止 AI 上下文无限膨胀。

8. **为 AI 设计 `EngineDisplayBlock` 反馈协议**
   - 把工具执行结果拆成两部分：`json` 给 AI 消费，`blocks` 给 UI 渲染。
   - 定义 `changelog`、`diff`、`profile`、`screenshot`、`task_status` 等标准块类型，让 Editor、Web 调试器、CLI 都能统一消费。

9. **预生成 `ENGINE_AI.md` 作为静态上下文**
   - 把 ECS 架构、组件说明、常用调参模式写成文档，作为 system prompt 的固定前缀注入。
   - 这样 AI 不需要每轮都重新学习引擎规则，还能享受 Prompt Caching 的成本优化。

10. **优先实现 MCP 协议而非自定义协议**
    - 引擎的 AI 接口应该首先是一套 MCP Server，`AgentBridge` 是引擎内部对这套 MCP 接口的封装。这样你的引擎可以直接被任何支持 MCP 的 AI 客户端调用。

11. **引入 Agent Loop 层**
    - 不要只给 AI 暴露工具，还要有一个循环层负责 "调用 LLM → 解析 tool_calls → 执行引擎工具 → 把结果回写 → 再调 LLM"。这是让 AI 从"写代码"进化到"自主调试"的关键。

#### 进阶偷（当项目超过原型规模）

12. **引入 Orchestrator 层调度多个专项 Agent**
    - 每个 Agent 绑定组件白名单，防止"越权操作"。

13. **Archetype 查询作为 AI 的第一类接口**
    - 让自然语言查询能直接映射到高效的 ECS archetype 迭代。

14. **构建 Deterministic Replay Pipeline**
    - 从 Day 1 就把所有输入按帧编号记录。未来 AI 可以自主跑回归测试。

15. **核心系统插入 Invariant 断言**
    - 让引擎本身成为 Agent 改动的"自动审核员"。

16. **为 AI 优化 Inspector 协议，而不是复用人类 GUI 协议**
    - 直接输出 JSON/Markdown，不要让 AI 去解析为人类设计的二进制 UI 数据。
