# 事件总线

> **前置依赖**：[[窗口与输入系统]]、[[系统调度与确定性]]
> **本模块增量**：让引擎内部系统之间能解耦通信；把原始硬件输入映射为游戏语义动作；支持延迟事件队列，保证迭代安全。
> **下一步**：[[Prefab与数据层]] — 事件驱动的状态变化最终需要被持久化和回放。

---

## Why：为什么引擎需要事件总线？

### 没有这个系统时，世界有多糟

1. **系统之间直接调用，耦合成团**
   - `PlayerSystem` 直接调用 `DamageSystem::takeDamage()`。一个月后，你想把伤害计算拆到服务器上，却发现调用链深埋在 20 个 cpp 文件里。

2. **同一帧内创建和销毁导致崩溃**
   - `GameplaySystem` 在遍历敌人列表时，某个敌人被子弹击中触发了 `destroy(entity)`。`std::vector` 里的元素被删除，后面的实体索引全部错位，遍历直接崩溃或跳过对象。

3. **输入与游戏逻辑硬耦合**
   - 代码里到处是 `if (key == GLFW_KEY_SPACE) player.jump()`。换到手柄上，你要把 `Space` 全部替换成 `ButtonSouth`。每次支持新设备都是一场重构噩梦。

4. **AI 无法注入输入验证行为**
   - 你想让 AI Agent 测试"连续按 10 次跳跃键"，但 AI 没有统一的事件注入接口，只能写裸的内存修改代码。

### 三个真实场景

| 场景 | 没有事件总线的困难 | 事件总线的答案 |
|------|-------------------|---------------|
| **场景 1：子弹击中敌人，敌人立即死亡并掉落物品** | 死亡逻辑在同一帧删除了敌人实体，掉落物系统遍历到已删除实体崩溃 | 所有销毁请求进入延迟事件队列，在本帧所有系统执行完毕后的安全点统一应用 |
| **场景 2：从键盘切换到手柄** | 所有 `if (key == GLFW_KEY_SPACE)` 都要重写 | 输入映射层把 `ButtonSouth` 和 `Space` 都映射为 `Jump` 动作，游戏逻辑只订阅 `Jump` 事件 |
| **场景 3：AI 想测试特定输入序列** | 只能写内存修改或模拟 OS 消息 | AI 通过 MCP 注入结构化事件：`inject_event("Jump", {triggered: true})` |

> **核心结论**：事件总线是引擎的"神经系统"。它让系统之间解耦、让输入与设备无关、让 AI 能安全地模拟和验证行为。

## What：最简化版本的事件总线长什么样？

下面的代码包含三个核心能力：
1. **Pub-Sub 消息系统**：系统之间通过事件类型注册和订阅
2. **延迟事件队列**：当前帧注册、下一帧（或安全点）执行
3. **输入事件到 ECS 的映射**：把原始按键映射为游戏语义动作

```cpp
#include <cstdint>
#include <vector>
#include <string>
#include <unordered_map>
#include <functional>
#include <queue>

// ============================================================
// 1. Pub-Sub 消息系统
// ============================================================
struct GameEvent {
    std::string type; // e.g., "OnDamage", "OnJump", "OnLevelUp"
    uint32_t    sender;
    uint32_t    target;
    std::string payload; // JSON 或二进制数据
};

class EventBus {
    std::unordered_map<std::string, std::vector<std::function<void(const GameEvent&)>>> subscribers;
public:
    void subscribe(const std::string& type, std::function<void(const GameEvent&)> cb) {
        subscribers[type].push_back(std::move(cb));
    }
    void publish(const GameEvent& e) {
        auto it = subscribers.find(e.type);
        if (it != subscribers.end()) {
            for (auto& cb : it->second) cb(e);
        }
    }
};

// ============================================================
// 2. 延迟事件队列
// ============================================================
class DelayedEventQueue {
    std::queue<GameEvent> pending;
public:
    void enqueue(const GameEvent& e) { pending.push(e); }
    void flush(EventBus& bus) {
        while (!pending.empty()) {
            bus.publish(pending.front());
            pending.pop();
        }
    }
    void clear() {
        while (!pending.empty()) pending.pop();
    }
};

// ============================================================
// 3. 输入事件到 ECS 的映射
// ============================================================
struct InputAction {
    std::string name;      // "MoveForward", "Jump", "Fire"
    float       value;     // 模拟量：0.0 ~ 1.0
    bool        triggered; // 本帧是否首次触发
    bool        performed; // 是否持续中
    bool        cancelled; // 本帧是否释放
};

class InputMapper {
    std::unordered_map<int, std::string> keyToAction;
public:
    void bindKey(int keyCode, const std::string& actionName) {
        keyToAction[keyCode] = actionName;
    }
    std::vector<InputAction> mapRawEvents(const std::vector<struct RawInputEvent>& rawEvents) {
        std::vector<InputAction> actions;
        for (auto& e : rawEvents) {
            auto it = keyToAction.find(e.keyCode);
            if (it != keyToAction.end()) {
                actions.push_back({it->second, 1.0f, e.pressed, e.pressed, !e.pressed});
            }
        }
        return actions;
    }
};
```

### 这个最小实现已经解决了什么

1. **系统解耦**：`DamageSystem` 不需要知道 `LootSystem` 的存在，只需要发布 `OnDeath` 事件。
2. **迭代安全**：结构修改请求（spawn/destroy/add/remove）先进入延迟队列，在 Tick 安全点统一应用。
3. **输入抽象**：游戏逻辑只订阅 `Jump` 动作，不关心 `Space` 键还是手柄 `A` 键。

### 还缺什么

- **没有事件优先级**：所有事件按 FIFO 处理，无法表达"输入事件优先于 AI 事件"。
- **没有输入状态机**：按键只有 pressed/released，没有 `started → performed → cancelled` 的完整生命周期。
- **没有 UI 竞争处理**：UI 和 Gameplay 可能同时消费同一个点击事件。
- **没有持久化/回放**：事件流没有被记录，无法重放。

## How：真实引擎的事件总线是如何一步一步复杂起来的？

### 阶段 1：能用 —— 从直接调用到 Pub-Sub

#### 触发原因

系统数量从 3 个增长到 15 个，直接函数调用导致编译依赖爆炸。某个 System 的接口变动会触发半个引擎重新编译。

#### 代码层面的变化

**A. 事件类型注册表**

不再用字符串硬编码 `"OnDamage"`，而是为每种事件生成一个 `EventTypeID`：

```cpp
enum class EventType : uint32_t {
    None = 0,
    OnDamage,
    OnDeath,
    OnJump,
    OnLevelUp,
    Count
};
```

**B. 事件队列与调度器集成**

延迟事件队列不再独立运行，而是被集成到 [[系统调度与确定性]] 的 `Scheduler` 中：

```cpp
void Scheduler::tick(World& world, float dt) {
    tickPhase(Phase::PreUpdate, world, dt);
    tickPhase(Phase::Update, world, dt);
    tickPhase(Phase::PostUpdate, world, dt);
    eventQueue.flush(eventBus); // 安全点：所有 System 执行完毕后才分发事件
    applyCommandBuffer(world);
}
```

这保证了：
- 事件消费不会在 System 执行中途触发。
- 事件引发的结构修改（如 `OnDeath` 导致 `destroy`）仍然可以进入 `CommandBuffer`，在下一安全点应用。

### 阶段 2：好用 —— 输入系统四层架构

#### 触发原因

原始按键/鼠标坐标无法直接驱动游戏逻辑。玩家按的是 `Space`，游戏想要的是 `Jump`；玩家用的是手柄左摇杆，游戏想要的是 `MoveForward/MoveRight`。如果没有映射层，输入逻辑会与硬件强耦合。

#### 代码层面的变化

**输入系统四层架构**：

```
┌─────────────────────────────────────────┐
│  硬件事件层（OS API：Win32 WndProc、X11）  │
│  输出：WM_KEYDOWN、鼠标坐标、手柄摇杆值      │
├─────────────────────────────────────────┤
│  设备抽象层（Device Manager）             │
│  职责：轮询硬件、维护状态块、统一数据格式     │
│  输出：标准化的状态块（key_state[256] 等）   │
├─────────────────────────────────────────┤
│  控制语义层（Control + Action）           │
│  职责：边缘检测、组合逻辑、动作语义映射       │
│  输出：InputAction 触发状态（triggered 等） │
├─────────────────────────────────────────┤
│  方案管理层（Scheme Manager）             │
│  职责：运行时切换输入方案、按键重绑定        │
│  输出：当前激活方案下的 Action 回调         │
└─────────────────────────────────────────┘
```

#### 2.1 Control 状态机

一个按键不是简单的 `true/false`。它的生命周期是：

```
┌─────┐   按下    ┌─────────┐  持续按住  ┌───────────┐   释放    ┌─────┐
│ idle│ ──────► │ started │ ─────────► │ performed │ ──────► │idle │
└─────┘         └─────────┘            └───────────┘         └─────┘
                                                     也可中途取消
                                                     ──────► cancelled
```

- `started`：本帧首次按下。适合"跳跃"（只触发一次，不连续跳）。
- `performed`：持续按住。适合"蓄力射击"。
- `cancelled`：本帧释放。适合检测"按键结束"，或 UI 强制消费事件。

#### 2.2 事件竞争：UI vs Gameplay

当鼠标 hover 在 UI 按钮上时，点击不应该同时触发"开枪"。解决方案：

1. UI 系统每帧报告 `want_capture_mouse` / `want_capture_keyboard`。
2. 输入系统的消费逻辑在分发前检查这些标志。
3. 如果 UI 已消费，对应的 `InputControl` 被强制置为 `cancelled`。

#### 2.3 IME（输入法）支持

中文、日文、韩文输入不能直接用 `KeyPress` 事件捕获。IME 需要：
- `startIME(window_handle)`：在输入框聚焦时激活输入法上下文。
- `tickIME(x, y, window_handle)`：每帧更新候选框位置，使其跟随输入框光标。
- `getCompositionString()`：获取拼音/假名的组合状态。
- `getResultString()`：获取用户最终确认的文字。

IME 层应做成**薄封装**：不缓存大量状态，直接透传 OS 原生参数，减少平台差异泄漏。

### 阶段 3：工业级 —— 事件持久化与确定性回放

#### 触发原因

网络同步、AI 自动测试、Bug 复现都需要"把输入事件流保存下来，在任何时刻重放"。

#### 代码层面的变化

**A. 输入事件序列化**

```cpp
struct InputFrameRecord {
    uint64_t frameNumber;
    float    logicDelta;
    std::vector<InputAction> actions;
};

class InputRecorder {
    std::vector<InputFrameRecord> history;
public:
    void record(uint64_t frame, float dt, const std::vector<InputAction>& actions) {
        history.push_back({frame, dt, actions});
    }
    bool replay(uint64_t frame, std::vector<InputAction>& outActions) {
        // 返回对应帧的输入动作
    }
};
```

**B. 事件流与 ChangeLog 的融合**

引擎内部有两种"变化"：
- **输入事件流**：外部对引擎的干预（按键、网络包、AI 命令）。
- **ChangeLog**：引擎内部的状态变化（组件字段修改）。

确定性回放的秘诀是：**只记录输入事件流**，然后重新运行引擎，让 ChangeLog 自然复现。如果两次运行的 ChangeLog 不一致，就说明存在非确定性 bug。

**C. 事件过滤与订阅白名单**

工业级 EventBus 支持订阅过滤：

```cpp
bus.subscribe("OnDamage", 
    [](const GameEvent& e) { /* 处理所有伤害事件 */ },
    [](const GameEvent& e) { return e.target == playerEntity; } // 只订阅与玩家相关的事件
);
```

## ECS 映射：事件总线在 ECS 下该如何表达？

### 输入事件的 ECS 表达

输入事件是**瞬态的**（transient），不应该长期存储在组件中。推荐两种模式：

**模式 A：事件队列 + CommandBuffer**
- 窗口回调将事件写入全局/World 级别的事件队列。
- `InputSystem` 在 Tick 开始时将队列中的事件转换为对 `InputComponent` 的修改，或通过 `CommandBuffer` 延迟应用。

**模式 B：瞬态组件（Transient Component）**
- 定义一个只存活一帧的组件：
  ```cpp
  struct InputFrame {
      std::vector<KeyEvent> keys;   // 或更紧凑的位掩码
      float mouse_x, mouse_y;
  };
  ```
- 每帧开始由系统清空并重填。Gameplay System 查询带 `InputFrame` 的实体进行处理。

### 输入映射的 ECS 化

```cpp
// 硬件输入组件（直接来自设备）
struct RawKeyboard { uint64_t key_mask; };
struct RawMouse    { float dx, dy; bool buttons[3]; };

// 游戏语义组件（经过映射层处理）
struct MoveInput   { float forward, right; };
struct JumpInput   { bool triggered; };
```

映射层作为一个 System 运行：读取 `RawKeyboard` + `RawMouse`，输出 `MoveInput` + `JumpInput`。这样 AI Agent 既可以直接操作底层硬件输入（测试路径），也可以直接修改语义组件（作弊/调试路径）。

### 事件总线与系统调度的关系

```cpp
// Scheduler 中的事件分发点
void Scheduler::tick(World& world, float dt) {
    tickPhase(Phase::PreUpdate, world, dt);
    tickPhase(Phase::Update, world, dt);
    tickPhase(Phase::PostUpdate, world, dt);
    eventQueue.flush(eventBus);     // 安全点 1：分发延迟事件
    applyCommandBuffer(world);       // 安全点 2：应用结构修改
    world.changelog.clear();
}
```

> **关键设计**：事件分发和 CommandBuffer 应用必须在所有 System 执行完毕之后。这保证了没有任何 System 会在迭代中途收到事件或看到半成品的结构修改。

## AI 友好设计检查清单

| 检查项 | 本模块的实现策略 |
|--------|----------------|
| **状态平铺** | 输入事件队列和 Action 列表都是平铺数组；订阅表是简单的 `EventType -> Handler[]` 映射。 |
| **自描述** | 输入动作名（`Jump`、`MoveForward`）、事件类型（`OnDamage`）通过反射注册，AI 不读头文件即可查询。 |
| **确定性** | 输入事件流按帧号严格排序；Replay 时按相同帧号注入相同事件序列。 |
| **工具边界** | AgentBridge 提供 `input.inject_action(name, value)`、`event.query_history(frame)` 等结构化接口。 |
| **Agent 安全** | 输入注入接口只影响事件队列，不直接修改任意组件；危险操作（如 `OnDestroyWorld`）可被 EventBus 订阅白名单拦截。 |

## 性能与 Trade-off：诚实地说清代价

### 选择 1：直接调用 vs 事件总线

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 直接函数调用 | 零开销、调试直观、调用栈清晰 | 编译耦合高、难以动态扩展 | 紧密耦合的核心系统（如 Transform → Render） |
| 事件总线 | 解耦、动态订阅、AI 可注入 | 间接跳转、调试困难、事件顺序不可预测（如果不加控制） | 跨模块通信、输入系统、AI 操作 |

> **反例警告**：不要把所有 System 之间的通信都改成事件总线。高频、强依赖的通信（如 TransformSystem 每帧给 RenderSystem 提供世界矩阵）应该保持直接数据流，否则事件分发开销会压垮帧率。

### 选择 2：逐帧处理输入 vs 缓冲输入

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 逐帧处理 | 确定性高、与 ECS Tick 天然对齐 | 快速连按可能在单帧内丢失 | **Gameplay 逻辑**（需要确定性回放） |
| 缓冲处理 | 不丢事件、响应性更好 | 增加延迟、回放时更难对齐时间戳 | **UI 交互**、**打字输入** |

> **最佳实践**：Gameplay 输入用逐帧 + 时间戳；UI 输入用缓冲 + 立即消费。两者共存，由不同的 System 负责。

## 如果我要 vibe coding，该偷哪几招？

1. **事件总线只用于跨模块通信**
   - 不要把每帧都发生的高频数据流（如 Transform 更新）走事件总线。只用它处理"发生了某件事"的通知。

2. **延迟事件队列是防止崩溃的生命线**
   - 所有结构修改请求（spawn/destroy/add/remove）都先进入延迟队列，在 Tick 安全点统一应用。

3. **输入映射层从 Day 1 就要有**
   - 游戏逻辑只订阅 `Jump`、`Fire`、`MoveForward`。`Space` 键和手柄 `A` 键的映射在更底层处理。

4. **Control 四态状态机**
   - `idle → started → performed → cancelled` 比简单的 bool 按键精确得多，几行代码就能避免"一帧多次触发"的 bug。

5. **UI 消费标志**
   - `want_capture_mouse` 是防止"点击按钮同时开枪"的最小解决方案，ImGui 已经帮你做了标志输出，直接复用。

6. **记录输入事件流用于 Replay**
   - 每帧记录 `{frame, dt, actions}` 到文件。遇到 bug 时，加载记录即可 100% 复现场景。这是 AI 自动跑回归测试的基石。
