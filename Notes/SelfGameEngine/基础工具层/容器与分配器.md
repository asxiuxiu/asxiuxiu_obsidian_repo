# 容器与分配器

## Why：为什么游戏引擎需要自定义容器和分配器？

初学者写引擎时，第一反应是"用 `std::vector`、`std::map`、`new`/`delete` 不就好了？"。这在原型阶段确实够用，但一旦进入生产环境，STL 的四个核心痛点会逐一爆发：

1. **分配器不可插拔**：`std::vector` 的内存来源几乎是硬编码的全局 `malloc`/`free`。你无法让临时渲染命令从"帧内存"里分配，也无法让 ECS 组件从固定对齐的 Chunk 里出生。
2. **异常安全冲突**：游戏引擎通常禁用异常（`-fno-exceptions`），但 STL 容器在内存不足时会抛 `std::bad_alloc`。编译开关和库实现一旦不一致，就是链接灾难。
3. **缓存不友好**：`std::list` 是指针跳转链，`std::unordered_map` 是 bucket + 链表/树，遍历时 cache miss 极高。而游戏引擎大量依赖"批量处理"（遍历所有粒子、所有碰撞体、所有 DrawCall）。
4. **调试与确定性缺失**：全局堆是子系统混用的黑箱。一帧之内 `malloc` 的时序、地址、碎片状态完全不可预测，导致离线回放和线上崩溃难以复现。

### 三个真实场景

| 场景             | 如果用 STL                             | 后果                                |
| -------------- | ----------------------------------- | --------------------------------- |
| **ECS 组件批量分配** | `new ComponentA[1024]`              | 构造全部执行一次；无法按 Entity Chunk 对齐；碎片随机 |
| **每帧临时渲染命令**   | `std::vector<DrawCmd> tempCmds`     | 每帧都隐式 `malloc`/`free`；容量抖动；析构所有命令 |
| **音频包线程间传递**   | `std::queue<AudioPacket>` + `mutex` | 锁竞争导致优先级反转；音频线程被阻塞，产生爆音           |

自定义容器和分配器，不是"重复造轮子"，而是**把内存从不可控的黑箱变成可编排的资源**。

### 问题：日常代码中最常用的动态数组，为什么 `std::vector` 不够用？

#### 最 naive 的方案

```cpp
std::vector<Entity> entities;
entities.push_back(e);
```

这在原型阶段完全够用。但进入生产环境后，`std::vector` 的三个设计决策与游戏引擎冲突：

1. **分配器不可插拔**：`std::vector` 的 `Allocator` 模板参数虽然存在，但标准分配器接口复杂（`rebind`、`construct`、`destroy` 等），且实际游戏中你需要让临时数组从帧内存分配、持久数组从对象池分配——`std::vector` 做不到简洁的切换。
2. **增长策略固定**：`std::vector` 的增长因子由实现定义（通常是 2x 或 1.5x），你无法针对引擎场景调优。例如 UE 使用 `NewMax + (3/8)*NewMax + 16`，比纯指数增长更平滑。
3. **异常安全冲突**：`std::vector` 在 `push_back` 失败时抛 `std::bad_alloc`。游戏引擎通常编译时禁用异常（`-fno-exceptions`），这会导致链接失败或未定义行为。

#### 改进：Array<T, Allocator>

```cpp
template <typename T, typename Allocator = DefaultAllocator>
struct Array {
    T* data = nullptr;
    usize count = 0;
    usize capacity = 0;
    Allocator allocator;

    void Reserve(usize newCap) {
        if (newCap <= capacity) return;
        T* newData = static_cast<T*>(
            allocator.Allocate(newCap * sizeof(T), alignof(T))
        );
        for (usize i = 0; i < count; ++i) {
            new (newData + i) T(static_cast<T&&>(data[i])); // move
            data[i].~T();
        }
        if (data) allocator.Free(data);
        data = newData;
        capacity = newCap;
    }

    void Push(const T& item) {
        if (count >= capacity) {
            usize newCap = capacity == 0 ? 4 : capacity + (capacity >> 1) + 16;
            Reserve(newCap);
        }
        new (data + count) T(item);
        ++count;
    }

    void Pop() {
        if (count > 0) {
            --count;
            data[count].~T();
        }
    }

    void Clear() {
        for (usize i = 0; i < count; ++i) data[i].~T();
        count = 0;
    }

    T& operator[](usize i) { return data[i]; }
    const T& operator[](usize i) const { return data[i]; }
    usize Size() const { return count; }
    usize Capacity() const { return capacity; }
    bool Empty() const { return count == 0; }
};
```

**状态变化图**：Push 触发扩容

```
初始：data=[a,b,c], count=3, capacity=3
         ▲
         └── 满了

Reserve(6)：分配新块，移动元素
  旧: [a,b,c] ──► 释放
  新: [a,b,c, _, _, _]
              ▲
              count=3, capacity=6

Push(d)：
  [a,b,c,d, _, _]
           ▲
           count=4
```

> **这段代码已经解决了什么**：分配器参数化（任何实现 `Allocate`/`Free` 的类型都可以注入）、无异常、增长策略可控、显式构造/析构（对有非平凡析构函数的类型安全）。
> **还缺什么**：没有 shrink-to-fit、没有插入/删除中间元素（`O(n)` 需自行实现）、没有范围检查（生产环境应加 `ASSERT(i < count)`）。
> **反例警告**：如果 `T` 的移动构造函数抛异常（在禁用异常的构建中不可能，但如果用拷贝构造函数且类型内部有裸指针管理），上述 `Reserve` 中的 move 循环在失败时会泄漏已移动但未析构的元素。**工业级实现需要"先在新内存构造，成功后统一析构旧元素"的两阶段提交**。

#### 与分配器整合：不同场景的不同选择

```cpp
// 场景 A：每帧生成的临时渲染命令 → 从帧内存分配
Array<DrawCmd, FrameArenaAllocator> frameCommands;

// 场景 B：持久存在的实体列表 → 从全局堆分配（默认）
Array<Entity> persistentEntities;

// 场景 C：固定预算的音频通道 → 从对象池分配
Array<AudioChannel, PoolAllocator<AudioChannel, 32>> channels;
```

同一个 `Array<T, Allocator>` 模板，通过更换分配器，适配完全不同的内存策略。这是 `std::vector` 的分配器参数无法简洁表达的。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「通用动态数组的分配器可控性」这个子问题。
> - **UE** 的 `TArray<T, Allocator>` 是行业标杆。它的分配器参数化极其灵活：`FHeapAllocator`（默认堆分配）、`TInlineAllocator<N>`（前 N 个元素放在栈上，超出后才堆分配）、`TNonRelocatableInlineAllocator`（对不可移动类型优化）。UE 的 `TArray` 还实现了 `Serialize(Ar)` 接口，可直接序列化到资产文件。
> - **Bevy** 作为 Rust 引擎，直接使用标准库的 `Vec<T>`。Rust 的 `Vec` 没有自定义分配器参数（直到 Rust nightly 的 `Allocator API` 稳定）。Bevy 的应对方式是：通用代码用 `Vec`，ECS 核心存储层用完全自定义的 `BlobArray` + `ThinArrayPtr`。
>
> **个人项目推荐**：实现 `Array<T, Allocator>` 模板，Allocator 只需实现 `Allocate(size, align)` 和 `Free(ptr)` 两个方法。默认用 `DefaultAllocator`（封装 `malloc`/`free`）。至少提供 `TInlineAllocator<N>` 变体——这对小数组（如 4x4 矩阵列表）能显著减少堆分配次数。

### 问题：通用键值查找时，为什么 `std::unordered_map` 不够用？

动态数组解决了"有序列表"的问题，但引擎中还有大量"按键查找"的需求：资源路径 → 句柄、组件类型 ID → 工厂函数、字符串 → 本地化文本。`std::unordered_map` 能用吗？

#### 最 naive 的方案

```cpp
std::unordered_map<std::string, ResourceHandle> resources;
resources["mesh/player.fbx"] = handle;
```

`std::unordered_map` 的底层是 **bucket + 链表/红黑树**。每个节点独立分配内存，遍历时要追着指针跳转，cache miss 极高。更糟糕的是：标准没有规定桶的数量、负载因子阈值、rehash 时机——不同平台的表现可能不一致。而且和 `std::vector` 一样，它的分配器参数化几乎是个摆设，你无法让它从帧内存或对象池中分配节点。

#### 改进：HashMap<K, V, Hash, Allocator>

采用**开放寻址法** + **线性探测**，所有 Entry 存储在一个连续数组中：

```cpp
template <typename K, typename V,
          typename Hash = DefaultHash<K>,
          typename Allocator = DefaultAllocator>
struct HashMap {
    struct Entry {
        u64 hash;      // 完整哈希值，用于快速拒绝（避免比较 Key）
        K key;
        V value;
        bool occupied; // false = 空槽
    };

    Entry* entries = nullptr;
    usize count = 0;
    usize capacity = 0;
    Allocator allocator;

    void Reserve(usize minCapacity) {
        usize newCap = NextPowerOfTwo(minCapacity * 2); // 保持 50% 负载因子
        Entry* newEntries = static_cast<Entry*>(
            allocator.Allocate(newCap * sizeof(Entry), alignof(Entry))
        );
        // 初始化所有槽为未占用
        for (usize i = 0; i < newCap; ++i) newEntries[i].occupied = false;

        // Rehash：迁移旧元素
        if (entries) {
            for (usize i = 0; i < capacity; ++i) {
                if (entries[i].occupied) {
                    InsertInto(newEntries, newCap,
                               entries[i].hash, entries[i].key,
                               static_cast<V&&>(entries[i].value));
                    entries[i].key.~K();
                    entries[i].value.~V();
                }
            }
            allocator.Free(entries);
        }
        entries = newEntries;
        capacity = newCap;
    }

    V* Find(const K& key) {
        if (!entries) return nullptr;
        u64 h = Hash::Hash(key);
        usize idx = h & (capacity - 1); // 容量是 2 的幂，可用位掩码
        for (usize probe = 0; probe < capacity; ++probe) {
            Entry& e = entries[idx];
            if (!e.occupied) return nullptr; // 空槽 = 不存在
            if (e.hash == h && e.key == key) return &e.value;
            idx = (idx + 1) & (capacity - 1); // 线性探测
        }
        return nullptr;
    }

    void Insert(const K& key, const V& value) {
        if (count * 2 >= capacity) Reserve(capacity == 0 ? 16 : capacity);
        u64 h = Hash::Hash(key);
        InsertInto(entries, capacity, h, key, value);
        ++count;
    }

private:
    static void InsertInto(Entry* table, usize cap, u64 h,
                           const K& k, const V& v) {
        usize idx = h & (cap - 1);
        while (table[idx].occupied) {
            idx = (idx + 1) & (cap - 1);
        }
        table[idx].hash = h;
        new (&table[idx].key) K(k);
        new (&table[idx].value) V(v);
        table[idx].occupied = true;
    }
};
```

**状态变化图**：Insert("B") 时的线性探测

```
容量 = 8, 负载因子 = 50%（即最多存 4 个元素）

初始状态（已存 A, C）：
  idx0: [hash(A), "A", val, occupied]
  idx1: [empty]
  idx2: [hash(C), "C", val, occupied]
  idx3~7: [empty]

Insert("B")：
  hash("B") & 7 = 1
  idx1 为空，直接放入：
  idx1: [hash(B), "B", val, occupied]

Insert("D")，假设 hash("D") & 7 = 1：
  idx1 已被 B 占据
  探测到 idx2：被 C 占据
  探测到 idx3：空，放入：
  idx3: [hash(D), "D", val, occupied]
```

> **这段代码已经解决了什么**：连续内存存储（cache 友好）、开放寻址法（无指针跳转）、分配器参数化、快速拒绝（先比 hash，hash 相等再比 key）。
> **还缺什么**：没有删除操作（开放寻址法的删除需要 tombstone 标记，会污染探测链）、没有迭代器（遍历需要跳过空槽）、rehash 时所有元素重新构造（对不可移动类型不友好）。
> **反例警告**：如果 Key 的哈希质量很差（如大量 Key 哈希到同一值），线性探测会退化为线性扫描，O(1) 变 O(n)。**必须提供一个好的默认哈希函数，并在调试构建中检测最长探测链长度**。

#### HashSet<T>：HashMap 的薄封装

```cpp
template <typename T, typename Hash = DefaultHash<T>, typename Allocator = DefaultAllocator>
struct HashSet {
    struct Dummy {}; // void 值类型的占位
    HashMap<T, Dummy, Hash, Allocator> map;

    void Insert(const T& key) { map.Insert(key, Dummy{}); }
    bool Contains(const T& key) { return map.Find(key) != nullptr; }
};
```

> **与 SparseSet 的关键区别**：
> - **SparseSet**：Entity ID 是连续的整数（0, 1, 2, ...），直接用 `sparse[entityID]` 数组索引 → 本质上是**完美哈希**，O(1) 且无需哈希计算
> - **HashMap**：Key 是任意类型，必须经过哈希函数 → 开放寻址法 → 平均 O(1)，但有哈希冲突和探测开销
>
> 在 ECS 中，Entity 到组件的映射用 SparseSet（因为 Entity ID 是连续整数）。在通用场景中（字符串 → 句柄、类型名 → 工厂），用 HashMap。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「通用键值查找的缓存友好性」这个子问题。
> - **UE** 的 `TMap<Key, Value>` 基于开放寻址法，但底层用 `TSparseArray` 存储实际元素。`TMap` 内部维护一个 `Hash` 数组存储哈希值，一个 `Elements` 数组（`TSparseArray`）存储键值对。`TSet<T>` 是 `TMap<T, void>` 的薄封装。UE 的哈希表支持**稳定句柄**（`FSetElementId`）——删除元素不会使其他句柄失效，这对需要持久引用的场景至关重要。
> - **Bevy** 在非 ECS 场景下直接使用 Rust 标准库的 `HashMap<K, V>`。Rust 的 `HashMap` 默认使用 SipHash（防 HashDoS），但性能不如 FNV 或 xxHash。Bevy 在某些性能敏感路径使用 `ahash` crate。在 ECS 核心层，Bevy 几乎不需要通用 HashMap——Entity→组件映射用 SparseSet，Archetype 查找用专用索引。
>
> **个人项目推荐**：默认实现开放寻址法 + 线性探测的 HashMap。哈希函数用 FNV-1a（简单快速，对游戏数据分布足够好）。容量保持为 2 的幂次，用位掩码代替取模。如果需要稳定句柄（如资源引用），在 HashMap 之上封装一个 Handle 层，或在 Value 中存一个 generation 字段用于校验。

### 问题 1：临时内存每帧产生、每帧丢弃，如何避免反复向操作系统申请？

#### 最 naive 的方案

每帧用 `std::vector<TempData> temp; temp.clear();`，以为 `clear()` 会释放内存。实际上它只是把 `size` 设为 0，`capacity` 纹丝不动——下一帧如果数据量变少，内存就浪费了；变多了，又会触发 `realloc` 拷贝。

#### 改进：FrameArena（帧分配器）

**核心思想**：一帧内所有临时内存共享一块大缓冲区，用一个指针 `offset` 往前推。帧结束时 `offset = 0`，O(1) 全部回收。

```cpp
struct FrameArena {
    static constexpr usize CAPACITY = 1024 * 1024; // 1 MiB
    u8* base = nullptr;
    usize offset = 0;

    void Init() { base = static_cast<u8*>(std::malloc(CAPACITY)); offset = 0; }
    void Shutdown() { std::free(base); base = nullptr; }

    void* Alloc(usize size, usize align = 8) {
        usize mask = align - 1;
        usize padded = (offset + mask) & ~mask;
        if (padded + size > CAPACITY) return nullptr; // 溢出，无回退
        offset = padded + size;
        return base + padded;
    }

    void Reset() { offset = 0; } // 每帧调用一次，O(1) 回收
};
```

**状态变化图**：`Alloc(24, 16)` 执行前后

```
分配前：
base ──► [..............................]
         ^offset=8

对齐后：
base ──► [........][....................]
                  ^padded=16

分配后：
base ──► [........][####################][............................]
                  ^返回的指针            ^offset=40
                  (24 bytes 已分配)
```

> **这段代码已经解决了什么**：O(1) 分配、O(1) 批量回收、零外部碎片、无系统调用（除 Init 时的一次 `malloc`）。
> **还缺什么**：没有多帧轮转（当前帧分配的内存不能异步被消费线程读取）、没有线程安全（需 TLS 或单生产者约束）、没有向系统分配器的动态扩容回退、**对齐浪费可能高达 `align - 1` 字节**。
> **反例警告**：如果一帧内临时分配总量超过 `CAPACITY`（如复杂粒子系统或大规模 ECS 查询），`Alloc` 会直接返回 `nullptr`。FrameArena 只适用于**内存总量可预估**的场景。

#### 工业级演进：从单帧 Arena 到多帧轮转与嵌套作用域

FrameArena 的最小版本够用，但生产环境会遇到两个新问题：

**问题 A：渲染线程需要消费游戏线程上一帧产生的数据**
如果游戏线程每帧 `Reset()`，渲染线程可能还在读上一帧的临时数据。解决方式是**双缓冲/三缓冲 Arena**：维护 `N` 个 `FrameArena`，游戏线程写入 Arena[i]，渲染线程读取 Arena[(i-1) mod N]。每帧只 `Reset` 最旧的那个。

**问题 B：子系统之间的临时内存互相污染**
如果所有临时内存都从一个全局 Arena 分配，物理系统的临时数组和渲染系统的临时命令混在一起，`Reset` 时无法只回收其中一部分。解决方式是**作用域标记（FMemMark）**：在 Arena 中压入一个标记点，子系统结束后把 `offset` 回滚到标记点，而非全局 `Reset`。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「临时内存的生命周期管理」这个子问题。
> - **UE** 在 RenderCore 层提供了 `FMemStackBase`（64KB 页线性栈），渲染线程用它分配 RenderGraph 的临时资源描述符。游戏线程则用 `FMalloc` 家族的通用分配器。UE 的 `FMemStack` 支持 `FMemMark` 嵌套回滚——你可以在子函数中分配临时内存，函数返回时自动回滚到进入时的偏移量。
> - **Bevy** 的 ECS 系统中，System 的临时查询结果（如 `Query<&Transform>` 返回的迭代器）不分配堆内存，而是直接引用 Archetype Table 的连续列数据。Bevy 的 `Commands` 缓冲区虽然需要临时存储，但它是按 System 在帧末统一 flush 的，本质也是一种 Arena 模式。
>
> **个人项目推荐**：默认用单帧 `FrameArena` 处理同帧临时数据；如果涉及异步消费（如游戏线程生成命令 → 渲染线程执行），升级到双缓冲 Arena。嵌套作用域标记可以在需要时添加，不必一开始就做。

### 问题 2：固定大小对象频繁创建销毁（如粒子、网络包、Entity 句柄），如何避免 malloc 碎片和构造开销？

#### 最 naive 的方案

`new Particle()` / `delete ptr`。每次都要向全局堆申请，释放后留下大小不一的空洞。如果粒子每帧生成 1000 个、销毁 1000 个，全局堆很快变成蜂窝。

#### 改进：ObjectPool（固定大小对象池）

**核心思想**：预分配一大块固定大小的 Slot，用空闲链表（free list）管理未使用的 Slot。分配和释放都是 O(1) 的指针操作。

```cpp
template <typename T, usize BLOCK_COUNT>
struct ObjectPool {
    union Slot {
        T object;
        Slot* nextFree;
        Slot() {}
        ~Slot() {}
    };

    Slot* memory = nullptr;
    Slot* freeList = nullptr;

    void Init() {
        memory = static_cast<Slot*>(std::malloc(sizeof(Slot) * BLOCK_COUNT));
        for (usize i = 0; i < BLOCK_COUNT; ++i) {
            memory[i].nextFree = (i + 1 < BLOCK_COUNT) ? &memory[i + 1] : nullptr;
        }
        freeList = &memory[0];
    }

    void Shutdown() { std::free(memory); }

    T* Allocate() {
        if (!freeList) return nullptr;
        Slot* slot = freeList;
        freeList = freeList->nextFree;
        return new (slot) T();
    }

    void Free(T* obj) {
        if (!obj) return;
        obj->~T();
        Slot* slot = reinterpret_cast<Slot*>(obj);
        slot->nextFree = freeList;
        freeList = slot;
    }
};
```

**状态变化图**：分配和释放一个对象

```
Init 后：
memory: [Slot0] → [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList

Allocate() 后：
memory: [Slot0: T()]  [Slot1] → [Slot2] → [Slot3] → nullptr
                       ▲
                      freeList
                      (返回 &Slot0.object)

Free(&Slot0) 后：
memory: [Slot0] ─┐
                 │
         ┌───────┘
         ▼
        [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList
```

> **这段代码已经解决了什么**：O(1) 分配/释放、零外部碎片、固定大小对象生命周期完全可控、构造/析构显式调用。
> **还缺什么**：没有跨池释放校验（把不是本池的对象 `Free` 进来会崩溃）、没有内存对齐保证（`alignof(T)` 可能大于默认 `malloc` 对齐）、没有动态扩容、**对象在内存中的地址永远只在预分配的 `memory` 块内，不支持运行时增长**。
> **反例警告**：如果对象大小差异很大（如 16 字节的粒子和 256 字节的 NPC），为每种大小各建一个 ObjectPool 会导致元数据（free list 指针）开销比例不同。ObjectPool 只适用于**对象大小固定且数量可预估**的场景。

#### 工业级演进：分代池与动态扩容

固定 ObjectPool 很快会遇到容量天花板。工业级引擎通常引入两层改进：

**分代索引（Generational Index）**：把裸指针返回给用户是危险的——如果 Slot 被释放后又重新分配，旧指针就变成了悬垂指针。解决方式是返回 `Handle { slotIndex, generation }`，每次复用 Slot 时 `generation++`。访问时校验 `generation` 是否匹配，不匹配则说明是旧指针。

```cpp
struct GenerationalHandle {
    u32 slotIndex;
    u32 generation;
};

struct GenerationalPool {
    struct Slot { T object; u32 generation; u32 nextFree; };
    std::vector<Slot> slots;
    u32 freeListHead = INVALID;
    // Allocate 返回 Handle；访问时通过 handle.slotIndex 取数据并校验 generation
};
```

**动态分块**：不要预分配一个固定的大数组，而是分配多个固定大小的 Block（如每块 1024 个 Slot）。`freeList` 跨 Block 链接。当所有 Block 都满时，追加新 Block。

> **参考：UE / Bevy / 前沿是怎么做的？**
>
> 我们在解决的是「固定大小对象的高频分配与悬垂指针安全」这个子问题。
> - **UE** 的 `FName` 系统本质上是一个巨型字符串对象池。所有字符串实际存储在全局的 `FNamePool` 中，按长度分块（Short / Medium / Large）。`FName` 本身只存一个整数索引，比较两个 `FName` 只需比较两个 `uint32`（O(1)）。这是对象池思想在字符串上的极致应用——把"比较字符串"从 O(n) 降为 O(1)。
> - **UE** 的 `TSparseArray` 是另一种固定大小对象池的变体：它维护一个 `dense` 数组存储实际元素，同时用 `freeIndices` 链表管理空洞。删除元素时只标记为空洞，不移动后续元素，因此其他元素的索引（句柄）不会失效。这对需要稳定 ID 的游戏对象集合非常有价值。
> - **Bevy** 的 `Entity` 分配器也采用分代索引：Entity 是一个 `(index, generation)` 对。当 Entity 被销毁后，它的 `index` 进入空闲列表，但 `generation` 递增。旧句柄的 `generation` 不匹配，访问时会被拒绝。
> - **前沿研究**：2025 年 Cox 等人对 Archetype ECS 与 SparseSet ECS 的对比研究表明，在 50,000 实体规模下，实体创建/销毁的瞬时开销是帧时间波动的主要来源之一。使用对象池预分配可将这部分开销从"每帧抖动"变为"启动时一次性成本"。
>
> **个人项目推荐**：默认用动态分块 + 分代索引的 ObjectPool。如果对象数量有严格上限（如音频通道固定 32 个），可以用固定池。`FName` 式的字符串池在游戏中有大量资源路径比较时是刚需，但个人项目初期可以用 `std::string_view` + 哈希表模拟。

### 问题 3：如何在 O(1) 下判断"某个 Entity 是否拥有某个组件"，同时保证遍历所有拥有该组件的 Entity 时 cache 友好？

#### 最 naive 的方案

用 `std::unordered_map<EntityID, ComponentT>`。查找是 O(1)，但遍历所有拥有该组件的 Entity 时，要遍历整个 map——不仅每个元素都可能有独立的内存分配（cache 灾难），而且你还不得不遍历那些"没有该组件"的 Entity 的空洞。

#### 改进：SparseSet（稀疏集）

**核心思想**：用两个数组实现 O(1) 增删和密集遍历，是 ECS 组件存储的最小骨架。

- `sparse[entityID]`：存储该 Entity 在 `dense` 数组中的索引（或空标记）。
- `dense[count]`：存储实际存在的 Entity ID，保证所有有效元素在数组前端连续排列。

```cpp
struct SparseSet {
    static constexpr u32 MAX_ENTITIES = 1024;
    static constexpr u32 INVALID = 0xFFFFFFFF;

    u32 sparse[MAX_ENTITIES];
    u32 dense[MAX_ENTITIES];
    u32 count = 0;

    SparseSet() {
        for (u32 i = 0; i < MAX_ENTITIES; ++i) sparse[i] = INVALID;
    }

    bool Has(u32 entity) const {
        if (entity >= MAX_ENTITIES) return false;
        u32 idx = sparse[entity];
        return idx != INVALID && idx < count && dense[idx] == entity;
    }

    void Add(u32 entity) {
        if (entity >= MAX_ENTITIES || Has(entity)) return;
        sparse[entity] = count;
        dense[count] = entity;
        ++count;
    }

    void Remove(u32 entity) {
        if (entity >= MAX_ENTITIES || !Has(entity)) return;
        u32 idx = sparse[entity];
        u32 lastEntity = dense[count - 1];
        dense[idx] = lastEntity;
        sparse[lastEntity] = idx;
        sparse[entity] = INVALID;
        --count;
    }

    const u32* Data() const { return dense; }
    u32 Count() const { return count; }
};
```

**状态变化图**：Add(Entity 5) → Add(Entity 2) → Remove(Entity 5)

```
初始状态：
sparse: [FF, FF, FF, FF, FF, FF, ...]
dense:  []
count:  0

Add(5):
sparse: [FF, FF, FF, FF, FF, 0,  FF, ...]
                             ^entity 5 的索引
dense:  [5]
count:  1

Add(2):
sparse: [FF, FF, 1,  FF, FF, 0,  FF, ...]
                   ^entity 2 的索引
dense:  [5, 2]
count:  2

Remove(5) — swap-and-pop 保持密集：
sparse: [FF, FF, 1,  FF, FF, FF, FF, ...]
                             ^entity 5 被标记为无效
dense:  [2]          (原末尾元素 2 被 swap 到索引 0)
count:  1
```

> **这段代码已经解决了什么**：O(1) 增删、O(count) 密集遍历（无空位检查）、极其缓存友好。`dense` 数组的连续性让批量处理组件时几乎没有 cache miss。
> **还缺什么**：`sparse` 数组的大小是 `MAX_ENTITIES`，Entity ID 范围很大时会浪费内存；没有与组件数据（ComponentArray）的联合存储。
> **反例警告**：如果 Entity ID 空间是 32 位（4294967296），直接开 `sparse` 数组会吃掉 16 GB 内存。工业级实现会用**分页稀疏数组**（Sparse Array of Pages）或**哈希表**回退。

#### 工业级演进：分页稀疏数组与组件数据联合存储

**分页稀疏数组**（Component Page）把 `sparse` 数组从线性大数组改为页表结构：

```
线性稀疏数组（浪费）：
sparse[4,294,967,296]  → 16 GB（不可接受）

分页稀疏数组：
pageTable[4096] → 每个 entry 指向一个 page（大小 1024）
                  或 nullptr（该页无实体）

EntityID = 0x00100080:
  pageIndex = 0x0010 = 16
  offset    = 0x0080 = 128
  sparse = pageTable[16][128]

实际内存：只有被使用过的页才会分配（4096 pages × 1024 uint32 ≈ 16 MB 全部分配，
          但实际按需分配，未使用页为 nullptr）。
```

**组件数据联合存储**：`SparseSet` 只解决了"哪些 Entity 拥有组件 T"的索引问题，但组件数据本身（如 `Transform` 结构体）还需要另一个数组。工业级做法是把 `dense` 数组和 `T components[]` 数组严格同步——`dense[i]` 对应的组件数据就是 `components[i]`。删除时用 `swap-and-pop` 同时更新两个数组。

> **参考：UE / Bevy / 前沿是怎么做的？**
>
> 我们在解决的是「O(1) 组件索引 + 密集遍历 + 大 Entity 空间内存效率」这个子问题。
> - **UE** 的 `TSet` 内部基于开放寻址法的哈希表，但有趣的是它用 `TSparseArray` 作为底层存储。`TSparseArray` 本质上就是一种 SparseSet 的变体——它维护 `dense` 数组存储实际元素，同时用 `freeIndices` 管理空洞，提供稳定的句柄（`FSparseArrayIndex`）。这与经典 SparseSet 的区别在于：UE 的稀疏数组更关注"稳定句柄"而非"Entity ID 映射"。
> - **Bevy** 的 `ComponentSparseSet` 采用经典三元结构：`sparse`（`Vec<Option<V>>`）+ `dense`（`Column`，紧凑组件数据）+ `entities`（记录每个 dense 槽位对应的 Entity）。Bevy 的分页不是通过显式 pageTable 实现的，而是依赖 Rust 的 `Vec` 本身的动态扩容。但 Bevy 的 `SparseArray` 在 `debug_assertions` 关闭时只存 `EntityIndex`（而非完整 Entity），节省了 generation 字段的内存。
> - **前沿（EnTT）**：EnTT 的 sparse set 实现被公认为工业级标杆。它的优化在于：sparse 数组不存 `uint32`，而是存 `entity_type`（紧凑打包）；dense 数组和组件数组严格分离但同步，支持"组（Group）"机制把多个组件的 sparse set 部分重叠，实现多组件查询时的 cache 友好。
> - **2025 年 Cox 等人论文**：在 10,000 实体以上规模，基于 Archetype（Table）的存储迭代速度几乎是纯 SparseSet 的两倍，但 SparseSet 的实体创建/销毁速度快 6.6 倍。这直接催生了「双轨存储」思想——用 Table 做默认存储，用 SparseSet 做高频变更组件的存储。
>
> **个人项目推荐**：默认用分页稀疏数组作为组件索引骨架（Entity 空间扩展到 32 位而不浪费内存）。组件数据与 `dense` 数组严格同步存储。如果你的 ECS 支持混合存储策略（如 Bevy），把频繁增删的标记组件放入 SparseSet，把批量迭代的主数据放入 Table（Archetype）。

### 问题 4：批量运算时，如何消除无关字段对 cache line 的污染？

#### 最 naive 的方案

`struct Particle { Vec3 pos; Vec3 vel; float life; }`，然后用 `std::vector<Particle>` 存储。更新位置时只读 `pos` 和 `vel`，但 `life` 和其他字段被一起加载到 cache line 中——64 字节的 cache line 只利用了 24 字节，其余都是"搭便车"的无效数据。

#### 改进：SoAVector（结构体数组 → 数组的结构体）

**核心思想**：把 `Particle` 拆成三个平行数组。批量运算时只读取需要的字段，消除无关字段的 cache line 污染。

```cpp
template <usize MAX_COUNT>
struct SoAParticles {
    f32 posX[MAX_COUNT];
    f32 posY[MAX_COUNT];
    f32 posZ[MAX_COUNT];
    f32 velX[MAX_COUNT];
    f32 velY[MAX_COUNT];
    f32 velZ[MAX_COUNT];
    f32 life[MAX_COUNT];
    usize count = 0;

    void Add(f32 x, f32 y, f32 z, f32 vx, f32 vy, f32 vz, f32 l) {
        if (count >= MAX_COUNT) return;
        posX[count] = x; posY[count] = y; posZ[count] = z;
        velX[count] = vx; velY[count] = vy; velZ[count] = vz;
        life[count] = l;
        ++count;
    }

    void UpdatePositions(f32 dt) {
        for (usize i = 0; i < count; ++i) {
            posX[i] += velX[i] * dt;
            posY[i] += velY[i] * dt;
            posZ[i] += velZ[i] * dt;
        }
    }
};
```

**内存布局对比**：AoS vs SoA（假设 `MAX_COUNT = 4`，每个 float 4 字节）

```
AoS (Array of Structs):
[ posX, posY, posZ, velX, velY, velZ, life ]  ← 32 bytes per particle
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]

如果只想更新 posX：
读取 cache line 时，vel/life 等无关数据被一起加载（假设 cache line 64 bytes，
每条 cache line 容纳 2 个粒子，但只用了 1 个 float）。

SoA (Structure of Arrays):
posX: [ x0, x1, x2, x3 ]  ← 16 bytes, 紧凑连续
posY: [ y0, y1, y2, y3 ]
posZ: [ z0, z1, z2, z3 ]
velX: [ vx0, vx1, vx2, vx3 ]
...

更新 posX 时：
16 bytes 完全有效，一条 cache line（64 bytes）可装 16 个 float，
即 16 个粒子的 posX。遍历效率显著提升。
```

> **这段代码已经解决了什么**：批量 SIMD 运算的天然友好布局、只读部分字段时 cache 利用率最大化。
> **还缺什么**：随机访问单个粒子的全部字段需要 7 次指针跳转（`posX[i]`, `posY[i]`...）、代码冗长、删除中间元素需要维护所有数组的同步。
> **反例警告**：如果系统需要频繁"读取一个粒子的全部字段"（如单个粒子与玩家的碰撞检测），SoA 会导致 7 次不连续的内存读取，反而比 AoS 慢。**SoA 是为批量同字段运算优化的，不是为随机全字段访问优化的**。

#### 工业级演进：类型擦除的列存储与 Archetype Table

手写 `SoAParticles` 在组件类型多的时候会变成代码灾难。工业级 ECS 引擎的做法是**类型擦除的列存储（Type-Erased Column Storage）**：

```cpp
struct Column {
    void* data;           // 原始字节数组，通过 componentSize 索引
    usize elementSize;
    usize count;
    usize capacity;
    // 附加元数据：added_tick, changed_tick 等
};

struct Table {
    std::vector<EntityID> entities;  // 行号 → 实体
    std::unordered_map<ComponentTypeID, Column> columns;  // 组件类型 → 列
};
```

同一 Table 中的所有实体拥有完全相同的组件组合（这就是 Archetype）。系统查询 `Query<Transform, Mesh>` 时，只需找到所有包含这两个组件的 Table，然后顺序读取对应的 `Column`——这就是 SoA 的工业级自动化版本。

**Bevy 的 Column 设计更进一步**：每个 `Column` 不仅存 `data: BlobArray`（类型擦除），还附带 `added_ticks` 和 `changed_ticks` 两个旁路数组。这意味着 Change Detection（变更检测）不需要额外的外部数据结构，而是直接在 Column 旁边完成，保证了 cache 局部性。

> **参考：UE / Bevy / 前沿是怎么做的？**
>
> 我们在解决的是「批量同字段运算的 cache 效率」这个子问题。
> - **UE** 的 `TArray` 本身是 AoS（因为它存储的是 `T` 对象数组），但 UE 在渲染层大量使用 SoA。例如 GPU 粒子系统（Niagara）的模拟数据就是严格的 SoA 布局，以便 SIMD 和 GPU 并行。UE 的 `TMap` / `TSet` 虽然哈希表结构不是 SoA，但它们的 `Elements` 数组（`TSparseArray`）内部是密集的。
> - **Bevy** 的 `Table` 是 SoA 的极致实现：每个 Archetype 对应一个 `Table`，Table 内按 `ComponentId` 分 `Column`。`Column` 用 `BlobArray` 做类型擦除（存储原始字节），用 `ThinArrayPtr` 剥离长度/容量字段（由外部 `Table` 统一管理）。这种"瘦指针 + 外部管理"的设计减少了每个 Column 的元数据开销。
> - **前沿（Unity DOTS / Flecs）**：Unity DOTS 的 Chunk 存储是 SoA 的变体——同一 Archetype 的实体按固定大小 Chunk（如 16KB）分块存储，每个 Chunk 内是 SoA 布局。Flecs 的 Table 也采用类似设计，但支持更灵活的分页策略。
> - **2025 年论文**：Archetype-based SoA 在 Tower Defense 模拟中比 OOP 和纯 AoS 实现实现了更高的帧稳定性，因为 cache miss 大幅降低。
>
> **个人项目推荐**：手写 SoA 在组件类型少（< 5 种）时是可行的。一旦进入 ECS 阶段，应直接实现类型擦除的 Column Storage（Table）。不要试图用 C++ 模板为每种组件组合生成特化代码——那会导致代码膨胀和编译时间爆炸。Bevy 的 `BlobArray` + `ThinArrayPtr` 模式是个人项目最优雅的参考。

### 问题 5：两个线程之间如何传递数据，既能避免锁竞争，又不引入复杂的同步机制？

#### 最 naive 的方案

`std::queue<T> + std::mutex`。生产者线程 `lock` → `push` → `unlock`，消费者线程 `lock` → `pop` → `unlock`。如果生产者优先级低、消费者优先级高（如音频线程），低优先级线程持有锁时可能被高优先级线程抢占，导致高优先级线程忙等——这就是**优先级反转**。

#### 改进：RingBuffer（无锁环形队列，SPSC）

**核心思想**：用两个原子索引 `head` / `tail` 和一块固定数组，实现单生产者-单消费者（SPSC）的零锁通信。

```cpp
#include <atomic>

template <typename T, usize SIZE>
struct RingBuffer {
    static_assert((SIZE & (SIZE - 1)) == 0, "SIZE must be power of 2");
    T buffer[SIZE];
    std::atomic<usize> head{0};
    std::atomic<usize> tail{0};

    bool Push(const T& item) {
        usize t = tail.load(std::memory_order_relaxed);
        usize next = (t + 1) & (SIZE - 1);
        if (next == head.load(std::memory_order_acquire)) return false; // 满
        buffer[t] = item;
        tail.store(next, std::memory_order_release);
        return true;
    }

    bool Pop(T& item) {
        usize h = head.load(std::memory_order_relaxed);
        if (h == tail.load(std::memory_order_acquire)) return false; // 空
        item = buffer[h];
        head.store((h + 1) & (SIZE - 1), std::memory_order_release);
        return true;
    }
};
```

**状态变化图**：Push A → Push B → Pop A（SIZE = 4）

```
初始状态：
[ _, _, _, _ ]
 h=0
 t=0

Push(A):
[ A, _, _, _ ]
 h=0
 t=1

Push(B):
[ A, B, _, _ ]
 h=0
 t=2

Pop() → A:
[ A, B, _, _ ]   (A 仍在内存中，但逻辑上已消费)
    h=1
 t=2
```

> **这段代码已经解决了什么**：单生产者-单消费者（SPSC）场景下的零锁通信、严格的 acquire/release 内存序保证可见性、无系统调用。
> **还缺什么**：仅支持 SPSC；MPSC/MPMC 需要更复杂的原子 CAS 或 seqlock；没有 ABA 防护（大小为 2 的幂次配合索引环绕可天然缓解，但 64 位索引溢出问题在极长运行时间后理论上存在）；**容量固定为 2 的幂次**。
> **反例警告**：如果在生产者线程中 Push 的数据大小不固定（如变长字符串），RingBuffer 无法直接使用。它只适用于**固定大小消息**的传递。多生产者场景下不加锁直接使用此代码会导致数据竞争。

#### 工业级演进：从 SPSC 到 MPMC 与命令队列

SPSC RingBuffer 够用，但游戏引擎中常见的生产者-消费者关系更复杂：

- **多生产者-单消费者（MPSC）**：Gameplay 系统在多线程中并行生成渲染命令，最终由渲染线程消费。解决方式是每个生产者有自己的 SPSC RingBuffer，消费者轮询所有 RingBuffer（"多路复用"）。这比真正的 MPMC 无锁队列简单得多，且避免了 CAS 竞争。
- **命令队列（Command Queue）**：ECS 中 System 不能直接在并行执行时修改 World（会导致数据竞争）。Bevy 的 `Commands` 本质就是一个线程本地的 RingBuffer/Vec，System 把"我要插入组件 A"、"我要销毁实体 B"等命令写入本地缓冲区，帧末由主线程串行 flush。这种"本地累积 + 批量提交"的模式是 RingBuffer 思想在 ECS 中的最高频应用。

> **参考：UE / Bevy / 前沿是怎么做的？**
>
> 我们在解决的是「跨线程零锁通信」这个子问题。
> - **UE** 的 `FMallocBinned2/3` 在分配器层面大量使用类似 RingBuffer 的思想：每个线程维护 `FPerThreadFreeBlockLists`（线程本地空闲块链表）。`Malloc` 和 `Free` 在热路径上是完全无锁的——直接从本地链表 Pop/Push。只有当本地链表满/空时，才需要加锁与全局回收器交互。这本质上是把"线程间的内存竞争"转化为"线程本地的栈操作"。
> - **Bevy** 的 `Event` 系统内部使用 RingBuffer 存储事件。`Commands` 虽然名义上是 Vec，但在并行 System 中每个线程写自己的命令缓冲区，最后合并——这与 MPSC 多路复用的思想一致。
> - **前沿（mimalloc）**：mimalloc 被 UE 5.5+ 采纳为默认分配器，其核心设计之一是"free list sharding"——每个 mimalloc page（64KB）内部有多个 free list（一个用于线程本地释放，一个用于跨线程释放）。跨线程释放只需一次 CAS，无需全局锁。这与 RingBuffer 的"把竞争分散到多个小单元"思想异曲同工。
>
> **个人项目推荐**：默认用 SPSC RingBuffer。如果需要 MPSC，优先用"每生产者一个 SPSC + 消费者轮询"的方案，不要直接写 MPMC 无锁队列（极易出错）。ECS 的命令队列模式（本地累积 + 批量提交）是应对多线程结构变更的最佳实践。

### 问题 6：通用对象的分配如何做到严格 O(1) 和确定性，同时避免全局堆碎片？

FrameArena 和 ObjectPool 都是专用分配器——它们只解决特定场景。但引擎中还有大量"通用小对象"（如网络包、AI 决策节点、UI 元素），它们的大小不固定、生命周期不统一。对于这些对象，我们需要一个通用分配器。

#### 最 naive 的方案

直接用 `malloc` / `free`。问题在问题 0 中已经说过了：时延不可预测、碎片、多线程竞争。

#### 改进：Binned Allocator（分箱分配器）

**核心思想**：把小块内存按固定大小分箱（如 8, 16, 24, 32, ... 直到 32KB），每个箱子是一个 ObjectPool。分配时把请求大小向上取整到最近的箱大小，从对应箱子里取一个空闲块。释放时把块归还到对应箱子。

```cpp
// 概念性伪代码
struct BinnedAllocator {
    static constexpr usize NUM_BINS = 32;
    static constexpr usize MAX_SMALL_SIZE = 32768;
    ObjectPool<void, /*per-bin-count*/> bins[NUM_BINS];

    void* Malloc(usize size) {
        if (size > MAX_SMALL_SIZE) return OSMalloc(size);
        uint32 binIndex = SizeToBinIndex(size); // O(1) 查表
        return bins[binIndex].Allocate();
    }
    void Free(void* ptr) {
        uint32 binIndex = PtrToBinIndex(ptr); // 通过地址反查
        bins[binIndex].Free(ptr);
    }
};
```

> **这段代码已经解决了什么**：小对象分配 O(1)、零外部碎片（同箱内块大小相同）、无需合并（释放直接归还箱子）。
> **还缺什么**：没有考虑多线程竞争（多个线程同时从同一个箱子分配仍需加锁）、没有 TLS 缓存（每次分配都要查表 + 箱子操作，cache 局部性一般）、大对象直接走 OS 分配。

#### 工业级演进：TLS 缓存 + 指针元数据外置 + Trim Epoch

**TLS 线程缓存**：每个工作线程维护独立的空闲块缓存数组。`Malloc` 时优先从本地缓存取，`Free` 时优先归还本地缓存。只有当本地缓存满/空时，才与全局分配器交互。热路径完全无锁。

**指针元数据外置**：释放时需要通过指针反查它属于哪个 Bin。不要给每个 Block 加头部信息（浪费内存），而是通过页对齐约定 + 哈希表反查：所有 Bin 的内存都从 OS 按页（如 64KB）申请，根据指针的页地址即可定位到 Bin 索引。

**Trim Epoch**：当工作线程进入空闲（如切后台）时，分配器需要回收该线程缓存中的内存。用单调递增的 epoch 标记哪些线程缓存需要回收，避免无差别扫描所有线程。

> **参考：UE / 前沿是怎么做的？**
>
> 我们在解决的是「通用小对象的确定性分配」这个子问题。
> - **UE** 的 `FMallocBinned2/3` 是工业级分箱分配器的标杆。`Binned2` 用 64KB 大页对齐，`Binned3` 每 Bin 独立预占 1GB 虚拟地址池并用位树管理 Block。两者都通过 `FPtrToPoolMapping` 用哈希 + 位运算反查指针所属 Bin，Block 头部零开销。UE 5.5+ 已将默认分配器切换为 **Mimalloc**，因为 Mimalloc 的 segment-based free list 在广泛基准测试中表现更稳定。
> - **前沿（Mimalloc）**：Mimalloc 的核心创新是"free list sharding"——每个 64KB page 内部有多个 free list（本地释放列表、跨线程释放列表）。跨线程释放只需一次 CAS，竞争被自然分散到成千上万个 page 中。在 2026 年的基准测试中，Mimalloc 在小对象高频分配场景下的 P99 延迟比 jemalloc 低 15%，比 tcmalloc 低 22%。UE 的切换决策验证了这一点。
> - **TLSF（Two-Level Segregated Fit）**：如果需要**严格 O(1)**（不是"平均 O(1)"，而是"最坏情况 O(1)"），TLSF 是另一个选择。它用两级位图索引空闲块，分配和释放都是严格常数时间。代价是位图和分割/合并逻辑引入额外元数据开销，内存利用率比 `malloc` 低 5%~15%。
>
> **个人项目推荐**：
> - **默认方案**：直接用 **Mimalloc** 作为全局分配器（单文件 `static.c` 即可嵌入）。它同时满足"工业级性能"和"极简集成"，UE 的选择已经替你验证过了。
> - **如果你需要完全自研**（如嵌入式平台或学习目的）：实现一个简化版 Binned Allocator + TLS 缓存。不要一开始就写 TLSF——它的位图维护复杂度远高于收益。Binned + TLS 缓存已经能让你的分配器在 99% 场景下媲美系统分配器。
> - **极端实时场景**（音频线程、网络线程）：如果 Mimalloc 的延迟仍然不可接受，退回到专用分配器（ObjectPool、RingBuffer、FrameArena）。通用分配器永远不如专用分配器快。

### 问题 7：ECS 组件存储该用 Archetype（Table）还是 SparseSet？如果两者都有优劣，能不能都要？

这是进入 ECS 阶段后最关键的设计抉择。前面的 SparseSet 和 SoA 分别给出了两种存储模型的最小实现，但它们适用于不同场景。

#### 两种模型的根本差异

| 维度        | Archetype（Table）                          | SparseSet                                      |
| --------- | ----------------------------------------- | ---------------------------------------------- |
| **内存布局**  | 同一 Archetype 的实体，组件数据在 Table 中按列连续存储（SoA） | 每个组件类型独立的 `sparse+dense` 结构，组件数据紧凑但不同组件类型之间不连续 |
| **迭代性能**  | 极佳：多组件查询时，只需顺序遍历 Table 的行，cache 命中率高      | 一般：多组件查询需要求多个 SparseSet 的交集，随机访问多              |
| **插入/移除** | 需要跨 Table 迁移数据（`swap_remove` + 复制），O(组件数) | O(1)，仅更新 sparse/dense 索引                       |
| **随机访问**  | 通过 `TableRow` 直接索引                        | 通过 `sparse` 数组两次索引                             |
| **适用场景**  | 大多数组件（Transform, Mesh, PhysicsBody 等）     | 稀有组件、频繁变更的标记组件（如 `IsHovered`、`DebugDraw`）      |

#### 2025 年的 benchmark 结论

Cox 等人在 2025 年的论文中对比了两种模型（使用 C++20 实现，Conway's Game of Life 作为负载）：

- **10,000 实体以下**：两者性能没有显著差异。
- **10,000 实体以上**：Archetype 的迭代速度几乎是 SparseSet 的两倍（50,000 实体时，Archetype 中位数 7.41ms vs SparseSet 13.82ms）。
- **实体创建/销毁**：SparseSet 比 Archetype 快 **6.6 倍**。

结论是：**没有绝对的最优，只有场景依赖的权衡**。

#### 改进：双轨混合存储（Hybrid Storage）

Bevy ECS v2 给出的答案是：**默认用 Table，可选用 SparseSet**。开发者为每种组件选择存储策略：

```cpp
// 概念性 C++ 模拟
enum class StorageType {
    Table,      // 默认：批量迭代快，增删慢
    SparseSet   // 可选：增删快，迭代慢
};

template <typename T>
struct ComponentDescriptor {
    static constexpr StorageType storage = StorageType::Table;
};

// 特例化：频繁增删的标记组件用 SparseSet
template <>
struct ComponentDescriptor<DebugDraw> {
    static constexpr StorageType storage = StorageType::SparseSet;
};
```

**Archetype 图缓存（Edges）**：Archetype 的致命弱点是"实体组件组合变更时需要迁移数据"。如果每次插入组件都重新计算"当前 Archetype + 新组件 = 哪个目标 Archetype"，开销很大。解决方式是缓存：每个 Archetype 维护一个 `Edges` 表，记录"插入 Bundle B 后到达哪个 Archetype"。这是 O(1) 的缓存查找，把结构变更的开销从"计算组合"降为"查表 + 数据复制"。

**多个 Archetype 共享同一个 Table**：如果两个 Archetype 的 Table 组件完全相同，但 SparseSet 组件不同，它们可以共享同一个 Table。这减少了 Table 的数量和内存碎片。

> **参考：Bevy / Flecs / 前沿是怎么做的？**
>
> 我们在解决的是「ECS 存储模型的选择」这个子问题。
> - **Bevy** 的 `Storages` 结构明确区分了 `tables` 和 `sparse_sets`。`Archetype` 不直接存数据，只存元数据（`table_id` + `edges` + `entities`）。`Table` 是列式存储，`ComponentSparseSet` 是三元稀疏集。`move_row` 跨表迁移时，Bevy 使用**双指针合并**（类似归并排序的合并步骤）按 `ComponentId` 排序匹配两表的列，O(n) 完成数据迁移。
> - **Flecs** 也是 Archetype-based，但它的 Table 支持更灵活的分页和继承机制。Flecs 的查询在多组件场景下比 EnTT 的 SparseSet 更快，但 EnTT 在单组件查询和实体增删上更快。
> - **Unity DOTS** 的 Chunk-based 存储是 Archetype 的变体：同一 Archetype 的实体按固定大小 Chunk（16KB）分块。Chunk 内是 SoA 布局，但 Chunk 之间不连续。这增加了少许碎片，但让内存分配更可控（固定大小 Chunk 可以用 ObjectPool 管理）。
> - **2025 年论文**：作者明确建议——如果应用场景中实体频繁变更组件组合（如动态环境、频繁生成销毁），SparseSet 更合适；如果场景中大量实体拥有稳定组件组合且主要负载是批量迭代（如 Tower Defense、大规模模拟），Archetype 更合适。**混合策略是两种模型优势的叠加**。
>
> **个人项目推荐**：
> - **默认采用 Archetype + Table 作为主力存储**，因为这是 ECS 批量迭代的甜蜜点。
> - **为标记组件（Tag Component）和频繁增删的组件提供 SparseSet 选项**。标记组件（如 `IsPlayer`、`IsDirty`）没有数据payload，用 SparseSet 可以几乎零开销地增删。
> - **必须实现 Archetype 图缓存（Edges）**。没有 Edges 的 Archetype ECS 在结构变更频繁时性能会急剧下降。以 Bundle（组件组合）为粒度缓存，而非单个组件——这能最大程度减少缓存条目数量。
> - **多个 Archetype 共享 Table** 是进阶优化，初期可以不做，但设计时要预留接口。

### 问题 8：专用容器多了之后，如何调试内存越界和 use-after-free，又不让调试代码污染最终构建？

#### 最 naive 的方案

在每次分配时多申请一些内存，填入魔数（Canary），释放时检查。但如果把这个逻辑直接写进 `FrameArena::Alloc`，最终构建也会带着这些开销。

#### 改进：装饰器模式 + AOP 式 Hook

**核心思想**：把调试功能做成可插拔的"包装层"，包裹在真正的分配器外面。开发构建用包装层，最终构建直接剥掉。

```cpp
// 调试分配器包装层
struct DebugAllocatorWrapper {
    IAllocator* inner;

    void* Alloc(usize size, usize align) {
        usize total = size + 2 * CANARY_SIZE;
        u8* raw = static_cast<u8*>(inner->Alloc(total, align));
        // 头部 Canary
        std::memcpy(raw, &CANARY_MAGIC, CANARY_SIZE);
        // 尾部 Canary
        std::memcpy(raw + CANARY_SIZE + size, &CANARY_MAGIC, CANARY_SIZE);
        // 毒值填充用户区域
        std::memset(raw + CANARY_SIZE, 0xFE, size);
        return raw + CANARY_SIZE;
    }

    void Free(void* ptr) {
        u8* raw = static_cast<u8*>(ptr) - CANARY_SIZE;
        // 校验 Canary...
        inner->Free(raw);
    }
};
```

**调试机制清单**：

| 机制 | 作用 | 实现方式 |
|------|------|---------|
| **毒值填充** | 分配时写入 `0xFE`，检测 use-after-free | `memset(ptr, 0xFE, size)` |
| **Canary 越界检测** | 块前后插入魔数，释放时校验 | 头部/尾部各 8 字节 |
| **调用栈捕获** | 记录每次分配的调用栈，泄漏时可回溯 | 平台相关（如 `CaptureStackBackTrace`） |
| **分配统计** | 每个子系统的 OS 申请量、已分配量、峰值 | `AllocatorMetrics` 接口 |

> **参考：UE / 前沿是怎么做的？**
>
> 我们在解决的是「内存调试与诊断」这个子问题。
> - **UE** 的 `FMalloc` 家族是装饰器模式的教科书案例。`FMallocStomp` 在每分配一页时用 guard page 检测越界；`FMallocLeakDetection` 追踪分配释放配对；`FMallocThreadSafeProxy` 给非线程安全分配器加锁。这些装饰器通过 `GMalloc` 指针链式组合：开发构建可能是 `Stomp → LeakDetection → ThreadSafeProxy → Binned3`，而最终构建只有 `Binned3`。
> - **UE** 的 `FMemory` 全局入口还提供了 `QuantizeSize` 接口，让容器在扩容时知道"最优请求大小"。例如 `TArray` 扩容时不会按"当前大小 × 1.5"申请，而是按 `FMemory::QuantizeSize` 对齐到 Bin 大小，减少内部碎片。
> - **前沿（Mimalloc）**：Mimalloc 的 debug build 自动提供 detailed statistics、padding detection、double-free detection。它的 guarded mode 可以在生产环境中以采样方式放置 guard page（每 N 次分配放一个），性能损失接近零。
>
> **个人项目推荐**：
> - 实现一个 `IAllocator` 接口（`Alloc` / `Free` / `Realloc`），所有专用分配器（FrameArena、ObjectPool、BinnedAllocator）都实现它。
> - 实现一个 `DebugAllocatorWrapper` 装饰器，用宏控制在 Debug/Development 构建中启用，Final 构建中完全剥离。
> - 不要为每个分配器单独写调试逻辑——那是代码重复。通过接口统一包装。
> - `QuantizeSize` 是个人项目容易忽略但收益很高的优化：`TArray` 类的容器在扩容时调用 `allocator.QuantizeSize(requestedSize)`，而不是直接 `requestedSize`，能显著减少内部碎片。

## 性能与 trade-off 总结：诚实对比

| 容器/分配器 | 对比基准 | 什么场景更好 | 什么场景更差 | 核心代价 |
|------------|---------|-------------|-------------|---------|
| **Array<T, Allocator>** | `std::vector<T>` | 分配器参数化、增长策略可控、无异常 | 需要 STL 兼容接口（如迭代器层级） | 自研维护成本、不如 STL 成熟 |
| **HashMap<K, V>** | `std::unordered_map<K, V>` | 连续内存、cache 友好、分配器可控 | Key 哈希质量差时探测链退化 | 无标准删除操作（需 tombstone） |
| **FrameArena** | `std::vector` + 每帧 `clear`/`shrink` | 大量临时对象，生命周期同帧 | 跨帧存活、总量不可预估 | 无单独释放，总容量固定 |
| **ObjectPool** | `new`/`delete` | 固定大小对象频繁创建销毁 | 对象大小差异大、数量剧增 | 无动态扩容，地址不连续 |
| **SparseSet** | `std::unordered_set<uint32_t>` | 密集遍历 + O(1) 增删 | Entity 空间极大、稀疏分布 | `sparse` 数组预分配内存 |
| **SoAVector** | AoS `std::vector<Struct>` | 批量 SIMD 同字段运算 | 随机访问单个对象全部字段 | 删除需同步多数组，代码冗长 |
| **RingBuffer** | `std::queue` + `mutex` | SPSC 线程通信 | MPMC、变长消息 | 容量固定为 2 的幂 |
| **Binned + TLS** | `malloc` / `free` | 通用小对象、多线程 | 块大小极度分散、内存紧张 | 元数据开销、内部碎片 |
| **Mimalloc** | jemalloc / tcmalloc | 高频小对象、长运行服务 | 需要完全自研控制 | 外部依赖（但极轻量） |
| **Archetype Table** | SparseSet ECS | 大规模批量迭代、稳定组件组合 | 频繁结构变更、小规模项目 | 结构变更需数据迁移 |
| **SparseSet (ECS)** | Archetype Table | 频繁增删、标记组件 | 大规模多组件查询 | 多组件查询需集合求交 |

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 满足 | FrameArena 的 `offset` 只要分配顺序固定就是确定性的；ObjectPool 的块布局是严格平铺的数组；SparseSet 的 `dense` 数组是 ECS 批量处理的基础；Archetype Table 的 Column 是类型擦除的连续字节 |
| **自描述** | ✅ 满足 | `AllocatorMetrics` 接口暴露 `consumedMemoryBytes`、`allocationCount`、`peakUsage` 等指标；Archetype 的 `edges` 缓存和组件组合是 AI 可直接观测的结构化数据 |
| **确定性** | ✅ 满足 | FrameArena 的 `Reset()` 消除了释放顺序的不确定性；ObjectPool 消除了外部碎片；SparseSet 的 `swap-and-pop` 保证遍历顺序虽变但无空洞；Archetype 图缓存使结构变更路径确定 |
| **工具边界** | ⚠️ 需封装 | AI 通过 MCP 请求"分配 1024 字节临时内存"时，必须返回一个作用域句柄（如 `ScopedArenaAlloc`），禁止返回裸指针。ECS 的 `Commands` 模式天然适合 Agent 操作——Agent 写入命令，引擎在帧末统一执行 |
| **Agent 安全 / 沙箱隔离** | ✅ 满足 | 通过为不同子系统生成独立内存池（`PoolAI`、`PoolScript`）实现编译期隔离；Agent 脚本层的分配错误不会破坏核心引擎。ECS 的 System 并行调度 + 命令队列模式天然支持多 Agent 的隔离与合并 |

## 与 ECS / 反射的关联

- **`Array<T, Allocator>`** 是引擎通用代码的骨架。阶段 2 的 `ComponentArray` 用 `std::vector` 临时实现，现在可以回探替换为 `Array<Component, PoolAllocator>` 或 `Array<Component, DefaultAllocator>`。`Array` 的分配器参数化让上层代码可以按场景选择内存来源，而无需修改容器逻辑。
- **`HashMap<K, V>`** 为反射系统（类型名 → 工厂函数）、资源管理（路径 → 句柄）、配置表（ID → 参数）提供了统一的键值查找基础设施。它比 `std::unordered_map` 更可控，且与引擎的分配器体系一致。
- **SparseSet** 是 ECS `ComponentArray` 的索引骨架。`dense` 数组遍历得到的每个 Entity，再到 `components[count]` 中取值，即可完成组件系统的批量查询。分页稀疏数组让 Entity 空间扩展到 32 位。
- **SoAVector** 是 ECS Archetype Table 的简化版。理解 SoA 后，Archetype 的"按组件类型分块存储"就不再是黑魔法。Bevy 的 `BlobArray` + `ThinArrayPtr` 提供了类型擦除 SoA 的最小实现模板。
- **FrameArena** 的 O(1) 线性分配 + 每帧 `Reset` 为 ECS 的临时查询结果（如 `Query<Transform, Mesh>` 返回的实体列表）提供了零抖动的内存保障。Bevy 的 Query 迭代器不分配堆内存，本质上是 Arena 思想的体现。
- **ObjectPool** 的固定大小分配消除了外部碎片，确保 `Entity` 句柄、网络消息等对象在固定内存预算内稳定运行。Bevy 的 Entity 分配器采用分代索引，UE 的 `FName` 采用字符串池——都是对象池思想在不同场景下的变体。
- **RingBuffer** 的 SPSC 无锁设计可直接用于 System 调度中的命令缓冲提交（Bevy `Commands`），避免互斥锁导致的优先级反转。
- **Archetype + 双轨存储** 是现代 ECS 的最佳实践。Table 做默认存储（迭代快），SparseSet 做可选存储（增删快），Edges 缓存消除结构变更的计算开销。2025 年的 benchmark 已经验证：混合策略在 10,000+ 实体规模下综合最优。
- 当容器和分配器就位后，[[组件系统架构]] 中的 ECS Chunk 存储、[[反射系统]] 中的类型注册表，都可以定制专属的高性能数据结构，而不再被 STL 的默认行为绑架。

### 问题：容器和分配器这么多，怎么按场景组合才不会混乱？

现在我们已经有了 `Array`、`HashMap`、`FrameArena`、`ObjectPool`、`SparseSet`、`RingBuffer`……但它们不是孤立的工具。一个真实的引擎场景中，往往是**容器 + 分配器 + 使用模式**的三元组合。如果随意搭配，可能会得到反效果：

| 错误组合 | 后果 |
|---------|------|
| `Array<T, FrameArenaAllocator>` 存跨帧数据 | 帧末 `Reset()` 直接释放，悬垂指针 |
| `HashMap<K, V>` 存 ECS 组件索引 | Entity ID 是连续整数，HashMap 的哈希计算纯属浪费，应该用 SparseSet |
| `ObjectPool<T, 1024>` 存变长字符串 | 对象池要求固定大小，字符串长度不固定，越写越乱 |
| `SparseSet` 存通用键值对 | Key 不是连续整数时，`sparse` 数组浪费巨大 |

#### 组合决策图

```
需要按键查找？
  ├── 是 → Key 是连续整数（如 EntityID）？
  │         ├── 是 → SparseSet（O(1) 无哈希，密集遍历）
  │         └── 否 → HashMap<K, V>（开放寻址，通用键）
  └── 否 → 需要 O(1) 随机访问？
            ├── 是 → Array<T, Allocator>（动态数组）
            └── 否 → 需要 FIFO 线程通信？
                      ├── 是 → RingBuffer（SPSC 无锁）
                      └── 否 → 需要批量同字段运算？
                                ├── 是 → SoA / Archetype Table
                                └── 否 → 纯临时数据？
                                          ├── 是 → FrameArena
                                          └── 否 → 固定大小高频对象？
                                                    ├── 是 → ObjectPool
                                                    └── 否 → 通用分配 → Mimalloc
```

#### 统一分配器接口：让组合成为可能

所有上述组合的前提是：分配器有一个**最小统一接口**。不需要像 STL 那样复杂的 `rebind`、`construct`、`destroy`——只要两个方法：

```cpp
struct IAllocator {
    virtual void* Allocate(usize size, usize align) = 0;
    virtual void  Free(void* ptr) = 0;
};
```

`Array`、`HashMap` 等通用容器通过模板参数接收分配器：`Array<T, MyAlloc>`。专用容器（`FrameArena`、`ObjectPool`）直接实现这个接口。这样：

```cpp
// 通用容器 + 专用分配器 = 场景定制
Array<DrawCmd, FrameArenaAllocator> frameCmds;      // 每帧丢弃
Array<Entity, DefaultAllocator>     entities;       // 持久存在
HashMap<StringId, Texture*, FNVHash, DefaultAllocator> textureMap; // 资源查找

// 专用容器直接面向场景
FrameArena perFrameTemp(1024 * 1024);               // 1MB 帧内存
ObjectPool<Particle, 4096> particlePool;            // 粒子池
RingBuffer<AudioPacket, 256> audioQueue;            // 音频线程队列
```

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「容器与分配器的系统化组合」这个子问题。
> - **UE** 的 `TArray<T, Allocator>` 正是这套组合哲学的体现。同一个 `TArray`，配 `FHeapAllocator` 就是普通堆数组，配 `TInlineAllocator<4>` 就是前 4 个元素在栈上的小型数组，配 `TAllocatorFixedSize` 就是从固定池分配。UE 的 `TMap` / `TSet` 同样参数化了分配器。这套设计的核心洞察是：**容器负责数据结构逻辑，分配器负责内存来源，两者通过最小接口解耦**。
> - **Bevy** 在 Rust 中没有自定义分配器参数（`Vec` 就是标准库 `Vec`），所以 Bevy 选择了另一种组合方式：通用代码用标准容器，性能敏感或特殊语义的代码用完全自定义的数据结构（`BlobArray`、`ComponentSparseSet`、`CommandQueue`）。这种"标准容器 + 专用结构"的两层策略，对个人项目的启发是：不要试图让所有容器都参数化分配器——那样会引入模板膨胀。只在**最常用**的容器（Array、HashMap）上做分配器参数化，其他场景用专用结构。
>
> **个人项目推荐**：
> - 给 `Array<T, Allocator>` 和 `HashMap<K, V, Hash, Allocator>` 实现分配器模板参数。Allocator 接口只需要 `Allocate(size, align)` 和 `Free(ptr)`。
> - 提供三个默认分配器：`DefaultAllocator`（封装 `malloc`/`free`）、`FrameArenaAllocator`（引用外部 Arena）、`PoolAllocator<T, N>`（引用外部 ObjectPool）。
> - 不要为每个容器写独立的分配逻辑——通过一个 `IAllocator` 接口统一。调试时用一个 `DebugAllocatorWrapper` 装饰器包裹任何分配器，即可全局开启 Canary 检测。
> - **最关键的决策**：ECS 存储（SparseSet / Archetype）不要用通用 `Array` 或 `HashMap` 组装——它们有自己独特的内存布局需求（SoA、类型擦除、分页稀疏数组）。专用问题用专用结构解决。

## 下一步预告

现在你拥有了一套完全可控的内存基础设施，并且理解了工业级引擎的存储策略演进：
- 通用动态数组交给 **`Array<T, Allocator>`**（按场景绑定分配器：帧内存、对象池或全局堆）
- 通用键值查找交给 **`HashMap<K, V>`**（开放寻址法，连续存储，避免 STL 的 bucket 指针跳转）
- 临时内存交给 **FrameArena**（多帧场景用双缓冲）
- 固定对象交给 **ObjectPool**（生产环境用分代索引 + 动态分块）
- 线程通信交给 **RingBuffer**（MPSC 场景用多路复用）
- ECS 索引交给 **SparseSet**（大 Entity 空间用分页稀疏数组）
- 批量运算交给 **SoA / Archetype Table**
- 通用分配交给 **Mimalloc** 或自研 **Binned + TLS**
- 频繁变更的标记组件用 **SparseSet**，主数据用 **Table**
- 所有组合通过 **`IAllocator`** 最小接口统一，调试时用 **`DebugAllocatorWrapper`** 装饰器全局开启 Canary 检测

这些容器不是孤立的工具，而是为上层系统铺好的路基。下一步：

- **[[文件IO与虚拟文件系统]]**：让引擎能加载外部配置和资源文件。
- **[[线程池与任务系统]]**：为批量计算和异步加载提供并发能力。

有了自己的容器和分配器，引擎才真正从"能跑"进入"可控"的阶段。
