# 容器与分配器

## Why：为什么游戏引擎需要自定义容器和分配器？

初学者写引擎时，第一反应是"用 `std::vector`、`std::map`、`new`/`delete` 不就好了？"。这在原型阶段确实够用，但一旦进入生产环境，STL 的四个核心痛点会逐一爆发：

1. **分配器不可插拔**：`std::vector` 的内存来源几乎是硬编码的全局 `malloc`/`free`。你无法让临时渲染命令从"帧内存"里分配，也无法让 ECS 组件从固定对齐的 Chunk 里出生。
2. **异常安全冲突**：游戏引擎通常禁用异常（`-fno-exceptions`），但 STL 容器在内存不足时会抛 `std::bad_alloc`。编译开关和库实现一旦不一致，就是链接灾难。
3. **缓存不友好**：`std::list` 是指针跳转链，`std::unordered_map` 是 bucket + 链表/树，遍历时 cache miss 极高。而游戏引擎大量依赖"批量处理"（遍历所有粒子、所有碰撞体、所有 DrawCall）。
4. **调试与确定性缺失**：全局堆是子系统混用的黑箱。一帧之内 `malloc` 的时序、地址、碎片状态完全不可预测，导致离线回放和线上崩溃难以复现。

### 三个真实场景

| 场景 | 如果用 STL | 后果 |
|------|-----------|------|
| **ECS 组件批量分配** | `new ComponentA[1024]` | 构造全部执行一次；无法按 Entity Chunk 对齐；碎片随机 |
| **每帧临时渲染命令** | `std::vector<DrawCmd> tempCmds` | 每帧都隐式 `malloc`/`free`；容量抖动；析构所有命令 |
| **音频包线程间传递** | `std::queue<AudioPacket>` + `mutex` | 锁竞争导致优先级反转；音频线程被阻塞，产生爆音 |

自定义容器和分配器，不是"重复造轮子"，而是**把内存从不可控的黑箱变成可编排的资源**。

## What：最小容器与分配器系统长什么样？

下面给出六个能直接编译运行的最小实现。它们不是玩具，而是真实引擎中对应模块的骨架。每个实现都附带**内存状态变化图**和**诚实的边界分析**。

### 1. FrameArena（帧分配器）

**核心思想**：一帧内所有临时内存共享一块大缓冲区，用一个指针 `offset` 往前推。帧结束时 `offset = 0`，O(1) 全部回收。

```cpp
struct FrameArena {
    static constexpr usize CAPACITY = 1024 * 1024; // 1 MiB
    u8* base = nullptr;
    usize offset = 0;

    void Init() { base = static_cast<u8*>(std::malloc(CAPACITY)); offset = 0; }
    void Shutdown() { std::free(base); base = nullptr; }

    void* Alloc(usize size, usize align = 8) {
        usize mask = align - 1;
        usize padded = (offset + mask) & ~mask;
        if (padded + size > CAPACITY) return nullptr; // 溢出，无回退
        offset = padded + size;
        return base + padded;
    }

    void Reset() { offset = 0; } // 每帧调用一次，O(1) 回收
};
```

**状态变化图**：`Alloc(24, 16)` 执行前后

```
分配前：
base ──► [..............................]
         ^offset=8

对齐后：
base ──► [........][....................]
                  ^padded=16

分配后：
base ──► [........][####################][............................]
                  ^返回的指针            ^offset=40
                  (24 bytes 已分配)
```

> **这段代码已经解决了什么**：O(1) 分配、O(1) 批量回收、零外部碎片、无系统调用（除 Init 时的一次 `malloc`）。
> **还缺什么**：没有多帧轮转（当前帧分配的内存不能异步被消费线程读取）、没有线程安全（需 TLS 或单生产者约束）、没有向系统分配器的动态扩容回退、**对齐浪费可能高达 `align - 1` 字节**。
> **反例警告**：如果一帧内临时分配总量超过 `CAPACITY`（如复杂粒子系统或大规模 ECS 查询），`Alloc` 会直接返回 `nullptr`。FrameArena 只适用于**内存总量可预估**的场景。

### 2. ObjectPool（固定大小对象池）

**核心思想**：预分配一大块固定大小的 Slot，用空闲链表（free list）管理未使用的 Slot。分配和释放都是 O(1) 的指针操作。

```cpp
template <typename T, usize BLOCK_COUNT>
struct ObjectPool {
    union Slot {
        T object;
        Slot* nextFree;
        Slot() {}
        ~Slot() {}
    };

    Slot* memory = nullptr;
    Slot* freeList = nullptr;

    void Init() {
        memory = static_cast<Slot*>(std::malloc(sizeof(Slot) * BLOCK_COUNT));
        for (usize i = 0; i < BLOCK_COUNT; ++i) {
            memory[i].nextFree = (i + 1 < BLOCK_COUNT) ? &memory[i + 1] : nullptr;
        }
        freeList = &memory[0];
    }

    void Shutdown() { std::free(memory); }

    T* Allocate() {
        if (!freeList) return nullptr;
        Slot* slot = freeList;
        freeList = freeList->nextFree;
        return new (slot) T();
    }

    void Free(T* obj) {
        if (!obj) return;
        obj->~T();
        Slot* slot = reinterpret_cast<Slot*>(obj);
        slot->nextFree = freeList;
        freeList = slot;
    }
};
```

**状态变化图**：分配和释放一个对象

```
Init 后：
memory: [Slot0] → [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList

Allocate() 后：
memory: [Slot0: T()]  [Slot1] → [Slot2] → [Slot3] → nullptr
                       ▲
                      freeList
                      (返回 &Slot0.object)

Free(&Slot0) 后：
memory: [Slot0] ─┐
                 │
         ┌───────┘
         ▼
        [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList
```

> **这段代码已经解决了什么**：O(1) 分配/释放、零外部碎片、固定大小对象生命周期完全可控、构造/析构显式调用。
> **还缺什么**：没有跨池释放校验（把不是本池的对象 `Free` 进来会崩溃）、没有内存对齐保证（`alignof(T)` 可能大于默认 `malloc` 对齐）、没有动态扩容、**对象在内存中的地址永远只在预分配的 `memory` 块内，不支持运行时增长**。
> **反例警告**：如果对象大小差异很大（如 16 字节的粒子和 256 字节的 NPC），为每种大小各建一个 ObjectPool 会导致元数据（free list 指针）开销比例不同。ObjectPool 只适用于**对象大小固定且数量可预估**的场景。

### 3. SparseSet（稀疏集）

**核心思想**：用两个数组实现 O(1) 增删和密集遍历，是 ECS 组件存储的最小骨架。

- `sparse[entityID]`：存储该 Entity 在 `dense` 数组中的索引（或空标记）。
- `dense[count]`：存储实际存在的 Entity ID，保证所有有效元素在数组前端连续排列。

```cpp
struct SparseSet {
    static constexpr u32 MAX_ENTITIES = 1024;
    static constexpr u32 INVALID = 0xFFFFFFFF;

    u32 sparse[MAX_ENTITIES];
    u32 dense[MAX_ENTITIES];
    u32 count = 0;

    SparseSet() {
        for (u32 i = 0; i < MAX_ENTITIES; ++i) sparse[i] = INVALID;
    }

    bool Has(u32 entity) const {
        if (entity >= MAX_ENTITIES) return false;
        u32 idx = sparse[entity];
        return idx != INVALID && idx < count && dense[idx] == entity;
    }

    void Add(u32 entity) {
        if (entity >= MAX_ENTITIES || Has(entity)) return;
        sparse[entity] = count;
        dense[count] = entity;
        ++count;
    }

    void Remove(u32 entity) {
        if (entity >= MAX_ENTITIES || !Has(entity)) return;
        u32 idx = sparse[entity];
        u32 lastEntity = dense[count - 1];
        dense[idx] = lastEntity;
        sparse[lastEntity] = idx;
        sparse[entity] = INVALID;
        --count;
    }

    const u32* Data() const { return dense; }
    u32 Count() const { return count; }
};
```

**状态变化图**：Add(Entity 5) → Add(Entity 2) → Remove(Entity 5)

```
初始状态：
sparse: [FF, FF, FF, FF, FF, FF, ...]
dense:  []
count:  0

Add(5):
sparse: [FF, FF, FF, FF, FF, 0,  FF, ...]
                             ^entity 5 的索引
dense:  [5]
count:  1

Add(2):
sparse: [FF, FF, 1,  FF, FF, 0,  FF, ...]
                   ^entity 2 的索引
dense:  [5, 2]
count:  2

Remove(5) — swap-and-pop 保持密集：
sparse: [FF, FF, 1,  FF, FF, FF, FF, ...]
                             ^entity 5 被标记为无效
dense:  [2]          (原末尾元素 2 被 swap 到索引 0)
count:  1
```

> **这段代码已经解决了什么**：O(1) 增删、O(count) 密集遍历（无空位检查）、极其缓存友好。`dense` 数组的连续性让批量处理组件时几乎没有 cache miss。
> **还缺什么**：`sparse` 数组的大小是 `MAX_ENTITIES`，Entity ID 范围很大时会浪费内存；没有与组件数据（ComponentArray）的联合存储。
> **反例警告**：如果 Entity ID 空间是 32 位（4294967296），直接开 `sparse` 数组会吃掉 16 GB 内存。工业级实现会用**分页稀疏数组**（Sparse Array of Pages）或**哈希表**回退。
> **ECS 映射**：`SparseSet` 就是 ECS `ComponentArray` 的索引层。`dense` 数组遍历得到的每个 Entity，再到对应的 `T components[count]` 数组中取值，即可完成"所有拥有 Component T 的实体"的批量处理。

### 4. SoAVector（结构体数组 → 数组的结构体）

**核心思想**：把 `struct Particle { Vec3 pos; Vec3 vel; float life; }` 拆成三个平行数组。批量运算时只读取需要的字段，消除无关字段的 cache line 污染。

```cpp
template <usize MAX_COUNT>
struct SoAParticles {
    f32 posX[MAX_COUNT];
    f32 posY[MAX_COUNT];
    f32 posZ[MAX_COUNT];
    f32 velX[MAX_COUNT];
    f32 velY[MAX_COUNT];
    f32 velZ[MAX_COUNT];
    f32 life[MAX_COUNT];
    usize count = 0;

    void Add(f32 x, f32 y, f32 z, f32 vx, f32 vy, f32 vz, f32 l) {
        if (count >= MAX_COUNT) return;
        posX[count] = x; posY[count] = y; posZ[count] = z;
        velX[count] = vx; velY[count] = vy; velZ[count] = vz;
        life[count] = l;
        ++count;
    }

    void UpdatePositions(f32 dt) {
        for (usize i = 0; i < count; ++i) {
            posX[i] += velX[i] * dt;
            posY[i] += velY[i] * dt;
            posZ[i] += velZ[i] * dt;
        }
    }
};
```

**内存布局对比**：AoS vs SoA（假设 `MAX_COUNT = 4`，每个 float 4 字节）

```
AoS (Array of Structs):
[ posX, posY, posZ, velX, velY, velZ, life ]  ← 32 bytes per particle
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]

如果只想更新 posX：
读取 cache line 时，vel/life 等无关数据被一起加载（假设 cache line 64 bytes，
每条 cache line 容纳 2 个粒子，但只用了 1 个 float）。

SoA (Structure of Arrays):
posX: [ x0, x1, x2, x3 ]  ← 16 bytes, 紧凑连续
posY: [ y0, y1, y2, y3 ]
posZ: [ z0, z1, z2, z3 ]
velX: [ vx0, vx1, vx2, vx3 ]
...

更新 posX 时：
16 bytes 完全有效，一条 cache line（64 bytes）可装 16 个 float，
即 16 个粒子的 posX。遍历效率显著提升。
```

> **这段代码已经解决了什么**：批量 SIMD 运算的天然友好布局、只读部分字段时 cache 利用率最大化。
> **还缺什么**：随机访问单个粒子的全部字段需要 7 次指针跳转（`posX[i]`, `posY[i]`...）、代码冗长、删除中间元素需要维护所有数组的同步。
> **反例警告**：如果系统需要频繁"读取一个粒子的全部字段"（如单个粒子与玩家的碰撞检测），SoA 会导致 7 次不连续的内存读取，反而比 AoS 慢。**SoA 是为批量同字段运算优化的，不是为随机全字段访问优化的**。
> **ECS 映射**：ECS 的 Archetype Chunk 本质上就是 SoA 的工业级实现。同一 Archetype 下的所有实体，其组件数据按类型分块存储在 Chunk 中，系统查询时只读取相关的组件列。

### 5. RingBuffer（无锁环形队列，SPSC）

**核心思想**：用两个原子索引 `head` / `tail` 和一块固定数组，实现单生产者-单消费者（SPSC）的零锁通信。

```cpp
#include <atomic>

template <typename T, usize SIZE>
struct RingBuffer {
    static_assert((SIZE & (SIZE - 1)) == 0, "SIZE must be power of 2");
    T buffer[SIZE];
    std::atomic<usize> head{0};
    std::atomic<usize> tail{0};

    bool Push(const T& item) {
        usize t = tail.load(std::memory_order_relaxed);
        usize next = (t + 1) & (SIZE - 1);
        if (next == head.load(std::memory_order_acquire)) return false; // 满
        buffer[t] = item;
        tail.store(next, std::memory_order_release);
        return true;
    }

    bool Pop(T& item) {
        usize h = head.load(std::memory_order_relaxed);
        if (h == tail.load(std::memory_order_acquire)) return false; // 空
        item = buffer[h];
        head.store((h + 1) & (SIZE - 1), std::memory_order_release);
        return true;
    }
};
```

**状态变化图**：Push A → Push B → Pop A（SIZE = 4）

```
初始状态：
[ _, _, _, _ ]
 h=0
 t=0

Push(A):
[ A, _, _, _ ]
 h=0
 t=1

Push(B):
[ A, B, _, _ ]
 h=0
 t=2

Pop() → A:
[ A, B, _, _ ]   (A 仍在内存中，但逻辑上已消费)
    h=1
 t=2
```

> **这段代码已经解决了什么**：单生产者-单消费者（SPSC）场景下的零锁通信、严格的 acquire/release 内存序保证可见性、无系统调用。
> **还缺什么**：仅支持 SPSC；MPSC/MPMC 需要更复杂的原子 CAS 或 seqlock；没有 ABA 防护（大小为 2 的幂次配合索引环绕可天然缓解，但 64 位索引溢出问题在极长运行时间后理论上存在）；**容量固定为 2 的幂次**。
> **反例警告**：如果在生产者线程中 Push 的数据大小不固定（如变长字符串），RingBuffer 无法直接使用。它只适用于**固定大小消息**的传递。多生产者场景下不加锁直接使用此代码会导致数据竞争。

## How：真实引擎的容器/内存系统是如何一步一步复杂起来的？

### 阶段 1：最小实现 → 能用（解决分配速度和碎片问题）

**触发原因**：`malloc`/`free` 的时延不可预测，全局堆碎片化，批量遍历 cache miss 严重。

**代码层面的变化**：

| 从 | 到 | 解决的问题 |
|----|---|-----------|
| `std::vector<Component>` | `SparseSet + ComponentArray` | O(1) 增删，密集遍历 |
| `new T` / `delete T` | `ObjectPool<T, N>` | O(1) 分配，零外部碎片 |
| `std::vector<TempData>` | `FrameArena` | 每帧 O(1) 批量回收 |
| `std::queue + mutex` | `RingBuffer` | 音频/网络线程零锁通信 |
| AoS `struct Particle` | `SoAVector` | SIMD 批量运算 cache 友好 |

这一阶段的核心收益是**速度**和**可预测性**。每个子系统开始按场景匹配算法，而不是一刀切地回退到 `malloc`。

---

### 阶段 2：能用 → 好用（解决迭代效率和调试问题）

**触发原因**：专用容器多了之后，如何让用户代码不硬编码分配器类型？如何检测内存越界和 use-after-free？

**代码层面的变化**：

#### 2.1 模板化分配器接口

```cpp
template <typename T, typename AllocatorT>
struct DynamicArray {
    T* data = nullptr;
    usize count = 0;
    usize capacity = 0;
    AllocatorT* allocator = nullptr;

    void Resize(usize newCap) {
        T* newData = static_cast<T*>(allocator->Alloc(newCap * sizeof(T), alignof(T)));
        for (usize i = 0; i < count; ++i) {
            new (&newData[i]) T(std::move(data[i]));
            data[i].~T();
        }
        allocator->Free(data);
        data = newData;
        capacity = newCap;
    }
};
```

通过为不同子系统生成独立内存池，实现编译期隔离：
- `RenderArena`：帧分配器 + 对象池
- `AIPool`：TLSF 分配器
- `AudioRing`：RingBuffer + 固定块池

一个子系统的内存泄漏或越界，不会直接污染其他子系统，调试范围从"全局堆"缩小到"具体子系统池"。

#### 2.2 调试与诊断 Hook 层

| 机制 | 作用 |
|------|------|
| **毒值填充** | 分配时清零 `0x00`，释放时写入 `0xFE`，检测 use-after-free |
| **Canary 越界检测** | 在块前后插入魔数，释放时校验 |
| **调用栈捕获** | 记录每次分配的调用栈，泄漏时可回溯到具体代码路径 |
| **AllocatorMetrics** | 统一统计 OS 申请量、用户已分配量、元数据开销、最大/最小块大小 |

这些调试能力通过 AOP 式 Hook 实现，Final 构建可完全剥离，零运行时开销。

---

### 阶段 3：好用 → 工业级（解决性能和规模问题）

**触发原因**：对象池容量固定、Buddy 分配的对数复杂度在实时线程中仍不够稳定、Entity ID 空间太大导致稀疏数组内存爆炸。

**代码层面的变化**：

#### 3.1 从固定 ObjectPool 到分代/动态池

```cpp
struct GenerationalPool {
    struct Slot { T object; u32 generation; Slot* nextFree; };
    std::vector<Slot*> blocks; // 动态追加新 block
    // Handle = { blockIndex, slotIndex, generation }
    // Free 时 generation 不变，下次 Allocate 时 generation++
};
```

**状态变化**：Handle 引入 generation 后，即使 Slot 被复用，旧 Handle 的 generation 不匹配，可检测 use-after-free。

#### 3.2 TLSF：严格 O(1) 的通用分配器

对于通用小对象和实时性要求高的系统（如网络包、AI 决策），Buddy 的对数复杂度仍然不够稳定。引入 **TLSF（Two-Level Segregated Fit）**：

- 通过一级索引（fl）和二级索引（sl）位图，在常数时间内定位到合适的空闲块。
- 分配和释放均为**严格 O(1)**，满足音频、网络等实时线程的要求。
- 支持对齐分配，适配 SIMD 和 GPU 资源对齐需求。

**诚实分析**：
- **优势**：最坏情况下仍是 O(1)，确定性极强。
- **代价**：位图和分割/合并逻辑引入额外元数据开销；内存利用率比 `malloc` 低 5%~15%（内部碎片）。
- **反例警告**：如果分配块大小极度分散（从 8 字节到 64 MB），TLSF 的索引表会变大，cache 局部性下降。它最适合**大小分布集中、实时性要求高**的场景。

#### 3.3 虚拟地址预分配 + 物理页延迟提交

向 OS 预留一大段连续虚拟地址（`VirtualAlloc` / `mmap`），真正写入时才触发页分配：
- 保证地址连续性（对 ECS Chunk 迁移和序列化极其重要）。
- 避免过早占用物理内存（对大世界编辑器友好）。

**诚实分析**：
- **优势**：地址连续性 + 按需提交。
- **代价**：虚拟地址空间是有限资源（32 位系统尤其紧张）；过度预留可能导致其他模块虚拟地址耗尽。

#### 3.4 ECS 专用：Chunk Allocator + Component Page

到了 ECS 工业级阶段，容器和分配器必须与 ECS 存储模型深度融合：

- **Chunk Allocator**：按固定大小（如 16 KB）分配 Chunk。每个 Chunk 存储同一 Archetype 的实体组件数据。
- **Component Page**：把 SparseSet 的 `sparse` 数组从线性大数组改为**页表结构**（`sparse[pageIndex][offsetInPage]`），Entity ID 空间扩展到 32 位而不浪费内存。

**状态变化图**：Component Page 的分页稀疏数组

```
线性稀疏数组（浪费）：
sparse[4,294,967,296]  → 16 GB（不可接受）

分页稀疏数组：
pageTable[4096] → 每个 entry 指向一个 page（大小 1024）
                  或 nullptr（该页无实体）

EntityID = 0x00100080:
  pageIndex = 0x0010 = 16
  offset    = 0x0080 = 128
  sparse = pageTable[16][128]

实际内存：只有被使用过的页才会分配（4096 pages × 1024 uint32 ≈ 16 MB 全部分配，
          但实际按需分配，未使用页为 nullptr）。
```

## 性能与 trade-off 总结：诚实对比

| 容器/分配器 | 对比基准 | 什么场景更好 | 什么场景更差 | 核心代价 |
|------------|---------|-------------|-------------|---------|
| **FrameArena** | `std::vector` + 每帧 `clear`/`shrink` | 大量临时对象，生命周期同帧 | 跨帧存活、总量不可预估 | 无单独释放，总容量固定 |
| **ObjectPool** | `new`/`delete` | 固定大小对象频繁创建销毁 | 对象大小差异大、数量剧增 | 无动态扩容，地址不连续 |
| **SparseSet** | `std::unordered_set<uint32_t>` | 密集遍历 + O(1) 增删 | Entity 空间极大、稀疏分布 | `sparse` 数组预分配内存 |
| **SoAVector** | AoS `std::vector<Struct>` | 批量 SIMD 同字段运算 | 随机访问单个对象全部字段 | 删除需同步多数组，代码冗长 |
| **RingBuffer** | `std::queue` + `mutex` | SPSC 线程通信 | MPMC、变长消息 | 容量固定为 2 的幂 |
| **TLSF** | `malloc` / `free` | 实时系统、确定性要求 | 块大小极度分散、内存紧张 | 元数据开销、利用率低 5%~15% |

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 满足 | FrameArena 的 `offset` 只要分配顺序固定就是确定性的；ObjectPool 的块布局是严格平铺的数组；SparseSet 的 `dense` 数组是 ECS 批量处理的基础 |
| **自描述** | ✅ 满足 | `AllocatorMetrics` 接口暴露 `consumedMemoryBytes`、`allocationCount`、`peakUsage` 等指标，AI 可观测 |
| **确定性** | ✅ 满足 | FrameArena 的 `Reset()` 消除了释放顺序的不确定性；ObjectPool 消除了外部碎片；SparseSet 的 `swap-and-pop` 保证遍历顺序虽变但无空洞 |
| **工具边界** | ⚠️ 需封装 | AI 通过 MCP 请求"分配 1024 字节临时内存"时，必须返回一个作用域句柄（如 `ScopedArenaAlloc`），禁止返回裸指针 |
| **Agent 安全 / 沙箱隔离** | ✅ 满足 | 通过为不同子系统生成独立内存池（`PoolAI`、`PoolScript`）实现编译期隔离，Agent 脚本层的分配错误不会破坏核心引擎 |

## 与 ECS / 反射的关联

- **SparseSet** 是 ECS `ComponentArray` 的索引骨架。`dense` 数组遍历得到的每个 Entity，再到 `components[count]` 中取值，即可完成组件系统的批量查询。
- **SoAVector** 是 ECS Archetype Chunk 的简化版。理解 SoA 后，Archetype 的"按组件类型分块存储"就不再是黑魔法。
- **FrameArena** 的 O(1) 线性分配 + 每帧 `Reset` 为 ECS 的临时查询结果（如 `Query<Transform, Mesh>` 返回的实体列表）提供了零抖动的内存保障。
- **ObjectPool** 的固定大小分配消除了外部碎片，确保 `Entity` 句柄、网络消息等对象在固定内存预算内稳定运行。
- **RingBuffer** 的 SPSC 无锁设计可直接用于 System 调度中的命令缓冲提交，避免互斥锁导致的优先级反转。
- 当容器和分配器就位后，[[组件系统架构]] 中的 ECS Chunk 存储、[[反射系统]] 中的类型注册表，都可以定制专属的高性能数据结构，而不再被 STL 的默认行为绑架。
