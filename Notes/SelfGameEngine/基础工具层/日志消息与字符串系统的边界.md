# 日志消息与字符串系统的边界

## 问题：日志消息不是字符串吗？

刚开始梳理引擎字符串系统时，一个很自然的想法是：日志系统大量使用字符串——日志级别（`"DEBUG"`）、分类名（`"Render"`）、格式化后的消息（`"Player hero pos=(114.5, 20.3)"`）——这些不都是字符串吗？既然我们已经有 SSO `String`、`StringView`、`StringId`，日志系统直接复用不就行了？

这个想法的问题在于：**它把"格式化过程"和"存储结构"混为一谈了**。日志系统确实会调用字符串格式化器来"生产"文本，但生产出来的日志消息在引擎中的角色，和组件名、资源路径这些"标识符字符串"完全不同。

## 两类字符串的根本区别

引擎中实际存在两种截然不同的"字符串"，它们的特征几乎完全相反：

| 特征 | 引擎标识符（组件名、资源名、配置键） | 日志消息（格式化后的输出） |
|------|-----------------------------------|-------------------------|
| **典型长度** | `"Transform"`（9）、`"volume"`（6）— 大多 < 16 字节 | `"Player hero pos=(114.5, 20.3)"`（30+）— 大多 30~200 字节 |
| **生命周期** | 持久（伴随组件/资源/配置项存在） | 临时（几帧后被新日志覆盖） |
| **总量特征** | 种类有限，反复引用 | 无限增长，只追加不修改 |
| **比较需求** | 高频（组件查找、事件匹配） | 几乎不比较 |
| **分配模式** | 零散、反复 `new`/`delete` | 顺序追加、批量淘汰 |
| **存储方式** | 长期持有，需要随机访问 | 流式写入，顺序消费 |
| **对应数据结构** | `StringId`（O(1) 比较）、`String`（SSO 内联） | **环形缓冲区顺序存储** |

> **核心洞察**：SSO `String` 和 `StringId` 解决的是"如何让一个短字符串在被反复引用和比较时又快又省"。日志消息既不短、也不被反复引用、更不需要比较——它需要的是"高速顺序写入 + 批量丢弃"。这是完全不同的数据结构问题。

## 为什么日志消息不适合用 `String` 存？

### 1. 生命周期错配

`String` 是一个**拥有型容器**。当你写 `String msg = format(...)` 时，引擎为这条消息分配了内存（即使是 SSO，也是 24 字节的完整对象构造），并在 `msg` 离开作用域时执行析构。

但日志消息的典型生命周期是：

```
生成 → 写入缓冲区 → 被后台线程消费 → 丢弃
   ↑_________________________________________|
                    （几毫秒到几十毫秒）
```

用 `String` 存日志，相当于为一段只活几毫秒的数据执行完整的"构造-析构"仪式。这不是 `String` 的设计目标。

### 2. 分配模式错配

`String` 的分配模型是"为每个对象单独管理内存"——即使 SSO 避免了堆分配，每次构造仍需执行 `strlen`/`memcpy`、模式判别、长度计算。

日志的分配模型应该是"在预分配的大块内存中顺序写入"：

```cpp
// 工业级做法：1MB 预分配环形缓冲区，日志顺序追加
char* ringBuffer = allocator.alloc(1024 * 1024); // 只分配一次
// 每条日志：head += write(entry, ringBuffer + head);
// 无 per-entry 分配，无析构，无 memcpy（或仅有一次性的顺序内存写）
```

### 3. 长度分布错配

[[可变字符串与 SSO]] 中提到的"80% 的字符串长度小于 16 字节"，统计的是**引擎标识符**（组件名、资源名、配置键、日志级别如 `"DEBUG"`）。**格式化后的日志消息不在此列**。

笔记中自己也给出了反例："每条日志平均 80 个字符"。80 字节远超 SSO 的 22 字节上限，必然走堆分配。如果每帧 100 条日志、每秒 60 帧，用 `String` 存储意味着每秒 6000 次堆分配——这正是日志系统要极力避免的。

## 日志消息的工业级存储架构

正确的做法是：**日志消息不走 `String`/`StringView`/`StringId` 中的任何一个**，而是用专为流式临时数据设计的存储结构。

### 方案 1：环形缓冲区顺序存储（最常用）

```cpp
struct LogRingBuffer {
    char*  buffer;      // 预分配大块（如 4MB）
    size_t capacity;
    size_t head;        // 写入位置
    size_t tail;        // 最老的有效数据
};

// 写入一条日志：在 buffer[head] 处顺序写入 entry header + 变长消息
// 无 per-entry 分配，满了覆盖最老数据
```

**关键特征**：
- 预分配一次，运行期零分配
- 日志条目变长内联存储（header + message 紧跟其后），无指针
- 覆盖旧日志时自然回收空间，无需逐条析构

这是 [[可视化日志系统]] 中"双缓冲队列"和"TLS 写缓冲"的底层基础。无论上层是 `std::vector` 交换还是 TLS 三指针，核心思想都是**顺序写入预分配缓冲**。

### 方案 2：延迟格式化（更极致）

如果连"格式化后的字符串"都不存储，只存模板和参数，内存效率还能更高：

```cpp
// 写入时
LOG_INFO("Player {} pos=({:.2f}, {:.2f})", name, x, y);

// 实际存储的条目
struct DeferredLogEntry {
    uint64_t    timestamp;
    StringId    formatId;       // "Player {} pos=({:.2f}, {:.2f})" 的 Intern ID
    LogArgStore args;           // 类型擦除后的参数（name, x, y）
};
```

**收益**：
- 如果日志被过滤掉（如 Release 关闭 DEBUG），**格式化开销完全避免**
- 存储的是紧凑参数（`float` 4 字节、`StringId` 8 字节），而非展开的 30+ 字节文本
- 多线程只竞争无锁环形 buffer 的写入位

**代价**：需要类型擦除机制，实现复杂度高。详见 [[字符串格式化]] 中 fmtlib 的"类型擦除接口"一节。

### 方案 3：结构化存储（JSON Lines）

[[可视化日志系统]] 中提出的 `.jsonl` 输出，本质上是把日志当作**结构化事件**而非**文本字符串**：

```json
{"timestamp":1234.56,"level":"Info","category":"Render",
 "fields":{"fps":60.0,"draw_calls":128}}
```

这里 `"Render"` 是 `StringId` 反查得到的，`"Info"` 是枚举值映射，而 `"fps"`、`"draw_calls"` 这类字段名本身也是 `StringId`。**真正的日志"消息"不是字符串，是键值对集合**。

## 日志系统中的字符串分工

厘清边界后，日志系统中各类字符串基础设施的分工就很明确了：

| 日志系统中的元素 | 推荐类型 | 理由 |
|----------------|---------|------|
| 日志级别（`"DEBUG"` → `LogLevel::Debug`） | `LogLevel` 枚举 | 编译期常量，零开销 |
| 日志分类/通道名（`"Render"`、`"Physics"`） | `StringId` | 短标识符，O(1) 比较，InternPool 统一管理 |
| 格式化模板（`"Player {} pos={}"`） | `StringId` | 编译期/启动期注册，延迟格式化时查表 |
| 格式化参数中的字符串（`name`、`path`） | `StringView` | 只读引用，零拷贝传入格式化器 |
| **格式化后的日志消息** | **环形缓冲区顺序存储** | **不走 `String`，不堆分配，临时流式数据** |
| 日志历史供 ImGui 显示 | `LogEntry` 结构体数组 | 纯数据，无指针，平铺缓存友好 |

> **一句话总结**：`StringId` 负责日志的"元数据标签"，`StringView` 负责格式化时的参数传递，栈格式化器负责"生产"文本，**环形缓冲区负责"容纳"文本**。四个环节各司其职，没有任何一个环节需要 `String` 容器。

## 回扣：日志系统笔记中的相关设计

[[可视化日志系统]] 中的几个关键设计，现在可以从"字符串系统边界"的视角重新理解：

1. **异步双缓冲队列 / TLS 写缓冲**：不是"用更好的锁保护 `String` 的写入"，而是"彻底消除 per-entry 的 `String` 构造，改用顺序内存写"。

2. **Sink 抽象**：`ConsoleSink` 和 `FileSink` 接收的是 `LogEntry`（结构化数据），不是 `const String&`。Sink 内部按需格式化——文件 Sink 可能输出文本，JSON Sink 直接序列化字段，两者都不持有 `String`。

3. **编译期过滤**：`if constexpr` 消除的不只是 `log()` 调用，而是**整个格式化 + 存储链条**。如果日志在编译期被剔除，`StringView` 参数不需要构造、`format()` 不需要执行、环形 buffer 不需要写入。

4. **结构化 `.jsonl`**：当日志以字段形式存储时， `"category"` 是 `StringId` 反查， `"msg"` 甚至可以不存在——真正的信息在 `"fields"` 键值对中。这比先格式化成 `String` 再拆分字段要高效得多。

## ECS 映射：日志在 ECS 中怎么表达？

```cpp
// LogEvent：瞬时事件，帧结束时清除
struct LogEvent {
    uint64_t  timestamp;
    LogLevel  level;
    StringId  categoryId;   // "Render"、"Physics" — StringId
    StringId  formatId;     // 模板 ID
    LogArgStore args;       // 类型擦除参数
};

// LogBuffer：ImGui 面板对应的实体，持有环形缓冲组件
struct LogBuffer {
    RingBuffer<LogEntry> entries;  // 不是 String 数组！
    uint32_t maxHistory = 4096;
};
```

**关键变化**：
- 日志不再是"调用全局 `g_logger.log(String msg)`"，而是"向 World 发送 `LogEvent`，Event 中不带 `String`"
- 消费端 Sink System 按需将 `LogEvent` + `formatId` + `args` 展开成文本——展开动作发生在消费线程，不在生产热点路径
- AI 直接 Query `LogEvent`，读取 `StringId` 和结构化字段，无需解析文本
