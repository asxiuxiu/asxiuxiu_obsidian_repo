# 字符串系统

## 为什么需要自定义字符串系统？

`std::string` 对引擎热路径来说有三个致命问题：

1. **分配器不可控**：`std::string` 的堆分配走全局 `new/delete`，无法接入引擎的内存池、线程局部分配器或追踪系统
2. **角色混乱**：`std::string` 试图同时扮演"标识符"（需要 O(1) 比较）、"只读引用"（需要零拷贝）和"可变文本"（需要动态拼接）三个角色，结果每个角色都做不好
3. **ECS 不兼容**：`std::string` 内部含堆指针，放入 ECS 组件会破坏密集数组的纯值特性，序列化时需要特殊处理

自定义字符串系统不是"造一个更好的 std::string"，而是**为三种不同角色分别设计专用类型**。

---

## 三种字符串角色的精确定义

引擎中的字符串按使用场景分为三种角色。这三种角色**不可互相替代**，必须在设计初期明确区分。

| 角色 | 核心需求 | 典型场景 | 对应类型 | 绝对不能做的事 |
|------|---------|---------|---------|-------------|
| **标识与键** | O(1) 比较、可哈希、纯值、无指针、确定性强 | 组件名、资源路径、事件名、配置键、日志分类 | `StringId` / `FName` | **不能**用于动态拼接（`"Entity_" + id`） |
| **只读引用** | 零拷贝、不拥有数据、可接受字面量/String/char* | System 函数参数、路径解析、子串操作、格式化参数 | `StringView` / `&str` | **不能**存储（不延长生命周期）、**不能**用于哈希表键 |
| **局部可变文本** | 动态拼接、格式化、修改、输出 | 日志消息、调试文本、文件路径构建、UI 动态内容 | `String` / `FString` | **不能**放入 ECS 组件、**不能**作为热路径比较键 |

### 为什么三种角色不可互相替代？

**StringView 不能替代 StringId**：`StringView` 的比较是 O(n)，且不具备可哈希性——你没法把 `StringView` 放进 `HashMap` 当键。更重要的是，`StringView` 不拥有数据，如果被引用的字符串被销毁，`StringView` 变成悬垂引用。标识符需要长期稳定存在。

**StringId 不能替代 String**：`StringId` 是不可变的。你无法用它来做 `"Player_" + std::to_string(id)` 这种动态拼接。标识符在创建时就确定内容，之后只读不写。

**String 不能替代 StringView**：`String` 的每次构造都有成本（即使是 SSO，`strlen` + `memcpy` 也躲不掉）。如果一个函数只是"看一眼"字符串内容（如 `getExtension(StringView path)`），要求传入 `String` 会强制调用方构造一个完整对象。

> **决策口诀**：先问"这个字符串要不要存进组件？" → 要，用 `StringId`；再问"这个字符串要不要修改？" → 要，用 `String`；否则默认用 `StringView`。

## 默认推荐路径

| 用途 | 推荐类型 | 理由 |
|------|---------|------|
| 动态文本（UI、错误信息、日志内容生产） | `String`（薄封装 `std::string` + 自定义分配器） | 实现成本低，现代标准库已有平台级 SSO，接入引擎内存池后堆路径可控 |
| 只读字符串参数（函数参数） | `StringView` | 零拷贝、无分配、可接受字面量/String/char* |
| 资源路径/组件名/配置键/事件名 | `StringId` | O(1) 比较、可哈希、纯值、Intern 池保证唯一性 |
| 日志/调试格式化输出 | 栈格式化器 / `std::format`（C++20） | 生产文本用栈格式化器，**存储文本用环形缓冲区**——日志消息不走 `String`。详见 [[日志消息与字符串系统的边界]] |
| 本地化管理 | 暂不实现 | 单机原型不需要，阶段 7 工具链再扩展 |

## ECS 映射：字符串在 ECS 中该怎么放？

```cpp
// ❌ 反模式：组件存储 String（哪怕是 SSO String）
struct NameComponent {
    String name; // 24-32 字节，可能含堆指针 → 破坏密集数组
};

// ✅ 正确：组件存储 StringId，System 参数用 StringView
struct NameComponent {
    StringId nameId; // u64，纯值，无指针，O(1) 比较
};

// 实际字符串内容存在 ECS Resource（全局状态）中
struct NameRegistry : public ECSResource {
    StringInternPool pool;
    HashMap<Entity, StringId> entityNames;
};
```

**核心原则**：ECS 组件中只存 `StringId`（u64），不存字符串对象。所有字符串内容上浮到 Resource 或外部系统中。

### 从 OOP 到 ECS 的重构映射

| OOP 设计 | ECS 重构 |
|---------|---------|
| Actor 的 Name 成员（FString） | `NameComponent { StringId nameId }` + `NameRegistry` Resource |
| 类型名字符串（如 UClass::GetFName） | `ComponentId`（u64）+ `TypeRegistry`（编译期注册） |
| 资源路径（FString 路径） | `AssetPathId`（StringId）+ `AssetRegistry` Resource |
| 日志 CategoryName（FName） | `LogCategoryId`（StringId），编译期常量 |
| 事件名字符串（如 `"OnDamage"`） | `EventTypeId`（StringId），编译期注册 |

各子笔记中涉及 ECS 映射的内容统一引用此处，不再重复展开。

## AI 友好设计红线

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ StringId 可直接作为 ECS 组件 | 8 字节纯值，无指针，序列化/拷贝 trivial |
| **自描述** | ✅ AI 可通过 TypeRegistry 反查名称 | Debug 构建维护 `StringId → const char*` 映射；Release 可剥离 |
| **确定性** | ✅ FNV-1a 哈希是纯算法 | 给定相同输入，输出始终相同。支持快照保存与回放 |
| **工具边界** | 应操作 `StringId` | AI 修改资源名或组件名时，优先操作 `StringId`，原始字符串仅作为初始注册输入 |
| **Agent 安全** | ⚠️ 需防范哈希冲突 | Debug 构建启用 intern 池碰撞检测；关键路径（如资源加载）使用二次校验 |

本模块的产出需要回探替换阶段 1 和阶段 2 中的临时实现：

| 临时实现（阶段 1/2） | 替换为（阶段 3.3） | 验证方法 |
|---------------------|-------------------|---------|

| 阶段 2 的组件名硬编码字符串（如 `"Transform"`） | `StringId` 编译期常量 | Inspector 仍能显示组件标签，且切换为 StringId 后无性能回退 |
| 日志格式化中的 `std::ostringstream` | 栈格式化器 / `std::format` | 输出格式一致，无堆分配 |
| 函数参数中的 `const std::string&` | `StringView` | 消除隐式构造开销 |

## 扩展路径（更极端工业场景）

- **无锁 Intern 池**：用 TLS 线程本地缓冲 + 批量提交消除多线程 `intern()` 的锁竞争。详见 [[字符串标识与 InternPool]]
- **大小写不敏感 FName**：UE 的 `FName` 默认大小写不敏感，适合资源路径比较。当前实现是大小写敏感的，可在 Intern 池中加入可选的 case-insensitive 模式
- **本地化与富文本**：引入 `FText` 级别的本地化支持（翻译表、文本收集、复数规则），以及 UI 富文本格式化（颜色、字体、超链接）
- **字符串压缩**：对序列化后的字符串数据做字典压缩（如 Zstd），减少存档和网络包体积
