# 字符串系统

## 问题 0：为什么引擎不能直接用 `std::string`？

刚开始写引擎原型时，几乎所有开发者都会下意识选择 `std::string`。它熟悉、安全、功能齐全——直到你开始用 Profiler 看帧时间。

假设你有一个 60 FPS 的游戏，每帧输出 100 条日志，每条日志平均 80 个字符：

```cpp
// 最 naive 的做法：全局使用 std::string
std::string msg = "Player " + name + " pos=(" + std::to_string(x) + ", " + std::to_string(y) + ")";
logger.info(msg);
```

这行代码在运行时发生了什么？

1. `std::to_string(x)` 创建一个临时的 `std::string`，触发堆分配
2. `+` 运算符每次拼接都可能导致重新分配和深拷贝
3. 最终 `msg` 的构造又是一次堆分配
4. 如果 `name` 也是 `std::string`，`"Player " + name` 会先构造一个临时对象

**结果**：单条日志触发了 5~8 次小内存分配。一帧 100 条日志就是 500~800 次分配。每秒 60 帧就是 **3~5 万次堆分配/秒**。

这会带来三个连锁问题：
- **分配器压力**：小对象分配占用了内存分配器的大量时间，且容易产生碎片
- **Cache 不友好**：`std::string` 内部通常是指针+长度+容量的三元组（24 字节），实际字符数据在堆的另一处。遍历字符串时，每次访问都要解引用指针，cache miss 率极高
- **比较成本高**：检查两个资源路径是否相同，需要逐字节比较，O(n) 复杂度。在组件查找、事件订阅键匹配时，这个成本被放大成千上万倍

> **诚实声明**：如果你的引擎目前只是一个窗口加几个立方体，`std::string` 完全够用。本模块解决的是"当字符串操作进入热点路径 Top10 时"的问题。阶段 1 的日志系统用 `std::string` 是正确选择——先让东西跑起来，再优化。

## 问题 1：如何消除小字符串的堆分配？

观察引擎中的实际字符串使用情况，你会发现一个规律：**80% 的字符串长度小于 16 字节**。组件名（`"Transform"`）、资源短名（`"hero"`）、配置键（`"volume"`）、日志级别（`"DEBUG"`）……这些都不长，但 `std::string` 无一例外地把它们扔到了堆上。

### Naive 方案：短字符串走栈，长字符串走堆

```cpp
// 问题：对象大小不固定，无法放入 ECS 组件或数组
struct NaiveString {
    char stack_buf[16];   // 短字符串内联
    char* heap_buf;       // 长字符串堆分配
    size_t len;
    bool is_heap;
};
```

这个方案的问题在于：
- `is_heap` 标志位让对象变成 32 字节（16 + 8 + 8 + 1 对齐到 32），cache line 利用率低
- 条件分支（`if (is_heap)`）在热路径上会降低分支预测命中率
- 放进 ECS 组件时，24 字节的内联空间被浪费了——ECS 组件通常是密集数组

### 改进方案：SSO（Small String Optimization）

SSO 的核心洞察是：**字符串对象本身已经有 24 字节的开销（指针+长度+容量），为什么不把这部分空间用来存字符？**

```cpp
// 这是 roadmap 阶段 3.3 的核心代码
class SmallString {
    static constexpr usize SSO_CAPACITY = 15;
    union {
        char  m_inline[SSO_CAPACITY + 1]; // 内联缓冲（16 字节）
        struct {
            char* m_data;     // 8 字节
            usize m_capacity; // 8 字节
        } m_heap;
    };
    usize m_len; // 8 字节

public:
    SmallString(const char* str = "") {
        m_len = std::strlen(str);
        if (m_len <= SSO_CAPACITY) {
            std::memcpy(m_inline, str, m_len + 1);
        } else {
            m_heap.m_capacity = m_len + 1;
            m_heap.m_data = new char[m_heap.m_capacity];
            std::memcpy(m_heap.m_data, str, m_len + 1);
        }
    }
    ~SmallString() { if (!isInline()) delete[] m_heap.m_data; }

    const char* c_str() const { return isInline() ? m_inline : m_heap.m_data; }
    usize length() const { return m_len; }
    bool  isInline() const { return m_len <= SSO_CAPACITY; }
};
```

**SSO 内存布局状态图**

**Inline 模式**（字符串长度 ≤ 15，无堆分配）：

```
SmallString 对象（24 字节，对齐到 8）
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+
| h | e | l | l | o | \0|   |   |   |   |   |   |   |   |   |   |  m_len = 5  |
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+
  └──────────────────────────────────────────────────────────┘
                        ↑
                   c_str() 返回 m_inline 首地址
```

**Heap 模式**（字符串长度 > 15，走堆分配）：

```
SmallString 对象（24 字节）                              堆内存
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+
|  data ptr (8B)  |  capacity (8B)  |      m_len = 20     |
+-----------------+-----------------+---------------------+
        │
        ▼
+-----------------------------------------------------------+
| t | h | i | s |   | i | s |   | a |   | l | o | n | g | ... | \0 |
+-----------------------------------------------------------+
```

> **这个方案还缺什么**：
> - ❌ Union 的 strict aliasing 问题：C++ 标准对 union 活跃成员有严格限制，错误访问是未定义行为。上面的代码为了教学清晰做了简化，工业级实现通常用 `std::aligned_storage` 或手动指针运算
> - ❌ 缺少拷贝/移动构造函数和赋值运算符（Rule of Five）
> - ❌ `m_len` 的存储位置需要精心设计——有些实现把长度放在 `m_inline[15]` 并用高位比特标记是否为堆模式，这样对象可以压缩到 16 字节
> - ❌ 没有自定义分配器接口，无法接入阶段 3.4 的 FrameArena 或 ObjectPool

**代价与适用边界**：
- SSO 消除了小分配，但让字符串对象本身变大了（24 字节 vs `std::string` 的 32 字节可能更小，但比 `const char*` 的 8 字节大得多）
- 如果引擎中绝大多数字符串都很长（如 JSON 配置全文、长资源路径），SSO 反而浪费了内联空间
- **默认推荐**：使用 SSO。80% 的引擎字符串是短字符串，SSO 的收益远超代价。

## 问题 2：如何让字符串比较从 O(n) 降到 O(1)？

SSO 解决了分配问题，但没有解决比较问题。假设你有一万个组件，每帧都要查找类型为 `"MeshRenderer"` 的实体：

```cpp
// 问题：每次比较都要逐字节遍历
for (auto& c : components) {
    if (c.typeName == "MeshRenderer") { ... } // O(n) 逐字节比较
}
```

即使 SSO 让 `c.typeName` 的内联访问很快，逐字节比较 `"MeshRenderer"`（12 字节）在热路径上依然昂贵——而且分支预测失败时会更慢。

### Naive 方案：编译期哈希

```cpp
// 编译期计算字符串哈希
constexpr u64 hashFNV1a(const char* str, u64 h = 0xcbf29ce484222325ULL) {
    return *str == '\0' ? h : hashFNV1a(str + 1, (h ^ static_cast<u8>(*str)) * 0x100000001b3ULL);
}

class StringId {
    u64 m_hash;
public:
    constexpr StringId(const char* str) : m_hash(hashFNV1a(str)) {}
    constexpr bool operator==(StringId o) const { return m_hash == o.m_hash; }
};

// 用法
constexpr StringId MESH_RENDERER("MeshRenderer");
for (auto& c : components) {
    if (c.typeId == MESH_RENDERER) { ... } // O(1) 整数比较！
}
```

这个方案已经很好了：比较从 O(n) 降到 O(1)，编译期常量可以在 switch-case 中使用。但运行时构造怎么办？

### 新问题：运行时字符串的哈希冲突

```cpp
// 运行时从配置文件读到的字符串
std::string path = config.readString("texture_path");
StringId id(path.c_str()); // 运行时哈希
```

FNV-1a 是 64 位哈希，碰撞概率理论上是 1/2^64，可以认为是"不可能"。但在一个确定性的 ECS 引擎中，**"几乎不可能"不等于"安全"**。如果 AI Agent 通过 MCP 接口修改资源名，恰好触发了哈希碰撞，两个不同的资源会被映射到同一个 ID，导致静默错误。

### 改进方案：StringId + Intern 池

```cpp
// 这是 roadmap 阶段 3.3 的推荐实现
class StringId {
    u64 m_hash;
    u32 m_index; // intern 池中的索引（可选，用于二次校验）
public:
    constexpr StringId(const char* str) : m_hash(hashFNV1a(str)), m_index(0) {}
    constexpr u64 hash() const { return m_hash; }
    constexpr bool operator==(StringId o) const {
        // 工业级做法：先比较 hash，再比较 index（如果 index 非零）
        return m_hash == o.m_hash;
    }
};

// Intern 池：运行时将字符串内容存入全局表，返回稳定 ID
class StringInternPool {
    HashMap<u64, const char*> m_pool; // hash -> 原始字符串
    // 注意：实际实现中，池本身需要线程安全（锁或 TLS）
public:
    StringId intern(const char* str) {
        u64 h = hashFNV1a(str);
        auto it = m_pool.find(h);
        if (it != m_pool.end()) {
            // 冲突检测：如果 hash 相同但内容不同，需要处理
            if (std::strcmp(it->second, str) != 0) {
                // 策略 A：崩溃（debug 构建）
                // 策略 B：二次哈希或线性探测
                // 策略 C：存储冲突链表
                assert(false && "StringId hash collision detected!");
            }
            return StringId(h);
        }
        char* copy = new char[std::strlen(str) + 1]; // 实际应用自定义分配器
        std::strcpy(copy, str);
        m_pool[h] = copy;
        return StringId(h);
    }
    const char* resolve(StringId id) const {
        auto it = m_pool.find(id.hash());
        return it != m_pool.end() ? it->second : nullptr;
    }
};
```

**这个方案已经解决了什么、还缺什么**：
- ✅ 编译期 StringId 零分配、O(1) 比较
- ✅ 运行期 intern 池保证唯一性，碰撞可被检测
- ✅ `resolve()` 让 Debug 构建可以将 StringId 反查为可读字符串
- ❌ intern 池是全局状态，在 ECS 中需要以 Resource 形式存在（参考 Bevy 的 TypeRegistry 设计）
- ❌ 线程安全：多线程同时 `intern()` 需要锁，会成为瓶颈。工业级方案使用 TLS 线程本地缓冲 + 批量提交
- ❌ 内存不释放：intern 池只增不减。对于动态生成的唯一字符串（如 `"Entity_12345"`），会造成泄漏

**代价与适用边界**：
- Intern 池消除了碰撞风险，但引入了全局同步开销
- 只在需要运行期构造 StringId 时使用 intern 池。编译期常量（如组件类型名、资源短名）直接用 `constexpr StringId`
- 对于 ECS 组件名、事件名、资源键等"有限枚举"，应在编译期全部注册，运行期只查不增

## 问题 3：如何零分配地格式化字符串？

日志是引擎中最频繁的字符串操作。我们已经用 SSO String 和 StringId 减少了分配和比较成本，但日志格式化本身仍然是热点：

```cpp
// 问题：std::ostringstream 动态分配内部缓冲，虚函数调用林立
LOG("Player {0} pos=({1}, {2})", name, x, y);
```

### Naive 方案：直接用 `snprintf`

```cpp
char buf[256];
snprintf(buf, 256, "Player %s pos=(%f, %f)", name.c_str(), x, y);
```

问题：
- `%f` vs `%lf` 在不同平台行为不同，传错类型是未定义行为
- 不支持自定义类型（如 `Vec3`）的格式化
- 缓冲区大小固定，过长会截断，过短浪费栈空间

### 改进方案：类型安全的栈格式化器

```cpp
// 这是 roadmap 阶段 3.3 的推荐实现
template<typename T>
void toStringBuf(char* buf, usize n, T v) {
    if constexpr (std::is_integral_v<T>) {
        // 整数统一用 %lld 避免 int/long 平台差异
        std::snprintf(buf, n, "%lld", static_cast<long long>(v));
    }
    else if constexpr (std::is_floating_point_v<T>) {
        std::snprintf(buf, n, "%.3f", static_cast<double>(v));
    }
    else if constexpr (std::is_same_v<T, const char*>) {
        std::snprintf(buf, n, "%s", v);
    }
    else if constexpr (std::is_same_v<T, SmallString>) {
        std::snprintf(buf, n, "%s", v.c_str());
    }
    else {
        std::snprintf(buf, n, "<??>"); // 未注册类型的安全回退
    }
}

// 特化：Vec3 自定义格式化
template<>
void toStringBuf(char* buf, usize n, Vec3 v) {
    std::snprintf(buf, n, "(%.2f, %.2f, %.2f)", v.x, v.y, v.z);
}

template<typename... Args>
int formatString(char* out, usize outSize, const char* fmt, Args&&... args) {
    constexpr usize N = sizeof...(Args);
    char argBufs[N][64] = {}; // 每个参数最多 64 字节的栈缓冲
    usize idx = 0;
    (..., (toStringBuf(argBufs[idx], 64, args), ++idx));

    usize pos = 0;
    for (const char* p = fmt; *p && pos < outSize - 1; ++p) {
        if (*p == '{' && *(p+1) >= '0' && *(p+1) <= '9' && *(p+2) == '}') {
            int ai = *(p+1) - '0';
            if (ai < static_cast<int>(N)) {
                for (const char* s = argBufs[ai]; *s && pos < outSize - 1; ++s)
                    out[pos++] = *s;
            }
            p += 2;
        } else {
            out[pos++] = *p;
        }
    }
    out[pos] = '\0';
    return static_cast<int>(pos);
}

// 用法
char buf[256];
formatString(buf, 256, "Player {0} pos={1}, hp={2}",
             "hero", Vec3{1.0f, 2.0f, 3.0f}, 100);
// buf = "Player hero pos=(1.00, 2.00, 3.00), hp=100"
```

**这个方案已经解决了什么、还缺什么**：
- ✅ 零堆分配：所有缓冲在栈上
- ✅ 类型安全：模板推导，不会把 `float` 当 `int` 传
- ✅ 可扩展：通过模板特化支持 `Vec3`、`Quat` 等自定义类型
- ❌ 不支持格式语法（如 `{0:8.2f}` 对齐/精度），需要更复杂的解析器
- ❌ 输出缓冲大小固定，超长内容会截断
- ❌ 变参数量受限于模板展开，不支持运行时变长参数

**代价与适用边界**：
- 这个格式化器适合日志、断言、调试输出等"已知格式+有限参数"的场景
- 对于复杂 UI 文本（多语言、富文本、动态布局），需要更强大的格式化库（如参考 UE 的 FText 体系）
- **默认推荐**：日志和调试用上述栈格式化器；UI 显示文本用 `SmallString` 或未来扩展的富文本系统

## 问题 4：工业级引擎是怎么做的？

现在我们已经从零构建了一个可用的字符串系统。但在做最终决策前，让我们看看工业界如何解决同一个问题——不是为了照搬，而是为了理解"在什么约束下选什么方案"。

> **我们正在解决的具体问题**："引擎需要几种不同用途的字符串表示，每种表示在性能、功能、内存之间做不同取舍。"

### 方案对照：UE 的三级字符串体系

UE 没有用一个类解决所有问题，而是设计了三个专门的字符串类型：

- **FString**：`TArray<TCHAR>` 的封装，可变、可格式化。相当于我们的 `SmallString`，但没有 SSO——UE 选择用分配器参数化（`TInlineAllocator<4>` 可将前几个元素放在栈上）来实现类似效果
- **FName**：全局字符串池的唯一标识。相当于我们的 `StringId + InternPool`，但工业级更强：分块分配（Short/Medium/Large）、无锁读取、大小写不敏感
- **FText**：支持本地化与文本收集。用于 UI 显示文本，内部持有 `FName` 键 + 翻译表引用

UE 的选择告诉我们：**字符串不是单一数据结构，而是根据不同用途分层**。资源路径、组件名用 FName（O(1) 比较）；日志和动态文本用 FString（灵活可变）；UI 文本用 FText（本地化支持）。

### 方案对照：chaos 的 StringFormat 系统

chaos 引擎在基础层自研了格式化系统，其核心设计是：

- 语法风格：`{index:align.width.type}` 占位符，类似 C# / Python
- 零堆分配：`alloca` + placement new，格式化器对象完全在栈上构造
- 类型扩展：模板特化 + 宏注册，新增类型只需十几行代码
- 跨平台封装：统一 MSVC 与 GCC/Clang 的 `snprintf` 行为差异

这与我们的栈格式化器思路一致，但 chaos 的实现更完整：支持对齐、精度、宽度等高级格式语法。代价是代码量更大（单头文件 600+ 行），且使用了 `alloca`（在递归或循环中不安全）。

### 方案对照：Bevy 的 ECS 原生做法

Bevy 作为 ECS 原生引擎，对字符串的处理方式与传统 OOP 引擎截然不同：

- **没有自定义字符串类型**：Bevy 直接复用 Rust 标准库的 `String` 和 `&str`。Rust 的 `String` 本身就有 SSO 优化（但不是 C++ 意义上的 union SSO，而是小容量时的不同布局策略）
- **组件不用字符串标识**：Bevy 用 `TypeId`（编译期生成的唯一类型标识，类似于 `std::type_info::hash_code`）而非字符串来标识组件类型。这从根本上消除了"组件名比较"这个问题
- **资源路径用 AssetPath**：`AssetPath<'static>` 是带有生命周期的字符串引用，配合 `AssetServer` 的 intern 机制，路径只在加载时解析一次，之后用 `Handle<A>`（类型安全的资源句柄）传递
- **反射用 TypeRegistry**：运行时通过 `TypeRegistry` 将 `TypeId` 映射到 `TypeRegistration`，字符串（类型路径）只在注册和序列化时出现，热路径上全是整数比较

Bevy 的方案给我们的启示是：**在 ECS 架构下，很多"字符串问题"可以通过"不用字符串"来解决**。组件类型、资源引用、系统标识，全部用整数 ID 代替字符串。字符串只保留在"人类可读"的边界上（配置、序列化、调试 UI）。

### 个人项目推荐路径

| 用途 | 推荐类型 | 理由 |
|------|---------|------|
| 日志/调试格式化 | 栈格式化器（`formatString`） | 零分配、类型安全、足够简单 |
| 资源路径/组件名/配置键 | `StringId`（编译期 + Intern 池） | O(1) 比较、确定性、可序列化 |
| 动态文本（UI、错误信息） | `SmallString`（SSO） | 小字符串无分配、支持修改 |
| 本地化管理 | 暂不实现 | 单机原型不需要，阶段 7 工具链再扩展 |

**默认路径**：先实现 `StringId` + `SmallString` + 栈格式化器三件套。`StringInternPool` 在 Debug 构建中启用，Release 构建可剥离以节省内存。

## 问题 5：在 ECS 架构下，字符串应该放在哪里？

这是本模块最关键的设计决策，直接关系到后续所有系统的 AI 可观测性和性能。

### ECS 中不该放什么

```cpp
// ❌ 错误：String 组件导致内存布局不紧凑、不可预测
struct NameComponent {
    SmallString name; // 24 字节对象，可能含堆指针
};
```

如果 `NameComponent` 包含 `SmallString`，ECS 的密集数组存储会被破坏：
- 组件数组不再是一个连续的 `NameComponent[N]`，因为某些元素可能指向堆
- 序列化时需要特殊处理指针字段
- AI 观察时无法简单地"把组件数据平铺成 JSON"

### ECS 中该放什么

```cpp
// ✅ 正确：StringId 是纯值类型，8 字节，可密集存储
struct NameComponent {
    StringId nameId; // u64，纯值，无指针
};

// 实际字符串内容存在 ECS Resource（全局状态）中
struct NameRegistry : public ECSResource {
    StringInternPool pool;
    HashMap<Entity, StringId> entityNames;
};
```

这样设计的收益：
- `NameComponent` 是 8 字节纯值，可以完美放入 ECS 的 SoA/Archetype 存储
- 字符串内容只在需要时通过 `NameRegistry` 反查
- 序列化时，`StringId` 可以直接写成 u64 或反查后的字符串
- AI 观察实体时，看到的是一个扁平的整数数组；AI 操作实体时，通过 `NameRegistry` 的接口安全地注册/查询名称

### 从 OOP 到 ECS 的重构映射

如果你的源码分析笔记基于 OOP 引擎（如 UE 的 Actor-Component 树），向 ECS 迁移时：

| OOP 设计 | ECS 重构 |
|---------|---------|
| Actor 的 Name 成员（FString） | `NameComponent { StringId nameId }` + `NameRegistry` Resource |
| 类型名字符串（如 UClass::GetFName） | `ComponentId`（u64）+ `TypeRegistry`（编译期注册） |
| 资源路径（FString 路径） | `AssetPathId`（StringId）+ `AssetRegistry` Resource |
| 日志 CategoryName（FName） | `LogCategoryId`（StringId），编译期常量 |

> **核心原则**：ECS 组件中只存 `StringId`（u64），不存字符串对象。所有字符串内容上浮到 Resource 或外部系统中。

## AI 友好设计红线

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ StringId 可直接作为 ECS 组件 | 8 字节纯值，无指针，序列化/拷贝 trivial |
| **自描述** | ✅ AI 可通过 TypeRegistry 反查名称 | Debug 构建维护 `StringId → const char*` 映射；Release 可剥离 |
| **确定性** | ✅ FNV-1a 哈希是纯算法 | 给定相同输入，输出始终相同。支持快照保存与回放 |
| **工具边界** | 应操作 `StringId` | AI 修改资源名或组件名时，优先操作 `StringId`，原始字符串仅作为初始注册输入 |
| **Agent 安全** | ⚠️ 需防范哈希冲突 | Debug 构建启用 intern 池碰撞检测；关键路径（如资源加载）使用二次校验 |

## 回探替换清单

本模块的产出需要回探替换阶段 1 和阶段 2 中的临时实现：

| 临时实现（阶段 1/2） | 替换为（阶段 3.3） | 验证方法 |
|---------------------|-------------------|---------|
| 日志系统中的 `std::string` 日志内容 | `SmallString` 或栈格式化器输出 | 日志面板正常滚动，Profiler 显示分配次数下降 |
| 阶段 2 的组件名硬编码字符串（如 `"Transform"`） | `StringId` 编译期常量 | Inspector 仍能显示组件标签，且切换为 StringId 后无性能回退 |
| 日志格式化中的 `std::ostringstream` | 栈格式化器 `formatString` | 输出格式一致，无堆分配 |
