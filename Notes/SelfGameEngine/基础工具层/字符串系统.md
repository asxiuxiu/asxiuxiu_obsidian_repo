# 字符串系统

## Why：为什么游戏引擎需要自定义字符串系统？

`std::string` 在普通业务代码里足够好用，但在游戏引擎的热路径上，它是一头隐藏的内存怪兽。

### std::string 在引擎里会有多糟？

假设你有一个 60 FPS 的游戏，每帧打印 100 条日志：
- 每条日志平均 64 字节
- `std::string` 的格式化几乎必然触发堆分配
- 结果：每秒 6000 次小内存分配，GC/碎片/缓存未命中的噩梦

### 三个真实场景

| 场景 | 发生了什么 | 痛点 |
|------|-----------|------|
| **每帧日志格式化** | `LOG("Player {0} pos=({1}, {2})", name, x, y)` | `std::ostringstream` 动态分配内部缓冲，虚函数调用林立 |
| **资源名查找** | `loadTexture("characters/hero_diffuse.dds")` | 相同字符串被反复哈希、比较、分配；std::string 比较是 O(n) 逐字节 |
| **组件类型名比较** | `if (typeName == "MeshRenderer")` | 运行时字符串比较昂贵；类型名往往很长，缓存不友好 |

### 核心痛点

- **堆分配不可控**：短字符串（如 `"x"`、`"id"`）也走堆，64 字节以下的分配占引擎 30% 以上
- **比较成本高**：`std::strcmp` 或逐字节比较，在分支预测失败时尤其慢
- **格式化类型不安全**：`printf` 的 `%d` vs `%lld` 在不同平台行为迥异，传错类型直接 UB
- **跨平台差异**：`snprintf` 返回值、本地化行为、宽字符转换，各编译器各唱各的调

## What：最小字符串系统长什么样？

一个可用的引擎字符串系统由三块积木构成：内联存储的短字符串、编译期/运行期哈希标识符、零堆分配的格式化器。下面是总计约 90 行的最小实现。

### 1. 小型字符串优化（SSO）String 类

```cpp
#include <cstring>
#include <cstdint>

class SmallString {
    static constexpr usize SSO_CAPACITY = 15;
    char  m_inline[SSO_CAPACITY + 1]; // 内联缓冲（16 字节）
    char* m_data;                     // 8 字节
    usize m_len;                      // 8 字节

public:
    SmallString(const char* str = "") {
        m_len = std::strlen(str);
        if (m_len <= SSO_CAPACITY) {
            std::memcpy(m_inline, str, m_len + 1);
            m_data = m_inline;
        } else {
            m_data = new char[m_len + 1];
            std::memcpy(m_data, str, m_len + 1);
        }
    }
    ~SmallString() { if (m_data != m_inline) delete[] m_data; }

    const char* c_str()  const { return m_data; }
    usize       length() const { return m_len; }
    bool        isInline() const { return m_data == m_inline; }
};
```

#### SSO 内存布局状态图

**Inline 模式**（字符串长度 ≤ 15，无堆分配）：

```
SmallString 对象（32 字节）
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+--------+
| h | e | l | l | o | \0|   |   |   |   |   |   |   |   |   |   | m_data ptr |  m_len |
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+--------+
  └──────────────────────────────────────────────────────────┘
                        ↑
                   m_data 指向 m_inline 首地址
```

**Heap 模式**（字符串长度 > 15，走堆分配）：

```
SmallString 对象（32 字节）                              堆内存
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+--------+
|   |   |   |   |   |   |   |   |   |   |   |   |   |   |   |   |  heap ptr  |   20   |
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+------------+--------+
                                                                    │
                                                                    ▼
                                                              +-------------------------+
                                                              | t | h | i | s | ... |\0|
                                                              +-------------------------+
```

> **诚实说明**：这个实现比 union 方案对象大 8 字节（32 字节 vs 24 字节），但完全避免了 C++ union 的严格别名（strict aliasing）和活跃成员未定义行为。对于 vibe coding 的早期阶段，可读性和安全性比极致压缩对象大小更重要。

### 2. StringId / HashedString

```cpp
constexpr u64 hashFNV1a(const char* str, u64 h = 0xcbf29ce484222325ULL) {
    return *str == '\0' ? h : hashFNV1a(str + 1, (h ^ static_cast<u8>(*str)) * 0x100000001b3ULL);
}

class StringId {
    u64 m_hash;
public:
    constexpr StringId() : m_hash(0) {}
    constexpr StringId(const char* str) : m_hash(hashFNV1a(str)) {}
    constexpr u64 value() const { return m_hash; }
    constexpr bool operator==(StringId o) const { return m_hash == o.m_hash; }
    constexpr bool operator!=(StringId o) const { return m_hash != o.m_hash; }
};

constexpr StringId operator"" _sid(const char* str, usize) { return StringId(str); }
// 用法：auto id = "hero_diffuse"_sid;
```

**这段代码已经解决了什么、还缺什么**：
- ✅ 解决了 O(1) 字符串比较、编译期哈希、资源表键类型、switch-case 分支支持
- ❌ 还缺：哈希冲突处理（工业级常用双重校验或 intern 池）、运行期字符串的线程安全注册、序列化时的可读性回退

### 3. 零分配字符串格式化器

下面的实现**没有虚函数、没有 alloca、没有堆分配**，完全在栈上完成：

```cpp
#include <cstdio>
#include <type_traits>

template<typename T>
void toStringBuf(char* buf, usize n, T v) {
    if constexpr (std::is_integral_v<T>)     std::snprintf(buf, n, "%lld", static_cast<long long>(v));
    else if constexpr (std::is_floating_point_v<T>) std::snprintf(buf, n, "%.3f", static_cast<double>(v));
    else if constexpr (std::is_same_v<T, const char*>) std::snprintf(buf, n, "%s", v);
    else std::snprintf(buf, n, "<??>");
}

template<typename... Args>
int formatString(char* out, usize outSize, const char* fmt, Args&&... args) {
    constexpr usize N = sizeof...(Args);
    char argBufs[N][64] = {};
    usize idx = 0;
    (..., (toStringBuf(argBufs[idx], 64, args), ++idx));

    usize pos = 0;
    for (const char* p = fmt; *p && pos < outSize - 1; ++p) {
        if (*p == '{' && *(p+1) >= '0' && *(p+1) <= '9' && *(p+2) == '}') {
            int ai = *(p+1) - '0';
            if (ai < static_cast<int>(N)) {
                for (const char* s = argBufs[ai]; *s && pos < outSize - 1; ++s)
                    out[pos++] = *s;
            }
            p += 2;
        } else {
            out[pos++] = *p;
        }
    }
    out[pos] = '\0';
    return static_cast<int>(pos);
}
```

**这段代码已经解决了什么、还缺什么**：
- ✅ 解决了热点路径零堆分配、类型安全（模板推导）、跨平台统一输出、索引越界不崩溃
- ❌ 还缺：对齐/宽度/精度语法 `{0:8.2f}`、自定义类型的扩展注册机制、wchar_t/UTF-8 支持、返回值长度预计算（当前可能截断）

## How：真实引擎的字符串系统是如何一步一步复杂起来的？

### 阶段 1：直接用 std::string → 能用但有坑

刚开始写引擎原型时，开发者往往直接用 `std::string` 传资源名、组件名、日志内容。项目小、帧率低，一切正常。

但随着功能增加，坑开始显现：
- **小字符串灾难**：`std::string` 通常有 32 字节左右的对象头（指针+长度+容量），短字符串仍然要堆分配，导致 cache line 利用率极低
- **隐式拷贝**：函数参数按值传递时，深层拷贝触发二次分配
- **比较爆炸**：组件查找、事件订阅键匹配时，逐字节比较成千上万个字符

### 阶段 2：SSO + StringId + 自定义格式化 → 好用

当性能分析显示字符串操作进入热点 Top10 时，引擎进入第二阶段：

1. **引入 SSO String**：80% 的字符串长度 < 16，内联后彻底消除小分配；剩余 20% 长路径（如资源全路径）走堆，但次数锐减
2. **引入 StringId**：所有资源表、组件类型字典、配置键改用 `uint64_t` 哈希；比较从 O(n) 降到 O(1)，且可以在编译期生成常量
3. **引入自定义 Formatter**：替代 `std::ostringstream` 和 `printf`；日志路径零堆分配，类型推导避免 `%d`/`%lld` 混淆

### 阶段 3：国际化 + 字符串池 + 反射集成 → 工业级

大型项目还需要：
- **本地化表（Localization）**：`StringId` 指向翻译键，运行时根据语言环境反查对应语言的 `SmallString` 或 UTF-8 文本
- **字符串池（String Interning）**：相同内容的字符串只存一份，减少内存占用；运行期构造的 `StringId` 与 intern 池联动，解决哈希冲突风险
- **反射集成**：类型系统的 `TypeName` 不再是裸字符串，而是反射注册时自动生成的 `StringId`；序列化器用 `StringId` 作为键写入 JSON/Binary，加载时再反查

## AI 友好设计红线

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ❌ String 类不太适合直接作为 ECS 组件 | 它可能含堆指针，导致组件布局不紧凑、序列化/拷贝复杂。建议 ECS 中存储 `StringId`，字符串内容放入资源表或 intern 池 |
| **自描述** | ✅ AI 可通过反查表获得可读名称 | 在 Debug/工具构建中，维护 `StringId → const char*` 的全局映射；Release 构建可剥离以节省内存 |
| **确定性** | ✅ 字符串比较和哈希是纯算法 | FNV-1a 在给定输入下输出确定，不依赖运行时状态，AI 修改资源名后可精确预测下游影响 |
| **工具边界** | 应操作 `StringId` | AI 修改资源名或组件名时，优先操作 `StringId`（或注册新的 `StringId`），而非直接改动原始字符串。原始字符串仅作为 Debug 反查或初始注册时的输入 |
| **Agent 安全** | ⚠️ 需防范哈希冲突 | 两个不同的字符串可能产生相同的 `StringId`（概率极低但非零）。AI 在修改关键资源名时，应使用 intern 池或二次校验，避免误命中 |
