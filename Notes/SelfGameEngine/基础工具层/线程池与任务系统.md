# 线程池与任务系统

## 问题 0：为什么引擎不能直接用 `std::thread`？

现代 CPU 几乎都是多核的。如果你的引擎只有一个主线程在跑，就等于把 70% 的算力直接扔进垃圾桶。

但"开个 `std::thread` 不就好了？"是一种幻觉。想象你要加载一个开放世界的场景，里面有 500 个小资源文件（纹理、模型、配置文件）。如果你为每个文件都创建一个线程：

```cpp
// 最 naive 的做法
for (auto& path : assetPaths) {
    std::thread t([path]() { LoadAsset(path); });
    t.detach();  // 或者 t.join()——但 join 会阻塞循环
}
```

这会带来三个立刻可见的灾难：

1. **线程爆炸**：500 个文件意味着 500 次线程创建和销毁。在大多数操作系统上，创建一个线程需要分配 1~8 MB 的栈空间，还要在内核里走一遍调度器注册流程。创建销毁的开销远超文件读取本身。
2. **竞争失控**：500 个线程同时往控制台打印加载进度，或者同时往一个"已加载计数器"里写 `++loadedCount`。没有同步机制就是数据竞争；加了锁就是 500 个线程抢一把锁，谁也跑不快。
3. **依赖地狱**："先解压纹理 → 再生成 Mipmap → 最后上传 GPU"，这种链式依赖用裸线程表达极其痛苦。你需要手动维护 `std::future`、手动 `.wait()`、手动处理异常——代码很快变成一团无法维护的回调 spaghetti。

**场景绑定**：如果你的游戏在 16ms 内要加载 50 个纹理，串行读取直接把帧时间拉到 200ms，玩家看到的是画面冻结 12 帧。这就是线程池要解决的第一个问题。

所以，引擎需要的是一个**任务系统**——不是"多线程优化"，而是**把计算从时间轴上解耦、合理映射到多核硬件**的基础设施。

## 问题 1：如何避免全局锁成为瓶颈？

### Naive 方案：一个全局任务队列

如果完全不懂，我会写一个固定线程池，所有任务塞进一个 `std::queue`，用 `std::mutex` 保护：

```cpp
class ThreadPool {
    std::vector<std::thread> m_workers;
    std::queue<std::function<void()>> m_tasks;
    std::mutex m_mutex;
    std::condition_variable m_cv;
    bool m_stop = false;

public:
    explicit ThreadPool(usize numThreads) {
        for (usize i = 0; i < numThreads; ++i) {
            m_workers.emplace_back([this]() {
                while (true) {
                    std::function<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(m_mutex);
                        m_cv.wait(lock, [this]() { return m_stop || !m_tasks.empty(); });
                        if (m_stop && m_tasks.empty()) return;
                        task = std::move(m_tasks.front());
                        m_tasks.pop();
                    }
                    task();
                }
            });
        }
    }

    ~ThreadPool() {
        { std::lock_guard<std::mutex> lock(m_mutex); m_stop = true; }
        m_cv.notify_all();
        for (auto& t : m_workers) t.join();
    }

    void Submit(std::function<void()> task) {
        { std::lock_guard<std::mutex> lock(m_mutex); m_tasks.push(std::move(task)); }
        m_cv.notify_one();
    }

    usize WorkerCount() const { return m_workers.size(); }
};
```

**这段代码已经解决了什么**：避免了线程频繁创建销毁；任务复用固定的工作线程；`mutex + condition_variable` 足够原型阶段使用。

**但立刻出现的问题**：当任务提交频率很高（比如每帧提交 1000 个粒子更新任务）时，所有 Worker 线程都在抢同一把 `m_mutex`。锁竞争会让多线程的优势消失殆尽——不是 "N 个核心并行"，而是 "N 个核心排队进一个门"。

更糟糕的是**伪共享**（False Sharing）：如果多个 Worker 同时读写 `m_tasks` 队列内部的数据结构，即使逻辑上没有竞争（因为 mutex 保证了互斥），CPU 缓存行（通常 64 字节）的无效化也会拖累性能。一个 Worker 从队列头部取出一个任务，另一个 Worker 在下一个缓存行预读数据——它们共享的缓存行被反复标记为"脏"，核心之间不断互相通知"我改了，你们作废"。

### 改进方案：每个 Worker 一个本地队列 + Work-Stealing

解决思路很简单：**不要让所有线程抢同一个队列**。让每个 Worker 拥有自己独立的任务队列，提交任务时优先放到某个 Worker 的本地队列。Worker 优先从自己队列取任务；只有当自己队列为空时，才去"偷"其他 Worker 队列尾部的任务。

这就是 **Work-Stealing**（工作窃取）的核心思想。它消除了全局锁竞争，同时天然实现了负载均衡——忙线程的队列不会爆满，闲线程会去帮忙。

但实现一个线程安全的 Work-Stealing 队列并不简单。我们需要一个**双端队列**（Deque），满足以下并发语义：
- 队列的**拥有者线程**在一端 `push` 和 `pop`（称为 "private end"）。
- **其他窃取线程**只在另一端 `steal`（称为 "public end"）。
- 拥有者线程的操作不需要原子操作（或只需轻量同步），窃取线程的操作需要无锁原子操作。

业界最广泛使用的实现是 **Chase-Lev 双端队列**（Chase & Lev, 2005）。它的核心是一个**循环数组**加上两个原子指针：`top`（窃取端）和 `bottom`（拥有者端）。

```
Chase-Lev Deque 的内存布局：

  array (动态扩容的循环缓冲区)
  ┌─────┬─────┬─────┬─────┬─────┬─────┐
  │ T0  │ T1  │ T2  │ T3  │ ... │     │
  └──┬──┴──┬──┴──┬──┴──┬──┴─────┴─────┘
     │     │     │
   steal   │    push/pop
   (top)   │   (bottom)
           │
     其他线程只能从 top 端 steal
     拥有者线程从 bottom 端 push/pop
```

**状态变化图**：任务窃取前后的队列状态

```
初始状态（Worker 0 的本地队列）：
  bottom = 4, top = 1
  [_, T1, T2, T3, T4, _, _, _]
   ^top          ^bottom

Worker 0 自己 pop() 一个任务：
  bottom = 3
  [_, T1, T2, T3, T4, _, _, _]
   ^top      ^bottom
  → Worker 0 执行 T4

Worker 0 的队列空了，bottom == top：
  bottom = 1, top = 1
  [_, T1, _, _, _, _, _, _]
   ^top/bottom

Worker 1 来 steal()：
  CAS(top, 1, 2) 成功
  → Worker 1 偷走 T1 执行
  top = 2, bottom = 1（暂时 top > bottom，表示"窃取冲突中"）
  
Worker 0 回来后看到 bottom < top，知道自己队列确实空了，
去尝试偷其他 Worker 的队列。
```

Chase-Lev 队列的精妙之处在于：**当队列中只有一个元素时，`pop` 和 `steal` 会竞争同一个位置**。此时拥有者线程的 `pop` 先递减 `bottom`，然后和 `top` 比较；如果发现 `bottom <= top`，说明队列已空或正在发生竞争，需要通过一次 `CAS` 来确认。这种设计保证了**无锁**（lock-free）——至少有一个线程能在有限步骤内完成操作。

> **关键细节**：Chase-Lev 队列需要**动态扩容**。当 `bottom - top` 接近数组容量时，拥有者线程会分配一个两倍大的新数组，把旧数据拷贝过去。注意：扩容时**不能**让窃取线程看到中间状态，所以扩容逻辑必须谨慎处理 `top` 和 `bottom` 的可见性。

### 代码演进：从全局队列到 Work-Stealing

下面是简化的 Work-Stealing 线程池核心代码（约 200 行，可直接嵌入引擎）：

```cpp
// flags: -O2 -std=c++20
#include <atomic>
#include <vector>
#include <thread>
#include <functional>
#include <memory>
#include <cassert>

// 简化的 Chase-Lev Work-Stealing Deque
class WorkStealingQueue {
    struct Array {
        usize capacity;
        std::unique_ptr<std::function<void()>[]> buffer;
        explicit Array(usize cap) : capacity(cap), buffer(new std::function<void()>[cap]) {}
        usize mask() const { return capacity - 1; }  // 容量必须是 2 的幂
        std::function<void()>* slot(usize idx) { return &buffer[idx & mask()]; }
    };

    std::atomic<usize> m_top{0};
    std::atomic<usize> m_bottom{0};
    std::unique_ptr<Array> m_array;

public:
    explicit WorkStealingQueue(usize initialCapacity = 256) {
        // 确保容量是 2 的幂
        usize cap = 1;
        while (cap < initialCapacity) cap <<= 1;
        m_array = std::make_unique<Array>(cap);
    }

    // 只有拥有者线程调用
    void Push(std::function<void()> task) {
        usize b = m_bottom.load(std::memory_order_relaxed);
        usize t = m_top.load(std::memory_order_acquire);
        Array* a = m_array.get();

        // 需要扩容？
        if (b - t > a->mask()) {
            auto newArr = std::make_unique<Array>(a->capacity * 2);
            for (usize i = t; i < b; ++i) {
                *newArr->slot(i) = std::move(*a->slot(i));
            }
            a = newArr.get();
            m_array = std::move(newArr);
        }

        *a->slot(b) = std::move(task);
        std::atomic_thread_fence(std::memory_order_release);
        m_bottom.store(b + 1, std::memory_order_relaxed);
    }

    // 只有拥有者线程调用
    std::function<void()> Pop() {
        usize b = m_bottom.load(std::memory_order_relaxed) - 1;
        Array* a = m_array.get();
        m_bottom.store(b, std::memory_order_relaxed);
        std::atomic_thread_fence(std::memory_order_seq_cst);

        usize t = m_top.load(std::memory_order_relaxed);
        std::function<void()> task;

        if (t <= b) {
            task = std::move(*a->slot(b));
            if (t == b) {
                // 最后一个元素，需要与 steal 竞争
                if (!m_top.compare_exchange_strong(t, t + 1,
                        std::memory_order_seq_cst, std::memory_order_relaxed)) {
                    // 竞争失败，已经被偷走了
                    m_bottom.store(b + 1, std::memory_order_relaxed);
                    return nullptr;
                }
                m_bottom.store(b + 1, std::memory_order_relaxed);
            }
            return task;
        } else {
            m_bottom.store(b + 1, std::memory_order_relaxed);
            return nullptr;
        }
    }

    // 其他 Worker 线程调用
    std::function<void()> Steal() {
        usize t = m_top.load(std::memory_order_acquire);
        std::atomic_thread_fence(std::memory_order_seq_cst);
        usize b = m_bottom.load(std::memory_order_acquire);

        if (t < b) {
            Array* a = m_array.get();
            std::function<void()> task = std::move(*a->slot(t));
            if (m_top.compare_exchange_strong(t, t + 1,
                    std::memory_order_seq_cst, std::memory_order_relaxed)) {
                return task;
            }
        }
        return nullptr;
    }
};

class ThreadPool {
    struct Worker {
        WorkStealingQueue queue;
        std::thread thread;
    };

    std::vector<std::unique_ptr<Worker>> m_workers;
    std::atomic<bool> m_stop{false};
    thread_local static Worker* s_localWorker;

public:
    explicit ThreadPool(usize numThreads) {
        for (usize i = 0; i < numThreads; ++i) {
            auto w = std::make_unique<Worker>();
            Worker* pw = w.get();
            w->thread = std::thread([this, pw]() {
                s_localWorker = pw;
                RunWorkerLoop();
            });
            m_workers.push_back(std::move(w));
        }
    }

    ~ThreadPool() {
        m_stop.store(true, std::memory_order_release);
        for (auto& w : m_workers) {
            w->queue.Push(nullptr);  // 用空任务唤醒
            if (w->thread.joinable()) w->thread.join();
        }
    }

    void Submit(std::function<void()> task) {
        // 轮询选择 Worker，简单实现；生产环境可用哈希或随机
        static std::atomic<usize> s_next{0};
        usize idx = s_next++ % m_workers.size();
        m_workers[idx]->queue.Push(std::move(task));
    }

    void SubmitLocal(std::function<void()> task) {
        if (s_localWorker) {
            s_localWorker->queue.Push(std::move(task));
        } else {
            Submit(std::move(task));
        }
    }

private:
    void RunWorkerLoop() {
        while (!m_stop.load(std::memory_order_acquire)) {
            std::function<void()> task = s_localWorker->queue.Pop();
            if (task) {
                task();
                continue;
            }
            // 本地队列为空，尝试窃取
            bool stolen = false;
            for (auto& w : m_workers) {
                if (w.get() == s_localWorker) continue;
                task = w->queue.Steal();
                if (task) {
                    task();
                    stolen = true;
                    break;
                }
            }
            if (!stolen) {
                std::this_thread::yield();
            }
        }
    }
};

thread_local ThreadPool::Worker* ThreadPool::s_localWorker = nullptr;
```

> **这段代码已经解决了什么**：消除了全局锁竞争；Worker 优先消费自己的任务，缓存局部性好；空闲 Worker 自动从其他 Worker 窃取任务，实现负载均衡。
> **还缺什么**：没有任务依赖图；窃取顺序是 O(N) 轮询，Worker 数量多时效率低（可用随机化改进）；`std::function` 有类型擦除开销，工业实现通常用自定义函数指针 + 内联存储；扩容时内存分配不是无锁的。
> **反例警告**：如果任务数量极少（每帧只有 2~3 个）且执行时间极短（< 1μs），线程池的调度开销可能超过并行收益。不要对所有循环都上并行化。

### 引擎对照：工业界怎么做 Work-Stealing？

> **参考：UE / Bevy / 开源方案对「工作窃取队列」是怎么做的？**
>
> 我们在解决的是「如何避免全局锁瓶颈」这个具体问题。
>
> - **UE5 LowLevelTasks**：底层调度器采用 Chase-Lev deque 的变体，每个 Worker 拥有本地队列。新任务通过 `TLS`（线程局部存储）快速定位到当前 Worker 的队列，无需全局锁。当 Worker 空闲时，按特定策略（通常是对称顺序）窃取其他 Worker 的尾部任务。UE5 还支持 **Oversubscription**（超订阅）：当所有 Worker 都因等待锁/事件而阻塞时，自动唤醒备用线程继续处理全局队列中的任务，防止级联死锁。
> - **Bevy（bevy_tasks）**：Rust 生态中的 `ComputeTaskPool` 同样基于工作窃取。它使用 `async_executor` 提供的工作窃取队列，与 `bevy_ecs` 的 `MultiThreadedExecutor` 集成。由于 Rust 的所有权系统，`!Send` 任务会被固定到特定线程执行，这与 C++ 中的线程亲和性设计异曲同工。
> - **开源参考**：Intel TBB、Taskflow、enkiTS 都实现了各自变体的 Chase-Lev deque。Google Filament 的实现不使用 `atomic_thread_fence`，而是完全依赖带有正确 memory order 的原子操作，这让 ThreadSanitizer 不会误报数据竞争。
>
> **默认推荐**：默认采用 Chase-Lev deque 的简化实现（如上代码）。如果你的引擎目标平台包含弱内存序架构（如 ARM），**必须**使用 `std::memory_order_seq_cst` 或经过验证的 fence 模式，否则可能出现难以复现的竞争 bug。在 x86_64 上可以适当放宽到 `acquire/release` 以提升性能，但前提是你要完全理解每条 relaxed 操作的后果。

## 问题 2：如何表达"先 A 后 B 再 C"的任务依赖？

### Naive 方案：`std::future` 链式调用

C++11 提供了 `std::future` 和 `.then()`（C++20），看起来可以表达依赖：

```cpp
auto fA = std::async(std::launch::async, TaskA);
auto fB = std::async(std::launch::async, TaskB);
auto fC = std::async(std::launch::async, [fA, fB]() {
    fA.wait(); fB.wait();
    TaskC();
});
```

**立刻出现的问题**：
1. **C 阻塞等待 A 和 B**，但 C 的任务对象在创建时就已经占用了线程池资源（即使它还没开始执行）。如果线程池只有 4 个线程，而你有 10 个这样的链式任务，很快就会耗尽线程。
2. **复杂 DAG 无法表达**："A → B 和 A → C，然后 B + C → D" 这种分支合并结构，用 Future 链写起来极其痛苦。
3. **大量临时对象**：每个 `.then()` 都创建一个新的 `std::promise/std::future` 对，内存分配和引用计数开销不容忽视。

### 改进方案：任务依赖图（Task Graph）+ 原子依赖计数器

游戏引擎中最自然的依赖表达方式是**有向无环图**（DAG）。每个任务是一个节点，边表示"前置依赖"。任务只有在前置任务全部完成后才能执行。

最简单的实现是**原子依赖计数器**：

```cpp
struct TaskNode {
    std::atomic<u32> remainingPrerequisites{0};  // 还剩多少前置任务未完成
    std::function<void()> work;
    std::vector<TaskNode*> subsequents;          // 后置任务列表

    void OnPrerequisiteCompleted() {
        if (--remainingPrerequisites == 0) {
            // 所有前置完成，将本任务加入调度队列
            Scheduler::Submit(this);
        }
    }

    void Execute() {
        work();
        for (auto* sub : subsequents) {
            sub->OnPrerequisiteCompleted();
        }
    }
};
```

**状态变化图**：A → B, A → C, B + C → D 的执行流程

```
初始状态：
  A.remaining = 0  (无前置，可立即执行)
  B.remaining = 1  (依赖 A)
  C.remaining = 1  (依赖 A)
  D.remaining = 2  (依赖 B 和 C)

Worker 执行 A：
  A.work() 完成
  A 通知 B：B.remaining = 0 → B 入队
  A 通知 C：C.remaining = 0 → C 入队

Worker 执行 B：
  B.work() 完成
  B 通知 D：D.remaining = 1 (还不能执行)

Worker 执行 C：
  C.work() 完成
  C 通知 D：D.remaining = 0 → D 入队

Worker 执行 D：
  D.work() 完成
```

这个方案足够表达任意 DAG，但它有个隐含假设：**任务图在运行时构建**。如果每帧都要重新分配 `TaskNode` 对象，内存分配会成为瓶颈。工业级引擎通常会使用**对象池**或**线性分配器**来批量分配任务节点，一帧结束后一次性释放。

更进一步的优化是**编译期/初始化期预构建任务图**。如果某些依赖关系是固定的（比如"每帧都是：物理 → 动画 → 渲染准备"），你可以在引擎启动时构建好图结构，每帧只更新任务体的函数指针，而不重建节点和边。

### 引擎对照：工业界怎么做任务依赖？

> **参考：UE / Bevy / EnTT 对「任务依赖图」是怎么做的？**
>
> 我们在解决的是「如何表达链式/分支/合并的任务依赖」这个具体问题。
>
> - **UE TaskGraph**：核心抽象是 `FBaseGraphTask`，每个任务持有 `Prerequisites`（前置任务列表）和 `Subsequents`（后置任务列表）。任务完成时调用 `Unlock()`（旧称 `DispatchSubsequents`），原子递减所有后置任务的依赖计数器。UE 的命名线程编码（`ENamedThreads`）也集成在 TaskGraph 中：任务可以指定 `GetDesiredThread()` 返回 `GameThread` 或 `RenderThread`，调度器会将任务路由到正确的线程队列。UE5 的 LowLevelTasks 作为新底层，保留了 TaskGraph 的 API 语义，但底层调度器替换为工作窃取模型——这是一种"接口兼容、实现替换"的优雅演进策略。
> - **Bevy Schedule**：Bevy 的依赖图不是任务级别的，而是 **System 级别**的。开发者通过 `.before()` / `.after()` 声明 System 之间的执行顺序，调度器在初始化期构建 `ScheduleGraph`（包含 `hierarchy` DAG 和 `dependency` DAG），然后通过拓扑排序生成 `SystemSchedule`。Bevy 的独到之处在于**静态分析数据冲突**：每个 System 的 `FilteredAccessSet` 记录了读/写的组件类型，调度器自动推断哪些 System 可以并行、哪些必须串行，无需开发者手动声明所有依赖。
> - **EnTT Graph**：EnTT 提供了独立的 `flow` 模块，允许开发者基于**资源访问模式**构建任务图。你声明每个任务"读"哪些资源、"写"哪些资源，EnTT 自动推导出依赖关系并生成执行图。如果两个任务只读同一个资源，它们可以并行；如果一个读一个写，它们必须串行。这种"以数据为中心"的依赖声明比显式的 `.before()` 更高级，但也更难以调试。
>
> **默认推荐**：默认采用「原子依赖计数器 + 显式前置/后置列表」的简化 Task Graph（如上代码）。这是理解更复杂系统的基础，且实现简单、行为可预测。当你的 System 数量超过 20 个、手动维护依赖变得痛苦时，再引入 Bevy 风格的**静态数据访问分析**来自动推导并行性。

## 问题 3：等待任务完成时，如何避免空转？

### Naive 方案：忙等（Busy-Wait）

```cpp
std::atomic<bool> done{false};
pool.Submit([&done]() { DoWork(); done.store(true); });
while (!done.load()) { std::this_thread::yield(); }
```

**立刻出现的问题**：`while + yield` 看起来比纯自旋好，但它仍然在消耗 CPU 周期。如果等待的任务还没开始执行，当前线程就在无谓地循环检查一个变量。在 16ms 一帧的预算里，忙等几毫秒就意味着帧率直接掉到 50fps。

### 改进 1：条件变量阻塞等待

```cpp
std::mutex mtx;
std::condition_variable cv;
bool done = false;

pool.Submit([&]() {
    DoWork();
    { std::lock_guard<std::mutex> lock(mtx); done = true; }
    cv.notify_one();
});

std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, [&]() { return done; });
```

这解决了忙等的问题——当前线程会被操作系统挂起，不消耗 CPU。但引入了**新的问题**：如果等待的任务还没开始执行，当前线程阻塞了，而线程池里可能还有其他空闲 Worker。这导致"一个线程在睡觉，另一个线程在闲着"的低效局面。

### 改进 2：Retraction（任务回缩）— UE 的核心优化

UE 的任务系统有一个极其精巧的优化：**等待任务时，先尝试把尚未执行的任务"抢回"到当前线程直接执行**。

具体逻辑是：

```
当前线程调用 Wait(TaskX)
  → 检查 TaskX 的状态
    ├─ 如果 TaskX 尚未开始执行（State == Ready/Scheduled）
    │    → "收回" TaskX：将 TaskX 从工作队列中移除
    │    → 在当前线程直接执行 TaskX.work()
    │    → 无需阻塞，零上下文切换
    └─ 如果 TaskX 已在其他线程运行
         → 进入阻塞等待
           ├─ 当前线程是命名线程（如 GameThread）
           │    → 帮助处理该线程队列中的其他任务（ProcessUntilTasksComplete）
           │    → 防止死锁并提高吞吐
           └─ 当前线程是工作线程
                → 通常直接休眠，等待唤醒
```

这个优化之所以重要，是因为游戏引擎中大量的等待发生在**父子任务**场景：一个任务拆分为 8 个子任务，父任务等待所有子任务完成。如果父任务只是阻塞等待，那么执行父任务的 Worker 就被浪费了；但如果父任务能"收回"那些还没开始的子任务并自己执行，就没有任何线程空转。

在自研引擎中，Retraction 可以这样简化实现：

```cpp
class TaskHandle {
    TaskNode* node;
public:
    void Wait(ThreadPool& pool) {
        // 尝试回缩：如果任务还在队列中，直接在当前线程执行
        if (pool.TryRetractAndExecute(node)) {
            return;  // 成功回缩，无需阻塞
        }
        // 回缩失败，任务已在其他线程运行
        // 当前线程帮助执行其他任务，直到本任务完成
        pool.HelpExecuteUntil([this]() { return node->IsCompleted(); });
    }
};
```

> **关键洞察**：Retraction 不是"优化"，而是**正确性保障**。在命名线程（如 GameThread）上等待任务时，如果 GameThread 纯阻塞，而某个被等待的任务又需要回调 GameThread（比如 `ENQUEUE_RENDER_COMMAND`），就会发生死锁——GameThread 在等任务，任务在等 GameThread。UE 的解决方案是：GameThread 等待时**主动参与**任务处理，确保依赖它的任务有机会执行。

### 引擎对照

> **参考：UE 对「等待优化」是怎么做的？**
>
> UE 的 `FTaskBase::Wait()` 内部调用 `TryRetractAndExecute(Timeout)`，这是 UE5 任务系统最重要的性能优化之一。当在命名线程（GameThread）上调用 `Wait()` 时，UE 不会纯阻塞，而是调用 `ProcessThreadUntilIdle(GameThread)` 帮助执行 GameThread 本地队列中的任务。这种"协作式等待"既避免了死锁，又提高了 CPU 利用率。
>
> Bevy 的 `scope_with_executor` 也有类似思想：主线程在 `scope` 内既提交任务，也执行分配给它的任务（通过 `thread_executor`），直到所有任务完成。但 Bevy 的实现更偏向"固定角色"（主线程始终是一个 Worker），而非 UE 的"动态回缩"。
>
> **默认推荐**：在简化版任务系统中，至少实现**两层等待策略**：
> 1. 先尝试忙等一小段时间（如 100μs），适用于任务即将完成的场景；
> 2. 忙等超时后，尝试回缩或帮助执行其他任务；
> 3. 最后才退回到条件变量阻塞。这种分层策略在大多数场景下都能获得接近最优的延迟和吞吐。

## 问题 4：哪些任务必须在特定线程上执行？

### Naive 方案：所有任务都丢给线程池

如果完全不管线程亲和性，会出现什么问题？

想象你在后台线程解码了一张纹理，接下来需要把它上传到 GPU。但 GPU API（如 OpenGL、D3D12、Vulkan）通常要求**所有 GPU 资源操作必须在创建 Context 的线程上执行**。如果你在后台线程直接调用 `glTexImage2D`，轻则 OpenGL 报错，重则程序崩溃。

再举一个例子：你的 ECS 系统允许后台线程通过 `Commands` 创建实体。但如果创建实体涉及内存分配器的状态更新、Entity ID 的分配、组件默认值的初始化——这些操作在没有同步保护的情况下被多个线程同时执行，数据竞争不可避免。

**结论**：引擎必须区分"可以在任意线程执行"的任务和"必须在特定线程执行"的任务。

### 改进方案：命名线程（Named Threads）+ 线程亲和性

核心思想是给线程贴标签，给任务指定标签。调度器只把任务投递到标签匹配的线程队列中。

**UE 的编码设计**非常精巧：它将线程索引、队列索引、任务优先级、线程优先级全部打包到一个 `int32` 中：

```
ENamedThreads 编码（UE 风格，概念性说明）：
  低 8 位（0xFF）：    线程索引（0=GameThread, 1=RenderThread, 0xFF=AnyThread）
  第 8 位（0x100）：   队列索引（0=主队列, 1=本地队列）
  第 9 位（0x200）：   任务优先级（0=普通, 1=高）
  第 10~11 位（0xC00）：线程优先级（0=普通, 1=高, 2=后台）
```

这种紧凑编码的好处是：**调度决策可以在单个整数比较中完成**，无需查表或字符串匹配。

在我们的简化引擎中，可以用枚举 + 轻量结构实现：

```cpp
enum class ThreadAffinity : u8 {
    Any = 0xFF,
    Game = 0,
    Render = 1,
    IO = 2,
};

struct TaskConfig {
    ThreadAffinity affinity = ThreadAffinity::Any;
    u8 priority = 0;  // 0=normal, 1=high
};

class NamedThreadPool {
    ThreadPool m_workers;  // AnyThread 的工作线程池
    std::thread m_gameThread;     // GameThread（实际就是主线程）
    std::thread m_renderThread;   // RenderThread
    // ...

    // 每个命名线程有自己的任务队列
    std::mutex m_gameQueueMutex;
    std::deque<std::function<void()>> m_gameQueue;

public:
    void Submit(TaskConfig config, std::function<void()> task) {
        if (config.affinity == ThreadAffinity::Any) {
            m_workers.Submit(std::move(task));
        } else if (config.affinity == ThreadAffinity::Game) {
            std::lock_guard<std::mutex> lock(m_gameQueueMutex);
            m_gameQueue.push_back(std::move(task));
        }
        // ... Render, IO 类似
    }

    // 由主线程在合适的时机调用（如帧开始或帧结束）
    void ProcessGameThreadQueue() {
        assert(IsOnGameThread());
        std::deque<std::function<void()>> local;
        {
            std::lock_guard<std::mutex> lock(m_gameQueueMutex);
            local.swap(m_gameQueue);
        }
        for (auto& task : local) task();
    }
};
```

> **关键设计**：命名线程的任务队列**不是由该线程自己消费**（除非它在主动 `ProcessQueue`），而是由调度器在其他时机统一投递。例如 GameThread 的队列由主循环在 `"帧更新前"` 统一消费，RenderThread 的队列由渲染循环在 `"渲染命令提交前"` 统一消费。

### 与 ECS 的映射

在 ECS 架构下，"命名线程"的概念可以映射为**System 的执行约束**：

- **AnyThread System**：纯计算逻辑，如批量更新粒子位置、并行计算包围盒。这些 System 可以被任务系统并行调度到任意 Worker。
- **GameThread System**：涉及 ECS 结构变更（创建/销毁实体、添加/移除组件）。这些 System 需要 `&mut World`，必须在主线程串行执行。
- **RenderThread System**：构建渲染命令列表、操作 GPU 资源。必须在渲染线程执行。

Bevy 的做法是：System 函数签名中的 `SystemParam` 隐式决定了它的线程约束。例如 `ResMut<AssetServer>` 是 `!Send` 的，调度器会自动将该 System 固定到主线程（通过 `thread_executor`）。这种"类型系统推导线程约束"的方式比显式标记更不易出错。

## 问题 5：如何自动并行化大循环？

### Naive 方案：手动拆分区间

```cpp
usize chunk = count / 4;
pool.Submit([&]() { for (usize i = 0; i < chunk; ++i) fn(i); });
pool.Submit([&]() { for (usize i = chunk; i < chunk*2; ++i) fn(i); });
// ... 还要处理 count 不能被 4 整除的情况
```

这太痛苦了。而且如果 `fn(i)` 的执行时间差异很大（比如有的粒子在复杂碰撞中，有的在空中自由落体），静态拆分会负载不均——一个 Worker 忙了 10ms，其他三个 1ms 就干完了。

### 改进方案：Parallel For + 自动批量化

```cpp
template<typename Func>
void ParallelFor(ThreadPool& pool, usize count, usize minBatchSize, Func&& fn) {
    usize numWorkers = pool.WorkerCount();
    if (numWorkers == 0 || count <= minBatchSize) {
        // 数据量太小，串行更高效
        for (usize i = 0; i < count; ++i) fn(i);
        return;
    }

    // 计算批次数量：不要拆得过细
    usize numBatches = (count + minBatchSize - 1) / minBatchSize;
    if (numBatches > numWorkers * 4) {
        numBatches = numWorkers * 4;  // 最多 4 倍于 Worker 数的批次
    }
    usize batchSize = count / numBatches;

    std::atomic<usize> nextIndex{0};
    std::atomic<usize> completed{0};

    for (usize b = 0; b < numBatches; ++b) {
        pool.Submit([&nextIndex, &completed, count, batchSize, &fn]() {
            while (true) {
                usize start = nextIndex.fetch_add(batchSize, std::memory_order_relaxed);
                if (start >= count) break;
                usize end = std::min(start + batchSize, count);
                for (usize i = start; i < end; ++i) {
                    fn(i);
                }
            }
            ++completed;
        });
    }

    // 等待所有批次完成（简化版：忙等 + yield）
    while (completed.load() < numBatches) {
        std::this_thread::yield();
    }
}
```

这里有两个关键设计：

1. **`minBatchSize` 防止过度拆分**：如果循环体执行时间极短（比如只是 `positions[i] += velocities[i] * dt`），拆成 1000 个批次意味着 1000 次任务提交和调度开销，远超并行收益。`minBatchSize` 保证每个批次至少处理一定数量元素。

2. **动态索引分配（Dynamic Indexing）**：使用 `std::atomic<usize> nextIndex` 作为全局游标，每个 Worker 完成一个批次后自动领取下一个批次。这解决了负载不均的问题——快的 Worker 会做更多批次，慢的 Worker 做更少。

> **状态变化图**：动态索引分配的执行过程
>
> ```
> 初始：count = 1000, batchSize = 64, numBatches = 16
> nextIndex = 0
>
> Worker 0 领取 [0, 64)：  nextIndex = 64
> Worker 1 领取 [64, 128)：nextIndex = 128
> Worker 2 领取 [128, 192)：nextIndex = 192
> ...
> Worker 0 先完成了，继续领取 [192, 256)：nextIndex = 256
> ```
>
> 注意：这里 `nextIndex` 是唯一的全局竞争点，但 `fetch_add` 是原子操作且竞争频率极低（每批次一次），不会像全局任务队列那样成为瓶颈。

### 引擎对照

> **参考：UE / Bevy 对「并行 For」是怎么做的？**
>
> - **UE `ParallelFor`**：提供 `EParallelForFlags::ForceSingleThread` 和 `EParallelForFlags::Unbalanced` 两种模式。默认使用**静态分割**（`count / numWorkers`），适合负载均匀的场景；`Unbalanced` 标志启用**动态分割**（类似上面的 `nextIndex` 模式），适合负载差异大的场景。UE 还有一个巧妙设计：调用 `ParallelFor` 的当前线程会先执行 `PreWork`，然后"帮助"其他 Worker 执行任务（work-sharing）。这避免了"主线程提交完任务就阻塞等待"的低效局面。
> - **Bevy `par_iter`**：Bevy 的并行迭代不是底层原语，而是基于 `ComputeTaskPool` 的 `Scope` 实现的。`Query::par_iter()` 将查询结果按批次分发给线程池，每个 Worker 处理一个批次。由于 Rust 的所有权系统，批次之间天然没有数据竞争——编译器保证的线程安全。
> - **enkiTS / Intel TBB**：这些开源库提供了更精细的控制，如 `parallel_for` 的粒度提示、范围分割策略（auto / simple / static）。
>
> **默认推荐**：默认采用**动态索引分配**（如上代码），因为它在大多数场景下都能获得良好的负载均衡。只有在确认循环体执行时间极度均匀时（如 SIMD 向量运算），才切换为静态分割以减少原子操作开销。始终提供 `minBatchSize` 参数，默认值设为 64~256（具体取决于循环体复杂度）。

## 问题 6：后台线程如何安全地修改主线程状态？

### Naive 方案：直接加锁修改

后台线程加载完纹理后，直接获取 `std::mutex` 锁，修改 ECS 的 `Material` 组件。

**立刻出现的问题**：
1. **帧时间抖动**：如果锁被后台线程持有了 5μs，主线程在这 5μs 内被阻塞。5μs 听起来不多，但如果每帧有 20 个后台任务在竞争不同的锁，累积起来就是 100μs——占 16ms 帧预算的 0.6%。更糟的是锁竞争不是可预测的，帧时间会不规律地抖动。
2. **死锁风险**：后台线程持有锁 A 等待锁 B，主线程持有锁 B 等待锁 A——经典的死锁。
3. **ECS 中间状态**：后台线程在修改组件的过程中（比如先改了纹理指针，还没改 UV 缩放），主线程的渲染系统可能正好读到这个半成品状态，画出错误的画面。

### 改进方案：主线程回调队列（Command Queue）

核心原则：**后台线程禁止直接修改 ECS 或渲染状态**。它们把"需要主线程执行的操作"投递到一个队列中，由主线程在固定时机统一消费。

```cpp
class MainThreadQueue {
    std::deque<std::function<void()>> m_queue;
    std::mutex m_mutex;

public:
    void Push(std::function<void()> callback) {
        std::lock_guard<std::mutex> lock(m_mutex);
        m_queue.push_back(std::move(callback));
    }

    void ExecuteAll() {
        std::deque<std::function<void()>> local;
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            local.swap(m_queue);
        }
        for (auto& cb : local) cb();
    }

    bool Empty() {
        std::lock_guard<std::mutex> lock(m_mutex);
        return m_queue.empty();
    }
};
```

**为什么 `swap` 是关键**：如果我们在持有锁的情况下逐个执行回调，而某个回调又试图往队列里 Push 新回调，就会发生**递归加锁**（如果 `m_mutex` 不是递归锁，直接死锁；如果是递归锁，性能很差）。`swap` 把队列内容换到局部变量中，立刻释放锁，然后无锁地执行回调。

**状态变化图**：异步加载纹理的完整链路

```
帧 N（主线程）：
  提交后台任务：DecodeTexture("hero_diffuse.png")

帧 N（后台线程）：
  读取文件 → 解码 PNG → 生成像素数据
  Push 回调：[UploadTextureToGPU(tex)]

帧 N+1（主线程，Render 阶段前）：
  ExecuteAll():
    → 调用 UploadTextureToGPU(tex)
    → 修改 ECS Material 组件，标记纹理就绪

主线程永远不会被 fread 阻塞，
后台线程永远不会直接触碰 GPU 或 ECS 状态。
```

### 与 Render Command Queue 的类比

这个模式在渲染领域有更强的版本：**Render Command Queue**。

UE 的 `ENQUEUE_RENDER_COMMAND` 是游戏引擎中最经典的跨线程通信模式之一。GameThread 构建一帧的渲染命令（如"在这个位置画这个网格"），把命令追加到一个追加-only 的队列中。RenderThread 在下一帧开始时读取队列并执行。关键安全保证是：
- Lambda 捕获的数据在 GameThread 侧是**只读**或**已同步**的。
- RenderThread 执行 Lambda 时不会回调 GameThread。
- 命令队列是**追加-only**，RenderThread 不会修改已入队的命令。

这种设计消除了 Game Thread 和 Render Thread 之间的数据竞争，代价是偶尔需要 `FlushRenderingCommands()` 进行显式同步。

在 ECS 引擎中，我们可以把这个思想推广到**所有跨线程状态修改**：
- 后台线程不直接操作 World，而是生成 "Command" 对象。
- 主线程在确定的同步点（如 `"PreUpdate"` 阶段）批量应用这些 Command。
- 这天然支持 **Undo/Redo**——因为所有修改都是通过 Command 对象表达的，可以反向执行。

> **Bevy 的对应概念**：Bevy 的 `Commands` 资源提供了几乎完全相同的语义。System 在并行执行时通过 `Commands` 发起结构变更（生成实体、添加组件），这些变更不会立即生效，而是被缓冲到 `CommandQueue` 中。调度器在合适的时机自动插入 `ApplyDeferred` 同步点，批量应用所有缓冲的 Command。这使得并行 System 可以安全地发起 World 变更，而不需要担心中间状态被其他 System 看到。

## 问题 7：ECS 架构下，任务系统应该如何设计？

前面的六个问题都是"通用任务系统"的范畴。但当你的引擎采用 ECS 架构时，任务系统需要与 ECS 的 System 调度深度融合。

### Naive 方案：每个 System 手动提交任务

```cpp
void UpdatePositionsSystem(World& world) {
    auto query = world.Query<Position, Velocity>();
    ParallelFor(g_threadPool, query.Count(), 64, [&](usize i) {
        auto [pos, vel] = query.Get(i);
        pos.value += vel.value * dt;
    });
}
```

这能工作，但当 System 数量达到几十个时，手动管理哪些 System 可以并行已经不现实。更糟的是，**System A 修改了组件 X，System B 读取了组件 X**——如果它们被手动并行化，就是数据竞争。

### 改进方案：基于数据访问 Mask 的自动并行调度

核心思想：**让 System 自己声明读什么、写什么，调度器自动推导并行性**。

```cpp
// 概念性代码：System 的数据访问声明
struct SystemDescriptor {
    const char* name;
    BitMask readComponents;   // 哪些组件是只读的
    BitMask writeComponents;  // 哪些组件是可写的
    bool isExclusive = false; // 是否需要独占 World（如创建/销毁实体）
    std::function<void(World&)> execute;
};

// 调度器在初始化期分析冲突
void BuildSchedule(const std::vector<SystemDescriptor>& systems) {
    // 1. 拓扑排序：根据显式 before/after 依赖构建 DAG
    // 2. 静态冲突分析：
    //    - 读-读：无冲突，可并行
    //    - 读-写 / 写-写：冲突，必须串行
    // 3. 生成执行批次（Wave）：
    //    Wave 0: SystemA(读X), SystemB(读X)  → 并行
    //    Wave 1: SystemC(写X)                → 串行，等 Wave 0 完成
    //    Wave 2: SystemD(读Y), SystemE(写Z)  → 并行（Y和Z不冲突）
}
```

**状态变化图**：System 调度的 Wave 执行模型

```
System 依赖与冲突分析：

显式依赖：A → C（A 必须在 C 之前）
数据冲突：
  A: 读 Position
  B: 读 Position
  C: 写 Position
  D: 读 Velocity
  E: 写 Health

Wave 0: [A, B, D, E]  （A/B 读 Position 不冲突；D/E 访问不同组件）
        ↓
Wave 1: [C]           （C 写 Position，必须等所有读 Position 的完成）
        ↓
Wave 2: []            （全部完成）
```

### 引擎对照

> **参考：Bevy / UE 对「ECS 任务调度」是怎么做的？**
>
> - **Bevy `MultiThreadedExecutor`**：这是 ECS 任务调度的典范实现。每个 System 的 `FilteredAccessSet` 记录了读/写的 `ComponentId` 和 `ResourceId`。调度器在构建期生成 `SystemSchedule`，预计算每个 System 的冲突位图（`conflicting_systems: FixedBitSet`）。运行时，`can_run()` 的判断变成了一系列位集合运算：`system_meta.conflicting_systems.is_disjoint(&running_systems)`。Bevy 还支持 `exclusive` 系统（需要 `&mut World`）和 `!Send` 系统（必须绑定到主线程），执行器自动串行化这些特殊系统。
> - **UE Tick Group**：UE 不是 ECS 引擎，但它的 Tick 分组机制提供了类似的语义。Actor/Component 的 `TickGroup` 决定了它在哪一帧的哪个阶段执行。同一 Tick Group 内的 Actor 可以通过 `ParallelFor` 并行 Tick，不同 Tick Group 之间按顺序执行。UE 的 `TaskGraph` 是底层支撑：Tick 函数被包装为 `TGraphTask`，根据依赖关系调度到工作线程或命名线程。
> - **EnTT `dispatcher` / `registry.view`**：EnTT 本身不提供内置的并行调度器，但它的 `view` 和 `group` 设计使得用户可以轻松结合外部任务系统（如 enkiTS）实现并行遍历。EnTT 的 `Graph` 模块（实验性）则提供了基于资源访问模式自动构建任务图的能力。
>
> **默认推荐**：在自研 ECS 引擎中，采用 **Bevy 风格的静态冲突分析 + Wave 调度**。原因如下：
> 1. 它把并行性的决策从"运行时动态锁"转移到"初始化期静态分析"，运行时零开销。
> 2. `BitMask` 冲突检测在 System 数量 < 128 时几乎是免费的（一条位运算指令）。
> 3. 显式的 `exclusive` 标记让"结构变更 System"和"数据更新 System"的界限清晰，不易出错。
>
> 实现路径：先支持手动标记 `before`/`after` 依赖 + 自动冲突检测；当 System 数量超过 20 个时，再引入完整的 DAG 拓扑排序和 Wave 生成。

---

> **默认推荐路径总结**：
>
> 线程池与任务系统的工业级默认路径是：**Chase-Lev Work-Stealing 线程池 + 原子依赖计数器 Task Graph + Retraction（任务回缩）+ 命名线程（Game/Render/IO）+ Parallel For 动态索引分配 + 主线程回调队列（Command Queue）**。该路径在 ECS 兼容前提下吸收了以下工业级设计：
> - **UE5 LowLevelTasks 的 Work-Stealing**：Chase-Lev deque 变体，每个 Worker 拥有本地队列，通过 TLS 快速定位，空闲时按对称顺序窃取。支持 Oversubscription（超订阅）防止级联死锁。UE5 已将底层调度器全面替换为该模型。
> - **UE `FTaskBase::Wait()` 的 Retraction**：等待任务时先尝试将未执行任务"抢回"到当前线程直接执行，零上下文切换。命名线程等待时主动帮助处理本地队列任务，既避免死锁又提高 CPU 利用率。这是 UE5 任务系统最重要的性能优化之一。
> - **UE `ENamedThreads` 的紧凑编码**：线程索引、队列索引、任务优先级、线程优先级全部打包到 `int32`，调度决策在单个整数比较中完成。命名线程的任务队列由主循环在固定时机统一消费。
> - **Bevy `MultiThreadedExecutor` 的静态冲突分析**：System 的 `FilteredAccessSet` 记录读/写的组件类型，调度器自动推断并行性。`exclusive` 系统和 `!Send` 系统自动串行化。我们把这套机制作为 ECS 调度的长期目标。
>
> **ECS 映射**：System 的数据访问声明（`readComponents` / `writeComponents` BitMask）在初始化期静态分析冲突，生成 Wave 执行计划。运行时零开销——调度变成了一系列位集合运算。ECS 命令队列（`Commands` 本地累积 + 批量提交）是 RingBuffer 思想在 ECS 结构变更中的核心应用，彻底消除并行 System 的数据竞争。
>
> **条件切换路径**：如果目标平台是单核或双核低端设备，可将 Work-Stealing 退化为单全局队列 + 条件变量（实现更简单，且锁竞争在核心数少时不严重）；如果需要严格确定性回放（网络物理同步），禁用 Work-Stealing，改用固定线程数的轮询分配。
>
> **扩展路径（更极端工业场景）**：
> - **完整 DAG 拓扑排序与 Wave 生成**：当 System 数量超过 20 个时，引入自动 DAG 构建和分层 Wave 调度。
> - **Bevy 风格的静态数据访问分析**：自动推导 System 读/写冲突，无需手动维护依赖。
> - **任务优先级与抢占**：高优先级任务（如音频解码）可抢占低优先级任务的 Worker。
> - **GPU 计算任务调度**：将 Compute Shader 派发纳入统一任务图，管理 CPU→GPU 的依赖和同步。
> - **I/O 完成端口 / io_uring**：在 Windows 上使用 IOCP，在 Linux 上使用 io_uring，取代后台线程轮询，实现真正的异步文件和网络 IO。

## 性能与 trade-off：诚实分析

以下表格只用于总结已经解释过的概念：

| 设计点 | 优势 | 代价 | 反例警告 |
|--------|------|------|----------|
| 全局 Mutex 队列 | 实现简单，行为可预测 | 高并发下锁竞争严重 | 任务提交频率 > 1000/帧时，mutex 成为瓶颈 |
| Chase-Lev Work-Stealing | 无锁、负载均衡、缓存友好 | 实现复杂；窃取可能破坏 cache locality | 任务之间共享 L3 cache 数据时，跨核窃取反而降低性能 |
| 原子依赖计数器 | 表达任意 DAG，无需动态内存 | 任务节点需要预分配；计数器竞争在密集 DAG 中不可忽视 | DAG 深度 > 10 时，级联唤醒的延迟可能累积 |
| Retraction（任务回缩） | 等待零阻塞，避免死锁 | 实现复杂；只有"未执行任务"才能回缩 | 如果所有任务都已在执行，Retraction 退化为普通等待 |
| 命名线程 | 保证 GPU/API 线程安全 | 命名线程空闲时无法帮助执行其他任务 | 不要把纯计算任务绑定到 RenderThread，否则会浪费渲染专用核心 |
| ParallelFor 动态索引 | 负载均衡，适应不均匀任务 | 每批次一次原子 `fetch_add` | 循环体太小（< 100ns）时，调度开销 > 并行收益 |
| 主线程回调队列 | 彻底消除跨线程数据竞争 | 引入至少一帧的延迟 | 对极低延迟要求的操作（如输入响应），不适合走回调队列 |
| ECS 静态冲突分析 | 运行时零开销、编译期可检测错误 | 需要为每种组件分配 ID；动态添加组件类型时需重建掩码 | 如果 System 通过 `void*` 或反射绕过掩码声明，静态分析会失效 |

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 是 | 任务系统的全部状态（队列内容、依赖计数器、完成标志）都可以用 POD 结构表达。任务本身最好是纯函数或闭包，不持有复杂的运行时状态 |
| **自描述** | ⚠️ 需补充 | AI 需要知道当前有多少任务在队列中、有多少线程在工作、System 冲突图的结构。应在调试构建中暴露 `pendingTaskCount`、`activeWorkerCount`、`systemConflictMatrix` 等指标 |
| **确定性** | ⚠️ 需约束 | Work-stealing 的窃取顺序由 OS 线程调度决定，非严格确定性。对于需要回放的网络物理同步，应使用固定线程数、固定任务拆分，并禁用 work-stealing（改用轮询分配） |
| **工具边界** | ✅ 是 | AI 通过 MCP 提交任务的接口很清晰：`submit_task(lambda)`、`parallel_for(range, fn)`。更合适的边界是让 AI 操作高层的 System 调度参数（如调整 `minBatchSize`、增删 System 依赖） |
| **Agent 安全** | ⚠️ 需设计 | 后台 Agent 不应直接提交原始 C++ lambda（不可审计）。应提供白名单机制：Agent 只能提交预定义的任务类型（如 `LoadTextureTask`、`RebuildNavMeshTask`），防止任务队列被恶意占满 |
