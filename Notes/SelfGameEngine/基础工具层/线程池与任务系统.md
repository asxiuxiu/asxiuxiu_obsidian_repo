# 线程池与任务系统

## Why：为什么引擎需要自己的线程池与任务系统？

现代 CPU 几乎都是多核的。如果你的引擎只有一个主线程在跑，就等于把 70% 的算力直接扔进垃圾桶。

但"开个 `std::thread` 不就好了？"同样是一种幻觉。当线程数量随意增长时，以下问题会接踵而至：

- **线程爆炸**：每个资源加载都 `new std::thread`，加载完就 `join`。1000 个小文件意味着 1000 次线程创建和销毁，开销远超文件读取本身。
- **竞争与伪共享**：多个线程无意识地读写同一缓存行（如全局计数器），导致 CPU 核心互相无效化缓存，性能不升反降。
- **回调地狱**：后台线程完成工作后，需要修改主线程的 ECS 状态。如果直接加锁写入，轻则帧时间抖动，重则数据竞争崩溃。
- **任务依赖难以表达**："先解压纹理 → 再生成 Mipmap → 最后上传 GPU"，这种链式依赖用裸线程表达极其痛苦。

### 三个真实场景

| 场景 | 没有线程池会怎样 | 后果 |
|------|----------------|------|
| 加载开放世界场景 | 主线程串行读取 500 个资源文件 | 加载画面卡住 10 秒 |
| 粒子系统更新 | 在主线程遍历 10 万个粒子 | 帧率从 60 掉到 15 |
| 音频解码 | 每帧都在音频线程做 FFT 分析 | 音频线程被阻塞，产生爆音 |

**结论**：任务系统不是"多线程优化"，而是**把计算从时间轴上解耦、合理映射到多核硬件**的基础设施。

## What：最小线程池与任务系统长什么样？

下面给出约 130 行的最小实现，包含：固定线程池、任务队列、并行 For、以及主线程回调投递。

### 1. ThreadPool（固定线程池）

```cpp
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <functional>
#include <vector>

class ThreadPool {
    std::vector<std::thread> m_workers;
    std::queue<std::function<void()>> m_tasks;
    std::mutex m_mutex;
    std::condition_variable m_cv;
    bool m_stop = false;

public:
    explicit ThreadPool(usize numThreads) {
        for (usize i = 0; i < numThreads; ++i) {
            m_workers.emplace_back([this]() {
                while (true) {
                    std::function<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(m_mutex);
                        m_cv.wait(lock, [this]() { return m_stop || !m_tasks.empty(); });
                        if (m_stop && m_tasks.empty()) return;
                        task = std::move(m_tasks.front());
                        m_tasks.pop();
                    }
                    task();
                }
            });
        }
    }

    ~ThreadPool() {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_stop = true;
        }
        m_cv.notify_all();
        for (auto& t : m_workers) t.join();
    }

    void Submit(std::function<void()> task) {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_tasks.push(std::move(task));
        }
        m_cv.notify_one();
    }

    usize WorkerCount() const { return m_workers.size(); }
};
```

**状态变化图**：任务提交与消费

```
初始状态：
ThreadPool(4):
  [Worker0]  [Worker1]  [Worker2]  [Worker3]
     │           │          │          │
     └───────────┴──────────┴──────────┘
              都在 wait 任务
  任务队列：[]

Submit(A):
  任务队列：[A]
  Worker0 被唤醒，取出 A 执行
  任务队列：[]

Submit(B) → Submit(C):
  任务队列：[B, C]
  Worker1 取 B，Worker2 取 C
  任务队列：[]
```

> **这段代码已经解决了什么**：避免线程频繁创建销毁、任务复用固定的工作线程、简单的 mutex + condition_variable 实现足够原型阶段使用。
> **还缺什么**：没有任务窃取（work-stealing），空闲线程只能从全局队列取任务；没有任务依赖图；没有任务取消机制；**全局 mutex 在任务极多时可能成为瓶颈**。
> **反例警告**：如果任务数量极小（如每帧只有 2~3 个）且执行时间极短（< 1μs），线程池的调度开销可能超过并行收益。不要对所有循环都上 `parallel_for`。

### 2. parallel_for（并行 For）

**核心思想**：将一个大数据集拆成若干块，每块作为一个任务提交到线程池。

```cpp
template<typename Func>
void parallel_for(ThreadPool& pool, usize count, Func&& fn) {
    usize numThreads = pool.WorkerCount();
    if (numThreads == 0 || count < 256) {
        // 数据量太小，直接串行更高效
        for (usize i = 0; i < count; ++i) fn(i);
        return;
    }

    usize chunkSize = count / numThreads;
    if (chunkSize == 0) chunkSize = 1;

    std::atomic<usize> completed{0};
    usize start = 0;
    while (start < count) {
        usize end = start + chunkSize;
        if (end > count) end = count;
        pool.Submit([start, end, &fn, &completed]() {
            for (usize i = start; i < end; ++i) fn(i);
            ++completed;
        });
        start = end;
    }

    // 忙等所有任务完成（原型阶段简单实现）
    usize totalTasks = (count + chunkSize - 1) / chunkSize;
    while (completed.load() < totalTasks) {
        std::this_thread::yield();
    }
}
```

> **这段代码已经解决了什么**：将大循环自动拆分为多个子区间并行执行，无需手写线程拆分逻辑。
> **还缺什么**：忙等（busy-wait）会浪费 CPU 周期；没有负载均衡（最后一个 chunk 可能比其他 chunk 大）；没有局部性优化（如按 cache line 对齐拆分）。
> **反例警告**：如果 `fn(i)` 内部有锁、原子操作或共享写，并行 For 的速度可能远低于串行。**并行 For 只适用于无状态、无共享写的纯函数式循环**。

### 3. 主线程回调队列（Main Thread Callback Queue）

**核心思想**：后台线程禁止直接修改 ECS 或渲染状态。它们把"需要主线程执行的操作"投递到一个队列中，由主线程在固定时机（如帧开始或帧结束）统一消费。

```cpp
#include <deque>
#include <functional>

class MainThreadQueue {
    std::deque<std::function<void()>> m_queue;
    std::mutex m_mutex;

public:
    void Push(std::function<void()> callback) {
        std::lock_guard<std::mutex> lock(m_mutex);
        m_queue.push_back(std::move(callback));
    }

    void ExecuteAll() {
        // 先 swap 出来，避免持有锁期间执行回调
        std::deque<std::function<void()>> local;
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            local.swap(m_queue);
        }
        for (auto& cb : local) cb();
    }

    bool Empty() {
        std::lock_guard<std::mutex> lock(m_mutex);
        return m_queue.empty();
    }
};
```

**状态变化图**：异步加载纹理的完整链路

```
帧 N（主线程）：
  Submit 任务：解码 PNG

帧 N（后台线程）：
  读取文件 → 解码 PNG → 生成像素数据
  Push 回调：[UploadTextureToGPU(tex)]

帧 N+1（主线程，Render 阶段前）：
  ExecuteAll():
    → 调用 UploadTextureToGPU(tex)
    → 修改 ECS Material 组件，标记纹理就绪

主线程永远不会被 fread 阻塞，
后台线程永远不会直接触碰 GPU 或 ECS 状态。
```

> **这段代码已经解决了什么**：彻底消除了后台线程与主线程之间的数据竞争风险；主线程可以在确定性时间点统一处理副作用。
> **还缺什么**：没有回调的优先级排序；没有超时/丢弃机制（如果主线程一帧内消费不完，队列会无限增长）。

## How：真实引擎的任务系统是如何一步一步复杂起来的？

### 阶段 1：裸 ThreadPool + parallel_for → 能用

这就是上面的代码。它解决了"多核不要浪费"的问题，已经能让资源加载和批量计算移到后台。

### 阶段 2：Job System + 依赖图 + 工作窃取 → 好用

**触发原因**：任务数量激增、任务之间有依赖关系、某些工作线程长期空闲而另一些队列爆满。

**代码层面的变化**：

1. **每个线程一个任务队列**：取代全局 mutex 队列。线程优先从自己队列取任务，空闲时从其他线程队列"窃取"（work-stealing）任务。这消除了全局锁竞争。
2. **Job 句柄 + 依赖计数器**：每个 Job 知道自己有多少前置 Job 未完成。前置 Job 完成时，原子递减计数器；计数器归零时，自动将当前 Job 加入队列。

```cpp
struct Job {
    std::atomic<u32> unfinishedJobs{1}; // 包含自己
    std::function<void()> work;
    Job* parent = nullptr;
    // 子任务完成时调用 Continue()
    void Continue() {
        if (--unfinishedJobs == 0) {
            if (parent) parent->Continue();
            // 将本 job 加入工作队列
        }
    }
};
```

3. **fiber / 协程支持**：某些引擎（如 Naughty Dog 的引擎）使用用户态线程（fiber）实现任务的无栈切换，工作线程可以在等待 IO 时立即切换到其他任务，而不阻塞 OS 线程。

### 阶段 3：与 ECS 调度深度融合 → 工业级

**触发原因**：System 数量达到几十个，手动管理哪些 System 可以并行已经不现实。

1. **基于组件读写 Mask 的依赖分析**：
   - 每个 System 声明自己"读"哪些组件、"写"哪些组件。
   - 调度器自动分析：读-读之间可以并行；读-写之间必须串行；写-写之间必须串行。
   - 生成依赖图后，用拓扑排序（Kahn 算法）将 System 分组，组内通过 Job System 并行执行。

2. **同步点（Sync Point）**：
   - 在 "Update" 和 "Render" 等大阶段之间设置显式同步点，确保所有并行任务完成后再进入下一阶段。
   - 这避免了隐式的 `std::barrier` 滥用，让时序完全可控。

这部分的详细设计见 [[系统调度与确定性]]。

## 性能与 trade-off：诚实分析

| 设计点 | 优势 | 代价 | 反例警告 |
|--------|------|------|----------|
| 固定 ThreadPool | 避免线程创建销毁开销 | 线程数需预先设定，过多会加剧上下文切换 | 线程数 > 物理核心数时，收益递减甚至为负 |
| parallel_for | 自动拆分大循环，代码简洁 | 拆分粒度不当会导致负载不均衡 | 循环体太小（< 1μs）时，调度开销 > 并行收益 |
| 主线程回调队列 | 消除跨线程数据竞争 | 引入一帧的延迟 | 对极低延迟要求的操作（如输入响应），不适合走回调队列 |
| Work-stealing | 消除全局锁，负载均衡 | 实现复杂；窃取可能破坏 cache locality | 任务之间共享 L3 cache 数据时，跨核窃取反而降低性能 |

## AI 友好设计红线检查

| 检查项          | 结论     | 说明                                                                                                                           |
| ------------ | ------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **状态平铺**     | ✅ 是    | 任务系统本身不持有复杂的运行时状态，只有任务队列和原子计数器。Job 的输入输出最好是 POD 或指针                                                                          |
| **自描述**      | ⚠️ 需补充 | AI 需要知道当前有多少任务在队列中、有多少线程在工作。应在调试构建中暴露 `pendingTaskCount`、`activeWorkerCount` 等指标                                             |
| **确定性**      | ⚠️ 需约束 | 并行 For 的拆分顺序和线程调度由 OS 决定，非严格确定性。对于需要回放的系统（如物理），应使用固定线程数、固定任务拆分，并记录任务完成顺序                                                     |
| **工具边界**     | ✅ 是    | AI 通过 MCP 提交任务的接口很清晰：`submit_task(lambda)`、`parallel_for(range, fn)`。但 fn 必须是 C++ 代码，AI 无法直接生成。更适合的边界是让 AI 操作高层的 System 调度参数 |
| **Agent 安全** | ✅ 是    | 通过主线程回调队列隔离，后台 Agent 不会直接破坏 ECS 状态。但仍需限制 Agent 能 Submit 的任务类型，防止任务队列被恶意占满                                                    |
