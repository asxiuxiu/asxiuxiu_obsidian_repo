# 字符串格式化

## 问题 0：为什么引擎不能直接用 `snprintf` 或 `std::ostringstream`？

日志是引擎中最频繁的字符串操作。我们已经用 SSO `String` 和 `StringId` 减少了分配和比较成本，但日志格式化本身仍然是热点：

```cpp
// 问题 1：std::ostringstream 动态分配内部缓冲，虚函数调用林立
std::ostringstream oss;
oss << "Player " << name << " pos=(" << x << ", " << y << ")";
logger.info(oss.str());

// 问题 2：snprintf 类型不安全
char buf[256];
snprintf(buf, 256, "Player %s pos=(%f, %f)", name.c_str(), x, y);
// 如果 name 是 std::string 忘记调 .c_str() → 未定义行为
// 如果 x 是 double 但用了 %f → 某些平台行为不同
```

引擎格式化器的核心矛盾：**类型安全 vs 零开销 vs 模块边界干净**。`snprintf` 零开销但类型不安全，`std::ostringstream` 类型安全但开销大。

## 问题 1：类型安全的栈格式化器

先思考一个架构问题：`formatString` 是字符串模块的基础设施，但它需要知道 `Vec3` 怎么格式化。如果我们在字符串模块里写死 `Vec3` 的格式化逻辑，字符串模块就必须 `#include "Vec3.h"`——底层模块依赖上层模块，方向反了。

工业级引擎不会这么做。正确的依赖方向是：**上层模块 include 底层模块，并为自己的类型扩展格式化能力**。C++ 中实现这一点的关键技术是 **ADL（Argument Dependent Lookup）**。

> [!tip] ADL 是什么？
>
> **ADL（Argument Dependent Lookup，参数依赖查找）** 是 C++ 的名称查找规则：当你调用一个**不写命名空间前缀**的函数时，编译器除了在当前作用域查找，还会去**每个实参所属的类型命名空间**里查找同名函数。
>
> **类比理解**：你在 `Foundation` 部门喊"谁会修电脑？"，普通查找只问本部门；但 ADL 会拿着你手里的设备，去**设备所属部门**也问一遍。如果你手里拿的是 `Math` 部门的 `Vec3`，编译器就会去 `Math` 命名空间找 `toStringBuf`。
>
> **为什么这里必须用 ADL？** 如果 `formatString` 里把 `toStringBuf` 写成 `Foundation::toStringBuf`，那它永远只会调用 `Foundation` 命名空间里的版本，永远找不到 `Math::toStringBuf`。`Math` 模块想为自己的 `Vec3` 提供格式化能力也无从下手——除非 `Foundation` 反过来 `#include "Vec3.h"`，那依赖方向就反了。
>
> 所以代码里**故意不写命名空间前缀**，靠 ADL 让编译器根据实参的类型自动"分发"到正确的命名空间。看下面代码时，重点留意 `formatString` 里那个裸的 `toStringBuf(...)` 调用。

```cpp
// Foundation/StringFormat.h —— String 模块只定义基础设施和基础类型
namespace Foundation {

// 基础类型的 toStringBuf（String 模块的"本职"）
inline void toStringBuf(char* buf, usize n, int v)           { std::snprintf(buf, n, "%d", v); }
inline void toStringBuf(char* buf, usize n, long long v)     { std::snprintf(buf, n, "%lld", v); }
inline void toStringBuf(char* buf, usize n, float v)         { std::snprintf(buf, n, "%.3f", static_cast<double>(v)); }
inline void toStringBuf(char* buf, usize n, double v)        { std::snprintf(buf, n, "%.3f", v); }
inline void toStringBuf(char* buf, usize n, const char* v)   { std::snprintf(buf, n, "%s", v); }
inline void toStringBuf(char* buf, usize n, StringView v)    { std::snprintf(buf, n, "%.*s", (int)v.length(), v.data()); }

// fallback：如果 ADL 没找到类型专属的 toStringBuf，编译在这里失败
template<typename T>
void toStringBuf(char* buf, usize n, T v);

template<typename... Args>
int formatString(char* out, usize outSize, const char* fmt, Args&&... args) {
    constexpr usize N = sizeof...(Args);
    char argBufs[N][64] = {}; // 每个参数最多 64 字节的栈缓冲
    usize idx = 0;
    // ADL 关键：toStringBuf 是非限定名调用，编译器会在参数类型的命名空间中查找重载
    (..., (toStringBuf(argBufs[idx], 64, args), ++idx));

    usize pos = 0;
    for (const char* p = fmt; *p && pos < outSize - 1; ++p) {
        if (*p == '{' && *(p+1) >= '0' && *(p+1) <= '9' && *(p+2) == '}') {
            int ai = *(p+1) - '0';
            if (ai < static_cast<int>(N)) {
                for (const char* s = argBufs[ai]; *s && pos < outSize - 1; ++s)
                    out[pos++] = *s;
            }
            p += 2;
        } else {
            out[pos++] = *p;
        }
    }
    out[pos] = '\0';
    return static_cast<int>(pos);
}

} // namespace Foundation
```

```cpp
// Math/Vector.h —— Math 模块依赖 String 模块，为自己的类型提供格式化
#pragma once
#include "Foundation/StringFormat.h"

namespace Math {

struct Vec3 { float x, y, z; };

// Vec3 的格式化由 Math 模块提供，String 模块完全不感知 Vec3 的存在
inline void toStringBuf(char* buf, usize n, Vec3 v) {
    std::snprintf(buf, n, "(%.2f, %.2f, %.2f)", v.x, v.y, v.z);
}

} // namespace Math
```

**为什么这样就不会有循环依赖？**

当用户在 Gameplay 代码中写：

```cpp
#include "Math/Vector.h"
char buf[256];
Foundation::formatString(buf, 256, "Player {0} pos={1}, hp={2}",
                         "hero", Math::Vec3{1.0f, 2.0f, 3.0f}, 100);
```

`formatString` 实例化时，内部调用 `toStringBuf(argBufs[idx], 64, vec3)`。由于 `vec3` 的类型是 `Math::Vec3`，**ADL 会自动到 `Math` 命名空间中查找** `toStringBuf` 的重载。

```
依赖方向（正确）：
    String 模块（底层）
        ↑ 被 include
    Math 模块（上层）—— 提供 Math::toStringBuf
        ↑ 被 include
    Gameplay 模块 —— 调用 formatString("...", vec3)
```

这个教学实现的优势：
- ✅ 零堆分配：所有缓冲在栈上
- ✅ 类型安全：模板推导，不会把 `float` 当 `int` 传
- ✅ 可扩展：通过 ADL 支持自定义类型，**无需修改 String 模块**

缺陷也很明显：
- ❌ 不支持格式语法（如 `{0:8.2f}` 对齐/精度）
- ❌ 输出缓冲大小固定，超长内容会截断
- ❌ 变参数量受限于模板展开

下面来看工业级实现是怎么解决这些问题的。

## 工业实现深度：fmtlib 的三层架构

fmtlib（C++20 `std::format` 的基础）是现代 C++ 格式化库的工业标杆。它的架构分为三层：

```
Layer 1: 类型感知接口（编译期）
    fmt::format("Player {} pos={}", name, pos)
    ↓
    format_string 编译期解析格式字符串
    ↓
    format_arg_store 存储类型化的参数引用

Layer 2: 类型擦除接口（运行期）
    format_args = make_format_args(args...)
    ↓
    每个参数被擦除为 {type tag, data pointer} 的 variant
    ↓
    vformat(fmt, format_args)  // 非模板函数，减少代码膨胀

Layer 3: 格式化引擎
    解析格式字符串 → 查找对应参数 → 调用该类型的 formatter::format
    ↓
    输出到 buffer（栈缓冲 + 动态增长）
```

### 层次 1：编译期格式字符串解析

C++20 `std::format` 的核心优化是：**格式字符串在编译期被解析成高效的指令序列**，运行期只做数据填充。

```cpp
// C++20：格式字符串在编译期检查
template<typename... Args>
std::string format(format_string<Args...> fmt, Args&&... args);

// format_string 的 consteval 构造函数
template<typename... Args>
struct format_string {
    consteval format_string(const char* s) {
        // 编译期解析格式字符串，检查：
        // 1. 语法是否正确（{} 是否匹配）
        // 2. 参数数量是否匹配
        // 3. 参数类型是否与格式说明符兼容
        parse_format_string<Args...>(s);
    }
    const char* str;
};

// 用法：编译期报错，不是运行期崩溃
auto s = std::format("{:d}", 3.14);  // 编译错误！float 不能用 :d
```

**关键洞察**：编译期解析后，运行期不需要再扫描 `"{...}"` 语法。格式字符串被转换成一组"指令"（如"输出参数 0，宽度 8，精度 2，格式 f"），运行期直接按指令执行。

### 层次 2：类型擦除

```cpp
// 类型感知入口（模板，编译期展开）
template<typename... Args>
void log(const char* file, int line, format_string<Args...> fmt, Args&&... args) {
    vlog(file, line, fmt.str, make_format_args(args...));
}

// 类型擦除后的接口（非模板，减少代码膨胀）
void vlog(const char* file, int line, string_view fmt, format_args args) {
    auto msg = vformat(fmt, args);
    writeToSink(msg);
}
```

**类型擦除的内部表示**：

```cpp
// 每个参数被包装成 2 个指针大小的结构
struct format_arg {
    enum type { int_type, uint_type, float_type, string_type, ... } type_tag;
    union {
        int i;
        uint64_t u;
        double d;
        string_view s;
        void* custom;
    } value;
    const void* formatter_vtable;  // 指向该类型的 formatter 函数
};
```

**为什么类型擦重要？**
- `log("pos={}", x)` 和 `log("pos={}", y)` 如果 x 和 y 类型不同，会生成两个模板实例
- 但它们的 `vlog` 调用是同一个非模板函数
- 日志系统有成百上千个调用点，类型擦除显著减少二进制体积

### 层次 3：basic_memory_buffer（栈缓冲 + 动态增长）

```cpp
template<typename T, size_t INLINE_SIZE = 512>
class basic_memory_buffer {
    T inline_buffer_[INLINE_SIZE];  // 栈上预分配
    T* ptr_;
    size_t size_;
    size_t capacity_;

public:
    basic_memory_buffer() : ptr_(inline_buffer_), size_(0), capacity_(INLINE_SIZE) {}
    
    void push_back(T c) {
        if (size_ < capacity_) {
            ptr_[size_++] = c;
        } else {
            grow();
            ptr_[size_++] = c;
        }
    }
    
private:
    void grow() {
        size_t new_cap = capacity_ * 2;
        T* new_ptr = new T[new_cap];
        memcpy(new_ptr, ptr_, size_ * sizeof(T));
        if (ptr_ != inline_buffer_) delete[] ptr_;
        ptr_ = new_ptr;
        capacity_ = new_cap;
    }
};
```

这是"SSO 思想在格式化输出中的应用"：小日志（< 512 字节）完全在栈上完成，只有超长日志才触发堆分配。

## 引擎在格式化上的具体方案

| 引擎 | 格式化系统 | 语法风格 | 类型扩展机制 | 关键特征 |
|------|-----------|---------|------------|---------|
| **UE** | `FString::Printf` | **C 风格 `%s`、`%f`** | 自定义类型通过 `LexToString(const FVector&)` 或 `ToString()` 成员函数提供格式化能力。日志输出时写 `UE_LOG(LogTemp, Log, TEXT("Pos=%s"), *LexToString(Vec))` | **底层模块不感知上层类型**。FString::Printf 不支持直接传入 FVector 等自定义对象，类型的格式化能力由类型所在模块提供 |
| **chaos** | 自研 `StringFormat` | **`{index:align.width.type}`** 占位符，类似 C# / Python | **模板特化 + 宏注册**。`CHAOS_STRING_FORMATTER_ATOM_TYPE_REGISTER` 注册原子类型，`CHAOS_STRING_FORMATTER_CLASS_REGISTER` 注册类类型。这些宏在类型所在模块的头文件中被调用（如 `Vector3.h` 中注册 `Vector3` 的格式化器） | `base` 层的 `string_format.h` 只提供宏定义和调度基础设施。零堆分配：用 `alloca` + placement new，格式化器对象完全在栈上构造。单头文件 600+ 行 |
| **Bevy** | Rust `std::fmt` | **`format!("{}", vec3)`** | **Display trait**。Rust 标准库要求类型自己实现 `std::fmt::Display` trait 来决定如何格式化为字符串。`format!("{}", vec3)` 会查找 `Vec3` 的 `Display` 实现 | 类型的格式化能力由类型自己提供，格式化器模块不感知具体类型。这与 UE 的 `LexToString`、chaos 的宏注册、以及我们 ADL 自由函数的设计是同一条架构原则 |

**关键观察**：
- 三个引擎在"类型格式化由谁提供"这个问题上达成了完全一致：**类型所在模块/类型自己提供，格式化基础设施不感知具体类型**
- UE 用 C 风格 `%s` + `LexToString`，chaos 用 `{index}` + 宏注册，Bevy 用 Display trait——形式不同，但架构原则相同
- fmtlib/C++20 std::format 是现代 C++ 的标准答案：编译期检查 + 类型擦除 + 栈缓冲

## 自研引擎格式化器的推荐路径

### 路径 A：C++20 标准（推荐）

如果目标 C++ 标准 ≥ C++20：**直接使用 `std::format`**，不要自研。`std::format` 已经是工业级实现。

```cpp
#include <format>

// 编译期检查格式字符串和参数类型
auto msg = std::format("Player {} pos=({:.2f}, {:.2f})", name, x, y);

// 自定义类型的 formatter 特化
template<>
struct std::formatter<Vec3> {
    constexpr auto parse(std::format_parse_context& ctx) { return ctx.begin(); }
    auto format(const Vec3& v, std::format_context& ctx) const {
        return std::format_to(ctx.out(), "({:.2f}, {:.2f}, {:.2f})", v.x, v.y, v.z);
    }
};
```

### 路径 B：C++11/14/17（fmtlib）

如果目标 C++ 标准 < C++20：**使用 fmtlib**（header-only，C++11 起支持）。

```cpp
#include <fmt/format.h>

auto msg = fmt::format("Player {} pos=({:.2f}, {:.2f})", name, x, y);

// 类型擦除接口（减少二进制膨胀）
void vlog(fmt::string_view fmt, fmt::format_args args) {
    auto msg = fmt::vformat(fmt, args);
    writeToSink(msg);
}

template<typename... Args>
void log(fmt::format_string<Args...> fmt, Args&&... args) {
    vlog(fmt, fmt::make_format_args(args...));
}
```

### 路径 C：特殊约束下自研（嵌入式/无外部依赖）

如果因为特殊约束必须自研：

```cpp
// 最小工业级格式化器：吸收 fmtlib 的核心设计
namespace Foundation {

// 1. 编译期格式字符串解析（C++14 constexpr，C++20 consteval）
template<typename... Args>
struct FormatString {
    const char* str;
};

// 2. 类型擦除参数存储
class FormatArgs {
    struct Arg { uint32_t type; const void* data; };
    Arg args_[MAX_ARGS];
    uint32_t numArgs_;
public:
    template<typename... T>
    FormatArgs(T&&... vals) { /* 擦除类型 */ }
};

// 3. 栈缓冲输出
class FormatBuffer {
    char inline_[512];
    char* ptr_;
    size_t cap_;
public:
    void append(StringView s);
    void appendInt(int64_t v, int base = 10);
    void appendFloat(double v, int precision);
};

// 4. 类型感知的公开接口 + 类型擦除的内部实现
template<typename... Args>
int formatTo(char* buf, size_t n, FormatString<Args...> fmt, Args&&... args) {
    FormatBuffer buffer(buf, n);
    FormatArgs fargs(args...);
    vformatTo(buffer, fmt.str, fargs);  // 非模板，减少膨胀
    return buffer.size();
}

} // namespace Foundation
```
