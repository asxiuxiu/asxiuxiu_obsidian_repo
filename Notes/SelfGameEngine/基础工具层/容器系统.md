# 容器系统

上一篇我们解决了"内存从哪里来"的问题——`FrameArena` 给临时数据、`ObjectPool` 给固定对象、`Mimalloc` 给通用小对象。但内存只是空地，**数据怎么组织和访问**才是直接影响性能的关键。

游戏引擎和通用程序对容器的需求完全不同：通用程序关心"能不能用"，引擎关心"遍历 10 万个元素时 cache 会不会爆"。一个 `std::unordered_map` 在业务代码里查几百次键毫无问题，但在引擎里遍历所有碰撞体时每跳一次指针都可能触发 cache miss，直接把帧时间从 8ms 拉到 15ms。

这一篇我们专注"数据组织"——从最简单的动态数组开始，一路走到 ECS 核心的 SparseSet 和 SoA。

---

## 问题 0：为什么引擎不能直接用 STL 容器？

初学者搭引擎时，动态数组用 `std::vector`，键值查找用 `std::unordered_map`。原型阶段没问题，但 STL 容器的三个设计与游戏引擎冲突：

1. **分配器不可插拔**：`std::vector` 的 `Allocator` 模板参数虽然存在，但标准分配器接口复杂（`rebind`、`construct`、`destroy` 等），且实际游戏中你需要让临时数组从帧内存分配、持久数组从全局堆分配——`std::vector` 做不到简洁的切换。我们在 [[内存分配器]] 中已经搭建了 `IAllocator` 接口，但 STL 容器无法直接使用它。

2. **缓存不友好**：`std::list` 是指针跳转链，`std::unordered_map` 是 bucket + 链表/树，遍历时 cache miss 极高。而引擎大量依赖"批量处理"——遍历所有粒子、所有碰撞体、所有 DrawCall。如果容器本身在内存里七零八落，批量处理就变成批量 cache miss。

3. **异常安全冲突**：游戏引擎通常禁用异常（`-fno-exceptions`），但 STL 在内存不足时会抛 `std::bad_alloc`。编译开关和库实现一旦不一致，就是链接灾难。

**场景绑定**：ECS 组件批量分配。如果你写 `new ComponentA[1024]`，构造会全部执行一次，但无法按 Entity Chunk 对齐，碎片随机分布。遍历这些组件时，cache miss 会直接吃掉帧时间。

有了这些痛点，我们开始逐个解决具体问题。

## 问题 1：为什么 `std::vector` 不够用？

### 最 naive 的方案

```cpp
std::vector<Entity> entities;
entities.push_back(e);
```

这在原型阶段完全够用。但进入生产环境后，`std::vector` 的三个设计决策与游戏引擎冲突：

1. **分配器不可插拔**：`std::vector` 的 `Allocator` 模板参数无法简洁地接入我们在 [[内存分配器]] 中定义的 `IAllocator` 接口。
2. **增长策略固定**：`std::vector` 的增长因子由实现定义（通常是 2x 或 1.5x），你无法针对引擎场景调优。
3. **异常安全冲突**：`std::vector` 在 `push_back` 失败时抛 `std::bad_alloc`。游戏引擎通常编译时禁用异常，这会导致链接失败或未定义行为。

### 改进：Array<T, Allocator>

```cpp
template <typename T, typename Allocator = DefaultAllocator>
struct Array {
    T* data = nullptr;
    usize count = 0;
    usize capacity = 0;
    Allocator allocator;

    void Reserve(usize newCap) {
        if (newCap <= capacity) return;
        T* newData = static_cast<T*>(
            allocator.Allocate(newCap * sizeof(T), alignof(T))
        );
        for (usize i = 0; i < count; ++i) {
            new (newData + i) T(static_cast<T&&>(data[i]));
            data[i].~T();
        }
        if (data) allocator.Free(data);
        data = newData;
        capacity = newCap;
    }

    void Push(const T& item) {
        if (count >= capacity) {
            usize newCap = capacity == 0 ? 4 : capacity + (capacity >> 1) + 16;
            Reserve(newCap);
        }
        new (data + count) T(item);
        ++count;
    }

    void Pop() {
        if (count > 0) {
            --count;
            data[count].~T();
        }
    }

    void Clear() {
        for (usize i = 0; i < count; ++i) data[i].~T();
        count = 0;
    }

    T& operator[](usize i) { return data[i]; }
    const T& operator[](usize i) const { return data[i]; }
    usize Size() const { return count; }
    usize Capacity() const { return capacity; }
    bool Empty() const { return count == 0; }
};
```

**状态变化图**：Push 触发扩容

```
初始：data=[a,b,c], count=3, capacity=3
         ▲
         └── 满了

Reserve(6)：分配新块，移动元素
  旧: [a,b,c] ──► 释放
  新: [a,b,c, _, _, _]
              ▲
              count=3, capacity=6

Push(d)：
  [a,b,c,d, _, _]
           ▲
           count=4
```

> **这段代码已经解决了什么**：分配器参数化（任何实现 `Allocate`/`Free` 的类型都可以注入，我们在 [[内存分配器]] 中定义的 `IAllocator` 派生类可直接使用）、无异常、增长策略可控、显式构造/析构（对有非平凡析构函数的类型安全）。
> **还缺什么**：没有 shrink-to-fit、没有插入/删除中间元素（`O(n)` 需自行实现）、没有范围检查（生产环境应加 `ASSERT(i < count)`）。
> **反例警告**：如果 `T` 的移动构造函数抛异常（在禁用异常的构建中不可能，但如果用拷贝构造函数且类型内部有裸指针管理），上述 `Reserve` 中的 move 循环在失败时会泄漏已移动但未析构的元素。**工业级实现需要"先在新内存构造，成功后统一析构旧元素"的两阶段提交**。

### 与分配器整合：不同场景的不同选择

```cpp
// 场景 A：每帧生成的临时渲染命令 → 从帧内存分配
Array<DrawCmd, FrameArenaAllocator> frameCommands;

// 场景 B：持久存在的实体列表 → 从全局堆分配（默认）
Array<Entity> persistentEntities;

// 场景 C：固定预算的音频通道 → 从对象池分配
Array<AudioChannel, PoolAllocator<AudioChannel, 32>> channels;
```

同一个 `Array<T, Allocator>` 模板，通过更换分配器，适配完全不同的内存策略。这是 `std::vector` 的分配器参数无法简洁表达的。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「通用动态数组的分配器可控性」这个子问题。
> - **UE** 的 `TArray<T, Allocator>` 是行业标杆。它的分配器参数化极其灵活：`FHeapAllocator`（默认堆分配）、`TInlineAllocator<N>`（前 N 个元素放在栈上，超出后才堆分配）、`TNonRelocatableInlineAllocator`（对不可移动类型优化）。UE 的 `TArray` 还实现了 `Serialize(Ar)` 接口，可直接序列化到资产文件。
> - **Bevy** 作为 Rust 引擎，直接使用标准库的 `Vec<T>`。Rust 的 `Vec` 没有自定义分配器参数（直到 Rust nightly 的 `Allocator API` 稳定）。Bevy 的应对方式是：通用代码用 `Vec`，ECS 核心存储层用完全自定义的 `BlobArray` + `ThinArrayPtr`。
>
> **默认推荐**：实现 `Array<T, Allocator>` 模板，Allocator 只需实现 `Allocate(size, align)` 和 `Free(ptr)` 两个方法（我们在 [[内存分配器]] 中已经定义了 `IAllocator` 接口）。默认用 `DefaultAllocator`（封装 `malloc`/`free`）。至少提供 `InlineAllocator<N>` 变体——这对小数组能显著减少堆分配次数。

解决了动态数组的问题，接下来引擎中还有大量"按键查找"的需求：资源路径 → 句柄、组件类型 ID → 工厂函数。`std::unordered_map` 能用吗？

## 问题 2：为什么 `std::unordered_map` 不够用？

### 最 naive 的方案

```cpp
std::unordered_map<std::string, ResourceHandle> resources;
resources["mesh/player.fbx"] = handle;
```

`std::unordered_map` 的底层是 **bucket + 链表/红黑树**。每个节点独立分配内存，遍历时要追着指针跳转，cache miss 极高。更糟糕的是：标准没有规定桶的数量、负载因子阈值、rehash 时机——不同平台的表现可能不一致。而且和 `std::vector` 一样，它的分配器参数化几乎是个摆设，你无法让它从帧内存或对象池中分配节点。

### 改进：HashMap<K, V, Hash, Allocator>

采用**开放寻址法** + **线性探测**，所有 Entry 存储在一个连续数组中：

```cpp
template <typename K, typename V,
          typename Hash = DefaultHash<K>,
          typename Allocator = DefaultAllocator>
struct HashMap {
    struct Entry {
        u64 hash;      // 完整哈希值，用于快速拒绝（避免比较 Key）
        K key;
        V value;
        bool occupied; // false = 空槽
    };

    Entry* entries = nullptr;
    usize count = 0;
    usize capacity = 0;
    Allocator allocator;

    void Reserve(usize minCapacity) {
        usize newCap = NextPowerOfTwo(minCapacity * 2); // 保持 50% 负载因子
        Entry* newEntries = static_cast<Entry*>(
            allocator.Allocate(newCap * sizeof(Entry), alignof(Entry))
        );
        for (usize i = 0; i < newCap; ++i) newEntries[i].occupied = false;

        if (entries) {
            for (usize i = 0; i < capacity; ++i) {
                if (entries[i].occupied) {
                    InsertInto(newEntries, newCap,
                               entries[i].hash, entries[i].key,
                               static_cast<V&&>(entries[i].value));
                    entries[i].key.~K();
                    entries[i].value.~V();
                }
            }
            allocator.Free(entries);
        }
        entries = newEntries;
        capacity = newCap;
    }

    V* Find(const K& key) {
        if (!entries) return nullptr;
        u64 h = Hash::Hash(key);
        usize idx = h & (capacity - 1); // 容量是 2 的幂，可用位掩码
        for (usize probe = 0; probe < capacity; ++probe) {
            Entry& e = entries[idx];
            if (!e.occupied) return nullptr; // 空槽 = 不存在
            if (e.hash == h && e.key == key) return &e.value;
            idx = (idx + 1) & (capacity - 1); // 线性探测
        }
        return nullptr;
    }

    void Insert(const K& key, const V& value) {
        if (count * 2 >= capacity) Reserve(capacity == 0 ? 16 : capacity);
        u64 h = Hash::Hash(key);
        InsertInto(entries, capacity, h, key, value);
        ++count;
    }

private:
    static void InsertInto(Entry* table, usize cap, u64 h,
                           const K& k, const V& v) {
        usize idx = h & (cap - 1);
        while (table[idx].occupied) {
            idx = (idx + 1) & (cap - 1);
        }
        table[idx].hash = h;
        new (&table[idx].key) K(k);
        new (&table[idx].value) V(v);
        table[idx].occupied = true;
    }
};
```

**状态变化图**：Insert("B") 时的线性探测

```
容量 = 8, 负载因子 = 50%（即最多存 4 个元素）

初始状态（已存 A, C）：
  idx0: [hash(A), "A", val, occupied]
  idx1: [empty]
  idx2: [hash(C), "C", val, occupied]
  idx3~7: [empty]

Insert("B")：
  hash("B") & 7 = 1
  idx1 为空，直接放入：
  idx1: [hash(B), "B", val, occupied]

Insert("D")，假设 hash("D") & 7 = 1：
  idx1 已被 B 占据
  探测到 idx2：被 C 占据
  探测到 idx3：空，放入：
  idx3: [hash(D), "D", val, occupied]
```

> **这段代码已经解决了什么**：连续内存存储（cache 友好）、开放寻址法（无指针跳转）、分配器参数化（直接复用 [[内存分配器]] 中的 `IAllocator`）、快速拒绝（先比 hash，hash 相等再比 key）。
> **还缺什么**：没有删除操作（开放寻址法的删除需要 tombstone 标记，会污染探测链）、没有迭代器（遍历需要跳过空槽）、rehash 时所有元素重新构造（对不可移动类型不友好）。
> **反例警告**：如果 Key 的哈希质量很差（如大量 Key 哈希到同一值），线性探测会退化为线性扫描，O(1) 变 O(n)。**必须提供一个好的默认哈希函数，并在调试构建中检测最长探测链长度**。

### HashSet<T>：HashMap 的薄封装

```cpp
template <typename T, typename Hash = DefaultHash<T>, typename Allocator = DefaultAllocator>
struct HashSet {
    struct Dummy {}; // void 值类型的占位
    HashMap<T, Dummy, Hash, Allocator> map;

    void Insert(const T& key) { map.Insert(key, Dummy{}); }
    bool Contains(const T& key) { return map.Find(key) != nullptr; }
};
```

> **与 SparseSet 的关键区别**：
> - **SparseSet**：Entity ID 是连续的整数（0, 1, 2, ...），直接用 `sparse[entityID]` 数组索引 → 本质上是**完美哈希**，O(1) 且无需哈希计算。
> - **HashMap**：Key 是任意类型，必须经过哈希函数 → 开放寻址法 → 平均 O(1)，但有哈希冲突和探测开销。
>
> 在 ECS 中，Entity 到组件的映射用 SparseSet（因为 Entity ID 是连续整数）。在通用场景中（字符串 → 句柄、类型名 → 工厂），用 HashMap。

解决了通用键值查找，接下来我们要面对 ECS 最核心的一个问题：如何在 O(1) 下判断"某个 Entity 是否拥有某个组件"，同时保证遍历所有拥有该组件的 Entity 时 cache 友好？

## 问题 3：如何在 O(1) 下判断 Entity 是否拥有组件，同时保证遍历密集？

### 最 naive 的方案

用 `std::unordered_map<EntityID, ComponentT>`。查找是 O(1)，但遍历所有拥有该组件的 Entity 时，要遍历整个 map——不仅每个元素都可能有独立的内存分配（cache 灾难），而且你还不得不遍历那些"没有该组件"的 Entity 的空洞。

### 改进：SparseSet（稀疏集）

**核心思想**：用两个数组实现 O(1) 增删和密集遍历，是 ECS 组件存储的最小骨架。

- `sparse[entityID]`：存储该 Entity 在 `dense` 数组中的索引（或空标记）。
- `dense[count]`：存储实际存在的 Entity ID，保证所有有效元素在数组前端连续排列。

```cpp
struct SparseSet {
    static constexpr u32 MAX_ENTITIES = 1024;
    static constexpr u32 INVALID = 0xFFFFFFFF;

    u32 sparse[MAX_ENTITIES];
    u32 dense[MAX_ENTITIES];
    u32 count = 0;

    SparseSet() {
        for (u32 i = 0; i < MAX_ENTITIES; ++i) sparse[i] = INVALID;
    }

    bool Has(u32 entity) const {
        if (entity >= MAX_ENTITIES) return false;
        u32 idx = sparse[entity];
        return idx != INVALID && idx < count && dense[idx] == entity;
    }

    void Add(u32 entity) {
        if (entity >= MAX_ENTITIES || Has(entity)) return;
        sparse[entity] = count;
        dense[count] = entity;
        ++count;
    }

    void Remove(u32 entity) {
        if (entity >= MAX_ENTITIES || !Has(entity)) return;
        u32 idx = sparse[entity];
        u32 lastEntity = dense[count - 1];
        dense[idx] = lastEntity;
        sparse[lastEntity] = idx;
        sparse[entity] = INVALID;
        --count;
    }

    const u32* Data() const { return dense; }
    u32 Count() const { return count; }
};
```

**状态变化图**：Add(Entity 5) → Add(Entity 2) → Remove(Entity 5)

```
初始状态：
sparse: [FF, FF, FF, FF, FF, FF, ...]
dense:  []
count:  0

Add(5)：
sparse: [FF, FF, FF, FF, FF, 0,  FF, ...]
                             ^entity 5 的索引
dense:  [5]
count:  1

Add(2)：
sparse: [FF, FF, 1,  FF, FF, 0,  FF, ...]
                   ^entity 2 的索引
dense:  [5, 2]
count:  2

Remove(5) — swap-and-pop 保持密集：
sparse: [FF, FF, 1,  FF, FF, FF, FF, ...]
                             ^entity 5 被标记为无效
dense:  [2]          (原末尾元素 2 被 swap 到索引 0)
count:  1
```

> **这段代码已经解决了什么**：O(1) 增删、O(count) 密集遍历（无空位检查）、极其缓存友好。`dense` 数组的连续性让批量处理组件时几乎没有 cache miss。
> **还缺什么**：`sparse` 数组的大小是 `MAX_ENTITIES`，Entity ID 范围很大时会浪费内存；没有与组件数据（ComponentArray）的联合存储。
> **反例警告**：如果 Entity ID 空间是 32 位（4294967296），直接开 `sparse` 数组会吃掉 16 GB 内存。工业级实现会用**分页稀疏数组**（Sparse Array of Pages）或**哈希表**回退。

### 为 ECS 铺垫：分页稀疏数组与组件数据联合存储

**分页稀疏数组**（Component Page）把 `sparse` 数组从线性大数组改为页表结构：

```
线性稀疏数组（浪费）：
sparse[4,294,967,296]  → 16 GB（不可接受）

分页稀疏数组：
pageTable[4096] → 每个 entry 指向一个 page（大小 1024）
                  或 nullptr（该页无实体）

EntityID = 0x00100080:
  pageIndex = 0x0010 = 16
  offset    = 0x0080 = 128
  sparse = pageTable[16][128]

实际内存：只有被使用过的页才会分配（4096 pages × 1024 uint32 ≈ 16 MB 全部分配，
          但实际按需分配，未使用页为 nullptr）。
```

**组件数据联合存储**：`SparseSet` 只解决了"哪些 Entity 拥有组件 T"的索引问题，但组件数据本身（如 `Transform` 结构体）还需要另一个数组。工业级做法是把 `dense` 数组和 `T components[]` 数组严格同步——`dense[i]` 对应的组件数据就是 `components[i]`。删除时用 `swap-and-pop` 同时更新两个数组。

> **参考：UE / Bevy / 前沿是怎么做的？**
>
> 我们在解决的是「O(1) 组件索引 + 密集遍历 + 大 Entity 空间内存效率」这个子问题。
> - **UE** 的 `TSet` 内部基于开放寻址法的哈希表，但有趣的是它用 `TSparseArray` 作为底层存储。`TSparseArray` 本质上就是一种 SparseSet 的变体——它维护 `dense` 数组存储实际元素，同时用 `freeIndices` 管理空洞，提供稳定的句柄。这与经典 SparseSet 的区别在于：UE 的稀疏数组更关注"稳定句柄"而非"Entity ID 映射"。
> - **Bevy** 的 `ComponentSparseSet` 采用经典三元结构：`sparse` + `dense`（Column，紧凑组件数据）+ `entities`（记录每个 dense 槽位对应的 Entity）。Bevy 的分页不是通过显式 pageTable 实现的，而是依赖 Rust 的 `Vec` 本身的动态扩容。但 Bevy 的 `SparseArray` 在 `debug_assertions` 关闭时只存 `EntityIndex`（而非完整 Entity），节省了 generation 字段的内存。
> - **前沿（EnTT）**：EnTT 的 sparse set 实现被公认为工业级标杆。它的优化在于：sparse 数组不存 `uint32`，而是存 `entity_type`（紧凑打包）；dense 数组和组件数组严格分离但同步，支持"组（Group）"机制把多个组件的 sparse set 部分重叠，实现多组件查询时的 cache 友好。
>
> **默认推荐**：默认用分页稀疏数组作为组件索引骨架（Entity 空间扩展到 32 位而不浪费内存）。组件数据与 `dense` 数组严格同步存储。这足以支撑阶段 2 的 `ComponentArray` 从 `std::vector` 升级到生产级实现。完整的 Archetype/Table 存储模型将在 [[组件系统架构]] 中深入讨论。

SparseSet 保证了索引层的高效，但组件数据的批量运算还有另一个瓶颈：无关字段对 cache line 的污染。当你只更新位置时，却把生命期、速度等无关字段一起加载进了 cache。这就是 SoA 要解决的问题。

## 问题 4：批量运算时，如何消除无关字段对 cache line 的污染？

### 最 naive 的方案

`struct Particle { Vec3 pos; Vec3 vel; float life; }`，然后用 `std::vector<Particle>` 存储。更新位置时只读 `pos` 和 `vel`，但 `life` 和其他字段被一起加载到 cache line 中——64 字节的 cache line 只利用了 24 字节，其余都是"搭便车"的无效数据。

### 改进：SoA（结构体数组 → 数组的结构体）

**核心思想**：把 `Particle` 拆成三个平行数组。批量运算时只读取需要的字段，消除无关字段的 cache line 污染。

```cpp
template <usize MAX_COUNT>
struct SoAParticles {
    f32 posX[MAX_COUNT];
    f32 posY[MAX_COUNT];
    f32 posZ[MAX_COUNT];
    f32 velX[MAX_COUNT];
    f32 velY[MAX_COUNT];
    f32 velZ[MAX_COUNT];
    f32 life[MAX_COUNT];
    usize count = 0;

    void Add(f32 x, f32 y, f32 z, f32 vx, f32 vy, f32 vz, f32 l) {
        if (count >= MAX_COUNT) return;
        posX[count] = x; posY[count] = y; posZ[count] = z;
        velX[count] = vx; velY[count] = vy; velZ[count] = vz;
        life[count] = l;
        ++count;
    }

    void UpdatePositions(f32 dt) {
        for (usize i = 0; i < count; ++i) {
            posX[i] += velX[i] * dt;
            posY[i] += velY[i] * dt;
            posZ[i] += velZ[i] * dt;
        }
    }
};
```

**内存布局对比**：AoS vs SoA（假设 `MAX_COUNT = 4`，每个 float 4 字节）

```
AoS (Array of Structs):
[ posX, posY, posZ, velX, velY, velZ, life ]  ← 32 bytes per particle
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]
[ posX, posY, posZ, velX, velY, velZ, life ]

如果只想更新 posX：
读取 cache line 时，vel/life 等无关数据被一起加载（假设 cache line 64 bytes，
每条 cache line 容纳 2 个粒子，但只用了 1 个 float）。

SoA (Structure of Arrays):
posX: [ x0, x1, x2, x3 ]  ← 16 bytes, 紧凑连续
posY: [ y0, y1, y2, y3 ]
posZ: [ z0, z1, z2, z3 ]
velX: [ vx0, vx1, vx2, vx3 ]
...

更新 posX 时：
16 bytes 完全有效，一条 cache line（64 bytes）可装 16 个 float，
即 16 个粒子的 posX。遍历效率显著提升。
```

> **这段代码已经解决了什么**：批量 SIMD 运算的天然友好布局、只读部分字段时 cache 利用率最大化。
> **还缺什么**：随机访问单个粒子的全部字段需要 7 次指针跳转（`posX[i]`, `posY[i]`...）、代码冗长、删除中间元素需要维护所有数组的同步。
> **反例警告**：如果系统需要频繁"读取一个粒子的全部字段"（如单个粒子与玩家的碰撞检测），SoA 会导致 7 次不连续的内存读取，反而比 AoS 慢。**SoA 是为批量同字段运算优化的，不是为随机全字段访问优化的**。

### 为 ECS 铺垫：类型擦除的列存储

手写 `SoAParticles` 在组件类型多的时候会变成代码灾难。工业级 ECS 引擎的做法是**类型擦除的列存储（Type-Erased Column Storage）**：

```cpp
struct Column {
    void* data;           // 原始字节数组，通过 componentSize 索引
    usize elementSize;
    usize count;
    usize capacity;
};

struct Table {
    Array<EntityID> entities;  // 行号 → 实体
    HashMap<ComponentTypeID, Column> columns;  // 组件类型 → 列
};
```

同一 Table 中的所有实体拥有完全相同的组件组合（这就是 Archetype）。系统查询 `Query<Transform, Mesh>` 时，只需找到所有包含这两个组件的 Table，然后顺序读取对应的 `Column`——这就是 SoA 的工业级自动化版本。

**Bevy 的 Column 设计更进一步**：每个 `Column` 不仅存 `data: BlobArray`（类型擦除），还附带 `added_ticks` 和 `changed_ticks` 两个旁路数组。这意味着 Change Detection（变更检测）不需要额外的外部数据结构，而是直接在 Column 旁边完成，保证了 cache 局部性。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「批量同字段运算的 cache 效率」这个子问题。
> - **UE** 的 `TArray` 本身是 AoS（因为它存储的是 `T` 对象数组），但 UE 在渲染层大量使用 SoA。例如 GPU 粒子系统（Niagara）的模拟数据就是严格的 SoA 布局，以便 SIMD 和 GPU 并行。UE 的 `TMap` / `TSet` 虽然哈希表结构不是 SoA，但它们的 `Elements` 数组（`TSparseArray`）内部是密集的。
> - **Bevy** 的 `Table` 是 SoA 的极致实现：每个 Archetype 对应一个 `Table`，Table 内按 `ComponentId` 分 `Column`。`Column` 用 `BlobArray` 做类型擦除（存储原始字节），用 `ThinArrayPtr` 剥离长度/容量字段（由外部 `Table` 统一管理）。这种"瘦指针 + 外部管理"的设计减少了每个 Column 的元数据开销。
> - **前沿（Unity DOTS / Flecs）**：Unity DOTS 的 Chunk 存储是 SoA 的变体——同一 Archetype 的实体按固定大小 Chunk（如 16KB）分块存储，每个 Chunk 内是 SoA 布局。Flecs 的 Table 也采用类似设计，但支持更灵活的分页策略。
>
> **默认推荐**：手写 SoA 在组件类型少（< 5 种）时是可行的。一旦进入 ECS 阶段，应直接实现类型擦除的 Column Storage（Table）。不要试图用 C++ 模板为每种组件组合生成特化代码——那会导致代码膨胀和编译时间爆炸。Bevy 的 `BlobArray` + `ThinArrayPtr` 模式是自研引擎最优雅的参考。完整的 Archetype Table 设计将在 [[组件系统架构]] 中深入讨论。

SoA 优化了单线程批量运算的 cache 效率，但引擎中还有一类场景必须跨线程通信——音频线程、渲染线程、网络线程。如果用锁，优先级反转会直接毁掉实时性。这就是 RingBuffer 要解决的问题。

## 问题 5：两个线程之间如何传递数据，既能避免锁竞争，又不引入复杂的同步机制？

### 最 naive 的方案

`std::queue<T> + std::mutex`。生产者线程 `lock` → `push` → `unlock`，消费者线程 `lock` → `pop` → `unlock`。如果生产者优先级低、消费者优先级高（如音频线程），低优先级线程持有锁时可能被高优先级线程抢占，导致高优先级线程忙等——这就是**优先级反转**。

### 改进：RingBuffer（无锁环形队列，SPSC）

**核心思想**：用两个原子索引 `head` / `tail` 和一块固定数组，实现单生产者-单消费者（SPSC）的零锁通信。

```cpp
#include <atomic>

template <typename T, usize SIZE>
struct RingBuffer {
    static_assert((SIZE & (SIZE - 1)) == 0, "SIZE must be power of 2");
    T buffer[SIZE];
    std::atomic<usize> head{0};
    std::atomic<usize> tail{0};

    bool Push(const T& item) {
        usize t = tail.load(std::memory_order_relaxed);
        usize next = (t + 1) & (SIZE - 1);
        if (next == head.load(std::memory_order_acquire)) return false; // 满
        buffer[t] = item;
        tail.store(next, std::memory_order_release);
        return true;
    }

    bool Pop(T& item) {
        usize h = head.load(std::memory_order_relaxed);
        if (h == tail.load(std::memory_order_acquire)) return false; // 空
        item = buffer[h];
        head.store((h + 1) & (SIZE - 1), std::memory_order_release);
        return true;
    }
};
```

**状态变化图**：Push A → Push B → Pop A（SIZE = 4）

```
初始状态：
[ _, _, _, _ ]
 h=0
 t=0

Push(A):
[ A, _, _, _ ]
 h=0
 t=1

Push(B):
[ A, B, _, _ ]
 h=0
 t=2

Pop() → A:
[ A, B, _, _ ]   (A 仍在内存中，但逻辑上已消费)
    h=1
 t=2
```

> **这段代码已经解决了什么**：单生产者-单消费者（SPSC）场景下的零锁通信、严格的 acquire/release 内存序保证可见性、无系统调用。
> **还缺什么**：仅支持 SPSC；MPSC/MPMC 需要更复杂的原子 CAS 或 seqlock；没有 ABA 防护（大小为 2 的幂次配合索引环绕可天然缓解，但 64 位索引溢出问题在极长运行时间后理论上存在）；**容量固定为 2 的幂次**。
> **反例警告**：如果在生产者线程中 Push 的数据大小不固定（如变长字符串），RingBuffer 无法直接使用。它只适用于**固定大小消息**的传递。多生产者场景下不加锁直接使用此代码会导致数据竞争。

### 工业级演进：从 SPSC 到 MPSC 与命令队列

SPSC RingBuffer 够用，但游戏引擎中常见的生产者-消费者关系更复杂：

- **多生产者-单消费者（MPSC）**：Gameplay 系统在多线程中并行生成渲染命令，最终由渲染线程消费。解决方式是每个生产者有自己的 SPSC RingBuffer，消费者轮询所有 RingBuffer（"多路复用"）。这比真正的 MPMC 无锁队列简单得多，且避免了 CAS 竞争。
- **命令队列（Command Queue）**：ECS 中 System 不能直接在并行执行时修改 World（会导致数据竞争）。Bevy 的 `Commands` 本质就是一个线程本地的缓冲区，System 把"我要插入组件 A"、"我要销毁实体 B"等命令写入本地缓冲区，帧末由主线程串行 flush。这种"本地累积 + 批量提交"的模式是 RingBuffer 思想在 ECS 中的最高频应用。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「跨线程零锁通信」这个子问题。
> - **UE** 的 `FMallocBinned2/3` 在分配器层面大量使用类似 RingBuffer 的思想：每个线程维护 `FPerThreadFreeBlockLists`（线程本地空闲块链表）。`Malloc` 和 `Free` 在热路径上是完全无锁的——直接从本地链表 Pop/Push。只有当本地链表满/空时，才需要加锁与全局回收器交互。这本质上是把"线程间的内存竞争"转化为"线程本地的栈操作"。
> - **Bevy** 的 `Event` 系统内部使用 RingBuffer 存储事件。`Commands` 虽然名义上是 Vec，但在并行 System 中每个线程写自己的命令缓冲区，最后合并——这与 MPSC 多路复用的思想一致。
> - **前沿（mimalloc）**：mimalloc 被 UE 5.5+ 采纳为默认分配器，其核心设计之一是"free list sharding"——每个 mimalloc page（64KB）内部有多个 free list（一个用于线程本地释放，一个用于跨线程释放）。跨线程释放只需一次 CAS，无需全局锁。这与 RingBuffer 的"把竞争分散到多个小单元"思想异曲同工。
>
> **默认推荐**：默认用 SPSC RingBuffer。如果需要 MPSC，优先用"每生产者一个 SPSC + 消费者轮询"的方案，不要直接写 MPMC 无锁队列（极易出错）。ECS 的命令队列模式（本地累积 + 批量提交）是应对多线程结构变更的最佳实践。

## 问题 6：容器这么多，怎么按场景组合才不会混乱？

现在我们已经有了 `Array`、`HashMap`、`SparseSet`、`SoA`、`RingBuffer`……但它们不是孤立的工具。一个真实的引擎场景中，往往是**容器 + 分配器 + 使用模式**的三元组合。如果随意搭配，可能会得到反效果：

| 错误组合 | 后果 |
|---------|------|
| `Array<T, FrameArenaAllocator>` 存跨帧数据 | 帧末 `Reset()` 直接释放，悬垂指针 |
| `HashMap<K, V>` 存 ECS 组件索引 | Entity ID 是连续整数，HashMap 的哈希计算纯属浪费，应该用 SparseSet |
| `SparseSet` 存通用键值对 | Key 不是连续整数时，`sparse` 数组浪费巨大 |

### 组合决策图

```
需要按键查找？
  ├── 是 → Key 是连续整数（如 EntityID）？
  │         ├── 是 → SparseSet（O(1) 无哈希，密集遍历）
  │         └── 否 → HashMap<K, V>（开放寻址，通用键）
  └── 否 → 需要 O(1) 随机访问？
            ├── 是 → Array<T, Allocator>（动态数组）
            └── 否 → 需要 FIFO 线程通信？
                      ├── 是 → RingBuffer（SPSC 无锁）
                      └── 否 → 需要批量同字段运算？
                                ├── 是 → SoA / Column Storage
                                └── 否 → 纯临时数据？
                                          ├── 是 → FrameArena（见 [[内存分配器]]）
                                          └── 否 → 固定大小高频对象？
                                                    ├── 是 → ObjectPool（见 [[内存分配器]]）
                                                    └── 否 → 通用分配 → Mimalloc（见 [[内存分配器]]）
```

### 统一分配器接口：让组合成为可能

所有上述组合的前提是：分配器有一个**最小统一接口**。我们在 [[内存分配器]] 中已经定义了：

```cpp
struct IAllocator {
    virtual void* Allocate(usize size, usize align) = 0;
    virtual void  Free(void* ptr) = 0;
};
```

`Array`、`HashMap` 等通用容器通过模板参数接收分配器：`Array<T, MyAlloc>`。专用容器（`FrameArena`、`ObjectPool`）直接实现这个接口。这样：

```cpp
// 通用容器 + 专用分配器 = 场景定制
Array<DrawCmd, FrameArenaAllocator> frameCmds;      // 每帧丢弃
Array<Entity, DefaultAllocator>     entities;       // 持久存在
HashMap<StringId, Texture*, FNVHash, DefaultAllocator> textureMap; // 资源查找

// 专用容器直接面向场景
FrameArena perFrameTemp(1024 * 1024);               // 1MB 帧内存
ObjectPool<Particle, 4096> particlePool;            // 粒子池
RingBuffer<AudioPacket, 256> audioQueue;            // 音频线程队列
```

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「容器与分配器的系统化组合」这个子问题。
> - **UE** 的 `TArray<T, Allocator>` 正是这套组合哲学的体现。同一个 `TArray`，配 `FHeapAllocator` 就是普通堆数组，配 `TInlineAllocator<4>` 就是前 4 个元素在栈上的小型数组，配 `TAllocatorFixedSize` 就是从固定池分配。UE 的 `TMap` / `TSet` 同样参数化了分配器。这套设计的核心洞察是：**容器负责数据结构逻辑，分配器负责内存来源，两者通过最小接口解耦**。
> - **Bevy** 在 Rust 中没有自定义分配器参数（`Vec` 就是标准库 `Vec`），所以 Bevy 选择了另一种组合方式：通用代码用标准容器，性能敏感或特殊语义的代码用完全自定义的数据结构（`BlobArray`、`ComponentSparseSet`、`CommandQueue`）。这种"标准容器 + 专用结构"的两层策略，启发是：不要试图让所有容器都参数化分配器——那样会引入模板膨胀。只在**最常用**的容器（Array、HashMap）上做分配器参数化，其他场景用专用结构。
>
> **默认推荐**：
> - 给 `Array<T, Allocator>` 和 `HashMap<K, V, Hash, Allocator>` 实现分配器模板参数。Allocator 接口只需要 `Allocate(size, align)` 和 `Free(ptr)`（我们在 [[内存分配器]] 中已经定义）。
> - 提供三个默认分配器：`DefaultAllocator`（封装 `malloc`/`free`）、`FrameArenaAllocator`（引用外部 Arena）、`PoolAllocator<T, N>`（引用外部 ObjectPool）。
> - 不要为每个容器写独立的分配逻辑——通过一个 `IAllocator` 接口统一。调试时用一个 `DebugAllocatorWrapper` 装饰器包裹任何分配器，即可全局开启 Canary 检测。
> - **最关键的决策**：ECS 存储（SparseSet / Archetype）不要用通用 `Array` 或 `HashMap` 组装——它们有自己独特的内存布局需求（SoA、类型擦除、分页稀疏数组）。专用问题用专用结构解决。

---

> **默认推荐路径总结**：
>
> 容器系统的工业级默认路径是：**`Array<T, Allocator>`（分配器参数化动态数组）+ `HashMap<K, V>`（开放寻址法 + 线性探测）+ `SparseSet`（分页稀疏数组 + swap-and-pop）+ 类型擦除 `Column Storage`（SoA 自动化）+ `RingBuffer`（SPSC 无锁队列）**。该路径在 ECS 兼容前提下吸收了以下工业级设计：
> - **UE `TArray<T, Allocator>` 的分配器参数化**：`FHeapAllocator`（默认堆）、`TInlineAllocator<N>`（前 N 个元素在栈上）、`TAllocatorFixedSize`（固定池）。核心洞察是**容器负责数据结构逻辑，分配器负责内存来源，两者通过最小接口解耦**。
> - **UE `TSparseArray`**：用 `dense` 数组存储实际元素，`freeIndices` 链表管理空洞，删除不移动后续元素，提供稳定句柄。这与经典 SparseSet 异曲同工。
> - **Bevy `BlobArray` + `ThinArrayPtr`**：类型擦除的列存储减少每个 Column 的元数据开销；`added_ticks` / `changed_ticks` 旁路数组让 Change Detection 不破坏 cache 局部性。
> - **EnTT 的 sparse set 标杆**：`entity_type` 紧凑打包，dense 数组与组件数组严格分离但同步，支持 Group 机制实现多组件查询的 cache 友好。
>
> **ECS 映射**：`SparseSet` 的 `dense` 数组与组件 `Column` 严格同步存储，是 ECS 组件索引的最小骨架。`SoA` 的列存储思想在 ECS 中通过 Archetype Table 自动实现——同一 Archetype 的实体按 Chunk 分块，每个 Chunk 内是 SoA 布局。ECS 命令队列模式（本地累积 + 批量提交）是 RingBuffer 思想在 ECS 中的最高频应用。
>
> **条件切换路径**：如果 Entity 空间确定小于 64K（如固定关卡的小规模场景），可将分页稀疏数组退化为线性稀疏数组以简化实现；如果 Key 的哈希质量极差且无法更换哈希函数，可将开放寻址法切换为链地址法（但会牺牲 cache 连续性）。
>
> **扩展路径（更极端工业场景）**：
> - **HashMap 删除与 tombstone**：引入 tombstone 标记支持开放寻址法的安全删除，或采用 Robin Hood 哈希进一步缩短最长探测链。
> - **SparseSet 分代索引**：为 `dense` 数组中的每个元素附加 generation 字段，防止 swap-and-pop 后旧索引的误用。
> - **MPSC 命令队列**：用"每生产者一个 SPSC + 消费者轮询"替代单 SPSC，支撑多线程并行 System 的命令提交。
> - **Archetype Table 完整实现**：在 [[组件系统架构]] 中深入讨论 Chunk-based 存储、实体迁移、查询缓存和并行调度。

## 回探替换清单：替换阶段 1/2 的临时实现

阶段 3 的核心使命是**回探替换**。以下是容器相关的替换点和验证方法：

| 临时实现（阶段 1/2） | 正式实现（阶段 3） | 替换位置 | 验证方法 |
|---------------------|---------------------|---------|---------|
| `std::vector<Entity>`（ComponentArray） | `Array<Entity, DefaultAllocator>` 或 `Array<Entity, PoolAllocator>` | 阶段 2 的 `ComponentArray` | Inspector 仍能正常显示实体列表，无内存泄漏 |
| `std::vector<DrawCmd>`（临时渲染命令） | `Array<DrawCmd, FrameArenaAllocator>` | 阶段 1 的渲染循环 | 每帧 `Reset()` 后无悬垂指针，FPS 稳定 |
| `std::unordered_map<TypeID, Factory>` | `HashMap<TypeID, Factory>` | 阶段 2 的硬编码组件工厂 | 新增组件类型后仍能正确创建 |

> **最关键的回探替换**：阶段 2 的 `ComponentArray`（`std::vector` 临时实现）→ 替换为引擎自定义的组件存储。这是本阶段**最关键的回探替换**。
>
> 具体的 ECS 存储模型（Archetype Table vs SparseSet 的完整架构决策）将在 [[组件系统架构]] 中深入讨论。本篇提供的是地基——SparseSet 的索引骨架、SoA 的列存储思想——有了这些，阶段 4.1 才能在此基础上搭建完整的 ECS 大厦。

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 满足 | SparseSet 的 `dense` 数组是 ECS 批量处理的基础；SoA 的列数据是平铺的连续数组；Array 的 `data/count/capacity` 是 AI 可直接观测的结构化数据 |
| **自描述** | ✅ 满足 | 容器内部状态（count/capacity/occupied）是 AI 可直接观测的结构化数据 |
| **确定性** | ✅ 满足 | SparseSet 的 swap-and-pop 保证遍历顺序虽变但无空洞；RingBuffer 的原子索引操作是确定性的 |
| **工具边界** | ⚠️ 需封装 | AI 通过 MCP 操作容器时，应通过 ECS 的 Commands 模式（延迟执行 + 批量提交），禁止直接修改裸指针 |
| **Agent 安全 / 沙箱隔离** | ✅ 满足 | ECS 的命令队列模式天然支持多 Agent 的隔离与合并；容器的边界检查（ASSERT）可防止越界访问 |

## 性能与 trade-off 总结

| 容器 | 对比基准 | 什么场景更好 | 什么场景更差 | 核心代价 |
|------|---------|-------------|-------------|---------|
| **Array<T, Allocator>** | `std::vector<T>` | 分配器参数化、增长策略可控、无异常 | 需要 STL 兼容接口（如迭代器层级） | 自研维护成本 |
| **HashMap<K, V>** | `std::unordered_map<K, V>` | 连续内存、cache 友好、分配器可控 | Key 哈希质量差时探测链退化 | 无标准删除操作（需 tombstone） |
| **SparseSet** | `std::unordered_set<uint32_t>` | 密集遍历 + O(1) 增删 | Entity 空间极大、稀疏分布 | sparse 数组预分配内存 |
| **SoA** | AoS `std::vector<Struct>` | 批量 SIMD 同字段运算 | 随机访问单个对象全部字段 | 删除需同步多数组，代码冗长 |
| **RingBuffer** | `std::queue` + `mutex` | SPSC 线程通信 | MPMC、变长消息 | 容量固定为 2 的幂 |

> **下一步**：[[文件IO与虚拟文件系统]] — 容器和分配器就位后，引擎需要加载外部数据。文件 IO 也需要可控的内存策略（如用 FrameArena 缓冲读取的临时数据）。
>
> 有了自己的容器体系，引擎才真正从"能跑"进入"可控"的阶段。阶段 2 中那些 `std::vector` 的临时实现，现在有了正式的替代品——而这片地基，将在 [[组件系统架构]] 中成长为完整的 ECS 存储大厦。
