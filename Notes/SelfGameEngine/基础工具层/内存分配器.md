# 内存分配器

## 问题 0：为什么引擎不能直接用 `malloc`/`new`？

初学者搭引擎时，动态内存用 `new`/`delete`，临时数组用 `std::vector`。原型阶段没问题，但一旦进入生产环境，三个与**分配**直接相关的痛点会爆炸：

1. **分配器不可插拔**：`std::vector` 的内存来源几乎是硬编码的全局 `malloc`。你无法让临时渲染命令从"帧内存"里分配，也无法让 ECS 组件从固定对齐的块里出生。内存变成了一座无法调度的大仓库。
2. **调试与确定性缺失**：全局堆是子系统混用的黑箱。一帧之内 `malloc` 的时序、地址、碎片状态完全不可预测，导致离线回放和线上崩溃难以复现。
3. **时延不可预测**：`malloc` 在底层可能向操作系统申请新页、合并空闲块、或触发全局锁竞争。对音频线程、渲染线程这种有实时性要求的场景，`malloc` 的一次停顿就是一次爆音或掉帧。

下面用两个真实场景让你感受到痛：

**场景一：每帧临时渲染命令**。每帧都创建 `std::vector<DrawCmd>`，你以为 `clear()` 释放了内存，实际上只是 `size` 归零，`capacity` 纹丝不动。下一帧数据量变多，又会触发 `realloc` 拷贝——16ms 的帧预算里，这种抖动直接让你掉帧。更隐蔽的是：`realloc` 可能在某个帧里突然去操作系统要新页，时延从几十纳秒跳到几百微秒。

**场景二：音频包线程间传递**。用 `std::queue<AudioPacket>` 加 `std::mutex`，如果生产者优先级低、消费者是音频线程（高优先级），低优先级线程持有锁时被抢占，高优先级音频线程只能忙等——这就是**优先级反转**，结果是爆音。问题的根源不是队列，而是队列背后 `new`/`delete` 触发的不可预测时延和锁竞争。

自定义分配器不是"重复造轮子"，而是**把内存从不可控的黑箱变成可编排的资源**。下面我们从最痛的问题开始逐个解决。

## 问题 1：临时内存每帧产生、每帧丢弃，如何避免反复向操作系统申请？

### 最 naive 的方案

每帧用 `std::vector<TempData> temp; temp.clear();`，以为 `clear()` 会释放内存。实际上它只是把 `size` 设为 0，`capacity` 纹丝不动——下一帧如果数据量变少，内存就浪费了；变多了，又会触发 `realloc` 拷贝。而且 `realloc` 时无法选择内存来源，只能走全局堆。

### 改进：FrameArena（帧分配器）

**核心思想**：一帧内所有临时内存共享一块大缓冲区，用一个指针 `offset` 往前推。帧结束时 `offset = 0`，O(1) 全部回收。

```cpp
struct FrameArena {
    static constexpr usize CAPACITY = 1024 * 1024; // 1 MiB
    u8* base = nullptr;
    usize offset = 0;

    void Init() { base = static_cast<u8*>(std::malloc(CAPACITY)); offset = 0; }
    void Shutdown() { std::free(base); base = nullptr; }

    void* Alloc(usize size, usize align = 8) {
        usize mask = align - 1;
        usize padded = (offset + mask) & ~mask;
        if (padded + size > CAPACITY) return nullptr; // 溢出，无回退
        offset = padded + size;
        return base + padded;
    }

    void Reset() { offset = 0; } // 每帧调用一次，O(1) 回收
};
```

**状态变化图**：`Alloc(24, 16)` 执行前后

```
分配前：
base ──► [..............................]
         ^offset=8

对齐后：
base ──► [........][....................]
                  ^padded=16

分配后：
base ──► [........][####################][............................]
                  ^返回的指针            ^offset=40
                  (24 bytes 已分配)
```

> **这段代码已经解决了什么**：O(1) 分配、O(1) 批量回收、零外部碎片、无系统调用（除 `Init` 时的一次 `malloc`）。
> **还缺什么**：没有多帧轮转（当前帧分配的内存不能异步被消费线程读取）、没有线程安全（需 TLS 或单生产者约束）、没有向系统分配器的动态扩容回退、**对齐浪费可能高达 `align - 1` 字节**。
> **反例警告**：如果一帧内临时分配总量超过 `CAPACITY`（如复杂粒子系统或大规模 ECS 查询），`Alloc` 会直接返回 `nullptr`。FrameArena 只适用于**内存总量可预估**的场景。

### 工业级演进：从单帧 Arena 到多帧轮转与嵌套作用域

FrameArena 的最小版本够用，但生产环境会遇到两个新问题：

**问题 A：渲染线程需要消费游戏线程上一帧产生的数据**
如果游戏线程每帧 `Reset()`，渲染线程可能还在读上一帧的临时数据。解决方式是**双缓冲/三缓冲 Arena**：维护 `N` 个 `FrameArena`，游戏线程写入 Arena[i]，渲染线程读取 Arena[(i-1) mod N]。每帧只 `Reset` 最旧的那个。

**问题 B：子系统之间的临时内存互相污染**
如果所有临时内存都从一个全局 Arena 分配，物理系统的临时数组和渲染系统的临时命令混在一起，`Reset` 时无法只回收其中一部分。解决方式是**作用域标记（FMemMark）**：在 Arena 中压入一个标记点，子系统结束后把 `offset` 回滚到标记点，而非全局 `Reset`。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「临时内存的生命周期管理」这个子问题。
> - **UE** 在 RenderCore 层提供了 `FMemStackBase`（64KB 页线性栈），渲染线程用它分配 RenderGraph 的临时资源描述符。游戏线程则用 `FMalloc` 家族的通用分配器。UE 的 `FMemStack` 支持 `FMemMark` 嵌套回滚——你可以在子函数中分配临时内存，函数返回时自动回滚到进入时的偏移量。
> - **Bevy** 的 ECS 系统中，System 的临时查询结果（如 `Query<&Transform>` 返回的迭代器）不分配堆内存，而是直接引用 Archetype Table 的连续列数据。Bevy 的 `Commands` 缓冲区虽然需要临时存储，但它是按 System 在帧末统一 flush 的，本质也是一种 Arena 模式。
>
> **默认推荐**：默认用单帧 `FrameArena` 处理同帧临时数据；如果涉及异步消费（如游戏线程生成命令 → 渲染线程执行），升级到双缓冲 Arena。嵌套作用域标记可以在需要时添加，不必一开始就做。

FrameArena 解决了"同帧临时数据"的问题，但引擎中还有一类对象——粒子、网络包、Entity 句柄——它们大小固定、生命周期长短不一，却频繁创建和销毁。这类对象如果走全局堆，碎片问题会非常严重。

## 问题 2：固定大小对象频繁创建销毁，如何避免 malloc 碎片和构造开销？

### 最 naive 的方案

`new Particle()` / `delete ptr`。每次都要向全局堆申请，释放后留下大小不一的空洞。如果粒子每帧生成 1000 个、销毁 1000 个，全局堆很快变成蜂窝。

### 改进：ObjectPool（固定大小对象池）

**核心思想**：预分配一大块固定大小的 Slot，用空闲链表（free list）管理未使用的 Slot。分配和释放都是 O(1) 的指针操作。

```cpp
template <typename T, usize BLOCK_COUNT>
struct ObjectPool {
    union Slot {
        T object;
        Slot* nextFree;
        Slot() {}
        ~Slot() {}
    };

    Slot* memory = nullptr;
    Slot* freeList = nullptr;

    void Init() {
        memory = static_cast<Slot*>(std::malloc(sizeof(Slot) * BLOCK_COUNT));
        for (usize i = 0; i < BLOCK_COUNT; ++i) {
            memory[i].nextFree = (i + 1 < BLOCK_COUNT) ? &memory[i + 1] : nullptr;
        }
        freeList = &memory[0];
    }

    void Shutdown() { std::free(memory); }

    T* Allocate() {
        if (!freeList) return nullptr;
        Slot* slot = freeList;
        freeList = freeList->nextFree;
        return new (slot) T();
    }

    void Free(T* obj) {
        if (!obj) return;
        obj->~T();
        Slot* slot = reinterpret_cast<Slot*>(obj);
        slot->nextFree = freeList;
        freeList = slot;
    }
};
```

**状态变化图**：分配和释放一个对象

```
Init 后：
memory: [Slot0] → [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList

Allocate() 后：
memory: [Slot0: T()]  [Slot1] → [Slot2] → [Slot3] → nullptr
                       ▲
                      freeList
                      (返回 &Slot0.object)

Free(&Slot0) 后：
memory: [Slot0] ─┐
                 │
         ┌───────┘
         ▼
        [Slot1] → [Slot2] → [Slot3] → nullptr
         ▲
        freeList
```

> **这段代码已经解决了什么**：O(1) 分配/释放、零外部碎片、固定大小对象生命周期完全可控、构造/析构显式调用。
> **还缺什么**：没有跨池释放校验（把不是本池的对象 `Free` 进来会崩溃）、没有内存对齐保证（`alignof(T)` 可能大于默认 `malloc` 对齐）、没有动态扩容、**对象在内存中的地址永远只在预分配的 `memory` 块内，不支持运行时增长**。
> **反例警告**：如果对象大小差异很大（如 16 字节的粒子和 256 字节的 NPC），为每种大小各建一个 ObjectPool 会导致元数据（free list 指针）开销比例不同。ObjectPool 只适用于**对象大小固定且数量可预估**的场景。

### 工业级演进：分代索引与动态分块

固定 ObjectPool 很快会遇到容量天花板。工业级引擎通常引入两层改进：

**分代索引（Generational Index）**：把裸指针返回给用户是危险的——如果 Slot 被释放后又重新分配，旧指针就变成了悬垂指针。解决方式是返回 `Handle { slotIndex, generation }`，每次复用 Slot 时 `generation++`。访问时校验 `generation` 是否匹配，不匹配则说明是旧指针。

```cpp
struct GenerationalHandle {
    u32 slotIndex;
    u32 generation;
};

struct GenerationalPool {
    struct Slot { T object; u32 generation; u32 nextFree; };
    Array<Slot> slots;
    u32 freeListHead = INVALID;
    // Allocate 返回 Handle；访问时通过 handle.slotIndex 取数据并校验 generation
};
```

**动态分块**：不要预分配一个固定的大数组，而是分配多个固定大小的 Block（如每块 1024 个 Slot）。`freeList` 跨 Block 链接。当所有 Block 都满时，追加新 Block。

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「固定大小对象的高频分配与悬垂指针安全」这个子问题。
> - **UE** 的 `FName` 系统本质上是一个巨型字符串对象池。所有字符串实际存储在全局的 `FNamePool` 中，按长度分块（Short / Medium / Large）。`FName` 本身只存一个整数索引，比较两个 `FName` 只需比较两个 `uint32`（O(1)）。这是对象池思想在字符串上的极致应用——把"比较字符串"从 O(n) 降为 O(1)。
> - **UE** 的 `TSparseArray` 是另一种固定大小对象池的变体：它维护一个 `dense` 数组存储实际元素，同时用 `freeIndices` 链表管理空洞。删除元素时只标记为空洞，不移动后续元素，因此其他元素的索引（句柄）不会失效。这对需要稳定 ID 的游戏对象集合非常有价值。
> - **Bevy** 的 `Entity` 分配器也采用分代索引：Entity 是一个 `(index, generation)` 对。当 Entity 被销毁后，它的 `index` 进入空闲列表，但 `generation` 递增。旧句柄的 `generation` 不匹配，访问时会被拒绝。
>
> **默认推荐**：默认用动态分块 + 分代索引的 ObjectPool。如果对象数量有严格上限（如音频通道固定 32 个），可以用固定池。`FName` 式的字符串池在游戏中有大量资源路径比较时是刚需，但最小实现阶段可以用 `std::string_view` + 哈希表模拟，后续在 [[字符串系统]] 中升级为正式 Intern 池。

ObjectPool 和 FrameArena 都是**专用分配器**——它们只解决特定场景。但引擎中还有大量"通用小对象"（网络包、AI 决策节点、UI 元素），它们的大小不固定、生命周期不统一。专用分配器无法覆盖它们。

## 问题 3：通用小对象分配如何做到严格 O(1) 和确定性？

FrameArena 和 ObjectPool 都是专用分配器——它们只解决特定场景。但引擎中还有大量"通用小对象"（如网络包、AI 决策节点、UI 元素），它们的大小不固定、生命周期不统一。对于这些对象，我们需要一个通用方案。

### 最 naive 的方案

直接用 `malloc` / `free`。问题在前面已经说过了：时延不可预测、碎片、多线程竞争。

### 改进：Binned Allocator（分箱分配器）

**核心思想**：把小块内存按固定大小分箱（如 8, 16, 24, 32, ... 直到 32KB），每个箱子是一个 ObjectPool。分配时把请求大小向上取整到最近的箱大小，从对应箱子里取一个空闲块。释放时把块归还到对应箱子。

```cpp
// 概念性伪代码
struct BinnedAllocator {
    static constexpr usize NUM_BINS = 32;
    static constexpr usize MAX_SMALL_SIZE = 32768;
    ObjectPool<void, /*per-bin-count*/> bins[NUM_BINS];

    void* Malloc(usize size) {
        if (size > MAX_SMALL_SIZE) return OSMalloc(size);
        uint32 binIndex = SizeToBinIndex(size); // O(1) 查表
        return bins[binIndex].Allocate();
    }
    void Free(void* ptr) {
        uint32 binIndex = PtrToBinIndex(ptr); // 通过地址反查
        bins[binIndex].Free(ptr);
    }
};
```

> **这段代码已经解决了什么**：小对象分配 O(1)、零外部碎片（同箱内块大小相同）、无需合并（释放直接归还箱子）。
> **还缺什么**：没有考虑多线程竞争（多个线程同时从同一个箱子分配仍需加锁）、没有 TLS 缓存（每次分配都要查表 + 箱子操作，cache 局部性一般）、大对象直接走 OS 分配。

### 工业级演进：TLS 缓存 + 指针元数据外置 + Trim Epoch

**TLS 线程缓存**：每个工作线程维护独立的空闲块缓存数组。`Malloc` 时优先从本地缓存取，`Free` 时优先归还本地缓存。只有当本地缓存满/空时，才与全局分配器交互。热路径完全无锁。

**指针元数据外置**：释放时需要通过指针反查它属于哪个 Bin。不要给每个 Block 加头部信息（浪费内存），而是通过页对齐约定 + 哈希表反查：所有 Bin 的内存都从 OS 按页（如 64KB）申请，根据指针的页地址即可定位到 Bin 索引。

**Trim Epoch**：当工作线程进入空闲（如切后台）时，分配器需要回收该线程缓存中的内存。用单调递增的 epoch 标记哪些线程缓存需要回收，避免无差别扫描所有线程。

> **参考：UE / 前沿是怎么做的？**
>
> 我们在解决的是「通用小对象的确定性分配」这个子问题。
> - **UE** 的 `FMallocBinned2/3` 是工业级分箱分配器的标杆。`Binned2` 用 64KB 大页对齐，`Binned3` 每 Bin 独立预占 1GB 虚拟地址池并用位树管理 Block。两者都通过 `FPtrToPoolMapping` 用哈希 + 位运算反查指针所属 Bin，Block 头部零开销。UE 5.5+ 已将默认分配器切换为 **Mimalloc**，因为 Mimalloc 的 segment-based free list 在广泛基准测试中表现更稳定。
> - **前沿（Mimalloc）**：Mimalloc 的核心创新是"free list sharding"——每个 64KB page 内部有多个 free list（本地释放列表、跨线程释放列表）。跨线程释放只需一次 CAS，竞争被自然分散到成千上万个 page 中。在 2026 年的基准测试中，Mimalloc 在小对象高频分配场景下的 P99 延迟比 jemalloc 低 15%，比 tcmalloc 低 22%。UE 的切换决策验证了这一点。
> - **TLSF（Two-Level Segregated Fit）**：如果需要**严格 O(1)**（不是"平均 O(1)"，而是"最坏情况 O(1)"），TLSF 是另一个选择。它用两级位图索引空闲块，分配和释放都是严格常数时间。代价是位图和分割/合并逻辑引入额外元数据开销，内存利用率比 `malloc` 低 5%~15%。
>
> **默认推荐**：
> - **默认方案**：直接用 **Mimalloc** 作为全局分配器（单文件 `static.c` 即可嵌入）。它同时满足"工业级性能"和"极简集成"，UE 的选择已经替你验证过了。
> - **如果你需要完全自研**（如嵌入式平台或学习目的）：实现一个简化版 Binned Allocator + TLS 缓存。不要一开始就写 TLSF——它的位图维护复杂度远高于收益。Binned + TLS 缓存已经能让你的分配器在 99% 场景下媲美系统分配器。
> - **极端实时场景**（音频线程、网络线程）：如果 Mimalloc 的延迟仍然不可接受，退回到专用分配器（ObjectPool、RingBuffer、FrameArena）。通用分配器永远不如专用分配器快。

## 问题 4：专用分配器多了之后，如何调试内存越界和 use-after-free？

### 最 naive 的方案

在每次分配时多申请一些内存，填入魔数（Canary），释放时检查。但如果把这个逻辑直接写进 `FrameArena::Alloc`，最终构建也会带着这些开销。

### 改进：装饰器模式 + AOP 式 Hook

**核心思想**：把调试功能做成可插拔的"包装层"，包裹在真正的分配器外面。开发构建用包装层，最终构建直接剥掉。

```cpp
// 调试分配器包装层
struct DebugAllocatorWrapper {
    IAllocator* inner;

    void* Alloc(usize size, usize align) {
        usize total = size + 2 * CANARY_SIZE;
        u8* raw = static_cast<u8*>(inner->Alloc(total, align));
        // 头部 Canary
        std::memcpy(raw, &CANARY_MAGIC, CANARY_SIZE);
        // 尾部 Canary
        std::memcpy(raw + CANARY_SIZE + size, &CANARY_MAGIC, CANARY_SIZE);
        // 毒值填充用户区域
        std::memset(raw + CANARY_SIZE, 0xFE, size);
        return raw + CANARY_SIZE;
    }

    void Free(void* ptr) {
        u8* raw = static_cast<u8*>(ptr) - CANARY_SIZE;
        // 校验 Canary...
        inner->Free(raw);
    }
};
```

**调试机制清单**：

| 机制 | 作用 | 实现方式 |
|------|------|---------|
| **毒值填充** | 分配时写入 `0xFE`，检测 use-after-free | `memset(ptr, 0xFE, size)` |
| **Canary 越界检测** | 块前后插入魔数，释放时校验 | 头部/尾部各 8 字节 |
| **调用栈捕获** | 记录每次分配的调用栈，泄漏时可回溯 | 平台相关（如 `CaptureStackBackTrace`） |
| **分配统计** | 每个子系统的 OS 申请量、已分配量、峰值 | `AllocatorMetrics` 接口 |

> **参考：UE / 前沿是怎么做的？**
>
> 我们在解决的是「内存调试与诊断」这个子问题。
> - **UE** 的 `FMalloc` 家族是装饰器模式的教科书案例。`FMallocStomp` 在每分配一页时用 guard page 检测越界；`FMallocLeakDetection` 追踪分配释放配对；`FMallocThreadSafeProxy` 给非线程安全分配器加锁。这些装饰器通过 `GMalloc` 指针链式组合：开发构建可能是 `Stomp → LeakDetection → ThreadSafeProxy → Binned3`，而最终构建只有 `Binned3`。
> - **UE** 的 `FMemory` 全局入口还提供了 `QuantizeSize` 接口，让容器在扩容时知道"最优请求大小"。例如 `TArray` 扩容时不会按"当前大小 × 1.5"申请，而是按 `FMemory::QuantizeSize` 对齐到 Bin 大小，减少内部碎片。
> - **前沿（Mimalloc）**：Mimalloc 的 debug build 自动提供 detailed statistics、padding detection、double-free detection。它的 guarded mode 可以在生产环境中以采样方式放置 guard page（每 N 次分配放一个），性能损失接近零。
>
> **默认推荐**：
> - 实现一个 `IAllocator` 接口（`Alloc` / `Free` / `Realloc`），所有专用分配器（FrameArena、ObjectPool、BinnedAllocator）都实现它。
> - 实现一个 `DebugAllocatorWrapper` 装饰器，用宏控制在 Debug/Development 构建中启用，Final 构建中完全剥离。
> - 不要为每个分配器单独写调试逻辑——那是代码重复。通过接口统一包装。
> - `QuantizeSize` 是容易忽略但收益很高的优化：容器在扩容时调用 `allocator.QuantizeSize(requestedSize)`，而不是直接 `requestedSize`，能显著减少内部碎片。

## 问题 5：分配器这么多，如何统一接口按场景组合？

现在我们已经有了 `FrameArena`、`ObjectPool`、`BinnedAllocator`、`DebugAllocatorWrapper`……但它们不是孤立的工具。一个真实的引擎场景中，往往是**分配器 + 使用模式**的组合。如果随意搭配，可能会得到反效果：

| 错误组合 | 后果 |
|---------|------|
| `FrameArena` 存跨帧数据 | 帧末 `Reset()` 直接释放，悬垂指针 |
| `ObjectPool<T, 1024>` 存变长字符串 | 对象池要求固定大小，字符串长度不固定，越写越乱 |
| 所有子系统共享一个全局 `FrameArena` | 子系统间内存互相污染，`Reset` 时无法单独回收 |

### 统一分配器接口：让组合成为可能

所有上述组合的前提是：分配器有一个**最小统一接口**。不需要像 STL 那样复杂的 `rebind`、`construct`、`destroy`——只要两个方法：

```cpp
struct IAllocator {
    virtual void* Allocate(usize size, usize align) = 0;
    virtual void  Free(void* ptr) = 0;
};
```

专用分配器（`FrameArena`、`ObjectPool`）直接实现这个接口。通用容器（在 [[容器系统]] 中讨论）通过模板参数接收分配器：`Array<T, MyAlloc>`。这样：

```cpp
// 专用分配器直接面向场景
FrameArena perFrameTemp(1024 * 1024);               // 1MB 帧内存
ObjectPool<Particle, 4096> particlePool;            // 粒子池

// 通用容器 + 专用分配器 = 场景定制
Array<DrawCmd, FrameArenaAllocator> frameCmds;      // 每帧丢弃
Array<Entity, DefaultAllocator>     entities;       // 持久存在
```

### 分配器选择决策图

```
对象生命周期是否严格等于一帧？
  ├── 是 → 内存总量是否可预估？
  │         ├── 是 → FrameArena（O(1) 分配/回收，零碎片）
  │         └── 否 → 退回到通用分配器（Mimalloc）
  └── 否 → 对象大小是否固定？
            ├── 是 → 数量是否有严格上限？
            │         ├── 是 → 固定 ObjectPool（最低开销）
            │         └── 否 → 动态分块 ObjectPool + 分代索引
            └── 否 → 对象是否极小（< 32KB）且高频？
                      ├── 是 → Binned Allocator / Mimalloc
                      └── 否 → 通用分配器（Mimalloc）
```

> **参考：UE / Bevy 是怎么做的？**
>
> 我们在解决的是「分配器的系统化组合」这个子问题。
> - **UE** 的 `FMalloc` 家族提供了统一的 `FMalloc` 接口（`Malloc` / `Free` / `Realloc` / `GetAllocationSize`）。所有分配器（`FMallocBinned2`、`FMallocBinned3`、`FMallocMimalloc`）都实现这个接口。UE 的 `TArray<T, Allocator>` 正是通过这套接口实现分配器参数化：同一个 `TArray`，配 `FHeapAllocator` 就是普通堆数组，配 `TInlineAllocator<4>` 就是前 4 个元素在栈上的小型数组。核心洞察是：**容器负责数据结构逻辑，分配器负责内存来源，两者通过最小接口解耦**。
> - **Bevy** 在 Rust 中没有自定义分配器参数（`Vec` 就是标准库 `Vec`），所以 Bevy 选择了另一种策略：通用代码用标准容器，性能敏感或特殊语义的代码用完全自定义的数据结构（`BlobArray`、`ComponentSparseSet`）。启发是：不要试图让所有容器都参数化分配器——那样会引入模板膨胀。只在最通用的容器上做分配器参数化，其他场景用专用结构。
>
> **默认推荐**：
> - 所有专用分配器实现 `IAllocator` 接口。`DefaultAllocator` 封装 `malloc`/`free` 作为默认回退。
> - 提供三个常用变体：`FrameArenaAllocator`（引用外部 Arena）、`PoolAllocator<T, N>`（引用外部 ObjectPool）。
> - 不要为每个分配器写独立的分配逻辑——通过 `IAllocator` 接口统一。调试时用一个 `DebugAllocatorWrapper` 装饰器包裹任何分配器，即可全局开启 Canary 检测。
> - **最关键的决策**：ECS 存储（SparseSet / Archetype）不要用通用 `Array` 或 `HashMap` 组装——它们有自己独特的内存布局需求（SoA、类型擦除、分页稀疏数组）。专用问题用专用结构解决。

---

> **默认推荐路径总结**：
>
> 内存分配体系的工业级默认路径是：**`FrameArena`（同帧临时数据）+ 动态分块 `ObjectPool`（固定大小对象）+ `Mimalloc`（通用小对象）+ `DebugAllocatorWrapper`（开发期诊断）+ 统一 `IAllocator` 接口**。该路径在 ECS 兼容前提下吸收了以下工业级设计：
> - **UE `FMalloc` 家族的装饰器模式**：`FMallocStomp`（guard page 越界检测）、`FMallocLeakDetection`（分配释放配对追踪）、`FMallocThreadSafeProxy`（线程安全包装）全部通过统一接口链式组合。开发构建可以是 `Stomp → LeakDetection → ThreadSafeProxy → Binned3`，最终构建只有 `Binned3`。我们直接吸收"接口统一、可链式组合"的思想。
> - **UE `FMemStackBase` 的帧内存管理**：渲染线程用 64KB 页线性栈分配 RenderGraph 临时资源描述符，支持 `FMemMark` 嵌套回滚。这验证了 FrameArena 在工业级引擎中的核心地位。
> - **UE 5.5+ 切换 Mimalloc 的决策**：在广泛基准测试中，Mimalloc 的小对象高频分配 P99 延迟比 jemalloc 低 15%，比 tcmalloc 低 22%。对于通用小对象分配，**不要自研**——直接用经过工业验证的 Mimalloc。
>
> **ECS 映射**：分配器本身是 ECS 的 Resource（如 `FrameArenaResource`、`ObjectPoolResource`），System 通过 `Res<FrameArena>` 获取临时内存。这保证了分配器状态对 AI 可见、可观测，且不同 System 的临时分配互不污染。
>
> **条件切换路径**：如果目标平台是嵌入式或禁止外部依赖（如某些主机平台），将 `Mimalloc` 切换为自研简化版 `BinnedAllocator` + TLS 缓存；如果对象数量有严格上限（如音频通道固定 32 个），可将动态分块 ObjectPool 退化为固定池。
>
> **扩展路径（更极端工业场景）**：
> - **TLS 线程缓存**：为 `BinnedAllocator` 引入线程本地空闲块缓存，热路径完全无锁。
> - **指针元数据外置**：通过页对齐约定 + 哈希表反查指针所属 Bin，Block 头部零开销。
> - **Trim Epoch**：工作线程进入空闲时自动回收线程缓存内存。
> - **GPU 显存上传堆**：为渲染资源（纹理、缓冲区）引入专用的 GPU 显存分配器（参考阶段 9.1 `内存管理全链路`）。
> - **分配器遥测**：集成崩溃报告系统，将内存峰值、泄漏热点、分配栈回溯自动上传到分析平台。

## 回探替换清单：替换阶段 1/2 的临时实现

阶段 3 的核心使命是**回探替换**。以下是分配器相关的替换点和验证方法：

| 临时实现（阶段 1/2） | 正式实现（阶段 3） | 替换位置 | 验证方法 |
|---------------------|---------------------|---------|---------|
| `new/delete` 裸分配（Entity 句柄） | `ObjectPool<EntitySlot, 1024>` + 分代索引 | 阶段 2 的 Entity 分配 | 销毁后旧句柄访问返回无效，无 use-after-free |
| 每帧 `std::vector<DrawCmd>` + `clear()` | `FrameArena` + 从 Arena 分配的临时数组 | 阶段 1 的渲染循环 | 每帧 `Reset()` 后无悬垂指针，FPS 稳定 |
| 全局 `malloc`（所有通用小对象） | `Mimalloc` 或自研 `BinnedAllocator` | 全局分配器入口 | 长时间运行后内存不膨胀，无泄漏 |

> **回探提示**：分配器的替换通常是"全局切入点"的替换——找到所有 `new` / `delete` / `malloc` / `free` 的调用点，改为通过 `IAllocator` 接口。由于分配器位于最底层，替换后需要全量回归测试。

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 满足 | FrameArena 的 `offset` 是确定性的；ObjectPool 的块布局是严格平铺的数组；分配器状态（已分配量、峰值）是 AI 可直接读取的结构化数据 |
| **自描述** | ✅ 满足 | `IAllocator` 接口可暴露 `allocationCount`、`peakUsage` 等指标 |
| **确定性** | ✅ 满足 | FrameArena 的 `Reset()` 消除了释放顺序的不确定性；ObjectPool 消除了外部碎片 |
| **工具边界** | ⚠️ 需封装 | AI 通过 MCP 请求"分配临时内存"时，必须返回作用域句柄（如 `ScopedArenaAlloc`），禁止返回裸指针 |
| **Agent 安全 / 沙箱隔离** | ✅ 满足 | 通过为不同子系统生成独立内存池实现编译期隔离；Agent 脚本层的分配错误不会破坏核心引擎 |

## 性能与 trade-off 总结

| 分配器                       | 对比基准                                | 什么场景更好        | 什么场景更差       | 核心代价           |
| ------------------------- | ----------------------------------- | ------------- | ------------ | -------------- |
| **FrameArena**            | `std::vector` + 每帧 `clear`/`shrink` | 大量临时对象，生命周期同帧 | 跨帧存活、总量不可预估  | 无单独释放，总容量固定    |
| **ObjectPool**            | `new`/`delete`                      | 固定大小对象频繁创建销毁  | 对象大小差异大、数量剧增 | 无动态扩容，地址不连续    |
| **Binned Allocator**      | `malloc`/`free`                     | 小对象高频分配，需要确定性 | 超大对象（> 32KB） | 内部碎片（向上取整到箱大小） |
| **Mimalloc**              | `jemalloc` / `tcmalloc`             | 高频小对象、长运行服务   | 需要完全自研控制     | 外部依赖（但极轻量）     |
| **DebugAllocatorWrapper** | 无                                   | 开发阶段捕获越界和泄漏   | 最终构建（有性能开销）  | 额外内存和运行时校验     |

> 有了自己的分配器体系，引擎才真正从"能跑"进入"可控"的阶段。阶段 2 中那些 `new`/`delete` 的临时实现，现在有了正式的替代品——而这片地基，将在 [[组件系统架构]] 中成长为完整的 ECS 存储大厦。
