# 数学基础

## 问题 0：为什么不能用 `std::array<float, 3>` 或外部数学库？

想象你正在组装第一辆自行车。前轮需要一个轴承，后轮也需要一个。最省事的办法是去商店买两个标准轴承——就像直接用 `std::array<float, 3>` 表示三维坐标，或者引入 GLM 这种成熟库。

这确实能让代码立刻跑起来。但当你开始追求速度、精度和可维护性时，标准轴承会出现三个致命的不匹配：

**第一，内存对齐失控。** 现代 CPU 的 SIMD 寄存器是 128 位宽，能同时装 4 个 `float`。但 `std::array<float, 3>` 只占 12 字节，编译器不会自动把它放到 16 字节对齐的地址上。当你想用一条指令同时计算 4 个向量的加法时，未对齐的内存访问要么直接崩溃（某些 ARM 设备），要么性能暴跌（x86 需要拆成两次加载）。

**第二，与 ECS 存储模型脱节。** 外部库通常把三维向量设计成带有复杂构造器、析构器或虚函数的 C++ 类。这些特性在单个对象层面很方便，但在 ECS 的批量迭代中，虚函数表指针会污染组件的内存布局，阻止系统对原始字节进行 SIMD 批量读写。你的 `Position` 组件理想情况下应该只是一块 12 字节的裸数据，没有任何隐藏开销。

**第三，浮点确定性无法保障。** 网络同步和回放系统要求：同样的输入序列，在 Windows PC 和 ARM 手机上必须计算出完全相同的物理位置和旋转角度。外部库为了最大化性能，会默认开启编译器的激进优化（如自动合并乘加指令），而不同平台对浮点舍入的处理规则不同。这意味着你无法在不修改库源码的情况下锁定行为。

因此，自研数学库不是"重复造轮子"，而是**可控性**的入场券。我们要做的不是一个比 GLM 更复杂的怪物，而是一个"刚好够用、完全可控、与 ECS 天生契合"的最小集合。

## 问题 1：最基础的向量与矩阵该怎么设计？

如果你完全不懂引擎数学，你可能会这样写：

```cpp
struct Vec3 {
    float x, y, z;
};

struct Mat4 {
    float m[4][4]; // 二维数组，直观但危险
};
```

这已经能表达"三维坐标"和"4×4 矩阵"了。但它会在三个地方埋雷。

### 埋雷点 1：二维数组的索引语义混乱

`m[row][col]` 看起来清晰，但在 C++ 中二维数组的内存布局是**行优先**的——第 0 行 4 个元素紧挨着，然后是第 1 行。而 GPU 的常量缓冲区和图形 API（OpenGL、DirectX、Vulkan）几乎**统一使用列主序**。如果你的 CPU 端用行主序计算了一个视图矩阵，原样传给 GPU，画面会直接撕裂或完全黑屏。

### 埋雷点 2：没有对齐保证

`Vec3` 只有 12 字节。把它塞进数组或直接 `new`，地址可能是任意 4 的倍数，而非 16 的倍数。后续接入 SIMD 时，每一处向量都需要额外检查地址，或者被迫使用性能更慢的非对齐加载指令。

### 埋雷点 3：缺少第四分量的"隐藏搭档"

`Vec3` 本身只有 x、y、z。但 SIMD 寄存器固定装 4 个 `float`。如果你把 `Vec3` 作为基础运算单元，每次 SIMD 操作都会浪费 1/4 的算力；如果你强行在 `Vec3` 里塞一个 `w` 字段变成 `Vec4`，又会让只存储位置的组件白白多占 33% 内存。

**改进方案：分层设计**

我们把数学类型分成两层：

1. **存储层**：`Vec3` 保持 12 字节，作为 ECS 组件和顶点数据的**存储格式**，不占额外空间。
2. **运算层**：`Vec4` 作为 16 字节对齐的 **SIMD 运算格式**，是 `Vec3` 在寄存器中的"工作形态"。

```cpp
// 存储层：紧凑、无隐式对齐要求（但可手动对齐）
struct Vec3 {
    f32 x, y, z;
};

// 运算层：16 字节对齐，可直接加载到 SIMD 寄存器
struct alignas(16) Vec4 {
    f32 x, y, z, w;
};
```

当系统需要批量更新位置时，先把 `Vec3` 数组的 x、y、z 分别扩展成 `Vec4`（w 填 0 或 1），然后一次性处理 4 个实体。这看起来多了一步转换，但这恰恰给了你选择权：内存紧张的场景用 `Vec3` 存，计算密集的场景用 `Vec4` 算。

对于矩阵，**统一采用列主序**：

```cpp
struct Mat4 {
    f32 m[16]; // 索引 = row + col * 4（列主序）

    static Mat4 Identity() {
        Mat4 out{};
        out.m[0] = out.m[5] = out.m[10] = out.m[15] = 1.0f;
        return out;
    }

    Vec3 TransformPoint(const Vec3& v) const {
        f32 x = m[0]*v.x + m[4]*v.y + m[8]*v.z + m[12];
        f32 y = m[1]*v.x + m[5]*v.y + m[9]*v.z + m[13];
        f32 z = m[2]*v.x + m[6]*v.y + m[10]*v.z + m[14];
        f32 w = m[3]*v.x + m[7]*v.y + m[11]*v.z + m[15];
        return w != 0.0f ? Vec3{x/w, y/w, z/w} : Vec3{x, y, z};
    }

    static Mat4 Translate(const Vec3& t) {
        Mat4 out = Identity();
        out.m[12] = t.x; out.m[13] = t.y; out.m[14] = t.z;
        return out;
    }

    static Mat4 Scale(const Vec3& s) {
        Mat4 out = Identity();
        out.m[0] = s.x; out.m[5] = s.y; out.m[10] = s.z;
        return out;
    }
};
```

列主序的内存布局：

```
列主序：m[col * 4 + row]

        Col 0   Col 1   Col 2   Col 3
       +------+------+------+------+
Row 0  |  m0  |  m4  |  m8  |  m12 |
Row 1  |  m1  |  m5  |  m9  |  m13 |
Row 2  |  m2  |  m6  |  m10 |  m14 |
Row 3  |  m3  |  m7  |  m11 |  m15 |
       +------+------+------+------+

线性内存中的顺序：
[m0, m1, m2, m3, m4, m5, m6, m7, m8, m9, m10, m11, m12, m13, m14, m15]
```

> **为什么列主序？** 因为 OpenGL 的 uniform 矩阵默认接收列主序，DirectX 的 HLSL 矩阵乘法语义也是列向量左乘矩阵。如果你坚持行主序，就必须在每次传 GPU 前转置，或者在 shader 里调整乘法顺序——这非常容易搞混，而且一旦搞混，画面黑屏时你很难定位是矩阵顺序错了。

> **参考：Bevy（通过 glam）的矩阵约定**
>
> Bevy 使用的 glam 库同样采用列主序存储和列向量语义。这意味着如果你在 C++ 端也采用列主序，将来与 Bevy 或 glam 的算法对照时会减少大量心智负担。工业界的主流图形库（OpenGL、HLSL、glam）在这一点上是统一的，不要逆势而行。

## 问题 2：旋转用欧拉角还是四元数？

最 naive 的方案：用三个浮点数表示 yaw（偏航）、pitch（俯仰）、roll（翻滚）。代码直观，符合人类直觉：

```cpp
struct EulerAngle {
    f32 yaw, pitch, roll; // 单位：弧度
};
```

但这个方案会在两个场景下彻底崩溃。

**崩溃场景 1：万向节锁（Gimbal Lock）**

当一个物体抬头 90 度（pitch = π/2）时，它的局部坐标系的"前方"轴与世界坐标系的"上方"轴重合。此时你再调整 yaw，物体不会左右转头，而是绕着世界坐标系的垂直轴翻滚——两个旋转自由度坍缩成了一个。这在 FPS 摄像机跟随和飞行模拟器中简直是灾难。

**崩溃场景 2：旋转插值不平滑**

如果你想让物体从朝向 A 平滑旋转到朝向 B，对欧拉角做线性插值会得到荒谬的路径。因为欧拉角的三个分量在球面上不是均匀分布的，直接插值可能让物体绕一大圈甚至倒转。

**改进方案：四元数**

四元数用 4 个浮点数 `(x, y, z, w)` 表示三维空间中的一个旋转。它不会万向节锁，且插值路径天然沿着球面大圆（最短路径）。

```cpp
struct Quat {
    f32 x, y, z, w;

    static Quat AxisAngle(const Vec3& axis, f32 radians) {
        f32 half = radians * 0.5f;
        f32 s = std::sin(half);
        Vec3 n = axis.Normalized();
        return {n.x * s, n.y * s, n.z * s, std::cos(half)};
    }
};
```

但四元数也有代价：你不能像欧拉角那样"单独改 yaw"然后立刻看到效果。如果需要人类可读的编辑器控件，通常的策略是**编辑时用欧拉角，存储和运算时用四元数**——在 Inspector 面板里显示 yaw、pitch、roll，用户修改后立即转换成四元数存入组件。

### 插值：NLERP vs SLERP

从一个旋转插值到另一个旋转，最 naive 的方法是直接对四元数的四个分量做线性插值再归一化（NLERP）：

```cpp
Quat Nlerp(const Quat& a, const Quat& b, f32 t) {
    f32 dot = a.x*b.x + a.y*b.y + a.z*b.z + a.w*b.w;
    f32 sign = dot < 0.0f ? -1.0f : 1.0f; // 取最短路径
    Quat r{
        a.x*sign + (b.x*sign - a.x)*t,
        a.y*sign + (b.y*sign - a.y)*t,
        a.z*sign + (b.z*sign - a.z)*t,
        a.w*sign + (b.w*sign - a.w)*t
    };
    // 归一化...
    return r.Normalized();
}
```

NLERP 的优点是快——只需要加减乘和一次开方。缺点是角速度不均匀：在 t=0.5 时，实际转过的角度不一定是总角度的一半。

SLERP（球面线性插值）解决了这个问题，但它需要计算 `acos` 和 `sin`，成本高得多。

> **默认推荐**：默认用 NLERP。对于动画混合、粒子朝向等 99% 的场景，NLERP 的速度优势远大于肉眼可察觉的角速度差异。只有在过场动画的精确镜头轨迹中，才需要 SLERP。

> **参考：工业引擎对这个问题是怎么做的？**
>
> UE 的动画系统对旋转插值做了多层优化：高速迭代用 NLERP，关键路径的骨骼重定向用 SLERP，且对 dot < 0 的情况强制翻转符号保证最短路径。
>
> Bevy（通过 glam）同样提供了 NLERP 和 SLERP 两种方法，默认在动画系统中使用球面插值以保证视觉平滑。
>
> **结论**：NLERP 作为默认方案足够覆盖绝大多数动画混合、粒子朝向和 gameplay 旋转插值场景；如果未来有过场动画的精确镜头轨迹需求，再引入 SLERP 作为可选路径。该决策与项目规模无关——UE 的动画系统在高速迭代路径上同样优先使用 NLERP。

## 问题 3：当场景有上万个物体时，标量运算会不会成为瓶颈？

假设你已经有了上面的 `Vec3` 和 `Mat4`，每个实体每帧都要更新世界变换矩阵。最 naive 的写法：

```cpp
for (auto& e : entities) {
    e.world_matrix = Mat4::Translate(e.position)
                   * Mat4::FromQuat(e.rotation)
                   * Mat4::Scale(e.scale);
}
```

这段代码在 100 个实体时毫无压力，但在 10000 个实体时，帧率会从 60fps 掉到 30fps 以下。为什么？

**瓶颈不在运算量，而在内存访问模式。**

现代 CPU 的频率已经很多年没有大幅提升，但内存延迟（从 CPU 请求数据到数据抵达寄存器）仍然是数百个时钟周期。CPU 缓解这个问题的方式是**缓存行预取**：当你访问 `entities[0].position` 时，CPU 会一口气把 `entities[0]` 到 `entities[15]` 附近的内存全部搬进高速缓存（通常 64 字节一行）。

问题在于，如果你的实体结构像这样：

```cpp
struct Entity {
    Vec3 position;
    Quat rotation;
    Vec3 scale;
    Mat4 world_matrix;
    Health health;
    NameTag name;
    // ... 其他 20 个字段
};
```

CPU 读取 `position` 时，会把整个实体对象的前 64 字节搬进缓存。但你的变换计算系统**只需要** position、rotation、scale，其他字段（health、name 等）被白白加载进缓存，占用了宝贵的缓存空间。这种现象叫做**过度取指（over-fetch）**。

**改进方案：Structure of Arrays（SoA）**

把每个字段拆成独立的连续数组：

```cpp
// 原来：Array of Structures（AoS）
// Entity entities[10000]; // 每个实体 200 字节，交错存储

// 现在：Structure of Arrays（SoA）
Vec3 positions[10000];
Quat rotations[10000];
Vec3 scales[10000];
Mat4 world_matrices[10000];
```

现在，当你遍历更新世界矩阵时，`positions[0]`、`positions[1]`、`positions[2]`... 在内存中是紧挨着的。CPU 预取机制能完美工作，缓存里的每一字节都是即将被用到的数据。

但这还不是终点。SoA 让我们可以进一步引入 SIMD，一次性处理 4 个实体：

```cpp
void UpdateWorldMatricesSoA(
    const Vec3* pos, const Quat* rot, const Vec3* scl,
    Mat4* out, usize count)
{
    for (usize i = 0; i < count; i += 4) {
        // 一次性加载 4 个 position.x 到 SIMD 寄存器
        // 一次性加载 4 个 position.y 到 SIMD 寄存器
        // ...
        // 4 个矩阵并行计算
    }
}
```

> **参考：UE 的 Transform 批量处理与 Bevy 的 ECS 存储**
>
> UE 的变换系统在逻辑层面把位移、旋转、缩放作为独立字段暴露，但在批量动画评估和骨骼变换传播时，上层会显式拆分为三个独立数组进行 SIMD 处理。这种"逻辑层 AoS，运算层 SoA"的混合策略，是大型引擎处理海量变换的核心技巧。
>
> Bevy 的原生 ECS 架构下，Transform 组件本身是一个 POD 结构，但由于 ECS 的 Archetype 存储把同类型组件紧密排列在同一张表的连续列中，批量查询天然获得了 SoA 的内存布局。Bevy 不需要像 OOP 引擎那样手动拆分数组——这是 ECS 架构在数学批量运算上的结构性优势。

### 诚实的 Trade-off：SoA 不是银弹

| 场景 | AoS 表现 | SoA 表现 | 原因 |
|------|---------|---------|------|
| 随机访问单个实体的完整变换 | ✅ 好 | ❌ 差 | SoA 需要读 3 个数组 + 1 个矩阵数组，cache miss 翻倍 |
| 批量更新所有实体的世界矩阵 | ❌ 差 | ✅ 好 | SoA 连续加载，SIMD 利用率接近 100% |
| 代码可读性与维护性 | ✅ 好 | ⚠️ 差 | `position[i]` 变成分别访问 x、y、z 数组，结构分散 |
| 实体频繁增删 | ✅ 简单 | ⚠️ 复杂 | SoA 需要同步维护多个数组的对应关系 |

> **反例警告**：如果你的游戏对象不到 500 个，或者每帧只更新少量可见对象，SoA 的代码复杂度和随机访问劣势会抵消其 SIMD 收益。不要过早优化。ECS 引擎虽然在存储层天然是 SoA，但你仍然可以选择在系统里逐个处理组件（标量路径），只有当 Profiler 证明变换更新是瓶颈时，才引入手动 SIMD 批量优化。

## 问题 4：跨平台时 SIMD 指令不同怎么办？

当你终于决定用 SIMD 加速批量矩阵运算时，最 naive 的方案是直接写 x86 的 SSE 内联函数：

```cpp
__m128 a = _mm_load_ps(px + i);
__m128 b = _mm_load_ps(py + i);
__m128 sum = _mm_add_ps(a, b);
```

这段代码在 x86 PC 上飞快，但在 ARM 手机上直接编译失败——ARM 的 NEON 指令集用的是 `float32x4_t` 和 `vaddq_f32`。

**改进方案：平台抽象层**

不要直接把 SSE 指令散落到渲染和物理的每一行代码里。引擎应该封装一层平台无关的 SIMD 类型，让上层只写一次统一的加法函数：

```cpp
namespace simd {
#if defined(ARCH_X86)
    using Vec4F = __m128;
    inline Vec4F Add(Vec4F a, Vec4F b) { return _mm_add_ps(a, b); }
    inline Vec4F Mul(Vec4F a, Vec4F b) { return _mm_mul_ps(a, b); }
#elif defined(ARCH_ARM)
    using Vec4F = float32x4_t;
    inline Vec4F Add(Vec4F a, Vec4F b) { return vaddq_f32(a, b); }
#else
    // 标量回退：用 4 个 float 模拟
    struct Vec4F { f32 v[4]; };
    inline Vec4F Add(Vec4F a, Vec4F b) {
        return {a.v[0]+b.v[0], a.v[1]+b.v[1], a.v[2]+b.v[2], a.v[3]+b.v[3]};
    }
#endif
}
```

这个抽象层的**关键设计决策**是：编译期多态，而非运行时函数指针。

有些方案建议运行时检测 CPU 特性，然后通过函数指针调用不同的实现。但数学运算（加法、乘法、点积）是每帧调用数百万次的极热路径。函数指针会阻止编译器内联，每一次调用都会引入间接跳转和寄存器保存/恢复的开销，最终可能比不用 SIMD 还慢。

> **参考：UE 与 glam 的 SIMD 策略**
>
> UE 在编译期通过平台宏选择 SSE、NEON 或 FPU 后端，Shipping 构建中所有 SIMD 调用都是直接内联的，没有运行时分支。同时，UE 5.2+ 在 x86 平台强制要求 SSE4.2，这意味着不需要运行时 CPUID 检测即可使用完整指令集。
>
> Bevy（通过 glam）采用了类似的编译期适配：在 x86 用 SSE2，在 ARM64 用 NEON，在 WASM 用 SIMD128，在标量平台自动回退。Rust 的编译目标三元组天然完成了平台分发。更关键的是，glam 即使在标量回退时仍保持 16 字节对齐和内部填充，确保对象尺寸和布局在所有架构上一致——这对网络同步和序列化极其重要。

> **默认推荐**：采用编译期条件编译 + 标量回退。不要为"可能支持 AVX 的极少数 CPU"做运行时动态分发——维护成本远高于收益。如果你的目标平台是 PC，直接要求 SSE4.2；如果是移动端，适配 NEON；其他平台用标量回退。同时学习 glam 的策略：标量回退时仍保持 16 字节对齐，避免架构差异导致内存布局变化。

### 对齐的务实折中

AVX 的 256 位寄存器需要 32 字节对齐。如果你要求所有向量数组都 32 字节对齐，某些内存分配器会产生显著的内存膨胀（对齐要求越高，内部碎片越大）。

UE 的做法是：**存储时只保证 16 字节对齐**，加载到寄存器时使用未对齐指令（如 `_mm_loadu_ps`）。在现代 CPU 上，未对齐加载的代价已经极低（通常 0~1 个额外周期），但内存节省是实实在在的。这是一种务实的工程折中。

> **默认推荐**：默认 16 字节对齐存储，使用未对齐加载指令。只有在经过 Profiler 确认某段 SIMD 代码是瓶颈且对齐加载能显著改善时，才局部升级到 32 字节对齐。

## 问题 5：网络同步和回放时，为什么同样的代码算出不同结果？

你的物理系统在两台机器上运行完全相同的逻辑：给刚体施加一个力，积分速度，更新位置。按理说，10 秒后两台机器上的位置应该完全一致。但现实是：它们差了 0.003 米——足以让碰撞检测在一台机器上触发，另一台上不触发。

**元凶是浮点运算的非确定性。**

编译器为了优化性能，会悄悄做两件事：

1. **重排运算顺序**：`a * b + c * d` 可能被改成 `c * d + a * b`。数学上等价，但浮点乘法和加法的舍入顺序不同，结果可能有 ±1 ULP 的差异。
2. **自动融合乘加（FMA）**：把 `a * b + c` 合并成一条 FMA 指令。FMA 只做一次舍入，而分开做有两次舍入，结果不同。更糟的是，x86 的 AVX2 支持 FMA，但旧 CPU 和某些 ARM 核心不支持，行为分裂。

**改进方案：确定性策略**

随机数是最容易统一的部分。彻底屏蔽 `std::random_device`（它依赖操作系统熵源，不同平台行为不同），改用自研的确定性算法：

```cpp
class RandomXORShift32 {
    u32 m_state;
public:
    explicit RandomXORShift32(u32 seed)
        : m_state(seed == 0 ? 0xBAD5EEDu : seed) {}

    u32 NextU32() {
        m_state ^= m_state << 13;
        m_state ^= m_state >> 17;
        m_state ^= m_state << 5;
        return m_state;
    }

    f32 NextFloat01() {
        constexpr u32 MAGIC = 0x3F800000u;
        u32 bits = MAGIC | (NextU32() >> 9);
        f32 f; std::memcpy(&f, &bits, sizeof(f));
        return f - 1.0f;
    }
};
```

`xorshift32` 只有 5 行位运算，没有乘法，没有平台差异。给定相同种子，任何编译器、任何 CPU 都会吐出完全相同的序列。

浮点确定性更棘手。可以采取以下**工业级策略**（按严格程度分层）：

1. **关闭激进优化**：在需要确定性的模块（物理积分、随机数种子、动画评估）中，局部关闭编译器的 fast-math 等激进优化标志。
2. **避免 FMA 分歧**：在关键路径上，显式写成 `(a * b) + c`，禁止编译器自动融合。或者统一开启/关闭 FMA 指令集。
3. **固定点数学（可选）**：对于极其严苛的物理同步（如格斗游戏的帧级判定），把位置和速度用 32 位或 64 位定点数表示，彻底消除浮点舍入问题。但这会牺牲动态范围，只适合特定子系统。

> **参考：UE 的 NaN 诊断**
>
> UE 的物理和动画系统在 Development 构建中开启 NaN 诊断，在每次向量运算后自动检查是否产生了非数，防止一个非法的除以零污染整个物理状态和渲染管线。这是一个低成本高回报的调试工具，建议在你的引擎中也加入可开关的 NaN/Inf 检查断言。

## 问题 6：GPU 带宽和顶点压缩需要半精度浮点，但 C++ 原生不支持？

现代 GPU 的顶点格式和纹理压缩广泛使用 IEEE 754 half-precision（1 位符号 + 5 位指数 + 10 位尾数），只占 16 位。如果你坚持用 `float`（32 位）存储所有顶点数据，一个 10 万面的角色模型在显存里会多占一倍空间，移动端直接爆显存。

但 C++23 之前标准库没有原生半精度类型。你需要自研一个轻量包装：

```cpp
struct Float16 {
    u16 bits;

    static Float16 FromFloat32(f32 value);
    f32 ToFloat32() const;
};
```

转换策略通常用**查表法**：

- `float → half`：借助 512 条目的 base_table 和 shift_table 快速映射指数和尾数，无需分支。
- `half → float`：借助 2048 条目的 mantissa_table 和 64 条目的 exponent_table 恢复完整精度。
- 全程使用 `std::memcpy` 进行位模式读写，避免 strict aliasing 违规。

这看起来是个边缘需求，但当你进入渲染管线阶段（阶段 5），会发现 half 类型是顶点压缩和 GPU 带宽优化的基石。

> **默认推荐**：先实现 `Float16` 的查表转换函数，但**不要**在 CPU 端数学运算中广泛使用它。half 的精度和范围都有限，做矩阵乘法会累积误差。它只应该出现在"数据存储和传输"路径（顶点缓冲区、纹理像素、网络同步的压缩包），不该出现在"运算"路径。

## 问题 7：ECS 架构下，数学类型作为组件有什么特殊考量？

前面我们解决了"数学库本身该怎么设计"。现在回到 ECS 语境：你的变换组件应该长什么样？

最 naive 的方案是直接复用上面的 `Vec3` 和 `Quat`：

```cpp
struct Transform {
    Vec3 translation;
    Quat rotation;
    Vec3 scale;
};
```

这在单个实体层面完全正确。但在 ECS 批量处理时，你很快就会遇到两个问题：

**问题 A：本地坐标 vs 世界坐标的分离**

如果一个实体有父节点，它的 `translation` 是相对于父节点的。但渲染系统需要世界坐标下的矩阵。如果每次渲染系统都重新计算一遍世界矩阵，父节点深嵌套时会产生大量重复计算。

**改进：分离存储**

把"本地变换"和"世界变换"拆成两个组件：

```cpp
struct Transform {
    Vec3 translation;
    Quat rotation;
    Vec3 scale;
};

struct GlobalTransform {
    Mat4 matrix; // 缓存的世界矩阵
};
```

一个专门的系统负责遍历层级树，把 `Transform` 转换成 `GlobalTransform`，并只在 `Transform` 或父节点变化时更新。其他系统（渲染、物理、碰撞）只读 `GlobalTransform`，不参与层级计算。

这种分离还有额外的好处：**AI 可观测性**。AI Agent 想修改一个物体的位置时，它修改的是 `Transform.translation`（本地坐标），不会影响层级关系；而 AI 想查询一个物体的世界位置时，它读取 `GlobalTransform.matrix`（已计算好的结果）。两个操作都有明确的语义边界。

> **参考：Bevy 的 Transform 分离与 UE 的相对/世界变换**
>
> Bevy 明确区分了本地 `Transform` 和世界 `GlobalTransform` 两个组件。`GlobalTransform` 由专门的系统在每帧自动更新。这种分离让渲染系统和其他游戏逻辑系统无需关心层级结构，直接读取世界坐标。
>
> UE 的场景组件也维护了相对变换和世界变换两个状态，但因为是 OOP 树形结构，更新逻辑嵌套在组件内部。迁移到 ECS 时，这两个状态应该被显式拆分为独立组件，由独立系统批量更新——这正是 Bevy 已经验证过的模式。

**问题 B：脏标记与批量更新**

如果每帧都全量计算所有 `GlobalTransform`，即使 90% 的物体没动，也在做无用功。

**改进：脏标记传播**

在 `Transform` 组件里加一个标志位：

```cpp
struct Transform {
    Vec3 translation;
    Quat rotation;
    Vec3 scale;
    u32 dirty : 1; // 本地变换是否被修改
};
```

当 Inspector 或 AI Agent 修改了 `Transform`，把 `dirty` 设为 true。变换传播系统只处理 `dirty == true` 的实体，并把脏标记沿父子链向下传播。这在静态场景占多数时，能节省大量 CPU 时间。

## 几何查询最小集

数学库的最后一块拼图是碰撞检测和裁剪的基础工具。不需要实现完整物理引擎，但渲染器和 gameplay 系统需要以下基础图元：

```cpp
struct AABB {
    Vec3 min;
    Vec3 max;

    bool Intersects(const AABB& other) const {
        return min.x <= other.max.x && max.x >= other.min.x
            && min.y <= other.max.y && max.y >= other.min.y
            && min.z <= other.max.z && max.z >= other.min.z;
    }

    bool Contains(const Vec3& point) const {
        return point.x >= min.x && point.x <= max.x
            && point.y >= min.y && point.y <= max.y
            && point.z >= min.z && point.z <= max.z;
    }
};

struct Ray {
    Vec3 origin;
    Vec3 direction; // 必须已归一化

    bool IntersectAABB(const AABB& box, f32& out_t) const {
        // Slab method：对三个轴分别求进入/离开时间
        f32 tmin = 0.0f, tmax = FLT_MAX;
        for (int i = 0; i < 3; ++i) {
            if (std::abs(direction[i]) < 1e-6f) {
                if (origin[i] < box.min[i] || origin[i] > box.max[i]) return false;
            } else {
                f32 ood = 1.0f / direction[i];
                f32 t1 = (box.min[i] - origin[i]) * ood;
                f32 t2 = (box.max[i] - origin[i]) * ood;
                tmin = std::max(tmin, std::min(t1, t2));
                tmax = std::min(tmax, std::max(t1, t2));
                if (tmin > tmax) return false;
            }
        }
        out_t = tmin;
        return true;
    }
};
```

AABB 和射线检测是视锥剔除、鼠标拾取、简单触发器的基石。它们不需要 SIMD 也能跑得很快（分支少、计算简单），所以保持标量实现即可。当实体数量达到数万级时，再考虑用 SIMD 批量做 AABB 相交测试。

---

> **默认推荐路径总结**：
>
> 数学库的工业级默认路径是：**纯 POD 的 `Vec3`/`Vec4`/`Mat4`/`Quat` + 列主序矩阵 + NLERP 默认插值 + 编译期 SIMD 抽象 + 确定性 `xorshift32` + 标量 AABB/Ray**。该路径在 ECS 兼容前提下吸收了以下工业级设计：
> - **UE 的 Transform 批量处理**：逻辑层用 AoS（`Transform` 组件），运算层显式拆分为 SoA 进行 SIMD 批量处理。这保证了 ECS 组件存储的紧凑性，同时不牺牲批量运算的 cache 效率。
> - **glam 的跨平台一致性**：标量回退时仍保持 16 字节对齐和内部填充，确保对象尺寸在所有架构上一致——这是网络同步和序列化的基石。
> - **UE 的 NaN 诊断**：在 Development 构建中开启可开关的 NaN/Inf 检查断言，防止非法运算污染物理状态和渲染管线。
>
> **ECS 映射**：`Vec3`/`Quat`/`Mat4` 均为纯 POD，无虚函数、无指针、无动态分配，可直接作为 ECS 组件被完美平铺到连续内存。`Transform`（本地）和 `GlobalTransform`（世界）分离为两个独立组件，由专用 System 批量更新——这正是 Bevy 已验证的 ECS 原生模式。
>
> **条件切换路径**：如果目标平台是仅支持标量的嵌入式设备，SIMD 抽象层完全退化为标量实现，但接口保持不变；如果引擎需要精确镜头控制（过场动画），将 NLERP 切换为 SLERP。
>
> **扩展路径（更极端工业场景）**：
> - **SIMD 多版本分发**：运行时检测 CPU 特性（SSE4.2 / AVX2 / NEON），函数指针指向不同实现版本（参考 `网络抽象与硬件检测` 中的 `InitSimdDispatch`）。
> - **固定点数学**：对于格斗游戏等需要帧级确定性判定的场景，将位置和速度用 32/64 位定点数表示，彻底消除浮点舍入问题。
> - **批量几何查询**：用 SIMD 批量做 AABB 相交测试、视锥体裁剪，支撑数万级实体的高效剔除。
> - **GPU 半精度顶点**：`Float16` 查表转换用于顶点压缩和 GPU 带宽优化，但限制在"数据存储和传输"路径。
