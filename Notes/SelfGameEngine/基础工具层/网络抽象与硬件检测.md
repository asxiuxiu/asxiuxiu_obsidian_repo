# 网络抽象与硬件检测

## Why：为什么引擎需要网络抽象与硬件检测？

这看起来是两个不相关的主题，但它们有一个共同点：**直面操作系统和硬件的差异**。标准库对这两块要么覆盖不全（C++ 没有标准 Socket API），要么抽象层级太高（`std::thread::hardware_concurrency()` 提供的信息太粗糙）。

### 没有网络抽象时世界有多糟

- **Windows 用 `WSAStartup`**，Linux/macOS 不用。你的网络代码里塞满 `#ifdef _WIN32`，可读性极差。
- **`send()` 的返回值处理不一致**：Windows 的 `send` 在连接断开时可能返回 `SOCKET_ERROR`，而 POSIX 返回实际发送字节数或 `-1`。没有统一封装，网络错误处理漏洞百出。
- **IPv4 vs IPv6**：直接操作 `sockaddr_in` 和 `sockaddr_in6` 会让地址解析代码膨胀三倍。

### 没有硬件检测时世界有多糟

- **SIMD 代码在旧 CPU 上直接崩溃**：你写了 AVX2 指令，但玩家的 CPU 只支持 SSE4.1，启动即 Illegal Instruction。
- **不知道核心数**：`parallel_for` 默认开 8 个线程，但玩家的机器只有 2 核，上下文切换把性能吃光。
- **主机/移动端限制**：某些平台禁止 `dlopen`/`LoadLibrary`，动态库加载需要平台特定的替代方案。

### 三个真实场景

| 场景 | 没有抽象层会怎样 | 后果 |
|------|----------------|------|
| 发布到 Steam Deck | 直接用 WinSock API | Linux 构建失败 |
| 粒子系统用 AVX2 | 老 CPU 玩家启动崩溃 | 差评如潮，退款率上升 |
| 加载第三方插件 | Windows 用 `.dll`，Linux 用 `.so` | 插件系统代码复制粘贴两份 |

## What：最小网络抽象 + 硬件检测长什么样？

下面给出约 130 行的最小实现，包含：跨平台 Socket 初始化/关闭、TCP 发送/接收封装、CPU SIMD 特性查询、动态库加载。

### 1. Socket 跨平台封装

```cpp
#if defined(PLATFORM_WINDOWS)
    #include <winsock2.h>
    #include <ws2tcpip.h>
    #pragma comment(lib, "ws2_32.lib")
    using SocketT = SOCKET;
    constexpr SocketT INVALID_SOCKET_HANDLE = INVALID_SOCKET;
#else
    #include <sys/socket.h>
    #include <netinet/in.h>
    #include <arpa/inet.h>
    #include <unistd.h>
    #include <fcntl.h>
    using SocketT = int;
    constexpr SocketT INVALID_SOCKET_HANDLE = -1;
#endif

struct NetResult {
    bool ok;
    int code;      // 错误码或发送/接收字节数
};

class Network {
    bool m_wsaInitialized = false;

public:
    bool Init() {
#if defined(PLATFORM_WINDOWS)
        WSADATA wsaData;
        m_wsaInitialized = (WSAStartup(MAKEWORD(2, 2), &wsaData) == 0);
        return m_wsaInitialized;
#else
        return true;
#endif
    }

    void Shutdown() {
#if defined(PLATFORM_WINDOWS)
        if (m_wsaInitialized) WSACleanup();
#endif
    }

    static void CloseSocket(SocketT s) {
#if defined(PLATFORM_WINDOWS)
        closesocket(s);
#else
        close(s);
#endif
    }

    // 设置非阻塞模式
    static bool SetNonBlocking(SocketT s) {
#if defined(PLATFORM_WINDOWS)
        u_long mode = 1;
        return ioctlsocket(s, FIONBIO, &mode) == 0;
#else
        int flags = fcntl(s, F_GETFL, 0);
        return fcntl(s, F_SETFL, flags | O_NONBLOCK) != -1;
#endif
    }

    static NetResult Send(SocketT s, const void* data, usize len) {
        int sent = ::send(s, static_cast<const char*>(data), static_cast<int>(len), 0);
        if (sent < 0) {
#if defined(PLATFORM_WINDOWS)
            int err = WSAGetLastError();
            if (err == WSAEWOULDBLOCK) return {true, 0}; // 非阻塞，稍后再试
            return {false, err};
#else
            if (errno == EAGAIN || errno == EWOULDBLOCK) return {true, 0};
            return {false, errno};
#endif
        }
        return {true, sent};
    }

    static NetResult Recv(SocketT s, void* buffer, usize maxLen) {
        int received = ::recv(s, static_cast<char*>(buffer), static_cast<int>(maxLen), 0);
        if (received < 0) {
#if defined(PLATFORM_WINDOWS)
            int err = WSAGetLastError();
            if (err == WSAEWOULDBLOCK) return {true, 0};
            return {false, err};
#else
            if (errno == EAGAIN || errno == EWOULDBLOCK) return {true, 0};
            return {false, errno};
#endif
        }
        if (received == 0) return {false, 0}; // 连接断开
        return {true, received};
    }
};
```

> **这段代码已经解决了什么**：封装了 Windows/Unix Socket API 的差异、统一了非阻塞模式设置、统一了错误码处理（区分"无数据可读"和"真正错误"）。
> **还缺什么**：没有地址解析（`getaddrinfo` 封装）、没有 UDP 支持、没有连接管理（connect/accept 的封装）。
> **反例警告**：如果你正在写真正的多人网络游戏，**不要自己从头写可靠传输层**。应使用 [ENet](http://enet.bespin.org/)、[KCP](https://github.com/skywind3000/kcp) 或 [GameNetworkingSockets](https://github.com/ValveSoftware/GameNetworkingSockets)。这个封装只是为了让你理解底层接口长什么样，以及在引擎中如何隐藏平台差异。

### 2. CPU / SIMD 特性检测

```cpp
#if defined(_MSC_VER)
    #include <intrin.h>
#else
    #include <cpuid.h>
#endif

struct CpuFeatures {
    bool sse2 = false;
    bool sse4_1 = false;
    bool avx = false;
    bool avx2 = false;
    u32 logicalCores = 1;
};

class Hardware {
public:
    static CpuFeatures Query() {
        CpuFeatures f;
        f.logicalCores = std::thread::hardware_concurrency();
        if (f.logicalCores == 0) f.logicalCores = 1;

#if defined(ARCH_64BIT) && (defined(_MSC_VER) || defined(__GNUC__))
        int regs[4] = {};
#if defined(_MSC_VER)
        __cpuid(regs, 1);
#else
        __cpuid(1, regs[0], regs[1], regs[2], regs[3]);
#endif
        f.sse2    = (regs[3] & (1 << 26)) != 0;
        f.sse4_1  = (regs[2] & (1 << 19)) != 0;

        // 检查 OS 是否支持 XSAVE（AVX 的前提）
        bool osxsave = (regs[2] & (1 << 27)) != 0;
        if (osxsave) {
            u64 xcr0 = 0;
#if defined(_MSC_VER)
            xcr0 = _xgetbv(0);
#else
            __asm__ __volatile__("xgetbv" : "=a"((u32)xcr0), "=d"(*(((u32*)&xcr0)+1)) : "c"(0));
#endif
            if ((xcr0 & 0x6) == 0x6) {
#if defined(_MSC_VER)
                __cpuidex(regs, 7, 0);
#else
                __cpuid_count(7, 0, regs[0], regs[1], regs[2], regs[3]);
#endif
                f.avx  = (regs[2] & (1 << 28)) != 0;
                f.avx2 = (regs[1] & (1 << 5)) != 0;
            }
        }
#endif
        return f;
    }
};
```

> **这段代码已经解决了什么**：在 x86/x64 平台上检测 SSE2/SSE4.1/AVX/AVX2 支持、查询逻辑核心数。
> **还缺什么**：没有 ARM NEON 检测、没有缓存大小查询（L1/L2/L3）、没有物理核心 vs 逻辑核心区分。
> **反例警告**：`cpuid` 只能在 x86/x64 上用。如果你的引擎要发布到 ARM（如 Apple Silicon、Android、Switch），必须另外实现平台分支（如读取 Linux `/proc/cpuinfo` 或使用 ARM 的 `getauxval(AT_HWCAP)`）。

### 3. 动态库加载（插件系统基础）

```cpp
#if defined(PLATFORM_WINDOWS)
    #include <windows.h>
    using DynLibHandle = HMODULE;
#else
    #include <dlfcn.h>
    using DynLibHandle = void*;
#endif

class DynamicLibrary {
public:
    static DynLibHandle Load(const char* path) {
#if defined(PLATFORM_WINDOWS)
        return LoadLibraryA(path);
#else
        return dlopen(path, RTLD_NOW);
#endif
    }

    static void Unload(DynLibHandle handle) {
        if (!handle) return;
#if defined(PLATFORM_WINDOWS)
        FreeLibrary(handle);
#else
        dlclose(handle);
#endif
    }

    static void* GetSymbol(DynLibHandle handle, const char* name) {
        if (!handle) return nullptr;
#if defined(PLATFORM_WINDOWS)
        return reinterpret_cast<void*>(GetProcAddress(handle, name));
#else
        return dlsym(handle, name);
#endif
    }
};
```

> **这段代码已经解决了什么**：统一了 Windows `.dll` 和 Linux/macOS `.so` / `.dylib` 的加载接口。
> **还缺什么**：没有路径后缀自动拼接（`.dll` vs `.so`）、没有符号版本管理、没有插件热重载的安全卸载机制（卸载一个正在被使用的 DLL 是未定义行为）。
> **反例警告**：某些平台（如 iOS、主机）禁止或严格限制动态库加载。在这些平台上，插件系统可能需要静态链接或使用脚本虚拟机替代。

## How：真实引擎的网络与硬件层是如何一步一步复杂起来的？

### 阶段 1：平台宏封装 → 能用

这就是我们上面的代码。它消除了最明显的平台差异，让你的引擎能在 Windows 和 Linux 上同时编译网络模块和 SIMD 代码。

### 阶段 2：异步网络 + SIMD 分发 + 插件管理 → 好用

**触发原因**：阻塞的网络 IO 会卡死帧循环；手动为每个 SIMD 级别写一份代码不现实；插件数量增加后需要生命周期管理。

**代码层面的变化**：

1. **网络 IO 与线程池结合**：
   - 将 Socket 的读写操作交给 [[线程池与任务系统]] 的后台线程。
   - 主线程只在回调队列中处理已接收的网络包。
   - 使用 `select` / `poll` / `epoll` / `kqueue` / IOCP 实现高效的多连接管理。

2. **SIMD 多版本分发（Dispatch）**：
   - 在启动时检测 CPU 支持的 SIMD 级别。
   - 根据检测结果，函数指针指向不同的实现版本（SSE2 版、AVX2 版、NEON 版）。

```cpp
using TransformFn = void(*)(const f32* in, f32* out, usize count);
TransformFn g_transformPoints = nullptr;

void InitSimdDispatch() {
    auto cpu = Hardware::Query();
    if (cpu.avx2)      g_transformPoints = TransformPoints_AVX2;
    else if (cpu.sse2) g_transformPoints = TransformPoints_SSE2;
    else               g_transformPoints = TransformPoints_Scalar;
}
```

3. **插件注册表**：
   - 动态库导出统一的 `RegisterPlugin()` 符号。
   - 引擎加载后调用注册函数，获取插件的元数据（名称、版本、依赖系统列表）。
   - 在 Editor 或 Debug 构建中支持热重载：卸载旧库 → 加载新库 → 重新注册。

### 阶段 3：工业级网络 + 全平台适配 → 生产

**触发原因**：多人联机需要低延迟、高吞吐、断线重连；平台数量扩大到主机、移动端、Web。

1. **可靠 UDP（如 KCP）**：在 UDP 之上实现顺序保证、重传、拥塞控制， latency 远低于 TCP。
2. **网络压缩与加密**：对高频状态同步包做增量压缩（如只发送变化字段），并对敏感数据做 AES/XChaCha20 加密。
3. **平台特定的硬件查询**：
   - Android：`getauxval(AT_HWCAP)` 检测 NEON。
   - Apple Silicon：运行时检测 `__builtin_cpu_supports("neon")`。
   - 主机：使用厂商 SDK 提供的专用 API 查询 GPU/CPU 能力。
4. **网关与消息路由**：客户端不直接连接 GameServer，而是通过 Gateway 做认证、负载均衡、断线快速切换。

这部分的详细设计见 [[网络基础设施]] 和 [[服务器集群架构]]。

## 性能与 Trade-off：诚实分析

| 设计点 | 优势 | 代价 | 反例警告 |
|--------|------|------|----------|
| 跨平台 Socket 宏封装 | 一份代码多平台编译 | 宏使调试困难；隐藏了平台特有的性能陷阱 | 高并发场景下，`select` 的性能远低于 `epoll` 或 IOCP，不能只做最简封装 |
| 运行时 SIMD 分发 | 一份源码生成多版本，自动适配硬件 | 启动时增加检测开销；函数指针间接调用有微小开销 | 对于极小函数（几个指令），函数指针开销可能抵消 SIMD 收益，应用内联模板替代 |
| 动态库插件 | 运行时扩展能力，无需重新编译引擎 | 平台限制多（iOS/主机禁止）；ABI 兼容风险高 | 不要在热路径上频繁穿越 DLL 边界，虚函数表间接调用开销显著 |

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 是 | `CpuFeatures` 是纯 POD；Socket 句柄是标量；插件注册表是结构化的数组/映射 |
| **自描述** | ✅ 是 | AI 可以通过 `Hardware::Query()` 直接读取 CPU 能力；通过网络抽象的封装接口操作 Socket，无需理解平台差异 |
| **确定性** | ⚠️ 需约束 | 网络包的到达时间本质是非确定性的。应在 [[系统调度与确定性]] 层将网络输入视为"外部事件序列"，按固定顺序注入世界，才能保持回放一致性 |
| **工具边界** | ✅ 是 | AI 操作网络的 MCP 接口应限定为：`send_packet(target, payload)`、`query_cpu_features()`、`load_plugin(path)` |
| **Agent 安全** | ⚠️ 需限制 | AI 不应被允许执行任意网络连接或加载未验证的动态库。插件加载必须白名单化，网络目标地址必须受控 |
