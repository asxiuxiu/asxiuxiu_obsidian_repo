# 引擎基础类型与平台抽象

## Why：为什么引擎需要自己的基础类型层？

C++ 标准库看起来提供了 `int`、`long`、`size_t` 等类型，但当你开始跨平台编译引擎时，噩梦才刚刚开始：

- **Windows 64 位**：`long` 是 32 位；**Linux 64 位**：`long` 是 64 位。你把一个 `long` 序列化到文件，在另一台机器上读取，数值直接错乱。
- **`size_t` 的陷阱**：它在 32 位和 64 位平台上宽度不同。如果你用 `size_t` 作为网络协议字段，不同架构的客户端连握手都过不了。
- **导出符号地狱**：Windows 需要 `__declspec(dllexport)`，Linux/macOS 需要 `__attribute__((visibility("default")))`。没有统一宏，跨平台动态库根本无法构建。
- **断言行为不一致**：`assert()` 在 Release 构建中可能被完全剥离，导致线上崩溃时拿不到任何文件/行号信息。

### 三个真实场景

| 场景 | 没有基础类型层会怎样 | 后果 |
|------|---------------------|------|
| 存档文件跨平台读取 | Windows `long` 写入 4 字节，Linux 按 8 字节读取 | 存档损坏，玩家进度丢失 |
| 网络同步 Entity ID | 一端用 `unsigned int`，另一端用 `uint64_t` | 协议不匹配，联机直接断开 |
| 插件系统跨平台 | Windows 导出了 1000 个符号，Linux 默认全隐藏 | 插件加载失败，链接器报错 |

**结论**：一个自研引擎必须在接触任何业务代码之前，先**封印平台差异**。这不是过度工程，而是后续所有模块能无痛跨平台的前提。

## What：最小基础类型层长什么样？

下面给出约 80 行的最小实现。它包含固定宽度整数、平台检测宏、导出符号宏、分层断言和调试填充。

```cpp
#include <cstdint>
#include <cstddef>

// ========== 1. 固定宽度整数 ==========
using u8  = uint8_t;
using u16 = uint16_t;
using u32 = uint32_t;
using u64 = uint64_t;
using i8  = int8_t;
using i16 = int16_t;
using i32 = int32_t;
using i64 = int64_t;
using f32 = float;
using f64 = double;

// 显式尺寸类型（避免 size_t 的跨平台宽度问题）
using usize = std::size_t;
using isize = std::ptrdiff_t;

// ========== 2. 平台检测 ==========
#if defined(_WIN32)
    #define PLATFORM_WINDOWS 1
#elif defined(__APPLE__)
    #define PLATFORM_MACOS 1
#elif defined(__linux__)
    #define PLATFORM_LINUX 1
#else
    #define PLATFORM_UNKNOWN 1
#endif

#if defined(_M_X64) || defined(__x86_64__)
    #define ARCH_64BIT 1
#else
    #define ARCH_32BIT 1
#endif

// ========== 3. 导出符号宏 ==========
#if defined(PLATFORM_WINDOWS)
    #ifdef ENGINE_BUILD_DLL
        #define ENGINE_API __declspec(dllexport)
    #else
        #define ENGINE_API __declspec(dllimport)
    #endif
#else
    #define ENGINE_API __attribute__((visibility("default")))
#endif

// ========== 4. 断言与调试钩子 ==========
#ifdef DEBUG
    #include <cstdlib>
    #include <cstdio>
    #define DEBUG_BREAK() __debugbreak()  // MSVC; GCC/Clang 可换 __builtin_trap()
    #define ASSERT(cond) \
        do { if (!(cond)) { \
            std::fprintf(stderr, "ASSERT FAILED: %s at %s:%d\n", #cond, __FILE__, __LINE__); \
            DEBUG_BREAK(); \
            std::abort(); \
        } } while(0)
#else
    #define ASSERT(cond) ((void)0)
#endif

// 编译期断言（C++11 起可用 static_assert，这里作为风格示例）
#define STATIC_ASSERT(cond, msg) static_assert(cond, msg)

// ========== 5. 调试填充（检测 use-after-free / 越界） ==========
#ifdef DEBUG
    constexpr u8 CANARY_ALLOC = 0xCD; // Clean memory
    constexpr u8 CANARY_FREE  = 0xFE; // Freed memory
    inline void DebugFill(void* ptr, usize size, u8 value) {
        std::memset(ptr, value, size);
    }
#else
    inline void DebugFill(void*, usize, u8) {}
#endif
```

### 这段代码已经解决了什么

| 能力 | 说明 |
|------|------|
| **跨平台类型统一** | `u32` 在任何平台都是 32 位无符号整数，网络协议和存档文件从此安全 |
| **编译期平台分支** | 通过 `PLATFORM_WINDOWS` 等宏，渲染后端、文件系统可以写平台无关的接口 |
| **动态库符号可控** | `ENGINE_API` 让核心模块的导出符号在 Windows/Linux 上一致 |
| **断言分层** | Debug 构建中断言带文件行号并可触发调试器；Release 构建零开销 |
| **调试验证** | `DebugFill` 为内存调试提供基础设施，检测未初始化读取和释放后使用 |

### 还缺什么

- 没有 **SIMD 类型封装**（如 `__m128` / `float32x4_t` 的平台抽象），这部分交给 [[数学基础]]。
- 没有 **运行时硬件能力查询**（CPU 是否支持 AVX2？有几个核心？），这部分交给 [[网络抽象与硬件检测]]。
- 没有 **原子类型和内存序封装**，标准库的 `std::atomic` 在原型阶段足够用，工业级时再做统一别名。

## How：真实引擎的基础层是如何一步一步复杂起来的？

### 阶段 1：能用 —— 类型别名 + 平台宏 + 断言

这就是上面的代码。目标是**消除最明显的跨平台陷阱**。在这个阶段，你的引擎可以在 Windows 和 Linux 上编译通过，网络协议字段宽度一致，动态库能正确导出符号。

### 阶段 2：好用 —— 日志集成断言、崩溃报告、对齐工具

当团队扩大、测试场景复杂化后，简单的 `fprintf + abort` 不够用了：

1. **断言与日志系统联动**：断言触发时，不仅打印到 stderr，还写入日志文件和屏幕面板（见 [[可视化日志系统]]）。
2. **崩溃报告（Crash Handler）**：注册信号处理器（`SIGSEGV`、`SIGABRT`、Windows 的 `SetUnhandledExceptionFilter`），在崩溃时捕获调用栈、模块列表、寄存器状态，输出到 `.dmp` 或文本文件。
3. **对齐工具**：为 SIMD 和 GPU 缓冲区提供统一的 `AlignUp`、`AlignDown`、`IsAligned` 工具。

```cpp
constexpr usize AlignUp(usize value, usize alignment) {
    usize mask = alignment - 1;
    return (value + mask) & ~mask;
}
constexpr bool IsPowerOfTwo(usize v) { return v && (v & (v - 1)) == 0; }
```

### 阶段 3：工业级 —— 编译器属性封装、ABI 隔离、构建时检测

走到这一步，引擎开始在多主机、多编译器（MSVC/GCC/Clang/主机工具链）上发布。

1. **编译器特性宏**：封装 `__forceinline`、`[[nodiscard]]`、`[[maybe_unused]]`、 `LIKELY`/`UNLIKELY` 等分支提示。
2. **ABI 隔离层**：引擎核心与第三方库（如物理引擎、音频中间件）使用不同的 CRT 或 STL 版本时，通过 `extern "C"` 的 C 接口和显式内存分配回调隔离 ABI 冲突。
3. **构建时静态检查**：
   - `static_assert(sizeof(void*) == 8, "64-bit only")` 防止误构建 32 位版本。
   - `static_assert(alignof(std::max_align_t) >= 16, "SIMD alignment required")` 确保分配器默认对齐满足 SIMD 需求。

## 性能与 Trade-off：诚实分析

| 设计点 | 优势 | 代价 | 反例警告 |
|--------|------|------|----------|
| 固定宽度别名（`u32` 等） | 跨平台一致性、网络/存档安全 | 代码中要多写一层类型名 | 无 |
| `ENGINE_API` 宏抽象 | 一份代码通吃 Windows/Linux | 宏定义增加了构建系统的复杂度（需要定义 `ENGINE_BUILD_DLL`） | 静态链接时宏定义不匹配会导致符号找不到 |
| `ASSERT` 在 Release 中剥离 | 零运行时开销 | 线上崩溃拿不到断言信息 | 不要把"可能发生的错误"放进 `ASSERT`，应用 `if + error handling` |

## AI 友好设计红线检查

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 是 | 基础类型全是编译期别名和宏，不引入任何运行时状态或动态分配 |
| **自描述** | ✅ 是 | `PLATFORM_WINDOWS` / `ARCH_64BIT` 等宏本身就是自描述的构建配置，AI 可通过编译数据库直接读取 |
| **确定性** | ✅ 是 | 固定宽度整数保证了不同平台上相同数值的位模式一致，是网络同步和回放系统的基石 |
| **工具边界** | ✅ 是 | AI 修改基础类型不会影响引擎行为（只要不改变宽度），风险极低 |
| **Agent 安全** | ✅ 是 | 纯头文件基础设施，无副作用，不存在误操作破坏全局状态的问题 |
