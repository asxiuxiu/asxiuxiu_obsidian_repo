# 引擎基础类型与平台抽象

## 问题 0：为什么引擎不能直接用 `int` 和 `long`？

假设你写了一个存档系统。玩家在 Windows 上攒了 21 亿金币，你用一个 `long` 把它写进存档文件——占 4 个字节。玩家把存档复制到 Linux 服务器上做联机验证，服务器上的程序也用 `long` 读取——但 Linux 64 位的 `long` 是 8 个字节。你的读取代码只读 4 字节，剩下的 4 字节被当成下一个字段的一部分。金币数变成一个天文数字，服务器判定"数据异常"，直接把存档 ban 了。

这不是虚构。C++ 标准故意不规定 `int`、`long`、`size_t` 的精确宽度：

- **Windows 64 位**：`long` = 32 位，`long long` = 64 位
- **Linux 64 位**：`long` = 64 位，`long long` = 64 位
- **`size_t`**：32 位平台 4 字节，64 位平台 8 字节——如果你把它塞进网络协议，两端根本握不了手

最 naive 的方案是什么？用 C99 的 `<cstdint>`：`int32_t`、`uint64_t`。这已经是工业共识（Barr Group 嵌入式 C 规范 5.2 条明确禁止在需要确定宽度的场景使用 `short` 和 `long`）。但 `<cstdint>` 的类型名太长，写起来累，而且它只是"有"，不是"好用"。

所以引擎的第一步，是给这些固定宽度类型起短别名：

```cpp
#include <cstdint>
#include <cstddef>

using u8  = uint8_t;   using i8  = int8_t;
using u16 = uint16_t;  using i16 = int16_t;
using u32 = uint32_t;  using i32 = int32_t;
using u64 = uint64_t;  using i64 = int64_t;
using f32 = float;     using f64 = double;

using usize = std::size_t;
using isize = std::ptrdiff_t;
```

这 12 行代码本身不产生任何运行时开销——全是编译期别名。但从此你的网络协议、存档格式、GPU 缓冲区布局，在不同平台上有了**位精确一致**的语义。这是后续一切确定性设计（网络同步、回放系统、AI Schema）的基石。

> **还遗留的问题**：类型统一了，但编译时怎么知道"当前在哪个平台"？不同平台的 API 不一样，引擎怎么写平台无关的代码？

## 问题 1：如何"封印"平台差异，避免 `#ifdef` 地狱？

每个写跨平台代码的人都写过这样的东西：

```cpp
// 最 naive 的方案
#ifdef _WIN32
    void* handle = LoadLibraryA("plugin.dll");
#elif defined(__linux__)
    void* handle = dlopen("plugin.so", RTLD_NOW);
#endif
```

问题在哪？`_WIN32` 这个宏在 Windows 上存在，在别的平台上**未定义**。如果你拼错了——比如写成 `#ifdef WIN32`（少了一个下划线）——编译器不会报错，只是静默跳过整段代码。你花了两小时调试，最后发现是宏名拼错了一个字符。

更隐蔽的是 `#ifdef` 的连锁反应。一个项目里有 50 个文件，每个文件都 `#ifdef _WIN32`，然后某个新同事在 macOS 上加了一段 `#ifdef __APPLE__`，结果和已有的平台判断逻辑冲突，编译通过但运行时行为诡异。

### 改进方案：预置为零 + 强制 `#if`

UE 的做法是：所有平台宏先统一设为 `0`，再用平台特定头文件覆盖为 `1`。这样你只能写 `#if PLATFORM_WINDOWS`，如果拼错成 `#if PLATFORM_WINDOS`，编译器会因为"未定义标识符"直接报错。

```cpp
// 改进后的平台检测（学习 UE 模式）
#define PLATFORM_WINDOWS 0
#define PLATFORM_LINUX   0
#define PLATFORM_MACOS   0
#define PLATFORM_ANDROID 0
#define PLATFORM_IOS     0

#if defined(_WIN32)
    #undef PLATFORM_WINDOWS
    #define PLATFORM_WINDOWS 1
#elif defined(__linux__)
    #undef PLATFORM_LINUX
    #define PLATFORM_LINUX 1
#elif defined(__APPLE__)
    #include <TargetConditionals.h>
    #if TARGET_OS_IPHONE
        #undef PLATFORM_IOS
        #define PLATFORM_IOS 1
    #else
        #undef PLATFORM_MACOS
        #define PLATFORM_MACOS 1
    #endif
#endif
```

这个模式的核心价值不是"多写几行宏"，而是**把未定义宏的静默失败变成编译期硬错误**。这一措施在任何规模的项目中成本都接近于零，但能避免最浪费时间的那类跨平台编译 bug。

### 构建配置的分层思想

到这里，你已经有了一套平台宏。但引擎的构建维度不止"平台"——还有 Debug/Release、是否启用渲染、是否启用网络、是否启用 AI 桥接。如果所有开关都塞进同一个头文件，很快就会变成一团乱麻。

UE 的解决方案是分层：平台宏归平台宏，构建类型归构建类型，功能开关归功能开关。Bevy 作为 Rust 引擎，给出了一个更系统的参考——它的 Feature 体系分为四层：

| 层级             | Bevy 的命名                         | 对应到 C++ 引擎的设计                         |
| -------------- | -------------------------------- | ------------------------------------- |
| **PROFILE**    | `2d`、`3d`、`ui`、`audio`           | 面向用户的"体验套餐"，CMake `option` 组合         |
| **COLLECTION** | `default_app`、`default_platform` | 中层功能聚合，如 `ENABLE_CORE`、`ENABLE_TOOLS` |
| **MODULE**     | `bevy_render`、`bevy_pbr`         | 与模块一一对应的开关，如 `ENABLE_MODULE_RENDER`   |
| **CAPABILITY** | `png`、`webgl2`、`trace_chrome`    | 最细粒度能力，如 `ENABLE_TRACE`、`USE_WAYLAND` |

Bevy 的精妙之处在于**命名一致性**：feature 名称 = crate 名称 = 模块名称。用户想开 sprite 功能，就写 `bevy_sprite` feature，代码里用 `bevy::sprite::Sprite`，底层 crate 也叫 `bevy_sprite`。对于 C++ 引擎，这意味着你的 CMake option、模块目录名、C++ namespace 应该尽量保持一致——比如渲染模块的开关叫 `ENABLE_RENDER`，目录叫 `modules/render/`，namespace 叫 `render::`。这种一致性大幅降低认知负担和配置出错概率。

Bevy 还有另一个值得借鉴的设计：**条件级联**（`?/` 语法）。开启 `trace` feature 不会强制引入所有 crate，只会给已经激活的 crate 打开追踪能力。映射到 C++，这意味着你的功能开关应该支持"叠加"语义——比如定义了 `ENABLE_TRACE` 后，只有已经编译的模块才会产生追踪输出，而不是强行拉入所有可选模块的依赖。

> **默认推荐**：平台宏采用 UE 的"预置为零 + `#if`"模式——这是将未定义宏的静默失败转化为编译期硬错误的工业级实践；功能开关采用 Bevy 的分层命名策略（PROFILE → MODULE → CAPABILITY），用 CMake `option` 实现。不要在一个头文件里混装所有宏。
>
> **ECS 映射**：平台宏与功能开关全部是编译期常量，不引入任何运行时状态，因此与 ECS 架构完全兼容。固定宽度类型保证组件内存布局跨平台位精确一致，这是 AI Schema 能够精确描述 ECS 组件的物理布局的前提。

## 问题 2：断言该怎么分层，才能在 Debug 时不漏报、在 Ship 时不拖性能？

最 naive 的方案是直接用标准库的 `assert()`：

```cpp
assert(ptr != nullptr);
*ptr = 42;  // 如果 assert 被关闭，这里直接崩溃，没有任何信息
```

`assert()` 的问题有两个：一是 Release 构建中可能被完全剥离，导致你失去对"内部不变量"的检查；二是它不分语义——"这个指针绝不应该为空"和"这个操作可能失败但我想试试"用同一个宏表达。

UE 的解决方案是三级断言语义，这已被广泛认可（Tom Looman 的 UE5 C++ 指南、Epic 官方论坛都将其列为最佳实践）：

| 宏                       | 表达式是否总是执行               | 失败时行为                  | 适用场景                            |
| ----------------------- | ----------------------- | ---------------------- | ------------------------------- |
| `check` / `checkf`      | **否**（仅在 Debug/Test 编译） | 断点 + 终止                | 内部不变量，如 `index < arraySize`     |
| `verify` / `verifyf`    | **是**                   | Debug 时断点，Release 仅返回值 | 带副作用的校验，如 `verify(file.Open())` |
| `ensure` / `ensureMsgf` | **是**                   | 记录错误 + 堆栈，首次断点后放行      | 容错恢复路径，如网络包校验失败                 |

```cpp
// check：内部不变量。Release 构建中整个表达式消失，零开销。
void ProcessArray(int32 index) {
    check(index >= 0 && index < ArraySize);  // 如果这里错了，是程序员 bug
    Data[index] = ...;
}

// verify：表达式必须执行，但失败处理可以只在 Debug 时生效。
bool SaveGame() {
    verify(FileHandle.Open(Path));  // FileHandle.Open() 必须执行！
    // ... 写入数据
}

// ensure："这里出了问题但游戏可以继续"。
void ReceivePacket(const Packet& pkt) {
    if (ensureMsgf(pkt.Size <= MAX_PACKET_SIZE, "Oversized packet: %d", pkt.Size)) {
        Process(pkt);
    }
}
```

`ensure` 的一个精妙设计是**"首次失败断点"**（参考 UE 源码中的 `EnsureFailed` 实现）：第一次触发时断点，让开发者看到问题；后续相同 ensure 不再断点，避免死循环淹没日志。这种"一次提醒，持续放行"的语义非常适合"不应该发生但发生了也能继续"的场景。

chaos 的做法则体现了另一种思路：通过**函数指针 `assertDelegate`** 让上层注册自定义处理器（弹出窗口、生成 dump、上传遥测）。未注册时回退到 `printf`。这给基础类型层增加了极小的间接性，但获得了巨大的扩展性。chaos 的五级构建裁剪（Debug/Check/Release/Profile/Ship）在工业级代码库中有其价值，但当前阶段先聚焦核心的三级语义；"断言可委托"的思想应作为预留接口保留——当引擎需要接入崩溃报告系统（如 Sentry、Backtrace）时，只需要替换一个函数指针，不需要改动任何断言调用点。

Bevy 作为 Rust 引擎，没有 C++ 宏体系，但它的调试哲学同样分层：`debug_assert!` 对应 `check`（Debug-only），`assert!` 对应 `verify`（始终执行），而 `tracing` 生态提供了类似 `ensure` 的非致命错误记录能力。更重要的是，Bevy 的 `bevy_internal` 设置了 `#[forbid(unsafe_code)]`，这意味着基础层本身的代码被要求"不能出错"——这种对基础层正确性的极高要求，反过来降低了对外部断言机制的依赖。

> **默认推荐**：采用 UE 的 `check` / `verify` / `ensure` 三级语义作为默认路径。`check` 用于内部不变量，`verify` 用于带副作用的校验，`ensure` 用于容错路径。该分层策略在 Shipping 构建中通过编译期剥离将 `check` 完全消去，保证 ECS System 的批量组件迭代路径零断言开销。如果未来需要接入崩溃报告，预留一个 `SetAssertHandler` 的函数指针接口（学习 chaos 的 assertDelegate 扩展思想）。

## 问题 3：动态库符号如何在跨平台上一致控制？

Windows 的动态库需要显式标记 `__declspec(dllexport)` 才能导出符号，而 Linux 默认导出所有符号。结果是：同一个模块在 Windows 上可能只导出 50 个符号（你标记的那些），在 Linux 上却导出 5000 个符号（全部）。这不仅增加二进制体积，还可能导致符号冲突。

最 naive 的方案是每个类/函数前手写平台条件：

```cpp
// 痛苦的 naive 方案
#ifdef _WIN32
    #ifdef BUILDING_MYMODULE
        #define MYMODULE_API __declspec(dllexport)
    #else
        #define MYMODULE_API __declspec(dllimport)
    #endif
#else
    #define MYMODULE_API __attribute__((visibility("default")))
#endif
```

这段代码的问题不是"错"，而是"每个模块都要复制粘贴一次"，而且 `BUILDING_MYMODULE` 这个宏需要在 CMake 里为每个模块单独定义，维护成本极高。

改进方案是统一封装：

```cpp
#if defined(PLATFORM_WINDOWS)
    #ifdef ENGINE_BUILD_DLL
        #define ENGINE_API __declspec(dllexport)
    #else
        #define ENGINE_API __declspec(dllimport)
    #endif
#else
    #define ENGINE_API __attribute__((visibility("default")))
#endif
```

但这还不够好。Linux 上默认导出所有符号的问题没解决。更工业级的做法是在编译选项里加 `-fvisibility=hidden`（GCC/Clang），然后只对需要导出的符号标记 `__attribute__((visibility("default")))`。这样 Windows 和 Linux 的行为就一致了：默认隐藏，显式导出。

对于 C++ 模板和内联函数，还有额外的陷阱：`dllexport` 对模板实例化的处理在不同编译器上行为不同。**默认推荐**：模板和内联函数**不要标记 ENGINE_API**，把它们放在头文件中直接内联，避免跨 DLL 的模板实例化问题。这与 UE 的 `CORE_API` / `ENGINE_API` 使用边界一致——仅对需要导出链接的类、非模板函数和全局变量标记导出宏。

## 问题 4：基础类型层应该"多厚"？

这是最容易过度工程的地方。你的基础类型层已经有了类型别名、平台宏、断言体系、导出宏。现在你可能 tempted 往里面塞更多东西：

- SIMD 类型封装（`__m128` / `float32x4_t`）
- 对齐工具（`AlignUp`、`IsAligned`）
- 安全指针（检测 use-after-free）
- 编译器属性封装（`FORCEINLINE`、`LIKELY`/`UNLIKELY`）
- 原子类型别名

**问题是：基础类型层应该是一个"零依赖"的头文件集合。** 如果它开始依赖其他模块（比如日志系统、内存管理器），那么任何使用基础类型的代码都会间接依赖这些上层模块，破坏模块层次。

UE 的做法值得学习：基础类型层（`CoreTypes.h`、`CoreMinimal.h`）保持极简——只有纯编译期别名和宏。`TSharedPtr`、`TWeakPtr` 属于内存管理模块，不在基础类型层。对齐工具放在数学库或内存分配器里。

chaos 则提供了一个反例参考：它的基础层包含 `RawPtr`，通过 salt 校验检测 use-after-free。这个功能很诱人，但它引入了运行时机制（salt 分配、校验逻辑），使基础层不再"纯编译期"。**默认推荐**：安全指针、智能指针、内存调试工具放到 [[内存分配器]] 中讨论，基础类型层保持零运行时开销。这是 UE `CoreTypes.h` 的工业级实践——基础层只有纯编译期别名和宏。

那对齐工具和编译器属性呢？建议采取"延迟引入"策略：

```cpp
// 对齐工具——等数学库需要 SIMD 对齐时再引入
constexpr usize AlignUp(usize value, usize alignment) {
    usize mask = alignment - 1;
    return (value + mask) & ~mask;
}
constexpr bool IsPowerOfTwo(usize v) { return v && (v & (v - 1)) == 0; }

// 编译器属性——在需要热路径优化时引入
#if defined(_MSC_VER)
    #define FORCEINLINE __forceinline
    #define LIKELY(x)   (x)
    #define UNLIKELY(x) (x)
#elif defined(__clang__) || defined(__GNUC__)
    #define FORCEINLINE inline __attribute__((always_inline))
    #define LIKELY(x)   __builtin_expect(!!(x), 1)
    #define UNLIKELY(x) __builtin_expect(!!(x), 0)
#endif
```

这些不是"现在就必须有"，而是"当性能分析显示这里需要时，随时可以拿出来用"。

> **默认推荐**：基础类型层 = 固定宽度别名 + 平台宏（预置为零）+ 断言三级体系 + 导出符号宏。仅此而已。对齐工具、属性宏、原子别名可以放在同一目录的扩展头文件中，但不属于"最小基础层"。安全指针、智能指针、内存调试工具放到 [[内存分配器]]。
>
> **吸收自 UE 的设计**：UE 的 `CoreTypes.h` / `CoreMinimal.h` 严格保持基础层极简，不引入任何运行时机制。我们直接吸收这一原则：基础层零依赖、零运行时开销、零动态分配。这保证了 ECS 架构中任何 System 引用基础类型头文件时，不会隐式拉入日志、内存管理或反射等上层模块的依赖。

## 问题 5：AI 友好的引擎，基础层该怎么设计？

前面四个问题都是传统引擎基础层会面对的。但作为为 AI 协作而设计的引擎，基础层还需要回答一个额外问题：**AI 能不能在不读 C++ 头文件的情况下，理解引擎的构建配置和类型系统？**

基础层虽然全是编译期宏，但它是整个引擎的"元信息源头"。如果 AI 无法获知"当前引擎启用了哪些模块"、"目标平台是什么"、"类型宽度是多少"，那么它在上层生成的代码很可能与实际情况不匹配。

### 自描述：让构建配置可查询

Bevy 的做法提供了重要启示：它的 Feature 体系（`2d`、`3d`、`trace`、`png` 等）不仅控制编译，还构成了引擎能力的**声明式清单**。Bevy 的 `bevy::platform::cfg` 模块进一步封装了平台特性检测，让代码可以通过统一的宏判断"当前是否支持 `std`"、"当前是否支持 `alloc`"。

映射到 C++ 引擎，这意味着：

1. **所有 CMake option 应该有规范化的命名和默认值**，不要散落在各个 `CMakeLists.txt` 里。集中到一个 `EngineConfig.cmake` 或 `config.h.in` 中，生成一个 `EngineConfig.h` 头文件。
2. **平台宏和功能宏应该能被外部工具读取**。CMake 可以生成 `compile_commands.json`，AI 可以通过解析这个文件获知当前的宏定义。更好的做法是维护一个 JSON/YAML 格式的"引擎配置清单"，AI 直接读取。

```cpp
// EngineConfig.h（由 CMake 自动生成）
#define ENGINE_VERSION_MAJOR 0
#define ENGINE_VERSION_MINOR 1
#define ENGINE_VERSION_PATCH 0

#define ENABLE_RENDER      1
#define ENABLE_PHYSICS     0
#define ENABLE_NETWORK     0
#define ENABLE_AI_BRIDGE   1

#define PLATFORM_WINDOWS   1
#define ARCH_64BIT         1
#define HAS_SSE4_2         1
```

### 确定性：固定宽度是回放的基石

基础类型层的最大贡献不是"让代码跨平台编译"，而是**让相同输入在不同平台上产生位精确一致的输出**。`u32` 在任何平台都是 32 位，这意味着：

- 网络协议字段的位模式一致 → 联机同步不出错
- 存档文件的二进制布局一致 → 跨平台读档安全
- 随机数种子产生的序列一致 → 确定性回放可行
- ECS 组件的内存布局一致 → AI 可以通过 Schema 准确描述数据结构

### 状态平铺与 Agent 安全

基础类型层本身没有运行时状态（全是编译期宏和别名），所以它天然满足"状态平铺"和"Agent 安全"。AI 修改基础类型别名（比如把 `u32` 改成 `u64`）不会破坏引擎运行时的全局状态——它只影响编译结果。这给了 AI 极大的操作自由度：调整类型宽度、开关功能宏、切换平台配置，都是**编译期安全的操作**。

## 参考：三引擎在基础层问题上的对照

不是每个子问题都需要三引擎对照。只有在存在方案分歧时，对照才有价值。

### 对照点 A：平台宏的"预置为零"策略

| 引擎 | 做法 | 默认推荐 |
|------|------|-------------|
| chaos | 三层宏架构（platform_macro / build_config / base_macro），强制 SSE/AVX/NEON 至少支持一个 | 三层拆分在大型代码库中有其价值，但最小基础层先聚焦核心平台宏 |
| UE | 所有 `PLATFORM_XXX` 先设为 `0`，再覆盖为 `1`；强制 `#if` | **推荐采用**。把未定义变成编译错误 |
| Bevy | `#[cfg]` + `bevy::platform::cfg` 封装，Feature 分层 | 命名一致性和分层思想值得借鉴，映射到 CMake option |

### 对照点 B：断言语义的分层

| 引擎    | 做法                                                                                       | 默认推荐                                          |
| ----- | ---------------------------------------------------------------------------------------- | ----------------------------------------------- |
| chaos | `assertDelegate` 函数指针 + 五级构建裁剪（Debug/Check/Release/Profile/Ship），Ship 版断言替换为 `((void)0)` | assertDelegate 的扩展性值得预留，五级裁剪对小型项目过度             |
| UE    | `check` / `verify` / `ensure` 三级语义，`.uedbg` section 冷路径分离                                | **推荐采用**三级语义。`.uedbg` section 在代码量超过 10 万行时值得引入 |
| Bevy  | `debug_assert!` / `assert!` / `tracing` 非致命记录                                            | Rust 的内存安全降低了断言需求，但"分层调试信息"的思想一致                |

### 对照点 C：基础层厚度

| 引擎 | 做法 | 默认推荐 |
|------|------|-------------|
| chaos | 基础层包含 `RawPtr`（salt 校验 use-after-free） | 功能诱人但增加运行时机制，不建议放在基础层 |
| UE | 基础层纯编译期：`int32`、`uint64`、`TCHAR`、`ENGINE_API` 等，零运行时 | **推荐采用**。安全指针放到内存管理模块 |
| Bevy | `bevy_internal` 设 `#[forbid(unsafe_code)]`，基础 crate 优先 `no_std` | `no_std` 优先 → 映射到 C++ 的"最小依赖优先"设计哲学 |

> **默认推荐路径总结**：
>
> 基础类型层的工业级默认路径是：**固定宽度别名（`u8`/`i32`/`f32` 等）+ 平台宏预置为零 + `#if` 强制检查 + `ENGINE_API` 统一导出 + `CHECK`/`VERIFY`/`ENSURE` 三级断言**。该路径在 ECS 兼容前提下吸收了 UE 的以下工业级设计：
> - **UE `CoreTypes.h` 的极简原则**：基础层零运行时机制、零动态分配、零上层依赖，任何 ECS System 头文件引用都不会隐式拉入日志或反射模块。
> - **UE 平台宏的预置为零模式**：将 `#ifdef` 的静默失败转化为编译期硬错误，消除跨平台宏名拼写的隐蔽 bug。
> - **UE 的三级断言语义**：`check` 在 Shipping 构建中完全消去，保证 ECS 批量迭代路径零断言开销；`verify` 保证副作用表达式始终执行；`ensure` 提供非致命容错路径。
> - **UE `HAL/Platform.h` 的分层思想**：平台检测、编译器属性、功能开关分文件管理，避免单个头文件膨胀。
>
> **条件切换路径**：如果目标平台是嵌入式或内核态（如游戏主机安全模式），可将 `CHECK` 的失败行为从 `abort()` 切换为自定义陷阱指令；如果引擎确定静态链接（无插件/DLL），`ENGINE_API` 可退化为空宏，但保留宏名以便未来切换为动态链接。
>
> **扩展路径（更极端工业场景）**：
> - **崩溃报告集成**：接入 `SetAssertHandler` 函数指针，将断言失败路由到 Sentry/Backtrace（参考 chaos 的 `assertDelegate`）。
> - **编译器属性扩展**：引入 `LIKELY`/`UNLIKELY`、`FORCEINLINE`、`RESTRICT` 等分支预测与内联控制宏，在热路径 System 迭代中发挥作用。
> - **原子类型别名**：为无锁 ECS 命令缓冲区或并行 System 调度定义 `AtomicU32` / `AtomicPtr` 等别名，统一 `std::atomic` 的跨平台表达。
> - **调试段分离**：参考 UE `.uedbg` section 将断言失败处理代码隔离到单独段，减少指令缓存污染。
> - **SIMD 能力探测**：扩展平台宏以检测 SSE4.2 / AVX2 / AVX-512 / NEON / WASM-SIMD，为数学库的自动向量化提供决策输入。

## 最小实现（约 120 行）

这是你的引擎在阶段 3 应该拥有的最小基础类型层。它完全自包含，不依赖任何上层模块：

```cpp
// FoundationTypes.h —— 阶段 3 最小基础类型层
#pragma once
#include <cstdint>
#include <cstddef>

// ========== 1. 固定宽度整数 ==========
using u8  = uint8_t;   using i8  = int8_t;
using u16 = uint16_t;  using i16 = int16_t;
using u32 = uint32_t;  using i32 = int32_t;
using u64 = uint64_t;  using i64 = int64_t;
using f32 = float;     using f64 = double;
using usize = std::size_t;
using isize = std::ptrdiff_t;

// ========== 2. 平台检测（预置为零 + #if） ==========
#define PLATFORM_WINDOWS 0
#define PLATFORM_LINUX   0
#define PLATFORM_MACOS   0
#define PLATFORM_ANDROID 0
#define PLATFORM_IOS     0
#define PLATFORM_UNKNOWN 0

#if defined(_WIN32)
    #undef PLATFORM_WINDOWS
    #define PLATFORM_WINDOWS 1
#elif defined(__linux__) && !defined(__ANDROID__)
    #undef PLATFORM_LINUX
    #define PLATFORM_LINUX 1
#elif defined(__APPLE__)
    #include <TargetConditionals.h>
    #if TARGET_OS_IPHONE
        #undef PLATFORM_IOS
        #define PLATFORM_IOS 1
    #else
        #undef PLATFORM_MACOS
        #define PLATFORM_MACOS 1
    #endif
#elif defined(__ANDROID__)
    #undef PLATFORM_ANDROID
    #define PLATFORM_ANDROID 1
#else
    #undef PLATFORM_UNKNOWN
    #define PLATFORM_UNKNOWN 1
#endif

#if defined(_M_X64) || defined(__x86_64__) || defined(__aarch64__)
    #define ARCH_64BIT 1
    #define ARCH_32BIT 0
#else
    #define ARCH_64BIT 0
    #define ARCH_32BIT 1
#endif

// ========== 3. 导出符号宏 ==========
#if PLATFORM_WINDOWS
    #ifdef ENGINE_BUILD_DLL
        #define ENGINE_API __declspec(dllexport)
    #else
        #define ENGINE_API __declspec(dllimport)
    #endif
#else
    #define ENGINE_API __attribute__((visibility("default")))
#endif

// ========== 4. 断言三级体系 ==========
#ifdef DEBUG
    #include <cstdlib>
    #include <cstdio>

    #if defined(_MSC_VER)
        #define DEBUG_BREAK() __debugbreak()
    #elif defined(__clang__) || defined(__GNUC__)
        #define DEBUG_BREAK() __builtin_trap()
    #else
        #define DEBUG_BREAK() std::abort()
    #endif

    inline bool DefaultEnsureHandler(const char* expr, const char* file, int line) {
        std::fprintf(stderr, "[ENSURE FAILED] %s at %s:%d\n", expr, file, line);
        return true;  // 返回 true 表示"继续执行"
    }

    #define CHECK_IMPL(cond, expr_str, file, line) \
        do { if (!(cond)) { \
            std::fprintf(stderr, "[CHECK FAILED] %s at %s:%d\n", expr_str, file, line); \
            DEBUG_BREAK(); \
            std::abort(); \
        } } while(0)

    // check：内部不变量，Release 中完全消失
    #define CHECK(cond) CHECK_IMPL(cond, #cond, __FILE__, __LINE__)

    // verify：表达式始终执行，失败处理仅在 Debug 生效
    #define VERIFY(cond) CHECK_IMPL(cond, #cond, __FILE__, __LINE__)

    // ensure：记录错误 + 继续执行（简化版，不含"首次断点"逻辑）
    // 使用 C++20 __VA_OPT__ 保证跨平台（MSVC/GCC/Clang 均支持）
    #define ENSURE_MSG(cond, fmt, ...) \
        do { if (!(cond)) { \
            std::fprintf(stderr, "[ENSURE] " fmt " at %s:%d\n" __VA_OPT__(,) __VA_ARGS__, __FILE__, __LINE__); \
        } } while(0)
    #define ENSURE(cond) ENSURE_MSG(cond, "%s", #cond)
#else
    #define CHECK(cond)      ((void)0)
    #define VERIFY(cond)     ((void)(cond))  // 表达式必须执行！
    #define ENSURE(cond)     ((void)0)
    #define ENSURE_MSG(...)  ((void)0)
#endif

#define STATIC_ASSERT(cond, msg) static_assert(cond, msg)

// ========== 5. 调试填充（检测 use-after-free / 越界） ==========
#ifdef DEBUG
    constexpr u8 CANARY_ALLOC = 0xCD;
    constexpr u8 CANARY_FREE  = 0xFE;
    inline void DebugFill(void* ptr, usize size, u8 value) {
        std::memset(ptr, value, size);
    }
#else
    inline void DebugFill(void*, usize, u8) {}
#endif
```

### 这段代码解决了什么、还缺什么

| 能力 | 状态 | 说明 |
|------|------|------|
| 跨平台类型统一 | ✅ | `u32` 在任何平台都是 32 位无符号整数 |
| 平台宏安全 | ✅ | `#if PLATFORM_WINDOWS` 拼错即编译报错 |
| 动态库符号可控 | ✅ | `ENGINE_API` 统一 Windows/Linux 导出语义 |
| 断言分层 | ✅ | `CHECK`/`VERIFY`/`ENSURE` 覆盖三种场景 |
| 调试验证 | ✅ | `DebugFill` 为内存调试提供基础设施 |
| SIMD 类型封装 | ⬜ | 交给 [[数学基础]] |
| 运行时硬件查询 | ⬜ | 交给 [[网络抽象与硬件检测]] |
| 原子类型别名 | ⬜ | 原型阶段 `std::atomic` 足够，工业级时再封装 |
| 崩溃报告钩子 | ⬜ | 预留 `SetAssertHandler` 接口，实际实现待 [[可视化日志系统]] 深化 |

## 性能与 Trade-off：诚实分析

| 设计点 | 优势 | 代价 | 反例警告 |
|--------|------|------|----------|
| 固定宽度别名 | 跨平台一致性、网络/存档安全 | 代码中多写一层类型名（但比 `uint32_t` 短） | 无 |
| 预置为零 + `#if` | 消除 `#ifdef` 的静默失败风险 | 每个平台宏多写两行 `#undef`/`#define` | 无 |
| `ENGINE_API` 宏抽象 | 一份代码通吃 Windows/Linux | 增加了构建系统复杂度（需定义 `ENGINE_BUILD_DLL`） | 静态链接时宏定义不匹配会导致符号找不到 |
| `CHECK` 在 Release 中剥离 | 零运行时开销 | 线上崩溃拿不到断言信息 | **不要把"可能发生的错误"放进 `CHECK`**，应用 `if + error handling` |
| `VERIFY` 保留表达式执行 | Shipping 构建中副作用不丢失 | 表达式本身仍有运行时开销 | 不要把昂贵的操作放进 `VERIFY` |

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ | 基础类型全是编译期别名和宏，不引入任何运行时状态或动态分配 |
