# 编辑器基础设施

> **前置依赖**：[[编辑器框架]]、[[反射系统]]、[[资源管理]]
> **本模块增量**：编辑器从"内部工具"升级为"可扩展的平台"：跨语言绑定、资产导入管线、高级 PIE 模式、多窗口布局管理。

## Why：为什么编辑器需要基础设施层？

在 [[编辑器框架]] 中，你已经能让 AI 和人类实时操作 ECS 世界：修改组件、拖拽 Gizmo、切换 PIE 模式。但这只是"能用"。

当引擎要支撑大型团队、跨平台开发和 AI Agent 协作时，编辑器必须从一个"内部工具"升级为**可扩展的平台**。你需要：

1. **跨语言前端**：让 UI 团队用 C#/Electron/Qt 快速迭代，而不必重新编译 C++ 核心。
2. **资产导入管线**：美术拖拽 FBX/PNG 进编辑器，自动转换为引擎内部格式。
3. **高级 PIE 模式**：暂停、单帧步进、多客户端联调。
4. **窗口布局管理**：浮动面板、选项卡组、布局持久化。

> **核心结论**：编辑器基础设施是工具链的"工业化升级"。它不影响核心运行时，但决定了开发团队的迭代效率和引擎的生态扩展能力。

## What：最简化版本的编辑器基础设施长什么样？

下面的代码展示编辑器基础设施的三大支柱：**数据桥**、**资产导入**、**窗口布局**。

```cpp
#include <string>
#include <vector>
#include <memory>

// ============================================================
// 1. 数据桥：C++ 与前端 UI 的共享数据镜像
// ============================================================
struct DataBridgeValue {
    std::string componentName;
    std::string fieldName;
    std::string jsonValue;
    bool dirty = false;
};

class DataBridge {
    std::vector<DataBridgeValue> values;
public:
    void markDirty(const std::string& comp, const std::string& field, const std::string& val) {
        values.push_back({comp, field, val, true});
    }
    std::vector<DataBridgeValue> flush() {
        auto out = std::move(values);
        values.clear();
        return out;
    }
};

// ============================================================
// 2. 资产导入管线：外部资产 → 引擎内部格式
// ============================================================
struct ImportTask {
    std::string sourcePath;   // e.g., "models/hero.fbx"
    std::string outputPath;   // e.g., "assets/hero.mesh"
    std::string importerType; // "FBX", "PNG", "WAV"
};

class AssetImporter {
public:
    virtual bool import(const ImportTask& task) = 0;
};

class AssetImportPipeline {
    std::unordered_map<std::string, std::unique_ptr<AssetImporter>> importers;
public:
    void registerImporter(const std::string& type, std::unique_ptr<AssetImporter> imp) {
        importers[type] = std::move(imp);
    }
    bool process(const ImportTask& task) {
        auto it = importers.find(task.importerType);
        if (it == importers.end()) return false;
        return it->second->import(task);
    }
};

// ============================================================
// 3. 窗口布局树
// ============================================================
enum class SplitDirection { Horizontal, Vertical };

struct ILayoutNode { virtual ~ILayoutNode() = default; };

struct LayoutSplit : ILayoutNode {
    SplitDirection dir;
    float ratio = 0.5f;
    std::shared_ptr<ILayoutNode> childA;
    std::shared_ptr<ILayoutNode> childB;
};

struct LayoutViewport : ILayoutNode {
    std::string cameraName;
    uint32_t width = 1280, height = 720;
};
```

### 这个最小实现已经解决了什么

1. **数据桥**：C++ 核心和前端 UI 之间的共享数据层，AI 也可以用 JSON 操作同一层数据。
2. **导入器插件体系**：为每种外部资产注册专用导入器，扩展新格式无需改核心。
3. **布局树**：支持 1/2/3/4 分屏、自定义比例、序列化为 JSON。

### 还缺什么

- **增量导入**：每次重新导入全部资产，没有基于 hash/时间戳的增量检测。
- **分布式导入**：大型场景烘焙无法利用多机并行。
- **.meta 文件**：没有持久化导入参数（如法线生成、LOD 级别）。
- **SWIG 绑定生成**：手写绑定代码繁琐易错。

## How：编辑器基础设施是如何一步一步复杂起来的？

### 阶段 1：能用 —— 数据桥 + 基础导入

#### 触发原因

编辑器前端需要高频读写组件字段，直接反射 get/set 太慢（每次都要跨语言边界）。

#### 代码层面的变化

**A. 数据桥与增量同步协议**

引入**数据桥（Data Bridge）**：C++ 侧维护一块与 Inspector 面板对应的数据镜像，C# 侧绑定到这块镜像。用户拖动滑块时只修改镜像，离开焦点或按下回车时才将镜像批量写回 ECS。

对于 AI Agent，这块数据桥恰好可以被替换为 **JSON/MCP 协议**：Agent 不需要理解 Sketum 或 WPF，只需要发送 `{"entity": 42, "component": "TransformComponent", "field": "position", "value": [10, 0, 0]}`。

**B. 资源导入管线（Asset Import Pipeline）**

编辑器不仅是场景操作工具，还是资源生产入口。

```
外部资产文件
    ↓ 拖拽到编辑器 / 文件系统监听
AssetImporter (模型 → Mesh + Material)
AssetImporter (纹理 → Texture2D + Mipmap)
    ↓
生成 .meta 文件（记录 GUID、导入参数、依赖关系）
    ↓
写入引擎资源库（Asset Database）
```

导入参数（如模型缩放、法线生成、纹理压缩格式）也通过 Inspector 编辑，并保存到 `.meta` 中。

**C. 元数据（.meta）与增量导入**

```cpp
struct AssetMeta {
    std::string guid;
    uint64_t sourceHash;
    uint64_t lastModified;
    std::string importerSettings; // JSON
};
```

导入器在运行前检查 `sourceHash`：
- 如果源文件未变更，跳过导入，直接复用缓存。
- 如果 `.meta` 中的导入参数变更，即使源文件未变，也触发重新导入。

### 阶段 2：好用 —— 跨语言绑定与高级 PIE

#### 触发原因

引擎要支撑大型团队和跨平台开发，需要更灵活的前端技术和更强大的调试模式。

#### 代码层面的变化

**A. 跨语言绑定与插件化编辑器前端**

工业级引擎通常将编辑器核心逻辑放在 C++（高性能、直接访问 ECS），而将 UI 放在 C# / Qt / Electron / SwiftUI（开发效率高、控件丰富）。

通过 SWIG、C++/CLI、或自定义绑定生成器，将 C++ 的反射系统完整暴露给前端语言：

```csharp
// C# 前端示例
TypeMeta meta = TypeMeta.newFromName("TransformComponent");
ReflectionInstance inst = editorSystem.getReflectionInstance(entityId);
FieldAccessor field = meta.getFieldByName("position");
field.set(inst.instance, new Vector3(10, 0, 0));
```

编辑器前端成为可选组件——你可以用默认的 C# UI，也可以替换为自己的 ImGui 前端或 Web 前端。

**B. PIE 的进阶模式**

- **Pause / Step Frame**：冻结游戏 Tick，逐帧推进，用于调试瞬态物理或动画状态。
- **Multiplayer PIE**：同时启动多个客户端实例（Listen Server + Clients），在编辑器内测试网络同步。
- **Play From Here**：从鼠标点击的位置和朝向生成玩家，而不是从固定的 PlayerStart 开始。

**C. 窗口布局与 Docking 系统**

用递归的 `LayoutNode` 树描述视口布局：

```cpp
struct ILayoutNode { virtual ~ILayoutNode() = default; };
struct LayoutSplit : ILayoutNode {
    SplitDirection dir; // Horizontal / Vertical
    float ratio;        // 0.0 ~ 1.0
    std::shared_ptr<ILayoutNode> childA;
    std::shared_ptr<ILayoutNode> childB;
};
struct LayoutViewport : ILayoutNode {
    ViewportConfig config;
};
```

支持 1/2/3/4 分屏、自定义比例、拖拽调整分割条位置。布局树可序列化为 JSON，保存用户的工作区偏好。

### 阶段 3：工业级 —— 分布式导入与节点图

#### 触发原因

大型项目有数十万资产，单线程导入需要数小时。编辑器还需要可视化脚本和复杂材质编辑。

#### 代码层面的变化

**A. 分布式导入与大型场景烘焙**

- 将导入任务分发到多台工作机或本地多进程。
- 使用共享缓存（如 Redis / 网络文件系统）存储中间结果。
- 导入管线变成 DAG：源文件 → 解析 → 中间格式 → 平台特化（如 PS5 纹理压缩）→ 最终资产。

**B. 节点图与可视化脚本**

- 材质编辑器、蓝图运行时、动画状态机都可以用统一的节点图框架实现。
- 节点图的数据模型也是 ECS 友好的：每个节点是一个 Entity，连接线是 `Wire` 组件，图执行是一个 System。

**C. 独立工具程序架构**

- `Converter`：命令行资产转换器，供 CI/CD 调用。
- `Packager`：资源打包工具，生成 Pak 文件。
- `Standalone Editor Tool`：不依赖完整引擎运行时的轻量工具。

## AI 友好设计检查清单

| 检查项 | 本模块的满足方式 |
|--------|---------------------|
| **状态平铺** | ✅ 数据桥的值是平铺的 JSON 键值对，布局树是平铺的节点数组 |
| **自描述** | ✅ 导入器类型、布局配置、PIE 模式状态都通过反射暴露 |
| **确定性** | ✅ 资产导入顺序由拓扑排序决定，同样的输入资产产生同样的输出 |
| **工具边界** | ✅ AI 通过 MCP 调用 `import_asset(path)`、`set_layout(json)` |
| **Agent 安全** | ✅ 导入器在沙箱进程/线程中运行，恶意资产不会崩溃主引擎 |

## 设计权衡表

| 决策 | 选项 A | 选项 B | 推荐 |
|------|--------|--------|------|
| 前端技术栈 | 原生 C++ UI（如 Qt/ImGui） | 脚本语言 UI（如 C#/Electron） | **核心逻辑 C++，UI 可插拔** |
| 数据桥同步 | 实时反射读写 | 镜像批量同步 | **镜像批量同步** |
| 导入粒度 | 整文件重新导入 | 基于 hash 的增量导入 | **增量导入** |
| PIE 隔离 | 深拷贝编辑世界 | 重新加载关卡资源 | **推荐重新加载** |
| 节点图实现 | 独立框架 | 基于 ECS 的节点 Entity | **基于 ECS**，与核心架构一致 |

## 如果我要 vibe coding，该偷哪几招？

1. **数据桥协议是 AI 友好性的分水岭**
   - 在 C++ 和 UI 之间建立一层结构化数据桥，人类用滑块操作它，AI 用 JSON 操作它。

2. **.meta 文件是增量导入的灵魂**
   - 记录 GUID、导入参数、源文件 hash。没有它，每次保存项目都要重新导入全部资产。

3. **PIE = 世界切换，不是窗口切换**
   - 切换编辑态和运行态时，保留窗口和渲染上下文，只换 World 实例。这是实现"秒级 Play/Stop"的关键。

4. **导入器插件化**
   - 不要在一个函数里处理所有资产类型。为 FBX、PNG、WAV 各写一个 `AssetImporter`，注册到同一个 Pipeline 中。

5. **布局树持久化**
   - 用户的窗口布局偏好应该自动保存为 JSON，下次启动时恢复。这是专业感的最小单位。
