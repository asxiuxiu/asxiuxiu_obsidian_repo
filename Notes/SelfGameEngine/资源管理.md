# 资源管理

## Why：为什么游戏引擎需要资源管理系统？

### 场景一：同步加载导致的卡顿

你在 RPG 中打开一扇门，进入一个地下城。如果没有资源管理系统，所有纹理、网格、音效都在主线程同步从磁盘读取。屏幕冻结 3 秒，玩家以为游戏崩溃了。

这不是"加载慢"的问题，而是**加载阻塞了渲染线程**的问题。现代游戏引擎要求资源加载必须在后台进行，主线程只收到一个"加载完成"的通知。

### 场景二：内存爆炸与重复加载

每个怪物都 `new` 一个 `Mesh`，每个武器都独立加载一份纹理。关卡中有 100 个哥布林，内存中就有 100 份相同的模型数据。更糟的是，当哥布林被击杀后，没人记得 `delete`，显存泄漏直到游戏崩溃。

资源管理系统的答案是：**用句柄（Handle）代替裸指针，用引用计数代替手动 new/delete，用全局对象表保证同一份数据只加载一次**。

### 场景三：迭代地狱

美术修改了一个材质的贴图，你需要关闭引擎、重新导出资源包、重新启动、重新加载关卡，才能看到效果。每次微调都耗费数分钟。

一个支持**热重载（Hot Reload）**的资源系统可以让美术在 Photoshop 中保存文件，引擎在 1 秒内自动刷新，无需重启。

### 结论

资源管理不是"锦上添花"的优化，而是**现代游戏引擎的必需品**。它的核心使命可以概括为：

> **在正确的时间，以正确的方式，把正确的数据加载到正确的位置，并在不再使用时安全释放。**

## What：最简化版本的资源管理长什么样？

下面的代码是一个**能编译、能运行**的最小资源管理骨架。它展示了引擎资源管理的三大基石：
1. **Handle + ObjectTable**：用轻量句柄代替裸指针
2. **异步加载状态机**：后台线程加载，主线程零阻塞
3. **引用计数生命周期**：自动释放无人引用的资源

```cpp
#include <cstdint>
#include <string>
#include <vector>
#include <unordered_map>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <functional>
#include <atomic>
#include <iostream>

// ============================================================
// 1. 轻量句柄 + 对象表（Handle-Table 模式）
// ============================================================
using ResourceHandle = uint32_t;
constexpr ResourceHandle INVALID_HANDLE = 0;

enum class ResourceState : uint8_t {
    Empty,
    Loading,
    Ready,
    Failed
};

struct ResourceEntry {
    std::string path;
    ResourceState state = ResourceState::Empty;
    uint32_t ref_count = 0;
    std::vector<uint8_t> data; // 模拟资源数据（真实引擎中可能是 RHITexture* 等）
};

class ResourceTable {
    std::unordered_map<ResourceHandle, ResourceEntry> m_table;
    std::atomic<ResourceHandle> m_next_handle{1};
public:
    ResourceHandle allocate(const std::string& path) {
        ResourceHandle h = m_next_handle.fetch_add(1);
        m_table[h] = { path, ResourceState::Loading, 1, {} };
        return h;
    }

    void addRef(ResourceHandle h) {
        if (auto it = m_table.find(h); it != m_table.end()) {
            it->second.ref_count++;
        }
    }

    void release(ResourceHandle h) {
        auto it = m_table.find(h);
        if (it == m_table.end()) return;
        if (--it->second.ref_count == 0) {
            std::cout << "[GC] Unloading resource: " << it->second.path << "\n";
            m_table.erase(it);
        }
    }

    ResourceEntry* get(ResourceHandle h) {
        auto it = m_table.find(h);
        return it != m_table.end() ? &it->second : nullptr;
    }
};

// ============================================================
// 2. 异步加载器（后台线程 + 任务队列）
// ============================================================
struct LoadTask {
    ResourceHandle handle;
    std::string path;
    std::function<void(ResourceHandle)> on_complete;
};

class AsyncAssetLoader {
    ResourceTable& m_table;
    std::thread m_worker;
    std::queue<LoadTask> m_queue;
    std::mutex m_mutex;
    std::condition_variable m_cv;
    bool m_running = true;

    void worker_loop() {
        while (m_running) {
            LoadTask task;
            {
                std::unique_lock<std::mutex> lock(m_mutex);
                m_cv.wait(lock, [this] { return !m_queue.empty() || !m_running; });
                if (!m_running) break;
                task = m_queue.front();
                m_queue.pop();
            }

            // 模拟磁盘 IO（ sleep 50ms ）
            std::this_thread::sleep_for(std::chrono::milliseconds(50));

            if (auto* entry = m_table.get(task.handle)) {
                // 模拟加载成功
                entry->data.resize(1024, 0xAB);
                entry->state = ResourceState::Ready;
                std::cout << "[Loader] Finished: " << task.path << " (handle=" << task.handle << ")\n";
                if (task.on_complete) {
                    task.on_complete(task.handle);
                }
            }
        }
    }

public:
    explicit AsyncAssetLoader(ResourceTable& table) : m_table(table) {
        m_worker = std::thread(&AsyncAssetLoader::worker_loop, this);
    }

    ~AsyncAssetLoader() {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_running = false;
        }
        m_cv.notify_all();
        if (m_worker.joinable()) m_worker.join();
    }

    void enqueue(LoadTask task) {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_queue.push(std::move(task));
        }
        m_cv.notify_one();
    }
};

// ============================================================
// 3. 资源管理器：统一入口
// ============================================================
class ResourceManager {
    ResourceTable m_table;
    AsyncAssetLoader m_loader;
public:
    ResourceManager() : m_loader(m_table) {}

    ResourceHandle loadAsync(const std::string& path,
                             std::function<void(ResourceHandle)> callback = nullptr) {
        // 已加载？直接增加引用计数并返回
        for (auto& [h, entry] : m_table) {
            if (entry.path == path && entry.state == ResourceState::Ready) {
                m_table.addRef(h);
                if (callback) callback(h);
                return h;
            }
        }

        ResourceHandle h = m_table.allocate(path);
        m_loader.enqueue({ h, path, std::move(callback) });
        return h;
    }

    ResourceHandle loadSync(const std::string& path) {
        // 简化：先异步发起，再忙等（真实引擎中会用信号量/Fence）
        ResourceHandle h = loadAsync(path);
        while (getState(h) == ResourceState::Loading) {
            std::this_thread::sleep_for(std::chrono::milliseconds(1));
        }
        return h;
    }

    void unload(ResourceHandle h) {
        m_table.release(h);
    }

    ResourceState getState(ResourceHandle h) {
        auto* entry = m_table.get(h);
        return entry ? entry->state : ResourceState::Empty;
    }

    ResourceEntry* get(ResourceHandle h) {
        return m_table.get(h);
    }
};

// ============================================================
// 使用示例
// ============================================================
int main() {
    ResourceManager rm;

    // 异步加载两个资源
    auto h1 = rm.loadAsync("textures/hero.png", [](ResourceHandle h) {
        std::cout << "[Callback] Hero texture ready! handle=" << h << "\n";
    });
    auto h2 = rm.loadAsync("meshes/sword.bin");

    // 模拟主线程运行几帧，等待加载完成
    for (int i = 0; i < 10; ++i) {
        std::this_thread::sleep_for(std::chrono::milliseconds(20));
        if (rm.getState(h1) == ResourceState::Ready && rm.getState(h2) == ResourceState::Ready) {
            std::cout << "Both resources ready at frame " << i << "\n";
            break;
        }
    }

    // 复制一份引用（例如给另一个实体使用同一把剑）
    rm.get(h2)->ref_count++; // 实际应走 addRef 接口

    // 释放
    rm.unload(h1);
    rm.unload(h2);
    rm.unload(h2); // 第二次 release 后引用计数归零，触发卸载

    return 0;
}
```

### 这个最小实现已经解决了什么？

| 能力 | 说明 |
|------|------|
| **零裸指针** | 上层只使用 `ResourceHandle`（`uint32_t`），不直接持有 `Texture*` 或 `Mesh*` |
| **异步加载** | 磁盘 IO 发生在后台线程，主线程可以正常渲染和更新游戏逻辑 |
| **引用计数** | 同一份资源被多个对象引用时自动共享，引用归零时自动卸载 |
| **状态可追踪** | `Empty / Loading / Ready / Failed` 四种状态，上层可以精确知道资源当前在哪一步 |

### 这个最小实现还缺什么？

- **没有依赖解析**：一个材质可能依赖 3 张纹理，当前代码不会自动加载依赖链
- **没有取消机制**：如果一个 NPC 在资源加载完之前就被销毁了，加载任务还在继续浪费 IO
- **没有热重载**：文件修改后引擎不会自动刷新
- **没有显存/GPU 上传管理**：`data` 还在 CPU 内存中，没有上传到 GPU
- **没有流送预算**：大纹理一次性全加载，可能导致显存爆掉

这些正是我们在 **How** 阶段要逐步补上的。

## How：真实引擎的资源管理是如何一步一步复杂起来的？

### 阶段 1：最小实现 → 能用（解决异步加载与生命周期）

#### 触发原因

当你的引擎从 demo 走向可玩原型时，会遇到三个迫切问题：
1. **加载有依赖的资源**（如一个材质需要纹理 + 着色器）
2. **取消无用的加载任务**（角色进入视野触发加载，还没加载完就死了）
3. **同步/异步加载的统一接口**（关卡初始化时需要阻塞加载核心资源，运行时则必须异步）

#### 代码层面的变化

**A. 依赖树与批量加载**

真实引擎中，资源不是孤立的。一个 `Material` 可能引用 `Texture`、`Shader`、`ParameterTable`。加载器必须能递归解析依赖，并保证**叶子资源先加载、根资源后组装**。

```cpp
struct AssetLoadRequest {
    std::string path;
    bool load_dependencies = true;
    std::function<void(ResourceHandle)> on_complete;
};

class DependencyGraph {
    std::unordered_map<std::string, std::vector<std::string>> m_deps;
public:
    void registerDeps(const std::string& path, const std::vector<std::string>& deps) {
        m_deps[path] = deps;
    }

    std::vector<std::string> topologicalOrder(const std::string& root) {
        // 返回从叶子到根的加载顺序
        std::vector<std::string> result;
        // ... 拓扑排序实现 ...
        return result;
    }
};
```

加载器的工作流程变为：
1. 解析根资源的依赖图
2. 按拓扑顺序将叶子资源插入加载队列的**头部**（深度优先）
3. 所有依赖就绪后，在主线程或后处理线程中组装根资源

**B. 任务令牌与取消机制**

为了避免动态分配任务对象，工业级实现通常使用**固定大小的任务池** + **Salt 校验**：

```cpp
constexpr uint32_t MAX_LOAD_TASKS = 1024;

struct LoadTaskSlot {
    std::atomic<bool> in_use{false};
    uint32_t salt = 0;
    std::atomic<bool> abandoned{false};
};

class TaskPool {
    LoadTaskSlot m_slots[MAX_LOAD_TASKS];
public:
    uint64_t allocate() {
        for (uint32_t i = 0; i < MAX_LOAD_TASKS; ++i) {
            bool expected = false;
            if (m_slots[i].in_use.compare_exchange_strong(expected, true)) {
                uint32_t salt = ++m_slots[i].salt;
                return (uint64_t(salt) << 32) | i;
            }
        }
        return 0; // 池耗尽
    }

    void abandon(uint64_t token) {
        uint32_t index = uint32_t(token);
        uint32_t salt = uint32_t(token >> 32);
        if (m_slots[index].salt == salt && m_slots[index].in_use) {
            m_slots[index].abandoned.store(true);
        }
    }

    bool isAbandoned(uint64_t token) {
        uint32_t index = uint32_t(token);
        return m_slots[index].abandoned.load();
    }
};
```

**为什么用 Salt？** 防止 ABA 问题：任务 A 完成后释放 slot，任务 B 立即复用同一 slot。如果外部仍持有旧 token，Salt 不匹配就能识别出 token 已过期。

**C. 双线程模型：加载线程 + 后处理线程**

当加载逻辑变复杂后，单后台线程会成为瓶颈。工业级引擎通常拆分为：
- **加载线程**：负责磁盘 IO、解压、反序列化
- **后处理线程**：负责依赖解析、引用链接、回调触发

两线程之间通过**无锁队列**（如 `moodycamel::BlockingReaderWriterQueue`）传递 `LoadedAssetInfo`：

```cpp
// 加载线程产生
struct LoadedAssetInfo {
    std::string path;
    ResourceHandle handle;
    std::vector<uint8_t> raw_data;
};

// 后处理线程消费
void postprocess_loop() {
    LoadedAssetInfo info;
    while (m_queue.wait_dequeue_timed(info, 100ms)) {
        // 解析依赖、触发回调、更新状态表
    }
}
```

这种拆分有两个好处：
1. 加载线程不会被复杂的回调逻辑阻塞，磁盘 IO 吞吐最大化
2. 后处理线程可以安全地操作主线程共享的数据结构（如 Handle Table），因为它在主线程的"安全窗口"之外运行，或通过原子操作/锁保护

### 阶段 2：能用 → 好用（解决迭代效率和开发体验）

#### 触发原因

原型验证阶段，美术和策划需要频繁修改资源并立即看到效果。每次重启引擎的迭代成本不可接受。同时，开发期需要从本地文件系统直接读取，而发行版需要打包为资源包（Pak）以优化 IO 和防止篡改。

#### 代码层面的变化

**A. 虚拟文件系统（VFS）与多模式归档**

引擎需要一种抽象的"文件容器"接口，让上层加载代码不关心数据到底来自本地文件夹、压缩包还是网络下载：

```cpp
class IArchive {
public:
    virtual ~IArchive() = default;
    virtual std::vector<uint8_t> readFile(const std::string& path) = 0;
    virtual bool exists(const std::string& path) = 0;
};

class FileSystemArchive : public IArchive { /* 本地文件 */ };
class PakArchive : public IArchive { /* 从 .pak 包中读取 */ };
class MixModeArchive : public IArchive {
    // 开发期优先本地文件，缺失时回退 Pak
};
```

**Pak 包的核心设计**：
- 文件末尾自带**加密指针**，指向包头位置（自描述格式，无需外部元数据）
- 支持 **Chunk 级压缩**（LZ4）和 **AES 加密**
- 支持 **热更包覆盖**：后加载的 patch 包中同名文件覆盖先加载的包

```cpp
struct PakEntry {
    std::string name;
    uint64_t offset;
    uint64_t compressed_size;
    uint64_t original_size;
    uint32_t crc32;
    bool is_compressed;
};

class PakArchive : public IArchive {
    std::unordered_map<std::string, PakEntry> m_index;
    std::vector<uint8_t> m_pkg_data; // 或直接持有文件句柄
public:
    std::vector<uint8_t> readFile(const std::string& path) override {
        auto it = m_index.find(path);
        if (it == m_index.end()) return {};
        // 定位 chunk → CRC 校验 → 解密 → 解压 → 返回
    }
};
```

**B. 热重载（Hot Reload）**

热重载不是"重新加载文件"那么简单，它需要解决三个问题：
1. **检测文件变化**：轮询修改时间戳，或使用 OS 文件系统通知（如 `ReadDirectoryChangesW` / `inotify`）
2. **保持外部引用有效**：重新加载后，所有引用该资源的 Handle 不能失效
3. **通知消费系统更新**：渲染系统需要在下一帧使用新资源

解决方案：**内存复用 + Dirty Handle 机制**

```cpp
class ResourceManager {
    std::unordered_map<std::string, uint64_t> m_last_modified_time;
    std::unordered_map<std::string, ResourceHandle> m_dirty_handles;

public:
    void hotReload(const std::string& path) {
        auto handle = findHandleByPath(path);
        if (!handle) return;

        // 1. 重新读取文件数据
        auto data = m_archive->readFile(path);

        // 2. 在现有 ObjectTable 条目中覆盖数据（保持 Handle 不变）
        if (auto* entry = m_table.get(handle)) {
            deserializeInto(entry, data);
        }

        // 3. 标记 Dirty，供渲染系统等消费端检测
        m_dirty_handles[path] = handle;
    }

    std::unordered_map<std::string, ResourceHandle> getAndClearDirtyHandles() {
        auto out = std::move(m_dirty_handles);
        m_dirty_handles.clear();
        return out;
    }
};
```

渲染系统每帧调用 `getAndClearDirtyHandles()`，如果发现某个 `TextureHandle` 在 dirty 列表中，就将材质中的旧 `RHITexture*` 替换为新创建的 `RHITexture*`。

> **关键洞察**：热重载的难点不在于"重新加载"，而在于**在不破坏现有引用关系的前提下，无缝替换底层数据**。

### 阶段 3：好用 → 工业级（解决性能和规模问题）

#### 触发原因

当引擎进入生产环境，面临开放世界、高分辨率纹理、大量模型时，资源管理遇到新的瓶颈：
1. **显存不足**：4K 纹理 × 数百张 = 数 GB 显存，无法一次性全部加载
2. **加载抖动**：进入新区域时一次性加载大量资源，导致帧率骤降
3. **跨平台差异**：主机有严格的内存预算和专用 IO 子系统（如 PS5 的 SSD IO）

#### 代码层面的变化

**A. 纹理流送（Texture Streaming）与 Mip 预算**

纹理流送的核心思想：**根据物体在屏幕上的大小和距离，只加载所需精度的 Mip 层级**。

```cpp
struct StreamableTexture {
    ResourceHandle handle;
    uint32_t max_mips = 11;      // 4K 纹理约为 11 级 mip
    uint32_t resident_mips = 0;  // 当前在 GPU 上的 mip 数
    uint32_t wanted_mips = 0;    // 根据距离计算出的目标 mip 数
    float min_distance_to_camera = 0.0f;
    bool is_visible = false;
};

class TextureStreamingManager {
    std::vector<StreamableTexture*> m_streaming_textures;
    uint64_t m_memory_budget = 512 * 1024 * 1024; // 512MB 预算

public:
    void tick(float delta_time) {
        // 1. 计算每个纹理的 wanted_mips
        for (auto* tex : m_streaming_textures) {
            if (tex->is_visible) {
                tex->wanted_mips = calculateWantedMips(tex->min_distance_to_camera);
            } else {
                tex->wanted_mips = 1; // 不可见时只保留最低精度
            }
        }

        // 2. 贪心算法：如果总预算超支，从低优先级纹理逐层 drop mip
        uint64_t total_needed = 0;
        for (auto* tex : m_streaming_textures) {
            total_needed += mipMemorySize(tex->handle, tex->wanted_mips);
        }

        if (total_needed > m_memory_budget) {
            dropMipsUntilFit();
        }

        // 3. 发起 StreamIn / StreamOut 请求
        for (auto* tex : m_streaming_textures) {
            if (tex->wanted_mips > tex->resident_mips) {
                requestStreamIn(tex);
            } else if (tex->wanted_mips < tex->resident_mips) {
                requestStreamOut(tex);
            }
        }
    }
};
```

流送的实现细节：
- **StreamIn**：异步加载更高精度的 mip 数据 → 在后台线程创建新的 `RHITexture` → 渲染线程在下一帧执行 `CopySharedMips`（保留旧纹理中仍然有效的低精度 mip）→ 原子替换引用
- **StreamOut**：创建更小尺寸的 `RHITexture` → `CopySharedMips` 从旧纹理拷贝有效数据 → 替换引用并释放旧显存

**B. GPU Upload 与异步拷贝队列**

现代 GPU API（D3D12/Vulkan）支持专门的 **Copy Queue** 用于 CPU → GPU 数据上传，不阻塞 Graphics Queue：

```cpp
class UploadManager {
public:
    void enqueueUpload(ResourceHandle handle, const std::vector<uint8_t>& cpu_data) {
        // 在 Copy Queue 上分配 staging buffer 并上传
        // 上传完成后 signal fence
    }

    bool isUploadComplete(ResourceHandle handle) {
        // 检查 fence 值
    }
};
```

工业级引擎还会使用**分页式 Upload Heap**（如 Vulkan 的 `VulkanUploadHeap`），每帧释放旧页面，避免 staging buffer 的碎片问题。

**C. 预加载与关卡流送**

为了避免玩家走到新区域时才临时加载资源，引擎会在玩家接近边界时**预加载（Prefetch）**下一区域的核心资源：

```cpp
class LevelStreamingManager {
public:
    void updatePlayerPosition(const Vec3& pos) {
        for (auto& level : m_levels) {
            float dist = distance(pos, level.center);
            if (dist < level.load_radius && level.state == LevelState::Unloaded) {
                level.state = LevelState::Loading;
                m_asset_manager->loadBatchAsync(level.asset_list);
            }
            if (dist < level.activate_radius && level.state == LevelState::Loading) {
                if (m_asset_manager->isBatchReady(level.asset_list)) {
                    level.state = LevelState::Active;
                    activateLevel(level.id);
                }
            }
            if (dist > level.unload_radius && level.state == LevelState::Active) {
                level.state = LevelState::Unloaded;
                deactivateLevel(level.id);
            }
        }
    }
};
```

## ECS 重构映射：如果源码不是 ECS，到 ECS 该怎么办？

传统 OOP 引擎中，资源管理通常以 `TextureManager`、`MeshManager`、`MaterialManager` 等单例形式存在。在 ECS 引擎中，这些概念需要被重新映射：

| 传统 OOP 概念 | ECS 表达 | 说明 |
|--------------|---------|------|
| `TextureManager` | `TextureRegistry`（Singleton Component 或 World 资源） | 全局 Handle Table，不绑定到特定实体 |
| `Mesh` | `MeshComponent` | 实体上的组件，存储 `ResourceHandle` 而非 `Mesh*` 指针 |
| `MaterialInstance` | `MaterialComponent` | 存储 PSO Handle、纹理 Handle 列表、Uniform 数据 |
| `AsyncAssetLoader` | `AssetLoadingSystem`（或独立服务） | 通常在主循环外独立运行，不直接作为 System tick |
| `TextureStreamingManager` | `StreamingSystem` | 每帧 tick，根据 CameraComponent 和 TransformComponent 计算 wanted_mips |
| `HotReloadManager` | `HotReloadSystem` | 监听文件变化，修改对应的 ResourceEntry |

### ECS 下的资源生命周期

```cpp
struct MeshComponent {
    ResourceHandle vertex_buffer;
    ResourceHandle index_buffer;
    uint32_t index_count;
};

struct MaterialComponent {
    ResourceHandle pipeline;
    ResourceHandle albedo_texture;
    ResourceHandle normal_texture;
    // ... 其他纹理句柄
};

// 渲染系统查询所有需要绘制的实体
class RenderSystem : public ISystem {
public:
    void tick(World& world, float dt) override {
        auto query = world.query<TransformComponent, MeshComponent, MaterialComponent>();
        for (auto [e, transform, mesh, material] : query) {
            // 从 ResourceManager 查询句柄对应的 GPU 资源是否已就绪
            if (!rm->isReady(mesh.vertex_buffer)) continue;
            if (!rm->isReady(material.albedo_texture)) continue;

            // 绑定并绘制
            cmd->bindPipeline(material.pipeline);
            cmd->bindVertexBuffer(mesh.vertex_buffer);
            cmd->drawIndexed(mesh.index_count);
        }
    }
};
```

**为什么 ECS 更适合资源管理？**
- **显式引用关系**：没有隐式的 `shared_ptr` 嵌套，`MeshComponent` 中的 `ResourceHandle` 一目了然
- **批量操作**：可以一次性查询所有引用某个纹理的实体，进行热重载替换或调试分析
- **AI 可观测性**：Agent 可以直接读取 `MaterialComponent` 中的所有句柄，无需遍历复杂的继承树

## AI 友好设计红线检查

| 检查项 | 资源管理系统的实现策略 |
|--------|----------------------|
| **状态平铺** | `ResourceEntry` 的所有字段（`path`、`state`、`ref_count`、`data`）都平铺在对象表中，可被序列化和 diff。不隐藏在任何单例的内部指针树中。 |
| **自描述** | 所有资源类型（Texture、Mesh、Shader）都在 `TypeRegistry` 中注册其字段描述。AI 可以在不读头文件的情况下知道 `TextureDesc` 包含哪些字段。 |
| **确定性** | 加载顺序由拓扑排序决定，同一资源列表在相同依赖图下产生相同的加载时序。`ResourceHandle` 的分配在快照回滚后必须能稳定复现（建议使用确定性 ID 生成策略）。 |
| **工具边界** | AgentBridge 提供结构化工具：`load_asset(path)`、`unload_asset(handle)`、`query_asset_state(handle)`、`get_dirty_handles()`。返回值统一为 JSON，包含 `path`、`state`、`ref_count`。 |
| **Agent 安全** | AI 不能直接调用底层 `free()` 或 `delete` 来销毁 GPU 资源。所有卸载必须通过 `ResourceManager::unload()` 走引用计数。提供 `resident_file_id_map` 保护核心资源（如默认纹理）不被误卸载。 |

## 设计权衡表

| 决策 | 选项 A | 选项 B | 推荐 |
|------|--------|--------|------|
| Handle 位数 | 32 位 | 64/128 位 | **32 位足够起步**，如需 ABA 防护可加入 Salt 扩展为 64 位 |
| 引用计数 | 侵入式（在资源对象内） | 集中式（在 Handle Table 内） | **集中式**。不污染资源对象，且便于 AI 观测全局引用情况 |
| 加载线程数 | 单线程 | 双线程（加载 + 后处理） | **双线程**。IO 和回调分离，磁盘吞吐最大化 |
| 热更检测 | 轮询修改时间 | OS 文件系统事件 | **开发期用 OS 事件**，发行版无热更需求 |
| 资源包格式 | 自定义 Pak | 直接用 ZIP/7z | **自定义 Pak**。支持 Chunk 级加密、流送、热更覆盖，ZIP 不适合运行时随机读取 |
| 纹理流送粒度 | Mip 级 | 整张贴图 | **Mip 级**。开放世界必备，工业标准做法 |
| GPU Upload | 主线程同步 Upload | Copy Queue 异步 | **Copy Queue**。不阻塞 Graphics Queue，现代 API 标配 |

## 如果我要 vibe coding，该偷哪几招？

1. **Handle-Table 模式**：用轻量 `uint32_t` 句柄代替裸指针，所有资源生命周期集中管理。这是引擎从 demo 走向产品的分水岭。
2. **Salt + 固定任务池**：32K 任务槽 + Salt 校验，零动态分配、零 ABA 风险。高并发异步加载的基础装备。
3. **双线程加载模型**：加载线程只做 IO/反序列化，后处理线程做依赖解析和回调。不要让磁盘等你的回调。
4. **热重载的 Dirty Handle 机制**：资源替换后旧引用不断裂，消费系统每帧检测 dirty 列表并刷新。美术爱死这个功能。
5. **Mip 级纹理流送**：开放世界和大纹理的救命稻草。贪心预算算法简单高效，几小时就能实现一个可用的原型。
6. **Copy Queue 异步上传**：现代 GPU API 下，不要让主线程等 Upload。用 Fence 同步即可。
7. **Pak 自描述格式**：文件末尾自带加密指针，无需外部 catalog 就能解析。分发和热更都非常方便。
