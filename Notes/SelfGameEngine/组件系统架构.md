## Why：为什么游戏引擎需要 ECS？

### 没有这个系统时，世界有多糟

在传统 OOP 游戏对象模型中，世界是这样运转的：

1. **状态深埋继承链**
   - `Player` 继承 `Actor`，`Actor` 继承 `GameObject`，`GameObject` 里有一个 `Transform*` 指针。你想知道"这个对象有哪些属性"，必须逐层读取头文件、追踪虚函数表、解析指针关系。AI 根本不可能在不读源码的情况下理解数据结构。

2. **组合困难，继承爆炸**
   - 一个对象既需要"可受伤"又需要"可拾取"，在 OOP 里只能创建 `DamageablePickupActor` 或塞进两个无关的子对象里。每增加一种能力组合，类树就多一个分支。

3. **批量操作是噩梦**
   - "给所有敌人加速 10%" 需要遍历 `std::vector<Actor*>`，对每个指针解引用，动态类型转换，再修改字段。CPU 在指针森林里跳来跳去，cache miss 严重。

4. **副作用不可追踪**
   - 某个虚函数 `OnTick()` 里悄悄修改了全局光照强度。你找不到"这一帧到底发生了什么变化"的清单。

### 三个真实场景

| 场景 | OOP 的困难 | ECS 的答案 |
|------|-----------|-----------|
| **场景 1：AI 要查询所有带 `Health` 且低于 30% 的实体** | 遍历所有 `Actor*`，动态转换，逐个检查 | 直接查询 `Query<Health>`，线性扫描连续内存 |
| **场景 2：网络同步只需要位置+旋转** | 序列化整个 `Actor` 对象，包含大量无关字段 | 只序列化 `Transform` 组件的紧凑数组 |
| **场景 needing 回放与确定性** | 随机数种子散落在各个对象构造函数里 | 所有状态平铺在世界中，输入固定则输出固定 |

### 这是"必须"还是"优化"

对于传统 3A 游戏，ECS 可以是"优化"——用 OOP 也能做出好游戏。但对于 **AI 协作开发（vibe coding）**，ECS 是**必须**：

- AI 需要"不读头文件就能理解数据结构" → 只有平铺的组件数据才能做到。
- AI 需要"批量修改一万个对象" → 只有连续内存上的线性扫描才能高效完成。
- AI 需要"验证改完对不对" → 只有确定性、无隐藏副作用的系统才能对比 diff。

> **核心结论**：ECS 不是为了性能而性能，而是为了让引擎状态对 AI **可观测、可操作、可验证**。

## What：最简化版本的 ECS 长什么样？

下面的代码是一个**真正可以编译运行**的极简 ECS。相比 [[从零开始的引擎骨架]] 里的 `std::vector` 裸数组，它增加了三个关键能力：

1. **Entity 是轻量 ID（带 generation 防悬空）**
2. **每个组件类型有自己的 `ComponentArray<T>`**
3. **System 可以批量查询所有拥有某组件的实体**

```cpp
#include <cstdint>
#include <vector>
#include <array>
#include <unordered_map>
#include <cassert>
#include <iostream>

// ============================================================
// 1. 轻量 Entity：ID + Generation（防止访问已销毁实体）
// ============================================================
struct Entity {
    uint32_t id{0xFFFFFFFF};
    uint32_t generation{0};
    bool valid() const { return id != 0xFFFFFFFF; }
    bool operator==(const Entity& o) const { return id == o.id && generation == o.generation; }
};

// ============================================================
// 2. 组件数据（纯 POD，无虚函数）
// ============================================================
struct Position { float x{0}, y{0}, z{0}; };
struct Velocity { float x{0}, y{0}, z{0}; };
struct Health   { float current{100.0f}; float max{100.0f}; };
```

### Sparse Set：为 ECS 量身定做的组件存储

在把组件存进 `World` 之前，必须先回答一个关键问题：**如何把一个轻量的 `Entity`（只是个整数 ID）映射到实际的组件数据上？**

最直觉的做法是用 `std::unordered_map<Entity, T>`，但它有两个致命缺陷：
1. **Cache 不友好**：哈希表节点散落在堆内存中，`System` 批量迭代时每一步都是指针追逐。
2. **无法线性扫描**：如果你想"遍历所有 `Position` 组件"，只能遍历哈希表的 `bucket`（[[C++编程/C++ unordered_map 的底层原理与性能陷阱\|详见 unordered_map 的 bucket 机制]]），性能远低于数组。

另一个想法是 `std::vector<std::optional<T>>`，以 `entity.id` 作为数组下标。这确实支持线性扫描，但问题是：
- 实体 ID 可能非常稀疏（比如世界里有 1000 个实体，ID 却分散到 100000），数组会膨胀成巨大的空洞。
- 增删实体时需要在数组中间留下 `nullopt` 空洞，无法保证"真正存在的组件"在内存中连续排列。

**Sparse Set（稀疏集）** 正是为了解决这个矛盾而生的数据结构。它的核心思想用两张表协作：
- **稀疏表（sparse）**：以 `entity.id` 为下标，只存一个整数索引（或无效标记）。这张表可以很大，但每个元素只有 4 字节，允许存在空洞。
- **密集表（dense）**：实际存放组件数据的连续数组。只有"真正拥有该组件的实体"才会被记录在这里，完全无空洞，完美 Cache 友好。

于是，从 `Entity` 到组件数据的查找路径变成：
1. `sparse[entity.id]` → 拿到该实体在 dense 数组中的索引。
2. `dense[index]` → 直接访问组件数据。

这种结构同时满足了三个需求：
- **O(1) 随机访问**：查 `sparse` 表再取 `dense` 元素，两步都是 O(1)。
- **O(1) 增删**：添加时 `dense.push_back`；删除时用 `dense` 末尾元素填充被删位置，更新对应 `sparse` 索引即可。
- **完美 Cache 友好的批量迭代**：`System` 只需要线性扫描 `dense` 数组，不需要关心稀疏的实体 ID 空间。

下面的 `ComponentArray<T>` 就是 Sparse Set 思想在 ECS 组件存储中的最小实现：

```cpp
// ============================================================
// 3. ComponentArray：Sparse Set 风格的密集存储
// ============================================================
template<typename T>
class ComponentArray {
    // entityIndex[entity.id] = 该 entity 在 dense 数组中的索引（或无效标记）
    std::vector<int32_t> entityIndex;
    std::vector<T> dense;           // 实际组件数据，连续存储
    std::vector<uint32_t> denseEntityIds; // dense[i] 对应哪个 entity.id

public:
    void set(Entity e, const T& value) {
        if (e.id >= entityIndex.size()) entityIndex.resize(e.id + 1, -1);
        int32_t idx = entityIndex[e.id];
        if (idx == -1) {
            idx = static_cast<int32_t>(dense.size());
            entityIndex[e.id] = idx;
            dense.push_back(value);
            denseEntityIds.push_back(e.id);
        } else {
            dense[idx] = value;
        }
    }

    void remove(Entity e) {
        if (e.id >= entityIndex.size() || entityIndex[e.id] == -1) return;
        int32_t idx = entityIndex[e.id];
        int32_t lastIdx = static_cast<int32_t>(dense.size()) - 1;
        // 用最后一个元素填充被删除的位置，保持密集
        dense[idx] = dense[lastIdx];
        denseEntityIds[idx] = denseEntityIds[lastIdx];
        entityIndex[denseEntityIds[lastIdx]] = idx;
        dense.pop_back();
        denseEntityIds.pop_back();
        entityIndex[e.id] = -1;
    }

    bool has(Entity e) const {
        return e.id < entityIndex.size() && entityIndex[e.id] != -1;
    }

    T* get(Entity e) {
        if (!has(e)) return nullptr;
        return &dense[entityIndex[e.id]];
    }

    size_t count() const { return dense.size(); }
    T* data() { return dense.data(); }
    uint32_t* entityIds() { return denseEntityIds.data(); }
};

// ============================================================
// 4. World：统一管理 Entity 和 ComponentArray
// ============================================================
class World {
    std::vector<uint32_t> entityGenerations;
    std::vector<uint32_t> freeEntities;

public:
    ComponentArray<Position> positions;
    ComponentArray<Velocity> velocities;
    ComponentArray<Health>   healths;

    Entity create() {
        if (!freeEntities.empty()) {
            uint32_t id = freeEntities.back(); freeEntities.pop_back();
            return {id, entityGenerations[id]};
        }
        uint32_t id = static_cast<uint32_t>(entityGenerations.size());
        entityGenerations.push_back(0);
        return {id, 0};
    }

    void destroy(Entity e) {
        if (e.id >= entityGenerations.size() || entityGenerations[e.id] != e.generation) return;
        positions.remove(e);
        velocities.remove(e);
        healths.remove(e);
        entityGenerations[e.id]++;
        freeEntities.push_back(e.id);
    }

    bool valid(Entity e) const {
        return e.id < entityGenerations.size() && entityGenerations[e.id] == e.generation;
    }
};

// ============================================================
// 5. System：批量迭代组件
// ============================================================
class MovementSystem {
public:
    void tick(World& world, float dt) {
        size_t n = world.positions.count();
        Position* pos = world.positions.data();
        uint32_t* ids = world.positions.entityIds();
        for (size_t i = 0; i < n; ++i) {
            if (Velocity* vel = world.velocities.get({ids[i], world.entityGenerations[ids[i]]})) {
                pos[i].x += vel->x * dt;
                pos[i].y += vel->y * dt;
                pos[i].z += vel->z * dt;
            }
        }
    }
};

// ============================================================
// 主循环
// ============================================================
int main() {
    World world;
    MovementSystem movement;

    Entity player = world.create();
    world.positions.set(player, {0, 0, 0});
    world.velocities.set(player, {1, 0, 0});
    world.healths.set(player, {80, 100});

    for (int i = 0; i < 3; ++i) {
        movement.tick(world, 0.016f);
        auto* p = world.positions.get(player);
        std::cout << "Frame " << i << ": (" << p->x << ", " << p->y << ")\n";
    }

    world.destroy(player);
    return 0;
}
```

### 这个最小实现已经解决了什么

| 能力 | 说明 |
|------|------|
| **Entity 是真正的轻量 ID** | 只有 8 字节（ID + generation），不携带任何状态 |
| **组件密集存储** | `ComponentArray<T>::dense` 是连续数组，批量迭代 cache 友好 |
| **快速增删** | Sparse Set 机制让组件增删都是 O(1)，且不影响其他组件 |
| **防悬空访问** | 销毁 Entity 时 generation 递增，旧 Entity 值自然失效 |
| **纯数据组件** | `Position`、`Velocity`、`Health` 都是 POD，可直接序列化、diff |

### 这个最小实现还缺什么

- **没有多组件联合查询**：`MovementSystem` 里需要手动检查 `velocities.get()`，无法表达 "给我所有同时有 Position 和 Velocity 的实体"
- **没有组件类型注册表**：`World` 里还是硬编码了 `positions`、`velocities`、`healths`
- **没有命令缓冲**：运行时增删组件会直接修改 `ComponentArray`
- **没有 Archetype 分组**：虽然同类型组件连续，但不同组件类型之间没有按 Entity 组合聚类

这些正是我们接下来要演进的步骤。

## ECS 重构映射：从 OOP Component-Based 到纯 ECS

很多成熟引擎（包括我们分析的源码）并非纯 ECS，而是**OOP 风格的 Component-Based 架构**。在提炼通用知识时，必须明确：如果目标是 AI 友好的 ECS 引擎，源码中的这些结构应该如何重构？

| 源码中的 OOP 结构 | 在 ECS 中的对应 | 重构原因 |
|------------------|----------------|---------|
| **重量级 `EntityProxy`**（携带 Prefab 数据、创建配置、父子 DataRef） | **轻量 `Entity`**（仅 ID + generation） | 实体不应有状态。Prefab 实例化数据、创建配置应作为独立的 `PrefabInstance` 组件；父子关系应作为 `Hierarchy` 组件。 |
| **多态 `Component` 基类**（带虚函数 `initialize()` / `clear()`） | **纯 POD Component**（无继承、无虚函数） | 虚函数表破坏数据连续性，且使组件大小不可预测。初始化/清理逻辑应移到 System 或专门的 `ComponentLifecycleSystem`。 |
| **`PossessedComponentHandle` + `DatumTable`**（引用计数对象池） | **`ComponentArray<T>` 密集数组** + Entity generation 校验 | Handle 解引用增加一次指针追逐。ECS 中实体 ID 本身就是"句柄"，通过 generation 防止 ABA 问题。 |
| **`EntityComponentManager`**（`Map<EntityID, Map<Group, Array<Handle>>>`） | **Archetype 表** 或 **Sparse Set 组件数组** | 多层哈希映射 cache 不友好，跨类型查询需要多次查找。Archetype 按组件组合聚类，单次线性扫描即可获取所有相关组件。 |
| **`Manager`**（System + Component 生命周期管理的混合体） | **`System`**（纯逻辑） + **`World`**（统一管理数据） | Manager 职责过重，既被调度又直接管理组件类型。ECS 中 System 只读/写 World 数据，不拥有任何组件。 |
| **`LevelTickManager` 的 9 阶段串行调度** | **Scheduler + 拓扑排序 + 可选并行** | 分阶段调度思想保留，但 System 之间应是纯数据依赖关系，支持 Job System 并行执行无依赖的 System。 |
| **延迟命令缓冲（`ConcurrentQueue<ComponentCreationData>`）** | **World::CommandBuffer** | 命令缓冲是优秀设计，直接迁移到 ECS 中。在 Tick 边界统一应用组件增删，保证迭代安全。 |

> **关键洞察**：源码中的许多设计（Handle 对象池、引用计数、延迟 GC）是为了解决 OOP 架构下对象生命周期复杂的问题。在纯 ECS 中，实体只是整数 ID，组件只是数组元素，生命周期管理被极大地简化了——大部分"复杂度"随着架构迁移而自然消失。

## How：真实引擎的 ECS 是如何一步一步复杂起来的？

### 阶段 1：能用 — 从 ComponentArray 到 Archetype 萌芽

#### 触发原因

当实体数量从几十个增长到几千个，System 数量从 2 个增长到 20 个时：

1. **联合查询太麻烦**：`MovementSystem` 需要 `Position + Velocity`，`RenderSystem` 需要 `Position + Mesh + Material`。每次都要写一堆 `if (world.X.has(e) && world.Y.has(e))`，既繁琐又容易漏。
2. **内存局部性不够好**：虽然单个 `ComponentArray` 内部是连续的，但 `Position` 数组和 `Velocity` 数组是两个独立的内存块。System 迭代时要在两个数组之间来回跳转。
3. **组件增删导致迭代器失效**：直接调用 `world.positions.remove(e)` 可能让正在遍历 `positions.dense` 的 System 读到错误数据。

#### 代码层面的变化

**A. 引入 Query 抽象**

```cpp
template<typename... Cs>
struct Query {
    World* world;
    // 返回所有同时满足 Cs... 的实体及其组件指针
    struct Iterator { Entity e; std::tuple<Cs*...> comps; };
    std::vector<Iterator> all();
};
```

最小实现可以通过遍历最稀疏的那个组件数组，再检查其他组件是否存在：

```cpp
// 简化版 Query：遍历 Position，筛选出同时有 Velocity 的
for (size_t i = 0; i < world.positions.count(); ++i) {
    Entity e = {world.positions.entityIds()[i], world.entityGenerations[world.positions.entityIds()[i]]};
    if (auto* vel = world.velocities.get(e)) {
        auto* pos = world.positions.get(e);
        // 处理 pos, vel
    }
}
```

> 这是**Query 的朴素实现**。它解决了"多组件联合查询"的表达问题，但性能上仍有优化空间。

**B. 引入命令缓冲（Command Buffer）**

```cpp
struct CommandBuffer {
    std::vector<std::pair<Entity, Position>> pendingPositions;
    std::vector<std::pair<Entity, Velocity>> pendingVelocities;
    std::vector<Entity> pendingDestroy;

    void stagePosition(Entity e, const Position& p) { pendingPositions.push_back({e, p}); }
    void stageVelocity(Entity e, const Velocity& v) { pendingVelocities.push_back({e, v}); }
    void stageDestroy(Entity e) { pendingDestroy.push_back(e); }

    void apply(World& w) {
        for (auto& [e, p] : pendingPositions) w.positions.set(e, p);
        for (auto& [e, v] : pendingVelocities) w.velocities.set(e, v);
        for (auto& e : pendingDestroy) w.destroy(e);
        clear();
    }
    void clear() {
        pendingPositions.clear();
        pendingVelocities.clear();
        pendingDestroy.clear();
    }
};
```

命令缓冲让 AI 或编辑器可以在任意时刻提交修改请求，真正的数据变更只在 Tick 边界发生。这是保证迭代安全、支持事务回滚的关键基础设施。

**C. 按 Archetype 分组（萌芽）**

为了解决跨类型查询的性能问题，可以把拥有相同组件组合的实体放到一起：

```cpp
using ComponentMask = uint32_t; // 位掩码，每位代表一种组件

struct Archetype {
    ComponentMask mask;
    std::vector<Entity> entities;
    std::vector<Position> positions;  // 只有 mask 包含 Position 时才非空
    std::vector<Velocity> velocities; // 同上
};

class World {
    std::unordered_map<ComponentMask, Archetype> archetypes;
public:
    Archetype& getOrCreateArchetype(ComponentMask mask) {
        return archetypes[mask];
    }
};
```

> 这是 Archetype 的最原始形态。它把"拥有 Position + Velocity 的实体"聚类在一起，`MovementSystem` 只需要遍历这个 Archetype 即可，无需检查 `velocities.has()`。

---

### 阶段 2：好用 — Archetype 存储与 Query 缓存

#### 触发原因

- 实体类型组合变得复杂：`Player` 有 8 个组件，`Enemy` 有 6 个，`Tree` 只有 2 个。
- Query 被反复执行：每帧都要查询 "所有带 `Transform + Mesh` 的实体"，每次都重新计算结果太浪费。
- 组件增删导致 Entity 在 Archetype 之间迁移：给敌人加一个 `Burning` 状态，它要从 `EnemyArchetype` 迁移到 `EnemyBurningArchetype`。

#### 代码层面的变化

**A. 真正的 Archetype 存储（SoA within Archetype）**

```cpp
struct Archetype {
    ComponentMask mask;
    size_t entityCount{0};
    size_t capacity{64};
    char* data{nullptr}; // 统一分配的内存块

    // 每个组件在 data 中的偏移量
    std::unordered_map<ComponentTypeID, size_t> offsets;
};
```

内存布局：

```
Archetype (mask = Position | Velocity)
├─ Entity[0]  Entity[1]  Entity[2]  ...  (64 个)
├─ Position[0] Position[1] Position[2] ... (64 个)
└─ Velocity[0] Velocity[1] Velocity[2] ... (64 个)
```

每个 Archetype 内部是 **SoA（Structure of Arrays）**。同一 Archetype 的所有组件都存储在一个连续内存块中，批量查询时只需要线性扫描即可。

**B. Entity Record：指向 Archetype 的索引**

```cpp
struct EntityRecord {
    Archetype* archetype{nullptr};
    size_t index{0}; // 在该 archetype 中的行号
};

class World {
    std::vector<EntityRecord> entityRecords;
    // ...
};
```

当 Entity 的组件组合变化时：
1. 从旧 Archetype 的 `index` 位置移除（用最后一个元素填充以保持密集）。
2. 更新被移动元素的 `EntityRecord`。
3. 把 Entity 插入新 Archetype 的末尾。

这个过程被称为 **Archetype Migration**，是现代 ECS（如 Bevy、Unity DOTS）的核心操作。

**C. Query 缓存与 Archetype Graph**

```cpp
class QueryCache {
    ComponentMask requiredMask;
    std::vector<Archetype*> matchedArchetypes;
public:
    void refresh(const std::unordered_map<ComponentMask, Archetype>& allArchetypes) {
        matchedArchetypes.clear();
        for (auto& [mask, arch] : allArchetypes) {
            if ((mask & requiredMask) == requiredMask) {
                matchedArchetypes.push_back(const_cast<Archetype*>(&arch));
            }
        }
    }
};
```

- 当新 Archetype 被创建时，自动检查它是否匹配已有的 Query。
- 高级实现（如 Bevy）会维护 **Archetype Graph**：从 `mask=A` 到 `mask=A|B` 的添加边、到 `mask=A-B` 的删除边，让组件增删时的 Archetype 查找从 O(N) 降到 O(1)。

### 阶段 3：工业级 — Chunk-based、并行调度、世界快照

#### 触发原因

- 实体数量达到 10 万+
- 需要在多核 CPU 上并行执行 System
- 需要网络同步、确定性回放、事务沙箱

#### 代码层面的变化

**A. Chunk-based 存储**

当单个 Archetype 的实体数量超过几千时，把它拆分成固定大小的 **Chunk**（如每个 Chunk 16KB，容纳 100~1000 个实体）：

```cpp
struct Chunk {
    size_t count{0};
    char data[CHUNK_SIZE]; // 按 SoA 布局存储组件
};

struct Archetype {
    ComponentMask mask;
    std::vector<Chunk*> chunks;
    size_t chunkCapacity; // 每个 Chunk 能容纳多少个实体
};
```

Chunk 的好处：
- **内存分配更稳定**：增删实体时只需要在 Chunk 内部移动，不需要重新分配整个 Archetype 的巨型数组。
- **更好的并行粒度**：一个 System 可以把不同 Chunk 分配给不同线程执行。
- **快照更高效**：只需要复制被修改的 Chunk，而不是整个世界。

**B. 并行 System 调度**

```cpp
class Scheduler {
    struct SystemNode {
        std::function<void(World&)> tick;
        ComponentMask reads;
        ComponentMask writes;
    };
    std::vector<SystemNode> systems;

public:
    void runParallel(World& world) {
        // 1. 检测依赖：如果两个 System 的 write mask 与另一个的 read/write mask 重叠，则不能并行
        // 2. 无依赖的 System 放入任务队列，由线程池并行执行
    }
};
```

> 例如：`RenderSystem`（读 Transform, Mesh, Material）和 `AudioSystem`（读 Transform, AudioSource）可以并行，因为它们不写相同的组件。但 `MovementSystem`（写 Position）和 `CollisionSystem`（读 Position）必须串行。

**C. 世界快照与回滚**

```cpp
class WorldSnapshot {
    // 只保存自上次快照以来发生变化的 Chunk
    std::unordered_map<Chunk*, std::vector<char>> dirtyChunks;
public:
    static WorldSnapshot capture(const World& w);
    static void restore(World& w, const WorldSnapshot& snap);
};
```

在 ECS 中，快照变得非常自然：
- 复制所有 Chunk 的数据块（memcpy，不需要逐个对象序列化）。
- 回滚时把 Chunk 数据原样写回。
- 由于 Entity 只是 ID，只要 `entityGenerations` 不变，所有引用关系就完全恢复。

## 设计权衡表

| 决策点 | 原型阶段（< 1000 实体） | 工业级阶段（> 10 万实体） |
|--------|----------------------|-------------------------|
| 存储模型 | `ComponentArray<T>` Sparse Set | Archetype + Chunk-based SoA |
| 查询方式 | 硬编码遍历 + 手动 `has()` 检查 | Query 缓存 + Archetype Graph 快速匹配 |
| 命令缓冲 | `std::vector` 延迟应用 | 分 Chunk 的增量变更队列 |
| 并发模型 | 单线程串行 | 基于组件读写 mask 的依赖分析 + Job System |
| 快照策略 | 全量 memcpy 所有 ComponentArray | 只复制脏 Chunk（Copy-on-Write） |
| Entity 校验 | generation 计数器 | generation + 版本化 Archetype 索引 |

## 如果我要 vibe coding，该偷哪几招？

### 骨架层必偷

1. **Entity 必须是轻量 ID，不能是对象**
   - 任何把"实体"设计成类的冲动都要压下去。实体只是一个整数，状态全部在组件里。

2. **组件必须是纯 POD，不能带虚函数**
   - 虚函数表 = 不可序列化 = AI 不可预测。初始化/清理逻辑交给 System。

3. **用 Sparse Set 做 ComponentArray 的起步实现**
   - 它比哈希表快，比 Archetype 简单，是原型阶段的最佳选择。

4. **命令缓冲从 Day 1 就引入**
   - 不要等到出 bug 才加。它是 AI 安全操作引擎的底线。

5. **给每个 System 标注读/写组件 mask**
   - 即使现在不并行，也要为未来并行调度留下元数据。

### ECS 重构层必偷

6. **如果源码是 OOP Component-Based，画一张 ECS 重构映射表**
   - 明确哪些类该消失、哪些状态该变成组件、哪些 Manager 该拆成 System。

7. **Archetype 不是必须的，但 Query 表达是必须的**
   - 原型阶段可以用 Sparse Set + 朴素 Query；规模上去后再引入 Archetype。

8. **组件增删触发 Archetype Migration 时，用"最后元素填充"保持密集**
   - 不要为了让 entity index 不变而留空洞，密集存储比稳定索引更重要。

### AI 层必偷

9. **让 `query_entities` 支持组件掩码查询**
   - 这是 AI 最常用的接口："给我所有带 Health 的敌人"。

10. **为 AI 提供 `get_archetype_summary()` 工具**
    - 返回当前世界有多少种 Archetype、每种有多少实体。这能让 AI 快速理解世界的结构，而不是盲目遍历。
