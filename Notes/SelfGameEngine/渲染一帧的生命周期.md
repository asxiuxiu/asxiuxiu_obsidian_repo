# 渲染一帧的生命周期

## Why：为什么必须理解"一帧是怎么画出来的"？

想象一下：你已经搭建了 ECS 世界、写好了 RHI 抽象层、管理好了纹理和网格资源，但运行 `main()` 后屏幕依然一片漆黑。这就像剧组有了演员、摄影机和剧本，但导演不知道"一场戏该怎么拍"。**渲染管线的生命周期，就是把"世界状态"翻译为"屏幕像素"的完整流程。**

### 没有这个闭环时世界有多糟

- **场景 1：美术导入了模型，但看不到任何东西**。因为没有明确的"在哪一帧、按什么顺序、用什么状态去画"的机制。
- **场景 2：网络同步调试发现画面不一致**。同一世界状态在两台机器上产生了不同的像素，因为 Draw Call 的提交顺序不确定。
- **场景 3：AI Agent 想帮你调画面，但完全无从下手**。它不知道这一帧画了多少个物体、用了哪些渲染目标、哪个 Pass 出了问题。

### 这是"必须"还是"优化"？

**必须**。没有渲染生命周期，引擎永远无法输出画面。它是阶段 4（第一帧画面渲染）的**最后一块拼图**。

---

## What：最简化版本的渲染生命周期长什么样？

下面的代码是一个**阶段 1 能用**的最小渲染系统。它假设你已经有了：
- `World` 和 ECS 查询能力（来自 [[组件系统架构]]）
- `RHICommandList`（来自 [[RHI抽象层]]）
- `ResourceHandle` 管理（来自 [[资源管理]]）
- `Transform` 组件的世界矩阵（来自 [[场景图与变换]]）

```cpp
// 阶段 1：最小可运行渲染系统（约 70 行）
struct MeshComponent {
    RHIBufferHandle vertex_buffer;
    RHIBufferHandle index_buffer;
    uint32_t        index_count;
};

struct MaterialComponent {
    RHIPipelineHandle pipeline;
    RHITextureHandle  albedo_texture;
};

struct TransformComponent {
    Mat4 world_matrix;
};

struct CameraComponent {
    Mat4 view_matrix;
    Mat4 projection_matrix;
};

struct DrawCall {
    RHIPipelineHandle pipeline;
    RHIBufferHandle   vb;
    RHIBufferHandle   ib;
    uint32_t          index_count;
    Mat4              mvp;
    uint64_t          sort_key; // 用于确定性排序
};

class MinimalRenderSystem {
public:
    void render(World& world, RHICommandList* cmd, const CameraComponent& camera) {
        // 1. ECS 查询：找出所有"有 Mesh + Material + Transform"的实体
        auto query = world.query<TransformComponent, MeshComponent, MaterialComponent>();

        std::vector<DrawCall> draws;
        draws.reserve(query.count_hint());

        // 2. 生成 DrawCall
        for (auto [transform, mesh, material] : query) {
            DrawCall dc;
            dc.pipeline     = material.pipeline;
            dc.vb           = mesh.vertex_buffer;
            dc.ib           = mesh.index_buffer;
            dc.index_count  = mesh.index_count;
            dc.mvp          = camera.projection_matrix *
                              camera.view_matrix *
                              transform.world_matrix;
            // 简单排序键：高 32 位 pipeline，低 32 位 texture
            dc.sort_key     = (uint64_t(material.pipeline.id) << 32) | material.albedo_texture.id;
            draws.push_back(dc);
        }

        // 3. 确定性排序：减少 PSO 切换，保证回放一致性
        std::sort(draws.begin(), draws.end(),
                  [](const DrawCall& a, const DrawCall& b) {
                      return a.sort_key < b.sort_key;
                  });

        // 4. 录制 RHI 命令
        cmd->begin_render_pass({ .clear_color = {0.1f, 0.1f, 0.1f, 1.0f} });
        for (const auto& dc : draws) {
            cmd->bind_pipeline(dc.pipeline);
            cmd->bind_vertex_buffer(dc.vb);
            cmd->bind_index_buffer(dc.ib);
            cmd->push_constants(&dc.mvp, sizeof(Mat4));
            cmd->draw_indexed(dc.index_count);
        }
        cmd->end_render_pass();
    }
};
```

### 这个最小实现已经解决了什么？

- **平铺状态到 Draw Call**：所有渲染输入都来自 ECS 组件，没有任何隐藏的继承树或单例状态。
- **确定性排序**：`std::sort` 保证相同场景永远产生相同的命令序列——这是网络同步和 AI 调试的基石。
- **与平台解耦**：底层只依赖 `RHICommandList`，具体是 D3D12 还是 Vulkan 对上层不可见。

### 还缺什么？

- 没有**视锥剔除**：所有实体都会被画，哪怕在相机背后。
- 没有**多 Pass 支持**：不能做 Shadow Map、GBuffer、后处理。
- 没有**资源 Barrier 管理**：如果某个 texture 刚被写入就被读取，可能产生 GPU 同步错误。
- 没有**多线程录制**：Draw Call 一多，CPU 会成为瓶颈。

这些缺口正是后面两阶段要填补的。

## How：真实引擎的渲染生命周期是如何一步一步复杂起来的？

### 阶段 1：最小实现 → 能用（解决"别画看不见的东西"）

**触发原因**：场景中只有 10% 的物体在相机视野内，但最小实现会把 100% 的实体都提交给 GPU。Overdraw 和顶点浪费会迅速拖垮帧率。

**代码层面的变化**：

在生成 DrawCall 之前，加入**视锥剔除系统**（Frustum Culling）。这是一个独立的 ECS System，在 RenderSystem 之前运行：

```cpp
class CullingSystem {
public:
    void cull(World& world, const CameraComponent& camera) {
        auto query = world.query<TransformComponent, MeshComponent>();
        for (auto [transform, mesh] : query) {
            AABB world_aabb = transform_aabb(mesh.local_aabb, transform.world_matrix);
            bool visible = frustum_intersects_aabb(camera.frustum, world_aabb);
            
            Entity e = world.entity_from_current_iteration();
            if (visible) {
                world.add_component<VisibleTag>(e);
            } else {
                world.remove_component<VisibleTag>(e);
            }
        }
    }
};
```

RenderSystem 的查询从 `<Transform, Mesh, Material>` 改为 `<Transform, Mesh, Material, VisibleTag>`：

```cpp
auto query = world.query<TransformComponent, MeshComponent,
                         MaterialComponent, VisibleTag>();
```

**新增数据结构**：
- `VisibleTag`：一个零大小的标记组件（Tag），表示该实体通过了剔除测试。
- `CullingSystem`：在 Tick 调度中注册为 RenderSystem 的前置依赖。

**为什么 ECS 更适合剔除？**
- 剔除和渲染**完全解耦**。你可以随时替换剔除算法（BVH、GPU Driven、PVS）而不影响 RenderSystem。
- `VisibleTag` 的存在让 AI 可以直接查询"当前帧有哪些实体可见"，无需理解复杂的视锥数学。

---

### 阶段 2：能用 → 好用（解决"Pass 太多、状态太乱"）

**触发原因**：随着需求增长，一帧需要多个 Pass：Shadow Map → Depth Prepass → GBuffer → SSAO → Deferred Lighting → Forward Transparent → PostProcess → UI。如果所有这些都在一个巨大的函数里硬编码，任何修改都会牵一发而动全身。

**代码层面的变化**：引入**RenderQueue** 和 **RenderGraph**。

#### 2.1 RenderQueue：按材质排序与合并

不再直接遍历 ECS 提交 Draw Call，而是先把所有可见物体的绘制请求收集到一个 `RenderQueue` 中，再按**排序键**统一提交：

```cpp
struct RenderQueueItem {
    uint64_t sort_key;   // 高 bits：材质/PSO；低 bits：深度
    DrawCall draw_call;
};

class RenderQueue {
    std::vector<RenderQueueItem> m_items;
public:
    void push(const DrawCall& dc, uint64_t sort_key) {
        m_items.push_back({sort_key, dc});
    }
    void sort_and_submit(RHICommandList* cmd) {
        std::sort(m_items.begin(), m_items.end(),
                  [](const auto& a, const auto& b) {
                      return a.sort_key < b.sort_key;
                  });
        for (auto& item : m_items) {
            item.draw_call.record(cmd);
        }
    }
};
```

排序策略示例：
- **不透明物体**：按材质/PSO 分组优先（减少状态切换），深度其次（利用 Early-Z）。
- **透明物体**：按深度从远到近排序（保证正确混合）。

#### 2.2 RenderGraph：声明式 Pass 管理

RenderGraph 把一帧的渲染过程抽象为**有向无环图（DAG）**：节点是 Pass，边是资源依赖。

```cpp
class RenderGraph {
public:
    void setup() {
        m_passes.clear();
        // 各子系统在此注册 Pass
    }
    
    void add_pass(const std::string& name,
                  const std::vector<std::string>& inputs,
                  const std::vector<std::string>& outputs,
                  std::function<void(RHICommandList*)> execute) {
        m_passes.push_back({name, inputs, outputs, execute});
    }
    
    void compile() {
        build_dependency_graph();  // 根据 input/output 建 DAG
        cull_dead_passes();        // 从最终输出反向遍历，剔除无关 Pass
        topological_sort();        // 确定执行顺序
        compute_resource_lifetime(); // 计算每个 transient 资源的生命周期
    }
    
    void execute(RHICommandList* cmd) {
        for (auto& pass : m_sorted_passes) {
            insert_barriers_if_needed(pass); // 自动推导资源状态转换
            pass.execute(cmd);
        }
    }
};
```

**典型帧图结构**：

```
ShadowPass          DepthPrepass
    \                   /
     \                 /
      GBufferPass ←───┘
          │
      SSAOPass
          │
      LightingPass
          │
      ForwardTransparentPass
          │
      PostProcessPass
          │
      UIPass
          │
      Present
```

**RenderGraph 解决的核心问题**：
- **自动 Pass 剔除**：如果 SSAO 被关闭，所有只连接到 SSAO 的 Pass 会在 `compile()` 时被自动剔除。
- **自动 Barrier 推导**：RenderGraph 知道每个资源的上一个使用者和下一个使用者，能自动插入 `RT→SRV`、`SRV→UAV` 等转换屏障。
- **Transient Resource Aliasing**：生命周期不重叠的临时纹理（如 SSAO 输出和 Bloom 中间缓冲）可以复用同一块 GPU 内存。

---

### 阶段 3：好用 → 工业级（解决"CPU 瓶颈和 GPU 利用率"）

**触发原因**：当场景规模达到数千 Draw Call 时，单线程 CPU 录制命令会成为瓶颈；同时，大量临时 RT 和同步点会导致 GPU 空闲。

**代码层面的变化**：

#### 3.1 多线程命令录制

现代 API（D3D12 / Vulkan）允许从多个线程同时录制独立的 `CommandList`，最后统一提交到 GPU：

```cpp
class CommandListPool {
    std::vector<RHICommandList*> m_pool;
public:
    RHICommandList* acquire() { /* 从池中取出一个复用的 CommandList */ }
    void release(RHICommandList* cmd) { /* 重置并归还 */ }
};

// RenderGraph 的每个 Pass 可以从池中获取独立 CommandList
void RenderGraph::execute_parallel() {
    std::vector<std::future<void>> futures;
    for (auto& pass : m_parallel_ready_passes) {
        futures.push_back(thread_pool.submit([&pass]() {
            auto* cmd = g_cmd_pool.acquire();
            pass.execute(cmd);
            pass.command_list = cmd;
        }));
    }
    for (auto& f : futures) f.wait();
    
    // 主线程按拓扑顺序提交所有 CommandList
    for (auto& pass : m_sorted_passes) {
        g_rhi->submit(pass.command_list);
    }
}
```

#### 3.2 GPU Driven Rendering / Indirect Draw

当物体数量达到万级时，CPU 逐一遍历实体生成 Draw Call 已经不可接受。工业级引擎会把可见性结果和绘制参数直接写到 **GPU Scene Buffer** 中，用 `ExecuteIndirect` 或 `MultiDrawIndirect` 让 GPU 自己决定画什么：

```cpp
// GPU 可见性缓冲：每个可见实体对应一个 IndirectCommand
struct GPUIndirectCommand {
    uint32_t index_count;
    uint32_t instance_count;
    uint32_t first_index;
    int32_t  vertex_offset;
    uint32_t first_instance;
};
```

CPU 的工作被压缩为：
1. 更新相机参数和全局常量缓冲。
2. 提交一个 Compute Shader 做 GPU Frustum Culling。
3. 调用一次 `ExecuteIndirect`，GPU 自动完成剩下的所有绘制。

#### 3.3 Async Compute 与多队列

某些 Pass（如 SSAO、Bloom、物理模拟）可以运行在独立的 **Compute Queue** 上，与 Graphics Queue 并行：

```
Graphics Queue:  GBuffer → [Barrier] → Lighting → Forward → Present
Compute Queue:           [SSAO Compute] ────────┘
```

RenderGraph 的 `compile()` 阶段会分析哪些 Pass 没有图形依赖，将其标记为 Async Compute，并在执行时插入 `Fence` 同步。

## ECS 重构映射：从 OOP 渲染管线到 ECS

如果源码分析中的渲染管线基于传统的 `RenderEntity` 继承或 `RenderPipeline` 硬编码调度，迁移到 ECS 时应做如下映射：

| 源码/传统概念 | ECS 下的表达 | 说明 |
|-------------|------------|------|
| `RenderEntity`（带 Mesh/Transform 的渲染对象） | 一个拥有 `MeshComponent + MaterialComponent + TransformComponent` 的 Entity | 没有专门的"渲染实体"类，组合即身份 |
| `VisibilityResult`（可见性结果列表） | `CullingSystem` 输出 `VisibleTag`；或写入 `RenderContext` 资源中的 `VisibleEntityList` | 平铺的可见性数据，可被 AI 直接查询 |
| `RenderPipeline`（硬编码 Pass 调度器） | `RenderGraph` 资源 + 多个独立 System（ShadowSystem, GBufferSystem, LightingSystem...） | 每个 System 只负责一类 Pass，通过 RenderGraph 组合 |
| `RenderView`（相机 + Viewport + 结果容器） | `CameraComponent` + `ViewportConfig` 组件 + `RenderViewResource` | 数据完全平铺，一台相机就是一个 Entity |
| `RenderTaskGraph`（多线程任务调度） | `Scheduler` 中注册的并行 System 任务 | 利用已有的 ECS Tick 调度器做并行，不另起炉灶 |

**为什么 ECS 更适合渲染管线？**
- **状态平铺可见**：AI 不需要解析复杂的继承树，只需要查询"哪些 Entity 有 `MeshComponent`"即可知道场景中所有可绘制对象。
- **组合灵活性**：一个 Entity 可以是 `Mesh + OpaqueMaterial`，也可以额外加上 `ShadowCasterTag`、`TransparentTag`、`DecalTag`，无需修改 Mesh 类。
- **Cache Locality**：RenderSystem 批量读取 `MeshComponent` 数组时，顶点缓冲句柄在内存中连续排列，排序和提交效率更高。

## AI 友好设计红线检查

| 检查项 | 渲染管线的实现策略 |
|--------|------------------|
| **状态平铺** | 所有可渲染状态都在 ECS 组件中（`MeshComponent`、`MaterialComponent`、`TransformComponent`、`VisibleTag`）。`RenderQueue` 和 `RenderGraph` 的 Pass 列表都是平铺数组，可被序列化和 diff。 |
| **自描述** | `MeshComponent`、`MaterialComponent` 的字段在 `TypeRegistry` 中注册。AI 可以不读头文件就知道：一个 Mesh 包含 `vertex_buffer`（句柄类型）和 `index_count`（uint32）。RenderGraph 的 Pass 名称和资源名也以字符串形式存储。 |
| **确定性** | **Draw Call 排序是确定性的**（按 sort_key 稳定排序）。Camera 的 View/Proj 矩阵在同一 Tick 相位中计算。这保证了给定相同的 ECS 快照，回放时产生的像素输出一致。 |
| **工具边界** | AgentBridge 提供结构化工具：`query_render_stats()` 返回 `{draw_calls, pso_switches, render_targets}`；`capture_frame()` 返回当前帧的 RenderGraph dump（JSON）。AI 不能直接调用底层 RHI 的 `draw()`，只能修改 ECS 组件，由 System 在下一 Tick 生效。 |
| **Agent 安全** | 禁止 AI 通过工具直接修改 `RHICommandList` 或 GPU 内存。所有渲染状态的变更必须走 ECS 组件 → System → RenderQueue 的正规路径。RenderGraph 的 `compile()` 阶段会验证资源依赖，防止非法的读写链。 |

## 设计权衡表

| 决策 | 选项 A | 选项 B | 推荐 |
|------|--------|--------|------|
| 剔除粒度 | 粗粒度 AABB | 精细 OBB / 逐三角形 | **AABB 起步**。90% 的收益来自粗剔除，精细剔除用 GPU Driven 解决 |
| 排序策略 | 按材质排序（减少 PSO 切换） | 按深度排序（减少 Overdraw） | **不透明按材质，透明按深度**。混合策略 |
| RenderGraph 编译频率 | 每帧重新 compile | 缓存静态图，只更新动态部分 | **小型引擎每帧 compile 简单图；工业级缓存静态子图** |
| Barrier 管理 | RHI 自动推导 | RenderGraph 全局优化 | **小型引擎选 A，工业级选 B** |
| 多线程录制 | 一 Pass 一 CommandList | 一阶段一 CommandList | **一 Pass 一 List**。粒度适中，合并开销小 |
| GPU Driven | CPU 遍历 Draw Call | GPU Indirect Draw | **数千 Draw Call 以上再引入 GPU Driven**，否则开发成本大于收益 |

## 如果我要 vibe coding，该偷哪几招？

1. **可见性 Tag + 查询驱动渲染**：用 `VisibleTag` 把剔除和绘制彻底解耦。这是 ECS 渲染管线的第一块基石。
2. **确定性 Draw Call 排序**：不要小看排序。它是网络同步一致性、AI 调试回放、单元测试画面的生命线。
3. **RenderQueue 中间层**：在 ECS 和 RHI 之间加一个 RenderQueue，按 sort_key 批量提交。几小时就能实现，但能让你的管线从 demo 级别跃升到可扩展架构。
4. **RenderGraph 的最小实现**：你不需要 Frostbite 级别的图编译器。一个能 `setup → compile → execute`、支持 `cullDeadPasses` 和自动 barrier 的最简 RenderGraph，大约 300 行代码就能让你的多 Pass 管线井井有条。
5. **Transient Resource Aliasing**：生命周期不重叠的临时 RT 复用同一块显存。这是控制显存占用的最便宜招数。
6. **CommandList Pool**：不要让每个线程自己 `new` CommandList。一个对象池就能支撑起多线程录制的骨架。
