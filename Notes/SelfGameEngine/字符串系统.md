## Why：为什么游戏引擎需要自定义字符串系统？

`std::string` 在普通业务代码里足够好用，但在游戏引擎的热路径上，它是一头隐藏的内存怪兽。

### std::string 在引擎里会有多糟？

假设你有一个 60 FPS 的游戏，每帧打印 100 条日志：
- 每条日志平均 64 字节
- `std::string` 的格式化几乎必然触发堆分配
- 结果：每秒 6000 次小内存分配，GC/碎片/缓存未命中的噩梦

### 三个真实场景

| 场景          | 发生了什么                                          | 痛点                                        |
| ----------- | ---------------------------------------------- | ----------------------------------------- |
| **每帧日志格式化** | `LOG("Player {0} pos=({1}, {2})", name, x, y)` | `std::ostringstream` 动态分配内部缓冲，虚函数调用林立     |
| **资源名查找**   | `loadTexture("characters/hero_diffuse.dds")`   | 相同字符串被反复哈希、比较、分配；std::string 比较是 O(n) 逐字节 |
| **组件类型名比较** | `if (typeName == "MeshRenderer")`              | 运行时字符串比较昂贵；类型名往往很长，缓存不友好                  |

### 核心痛点

- **堆分配不可控**：短字符串（如 `"x"`、`"id"`）也走堆，64 字节以下的分配占引擎 30% 以上
- **比较成本高**：`std::strcmp` 或逐字节比较，在分支预测失败时尤其慢
- **格式化类型不安全**：`printf` 的 `%d` vs `%lld` 在不同平台行为迥异，传错类型直接 UB
- **跨平台差异**：`snprintf` 返回值、本地化行为、宽字符转换，各编译器各唱各的调

## What：最小字符串系统长什么样？

一个可用的引擎字符串系统由三块积木构成：内联存储的短字符串、编译期/运行期哈希标识符、零堆分配的格式化器。

### 1. 小型字符串优化（SSO）String 类

```cpp
#include <cstring>
#include <algorithm>

class SmallString {
    static constexpr size_t SSO_CAPACITY = 15;
    
    union {
        struct {
            char*  data;
            size_t size;
            size_t capacity;
        } heap;
        struct {
            char   data[SSO_CAPACITY + 1];
            uint8_t size;  // 最高位标记是否内联
        } sso;
    };
    
    bool isInline() const { return (sso.size & 0x80) != 0; }
    void setInlineSize(uint8_t sz) { sso.size = sz | 0x80; }
    uint8_t inlineSize() const { return sso.size & 0x7F; }
    
public:
    SmallString() { sso.data[0] = '\0'; setInlineSize(0); }
    
    SmallString(const char* str) {
        size_t len = strlen(str);
        if (len <= SSO_CAPACITY) {
            memcpy(sso.data, str, len + 1);
            setInlineSize(static_cast<uint8_t>(len));
        } else {
            heap.capacity = len * 2;
            heap.size = len;
            heap.data = new char[heap.capacity + 1];
            memcpy(heap.data, str, len + 1);
            sso.size = 0; // 标记为堆模式
        }
    }
    
    ~SmallString() { if (!isInline() && heap.data) delete[] heap.data; }
    
    const char* c_str() const {
        return isInline() ? sso.data : heap.data;
    }
    size_t length() const {
        return isInline() ? inlineSize() : heap.size;
    }
};
```

**这段代码已经解决了什么、还缺什么**：
- ✅ 解决了 ≤15 字符的零堆分配存储，日志标签、资源扩展名、属性键直接内联
- ❌ 还缺：移动语义、写时拷贝防护、resize/append 接口、constexpr 构造、对齐优化（当前 union 以最大成员对齐，可能浪费）

### 2. StringId / HashedString

```cpp
#include <cstdint>

// FNV-1a 64bit
constexpr uint64_t hashStringFnv1a(const char* str, uint64_t hash = 0xcbf29ce484222325ULL) {
    return *str == '\0' ? hash : hashStringFnv1a(str + 1, (hash ^ *str) * 0x100000001b3ULL);
}

class StringId {
    uint64_t m_hash;
public:
    constexpr StringId() : m_hash(0) {}
    constexpr StringId(const char* str) : m_hash(hashStringFnv1a(str)) {}
    
    constexpr uint64_t value() const { return m_hash; }
    constexpr bool operator==(StringId other) const { return m_hash == other.m_hash; }
    constexpr bool operator!=(StringId other) const { return m_hash != other.m_hash; }
};

// 编译期生成
constexpr StringId operator"" _sid(const char* str, size_t) {
    return StringId(str);
}

// 示例用法
// auto id = "hero_diffuse"_sid;
```

**可选：Debug 反查数据库**

```cpp
#include <unordered_map>

#ifdef DEBUG_BUILD
class StringIdDebugDB {
    static std::unordered_map<uint64_t, const char*>& db() {
        static std::unordered_map<uint64_t, const char*> instance;
        return instance;
    }
public:
    static void registerString(uint64_t hash, const char* str) { db()[hash] = str; }
    static const char* lookup(uint64_t hash) {
        auto it = db().find(hash);
        return it != db().end() ? it->second : "<unknown>";
    }
};
#define SID(str) (StringIdDebugDB::registerString(hashStringFnv1a(str), str), StringId(str))
#else
#define SID(str) StringId(str)
#endif
```

**这段代码已经解决了什么、还缺什么**：
- ✅ 解决了 O(1) 字符串比较、编译期哈希、资源表键类型、switch-case 分支支持
- ❌ 还缺：哈希冲突处理（工业级常用双重校验或 intern 池）、运行期字符串的自动注册宏的线程安全、序列化时的可读性回退

### 3. 零分配字符串格式化器

```cpp
#include <cstdarg>
#include <cstdio>
#include <type_traits>

// 抽象格式化接口（类型擦除入口）
class IArgFormatter {
public:
    virtual int format(char* buf, size_t bufSize) const = 0;
    virtual ~IArgFormatter() = default;
};

template<typename T>
class ArgFormatter : public IArgFormatter {
    T m_value;
public:
    explicit ArgFormatter(T v) : m_value(v) {}
    int format(char* buf, size_t bufSize) const override {
        if constexpr (std::is_integral_v<T>) {
            return snprintf(buf, bufSize, "%lld", static_cast<long long>(m_value));
        } else if constexpr (std::is_floating_point_v<T>) {
            return snprintf(buf, bufSize, "%f", static_cast<double>(m_value));
        } else if constexpr (std::is_same_v<T, const char*>) {
            return snprintf(buf, bufSize, "%s", m_value);
        } else {
            return snprintf(buf, bufSize, "<??>");
        }
    }
};

// 构造同构指针数组（类型擦除）
template<typename... Args>
void buildFormatterArray(IArgFormatter** out, Args&&... args) {
    size_t i = 0;
    (..., (out[i++] = new (alloca(sizeof(ArgFormatter<std::decay_t<Args>))))
        ArgFormatter<std::decay_t<Args>>(std::forward<Args>(args))));
}

// 极简 zero-allocation 格式化（仅支持 {0} 占位符）
template<typename... Args>
int formatString(char* outBuf, size_t outSize, const char* fmt, Args&&... args) {
    constexpr size_t MAX_ARGS = 8;
    IArgFormatter* formatters[MAX_ARGS] = {};
    static_assert(sizeof...(Args) <= MAX_ARGS, "Too many args");
    
    buildFormatterArray(formatters, std::forward<Args>(args)...);
    
    size_t outPos = 0;
    for (const char* p = fmt; *p && outPos < outSize - 1; ++p) {
        if (*p == '{' && *(p+1) >= '0' && *(p+1) <= '9' && *(p+2) == '}') {
            int idx = *(p+1) - '0';
            if (idx < sizeof...(Args)) {
                char tmp[256];
                int n = formatters[idx]->format(tmp, sizeof(tmp));
                size_t toCopy = static_cast<size_t>(n) < outSize - outPos - 1
                    ? static_cast<size_t>(n) : outSize - outPos - 1;
                memcpy(outBuf + outPos, tmp, toCopy);
                outPos += toCopy;
            } else {
                // 越界防御：保留占位符原文
                outBuf[outPos++] = *p;
                continue;
            }
            p += 2;
        } else {
            outBuf[outPos++] = *p;
        }
    }
    outBuf[outPos] = '\0';
    return static_cast<int>(outPos);
}
```

**这段代码已经解决了什么、还缺什么**：
- ✅ 解决了热点路径零堆分配、类型安全（模板推导）、跨平台统一输出、索引越界不崩溃
- ❌ 还缺：对齐/宽度/精度语法 `{0:8.2f}`、自定义类型的扩展注册机制、wchar_t/UTF-8 支持、返回值长度预计算（当前可能截断）

## How：真实引擎的字符串系统是如何一步一步复杂起来的？

### 阶段 1：直接用 std::string → 能用但有坑

刚开始写引擎原型时，开发者往往直接用 `std::string` 传资源名、组件名、日志内容。项目小、帧率低，一切正常。

但随着功能增加，坑开始显现：
- **小字符串灾难**：`std::string` 通常有 32 字节左右的对象头（指针+长度+容量），短字符串仍然要堆分配，导致 cache line 利用率极低
- **隐式拷贝**：函数参数按值传递时，深层拷贝触发二次分配
- **比较爆炸**：组件查找、事件订阅键匹配时，逐字节比较成千上万个字符

### 阶段 2：SSO + StringId + 自定义格式化 → 好用

当性能分析显示字符串操作进入热点 Top10 时，引擎进入第二阶段：

1. **引入 SSO String**：80% 的字符串长度 < 16，内联后彻底消除小分配；剩余 20% 长路径（如资源全路径）走堆，但次数锐减
2. **引入 StringId**：所有资源表、组件类型字典、配置键改用 `uint64_t` 哈希；比较从 O(n) 降到 O(1)，且可以在编译期生成常量
3. **引入自定义 Formatter**：替代 `std::ostringstream` 和 `printf`；日志路径零堆分配，类型推导避免 `%d`/`%lld` 混淆

### 阶段 3：国际化 + 字符串池 + 反射集成 → 工业级

大型项目还需要：
- **本地化表（Localization）**：`StringId` 指向翻译键，运行时根据语言环境反查对应语言的 `SmallString` 或 UTF-8 文本
- **字符串池（String Interning）**：相同内容的字符串只存一份，减少内存占用；运行期构造的 `StringId` 与 intern 池联动，解决哈希冲突风险
- **反射集成**：类型系统的 `TypeName` 不再是裸字符串，而是反射注册时自动生成的 `StringId`；序列化器用 `StringId` 作为键写入 JSON/Binary，加载时再反查

## AI 友好设计红线

在将字符串系统集成到引擎架构时，以下规则直接影响 AI 辅助开发的安全性和可预测性：

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ❌ String 类不太适合直接作为 ECS 组件 | 它可能含堆指针，导致组件布局不紧凑、序列化/拷贝复杂。建议 ECS 中存储 `StringId`，字符串内容放入资源表或 intern 池 |
| **自描述** | ✅ AI 可通过反查表获得可读名称 | 在 Debug/工具构建中，维护 `StringId → const char*` 的全局映射；Release 构建可剥离以节省内存 |
| **确定性** | ✅ 字符串比较和哈希是纯算法 | FNV-1a 在给定输入下输出确定，不依赖运行时状态，AI 修改资源名后可精确预测下游影响 |
| **工具边界** | 应操作 `StringId` | AI 修改资源名或组件名时，优先操作 `StringId`（或注册新的 `StringId`），而非直接改动原始字符串。原始字符串仅作为 Debug 反查或初始注册时的输入 |
