# 数学基础

## Why：为什么游戏引擎需要自己的数学库？

想象一下，你正在用标准库的零散工具拼凑一个 3D 引擎：

- **Transform 更新**：每个游戏对象都有位置、旋转、缩放，每帧都需要计算世界变换矩阵。没有统一的 `Mat4 × Mat4` 和 `Quat → Mat4`，代码里将充斥着手写 16 重循环和极易出错的索引计算。
- **渲染 MVP 矩阵**：摄像机视图矩阵和投影矩阵必须精确组装，然后传递给 GPU。如果矩阵是行主序还是列主序搞混了，或者 `perspective` 的 near/far 平面符号约定不一致，画面会直接撕裂或完全黑屏。
- **物理积分**：刚体的速度和位置更新需要大量的 `Vec3` 点乘、叉乘。如果向量运算涉及频繁的临时对象分配或隐式类型转换，帧率会断崖式下跌。

那么，直接用 **GLM** 不行吗？对于小型项目或个人学习，GLM 确实足够。但当我们迈向工业级引擎时，它会暴露以下软肋：

| 问题            | 说明                                                                                           |
| ------------- | -------------------------------------------------------------------------------------------- |
| **对齐控制不可控**   | SIMD 类型（如 `__m128`）通常要求 16 字节对齐，GPU 常量缓冲区可能需要 256 字节对齐。GLM 的对齐宏行为在不同编译器下并不稳定。                |
| **half 类型缺失** | 在 C++23 之前，标准库没有 `std::float16_t`。现代 GPU 纹理和顶点压缩大量依赖半精度浮点，引擎必须自己管理 `float16` ↔ `float32` 转换。 |
| **SIMD 不友好**  | GLM 默认是标量实现（AoS），没有针对 SSE/AVX/NEON 做结构化优化。对于需要批量变换的粒子系统或骨骼动画，性能瓶颈明显。                         |
| **浮点确定性难保证**  | 网络游戏或回放系统要求相同输入产生完全相同的结果。不同编译器对浮点优化的策略不同（如 FMA 指令的取舍），外部库很难提供严格的跨平台确定性保障。                    |

因此，一个自研的数学库不是炫技，而是**可控性、性能、确定性**三位一体的基础设施。

## What：最小数学库长什么样？

下面给出三块**能跑的最小实现**，总计不超过 150 行。它们不是工业级的完整答案，但已经足以支撑起最简单的 Transform 和渲染管线。

### 1. 对齐工具 — 内存安全的底层基石

无论是 SIMD 加载、GPU 资源映射，还是内存池分配，都需要精确的对齐计算。核心思想是位运算向上/向下取整：

$$\text{AlignUp}(value, alignment) = (value + mask) \ \& \ \sim mask, \quad mask = alignment - 1$$

```cpp
#include <cstdint>
#include <cstddef>

// 根据指针宽度选择足够大的整数类型，防止 64 位地址截断
using AlignInt = std::conditional_t<sizeof(void*) == 8, std::uint64_t, std::uint32_t>;

inline constexpr std::size_t AlignUp(std::size_t value, std::size_t alignment)
{
    std::size_t mask = alignment - 1;
    return (value + mask) & ~mask;
}

inline constexpr std::size_t AlignDown(std::size_t value, std::size_t alignment)
{
    std::size_t mask = alignment - 1;
    return value & ~mask;
}

inline constexpr bool IsAligned(std::size_t value, std::size_t alignment)
{
    return (value & (alignment - 1)) == 0;
}

// 常用 SIMD/GPU 对齐常量
inline constexpr std::size_t kAlignSIMD16 = 16;
inline constexpr std::size_t kAlignSIMD64 = 64;
inline constexpr std::size_t kAlignGPUBuffer = 256;
```

> **这段代码已经解决了什么**：所有内存分配器和资源上传器都可以依赖统一的对齐接口，避免手动写 `(x + 15) / 16 * 16` 这类易错公式。编译器通常将其优化为 1~2 条位运算指令。
>
> **还缺什么**：没有将 `AlignUp` 重载到指针类型（`T* AlignUp(T* ptr, size_t alignment)`），也没有在分配器里自动填充对齐边界上的哨兵字节以检测越界。

### 2. Vec3 + Mat4 骨架 — 足够用于 Transform 和 MVP

```cpp
#include <cmath>

struct Vec3
{
    float x, y, z;

    Vec3 operator+(const Vec3& o) const { return {x + o.x, y + o.y, z + o.z}; }
    Vec3 operator-(const Vec3& o) const { return {x - o.x, y - o.y, z - o.z}; }
    Vec3 operator*(float s)      const { return {x * s, y * s, z * s}; }

    float Dot(const Vec3& o)   const { return x * o.x + y * o.y + z * o.z; }
    Vec3  Cross(const Vec3& o) const
    {
        return {
            y * o.z - z * o.y,
            z * o.x - x * o.z,
            x * o.y - y * o.x
        };
    }

    float LengthSq() const { return Dot(*this); }
    float Length()   const { return std::sqrt(LengthSq()); }
    Vec3  Normalized() const
    {
        float len = Length();
        return len > 0.0f ? (*this) * (1.0f / len) : Vec3{0, 0, 0};
    }
};

// 列主序 4x4 矩阵（OpenGL / Vulkan / D3D 常量缓冲区通用约定）
struct Mat4
{
    float m[16]; // 索引 = 行 + 列 * 4

    static Mat4 Identity()
    {
        Mat4 out{};
        out.m[0] = out.m[5] = out.m[10] = out.m[15] = 1.0f;
        return out;
    }

    Mat4 operator*(const Mat4& o) const
    {
        Mat4 out{};
        for (int i = 0; i < 4; ++i)
            for (int j = 0; j < 4; ++j)
                for (int k = 0; k < 4; ++k)
                    out.m[i + j * 4] += m[i + k * 4] * o.m[k + j * 4];
        return out;
    }

    Vec3 TransformPoint(const Vec3& v) const
    {
        float x = m[0] * v.x + m[4] * v.y + m[8]  * v.z + m[12];
        float y = m[1] * v.x + m[5] * v.y + m[9]  * v.z + m[13];
        float z = m[2] * v.x + m[6] * v.y + m[10] * v.z + m[14];
        float w = m[3] * v.x + m[7] * v.y + m[11] * v.z + m[15];
        return w != 0.0f ? Vec3{x / w, y / w, z / w} : Vec3{x, y, z};
    }
};
```

> **这段代码已经解决了什么**：可以计算世界变换矩阵（TRS 组合）、组装 MVP 矩阵、将本地坐标变换到裁剪空间。它是完全自包含的，不依赖任何第三方库。
>
> **还缺什么**：没有四元数（Quat），所以旋转插值要用欧拉角或矩阵，会产生万向节锁；没有逆矩阵和转置，阴影映射或法线矩阵无法计算；没有 `operator*` 的返回值优化（RVO）之外的显式 SIMD 加速。

### 3. 确定性随机数生成器 — xorshift32

对于回放系统、程序化生成、网络同步，随机数必须是**完全可复现的**。xorshift32 是一个状态仅 32 位、周期 $2^{32}-1$、无除法浮点运算的轻量级方案。

```cpp
#include <cstdint>

class RandomXORShift32
{
public:
    explicit RandomXORShift32(std::uint32_t seed)
    {
        // 避免 seed = 0 导致全零死锁
        m_state = seed == 0 ? 0xBAD5EEDu : seed;
    }

    std::uint32_t NextUint32()
    {
        m_state ^= m_state << 13;
        m_state ^= m_state >> 17;
        m_state ^= m_state << 5;
        return m_state;
    }

    // 返回 [0, 1) 范围内的 float，保证相同种子跨平台结果一致
    float NextFloat01()
    {
        // 取高 23 位作为尾数，配合隐式前导 1.0，恰好覆盖 [1.0, 2.0)
        constexpr std::uint32_t MAGIC = 0x3F800000u; // 1.0f 的 IEEE 754 位模式
        std::uint32_t bits = MAGIC | (NextUint32() >> 9);
        float f;
        static_assert(sizeof(f) == sizeof(bits));
        std::memcpy(&f, &bits, sizeof(f));
        return f - 1.0f;
    }

    void Reset(std::uint32_t seed) { m_state = seed == 0 ? 0xBAD5EEDu : seed; }

private:
    std::uint32_t m_state;
};
```

> **这段代码已经解决了什么**：给定相同种子，在任何平台、任何编译器下都会吐出完全相同的整数序列和浮点序列。没有调用 `std::rand()` 这类依赖 CRT 实现的函数。
>
> **还缺什么**：周期只有 $2^{32}-1$，对于需要大量随机数的大型开放世界不够用（应升级到 xorshift128+ 或 PCG）；没有提供正态分布或泊松分布；`NextFloat01` 的精度只有 $2^{-23}$，对高精度需求（如次像素抖动）可能不足。

## How：真实引擎的数学库是如何一步一步复杂起来的？

### 阶段 1：标量实现 → 能用

这就是我们上面给出的代码。目标是**正确性优先**：矩阵乘法顺序不能错、列主序和行主序约定要一致、随机数种子要可控。这个阶段的数学库通常只有几百行，但已经能支撑起一个完整的小 demo。

### 阶段 2：SIMD 加速 + 内存对齐 → 好用

当场景中有上千个对象、上万个粒子时，标量 `Vec3` 会成为瓶颈。引擎开始引入 SIMD：

- **SSE/AVX/NEON**：将 `Vec4`（而不是 `Vec3`）作为基础 SIMD 类型，因为 128 位寄存器刚好装 4 个 float。`Vec3` 运算时把第四分量置 0 或 1，用一条指令完成 4 个分量的加减乘除。
- **AoS vs SoA**：
  - **AoS（Array of Structs）**：`Vec4 positions[1024]`，代码直观，适合单个个体的随机访问。
  - **SoA（Struct of Arrays）**：将 x、y、z、w 分别存成 4 个连续数组，SIMD 可以一次性加载 4 个对象的 x 分量到寄存器，批量计算效率极高。
  - **ECS 天然适合 SoA**：当 [[组件系统架构]] 把 `Position`、`Velocity` 平铺成连续数组时，物理系统可以一次性读取 4 个 `Velocity` 的 x 分量，一条指令完成 4 个位置更新。

```cpp
// SoA 批量更新的伪代码
void UpdatePositionsSoA(float* px, float* py, float* pz,
                        const float* vx, const float* vy, const float* vz,
                        float dt, std::size_t count)
{
    for (std::size_t i = 0; i < count; i += 4)
    {
        // SSE 一次性加载 4 个 float
        __m128 dtx = _mm_set1_ps(dt);
        __m128 px4 = _mm_loadu_ps(px + i);
        __m128 vx4 = _mm_loadu_ps(vx + i);
        _mm_storeu_ps(px + i, _mm_add_ps(px4, _mm_mul_ps(vx4, dtx)));
        // ... 对 py, pz 同理
    }
}
```

### 阶段 3：平台抽象 + half 类型 + 确定性保障 → 工业级

走到这一步，引擎开始面临跨平台发布（PC、主机、移动端）和严苛的网络同步要求。

#### 平台抽象 SIMD

为了不把 `__m128`、`_mm_add_ps` 这类内联指令硬编码到渲染和物理系统的每一处，引擎会封装一层平台无关的 SIMD 类型：

```cpp
namespace simd {
    using Vec4F = __m128;        // x86
    // using Vec4F = float32x4_t; // ARM NEON
    inline Vec4F Add(Vec4F a, Vec4F b) { return _mm_add_ps(a, b); }
}
```

这样，渲染器写一次 `simd::Add`，就能在 x86 和 ARM 上同时工作。

#### half（float16）类型

现代 GPU 的顶点格式和纹理压缩广泛使用 IEEE 754 half-precision（1 位符号 + 5 位指数 + 10 位尾数）。在 C++23 之前，标准库没有原生支持，因此引擎通常引入一个轻量级的 16 位包装类。核心转换策略是查表法：

- `float → half`：借助 512 条目的 base_table 和 shift_table 快速映射指数和尾数。
- `half → float`：借助 2048 条目的 mantissa_table 和 64 条目的 exponent_table 恢复完整精度。
- 全程使用 `std::memcpy` 进行位模式读写，避免 strict aliasing 违规。

这能显著减少顶点缓冲区带宽和显存占用，对移动端尤其重要。

#### 确定性保障

- **随机数**：统一使用自研的 xorshift / PCG，彻底屏蔽 `std::random_device` 和操作系统熵源的差异。
- **浮点规则**：
  - 关闭编译器的 `-ffast-math` 等激进优化，因为它们会重排运算顺序。
  - 统一使用 `std::fma` 或显式避免 FMA，防止 x86 和 ARM 在是否融合乘加上的行为差异。
  - 对物理积分等关键路径，采用固定点数学（Fixed Point）或软浮点库（如 Verificatum 的浮点模拟）。

### AI 友好设计红线检查

在将数学库与 [[组件系统架构]] 和 AI 工具链结合时，需要验证以下设计约束：

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **状态平铺** | ✅ 是 | `Vec3`、`Mat4` 都是纯 POD（Plain Old Data），没有虚函数、没有指针、没有动态分配。它们可以作为 ECS 组件被完美平铺到连续内存中，SIMD 批量处理效率极高。 |
| **自描述** | ⚠️ 依赖 [[反射与序列化]] | AI 本身无法从 C++ 头文件中自动知道 `Transform` 组件里有 `position: Vec3`。必须依赖 TypeRegistry 将字段名、类型、偏移量注册为结构化元数据。 |
| **确定性** | ⚠️ 需显式策略 | 随机数种子可控（`RandomXORShift32` 已满足）。但浮点运算的跨平台一致性需要额外的编译器标志和数学规则约束，不能自动保证。 |
| **工具边界** | ✅ 是 | AI 修改一个 `position` 组件时，输入输出都是结构化的三元组 `(x, y, z)` 或十六元组矩阵。没有隐式的全局状态，没有副作用函数。 |
