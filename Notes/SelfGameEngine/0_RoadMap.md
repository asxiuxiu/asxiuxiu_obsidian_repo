# SelfGameEngine 构建手册

> 本手册是一本**可执行的引擎构建指南**。每一步都必须是**可落地、可看见、可验证**的。我们不追求先读完所有理论再动手，而是**每完成一个阶段，屏幕上就要多出一些东西**。
>
> **核心设计理念**：为 **AI 协作开发** 而设计的 ECS 引擎。每篇笔记默认回扣两个问题：**AI 如何观察它？** **AI 如何操作它？**
>
> **构建哲学**：单机优先，网络预留。先让一个人在屏幕上看见东西，再考虑两个人联网。

**每个阶段的验收标准是"能看见的"**：
- **阶段 1**：一个能打开的窗口，有实时滚动的 ImGui 日志面板和 FPS 计数器
- **阶段 2**：ImGui 左侧面板列出 ECS 实体和组件，选中后 Inspector 显示字段值
- **阶段 3**：用新实现的基础库替换阶段 1/2 的临时代码，日志输出"替换完成，零回归"
- **阶段 4**：窗口里有 3~5 个彩色立方体在旋转，Inspector 实时修改 Transform 立即生效，场景可保存加载
- **阶段 5**：屏幕中央有带纹理的旋转立方体，2D UI 文字/按钮可点击，后处理 Bloom 生效
- **阶段 6**：自研 UI 框架替代 ImGui，文本清晰、布局自动、控件可交互、面板可拖拽、编辑器核心面板重写完成
- **阶段 7**：窗口里有能行走/跳跃/播放动画的角色，能碰撞和寻路，输入映射可热切换
- **阶段 8**：外部 AI 通过 MCP 发送命令修改场景，Inspector 实时同步，支持 Undo/Redo
- **阶段 9**：CI 自动编译打包，性能面板显示各系统耗时，[可选] 多人客户端联机

## 阶段 1：Hello Window

**里程碑**：`main()` 启动后弹出一个可交互窗口，窗口内有 ImGui 绘制的日志面板和 FPS 显示。从此不再依赖 `printf` 硬编码验证。

> **为什么 ImGui 在这个阶段？** 因为阶段 2 的 ECS Inspector 需要它，而 ImGui 需要一个图形后端才能渲染。这些前置依赖不能隐式跳过，必须显式接入。

**验收标准（能看见的）**：
- [x] 运行后弹出一个窗口（GLFW / SDL / Win32）
- [x] 窗口内有一个 ImGui 面板，显示实时滚动的日志（至少 Debug / Info / Error 三级过滤）
- [x] 日志能输出到控制台、文件、ImGui 面板三端
- [x] 窗口标题栏或面板显示当前 FPS 和帧时间
- [x] 按 ESC 键可以干净退出，无内存泄漏

### 1.1 窗口与输入系统

> **下一步**：窗口有了，但屏幕是黑的。要显示 ImGui 面板，需要一个最简图形后端。

### 1.2 最简图形后端

- [x] [[最简图形后端]] — OpenGL / bgfx 初始化、交换链、清屏、为 ImGui 准备渲染上下文

> **前置说明**：我们不在这里深入讲着色器、管线状态、资源屏障——那些属于阶段 5。这里只需要"能把 ImGui 画到屏幕上"的最小图形能力。
>
> **下一步**：图形后端就位后，接入 ImGui，获得调试面板能力。

### 1.3 ImGui 接入与基础面板

- [x] [[ImGui接入与基础面板]] — ImGui 图形后端绑定、字体加载、基础窗口管理、Docking 初探

> **下一步**：日志系统可以将输出目标扩展到 ImGui 面板了。

### 1.4 可视化日志系统

> **本阶段增量**：引擎拥有了第一个"可视化调试工具"——ImGui 日志面板。从此所有系统验证都通过日志或面板反馈，不再硬编码 `printf`。
> **下一步**：在 ImGui 面板里直接观察 ECS 世界。

## 阶段 2：最小 ECS 骨架

**里程碑**：在阶段 1 的 ImGui 窗口里，通过面板直接观察 ECS 世界状态。有一个 System 在 Tick 中批量修改组件，且变化能实时反映在面板上。

> **为什么 ECS 这么早？** 因为后续所有系统（数学、渲染、物理）都是围绕 ECS 设计的。越早建立 ECS 骨架，后续模块就有统一的验证舞台。

**验收标准（能看见的）**：
- [x] ImGui 左侧面板显示"Entity 列表"（ID + 组件标签）
- [x] 选中 Entity 后，右侧面板显示该 Entity 的所有组件字段数值
- [x] 有一个 System 在 Tick 中修改某个组件数值（如位置递增）
- [x] 数值变化实时反映在 Inspector 面板上
- [x] 能按按钮"添加 Entity"、"删除选中 Entity"

### 2.1 最小 ECS 数据层

- [x] [[最小ECS数据层]] — Entity 句柄、ComponentArray（初始用 `std::vector` 临时实现）、映射表、World 生命周期

> **临时性声明**：本章节的 `ComponentArray` 使用 `std::vector` 做临时存储，阶段 4.1 会回探替换为 Sparse Set / Archetype。
> **下一步**：没有 Inspector 的 ECS 是黑盒。

### 2.2 极简 Inspector 与 ECS 可视化

- [x] [[极简Inspector与ECS可视化]] — Entity 列表、组件字段遍历（初始用硬编码反射）、读写反馈、运行时增删组件

> **临时性声明**：本章节的组件字段显示使用硬编码枚举，阶段 4.4 会回探替换为自动反射系统。
> **本阶段增量**：引擎拥有了可交互的 ECS 世界，所有后续系统（数学、渲染、物理）都可以在这个 Inspector 里被可视化验证。
> **下一步**：ECS 骨架搭好了，但底层用的是 `std::vector`、硬编码反射、平台相关类型。现在需要构建"正式"的基础工具层，并**回探替换**之前的临时实现。

## 阶段 3：基础工具层（回探与深化）

**里程碑**：数学库、字符串、容器/分配器、文件IO、线程池全部就位，且**已回探替换**阶段 1、2 中的临时实现。所有验证通过日志或 Inspector 可视化完成。

> **阶段性质说明**：这不是"第一次写"这些模块，而是**回探替换**。阶段 1 的日志用了 `std::string`，阶段 2 的 ComponentArray 用了 `std::vector`，阶段 2 的 Inspector 用了硬编码字段枚举。现在我们要写出"正式的"引擎基础实现，并用它们替换之前的临时代码。
>
> **回探策略**：每完成一个基础模块，在笔记末尾附带一个"回探替换清单"，明确列出阶段 1/2 中哪些临时代码需要被替换、替换后的验证方法。

**验收标准（能看见的）**：
- [ ] 在 Inspector 或日志中验证 `Vec3 * Mat4` 的正确性
- [ ] 加载一个 JSON/文本配置文件，日志输出解析结果
- [ ] 线程池执行 1000 个任务，日志输出耗时统计
- [ ] 用 FrameArena 分配 1MB 临时内存，日志验证无泄漏
- [ ] **回探验证**：ComponentArray 从 `std::vector` 替换为引擎自定义容器后，阶段 2 的 Inspector 仍能正常显示实体列表
- [ ] **回探验证**：日志系统从 `std::string` 替换为引擎字符串基础设施后（`StringId` 分类名 + 栈格式化器生产 + 环形缓冲区存储），阶段 1 的 ImGui 日志面板仍能正常滚动

### 3.1 引擎基础类型与平台抽象

- [x] [[引擎基础类型与平台抽象]] — 固定宽度整数、导出符号、断言与调试钩子、平台检测宏

> **回探替换点**：阶段 1 代码中所有裸 `int` / `uint32_t` → 替换为引擎固定宽度类型；所有 `assert()` → 替换为引擎断言宏。
> **下一步**：在类型基础之上构建数学运算。

### 3.2 数学基础

- [x] [[数学基础]] — Vec3 / Mat4 / Quat、四元数插值、AABB 射线检测、确定性随机

> **回探替换点**：阶段 2 的 Transform 组件中的 `float` 数组 → 替换为 `Vec3` / `Quat` / `Mat4` 结构。
> **下一步**：ECS 组件名、资源路径都需要稳定的字符串。

### 3.3 字符串系统

- [x] [[字符串系统]] — **总览与唯一权威定义**：三种字符串角色（StringId/StringView/String）的精确契约、默认推荐路径、ECS 映射、OOP→ECS 重构映射、回探替换清单
- [x] [[可变字符串与 SSO]] — 可变字符串容器的**两种工业级优化路径**（union SSO vs 分配器参数化）、三引擎对照（UE FString 纯堆/chaos 薄封装/Bevy 架构规避）、StringView 与 COW、自研推荐
- [x] [[字符串标识与 InternPool]] — StringId 设计、字符串哈希算法选择（FNV-1a/xxHash/CityHash）、UE FNamePool 工业级架构（分块/无锁/TLS）、chaos StringID 编译期哈希、Bevy TypeId/Handle 规避策略
- [x] [[字符串格式化]] — ADL 自由函数栈格式化器、fmtlib 三层架构（编译期解析/类型擦除/栈缓冲）、C++20 std::format、三引擎格式化对照
- [x] [[日志消息与字符串系统的边界]] — 日志消息虽经格式化器生产，但存储架构（环形缓冲区/延迟格式化/结构化输出）与标识符完全不同

> **回探替换点**：阶段 1 日志系统中的 `std::string` 日志内容 → 日志级别/分类名替换为 `StringId`，日志消息存储替换为**栈格式化器 + 环形缓冲区**（不走 `String`）；阶段 2 的组件名硬编码字符串 → 替换为 StringId。
> **下一步**：ECS 需要专用的内存策略。

### 3.4 内存分配器与容器系统

- [x] [[内存分配器]] — FrameArena、ObjectPool、BinnedAllocator、统一分配器接口 `IAllocator`、调试装饰器 `DebugAllocatorWrapper`
- [x] [[容器系统]] — Array、HashMap、SparseSet、SoA、RingBuffer、与 `std::vector` 的迁移边界

> **回探替换点**：阶段 2 的 `ComponentArray`（`std::vector` 临时实现）→ 替换为引擎自定义的组件存储（Pool / Sparse Set 初版）。这是本阶段**最关键的回探替换**。
> **下一步**：引擎需要加载外部资源。

### 3.5 文件 IO 与虚拟文件系统

- [x] [[文件IO与虚拟文件系统]] — Path 抽象、VFS 多后端挂载、Pak 归档格式、热重载、依赖图与拓扑排序、异步加载基础、AI 可观测性设计

> **回探替换点**：阶段 1 的日志文件输出（裸 `fopen`）→ 替换为 VFS 路径写入。
> **下一步**：异步加载和并行计算的基础。

### 3.6 线程池与任务系统

- [x] [[线程池与任务系统]] — ThreadPool、并行 For、任务依赖、主线程回调投递

> **回探替换点**：阶段 1 的日志后台刷盘线程（裸 `std::thread`）→ 替换为线程池任务；阶段 2 的 System Tick（单线程）→ 预留并行 System 的调度接口。
> **下一步**：补齐基础层的最后一块拼图。

### 3.7 网络抽象与硬件检测（单机预留接口）

- [x] [[网络抽象与硬件检测]] — Socket 抽象（仅封装，不深究同步）、CPU / SIMD 查询、动态库加载

> **单机路径说明**：本章节的 Socket 抽象只做最底层封装，为阶段 8 的网络扩展预留接口。单机开发阶段不实现网络同步逻辑。
> **回探替换点**：阶段 1 的平台检测代码 → 替换为统一的 CPU/SIMD 查询接口。
> **下一步**：执行系统性的回探替换。

### 3.8 基础层回探与替换指南

- [ ] [[基础层回探与替换指南]] — 回探清单、替换顺序、回归验证方法、常见陷阱

> **本阶段增量**：引擎的底层全部替换为自研实现。`std::vector` → 自定义容器，`std::string` → 引擎字符串，硬编码反射 → 预留反射接口，裸线程 → 线程池。阶段 1 和 2 的所有功能在替换后**零回归**。
> **下一步**：让 ECS 世界真正运转起来——用正式的底层实现驱动上层。

## 阶段 4：核心运行时闭环

**里程碑**：ECS 查询/存储、场景图、反射、Tick 调度、事件总线、Prefab、Plugin 全部连通。窗口里有会动的几何体，Inspector 能实时修改组件数值并看到变化。

**验收标准（能看见的）**：
- [ ] 窗口里有 3~5 个彩色立方体在旋转或平移
- [ ] 在 Inspector 中修改某个立方体的 `Transform.position`，它立刻移动
- [ ] 立方体之间有父子层级关系，修改父节点时子节点跟随
- [ ] 按空格键可以触发一个事件，让所有立方体变颜色
- [ ] 能保存当前场景为一个文件，重启程序后能加载恢复
- [ ] 能通过 Plugin 机制加载一个新 System，无需修改核心代码

### 4.1 组件系统架构

- [x] [[组件系统架构]] — Sparse Set、Query、Archetype、Chunk-based 存储与并行调度

> **与阶段 2 的关系**：这是阶段 2 `ComponentArray` 的最终形态。阶段 2 用 `std::vector` 验证了 ECS 概念，现在用 Sparse Set / Archetype 替换以获得生产级性能。
> **下一步**：让实体在三维空间中建立关系。

### 4.2 场景图与变换

### 4.3 实体生命周期与注册表

### 4.4 反射系统

## 阶段 6：自研 UI 框架与编辑器基础

**里程碑**：拥有一套 Retained-Mode UI 框架，能替代 ImGui 渲染出清晰、可定制、支持高级布局的编辑器界面。编辑器核心面板（日志、Inspector、场景视图）全部用自研 UI 重写，达到 UE Slate / Unity UI Toolkit 级别的丝滑体验。

> **为什么是阶段 6？** 因为阶段 5 的 2D 渲染管线已经为 UI 提供了底层绘制能力（SDF 字体、批次合批、RenderTarget），而阶段 7 之后的动画、Gameplay、工具链都需要一个高质量的编辑器来承载。ImGui 的即时模式架构在调试面板阶段够用，但面对复杂的资产浏览器、节点图、多面板 Docking 时，它的样式简陋、字体模糊、布局僵硬问题会成为瓶颈。自研 UI 框架是编辑器体验的决定性分水岭。

**验收标准（能看见的）**：
- [ ] 窗口里有一个自研 UI 渲染的文本标签，使用 SDF 字体，缩放后仍清晰锐利
- [ ] 垂直/水平布局容器能自动排列子元素，支持边距、对齐、填充策略
- [ ] 按钮、输入框、滑动条可交互，有 hover / press / focus 状态视觉反馈
- [ ] 支持样式系统：颜色、圆角、边框、阴影、字体大小可通过主题统一配置和动态切换
- [ ] 支持层级化面板系统：可拖拽分割、标签页切换、浮动窗口（Docking 初版）
- [ ] 编辑器核心面板（日志、Inspector、场景视图）用新 UI 框架重写，不再依赖 ImGui
- [ ] 日志面板支持多级别过滤、搜索高亮、自动滚动；Inspector 支持组件折叠、字段编辑
- [ ] UI 框架与阶段 5 的 2D 渲染管线打通，使用同一套批次合批和渲染后端

### 6.1 UI 渲染核心

- [ ] [[UI渲染核心]] — SDF 字体渲染、矢量图元（矩形/圆角/路径）、9-slice 贴图、批次合批与裁剪、与阶段 5 的 2D 渲染管线对接

> **与阶段 5 的关系**：阶段 5 的 `2D渲染基础与批次合批` + `字体渲染系统` 已经实现了基础 2D 批次和 SDF/MSDF 字体，但那是面向游戏内 UI 的。本章节的渲染核心面向编辑器 UI，需要额外支持：大量动态文本、复杂裁剪、高频状态变化下的稳定帧率。
> **下一步**：能画了，但放哪里？需要布局系统决定每个元素的位置和大小。

- [ ] [[布局系统]] — 约束传递（Constraint / Measure / Arrange）、Flex/Box 布局、边距与对齐、自动尺寸与滚动区域、性能优化（脏标记与增量布局）

> **参考来源**：UE Slate 的布局系统采用 Constraint → Measure → Arrange 三阶段；Web 的 Flexbox 是行业最广泛验证的布局模型。个人项目推荐：核心采用 Flexbox 语义（因为开发者熟悉），底层实现用 Slate 式的约束传递（因为性能可控）。

> **参考来源**：ImGui 的输入处理是全局的，没有焦点和冒泡概念。UE Slate 有完整的 `FReply`、`FPointerEvent` 路由体系。自研 UI 必须引入 retained-mode 的事件路由，否则复杂面板（如 Inspector 里的嵌套控件）无法正确响应输入。

> **AI 友好设计**：样式系统的状态应该是平铺的、可序列化的 JSON/Schema，AI 可以通过 MCP 接口查询和修改任意控件的样式属性，而不需要理解 C++ 的头文件。

### 6.5 基础控件库

- [ ] [[基础控件库]] — 文本标签（TextBlock）、按钮（Button）、输入框（TextBox）、滑动条（Slider）、复选框（CheckBox）、下拉框（ComboBox）、列表视图（ListView）、树形视图（TreeView）、可折叠组（CollapsiblePanel）

> **与 ImGui 的差异**：ImGui 的控件是函数调用（`ImGui::Button()`），而自研 UI 的控件是 retained 对象树中的节点。每个控件有持久状态（如输入框的光标位置、列表的滚动偏移），ImGui 需要每帧重建这些状态，而自研 UI 天然保留它们。
