# 引用、指针与参数传递

> [!info] 一句话概括
> 传值会拷贝，传指针能共享但管理负担重，传引用在语法上像传值、性能上像指针；理解三者的取舍，是设计引擎接口的第一步。

---

## 问题 0：想把对象「拿进」函数处理，最自然的写法是什么？它会在哪里出问题？

初学函数时，我们会直觉地认为：把变量写进参数列表，函数就能拿到它。

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

void append_log(std::string log) {
    log += " [processed]";
    std::cout << "inside:  " << log << '\n';
}

int main() {
    std::string msg = "player moved";
    append_log(msg);
    std::cout << "outside: " << msg << '\n';
    return 0;
}
```

运行后会发现：`outside` 打印的仍然是 `player moved`，函数里的修改没有带出来。这是因为 **传值（pass-by-value）** 时，函数得到的是原对象的一个副本；函数内部改的是副本，原对象安然无恙。

在简单场景下这没问题，甚至正是我们想要的行为。但在引擎代码里，传值会暴露两个致命问题：

1. **拷贝爆炸**：一个 `Transform` 或 `std::vector<Vertex>` 拷贝一次可能涉及深拷贝、内存分配，主线程 16ms 一帧根本经不起折腾。
2. **多态切片**：如果参数类型是基类，传子类对象时只会保留基类部分，虚函数表也丢了，动态派发失效。

所以引擎里几乎不会把「可能需要修改原对象」或「只读访问大对象」的场景设计为传值。

---

## 问题 1：那我用指针把地址传进去，总行了吧？

指针确实解决了传值的拷贝和切片问题。函数拿到的是对象的地址，通过地址可以直接读写原对象；如果参数类型是基类指针，还能指向不同子类对象，实现多态。

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

void append_log(std::string* log) {
    if (log != nullptr) {          // 每次都要检查
        *log += " [processed]";
    }
}

int main() {
    std::string msg = "player moved";
    append_log(&msg);
    std::cout << msg << '\n';
    return 0;
}
```

但指针的便利是有代价的，这个代价在大型代码库里会越滚越大：

- **可能为空**：调用者和被调用者都要考虑 `nullptr` 分支，漏判就会崩溃。
- **所有权模糊**：函数接收指针后，要不要 `delete`？能不能保存下来以后用？仅凭函数签名无法回答。
- **悬垂风险**：指针指向的对象被释放后，指针本身不会自动失效，继续使用就是未定义行为。
- **语法噪音**：每次访问都要写 `*` 或 `->`，链式调用一长就很难读。

指针是一把锋利的刀，能切菜也能切手。C++ 后来引入的「引用」，就是想保留指针「不拷贝、能修改原对象」的优点，同时尽量消除它的管理负担。

---

## 问题 2：引用是怎么做到「像传值一样方便，又像指针一样高效」的？

**引用（reference）** 本质上是一个对象的别名。一旦绑定到某个对象，就不能再改绑到别的对象；它也不能为空，因为定义时必须初始化。语法上，使用引用和使用原对象几乎没有区别。

把刚才的指针版本改成引用版本：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

void append_log(std::string& log) {
    log += " [processed]";   // 不需要 *，不需要判空
}

int main() {
    std::string msg = "player moved";
    append_log(msg);          // 不需要 &
    std::cout << msg << '\n';
    return 0;
}
```

对比两个版本，引用的好处一目了然：

| 维度 | 传值 | 传指针 | 传引用 |
|------|------|--------|--------|
| 是否拷贝原对象 | 是 | 否 | 否 |
| 能否修改原对象 | 不能 | 能 | 能 |
| 调用语法 | 最简单 | 需要 `&` / `*` | 最简单 |
| 是否可能为空 | 否 | 是 | 否 |
| 是否可能重新绑定 | — | 是 | 否 |

表格只是帮你整理记忆。真正关键的是理解：**引用在语义上「就是」那个对象，而不是指向那个对象的间接物**。这意味着：

- 函数签名 `void f(T& x)` 明确告诉调用者：我会修改 `x`。
- 函数签名 `void f(const T& x)` 明确告诉调用者：我只会读取 `x`，而且不会拷贝。
- 调用者不必担心空指针，也不必写取地址符。

## 问题 3：引用这么好，它能完全替代指针吗？

不能。引用解决的是「参数传递」这个特定问题，但指针还有引用无法覆盖的场景：

1. **需要重新绑定**：引用一旦初始化就终身绑定，不能像指针那样先指向 A，再指向 B。
2. **可能为空**：有时候「没有对象」本身就是一种有效状态，比如搜索失败返回空指针。引用不能表达「我没有绑定任何东西」。
3. **动态内存与数组算术**：`new T` 返回的是指针；指针可以做 `++`、`--`、随机访问，引用不行。
4. **多态容器存储**：容器里通常存指针（或智能指针），因为引用不能作为容器元素（数组元素必须可默认构造、可重新赋值）。
5. **可选输出参数**：有些 API 用 `T* out = nullptr` 表示「可选输出」，调用者不想拿结果就传空。

一个简单例子：交换两个变量，引用最自然；但如果你想实现一个「也许交换、也许不交换」的策略，用指针会更灵活。

```cpp
// flags: -O0 -g
#include <iostream>

void maybe_swap(int* a, int* b, bool do_it) {
    if (do_it && a && b) {
        int tmp = *a;
        *a = *b;
        *b = tmp;
    }
}

int main() {
    int x = 1, y = 2;
    maybe_swap(&x, &y, true);
    std::cout << x << ' ' << y << '\n';   // 2 1
    maybe_swap(&x, &y, false);
    std::cout << x << ' ' << y << '\n';   // 2 1
    return 0;
}
```

所以正确的心态不是「用引用取代指针」，而是：**能确定对象必然存在、不需要重新绑定时，优先用引用；需要表达可选、可空、可重新绑定或做底层内存操作时，用指针**。

## 问题 4：传 `const T&` 还是传值？悬垂引用又是怎么来的？

如果只读访问一个大对象，传 `const T&` 几乎是标准答案。它既避免了拷贝，又通过 `const` 向调用者承诺不会修改。

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>
#include <vector>

bool contains(const std::vector<std::string>& logs, const std::string& target) {
    for (const auto& log : logs) {
        if (log == target) return true;
    }
    return false;
}

int main() {
    std::vector<std::string> logs = {"move", "jump", "attack"};
    std::cout << std::boolalpha
              << contains(logs, "jump") << '\n'   // true
              << contains(logs, "dash") << '\n';  // false
    return 0;
}
```

但有一个隐藏陷阱：**引用必须绑定到一个真实存在的对象**。如果绑定到临时对象，临时对象表达式结束后就被销毁，引用就会悬垂。

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

const std::string& get_prefix() {
    std::string local = "[LOG] ";   // 局部对象
    return local;                    // 返回局部对象的引用：悬垂！
}

int main() {
    const std::string& prefix = get_prefix();
    std::cout << prefix << '\n';     // 未定义行为，可能崩溃或打印乱码
    return 0;
}
```

这个例子里的 `prefix` 指向的 `local` 在 `get_prefix` 返回时已经销毁。这是新手最容易踩的坑之一：**返回局部对象的引用**。

另一个常见变体是：用 `const T&` 接收一个函数返回的临时对象，然后把它保存到另一个线程或更长的生命周期里。只要引用的生命周期比被引用对象长，就会出问题。

传值 vs 传 `const&` 的简单决策规则：

- **小对象**（`int`、`float`、`Vec3`、小型枚举）：传值。拷贝便宜，还能避免悬垂和别名问题。
- **大对象或拷贝昂贵的对象**（`std::string`、`std::vector`、复杂结构体）：传 `const T&`。
- **需要修改原对象**：传 `T&`。
- **可选参数或可能为空**：传 `T*`。

## 问题 5：引用的底层和指针到底有什么区别？

从实现层面看，编译器通常会用指针来实现引用；但 **语义层面完全不同**：

- 指针是一个独立变量，保存地址；引用是对象的别名，没有独立身份。
- 指针可以不初始化、可以为空、可以重新赋值；引用定义时必须绑定且终身绑定。
- `sizeof(T*)` 是指针大小；`sizeof(T&)` 是被引用对象的大小（语义上）。
- 指针可以做算术；引用不能。

这些语义差异决定了：在 API 设计里，引用传达的是「必然存在、不可为空」的强契约，指针传达的是「可能为空、可能重新指向」的弱契约。选择哪一种，是在向阅读代码的人传递意图。

---

## 总结

- 传值最安全但会拷贝，引擎里对大对象和多态对象通常不合适。
- 传指针能解决拷贝和多态问题，但引入空指针、所有权、悬垂等管理负担。
- 传引用在性能上等价于传指针，在语法上像传值，并且天然非空、不可重新绑定。
- 引用的核心限制：必须初始化、不能重新绑定、不能为空；因此指针在可选输出、动态内存、容器存储、数组算术等场景仍然不可替代。
- 只读大对象优先传 `const T&`；可能修改原对象传 `T&`；可能为空或需要重新绑定传 `T*`。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 接口参数约定 | 数学类型（`Vec3`、`Mat4`）等小对象传值；日志字符串、资源描述等大对象传 `const T&`；输出参数优先用 `T&`，可选输出用 `T*` |
| **UE** | `TArray`、`FString` 等容器接口 | `const TArray<T>&` 只读遍历、`TArray<T>&` 修改、`T*` 表示可选元素或指针容器；蓝图反射层大量用引用传递输出值 |

> [!note] 关键取舍
> SelfGameEngine 倾向于用引用表达「强存在」契约，减少运行时判空；UE 由于历史原因和 C++ 反射层更复杂，指针和引用并用。两者共同点是：**大对象只读一定走 `const&`**，这是避免帧时间被拷贝吃掉的红线。
