# 完美转发

> [!info] 一句话概括
> **完美转发**解决的是：在泛型代码里把参数原封不动地传给另一个函数，既保持它的值类别（左值/右值），又不引入多余的拷贝或移动。

---

## 问题 0：为什么不能直接 `std::move` 所有参数？

想象你要写一个工厂函数，根据参数构造一个对象：

```cpp
template<typename T, typename Arg>
T make(Arg arg) {
    return T(arg);  // 按值传递，已经多了一次拷贝
}
```

如果 `Arg` 是一个很大的对象，按值传参会先拷贝一次，构造函数里再拷贝/移动一次，效率很差。

你可能会想：那用 `std::move` 不就行了？

```cpp
template<typename T, typename Arg>
T make(Arg&& arg) {
    return T(std::move(arg));  // 不管 arg 是左值右值，都当成右值
}
```

但这会出问题：如果调用者传了一个左值，你内部把它 `move` 走了，调用者之后再用这个左值就是未定义行为。

```cpp
std::string s = "hello";
auto x = make<MyString>(s);  // s 被 move 走了！
std::cout << s;  // ❌ 未定义行为
```

> [!abstract]
> **完美转发（Perfect Forwarding）**指的是：函数模板接收任意参数，然后把这些参数以**原来的值类别**（左值还是左值，右值还是右值）转发给另一个函数。核心工具是 `std::forward` 和**万能引用（universal reference）**。

---

## 问题 1：什么是万能引用？

当你看到 `T&&` 时，它有可能是两种含义：

1. **右值引用**：当 `T` 是具体类型时，如 `int&&`、`std::string&&`。
2. **万能引用**：当 `T` 是模板参数，且形参写成 `T&&` 时，它能同时绑定左值和右值。

```cpp
// flags: -O0 -g
#include <iostream>
#include <utility>

template<typename T>
void wrapper(T&& arg) {
    foo(std::forward<T>(arg));  // 保持 arg 原来的值类别
}

void foo(int& x)  { std::cout << "lvalue\n"; }
void foo(int&& x) { std::cout << "rvalue\n"; }

int main() {
    int a = 1;
    wrapper(a);       // T 推导为 int&，arg 是 int&，转发给 foo(int&)
    wrapper(1);       // T 推导为 int， arg 是 int&&，转发给 foo(int&&)
    return 0;
}
```

### 引用折叠规则

为什么 `T&&` 能同时是左值引用和右值引用？因为 C++ 有**引用折叠**规则：

| 组合 | 折叠结果 |
|------|---------|
| `T& &`  | `T&` |
| `T& &&` | `T&` |
| `T&& &` | `T&` |
| `T&& &&`| `T&&` |

当传入左值 `a` 时，`T` 被推导为 `int&`，那么 `T&&` 就是 `int& &&`，折叠后变成 `int&`。
当传入右值 `1` 时，`T` 被推导为 `int`，那么 `T&&` 就是 `int&&`。

> [!warning]
> 只有**模板参数**的 `T&&` 才是万能引用。`const T&&`、`std::vector<T>&&` 这种具体类型的右值引用不是万能引用。

## 问题 2：`std::forward` 到底做了什么？

`std::forward<T>(arg)` 的本质是：
- 如果 `T` 是左值引用类型，返回 `static_cast<T&>(arg)`，保持左值。
- 如果 `T` 不是左值引用类型，返回 `static_cast<T&&>(arg)`，把它变成右值引用。

所以它和 `std::move` 的区别很清楚：

| 工具 | 作用 | 使用场景 |
|------|------|---------|
| `std::move(x)` | 无条件把 `x` 转成右值 | 你想转移所有权，确定 `x` 之后不再用 |
| `std::forward<T>(x)` | 有条件地保持 `x` 的值类别 | 泛型转发，不知道 `x` 是左值还是右值 |

> [!tip]
> 记住一个简单规则：**万能引用 + std::forward 是套餐**。看到 `T&&` 形参，函数体里转发时就用 `std::forward<T>`。

## 问题 3：原地构造 `emplace` 为什么快？

标准容器的 `emplace_back` 就是完美转发的典型应用：

```cpp
std::vector<std::pair<int, std::string>> v;
v.emplace_back(42, "hello");  // 直接在 vector 内存里构造 pair，无临时对象
```

对比 `push_back`：

```cpp
v.push_back(std::make_pair(42, "hello"));
// 1. 构造临时 pair
// 2. move/copy 进 vector
```

`emplace_back` 把参数 `(42, "hello")` 完美转发给 `pair` 的构造函数，**直接在 vector 分配的内存里构造对象**，省去了临时对象和一次移动。

Day 38 的手写 `Vector::emplace_back` 就是要实现这个效果。

## 问题 4：`MakeShared` 为什么需要完美转发？

Day 10 的 `MakeShared` 要把任意构造参数转发给 `T` 的构造函数：

```cpp
template<typename T, typename... Args>
SharedPtr<T> MakeShared(Args&&... args) {
    // 一次性分配控制块 + 对象
    auto* storage = allocator::allocate(sizeof(ControlBlock) + sizeof(T));
    auto* obj = reinterpret_cast<T*>(storage + sizeof(ControlBlock));
    new (obj) T(std::forward<Args>(args)...);  // 完美转发构造参数
    return SharedPtr<T>(...);
}
```

如果不用 `std::forward`，而是直接 `T(args)...`：
- 当 `args` 包含右值时，会多一次拷贝。
- 当 `args` 包含不可拷贝的类型（如 `unique_ptr`）时，编译失败。

所以完美转发是 `MakeShared`、`emplace`、`工厂函数` 等泛型接口的必备技能。

## 问题 5：完美转发有哪些常见坑？

### 坑 1：`{ }` 初始化不能完美转发

```cpp
template<typename T>
void make_vec(T&& arg) {
    std::vector<int> v(std::forward<T>(arg));
}

make_vec({1, 2, 3});  // ❌ 编译错误：无法推导 {1,2,3} 的类型
```

大括号初始化列表没有独立类型，万能引用无法推导。解决方式是先构造对象再传，或显式指定类型。

### 坑 2：0 和 NULL 会被推导成 int

```cpp
template<typename T>
void foo(T&& x) { bar(std::forward<T>(x)); }

foo(NULL);  // T 推导为 int，不是指针！
```

现代 C++ 用 `nullptr` 避免这个问题。

### 坑 3：右值引用成员不是万能引用

```cpp
template<typename T>
class Wrapper {
    T&& value;  // 不是万能引用！T 确定后这就是右值引用成员
};
```

只有函数模板的 `T&&` 形参才是万能引用。

## 总结

- **完美转发**让泛型函数把参数以原值类别传给下游函数。
- 万能引用 `T&&` 通过**引用折叠**同时绑定左值和右值。
- `std::forward<T>` 配合万能引用使用；`std::move` 用于确定要转移所有权的场景。
- `emplace` 和 `MakeShared` 都依赖完美转发实现原地构造和零拷贝转发。
- 注意大括号初始化、`NULL`、右值引用成员等转发失败场景。

---

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 泛型容器接口 | `Array::emplace`、`HashMap::emplace` 通过完美转发把参数直接传给元素构造函数；工厂函数如 `MakeShared` 用它来避免不必要的拷贝 |
| **UE** | `MakeShared`、`Emplace` | UE 的 `TSharedPtr` 和容器大量依赖完美转发；UE 源码中常见 `Forward<T>(Args...)` 模式 |

> [!note] 关键取舍
> 完美转发不是语法糖，而是泛型接口性能的关键。没有它，`emplace` 就退化成 `push_back`，`MakeShared` 无法高效转发构造参数。理解万能引用和引用折叠，是阅读现代 C++ 和引擎源码的必备门槛。
