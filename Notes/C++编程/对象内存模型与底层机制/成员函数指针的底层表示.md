# 成员函数指针的底层表示

> [!info] 一句话概括
> 成员函数指针不是「指向代码的地址」那么简单——它是一个复合结构，内部至少包含函数代码地址和 this 指针调整量。多重继承下它可能膨胀到 16 字节甚至更大，这正是引擎委托系统选择自己实现类型擦除、而不直接用 `std::function` 的底层原因之一。

---

## 问题 0：我想把成员函数注册到回调系统，能像普通函数指针那样用吗？

想象你在写一个游戏引擎的事件系统。玩家受伤时，需要触发一系列回调：播放音效、更新 UI、记录日志。你设计了一个通用的事件分发器：

```cpp
class EventDispatcher {
public:
    using Callback = void(*)();
    void register(Callback cb) { callbacks.push_back(cb); }
    void fire() { for (auto cb : callbacks) cb(); }
private:
    std::vector<Callback> callbacks;
};
```

现在你想把 `Player::onDamage` 注册进去：

```cpp
class Player {
public:
    void onDamage() { std::cout << "Player took damage!\n"; }
};

Player player;
EventDispatcher dispatcher;

// 编译错误！
dispatcher.register(&Player::onDamage);
```

编译器会报错，因为 `&Player::onDamage` 的类型不是 `void(*)()`，而是 `void (Player::*)()`——**成员函数指针**。它和普通的函数指针是完全不同的类型，不能隐式转换。

为什么会这样？因为 `Player::onDamage` 的代码里隐式用到了 `this` 指针（比如访问 `this->health`）。普通函数指针不携带任何对象信息，调用时不知道 `this` 该指向谁。成员函数指针则必须同时记住「函数代码在哪」和「调用时把哪个对象当作 this」。

那成员函数指针到底长什么样？为什么它比普通函数指针复杂得多？

---

## 问题 1：成员函数指针为什么比函数指针大？

先来看一个最直接的验证：

```cpp
// flags: -O0 -g
#include <iostream>

class SimpleClass {
public:
    void foo() {}
};

void freeFunction() {}

int main() {
    std::cout << "sizeof(free function pointer)  = " << sizeof(void(*)()) << "\n";
    std::cout << "sizeof(member function pointer) = " << sizeof(void (SimpleClass::*)()) << "\n";
    return 0;
}
```

在 64 位 Linux/Windows 上，普通函数指针通常是 8 字节。但成员函数指针呢？

- **MSVC**：通常是 8 字节（单继承场景下做了优化）
- **GCC/Clang**：通常是 16 字节

这已经说明了问题：成员函数指针内部存储的信息比普通函数指针多。但 16 字节还不是上限。让我们看一个更极端的情况——多重继承：

```cpp
// flags: -O0 -g
#include <iostream>

class Base1 {
public:
    virtual void f1() {}
};

class Base2 {
public:
    virtual void f2() {}
};

class Derived : public Base1, public Base2 {
public:
    void derivedFunc() {}
};

int main() {
    std::cout << "sizeof(void (Base1::*)())      = " << sizeof(void (Base1::*)()) << "\n";
    std::cout << "sizeof(void (Base2::*)())      = " << sizeof(void (Base2::*)()) << "\n";
    std::cout << "sizeof(void (Derived::*)())    = " << sizeof(void (Derived::*)()) << "\n";
    return 0;
}
```

在 MSVC 上，`Derived` 的成员函数指针可能达到 **16~24 字节**。在 GCC/Clang 上，可能已经是 16 字节（因为它们的实现本身就预留了 this 调整量）。

### 成员函数指针里到底藏了什么？

成员函数指针的底层结构通常包含至少两个字段：

| 字段 | 含义 |
|------|------|
| **函数地址（code ptr）** | 成员函数体代码在内存中的地址 |
| **this 调整量（delta）** | 调用时需要把对象地址偏移多少，才能拿到正确的 this 指针 |

在单继承场景下，派生类的成员函数如果被基类指针调用，this 指针不需要调整（delta = 0），所以有些编译器（如 MSVC）会把成员函数指针优化成和普通函数指针一样大——只存函数地址。

但在多重继承场景下，情况复杂了。考虑：

```cpp
Derived d;
Base2* b2 = &d;  // this 指针需要偏移，跳过 Base1 的部分
```

当你通过 `Base2` 的成员函数指针调用 `Derived` 的函数时，编译器需要先把对象地址往回（或往前）调整一段距离，才能找到 `Derived` 的完整对象。这个调整量必须记录在成员函数指针里。

更复杂的是，如果涉及**虚继承**，this 调整量甚至不是编译期常量——它取决于对象的具体类型和虚基类表。这时成员函数指针可能还需要一个额外的字段来记录「虚基类调整信息」，大小进一步膨胀。

## 问题 2：成员函数指针的调用机制是什么？

语法上，成员函数指针的调用有两种写法：

```cpp
SimpleClass obj;
void (SimpleClass::*fp)() = &SimpleClass::foo;

(obj.*fp)();        // 对象调用
SimpleClass* p = &obj;
(p->*fp)();         // 指针调用
```

注意括号是必须的，因为 `.*` 和 `->*` 的优先级低于函数调用括号 `()`。

从汇编层面看，`(obj.*fp)()` 的调用过程等价于：

```cpp
// 伪代码，展示编译器视角
// 1. 从成员函数指针中取出函数地址
void (*code_ptr)() = fp.code_address;

// 2. 从成员函数指针中取出 this 调整量
ptrdiff_t delta = fp.this_delta;

// 3. 调整 this 指针
SimpleClass* adjusted_this = (SimpleClass*)((char*)&obj + delta);

// 4. 调用函数，把调整后的 this 作为第一个参数传递
(*code_ptr)(adjusted_this);
```

对于普通成员函数，delta 通常是 0，调用基本等同于「普通函数调用 + 传 this 指针」。但对于多重继承中的成员函数指针，delta 可能非零，编译器必须在调用前插入地址调整指令。

我们来写一个可以验证 this 指针调整的代码：

```cpp
// flags: -O0 -g
#include <iostream>

class Base1 {
public:
    int x = 1;
    void printThis1() { std::cout << "Base1::printThis1  this = " << this << "\n"; }
};

class Base2 {
public:
    int y = 2;
    void printThis2() { std::cout << "Base2::printThis2  this = " << this << "\n"; }
};

class Derived : public Base1, public Base2 {
public:
    int z = 3;
    void printThisD() { std::cout << "Derived::printThisD this = " << this << "\n"; }
};

int main() {
    Derived d;
    std::cout << "&d                = " << &d << "\n";
    std::cout << "(Base1*)&d        = " << (Base1*)&d << "\n";
    std::cout << "(Base2*)&d        = " << (Base2*)&d << "\n\n";

    // 成员函数指针调用
    void (Derived::*fp1)() = &Derived::printThis1;  // 继承自 Base1
    void (Derived::*fp2)() = &Derived::printThis2;  // 继承自 Base2
    void (Derived::*fpd)() = &Derived::printThisD;  // Derived 自己的

    (d.*fp1)();
    (d.*fp2)();
    (d.*fpd)();

    return 0;
}
```

运行后你会发现一个关键现象：`fp2` 调用时打印的 `this` 指针与 `(Base2*)&d` 的值相同，而**不等于 `&d`**。这说明编译器在调用 `fp2` 时，自动把 `this` 指针从 `Derived*` 偏移到了 `Base2*` 的位置。

这个 this 调整信息，就存储在成员函数指针 `fp2` 的内部。如果你用 `sizeof` 比较单继承和多继承场景下的成员函数指针，就能直观感受到这个额外信息带来的空间膨胀。

## 问题 3：成员函数指针在引擎中有什么用？为什么它是委托系统的基础？

理解了成员函数指针的底层结构后，我们就能解释引擎中很多设计决策的来龙去脉。

### 委托系统需要「对象 + 成员函数」绑定

游戏引擎中随处可见「事件-回调」模式。比如：

```cpp
// 按钮点击时调用 Player::onButtonClick
button.onClick.bind(&player, &Player::onButtonClick);
```

这个 `bind` 操作本质上就是在内存中存储两个东西：
1. 对象的地址（`this` 指针）
2. 成员函数指针（包含函数地址和 this 调整量）

当事件触发时，委托系统把存储的对象地址和成员函数指针组合起来，执行 `(obj.*mfp)()` 的调用逻辑。成员函数指针的存在，让「绑定任意对象的任意成员函数」成为可能。

### std::function 的类型擦除

C++11 的 `std::function` 能把任意可调用对象（普通函数、lambda、成员函数绑定结果）封装成统一类型，底层用的就是**类型擦除（type erasure）**。它的内部通常有一个小对象优化缓冲区（比如 24 字节），对于成员函数绑定器（`std::bind`），这个缓冲区里存放的正是：

- 对象指针（或对象副本）
- 成员函数指针
- 调用代理函数（负责解包并执行 `(obj.*mfp)(args...)`）

### UE 委托系统的轻量级替代

UE 没有直接使用 `std::function`，而是自己实现了一套委托系统（`TDelegate`、`DECLARE_DELEGATE` 等）。原因之一就是 `std::function` 的类型擦除开销：

- `std::function` 为了支持任意可调用对象，内部通常有虚函数调用（通过函数指针间接调用代理函数）
- 小对象优化有大小限制，超出后需要堆分配
- 成员函数指针本身的大小在不同平台不固定，增加了跨平台复杂度

UE 的委托系统针对「绑定对象 + 成员函数」这个最常见场景做了特化：

```cpp
// 简化示意
class TDelegate {
    void* ObjectPtr;                    // this 指针
    void (*FunctionPtr)(void* this, ...); // 经过适配的函数指针
    // 或直接使用成员函数指针的内部表示
};
```

通过宏生成的代码在编译期就知道回调的类型，避免了 `std::function` 的运行时类型擦除开销。底层仍然依赖「对象地址 + 成员函数地址 + 可能的 this 调整」这三要素，但布局和控制权完全在自己手里。

### Lambda 的替代趋势

现代 C++ 中，lambda 表达式在很多场景下已经替代了成员函数指针 + `std::bind` 的组合：

```cpp
// 老写法：成员函数指针 + bind
auto cb = std::bind(&Player::onDamage, &player, std::placeholders::_1);

// 新写法：lambda（编译器会生成一个匿名闭包类）
auto cb = [&player](int damage) { player.onDamage(damage); };
```

lambda 的优势在于编译器能看到完整的调用链路，更容易内联优化。但 lambda 本质上仍然是一个带有 `operator()` 的匿名对象，如果把它塞进 `std::function`，仍然会触发类型擦除。引擎中追求零开销抽象时，模板 + lambda 的组合比 `std::function` 更受青睐。

## 问题 4：成员函数指针的边界与常见陷阱

### 陷阱 1：不能隐式转换

```cpp
void (*fp)() = &SimpleClass::foo;        // 错误
default void (SimpleClass::*mfp)() = &SimpleClass::foo;  // 正确
```

成员函数指针和普通函数指针是两种完全不同的类型，不能互相转换。如果你需要一个「既能存普通函数、又能存成员函数」的统一类型，必须用 `std::function` 或自己实现类型擦除。

### 陷阱 2：取地址的语法陷阱

```cpp
void (SimpleClass::*mfp)() = &SimpleClass::foo;  // 正确
void (SimpleClass::*mfp)() = SimpleClass::foo;   // 错误，缺少 &
```

取成员函数地址**必须加 `&`**。普通函数可以隐式退化为指针（`void (*fp)() = freeFunction;`），但成员函数不行。

### 陷阱 3：虚函数与成员函数指针

成员函数指针可以指向虚函数，但调用时仍会通过 vtable 解析：

```cpp
// flags: -O0 -g
#include <iostream>

class Base {
public:
    virtual void foo() { std::cout << "Base\n"; }
};

class Derived : public Base {
public:
    void foo() override { std::cout << "Derived\n"; }
};

int main() {
    void (Base::*mfp)() = &Base::foo;  // 指向虚函数

    Base b;
    Derived d;

    (b.*mfp)();  // 输出 Base
    (d.*mfp)();  // 输出 Derived！成员函数指针仍支持动态绑定

    return 0;
}
```

这说明成员函数指针里存储的函数地址，在虚函数场景下可能是一个 vtable 槽位索引或跳板函数地址，而不是最终的实现地址。编译器必须在调用时保留虚函数的分发逻辑。

### 陷阱 4：不能比较大小

C++ 标准规定，**成员函数指针之间只能做 `==` 和 `!=` 比较**，不能做 `<`、`>` 或算术运算。因为成员函数指针的内部结构是不透明的，不同编译器的布局不同，比较大小没有意义。

## 总结

- **成员函数指针不是普通函数指针**。它内部至少包含「函数代码地址」和「this 指针调整量」两个信息，多重继承下可能膨胀到 16~24 字节。
- **调用机制是「调整 this + 间接调用」**。编译器从成员函数指针中读取 delta，调整对象地址，再把调整后的 this 作为隐式参数传递给函数。
- **它是委托系统的实现基础**。无论是 `std::function` 的类型擦除，还是 UE 自研的 `TDelegate`，底层都需要「对象地址 + 成员函数地址 + this 调整」三要素来支持「绑定对象的方法」。
- **现代替代方案是 lambda + 模板**。lambda 让编译器能看到完整调用链路，更容易内联和优化；模板参数传递避免了 `std::function` 的类型擦除和虚函数开销。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 事件/委托系统的底层绑定 | 委托系统存储 `this` 指针 + 成员函数指针（或编译期生成的 lambda 闭包），事件触发时执行 `(obj.*mfp)(args)`。对性能极敏感的路径使用模板 + lambda，避免类型擦除 |
| **UE** | `TDelegate` / `DECLARE_DELEGATE` 宏体系 | UE 不依赖 `std::function`，而是通过宏生成特化的委托类。内部用 `TMethodPtr` 存储成员函数指针的底层表示（考虑跨平台和多继承的 this 调整），配合 `TWeakObjectPtr` 做对象生命周期安全检测 |
| **UE** | 蓝图动态委托 vs C++ 静态委托 | C++ 委托在编译期确定类型，直接内联调用；蓝图委托因需运行时类型安全，走 UObject 反射 + 虚函数路径。成员函数指针的知识帮助你理解为什么 C++ 委托比蓝图委托快一个数量级 |

> [!note] 关键取舍
> `std::function` 为了「通用」牺牲了性能：类型擦除带来间接调用、小对象优化限制、堆分配 fallback。UE 选择自己实现委托，本质上是在「通用性」和「性能」之间做了更激进的偏向——只支持最常见的回调模式（普通函数、lambda、对象+成员函数），换取编译期确定性和零虚函数开销。
> SelfGameEngine 的委托系统如果走更激进的模板化路线，可以在编译期完全内联回调调用，连成员函数指针的存储都省去——但代价是二进制体积增加和编译时间延长。
