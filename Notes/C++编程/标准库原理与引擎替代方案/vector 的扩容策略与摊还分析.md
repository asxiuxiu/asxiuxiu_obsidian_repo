# std::vector 的扩容策略与摊还分析

> [!info] 一句话概括
> `std::vector` 的扩容不是「满了就加 1」，而是按**增长因子**一次性申请更大内存，把旧元素搬过去——这个策略让单次 `push_back` 的平均代价稳定在 O(1)，但扩容瞬间的停顿和元素迁移成本，是手写引擎容器时必须仔细权衡的地方。

---

## 问题 0：为什么 vector 要扩容？

`vector` 承诺元素在内存中连续存储。当你 `push_back` 时，如果尾部已经没有可用空间，就必须：

1. 申请一块更大的连续内存。
2. 把旧元素搬过去（拷贝或移动）。
3. 释放旧内存。
4. 在新尾部构造新元素。

如果每次只多申请 1 个元素的空间，那么每次 `push_back` 都要搬迁所有已有元素，n 次插入的总代价是 O(n²)。

所以 vector 采用**超额分配（over-allocation）**策略：每次扩容时申请比当前需求更大的空间，把搬迁成本均摊到多次插入上。

---

## 问题 1：增长因子选多少？

### 翻倍策略（growth factor = 2）

很多实现（如 GCC 的 libstdc++）采用约 2 倍增长：

```
容量序列：1 → 2 → 4 → 8 → 16 → 32 → ...
```

优点：均摊分析简单，push_back 平均 O(1)。

缺点：
- 内存浪费严重：最终容量可能是实际元素数量的 2 倍。
- 内存碎片：旧内存块大小不一，释放后可能无法被后续分配复用。

### 1.5 倍策略（growth factor ≈ 1.5）

一些实现（如 MSVC 的某些版本）采用 1.5 倍增长：

```
容量序列：1 → 2 → 3 → 4 → 6 → 9 → 13 → ...
```

优点：
- 内存浪费更少。
- 存在一种理论性质：新分配的内存块大小不会小于之前所有释放块的总和，从而可以利用内存碎片（这个性质在 2 倍策略下不成立）。

缺点：均摊常数比 2 倍策略稍大，但仍然是 O(1)。

### 工程中的选择

| 引擎/库 | 增长因子 | 备注 |
|--------|---------|------|
| GCC libstdc++ | 2 | 经典实现 |
| MSVC | 1.5 | 内存效率优先 |
| UE TArray | 约 1.5-2 | 可配置，考虑序列化和对齐 |
| SelfGameEngine | 可配置 | 通常 1.5 或 2，取决于使用场景 |

> [!tip]
> Day 37 的练习不必纠结「最佳增长因子」。关键是理解：增长因子 > 1 才能让均摊复杂度降到 O(1)，而具体选 1.5 还是 2 是工程权衡。

## 问题 2：reallocate 时元素怎么迁移？

当 vector 扩容时，已有元素需要搬到新内存。这里有两种方式：

### 拷贝构造（C++11 之前）

```cpp
// 概念代码
for (size_t i = 0; i < size; ++i) {
    new (new_data + i) T(old_data[i]);  // 拷贝构造
    old_data[i].~T();                    // 析构旧元素
}
```

### 移动构造（C++11 之后）

如果 `T` 的移动构造是 `noexcept` 的，vector 会优先使用移动：

```cpp
for (size_t i = 0; i < size; ++i) {
    new (new_data + i) T(std::move(old_data[i]));  // 移动构造
    old_data[i].~T();                                 // 析构旧元素
}
```

> [!warning]
> 如果 `T` 的移动构造不是 `noexcept`，扩容过程中一旦发生异常，vector 可能无法保证强异常安全。因此标准库容器通常回退到拷贝构造。这也是为什么在写引擎容器时，要尽可能把移动构造标为 `noexcept`。

## 问题 3：摊还分析是怎么回事？

假设增长因子为 2，容量序列是 1, 2, 4, 8, ..., n。

把 n 个元素依次 push_back 的总成本：
- 每次插入：构造新元素，O(1)。
- 扩容时：搬迁已有元素。

总搬迁元素数 ≈ 1 + 2 + 4 + 8 + ... + n/2 = n - 1

所以 n 次插入的总成本是 O(n)，单次均摊 O(1)。

> [!abstract]
> **摊还分析（Amortized Analysis）**不是看单次操作的最坏情况，而是看连续 n 次操作的平均代价。vector 的 push_back 单次最坏是 O(n)（扩容时），但均摊是 O(1)。

## 问题 4：`size`、`capacity`、`reserve`、`shrink_to_fit` 是什么关系？

| 概念 | 含义 |
|------|------|
| `size()` | 当前有效元素数量 |
| `capacity()` | 不扩容的情况下最多能容纳的元素数量 |
| `reserve(n)` | 提前分配至少容纳 n 个元素的空间，避免后续多次扩容 |
| `shrink_to_fit()` | 请求把 capacity 缩到 size（不一定真的缩，是实现定义的） |

```cpp
// flags: -O0 -g
#include <iostream>
#include <vector>

int main() {
    std::vector<int> v;
    v.reserve(100);
    std::cout << "size=" << v.size() << " capacity=" << v.capacity() << "\n";

    for (int i = 0; i < 10; ++i) v.push_back(i);
    std::cout << "size=" << v.size() << " capacity=" << v.capacity() << "\n";

    v.shrink_to_fit();
    std::cout << "size=" << v.size() << " capacity=" << v.capacity() << "\n";
    return 0;
}
```

> [!tip]
> 如果你大概知道要存多少元素，先 `reserve` 能显著减少扩容次数。这在引擎加载资源、批量生成粒子等场景中非常重要。

## 问题 5：手写 `Vector` 时要注意什么？

Day 37 的练习是手写一个简化版 `Vector`。关键检查点：

1. **三个核心指针/索引**
   - `start_`：内存起始。
   - `finish_`：最后一个元素的下一个位置（即 `size()` 的边界）。
   - `end_of_storage_`：已分配内存的边界（即 `capacity()` 的边界）。

2. **`push_back` 的扩容逻辑**
   - 检查 `finish_ == end_of_storage_`，如果是则扩容。
   - 扩容后在新位置构造新元素，不要先构造再搬迁。

3. **元素迁移的异常安全**
   - 尽量使用移动构造（如果 `noexcept`）。
   - 如果迁移过程中异常，保证不泄漏已分配的内存。

4. **`pop_back` 的正确性**
   - 只减少 `finish_` 并析构最后一个元素，不释放内存。

5. **`resize` 与 `reserve` 的区别**
   - `reserve` 只改变 capacity，不改变 size。
   - `resize` 改变 size，多出的元素默认构造或拷贝构造。

## 总结

- vector 通过**超额分配**把 `push_back` 的均摊复杂度降到 O(1)。
- 常见增长因子有 2 倍和 1.5 倍，前者简单、后者更省内存。
- 扩容时需要搬迁元素，优先使用 `noexcept` 移动构造。
- `reserve` 是性能优化利器，适合知道大致数量时使用。
- 手写 `Vector` 要理清 size/capacity 的边界，并保证扩容的异常安全。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 自定义 `Array`、`DynamicArray` | 自研容器可配置增长因子、自定义分配器、精确控制扩容时的元素迁移策略 |
| **UE** | `TArray` | `TArray` 支持 `Reserve`、`Shrink`、`Empty` 等操作；增长因子和分配策略针对不同平台有优化 |

> [!note] 关键取舍
> `std::vector` 为了通用性牺牲了一些可控性：它不知道你的对象是否适合移动、不知道你的分配器是否有特殊对齐要求。引擎自研容器会根据数据特性选择增长因子、批量构造/析构策略，甚至禁用异常安全保证以换取性能。理解 vector 的底层行为，是设计这些优化方案的前提。
