# 智能指针与所有权模型

> [!info] 一句话概括
> 智能指针用**类型系统表达所有权**：`unique_ptr` 表示独占，`shared_ptr` 表示共享，`weak_ptr` 表示旁观——理解了所有权，才算真正理解了 C++ 资源管理。

---

## 问题 0：裸指针管理资源有什么问题？

```cpp
void leak() {
    int* p = new int(42);
    if (some_error()) return;  // 提前返回 → 内存泄漏
    delete p;
}

void double_free() {
    int* p = new int(42);
    delete p;
    delete p;  // 重复释放 → 未定义行为
}
```

裸指针的痛点不是指针本身，而是**所有权不明确**：谁负责 `delete`？什么时候 `delete`？异常或提前返回时怎么办？

> [!abstract]
> **所有权（Ownership）**回答的是「谁负责释放这个资源」。C++ 通过智能指针把所有权编码进类型，让编译器和作用域自动保证释放责任不遗漏、不重复。

## 问题 1：`unique_ptr`：独占所有权

`std::unique_ptr<T>` 表示「我是这个资源的唯一主人」。它**不可拷贝**，只能**移动**。

```cpp
// flags: -O0 -g
#include <iostream>
#include <memory>

class Resource {
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
};

int main() {
    {
        std::unique_ptr<Resource> res = std::make_unique<Resource>();
        // res 是唯一所有者
    }  // 离开作用域，自动 delete

    // std::unique_ptr<Resource> copy = res;  // ❌ 编译错误：不可拷贝
    return 0;
}
```

### 核心操作

| 操作 | 含义 | 所有权变化 |
|------|------|-----------|
| `std::make_unique<T>(args)` | 创建并初始化 | 创建者获得所有权 |
| `std::move(up)` | 转移所有权 | 源失去所有权，目标获得 |
| `up.reset()` | 释放当前资源 | 所有权清空 |
| `up.release()` | 交出裸指针 | 所有权转移给调用者，需手动 `delete` |
| `up.get()` | 借用裸指针 | 所有权不变 |

> [!warning]
> 不要对 `get()` 返回的裸指针调用 `delete`。`get()` 只是**借用**，`unique_ptr` 仍然拥有资源并会在析构时释放。

### 自定义删除器

对于 `FILE*` 这类 C 资源，默认 `delete` 不适用，可以指定删除器：

```cpp
// flags: -O0 -g
#include <cstdio>
#include <memory>

struct FileDeleter {
    void operator()(FILE* f) const {
        if (f) std::fclose(f);
    }
};

int main() {
    std::unique_ptr<FILE, FileDeleter> file(
        std::fopen("test.txt", "r")
    );
    // 离开作用域自动 fclose
    return 0;
}
```

## 问题 2：`shared_ptr`：共享所有权

`std::shared_ptr<T>` 表示「多个人共同拥有这个资源」。它内部维护一个**引用计数**，最后一个所有者释放时销毁对象。

```cpp
// flags: -O0 -g
#include <iostream>
#include <memory>

int main() {
    std::shared_ptr<int> a = std::make_shared<int>(42);
    {
        std::shared_ptr<int> b = a;  // 引用计数 +1
        std::cout << *b << "\n";
    }  // b 离开作用域，引用计数 -1

    // a 仍然有效，因为对象还没被释放
    std::cout << *a << "\n";
    return 0;
}
```

### 控制块（Control Block）

`shared_ptr` 内部有两部分：
- **被管理对象**：`T` 本身。
- **控制块**：存放引用计数、弱引用计数、删除器等元数据。

```cpp
// 推荐：一次分配
auto p1 = std::make_shared<Resource>();
// [控制块 | Resource 对象] 连续存储

// 不推荐：两次分配
auto p2 = std::shared_ptr<Resource>(new Resource());
// 控制块和对象分开，多一次 malloc，cache 不友好
```

> [!tip]
> 优先使用 `std::make_shared`。它更高效、异常安全，且控制块与对象连续存储。

### 引用计数是原子的

`shared_ptr` 的拷贝和析构会修改引用计数，这个操作是**原子的**，所以可以在多线程间共享 `shared_ptr` 本身。但要注意：
- 多线程同时读写**同一个 `shared_ptr` 对象**需要同步（拷贝不算读写对象本身）。
- 多线程读写**被指向的对象**需要另外的同步机制（如 mutex）。

## 问题 3：`weak_ptr`：打破循环引用

如果两个对象互相持有 `shared_ptr`，引用计数永远降不到 0，导致内存泄漏：

```cpp
// flags: -O0 -g
#include <iostream>
#include <memory>

struct Node {
    std::shared_ptr<Node> next;  // 循环引用！
    ~Node() { std::cout << "Node destroyed\n"; }
};

int main() {
    auto a = std::make_shared<Node>();
    auto b = std::make_shared<Node>();
    a->next = b;
    b->next = a;  // 引用计数互锁，泄漏
    return 0;
}
```

`weak_ptr` 不增加引用计数，只是「旁观」对象是否还活着：

```cpp
struct Node {
    std::weak_ptr<Node> next;  // 弱引用，不增加引用计数
    ~Node() { std::cout << "Node destroyed\n"; }
};

int main() {
    auto a = std::make_shared<Node>();
    auto b = std::make_shared<Node>();
    a->next = b;
    b->next = a;  // ✅ 不会循环引用
    return 0;
}
```

使用 `weak_ptr` 时需要先 `lock()` 提升为 `shared_ptr`，并检查是否成功：

```cpp
if (auto locked = weak.lock()) {
    // 对象还活着，可以安全使用
} else {
    // 对象已被释放
}
```

## 问题 4：写 `SharedPtr` 前要理解什么？

Day 9 要求手写简化版 `SharedPtr`，Day 10 要求加上 `MakeShared`。核心概念：

1. **控制块**：至少包含 `use_count`。简化版可以不支持弱引用和自定义删除器。
2. **原子引用计数**：多线程环境下拷贝/析构 `SharedPtr` 必须原子地增减计数。Day 9 的简化版可能不考虑线程安全，但工业级实现必须考虑。
3. **不要混合裸指针**：从同一个裸指针创建两个独立的 `SharedPtr` 会产生两个控制块，最终 double-free。
4. **`MakeShared` 的优化**：一次性分配控制块 + 对象，减少 malloc 次数、提高 cache 友好性。

```cpp
// 错误示范
template<typename T>
class NaiveSharedPtr {
    T* ptr_;
    int* count_;
public:
    NaiveSharedPtr(T* p) : ptr_(p), count_(new int(1)) {}
    ~NaiveSharedPtr() {
        if (--(*count_) == 0) {
            delete ptr_;
            delete count_;
        }
    }
    // 拷贝构造、拷贝赋值需要 +1 引用计数
};
```

> [!warning]
> 上面的简化版没有处理自赋值、异常安全、移动语义等细节。Day 9 的练习会逐步引导你补全，但前提是先理解「共享所有权 + 控制块 + 引用计数」这三个核心概念。

## 总结

| 智能指针 | 所有权 | 拷贝 | 典型使用场景 |
|---------|-------|------|------------|
| `unique_ptr<T>` | 独占 | 不可拷贝，可移动 | 默认选择，工厂函数返回值，Pimpl |
| `shared_ptr<T>` | 共享 | 可拷贝，引用计数 | 多所有者，缓存、观察者 |
| `weak_ptr<T>` | 无 | 可拷贝，不增加计数 | 打破循环引用，缓存观察者 |

- 智能指针的价值在于把**释放责任**编码进类型，让作用域和析构自动处理。
- `unique_ptr` 零开销，`shared_ptr` 有引用计数原子操作开销。
- `make_unique` / `make_shared` 更高效、异常安全，优先使用。
- 写 `SharedPtr` 前必须理解控制块和引用计数；写 `MakeShared` 前还需要理解完美转发。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄、对象所有权 | 自定义 `UniquePtr`/`SharedPtr` 替代标准库，确保不依赖异常、控制分配器、可嵌入自定义调试机制 |
| **UE** | `TSharedPtr`、`TWeakObjectPtr`、`TUniquePtr` | UE 自研智能指针体系，与 UObject 垃圾回收共存；`TSharedFromThis` 用于在对象内部安全地获取 `TSharedPtr` |

> [!note] 关键取舍
> SelfGameEngine 自研智能指针是为了和自定义分配器、禁用异常的策略保持一致。UE 则同时存在 UObject GC 和 `TSharedPtr` 两套所有权体系，理解标准智能指针模型是理解这些工业级设计的前提。
