# 运算符重载的原则与陷阱

> [!info] 一句话概括
> 运算符重载让自定义类型拥有像内置类型一样的直观语法，但只有**语义一致、行为对称、不滥用**时，它才是便利而不是陷阱。

---

## 问题 0：我想让 `Vec3` 像数字一样相加，最直观的写法是什么？会出什么问题？

在引擎里，数学类型无处不在。如果完全不懂运算符重载，我们可能会写一个普通函数：

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec3 {
    float x, y, z;
};

Vec3 add(const Vec3& a, const Vec3& b) {
    return {a.x + b.x, a.y + b.y, a.z + b.z};
}

int main() {
    Vec3 a{1.0f, 2.0f, 3.0f};
    Vec3 b{4.0f, 5.0f, 6.0f};
    Vec3 c = add(a, b);
    std::cout << c.x << ", " << c.y << ", " << c.z << "\n";
    return 0;
}
```

这能跑，但写起来不像数学。我们自然会想：能不能写成 `Vec3 c = a + b;`？

运算符重载就是干这个的。但初学时很容易把它当成"把函数名换成符号"，忽略了一个核心事实：**运算符在 C++ 里不是孤立的语法糖，它承载着读者对内置类型的语义预期**。如果重载后 `+` 做的事不像"加法"，代码会比 `add(a, b)` 更难读。

---

## 问题 1：重载 `+` 时，写成成员函数和非成员函数有什么区别？

最 naive 的做法是把 `operator+` 写成成员函数：

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec3 {
    float x, y, z;

    Vec3 operator+(float s) const {
        return {x + s, y + s, z + s};
    }
};

int main() {
    Vec3 v{1.0f, 2.0f, 3.0f};
    Vec3 a = v + 2.0f;      // OK
    // Vec3 b = 2.0f + v;   // 编译错误：成员 operator+ 要求左侧必须是 Vec3
    return 0;
}
```

成员运算符的左侧操作数**必须是当前类的对象**。所以当左侧是 `float`、右侧是 `Vec3` 时，`2.0f + v` 无法匹配。

这在数学类型里很常见：`v * 2` 和 `2 * v` 应该都能写。解决方法是把 `operator+` 做成非成员函数：

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec3 {
    float x, y, z;

    // 复合赋值运算符通常作为成员函数：直接修改自身，效率最高
    Vec3& operator+=(const Vec3& rhs) {
        x += rhs.x; y += rhs.y; z += rhs.z;
        return *this;
    }

    Vec3& operator+=(float s) {
        x += s; y += s; z += s;
        return *this;
    }
};

// 对称的 + 用非成员函数实现，基于 += 避免重复
Vec3 operator+(const Vec3& a, const Vec3& b) {
    Vec3 r = a;
    r += b;
    return r;
}

Vec3 operator+(const Vec3& v, float s) {
    Vec3 r = v;
    r += s;
    return r;
}

Vec3 operator+(float s, const Vec3& v) {
    return v + s;  // 复用上面的版本
}

int main() {
    Vec3 v{1.0f, 2.0f, 3.0f};
    Vec3 a = v + 2.0f;
    Vec3 b = 2.0f + v;
    std::cout << a.x << " " << b.x << "\n";  // 3 3
    return 0;
}
```

这里有两个惯例：

1. **复合赋值运算符（+=、-=、*= 等）作为成员函数**：它们要修改自身，左侧固定是当前对象，不需要对称性。
2. **普通算术运算符（+、-、* 等）作为非成员函数**：保证左右操作数的类型对称。

如果只写成员 `operator+`，你等于告诉用户"这个运算只对 `Vec3` 在左侧时成立"，这通常不是数学库想要的语义。

## 问题 2：赋值运算符为什么要检查自赋值？

运算符重载里最容易被轻视的是 `operator=`。初学时可能会这样写：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class String {
    char* data_;
    size_t len_;
public:
    String(const char* s) : len_(std::strlen(s)) {
        data_ = new char[len_ + 1];
        std::memcpy(data_, s, len_ + 1);
    }
    ~String() { delete[] data_; }

    // 没有自赋值保护的拷贝赋值
    String& operator=(const String& rhs) {
        delete[] data_;                // ① 先释放自己的资源
        len_ = rhs.len_;
        data_ = new char[len_ + 1];    // ② 再分配新内存
        std::memcpy(data_, rhs.data_, len_ + 1);
        return *this;
    }

    const char* c_str() const { return data_; }
};

int main() {
    String s("hello");
    s = s;  // 自赋值：rhs 就是 *this
    std::cout << s.c_str() << "\n";
    return 0;
}
```

如果执行 `s = s`，第一步 `delete[] data_` 会把 `rhs` 也依赖的内存释放掉（因为 `rhs` 就是 `s`），接下来访问 `rhs.data_` 就是未定义行为。

修复方法是在开头检查自赋值：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class String {
    char* data_;
    size_t len_;
public:
    String(const char* s) : len_(std::strlen(s)) {
        data_ = new char[len_ + 1];
        std::memcpy(data_, s, len_ + 1);
    }
    ~String() { delete[] data_; }

    String& operator=(const String& rhs) {
        if (this == &rhs) return *this;  // 自己赋给自己，直接返回
        delete[] data_;
        len_ = rhs.len_;
        data_ = new char[len_ + 1];
        std::memcpy(data_, rhs.data_, len_ + 1);
        return *this;
    }

    const char* c_str() const { return data_; }
};

int main() {
    String s("hello");
    s = s;  // 自赋值：被 guard 安全处理
    std::cout << s.c_str() << "\n";
    return 0;
}
```

对于像 `Vec3` 这种纯值类型，自赋值 `v = v` 不会造成问题；但对于**持有资源**的类型，自赋值检查是 `operator=` 的必修课。

## 问题 3：哪些运算符不能随便重载？

C++ 允许重载大部分运算符，但不代表都应该重载。最危险的几个是：

- `&&` 和 `||`：内置版本有**短路求值**（左侧为真/假时右侧不执行），重载后变成普通函数调用，两侧操作数一定会被求值。
- `,`（逗号运算符）：内置版本有从左到右的求值顺序保证，重载后求值顺序由函数调用规则决定，不再保证。
- `&`（取地址）：重载后会让 `&obj` 不再返回对象的真实地址，破坏很多底层假设。

来看 `&&` 的陷阱：

```cpp
// flags: -O0 -g
#include <iostream>

struct Guard {
    bool ok;
    explicit Guard(bool b) : ok(b) {}
};

bool operator&&(const Guard& a, const Guard& b) {
    std::cout << "operator&& called\n";
    return a.ok && b.ok;  // 这里虽然内部还是短路，但 b 已经被构造出来了
}

int main() {
    Guard g(false);
    // 内置 &&：右侧不会执行
    // 重载后：右侧必须先求值，才能传进函数
    if (g && Guard(true)) {
        std::cout << "inside\n";
    }
    return 0;
}
```

输出会显示 `operator&& called`，而且右侧的 `Guard(true)` 一定会被构造。如果你把 `&&` 用于"左侧检查通过后右侧才能安全执行"的逻辑（比如判空后再访问），重载后就会直接崩溃。

另一个常见陷阱是**改变运算符的语义**。比如把 `+` 重载成"连接网络消息"、把 `<<` 重载成"写入日志文件"——在特定领域这是合理的（流输出 `<<` 就是如此），但如果 `+` 的行为让读者无法一眼猜到，就会增加认知负担。

## 问题 4：为什么重载后会出现"隐藏"的隐式转换？

假设我们给 `Vec3` 加了一个类型转换运算符：

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec3 {
    float x, y, z;
    operator bool() const { return x != 0.0f || y != 0.0f || z != 0.0f; }
};

void print(int n) { std::cout << "int: " << n << "\n"; }

int main() {
    Vec3 v{1.0f, 0.0f, 0.0f};
    print(v);   // 通过 operator bool -> bool -> int 的隐式转换链
    return 0;
}
```

本意可能是让 `Vec3` 能在 `if (v)` 里判断是否为零向量，结果它却能一路转换成 `int` 去调用 `print(int)`。这就是隐式类型转换运算符的副作用：编译器会把它和标准转换链组合起来，匹配到你意想不到的重载。

这个问题会在下一篇笔记《[[Notes/C++编程/运算符重载基础/explicit 与隐式转换控制|explicit 与隐式转换控制]]》里专门讨论。这里先记住一条原则：**如果你只想让类型在条件判断、显式转换等少数场景下"变成另一个类型"，请把转换运算符声明为 `explicit`**。

## 总结

- 运算符重载不是"让代码更酷"，而是让自定义类型符合读者的直觉语义。
- 需要对称性的运算符（`+`、`-`、`*` 等）优先实现为**非成员函数**，并通过复合赋值运算符复用逻辑。
- 会修改左侧操作数的运算符（`=`、`+=`、`-=` 等）通常作为**成员函数**；`operator=` 要处理自赋值。
- `&&`、`||`、`,`、`&` 等运算符有内置的特殊语义，重载后这些语义会消失，一般不要碰。
- 类型转换运算符和单参数构造函数都会引入隐式转换，是运算符重载中最隐蔽的陷阱来源。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 数学库（`Vec3`、`Mat4`、`Quat`） | `operator+`、`-`、`*` 实现为非成员函数保证对称；`operator*=` 等作为成员函数就地修改；避免重载 `&&`/`\|\|` 等逻辑运算符 |
| **UE** | `FVector`、`FMatrix`、`FQuat` 的算术运算 | `FVector::operator+` 等作为成员/非成员搭配，保持与数学直觉一致；`TSharedPtr` 等句柄类使用 `explicit operator bool` 防止意外转换 |

> [!note] 关键取舍
> 引擎数学库通常会把"数学上对称"的运算符做成非成员函数，把"修改自身"的运算符做成成员函数。这不是语法偏好，而是在表达"这个运算是二元对称关系"还是"对象对自己做修改"的设计意图。
