# 默认与删除的函数（= default / = delete）

> [!info] 一句话概括
> `= default` 是说「我要编译器生成这个特殊成员函数」；`= delete` 是说「我要显式禁止这个函数存在」。两者都是把编译器行为从「隐式猜测」变成「显式声明」，让类的拷贝控制和可构造性在接口层面一目了然。

---

## 问题 0：编译器自动生成函数，为什么还需要显式控制？

我在前面一篇笔记里写过：只要类里写了任何自定义构造函数，编译器就不再自动生成默认构造函数。这会带来一个常见的维护问题——代码阅读者不知道某个类「到底能不能默认构造」。

比如下面这个资源句柄：

```cpp
// flags: -O0 -g
#include <iostream>

class Texture {
public:
    Texture(const char* path) {
        std::cout << "加载纹理: " << path << '\n';
    }
    ~Texture() { std::cout << "释放纹理\n"; }
};

int main() {
    // Texture t;  // 编译错误：没有默认构造函数
    Texture t("hero.png");
    return 0;
}
```

`Texture` 写了 `Texture(const char*)` 这个自定义构造后，`Texture t;` 就编译不过了。这个行为本身没错——空纹理确实语义不明——但问题是：这个「不能默认构造」的事实是编译器**隐式**决定的，类的声明里没有明确写出来。

在小型项目里这无所谓，但在引擎代码里，一个类可能有几十个特殊成员函数相关的隐含规则：默认构造能不能用？拷贝构造有没有？移动构造有没有？这些全要靠读者去试、去猜，或者看注释。

C++11 引入的 `= default` 和 `= delete` 就是要把这些规则**显式化**：

- `= default`：明确告诉编译器「我要这个函数，用默认生成的版本」。
- `= delete`：明确告诉编译器「我不要这个函数，谁调用谁编译错误」。

这样类的接口本身就表达了设计意图，不再需要靠注释或试错来推断。

---

## 问题 1：`= default` 和「不写」有什么区别？

最直观的场景是：我写了自定义构造，但又想保留默认构造。用 `= default` 就可以显式恢复：

```cpp
// flags: -O0 -g
#include <iostream>

class Texture {
public:
    Texture() = default;  // 显式恢复默认构造
    Texture(const char* path) {
        std::cout << "加载纹理: " << path << '\n';
    }
};

int main() {
    Texture t1;              // OK
    Texture t2("hero.png");  // OK
    return 0;
}
```

这里 `Texture() = default;` 和「完全不写任何构造函数时编译器隐式生成默认构造」效果一样，但语义完全不同：**它明确告诉读者，这个默认构造是设计者故意保留的，不是遗漏。**

`= default` 还可以放在类外定义，用于控制默认版本的生成位置：

```cpp
// flags: -O0 -g
#include <iostream>

class Buffer {
public:
    Buffer() = default;
    Buffer(const Buffer&) = default;
    Buffer& operator=(const Buffer&) = default;
    ~Buffer() = default;
};

int main() {
    Buffer a;
    Buffer b = a;  // 调用 = default 的拷贝构造
    return 0;
}
```

对于**平凡类型（trivially copyable）**，编译器生成的拷贝构造往往比手写的循环复制更快，因为它可能被优化成 `memcpy`。`= default` 让编译器有机会生成这种高效版本，而不是被迫执行你手写的、可能带副作用的代码。

```cpp
// flags: -O0 -g
#include <cstring>
#include <iostream>

struct Vec3 {
    float x, y, z;
};

struct Wrapper {
    Vec3 v;
    Wrapper(float x, float y, float z) : v{x, y, z} {}
    Wrapper(const Wrapper&) = default;  // 编译器可用 memcpy 级别的优化
};

int main() {
    Wrapper a(1.0f, 2.0f, 3.0f);
    Wrapper b = a;
    std::cout << b.v.x << '\n';
    return 0;
}
```

> 注意：`= default` 只能用于**特殊成员函数**（默认构造、析构、拷贝/移动构造、拷贝/移动赋值）。你不能写 `void foo() = default;`，编译器会报错。

那么，如果我不仅想让编译器生成某个函数，还想**彻底禁掉**某个函数呢？

## 问题 2：`= delete` 怎么让「禁止拷贝」变得干净？

在 C++11 之前，要让一个类不可拷贝，通常的做法是把拷贝构造和拷贝赋值声明为 `private`，并且不提供实现：

```cpp
class NonCopyable {
    NonCopyable(const NonCopyable&);            // private，不实现
    NonCopyable& operator=(const NonCopyable&); // private，不实现
public:
    NonCopyable() {}
};
```

这个做法有几个问题：
1. 友元类和成员函数仍然可以访问，不小心就会绕过限制。
2. 成员函数内部如果误用，链接期才会报错（找不到实现），而不是编译期。
3. 意图不够明显，看到 private 拷贝构造，新手可能以为是「还没实现」而不是「故意禁止」。

`= delete` 把这些问题一次性解决：

```cpp
// flags: -O0 -g
#include <iostream>

class NonCopyable {
public:
    NonCopyable() { std::cout << "构造\n"; }
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};

int main() {
    NonCopyable a;
    // NonCopyable b = a;  // 编译错误：尝试引用已删除的函数
    // NonCopyable c; c = a;  // 编译错误
    return 0;
}
```

`= delete` 的核心价值在于：**把错误从运行期/链接期前移到编译期，并且让意图无可置疑。** 任何尝试拷贝 `NonCopyable` 的代码都会在编译时被直接拒绝。

`= delete` 不只用于拷贝控制。它可以禁用任何函数的任何重载：

```cpp
// flags: -O0 -g
#include <iostream>

void process(int) = delete;
void process(double x) {
    std::cout << "double: " << x << '\n';
}

int main() {
    // process(42);       // 编译错误：int 版本被删除
    process(3.14);        // OK
    process(42.0f);       // OK：float 提升为 double
    return 0;
}
```

这个技巧在泛型代码里特别有用，比如禁止 `foo(char*)` 但保留 `foo(const char*)`，或者禁止某个会隐式导致歧义的重载。

## 问题 3：删除一个特殊成员函数，会怎样影响其他特殊成员函数？

这里有一个容易踩的坑：**`= delete` 一个特殊成员函数，并不等于「没声明」它。** 它仍然是已声明的，只是不可用。而「已声明」这件事会影响编译器对其他特殊成员函数的生成决策。

最典型的例子是移动语义。如果你只删除了拷贝构造，但没有显式声明移动构造，编译器**不会**自动生成移动构造：

```cpp
// flags: -O0 -g
#include <iostream>

class NoCopy {
public:
    NoCopy() = default;
    NoCopy(const NoCopy&) = delete;  // 删除拷贝
    NoCopy& operator=(const NoCopy&) = delete;
};

int main() {
    NoCopy a;
    // NoCopy b = std::move(a);  // 编译错误：没有移动构造
    // NoCopy c = a;             // 编译错误：拷贝已删除
    return 0;
}
```

原因：C++11 规定，如果用户声明了拷贝构造、拷贝赋值、析构中的任意一个，移动构造和移动赋值就不再自动生成。`= delete` 也算「用户声明」。所以 `NoCopy` 虽然没有移动构造，但也不是因为它被删除了，而是因为它根本没被生成。

如果你想让类**可移动但不可拷贝**，必须显式保留移动操作：

```cpp
// flags: -O0 -g
#include <iostream>
#include <utility>

class MovableOnly {
public:
    MovableOnly() = default;
    MovableOnly(const MovableOnly&) = delete;
    MovableOnly& operator=(const MovableOnly&) = delete;

    MovableOnly(MovableOnly&&) = default;
    MovableOnly& operator=(MovableOnly&&) = default;
};

int main() {
    MovableOnly a;
    MovableOnly b = std::move(a);  // OK
    return 0;
}
```

这个规则直接推导出引擎里 `NonCopyable` 和 `NonMovable` 基类的实现：如果你只想禁止拷贝但保留移动，要同时 `= default` 移动操作；如果你连移动也想禁掉，可以 `= delete` 移动操作，或者干脆不声明（因为删除拷贝后移动也不会自动生成）。

## 问题 4：`= default` 和 `= delete` 在引擎里怎么落地？

理解了机制之后，真正工程化的做法是把它封装成基类。引擎里大量资源类都需要「不可拷贝」的语义，比如文件句柄、GPU 资源、锁等。与其在每个类里重复写四行 `= delete`，不如用一个基类统一表达：

```cpp
// flags: -O0 -g
#include <iostream>
#include <utility>

class NonCopyable {
public:
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};

class NonMovable : public NonCopyable {
public:
    NonMovable() = default;
    NonMovable(NonMovable&&) = delete;
    NonMovable& operator=(NonMovable&&) = delete;
};

class FileHandle : public NonCopyable {
    int fd = -1;
public:
    FileHandle(int f) : fd(f) {}
    ~FileHandle() { std::cout << "关闭 fd=" << fd << '\n'; }
};

int main() {
    FileHandle f(3);
    // FileHandle f2 = f;       // 编译错误：基类子对象不可拷贝
    // FileHandle f3 = std::move(f);  // 编译错误：没有可用的移动/拷贝构造
    return 0;
}
```

这里 `FileHandle` 继承自 `NonCopyable`，默认也会继承「不可拷贝」的语义。注意：**继承的是「行为」，不是「函数本身」**——派生类的拷贝构造仍然由编译器生成规则决定，但基类子对象不可拷贝，所以整体也无法拷贝。

如果你希望派生类可移动但不可拷贝，可以让 `NonCopyable` 同时 `= default` 移动操作：

```cpp
class NonCopyable {
public:
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
    NonCopyable(NonCopyable&&) = default;
    NonCopyable& operator=(NonCopyable&&) = default;
};
```

这是引擎里更常见的做法：禁止拷贝（因为资源所有权唯一），但允许移动（因为移动是安全的所有权转移）。

## 总结

- `= default` 显式要求编译器生成特殊成员函数的默认版本，常用于恢复默认构造或保留高效的平凡拷贝。
- `= delete` 显式禁止某个函数，常用于禁用拷贝/赋值，也可用于禁用特定重载。
- `= delete` 一个特殊成员函数仍算作「用户声明」，会抑制编译器对其他特殊成员函数（尤其是移动操作）的自动生成。
- 把 `= default` / `= delete` 封装成 `NonCopyable` / `NonMovable` 基类，是引擎代码中管理资源所有权和生命周期的标准做法。

---

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄禁止拷贝 | `FileHandle`、`GPUBuffer`、`Mutex` 等继承 `NonCopyable`，确保所有权唯一 |
| **SelfGameEngine** | 可移动但不可拷贝 | `UniquePtr<T>` 删除拷贝但默认移动，实现所有权的转移而非共享 |
| **UE** | `TNonCopyable` 基类 | UE 提供的 `FNonCopyable` 被广泛继承，用于 UObject 派生类、资源加载句柄等 |
| **UE** | 禁用危险重载 | `TSharedPtr` 通过 `= delete` 禁止从原始指针的隐式构造，避免引用计数不一致 |

> [!note] 关键取舍
> `= default` 不等于「自动生成的就一定好」，它只是把决定权显式交还给设计者。对于需要自定义析构的类（比如管理动态内存），通常也要自定义或显式删除拷贝/移动操作，遵循 Rule of Three/Five。`= delete` 的价值在于**把错误提前到编译期**——在引擎这种大型代码库中，编译期拒绝远比运行期崩溃便宜。
