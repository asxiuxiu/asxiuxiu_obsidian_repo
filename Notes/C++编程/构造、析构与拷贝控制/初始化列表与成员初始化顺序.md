# 初始化列表与成员初始化顺序

> [!info] 一句话概括
> 初始化列表不是「构造函数的语法糖」，而是成员**真正诞生**的地方——它决定了成员是直接构造还是事后修补，也暗藏着「书写顺序 ≠ 执行顺序」的经典陷阱。

---

## 问题 0：在构造函数体里用赋值语句初始化成员，有什么问题？

假设你在写一个引擎的 `TextureHandle` 类，它内部持有一个 OpenGL 纹理 ID：

```cpp
// flags: -O0 -g
#include <iostream>

class TextureHandle {
    unsigned int id_;
public:
    TextureHandle(unsigned int id) {
        id_ = id;  // 这是「赋值」，不是「初始化」
        std::cout << "TextureHandle constructed\n";
    }
};

int main() {
    TextureHandle tex(42);
    return 0;
}
```

这段代码能跑，但有一个隐藏的代价：在 `id_ = id` 执行之前，`id_` 已经**默认构造**过一次了。对于 `unsigned int` 这种内置类型，「默认构造」意味着它的值是不确定的（未初始化），然后你才用赋值把它覆盖掉。

对于更复杂的类型，这个代价会更明显：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

class Player {
    std::string name_;
public:
    Player(const std::string& name) {
        name_ = name;  // 先默认构造空字符串，再拷贝赋值
        std::cout << "Player body executed\n";
    }
};

int main() {
    Player p("Alice");
    return 0;
}
```

如果你在这段代码里打断点或插入打印，会发现 `std::string` 的默认构造函数**在 `Player` 构造函数体执行之前就被调用了**。`name_` 先变成空字符串，然后 `name_ = name` 把它重新赋值为 `"Alice"`——一次完全多余的默认构造 + 析构 + 拷贝赋值。

但这还不是最致命的。有些成员根本**不允许**你在构造函数体里赋值：

```cpp
class BadExample {
    const int id_;      // const 成员：一旦诞生就不能改
    int& ref_;          // 引用成员：一旦绑定就不能重新指向
    std::string& name_; // 同上
public:
    BadExample(int id, int& ref) {
        id_ = id;   // ❌ 编译错误：给 const 成员赋值
        ref_ = ref; // ❌ 编译错误：引用必须在初始化时绑定
    }
};
```

**这些成员必须在「诞生」的那一刻就确定值，事后赋值根本来不及。**

那么问题来了：怎么让成员在诞生时就拿到正确的值，而不是先默认构造再事后修补？

---

## 问题 1：初始化列表是什么？它跟函数体里的赋值有什么区别？

C++ 的答案是：**初始化列表（member initializer list）**，它写在构造函数的参数列表和函数体之间，用冒号 `:` 开头：

```cpp
class Player {
    std::string name_;
public:
    Player(const std::string& name)
        : name_(name)  // ← 初始化列表：直接调用拷贝构造
    {
        // 函数体里什么都不用做
    }
};
```

这里的 `name_(name)` 不是「调用某个函数」，而是**直接构造 `name_`**——编译器把它翻译成 `std::string name_(name);`，一步到位，没有默认构造的中间步骤。

对比两种写法：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

struct ViaAssignment {
    std::string name;
    ViaAssignment(const std::string& n) {
        name = n;  // 默认构造 → 拷贝赋值（两步）
    }
};

struct ViaInitList {
    std::string name;
    ViaInitList(const std::string& n)
        : name(n)  // 直接拷贝构造（一步）
    {}
};

int main() {
    std::cout << "--- ViaAssignment ---\n";
    ViaAssignment a("Alice");
    std::cout << "--- ViaInitList ---\n";
    ViaInitList b("Alice");
    return 0;
}
```

运行后你会发现 `ViaAssignment` 比 `ViaInitList` 多调用了 `std::string` 的默认构造函数。对于引擎中高频构造的对象（比如每帧生成的大量粒子、事件包），这种多余的构造开销会累积成明显的性能问题。

更重要的是，初始化列表能拯救那些在函数体里根本无法赋值的成员：

```cpp
// flags: -O0 -g
#include <iostream>

class GoodExample {
    const int id_;
    int& ref_;
public:
    GoodExample(int id, int& ref)
        : id_(id)    // ✅ const 成员：初始化时确定值
        , ref_(ref)  // ✅ 引用成员：初始化时绑定目标
    {
        // 函数体里可以安全使用它们
        std::cout << "id=" << id_ << ", ref=" << ref_ << "\n";
    }
};

int main() {
    int x = 100;
    GoodExample g(42, x);
    return 0;
}
```

**核心区别**：构造函数体执行时，所有成员已经完成「诞生」（构造）了。初始化列表是成员**诞生之前的最后一句话**——它决定了成员以什么参数出生。函数体里的代码只能「修改」已经出生的成员，而初始化列表能「定制」成员的出生过程。

知道了初始化列表能直接构造成员，一个很自然的想法是：如果我有多个成员，初始化列表里的顺序会不会影响结果？比如，我能不能用先初始化的成员去初始化后初始化的成员？

---

## 问题 2：初始化列表里的书写顺序真的就是执行顺序吗？

这是 C++ 中最经典的构造函数陷阱之一。看下面这段代码：

```cpp
// flags: -O0 -g
#include <iostream>

class Trap {
    int a_;
    int b_;
public:
    Trap(int x)
        : b_(x)      // 列表里先写 b_
        , a_(b_ + 1) // 再用 b_ 初始化 a_
    {
        std::cout << "a=" << a_ << ", b=" << b_ << "\n";
    }
};

int main() {
    Trap t(10);
    return 0;
}
```

直觉上，`b_` 先被初始化为 `10`，然后 `a_` 被初始化为 `11`。但如果你的编译器开了足够高的警告级别（比如 GCC 的 `-Wreorder`），它会发出警告：

```
warning: 'Trap::b_' will be initialized after 'Trap::a_' [-Wreorder]
```

**真相是：成员的初始化顺序永远由它们在类中的声明顺序决定，与初始化列表的书写顺序无关。**

在这个例子中，类里先声明了 `a_`，后声明了 `b_`，所以实际执行顺序是：
1. `a_(b_ + 1)` —— 此时 `b_` 还是未初始化的垃圾值
2. `b_(x)` —— 然后才把 `b_` 初始化为 `10`

`a_` 得到了一个未定义的值，程序行为是未定义行为（UB）。

验证一下：

```cpp
// flags: -O0 -g -Wall
#include <iostream>

class OrderDemo {
    int first_;   // 声明顺序：第 1
    int second_;  // 声明顺序：第 2
public:
    OrderDemo(int v)
        : second_(v)      // 列表里先写 second_
        , first_(second_) // 列表里后写 first_
    {
        std::cout << "first=" << first_ << ", second=" << second_ << "\n";
    }
};

int main() {
    OrderDemo o(42);
    return 0;
}
```

运行结果中 `first_` 的值大概率不是 `42`，而是一个随机数——因为 `first_` 在 `second_` 之前初始化，但它试图读取尚未初始化的 `second_`。

> [!warning] 黄金法则
> **初始化列表的书写顺序必须与成员声明顺序完全一致。** 这不是风格问题，而是正确性问题。不一致时，编译器只会给警告（如果你开了 `-Wreorder`），不会报错，但程序可能默默产生未定义行为。

为什么 C++ 要这样设计？因为析构顺序必须与构造顺序严格相反（后构造的先析构），而声明顺序是编译期就能确定的、唯一稳定的排序依据。如果允许初始化列表改变顺序，析构时就会面临「按哪个顺序析构」的歧义。

既然初始化顺序绑定在声明顺序上，那如果我想给成员一个「默认值」，但某些构造函数想覆盖它，该怎么办？每次都在初始化列表里重复写吗？

## 问题 3：默认成员初始化器与委托构造，怎么和初始化列表配合？

### 默认成员初始化器（C++11）

C++11 允许你在声明成员时就给它一个默认值：

```cpp
// flags: -O0 -g
#include <iostream>

class Config {
    int threads_ = 4;           // 默认成员初始化器
    bool debug_  = false;
    std::string name_ = "default";
public:
    Config() = default;         // threads_=4, debug_=false, name_="default"

    Config(int threads)         // threads_=传入值, debug_=false, name_="default"
        : threads_(threads)
    {}

    Config(int threads, bool debug, const std::string& name)  // 全部覆盖
        : threads_(threads)
        , debug_(debug)
        , name_(name)
    {}
};

int main() {
    Config c1;           // 全部默认值
    Config c2(8);        // 只覆盖 threads
    Config c3(16, true, "game");  // 全部覆盖
    return 0;
}
```

**优先级规则**：如果初始化列表显式指定了某个成员，默认成员初始化器被忽略；如果没指定，才使用默认成员初始化器。这跟函数默认参数的逻辑类似。

> [!tip] 引擎实践
> 在引擎中，类往往有很多配置成员（如缓冲区大小、线程数、标志位）。使用默认成员初始化器可以把「默认值」写在声明处，一眼就能看到，而不必在每个构造函数里重复。只有当某个构造函数需要特殊值时，才在初始化列表中覆盖。

但要注意：**默认成员初始化器和初始化列表同时存在时，初始化顺序仍然由声明顺序决定，与「哪个来源提供值」无关。**

### 委托构造（C++11）

当一个类有多个构造函数，且它们共享大量初始化逻辑时，C++11 允许一个构造函数「委托」给另一个构造函数：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

class Player {
    std::string name_;
    int level_;
    int hp_;
public:
    // 主构造函数：承担所有初始化责任
    Player(const std::string& name, int level, int hp)
        : name_(name)
        , level_(level)
        , hp_(hp)
    {
        std::cout << "Player: " << name_ << " Lv" << level_ << " HP" << hp_ << "\n";
    }

    // 委托构造：把具体初始化交给上面的主构造函数
    Player(const std::string& name)
        : Player(name, 1, 100)  // 必须是初始化列表里的唯一一项
    {}

    // 默认构造也委托
    Player()
        : Player("Unknown", 1, 100)
    {}
};

int main() {
    Player p1("Alice", 10, 500);
    Player p2("Bob");
    Player p3;
    return 0;
}
```

**委托构造的规则**：
- 委托构造的初始化列表**只能包含**对同类另一个构造函数的调用，不能同时初始化其他成员
- 委托构造完成后，才进入当前构造函数的函数体
- 如果委托链最终没有初始化所有成员（比如委托了一个不存在的构造函数），编译错误

这在引擎中非常实用：资源句柄类通常有「完整参数版」和「便捷版」构造函数，委托构造避免了初始化逻辑的重复和发散。

### 基类构造的顺序

最后提一下继承场景。如果类有基类，初始化列表里也需要（或隐式）调用基类构造函数：

```cpp
class Derived : public Base {
public:
    Derived(int x)
        : Base(x)   // 先构造 Base，再构造 Derived 的成员
        , member_(x)
    {}
};
```

完整的构造顺序是：
1. 基类构造函数（按继承声明顺序）
2. 成员构造函数（按成员声明顺序）
3. 当前构造函数体

析构时严格逆序。

## 总结

- **初始化列表是成员「诞生」的地方，函数体是「出生后做事」的地方**。const、引用、无默认构造的成员必须在初始化列表中初始化，否则编译失败。
- **初始化列表的效率更高**：直接构造 vs 默认构造+赋值，对于复杂类型能省去一次多余的构造和析构。
- **初始化顺序由声明顺序决定，与列表书写顺序无关**。书写顺序必须与声明顺序一致，否则可能读取未初始化的成员，产生未定义行为。
- **默认成员初始化器（C++11）** 适合给成员提供「通用默认值」，初始化列表中的显式指定会覆盖它。
- **委托构造（C++11）** 适合把多个构造函数的共同逻辑收敛到一个「主构造函数」，避免代码重复。
- **构造顺序**：基类 → 成员（按声明顺序）→ 构造函数体；析构严格逆序。

> [!note] 关键取舍
> SelfGameEngine 的组件存储追求零开销，初始化列表的「直接构造」特性与 `placement new` 批量构造完美配合——没有虚指针、没有默认构造的隐藏调用，每个字节和每个 CPU 周期都可预测。
