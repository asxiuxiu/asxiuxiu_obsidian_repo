# 构造函数：默认、显式、委托与继承构造

> [!info] 一句话概括
> 构造函数不只是「对象出生时的初始化函数」。默认构造、explicit、委托构造、继承构造这四项机制，分别回答了「对象如何被默认创建」「如何防止意外类型转换」「如何消除构造代码重复」「派生类如何正确初始化基类」四个工程问题。

---

## 问题 0：不写构造函数时，对象到底是怎么被创建出来的？

假设我正在写一个 `Vec2`：

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec2 {
    float x;
    float y;
};

int main() {
    Vec2 v;
    std::cout << v.x << ", " << v.y << '\n';
    return 0;
}
```

这段代码能编译，但运行后 `v.x` 和 `v.y` 的值是不确定的。因为它们没有被初始化，只是拿到了栈上那块内存里原本就有的「垃圾值」。

如果我把 `Vec2` 改成带默认值的类：

```cpp
struct Vec2 {
    float x = 0.0f;
    float y = 0.0f;
};
```

再创建 `Vec2 v;`，`x` 和 `y` 就会被初始化为 0。这里真正干活的是编译器自动生成的**默认构造函数（default constructor）**——一个不需要参数就能调用的构造函数。

默认构造函数解决了一个很朴素的问题：**我有时候只是想创建一个对象，不想每次都手动传初始化值。** 但前提是：要么类里有类内初始值（如上面的 `= 0.0f`），要么成员类型自己有默认构造函数（比如 `std::string`）。

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

struct Player {
    std::string name;  // string 有默认构造，会空初始化
    int hp = 100;      // 类内初始值
};

int main() {
    Player p;  // 调用编译器生成的默认构造
    std::cout << p.name << " hp=" << p.hp << '\n';
    return 0;
}
```

不过这里有个陷阱：**只要你自己写了任何构造函数，编译器就不再自动生成默认构造函数。** 比如我给 `Player` 加一个带参数的构造：

```cpp
struct Player {
    std::string name;
    int hp;
    Player(const std::string& n, int h) : name(n), hp(h) {}
};

Player p;  // 编译错误：没有默认构造函数
```

这引出了下一个问题：如果我既想保留「默认创建」的能力，又想提供自定义构造，该怎么办？

## 问题 1：单参数构造函数为什么需要 `explicit`？

现在我想给 `Vec2` 加一个「从标量创建」的构造函数，让 `Vec2 v(5.0f)` 表示 `(5.0f, 5.0f)`：

```cpp
struct Vec2 {
    float x, y;
    Vec2(float s) : x(s), y(s) {}
};
```

这看起来没问题，但它会引入一个隐式转换通道：

```cpp
void draw(Vec2 pos);

draw(3.0f);  // 编译通过！3.0f 被隐式转成 Vec2(3.0f, 3.0f)
```

更隐蔽的是：

```cpp
Vec2 v = 2.0f;  // 也编译通过
```

代码里明明写的是 `float`，编译器却悄悄调用了一次 `Vec2(float)` 构造。这种**隐式转换**在简单类型里还能接受，但在引擎代码里往往是灾难。比如 `String(int capacity)` 表示「预分配容量」的构造函数，结果 `String s = 0;` 被解释成「创建一个空字符串」，语义完全跑偏。

`explicit` 关键字就是用来堵住这个洞的。它告诉编译器：**这个构造函数只能被显式调用，不能用于隐式转换。**

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec2 {
    float x, y;
    explicit Vec2(float s) : x(s), y(s) {}
};

void draw(Vec2 pos) {
    std::cout << pos.x << ", " << pos.y << '\n';
}

int main() {
    Vec2 v1(3.0f);      // OK：显式调用
    // Vec2 v2 = 3.0f;  // 编译错误： explicit 禁止隐式转换
    // draw(3.0f);      // 编译错误
    draw(Vec2(3.0f));   // OK：显式构造后再传入
    return 0;
}
```

在引擎里，只要构造函数接收单个参数，就应该认真考虑是否需要 `explicit`。一个常见的准则是：**如果构造函数的参数不是「逻辑上等价于这个类型」的东西，就加 `explicit`。** 比如 `String(const char*)` 通常不加，因为 `const char*` 确实就是字符串的内容；但 `String(int)` 几乎一定要加。

那么，如果类有很多构造函数，每个里面都重复写着相似的初始化逻辑，该怎么处理？

## 问题 2：构造函数里的重复代码怎么消除？

假设 `Vec2` 支持多种构造方式：

```cpp
struct Vec2 {
    float x, y;

    Vec2() : x(0), y(0) {
        std::cout << "default\n";
    }

    Vec2(float s) : x(s), y(s) {
        std::cout << "scalar\n";
    }

    Vec2(float x_, float y_) : x(x_), y(y_) {
        std::cout << "two floats\n";
    }
};
```

每个构造函数都打印了一条日志。如果初始化逻辑更复杂——比如分配资源、检查不变式、注册到全局管理器——这段代码就会在多个构造函数里重复。一旦修改，就要改多处。

C++11 引入了**委托构造（delegating constructor）**：一个构造函数可以在初始化列表里调用同一个类的另一个构造函数，把「真正干活」的逻辑集中到一个地方。

```cpp
// flags: -O0 -g
#include <iostream>

struct Vec2 {
    float x, y;

    Vec2(float x_, float y_) : x(x_), y(y_) {
        std::cout << "主构造: (" << x << ", " << y << ")\n";
    }

    Vec2() : Vec2(0.0f, 0.0f) {
        std::cout << "委托到主构造\n";
    }

    explicit Vec2(float s) : Vec2(s, s) {}
};

int main() {
    Vec2 a;          // 先走 Vec2(float, float)，再走 Vec2()
    Vec2 b(5.0f);    // 走 Vec2(float, float)
    Vec2 c(1.0f, 2.0f);
    return 0;
}
```

委托构造的规则很严格：**委托构造的初始化列表里只能有一个目标构造，不能同时初始化成员。** 也就是说，下面这种写法是非法的：

```cpp
Vec2() : Vec2(0.0f, 0.0f), x(1.0f) {}  // 错误：委托构造时不能再初始化成员
```

因为目标构造已经负责初始化全部成员了，再写一次会造成语义冲突。

委托构造让「主构造函数」模式在 C++ 里成为可能：保留一个最全参数的构造函数，其他构造函数都委托给它。这和工厂函数有点像，但更轻量，不需要引入额外的静态函数。

那么，如果是派生类，构造函数该怎么处理基类部分？

## 问题 3：派生类构造时，基类部分由谁初始化？

考虑一个简单继承：

```cpp
struct Entity {
    int id;
    Entity(int id_) : id(id_) {}
};

struct Player : Entity {
    std::string name;
    Player(const std::string& n) : name(n) {}
};
```

这段代码会编译失败。因为 `Player` 的构造函数没有显式初始化基类 `Entity`，编译器会尝试调用 `Entity` 的默认构造函数，但 `Entity` 只有自己写的 `Entity(int)`，没有默认构造。

派生类构造的正确写法是在初始化列表里调用基类构造：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

struct Entity {
    int id;
    Entity(int id_) : id(id_) { std::cout << "Entity ctor\n"; }
};

struct Player : Entity {
    std::string name;
    Player(int id_, const std::string& n)
        : Entity(id_), name(n) {
        std::cout << "Player ctor\n";
    }
};

int main() {
    Player p(42, "Alice");
    std::cout << "id=" << p.id << " name=" << p.name << '\n';
    return 0;
}
```

执行顺序是：先 `Entity(int)`，再 `Player` 的成员 `name`，最后进入 `Player` 的构造函数体。这个顺序由 C++ 强制规定，和初始化列表里的书写顺序无关。

如果基类有多个构造函数，派生类可能希望「原封不动地继承」其中一些，让基类的构造在派生类里直接可用。C++11 提供了**继承构造（inheriting constructor）**：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

struct Entity {
    int id;
    Entity(int id_) : id(id_) { std::cout << "Entity(int)\n"; }
    Entity(const std::string& label) : id(std::stoi(label)) {
        std::cout << "Entity(string)\n";
    }
};

struct Player : Entity {
    using Entity::Entity;  // 继承 Entity 的所有构造函数
    std::string name;
};

int main() {
    Player p1(42);           // 调用继承来的 Entity(int)
    Player p2("100");        // 调用继承来的 Entity(string)
    std::cout << "p1.id=" << p1.id << " p2.id=" << p2.id << '\n';
    return 0;
}
```

注意：继承构造只是把基类的构造函数「暴露」到派生类的作用域里，它并不会初始化派生类自己的成员。`Player` 的 `name` 会由 `std::string` 自己的默认构造函数初始化。如果派生类有非默认可构造的成员，继承构造可能会失效。

继承构造在引擎里比较少见，因为大多数基类都需要派生类参与初始化（比如传递类型信息、注册到反射系统）。但它适合那种「基类就是全部状态，派生类只是加几个行为」的轻量级封装场景。

## 问题 4：默认构造函数什么时候会「消失」？

回到问题 0 的结尾：一旦我写了任何自定义构造函数，编译器就不再自动生成默认构造函数。这其实是 C++ 的一个默认策略：**只要你显式声明了构造方式，编译器就假设你已经接管了对象创建逻辑，不再替你瞎猜。**

```cpp
// flags: -O0 -g
#include <iostream>

struct Buffer {
    int* data;
    Buffer(int size) : data(new int[size]) {}
    ~Buffer() { delete[] data; }
};

int main() {
    // Buffer b;  // 编译错误：没有默认构造函数
    Buffer b(16);  // OK
    return 0;
}
```

这个设计是合理的：如果 `Buffer` 必须知道大小才能分配内存，那么「默认构造一个不知道大小的 Buffer」本身就是语义错误。编译器不自动生成默认构造，反而能避免隐藏 bug。

但有时候我确实需要一个默认构造，比如「空 Buffer」或「占位对象」。下一篇笔记会讲如何用 `= default` 显式要求编译器生成默认构造，以及如何用 `= delete` 彻底禁止某种构造。这里只需要记住：**默认构造不是免费的午餐，它会被你写的任何自定义构造函数抑制。**

---

## 总结

- **默认构造函数**让对象可以无参创建，但一旦被任何自定义构造函数抑制，就需要显式恢复或提供替代方案。
- **`explicit`** 堵住单参数构造函数的隐式转换通道，是引擎接口设计中的重要防御手段。
- **委托构造**把重复初始化逻辑收敛到一个主构造函数，避免代码重复和状态不一致。
- **继承构造**让派生类直接使用基类的构造函数，但派生类成员仍需自行可默认构造。
- 构造函数的调用顺序由语言强制规定：基类 → 成员 → 派生类构造体，和初始化列表书写顺序无关。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄类的显式构造 | `TextureHandle(int width, int height)` 通常标记 `explicit`，防止 `LoadTexture(1024)` 被误用 |
| **SelfGameEngine** | 数学类型的多路构造 | `Vec3()` 委托到 `Vec3(0, 0, 0)`，`Vec3(float)` 委托到 `Vec3(s, s, s)`，统一初始化路径 |
| **UE** | `UObject` 派生类的构造 | 生成的构造函数通过基类初始化列表调用 `UObject` 的构造，继承构造极少使用，因为反射注册需要派生类参与 |
| **UE** | `explicit` 防御宏生成 | UHT 生成的单参数构造函数常被标记 `explicit`，避免蓝图/C++ 边界处的隐式转换 |

> [!note] 关键取舍
> 在引擎代码中，**单参数构造函数默认加 `explicit`** 是一条安全准则。只有当参数类型在语义上就是该类型的「内容表示」时（如 `String(const char*)`、 `Vec2(std::initializer_list<float>)` 的语义化场景），才考虑省略。委托构造则适合把「零初始化 + 标量初始化 + 全量初始化」收敛到同一份逻辑，避免拷贝控制中的重复代码。
