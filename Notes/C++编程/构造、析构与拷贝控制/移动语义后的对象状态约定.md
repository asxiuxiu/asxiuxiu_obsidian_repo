# 移动语义后的对象状态约定

> [!info] 一句话概括
> C++ 的移动操作只是把资源所有权从源对象「搬」到目标对象；搬完之后，源对象仍必须处于**合法但值未指明**的状态，保证可以安全析构和重新赋值，但不能假定它还有原来的值。

---

## 问题 0：移动后源对象还能继续用吗？

假设你写了一个 `String` 类，内部管理一块动态内存。某天你写出这样的代码：

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>
#include <string>

int main() {
    std::string a = "hello";
    std::string b = std::move(a);

    std::cout << "b: " << b << "\n";
    std::cout << "a: " << a << "\n"; // 这里安全吗？

    a = "world"; // 还能赋值吗？
    std::cout << "a again: " << a << "\n";
    return 0;
}
```

运行后你会发现，`a` 可能打印空串，也可能打印 `"hello"`，具体取决于标准库实现。但**这段代码不是未定义行为**——`a` 仍然是合法对象，可以析构、可以赋值，只是它的值不再由你控制。

> [!abstract]
> C++ 标准规定：除非类型文档另有说明，否则移动构造或移动赋值后，源对象处于**合法但未指明的状态（valid but unspecified state）**。也就是说，所有不依赖具体值的操作（析构、赋值、清空）都是安全的；但读取它的当前值没有意义。

---

## 问题 1：标准说的「可析构保证」到底是什么？

容器扩容时会把元素从旧内存移动到新内存。如果移动后源对象的析构函数会崩溃，那么扩容就会直接失败。因此，**移动操作必须保证源对象能被安全析构**。

下面是一个最小化的 `Buffer` 实现，它在移动后把源对象置成空状态：

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>
#include <utility>

class Buffer {
public:
    explicit Buffer(size_t n) : ptr_(new int[n]), size_(n) {}
    ~Buffer() { delete[] ptr_; }

    // 移动构造：偷走资源，把源对象置空
    Buffer(Buffer&& other) noexcept : ptr_(other.ptr_), size_(other.size_) {
        other.ptr_ = nullptr;
        other.size_ = 0;
    }

    // 移动赋值：释放旧资源，再偷走新资源
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] ptr_;
            ptr_ = other.ptr_;
            size_ = other.size_;
            other.ptr_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }

    size_t size() const { return size_; }

private:
    int* ptr_;
    size_t size_;
};

int main() {
    Buffer a(100);
    Buffer b(std::move(a));

    std::cout << "b.size=" << b.size() << "\n";
    std::cout << "a.size=" << a.size() << "\n"; // 未指明，但这里显示为 0

    a = Buffer(10); // 重新赋值安全
    std::cout << "after assign a.size=" << a.size() << "\n";
    return 0;
}
```

这里的关键是：移动后 `other.ptr_` 变成了 `nullptr`，析构 `delete[] nullptr` 是安全的。如果忘记置空，`a` 和 `b` 的析构函数会各执行一次 `delete[] ptr_`，同一块内存被释放两次，触发未定义行为。

---

## 问题 2：为什么通常要把源对象「置空」？

最直接的回答是：**为了避免双重释放**。但深层原因和所有权语义有关：移动意味着资源所有权已经转移，源对象不再拥有那份资源。把指针设为 `nullptr`、把句柄设为无效值，是表达「我不再拥有它」的最自然方式。

看一个反例，移动后没有置空：

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>

// 错误示范：移动后没有重置源对象
class BadBuffer {
public:
    explicit BadBuffer(size_t n) : ptr_(new int[n]) {}
    ~BadBuffer() { delete[] ptr_; }

    BadBuffer(BadBuffer&& other) noexcept : ptr_(other.ptr_) {
        // ❌ 没有把 other.ptr_ 置空
    }

private:
    int* ptr_;
};

int main() {
    BadBuffer a(10);
    BadBuffer b(std::move(a));
    // a 和 b 的 ptr_ 指向同一块内存，析构时双重释放 → UB
    return 0;
}
```

> [!warning]
> 不是所有 moved-from 对象都必须为空。标准只要求「合法且可析构」。例如 `std::unique_ptr` 移动后确实为 `nullptr`，但 `std::string` 可能保留一个未指明的内部缓冲区。具体怎么置空，由类型的不变式决定。

## 问题 3：自移动赋值怎么办？

如果写出 `b = std::move(b);`，源对象和目标对象是同一个。移动赋值运算符必须在不破坏对象的前提下安全处理这种情况。

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>
#include <utility>

class Buffer {
public:
    explicit Buffer(size_t n) : ptr_(new int[n]), size_(n) {}
    ~Buffer() { delete[] ptr_; }

    Buffer(Buffer&& other) noexcept
        : ptr_(std::exchange(other.ptr_, nullptr))
        , size_(std::exchange(other.size_, 0)) {}

    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] ptr_;
            ptr_ = std::exchange(other.ptr_, nullptr);
            size_ = std::exchange(other.size_, 0);
        }
        return *this;
    }

    size_t size() const { return size_; }

private:
    int* ptr_;
    size_t size_;
};

int main() {
    Buffer b(50);
    b = std::move(b); // 自移动赋值
    std::cout << "b.size=" << b.size() << "\n"; // 仍然是 50
    return 0;
}
```

> [!tip]
> 自移动赋值检查通常写成 `if (this != &other)`。如果不检查，会先 `delete[] ptr_`（也就是 `other.ptr_`），然后再从已经释放的地址复制数据，造成 use-after-free。

## 问题 4：`MoveInsertable` 是什么？

容器（如 `std::vector`）扩容时需要把旧元素搬到新内存。它不会调用拷贝构造，而是优先调用**移动构造**。C++ 标准把「能在容器指定位置通过右值构造对象」的能力称为 **MoveInsertable**。

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>
#include <utility>
#include <vector>

class Buffer {
public:
    explicit Buffer(size_t n) : ptr_(new int[n]), size_(n) {
        std::cout << "construct " << size_ << "\n";
    }
    ~Buffer() { delete[] ptr_; }

    Buffer(Buffer&& other) noexcept
        : ptr_(std::exchange(other.ptr_, nullptr))
        , size_(std::exchange(other.size_, 0)) {
        std::cout << "move construct\n";
    }

    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] ptr_;
            ptr_ = std::exchange(other.ptr_, nullptr);
            size_ = std::exchange(other.size_, 0);
        }
        return *this;
    }

private:
    int* ptr_;
    size_t size_;
};

int main() {
    std::vector<Buffer> v;
    v.reserve(2);
    v.emplace_back(10);
    v.emplace_back(20);

    std::cout << "---- reallocate ----\n";
    v.reserve(100); // 触发重新分配，元素被移动
    return 0;
}
```

`std::vector` 在重新分配时会用 `std::move` 把旧元素搬到新内存。如果 `Buffer` 的移动构造函数标记为 `noexcept`，扩容会直接使用移动；如果可能抛异常，为了保证强异常安全，某些实现会回退到拷贝。这也是为什么移动构造函数通常要加 `noexcept`。

## 问题 5：`std::exchange` 在移动操作中有什么用？

`std::exchange(obj, new_val)` 是一个小工具：它把 `obj` 的旧值返回，同时把 `obj` 设为新值。在移动操作中，它可以「一次性」完成资源转移和源对象重置，避免手动写两遍赋值。

```cpp
// flags: -std=c++20 -Wall -O2
#include <iostream>
#include <utility>

class Handle {
public:
    explicit Handle(int id) : id_(id) {}

    Handle(Handle&& other) noexcept
        : id_(std::exchange(other.id_, -1)) {}

    Handle& operator=(Handle&& other) noexcept {
        if (this != &other) {
            id_ = std::exchange(other.id_, -1);
        }
        return *this;
    }

    int id() const { return id_; }

private:
    int id_;
};

int main() {
    Handle a(42);
    Handle b(std::move(a));

    std::cout << "b.id=" << b.id() << "\n";
    std::cout << "a.id=" << a.id() << "\n"; // -1，表示已无效
    return 0;
}
```

> [!abstract]
> `std::exchange` 的签名可以简单理解为 `T exchange(T& obj, U&& new_val)`：返回 `obj` 的旧值，并把 `obj` 设为 `new_val`。它不仅用于移动语义，也常用于实现「先取值再重置」的惯用法，比如 `std::atomic::exchange`、状态机切换。

## 总结

- 移动操作转移资源所有权，移动后的源对象进入**合法但未指明**的状态。
- 对 moved-from 对象能做的安全操作通常只有：析构、赋值、清空；读取其具体值没有意义。
- 实现移动操作时，**必须保证源对象可析构**，最常见的做法是把内部资源句柄置空。
- **自移动赋值必须安全**，通常通过 `this != &other` 判断避免自我破坏。
- 容器的 `MoveInsertable` 概念依赖类型的移动构造函数；`noexcept` 移动能让容器扩容走移动路径而非拷贝路径。
- `std::exchange` 能简化「转移并置空」的代码，减少手动赋值出错的可能。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄、自定义容器扩容 | `TextureHandle`、`Buffer` 等 RAII 句柄的移动构造会把源句柄置为无效；`Array` 扩容时用 `std::move` 迁移元素，要求元素类型 MoveInsertable 且 `noexcept` |
| **UE** | `TUniquePtr`、`TArray` 重分配 | `TUniquePtr` 移动后源指针为 `nullptr`；`TArray` 扩容时若元素移动构造函数 `noexcept` 则使用移动，否则回退拷贝以保证异常安全 |
| **Bevy/现代 ECS** | 组件移动与内存迁移 | ECS 批量迁移组件时依赖类型的移动语义；moved-from 组件必须可析构，否则会造成内存泄漏或重复释放 |

> [!note] 关键取舍
> 移动语义的收益来自「 steal 资源」，但代价是必须管理 moved-from 状态。对于引擎中高频创建/销毁的资源句柄，统一约定「移动后置空」是最不容易出错的设计；对于值语义对象（如数学向量），移动和拷贝往往没有区别，也无需特殊处理。
