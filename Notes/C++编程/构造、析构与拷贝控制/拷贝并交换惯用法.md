# 拷贝并交换惯用法

> [!info] 一句话概括
> 拷贝并交换（Copy-and-Swap）用「先拷贝到临时对象，再交换」的策略，把拷贝赋值的异常安全和自赋值处理**委托给拷贝构造和 swap**——你不需要写自赋值检查，也不需要担心 `new` 抛异常后对象变成残骸。

---

## 问题 0：「先 delete 再 new」的拷贝赋值，在异常面前有多脆弱？

回顾我们之前写的「看起来正确」的拷贝赋值：

```cpp
UnsafeString& operator=(const UnsafeString& other) {
    if (this == &other) return *this;  // 自赋值检查
    delete[] data_;                     // 释放旧资源
    len_ = other.len_;
    data_ = new char[len_ + 1];         // 分配新资源
    std::strcpy(data_, other.data_);    // 复制内容
    return *this;
}
```

这段代码处理了自赋值，但还留着一个暗雷：**如果 `new char[len_ + 1]` 抛出 `std::bad_alloc`（内存不足），对象会变成什么状态？**

执行流程：
1. `delete[] data_` —— 旧内存已释放
2. `len_ = other.len_` —— 长度已更新
3. `new char[len_ + 1]` —— **抛出异常！**
4. `data_` 指向已经被释放的旧地址（悬垂指针）
5. `len_` 已经是新值，但 `data_` 没有对应的新内存

此时 `*this` 处于一个**部分更新的损坏状态**：调用者捕获异常后，如果继续用这个对象（比如再次赋值、或者等它析构），行为是未定义的。析构时会 `delete[] data_` —— 对同一块内存第二次释放（因为 `data_` 还指向之前 `delete[]` 过的地址）。

这个问题在引擎中尤其危险：加载一个大纹理或网格时，如果内存分配失败，你不仅没加载成功，还把之前持有的资源给毁了，游戏可能直接崩溃。

有没有一种写法，能保证「要么全成功，要么全不变」？

---

## 问题 1：什么是异常安全保证等级？

在讲解决方案之前，先明确目标。C++ 中异常安全分为三个等级：

| 等级 | 含义 | 类比 |
|------|------|------|
| **基本保证（Basic Guarantee）** | 异常发生后，对象不泄漏资源，不破坏不变式，但状态可能已经改变 | 转账失败，钱没丢，但可能扣了手续费 |
| **强保证（Strong Guarantee）** | 异常发生后，对象状态**完全不变**，就像操作从未发生过 | 转账失败，双方账户都回到操作前状态 |
| **不抛异常（No-Throw Guarantee）** | 操作绝对不会抛异常 | 用现金交易，没有中间环节能失败 |

我们刚才的「先 delete 再 new」版本，连**基本保证**都达不到：异常发生后对象处于损坏状态，析构时 double-free。

加上自赋值检查的版本好一些，但如果 `new` 抛异常，仍然连基本保证都守不住。

拷贝并交换惯用法的目标就是：用**拷贝构造**（可能抛异常，但抛异常时不影响原对象）+ **noexcept swap**（绝不抛异常）来实现**强异常安全**。

---

## 问题 2：拷贝并交换惯用法是怎么工作的？

核心思路出奇地简单：

```cpp
SafeString& operator=(SafeString other) {  // 注意：传值，不是 const&！
    swap(other);                            // 与临时对象交换
    return *this;                           // 临时对象离开作用域，自动析构旧资源
}
```

三行代码，但它背后发生了这些事情：

1. **参数传值**：`other` 是用**拷贝构造**创建的局部副本。如果拷贝构造抛异常（比如内存不足），异常发生在 `operator=` 函数体外，`*this` **完全不受影响**。
2. **noexcept swap**：`swap(*this, other)` 只交换指针和整数值，不涉及内存分配，标记为 `noexcept`。这一步**绝对不会失败**。
3. **临时对象析构**：函数结束时，`other`（现在持有 `*this` 的旧资源）自动析构，释放旧内存。

用图来表示：

```
调用前：
  this → [data: "Hello" | len: 5]
  other（参数传值拷贝 other_in）→ [data: "World" | len: 5]

swap 后：
  this → [data: "World" | len: 5]
  other → [data: "Hello" | len: 5]

函数返回，other 析构：
  this → [data: "World" | len: 5]   ✅ 成功获得新值
  "Hello" 的内存被 other 的析构释放   ✅ 旧资源正确回收
```

如果拷贝构造在步骤 1 抛异常：

```
拷贝构造抛异常：
  this → [data: "Hello" | len: 5]   ✅ 完全没变
  other 不存在（拷贝构造失败，没有创建出来）
```

这就是**强异常安全**——要么成功替换，要么状态完全不变。

### 完整实现示例

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>
#include <utility>

class SafeString {
    char* data_;
    size_t len_;

    // 私有辅助：交换两个 SafeString 的内部状态
    void swap(SafeString& other) noexcept {
        using std::swap;
        swap(data_, other.data_);
        swap(len_, other.len_);
    }

public:
    SafeString(const char* str = "")
        : len_(std::strlen(str))
        , data_(new char[len_ + 1])
    {
        std::strcpy(data_, str);
    }

    // 拷贝构造
    SafeString(const SafeString& other)
        : len_(other.len_)
        , data_(new char[other.len_ + 1])
    {
        std::strcpy(data_, other.data_);
    }

    // 移动构造
    SafeString(SafeString&& other) noexcept
        : data_(other.data_)
        , len_(other.len_)
    {
        other.data_ = nullptr;
        other.len_ = 0;
    }

    // 拷贝并交换惯用法：拷贝赋值
    SafeString& operator=(SafeString other) {
        swap(other);
        return *this;
    }

    // 移动赋值（也可以用 swap 实现）
    SafeString& operator=(SafeString&& other) noexcept {
        if (this != &other) {
            SafeString tmp(std::move(other));
            swap(tmp);
        }
        return *this;
    }

    ~SafeString() {
        delete[] data_;
    }

    const char* c_str() const { return data_; }
    size_t length() const { return len_; }
};

int main() {
    SafeString a("Hello");
    SafeString b("World");

    b = a;                    // 拷贝赋值
    std::cout << b.c_str() << "\n";

    b = b;                    // 自赋值：完全安全！
    std::cout << b.c_str() << "\n";

    SafeString c("Temporary");
    c = SafeString("Moved");  // 移动赋值
    std::cout << c.c_str() << "\n";

    return 0;
}
```

> [!tip] 自赋值自动安全
> 注意拷贝赋值的参数是 `SafeString other`（传值），不是 `const SafeString&`。这意味着 `b = b` 时：
> 1. `other` 是 `b` 的副本（拷贝构造自己，成功）
> 2. `swap` 交换两个完全相同的副本（无变化）
> 3. `other` 析构释放副本的内存（和 `b` 原来的内存内容相同，但地址不同）
> 结果：`b` 的状态完全不变。**不需要显式写 `if (this == &other)` 检查！**

## 问题 3：为什么 swap 必须标记为 noexcept？

拷贝并交换的安全依赖于一个前提：**swap 操作绝对不能抛异常**。如果 swap 抛异常，强保证就破了——因为 swap 执行到一半失败时，`*this` 和临时对象的状态可能都处于中间态。

幸运的是，绝大多数类型的 swap 都只是交换几个指针和标量值：

```cpp
void swap(SafeString& other) noexcept {
    using std::swap;
    swap(data_, other.data_);  // 交换两个 char* 指针，不可能抛异常
    swap(len_, other.len_);    // 交换两个 size_t，不可能抛异常
}
```

`std::swap` 在 C++11 起被标记为 `noexcept`（当交换的类型是 `noexcept` 可移动时）。自定义 swap 时，**务必加上 `noexcept`**——这是强异常安全的契约基础。

> [!warning] 如果 swap 不能 noexcept 怎么办？
> 如果类的 swap 涉及可能抛异常的操作（比如交换两个可能分配内存的标准容器），拷贝并交换就无法提供强保证。此时只能退而求其次追求基本保证，或者重新设计类的内部表示，让 swap 变为 noexcept（比如用指针间接层，真正交换的只是指针）。

## 问题 4：拷贝并交换还有什么额外的好处？

### 代码复用

拷贝并交换让拷贝赋值**复用**了拷贝构造和析构的实现。你只需要确保拷贝构造是正确的，拷贝赋值就自动正确。不需要在两个地方维护「分配内存 + 复制内容」的逻辑。

### 自然支持移动赋值

如果已经实现了移动构造和 noexcept swap，移动赋值也可以直接用 swap 实现：

```cpp
SafeString& operator=(SafeString&& other) noexcept {
    SafeString tmp(std::move(other));  // 移动构造临时对象
    swap(tmp);                          // noexcept swap
    return *this;
}
```

或者更简洁地，利用参数传值的版本已经能处理移动：

```cpp
// 如果调用者传右值，C++11 会先尝试移动构造参数
SafeString& operator=(SafeString other) {  // 右值 → 移动构造参数
    swap(other);
    return *this;
}
```

> 但这种「统一版本」有一个性能细节：移动构造参数 + swap 比直接移动赋值多一次 swap 操作。对于只包含几个指针的类型，swap 开销可以忽略；但对于大型对象，可能需要单独实现移动赋值。

### 一致性

拷贝并交换强制你实现一个正确的拷贝构造和一个 noexcept swap——这两个操作本身也是有用的：
- 拷贝构造：函数传值、返回对象、容器扩容时都需要
- swap：标准算法（`std::sort`、自定义排序器）中经常用到

---

## 总结

- **拷贝并交换的核心公式**：`operator=(T other) { swap(other); return *this; }`——参数传值创建副本，noexcept swap 交换状态，临时对象析构释放旧资源。
- **强异常安全**：拷贝构造可能抛异常，但抛异常时原对象完全不受影响；swap 是 noexcept，一旦拷贝成功，后续操作绝不失败。
- **自赋值自动安全**：不需要显式 `if (this == &other)` 检查，因为拷贝自己生成独立副本，swap 后状态不变。
- **swap 必须 noexcept**：这是强保证的契约基础。如果 swap 可能抛异常，整个惯用法退化为基本保证。
- **额外收益**：拷贝赋值复用拷贝构造和析构的实现；统一了异常安全和资源回收逻辑；为移动赋值提供了自然实现路径。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 自定义容器的赋值实现 | 引擎的 `Array<T>`、`String`、`HashMap` 等容器在实现拷贝赋值时普遍采用拷贝并交换，确保在内存不足等异常场景下容器状态不被破坏。`swap` 通常只交换指针和容量计数器，保证 noexcept |
| **UE** | `TArray` 等容器的赋值操作 | `TArray::operator=` 的实现使用了类似的策略：先分配新内存并复制元素，成功后再释放旧内存。如果中途失败，原数组数据保持完整。UE 在禁用异常的环境下通过返回值和断言处理错误，但「先成功再替换」的逻辑与拷贝并交换一脉相承 |

> [!note] 关键取舍
> SelfGameEngine 若使用异常（或至少保留异常安全思维），拷贝并交换是容器赋值的标准实现。即使在禁用异常的环境中（如 UE），「先构建新状态，再原子性替换旧状态」的逻辑仍然是避免数据损坏的核心策略——只是错误处理从异常变成返回值检查和断言。
> 对于只移动（Move-Only）的类型（如 GPU 资源句柄），拷贝并交换不适用（没有拷贝构造），但 `noexcept swap` 的思想仍然体现在移动赋值中：先移动构造临时对象，再 swap，最后让临时对象析构释放旧资源。
