# 拷贝构造函数与拷贝赋值运算符

> [!info] 一句话概括
> 拷贝构造是「以现有对象为模板制造新对象」，拷贝赋值是「用现有对象覆盖已有对象」——两者面临的共同挑战是：当对象内部管理着堆内存或外部资源时，编译器自动生成的「逐成员复制」会变成**浅拷贝**，引发双重释放或资源竞争。

---

## 问题 0：直接用 = 复制一个对象，到底发生了什么？

假设你在写一个简易的字符串类，内部用 `char*` 管理动态内存：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class NaiveString {
    char* data_;
    size_t len_;
public:
    NaiveString(const char* str) {
        len_ = std::strlen(str);
        data_ = new char[len_ + 1];
        std::strcpy(data_, str);
    }

    ~NaiveString() {
        delete[] data_;
    }
};

int main() {
    NaiveString a("Hello");
    NaiveString b = a;  // 这行代码到底做了什么？
    return 0;
}
```

如果你编译运行这段代码，程序大概率会在退出时崩溃，或者输出乱码。为什么？

因为 `NaiveString b = a;` 触发的是**拷贝构造**——编译器自动生成的拷贝构造函数会**逐成员复制**：把 `a.data_` 的地址值复制给 `b.data_`，把 `a.len_` 的值复制给 `b.len_`。结果是：`a` 和 `b` 的 `data_` 指针指向了**同一块堆内存**。

当 `main` 函数结束时，局部对象按构造的逆序析构：
1. 先析构 `b`：`delete[] data_` —— 释放了那块内存
2. 再析构 `a`：`delete[] data_` —— **同一块内存又被释放了一次**

这就是**双重释放（double-free）**，是未定义行为。轻则程序崩溃，重则内存损坏引发安全漏洞。

这个问题的根源是**浅拷贝（shallow copy）**：复制了指针的值，但没有复制指针指向的内容。对于只包含基本类型（`int`、`float`）或值类型（`std::string`、`std::vector`）的对象，编译器自动生成的逐成员拷贝完全够用；但一旦类内部管理着**动态分配的资源**（堆内存、文件句柄、GPU 资源 ID 等），浅拷贝就会爆炸。

那么，怎么区分「什么时候该让编译器自动生成，什么时候必须手写」？又该怎么手写？

---

## 问题 1：拷贝构造和拷贝赋值，到底在哪些场景下被调用？

很多人分不清拷贝构造和拷贝赋值，因为它们的语法都用了 `=`。但实际上，**一个是「制造新对象」，一个是「修改老对象」**。

### 拷贝构造函数（Copy Constructor）

**签名**：`ClassName(const ClassName& other);`

触发场景：**用一个已有对象去初始化一个全新的对象**。

```cpp
// flags: -O0 -g
#include <iostream>

class Demo {
public:
    Demo() { std::cout << "default ctor\n"; }
    Demo(const Demo&) { std::cout << "copy ctor\n"; }
    Demo& operator=(const Demo&) { std::cout << "copy assign\n"; return *this; }
};

Demo makeDemo() {
    Demo d;
    return d;  // 可能触发拷贝构造（NRVO 优化前）
}

int main() {
    Demo a;        // 默认构造
    Demo b = a;    // 拷贝构造（b 是全新对象，用 a 初始化它）
    Demo c(a);     // 拷贝构造（显式写法）
    Demo d = makeDemo();  // 可能是拷贝构造，也可能是移动构造/优化省略
    return 0;
}
```

关键点：**`Demo b = a;` 是拷贝构造，不是赋值**——尽管写了 `=` 符号，但因为 `b` 在这里第一次被创建，所以调用的是构造函数，不是赋值运算符。

### 拷贝赋值运算符（Copy Assignment Operator）

**签名**：`ClassName& operator=(const ClassName& other);`

触发场景：**两个已经存在的对象之间用 = 进行赋值**。

```cpp
// flags: -O0 -g
#include <iostream>

class Demo {
public:
    Demo() { std::cout << "default ctor\n"; }
    Demo(const Demo&) { std::cout << "copy ctor\n"; }
    Demo& operator=(const Demo&) { std::cout << "copy assign\n"; return *this; }
};

int main() {
    Demo a;
    Demo b;
    b = a;  // 两个对象都已存在 → 拷贝赋值
    return 0;
}
```

### 快速区分法则

| 场景 | 调用的是 |
|------|---------|
| `Demo b = a;`（b 尚未存在） | **拷贝构造** |
| `Demo b(a);` | **拷贝构造** |
| `b = a;`（b 已存在） | **拷贝赋值** |
| 函数参数传值 `void f(Demo d)` | **拷贝构造** |
| 函数返回值 `return d;` | **拷贝构造**（可能被优化省略） |

> [!tip] 核心记忆法
> 只要等号左边是一个**新诞生的对象**，就是构造；如果等号两边都是**已经活着的对象**，就是赋值。

## 问题 2：编译器什么时候自动生成拷贝控制函数？自动生成的版本做了什么？

### 旧规则（C++98/03）

在 C++11 之前，只要你没有声明拷贝构造或拷贝赋值，编译器就会**无条件**为你生成一个。生成的版本执行**逐成员拷贝（member-wise copy）**：对每个成员调用它自己的拷贝构造或拷贝赋值。

这听起来很贴心，但问题是：**对于指针成员，逐成员拷贝就是浅拷贝**——只复制地址，不复制内容。

### 新规则（C++11 及以后）

C++11 引入了移动语义，同时改变了编译器自动生成拷贝控制函数的规则。现在，编译器**不会**在以下情况下生成默认的拷贝构造/拷贝赋值：

- 类声明了移动构造或移动赋值
- 类有引用成员或 const 成员（无法拷贝赋值）
- 类成员的拷贝构造/拷贝赋值被删除或不可访问
- 类有基类的拷贝构造/拷贝赋值被删除或不可访问

> 这些规则的细节将在 [[Notes/C++编程/构造、析构与拷贝控制/Rule of Three Five Zero|Rule of Three / Five / Zero]] 中系统整理。

### 验证编译器生成行为

```cpp
// flags: -O0 -g -std=c++20
#include <iostream>

class HasReference {
    int& ref_;  // 引用成员
public:
    HasReference(int& r) : ref_(r) {}
    // 编译器不会生成拷贝赋值，因为引用不能重新绑定
};

class HasUniquePtr {
    // 假设有一个不可拷贝的成员
    HasReference* ptr_;
public:
    HasUniquePtr(HasReference* p) : ptr_(p) {}
    // 如果成员类型删除了拷贝操作，编译器也不会生成
};

int main() {
    int x = 42;
    HasReference a(x);
    // HasReference b = a;  // ❌ C++11 起：拷贝构造被删除或不可用
    return 0;
}
```

在实际工程中，**不要依赖编译器自动生成**——如果你的类管理着资源，就应该显式声明拷贝控制函数（或者显式删除它们，表达「不可拷贝」的意图）。

## 问题 3：手写拷贝赋值时，为什么要处理自赋值？

现在我们知道，对于管理动态内存的类，必须手写拷贝控制函数。先看一个「看起来正确」的拷贝赋值实现：

```cpp
class UnsafeString {
    char* data_;
    size_t len_;
public:
    // ... 构造、析构、拷贝构造省略 ...

    UnsafeString& operator=(const UnsafeString& other) {
        delete[] data_;           // 先释放自己的旧资源
        len_ = other.len_;
        data_ = new char[len_ + 1];
        std::strcpy(data_, other.data_);
        return *this;
    }
};
```

这个实现有个致命的漏洞：**自赋值（self-assignment）**。

```cpp
UnsafeString s("Hello");
s = s;  // 自赋值！
```

执行流程：
1. `delete[] data_` —— 释放了 `s` 自己的内存
2. `other.data_` —— 但 `other` 就是 `s` 自己！所以 `other.data_` 现在指向**已经被释放的内存**
3. `std::strcpy(data_, other.data_)` —— **读取已释放的内存** → 未定义行为

修复方案很简单：在开头加一个自赋值检查：

```cpp
SafeString& operator=(const SafeString& other) {
    if (this == &other) {  // 自赋值检查
        return *this;
    }
    delete[] data_;
    len_ = other.len_;
    data_ = new char[len_ + 1];
    std::strcpy(data_, other.data_);
    return *this;
}
```

但这还不是最安全的写法——如果 `new char[]` 抛出 `std::bad_alloc` 异常，`this->data_` 已经被 `delete[]` 了，对象处于一个**部分销毁的无效状态**。这个问题需要「拷贝并交换惯用法」来解决，详见 [[Notes/C++编程/构造、析构与拷贝控制/拷贝并交换惯用法|拷贝并交换惯用法]]。

## 问题 4：参数和返回值的引用，为什么缺一不可？

看拷贝构造和拷贝赋值的签名：

```cpp
MyClass(const MyClass& other);           // 拷贝构造
MyClass& operator=(const MyClass& other); // 拷贝赋值
```

### 为什么参数必须是 `const MyClass&`？

**不能传值**：如果参数写成 `MyClass other`（传值），调用拷贝构造时会发生什么？你需要把一个 `MyClass` 对象传进函数 → 这需要调用拷贝构造来复制参数 → 而这个拷贝构造的参数又是传值 → 又需要调用拷贝构造……**无限递归**。

**不能省略 const**：拷贝语义承诺「源对象在拷贝过程中不被修改」。如果参数是 `MyClass&`，你不仅违背了拷贝的直觉语义，还无法接收临时对象或 const 对象。

### 为什么拷贝赋值返回 `MyClass&`？

为了支持**链式赋值**：`a = b = c;`

赋值运算符的语义是「把右边的值赋给左边，然后返回左边的引用」。如果返回 `void`，链式赋值就断掉了。返回引用（不是值）则避免了不必要的拷贝构造。

## 总结

- **拷贝构造**用于「用已有对象初始化新对象」；**拷贝赋值**用于「两个已有对象之间的赋值」。两者都用 `=` 语法，但前者发生在对象诞生时，后者发生在对象已存在时。
- **浅拷贝**复制指针地址而非内容，会导致双重释放、悬垂指针等问题；**深拷贝**为指针成员独立分配新内存并复制内容。
- 编译器自动生成的拷贝控制函数执行逐成员拷贝。C++11 起，如果类含有不可拷贝成员、引用成员、const 成员，或声明了移动操作，编译器**可能不再生成**默认拷贝函数。
- **自赋值检查**（`if (this == &other) return *this;`）是拷贝赋值的基本安全要求，但仅靠它还不能保证强异常安全。
- 拷贝控制的参数必须是 `const&`，返回值必须是引用——这是语法和语义的双重约束。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄类的拷贝策略 | 渲染资源（`Texture`、`Buffer`）通常持有 GPU 句柄，拷贝意味着创建新的 GPU 资源副本，开销极大。因此引擎中大量使用 `= delete` 禁用拷贝，强制使用移动语义或引用计数；仅当资源本身支持廉价复制（如小尺寸 Uniform Block）时才实现深拷贝 |
| **UE** | `UObject` 的不可拷贝设计 | `UObject` 及其派生类**禁止值拷贝**——没有拷贝构造/拷贝赋值，复制对象必须通过编辑器序列化或 `DuplicateObject` 实现。这是因为 `UObject` 与 GC、反射、蓝图系统深度绑定，值拷贝会破坏对象身份和引用关系 |

> [!note] 关键取舍
> SelfGameEngine 中，轻量级值类型（`Vec3`、`Mat4`、`Color`）全部由编译器自动生成拷贝控制，逐成员拷贝就是正确行为；而管理外部资源（GPU 句柄、文件描述符、锁）的类则显式 `= delete` 拷贝，强制程序员思考「转移还是共享」。
> UE 的 `UObject` 体系则从根本上取消了值语义，所有对象都在堆上、由 GC 管理，拷贝只能通过专门的克隆 API 完成——这是大型编辑器工具链的必然选择，但代价是失去了值类型的性能和可预测性。
