# 类的作用域与友元

> [!info] 一句话概括
> 类本身也是一个作用域，成员名字在这个作用域内被管理；**友元（`friend`）**则是在不破坏整体封装的前提下，给特定外部函数或类开一扇"后门"，让它能访问私有成员。

---

## 问题 0：如果类的成员直接放在全局作用域里，会出什么问题？

假设我们在写一个图形引擎，需要表示"纹理句柄"和"网格句柄"。如果不利用类作用域，我们可能会这样写：

```cpp
// flags: -O0 -g
#include <iostream>

struct TextureHandle {
    int id;
};

struct MeshHandle {
    int id;
};

// 为了区分，不得不在全局加前缀
bool IsTextureValid(TextureHandle h) { return h.id >= 0; }
bool IsMeshValid(MeshHandle h) { return h.id >= 0; }
```

两个不相关的类型都可能有 `id` 字段和 `IsValid` 函数。如果没有类作用域隔离，命名冲突会迅速爆炸：你要么给每个函数加前缀（`Texture_IsValid`、`Mesh_IsValid`），要么祈祷没有重名。

C++ 的解决方式很自然：**把成员和关联函数放进类里**。类本身就是一个作用域。

---

## 问题 1：类作用域到底是什么？它和普通的命名空间有什么区别？

类作用域意味着：**类名就是成员的命名空间**。你可以通过 `类名::成员名` 访问静态成员、嵌套类型和成员函数；非静态成员则需要通过对象来访问。

```cpp
// flags: -O0 -g
#include <iostream>

class TextureHandle {
    int id_;
public:
    explicit TextureHandle(int id) : id_(id) {}

    bool isValid() const { return id_ >= 0; }
    int getId() const { return id_; }

    // 静态成员：属于类，不属于某个对象
    static constexpr int InvalidId = -1;
};

int main() {
    TextureHandle t(5);
    std::cout << "valid: " << t.isValid() << "\n";
    // 通过类作用域访问静态成员
    std::cout << "invalid id: " << TextureHandle::InvalidId << "\n";
    return 0;
}
```

类作用域和命名空间有相似之处：它们都能避免命名冲突。但类作用域更特殊：

- 类里可以声明非静态成员，这些成员必须依附于对象；
- 类的访问控制（`public` / `private`）也作用于这个作用域；
- 类作用域支持嵌套：一个类里可以再定义另一个类。

---

## 问题 2：嵌套类适合解决什么问题？

有时候，一个类的内部需要一种"只在这个类里有意义"的类型。把它定义成全局类会污染命名空间，定义成嵌套类则最自然。

```cpp
// flags: -O0 -g
#include <iostream>

class LinkedList {
public:
    // Node 只在 LinkedList 内部有意义
    struct Node {
        int value;
        Node* next;
    };

    void pushFront(int v) {
        Node* n = new Node{v, head_};
        head_ = n;
    }

    void print() const {
        Node* cur = head_;
        while (cur) {
            std::cout << cur->value << " -> ";
            cur = cur->next;
        }
        std::cout << "null\n";
    }

private:
    Node* head_ = nullptr;
};

int main() {
    LinkedList list;
    list.pushFront(3);
    list.pushFront(2);
    list.pushFront(1);
    list.print();  // 1 -> 2 -> 3 -> null
    return 0;
}
```

嵌套类 `LinkedList::Node` 是 `LinkedList` 的公开类型，外部可以这样用：`LinkedList::Node* n;`。但通常我们更希望把 `Node` 藏在类的私有部分，因为外部根本不需要知道链表的内部节点类型。

嵌套类也有一个常见误解：很多人以为嵌套类能自动访问外部类的私有成员。实际上，**嵌套类本身只是作用域上嵌套，访问权限和普通类一样**。如果 `Node` 需要访问 `LinkedList` 的私有成员，需要把 `Node` 声明为 `LinkedList` 的友元，或者通过公开接口访问。

## 问题 3：如果访问控制能保护内部状态，为什么 C++ 还要提供 `friend`？

`friend` 看起来像是在"破坏封装"：它让一个外部函数或类可以访问当前类的私有成员。既然要封装，为什么又允许开后门？

因为**封装不是要把所有人都挡在外面，而是要把接口控制在有责任的代码手里**。

最典型的场景是：**两个类需要紧密协作，但又不希望把内部细节暴露给全世界**。比如，一个 `Matrix` 类和一个 `MatrixMultiplier` 类，乘法器需要直接读取矩阵的内部存储，但你不想把矩阵的底层数组公开给所有用户。

```cpp
// flags: -O0 -g
#include <iostream>

class Matrix {
    // MatrixMultiplier 被允许访问 Matrix 的私有成员
    friend class MatrixMultiplier;

    int data_[4];
public:
    Matrix(int a, int b, int c, int d) : data_{a, b, c, d} {}
    int at(int i) const { return data_[i]; }
};

class MatrixMultiplier {
public:
    static int dot(const Matrix& a, const Matrix& b) {
        // 直接访问私有成员，不需要经过 getter
        return a.data_[0] * b.data_[0] +
               a.data_[1] * b.data_[1] +
               a.data_[2] * b.data_[2] +
               a.data_[3] * b.data_[3];
    }
};

int main() {
    Matrix m1(1, 2, 3, 4);
    Matrix m2(5, 6, 7, 8);
    std::cout << "dot = " << MatrixMultiplier::dot(m1, m2) << "\n";
    return 0;
}
```

`friend` 的关键在于：**谁是你的朋友，由类自己决定**。不是外部代码强行闯入，而是类主动声明"我信任这个函数/类"。

另一个常见用法是重载运算符。以 `operator<<` 输出自定义类型为例：

```cpp
// flags: -O0 -g
#include <iostream>

class Vec2 {
    float x_, y_;
public:
    Vec2(float x, float y) : x_(x), y_(y) {}
    // 友元声明：让这个非成员函数能访问 Vec2 的私有成员
    friend std::ostream& operator<<(std::ostream& os, const Vec2& v) {
        os << "(" << v.x_ << ", " << v.y_ << ")";
        return os;
    }
};

int main() {
    Vec2 v(1.0f, 2.0f);
    std::cout << v << "\n";  // (1, 2)
    return 0;
}
```

> [!warning] 一个常见误解
> 因为 `operator<<` 写在 `Vec2` 类体里，很多人会以为它是 `Vec2` 的成员函数。其实，`friend` 开头的函数**不是成员函数**，而是普通函数；`Vec2` 只是通过 `friend` 主动授权它访问私有成员。
>
> 为什么不能做成成员函数？如果去掉 `friend`，它就变成 `Vec2::operator<<(std::ostream&)`，调用时左操作数是 `Vec2`，只能写成 `v << std::cout`，而没法写成习惯的 `std::cout << v`。所以输出运算符通常以非成员函数实现；如果把它放到类外定义又不加 `friend`，编译器又会因为 `x_`、`y_` 是私有的而报错。`friend` 就是解决这个矛盾最自然的方式。

## 问题 4：`friend` 什么时候该用，什么时候是滥用？

`friend` 是一把双刃剑。用得好，它能让紧密协作的类保持干净的公开接口；用得不好，它会让封装形同虚设。

**适合用 `friend` 的场景：**

- 两个类语义上强耦合，需要直接访问内部状态（如容器和迭代器）；
- 运算符重载需要访问私有成员；
- 单元测试需要验证内部不变量，但不想为了测试暴露公开接口。

**应该避免的场景：**

- 只是为了"省事"，绕过 getter/setter 直接改私有成员；
- 把大量无关类都声明为友元，导致类的边界模糊；
- 用友元替代良好的接口设计。

```cpp
// flags: -O0 -g
#include <iostream>

// 反面示例：滥用 friend
class BankAccount {
    friend class EveryClass;  // 糟糕：谁都能访问
    int balance_;
public:
    BankAccount(int b) : balance_(b) {}
};

class EveryClass {
public:
    static void steal(BankAccount& acc) {
        acc.balance_ = 0;  // 友元滥用导致封装失效
    }
};

int main() {
    BankAccount acc(1000);
    EveryClass::steal(acc);
    // 现在 balance_ 变成 0，但没有任何公开接口能解释这件事
    return 0;
}
```

好的原则是：**友元应该是"有名字的、受控的"**。你明确知道谁是朋友，并且这个朋友有合理的理由访问内部。如果朋友名单太长，说明你的类设计可能需要拆分。

---

## 总结

- 类本身是一个作用域，能避免成员命名冲突，并通过访问控制保护实现细节。
- 嵌套类适合表达"只在外部类内部有意义"的类型，但它不会自动获得外部类的私有访问权限。
- `friend` 是类主动开放的访问权限，常用于强耦合的类协作和运算符重载。
- 滥用 `friend` 会破坏封装；合理的 `friend` 应该是少量的、有明确目的的、可控的。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 容器与迭代器、测试友元 | `Array<T>::Iterator` 常作为嵌套类定义，并声明为友元以直接访问内部存储；单元测试类通过 `friend class Test_xxx` 验证内部不变量，避免为测试暴露公开 getter |
| **UE** | 反射生成代码与属性访问 | UHT 生成的反射代码需要访问 `UCLASS` 的私有成员，UE 通过 `friend` 或.generated.h 中的友元声明让 generated code 合法访问；同时 `FArchive` 序列化系统也常与 UObject 私有状态紧密协作 |

> [!note] 关键取舍
> 在引擎中，`friend` 最常见的正当用途有两个：一是容器-迭代器这种天然紧密耦合的抽象，二是测试。SelfGameEngine 可能会更倾向于把测试友元集中、有限地使用，而 UE 因为反射代码生成，友元使用范围更广。理解这一点，能帮你判断"这个 friend 是必要的协作，还是偷懒的破坏封装"。
