# 构造函数与析构函数基础

> [!info] 一句话概括
> **构造函数**保证对象诞生时就有合法状态，**析构函数**保证对象死亡时自动清理资源；两者合起来构成了 C++ 最核心的资源管理思想——RAII（资源获取即初始化）。

---

## 问题 0：如果类只能靠外部代码逐个初始化成员，会出什么问题？

假设我们写了一个简单的字符串包装类，靠外部代码手动设置指针和长度：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
public:
    char* data;
    size_t len;
};

int main() {
    MyString s;
    // s.data 是什么？s.len 是什么？
    // 它们只是未初始化的随机值
    std::cout << "len = " << s.len << "\n";
    return 0;
}
```

这个程序能编译，但 `s.data` 和 `s.len` 的值是不确定的。在真实代码里，这种"未初始化就使用"的bug会导致：

- 读到垃圾值；
- 解引用野指针崩溃；
- 调用者忘了分配内存，析构时 `delete` 一个无效地址。

更麻烦的是，**初始化责任被推给了每一个使用这个类的人**。有一百处创建 `MyString` 的地方，就可能有一百种错误的初始化方式。

所以问题变成了：能不能让对象"一出生"就是完整的、合法的？

---

## 问题 1：构造函数是怎么保证对象诞生时就有合法状态的？

构造函数是类的一种特殊成员函数，**名字和类名相同，没有返回值类型**。它在对象创建时自动调用，用来完成初始化工作。

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
    char* data_;
    size_t len_;
public:
    // 默认构造函数：没有参数，提供合理的默认值
    MyString() : data_(nullptr), len_(0) {
        std::cout << "默认构造：空字符串\n";
    }

    // 带参构造函数：根据外部输入创建对象
    MyString(const char* str) {
        len_ = std::strlen(str);
        data_ = new char[len_ + 1];
        std::strcpy(data_, str);
        std::cout << "带参构造：\"" << data_ << "\"\n";
    }

    void print() const {
        std::cout << (data_ ? data_ : "(null)") << "\n";
    }
};

int main() {
    MyString s1;           // 调用默认构造函数
    MyString s2("hello");  // 调用带参构造函数
    s1.print();
    s2.print();
    return 0;
}
```

运行结果会显示两个构造函数依次被调用。关键点是：**调用者不需要手动调用初始化函数，编译器会在对象诞生时自动完成**。

构造函数有几种常见形态：

- **默认构造函数**：不接受参数，或者所有参数都有默认值；
- **带参构造函数**：根据外部输入初始化对象；
- **拷贝构造函数**：用一个同类对象初始化新对象（后续笔记会深入）；
- **移动构造函数**："偷取"另一个对象的资源（后续笔记会深入）。

在本阶段，我们先聚焦默认构造和带参构造。

## 问题 2：对象销毁时发生了什么？为什么需要析构函数？

对象的生命周期总有尽头：离开作用域、被 `delete`、或者程序结束。销毁时，如果对象持有某些资源——比如动态分配的内存、打开的文件句柄、获得的锁——这些资源必须被归还。

**析构函数**就是负责"临终清理"的特殊成员函数：

- 名字是 `~类名`；
- 没有参数，没有返回值；
- 一个类只能有一个析构函数；
- 对象销毁时自动调用。

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
    char* data_;
    size_t len_;
public:
    MyString(const char* str) {
        len_ = std::strlen(str);
        data_ = new char[len_ + 1];
        std::strcpy(data_, str);
        std::cout << "构造：\"" << data_ << "\"\n";
    }

    ~MyString() {
        delete[] data_;
        std::cout << "析构：释放内存\n";
    }
};

int main() {
    {
        MyString s("hello");
        // s 离开作用域，自动调用析构函数
    }
    std::cout << "s 已经销毁\n";
    return 0;
}
```

输出会显示：先构造，然后析构，然后才打印 "s 已经销毁"。这说明**析构函数在对象离开作用域时自动执行**，调用者不需要、也不应该手动调用。

如果没有析构函数，`new char[...]` 分配的内存就永远不会被释放——这就是内存泄漏。

## 问题 3：把资源释放放到析构函数里，为什么比手动管理更好？

手动管理资源最容易犯的错误是"漏释放"。比如：

```cpp
void process() {
    FILE* f = fopen("data.txt", "r");
    if (some_error()) {
        return;  // 糟糕，f 没关
    }
    // ... 其他可能提前返回的路径 ...
    fclose(f);
}
```

每一个提前返回、每一个异常抛出、每一个 `continue` / `break`，都可能让 `fclose` 被跳过。代码越长，遗漏的概率越高。

C++ 的解决方案是：**把资源的生命周期绑定到对象的生命周期**。对象创建时获取资源，对象销毁时释放资源。这就是 **RAII（Resource Acquisition Is Initialization，资源获取即初始化）**。

```cpp
// flags: -O0 -g
#include <cstdio>
#include <iostream>
#include <stdexcept>

class FileGuard {
    FILE* file_;
public:
    explicit FileGuard(const char* path) {
        file_ = std::fopen(path, "r");
        if (!file_) {
            throw std::runtime_error("无法打开文件");
        }
        std::cout << "打开文件\n";
    }

    ~FileGuard() {
        if (file_) {
            std::fclose(file_);
            std::cout << "关闭文件\n";
        }
    }

    // 禁止拷贝，避免重复关闭
    FileGuard(const FileGuard&) = delete;
    FileGuard& operator=(const FileGuard&) = delete;
};

int main() {
    try {
        FileGuard fg("nonexistent.txt");
    } catch (const std::exception& e) {
        std::cout << "捕获异常：" << e.what() << "\n";
    }
    return 0;
}
```

这个例子演示了 RAII 的威力：

- 构造函数负责"获取资源"；
- 析构函数负责"释放资源"；
- 即使中间发生异常，栈展开时也会调用析构函数，资源不会泄漏。

> 注意：上面例子提前使用了 `= delete` 和异常，这些是后续阶段的内容。这里只需要理解"析构函数是资源清理的可靠入口"即可。

## 问题 4：构造函数和析构函数有哪些常见陷阱？

**陷阱 1：忘记写析构函数，导致资源泄漏**

只要类里管理了堆内存、文件句柄、锁、网络连接等资源，就必须考虑是否需要自定义析构函数。这是 [[Notes/C++编程/构造、析构与拷贝控制/Rule of Three Five Zero|Rule of Three, Five and Zero]] 的起点。

**陷阱 2：在构造函数里抛异常，导致部分构造的对象泄漏**

如果构造函数申请了多个资源，第一个成功、第二个失败并抛异常，第一个资源可能泄漏。解决方法是"先全部尝试获取，失败时手动回滚"，或者用更小的 RAII 对象把每个资源单独封装。

**陷阱 3：手动调用析构函数，或者重复调用**

析构函数应该由编译器自动调用。手动调用 `s.~MyString()` 通常只在"placement new"等高级场景出现，否则极易导致重复释放。

```cpp
// flags: -O0 -g
#include <iostream>

class Demo {
public:
    Demo()  { std::cout << "构造\n"; }
    ~Demo() { std::cout << "析构\n"; }
};

int main() {
    Demo d;
    // d.~Demo();  // 错误：会导致 d 生命周期结束时再次析构
    return 0;
}
```

**陷阱 4：默认构造函数被隐式删除**

如果你写了带参构造函数但没有写默认构造函数，编译器不会自动生成默认构造。此时 `MyString s;` 会编译失败。

```cpp
// flags: -O0 -g
#include <iostream>

class OnlyParam {
public:
    OnlyParam(int x) { std::cout << x << "\n"; }
    // 没有默认构造函数
};

int main() {
    // OnlyParam o;  // 编译错误：没有合适的默认构造函数
    OnlyParam o(42);  // OK
    return 0;
}
```

如果你确实需要默认构造，可以显式要求编译器生成：`OnlyParam() = default;`。

## 总结

- 构造函数在对象创建时自动执行，确保对象一诞生就处于合法状态。
- 析构函数在对象销毁时自动执行，负责清理对象持有的资源。
- RAII 把资源生命周期绑定到对象生命周期，是 C++ 避免资源泄漏的核心机制。
- 忘记析构函数、手动调用析构函数、以及默认构造被隐式删除，是这一阶段最常见的坑。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄类 | `FileGuard`、`LockGuard`、`ScopedProfiler` 等轻量级 RAII 类，用构造函数获取资源、析构函数释放资源，避免在庞大渲染管线中手动管理句柄 |
| **UE** | `UObject` 生命周期 | `UObject` 由垃圾回收管理，构造函数负责初始化默认值，析构函数则不直接释放外部资源；但 `TUniquePtr`、`FScopeLock` 等工具类仍严格遵循 RAII |

> [!note] 关键取舍
> 在 SelfGameEngine 这类追求确定性和零开销的引擎里，RAII 是资源管理的基本单元；而 UE 的 `UObject` 因为有反射和垃圾回收，资源释放往往交给 GC。但即便如此，非 `UObject` 的资源（文件、锁、内存）仍然需要用 RAII 封装。理解"构造函数获取、析构函数释放"这个基本模式，是后续理解智能指针、自定义分配器和异常安全的基础。
