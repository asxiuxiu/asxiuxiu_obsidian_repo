# 构造函数与析构函数基础

> [!info] 一句话概括
> **构造函数**保证对象诞生时就有合法状态，**析构函数**保证对象死亡时自动清理资源；两者合起来构成了 C++ 最核心的资源管理思想——RAII（资源获取即初始化）。

---

## 问题 0：如果类只能靠外部代码逐个初始化成员，会出什么问题？

假设我们写了一个简单的字符串包装类，靠外部代码手动设置指针和长度：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
public:
    char* data;
    size_t len;
};

int main() {
    MyString s;
    // s.data 是什么？s.len 是什么？
    // 它们只是未初始化的随机值
    std::cout << "len = " << s.len << "\n";
    return 0;
}
```

这个程序能编译，但 `s.data` 和 `s.len` 的值是不确定的。

> [!note] 为什么不是 0？
> 这里 `MyString s;` 属于**默认初始化**。对于基本类型成员（`char*`、`size_t`、`int` 等），默认初始化不会自动给它们赋值；笔记这里故意用"最裸"的写法来展示风险。
> 
> 基本类型成员会被自动置 0 的常见情况如下：

| 初始化方式 | 示例 | `data` / `len` 是否被置 0 |
|---|---|---|
| 默认初始化（局部变量） | `MyString s;` | ❌ 不确定 |
| 值初始化 | `MyString s{};` | ✅ 置 0 |
| 类内初始值（C++11） | `char* data = nullptr; size_t len = 0;` | ✅ 使用类内值 |
| 全局 / 静态对象 | `static MyString s;` | ✅ 零初始化 |
| 构造函数初始化列表 | `MyString() : data(nullptr), len(0) {}` | ✅ 按列表初始化 |

> 正因为 `MyString s;` 这种写法是业务代码里最自然的写法，类本身才必须负责把基本成员初始化成合法值——否则每一处使用这个类的人都要记住额外规则，迟早会出错。

在真实代码里，这种"未初始化就使用"的bug会导致：

- 读到垃圾值；
- 解引用野指针崩溃；
- 调用者忘了分配内存，析构时 `delete` 一个无效地址。

更麻烦的是，**初始化责任被推给了每一个使用这个类的人**。有一百处创建 `MyString` 的地方，就可能有一百种错误的初始化方式。

所以问题变成了：能不能让对象"一出生"就是完整的、合法的？

## 问题 1：构造函数是怎么保证对象诞生时就有合法状态的？

构造函数是类的一种特殊成员函数，**名字和类名相同，没有返回值类型**。它在对象创建时自动调用，用来完成初始化工作。

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
    char* data_;
    size_t len_;
public:
    // 默认构造函数：没有参数，提供合理的默认值
    MyString() : data_(nullptr), len_(0) {
        std::cout << "默认构造：空字符串\n";
    }

    // 带参构造函数：根据外部输入创建对象
    MyString(const char* str) {
        len_ = std::strlen(str);
        data_ = new char[len_ + 1];
        std::strcpy(data_, str);
        std::cout << "带参构造：\"" << data_ << "\"\n";
    }

    void print() const {
        std::cout << (data_ ? data_ : "(null)") << "\n";
    }
};

int main() {
    MyString s1;           // 调用默认构造函数
    MyString s2("hello");  // 调用带参构造函数
    s1.print();
    s2.print();
    return 0;
}
```

运行结果会显示两个构造函数依次被调用。关键点是：**调用者不需要手动调用初始化函数，编译器会在对象诞生时自动完成**。

构造函数有几种常见形态：

- **默认构造函数**：不接受参数，或者所有参数都有默认值；
- **带参构造函数**：根据外部输入初始化对象；
- **拷贝构造函数**：用一个同类对象初始化新对象（后续笔记会深入）；
- **移动构造函数**："偷取"另一个对象的资源（后续笔记会深入）。

在本阶段，我们先聚焦默认构造和带参构造。

## 问题 2：对象销毁时发生了什么？为什么需要析构函数？

对象的生命周期总有尽头：离开作用域、被 `delete`、或者程序结束。销毁时，如果对象持有某些资源——比如动态分配的内存、打开的文件句柄、获得的锁——这些资源必须被归还。

**析构函数**就是负责"临终清理"的特殊成员函数：

- 名字是 `~类名`；
- 没有参数，没有返回值；
- 一个类只能有一个析构函数；
- 对象销毁时自动调用。

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class MyString {
    char* data_;
    size_t len_;
public:
    MyString(const char* str) {
        len_ = std::strlen(str);
        data_ = new char[len_ + 1];
        std::strcpy(data_, str);
        std::cout << "构造：\"" << data_ << "\"\n";
    }

    ~MyString() {
        delete[] data_;
        std::cout << "析构：释放内存\n";
    }
};

int main() {
    {
        MyString s("hello");
        // s 离开作用域，自动调用析构函数
    }
    std::cout << "s 已经销毁\n";
    return 0;
}
```

输出会显示：先构造，然后析构，然后才打印 "s 已经销毁"。这说明**析构函数在对象离开作用域时自动执行**，调用者不需要、也不应该手动调用。

如果没有析构函数，`new char[...]` 分配的内存就永远不会被释放——这就是内存泄漏。

## 问题 3：把资源释放放到析构函数里，为什么比手动管理更好？

手动管理资源最容易犯的错误是"漏释放"。比如：

```cpp
void process() {
    FILE* f = fopen("data.txt", "r");
    if (some_error()) {
        return;  // 糟糕，f 没关
    }
    // ... 其他可能提前返回的路径 ...
    fclose(f);
}
```

每一个提前返回、每一个异常抛出、每一个 `continue` / `break`，都可能让 `fclose` 被跳过。代码越长，遗漏的概率越高。

C++ 的解决方案是：**把资源的生命周期绑定到对象的生命周期**。对象创建时获取资源，对象销毁时释放资源。这就是 **RAII（Resource Acquisition Is Initialization，资源获取即初始化）**。

```cpp
// flags: -O0 -g
#include <cstdio>
#include <iostream>
#include <stdexcept>

class FileGuard {
    FILE* file_;
public:
    // explicit: 禁止 const char* 隐式转换成 FileGuard，避免不经意间触发文件打开
    explicit FileGuard(const char* path) {
        file_ = std::fopen(path, "r");
        if (!file_) {
            throw std::runtime_error("无法打开文件");
        }
        std::cout << "打开文件\n";
    }

    ~FileGuard() {
        if (file_) {
            std::fclose(file_);
            std::cout << "关闭文件\n";
        }
    }

    // 禁止拷贝，避免重复关闭
    FileGuard(const FileGuard&) = delete;
    FileGuard& operator=(const FileGuard&) = delete;
};

int main() {
    try {
        FileGuard fg("nonexistent.txt");
    } catch (const std::exception& e) {
        std::cout << "捕获异常：" << e.what() << "\n";
    }
    return 0;
}
```

这个例子演示了 RAII 的威力：

- 构造函数负责"获取资源"；
- 析构函数负责"释放资源"；
- 即使中间发生异常，栈展开时也会调用析构函数，资源不会泄漏。

> [!note] 为什么 `FileGuard` 的构造函数前面要加 `explicit`？
> 单参数构造函数如果不加 `explicit`，编译器会允许该参数类型**隐式转换**成这个类的对象。例如 `FileGuard fg = "data.txt";` 或 `process("data.txt")`（形参为 `FileGuard`）都会悄悄打开文件。
>
> `explicit` 的作用是关闭这种隐式转换，强制调用者显式写出 `FileGuard{"data.txt"}`。对于会触发资源获取、副作用明显的构造函数，通常都建议加 `explicit`。

> 注意：上面例子提前使用了 `= delete` 和异常，这些是后续阶段的内容。这里只需要理解"析构函数是资源清理的可靠入口"即可。

## 问题 4：构造函数和析构函数有哪些常见陷阱？

**陷阱 1：忘记写析构函数，导致资源泄漏**

只要类里管理了堆内存、文件句柄、锁、网络连接等资源，就必须考虑是否需要自定义析构函数。这是 [[Notes/C++编程/构造、析构与拷贝控制/Rule of Three Five Zero|Rule of Three, Five and Zero]] 的起点。

**陷阱 2：在构造函数里抛异常，导致部分构造的对象泄漏**

如果构造函数申请了多个资源，第一个成功、第二个失败并抛异常，第一个资源可能泄漏。**对象只有在构造函数完全执行成功后才会被调用析构函数**，所以部分构造成功的对象不会自动清理已分配的资源。

下面用一个简化版"纹理加载"类演示这个问题：

```cpp
// flags: -O0 -g
#include <iostream>
#include <stdexcept>

// 模拟：加载像素成功，创建纹理失败
char* load_pixels(const char*) {
    std::cout << "加载像素\n";
    return new char[1024]{};
}

int create_texture(const char*) {
    std::cout << "创建纹理失败，抛异常\n";
    throw std::runtime_error("GPU 纹理创建失败");
}

class Texture {
    char* pixels_;
    int texture_id_;
public:
    Texture(const char* path) {
        pixels_ = load_pixels(path);     // 第 1 步：成功，分配了内存
        texture_id_ = create_texture(pixels_);  // 第 2 步：抛异常
    }

    ~Texture() {
        delete[] pixels_;
        std::cout << "析构：释放像素\n";
    }
};

int main() {
    try {
        Texture t("hero.png");
    } catch (const std::exception& e) {
        std::cout << "捕获异常：" << e.what() << "\n";
    }
    return 0;
}
```

运行后会发现：**"析构：释放像素" 永远不会打印**，`pixels_` 指向的内存泄漏了。因为构造函数还没跑完就抛异常，编译器认为这个对象"不存在"，不会调用析构函数。

> 这种泄漏在真实引擎里非常常见：比如加载模型时，顶点缓冲区已经分配，但索引缓冲区创建失败；或者文件已经打开，但内存映射创建失败。

**怎么解决？**

**方案 A：手动回滚（能解决问题，但容易漏）**

在构造函数里用 `try-catch` 捕获异常，手动释放已经获取的资源，然后再重新抛出：

```cpp
Texture(const char* path) {
    pixels_ = load_pixels(path);
    try {
        texture_id_ = create_texture(pixels_);
    } catch (...) {
        delete[] pixels_;   // 手动释放已分配的资源
        throw;              // 把异常继续往上抛
    }
}
```

这个方案的问题是：资源步骤越多，回滚逻辑越复杂。如果有 5 个资源，就要写 5 段回滚代码，任何一个遗漏都会泄漏。

**方案 B：把每个资源封装成更小的 RAII 对象（推荐）**

让每个资源的生命周期由自己的小型 RAII 类管理。即使外层构造失败，已经构造成功的成员对象也会被正常析构。

```cpp
// flags: -O0 -g
#include <iostream>
#include <stdexcept>

char* load_pixels(const char*) {
    std::cout << "加载像素\n";
    return new char[1024]{};
}

int create_texture(const char*) {
    std::cout << "创建纹理失败，抛异常\n";
    throw std::runtime_error("GPU 纹理创建失败");
}

// 小 RAII 类：只负责像素内存
class PixelBuffer {
    char* data_;
public:
    PixelBuffer(const char* path) : data_(load_pixels(path)) {}
    ~PixelBuffer() {
        delete[] data_;
        std::cout << "PixelBuffer 析构：释放像素\n";
    }
    char* get() const { return data_; }

    PixelBuffer(const PixelBuffer&) = delete;
    PixelBuffer& operator=(const PixelBuffer&) = delete;
};

class Texture {
    PixelBuffer pixels_;     // 先构造，失败时外层对象不存在，但它自己已经完整构造过
    int texture_id_;
public:
    Texture(const char* path)
        : pixels_(path)
        , texture_id_(create_texture(pixels_.get()))  // 这里抛异常
    {}
};

int main() {
    try {
        Texture t("hero.png");
    } catch (const std::exception& e) {
        std::cout << "捕获异常：" << e.what() << "\n";
    }
    return 0;
}
```

运行后会看到：即使 `Texture` 构造失败，`PixelBuffer` 的析构函数仍然被调用，像素内存被正确释放。

**核心原则**：

- 构造函数里尽量避免同时直接管理多个裸资源；
- 每个资源尽量交给独立的 RAII 对象（如 `PixelBuffer`、`FileGuard`、智能指针等）；
- 如果确实要在一个类里管理多个裸资源，异常发生时要手动回滚，并且每增加一个资源都要检查回滚路径是否完整。

**陷阱 3：手动调用析构函数，或者重复调用**

析构函数应该由编译器自动调用。手动调用 `s.~MyString()` 通常只在"placement new"等高级场景出现，否则极易导致重复释放。

```cpp
// flags: -O0 -g
#include <iostream>

class Demo {
public:
    Demo()  { std::cout << "构造\n"; }
    ~Demo() { std::cout << "析构\n"; }
};

int main() {
    Demo d;
    // d.~Demo();  // 错误：会导致 d 生命周期结束时再次析构
    return 0;
}
```

**陷阱 4：默认构造函数被隐式删除**

如果你写了带参构造函数但没有写默认构造函数，编译器不会自动生成默认构造。此时 `MyString s;` 会编译失败。

```cpp
// flags: -O0 -g
#include <iostream>

class OnlyParam {
public:
    OnlyParam(int x) { std::cout << x << "\n"; }
    // 没有默认构造函数
};

int main() {
    // OnlyParam o;  // 编译错误：没有合适的默认构造函数
    OnlyParam o(42);  // OK
    return 0;
}
```

如果你确实需要默认构造，可以显式要求编译器生成：`OnlyParam() = default;`。

## 常见需要自定义析构函数的资源清单

判断一个类是否需要自定义析构函数，核心标准只有一个：**这个类的对象是否直接或间接拥有一个需要显式释放的资源**。如果资源不归这个对象管理（比如只是临时引用一下），就不需要写析构函数。

| 资源类型               | 典型成员 / 句柄                                    | 释放方式                                            |
| ------------------ | -------------------------------------------- | ----------------------------------------------- |
| 堆内存                | `char*`、`T*`、`new[]` 分配的内存                   | `delete` / `delete[]` / 自定义释放器                  |
| 文件句柄               | `FILE*`、`int fd`、`HANDLE`                    | `fclose`、`close`、`CloseHandle`                  |
| 网络 / 套接字           | `int sockfd`、`SOCKET`                        | `close` / `closesocket`                         |
| 线程 / 进程句柄          | `std::thread`（未 join）、`HANDLE`、`pid_t`       | `join` / `detach`、`CloseHandle`、`waitpid`       |
| 锁 / 同步原语           | 手动管理的 `pthread_mutex_t*`、`CRITICAL_SECTION*` | `pthread_mutex_destroy`、`DeleteCriticalSection` |
| 数据库连接              | `sqlite3*`、`PGconn*`、`MYSQL*`                | 对应的 close / disconnect API                      |
| GPU / 图形资源         | `GLuint`、`VkBuffer`、`ID3D12Resource*`        | `glDelete*`、`vkDestroy*`、`Release`              |
| 音频资源               | `ALuint`、`Mix_Chunk*`                        | 对应的 free / destroy API                          |
| 临时文件 / 目录          | 创建出来的临时路径                                    | `std::remove`、`std::filesystem::remove`         |
| 第三方库 opaque handle | `libxxx_handle_t*` 等                         | 库提供的 destroy / free 函数                          |

> [!tip] 什么时候**不需要**自定义析构函数？
> - 类里只有普通值类型（`int`、`float`、自定义结构体）；
> - 类里持有智能指针（`std::unique_ptr`、`std::shared_ptr`）或其他 RAII 对象（`std::string`、`std::vector`、`std::lock_guard`）；
> - 类只是引用外部资源，但不负责释放（例如传入的 `const char*` 字符串字面量、观察者模式中的裸指针）。
>
> 这些情况要么编译器生成的析构函数已经足够，要么释放责任在别人身上。

## RAII 的演进：从手写 Guard 到标准库

前面为了展示 RAII 的本质，我们用了很多**手写**的 `FileGuard`、`PixelBuffer` 和裸 `new/delete`。但在现代 C++ 工程里，真正天天写的代码往往不是这样。

**RAII 的思想没有变**：资源获取放在构造函数，资源释放放在析构函数，靠对象作用域保证清理。只是现代 C++ 已经把这些模式封装进了标准库，大部分时候你直接用现成的即可。

### 现代 C++ 里，手写 RAII 类已经变少了

| 场景    | 过去手写的 RAII               | 现代 C++ 首选                            |
| ----- | ------------------------ | ------------------------------------ |
| 动态内存  | 自定义类里 `new`/`delete`     | `std::unique_ptr`、`std::shared_ptr`  |
| 可变长数组 | 自定义类里 `new[]`/`delete[]` | `std::vector`                        |
| 字符串   | 自定义类里 `new[]`/`delete[]` | `std::string`                        |
| 文件    | `FileGuard` 封装 `FILE*`   | `std::ifstream`、`std::ofstream`      |
| 锁     | 自定义类封装 `mutex`           | `std::lock_guard`、`std::unique_lock` |
| 线程    | 自定义类管理 `join`/`detach`   | `std::jthread`（C++20）/`std::thread`  |

例如，你笔记里那个 `MyString` 类，在现代 C++ 里通常直接写成：

```cpp
// flags: -O0 -g
#include <iostream>
#include <string>

int main() {
    std::string s1;            // 默认构造，空字符串
    std::string s2("hello");   // 带参构造
    std::cout << s1 << "\n";
    std::cout << s2 << "\n";
    return 0;  // s1、s2 离开作用域，自动释放内存
}
```

`std::string` 内部仍然遵循 RAII：构造时分配内存，析构时释放内存。但你不再需要自己写析构函数、拷贝构造、移动构造等一堆配套函数。

### 现在还需要手写 RAII 类的地方

标准库覆盖不了的资源，仍然需要自己封装。典型的有：

- **GPU / 图形资源**：`GLuint`、`VkBuffer`、`ID3D12Resource*` 等
- **第三方 C 库的 opaque handle**：比如 `sqlite3*`、`libxxx_handle_t`
- **文件描述符 / socket**：`int fd`、`SOCKET`
- **临时文件 / 目录**：需要析构时自动删除
- **引擎内部工具**：作用域分析器、锁、自定义分配器的 RAII 包装

判断标准还是一样的：**只要你持有的资源需要一个明确的清理动作，而标准库没有现成封装，就应该写一个轻量 RAII 类**。

### 为什么这节课还要学手写 RAII？

因为智能指针、`std::vector`、`std::lock_guard` 这些工具，本质上都是 RAII 思想的实现。如果你不理解“构造函数获取、析构函数释放”这个底层机制，遇到下面这些情况会很难排查：

- 自定义删除器的 `std::unique_ptr` 行为异常；
- `std::vector` 里的元素析构顺序导致 bug；
- 多线程里锁的持有范围超出预期；
- 自己封装第三方库的 handle 时，忘记处理异常安全。

> [!tip] 现代 C++ 的 RAII 口诀
> **能交给标准库的，绝不自己写；标准库覆盖不到的，写一个小而专注的 RAII 类。**

---

## 总结

- 构造函数在对象创建时自动执行，确保对象一诞生就处于合法状态。
- 析构函数在对象销毁时自动执行，负责清理对象持有的资源。
- RAII 把资源生命周期绑定到对象生命周期，是 C++ 避免资源泄漏的核心机制。
- 现代 C++ 中，RAII 思想仍然是最核心的资源管理方式，但多数场景应优先使用标准库封装好的工具。
- 忘记析构函数、手动调用析构函数、以及默认构造被隐式删除，是这一阶段最常见的坑。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 资源句柄类 | `FileGuard`、`LockGuard`、`ScopedProfiler` 等轻量级 RAII 类，用构造函数获取资源、析构函数释放资源，避免在庞大渲染管线中手动管理句柄 |
| **UE** | `UObject` 生命周期 | `UObject` 由垃圾回收管理，构造函数负责初始化默认值，析构函数则不直接释放外部资源；但 `TUniquePtr`、`FScopeLock` 等工具类仍严格遵循 RAII |

> [!note] 关键取舍
> 在 SelfGameEngine 这类追求确定性和零开销的引擎里，RAII 是资源管理的基本单元；而 UE 的 `UObject` 因为有反射和垃圾回收，资源释放往往交给 GC。但即便如此，非 `UObject` 的资源（文件、锁、内存）仍然需要用 RAII 封装。理解"构造函数获取、析构函数释放"这个基本模式，是后续理解智能指针、自定义分配器和异常安全的基础。
