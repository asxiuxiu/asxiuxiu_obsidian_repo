# 类模板基础与模板参数推导

> [!info] 一句话概括
> **类模板**让一份类定义能同时服务多种类型，而**模板参数推导**让编译器帮你自动填上这些类型——这是引擎里 `TArray`、`TMap`、`Array<T, N>` 等泛型容器的根基。

---

## 问题 0：为什么需要类模板？

假设你要写一个固定大小的数组类，先为 `int` 写一版：

```cpp
class IntArray {
    int data_[10];
public:
    int& operator[](size_t i) { return data_[i]; }
    size_t size() const { return 10; }
};
```

没过多久你会发现还需要 `float` 版、`Vec3` 版、`String` 版……它们的逻辑完全一样，只是元素类型不同。如果每个类型都手写一份，代码会爆炸，而且任何 bug 修复都要改 N 个地方。

这就是类模板要解决的问题：**把「类型」也变成参数**，让编译器根据你传入的类型自动生成对应的类。

> [!abstract]
> **类模板（Class Template）**是一种带参数的类定义。模板参数可以是类型（如 `T`）、整数常量（如 `N`）、甚至其他模板。编译器在实例化时根据实际参数生成具体的类。

## 问题 1：类模板长什么样？

把上面的 `IntArray` 改成模板：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstddef>

template<typename T, size_t N>
class Array {
    T data_[N];
public:
    T& operator[](size_t i) { return data_[i]; }
    const T& operator[](size_t i) const { return data_[i]; }
    size_t size() const { return N; }
};

int main() {
    Array<int, 10> arr;
    arr[0] = 42;
    std::cout << arr[0] << "\n";
    std::cout << arr.size() << "\n";

    Array<double, 4> darr;
    darr[0] = 3.14;
    return 0;
}
```

这里有两个模板参数：
- `typename T`：类型参数，可以是 `int`、`double`、`Vec3` 等。
- `size_t N`：非类型参数（non-type template parameter），必须能在编译期确定，这里是数组大小。

> [!tip]
> `typename` 和 `class` 在模板参数列表里完全等价。你可以写 `template<class T>` 或 `template<typename T>`。现代 C++ 推荐用 `typename`，因为 `class` 容易让人误以为只能传类类型。

### 模板参数的三大类别

| 类别 | 示例 | 说明 |
|------|------|------|
| 类型参数 | `typename T` | 代表一个类型 |
| 非类型参数 | `size_t N`、`int Value` | 编译期常量，C++20 起支持更多类型（如浮点、字面量类型） |
| 模板模板参数 | `template<typename> class Container` | 参数本身是一个模板，用于 policy-based design |

对于 Day 5 的练习，你只需要掌握前两类。

## 问题 2：编译器怎么知道 `T` 和 `N` 是什么？

### 显式指定

最直接的方式是写清楚：

```cpp
Array<int, 10> arr;        // T = int,    N = 10
Array<double, 4> darr;     // T = double, N = 4
```

### 类模板参数推导（C++17）

C++17 之前，即使初始化列表能推断类型，你也必须显式写类型：

```cpp
// C++11/14
std::pair<int, double> p(1, 2.0);

// C++17
std::pair p(1, 2.0);  // 编译器推导为 pair<int, double>
```

这个特性对标准库容器很方便，但对我们自己写的 `Array<T, N>`，由于 `N` 是大小、无法从构造函数参数推导，通常还是显式指定：

```cpp
Array<int, 10> arr;  // 必须显式指定 N
```

### 函数模板的推导更自然

虽然 Day 5 写的是类模板，但顺带理解函数模板推导很重要：

```cpp
// flags: -O0 -g
#include <iostream>

template<typename T>
T max(T a, T b) {
    return a > b ? a : b;
}

int main() {
    auto x = max(3, 4);       // T 推导为 int
    auto y = max(3.0, 4.0);   // T 推导为 double
    // auto z = max(3, 4.0);  // 编译错误：T 无法同时是 int 和 double
    return 0;
}
```

> [!warning]
> 函数模板推导时，`T` 通常会**退化**（decay）。比如数组会退化为指针，`const` 和引用会被剥离。如果你需要保留这些修饰，要用 `T&` 或 `const T&` 作为参数类型。

## 问题 3：写 `Array<T, N>` 要注意什么？

Day 5 的题目是手写一个固定大小数组模板。核心点：

1. **默认构造与元素初始化**
   - `T data_[N];` 不会默认初始化内置类型元素（它们是未确定的垃圾值）。
   - 如果你希望元素默认初始化，可以用 `T data_[N]{};`（值初始化）。

2. **`operator[]` 的 const 重载**
   - 非 const 对象返回 `T&`，允许修改。
   - const 对象返回 `const T&`，只读访问。

3. **显式构造（explicit）**
   - 如果你的 `Array` 有单参数构造函数（比如从 `std::initializer_list` 构造），要加 `explicit` 防止隐式转换。
   - 对于 `Array<T, N>`，通常没有单参数构造，所以 `explicit` 不是重点，但 Day 5 把它列为补充阅读，说明要留意。

4. **边界检查（可选）**
   - 简单版本可以不做，只提供 `operator[]`。
   - 健壮版本可以提供 `at()` 做边界检查并抛异常或断言。

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstddef>
#include <stdexcept>

template<typename T, size_t N>
class Array {
    T data_[N]{};  // 值初始化
public:
    T& operator[](size_t i) { return data_[i]; }
    const T& operator[](size_t i) const { return data_[i]; }
    size_t size() const { return N; }

    T& at(size_t i) {
        if (i >= N) throw std::out_of_range("Array::at");
        return data_[i];
    }
};

int main() {
    Array<int, 5> arr;
    arr[0] = 10;
    std::cout << arr[0] << " " << arr.size() << "\n";
    return 0;
}
```

## 问题 4：模板代码为什么通常在头文件里？

这是新手最常踩的坑之一。如果你把类模板声明放在 `.h`，定义放在 `.cpp`，然后在外部实例化：

```cpp
// Array.h
template<typename T, size_t N>
class Array { ... };

// Array.cpp
// 这里只有模板定义，没有具体实例
```

链接时会报错，因为编译器在编译 `main.cpp` 时看不到模板定义，无法生成 `Array<int, 10>` 的具体代码。

解决方案：
1. **定义放在头文件**（最常见）。
2. **显式实例化**（explicit instantiation）：在 `.cpp` 里写 `template class Array<int, 10>;`，但这样只支持已知类型。
3. C++20 **modules**：把模板接口放到模块接口单元，但支持度和工具链成熟度还在演进。

对于练习和引擎中的模板容器，方案 1 是默认选择。

## 总结

- **类模板**把类型和编译期常量参数化，避免为每种类型手写重复类。
- 模板参数分**类型参数**、**非类型参数**、**模板模板参数**三类；Day 5 主要用前两类。
- **显式指定**是最稳妥的实例化方式；C++17 的类模板参数推导对标准库更友好。
- 函数模板推导时会退化，注意 `T` vs `T&` vs `const T&` 的区别。
- 模板定义通常要放在头文件，否则链接时找不到实例化代码。
- 写 `Array<T, N>` 时注意默认初始化、`const` 重载、`explicit` 和边界检查。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 泛型容器 | `Array<T>`、`HashMap<K, V>` 等自定义容器完全基于类模板；非类型参数用于固定大小缓冲区、编译期配置 |
| **UE** | `TArray`、`TMap`、`TSet` | UE 的容器大量依赖类模板和模板特化；`TAllocator` 等策略类通过模板模板参数注入分配策略 |

> [!note] 关键取舍
> SelfGameEngine 选择自研模板容器，是为了完全控制内存布局、分配器和异常策略。理解类模板基础后，才能进一步学习模板特化、SFINAE、Concepts 等进阶技巧，最终写出和 UE `TArray` 同量级的工业级容器。
