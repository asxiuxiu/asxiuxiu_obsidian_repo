# 原子操作与无锁编程基础

> [!info] 一句话概括
> **原子操作**是不可再分的最小读写单元；`std::atomic` 是 C++11 提供的原子类型模板；无锁编程则是用原子操作而不是锁来实现线程安全，核心是 CAS（Compare-And-Swap）循环。

---

## 问题 0：为什么需要原子操作？`volatile` 不行吗？

很多人初学到并发时会有一个误解：给变量加个 `volatile`，它就不会被编译器优化到寄存器里，其他线程就能看到了，对吧？

不对。`volatile` 的作用是告诉编译器"这个变量可能随时被外部因素改变"，比如硬件寄存器、信号处理程序。它**不保证原子性**，也**不保证操作顺序**。

```cpp
// flags: -O0 -g
#include <thread>

volatile int counter = 0;

void worker() {
    for (int i = 0; i < 100000; ++i) {
        ++counter;  // 仍然不是原子操作！
    }
}

int main() {
    std::thread t1(worker);
    std::thread t2(worker);
    t1.join();
    t2.join();
    return 0;
}
```

`++counter` 底层还是读-改-写三步，`volatile` 不能阻止两个线程的三步交错执行。最终结果仍然不是 `200000`。

真正需要的是一个"不可拆分"的操作：要么完成，要么没做，中间不会被其他线程看到半成品。这就是**原子操作（Atomic Operation）**。

---

## 问题 1：什么是 std::atomic？

**`std::atomic<T>`** 是 C++11 在 `<atomic>` 头文件中提供的类模板，它把类型 `T` 的变量包装成"原子变量"。

对原子变量执行的读、写、交换、比较交换等操作，在底层会被编译成特殊的 CPU 指令（如 x86 的 `LOCK` 前缀指令、ARM 的 `LDREX`/`STREX`），保证操作是不可分割的。

```cpp
// flags: -O0 -g
#include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> counter{0};

void worker() {
    for (int i = 0; i < 100000; ++i) {
        ++counter;  // 原子自增，线程安全
    }
}

int main() {
    std::thread t1(worker);
    std::thread t2(worker);
    t1.join();
    t2.join();
    std::cout << counter << "\n";  // 几乎一定是 200000
    return 0;
}
```

> [!abstract]
> **std::atomic** 提供了两个核心保证：
> 1. **原子性**：对变量的操作不会被其他线程看到中间状态。
> 2. **可见性**：通过指定内存序，可以控制一个线程的写入何时对其他线程可见。

---

## 问题 2：load、store、exchange 有什么区别？

`std::atomic` 最常用的三个操作：

- **`load()`**：原子地读取值。
- **`store()`**：原子地写入值。
- **`exchange()`**：原子地写入新值，并返回旧值。

```cpp
// flags: -O0 -g
#include <atomic>
#include <iostream>

int main() {
    std::atomic<int> x{10};

    int a = x.load();           // a = 10
    x.store(20);                // x = 20
    int old = x.exchange(30);   // old = 20, x = 30

    std::cout << a << " " << old << " " << x << "\n";
    return 0;
}
```

这三个操作都是原子的，不会读取到"半个值"，也不会在写入过程中被其他线程打断。

> [!tip]
> `std::atomic<int>` 也支持隐式转换和赋值运算符，比如 `int a = x;` 等价于 `x.load()`，`x = 20;` 等价于 `x.store(20)`。但显式调用 `load()`/`store()` 更清楚，也更容易指定内存序。

## 问题 3：什么是 CAS？compare_exchange_weak 和 strong 有什么区别？

**CAS（Compare-And-Swap）** 是无锁编程的核心原语。它的语义是：

> "如果当前值等于我预期的旧值，就把它换成新值；否则告诉我实际值是多少。"

`std::atomic` 提供了两个版本：

- **`compare_exchange_strong(expected, desired)`**：严格按上述语义执行。
- **`compare_exchange_weak(expected, desired)`**：语义相同，但可能**伪失败（spurious failure）**，即即使当前值等于预期值，也返回失败。

```cpp
// flags: -O0 -g
#include <atomic>
#include <iostream>

int main() {
    std::atomic<int> x{10};

    int expected = 10;
    bool ok = x.compare_exchange_strong(expected, 20);
    // ok = true, x = 20, expected 仍为 10

    std::cout << std::boolalpha << ok << " " << x << "\n";
    return 0;
}
```

伪失败是什么意思？在某些架构（尤其是 ARM）上，为了实现更好的性能，`compare_exchange_weak` 可能在没有任何线程竞争的情况下也返回失败。这听起来很讨厌，但在循环里它通常不是问题。

> [!abstract]
> **选择原则**：
> - 在 CAS 循环中，优先使用 `compare_exchange_weak`，因为它在某些平台上更快。
> - 如果不希望有伪失败（比如只执行一次 CAS 就结束），使用 `compare_exchange_strong`。

## 问题 4：CAS 循环怎么用？

单次的 CAS 只能做"如果还是旧值就更新"。但很多算法需要"读取当前值，算出新值，再尝试更新"，如果失败就重试。这就是 **CAS 循环（CAS Loop）**。

```cpp
// flags: -O0 -g
#include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> counter{0};

void increment() {
    int old = counter.load();
    int desired;
    do {
        desired = old + 1;
    } while (!counter.compare_exchange_weak(old, desired));
    // 如果失败，old 会被更新为当前值，下一轮重试
}

void worker() {
    for (int i = 0; i < 100000; ++i) {
        increment();
    }
}

int main() {
    std::thread t1(worker);
    std::thread t2(worker);
    t1.join();
    t2.join();
    std::cout << counter << "\n";
    return 0;
}
```

这里的关键是：`compare_exchange_weak` 失败时，会把 `old` 更新为当前实际值，所以下一轮循环基于最新值重新计算。

CAS 循环是无锁算法的骨架。自旋锁、无锁队列、无锁栈，底层都是这种模式。

## 问题 5：ABA 问题是什么？

CAS 看起来很完美，但它有一个经典陷阱：**ABA 问题**。

场景：线程 T1 读到值 A，准备去换成 C。期间线程 T2 把 A 改成 B，又改回 A。T1 的 CAS 检查"当前值是不是 A"，发现是 A，于是成功更新。但中间 T2 的修改被掩盖了。

```text
T1: 读 A
        T2: A → B
        T2: B → A
T1: CAS(A → C)  成功，但过程被 T2 修改过
```

在某些数据结构（如无锁栈）中，ABA 会导致指针指向已经被回收的内存，造成崩溃。

常见解决方案：

1. **版本号（Tagged Pointer）**：把指针和计数器打包在一起，每次修改计数器加一。
2. **延迟回收（Hazard Pointer / RCU）**：确保节点不会被提前释放。
3. **直接使用带版本号的原子类型**（如 `std::atomic<uint64_t>` 高 32 位存版本号，低 32 位存指针）。

> [!warning]
> ABA 问题不是 CAS 本身的 bug，而是"只比较值是否相等"这个语义在复杂数据结构中的局限。写无锁结构时必须考虑。

## 问题 6：如何用原子操作实现自旋锁？

自旋锁（SpinLock）是最简单的无锁/锁混合原语：线程反复 CAS 一个标志位，直到获得锁；释放时把标志位清 0。

```cpp
// flags: -O0 -g
#include <atomic>
#include <thread>
#include <iostream>

class SpinLock {
    std::atomic<bool> flag_{false};
public:
    void lock() {
        // 先测试再测试并设置（TTAS）
        while (true) {
            while (flag_.load()) { }  // 先自旋等待
            if (!flag_.exchange(true)) {
                return;  // 成功抢到锁
            }
        }
    }

    void unlock() {
        flag_.store(false);
    }
};

SpinLock spin;
int counter = 0;

void worker() {
    for (int i = 0; i < 100000; ++i) {
        spin.lock();
        ++counter;
        spin.unlock();
    }
}

int main() {
    std::thread t1(worker);
    std::thread t2(worker);
    t1.join();
    t2.join();
    std::cout << counter << "\n";
    return 0;
}
```

> [!tip]
> 上面的 `while (flag_.load())` 是**测试-测试并设置（TTAS）**优化：先便宜地读，发现锁被占住才继续自旋，避免每次循环都发起昂贵的 `exchange`。

自旋锁适合**临界区极短**的场景。如果临界区长，自旋会浪费 CPU；这时应该用互斥锁，竞争失败时让线程睡眠。

## 总结

- **`std::atomic<T>`** 提供不可分割的读写操作，是并发编程的基础工具。
- **`load`/`store`/`exchange`** 覆盖基本读写和交换场景。
- **`compare_exchange_weak`/`strong`** 实现 CAS，是无锁算法的核心。
- **CAS 循环**通过"读-算-试更新-重试"实现复杂的无锁逻辑。
- **ABA 问题**是 CAS 只比较值带来的隐患，需要用版本号或延迟回收解决。
- **自旋锁**是最简单的原子锁实现，适合短临界区。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 自旋锁、无锁任务队列、引用计数 | `SpinLock` 封装 `std::atomic<bool>`；无锁队列用 CAS 循环维护头尾指针；共享指针的引用计数用原子自增自减 |
| **UE** | `FSpinLock`、任务图同步、引用计数 | UE 提供 `FSpinLock` 用于低延迟同步；`TSharedPtr` 的引用计数依赖原子操作；任务系统中的无锁队列也是 CAS 实现 |

> [!note] 关键取舍
> 无锁编程看起来性能更高，但正确性极难保证。工业引擎中，锁仍然是默认选择；无锁结构只在"被证明有性能瓶颈"且"有成熟算法"时才使用。理解原子操作和 CAS，是为了在需要时能安全地使用或审查无锁代码。
