# 工作窃取队列与线程池设计

> [!info] 一句话概括
> 线程池不是「开几个线程就行」那么简单：任务怎么排队、线程怎么拿任务、负载不均时怎么办、任务有没有特殊归属要求，这些设计选择直接决定了一个并发系统能不能 scale。

---

## 问题 0：为什么需要线程池？

假设你每次提交一个异步任务就 `std::thread t(task); t.detach();` 一把。任务多了会发生什么事？

1. **线程创建和销毁非常贵**。操作系统要分配内核对象、栈空间、调度结构；这笔开销可能比任务本身的计算量还大。
2. **线程数量失控**。如果每秒涌进来几千个任务，你就会有几千个线程在跑。CPU 调度器会被上下文切换淹没，cache 也全被冲垮。
3. **资源不可预测**。每个线程默认栈 1MB（Linux）或 1MB+（Windows），一千个线程就是 1GB 虚拟内存。

更聪明的做法是：**预先创建固定数量的线程，让它们反复从同一个任务队列里取活干**。这就是线程池（Thread Pool）。

> [!abstract]
> **线程池**是一组长期存活的工作线程，加上一个共享任务队列。调用方把任务丢进队列，工作线程竞争（或被分配）任务执行；线程不会被频繁创建和销毁。

---

## 问题 1：最简单的线程池长什么样？

一个教学级线程池通常包含：

- 一个 `std::queue<Task>` 存放待执行任务。
- 一个 `std::mutex` 保护队列。
- 一个 `std::condition_variable` 让工作线程在队列空时阻塞等待。
- 一个 `stop_` 标志控制优雅退出。

```cpp
// flags: -O0 -g -pthread
#include <condition_variable>
#include <functional>
#include <future>
#include <iostream>
#include <mutex>
#include <queue>
#include <thread>
#include <vector>

class SimpleThreadPool {
public:
    using Task = std::function<void()>;

    explicit SimpleThreadPool(size_t n) {
        for (size_t i = 0; i < n; ++i) {
            workers_.emplace_back([this] {
                while (true) {
                    Task task;
                    {
                        std::unique_lock<std::mutex> lock(mtx_);
                        cv_.wait(lock, [this] { return stop_ || !tasks_.empty(); });
                        if (stop_ && tasks_.empty()) return;
                        task = std::move(tasks_.front());
                        tasks_.pop();
                    }
                    task();
                }
            });
        }
    }

    template<typename F>
    void submit(F&& f) {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            tasks_.emplace(std::forward<F>(f));
        }
        cv_.notify_one();
    }

    ~SimpleThreadPool() {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& t : workers_) t.join();
    }

private:
    std::vector<std::thread> workers_;
    std::queue<Task> tasks_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_ = false;
};

int main() {
    SimpleThreadPool pool(4);
    for (int i = 0; i < 8; ++i) {
        pool.submit([i] { std::cout << "task " << i << "\n"; });
    }
    return 0;
}
```

这个版本能跑，但有一个明显瓶颈：**所有线程都抢同一把锁、同一个队列**。任务量很大时，锁竞争会成为性能天花板。关于条件变量和谓词等待的细节，参见 [[条件变量与虚假唤醒#什么是虚假唤醒？|条件变量与虚假唤醒]]。

## 问题 2：任务队列还能怎么设计？

### 方案 A：单一全局队列

上面的代码就是。优点是实现简单、负载天然均衡；缺点是锁竞争激烈、cache line  bouncing 严重。

### 方案 B：每个线程一个本地队列

给每个工作线程配一个自己的任务队列。提交任务时按某种规则放到某个线程的队列里；工作线程优先处理自己队列里的任务。这样可以大幅降低锁竞争——每个队列只需要自己的锁。

但会出现新问题：如果某个线程的队列空了，其他线程的队列可能还有大量任务。这个线程就会闲着。

### 方案 C：工作窃取（Work-Stealing）

工作窃取是方案 B 的改进：每个线程有自己的本地队列，但当自己的队列空时，可以去**偷**其他线程队列里的任务。

```
Thread 0 queue: [T1] [T2]
Thread 1 queue: [T3] [T4] [T5] [T6]
Thread 2 queue: empty

Thread 2 没活干 → 从 Thread 1 的队列尾部偷一个任务 T6
```

> [!abstract]
> **工作窃取（Work-Stealing）**的核心思想是：每个工作线程优先处理自己队列里的任务，空闲时从其他线程的队列尾部窃取任务，以此平衡负载并减少全局同步。

## 问题 3：窃取队列为什么要做成双端队列？

工作窃取队列（work-stealing deque）通常设计成**双端队列（deque）**，并且两端的行为不一样：

- **本地线程**：从队列的**尾部（tail）**`push` 和 `pop`。
- **其他窃取线程**：从队列的**头部（head）**`steal`。

这样设计有两个好处：

1. **减少冲突**：本地线程操作尾部，窃取线程操作头部，两者大多数情况下不会碰到一起。
2. **LIFO 对本地友好**：线程自己 `pop` 最新任务，有利于 cache 局部性（刚生成的子任务往往和父任务共享数据）。

无锁实现通常用 Chase-Lev 算法，它用一个环形数组 + 原子指针实现。C++ 标准库没有直接提供无锁 work-stealing deque，所以工业引擎大多自己实现。

> [!warning]
> 工作窃取不是万能药。窃取本身需要同步（即使是原子的），如果任务粒度太细，窃取的 overhead 会超过收益。任务粒度要足够大，才能让「窃取一次」摊销掉同步成本。

## 问题 4：什么是任务亲和性？

有些任务不适合被任意线程执行。比如：

- 一个 OpenGL / D3D 上下文只能被创建它的线程使用。
- 一个音频缓冲区只能在音频线程填充。
- 某个线程已经缓存了大量数据，换线程执行会导致 cache miss。

这种「任务偏好由某个特定线程或某类线程执行」的性质，叫做**任务亲和性（Task Affinity）**。

设计线程池时，常见的亲和性策略：

| 策略 | 含义 | 适用场景 |
|------|------|---------|
| 无亲和性 | 任意线程可执行 | 通用计算 |
| 线程索引亲和 | 指定必须由某个线程 ID 执行 | IO 完成端口、渲染上下文 |
| NUMA 节点亲和 | 任务必须运行在特定 CPU 插槽上 | 大规模并行、内存局部性敏感 |

```cpp
// flags: -O0 -g
// 示意：带线程索引亲和的任务包装
struct AffinityTask {
    std::function<void()> task;
    int preferred_thread = -1;  // -1 表示任意线程
};
```

> [!tip]
> 游戏引擎里最常见的亲和性例子是「主线程必须执行渲染提交」和「音频线程必须独立」。设计线程池 API 时，最好在一开始就留出亲和性参数，而不是后期硬塞。

## 问题 5：线程池应该开多少个线程？

这是最容易拍脑袋的地方。常见的经验法则：

| 场景 | 推荐线程数 | 原因 |
|------|-----------|------|
| CPU 密集型任务 | `std::thread::hardware_concurrency()` | 超过 CPU 核心数只会增加上下文切换 |
| IO 密集型任务 | 核心数 × 2 或更多 | 线程阻塞在 IO 时，其他线程可以继续跑 |
| 混合负载 | 需要实测 + 动态调整 | 没有统一公式 |

但 `hardware_concurrency()` 只是 hint，可能返回 0。而且现代 CPU 有超线程（Hyper-Threading），一个物理核心可以跑两个逻辑线程。对于 CPU 密集型任务，有时绑定到物理核心比超线程效果更好。

> [!abstract]
> **线程数没有银弹**。它取决于任务是 CPU 密集还是 IO 密集、是否需要亲和性、系统上还有哪些线程在竞争。上线前必须压测。

## 问题 6：手写 Day 66 线程池要抓住哪些点？

回到练习目标：手写一个固定大小、带任务队列的线程池。不必一步到位做到工作窃取，但要理解这些设计维度：

1. **队列 + 锁 + 条件变量**三件套。
2. **谓词等待**：`cv.wait(lock, [this] { return stop_ || !tasks_.empty(); })`。
3. **停止机制**：设置 `stop_` 后 `notify_all`，让线程安全退出。
4. **执行任务前释放锁**：避免阻塞其他线程入队。
5. **返回 future**（可选但推荐）：用 `std::packaged_task` 包装任务，让调用方异步拿结果。关于 future/promise 的用法，参见 [[异步编程模型与 future-promise#future 和 promise 是什么关系？|异步编程模型与 future/promise]]。

## 总结

- **线程池**通过复用固定数量的线程，避免线程创建销毁开销和资源爆炸。
- 最简单的线程池是「单一全局队列 + 锁 + 条件变量」，但锁竞争是瓶颈。
- **工作窃取**让每个线程有自己的本地队列，空闲时从其他线程队列头部偷任务，是工业级线程池的标配。
- **任务亲和性**不可忽视：渲染、音频、IO 上下文往往要求特定线程执行。
- 线程数要按负载类型选择，CPU 密集 ≈ 核心数，IO 密集可以更多，最终靠压测。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 任务调度、资源加载、并行 System | 每个工作线程维护本地双端队列，主线程提交任务时按亲和性路由；空闲线程窃取其他队列尾部任务；渲染提交任务绑定到主线程 |
| **UE** | `FQueuedThreadPool`、`FTaskGraph` | UE 提供多种线程池实现；任务图系统支持任务依赖、亲和性标记和优先级；大规模场景下通过工作窃取和 NUMA 感知减少调度开销 |

> [!note] 关键取舍
> 教学代码用全局锁队列就够了，但工业引擎几乎一定会走向「本地队列 + 工作窃取」。理解两者的 trade-off，才能根据项目阶段做出合理选择：先让功能正确，再让性能 scale。
