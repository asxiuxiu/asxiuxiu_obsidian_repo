# 条件变量与虚假唤醒

> [!info] 一句话概括
> **条件变量**让线程可以「 sleeping 等待某个条件成立」，而不是忙等浪费 CPU；但等待必须配合**谓词循环**，因为线程可能被**虚假唤醒**（spurious wakeup）——不是条件真的满足了，而是操作系统把你叫醒了。

---

## 问题 0：为什么需要条件变量？

想象你写了一个任务队列，一个工作线程从队列里取任务执行：

```cpp
std::queue<Task> queue;
std::mutex mtx;

void worker() {
    while (true) {
        std::lock_guard<std::mutex> lock(mtx);
        if (!queue.empty()) {
            auto task = queue.front();
            queue.pop();
            lock.unlock();
            task.run();
        }
        // else 怎么办？队列空了就只能继续循环
    }
}
```

如果队列长时间为空，这个线程会**忙等（busy-wait）**：不停加锁、检查、解锁，白白消耗 CPU。

更好的做法是：当队列为空时，线程**阻塞自己**，等有新任务时再被叫醒。这就是条件变量做的事。

> [!abstract]
> **条件变量（Condition Variable）**是一种同步原语，它允许一个线程在持有锁的情况下等待某个条件成立，并在等待期间释放锁、让出 CPU；当其他线程改变了条件后，再通知等待线程醒来检查。

## 问题 1：条件变量的基本用法

C++ 标准库提供 `std::condition_variable`，通常和 `std::mutex` 一起使用：

```cpp
// flags: -O0 -g
#include <iostream>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>

struct Task {
    int id;
    void run() { std::cout << "run task " << id << "\n"; }
};

std::queue<Task> queue;
std::mutex mtx;
std::condition_variable cv;
bool stop = false;

void producer() {
    for (int i = 0; i < 5; ++i) {
        {
            std::lock_guard<std::mutex> lock(mtx);
            queue.push({i});
        }
        cv.notify_one();  // 通知一个等待线程
    }

    {
        std::lock_guard<std::mutex> lock(mtx);
        stop = true;
    }
    cv.notify_all();  // 通知所有等待线程退出
}

void worker() {
    while (true) {
        std::unique_lock<std::mutex> lock(mtx);
        cv.wait(lock, [] { return !queue.empty() || stop; });  // 带谓词的等待

        if (stop && queue.empty()) break;

        auto task = queue.front();
        queue.pop();
        lock.unlock();
        task.run();
    }
}

int main() {
    std::thread t1(worker);
    std::thread t2(producer);
    t1.join();
    t2.join();
    return 0;
}
```

### 关键流程

1. **等待方**：
   - 获取锁。
   - 检查条件是否满足；不满足则调用 `cv.wait(lock, predicate)`。
   - `wait` 会自动释放锁并阻塞；被通知后重新获取锁，再次检查谓词。

2. **通知方**：
   - 获取锁。
   - 修改共享状态（如入队任务）。
   - 释放锁（或在 lock 作用域内）调用 `cv.notify_one()` 或 `cv.notify_all()`。

> [!warning]
> 通知时是否持有锁，取决于具体场景。持有锁通知更安全（保证通知时状态已稳定），但可能增加唤醒线程的等待时间。`std::condition_variable` 的 `wait` 在退出时会重新加锁，所以通知方最好已经释放锁或即将释放。

## 问题 2：什么是虚假唤醒？

虚假唤醒指的是：**线程在没有被 `notify` 的情况下也可能从 `wait` 中返回**。

原因包括：
- 操作系统或硬件层面的信号/中断。
- `notify_all` 唤醒多个线程，但只有一个能拿到锁，其他线程拿到锁后发现条件已经不满足。
- 某些平台实现上的效率优化。

如果你这样写：

```cpp
// 错误示范
cv.wait(lock);
auto task = queue.front();  // 可能队列还是空的！
```

一旦被虚假唤醒，就会访问空队列，导致未定义行为。

### 正确写法：谓词循环

```cpp
// 正确示范
cv.wait(lock, [] { return !queue.empty(); });
auto task = queue.front();
```

`wait(lock, predicate)` 等价于：

```cpp
while (!predicate()) {
    cv.wait(lock);
}
```

即使被虚假唤醒，只要谓词不满足，就会继续等待。

> [!abstract]
> **谓词循环（Predicate Loop）**是使用条件变量的黄金法则：无论什么原因醒来，都要重新检查条件是否真的满足。

---

## 问题 3：`notify_one` 还是 `notify_all`？

| 方法 | 含义 | 使用场景 |
|------|------|---------|
| `notify_one()` | 唤醒一个等待线程 | 只有一个任务，任意一个工作线程处理即可 |
| `notify_all()` | 唤醒所有等待线程 | 多个线程等待不同条件，或需要全部退出（如 shutdown） |

> [!tip]
> 如果你不确定用哪个，通常 `notify_one` 就够了。只有当多个等待线程需要同时重新评估条件时（比如状态从「可用」变成「不可用」 affecting 所有读者），才需要 `notify_all`。

## 问题 4：手写线程池时要注意什么？

Day 66 的手写线程池会大量用到条件变量。核心要点：

1. **任务队列 + mutex + condition_variable 三件套**。
2. **谓词循环**：`cv.wait(lock, [this] { return stop_ || !tasks_.empty(); })`。
3. **停止机制**：设置 `stop_` 标志后调用 `notify_all`，让线程池安全退出。
4. **避免在持有锁时执行任务**：取出任务后先 `unlock`，再运行任务，否则其他线程无法入队。

---

## 总结

- 条件变量让线程可以阻塞等待条件，避免忙等。
- 使用条件变量必须配合**谓词循环**，防止虚假唤醒导致错误。
- `notify_one` 唤醒一个线程，`notify_all` 唤醒所有线程。
- 手写线程池时，取出任务后要先释放锁再执行，避免阻塞其他线程。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 任务系统、资源加载队列 | 工作线程通过条件变量等待任务，主线程提交任务后 `notify_one`；shutdown 时用 `notify_all` |
| **UE** | 任务图、异步加载 | UE 的任务系统底层使用条件变量或事件对象进行线程间等待与唤醒 |

> [!note] 关键取舍
> 条件变量比忙等省 CPU，但会带来线程切换开销。对于超低延迟场景（如音频线程、渲染提交），可能改用无锁队列 + 自旋/混合策略。理解条件变量是设计这些调度策略的基础。
