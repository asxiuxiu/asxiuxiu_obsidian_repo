# 原始内存操作与对象生命周期的边界

> [!info] 一句话概括
> `memcpy`、`memset`、`memmove` 等原始内存操作是引擎中的「重型搬运机械」——它们无视类型、无视构造、无视虚表，直接按字节读写内存。用对了能救性能，用错了会让对象在编译器眼皮底下悄无声息地崩掉。

---

## 问题 0：引擎里要批量搬运数据，直接用 `=` 或 `std::copy` 不行吗？

想象你在写一个粒子系统的序列化模块。每帧有几万个粒子需要保存到回放缓冲区，或者从网络包中直接还原。你写了这样的代码：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>
#include <vector>

struct Particle {
    float x, y, z;
    float vx, vy, vz;
    int   lifetime;
};

int main() {
    std::vector<Particle> src(10000);
    std::vector<Particle> dst(10000);

    // naive：逐个赋值
    for (size_t i = 0; i < src.size(); ++i) {
        dst[i] = src[i];
    }

    std::cout << "copied " << src.size() << " particles\n";
    return 0;
}
```

这段代码能跑，但在引擎里这可能是性能自杀。`dst[i] = src[i]` 会调用 `Particle` 的拷贝赋值运算符——虽然在这个结构体上编译器很可能内联为逐字段复制，但一旦 `Particle` 稍微复杂一点（比如加入 `std::string` 名字标签），每次赋值都会触发深拷贝的堆分配。

更致命的场景是**自定义分配器**。你的帧分配器（FrameArena）返回的是一块**未初始化**的原始内存：

```cpp
void* raw = frameArena.Allocate(sizeof(Particle) * count);
// raw 指向的内存里全是垃圾值，没有任何对象存在
```

这时候你甚至不能写 `dst[i] = src[i]`，因为 `dst[i]` 在 C++ 的语义里是一个对象，而 `raw` 里还没有构造出任何对象。对未初始化内存调用 `operator=` 是**未定义行为**。

这时候你就需要一种「不调用构造函数、不检查类型、直接把字节搬过去」的操作——这就是 `std::memcpy` 存在的意义。它不关心 `Particle` 里有什么，只关心「从地址 A 搬 N 个字节到地址 B」。

但「不关心类型」是一把双刃剑。它跑得飞快，但也意味着编译器不会帮你检查任何错误。当你从网络包 `memcpy` 进一个结构体时，如果协议里字段顺序和编译器实际布局不一致，数据就会错位填充——这一点在 [[Notes/C++编程/对象内存布局：从 struct 到 class|对象内存布局]] 里已经踩过坑了。

知道了 `memcpy` 为什么必要，下一个自然的问题是：既然它这么快，我能不能把引擎里所有「复制」都换成 `memcpy`？

---

## 问题 1：`memcpy` 这么快，是不是所有数据复制都可以用它？

假设你的粒子系统升级了，现在每个粒子带一个名字：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>
#include <string>

struct NamedParticle {
    float x, y, z;
    std::string name;  // 不再是 POD
};

int main() {
    NamedParticle a;
    a.x = 1.0f; a.name = "explosion";

    NamedParticle b;
    std::memcpy(&b, &a, sizeof(NamedParticle));

    std::cout << "b.name = " << b.name << "\n";
    return 0;
}
```

如果你运行这段代码，大概率会在 `main` 退出时崩溃。`std::string` 内部通常有一个指向堆内存的指针（存储真正的字符数据）。`memcpy` 把 `a.name` 的**内部指针**原样复制到了 `b.name` 里——现在 `a` 和 `b` 的 `name` 指向了**同一块堆内存**。

当 `main` 结束时，`b` 先析构，释放了那块堆内存；然后 `a` 析构，**再次释放同一块内存**——双重释放，程序崩溃。

这就是 `memcpy` 的核心限制：它只做**浅拷贝**（按字节复制），完全不管对象内部的资源所有权。如果一个类型管理着堆内存、文件句柄、网络连接，或者有虚函数，用 `memcpy` 复制它几乎一定会炸。

### 什么类型可以安全地用 `memcpy`？

C++11 引入了一个概念来回答这个问题：**trivially copyable（可平凡复制）**。一个类型是 trivially copyable，当且仅当编译器可以用 `memcpy` 安全地复制它。简单来说，就是没有虚函数、没有自定义拷贝/移动/析构函数、所有成员和基类也都是 trivially copyable 的类型。

你可以用 `std::is_trivially_copyable` 在编译期检查：

```cpp
// flags: -O0 -g
#include <iostream>
#include <type_traits>
#include <string>

struct PlainParticle {
    float x, y, z;
    int   lifetime;
};

struct NamedParticle {
    float x, y, z;
    std::string name;
};

int main() {
    std::cout << std::boolalpha;
    std::cout << "PlainParticle is trivially_copyable: "
              << std::is_trivially_copyable_v<PlainParticle> << "\n";
    std::cout << "NamedParticle is trivially_copyable: "
              << std::is_trivially_copyable_v<NamedParticle> << "\n";
    return 0;
}
```

输出会是 `true` 和 `false`。在引擎代码中，ECS 的组件数组之所以要求组件必须是 trivially copyable 的，就是为了允许批量 `memcpy` 搬运——这是 SoA 布局性能的关键之一。

但还有一个更隐蔽的陷阱：`memcpy` 不调用构造函数，意味着**被复制的目标内存必须是已经存在的对象**，或者你明确知道自己在做什么。如果你把数据 `memcpy` 进一块未初始化的内存，然后这块内存后来被当作正常对象析构，析构函数会看到一些「看起来像有效值但实际是垃圾」的状态。

那么问题来了：如果我想要把一块内存清零，用 `memset` 行不行？这和 `memcpy` 有什么相似的陷阱？

---

## 问题 2：`memset` 能一键清零，为什么在有虚函数的类里用它初始化会崩溃？

这是很多 C++ 程序员的经典踩坑。你写了一个基类，为了让所有成员有个确定的初始状态，在构造函数里用了 `memset`：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

class Entity {
public:
    Entity() {
        // naive：把整个对象清零
        std::memset(this, 0, sizeof(*this));
    }

    virtual void Update() {
        std::cout << "Entity::Update, health=" << health << "\n";
    }

    int health = 100;
    int armor  = 50;
};

class Player : public Entity {
public:
    void Update() override {
        std::cout << "Player::Update\n";
    }
};

int main() {
    Player p;
    p.Update();  // 这里会崩溃
    return 0;
}
```

这段代码在 `p.Update()` 时几乎一定会**段错误**。原因在 [[Notes/C++编程/虚函数与多态本质|虚函数与多态本质]] 里讲过：有虚函数的类，对象开头会藏着一个**虚指针（vptr）**，指向该类的虚函数表（vtable）。

当 `Player` 的构造函数执行时，编译器会先把 vptr 指向 `Player` 的虚表。然后你调用了 `std::memset(this, 0, sizeof(*this))`——这个操作**把 vptr 也清零了**。现在对象开头的 8 个字节（64 位下）全是 0，vptr 指向了地址 `0x0`。当你调用 `p.Update()` 时，运行时通过 vptr 查虚表，去访问空指针，直接崩溃。

### `memset` 的真正适用场景

`memset` 只能安全地用于**纯字节数组**或**显式标记为 POD/trivial 的类型**。例如：

- 清零一个 `float` 数组
- 初始化一个自定义分配器返回的原始内存块（在构造对象之前）
- 重置一个网络包缓冲区

但绝不可以用在已经有对象语义的地方，尤其是对象已经构建完成、vtable 已经设置之后。

### `memset` 的另外一个隐蔽陷阱：浮点数的 0

```cpp
float f;
std::memset(&f, 0, sizeof(f));  // f 变成 0.0f

int x;
std::memset(&x, 0, sizeof(x));  // x 变成 0
```

凑巧 `memset` 到 0 对整数和 IEEE-754 浮点数都是有效的零值。但如果你写 `std::memset(&x, 1, sizeof(x))`，你得到的不是 `x = 1`，而是 `x = 0x01010101`（16843009）。`memset` 是按**字节**设置的，不是按**语义**设置的。这是很多人混淆它的根本原因。

好，现在你知道了 `memcpy` 不能乱用、`memset` 会踩虚表。但还有一个更微妙的边界：`memcpy` 和 `memmove` 看起来几乎一样，为什么标准库要提供两个？

## 问题 3：`memcpy` 和 `memmove` 有什么区别？内存重叠时会发生什么？

假设你有一个数组，需要把前面的元素往后挪一个位置：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

int main() {
    int arr[] = {1, 2, 3, 4, 5};

    // naive：想把 [1,2,3,4] 后移到 [2,3,4,5] 的位置
    std::memcpy(arr + 1, arr, 4 * sizeof(int));

    for (int x : arr) {
        std::cout << x << " ";
    }
    std::cout << "\n";
    return 0;
}
```

这段代码的行为是**未定义行为**。`memcpy` 的文档明确规定：**源和目标内存区域不能重叠**。重叠时，有些字节在被复制之前可能已经被覆盖了，导致结果错乱。不同编译器、不同优化级别下，输出可能完全不同。

正确的做法是使用 `std::memmove`：

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>

int main() {
    int arr[] = {1, 2, 3, 4, 5};

    // 正确：允许重叠
    std::memmove(arr + 1, arr, 4 * sizeof(int));

    for (int x : arr) {
        std::cout << x << " ";
    }
    std::cout << "\n";
    return 0;
}
```

`memmove` 的内部实现会检测重叠方向，决定是从前向后复制还是从后向前复制，确保每个源字节在被覆盖之前先被复制到目标位置。

### 引擎中的决策原则

| 场景 | 该用什么 |
|------|---------|
| 源和目标**确定不重叠** | `std::memcpy`（可能更快，编译器可能内联为 SIMD 指令） |
| 源和目标**可能重叠**或**不确定** | `std::memmove`（安全，内部做方向判断） |
| 需要同时移动和构造对象 | `std::uninitialized_move` 或 placement new |

> [!warning] 关键取舍
> `memcpy` 在 x86-64 上经常被编译器内联为 `rep movsb` 或 AVX 向量指令，性能极高。`memmove` 因为有重叠检测分支，理论上略慢。但在现代 CPU 的分支预测和微架构优化下，两者差距通常很小。**除非你在写极致性能的热路径代码并且已经 profile 确认瓶颈，否则优先用 `memmove` 更安全。**

## 问题 4：自定义分配器返回的原始内存，该如何安全地构造对象？

这是把前面所有问题串起来的核心场景。你的引擎有一个帧分配器：

```cpp
class FrameArena {
public:
    void* Allocate(size_t size);  // 返回未初始化的原始内存
    void  Reset();                 // 一帧结束，全部释放
};
```

你拿到一块原始内存，想在上面创建对象。最 naive 的做法是 `memcpy` 一个已有对象进去：

```cpp
void* raw = arena.Allocate(sizeof(Transform));
Transform existing = ...;
std::memcpy(raw, &existing, sizeof(Transform));  // 危险！
```

如果 `Transform` 不是 trivially copyable（比如它内部有一个 `std::string` 名字），这就是问题 1 里的双重释放陷阱。即使它是 trivially copyable，这块内存上的对象也没有经过**正式构造**——从 C++ 对象模型的角度，这块内存上还没有「对象」存在，你只是把字节摆对了。

### 正确的做法：placement new

C++ 提供了 **placement new**，它在**已分配的内存地址上**调用构造函数，而不是分配新内存：

```cpp
// flags: -O0 -g
#include <iostream>
#include <new>

struct Transform {
    float x, y, z;
    Transform(float x_, float y_, float z_) : x(x_), y(y_), z(z_) {
        std::cout << "Transform constructed\n";
    }
};

int main() {
    alignas(Transform) char buffer[sizeof(Transform)];

    // 在 buffer 的地址上构造 Transform
    Transform* t = new (buffer) Transform(1.0f, 2.0f, 3.0f);

    std::cout << "t->x = " << t->x << "\n";

    // 必须显式调用析构函数（因为内存不是 new 分配的）
    t->~Transform();

    return 0;
}
```

placement new 的关键点：
1. **不分配内存**，只在指定地址上构造
2. **必须配对显式析构**——因为内存不是从堆分配的，delete 不会知道如何释放
3. **内存必须对齐**——如果分配的内存没有满足类型的对齐要求，placement new 是未定义行为（C++17 起可以用 `std::aligned_alloc` 或 `alignas` 保证）

### 批量构造：从原始内存到对象数组

引擎中更常见的是批量构造 ECS 组件：

```cpp
// flags: -O0 -g
#include <iostream>
#include <new>
#include <cstring>

struct Vec3 {
    float x, y, z;
    Vec3() : x(0), y(0), z(0) {}
    explicit Vec3(float v) : x(v), y(v), z(v) {}
};

int main() {
    constexpr size_t N = 4;
    alignas(Vec3) char buffer[sizeof(Vec3) * N];

    Vec3* arr = reinterpret_cast<Vec3*>(buffer);

    // 逐个 placement new 构造
    for (size_t i = 0; i < N; ++i) {
        new (&arr[i]) Vec3(static_cast<float>(i));
    }

    for (size_t i = 0; i < N; ++i) {
        std::cout << arr[i].x << " ";
    }
    std::cout << "\n";

    // 逐个显式析构
    for (size_t i = 0; i < N; ++i) {
        arr[i].~Vec3();
    }

    return 0;
}
```

对于 trivially copyable 的类型（比如这个 `Vec3`），你可以**先 placement new 构造一个原型**，然后用 `memcpy` 批量复制到其他位置——这比每个元素都调用构造函数快得多：

```cpp
Vec3 prototype(1.0f);
new (arr) Vec3(prototype);  // 构造第一个

// 后面的用 memcpy 批量复制（前提是 trivially copyable）
for (size_t i = 1; i < N; ++i) {
    std::memcpy(&arr[i], &arr[0], sizeof(Vec3));
}
```

这也是引擎中 ECS 组件数组的常见优化路径：**trivially copyable 的组件允许批量 memcpy 初始化，非 trivial 的组件必须逐个 placement new 构造。**

## 总结

- **`memcpy` 只认字节，不认对象**。它不会调用任何构造函数或拷贝赋值，也不会管虚表指针、堆指针。只用于 trivially copyable 类型的复制，否则就是未定义行为。
- **`memset` 只认字节，不设语义**。它能清零内存，但如果对象已经构建（尤其是有虚函数的类），它会覆盖 vptr 导致崩溃。只用于原始缓冲区或 POD 数组。
- **`memcpy` 要求内存不重叠**，重叠时必须用 `memmove`。在不确定是否重叠的场景下，优先用 `memmove`。
- **placement new 是在已有内存上构造对象的唯一正确方式**。自定义分配器返回的原始内存，必须先经过 placement new 才有对象语义。对应的，析构时必须显式调用析构函数，不能用 `delete`。
- **trivially copyable 是引擎性能的分水岭**。ECS 组件、数学类型、网络协议结构体如果能保持 trivially copyable，就能享受批量 `memcpy`/`memset` 的极致性能；一旦引入虚函数或动态内存管理，就必须退回到逐个构造/析构的慢路径。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | ECS 组件数组的批量初始化 | 组件注册时检查 `std::is_trivially_copyable_v<T>`，trivial 组件允许 `memcpy` 批量复制，非 trivial 组件逐个 `placement new` 构造；帧分配器返回的内存一律通过 placement new 初始化 |
| **UE** | `UScriptStruct` 的内存操作与反射序列化 | UHT 生成的反射代码通过 `UScriptStruct::InitializeStruct` 和 `CopyScriptStruct` 操作内存，内部区分 POD 与非 POD 类型；POD 结构体走 `FMemory::Memcpy`，非 POD 走逐个构造/析构；`TMemStack` 等分配器返回的内存配合 placement new 使用 |

> [!note] 关键取舍
> SelfGameEngine 在 ECS 设计中对 trivially copyable 有硬性要求，因为每帧要遍历并批量复制数百万个组件，任何逐个构造的开销都会成为瓶颈。
> UE 的 `UObject` 体系接受非 trivial 类型（带有虚函数、反射信息、GC 标记），因为编辑器需要运行时类型信息和蓝图交互，性能换灵活性。
> 两套设计背后的分界线，正是「能否安全地使用原始内存操作」——理解这条线，才能判断你的类型该往哪边走。
