# 虚函数与多态本质

> [!info] 一句话概括
> 虚函数的本质是「编译器为每个类生成一张只读函数表（vtable），在每个对象头部插入一个指针（vptr）指向它」。动态绑定的代价是运行时两次内存读取加一次间接跳转——这也是现代引擎转向数据导向设计的底层动机。

---

## 问题 0：不用虚函数，怎么让基类指针调用不同子类的实现？

想象你在写一个游戏渲染系统。场景里有 `Mesh`、`Sprite`、`ParticleSystem`，它们都需要被渲染。你想写一个统一的接口：不管具体是什么类型，只要是个「可渲染对象」，就丢进同一个列表里批量绘制。

如果你完全不知道虚函数，可能会这样写：

```cpp
// flags: -O0 -g
#include <iostream>
#include <vector>

enum class RenderType { Mesh, Sprite, Particle };

struct Renderable {
    RenderType type;
    // ... 共享数据
};

void render(const Renderable* r) {
    switch (r->type) {
        case RenderType::Mesh:     /* 绘制网格 */ break;
        case RenderType::Sprite:   /* 绘制精灵 */ break;
        case RenderType::Particle: /* 绘制粒子 */ break;
    }
}

int main() {
    std::vector<Renderable*> scene;
    // ... 填充各种对象
    for (auto* obj : scene) {
        render(obj);
    }
    return 0;
}
```

这个方案能跑，但隐患很大：**每新增一种可渲染类型，你都要找到所有 `switch` 语句逐一修改**。你的渲染逻辑散落在各个 `case` 分支里，新增一个 `Terrain` 类型时，要改 `render()`、`cull()`、`getBounds()`……到处都要改。这违反了**开闭原则**——对扩展开放，对修改关闭。

更糟的是，如果两个程序员各自新增一种类型，很容易漏改对方的 `switch`，运行时进了 `default` 分支，莫名其妙地没渲染出来。

> 有没有一种办法，让「调用哪个实现」这个决策从「程序员手工维护的 switch」变成「对象自己知道该调用谁」？

## 问题 1：虚函数的底层机制——编译器在背后塞了什么？

C++ 给出的方案是**虚函数**。你只要在基类里把一个函数标记为 `virtual`，编译器就会在背后做两件事：

1. **为每个含虚函数的类生成一张虚函数表（vtable）**
2. **在每个对象头部悄悄插入一个虚表指针（vptr）**

### vtable：类的「电话簿」

vtable 本质上是一个**函数指针数组**，每个 slot 对应一个虚函数。编译器在编译期就确定好了每个虚函数在表中的偏移位置。比如：

```cpp
class Base {
public:
    virtual void foo() { std::cout << "Base::foo\n"; }
    virtual void bar() { std::cout << "Base::bar\n"; }
};

class Derived : public Base {
public:
    void foo() override { std::cout << "Derived::foo\n"; }
    // bar 继承 Base 的实现，vtable 中仍指向 Base::bar
};
```

编译器生成的 vtable 布局大致如下：

```
Base::vtable    →  [ &Base::foo , &Base::bar  ]
                      偏移0        偏移8（64位）

Derived::vtable →  [ &Derived::foo , &Base::bar  ]
                      偏移0          偏移8
```

注意一个关键事实：**vtable 是「每个类」一张，不是「每个对象」一张**。创建 1000 个 `Derived` 对象，内存里仍然只有一张 `Derived::vtable`，所有对象共享它。这张表存放在进程的**只读数据段**（Linux 的 `.rodata`，Windows 的 `.rdata`），运行时不可修改。

### vptr：对象的「指南针」

每个包含虚函数的对象，在内存布局的最开头（通常偏移为 0）会多出一个指针：

```
Base 对象：
┌─────────┬────────────┐
│  vptr   │  Base数据  │
│ (8字节) │            │
└────┬────┴────────────┘
     │
     ▼
Base::vtable [ &Base::foo, &Base::bar ]

Derived 对象：
┌─────────┬────────────┬──────────────┐
│  vptr   │  Base数据  │ Derived数据  │
│ (8字节) │            │              │
└────┬────┴────────────┴──────────────┘
     │
     ▼
Derived::vtable [ &Derived::foo, &Base::bar ]
```

你可以用代码验证 vptr 对对象大小的影响：

```cpp
// flags: -O0 -g
#include <iostream>

class NoVirtual {
    int x;
};

class WithVirtual {
    int x;
public:
    virtual void foo() {}
};

int main() {
    std::cout << "sizeof(NoVirtual)   = " << sizeof(NoVirtual) << "\n";
    std::cout << "sizeof(WithVirtual) = " << sizeof(WithVirtual) << "\n";
    return 0;
}
```

在 64 位系统上，通常输出 `4` 和 `16`（`int` 4 字节 + vptr 8 字节，再对齐到 8 字节）。这就是 vptr 的「空间税」。

### 动态绑定的调用链路

当你通过**指针或引用**调用虚函数时：

```cpp
Base* p = new Derived();
p->foo();  // 输出 "Derived::foo"
```

编译器生成的代码逻辑等价于：

```cpp
// 伪代码，展示编译器视角
(*p->__vptr[0])(p);  // 从对象取 vptr → 从 vtable 取第 0 个 slot → 调用
```

注意一个常见的误解：`p->foo()` 的**目标地址**是运行时确定的，但**偏移量（第几个 slot）**是编译期确定的。编译器在编译时就知道 `foo` 是 `Base` 声明的第一个虚函数，所以硬编码了偏移 0。运行时只决定「这张表是谁的表」。

> 如果通过**对象本身**调用虚函数（`obj.foo()`），编译器会生成静态绑定，直接 `call` 固定地址，不走 vtable。只有指针和引用才会触发动态绑定。

## 问题 2：对象构造和析构时，vptr 为什么会「变身」？

vtable 和 vptr 不是凭空出现的。vptr 由**构造函数写入**，而且在构造过程中**会改变指向**。

### 构造函数中的 vptr 切换

```cpp
class Base {
public:
    Base()  { /* ... */ }
    virtual void foo() { std::cout << "Base\n"; }
};

class Derived : public Base {
public:
    Derived() { /* ... */ }
    void foo() override { std::cout << "Derived\n"; }
};
```

当你执行 `Derived d;` 时，构造顺序等价于：

```cpp
// 编译器在 Base::Base() 开头插入：
// this->__vptr = &Base::vtable;
Base::Base() { /* 执行 Base 构造体 */ }

// 编译器在 Derived::Derived() 中、Base() 返回后插入：
// this->__vptr = &Derived::vtable;
Derived::Derived() { /* 执行 Derived 构造体 */ }
```

这意味着：**进入 `Base::Base()` 时，对象的 vptr 指向 `Base::vtable`；离开 `Base::Base()` 后，才切换成 `Derived::vtable`**。

这个设计不是多此一举，而是安全措施。如果在 `Base::Base()` 里调用了虚函数，此时 `Derived` 的成员还没有初始化，如果让虚函数「下降」到 `Derived::foo()`，`foo()` 可能会访问未初始化的成员导致崩溃。所以 C++ 规定：**构造期间，对象的动态类型就是当前正在构造的基类版本**。

用代码验证：

```cpp
// flags: -O0 -g
#include <iostream>

class Base {
public:
    Base() { foo(); }  // 构造期间调用虚函数
    virtual void foo() { std::cout << "Base::foo\n"; }
};

class Derived : public Base {
public:
    void foo() override { std::cout << "Derived::foo\n"; }
};

int main() {
    Derived d;  // 输出 "Base::foo"，不是 "Derived::foo"！
    return 0;
}
```

### 析构时的逆向过程

析构顺序与构造相反。编译器在 `Derived::~Derived()` 末尾插入代码，把 vptr 改回 `Base::vtable`，然后再调用 `Base::~Base()`。这保证了在基类析构体中调用虚函数时，不会误入已经销毁的派生类实现。

> **结论**：构造和析构期间，vptr 会「变身」两次。这是 C++ 的类型安全机制，但也意味着**不要在构造/析构函数中调用依赖于派生类实现的虚函数**——你得到的永远是当前层的版本。

## 问题 3：虚函数调用的 CPU 代价——为什么引擎要禁用虚函数？

vtable 机制解决了分发问题，但带来了运行时开销。从 CPU 视角看，一次虚函数调用和普通函数调用完全是两回事。

### 汇编层面的对比

**普通函数调用（静态绑定）**：

```asm
call 0x401240        ; E8 xx xx xx xx，目标地址直接编码在指令中
```

**虚函数调用（动态绑定）**：

```asm
mov  rax, [rbx]      ; 从对象首地址读取 vptr（1次内存访问）
mov  rax, [rax+0x8]  ; 从 vtable 读取第 N 个 slot（2次内存访问）
call rax             ; FF D0，间接跳转，目标地址在寄存器中
```

多了两次内存读取和一次间接跳转。这看起来不多，但在批量循环里会叠加成显著开销。

### 分支预测器视角：间接跳转是「盲猜」

CPU 的分支预测器对直接调用（`call 固定地址`）很友好——目标地址在取指阶段就知道了。但虚函数的 `call rax` 是**间接跳转**，目标地址藏在寄存器里，必须等前面两条 `mov` 执行完才能知道。

如果循环里处理的对象类型是混杂的：

```cpp
std::vector<Base*> mixed;  // Mesh*, Sprite*, Particle* 混在一起
for (auto* obj : mixed) {
    obj->render();  // 目标地址：Mesh::render? Sprite::render? Particle::render?
}
```

分支预测器看到的地址序列近乎随机，命中率暴跌。一旦猜错，流水线中预取的错误路径指令全部作废，重新取指的代价是 **10~20+ 个周期**。

### 指针追逐：Cache 不友好的根源

虚函数调用还伴随典型的**指针追逐**模式：

```
对象内存（L1 Cache?） → vptr（可能 Cache Miss） → vtable（可能跨缓存行） → 函数代码
```

每一步都可能触发缓存未命中。在现代 CPU 上，一次主内存访问需要 **100~300 个周期**，而一次寄存器访问只要 **~1 个周期**。

> 虚函数 = 用可预测的分支换取灵活的多态，代价是运行时查表 + 间接跳转 + 指针追逐。在游戏引擎这种追求确定性性能的领域，这个代价有时是不可接受的。

## 问题 4：虚析构函数为什么是「基类设计红线」？

vtable 不仅管活着时的调用，还管销毁时的行为。如果基类的析构函数不是虚函数，通过基类指针 `delete` 派生类对象时会出大问题。

```cpp
// flags: -O0 -g
#include <iostream>

class Base {
public:
    ~Base() { std::cout << "~Base\n"; }  // 非虚析构！
};

class Derived : public Base {
public:
    ~Derived() { std::cout << "~Derived\n"; }
};

int main() {
    Base* p = new Derived();
    delete p;  // 只输出 "~Base"，Derived 的析构被跳过了！
    return 0;
}
```

为什么？因为 `delete p` 时，如果 `~Base()` 不是虚函数，编译器直接生成静态绑定的 `call Base::~Base()`，根本不会查 vtable。`Derived::~Derived()` 永远不会执行，派生类里分配的资源（内存、文件句柄、GPU 资源）全部泄漏。

修复方法很简单：

```cpp
class Base {
public:
    virtual ~Base() = default;  // 虚析构
};
```

只要析构函数是虚的，`delete p` 就会走 vtable 查找，先调用 `Derived::~Derived()`，再调用 `Base::~Base()`，保证完整的析构链。

> [!danger] 设计红线
> 任何**打算被继承**的类，析构函数必须是 `virtual`（或 `= default` 让编译器生成虚析构）。不遵守这条规则，通过基类指针删除派生类对象的行为是未定义的。

### 纯虚函数在 vtable 里长什么样？

纯虚函数（`virtual void foo() = 0;`）对应的 vtable slot 通常填入一个**占位函数**（如 `__cxa_pure_virtual`），它的作用是在运行时被意外调用时直接终止程序，提示「你调用了纯虚函数」。

一个有趣的事实：**纯虚函数可以有实现**。`= 0` 只是告诉编译器「这个类是抽象类，不能实例化」，并不禁止你提供函数体：

```cpp
class Base {
public:
    virtual void foo() = 0;  // 纯虚
};

void Base::foo() {  // 合法！提供默认实现
    std::cout << "Base::foo (default)\n";
}

class Derived : public Base {
public:
    void foo() override {
        Base::foo();  // 可以调用基类的默认实现
        std::cout << "Derived::foo\n";
    }
};
```

> [!note] 我曾经怎么误解的
> 我一度以为「纯虚函数 = 没有实现，必须在派生类里重写」。实际上 `= 0` 控制的是**能否实例化**，不是**能否有函数体**。这个误解让我在设计接口基类时走了弯路——其实可以让纯虚函数提供默认行为，派生类按需调用。

## 问题 5：多重继承时，一个对象有几个 vptr？

单继承时对象只有一个 vptr，但多重继承时情况复杂得多：

```cpp
class Base1 {
public:
    virtual void f1() {}
};

class Base2 {
public:
    virtual void f2() {}
};

class Derived : public Base1, public Base2 {
public:
    void f1() override {}
    void f2() override {}
};
```

`Derived` 对象的内存布局大致是：

```
┌─────────┬──────────┬─────────┬──────────┐
│ vptr1   │ Base1数据 │ vptr2   │ Base2数据 │
│(Base1   │          │(Base2   │          │
│ vtable) │          │ vtable) │          │
└─────────┴──────────┴─────────┴──────────┘
         ↑
       this（Derived*）
```

**每个有虚函数的基类子对象都对应一个 vptr**。当你把 `Derived*` 转换成 `Base2*` 时，`this` 指针需要偏移（跳过 Base1 的部分），指向第二个 vptr 所在的位置。这就是「指针漂移」的底层原因，也是 `static_cast` 和 `dynamic_cast` 在多继承时行为不同的根源。

多重继承的复杂性——多个 vptr、this 指针调整、vtable 的交错布局——是现代 C++ 引擎**普遍禁用或严格限制多重继承**的底层原因。SelfGameEngine 和 UE 都倾向于用单继承 + 接口组合来替代。

## 总结

- **虚函数的本质是「运行时查表」**：编译器为每个类生成一张只读的 vtable，每个对象藏一个 vptr 指向它。动态绑定通过「对象 → vptr → vtable → 函数」完成，比静态绑定多两次内存访问和一次间接跳转。
- **vptr 在构造/析构期间会切换**：进入基类构造函数时指向基类 vtable，离开后才切到派生类 vtable。这保证了构造/析构期间的安全性，也意味着此时虚函数不会「下降」到派生类。
- **虚析构是基类的设计红线**：任何会被继承的类，析构函数必须是虚的，否则 `delete` 基类指针时派生类资源泄漏。
- **虚函数的 CPU 代价不可忽视**：间接跳转让分支预测器「失明」，指针追逐破坏 Cache 局部性。这是现代游戏引擎从 OOP 转向 ECS/DOD 架构的核心动机之一。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | ECS 组件存储禁用虚函数 | 组件数组追求 SoA 紧凑布局，vptr 会破坏数据连续性和缓存局部性。组件行为由 System 批量处理，类型在编译期或循环外确定 |
| **SelfGameEngine** | 需要多态语义时的零开销替代 | 组件基类使用 CRTP（奇异递归模板模式）实现静态多态，编译期确定调用目标，零运行时开销 |
| **UE** | `UObject` 的反射与序列化 | `UProperty`/`UFUNCTION` 宏配合 UHT 代码生成，在编译期生成反射信息表。虚函数大量用于 `UObject` 的派生类覆盖（如 `Tick`、`BeginPlay`） |
| **UE** | 编辑器工具链需要运行时多态 | 编辑器中需要运行时识别类型、动态绑定行为，虚函数 + 宏生成反射表是合理选择 |

> [!note] 关键取舍
> SelfGameEngine 在 ECS 组件存储中刻意避免虚函数，因为虚指针会破坏 SoA 布局的紧凑性，批量处理时 Cache Miss 和分支预测失败的代价不可接受。
> UE 则大量使用虚函数 + 宏生成反射信息，因为编辑器工具链需要运行时类型信息和动态派发能力。
> 同一个机制（虚函数），在两个引擎中做出了相反的选择——理解底层原理，才能判断什么场景该用、什么场景该禁。
