# 运算符重载的原则与陷阱

> [!info] 一句话概括
> **运算符重载** 让自定义类型也能用 `+`、`==`、`[]` 等符号表达操作。它本质上是一种社会契约：重载后的行为必须和人们对这个符号的直觉一致。

---

## 问题 0：为什么需要运算符重载？

假设你在写游戏数学库里的 `Vec2`：

```cpp
class Vec2 { public: float x, y; };
```

没有运算符重载时，向量加法只能写成函数调用；运算符重载让表达式回归自然：

```cpp
Vec2 f = gravity + velocity * dt + position;
```

> [!abstract]
> **运算符重载（Operator Overloading）** 是一种特殊函数，函数名由 `operator` 加运算符组成。它允许用户自定义类型以接近内置类型的方式参与运算。

---

## 问题 1：哪些运算符该重载？

C++ 中不能重载 `.`、`.*`、`::`、`?:`、`sizeof`。`&&` 和 `||` 能重载但不建议，因为会失去短路求值。

值语义类型（向量、矩阵、复数、颜色）通常值得重载：

| 类别 | 运算符 | 例子 |
| :-- | :-- | :-- |
| 算术 | `+`、`-`、`*`、`/` | `Vec3`、`Mat4` |
| 复合赋值 | `+=`、`-=`、`*=`、`/=` | 与算术运算符配套 |
| 比较 | `==`、`!=`、`<`、`<=`、`>`、`>=` | 排序、哈希键 |
| 一元 | `-`、`+` | 向量取反 |
| 下标 | `[]` | 容器、向量分量 |

## 问题 2：成员函数还是非成员函数？

### 成员函数破坏对称性

```cpp
// flags: -O0 -g
#include <iostream>

class Vec2 {
public:
    float x, y;
    Vec2(float x_, float y_) : x(x_), y(y_) {}
    Vec2 operator+(const Vec2& rhs) const {
        return Vec2{x + rhs.x, y + rhs.y};
    }
};

int main() {
    Vec2 a{1, 2}, b{3, 4};
    Vec2 c = a + b;   // ✅
    // Vec2 d = a + 1.0f;  // ❌ 右侧必须是 Vec2
    return 0;
}
```

成员函数版本的左侧必须是本类型，想让 `a + 1.0f` 合法就得引入隐式转换构造函数，带来新的风险。

### 非成员函数保持对称

```cpp
// flags: -O0 -g
#include <iostream>

class Vec2 {
public:
    float x, y;
    Vec2(float x_, float y_) : x(x_), y(y_) {}
};

Vec2 operator+(const Vec2& lhs, const Vec2& rhs) {
    return Vec2{lhs.x + rhs.x, lhs.y + rhs.y};
}

int main() {
    Vec2 a{1, 2}, b{3, 4};
    Vec2 c = a + b;            // ✅
    Vec2 d = a + Vec2{1, 1};   // ✅
    return 0;
}
```

### 常见分工

| 运算符 | 推荐形式 | 原因 |
| :-- | :-- | :-- |
| `+=`、`-=` 等 | 成员函数 | 直接修改 `this` |
| `+`、`-`、`==` 等 | 非成员函数 | 保持对称性 |
| `[]`、`()`、`->` | 成员函数 | 语法要求 |
| `<<`、`>>` | 非成员函数 | 左侧是标准库流对象 |

## 问题 3：与内置语义的一致性

C++ 不检查你重载 `+` 时是否真的在做加法。恶魔代码可以编译：

```cpp
// flags: -O0 -g
#include <iostream>

class Number {
public:
    int value;
    explicit Number(int v) : value(v) {}
};
Number operator+(const Number& lhs, const Number& rhs) {
    return Number{lhs.value - rhs.value};  // 😈 名为加法，实为减法
}
int main() {
    Number a{5}, b{3};
    std::cout << (a + b).value << "\n";  // 输出 2
    return 0;
}
```

一致性要求：`==` 是等价关系，`<` 是严格弱序，`+` 满足交换律，`+=` 执行后状态与 `a + b` 一致。

## 问题 4：自赋值与自增前后置

### 自赋值检查

```cpp
// flags: -O0 -g
#include <iostream>
#include <cstring>
#include <cstddef>

class Buffer {
    char* data_;
    size_t size_;
public:
    Buffer(const char* str) {
        size_ = std::strlen(str);
        data_ = new char[size_ + 1];
        std::strcpy(data_, str);
    }
    ~Buffer() { delete[] data_; }
    Buffer& operator=(const Buffer& rhs) {
        if (this == &rhs) return *this;  // ✅ 自赋值检查
        char* new_data = new char[rhs.size_ + 1];
        std::strcpy(new_data, rhs.data_);
        delete[] data_;
        data_ = new_data;
        size_ = rhs.size_;
        return *this;
    }
};

int main() {
    Buffer a("hello");
    a = a;
    return 0;
}
```

没有自赋值检查，`delete[] data_` 会释放掉自己还要拷贝的源内存。更优雅的做法是用 [[Notes/C++编程/构造、析构与拷贝控制/拷贝并交换惯用法|拷贝并交换惯用法]]。

### 自增前后置

```cpp
// flags: -O0 -g
#include <iostream>

class Counter {
    int value_;
public:
    explicit Counter(int v) : value_(v) {}
    Counter& operator++() {       // 前置：返回引用
        ++value_;
        return *this;
    }
    Counter operator++(int) {     // 后置：返回旧值副本
        Counter old(*this);
        ++value_;
        return old;
    }
    int value() const { return value_; }
};

int main() {
    Counter c(0);
    Counter& r = ++c;      // r 引用 c，c == 1
    Counter old = c++;     // old == 1, c == 2
    std::cout << c.value() << " " << old.value() << "\n";
    return 0;
}
```

| 版本 | 签名 | 返回值 | 效率 |
| :-- | :-- | :-- | :-- |
| 前置 `++x` | `T& operator++()` | 引用 | 高 |
| 后置 `x++` | `T operator++(int)` | 临时对象 | 较低 |

优先使用前置自增，除非确实需要旧值。

## 问题 5：类型转换运算符与 explicit

隐式类型转换运算符会在意想不到的地方触发：

```cpp
// flags: -O0 -g
#include <iostream>

class Vec2 {
public:
    float x, y;
    Vec2(float x_, float y_) : x(x_), y(y_) {}
    // operator bool() const { return x != 0.0f || y != 0.0f; }      // 隐式：危险
    explicit operator bool() const { return x != 0.0f || y != 0.0f; } // 显式：安全
};

int main() {
    Vec2 v{1, 0};
    if (v) { std::cout << "non-zero\n"; }  // ✅ 布尔上下文仍可用
    // int n = v + 5;                       // ✅ 编译错误（显式转换阻止）
    bool b = static_cast<bool>(v);          // ✅ 显式转换
    return 0;
}
```

> [!abstract]
> **explicit 转换运算符** 在 `operator Type()` 前加 `explicit`，阻止大多数隐式转换，但保留 `static_cast` 和布尔上下文中的显式使用。

它与 [[Notes/C++编程/类型系统与类型推导/explicit 关键字|explicit 与隐式转换构造函数]] 是对偶的：一个控制「其他类型 → 本类型」，一个控制「本类型 → 其他类型」。

## 问题 6：Day 22 练习关注什么？

手写 `Vec2`/`Vec3` 时的设计决策：

| 决策点 | 建议 |
| :-- | :-- |
| `+`、`-`、`*`、`/` | 非成员函数，保持对称 |
| `+=`、`-=` 等 | 成员函数，直接修改 `this` |
| `==`、`!=` | 非成员函数，分量比较 |
| 数乘 `v * 2.0f` | 同时提供 `operator*(Vec2, float)` 和 `operator*(float, Vec2)` |
| 下标 `v[0]` | 成员函数，返回引用 |
| 类型转换 | 尽量显式，避免隐式转换链 |

---

## 总结

- 运算符重载是一种**社会契约**，行为必须与符号直觉一致。
- 二元运算符优先用**非成员函数**，保证左右操作数对称。
- 自赋值检查是资源管理类的必要防御，拷贝并交换是更优雅的实现。
- 前置自增返回引用、效率更高；后置自增返回旧值副本。
- 类型转换运算符尽量加 `explicit`，避免隐式转换链导致意外行为。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
| :-- | :-- | :-- |
| **SelfGameEngine** | 数学库 `Vec2`/`Vec3`/`Mat4`/`Quat` | 算术运算符用非成员函数保证对称；比较运算符配合 `constexpr`；类型转换严格显式 |
| **UE** | `FVector`、`FMatrix`、`FQuat` | 运算符遵循分量级数学语义；`operator==` 常配合 `IsNearlyEqual` 处理浮点误差 |


> [!note] 关键取舍
> 游戏引擎中的数学类型通常禁止隐式类型转换，因为一次意外的 `float → Vec3` 转换可能导致渲染结果完全错误。理解运算符重载的对称性和语义一致性，才能避免「语法漂亮、运行时爆炸」的陷阱。
