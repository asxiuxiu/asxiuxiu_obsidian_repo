# 观察者模式与委托

> [!info] 一句话概括
> **观察者模式**让一组对象被动响应另一个对象的状态变化，而不需要彼此硬编码依赖；**委托（Delegate）**则是把「响应逻辑」封装成可插拔的回调对象，是引擎事件系统中实现观察者模式的核心工具。

---

## 问题 0：观察者模式解决什么问题？

想象你在写一个游戏菜单系统。当玩家点击「设置」按钮时，很多事情要发生：

- 打开设置界面。
- 暂停游戏逻辑。
- 播放点击音效。
- 记录一个埋点事件。

如果你把所有逻辑都写在按钮的点击处理函数里：

```cpp
void SettingsButton::on_click() {
    open_settings_ui();
    pause_game();
    play_click_sound();
    report_event("open_settings");
}
```

按钮就耦合了界面系统、游戏系统、音频系统、埋点系统。每新增一个响应，你都要修改按钮类。这在大型引擎里是不可维护的。

> **观察者模式（Observer Pattern）** 解决的就是这种「一对多依赖」的解耦问题：一个对象（Subject）状态变化时，自动通知一组依赖它的对象（Observers），而不需要 Subject 知道 Observers 具体是谁。

---

## 问题 1：Subject 和 Observer 是什么关系？

观察者模式有两个核心角色：

- **Subject（被观察者 / 主题）**：持有状态变化，维护一个观察者列表，负责在状态变化时通知它们。
- **Observer（观察者）**：对 Subject 的某种变化感兴趣，注册到 Subject 上，收到通知后执行自己的响应逻辑。

一个直观的实现是：Subject 保存一组 `Observer*` 指针，状态变化时遍历调用每个 Observer 的 `on_notify()`。

```cpp
// flags: -O0 -g
#include <iostream>
#include <vector>

class Observer {
public:
    virtual void on_notify(const std::string& event) = 0;
    virtual ~Observer() = default;
};

class Subject {
    std::vector<Observer*> observers_;
public:
    void add_observer(Observer* o) { observers_.push_back(o); }

    void notify(const std::string& event) {
        for (Observer* o : observers_) {
            o->on_notify(event);
        }
    }
};

class SoundSystem : public Observer {
public:
    void on_notify(const std::string& event) override {
        std::cout << "[Sound] play sound for " << event << "\n";
    }
};

int main() {
    Subject button;
    SoundSystem sound;
    button.add_observer(&sound);
    button.notify("click");
    return 0;
}
```

这个实现已经体现了观察者模式的核心：**Subject 只负责广播，Observer 自己决定怎么响应**。

> [!abstract]
> **观察者模式**是一种行为设计模式，定义了对象之间的一对多依赖关系：当一个对象改变状态时，所有依赖它的对象都会收到通知并自动更新。

---

## 问题 2：传统观察者模式有什么痛点？

上面的实现虽然清晰，但在实际工程中有几个问题：

### 痛点 1：每个 Observer 都要继承接口

如果你想让一个普通函数、Lambda 或成员函数响应事件，必须先写一个继承 `Observer` 的包装类。这很啰嗦。

### 痛点 2：生命周期管理困难

如果 Observer 在 Subject 之前析构，Subject 里还保存着它的野指针，下次通知时就会崩溃。这个问题被称为**悬垂观察者（dangling observer）**。

```cpp
{
    SoundSystem sound;
    button.add_observer(&sound);
}  // sound 在这里析构

button.notify("click");  // ❌ 使用野指针，未定义行为
```

### 痛点 3：Observer 和事件类型强耦合

如果系统有很多种事件（点击、悬浮、取消），Observer 接口就要不断膨胀，或者 Observer 不得不处理自己根本不关心的事件。

---

## 问题 3：委托（Delegate）是什么？

**委托（Delegate）**可以理解为「一个封装好的回调对象」。它把「谁响应」这个问题从继承体系里解放出来：

- 可以绑定普通函数。
- 可以绑定 Lambda。
- 可以绑定成员函数 + 对象指针。
- 可以是单播（只存一个回调）或多播（存多个回调）。

委托让观察者模式不再强制要求 Observer 是一个类，而只需要是一个**可调用对象**。这正是 C++ 引擎里 Delegate 系统的价值。

> [!abstract]
> **委托（Delegate）**在引擎语境下通常指一个轻量级的回调封装对象。它把函数指针、Lambda、成员函数绑定等统一表示，使得事件源无需关心回调的具体形态。

---

## 问题 4：委托和普通回调有什么区别？

| 特性 | 普通函数指针回调 | 委托（Delegate） |
|------|----------------|----------------|
| 能否携带状态 | ❌ 一般不能 | ✅ 可以绑定成员函数、Lambda、捕获变量 |
| 类型安全 | 弱，靠约定 | 强，通常基于模板生成强类型委托 |
| 生命周期 | 调用方自己管 | 好的委托系统会提供解注册、弱引用、句柄等机制 |
| 多播能力 | 需要额外容器 | 原生支持 `MulticastDelegate` |

简单说：**回调是一个函数地址，委托是一个更完整的「调用对象 + 生命周期 + 分发机制」的封装**。

---

## 问题 5：生命周期管理——观察者先析构怎么办？

这是观察者模式最危险的地方。常见解决方案有几种：

### 方案 A：手动解注册

Observer 析构前主动从 Subject 移除自己。

```cpp
class SoundSystem : public Observer {
public:
    ~SoundSystem() {
        if (subject_) subject_->remove_observer(this);
    }
};
```

问题是：如果 Observer 忘了调用，或者析构顺序复杂，还是会崩溃。

### 方案 B：使用弱引用或句柄

Subject 不直接保存 Observer 指针，而是保存一个弱引用或委托句柄。通知前检查对象是否仍然有效。

```cpp
// 概念示意
template<typename Observer>
class WeakObserver {
    std::weak_ptr<Observer> ptr_;
public:
    void notify(const std::string& event) {
        if (auto sp = ptr_.lock()) {
            sp->on_notify(event);
        }
    }
};
```

这种方案在 UE 的 `TWeakObjectPtr`、自研引擎的句柄系统中很常见。

### 方案 C：委托句柄 + 自动失效

引擎里的 Delegate 通常返回一个句柄（Handle）。当 Observer 被销毁时，句柄自动失效，Subject 通知时会跳过失效项。

```cpp
// 概念示意
class DelegateHandle {
    uint32_t id_;
    std::shared_ptr<bool> alive_;  // 当 Observer 销毁时 alive_ 也销毁
};
```

> [!warning]
> 无论哪种方案，核心原则只有一个：**Subject 不能无条件信任它保存的 Observer 指针仍然有效**。在事件分发前必须做有效性检查，或者在架构上保证 Observer 活得比 Subject 长。

## 问题 6：引擎中的 Delegate 是怎么设计的？

工业级引擎的 Delegate 系统通常不是直接继承 `std::function`，而是基于自身的类型安全和生命周期需求重新设计。

### 单播委托（Single-cast Delegate）

只绑定一个回调，类似一个类型安全的 `std::function`。

```cpp
// 概念示意
Delegate<void(int)> on_click;
on_click.bind(&handler, &Handler::on_click);
on_click.execute(42);
```

### 多播委托（Multicast Delegate）

绑定多个回调，通知时逐个调用。观察者模式在引擎里通常就是用多播委托实现的。

```cpp
// 概念示意
MulticastDelegate<void(int)> on_click;
on_click.add(&handler1, &Handler::on_click);
on_click.add(lambda_callback);
on_click.broadcast(42);
```

### 关键设计点

1. **强类型**：委托签名在编译期确定，避免 `void*` 满天飞。
2. **生命周期安全**：通过句柄、弱引用或 RAII 包装自动处理 Observer 先死的情况。
3. **性能**：高频事件避免 `std::function` 的虚函数开销，可能用手写 SOO Function 或函数指针数组。
4. **解耦**：Subject 只暴露委托，Observer 只订阅委托，两者不需要知道彼此存在。

---

## 问题 7：虚函数在这里还重要吗？

重要。观察者模式最早的实现就是基于虚函数的多态：每个 Observer 继承接口，Subject 通过基类指针调用虚函数。理解虚函数的底层派发机制（虚表、虚指针、间接跳转），才能理解为什么引擎 Delegate 系统要费力做非虚函数优化。

如果你还不太清楚虚函数表和动态派发的代价，可以先读 [[Notes/C++编程/对象内存模型与底层机制/虚函数与多态本质|虚函数与多态本质]]。

## 问题 8：Day 25 练习需要抓住什么？

Day 25 的目标是手写一个事件系统（多播委托 + 观察者模式）。在动手前，建议先回答这几个问题：

1. **Subject 用什么数据结构保存委托？** `std::vector` 最简单，但删除时可能需要处理迭代器失效。
2. **委托怎么表示？** 手写类型擦除 Function，还是直接用 `std::function`？
3. **生命周期怎么处理？** 是否支持在通知过程中安全地添加/移除观察者？
4. **事件参数怎么传递？** 值传递、引用传递、还是打包成事件对象？

> [!tip]
> 初学者版本可以先用 `std::function` + `std::vector` 实现一个能跑的多播委托，重点体会「Subject 广播、Observer 订阅」的解耦思想。进阶版本再考虑手写 Delegate、生命周期句柄、广播过程中安全增删等工程细节。

## 总结

- **观察者模式**解耦了「事件发生方」和「事件响应方」，Subject 一对多通知 Observer。
- 传统观察者模式要求 Observer 继承接口，生命周期管理复杂，事件类型耦合强。
- **委托（Delegate）**把回调封装成可插拔对象，支持函数指针、Lambda、成员函数绑定，是引擎实现观察者模式的核心工具。
- 生命周期管理是事件系统的头号难题，常见方案包括手动解注册、弱引用、委托句柄。
- 引擎 Delegate 通常在手写类型擦除和生命周期安全之间做取舍，不是简单套用 `std::function`。

## 引擎映射

| 引擎 | 应用场景 | 具体体现 |
|------|---------|---------|
| **SelfGameEngine** | 模块间事件通信、输入系统、UI 回调 | 手写 `Delegate` / `MulticastDelegate`，内部用类型擦除 `Function` 存储回调；输入轴事件、按钮点击事件通过委托分发；生命周期用句柄或弱引用管理 |
| **UE** | `DECLARE_DELEGATE` / `DECLARE_MULTICAST_DELEGATE` / 动态多播委托 | 宏生成强类型委托类；`AddDynamic` / `BindDynamic` 绑定 UObject 成员函数；`TWeakObjectPtr` 处理 UObject 被垃圾回收后的委托失效；蓝图事件本质也是动态多播委托 |
| **Bevy** | ECS 事件系统、观察者系统（Observer System） | Rust 没有成员函数指针，事件通过 `Event<T>` 资源和 `EventReader` / `EventWriter` 分发；Observer 是响应组件变化的系统函数，生命周期由 ECS 的实体/组件系统自然管理 |

> [!note] 关键取舍
> 不同引擎对观察者模式的实现差异很大，但核心矛盾是一致的：**灵活性 vs 性能 vs 生命周期安全**。
> - 灵活性最强：`std::function` 或动态多播委托，任何可调用对象都能绑定。
> - 性能最好：函数指针数组或手写 SOO Function，避免虚函数间接跳转。
> - 生命周期最稳：弱引用、委托句柄、ECS 组件生命周期天然管理。
> 自研引擎时，要根据事件路径的热度选择合适的方案，而不是把所有事件都塞进同一个容器。
